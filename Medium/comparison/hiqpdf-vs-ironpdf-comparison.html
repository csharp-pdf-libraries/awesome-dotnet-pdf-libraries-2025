What is HiQPdf and how does it work?

HiQPdf is a commercial HTML-to-PDF library offering both a legacy WebKit engine (Classic) and a modern Chromium engine (Next). It targets .NET Standard 2.0, .NET Framework 4.6.2+, and .NET Core 9/8/7/6/5.

The product is actively developed but fragmented into multiple package families: HiQPdf (Classic, WebKit, Windows-only), HiQPdf.NG (new generation, still WebKit), HiQPdf.Next (Chromium-based, cross-platform), and HiQPdf.Client (multi-platform client/server). Each has different capabilities, platform support, and deployment requirements. The official NuGet profile lists 20+ packages; determining the correct combination requires documentation study.

Where it gets complicated is package selection. Classic and NG packages are Windows x64 only with an older WebKit engine that lacks modern CSS and JavaScript support. Next packages require separate runtime packages (Core.Windows, Core.Linux). The Client package requires a separate server component. Different feature sets exist across package families, so you must verify capabilities per package.

Problem 2 — Why does HTML rendering quality vary between HiQPdf engines?

Answer: You build your templates using modern CSS Grid and Flexbox, test them in Chrome, and they look perfect. Then you generate PDFs with HiQPdf Classic or NG and the layouts collapse. The WebKit engine in Classic and NG packages lacks modern CSS and JavaScript support — no CSS Grid, partial Flexbox, no ES6+. You switch to HiQPdf Next for its Chromium engine, but now you need different packages, different APIs, and platform-specific runtime dependencies.

The diagnostic path looks like this: first, determine which HiQPdf engine you're actually using (Classic WebKit vs Next Chromium). Then check whether the CSS features in your templates are supported by that engine. Classic and NG lack modern JavaScript (ES6+), CSS Grid, and have only partial Flexbox support. Next supports modern web standards but requires multi-package coordination.

IronPDF uses the Chromium rendering engine across all platforms from a single package. If your HTML displays correctly in Chrome print preview, IronPDF will match it exactly. HiQPdf Classic — HTML string to PDF (WebKit engine)

using HiQPdf;
using System;
using System.Diagnostics;

namespace HiQPdfClassicExample
{
    class Program
    {
        static void Main(string[] args)
        {
            // Note: Classic engine is Windows x64 only
            // Requires HiQPdf NuGet package (not HiQPdf.NG or HiQPdf.Next)

            var stopwatch = Stopwatch.StartNew();

            // Create converter instance
            HtmlToPdf htmlToPdfConverter = new HtmlToPdf();

            // Configure page settings
            htmlToPdfConverter.Document.PageSize = PdfPageSize.A4;
            htmlToPdfConverter.Document.PageOrientation = PdfPageOrientation.Portrait;
            htmlToPdfConverter.Document.Margins = new PdfMargins(10, 10, 10, 10);

            // Set rendering settings
            htmlToPdfConverter.BrowserWidth = 1024;
            htmlToPdfConverter.BrowserHeight = 768;

            // HTML content
            string htmlContent = @"
                <html>
                <head>
                    <style>
                        body { font-family: Arial; }
                        h1 { color: navy; }
                    </style>
                </head>
                <body>
                    <h1>Performance Test Invoice #12345</h1>
                    <table border='1'>
                        <tr><td>Item</td><td>Quantity</td><td>Price</td></tr>
                        <tr><td>Widget A</td><td>5</td><td>$25.00</td></tr>
                        <tr><td>Widget B</td><td>3</td><td>$15.00</td></tr>
                    </table>
                    <p><b>Total: $200.00</b></p>
                </body>
                </html>";

            // Convert HTML to PDF (synchronous blocking call)
            byte[] pdfBytes = htmlToPdfConverter.ConvertHtmlToMemory(htmlContent, null);

            stopwatch.Stop();

            // Save result
            System.IO.File.WriteAllBytes("output-classic.pdf", pdfBytes);

            Console.WriteLine($"HiQPdf Classic rendering time: {stopwatch.ElapsedMilliseconds}ms");
            // Typical: 80-150ms for simple HTML, 400-1200ms for complex pages
        }
    }
}

Problem 3 — How do you handle batch processing when HiQPdf lacks async and parallel support?

Answer: You need to generate hundreds or thousands of PDFs in a batch job. HiQPdf's API is synchronous — no async variants, thread blocks during conversion. Documentation doesn't explicitly demonstrate parallel batch processing patterns. You're stuck with serial processing, and for 100 simple documents that means 15-35 seconds depending on the engine.

The HiQPdf workaround is creating a single converter instance and reusing it across the batch to amortize initialization cost. You can try wrapping calls in Parallel.ForEach, but thread safety isn't explicitly documented. Memory accumulation is also a concern — monitor for large batches. There are no native async methods (ConvertHtmlToMemoryAsync) demonstrated in the documentation, limiting scalability in ASP.NET web scenarios.

IronPDF explicitly supports Parallel.ForEach for batch processing and provides native async methods (RenderHtmlAsPdfAsync, RenderUrlAsPdfAsync) for web applications. The Chromium engine is thread-safe and scales across cores. On a quad-core machine, parallel processing typically yields a 3-4x speedup over serial — 5-12 seconds for 100 simple documents compared to 15-35 seconds serial. For optimization techniques, see the API reference.

HiQPdf — serial batch rendering

using HiQPdf.Next.HtmlToPdf;
using System;
using System.Collections.Generic;
using System.Diagnostics;
using System.Linq;

namespace HiQPdfBatchExample
{
    class Program
    {
        static void Main(string[] args)
        {
            // Create converter instance once for batch
            var converter = new HtmlToPdfConverter();

            // Prepare batch of HTML documents
            var htmlBatch = Enumerable.Range(1, 100).Select(i =>
                $"<html><body><h1>Invoice #{i}</h1><p>Total: ${i * 10}.00</p></body></html>"
            ).ToList();

            var stopwatch = Stopwatch.StartNew();
            var results = new List<byte[]>();

            // Serial processing (no built-in parallel support demonstrated in docs)
            foreach (var html in htmlBatch)
            {
                var pdfBytes = converter.ConvertHtmlToMemory(html, new HtmlToPdfOptions());
                results.Add(pdfBytes);
            }

            stopwatch.Stop();

            Console.WriteLine($"HiQPdf Next batch (100 docs): {stopwatch.ElapsedMilliseconds}ms");
            Console.WriteLine($"Average per doc: {stopwatch.ElapsedMilliseconds / 100.0}ms");
            // Typical serial: 15-30 seconds for 100 simple docs
            // Parallel.ForEach may work but not explicitly documented
        }
    }
}

IronPDF — parallel batch rendering with async support

using IronPdf;
using System;
using System.Diagnostics;
using System.Linq;
using System.Threading.Tasks;

namespace IronPdfBatchExample
{
    class Program
    {
        static async Task Main(string[] args)
        {
            // Prepare batch of HTML documents
            var htmlBatch = Enumerable.Range(1, 100).Select(i =>
                $"<html><body><h1>Invoice #{i}</h1><p>Total: ${i * 10}.00</p></body></html>"
            ).ToList();

            var stopwatch = Stopwatch.StartNew();

            // Parallel processing using Parallel.ForEach (recommended for batch)
            var pdfResults = new System.Collections.Concurrent.ConcurrentBag<PdfDocument>();

            Parallel.ForEach(htmlBatch, new ParallelOptions { MaxDegreeOfParallelism = 4 }, html =>
            {
                var renderer = new ChromePdfRenderer();
                var pdf = renderer.RenderHtmlAsPdf(html);
                pdfResults.Add(pdf);
            });

            stopwatch.Stop();

            Console.WriteLine($"IronPDF parallel batch (100 docs): {stopwatch.ElapsedMilliseconds}ms");
            Console.WriteLine($"Average per doc: {stopwatch.ElapsedMilliseconds / 100.0}ms");
            // Typical parallel (4 cores): 5-12 seconds for 100 simple docs
            // ~3-4x speedup from parallelization on quad-core

            // Alternative: Async pattern for web applications
            var asyncStopwatch = Stopwatch.StartNew();
            var asyncTasks = htmlBatch.Select(async html =>
            {
                var renderer = new ChromePdfRenderer();
                return await renderer.RenderHtmlAsPdfAsync(html);
            });

            var asyncResults = await Task.WhenAll(asyncTasks);
            asyncStopwatch.Stop();

            Console.WriteLine($"IronPDF async batch (100 docs): {asyncStopwatch.ElapsedMilliseconds}ms");
            // Async pattern similar performance but better resource utilization in ASP.NET
        }
    }
}

FAQ and Troubleshooting Tips

Which HiQPdf package should I install for cross-platform support?

HiQPdf.Next.HtmlToPdf is the cross-platform option, but you also need platform-specific runtime packages — HiQPdf.Next.Core.Windows for Windows, HiQPdf.Next.Core.Linux for Linux. Version mismatch between the converter and runtime packages causes failures. IronPDF requires a single dotnet add package IronPdf command for all platforms — native libraries auto-select based on your target environment.

Can I migrate from HiQPdf Classic to HiQPdf Next without rewriting code?

Not as a drop-in replacement. Classic uses the HtmlToPdf class with Document.PageSize and Document.Margins properties. Next uses HtmlToPdfConverter with HtmlToPdfOptions for configuration. The API surface differs, so migration requires code changes. With IronPDF, you use ChromePdfRenderer and RenderingOptions consistently across all platforms — no engine-switching decision to revisit later.

Why is HiQPdf Next slower than Classic for simple HTML?

Classic uses an older WebKit engine that's typically faster for simple HTML (80-150ms) but lacks modern CSS and JavaScript support. Next uses Chromium, which adds overhead (150-300ms for simple HTML, 800-1500ms cold start) but handles modern web standards fully. IronPDF also uses Chromium (120-250ms for simple HTML, 700-1200ms cold start) with comparable warm performance to HiQPdf Next, but without the package fragmentation. See the headers and footers guide for optimizing rendering output.

Does HiQPdf support async/await for web application scenarios?

Neither HiQPdf Classic nor HiQPdf Next provide native async methods. Conversion calls are synchronous and block the calling thread. In ASP.NET applications, this limits scalability under concurrent load. IronPDF provides native async methods — RenderHtmlAsPdfAsync, RenderUrlAsPdfAsync — designed for web application scenarios where thread pool efficiency matters.

How do the API concepts map between HiQPdf and IronPDF?

The key mappings: HiQPdf Classic's HtmlToPdf and Next's HtmlToPdfConverter both map to IronPDF's ChromePdfRenderer. ConvertHtmlToMemory maps to RenderHtmlAsPdf. ConvertUrl maps to RenderUrlAsPdf. HiQPdf's Document.PageSize (Classic) or HtmlToPdfOptions.PdfPageSize (Next) maps to RenderingOptions.PaperSize. HiQPdf has no equivalent for IronPDF's native RenderHtmlAsPdfAsync. The full API mapping is available in the IronPDF API documentation.

When should you use HiQPdf vs IronPDF?

Use HiQPdf when you have a Windows-only deployment and want the fastest possible rendering for simple HTML content — Classic's WebKit engine at 80-150ms per page is hard to beat for basic layouts. If you're already invested in HiQPdf's ecosystem and your package coordination is stable, there may not be a compelling reason to migrate for existing projects.

Use IronPDF when you need unified cross-platform support without package coordination, clear async/await patterns for web applications, explicit parallel processing support for batch scenarios, or simpler deployment. IronPDF provides Chromium rendering across all platforms from a single NuGet package with performance comparable to HiQPdf Next (both use Chromium) but without the package fragmentation. Native async methods, explicit Parallel.ForEach support, 24/5 engineer chat support, and a comprehensive PDF manipulation API come standard.

The red flag: you're managing multiple HiQPdf packages across platforms, hitting version mismatches between core and runtime packages, or working around the lack of async APIs in a web application. The deployment matrix — different packages for Framework vs Core, Windows vs Linux, Classic vs Next — adds real operational risk during updates.

The green flag: your deployment is simple, Windows-only, and HiQPdf Classic's WebKit performance meets your needs. In that case, the library does what it says. But the moment you need cross-platform, async, or parallel batch support, the package coordination overhead starts to outweigh the performance benefits.
