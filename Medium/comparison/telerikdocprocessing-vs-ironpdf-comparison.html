Telerik Document Processing vs IronPDF in C#: Which Should Your Team Choose?

Your application needs to generate PDF reports from HTML dashboard exports, and a developer suggests Telerik Document Processing because "it has HTML support." Initial implementation reveals that HtmlFormatProvider imports HTML into a RadFlowDocument—an intermediate word-processing document model—which PdfFormatProvider then exports to PDF. Because the flow document model is a paginated word-processing model rather than a browser layout model, modern CSS layout features such as CSS Grid and Flexbox typically don't translate directly. The two-step conversion (HTML → flow document → PDF) introduces a layout-mapping step where complex HTML structures may not survive the intermediary format's constraints.

Teams I have worked with hit this pattern when they expect Telerik Document Processing to function as an HTML-to-PDF renderer when its architecture serves a different purpose. In this article I'll walk through the architectural differences, show you the code, and help you decide which library fits your workflow.

What Is Telerik Document Processing?

Telerik Document Processing is an actively maintained set of .NET document libraries from Progress Telerik with regular release updates. Recent releases have moved the supported .NET baseline forward; check the current Telerik release notes for the exact .NET versions supported by your version. The product ships as part of Telerik's broader DevCraft suite and is commercially supported.

The library is designed for programmatic document construction—drawing shapes, positioning text precisely, building tables from data structures—across PDF, Word, and Excel through RadFixedDocument, RadFlowDocument, and RadSpreadDocument. HTML-to-PDF is one of several supported flows rather than the primary design target. Telerik's HTML conversion is a two-step process: HtmlFormatProvider imports HTML into a RadFlowDocument (a word-processing flow model), then PdfFormatProvider exports that flow document to PDF. Because HTML elements must map to flow-document equivalents and CSS must translate to flow-document formatting, the layout constraints of the flow model define what HTML structures can be expressed in the output.

What Is IronPDF?

IronPDF uses Chromium—the same rendering engine that powers Google Chrome—to convert HTML directly to PDF. There's no intermediate flow-document format and no layout-model translation. The HTML you write renders the way Chrome renders it because it is using Chrome's rendering pipeline. CSS Grid, Flexbox, modern JavaScript, and web fonts work in the PDF the same way they work in a browser tab.

This direct rendering approach eliminates the troubleshooting loop of "why did my layout break during conversion?" If it displays correctly in Chrome, it will appear identically in the PDF. The ChromePdfRenderer class produces PDFs that match Chrome browser output exactly, and for teams building from HTML templates, this architectural match between development environment and output determines whether layout debugging is straightforward or perpetually mysterious.

How Does HTML-to-PDF Conversion Differ Between the Two?

This is where the architectural difference is most visible. Telerik converts HTML through a two-step pipeline (HTML → RadFlowDocument → PDF), while IronPDF renders HTML directly through Chromium in a single pass (HTML → PDF).

The difference matters most with modern CSS. CSS Grid, Flexbox, transforms, transitions, and animations do not have direct equivalents in the flow-document model, so they typically do not survive the import step. JavaScript does not execute either—single-page applications, dynamic content loading, and AJAX-populated sections will not run because the conversion does not include a browser engine. Verify against your specific Telerik version for any features that have been added.

Telerik Document Processing — HTML via Flow Document

using Telerik.Windows.Documents.Flow.FormatProviders.Html;
using Telerik.Windows.Documents.Flow.FormatProviders.Pdf;
using Telerik.Windows.Documents.Flow.Model;
using System.IO;
using System.Net.Http;
using System.Threading.Tasks;

public class TelerikHtmlConverter
{
    public async Task<byte[]> ConvertHtmlStringToPdfAsync(string htmlContent)
    {
        // Step 1: HTML → RadFlowDocument (flow document model)
        HtmlFormatProvider htmlProvider = new HtmlFormatProvider();
        RadFlowDocument flowDocument = htmlProvider.Import(htmlContent);

        // Step 2: RadFlowDocument → PDF
        PdfFormatProvider pdfProvider = new PdfFormatProvider();
        byte[] pdfBytes = pdfProvider.Export(flowDocument);

        return pdfBytes;
    }

    public async Task<byte[]> ConvertUrlToPdfAsync(string url)
    {
        // Download the HTML before passing it to HtmlFormatProvider
        using (var httpClient = new HttpClient())
        {
            string html = await httpClient.GetStringAsync(url);

            HtmlFormatProvider htmlProvider = new HtmlFormatProvider();
            htmlProvider.ImportSettings.LoadFromUri += (sender, args) =>
            {
                // Resolve relative URLs and supply resource bytes here
            };

            RadFlowDocument flowDocument = htmlProvider.Import(html);

            PdfFormatProvider pdfProvider = new PdfFormatProvider();
            return pdfProvider.Export(flowDocument);
        }
    }

    public async Task<byte[]> ConvertComplexHtmlAsync(string htmlWithModernCss)
    {
        // HTML using CSS Grid and Flexbox layout
        string html = @"
            <style>
                .grid-container {
                    display: grid;
                    grid-template-columns: 1fr 2fr 1fr;
                    gap: 20px;
                }
                .flex-item {
                    display: flex;
                    justify-content: space-between;
                }
            </style>
            <div class='grid-container'>
                <div>Column 1</div>
                <div>Column 2</div>
                <div>Column 3</div>
            </div>";

        HtmlFormatProvider htmlProvider = new HtmlFormatProvider();
        RadFlowDocument flowDocument = htmlProvider.Import(html);

        PdfFormatProvider pdfProvider = new PdfFormatProvider();
        return pdfProvider.Export(flowDocument);
    }

    public byte[] ConvertHtmlWithImages(string htmlContent)
    {
        HtmlFormatProvider htmlProvider = new HtmlFormatProvider();

        // Provide image bytes via the LoadFromUri callback
        htmlProvider.ImportSettings.LoadFromUri += (sender, e) =>
        {
            if (e.Uri.EndsWith(".jpg") || e.Uri.EndsWith(".png"))
            {
                using (var client = new System.Net.WebClient())
                {
                    e.SetData(client.DownloadData(e.Uri));
                }
            }
        };

        RadFlowDocument flowDocument = htmlProvider.Import(htmlContent);

        PdfFormatProvider pdfProvider = new PdfFormatProvider();
        return pdfProvider.Export(flowDocument);
    }
}

A few architectural points are worth knowing before relying on this pipeline for HTML-heavy work:

  • CSS Grid and Flexbox are layout-model features the flow document does not natively express, so layouts using them typically reflow to a linear sequence—verify against your version.
  • The mental model spans two document types: RadFlowDocument for the word-processing flow model and RadFixedDocument for the fixed PDF model, so debugging an HTML-to-PDF flow may require reasoning about both.
  • External resources (images, CSS, fonts) are resolved through the LoadFromUri callback on ImportSettings rather than by a browser fetcher, so resource resolution is application code rather than automatic.
  • JavaScript is not executed by the converter; dynamic content, AJAX-loaded sections, and SPAs need to be pre-rendered before import.
  • The flow-to-PDF step applies its own pagination and reflow rules, which may differ from what a browser layout engine would produce.
  • There is no Chrome-equivalent preview during development because the rendered output is shaped by the flow-document model rather than a browser layout.

IronPDF — Direct Chromium Rendering

using IronPdf;
using System.Threading.Tasks;

public class IronPdfHtmlConverter
{
    private readonly ChromePdfRenderer _renderer;

    public IronPdfHtmlConverter()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
        _renderer = new ChromePdfRenderer();
    }

    public async Task<byte[]> ConvertHtmlStringToPdfAsync(string htmlContent)
    {
        // Direct HTML → PDF, no intermediate format
        var pdf = await _renderer.RenderHtmlAsPdfAsync(htmlContent);
        return pdf.BinaryData;
    }

    public async Task<byte[]> ConvertUrlToPdfAsync(string url)
    {
        // Browser engine fetches URL, loads resources, executes JavaScript
        var pdf = await _renderer.RenderUrlAsPdfAsync(url);
        return pdf.BinaryData;
    }

    public async Task<byte[]> ConvertComplexHtmlAsync()
    {
        // Modern CSS works because Chromium supports it
        string html = @"
            <!DOCTYPE html>
            <html>
            <head>
                <style>
                    .grid-container {
                        display: grid;
                        grid-template-columns: 1fr 2fr 1fr;
                        gap: 20px;
                    }
                    .flex-item {
                        display: flex;
                        justify-content: space-between;
                    }
                </style>
            </head>
            <body>
                <div class='grid-container'>
                    <div>Column 1</div>
                    <div>Column 2</div>
                    <div>Column 3</div>
                </div>
            </body>
            </html>";

        var pdf = await _renderer.RenderHtmlAsPdfAsync(html);
        return pdf.BinaryData;
    }

    public async Task<byte[]> ConvertHtmlWithImages(string htmlWithImages)
    {
        // Images load automatically - browser engine handles resource fetching
        var pdf = await _renderer.RenderHtmlAsPdfAsync(htmlWithImages);
        return pdf.BinaryData;
    }
}

Chromium rendering ensures consistent output across platforms. Chrome preview equals PDF output—what you see is exactly what the PDF will contain. For HTML string rendering details, see the HTML string to PDF guide.

How Do You Merge PDFs and Apply Digital Signatures?

Both libraries support merging PDFs, extracting text, digital signatures, and form filling. Telerik provides these through its programmatic RadFixedDocument API, which is comprehensive and well-documented for code-driven PDF operations. IronPDF provides the same operations but keeps the workflow HTML-centric, with PDF manipulation built on top of the rendering engine.

One practical difference is installation footprint. Telerik's HTML-to-PDF flow uses the Telerik.Documents.Flow package (which contains HtmlFormatProvider and RadFlowDocument) together with Telerik.Documents.Flow.FormatProviders.Pdf for the PDF export. Fixed-document operations such as merging or watermarking add Telerik.Documents.Fixed. Each component pulls its own dependencies and needs to be kept on aligned versions.

How Do the Two Libraries Compare Feature by Feature?

  • Status and support. Both products are actively maintained commercial libraries. Telerik ships on a regular release cadence as part of the DevCraft suite with ticket-based support; IronPDF ships frequent updates with engineering-level support.
  • Supported .NET targets. Both libraries cover .NET Framework and modern .NET. Check each vendor's current release notes for the exact supported runtime range, since both have moved their baselines forward.
  • HTML rendering approach. Telerik takes an indirect path through RadFlowDocument; IronPDF renders directly through the Chromium engine.
  • CSS3 coverage. Telerik's CSS support is bounded by what the flow-document model can express; IronPDF inherits Chromium's CSS coverage, including CSS Grid and Flexbox.
  • JavaScript. Telerik does not execute JavaScript during HTML import. IronPDF runs scripts as part of the Chromium render pass.
  • External resources and web fonts. Telerik resolves remote resources through the LoadFromUri callback on ImportSettings; IronPDF fetches resources through Chromium the same way a browser does.
  • PDF operations. Both libraries support creating PDFs from scratch, merging, text extraction, digital signatures, and form filling. Telerik exposes these on its RadFixedDocument programmatic API; IronPDF exposes them on PdfDocument.
  • Office-document interop. Telerik can also convert Word and Excel documents through RadWordsProcessing and RadSpreadProcessing—a capability IronPDF does not provide directly.
  • Primary design intent. Telerik is designed primarily for programmatic document construction across PDF, Word, and Excel; IronPDF is designed primarily for HTML-driven PDF rendering.
  • Async surface. IronPDF exposes async/await throughout its public API; Telerik's HTML-to-PDF flow is largely synchronous.
  • Package footprint. Telerik's HTML-to-PDF flow combines Telerik.Documents.Flow, Telerik.Documents.Flow.FormatProviders.Pdf, and (for fixed-document operations) Telerik.Documents.Fixed; IronPDF ships as a single IronPdf package with its native binaries bundled, which makes its disk footprint larger.
  • Preview workflow. Telerik output reflects the flow-document layout, so previewing in a browser does not match the PDF; IronPDF output matches Chrome's print preview because it is the same engine.
  • Conversion steps. Telerik HTML-to-PDF is a two-step pipeline (HTML → RadFlowDocument → PDF); IronPDF is a single Chromium pass (HTML → PDF).
  • Debugging surface. Telerik debugging spans both the flow-document model and the fixed-document model; IronPDF debugging is grounded in the browser layout you can preview in Chrome.

Frequently Asked Questions

Why does my CSS Grid layout vanish when using Telerik Document Processing?

Telerik's HtmlFormatProvider imports HTML into a RadFlowDocument—a word-processing flow model that predates CSS Grid. The flow document does not represent grid layout, so grid-based HTML typically reflows into sequential paragraphs during import. A common workaround is to rewrite the layout using tables, which the flow-document model does represent. IronPDF's Chromium engine supports CSS Grid the same way Chrome does because it is the same engine.

Can Telerik Document Processing execute JavaScript during conversion?

The HTML-to-PDF flow through HtmlFormatProvider imports static HTML; it does not execute scripts. Single-page applications, dynamic content loading, and AJAX-populated sections need to be pre-rendered before import. IronPDF's Chromium engine runs the V8 JavaScript engine during rendering, so dynamic content can be produced inline.

Why does Telerik require so many NuGet packages for HTML-to-PDF?

The HTML-to-PDF flow combines Telerik.Documents.Flow (which contains HtmlFormatProvider and RadFlowDocument) with Telerik.Documents.Flow.FormatProviders.Pdf for the PDF export step. If you also need fixed-document operations such as merging or watermarking, add Telerik.Documents.Fixed. IronPDF consolidates HTML rendering and PDF manipulation into a single NuGet package: dotnet add package IronPdf.

How do I debug layout issues in Telerik's two-step conversion?

When HTML renders unexpectedly through Telerik, the difference can come from the HTML → flow document import, the flow document → PDF export, or the interaction between both steps. One useful debugging strategy is to export the intermediate flow document to DOCX, inspect it, and check whether the HTML constructs you used have direct flow-document equivalents. IronPDF removes this layer by using a single Chromium render pass—if it looks right in Chrome's print preview, it should look the same in the PDF.

Does Telerik Document Processing handle external images automatically?

External images are resolved through the LoadFromUri event on ImportSettings. The application supplies image bytes in the event handler, resolves relative URLs against a base URL, and handles its own error reporting. IronPDF's Chromium engine fetches resources the same way a browser does, including relative and absolute URLs.

Which library is better for programmatic PDF construction rather than HTML conversion?

Telerik Document Processing is well suited to programmatic PDF construction. Its RadFixedDocument API provides precise control over positioning, drawing, and structure, and the broader Telerik suite covers Word and Excel as well. If your workflow centers on building PDFs from data structures rather than HTML templates, and your organization already licenses the Telerik DevCraft suite, Telerik's programmatic tools are a strong fit. IronPDF is the stronger choice when HTML-to-PDF is the primary requirement.

Which Should Your Team Choose?

Telerik Document Processing built its reputation on programmatic PDF construction—creating documents through code with precise control over positioning, drawing, and structure. The RadFixedDocument API excels when requirements demand coordinate-level positioning or when C# developers own the entire document creation workflow. For teams that primarily build PDFs from data structures, need the full suite of Telerik document libraries (Word, Excel, PDF), and use HTML conversion for less than 10% of PDF generation, Telerik provides comprehensive, well-documented tools that integrate with the broader Telerik ecosystem.

IronPDF inverts that priority: HTML rendering is the foundation rather than an add-on path. The Chromium engine renders HTML directly with the same code that powers Chrome browsers, so troubleshooting tends to look like "does it work in Chrome?" rather than "how do I map this HTML to flow-document concepts?" For applications where HTML templates drive PDF generation, where modern CSS features are essential, or where JavaScript execution is required, that design difference is the main reason to choose IronPDF.
