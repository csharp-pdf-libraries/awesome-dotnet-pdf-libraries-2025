Fluid vs IronPDF: Solving Common .NET PDF Pain Points

A team I worked with built their entire report pipeline around Fluid templates. The Liquid-based engine handled dynamic HTML beautifully — product catalogs, invoice layouts, even multi-section compliance reports all rendered from clean template syntax. Then the CTO asked a simple question: "Can we email these as PDFs?" That's when the team discovered Fluid generates HTML strings, not PDF files. The templating layer they'd invested months in couldn't produce a single binary PDF without bolting on an entirely separate rendering library.

I've seen this pattern repeatedly in .NET document workflows. Fluid is a battle-tested template engine that excels at generating dynamic HTML from Liquid syntax. IronPDF is a Chromium-based PDF renderer that converts HTML — from any source — into production-ready PDFs. This comparison unpacks three common pain points that surface when teams try to bridge the gap between template output and PDF delivery.

What is Fluid and why do .NET teams use it?

Fluid is an actively maintained open-source template engine for .NET that implements the Liquid template language. The library ships as the Fluid.Core NuGet package. Teams adopt it because it provides a secure, sandboxed way to render dynamic content — templates cannot execute arbitrary C# code, which makes Fluid safe for CMS platforms, email generation, and user-facing template editors.

The workflow is straightforward: define a Liquid template with placeholders like {{ report.title }} and loop constructs like {% for item in items %}, parse it with FluidParser, bind a data model via TemplateContext, and call Render() to get an HTML string. That string can go to a browser, an email client, or — with additional tooling — a PDF renderer.

Fluid targets .NET Standard 2.0+, runs on .NET Core, .NET 5+, Linux, macOS, and Docker. It installs as a single lightweight NuGet package with no native dependencies. Community support comes through GitHub issues, but there is no commercial SLA or dedicated support team.

Why can't Fluid produce PDF files on its own?

Fluid is a text-output engine. It was never designed to produce binary PDF files. The library outputs strings — HTML, plain text, or any text format your template defines — but it has no built-in PDF rendering engine. There is no way to specify page breaks, add headers and footers, apply watermarks, set encryption, create digital signatures, or generate fillable form fields using Fluid alone.

This means teams that need PDF deliverables must build a two-stage pipeline: Fluid generates the HTML, then a separate PDF library converts that HTML into a PDF document. This creates architectural complexity — template errors surface at parse time, but PDF rendering errors surface later in a completely different tool. Debugging layout issues requires checking both the template logic and the PDF renderer's interpretation of the output.

CSS must be inline or embedded within the template itself, because Fluid doesn't manage external assets. JavaScript execution depends entirely on the downstream PDF renderer. The result is dependency sprawl: production apps end up requiring Fluid plus a PDF library plus potentially a Razor-to-HTML helper if using MVC.

How do you solve the "HTML template to PDF" pipeline problem?

IronPDF consolidates the entire HTML-to-PDF pipeline into a single library. You pass HTML — whether generated by Fluid, Razor, a StringBuilder, or a static file — to ChromePdfRenderer, and it produces a PdfDocument object using a Chromium rendering engine. The output matches what users see in Chrome, which matters when design fidelity is non-negotiable.

Fluid: Dynamic HTML Template Rendering (No PDF Output)

using Fluid;
using System;
using System.Collections.Generic;

namespace FluidTemplateExample
{
    class Program
    {
        static void Main(string[] args)
        {
            // Define template source
            var templateSource = @"
                <html>
                <head><title>{{ report.title }}</title></head>
                <body>
                    <h1>{{ report.title }}</h1>
                    <p>Generated: {{ report.date }}</p>
                    <table>
                        <tr><th>Product</th><th>Quantity</th><th>Price</th></tr>
                        {% for item in report.items %}
                        <tr>
                            <td>{{ item.name }}</td>
                            <td>{{ item.quantity }}</td>
                            <td>{{ item.price }}</td>
                        </tr>
                        {% endfor %}
                    </table>
                </body>
                </html>";

            // Parse template
            var parser = new FluidParser();
            if (!parser.TryParse(templateSource, out var template, out var error))
            {
                Console.WriteLine($"Template parse error: {error}");
                return;
            }

            // Prepare data model
            var reportData = new
            {
                report = new
                {
                    title = "Q1 Sales Report",
                    date = DateTime.Now.ToString("yyyy-MM-dd"),
                    items = new[]
                    {
                        new { name = "Widget A", quantity = 100, price = "$50.00" },
                        new { name = "Widget B", quantity = 250, price = "$30.00" },
                        new { name = "Widget C", quantity = 75, price = "$120.00" }
                    }
                }
            };

            // Configure context
            var options = new TemplateOptions();
            options.MemberAccessStrategy.Register(reportData.GetType());
            var context = new TemplateContext(reportData, options);

            // Render to HTML string
            var htmlOutput = template.Render(context);
            Console.WriteLine(htmlOutput);

            // NOTE: htmlOutput is just a string - no PDF yet
            // You would need to pass this to a PDF renderer separately
        }
    }
}

IronPDF: HTML to PDF Rendering in One Pass

using IronPdf;
using System;
using System.Text;

namespace IronPdfTemplateExample
{
    class Program
    {
        static void Main(string[] args)
        {
            // License configuration (free 30-day trial available)
            IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

            // Build HTML content (could come from Fluid, Razor, or static source)
            var htmlContent = BuildReportHtml();

            // Create PDF renderer
            var renderer = new ChromePdfRenderer();

            // Configure rendering options
            renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
            renderer.RenderingOptions.MarginTop = 40;
            renderer.RenderingOptions.MarginBottom = 40;
            renderer.RenderingOptions.PrintHtmlBackgrounds = true;

            // Render HTML to PDF
            var pdf = renderer.RenderHtmlAsPdf(htmlContent);

            // Save PDF
            pdf.SaveAs("Q1_Sales_Report.pdf");
            Console.WriteLine("PDF created successfully");

            // Cleanup
            pdf.Dispose();
        }

        static string BuildReportHtml()
        {
            // In production, this HTML could come from Fluid templates,
            // Razor views, or any HTML generation method
            var sb = new StringBuilder();
            sb.AppendLine("<html><head>");
            sb.AppendLine("<style>body { font-family: Arial; } table { border-collapse: collapse; width: 100%; }");
            sb.AppendLine("th, td { border: 1px solid #ddd; padding: 8px; text-align: left; }</style>");
            sb.AppendLine("</head><body>");
            sb.AppendLine("<h1>Q1 Sales Report</h1>");
            sb.AppendLine($"<p>Generated: {DateTime.Now:yyyy-MM-dd}</p>");
            sb.AppendLine("<table><tr><th>Product</th><th>Quantity</th><th>Price</th></tr>");
            sb.AppendLine("<tr><td>Widget A</td><td>100</td><td>$50.00</td></tr>");
            sb.AppendLine("<tr><td>Widget B</td><td>250</td><td>$30.00</td></tr>");
            sb.AppendLine("<tr><td>Widget C</td><td>75</td><td>$120.00</td></tr>");
            sb.AppendLine("</table></body></html>");
            return sb.ToString();
        }
    }
}

IronPDF handles the complete HTML-to-PDF pipeline in one pass. The HTML source is flexible — you can generate it with Fluid if template-driven content makes sense, or use Razor, string builders, or static files. How do you create fillable PDF forms from HTML templates?

Fluid can generate HTML form markup with input fields, labels, and placeholders — but the output is just HTML. It cannot create interactive PDF form fields. To produce a fillable PDF with AcroForm fields, you need a PDF library that understands form specifications.

IronPDF solves this with a single rendering option: CreatePdfFormsFromHtml = true. When enabled, HTML form elements automatically convert into editable PDF fields.

Fluid: Template-Based Form HTML (No PDF Forms)

using Fluid;
using System;

namespace FluidFormExample
{
    class Program
    {
        static void Main(string[] args)
        {
            var formTemplate = @"
                <html>
                <body>
                    <h2>{{ form.title }}</h2>
                    <form>
                        {% for field in form.fields %}
                        <label>{{ field.label }}</label>
                        <input type='{{ field.type }}' name='{{ field.name }}'
                               placeholder='{{ field.placeholder }}' /><br/>
                        {% endfor %}
                        <button type='submit'>Submit</button>
                    </form>
                </body>
                </html>";

            var parser = new FluidParser();
            if (!parser.TryParse(formTemplate, out var template, out var error))
            {
                Console.WriteLine($"Parse error: {error}");
                return;
            }

            var formModel = new
            {
                form = new
                {
                    title = "Customer Registration",
                    fields = new[]
                    {
                        new { label = "Full Name", type = "text", name = "fullName", placeholder = "John Doe" },
                        new { label = "Email", type = "email", name = "email", placeholder = "john@example.com" },
                        new { label = "Phone", type = "tel", name = "phone", placeholder = "555-1234" }
                    }
                }
            };

            var options = new TemplateOptions();
            options.MemberAccessStrategy.Register(formModel.GetType());
            var context = new TemplateContext(formModel, options);

            var htmlForm = template.Render(context);
            Console.WriteLine(htmlForm);
            // This HTML form still needs a PDF renderer to become a PDF
        }
    }
}

IronPDF: Interactive PDF Form Creation from HTML

using IronPdf;
using System;

namespace IronPdfFormExample
{
    class Program
    {
        static void Main(string[] args)
        {
            IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

            var htmlWithForm = @"
                <html>
                <body>
                    <h2>Customer Registration</h2>
                    <form>
                        <label>Full Name:</label>
                        <input type='text' name='fullName' placeholder='John Doe' /><br/>
                        <label>Email:</label>
                        <input type='email' name='email' placeholder='john@example.com' /><br/>
                        <label>Phone:</label>
                        <input type='tel' name='phone' placeholder='555-1234' /><br/>
                        <button type='submit'>Submit</button>
                    </form>
                </body>
                </html>";

            var renderer = new ChromePdfRenderer();
            renderer.RenderingOptions.CreatePdfFormsFromHtml = true; // Convert HTML inputs to PDF form fields

            var pdf = renderer.RenderHtmlAsPdf(htmlWithForm);
            pdf.SaveAs("CustomerRegistrationForm.pdf");

            Console.WriteLine("Interactive PDF form created");
            pdf.Dispose();
        }
    }
}

With CreatePdfFormsFromHtml enabled, IronPDF converts HTML form elements into editable PDF fields. For complex scenarios — multi-page forms, calculated fields, digital signatures — refer to IronPDF's form handling documentation.

How do you add page numbers, headers, and footers to multi-page reports?

This is one of the most common pain points in the Fluid-to-PDF workflow. Fluid cannot inject "page X of Y" content because it has no awareness of pagination. CSS @page rules and running headers placed in templates depend entirely on whether the downstream PDF renderer supports them. Developers must manually guess where content breaks across pages.

IronPDF's rendering engine understands {page} and {total-pages} tokens natively, applying them automatically during PDF generation. This eliminates guesswork around pagination entirely.

Fluid: Template with Header/Footer Placeholders (Limited Control)

using Fluid;
using System;

namespace FluidMultiPageExample
{
    class Program
    {
        static void Main(string[] args)
        {
            var pageTemplate = @"
                <html>
                <head>
                    <style>
                        @page { margin: 1in; }
                        header { position: running(pageHeader); }
                        footer { position: running(pageFooter); }
                    </style>
                </head>
                <body>
                    <header>{{ header.text }}</header>
                    {% for section in sections %}
                    <div class='page-section'>
                        <h2>{{ section.title }}</h2>
                        <p>{{ section.content }}</p>
                    </div>
                    {% endfor %}
                    <footer>Page {{ footer.pageNumber }}</footer>
                </body>
                </html>";

            var parser = new FluidParser();
            if (!parser.TryParse(pageTemplate, out var template, out var error))
            {
                Console.WriteLine($"Error: {error}");
                return;
            }

            var data = new
            {
                header = new { text = "Annual Report 2025" },
                sections = new[]
                {
                    new { title = "Executive Summary", content = "Lorem ipsum..." },
                    new { title = "Financial Highlights", content = "Revenue grew by..." },
                    new { title = "Future Outlook", content = "We anticipate..." }
                },
                footer = new { pageNumber = "{{pageNumber}}" } // Placeholder - not evaluated
            };

            var options = new TemplateOptions();
            options.MemberAccessStrategy.Register(data.GetType());
            var context = new TemplateContext(data, options);

            var html = template.Render(context);
            Console.WriteLine(html);
            // CSS @page rules won't work in most browsers; PDF renderer must support them
        }
    }
}

IronPDF: Native Header/Footer with Dynamic Page Numbers

using IronPdf;
using System.Text;

namespace IronPdfHeaderFooterExample
{
    class Program
    {
        static void Main(string[] args)
        {
            IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

            var reportHtml = BuildMultiSectionReport();

            var renderer = new ChromePdfRenderer();

            // Configure headers and footers with dynamic content
            renderer.RenderingOptions.TextHeader = new IronPdf.Rendering.TextHeaderFooter
            {
                CenterText = "Annual Report 2025",
                Font = IronPdf.Rendering.FontTypes.Helvetica,
                FontSize = 10
            };

            renderer.RenderingOptions.TextFooter = new IronPdf.Rendering.TextHeaderFooter
            {
                LeftText = "Confidential",
                RightText = "Page {page} of {total-pages}",
                FontSize = 9
            };

            var pdf = renderer.RenderHtmlAsPdf(reportHtml);
            pdf.SaveAs("AnnualReport2025.pdf");

            pdf.Dispose();
        }

        static string BuildMultiSectionReport()
        {
            var sb = new StringBuilder();
            sb.AppendLine("<html><body>");
            sb.AppendLine("<h2>Executive Summary</h2><p>Lorem ipsum dolor sit amet...</p>");
            sb.AppendLine("<div style='page-break-before: always;'></div>");
            sb.AppendLine("<h2>Financial Highlights</h2><p>Revenue grew by 15%...</p>");
            sb.AppendLine("<div style='page-break-before: always;'></div>");
            sb.AppendLine("<h2>Future Outlook</h2><p>We anticipate strong performance...</p>");
            sb.AppendLine("</body></html>");
            return sb.ToString();
        }
    }
}

How do the two libraries compare feature-by-feature?

The list below covers every major capability. The key takeaway is that Fluid and IronPDF operate at different layers of the document pipeline — Fluid handles template logic and HTML generation, while IronPDF handles PDF rendering, manipulation, and security.

  • Project model. Fluid is open-source with community support via GitHub Issues; IronPDF is commercial with SLA-backed support options. Both are actively developed.
  • Documentation and samples. Fluid leans on the broader Liquid reference plus its own README; IronPDF ships extensive C#-specific tutorials and code samples.
  • Layer of the stack. Fluid is a Liquid template engine that emits HTML or text; IronPDF is a PDF renderer that takes HTML in and produces PDFs out. They are complementary, not competitors.
  • PDF output. Fluid does not produce PDFs at all; IronPDF produces PDFs as its primary output via ChromePdfRenderer.
  • Pagination, headers, and footers. Fluid has no concept of pages; IronPDF supports CSS page breaks, auto-detection, and dynamic {page} / {total-pages} tokens in TextHeader and TextFooter.
  • PDF manipulation. Watermarks, merge, split, text extraction, and editing existing PDFs are out of scope for Fluid. IronPDF exposes APIs for each — for example PdfDocument.Merge and ApplyWatermark.
  • Forms and signatures. Fluid can render HTML forms but cannot create AcroForm fields or sign PDFs. IronPDF turns HTML inputs into PDF form fields via CreatePdfFormsFromHtml and supports digital signatures.
  • Security. Fluid has no PDF-level security model. IronPDF supports AES 128/256-bit encryption, user/owner passwords, and print/copy/edit permission restrictions through SecuritySettings.
  • JavaScript and CSS. Fluid does not execute JavaScript and requires CSS to be inline in the template if any downstream renderer is to use it. IronPDF runs JavaScript pre-render via Chromium and supports embedded or external CSS.
  • Platform reach. Both run on .NET Standard 2.0+ / .NET Core 3.1+ / .NET 5+, on Windows, Linux, macOS, and Docker. IronPDF additionally supports .NET Framework 4.6.2+.

What are the most common questions about Fluid and IronPDF?

Can I use Fluid and IronPDF together?

Yes. Fluid generates the HTML string from your Liquid template, and IronPDF converts that string into a PDF. Pass the output of template.Render(context) directly to ChromePdfRenderer.RenderHtmlAsPdf(). This is a natural pairing — Fluid handles the data binding and template logic, IronPDF handles the rendering and PDF features.

Why does my Fluid template use plus: instead of the + operator?

Liquid syntax does not support native operators like + or *. Instead, you use filters: {{ value | plus: 5 }} and {{ value | times: 3 }}. This is a design choice that maintains Liquid's security model — templates cannot execute arbitrary code. Developers familiar with Razor or C# expressions need to adjust to this filter-based approach.

Does IronPDF require a paid license?

IronPDF offers a free 30-day trial. For production use, a commercial license is required. The license includes commercial support with SLA options. Fluid is free and open-source under the MIT license with community support via GitHub issues.

How do I handle null values in Fluid templates?

Fluid uses nil instead of null. Conditionals check for nil: {% if variable != nil %}. This is a Liquid convention, not a bug. Developers expecting C#-style null handling need to adapt their template logic accordingly.

Can IronPDF render CSS Grid and Flexbox layouts?

Yes. IronPDF uses a Chromium rendering engine, so it supports modern CSS including CSS Grid, Flexbox, and JavaScript-driven charts. The PDF output matches what users see in Chrome. This is a significant advantage over renderers that rely on older HTML parsing engines with limited CSS support. See the IronPDF rendering documentation for layout examples.

When should you use Fluid vs IronPDF?

Use Fluid when your pipeline ends with HTML strings that go to a web browser or email client. Fluid is battle-tested in CMS platforms and email generation workflows where the output is HTML or plain text. It handles dynamic content rendering efficiently with a secure, sandboxed template model.

Use IronPDF when the deliverable is a PDF document — invoices, certificates, compliance reports, contracts. The two-stage "Fluid then PDF renderer" approach adds architectural complexity that IronPDF eliminates by accepting HTML from any source and producing pixel-perfect PDFs with headers, footers, forms, signatures, and encryption in a single library.

Migration from a Fluid-plus-separate-renderer setup to IronPDF becomes necessary when: teams maintain both Fluid and a separate PDF library with growing dependency sprawl; pagination issues require PDF-aware layout control; or compliance mandates demand native PDF features like digital signatures and redaction. In those cases, IronPDF's unified API reduces the testing surface area and simplifies the deployment footprint.
