PuppeteerSharp vs IronPDF: Solving Common .NET PDF Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

What Is PuppeteerSharp?

PuppeteerSharp is a .NET port of the popular Node.js Puppeteer library. It launches headless Chromium as an external process and controls it via the DevTools Protocol. Teams discover it when they need HTML-to-PDF conversion and recognize Puppeteer's popularity in the Node ecosystem.

It works -- until it doesn't. Common failure modes include Chrome binary downloads timing out in CI/CD pipelines, X-server requirements breaking Linux containers, PDF generation mysteriously hanging on page loads with external assets, memory leaks from undisposed browser instances piling up, and cryptic DevTools Protocol errors that offer no actionable guidance.

PuppeteerSharp is actively maintained (version 20.2.6 as of December 2025), has 3.8k GitHub stars, and tracks upstream Puppeteer changes. But it is fundamentally a browser automation tool adapted for PDF generation, not a PDF-focused library.

IronPDF takes the opposite approach. It embeds a Chromium rendering engine as a .NET library component -- no external processes, no browser binaries to manage, no DevTools Protocol complexity. Both produce Chrome-quality PDFs, but the architectural differences create very different operational experiences. You can explore the HTML to PDF tutorial to see IronPDF's approach firsthand.

How Do I Convert HTML to PDF with Each Library?

Answer:

This is the most common task, and it reveals the fundamental architectural difference. PuppeteerSharp requires explicit browser lifecycle management. IronPDF treats PDF generation as a single method call.

PuppeteerSharp -- Browser Automation Approach

using PuppeteerSharp;
using System;
using System.IO;
using System.Threading.Tasks;

public class PuppeteerPdfGenerator
{
    public static async Task GenerateBasicPdf()
    {
        // Download Chrome binary if not present (can take 1-2 minutes on first run)
        var browserFetcher = new BrowserFetcher();
        await browserFetcher.DownloadAsync();

        // Launch headless Chrome (creates external process)
        await using var browser = await Puppeteer.LaunchAsync(new LaunchOptions
        {
            Headless = true,
            Args = new[] { "--no-sandbox", "--disable-setuid-sandbox" } // Required in containers
        });

        // Create new page (separate browser context)
        await using var page = await browser.NewPageAsync();

        // Set content and wait for resources
        await page.SetContentAsync(@"
            <!DOCTYPE html>
            <html>
            <head>
                <style>
                    body { font-family: Arial, sans-serif; margin: 40px; }
                    h1 { color: #333; }
                </style>
            </head>
            <body>
                <h1>Sample Report</h1>
                <p>This is a basic HTML to PDF example.</p>
            </body>
            </html>
        ");

        // Wait for fonts to load (required for accurate rendering)
        await page.EvaluateExpressionHandleAsync("document.fonts.ready");

        // Generate PDF
        await page.PdfAsync("puppeteersharp-basic.pdf", new PdfOptions
        {
            Format = PaperFormat.A4,
            PrintBackground = true,
            MarginOptions = new MarginOptions
            {
                Top = "20px",
                Bottom = "20px",
                Left = "20px",
                Right = "20px"
            }
        });

        // Browser and page are disposed via await using
    }
}

// Usage
await PuppeteerPdfGenerator.GenerateBasicPdf();

There are several troubleshooting points in this code. The binary download can timeout since DownloadAsync() fetches roughly 200 MB. The --no-sandbox flag is a security risk but is required in Docker containers without proper user namespaces. Without document.fonts.ready, PDFs may render with missing or fallback fonts. And forgetting await using or Dispose() leaves Chrome processes running at around 150-200 MB each.

IronPDF -- Embedded Rendering Approach

using IronPdf;

// Generate PDF from HTML string (no browser process, no external binaries)
var renderer = new ChromePdfRenderer();
using var pdf = renderer.RenderHtmlAsPdf(@"
    <!DOCTYPE html>
    <html>
    <head>
        <style>
            body { font-family: Arial, sans-serif; margin: 40px; }
            h1 { color: #333; }
        </style>
    </head>
    <body>
        <h1>Sample Report</h1>
        <p>This is a basic HTML to PDF example.</p>
    </body>
    </html>
");

pdf.SaveAs("ironpdf-basic.pdf");

IronPDF's embedded engine handles fonts, resources, and rendering internally. No --no-sandbox flags, no X-server requirements, no process lifecycle management. The using statement ensures memory cleanup. See the HTML string to PDF guide for advanced scenarios.

How Do I Render URLs with External Resources?

Answer:

Rendering live URLs introduces timeout and wait-condition complexity. PuppeteerSharp requires careful configuration around navigation strategies. IronPDF handles it with a simpler API.

PuppeteerSharp -- Navigation and Wait Strategies

using PuppeteerSharp;
using System;
using System.Threading.Tasks;

public class PuppeteerUrlRenderer
{
    public static async Task RenderUrlWithWait()
    {
        var browserFetcher = new BrowserFetcher();
        await browserFetcher.DownloadAsync();

        await using var browser = await Puppeteer.LaunchAsync(new LaunchOptions
        {
            Headless = true,
            Args = new[]
            {
                "--no-sandbox",
                "--disable-setuid-sandbox",
                "--disable-dev-shm-usage" // Prevent shared memory issues in containers
            },
            Timeout = 120000 // 2-minute launch timeout
        });

        await using var page = await browser.NewPageAsync();

        // Set longer timeout for navigation
        page.DefaultNavigationTimeout = 60000; // 60 seconds

        try
        {
            // Navigate and wait for network idle (all resources loaded)
            await page.GoToAsync("https://example.com/report", new NavigationOptions
            {
                WaitUntil = new[] { WaitUntilNavigation.Networkidle0 }, // Wait for network idle
                Timeout = 60000
            });

            // Wait for specific elements if needed
            await page.WaitForSelectorAsync("body", new WaitForSelectorOptions
            {
                Timeout = 10000
            });

            // Additional wait for fonts and images
            await page.EvaluateExpressionHandleAsync("document.fonts.ready");
            await Task.Delay(500); // Additional buffer for images

            // Generate PDF
            await page.PdfAsync("puppeteersharp-url.pdf", new PdfOptions
            {
                Format = PaperFormat.A4,
                PrintBackground = true,
                PreferCSSPageSize = true,
                MarginOptions = new MarginOptions
                {
                    Top = "20px",
                    Bottom = "20px",
                    Left = "20px",
                    Right = "20px"
                }
            });
        }
        catch (NavigationException ex)
        {
            Console.WriteLine($"Navigation failed: {ex.Message}");
        }
        catch (TimeoutException ex)
        {
            Console.WriteLine($"Operation timed out: {ex.Message}");
        }
    }
}

await PuppeteerUrlRenderer.RenderUrlWithWait();

The default 30-second timeout is often insufficient for real-world sites with analytics, ads, or slow CDNs. Choosing between Networkidle0 (zero connections for 500ms) and Networkidle2 (two or fewer connections) determines whether you get premature rendering or excessive waits. External analytics scripts can delay network idle indefinitely. And --disable-dev-shm-usage is required to prevent Chrome from using /dev/shm, which is often limited in containers.

IronPDF -- Simplified URL Rendering

using IronPdf;

var renderer = new ChromePdfRenderer();

// Configure wait conditions if needed
renderer.RenderingOptions.Timeout = 60; // seconds
renderer.RenderingOptions.WaitFor.RenderDelay(500); // Wait 500ms after load

// Render URL to PDF (handles navigation, resources, fonts automatically)
using var pdf = renderer.RenderUrlAsPdf("https://example.com/report");
pdf.SaveAs("ironpdf-url.pdf");

IronPDF's embedded engine manages the full rendering pipeline. No navigation strategies, no process lifecycle, no DevTools Protocol complexity. For advanced wait conditions, see the JavaScript and render delays guide.

How Do I Handle Authentication When Generating PDFs?

Answer:

Rendering authenticated pages is a common requirement for internal dashboards and secure reports. PuppeteerSharp requires manual cookie and header management. IronPDF exposes authentication through rendering options.

PuppeteerSharp -- Cookie and Header Management

using PuppeteerSharp;
using System;
using System.Collections.Generic;
using System.Threading.Tasks;

public class PuppeteerAuthRenderer
{
    public static async Task RenderWithAuth()
    {
        var browserFetcher = new BrowserFetcher();
        await browserFetcher.DownloadAsync();

        await using var browser = await Puppeteer.LaunchAsync(new LaunchOptions
        {
            Headless = true,
            Args = new[] { "--no-sandbox", "--disable-setuid-sandbox" }
        });

        await using var page = await browser.NewPageAsync();

        // Set authentication cookie
        await page.SetCookieAsync(new CookieParam
        {
            Name = "auth_token",
            Value = "your-token-here",
            Domain = "example.com",
            Path = "/",
            HttpOnly = true,
            Secure = true
        });

        // Set custom headers (e.g., Bearer token)
        await page.SetExtraHttpHeadersAsync(new Dictionary<string, string>
        {
            { "Authorization", "Bearer your-bearer-token" },
            { "Custom-Header", "custom-value" }
        });

        // Navigate with authentication
        await page.GoToAsync("https://example.com/secure-report", new NavigationOptions
        {
            WaitUntil = new[] { WaitUntilNavigation.Networkidle0 },
            Timeout = 60000
        });

        await page.PdfAsync("puppeteersharp-auth.pdf", new PdfOptions
        {
            Format = PaperFormat.A4,
            PrintBackground = true
        });
    }
}

await PuppeteerAuthRenderer.RenderWithAuth();

Watch out for cookie domain mismatches -- setting cookies with the wrong domain silently fails. Cookies with Secure = true only work over HTTPS. Extra headers only apply to the navigation request, not subsequent resource loads. For complex auth flows like OAuth or multi-step logins, you must script the full browser interaction including form fills, clicks, and redirects.

IronPDF -- Authentication in Rendering Options

using IronPdf;
using IronPdf.Engines.Chrome;

var renderer = new ChromePdfRenderer();

// Set authentication credentials
renderer.RenderingOptions.CustomCookies.Add("auth_token", "your-token-here");
renderer.RenderingOptions.CustomHttpHeaders.Add("Authorization", "Bearer your-bearer-token");

using var pdf = renderer.RenderUrlAsPdf("https://example.com/secure-report");
pdf.SaveAs("ironpdf-auth.pdf");

IronPDF's rendering options handle authentication configuration without manual browser scripting. See the HTML to PDF tutorial for comprehensive authentication examples.

What Are the Most Common PuppeteerSharp Troubleshooting Scenarios?

Answer:

I see four issues that come up repeatedly when teams use PuppeteerSharp for PDF generation. These are architectural consequences of wrapping an external browser process, and each one has a different mitigation strategy.

Chrome Binary Download Failures

BrowserFetcher.DownloadAsync() times out or fails in CI/CD pipelines. The workaround is to pre-download Chrome and bundle it with your application, or use a Docker image with Chrome pre-installed.

// Pre-download Chrome and bundle with application
var browserFetcher = new BrowserFetcher(new BrowserFetcherOptions
{
    Path = "/path/to/bundled/chrome"
});

// Specify exact Chrome revision to avoid downloads
await browserFetcher.DownloadAsync(BrowserFetcher.DefaultChromiumRevision);

With IronPDF, this is not applicable -- there are no external binary downloads required.

"Cannot open display" Error in Docker

This error means Chrome cannot find an X-server. You either install X virtual framebuffer in your Dockerfile or use the --no-sandbox flag, which carries security risks.

# Install X virtual framebuffer in Dockerfile
RUN apt-get update && apt-get install -y xvfb

# Run with xvfb-run
xvfb-run dotnet MyApp.dll

With IronPDF, this is not applicable -- there is no X-server requirement.

Memory Leaks from Undisposed Browser Instances

Application memory grows continuously and eventually throws OutOfMemoryException. The cause is missing disposal of browser and page objects.

// WRONG - leaks browser process
public async Task GeneratePdfBadExample()
{
    var browser = await Puppeteer.LaunchAsync(new LaunchOptions { Headless = true });
    var page = await browser.NewPageAsync();
    await page.SetContentAsync("<h1>Test</h1>");
    await page.PdfAsync("output.pdf");
    // Missing: await browser.DisposeAsync()
    // Missing: await page.DisposeAsync()
}

// CORRECT - uses await using for automatic disposal
public async Task GeneratePdfGoodExample()
{
    await using var browser = await Puppeteer.LaunchAsync(new LaunchOptions { Headless = true });
    await using var page = await browser.NewPageAsync();
    await page.SetContentAsync("<h1>Test</h1>");
    await page.PdfAsync("output.pdf");
    // Automatic disposal via await using
}

IronPDF handles resource cleanup internally. A standard using statement ensures PDF document disposal:

public void GeneratePdf()
{
    var renderer = new ChromePdfRenderer();
    using var pdf = renderer.RenderHtmlAsPdf("<h1>Test</h1>");
    pdf.SaveAs("output.pdf");
    // Automatic cleanup via using statement
}

Navigation Timeout on Pages with Slow External Resources

TimeoutException: Navigation timeout of 30000 ms exceeded is one of the most common errors. The fix is to block problematic domains and increase timeouts.

// Block problematic domains (analytics, ads)
await page.SetRequestInterceptionAsync(true);
page.Request += (sender, e) =>
{
    var blockedDomains = new[] { "google-analytics.com", "doubleclick.net", "facebook.com" };
    if (blockedDomains.Any(domain => e.Request.Url.Contains(domain)))
    {
        e.Request.AbortAsync();
    }
    else
    {
        e.Request.ContinueAsync();
    }
};

// Increase timeouts
await page.GoToAsync("https://slow-site.com", new NavigationOptions
{
    WaitUntil = new[] { WaitUntilNavigation.DOMContentLoaded },
    Timeout = 90000 // 90 seconds
});

IronPDF has built-in timeout handling with configurable delays:

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.Timeout = 90; // seconds
renderer.RenderingOptions.WaitFor.RenderDelay(1000); // Wait 1 second after load

using var pdf = renderer.RenderUrlAsPdf("https://slow-site.com");
pdf.SaveAs("output.pdf");

Frequently Asked Questions

Is PuppeteerSharp free to use?

Yes. PuppeteerSharp is open source and free. It is community-maintained with sponsorship-based support. However, support is community-dependent through GitHub issues and Stack Overflow. Complex debugging often requires understanding Chrome DevTools Protocol internals. IronPDF is commercial with a 30-day free trial and includes 24/5 engineer support with optional 24/7 premium tiers and enterprise SLAs.

Can PuppeteerSharp merge, split, or watermark PDFs?

No. PuppeteerSharp generates PDFs but does not merge them, split them, extract text, add watermarks, apply digital signatures, or fill forms. You need an additional library like IronPDF, iText, or PdfSharp for any post-generation operations. IronPDF provides PdfDocument.Merge(), PdfDocument.ExtractAllText(), PdfDocument.ApplyWatermark(), PdfDocument.Sign(), and form-filling capabilities out of the box.

Does PuppeteerSharp work in Docker containers without extra configuration?

Not typically. PuppeteerSharp requires X11 on Linux for headless Chrome. In Docker containers, you must install xvfb or use the --no-sandbox flag, which has security implications. You also need --disable-dev-shm-usage to prevent shared memory issues. IronPDF runs natively on Linux with no X-server requirements and minimal container configuration.

What happens if I forget to dispose the browser in PuppeteerSharp?

Each undisposed browser instance leaks a Chrome process consuming roughly 150-200 MB of memory and file handles. In high-traffic applications, 10-20 leaked processes can exhaust server memory. PuppeteerSharp's async architecture with nested operations makes it easy to miss disposal. Always use await using for both browser and page objects. IronPDF handles resource management internally through its library-based architecture, and a standard using statement handles cleanup.

Do both libraries produce the same rendering quality?

Yes. Both use Chrome's rendering engine, so the output quality is identical. The difference is architectural: PuppeteerSharp launches headless Chrome as an external process and controls it via the DevTools Protocol, while IronPDF embeds the Chromium rendering engine as a library component. Same Chrome quality, different operational characteristics.

When Should I Use PuppeteerSharp vs IronPDF?

PuppeteerSharp is the right choice if you need full browser automation capabilities -- scripting clicks, form fills, screenshots, and navigation flows in addition to PDF generation. Its process-based architecture provides isolation and the full Chrome API surface. For teams already using browser automation or requiring fine-grained browser control, the operational complexity is justified.

However, if your primary need is "convert HTML to PDF," PuppeteerSharp's browser automation model adds troubleshooting overhead without functional benefit. Chrome download timeouts, X-server errors, process leaks, and navigation hangs are all architectural consequences of wrapping an external browser process.

IronPDF's library-based approach handles the full PDF lifecycle in-process. It renders HTML with the same Chrome quality but without operational friction. Built-in PDF manipulation (merge, split, watermark, sign, encrypt) and 24/5 engineer support streamline production deployments. IronPDF supports .NET Core, .NET 5+, and .NET Framework 4.6.2+, and integrates naturally with ASP.NET, Azure Functions, AWS Lambda, and containerized deployments. For teams that need reliable HTML-to-PDF conversion without browser automation complexity, IronPDF's embedded architecture is the pragmatic choice.

The question to ask your team: does your application need full browser automation with PDF as a byproduct, or is HTML-to-PDF rendering the primary requirement?
