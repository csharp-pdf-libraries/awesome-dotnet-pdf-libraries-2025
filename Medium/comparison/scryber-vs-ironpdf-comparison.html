Scryber.Core vs IronPDF: Solving Common .NET PDF Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Imagine debugging a PDF rendering issue at 2 AM because a flexbox layout renders differently than expected, and your library's documentation says "CSS support is partial — some features may not work as intended." One team I heard about spent days tracking down why display: grid worked in their HTML preview but broke pagination in the final PDF. The root cause? Their PDF library used a custom CSS parser that didn't implement the Grid spec fully.

Scryber.Core is an open-source, template-driven PDF library for .NET that emphasizes programmatic document layout and data binding with a handlebars-style template syntax. It's built entirely in C# with no browser dependencies, which makes it portable and lightweight. However, this architectural choice creates a predictability problem: HTML/CSS support depends on the maintainer implementing each spec feature manually. If you're reaching for Scryber, this comparison will help you understand when its approach fits — and when Chrome-based rendering would save troubleshooting time.

What Is IronPDF?

Answer:

IronPDF uses Google Chrome's rendering engine (Chromium/Blink) to generate PDFs, which means it supports whatever Chrome supports: CSS Grid, Flexbox, CSS variables, Web Fonts, SVG, and complex JavaScript. When you call ChromePdfRenderer.RenderHtmlAsPdf(), your HTML renders exactly as it would in Chrome's "Print to PDF" feature. No second-guessing whether a specific CSS property is implemented; if it works in Chrome DevTools, it works in your PDF.

The trade-off is footprint: IronPDF bundles the Chromium engine (~50MB compressed NuGet package vs Scryber's ~5MB). For teams that value "write HTML once, render everywhere" over minimal package size, that's an acceptable trade. IronPDF targets .NET 10, 9, 8, 7, 6, Core, Framework, and runs on Windows, Linux, macOS, Docker, and cloud platforms without modification.

What Are the Key Limitations of Scryber.Core?

Answer:

Scryber.Core is an open-source project under LGPL 3.0 license; commercial use is allowed but any modifications to Scryber itself must be shared under LGPL. It has a single active maintainer (richard-scryber) on GitHub. Release cadence has slowed compared to early versions — the latest major version is 9.x, released November 2025. Community support is via GitHub issues only; there is no dedicated support team.

Scryber.Core implements a custom PDF rendering engine, not a browser engine, which means HTML/CSS support is incomplete. CSS Grid is not mentioned in documentation — verify current support before relying on it. Complex CSS selectors (:nth-of-type, attribute selectors, pseudo-elements beyond basic) have partial support. JavaScript execution is not supported at all — any dynamic content must be pre-rendered server-side. Modern web fonts require manual import via @font-face and may encounter rendering issues. There is no support for SVG filters, CSS animations, or advanced transforms.

The layout engine uses custom flow logic, not a browser's layout algorithm, leading to subtle differences from HTML preview. Pagination behavior differs from the CSS Paged Media spec — you control breaks via Scryber-specific attributes or CSS properties that may behave differently than expected. Table rendering occasionally miscalculates cell heights with complex content, requiring manual padding/margin adjustments. Font metrics are calculated differently than browsers, causing line-height discrepancies in dense text.

The template syntax mixes HTML with Handlebars-style binding ({{variable}}), which requires learning Scryber's expression engine rather than using standard JavaScript or Razor. Data binding happens at render time via Document.Params dictionary; there is no strongly-typed model binding like MVC Razor. Performance scales with document complexity — large documents (100+ pages) can be slow due to a single-threaded layout engine. No built-in support for HTTP headers, cookies, or authentication when fetching remote resources. Debugging layout issues requires understanding Scryber's internal box model, which differs from browser DevTools.

How Does HTML Rendering with Data Binding Compare?

Answer:

Scryber.Core — Rendering HTML with Data Binding

Scryber's approach uses template files with handlebars syntax. Here's generating an invoice with an items list:

using Scryber.Components;
using Scryber.Components.Mvc;
using System;
using System.Collections.Generic;
using System.IO;

public class ScryberInvoiceGenerator
{
    public void GenerateInvoice(int invoiceId, string outputPath)
    {
        // HTML template file with Scryber syntax
        string templatePath = "InvoiceTemplate.html";
        string templateHtml = @"
<!DOCTYPE html>
<html xmlns='http://www.w3.org/1999/xhtml'>
<head>
    <title>Invoice {{invoiceNumber}}</title>
    <style>
        body { font-family: Arial, sans-serif; margin: 20px; }
        /* Note: CSS Grid not supported - verify in docs */
        .invoice-header {
            display: block; /* Flexbox support limited */
        }
        table { width: 100%; border-collapse: collapse; }
        td, th { border: 1px solid #ddd; padding: 8px; }
    </style>
</head>
<body>
    <div class='invoice-header'>
        <h1>Invoice #{{invoiceNumber}}</h1>
        <p>Date: {{invoiceDate}}</p>
    </div>

    <table>
        <thead>
            <tr><th>Item</th><th>Quantity</th><th>Price</th></tr>
        </thead>
        <tbody>
            <!-- Scryber repeat syntax for data binding -->
            <template data-bind='{{#each items}}'>
                <tr>
                    <td>{{name}}</td>
                    <td>{{quantity}}</td>
                    <td>${{price}}</td>
                </tr>
            </template>
        </tbody>
    </table>

    <p><strong>Total: ${{total}}</strong></p>
</body>
</html>";

        // Save template to file (Scryber requires file path)
        File.WriteAllText(templatePath, templateHtml);

        // Fetch invoice data
        var invoiceData = FetchInvoiceData(invoiceId);

        try
        {
            // Parse document from HTML file
            using (var doc = Document.ParseDocument(templatePath))
            {
                // Bind data via Params dictionary (no strong typing)
                doc.Params["invoiceNumber"] = invoiceData.InvoiceNumber;
                doc.Params["invoiceDate"] = invoiceData.Date.ToString("yyyy-MM-dd");
                doc.Params["total"] = invoiceData.Total;

                // Items collection must be structured for Scryber's {{#each}}
                doc.Params["items"] = invoiceData.Items.Select(i => new
                {
                    name = i.ItemName,
                    quantity = i.Quantity,
                    price = i.Price
                }).ToArray();

                // Save as PDF
                using (var stream = new FileStream(outputPath, FileMode.Create))
                {
                    doc.SaveAsPDF(stream);
                }
            }
        }
        finally
        {
            // Clean up template file
            if (File.Exists(templatePath))
                File.Delete(templatePath);
        }
    }

    private InvoiceData FetchInvoiceData(int invoiceId)
    {
        // Implementation details...
        return new InvoiceData();
    }
}

public class InvoiceData
{
    public string InvoiceNumber { get; set; }
    public DateTime Date { get; set; }
    public decimal Total { get; set; }
    public List<LineItem> Items { get; set; }
}

public class LineItem
{
    public string ItemName { get; set; }
    public int Quantity { get; set; }
    public decimal Price { get; set; }
}

Common Scryber troubleshooting issues to be aware of:

Scryber's Document.ParseDocument() expects a file path, not an HTML string. For in-memory templates, you must write to a temp file first (workaround shown above). This complicates unit testing and adds I/O overhead. The {{#each}} syntax requires specific object structure — if your data model doesn't match, you must project it to anonymous types. Modern layout techniques like Flexbox have limited support: if your HTML preview uses display: flex with complex justify/align properties, expect layout differences in PDF. JavaScript is not executed at all, so any client-side logic must be pre-computed server-side. Custom fonts via @font-face work but require absolute paths or embedded base64, and font metrics may differ slightly from browser preview.

IronPDF — Rendering HTML with Data Binding

IronPDF takes HTML strings, files, or URLs directly. No template file requirement, no special syntax:

using IronPdf;
using System.Linq;

public void GenerateInvoice(int invoiceId, string outputPath)
{
    var invoiceData = FetchInvoiceData(invoiceId);

    // Standard HTML with inline data
    string invoiceHtml = $@"
<!DOCTYPE html>
<html>
<head>
    <style>
        body {{ font-family: Arial, sans-serif; margin: 20px; }}
        .invoice-header {{
            display: flex;
            justify-content: space-between;
            align-items: center;
        }}
        table {{ width: 100%; border-collapse: collapse; margin-top: 20px; }}
        td, th {{ border: 1px solid #ddd; padding: 8px; }}
    </style>
</head>
<body>
    <div class='invoice-header'>
        <h1>Invoice #{invoiceData.InvoiceNumber}</h1>
        <p>Date: {invoiceData.Date:yyyy-MM-dd}</p>
    </div>

    <table>
        <thead>
            <tr><th>Item</th><th>Quantity</th><th>Price</th></tr>
        </thead>
        <tbody>
            {string.Join("", invoiceData.Items.Select(item => $@"
            <tr>
                <td>{item.ItemName}</td>
                <td>{item.Quantity}</td>
                <td>${item.Price:F2}</td>
            </tr>"))}
        </tbody>
    </table>

    <p><strong>Total: ${invoiceData.Total:F2}</strong></p>
</body>
</html>";

    // Render HTML to PDF
    var renderer = new ChromePdfRenderer();
    using var pdf = renderer.RenderHtmlAsPdf(invoiceHtml);
    pdf.SaveAs(outputPath);
}

Learn more about HTML to PDF conversion and handling HTML files.

IronPDF advantages for troubleshooting: HTML renders identically to Chrome browser — preview in browser, get same output in PDF. No template file I/O; work with HTML strings directly in memory. Full CSS Grid, Flexbox, and modern layout support. JavaScript executes before PDF generation if needed.

How Does Complex CSS Rendering Compare?

Answer:

Scryber.Core — Handling Complex CSS

Scryber's CSS support requires workarounds for modern features:

using Scryber.Components;
using System.IO;

public void RenderComplexLayout(string outputPath)
{
    string templateHtml = @"
<!DOCTYPE html>
<html xmlns='http://www.w3.org/1999/xhtml'>
<head>
    <style>
        /* Modern CSS features with caveats */
        :root {
            /* CSS variables: supported */
            --primary-color: #3498db;
            --spacing: 20px;
        }

        body {
            font-family: 'Segoe UI', Tahoma, sans-serif;
            margin: var(--spacing);
            color: #333;
        }

        /* CSS Grid: NOT supported - verify in docs */
        /* If you need grid layout, use tables or floats */
        .grid-container {
            /* display: grid; - this will be ignored or cause issues */
            display: block; /* Fallback to block layout */
        }

        /* Flexbox: LIMITED support */
        .flex-container {
            display: flex; /* Basic flex works */
            /* Complex properties may not work as expected: */
            /* flex-wrap: wrap; - verify behavior */
            /* justify-content: space-between; - may not render correctly */
        }

        /* Use traditional layouts for reliability */
        .two-column {
            width: 100%;
        }
        .two-column .column {
            float: left;
            width: 48%;
            margin-right: 2%;
        }

        /* Advanced selectors: partial support */
        /* :nth-child(odd) - may work */
        /* ::before, ::after - limited support */

        .table-striped tr:nth-child(even) {
            background-color: #f2f2f2; /* Test this */
        }
    </style>
</head>
<body>
    <h1 style='color: var(--primary-color);'>Complex Layout Test</h1>

    <!-- Use table-based layout for reliability -->
    <table style='width: 100%;'>
        <tr>
            <td style='width: 50%; vertical-align: top;'>
                <p>Column 1 content</p>
            </td>
            <td style='width: 50%; vertical-align: top;'>
                <p>Column 2 content</p>
            </td>
        </tr>
    </table>
</body>
</html>";

    File.WriteAllText("complex_layout.html", templateHtml);

    using (var doc = Document.ParseDocument("complex_layout.html"))
    {
        // Additional rendering options
        // Verify available properties in Scryber documentation
        // Some options may not exist or behave differently

        using var stream = new FileStream(outputPath, FileMode.Create);
        doc.SaveAsPDF(stream);
    }
}

Scryber CSS troubleshooting checklist: always preview in browser first, then check if Scryber renders the same layout. Test CSS features in isolation before combining them in production templates. Use traditional layouts (tables, floats) instead of Grid/Flexbox for complex designs. Check GitHub issues for known CSS limitations before implementing. Expect pagination differences compared to the CSS Paged Media spec. Validate fonts load by checking output carefully — silent font fallback happens.

IronPDF — Rendering Complex CSS

IronPDF uses Chrome, so modern CSS just works:

using IronPdf;

public void RenderComplexLayout(string outputPath)
{
    string html = @"
<!DOCTYPE html>
<html>
<head>
    <style>
        :root {
            --primary-color: #3498db;
            --spacing: 20px;
        }

        body {
            font-family: 'Segoe UI', Tahoma, sans-serif;
            margin: var(--spacing);
        }

        /* CSS Grid: fully supported */
        .grid-container {
            display: grid;
            grid-template-columns: repeat(3, 1fr);
            gap: var(--spacing);
        }

        /* Flexbox: fully supported */
        .flex-container {
            display: flex;
            justify-content: space-between;
            flex-wrap: wrap;
            align-items: center;
        }

        /* Modern selectors: all supported */
        .grid-item:nth-child(3n) {
            background: linear-gradient(135deg, #667eea 0%, #764ba2 100%);
            color: white;
        }

        .card::before {
            content: '→';
            margin-right: 10px;
        }
    </style>
</head>
<body>
    <h1 style='color: var(--primary-color);'>Complex Layout</h1>

    <div class='grid-container'>
        <div class='card'>Item 1</div>
        <div class='card'>Item 2</div>
        <div class='card'>Item 3</div>
    </div>

    <div class='flex-container'>
        <p>Flex Item 1</p>
        <p>Flex Item 2</p>
        <p>Flex Item 3</p>
    </div>
</body>
</html>";

    var renderer = new ChromePdfRenderer();
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs(outputPath);
}

No workarounds needed. Preview in Chrome DevTools, get identical PDF output.

How Do You Install Each Library?

Answer:

Scryber.Core Installation

# Install base package
Install-Package Scryber.Core

# Optional: MVC extensions
Install-Package Scryber.Core.Mvc

# Optional: OpenType font support
Install-Package Scryber.Core.OpenType

# Namespace imports:
using Scryber.Components;
using Scryber.Components.Mvc; // If using MVC extensions

Lightweight installation (~5-10MB), no native dependencies. Requires template files for most use cases (HTML files on disk).

IronPDF Installation

# Single package with native dependencies
Install-Package IronPdf

# Namespace import:
using IronPdf;

Larger package (~50MB) due to bundled Chromium engine. Works with HTML strings, files, or URLs. No external dependencies or template file requirements.

Frequently Asked Questions

Answer:

Can Scryber.Core execute JavaScript during PDF generation?

No. Scryber.Core does not support JavaScript execution at all. Any dynamic content must be pre-rendered server-side before binding. IronPDF executes JavaScript via its embedded Chromium engine — useful for charts rendered by Chart.js, dynamic forms, or content loaded via AJAX. See the JavaScript in HTML to PDF guide for details.

Does Scryber.Core support CSS Grid and Flexbox?

CSS Grid is not mentioned in Scryber's documentation — verify current support before relying on it. Flexbox has limited support; basic display: flex may work, but complex properties like flex-wrap, justify-content: space-between, and align-items may not render correctly. IronPDF supports both fully through Chrome's rendering engine.

What are the licensing differences?

Scryber.Core is open-source under LGPL 3.0. Commercial use is allowed, but any modifications to Scryber itself must be shared under LGPL terms. Your application code does not need to be open-sourced if you link without modifying the library. IronPDF is a commercial product with paid licenses and includes 24/5 engineering support with SLA availability.

How does package size compare?

Scryber.Core is approximately 5MB with no native dependencies. IronPDF is approximately 50MB due to bundling the Chromium engine. The size difference reflects the architectural choice: Scryber uses a custom rendering engine while IronPDF embeds a full browser engine for pixel-perfect rendering fidelity.

Can I debug PDF layout issues with browser DevTools?

With IronPDF, yes. Because IronPDF uses Chrome's rendering engine, you can design and debug your HTML in Chrome DevTools and get identical output in the generated PDF. With Scryber.Core, debugging layout issues requires understanding Scryber's internal box model, which differs from what browser DevTools shows. This means a guess-and-check workflow when CSS doesn't render as expected.

When Should You Use Scryber.Core?

Answer:

Scryber.Core fits specific scenarios where its trade-offs align with project constraints. If package size matters more than CSS compatibility — embedded systems, IoT devices, or environments with strict size limits — Scryber's ~5MB footprint beats IronPDF's ~50MB significantly. Teams already using XML/HTML templates with data binding may find Scryber's handlebars-style syntax familiar. Open-source projects under LGPL or compatible licenses can use Scryber without licensing costs.

Simple document layouts — reports with basic tables, text, and images, no complex responsive layouts — work reliably in Scryber. If your PDF templates look like "formatted Word documents" rather than "web pages," Scryber's limitations won't surface often.

When Should You Use IronPDF?

Answer:

For most modern .NET teams, IronPDF's architecture aligns better with contemporary development practices. If your workflow is "design in browser, export to PDF," IronPDF's Chrome-based rendering eliminates the "looks good in browser, breaks in PDF" problem. Teams using modern CSS (Grid, Flexbox, transforms, animations) avoid Scryber's partial-support uncertainty.

Any scenario requiring JavaScript execution — charts rendered by Chart.js, dynamic forms with validation, content loaded via AJAX — requires a JavaScript-capable renderer. Scryber can't execute JS; IronPDF can. Enterprise teams needing guaranteed response times and engineering escalation can't rely on GitHub issues. IronPDF provides 24/5 support (24/7 in Premium) with SLAs.

When CSS doesn't render as expected, Chrome DevTools let you inspect, modify, and test fixes in real time. Scryber requires guess-and-check iterations because its rendering engine doesn't expose browser-like inspection tools. For teams building new PDF workflows, the question is: do you optimize for package size (Scryber) or development speed (IronPDF)? Most teams choose speed.
