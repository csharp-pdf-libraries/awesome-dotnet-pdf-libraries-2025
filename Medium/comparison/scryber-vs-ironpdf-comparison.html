Scryber.Core vs IronPDF: Solving Common .NET PDF Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Two pipelines render the same invoice template. The first writes display: grid in a stylesheet that a custom layout engine reads; the second writes the same line and hands the HTML to Chromium. The first relies on whatever CSS the engine's maintainer has implemented; the second inherits whatever Chrome supports today. That gap — between "valid CSS" and "CSS that this particular engine actually paginates" — is the architectural choice this article is about.

Scryber.Core is an open-source, template-driven PDF library for .NET that emphasizes programmatic document layout and data binding with a handlebars-style template syntax. It's built entirely in C# with no browser dependencies, which makes it portable and lightweight. However, this architectural choice creates a predictability problem: HTML/CSS support depends on the maintainer implementing each spec feature manually. If you're reaching for Scryber, this comparison will help you understand when its approach fits — and when Chrome-based rendering would save troubleshooting time.

What Is IronPDF?

IronPDF uses Google Chrome's rendering engine (Chromium/Blink) to generate PDFs, which means it supports whatever Chrome supports: CSS Grid, Flexbox, CSS variables, web fonts, SVG, and JavaScript execution. When you call ChromePdfRenderer.RenderHtmlAsPdf(), your HTML renders the way Chrome's own "Print to PDF" feature would render it. If a CSS property works in Chrome DevTools, it should work in your PDF.

The trade-off is footprint: IronPDF bundles the Chromium engine (~50MB compressed NuGet package vs Scryber's ~5MB). For teams that value "write HTML once, render everywhere" over minimal package size, that's an acceptable trade. IronPDF targets .NET 10, 9, 8, 7, 6, Core, Framework, and runs on Windows, Linux, macOS, Docker, and cloud platforms without modification.

What Are the Key Limitations of Scryber.Core?

Scryber.Core is an open-source project under LGPL 3.0; commercial use is allowed but modifications to Scryber itself must be shared under LGPL. The project is maintained on GitHub, and community support is via GitHub issues — there is no dedicated commercial support team.

Scryber.Core implements a custom PDF rendering engine rather than embedding a browser engine, so HTML/CSS support is scoped to whatever the maintainer has implemented. CSS Grid is not documented as supported — verify against the current release before relying on it. Advanced selectors (:nth-of-type, attribute selectors, pseudo-elements beyond the basics) typically have partial support. JavaScript execution is not supported — any dynamic content must be pre-rendered server-side. Modern web fonts via @font-face are accepted but font metrics may differ from browser rendering. SVG filters, CSS animations, and advanced transforms are generally outside scope.

The layout engine uses its own flow logic, so pagination and box-model behavior may differ from the CSS Paged Media spec and from browser DevTools. Tables with complex content can require manual padding/margin tuning. Line-height in dense text may shift slightly compared to a browser-based renderer.

The template syntax mixes XHTML with handlebars-style binding ({{variable}}), and data binding happens at render time via the Document.Params dictionary rather than a strongly-typed MVC-Razor model. Large documents (100+ pages) can render slowly because the layout pipeline is largely single-threaded. There is no built-in HTTP client for fetching remote HTML with custom headers, cookies, or authentication — you fetch it yourself and hand the string to ParseDocument. Debugging layout issues means reasoning about Scryber's internal box model rather than inspecting in browser DevTools.

How Does HTML Rendering with Data Binding Compare?

Scryber.Core — Rendering HTML with Data Binding

Scryber parses an XHTML string through a StringReader via Document.ParseDocument(reader, string.Empty, ParseSourceType.DynamicContent), with handlebars-style placeholders bound through doc.Params:

using Scryber.Components;
using System;
using System.Collections.Generic;
using System.IO;
using System.Linq;

public class ScryberInvoiceGenerator
{
    public void GenerateInvoice(int invoiceId, string outputPath)
    {
        string templateHtml = @"
<!DOCTYPE html>
<html xmlns='http://www.w3.org/1999/xhtml'>
<head>
    <title>Invoice {{invoiceNumber}}</title>
    <style>
        body { font-family: Arial, sans-serif; margin: 20px; }
        /* CSS Grid: verify support against your Scryber version */
        .invoice-header { display: block; }
        table { width: 100%; border-collapse: collapse; }
        td, th { border: 1px solid #ddd; padding: 8px; }
    </style>
</head>
<body>
    <div class='invoice-header'>
        <h1>Invoice #{{invoiceNumber}}</h1>
        <p>Date: {{invoiceDate}}</p>
    </div>

    <table>
        <thead>
            <tr><th>Item</th><th>Quantity</th><th>Price</th></tr>
        </thead>
        <tbody>
            <template data-bind='{{#each items}}'>
                <tr>
                    <td>{{name}}</td>
                    <td>{{quantity}}</td>
                    <td>${{price}}</td>
                </tr>
            </template>
        </tbody>
    </table>

    <p><strong>Total: ${{total}}</strong></p>
</body>
</html>";

        var invoiceData = FetchInvoiceData(invoiceId);

        // Parse the HTML string in-memory via a StringReader; no temp file required.
        using (var reader = new StringReader(templateHtml))
        using (var doc = Document.ParseDocument(reader, string.Empty, ParseSourceType.DynamicContent))
        using (var stream = new FileStream(outputPath, FileMode.Create))
        {
            doc.Params["invoiceNumber"] = invoiceData.InvoiceNumber;
            doc.Params["invoiceDate"] = invoiceData.Date.ToString("yyyy-MM-dd");
            doc.Params["total"] = invoiceData.Total;

            // Project to anonymous types whose property names match the {{#each}} keys.
            doc.Params["items"] = invoiceData.Items.Select(i => new
            {
                name = i.ItemName,
                quantity = i.Quantity,
                price = i.Price
            }).ToArray();

            doc.SaveAsPDF(stream);
        }
    }

    private InvoiceData FetchInvoiceData(int invoiceId)
    {
        // Implementation details...
        return new InvoiceData();
    }
}

public class InvoiceData
{
    public string InvoiceNumber { get; set; }
    public DateTime Date { get; set; }
    public decimal Total { get; set; }
    public List<LineItem> Items { get; set; }
}

public class LineItem
{
    public string ItemName { get; set; }
    public int Quantity { get; set; }
    public decimal Price { get; set; }
}

A few practical notes when working with Scryber templates:

The {{#each}} binding expects a collection whose item property names match the placeholders, which often means projecting your domain models to anonymous types. Flexbox support is partial — complex justify-content / align-items combinations may render differently than a browser preview, so verify against your version. JavaScript is not executed, so any client-side logic must be pre-computed server-side before binding. Custom fonts via @font-face work but font metrics can differ slightly from a browser, which sometimes shifts line-height in dense text.

IronPDF — Rendering HTML with Data Binding

IronPDF takes HTML strings, files, or URLs directly. No template file requirement, no special syntax:

using IronPdf;
using System.Linq;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

public void GenerateInvoice(int invoiceId, string outputPath)
{
    var invoiceData = FetchInvoiceData(invoiceId);

    // Standard HTML with inline data
    string invoiceHtml = $@"
<!DOCTYPE html>
<html>
<head>
    <style>
        body {{ font-family: Arial, sans-serif; margin: 20px; }}
        .invoice-header {{
            display: flex;
            justify-content: space-between;
            align-items: center;
        }}
        table {{ width: 100%; border-collapse: collapse; margin-top: 20px; }}
        td, th {{ border: 1px solid #ddd; padding: 8px; }}
    </style>
</head>
<body>
    <div class='invoice-header'>
        <h1>Invoice #{invoiceData.InvoiceNumber}</h1>
        <p>Date: {invoiceData.Date:yyyy-MM-dd}</p>
    </div>

    <table>
        <thead>
            <tr><th>Item</th><th>Quantity</th><th>Price</th></tr>
        </thead>
        <tbody>
            {string.Join("", invoiceData.Items.Select(item => $@"
            <tr>
                <td>{item.ItemName}</td>
                <td>{item.Quantity}</td>
                <td>${item.Price:F2}</td>
            </tr>"))}
        </tbody>
    </table>

    <p><strong>Total: ${invoiceData.Total:F2}</strong></p>
</body>
</html>";

    // Render HTML to PDF
    var renderer = new ChromePdfRenderer();
    using var pdf = renderer.RenderHtmlAsPdf(invoiceHtml);
    pdf.SaveAs(outputPath);
}

Learn more about HTML to PDF conversion and handling HTML files.

IronPDF advantages for troubleshooting: HTML renders identically to Chrome browser — preview in browser, get same output in PDF. No template file I/O; work with HTML strings directly in memory. Full CSS Grid, Flexbox, and modern layout support. JavaScript executes before PDF generation if needed.

How Does Complex CSS Rendering Compare?

Scryber.Core — Handling Complex CSS

Scryber's CSS coverage is a subset of the spec, so modern layout features often need traditional fallbacks:

using Scryber.Components;
using Scryber.Drawing;
using Scryber.PDF;
using System.IO;

public void RenderComplexLayout(string outputPath)
{
    string templateHtml = @"
<!DOCTYPE html>
<html xmlns='http://www.w3.org/1999/xhtml'>
<head>
    <style>
        :root {
            /* CSS variables are supported */
            --primary-color: #3498db;
            --spacing: 20px;
        }

        body {
            font-family: 'Segoe UI', Tahoma, sans-serif;
            margin: var(--spacing);
            color: #333;
        }

        /* CSS Grid is not documented as supported — verify against your version */
        .grid-container {
            display: block;
        }

        /* Flexbox: basic support; verify complex alignment properties */
        .flex-container {
            display: flex;
        }

        /* Float-based two-column fallback for portable layouts */
        .two-column { width: 100%; }
        .two-column .column {
            float: left;
            width: 48%;
            margin-right: 2%;
        }

        .table-striped tr:nth-child(even) {
            background-color: #f2f2f2;
        }
    </style>
</head>
<body>
    <h1 style='color: var(--primary-color);'>Complex Layout Test</h1>

    <table style='width: 100%;'>
        <tr>
            <td style='width: 50%; vertical-align: top;'>
                <p>Column 1 content</p>
            </td>
            <td style='width: 50%; vertical-align: top;'>
                <p>Column 2 content</p>
            </td>
        </tr>
    </table>
</body>
</html>";

    using (var reader = new StringReader(templateHtml))
    using (var doc = Document.ParseDocument(reader, string.Empty, ParseSourceType.DynamicContent))
    using (var stream = new FileStream(outputPath, FileMode.Create))
    {
        doc.SaveAsPDF(stream);
    }
}

A practical workflow with Scryber: preview your HTML in a browser, then validate that Scryber renders the same layout. Test CSS features in isolation before combining them in production templates. Where Grid or complex Flexbox is involved, fall back to tables or floats for predictable output. Pagination behavior may differ from the CSS Paged Media spec, and font loading can fall back silently — so confirm output rather than assuming the browser preview is authoritative.

IronPDF — Rendering Complex CSS

IronPDF uses Chrome, so modern CSS just works:

How Do the APIs Map Between Libraries?

  • Document.ParseDocument(reader, string.Empty, ParseSourceType.DynamicContent) — maps to ChromePdfRenderer.RenderHtmlAsPdf(html). IronPDF accepts an HTML string directly without a StringReader or source-type flag.
  • doc.Params[key] = value — replaced by C# string interpolation or any standard templating engine (Razor, Scriban, Handlebars.Net) before handing the HTML to RenderHtmlAsPdf.
  • {{variable}} placeholder syntax — replaced by C# interpolation {variable} in a $"..." string.
  • {{#each collection}} with a <template> element — replaced by LINQ .Select(...) + string.Join("", ...) to build the inner markup.
  • doc.SaveAsPDF(stream) — maps to pdf.SaveAs(path), or use pdf.Stream / pdf.BinaryData for in-memory output.
  • Scryber expression engine — not needed; you compute values in C# and bind them through interpolation.
  • Custom font embedding — replaced by standard CSS @font-face, which Chrome handles natively.
  • Page size and margins declared in @page CSS inside the template — map to renderer.RenderingOptions.PaperSize, PaperOrientation, and the MarginTop/MarginBottom/MarginLeft/MarginRight properties on ChromePdfRenderer.

What Does the Full Feature Comparison Look Like?

  • Project status and licensing. Scryber.Core is open source under LGPL 3.0, maintained on GitHub. IronPDF is a commercial product with a paid licensing model. Both target current .NET, including .NET 8, 9, and 10.
  • Documentation and support. Scryber relies on GitHub-hosted docs and community issues. IronPDF ships with hosted tutorials and offers paid support plans.
  • HTML and CSS engine. Scryber accepts XHTML with a CSS subset interpreted by its own layout engine. IronPDF uses Chrome's rendering engine, so CSS3, Grid, Flexbox, modern selectors, and pseudo-elements behave the same as in the browser.
  • JavaScript. Scryber does not execute JavaScript — content must be pre-rendered server-side. IronPDF runs JavaScript via Chromium and exposes WaitFor hooks for async-rendered content.
  • PDF operations. IronPDF has first-class APIs for merging, splitting, HTML- or image-based watermarks, and digital signatures. Equivalent operations in Scryber are limited — verify what your version exposes against the current docs.
  • Security. IronPDF supports AES-256 encryption, separate user/owner passwords, and granular permissions through SecuritySettings. Scryber supports password protection; granular permission flags should be verified against your version.
  • Layout fidelity. Scryber's custom engine introduces small differences in font metrics, table cell heights, and pagination relative to a browser. IronPDF inherits Chrome's metrics, which makes "preview in browser, render in PDF" predictable.
  • Templating model. Scryber uses XHTML templates plus its handlebars-style binding through doc.Params. IronPDF works with any string-producing template approach (Razor, Scriban, plain interpolation), since it consumes whatever HTML you produce.
  • Debugging. Scryber's box model is internal and lacks a DevTools surface. IronPDF lets you iterate in Chrome DevTools and re-render with the same engine.
  • Performance. Scryber's layout pipeline is largely single-threaded, which can slow large documents. IronPDF benefits from Chromium's optimized rendering pipeline.

How Do You Install Each Library?

Scryber.Core Installation

# Install base package
Install-Package Scryber.Core

# Namespace imports:
using Scryber.Components;
using Scryber.Drawing;
using Scryber.PDF;

Lightweight package, no native dependencies. Accepts HTML strings in memory via a StringReader handed to Document.ParseDocument(reader, string.Empty, ParseSourceType.DynamicContent); no template-file-on-disk requirement.

IronPDF Installation

# Single package with native dependencies
Install-Package IronPdf

# Namespace import:
using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

Larger package due to the bundled Chromium engine. Works with HTML strings, files, or URLs.

Frequently Asked Questions

Can Scryber.Core execute JavaScript during PDF generation?

No. Scryber.Core does not document JavaScript execution as a feature — its rendering pipeline is static, so any dynamic content must be pre-rendered server-side before binding. IronPDF executes JavaScript via its embedded Chromium engine — useful for charts rendered by Chart.js, dynamic forms, or content loaded via AJAX. See the JavaScript in HTML to PDF guide for details.

Does Scryber.Core support CSS Grid and Flexbox?

CSS Grid is not mentioned in Scryber's documentation — verify current support before relying on it. Flexbox has limited support; basic display: flex may work, but complex properties like flex-wrap, justify-content: space-between, and align-items may not render correctly. IronPDF supports both fully through Chrome's rendering engine.

What are the licensing differences?

Scryber.Core is open source under LGPL 3.0. Commercial use is allowed, but any modifications to Scryber itself must be shared under LGPL terms. Your application code does not need to be open-sourced if you link without modifying the library. IronPDF is a commercial product with paid licenses and includes paid support plans.

How does package size compare?

Scryber.Core is approximately 5MB with no native dependencies. IronPDF is approximately 50MB due to bundling the Chromium engine. The size difference reflects the architectural choice: Scryber uses a custom rendering engine while IronPDF embeds a full browser engine for browser-aligned rendering fidelity.

Can I debug PDF layout issues with browser DevTools?

With IronPDF, yes. Because IronPDF uses Chrome's rendering engine, you can design and debug your HTML in Chrome DevTools and get visually consistent output in the generated PDF. With Scryber.Core, debugging layout issues typically involves reasoning about Scryber's internal box model, which differs from what browser DevTools shows.

When Should You Use Scryber.Core?

Scryber.Core fits specific scenarios where its trade-offs align with project constraints. If package size matters more than CSS compatibility — embedded systems, IoT devices, or environments with strict size limits — Scryber's smaller footprint is a real advantage over a Chromium-bundled library. Teams already using XHTML templates with data binding may find Scryber's handlebars-style syntax familiar. Open-source projects under LGPL or compatible licenses can use Scryber without licensing costs.

Simple document layouts — reports with basic tables, text, and images, no complex responsive layouts — work reliably in Scryber. If your PDF templates look like "formatted Word documents" rather than "web pages," Scryber's limitations won't surface often.

When Should You Use IronPDF?

For most modern .NET teams, IronPDF's architecture aligns better with contemporary development practices. If your workflow is "design in browser, export to PDF," IronPDF's Chrome-based rendering eliminates the "looks good in browser, breaks in PDF" problem. Teams using modern CSS (Grid, Flexbox, transforms, animations) avoid Scryber's partial-support uncertainty.

Any scenario requiring JavaScript execution — charts rendered by Chart.js, dynamic forms with validation, content loaded via AJAX — requires a JavaScript-capable renderer. Scryber's pipeline is static; IronPDF runs the script through Chromium before snapshotting the page. Teams that need contractual response times and direct vendor escalation may also prefer a commercial library with paid support plans over a GitHub-issue workflow.

When CSS doesn't render as expected, Chrome DevTools let you inspect, modify, and test fixes in real time. Scryber's custom rendering pipeline does not expose a browser-style inspection surface, so debugging is more iterative. For teams building new PDF workflows, the question is: do you optimize for package size (Scryber) or for matching browser rendering exactly (IronPDF)? Most teams I have seen choose the latter.
