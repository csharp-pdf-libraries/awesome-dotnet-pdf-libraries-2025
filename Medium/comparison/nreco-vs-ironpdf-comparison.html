NReco PDF Generator vs IronPDF: What the Docs Do Not Tell You

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Picture a logistics app generating 500 shipping labels per hour during peak season. The team chose NReco.PdfGenerator two years ago because it was free for single-server deployments and the syntax looked straightforward. Six months in, production logs started filling with cryptic "Exit with code 1, due to unknown error" messages. No stack traces. No meaningful context. Just intermittent failures that vanished on retry.

The underlying problem: NReco spawns wkhtmltopdf.exe as an external process for every PDF conversion. When file permissions get misaligned, temp directories fill up, or Windows security policies tighten, the wrapper can't tell you what actually failed — it just sees an exit code from a black-box executable it doesn't control.

This isn't a story about NReco being broken. It's a story about technical debt from architectural decisions made over a decade ago, when Qt WebKit seemed like a reasonable rendering engine and process-based wrappers felt acceptable. But WebKit was deprecated in 2013. Wkhtmltopdf saw its last significant update in 2020. The ecosystem moved on.

I want to walk through what still works, where the friction surfaces, and when the math tips toward modern alternatives so your team can make a confident decision in 2026.

What Is IronPDF?

Answer:

IronPDF renders HTML to PDF using an embedded Chromium engine — the same Blink renderer that powers Chrome, Edge, and Electron apps. This means HTML5, CSS Grid, Flexbox, ES6 JavaScript, and WOFF2 fonts work exactly as they do in a modern browser. No "this CSS property isn't supported" surprises. No external executables to spawn or monitor.

The library provides both HTML-to-PDF rendering and comprehensive PDF manipulation in one package: merge documents, extract text, add watermarks, apply digital signatures, fill forms programmatically. For teams managing document workflows beyond simple generation, this consolidates tooling instead of forcing integration across multiple libraries.

What Are the Key Limitations of NReco PDF Generator?

Answer:

Last NuGet update January 2023. Built on wkhtmltopdf 0.12.6 from June 2020. Qt WebKit engine deprecated since 2013 — no CSS Grid, no Flexbox, no ES6 modules, no modern web standards introduced in the past decade.

No PDF merging, splitting, or manipulation — you need iTextSharp or another library. No built-in watermarking. No digital signatures. No form filling. No text extraction. HTML-to-PDF only, nothing else.

Process-based architecture creates race conditions under load. Temp file management causes production errors when permissions aren't perfect. JavaScript support exists but executes in Qt WebKit context (vintage 2013 behavior). CSS3 features like flex, grid, @supports don't work. WOFF2 fonts unsupported.

Community support only — no SLA, no dedicated engineering team. GitHub issues show process timeouts, permission errors, IIS deployment problems. Many remain unresolved. The product appears in maintenance mode rather than active development.

External .exe dependency complicates Docker deployments (manual binary management). Windows-only by default — cross-platform requires paid LT version plus separate wkhtmltopdf installation. Process spawning adds latency (hundreds of milliseconds baseline). Diagnostic information poor — "exit code 1" tells you nothing useful when debugging.

How Do You Convert an HTML String to PDF with Headers and Footers?

Answer:

NReco — HTML String to PDF with Custom Headers

using NReco.PdfGenerator;
using System;
using System.IO;

namespace PdfGenerationExample
{
    class Program
    {
        static void Main(string[] args)
        {
            // Configure converter instance
            var pdfConverter = new HtmlToPdfConverter();

            // Set page dimensions
            pdfConverter.PageWidth = 210;  // A4 width in mm
            pdfConverter.PageHeight = 297; // A4 height in mm
            pdfConverter.Margins = new PageMargins
            {
                Top = 20,
                Bottom = 20,
                Left = 15,
                Right = 15
            };

            // Header configuration - text only, no HTML
            pdfConverter.PageHeaderHtml = "Report Generated: " + DateTime.Now.ToShortDateString();

            // HTML content
            string htmlContent = @"
                <html>
                <head>
                    <style>
                        body { font-family: Arial; }
                        .invoice { border: 1px solid #ccc; padding: 20px; }
                    </style>
                </head>
                <body>
                    <div class='invoice'>
                        <h1>Monthly Report</h1>
                        <p>Generated on " + DateTime.Now + @"</p>
                    </div>
                </body>
                </html>";

            // Execute conversion (spawns wkhtmltopdf.exe process)
            byte[] pdfBytes = pdfConverter.GeneratePdf(htmlContent);

            // Save output
            File.WriteAllBytes("nreco_output.pdf", pdfBytes);
        }
    }
}

Headers are text-only; no HTML styling or complex layouts supported. Process spawning adds 200-500ms baseline latency per conversion. The external .exe must have file system write permissions to temp directories. No diagnostic output when the process fails — just exit codes. CSS Grid, Flexbox, and modern web fonts won't render. JavaScript execution limited to Qt WebKit capabilities (ECMAScript 5 era).

IronPDF — HTML String to PDF with HTML Headers

using IronPdf;

// Initialize renderer
var renderer = new ChromePdfRenderer();

// Configure with full HTML header support
renderer.RenderingOptions.HtmlHeader = new HtmlHeaderFooter
{
    Height = 20,
    HtmlFragment = "<div style='text-align:center; font-size:10px;'>Report Generated: {date}</div>",
    DrawDividerLine = true
};

// HTML content with modern CSS
string htmlContent = @"
    <style>
        .invoice {
            display: grid;
            grid-template-columns: 1fr 1fr;
            gap: 20px;
        }
    </style>
    <div class='invoice'>
        <h1>Monthly Report</h1>
        <p>Generated on {date}</p>
    </div>";

// Render with Chromium engine
var pdf = renderer.RenderHtmlAsPdf(htmlContent);
pdf.SaveAs("ironpdf_output.pdf");

IronPDF headers support full HTML/CSS styling and dynamic placeholders like {page}, {total-pages}, {date}. The ChromePdfRenderer handles CSS Grid and modern layouts without workarounds. Learn more about HTML string rendering techniques.

How Do You Convert a URL to PDF with JavaScript Execution?

Answer:

NReco — Rendering URLs with JavaScript Wait

using NReco.PdfGenerator;
using System;
using System.IO;

namespace UrlToPdfExample
{
    class Program
    {
        static void Main(string[] args)
        {
            var converter = new HtmlToPdfConverter();

            // Enable JavaScript in Qt WebKit context
            converter.EnableJavaScript = true;

            // Wait time for JS execution (milliseconds)
            converter.JavaScriptDelay = 1000;

            // Set output options
            converter.Orientation = PageOrientation.Portrait;
            converter.Size = PageSize.A4;

            // Margins configuration
            converter.Margins.Top = 10;
            converter.Margins.Bottom = 10;
            converter.Margins.Left = 10;
            converter.Margins.Right = 10;

            // Additional wkhtmltopdf flags
            converter.Quiet = false; // Enable console output for debugging

            try
            {
                // Spawns external process, blocks until complete
                var pdfBytes = converter.GeneratePdfFromFile(
                    "https://example.com/dashboard",
                    null
                );

                File.WriteAllBytes("url_output.pdf", pdfBytes);
            }
            catch (PdfGenerationException ex)
            {
                // Exception message typically just shows exit code
                Console.WriteLine($"Generation failed: {ex.Message}");
            }
        }
    }
}

Qt WebKit understands ECMAScript 5 only (no arrow functions, no const/let, no async/await). The external process call blocks the thread — no true async support. JavaScript timing is a fixed delay, not "wait until page load complete." Network errors from wkhtmltopdf.exe appear as generic exit codes. TLS 1.2+ support depends on the Qt WebKit version compiled in 2020.

IronPDF — Async URL Rendering with Wait-for-Load

using IronPdf;
using System.Threading.Tasks;

async Task GenerateFromUrl()
{
    var renderer = new ChromePdfRenderer();

    // Modern JavaScript execution
    renderer.RenderingOptions.EnableJavaScript = true;
    renderer.RenderingOptions.RenderDelay = 500; // ms after page load
    renderer.RenderingOptions.WaitFor.RenderDelay(1000); // or wait for specific element

    // True async rendering
    var pdf = await renderer.RenderUrlAsPdfAsync(new Uri("https://example.com/dashboard"));
    await pdf.SaveAsAsync("url_output.pdf");
}

IronPDF's Chromium engine understands modern JavaScript (ES2020+) and handles dynamic SPAs that rely on fetch, promises, and async rendering. The async methods don't block threads. Detailed walkthrough: Pixel-perfect HTML to PDF rendering.

How Do You Convert a Local HTML File with External Assets?

Answer:

NReco — File Conversion with Asset References

using NReco.PdfGenerator;
using System;
using System.IO;

namespace FileToPdfExample
{
    class Program
    {
        static void Main(string[] args)
        {
            var converter = new HtmlToPdfConverter();

            // Configure dimensions
            converter.Size = PageSize.Letter;
            converter.Orientation = PageOrientation.Portrait;
            converter.Margins = new PageMargins(15, 15, 15, 15);

            // Zoom affects text size
            converter.Zoom = 1.0f;

            // Background graphics
            converter.GenerateGrayscale = false;
            converter.LowQuality = false;

            // Footer with page numbers - text only
            converter.PageFooterHtml = "Page {page} of {topage}";

            try
            {
                // File path must be absolute
                string absolutePath = Path.GetFullPath("template.html");

                // Relative asset paths in HTML must be absolute
                // or relative to the HTML file location
                byte[] pdfContent = converter.GeneratePdfFromFile(
                    absolutePath,
                    null  // No separate output path needed for byte array
                );

                File.WriteAllBytes("file_output.pdf", pdfContent);
            }
            catch (Exception ex)
            {
                Console.WriteLine($"Conversion error: {ex.Message}");
                // Debugging: check if wkhtmltopdf binaries extracted correctly
                // Default location: App_Data/wkhtmltopdf/
            }
        }
    }
}

Asset paths must be absolute or correctly relative to the HTML file location. Qt WebKit can't load WOFF2 fonts (only TTF/WOFF). CSS imports via @import may fail depending on file:// protocol restrictions. Image formats limited to JPEG, PNG, GIF, BMP — no WebP, AVIF. The process needs read permissions on all referenced files.

IronPDF — File Rendering with Auto Asset Resolution

using IronPdf;

var renderer = new ChromePdfRenderer();

// Footer with flexible placeholders
renderer.RenderingOptions.HtmlFooter = new HtmlHeaderFooter
{
    Height = 15,
    HtmlFragment = "<center><i>Page {page} of {total-pages}</i></center>"
};

// Automatically resolves relative paths from file's directory
var pdf = renderer.RenderHtmlFileAsPdf("template.html");
pdf.SaveAs("file_output.pdf");

IronPDF automatically sets the base URL to the HTML file's directory, so relative paths to CSS, images, and fonts work without manual configuration. Supports all modern image formats (WebP, AVIF) and font formats (WOFF2, variable fonts). Complete file conversion guide: HTML File to PDF Tutorial.

How Do You Handle Batch Processing of Multiple HTML Files?

Answer:

NReco — Sequential File Batch Processing

using NReco.PdfGenerator;
using System;
using System.Collections.Generic;
using System.IO;
using System.Linq;

namespace BatchProcessingExample
{
    class Program
    {
        static void Main(string[] args)
        {
            var converter = new HtmlToPdfConverter();

            // Shared configuration
            converter.Size = PageSize.A4;
            converter.Margins = new PageMargins(20, 20, 20, 20);

            // Timeout for each process spawn
            converter.ExecutionTimeout = 60000; // 60 seconds

            string[] htmlFiles = Directory.GetFiles("invoices/", "*.html");

            var results = new List<(string filename, bool success, string error)>();

            // Must process sequentially - no parallel support
            foreach (var htmlFile in htmlFiles)
            {
                try
                {
                    byte[] pdfBytes = converter.GeneratePdfFromFile(htmlFile, null);

                    string outputPath = Path.ChangeExtension(htmlFile, ".pdf");
                    File.WriteAllBytes(outputPath, pdfBytes);

                    results.Add((htmlFile, true, null));
                }
                catch (PdfGenerationException ex)
                {
                    // Common error: "Exit with code 1" - not actionable
                    results.Add((htmlFile, false, ex.Message));
                }
                catch (Exception ex)
                {
                    results.Add((htmlFile, false, ex.Message));
                }
            }

            // Report results
            int succeeded = results.Count(r => r.success);
            int failed = results.Count(r => !r.success);

            Console.WriteLine($"Completed: {succeeded} succeeded, {failed} failed");

            foreach (var result in results.Where(r => !r.success))
            {
                Console.WriteLine($"Failed: {result.filename} - {result.error}");
            }
        }
    }
}

Process spawning prevents efficient parallelization (creates process handle leaks under load). Timeout management is per-file, with no batch-level controls. Memory overhead accumulates because each conversion spawns a new .exe. Error handling provides minimal context — just exit codes. No progress callbacks or cancellation support. Temp file cleanup failures can cause cascading problems in long-running batches.

IronPDF — Parallel Batch with Thread Safety

using IronPdf;
using System.IO;
using System.Linq;

var renderer = new ChromePdfRenderer();
var htmlFiles = Directory.GetFiles("invoices/", "*.html");

// Thread-safe parallel processing
Parallel.ForEach(htmlFiles, htmlFile =>
{
    var pdf = renderer.RenderHtmlFileAsPdf(htmlFile);
    pdf.SaveAs(Path.ChangeExtension(htmlFile, ".pdf"));
});

IronPDF's Chromium engine is thread-safe and designed for parallel execution. No external processes to manage, no temp file conflicts, no mysterious exit codes. The library handles resource pooling internally.

How Do You Install Each Library?

Answer:

NReco.PdfGenerator Installation

# Windows-only version (includes wkhtmltopdf binaries)
dotnet add package NReco.PdfGenerator

# Cross-platform version (requires manual wkhtmltopdf installation)
dotnet add package NReco.PdfGenerator.LT

using NReco.PdfGenerator;

// Windows: works immediately after NuGet restore
// Linux/macOS: requires wkhtmltopdf installation and path configuration
var converter = new HtmlToPdfConverter();

// For LT version on Linux
converter.WkHtmlToPdfExeName = "wkhtmltopdf";
converter.PdfToolPath = "/usr/local/bin/"; // where wkhtmltopdf is installed

NReco requires Visual C++ 2015 Runtime on Windows. Docker deployments need manual binary management. The free version limits usage to single-server production environments.

dotnet add package IronPdf

using IronPdf;

// Works immediately on Windows, Linux, macOS, Docker
// No external executables, no runtime dependencies
var renderer = new ChromePdfRenderer();

IronPDF is fully managed .NET code with embedded Chromium engine. No native dependencies to install. Cross-platform support included without separate packages. Complete installation steps: Chrome PDF Rendering Engine Guide.

Frequently Asked Questions

Can NReco render CSS Grid or Flexbox layouts?

Answer:

No. NReco is built on wkhtmltopdf 0.12.6, which uses the Qt WebKit engine deprecated since 2013. CSS Grid, Flexbox, @supports, and other modern CSS features introduced in the past decade are not supported. You are limited to table-based layouts. IronPDF's Chromium Blink engine supports full modern CSS including Grid, Flexbox, and variable fonts.

Is NReco thread-safe for parallel PDF generation?

Answer:

Not recommended. NReco spawns an external wkhtmltopdf.exe process for every conversion. Under parallel load, this creates process handle leaks, temp file conflicts, and race conditions. IronPDF is fully thread-safe and designed for parallel execution with Parallel.ForEach, handling resource pooling internally without external processes.

Does NReco support async/await?

Answer:

No. NReco's process-based architecture blocks the calling thread while the external wkhtmltopdf.exe runs. There are no native async methods. IronPDF provides full async support with methods like RenderHtmlAsPdfAsync and RenderUrlAsPdfAsync that don't block threads.

Can I merge or digitally sign PDFs with NReco?

Answer:

No. NReco is HTML-to-PDF only. There is no PDF merging, splitting, text extraction, form filling, watermarking, digital signatures, encryption, or password protection. You would need a separate library like iTextSharp for those features. IronPDF includes all of these capabilities in a single package.

How does NReco handle Docker deployments compared to IronPDF?

Answer:

NReco requires manual wkhtmltopdf binary management in Docker containers. You need the paid LT version for cross-platform support, plus a separate wkhtmltopdf installation and path configuration. IronPDF works out of the box in Docker with no external executables or native dependencies to install.

When Should You Stay on NReco, and When Should You Migrate?

If your HTML templates use table-based layouts (no CSS Grid/Flexbox), you rarely update them, and you're running in a stable Windows environment where the process spawning hasn't caused issues, staying on NReco can make sense — especially if you're on the free tier and budgets are tight.

Migration becomes necessary when CSS Grid/Flexbox layouts fail to render, modern JavaScript breaks, process timeouts appear in production logs, or you need PDF manipulation features NReco doesn't provide (merging, digital signatures, form filling). Docker deployments or cross-platform requirements also push toward modern alternatives because the manual wkhtmltopdf binary management adds operational overhead.

The pragmatic decision point: how much engineering time are you spending working around NReco's limitations versus investing in a migration? For teams generating hundreds of PDFs daily with evolving template requirements, the return on migration effort typically appears within a quarter.

IronPDF consolidates HTML rendering and PDF manipulation into one library with a Chromium-based engine that handles modern web standards. Thread-safe parallel processing replaces process spawning. Full async support prevents thread blocking. No external executables to debug when something fails.
