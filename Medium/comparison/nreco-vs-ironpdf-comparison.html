NReco PDF Generator vs IronPDF: What the Docs Do Not Tell You

One architectural fact frames this comparison: NReco.PdfGenerator spawns wkhtmltopdf.exe as an external process for every conversion, and the upstream wkhtmltopdf project was archived on January 2, 2023 with 0.12.6 (June 2020) as its final release. The wrapper itself still works, but the engine it wraps is no longer maintained, and that constraint shapes everything downstream — security posture, CSS support, JavaScript runtime, and how diagnostics surface when something fails.

When the spawned process exits non-zero, the wrapper can only surface that exit code; it cannot reach inside an executable it does not control. When file permissions shift, temp directories fill up, or Windows security policies tighten, that diagnostic gap becomes visible.

This isn't a story about NReco being broken — it is a working wrapper around an engine choice made more than a decade ago, when Qt WebKit was a reasonable rendering target. The ecosystem moved on.

The rest of this article walks through what still works in NReco, where the friction surfaces, and when the math tips toward a modern alternative.

What Is IronPDF?

IronPDF renders HTML to PDF using an embedded Chromium engine — the same Blink renderer that powers Chrome, Edge, and Electron apps. HTML5, CSS Grid, Flexbox, ES6 JavaScript, and WOFF2 fonts render the same way they do in a modern browser. No external executables to spawn or monitor.

The library provides both HTML-to-PDF rendering and comprehensive PDF manipulation in one package: merge documents, extract text, add watermarks, apply digital signatures, fill forms programmatically. For teams managing document workflows beyond simple generation, this consolidates tooling instead of forcing integration across multiple libraries.

What Are the Key Limitations of NReco PDF Generator?

NReco wraps wkhtmltopdf 0.12.6 (June 2020). The wkhtmltopdf project itself was archived by its maintainers on January 2, 2023, so the underlying engine no longer receives upstream updates. Qt WebKit, the renderer wkhtmltopdf is built on, has been deprecated since 2013, which typically means no CSS Grid, no Flexbox, and no support for newer JavaScript or layout features added since.

NReco's scope is HTML-to-PDF generation. It does not provide PDF merging, splitting, watermarking, digital signatures, form filling, or text extraction — those tasks require a separate library such as iText or PdfSharp.

The process-based architecture has two consequences worth planning around. Spawning wkhtmltopdf.exe per conversion adds baseline latency, and parallel load can run into temp-file or permission contention. JavaScript executes inside the Qt WebKit context bundled with that 2020 wkhtmltopdf build, so modern flex, grid, @supports, and WOFF2 fonts may not render reliably — verify against your templates.

Community support only; there is no SLA or dedicated engineering channel. The free version is licensed for single-server deployments — cross-platform and multi-server scenarios use the paid LT package and require a separate wkhtmltopdf installation. Diagnostics are limited to the wkhtmltopdf exit code, which can make IIS or Docker debugging slower.

How Do You Convert an HTML String to PDF with Headers and Footers?

NReco — HTML String to PDF with Custom Headers

using NReco.PdfGenerator;
using System;
using System.IO;

namespace PdfGenerationExample
{
    class Program
    {
        static void Main(string[] args)
        {
            // Configure converter instance
            var pdfConverter = new HtmlToPdfConverter();

            // Set page dimensions
            pdfConverter.PageWidth = 210;  // A4 width in mm
            pdfConverter.PageHeight = 297; // A4 height in mm
            pdfConverter.Margins = new PageMargins
            {
                Top = 20,
                Bottom = 20,
                Left = 15,
                Right = 15
            };

            // PageHeaderHtml accepts a plain text string (no HTML markup)
            pdfConverter.PageHeaderHtml = "Report Generated: " + DateTime.Now.ToShortDateString();

            // HTML content
            string htmlContent = @"
                <html>
                <head>
                    <style>
                        body { font-family: Arial; }
                        .invoice { border: 1px solid #ccc; padding: 20px; }
                    </style>
                </head>
                <body>
                    <div class='invoice'>
                        <h1>Monthly Report</h1>
                        <p>Generated on " + DateTime.Now + @"</p>
                    </div>
                </body>
                </html>";

            // GeneratePdf invokes the bundled wkhtmltopdf executable
            byte[] pdfBytes = pdfConverter.GeneratePdf(htmlContent);

            // Save output
            File.WriteAllBytes("nreco_output.pdf", pdfBytes);
        }
    }
}

PageHeaderHtml takes a plain string, so HTML styling in the header is not available — use IronPDF or another tool if you need a styled header. Process spawning adds baseline latency per conversion (typically a few hundred milliseconds), and the executable needs write access to a temp directory. When the wkhtmltopdf process fails, the wrapper surfaces an exit code rather than a structured diagnostic. CSS Grid, Flexbox, and WOFF2 web fonts are not supported by the Qt WebKit engine bundled with this wkhtmltopdf build, and JavaScript runs in that older runtime.

IronPDF headers support full HTML/CSS styling and dynamic placeholders like {page}, {total-pages}, {date}. The ChromePdfRenderer handles CSS Grid and modern layouts without workarounds. Learn more about HTML string rendering techniques.

How Do You Convert a URL to PDF with JavaScript Execution?

NReco — Rendering URLs with JavaScript Wait

using NReco.PdfGenerator;
using System;
using System.IO;

namespace UrlToPdfExample
{
    class Program
    {
        static void Main(string[] args)
        {
            var converter = new HtmlToPdfConverter();

            // Enable JavaScript execution in the wkhtmltopdf engine
            converter.EnableJavaScript = true;

            // Fixed delay applied after page load (milliseconds)
            converter.JavaScriptDelay = 1000;

            // Page setup
            converter.Orientation = PageOrientation.Portrait;
            converter.Size = PageSize.A4;

            // Margins (mm)
            converter.Margins.Top = 10;
            converter.Margins.Bottom = 10;
            converter.Margins.Left = 10;
            converter.Margins.Right = 10;

            // Forward wkhtmltopdf stderr for debugging
            converter.Quiet = false;

            try
            {
                // GeneratePdfFromFile also accepts a URL as input
                var pdfBytes = converter.GeneratePdfFromFile(
                    "https://example.com/dashboard",
                    null
                );

                File.WriteAllBytes("url_output.pdf", pdfBytes);
            }
            catch (PdfGenerationException ex)
            {
                Console.WriteLine($"Generation failed: {ex.Message}");
            }
        }
    }
}

The Qt WebKit JavaScript runtime in this wkhtmltopdf build is older than ES6 — features like arrow functions, const/let, and async/await may not work. JavaScriptDelay is a fixed wall-clock wait, not an event-based "wait until ready" hook, so timing-sensitive single-page apps can be tricky. GeneratePdfFromFile is synchronous and blocks the calling thread. Network and TLS support depend on the Qt build compiled into the 2020 wkhtmltopdf release.

IronPDF — Async URL Rendering with Wait-for-Load

using IronPdf;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

async Task GenerateFromUrl()
{
    var renderer = new ChromePdfRenderer();

    renderer.RenderingOptions.EnableJavaScript = true;
    renderer.RenderingOptions.WaitFor.RenderDelay(1000); // ms after page load

    var pdf = await renderer.RenderUrlAsPdfAsync(new Uri("https://example.com/dashboard"));
    await pdf.SaveAsAsync("url_output.pdf");
}

IronPDF's Chromium engine understands modern JavaScript (ES2020+) and handles dynamic SPAs that rely on fetch, promises, and async rendering. The async methods don't block threads. Detailed walkthrough: Pixel-perfect HTML to PDF rendering.

How Do You Convert a Local HTML File with External Assets?

NReco — File Conversion with Asset References

using NReco.PdfGenerator;
using System;
using System.IO;

namespace FileToPdfExample
{
    class Program
    {
        static void Main(string[] args)
        {
            var converter = new HtmlToPdfConverter();

            // Page setup
            converter.Size = PageSize.Letter;
            converter.Orientation = PageOrientation.Portrait;
            converter.Margins = new PageMargins(15, 15, 15, 15);

            converter.Zoom = 1.0f;

            converter.GenerateGrayscale = false;
            converter.LowQuality = false;

            // wkhtmltopdf placeholder syntax uses square brackets
            converter.PageFooterHtml = "Page [page] of [topage]";

            try
            {
                // GeneratePdfFromFile expects an absolute path
                string absolutePath = Path.GetFullPath("template.html");

                // Asset paths in HTML resolve relative to the HTML file location
                byte[] pdfContent = converter.GeneratePdfFromFile(
                    absolutePath,
                    null
                );

                File.WriteAllBytes("file_output.pdf", pdfContent);
            }
            catch (Exception ex)
            {
                Console.WriteLine($"Conversion error: {ex.Message}");
                // Default extraction location: App_Data/wkhtmltopdf/
            }
        }
    }
}

Asset paths must be absolute or correctly relative to the HTML file location. The bundled Qt WebKit engine supports TTF and WOFF fonts but typically not WOFF2 — verify against your font set. @import CSS rules over file:// can be limited depending on the wkhtmltopdf flags in use. Common image formats (JPEG, PNG, GIF, BMP) are supported; newer formats such as WebP and AVIF are not. The process needs read permissions on all referenced files.

IronPDF — File Rendering with Auto Asset Resolution

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();

// Footer with HTML markup and IronPDF placeholders
renderer.RenderingOptions.HtmlFooter = new HtmlHeaderFooter
{
    MaxHeight = 15,
    HtmlFragment = "<center><i>Page {page} of {total-pages}</i></center>"
};

// Relative paths in the HTML resolve against the file's directory automatically
var pdf = renderer.RenderHtmlFileAsPdf("template.html");
pdf.SaveAs("file_output.pdf");

IronPDF automatically sets the base URL to the HTML file's directory, so relative paths to CSS, images, and fonts work without manual configuration. Supports all modern image formats (WebP, AVIF) and font formats (WOFF2, variable fonts). Complete file conversion guide: HTML File to PDF Tutorial.

How Do You Handle Batch Processing of Multiple HTML Files?

NReco — Sequential File Batch Processing

using NReco.PdfGenerator;
using System;
using System.Collections.Generic;
using System.IO;
using System.Linq;

namespace BatchProcessingExample
{
    class Program
    {
        static void Main(string[] args)
        {
            var converter = new HtmlToPdfConverter();

            // Shared configuration
            converter.Size = PageSize.A4;
            converter.Margins = new PageMargins(20, 20, 20, 20);

            // Per-conversion timeout
            converter.ExecutionTimeout = 60000; // 60 seconds

            string[] htmlFiles = Directory.GetFiles("invoices/", "*.html");

            var results = new List<(string filename, bool success, string error)>();

            // Sequential loop - each iteration spawns a new wkhtmltopdf process
            foreach (var htmlFile in htmlFiles)
            {
                try
                {
                    byte[] pdfBytes = converter.GeneratePdfFromFile(htmlFile, null);

                    string outputPath = Path.ChangeExtension(htmlFile, ".pdf");
                    File.WriteAllBytes(outputPath, pdfBytes);

                    results.Add((htmlFile, true, null));
                }
                catch (PdfGenerationException ex)
                {
                    results.Add((htmlFile, false, ex.Message));
                }
                catch (Exception ex)
                {
                    results.Add((htmlFile, false, ex.Message));
                }
            }

            // Report results
            int succeeded = results.Count(r => r.success);
            int failed = results.Count(r => !r.success);

            Console.WriteLine($"Completed: {succeeded} succeeded, {failed} failed");

            foreach (var result in results.Where(r => !r.success))
            {
                Console.WriteLine($"Failed: {result.filename} - {result.error}");
            }
        }
    }
}

The per-conversion process model makes parallelism harder than a thread-pool: each call spawns a new wkhtmltopdf.exe, and timeout/cancellation are per-file rather than batch-level. Error context is limited to the wkhtmltopdf exit code and any stderr it forwards. Long-running batches should also watch for temp-file accumulation when the executable is killed mid-conversion.

IronPDF — Parallel Batch with Thread Safety

using IronPdf;
using System.IO;
using System.Linq;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var htmlFiles = Directory.GetFiles("invoices/", "*.html");

Parallel.ForEach(htmlFiles, htmlFile =>
{
    var pdf = renderer.RenderHtmlFileAsPdf(htmlFile);
    pdf.SaveAs(Path.ChangeExtension(htmlFile, ".pdf"));
});

The IronPDF renderer is documented as thread-safe and is designed to be reused across parallel calls. There is no external executable to spawn, no temp-file contention, and exit-code-style failures are replaced with structured .NET exceptions. Resource pooling is handled by the library.

What Does the Feature Comparison Look Like?

  • Current status. NReco's last NuGet release is January 2023 and wraps wkhtmltopdf 0.12.6 (June 2020). IronPDF ships ongoing releases through NuGet.
  • HTML engine. NReco uses Qt WebKit via wkhtmltopdf — limited support for CSS Grid and Flexbox. IronPDF uses Chromium Blink, with the HTML5/CSS3/modern-JavaScript surface you would expect from Chrome.
  • Rendering surface. NReco is best matched to table-based layouts and legacy fonts. IronPDF targets the same rendering pipeline as a current desktop browser.
  • Installation. NReco extracts the wkhtmltopdf binaries on first run and requires the Visual C++ runtime on Windows. IronPDF is a single managed NuGet package that bundles its rendering engine.
  • Support. NReco is supported through community channels. IronPDF offers commercial engineering support with SLA options on paid tiers.
  • Future viability. NReco wraps an upstream project (wkhtmltopdf) that was archived on January 2, 2023, so engine-level updates are no longer expected. IronPDF tracks Chromium and continues active feature work.

Drilling deeper, here is how the two libraries compare across the categories teams typically evaluate:

  • Status and roadmap. NReco — last release January 2023, no public roadmap, upstream wkhtmltopdf archived. IronPDF — ongoing releases with a public changelog and roadmap.
  • Support model. NReco — community forums and documentation. IronPDF — commercial engineering support with SLAs available on enterprise tiers.
  • HTML and CSS. NReco — Qt WebKit; CSS Grid and Flexbox are typically not supported in this build. IronPDF — Chromium Blink, with the full modern CSS surface.
  • JavaScript. NReco — JavaScript runs in the older Qt WebKit runtime bundled with wkhtmltopdf 0.12.6. IronPDF — current Chromium V8.
  • Fonts. NReco — TTF and WOFF; WOFF2 typically unsupported. IronPDF — all common formats including WOFF2 and variable fonts.
  • Headers and footers. NReco — plain-text only; placeholders use square brackets like [page] and [topage]. IronPDF — full HTML/CSS markup with {page}, {total-pages}, {date}, etc.
  • Scope. NReco — HTML-to-PDF only; merging, splitting, watermarks, signatures, form filling, encryption, and text/image extraction require an additional library such as iText. IronPDF — these operations are part of the same package.
  • Security and TLS. NReco — depends on the Qt build compiled into the 2020 wkhtmltopdf release. IronPDF — modern TLS via Chromium.
  • Concurrency. NReco — synchronous API; the wkhtmltopdf.exe process model can run into temp-file or permission contention under load. IronPDF — async methods and a renderer documented as thread-safe.
  • Deployment. NReco on Windows includes the wkhtmltopdf binary; cross-platform requires the paid LT package and a separate wkhtmltopdf install. Docker images need that binary added explicitly. IronPDF ships engine binaries via NuGet for Windows, Linux, and macOS, including Docker base images.
  • .NET targets. Both run on modern .NET; IronPDF documents support up to current .NET LTS releases.

What Does the API Mapping Between NReco and IronPDF Look Like?

  • HtmlToPdfConverter — replaced by ChromePdfRenderer.
  • GeneratePdf(string html) — RenderHtmlAsPdf(string html); IronPDF returns a PdfDocument rather than a byte[], so call .BinaryData if you need the raw bytes.
  • GeneratePdfFromFile(string path, null) — RenderHtmlFileAsPdf(string path) for a local file, or RenderUrlAsPdf(string url) when the input is an HTTP(S) URL.
  • EnableJavaScript — RenderingOptions.EnableJavaScript.
  • JavaScriptDelay — RenderingOptions.WaitFor.RenderDelay(ms); IronPDF also exposes WaitFor.JavaScript and WaitFor.HtmlElementById for event-based waits.
  • PageHeaderHtml (plain text) — RenderingOptions.HtmlHeader = new HtmlHeaderFooter { HtmlFragment = ... } (full HTML/CSS markup).
  • PageFooterHtml (plain text) — RenderingOptions.HtmlFooter = new HtmlHeaderFooter { HtmlFragment = ... }.
  • Margins / PageMargins — individual RenderingOptions.MarginTop, MarginBottom, MarginLeft, MarginRight values in millimetres.
  • Size (PageSize enum) — RenderingOptions.PaperSize (PdfPaperSize); custom sizes via SetCustomPaperSizeinMilimeters.
  • Orientation — RenderingOptions.PaperOrientation (PdfPaperOrientation).
  • Zoom — RenderingOptions.Zoom; note NReco uses a float multiplier (1.5 = 150%), IronPDF uses an integer percentage (150 = 150%).
  • GenerateGrayscale — RenderingOptions.GrayScale.
  • CustomWkHtmlArgs / CustomWkHtmlPageArgs — no direct equivalent; map common flags to typed RenderingOptions (for example FitToPaperMode, PrintHtmlBackgrounds, WaitFor.RenderDelay).
  • PdfToolPath / WkHtmlToPdfExeName — not applicable; IronPDF bundles its own engine.
  • ExecutionTimeout — RenderingOptions.Timeout in seconds, or a CancellationToken on the async APIs.
  • Header/footer placeholders — replace wkhtmltopdf's [page], [topage], [date] with IronPDF's {page}, {total-pages}, {date}.

Frequently Asked Questions

Can NReco render CSS Grid or Flexbox layouts?

No. NReco is built on wkhtmltopdf 0.12.6, which uses the Qt WebKit engine deprecated since 2013. CSS Grid, Flexbox, @supports, and other modern CSS features introduced in the past decade are not supported. You are limited to table-based layouts. IronPDF's Chromium Blink engine supports full modern CSS including Grid, Flexbox, and variable fonts.

Is NReco thread-safe for parallel PDF generation?

The library wraps an external wkhtmltopdf.exe invocation per conversion, which means concurrency is bounded by process spawning rather than threading. Under heavy parallel load this can lead to temp-file contention or permission errors — verify against your environment before scaling out. IronPDF's renderer is documented as thread-safe and is designed to be used with Parallel.ForEach, with resource pooling handled inside the library.

Does NReco support async/await?

No. NReco's API is synchronous and blocks the calling thread for the duration of the wkhtmltopdf invocation. IronPDF provides async equivalents such as RenderHtmlAsPdfAsync and RenderUrlAsPdfAsync.

Can I merge or digitally sign PDFs with NReco?

No. NReco is HTML-to-PDF only. There is no PDF merging, splitting, text extraction, form filling, watermarking, digital signatures, encryption, or password protection. You would need a separate library like iTextSharp for those features. IronPDF includes all of these capabilities in a single package.

How does NReco handle Docker deployments compared to IronPDF?

NReco requires manual wkhtmltopdf binary management in Docker containers. You need the paid LT version for cross-platform support, plus a separate wkhtmltopdf installation and path configuration. IronPDF works out of the box in Docker with no external executables or native dependencies to install.

When Should You Stay on NReco, and When Should You Migrate?

If your HTML templates use table-based layouts (no CSS Grid/Flexbox), you rarely update them, and you're running in a stable Windows environment where the process spawning hasn't caused issues, staying on NReco can make sense — especially if you're on the free tier and budgets are tight.

Migration becomes necessary when CSS Grid/Flexbox layouts fail to render, modern JavaScript breaks, process timeouts appear in production logs, or you need PDF manipulation features NReco doesn't provide (merging, digital signatures, form filling). Docker deployments or cross-platform requirements also push toward modern alternatives because the manual wkhtmltopdf binary management adds operational overhead.

The pragmatic decision point: how much engineering time are you spending working around NReco's limitations versus investing in a migration? For teams generating hundreds of PDFs daily with evolving template requirements, the return on migration effort typically appears within a quarter.

IronPDF consolidates HTML rendering and PDF manipulation into one library with a Chromium-based engine that handles modern web standards. Thread-safe parallel processing replaces process spawning. Full async support prevents thread blocking. No external executables to debug when something fails.
