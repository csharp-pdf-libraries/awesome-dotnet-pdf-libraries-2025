QuestPDF vs IronPDF: The .NET Developer's Decision Guide

"Do we program our layout in C#, or render the HTML we already have?" That single question separates QuestPDF from IronPDF more cleanly than any feature matrix. Both libraries are actively maintained, both produce PDFs, and both target the same .NET runtimes — but they solve fundamentally different problems. QuestPDF asks you to express document layout through a fluent C# API; IronPDF embeds a Chromium engine and renders your existing HTML, CSS, and JavaScript the way the browser would. Pick the one whose mental model matches the assets your team already has, and migration cost stays low; pick the other, and you will spend weeks rewriting either C# layout code or HTML templates.

This comparison exists because teams often confuse "PDF generation" with "HTML-to-PDF rendering." If you are building invoice templates from scratch and love strongly-typed layout DSLs, QuestPDF might be your tool. If you are converting web reports, dashboards, or any HTML you already have into PDFs, IronPDF's Chrome engine will save you from reinventing CSS layout rules in C#. This guide covers the architectural differences, real code for typical operations, and the tradeoffs that determine which tool fits your workflow.

What Is IronPDF?

IronPDF is a .NET library that embeds a Chromium-based rendering engine to convert HTML, CSS, and JavaScript into PDFs. If content displays correctly in Chrome, IronPDF renders it identically in PDF form. The library supports .NET Core, .NET 5+, and .NET Framework 4.6.2+, running on Windows, Linux, and macOS. Instead of programming document layouts in C#, you write HTML — or point IronPDF at existing web pages, Razor views, or HTML strings — and the Chrome engine handles all layout, font rendering, and responsive design.

IronPDF handles the full PDF lifecycle: generate from HTML, merge/split documents, add watermarks, apply digital signatures, extract text, fill forms, and encrypt with passwords. Because it uses Chrome's rendering stack, it supports modern CSS Grid, Flexbox, SVG, web fonts, and JavaScript-driven content. See the complete HTML-to-PDF tutorial for getting started.

What Are the Key Limitations of QuestPDF?

QuestPDF is actively maintained with regular releases. It is open-source under a tiered license model: the Community License (MIT terms) is free for individuals, nonprofits, and companies under $1M USD annual gross revenue, with paid Professional and Enterprise tiers above that threshold. The project has strong community adoption on GitHub, with PDF/A and PDF/UA conformance support among recent additions — verify against the version you adopt.

No HTML rendering engine: QuestPDF does not parse or render HTML/CSS. You cannot pass an HTML string, file, or URL and get a PDF. All document structure must be expressed using QuestPDF's fluent API in C#. If you have existing HTML reports, dashboards, or templates, you will need to rewrite them entirely in QuestPDF's DSL. This is a deliberate design choice, not a defect — but it is the single biggest factor in whether QuestPDF fits your workflow.

No web content conversion: Because there is no browser engine, QuestPDF cannot render JavaScript-driven content, responsive CSS layouts, or web fonts the way a browser would. It is a layout engine, not a browser engine.

No visual designer tooling out of the box: QuestPDF is code-first. Non-developers cannot edit templates in a visual editor or WYSIWYG tool without additional tooling. Compare this to HTML workflows where designers can iterate in browsers and hand off working HTML/CSS to developers.

Native dependency considerations: QuestPDF relies on SkiaSharp for rendering, which ships platform-specific native binaries. In containerized or restricted environments, native dependencies can require extra setup steps — verify base image and platform support against your deployment target.

Different mental model for web developers: Teams with HTML/CSS expertise face a paradigm shift. Instead of applying CSS rules, you call C# methods like .Column(), .Row(), .PaddingVertical(). This is productive if you think in code, but represents new learning if your team already has HTML templates or web design skills.

Community-led support model: QuestPDF is supported through GitHub discussions and community channels. Commercial support arrangements are tied to its licensing tiers; check the project's current support policy if you require formal SLAs for bug fixes.

No existing HTML reuse: If your organization has invested in HTML templates for reports, emails, or web views, you cannot leverage them directly. Migrating to QuestPDF requires reimplementing every layout in C#. For teams with dozens of templates, this is meaningful rewrite effort.

Fluent API verbosity for complex layouts: QuestPDF's API is concise for simple documents but can become verbose for complex nested structures. A multi-column invoice with dynamic tables, conditional sections, and precise positioning requires careful composition of nested fluent calls. Whether this is more or less intuitive than CSS depends on your team's background.

How Does Creating a Simple Report Compare?

QuestPDF — Programmatic Layout

When you need to build a report from scratch without HTML, QuestPDF's fluent API provides fine-grained control. Here is a complete example generating a simple report with header, table, and footer:

using QuestPDF.Fluent;
using QuestPDF.Helpers;
using QuestPDF.Infrastructure;
using System;
using System.Collections.Generic;

// Set license type (required)
QuestPDF.Settings.License = LicenseType.Community;

// Sample data model
public class ReportData
{
    public string ReportTitle { get; set; }
    public DateTime GeneratedDate { get; set; }
    public List<ReportItem> Items { get; set; }
}

public class ReportItem
{
    public string ProductName { get; set; }
    public int Quantity { get; set; }
    public decimal UnitPrice { get; set; }
    public decimal Total => Quantity * UnitPrice;
}

// Generate the report
var data = new ReportData
{
    ReportTitle = "Q4 Sales Report",
    GeneratedDate = DateTime.Now,
    Items = new List<ReportItem>
    {
        new ReportItem { ProductName = "Widget A", Quantity = 100, UnitPrice = 25.50m },
        new ReportItem { ProductName = "Gadget B", Quantity = 50, UnitPrice = 47.99m }
    }
};

Document.Create(container =>
{
    container.Page(page =>
    {
        page.Size(PageSizes.A4);
        page.Margin(2, Unit.Centimetre);
        page.PageColor(Colors.White);
        page.DefaultTextStyle(x => x.FontSize(11));

        page.Header()
            .Column(column =>
            {
                column.Item().Text(data.ReportTitle)
                    .FontSize(20).Bold();
                column.Item().Text($"Generated: {data.GeneratedDate:yyyy-MM-dd}")
                    .FontSize(9).FontColor(Colors.Grey.Medium);
            });

        page.Content()
            .PaddingVertical(1, Unit.Centimetre)
            .Column(column =>
            {
                column.Item().Table(table =>
                {
                    table.ColumnsDefinition(columns =>
                    {
                        columns.RelativeColumn(3);
                        columns.RelativeColumn(1);
                        columns.RelativeColumn(1);
                        columns.RelativeColumn(1);
                    });

                    table.Header(header =>
                    {
                        header.Cell().Background(Colors.Grey.Lighten2).Padding(5).Text("Product");
                        header.Cell().Background(Colors.Grey.Lighten2).Padding(5).Text("Qty");
                        header.Cell().Background(Colors.Grey.Lighten2).Padding(5).Text("Price");
                        header.Cell().Background(Colors.Grey.Lighten2).Padding(5).Text("Total");
                    });

                    foreach (var item in data.Items)
                    {
                        table.Cell().Border(1).Padding(5).Text(item.ProductName);
                        table.Cell().Border(1).Padding(5).AlignRight().Text(item.Quantity.ToString());
                        table.Cell().Border(1).Padding(5).AlignRight().Text($"${item.UnitPrice:F2}");
                        table.Cell().Border(1).Padding(5).AlignRight().Text($"${item.Total:F2}");
                    }
                });
            });

        page.Footer()
            .AlignCenter()
            .Text(x =>
            {
                x.Span("Page ");
                x.CurrentPageNumber();
            });
    });
})
.GeneratePdf("questpdf-report.pdf");

Technical limitations with this approach: no HTML reuse (if you already have report templates in HTML/Razor, you must rewrite them entirely in QuestPDF's API), web developers must learn QuestPDF's layout DSL instead of using existing CSS knowledge, you cannot test layouts in Chrome DevTools, non-developers cannot edit templates visually, nested tables with conditional sections require extensive fluent API composition, and you cannot render charts or dynamic content generated by JS libraries like Chart.js or D3.

IronPDF — HTML Rendering

IronPDF converts existing HTML directly to PDF. If you already have HTML reports, dashboards, or templates, you can render them without rewriting. Here is the same report generated from HTML:

using IronPdf;
using System;
using System.Collections.Generic;
using System.Text;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// Sample data model (same as QuestPDF example)
public class ReportData
{
    public string ReportTitle { get; set; }
    public DateTime GeneratedDate { get; set; }
    public List<ReportItem> Items { get; set; }
}

public class ReportItem
{
    public string ProductName { get; set; }
    public int Quantity { get; set; }
    public decimal UnitPrice { get; set; }
    public decimal Total => Quantity * UnitPrice;
}

// Generate report data
var data = new ReportData
{
    ReportTitle = "Q4 Sales Report",
    GeneratedDate = DateTime.Now,
    Items = new List<ReportItem>
    {
        new ReportItem { ProductName = "Widget A", Quantity = 100, UnitPrice = 25.50m },
        new ReportItem { ProductName = "Gadget B", Quantity = 50, UnitPrice = 47.99m }
    }
};

// Build HTML with embedded CSS
var html = $@"
<!DOCTYPE html>
<html>
<head>
    <style>
        body {{ font-family: Arial, sans-serif; margin: 40px; }}
        h1 {{ color: #333; }}
        .meta {{ color: #666; font-size: 0.9em; margin-bottom: 20px; }}
        table {{ width: 100%; border-collapse: collapse; margin-top: 20px; }}
        th {{ background-color: #f0f0f0; padding: 10px; text-align: left; border: 1px solid #ddd; }}
        td {{ padding: 10px; border: 1px solid #ddd; }}
        .number {{ text-align: right; }}
    </style>
</head>
<body>
    <h1>{data.ReportTitle}</h1>
    <div class='meta'>Generated: {data.GeneratedDate:yyyy-MM-dd}</div>
    <table>
        <thead>
            <tr><th>Product</th><th>Qty</th><th>Price</th><th>Total</th></tr>
        </thead>
        <tbody>
            {string.Join("", data.Items.Select(item => $@"
            <tr>
                <td>{item.ProductName}</td>
                <td class='number'>{item.Quantity}</td>
                <td class='number'>${item.UnitPrice:F2}</td>
                <td class='number'>${item.Total:F2}</td>
            </tr>"))}
        </tbody>
    </table>
</body>
</html>";

// Render HTML to PDF using Chrome engine
var renderer = new ChromePdfRenderer();
using var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("ironpdf-report.pdf");

IronPDF leverages Chrome's rendering engine, so any HTML/CSS that works in a browser renders identically in the PDF. You can test layouts in Chrome DevTools, use CSS Grid/Flexbox, include web fonts, and even render JavaScript-generated content. For more complex scenarios with external assets, see HTML string to PDF with base paths.

How Does Adding a Watermark Compare?

QuestPDF — Watermark via Background Layer

QuestPDF does not have a dedicated watermark API. Instead, you add watermarks by composing background layers using the document structure:

using QuestPDF.Fluent;
using QuestPDF.Helpers;
using QuestPDF.Infrastructure;
using System;

QuestPDF.Settings.License = LicenseType.Community;

Document.Create(container =>
{
    container.Page(page =>
    {
        page.Size(PageSizes.A4);
        page.Margin(2, Unit.Centimetre);

        // Apply watermark as background element
        page.Background()
            .AlignCenter()
            .AlignMiddle()
            .Rotate(-45)
            .Text("CONFIDENTIAL")
            .FontSize(72)
            .FontColor(Colors.Grey.Lighten3);

        // Content layer renders on top of background
        page.Content()
            .Column(column =>
            {
                column.Item().Text("Quarterly Financial Report").FontSize(24).Bold();
                column.Item().PaddingTop(10).Text("This is a sample document with a watermark.");
                column.Item().PaddingTop(10).Text("Lorem ipsum dolor sit amet, consectetur adipiscing elit.");
            });

        page.Footer()
            .AlignCenter()
            .Text(x =>
            {
                x.Span("Page ");
                x.CurrentPageNumber();
            });
    });
})
.GeneratePdf("questpdf-watermark.pdf");

Technical limitations: no .ApplyWatermark() method (you must structure background layers manually), watermark positioning relies on fluent API alignment, applying different watermarks to specific pages requires conditional logic, transparency is adjusted through color values rather than explicit opacity parameters, and adding image watermarks requires .Image() elements composed as background layers.

IronPDF — Apply Watermark API

IronPDF provides dedicated watermarking methods that accept HTML strings, allowing rich watermarks with full CSS control:

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// Load or create PDF
var renderer = new ChromePdfRenderer();
using var pdf = renderer.RenderHtmlAsPdf("<h1>Quarterly Financial Report</h1><p>This document contains sensitive information.</p>");

// Apply HTML-based watermark with rotation and opacity (0-100)
pdf.ApplyWatermark("<div style='font-size: 72px; color: #cccccc; font-weight: bold;'>CONFIDENTIAL</div>",
    rotation: -45,
    opacity: 30);

pdf.SaveAs("ironpdf-watermark.pdf");

IronPDF's watermark API supports HTML content, so you can include images, custom fonts, and complex styling. The method accepts rotation angle and opacity as parameters, making it straightforward to create professional watermarks. For more advanced scenarios like positioning watermarks at specific locations, see the custom watermark guide.

How Does Merging Multiple PDFs Compare?

QuestPDF — Generate Combined Document or Use Document Operations API

For sources that originate inside your QuestPDF code, you generate a single document programmatically that concatenates the content. For combining pre-existing PDF files, the Document Operations API (added in 2024.12.0) supports merge, page extraction, overlay/underlay, and attachments. Here is the in-code composition pattern:

using QuestPDF.Fluent;
using QuestPDF.Helpers;
using QuestPDF.Infrastructure;
using System;
using System.Collections.Generic;

QuestPDF.Settings.License = LicenseType.Community;

// Sample data for different sections
public class Section
{
    public string Title { get; set; }
    public string Content { get; set; }
}

var sections = new List<Section>
{
    new Section { Title = "Executive Summary", Content = "Overview of quarterly performance..." },
    new Section { Title = "Financial Details", Content = "Detailed breakdown of revenue and expenses..." },
    new Section { Title = "Future Outlook", Content = "Projections for next quarter..." }
};

Document.Create(container =>
{
    container.Page(page =>
    {
        page.Size(PageSizes.A4);
        page.Margin(2, Unit.Centimetre);
        page.DefaultTextStyle(x => x.FontSize(11));

        page.Content()
            .Column(column =>
            {
                foreach (var section in sections)
                {
                    column.Item().PaddingVertical(10).Column(sectionColumn =>
                    {
                        sectionColumn.Item().Text(section.Title).FontSize(18).Bold();
                        sectionColumn.Item().PaddingTop(5).Text(section.Content);
                    });

                    // Page break between sections
                    column.Item().PageBreak();
                }
            });

        page.Footer()
            .AlignCenter()
            .Text(x =>
            {
                x.Span("Page ");
                x.CurrentPageNumber();
            });
    });
})
.GeneratePdf("questpdf-combined.pdf");

For loading and combining pre-existing PDF files, QuestPDF 2024.12.0+ exposes a separate Document Operations API:

// QuestPDF Document Operations API (2024.12.0+)
DocumentOperation
    .LoadFile("section1.pdf")
    .TakePages("1-z")
    .MergeFile("section2.pdf", "1-z")
    .MergeFile("section3.pdf", "1-z")
    .Save("combined-report.pdf");

The Document Operations API covers merge, page extraction, overlay/underlay, attachments, encryption, and linearization. It does not edit existing page content, fill imported form fields, perform OCR, or apply digital signatures — for those workflows you would still reach for another tool.

IronPDF — Merge Existing PDFs

IronPDF can load and merge existing PDF files programmatically, preserving all content and metadata:

using IronPdf;
using System.Collections.Generic;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// Load existing PDFs from files
var pdf1 = PdfDocument.FromFile("section1.pdf");
var pdf2 = PdfDocument.FromFile("section2.pdf");
var pdf3 = PdfDocument.FromFile("section3.pdf");

// Merge into single document
var merged = PdfDocument.Merge(new List<PdfDocument> { pdf1, pdf2, pdf3 });
merged.SaveAs("combined-report.pdf");

// Clean up
pdf1.Dispose();
pdf2.Dispose();
pdf3.Dispose();
merged.Dispose();

IronPDF preserves all content, formatting, metadata, and embedded resources when merging. This is essential for workflows that need to combine PDFs from multiple sources (generated reports, scanned documents, third-party files). For additional merging scenarios and page extraction, see the merge and split PDFs guide.

How Does Applying Security Settings Compare?

QuestPDF — Encryption via Document Operations API

QuestPDF 2024.12.0+ supports password protection, 40/128/256-bit encryption, and permission flags through its Document Operations API. The pattern is to generate the PDF first with the fluent layout API, then run the file through DocumentOperation to apply encryption settings:

using QuestPDF.Fluent;
using QuestPDF.Helpers;
using QuestPDF.Infrastructure;

QuestPDF.Settings.License = LicenseType.Community;

// Step 1: Generate PDF with QuestPDF's fluent API
var outputPath = "unprotected-report.pdf";
Document.Create(container =>
{
    container.Page(page =>
    {
        page.Size(PageSizes.A4);
        page.Margin(2, Unit.Centimetre);
        page.Content()
            .Column(column =>
            {
                column.Item().Text("Confidential Report").FontSize(20).Bold();
                column.Item().PaddingTop(10).Text("This document contains sensitive information.");
            });
    });
})
.GeneratePdf(outputPath);

// Step 2: Apply encryption via Document Operations API (2024.12.0+)
DocumentOperation
    .LoadFile(outputPath)
    .Encrypt(new Encryption256Bit
    {
        UserPassword = "user123",
        OwnerPassword = "owner456",
        AllowPrinting = false,
        AllowContentExtraction = false
    })
    .Save("protected-report.pdf");

QuestPDF's Document Operations API covers user/owner passwords, 40/128/256-bit encryption, and permission flags (printing, content extraction, editing). It does not currently apply digital signatures or certificate-based signing — for PAdES or X.509 signing workflows you would still need a separate library.

IronPDF — Built-in Security Settings

IronPDF includes comprehensive security features directly in the API:

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// Generate or load PDF
var renderer = new ChromePdfRenderer();
using var pdf = renderer.RenderHtmlAsPdf("<h1>Confidential Report</h1><p>This document contains sensitive information.</p>");

// Apply security settings
pdf.SecuritySettings.RemovePasswordsAndEncryption(); // Clear existing security
pdf.SecuritySettings.MakePdfDocumentReadOnly("user123");
pdf.SecuritySettings.OwnerPassword = "owner456";
pdf.SecuritySettings.UserPassword = "user123";
pdf.SecuritySettings.AllowUserCopyPasteContent = false;
pdf.SecuritySettings.AllowUserAnnotations = false;
pdf.SecuritySettings.AllowUserPrinting = PdfPrintSecurity.NoPrinting;

pdf.SaveAs("secured-report.pdf");

IronPDF handles password protection, encryption (128-bit and 256-bit AES), and granular permissions in a single API. No external libraries required. For complete security workflows including metadata sanitization and digital signatures, see the PDF security tutorial.

What Does the Full Feature Comparison Look Like?

Two libraries with the same output format and very different scopes. Below is a feature-by-feature breakdown grouped by category.

Project status and support. Both libraries are actively maintained. QuestPDF is supported through GitHub discussions and community channels, with commercial support tied to its licensing tiers. IronPDF offers commercial support contracts with response-time and SLA options for paying customers — verify current support tiers and SLA terms against vendor pricing before committing.

Documentation. QuestPDF documentation focuses on getting started with the fluent API and common layout patterns. IronPDF publishes tutorials, how-to guides, and an API reference covering HTML-to-PDF, PDF manipulation, and security workflows.

  • HTML rendering. QuestPDF — not supported (by design). IronPDF — full Chromium engine.
  • CSS support. QuestPDF — none; layout is expressed through the fluent API. IronPDF — CSS3 including Grid, Flexbox, and custom fonts.
  • JavaScript execution. QuestPDF — none. IronPDF — yes, with render delays and wait conditions.
  • Responsive design. QuestPDF — manual via fluent API. IronPDF — native, since responsive CSS is interpreted by Chromium.
  • Web fonts. QuestPDF — embedded fonts only. IronPDF — web fonts via standard CSS.
  • SVG. QuestPDF — basic via image elements. IronPDF — full Chromium SVG rendering.
  • Existing HTML/Razor/ASPX reuse. QuestPDF — must rewrite in C#. IronPDF — render existing markup directly.

PDF operations. QuestPDF's primary surface is generating new documents through the fluent layout API. Since 2024.12.0 a separate Document Operations API also covers merging, page extraction, overlay/underlay, attachments, and linearization on existing files. It does not edit existing page content, fill imported form fields, perform OCR, or apply digital signatures. IronPDF covers merging, splitting, page extraction, watermarking via a dedicated ApplyWatermark API, digital signatures with X.509 certificates, form filling, text and image extraction, and redaction in the same API.

Security and encryption. QuestPDF's Document Operations API (2024.12.0+) supports user/owner passwords, 40/128/256-bit encryption, and permission flags applied as a post-generation step. Digital signatures are not yet supported. IronPDF provides user/owner passwords, AES encryption (128-bit and 256-bit), granular permissions (print, copy, edit, annotations), digital signing with X.509 certificates, and metadata controls in the same API.

Native dependencies. QuestPDF uses SkiaSharp, which ships native binaries per platform. IronPDF bundles a Chromium runtime. Both have container and Linux deployment paths — verify the current base-image and runtime requirements for your target platform.

Learning curve. QuestPDF is straightforward for developers who prefer strongly-typed, code-first layout. IronPDF is straightforward for teams who already know HTML and CSS.

Platform support. Both support .NET Standard 2.0, .NET Framework, modern .NET (5/6/7/8+), Windows, Linux, and macOS, and run in containers and major cloud platforms. Confirm the exact .NET target framework matrix against each library's current release notes.

Frequently Asked Questions

Can I use QuestPDF to convert existing HTML templates to PDF?

No. QuestPDF does not include an HTML rendering engine. All document structure must be expressed using QuestPDF's fluent C# API. If you have existing HTML reports or templates, you would need to rewrite them entirely in QuestPDF's DSL. IronPDF's Chrome-based engine renders any HTML/CSS/JS directly to PDF without rewriting.

Does QuestPDF support PDF security features like passwords and encryption?

Yes, since 2024.12.0. QuestPDF's Document Operations API supports user/owner passwords, 40/128/256-bit encryption, and permission flags (printing, content extraction, editing) applied as a post-generation step on the output file. Digital signatures and certificate-based signing are not currently supported. IronPDF includes the security settings directly in the same API as generation and adds X.509 digital signing alongside encryption and permissions.

Can QuestPDF merge existing PDF files from disk?

Yes, via the Document Operations API introduced in 2024.12.0. DocumentOperation.LoadFile().MergeFile() loads and combines pre-existing PDFs, and the same API covers page extraction, overlay/underlay, attachments, and linearization. Content editing of imported pages, form filling, and digital signatures are still out of scope. IronPDF's PdfDocument.Merge() covers the same merging case and adds the editing operations QuestPDF does not.

Which tool has a lower learning curve for web developers?

Teams with HTML/CSS expertise will find IronPDF more approachable because it leverages existing web skills. You write standard HTML and CSS, test in Chrome DevTools, and render to PDF. QuestPDF requires learning a fluent C# API where layout is expressed through method calls like .Column(), .Row(), and .PaddingVertical() rather than CSS rules. QuestPDF is more natural for developers who prefer strongly-typed, code-first approaches.

Which Tool Should You Choose?

QuestPDF is a strong fit when you are building document layouts from scratch in strongly-typed C# code. If your team prefers fluent APIs, does not have existing HTML templates, and wants compile-time safety for document structure, it is a solid choice. It is actively maintained, has wide community adoption, and handles programmatic layout cleanly. Just confirm that its tiered license model (free Community License under the $1M revenue threshold, paid Professional and Enterprise tiers above) fits your organization.

If your organization already has HTML reports, web-based dashboards, or any content expressed in HTML/CSS, migrating those to QuestPDF means rewriting them in C#. IronPDF's Chromium rendering engine lets you reuse existing templates, apply standard web design skills, and iterate in browser DevTools before generating PDFs. QuestPDF's Document Operations API (2024.12.0+) covers merging, page extraction, and encryption on existing files; IronPDF additionally covers digital signing, form filling, OCR, and content editing in the same package.

IronPDF's Chromium-based rendering aims for browser-faithful output for any HTML that renders correctly in Chrome. It supports modern CSS (Grid, Flexbox), web fonts, JavaScript-driven content, and responsive designs without requiring C# layout code, and bundles PDF manipulation features (merge/split, watermarking, digital signatures, password protection, form filling, and text extraction) in a single NuGet package with commercial support tiers available.
