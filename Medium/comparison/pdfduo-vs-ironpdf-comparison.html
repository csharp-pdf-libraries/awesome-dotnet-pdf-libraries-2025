PDF Duo .Net vs IronPDF in C#: Why It's Time to Leave a Decade-Old Library Behind

The .NET PDF landscape has shifted dramatically. A library whose last public release activity appears to be from over a decade ago cannot handle what modern C# applications require of an HTML-to-PDF engine.

Teams I have worked with describe a recurring pattern: a reporting system built around PDF Duo .Net years ago, deployed to an older .NET Framework, and left alone because "if it works, don't fix it." It churns out documents quietly for years.

Then compliance or product requirements shift. Reports need QR codes for document verification. CSS tables need better responsive sizing. Font embedding for international character sets becomes mandatory. The team opens the PDF generation module and realizes they are staring at archaeology — an engine whose feature set predates the modern web platform.

The architectural gap, not subjective "feels old" complaints, is what makes the conversation concrete. Here is what that gap looks like.

What is PDF Duo .Net and what is it designed for?

PDF Duo .Net is a lightweight HTML-to-PDF conversion library for .NET Framework, distributed as the PDFDuo.NET NuGet package with a PDFDuo namespace. Public release activity appears to have stopped over a decade ago, with no visible repository, no current vendor channels, and no record of recent security patches or .NET Core/.NET 5+ compatibility work.

In its time, PDF Duo addressed a specific problem: converting basic HTML with table-based layouts and system fonts into PDF documents. It targeted older .NET Framework versions and a CSS feature set that predates modern layout primitives.

This is not about PDF Duo being poorly made. For the web standards of its era, it solved problems effectively. But the standards that define modern HTML output — CSS Grid, Flexbox, ES6, WOFF2, variable fonts — were not established when PDF Duo's parser was written, and an unmaintained engine cannot pick them up retroactively.

Teams keeping ancient libraries in production aren't making technical decisions. They're making risk calculations: how much does this part of the system matter, and what breaks if we change it? This guide provides measurements to support that calculation.

How does an older custom-parser library compare to a modern Chromium-based renderer?

The gap shows up across every dimension that matters for production PDF generation.

  • .NET runtime target. PDF Duo was built against legacy .NET Framework versions; IronPDF runs on .NET Framework alongside .NET 6 through .NET 10+, so a framework migration does not have to wait on the PDF layer.
  • HTML parsing. PDF Duo uses a custom in-process HTML/CSS parser; IronPDF delegates rendering to Chromium, which means CSS3 layout primitives render the same way they do in Chrome.
  • Font support. PDF Duo handles basic font embedding; IronPDF supports WOFF2 and variable fonts via the browser engine, which avoids character-rendering surprises with international scripts.
  • Error diagnostics. PDF Duo surfaces exception messages only; IronPDF exposes detailed logs and standard .NET debugging surfaces.
  • Feature velocity. PDF Duo has no visible release cadence; IronPDF ships regular updates, which compounds in favor of the maintained dependency over time.

IronPDF uses Chromium's Blink rendering engine — the same technology behind Chrome, Edge, and Brave — to convert HTML to PDF. This means the library supports modern web standards by default: HTML5 semantic elements, CSS Grid layouts, ES2020+ JavaScript, WebAssembly modules, WOFF2 fonts. When Chrome browser adds support for new CSS features, IronPDF inherits them in the next release because they share the underlying rendering engine.

Beyond HTML-to-PDF conversion, IronPDF provides programmatic PDF manipulation: merge multiple documents, extract text and images, add watermarks, apply digital signatures, fill form fields, encrypt with passwords. For teams managing document workflows beyond simple generation, this consolidates functionality into a single well-maintained library.

How do the HTML conversion speeds compare?

The architectural difference between a custom HTML/CSS parser and a modern Chromium engine shows up clearly when you sketch out a 100-invoice batch on each side. The shapes below are illustrative — the goal is to show the API surface and the patterns each engine forces on you, not to publish vendor-specific timings.

PDF Duo — Basic HTML to PDF

// NuGet: Install-Package PDFDuo.NET
using PDFDuo;
using System;
using System.Diagnostics;
using System.IO;

namespace PdfDuoPerformance
{
    class Program
    {
        static void Main(string[] args)
        {
            var stopwatch = new Stopwatch();
            int iterations = 100;
            long totalMilliseconds = 0;

            for (int i = 0; i < iterations; i++)
            {
                stopwatch.Restart();

                // Table-based HTML, the layout shape this engine targets
                string html = @"
                    <html>
                    <head>
                        <style>
                            body { font-family: Arial; font-size: 12pt; }
                            .header { font-weight: bold; }
                        </style>
                    </head>
                    <body>
                        <div class='header'>Invoice #" + i + @"</div>
                        <p>Date: " + DateTime.Now.ToShortDateString() + @"</p>
                        <table border='1'>
                            <tr><td>Item</td><td>Price</td></tr>
                            <tr><td>Widget</td><td>$10.00</td></tr>
                        </table>
                    </body>
                    </html>";

                try
                {
                    // PDFDuo writes the PDF to a path directly; no in-memory PDF object is returned.
                    var converter = new HtmlToPdfConverter();
                    converter.ConvertHtmlString(html, $"output_{i}.pdf");

                    stopwatch.Stop();
                    totalMilliseconds += stopwatch.ElapsedMilliseconds;
                }
                catch (Exception ex)
                {
                    Console.WriteLine($"Conversion {i} failed: {ex.Message}");
                }
            }

            double averageMs = totalMilliseconds / (double)iterations;
            Console.WriteLine($"Average conversion time: {averageMs}ms");
            Console.WriteLine($"Total time: {totalMilliseconds}ms");
        }
    }
}

What the architecture implies for runtime behavior:

  • A custom HTML/CSS parser typically carries different overhead characteristics than a tuned browser engine — verify against your own workload.
  • The exposed API is synchronous and writes directly to a file path.
  • CSS3 layout primitives such as Grid and Flexbox are not part of the supported feature set documented for this engine.
  • Memory is allocated per call; there is no documented pooling primitive for batch scenarios.

IronPDF — Modern HTML Conversion Benchmark

Complete API reference: ChromePdfRenderer documentation. For converting HTML strings to PDF, IronPDF provides a clean async API built on Chromium.

using IronPdf;
using System;
using System.Diagnostics;
using System.Threading.Tasks;

class BenchmarkProgram
{
    static async Task Main(string[] args)
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var stopwatch = new Stopwatch();
        int iterations = 100;
        long totalMilliseconds = 0;

        var renderer = new ChromePdfRenderer();

        for (int i = 0; i < iterations; i++)
        {
            stopwatch.Restart();

            // Modern HTML with CSS Grid, Flexbox, modern fonts
            string html = $@"
                <style>
                    body {{ font-family: -apple-system, system-ui; }}
                    .invoice {{
                        display: grid;
                        grid-template-columns: 1fr 1fr;
                        gap: 20px;
                    }}
                </style>
                <div class='invoice'>
                    <div><strong>Invoice #{i}</strong></div>
                    <div>{DateTime.Now:yyyy-MM-dd}</div>
                </div>";

            var pdf = await renderer.RenderHtmlAsPdfAsync(html);
            await pdf.SaveAsAsync($"output_{i}.pdf");
            pdf.Dispose();

            stopwatch.Stop();
            totalMilliseconds += stopwatch.ElapsedMilliseconds;
        }

        double averageMs = totalMilliseconds / (double)iterations;
        Console.WriteLine($"Average: {averageMs}ms | Total: {totalMilliseconds}ms");
    }
}

What the IronPDF architecture gives you for batch work:

  • CSS Grid, Flexbox, modern selectors and WOFF2 fonts are handled by the underlying Chromium engine, so no separate parser logic has to be added.
  • The async API (RenderHtmlAsPdfAsync, SaveAsAsync) lets you compose the workload with Parallel.ForEach or Task.WhenAll without blocking calling threads.
  • Renderer instances can be reused across requests, and PdfDocument implements IDisposable so Chromium resources release deterministically.
  • Updates ship on a regular cadence, which means rendering improvements that land in upstream Chromium become available without you swapping libraries.

What about memory usage in batch processing?

Memory behavior is the dimension where unmaintained libraries quietly become risky. Without patches and profiling work, any latent retention bugs compound in long-running services. The sketch below shows the shape of a 500-document batch on each side — measure the actual numbers against your own workload before drawing conclusions.

PDF Duo — Memory Profile (Single-Threaded Sequential)

// NuGet: Install-Package PDFDuo.NET
using PDFDuo;
using System;
using System.Diagnostics;

class MemoryTest
{
    static void Main(string[] args)
    {
        var process = Process.GetCurrentProcess();
        long startMemory = process.WorkingSet64;

        Console.WriteLine($"Starting memory: {startMemory / 1024 / 1024}MB");

        // Generate 500 PDFs sequentially
        for (int i = 0; i < 500; i++)
        {
            string html = GenerateInvoiceHtml(i);

            var converter = new HtmlToPdfConverter();
            converter.ConvertHtmlString(html, $"invoice_{i}.pdf");

            if (i % 100 == 0)
            {
                long currentMemory = process.WorkingSet64;
                long used = (currentMemory - startMemory) / 1024 / 1024;
                Console.WriteLine($"After {i} PDFs: {used}MB used");

                // Force GC to see if memory releases
                GC.Collect();
                GC.WaitForPendingFinalizers();
                GC.Collect();

                long afterGC = process.WorkingSet64;
                long afterGCUsed = (afterGC - startMemory) / 1024 / 1024;
                Console.WriteLine($"After GC: {afterGCUsed}MB used");
            }
        }

        long finalMemory = process.WorkingSet64;
        long totalUsed = (finalMemory - startMemory) / 1024 / 1024;
        Console.WriteLine($"Final memory used: {totalUsed}MB");
    }

    static string GenerateInvoiceHtml(int id)
    {
        return $"<html><body><h1>Invoice {id}</h1></body></html>";
    }
}

What the architecture implies for memory pressure:

  • A new HtmlToPdfConverter is constructed per call in the documented pattern, with no documented pooling primitive.
  • The API is synchronous, so concurrent batch processing has to be added at the caller level rather than handed to the library.
  • Long-running services should profile retention against their own workload — an unmaintained engine has not had memory regressions filed and patched.

IronPDF — Memory Profile (Parallel Processing)

Detailed rendering options: ChromePdfRenderOptions class.

using IronPdf;
using System;
using System.Diagnostics;
using System.Linq;
using System.Threading.Tasks;

class MemoryTest
{
    static async Task Main(string[] args)
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var process = Process.GetCurrentProcess();
        long startMemory = process.WorkingSet64 / 1024 / 1024;

        Console.WriteLine($"Starting memory: {startMemory}MB");

        var renderer = new ChromePdfRenderer();
        var tasks = Enumerable.Range(0, 500).Select(i => GeneratePdfAsync(renderer, i));

        await Task.WhenAll(tasks);

        GC.Collect();
        GC.WaitForPendingFinalizers();

        long finalMemory = process.WorkingSet64 / 1024 / 1024;
        Console.WriteLine($"Final memory: {finalMemory}MB ({finalMemory - startMemory}MB used)");
    }

    static async Task GeneratePdfAsync(ChromePdfRenderer renderer, int id)
    {
        string html = $"<html><body><h1>Invoice {id}</h1></body></html>";

        using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
        await pdf.SaveAsAsync($"invoice_{id}.pdf");
        // Explicit disposal releases Chromium resources
    }
}

What IronPDF's architecture gives you here:

  • Chromium engine instances are reused across renders rather than reconstructed per document.
  • PdfDocument implements IDisposable, so a using statement around each render releases native resources deterministically.
  • The async API composes with Task.WhenAll or Parallel.ForEachAsync for thread-safe batch work.
  • Long-running services typically settle into a steady-state working set instead of growing linearly with throughput — measure to confirm on your hardware.

Feature support comparison:

  • CSS Grid. Not part of PDF Duo's documented feature set. IronPDF inherits Chromium's Grid implementation, so modern multi-column layouts render the same as in Chrome.
  • Flexbox. Not part of PDF Duo's documented feature set. IronPDF supports Flexbox via Chromium, which keeps responsive layouts intact in PDF output.
  • CSS Variables. Not documented for PDF Duo. IronPDF resolves var() at render time through Chromium, so theme-driven templates carry over.
  • WOFF2 fonts. Not documented for PDF Duo. IronPDF supports the same web font formats Chromium does, including WOFF2 and variable fonts.
  • Media queries and modern selectors. PDF Duo's parser predates these constructs; IronPDF treats them as Chromium does, including @media print and :not().
  • Transforms and transitions. Static transforms render in IronPDF via Chromium. PDF Duo does not document these.
  • SVG and modern image formats. IronPDF handles SVG and WebP through the browser engine. PDF Duo's image support is documented for raster formats from its era (JPEG, PNG, GIF, BMP).
  • JavaScript. IronPDF can execute JavaScript via Chromium when EnableJavaScript is set. PDF Duo does not advertise script execution.

What are the maintenance and security concerns?

PDF Duo's public release activity appears to have stopped over a decade ago. That shapes the risk surface in predictable ways:

  1. Unicode handling for modern character sets (emoji, extended scripts) may not match current expectations — verify against your input data.
  2. The HTML parser has no documented vulnerability disclosure process or recent security audit.
  3. Processing untrusted HTML on an unmaintained engine is a hardening concern; treat the parser as untrusted input territory.
  4. Long-running services should profile retention themselves; an unmaintained engine has not had retention regressions filed and patched.
  5. .NET Core / .NET 5+ compatibility is not advertised — assume a Framework-only target unless you confirm otherwise.
  6. TLS handling depends on the OS-level stack of the era it was written for; URL fetches may need verification on modern Windows.

For teams handling sensitive documents — financial reports, legal contracts, healthcare records — an unmaintained dependency with no current vendor channel is typically flagged in a compliance audit. That is the practical concern, more than any single CVE.

How do the features compare?

The scope difference is the headline. PDF Duo focuses on HTML-to-PDF conversion plus basic merging. IronPDF covers the full document lifecycle. Grouped by category:

  • Maintenance status. PDF Duo has no recent public release activity and no current vendor channel. IronPDF ships on a regular cadence with security and feature updates.
  • Documentation and support. PDF Duo's documentation is sparse and dates from its release era. IronPDF maintains current API reference, how-tos, and a paid support channel.
  • HTML-to-PDF engine. PDF Duo uses a custom HTML/CSS parser scoped to the standards of its release era. IronPDF uses Chromium, which tracks current HTML5/CSS3/ES2020+ behavior.
  • URL to PDF. Both engines accept a URL input — PDF Duo via ConvertUrl(url, path), IronPDF via RenderUrlAsPdf(url) with async support.
  • Modern fonts. PDF Duo does not document WOFF2 support. IronPDF inherits Chromium's font handling, including WOFF2 and variable fonts.
  • Headers and footers. Not documented in PDF Duo's published API. IronPDF exposes RenderingOptions.HtmlHeader and HtmlFooter with placeholders such as {page} and {total-pages}.
  • Watermarks. Not part of PDF Duo's documented surface. IronPDF provides PdfDocument.ApplyWatermark(html, ...).
  • PDF manipulation. PDF Duo's published surface covers HTML conversion plus PdfMerger. IronPDF additionally handles split, text extraction, form filling, digital signatures, and encryption.
  • .NET runtime. PDF Duo targets older .NET Framework versions; .NET Core / .NET 5+ compatibility is not advertised. IronPDF supports .NET Framework alongside .NET 6 through .NET 10+.
  • Async and threading. PDF Duo's published API is synchronous. IronPDF exposes Async variants and is documented for thread-safe parallel use.
  • Cross-platform and containers. PDF Duo's published target is Windows-era .NET Framework. IronPDF runs on Windows, Linux, and macOS, with documented Docker guidance.

How do the APIs map?

If you are migrating from PDF Duo to IronPDF, most of the work is mechanical class and method renames rather than architectural redesign. The mapping:

  • HtmlToPdfConverter — becomes ChromePdfRenderer.
  • converter.ConvertHtmlString(html, outputPath) — becomes renderer.RenderHtmlAsPdf(html).SaveAs(outputPath). PDF Duo writes directly to a path; IronPDF returns a PdfDocument you can save, post-process, or stream.
  • converter.ConvertUrl(url, outputPath) — becomes renderer.RenderUrlAsPdf(url).SaveAs(outputPath).
  • PdfMerger + AddFile + Merge — becomes PdfDocument.FromFile(path) per source plus a single PdfDocument.Merge(pdfs) call, then SaveAs on the result.
  • PageSize, PageOrientation — become RenderingOptions.PaperSize and RenderingOptions.PaperOrientation.
  • Margins object (e.g. new Margins(t, r, b, l)) — flatten to RenderingOptions.MarginTop / MarginRight / MarginBottom / MarginLeft.
  • Custom font embedding — not needed in IronPDF; fonts referenced from HTML/CSS are resolved by Chromium.
  • JavaScript — opt in with RenderingOptions.EnableJavaScript = true.
  • Async and parallel work — use RenderHtmlAsPdfAsync / SaveAsAsync and compose with Task.WhenAll or Parallel.ForEachAsync.
  • Watermarks (no PDF Duo equivalent) — pdf.ApplyWatermark(html, opacityPercent, vAlign, hAlign); opacity is an integer 0–100.

What does migration cost vs staying?

This is the calculation that matters most. The cost of migration is finite and predictable. The cost of staying is ongoing and growing.

Hidden Costs of Unmaintained Libraries

Time typically spent working around limitations:

  • Rewriting HTML templates to avoid CSS3 features the engine does not document support for.
  • Manual visual testing because the browser preview diverges from the rendered PDF.
  • Troubleshooting rendering issues without modern diagnostics or a vendor channel to escalate to.

Opportunity cost:

  • Modern layout primitives (CSS Grid, Flexbox) are off the table, so templates stay on table-based layouts.
  • Custom web fonts are off the table, so typography is limited to system fonts.
  • Inbound HTML from external sources cannot be trusted to render, since modern CSS may fail silently.

Technical debt accumulation:

  • Pinned to legacy .NET Framework targets if the library cannot move forward.
  • No documented stream of security patches, which keeps the dependency on the audit radar.
  • No feature improvements arriving, so workarounds accumulate over time rather than getting absorbed by the library.

FAQ

Is PDF Duo .Net still maintained or supported?

Public release activity for PDF Duo .Net appears to have stopped over a decade ago. There is no visible repository, no current vendor channel, and no documented stream of security patches. Treat it as an unmaintained dependency for planning purposes.

Can PDF Duo .Net run on .NET 6 or later?

PDF Duo's published target is older .NET Framework versions. .NET Core and .NET 5+ compatibility is not advertised, so running it on modern .NET runtimes typically requires compatibility shims and verification on your own workload.

How long does it take to migrate from PDF Duo to IronPDF?

For a typical system with 5-10 PDF templates, the estimated migration effort is 6-10 hours total. This includes installing IronPDF (15 minutes), updating API calls (2-3 hours), testing (1-2 hours), and removing old workarounds (2-4 hours). The API mapping is straightforward — HtmlToPdfConverter becomes ChromePdfRenderer, and ConvertHtmlString becomes RenderHtmlAsPdf.

Does IronPDF support CSS Grid and Flexbox?

Yes. IronPDF uses Chrome's Blink rendering engine, which means it supports modern CSS features including CSS Grid, Flexbox, CSS Variables, media queries, modern selectors, transforms, and transitions. The rendering output is intended to match what you see in the Chrome browser, since the underlying engine is shared.

What are the security risks of using PDF Duo .Net in production?

The practical concerns are: unpatched HTML parser issues with no current disclosure process, potential XSS exposure when processing untrusted HTML, possible Unicode and encoding gaps, retention behavior in long-running services, and TLS/HTTPS compatibility on modern Windows. Without a current vendor channel, the full scope is hard to characterize, and a compliance audit will typically flag an unmaintained dependency on those grounds alone.

When should you finally migrate?

Staying on PDF Duo makes sense if: (1) you're generating fewer than 50 PDFs per month from frozen templates, (2) those templates use only table-based layouts with system fonts, (3) you're running .NET Framework 3.5 and have no plans to upgrade, and (4) security isn't a concern because the HTML is fully controlled and never comes from external sources.

Migration becomes necessary when: rendering quality degrades because templates rely on modern HTML/CSS, you need features PDF Duo does not provide (watermarks, signatures, encryption, text extraction), you are upgrading to .NET Core / .NET 6+, or security audits flag unmaintained dependencies. The architectural difference between a custom parser and a Chromium engine usually shows up in throughput and memory stability under load, but the real driver is typically blocked feature development.
