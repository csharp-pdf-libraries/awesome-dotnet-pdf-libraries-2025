Kaizen.io vs IronPDF in C#: Which Should Your Team Choose?

Two different products, two different shapes. Kaizen.io HTML-to-PDF is a self-hosted Docker container that exposes a single REST endpoint; integration from C# is hand-rolled HttpClient against POST /html-to-pdf. IronPDF is a NuGet package that runs in-process and returns a PdfDocument. Same broad goal — HTML in, PDF out — but the integration model, the API surface, and what you can do after conversion all diverge sharply.

Teams I have worked with hit a familiar pattern: an HTML-to-PDF tool gets chosen because the first requirement is HTML-to-PDF, and the second, third, and fourth requirements arrive later. I want to walk through the specific questions .NET teams ask when evaluating Kaizen.io and IronPDF, with code and facts for each scenario.

What is Kaizen.io and how does it generate PDFs?

Kaizen.io HTML-to-PDF ships as a Docker image (kaizenio.azurecr.io/html-to-pdf) that runs as a container exposing port 8080 with a single endpoint, POST /html-to-pdf. There is no official .NET SDK or NuGet package — every C# call is a hand-rolled HttpClient request that posts a JSON body containing an html field. Licensing is configured by setting the KAIZEN_PDF_LICENSE environment variable on the container; without it, output PDFs carry a watermark.

The v1.x JSON body accepts only the html field. URL input, custom stylesheets, headers and footers, page-size, margins, orientation, and JavaScript wait conditions are listed as roadmap items rather than shipping features on the product page. Anything beyond rendering a string of HTML therefore has to be expressed inside the HTML itself, typically through @page CSS and absolutely-positioned divs.

Typical usage from C# looks like this:

using System.IO;
using System.Net.Http;
using System.Text;
using System.Text.Json;

// Container must be running on localhost:8080
//   docker run -d -p 8080:8080 -e KAIZEN_PDF_LICENSE=... \
//     kaizenio.azurecr.io/html-to-pdf:latest

using var http = new HttpClient();
var payload = JsonSerializer.Serialize(new { html = "<html><body><h1>Hello World</h1></body></html>" });
var content = new StringContent(payload, Encoding.UTF8, "application/json");
var response = await http.PostAsync("http://localhost:8080/html-to-pdf", content);
response.EnsureSuccessStatusCode();
var pdfBytes = await response.Content.ReadAsByteArrayAsync();
File.WriteAllBytes("output.pdf", pdfBytes);

How does IronPDF differ from Kaizen.io architecturally?

IronPDF is a native .NET library installed via the IronPdf NuGet package. It embeds a Chromium rendering engine in-process and returns a rich PdfDocument object instead of raw HTTP bytes. That object is the entry point to merging, splitting, watermarking, encrypting, signing, form filling, and text extraction — all in the same library. Kaizen.io is scoped to the conversion step, fronted by a container; IronPDF covers the conversion step plus the operations that typically follow it, in-process.

For .NET applications, the architectural difference shows up in two places. First, no container: there is no sidecar to operate, no port to manage, no HTTP round-trip on every PDF. Second, no second library: when Kaizen returns raw bytes, any post-conversion work needs another PDF library that accepts byte arrays. With IronPDF, the result of RenderHtmlAsPdf is already a manipulable document.

Installation is a single NuGet command:

dotnet add package IronPdf

using IronPdf;

That difference — out-of-process REST endpoint versus in-process .NET library with a full PDF document model — drives every other comparison point below.

How does HTML-to-PDF conversion compare between Kaizen.io and IronPDF?

Both tools render HTML and produce a PDF. The difference is in the integration shape and what you can do with the result.

Kaizen.io — HTML String to PDF

using System.IO;
using System.Net.Http;
using System.Text;
using System.Text.Json;
using System.Threading.Tasks;

class Program
{
    static async Task Main()
    {
        using var http = new HttpClient();
        var payload = JsonSerializer.Serialize(new { html = "<html><body><h1>Hello World</h1></body></html>" });
        var content = new StringContent(payload, Encoding.UTF8, "application/json");
        var response = await http.PostAsync("http://localhost:8080/html-to-pdf", content);
        response.EnsureSuccessStatusCode();
        var pdfBytes = await response.Content.ReadAsByteArrayAsync();
        File.WriteAllBytes("output.pdf", pdfBytes);
    }
}

The call returns raw bytes via HTTP. To save, you write the bytes to disk yourself. To do anything else with the PDF — merge, encrypt, watermark — you need a separate library that accepts byte arrays.

IronPDF — HTML String to PDF

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var html = "<html><body><h1>Hello World</h1></body></html>";
var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");

IronPDF returns a PdfDocument. You can SaveAs, but you can also call MergeFast, ApplyStamp, Encrypt, or feed it into form-filling APIs. The HTML to PDF documentation shows advanced options for margins, page size, and rendering behavior.

How do URL rendering and headers/footers compare?

This is where the v1.x API gap shows up. URL-to-PDF and header/footer fields are roadmap items in Kaizen.io, not shipping features, so the typical workaround is to fetch the page from C#, then bake fake header and footer divs into the HTML using fixed-position CSS before posting. Page-number placeholders like {page} and {total} are not supported by the API at all. IronPDF exposes URL rendering as a first-class method, real header and footer zones, and resolves {page} / {total-pages} placeholders at render time.

Kaizen.io — URL to PDF with simulated headers and footers

using System.IO;
using System.Net.Http;
using System.Text;
using System.Text.Json;
using System.Threading.Tasks;

class Program
{
    static async Task Main()
    {
        using var http = new HttpClient();

        // No URL endpoint in v1.x: fetch the page client-side.
        var pageHtml = await http.GetStringAsync("https://example.com");

        // No header/footer or margin fields: inject via CSS.
        var wrapped = $@"<!doctype html>
<html><head><style>
  @page {{ margin: 20mm; }}
  .k-header {{ position: fixed; top: -15mm; left: 0; right: 0; text-align: center; }}
  .k-footer {{ position: fixed; bottom: -15mm; left: 0; right: 0; text-align: center; }}
</style></head><body>
  <div class='k-header'>Company Header</div>
  <div class='k-footer'>Footer (page numbers unsupported in v1.x)</div>
  {pageHtml}
</body></html>";

        var payload = JsonSerializer.Serialize(new { html = wrapped });
        var content = new StringContent(payload, Encoding.UTF8, "application/json");
        var response = await http.PostAsync("http://localhost:8080/html-to-pdf", content);
        response.EnsureSuccessStatusCode();
        var pdfBytes = await response.Content.ReadAsByteArrayAsync();
        File.WriteAllBytes("webpage.pdf", pdfBytes);
    }
}

Three things are worth noting. First, the URL fetch is your code — Kaizen never sees the URL. Second, header and footer styling is CSS-by-convention rather than a typed option. Third, page numbers cannot be produced server-side because the API has no placeholders.

IronPDF — URL to PDF with headers and footers

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.TextHeader.CenterText = "Company Header";
renderer.RenderingOptions.TextFooter.CenterText = "Page {page} of {total-pages}";
renderer.RenderingOptions.MarginTop = 20;
renderer.RenderingOptions.MarginBottom = 20;
var pdf = renderer.RenderUrlAsPdf("https://example.com");
pdf.SaveAs("webpage.pdf");

IronPDF exposes both a text-based header/footer (shown above) and an HTML-based one (HtmlHeader / HtmlFooter with an HtmlFragment property). The {page} and {total-pages} placeholders resolve at render time, so "Page 1 of 12" no longer needs hand-substitution before rendering. IronPDF's URL rendering capabilities include configurable timeouts, JavaScript wait conditions, and custom HTTP headers.

How does merging, signing, and advanced PDF manipulation compare?

This is where the gap between a single-endpoint conversion service and a full .NET PDF library becomes stark. Kaizen.io's documented v1.x API is a single endpoint that accepts an html field and returns PDF bytes. Merging, splitting, watermarking, form filling, text extraction, and digital signatures are not part of that surface — confirm the current product page for any additions.

IronPDF provides all of these as native operations on the PdfDocument object. When Kaizen returns raw bytes, you still need another library to manipulate those bytes. With IronPDF, the conversion result is already a rich object you can merge, split, sign, encrypt, or annotate in the same code block.

For HTML files with external assets, the difference also matters. Kaizen accepts a single string of HTML in the JSON body; resolving relative asset paths means reading the file yourself and either inlining the assets or rewriting the URLs to absolute references before posting. IronPDF's RenderHtmlFileAsPdf automatically resolves relative asset paths using the HTML file's directory as the base. See the HTML file rendering documentation for details on asset resolution.

Kaizen.io — HTML file rendering

using System.IO;
using System.Net.Http;
using System.Text;
using System.Text.Json;
using System.Threading.Tasks;

class Program
{
    static async Task Main()
    {
        var body = File.ReadAllText("input.html");

        // No file endpoint and no page-size field: embed @page CSS.
        var html = "<style>@page { size: A4 portrait; }</style>" + body;

        using var http = new HttpClient();
        var payload = JsonSerializer.Serialize(new { html });
        var content = new StringContent(payload, Encoding.UTF8, "application/json");
        var response = await http.PostAsync("http://localhost:8080/html-to-pdf", content);
        response.EnsureSuccessStatusCode();
        File.WriteAllBytes("document.pdf", await response.Content.ReadAsByteArrayAsync());
    }
}

IronPDF — HTML file to PDF

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = PdfPaperSize.A4;
renderer.RenderingOptions.PaperOrientation = PdfPaperOrientation.Portrait;
var pdf = renderer.RenderHtmlFileAsPdf("input.html");
pdf.SaveAs("document.pdf");

How do the features compare side by side?

  • Distribution and install. Kaizen.io ships as the Docker image kaizenio.azurecr.io/html-to-pdf; you run, monitor, and update the container yourself. IronPDF is the IronPdf NuGet package and runs in-process inside your .NET application.
  • C# integration model. Kaizen.io has no .NET SDK; integration is hand-rolled HttpClient POST against the container's REST endpoint. IronPDF exposes a strongly-typed ChromePdfRenderer API with compile-time-checked properties.
  • API surface. Kaizen.io v1.x accepts a single html field in the JSON body. IronPDF returns a PdfDocument with methods for save, merge, split, stamp, encrypt, sign, fill, and extract.
  • URL input. Roadmap in Kaizen v1.x; you fetch the page yourself and post the resulting HTML. IronPDF exposes RenderUrlAsPdf(url) directly.
  • Headers and footers. Not part of the v1.x API; the typical workaround is fixed-position divs in the HTML you post. IronPDF supports both text-based (TextHeader / TextFooter) and HTML-based (HtmlHeader / HtmlFooter) header and footer models, with {page} and {total-pages} placeholders resolved at render time.
  • Page numbers. No {page} / {total} placeholder support in the Kaizen API. IronPDF resolves {page} and {total-pages} server-side.
  • Page size, orientation, margins. Embed in @page CSS for Kaizen. Configure via RenderingOptions.PaperSize / PaperOrientation / MarginTop for IronPDF.
  • Merge, split, watermark, sign, form fill, extract. Not part of Kaizen.io's documented API; native on IronPDF's PdfDocument.
  • Password protection and permissions. Not part of Kaizen.io's documented API; IronPDF exposes SecuritySettings.UserPassword, SecuritySettings.OwnerPassword, and granular permission flags.
  • Async support. Kaizen calls are already async because they are HTTP. IronPDF exposes first-class RenderHtmlAsPdfAsync and RenderUrlAsPdfAsync.
  • Data privacy. Kaizen runs in your own container, so HTML and PDF bytes stay inside your network — but they still leave the calling process and travel over a (usually localhost) socket. IronPDF runs in the calling process; data never leaves it.
  • Licensing. Kaizen uses the KAIZEN_PDF_LICENSE env var on the container, with a watermarked free tier. IronPDF uses IronPdf.License.LicenseKey set once at startup.

FAQ and Troubleshooting Tips

Does Kaizen.io require any external service or container at runtime?

Yes. Kaizen.io HTML-to-PDF ships only as the Docker image kaizenio.azurecr.io/html-to-pdf. You must run that container somewhere reachable from your application — local Docker, sidecar, or a separate host — and post JSON to its POST /html-to-pdf endpoint on port 8080. There is no in-process .NET option.

Can I use Kaizen.io for PDF merging, splitting, or digital signatures?

Kaizen.io's documented v1.x API is a single endpoint that accepts an html field. Additional PDF operations like merging, splitting, watermarking, form filling, text extraction, and digital signatures are not part of that surface — verify against the latest vendor docs. IronPDF provides all of these as native operations on the PdfDocument object returned from any conversion.

How do I handle concurrent PDF generation with Kaizen.io?

Concurrency for Kaizen is bounded by what your container instance can serve before requests queue or time out, plus whatever throughput your HttpClient pool sustains. Scaling means running more containers and load-balancing across them. IronPDF runs in-process on standard .NET threading; concurrency is bounded by CPU and RAM. See the HTML to PDF tutorial for async patterns and batch processing.

What is the placeholder syntax difference?

Kaizen v1.x has no server-side placeholder support, so any "{page}" or "{total}" tokens in the HTML you post are rendered literally — page numbers were typically substituted client-side before posting. IronPDF resolves {page} and {total-pages} at render time. The most common surprise after migration is a literal "Page 1 of {total}" footer where the placeholder needs to be updated to {total-pages}.

Can I migrate from Kaizen.io to IronPDF incrementally?

Yes. The shape of the migration is replacing the HTTP plumbing with a ChromePdfRenderer instance. HttpClient.PostAsync(".../html-to-pdf", { html }) maps to renderer.RenderHtmlAsPdf(html). Inline @page CSS for size, orientation, and margins maps to renderer.RenderingOptions.PaperSize / PaperOrientation / MarginTop. Fake header and footer divs map to RenderingOptions.HtmlHeader / HtmlFooter. Manual placeholder substitution maps to {page} / {total-pages}. The KAIZEN_PDF_LICENSE env var maps to IronPdf.License.LicenseKey. The HTTP byte[] body maps to pdf.BinaryData or pdf.SaveAs(path). For migration guidance, see the HTML string to PDF documentation.

Which should your team choose — Kaizen.io or IronPDF?

Kaizen.io makes sense when the workload is genuinely conversion-only and the operations team is comfortable running a container. The narrow API surface and small footprint can be a virtue when the team wants a sidecar that does one thing, and the watermarked free tier is convenient for evaluation.

For .NET teams whose PDF requirements have a habit of expanding past conversion — adding watermarks for drafts, merging monthly reports, password-protecting financial statements, filling AcroForm fields, extracting text for indexing — a single-endpoint converter means a second PDF library shows up in the project sooner or later. Two PDF dependencies means two licenses, two byte-shuffling boundaries, and two sets of edge cases. Container-based deployment also adds a moving part: pulls, restarts, port management, license env var, and monitoring all become operational surface area.

Migration becomes mandatory when you need PDF manipulation beyond HTML-to-PDF without introducing a second library, when password protection or signing requirements arrive, when you want page-number placeholders without hand-substituting strings, or when you want a single PdfDocument object as the unit of work across your PDF code.

IronPDF provides native .NET PDF operations through an embedded Chromium engine. The HTML to PDF conversion works in-process, with no container or HTTP hop, and the returned PdfDocument covers the operations that typically follow conversion. For teams that view PDF generation as a library concern that spans more than the rendering step, IronPDF eliminates both the second-library handoff and the container dependency.
