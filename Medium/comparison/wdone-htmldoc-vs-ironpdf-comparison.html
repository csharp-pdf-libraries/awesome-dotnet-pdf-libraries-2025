HTMLDOC vs IronPDF: The Decision Guide for .NET Teams

A practical comparison of a legacy HTML 3.2 converter and a modern Chromium-based .NET PDF library.

You open the technical specification: "Generate PDF reports from HTML templates." You reach for HTMLDOC because it's free and established. Then you see the requirements list: modern CSS layouts, JavaScript-driven charts, web fonts, responsive design. You check HTMLDOC's documented feature scope: HTML 3.2 with limited HTML 4.01, no CSS engine, no forms.

The gap between "can generate PDFs from HTML" and "can handle 2026 web content" determines whether a tool is viable. HTMLDOC is intentionally scoped as a legacy converter for simple, static HTML documentation. For .NET applications targeting modern web standards, the architectural mismatch usually surfaces during prototyping rather than later in production.

I put together this guide to walk through the real differences so you can make an informed choice before committing to either tool.

What Is HTMLDOC?

HTMLDOC is a standalone command-line tool and GUI application for converting HTML to PDF, PostScript, and EPUB. It supports HTML 3.2 with limited HTML 4.01 features. It has no CSS support, no JavaScript execution, and no modern web rendering. It is written in C and is open-source under GPL v2.

Binaries are available for Windows 10+, macOS 11+, and Linux (Snap). It is not a library -- it is a separate executable that must be invoked via Process.Start or shell execution from your .NET application. It is intentionally scoped as a legacy HTML converter, designed for documentation and reports built with HTML 3.2-era markup.

What Is IronPDF?

IronPDF is a .NET library that embeds a Chromium rendering engine. It processes HTML the same way Chrome does, supporting CSS3, JavaScript, web fonts, and modern DOM APIs. No external executable needed -- just a NuGet package that integrates directly into your C# application.

The library targets .NET Framework 4.6.2+, .NET Core, .NET Standard 2.0, and current .NET (6, 8, 9, 10). It works on Windows, Linux, macOS, in Docker, serverless functions, and traditional ASP.NET.

What Are the Key Limitations of HTMLDOC?

HTMLDOC is still maintained by its original author and continues to receive periodic releases. It is stable and free. However, it is intentionally scoped as a legacy HTML converter -- the documentation explicitly states it does not support CSS, forms, full Unicode, or modern web features.

Here is what is missing:

  • No CSS support: Cannot process <style> tags or external stylesheets; styling via deprecated HTML attributes only
  • No JavaScript execution: Static HTML only; cannot render charts, dynamic content, or interactive elements
  • HTML 3.2 + limited 4.01: Missing modern HTML5 elements (<header>, <nav>, <article>, <canvas>, <video>)
  • No web fonts: Limited to system fonts; no WOFF/WOFF2 support
  • No forms support: Cannot render PDF forms or input elements
  • Limited Unicode: Lacks support for CJK characters, Arabic, full emoji; limited to Western character sets
  • No flexbox or Grid: Cannot process modern CSS layout systems

On the technical side, HTMLDOC requires file-based I/O (no in-memory streaming), spawns external processes via Process.Start (adding latency and complexity), and relies on exit codes for error handling. It is not a .NET library -- interop happens through process execution and the file system. You must deploy platform-specific binaries (htmldoc.exe on Windows, htmldoc on Linux/macOS) alongside your application and handle process lifecycle management yourself.

Support is community-based via GitHub issues. There is no commercial support or SLA. The project is maintained by a single developer.

How Does the Code Compare for HTML String to PDF?

This is where the architectural difference becomes immediately visible. HTMLDOC requires writing HTML to a temp file, spawning a process, waiting for it, checking exit codes, and cleaning up. IronPDF does it in three lines.

HTMLDOC -- HTML String to PDF (Process Invocation)

using System;
using System.Diagnostics;
using System.IO;

namespace HtmldocExample
{
    class Program
    {
        static void Main(string[] args)
        {
            // HTML content authored to HTMLDOC's supported subset
            // (HTML 3.2 / limited 4.01, no CSS, no JavaScript)
            string htmlContent = @"
                <html>
                <head>
                    <title>Invoice</title>
                </head>
                <body>
                    <h1>Invoice #12345</h1>
                    <table border='1' cellpadding='5'>
                        <tr>
                            <td><b>Item</b></td>
                            <td><b>Quantity</b></td>
                            <td><b>Price</b></td>
                        </tr>
                        <tr>
                            <td>Widget A</td>
                            <td>5</td>
                            <td>$25.00</td>
                        </tr>
                    </table>
                    <p><b>Total: $125.00</b></p>
                </body>
                </html>";

            // HTMLDOC reads input from a file path; write HTML to a temp file
            string tempHtmlPath = Path.Combine(Path.GetTempPath(), "temp_invoice.html");
            File.WriteAllText(tempHtmlPath, htmlContent);

            // Output PDF path
            string outputPdfPath = "output.pdf";

            // Prepare HTMLDOC command-line arguments
            // Format: htmldoc [options] input.html -f output.pdf
            var startInfo = new ProcessStartInfo
            {
                FileName = "htmldoc", // Must be in PATH or use full path
                Arguments = $"--webpage --no-title --no-toc -f \"{outputPdfPath}\" \"{tempHtmlPath}\"",
                RedirectStandardOutput = true,
                RedirectStandardError = true,
                UseShellExecute = false,
                CreateNoWindow = true
            };

            try
            {
                // Execute HTMLDOC process
                using (var process = Process.Start(startInfo))
                {
                    // Wait for the htmldoc process to exit
                    process.WaitForExit();

                    // Check exit code
                    if (process.ExitCode != 0)
                    {
                        string error = process.StandardError.ReadToEnd();
                        Console.WriteLine($"HTMLDOC failed with exit code {process.ExitCode}");
                        Console.WriteLine($"Error: {error}");
                        return;
                    }
                }

                Console.WriteLine($"PDF generated: {outputPdfPath}");
            }
            catch (Exception ex)
            {
                Console.WriteLine($"Error executing HTMLDOC: {ex.Message}");
                // Typical causes to check: htmldoc binary not on PATH,
                // file path quoting/encoding, or temp directory permissions.
            }
            finally
            {
                // Cleanup temp file
                try
                {
                    if (File.Exists(tempHtmlPath))
                        File.Delete(tempHtmlPath);
                }
                catch { /* Ignore cleanup errors */ }
            }
        }
    }
}

IronPDF -- HTML String to PDF (Native Library)

{

                        h1 { color: #2c3e50; }
                        table { border-collapse: collapse; width: 100%; }
                        th, td { border: 1px solid #ddd; padding: 8px; text-align: left; }
                        th { background-color: #3498db; color: white; }
                    </style>
                </head>
                <body>
                    <h1>Invoice #12345</h1>
                    <table>
                        <thead>
                            <tr>
                                <th>Item</th>
                                <th>Quantity</th>
                                <th>Price</th>
                            </tr>
                        </thead>
                        <tbody>
                            <tr>

                            </tr>

How Does HTML File to PDF Conversion Compare?

When you are converting an HTML file that references external assets like images, CSS files, and JavaScript, the differences get even more pronounced. HTMLDOC does not process external CSS, requires absolute file paths or base64 data URIs for images, and does not support modern image formats like SVG or WebP.

HTMLDOC -- HTML File to PDF

using System;
using System.Diagnostics;
using System.IO;

namespace HtmldocFileExample
{
    class Program
    {
        static void Main(string[] args)
        {
            // Assume we have an HTML file: invoice.html
            // HTMLDOC reference: external CSS via <link> is not processed,
            // and images are resolved only when referenced via absolute
            // file:// URLs or embedded as base64 data URIs. Modern image
            // formats such as WebP and SVG are not supported.

            string inputHtmlPath = "invoice.html";
            string outputPdfPath = "invoice.pdf";

            var startInfo = new ProcessStartInfo
            {
                FileName = "htmldoc",
                Arguments = $"--webpage --no-title --no-toc -f \"{outputPdfPath}\" \"{inputHtmlPath}\"",
                RedirectStandardOutput = true,
                RedirectStandardError = true,
                UseShellExecute = false,
                CreateNoWindow = true
            };

            try
            {
                using (var process = Process.Start(startInfo))
                {
                    process.WaitForExit();

                    if (process.ExitCode != 0)
                    {
                        string error = process.StandardError.ReadToEnd();
                        Console.WriteLine($"Error: {error}");
                        return;
                    }
                }

                Console.WriteLine($"PDF generated: {outputPdfPath}");
            }
            catch (Exception ex)
            {
                Console.WriteLine($"Error: {ex.Message}");
            }
        }
    }
}

IronPDF -- HTML File to PDF

using IronPdf;

namespace IronPdfFileExample
{
    class Program
    {
        static void Main(string[] args)
        {
            IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

How Do Headers and Footers Compare?

HTMLDOC headers and footers are configured via command-line options and limited to simple text strings with tokens. You cannot use custom HTML or CSS for styling. IronPDF lets you write full HTML/CSS fragments for headers and footers, including images, custom fonts, and dynamic placeholders.

HTMLDOC -- Headers and Footers

using System;
using System.Diagnostics;
using System.IO;

namespace HtmldocHeaderFooter
{
    class Program
    {
        static void Main(string[] args)
        {
            // HTMLDOC headers/footers are configured via command-line options.
            // Each is a three-part string (left/center/right) made of literal
            // text and token characters; HTML/CSS markup is not interpreted.

            string htmlContent = "<html><body><h1>Report</h1><p>Content...</p></body></html>";
            string tempHtmlPath = Path.Combine(Path.GetTempPath(), "report.html");
            File.WriteAllText(tempHtmlPath, htmlContent);

            string outputPdfPath = "report.pdf";

            // HTMLDOC header/footer options:
            // --header left center right
            // --footer left center right
            // Tokens: / (page number), 1 (total pages), t (title), d (date)
            var startInfo = new ProcessStartInfo
            {
                FileName = "htmldoc",
                Arguments = $"--webpage " +
                           $"--header ... " +     // Left: empty, Center: "Company Report", Right: empty
                           $"--footer .1. " +     // Left: empty, Center: "Page X", Right: empty
                           $"-f \"{outputPdfPath}\" \"{tempHtmlPath}\"",
                RedirectStandardOutput = true,
                RedirectStandardError = true,
                UseShellExecute = false,
                CreateNoWindow = true
            };

            try
            {
                using (var process = Process.Start(startInfo))
                {
                    process.WaitForExit();

                    if (process.ExitCode != 0)
                    {
                        Console.WriteLine($"Error: {process.StandardError.ReadToEnd()}");
                        return;
                    }
                }

                Console.WriteLine($"PDF with headers/footers: {outputPdfPath}");
            }
            catch (Exception ex)
            {
                Console.WriteLine($"Error: {ex.Message}");
            }
            finally
            {
                try { File.Delete(tempHtmlPath); } catch { }
            }
        }
    }
}

IronPDF -- Headers and Footers

using IronPdf;

namespace IronPdfHeaderFooter
{
    class Program
    {
        static void Main(string[] args)
        {
            IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

            // Instantiate renderer
            var renderer = new ChromePdfRenderer();

            // Configure HTML header with full CSS support
            renderer.RenderingOptions.HtmlHeader = new HtmlHeaderFooter
            {
                MaxHeight = 20, // millimeters
                HtmlFragment = @"
                    <div style='text-align:center; font-family: Arial; font-size: 12pt;'>
                        <img src='data:image/png;base64,iVBORw0KGgoAAAANSUhEUg...' height='30'/>
                        <b>Company Report - {date}</b>
                    </div>",
                DrawDividerLine = true
            };

            // Configure HTML footer with page numbers
            renderer.RenderingOptions.HtmlFooter = new HtmlHeaderFooter
            {
                MaxHeight = 15,
                HtmlFragment = "<center style='font-size: 9pt; color: #666;'>Page {page} of {total-pages}</center>",
                DrawDividerLine = true
            };

            // Set margins
            renderer.RenderingOptions.MarginTop = 25;    // mm
            renderer.RenderingOptions.MarginBottom = 20; // mm

            // Render to PDF
            var pdf = renderer.RenderHtmlAsPdf("<h1>Report</h1><p>Content...</p>");
            pdf.SaveAs("report.pdf");
        }
    }
}

How Do the Features Compare Overall?

The two tools sit at opposite ends of the HTML-to-PDF spectrum: HTMLDOC is a legacy command-line converter scoped to HTML 3.2-era content, and IronPDF is a Chromium-backed .NET library aimed at modern web rendering. The points below summarise how that plays out across status, support, content creation, PDF operations, security, and development integration. Capabilities are drawn from each project's documentation.

  • Project status. HTMLDOC is in long-term maintenance mode and intentionally scoped to its legacy feature set; IronPDF is under active development with regular releases.
  • Licensing. HTMLDOC is GPL v2 (a commercial license is also offered by the author for proprietary use); IronPDF is a commercial library with permissive distribution terms.
  • Support model. HTMLDOC support is community-driven via the GitHub project; IronPDF offers commercial support channels and SLA options on paid tiers.
  • HTML support. HTMLDOC targets HTML 3.2 with limited HTML 4.01; IronPDF processes HTML5 the same way Chromium does.
  • CSS support. HTMLDOC has no CSS engine -- styling relies on deprecated HTML attributes; IronPDF supports CSS3 including Flexbox and Grid.
  • JavaScript. HTMLDOC does not execute JavaScript; IronPDF runs JavaScript via the embedded Chromium engine.
  • Fonts. HTMLDOC is limited to system fonts and does not load WOFF/WOFF2 web fonts; IronPDF supports web fonts and font embedding.
  • Image formats. HTMLDOC handles PNG, JPEG, and GIF; IronPDF additionally supports modern formats such as WebP and SVG.
  • Unicode. HTMLDOC has limited Unicode coverage in the default build (focused on Western character sets); IronPDF renders the full Unicode range supported by Chromium.
  • HTML forms in source. HTMLDOC does not render <form> input elements; IronPDF renders them and can produce interactive PDF AcroForms.
  • Headers and footers. HTMLDOC uses three-part text strings with token characters; IronPDF accepts full HTML/CSS fragments via HtmlHeaderFooter.
  • PDF post-processing. HTMLDOC focuses on the conversion step and does not expose merge, split, text extraction, watermarking, digital signing, or form fill/flatten APIs from .NET; IronPDF exposes these via PdfDocument.
  • Security. HTMLDOC supports basic password encryption via command-line flags; IronPDF supports AES-256 encryption with granular permission flags through SecuritySettings.
  • Integration type. HTMLDOC is invoked as an external process (typically via Process.Start); IronPDF is a native .NET library distributed as a NuGet package.
  • Async support. HTMLDOC is synchronous from the caller's perspective (Process.WaitForExit); IronPDF exposes RenderHtmlAsPdfAsync and related async methods.
  • I/O model. HTMLDOC requires file-based input and output; IronPDF supports in-memory rendering via BinaryData and stream APIs.
  • .NET targets. HTMLDOC has no managed integration -- you call the binary; IronPDF targets .NET Framework 4.6.2+, .NET Standard 2.0, and current .NET (6, 8, 9, 10).
  • Cross-platform deployment. HTMLDOC ships as platform-specific binaries that must be installed alongside the application; IronPDF ships as a single NuGet package that works on Windows, Linux, macOS, Docker, and serverless hosts.
  • Reader-useful caveats. Choosing HTMLDOC means accepting the GPL license terms (or purchasing a commercial license from the author), shipping and managing the platform-specific binary, and authoring HTML inside its supported subset.

Frequently Asked Questions

Is HTMLDOC still actively maintained?

Yes -- HTMLDOC continues to receive periodic releases from its original author, and the GitHub repository is the source of truth for current release dates. It is intentionally scoped as a legacy HTML converter; the documentation explicitly states it does not support CSS, forms, full Unicode, or modern web features. It remains a solid tool for its original purpose -- converting HTML 3.2 documentation to PDF, PostScript, or EPUB.

Can I use HTMLDOC with modern HTML and CSS?

No. HTMLDOC supports HTML 3.2 with limited HTML 4.01 features. It cannot process <style> tags, external stylesheets, CSS Grid, Flexbox, web fonts, or any modern HTML5 elements like <header>, <nav>, or <article>. Styling is limited to deprecated HTML attributes. If your content uses CSS for layout or styling, HTMLDOC cannot render it.

How does IronPDF handle JavaScript-driven content like charts?

IronPDF embeds a Chromium rendering engine that executes JavaScript before generating the PDF, just like Chrome would. This means JavaScript-driven charts, dynamic content, and interactive elements all render correctly. HTMLDOC has no JavaScript execution capability at all -- it processes static HTML only.

What is the deployment difference between the two tools?

HTMLDOC requires installing a platform-specific binary (htmldoc.exe on Windows, htmldoc on Linux/macOS) on every deployment target, ensuring it is in the system PATH, and managing process execution permissions. IronPDF installs as a single NuGet package (dotnet add package IronPdf) that works cross-platform from the same package with no external executable dependencies.

When should I choose HTMLDOC over IronPDF?

HTMLDOC is appropriate if your HTML is strictly HTML 3.2 era (tables, basic formatting), you need PostScript or EPUB output, your content contains no CSS or JavaScript, you are converting legacy documentation or simple reports, and you need a free open-source solution and accept its limitations. If your HTML uses any CSS for styling, needs JavaScript execution, requires web fonts or full Unicode, or you want native .NET library integration, HTMLDOC is not the right tool.

Conclusion

HTMLDOC serves a specific niche: converting simple, legacy HTML documentation (HTML 3.2 era) to PDF, PostScript, or EPUB. It is intentionally scoped -- the documentation explicitly states it does not support CSS, JavaScript, forms, or full Unicode. For teams with modern HTML content containing CSS layouts, JavaScript charts, web fonts, or responsive design, HTMLDOC cannot process that content.

Migration becomes mandatory when your HTML uses any CSS for styling or layout, you need JavaScript execution for dynamic content, you want native .NET library integration instead of spawning external processes, you need to avoid file-based I/O and temp file management, or you require PDF editing capabilities like merge, split, watermark, and sign.

IronPDF closes the architectural mismatch by embedding a Chromium rendering engine. It processes modern HTML5, CSS3, JavaScript, web fonts, and Unicode the same way Chrome does. It is a native .NET library with async/await support, in-memory processing, a comprehensive PDF manipulation API, and commercial engineer support on paid tiers. The library runs wherever .NET runs -- Windows, Linux, macOS, Docker, serverless -- with no external executable dependencies.
