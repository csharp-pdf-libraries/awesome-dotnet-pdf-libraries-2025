iTextSharp vs IronPDF: The .NET Developer's Decision Guide for 2025

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

If you're still running iTextSharp in production and wondering whether it's time to move on, this guide lays out the facts. I've built IronPDF from the ground up, so I'm biased — but every claim here is grounded in what the libraries actually do, with code you can run yourself.

What Is iTextSharp?

iTextSharp is a .NET port of the iText Java library that was widely used for programmatic PDF generation from roughly 2009 through 2016. It takes a low-level, coordinate-based approach to PDF construction — you specify exact X/Y coordinates for text, images, and shapes. This gives precise control but requires deep PDF spec knowledge and produces verbose code.

HTML rendering requires the separate XMLWorker package, which parses HTML/CSS but has limited support for modern standards — no Flexbox, Grid, or advanced CSS3. Teams often hit friction around thread safety (PdfWriter isn't thread-safe by default), 1-based page indexing (easy to introduce off-by-one errors), and fragmented documentation spread across Stack Overflow since official support ended.

iTextSharp 5.x reached end-of-life and transitioned to maintenance mode. The GitHub repository is marked [DEPRECATED] with the note "only security fixes will be added." The last release (5.5.13.3, February 2024) backported security patches — no new features since 5.5.11. iText 7 (the successor, now at version 9) is a different library with breaking API changes and dual AGPL/commercial licensing.

What Are the Key Limitations of iTextSharp?

Product Status

iTextSharp is officially deprecated and receives only security patches — no bug fixes, no feature development. The GitHub repository redirects teams to iText 7, which is a different product with incompatible APIs and commercial licensing requirements. Teams on iTextSharp 5.x are on a sunset path with no upgrade route that preserves existing code.

Missing Capabilities

Native HTML rendering requires the separate itextsharp.xmlworker package, which supports basic HTML 4 and CSS 2.1 but lacks modern standards. No Flexbox, CSS Grid, viewport units, or JavaScript execution — complex web layouts require manual PDF reconstruction. No built-in async methods for I/O operations. No URL-to-PDF conversion without custom HTTP client code.

Technical Issues

Thread safety requires manual synchronization — PdfWriter and PdfStamper aren't thread-safe by default, leading to corruption in concurrent environments. 1-based page indexing differs from C# conventions (0-based arrays/lists), causing off-by-one bugs. Error handling uses return codes in some APIs and exceptions in others — no consistent pattern.

Architecture Problems

Coordinate-based layout requires calculating pixel positions manually, making dynamic content brittle. Changing font size or adding a paragraph means recalculating positions downstream. HTML/CSS workflows require converting designs to procedural PDF construction code. Separate packages for PDF/A, XFA forms, and extras create dependency management complexity.

Support Status

No official support for iTextSharp 5.x — commercial support applies only to iText 7/9. Community support exists on Stack Overflow but answers reference the deprecated API. Documentation is scattered: official docs redirect to iText 7, tutorials are outdated, and migration guides assume you're moving to iText 7.

What Is IronPDF and How Does It Differ?

IronPDF takes a web-first approach: it uses a Chromium rendering engine to convert HTML/CSS/JavaScript directly to PDF, treating PDF generation as "print to PDF from a browser." This means your HTML templates render pixel-perfect without manual coordinate calculations. The library handles document structure, pagination, and resource loading automatically.

Installation is a single NuGet package (IronPdf) with no external dependencies — Chromium is bundled and managed internally. The core workflow is: create a ChromePdfRenderer, call RenderHtmlAsPdf() with your HTML string/file/URL, and save the resulting PdfDocument. All operations use 0-based page indexing (standard C# convention) and throw exceptions on errors.

How Does HTML-to-PDF Conversion Compare?

iTextSharp — HTML to PDF

// Install packages:
// Install-Package iTextSharp
// Install-Package itextsharp.xmlworker

using System;
using System.IO;
using System.Text;
using iTextSharp.text;
using iTextSharp.text.pdf;
using iTextSharp.tool.xml;

public class HtmlToPdfConverter
{
    public void ConvertHtmlToPdf()
    {
        var htmlContent = @"
            <html>
            <head>
                <style>
                    body { font-family: Arial; margin: 20px; }
                    h1 { color: #2c3e50; font-size: 24px; }
                    .invoice-table { width: 100%; border-collapse: collapse; }
                    .invoice-table th { background-color: #3498db; color: white; }
                </style>
            </head>
            <body>
                <h1>Invoice #2026-001</h1>
                <table class='invoice-table'>
                    <tr><th>Item</th><th>Amount</th></tr>
                    <tr><td>Consulting</td><td>$5,000</td></tr>
                </table>
            </body>
            </html>";

        using (var stream = new FileStream("invoice.pdf", FileMode.Create))
        {
            var document = new Document(PageSize.A4, 50, 50, 25, 25);
            var writer = PdfWriter.GetInstance(document, stream);
            document.Open();

            try
            {
                using (var htmlStream = new MemoryStream(Encoding.UTF8.GetBytes(htmlContent)))
                {
                    XMLWorkerHelper.GetInstance().ParseXHtml(writer, document, htmlStream);
                }
            }
            finally
            {
                document.Close();
                writer.Close();
            }
        }
    }
}

The XMLWorker approach requires a separate package, supports only CSS 2.1 (partial), needs manual resource management with explicit document.Open() and document.Close(), and has no JavaScript execution. That's 8+ lines of setup before actual HTML parsing begins.

IronPDF — HTML to PDF

// Install package:
// Install-Package IronPdf

using IronPdf;

public class HtmlToPdfConverter
{
    public void ConvertHtmlToPdf()
    {
        var htmlContent = @"
            <html>
            <head>
                <style>
                    body { font-family: Arial; margin: 20px; }
                    h1 { color: #2c3e50; font-size: 24px; }
                    .invoice-table { width: 100%; border-collapse: collapse; }
                    .invoice-table th { background-color: #3498db; color: white; }
                </style>
            </head>
            <body>
                <h1>Invoice #2026-001</h1>
                <table class='invoice-table'>
                    <tr><th>Item</th><th>Amount</th></tr>
                    <tr><td>Consulting</td><td>$5,000</td></tr>
                </table>
            </body>
            </html>";

        var renderer = new ChromePdfRenderer();
        var pdf = renderer.RenderHtmlAsPdf(htmlContent);
        pdf.SaveAs("invoice.pdf");
    }
}

IronPDF's Chrome rendering engine handles modern CSS automatically — full CSS3, JavaScript, Flexbox, Grid, media queries. Three lines of actual logic. For advanced rendering options like custom headers, footers, and paper sizes, see the HTML to PDF documentation.

How Does PDF Merging Compare?

iTextSharp — Merge PDFs

// Install-Package iTextSharp

using System;
using System.IO;
using iTextSharp.text;
using iTextSharp.text.pdf;

public class PdfMerger
{
    public void MergePdfs()
    {
        var outputFile = "merged.pdf";
        var inputFiles = new[] { "report1.pdf", "report2.pdf", "report3.pdf" };

        using (var stream = new FileStream(outputFile, FileMode.Create))
        {
            var document = new Document();
            var copy = new PdfCopy(document, stream);
            document.Open();

            try
            {
                foreach (var inputFile in inputFiles)
                {
                    var reader = new PdfReader(inputFile);

                    try
                    {
                        for (int i = 1; i <= reader.NumberOfPages; i++)
                        {
                            var importedPage = copy.GetImportedPage(reader, i);
                            copy.AddPage(importedPage);
                        }

                        copy.FreeReader(reader);
                    }
                    finally
                    {
                        reader.Close();
                    }
                }
            }
            finally
            {
                document.Close();
            }
        }
    }
}

No single-call merge method — you must loop through files and pages manually. Uses 1-based page indexing, requires multiple nested using/try-finally blocks, and each file is processed sequentially with manual state tracking.

IronPDF — Merge PDFs

// Install-Package IronPdf

using IronPdf;

public class PdfMerger
{
    public void MergePdfs()
    {
        var pdf1 = PdfDocument.FromFile("report1.pdf");
        var pdf2 = PdfDocument.FromFile("report2.pdf");
        var pdf3 = PdfDocument.FromFile("report3.pdf");

        var merged = PdfDocument.Merge(pdf1, pdf2, pdf3);
        merged.SaveAs("merged.pdf");
    }
}

That's the entire merge operation — load, merge, save. For splitting PDFs and advanced merge scenarios, see the merge and split documentation.

How Does Watermarking Compare?

iTextSharp — Apply Watermark

// Install-Package iTextSharp

using System;
using System.IO;
using iTextSharp.text;
using iTextSharp.text.pdf;

public class PdfWatermarker
{
    public void AddWatermark()
    {
        var inputFile = "original.pdf";
        var outputFile = "watermarked.pdf";

        using (var reader = new PdfReader(inputFile))
        using (var stream = new FileStream(outputFile, FileMode.Create))
        {
            var stamper = new PdfStamper(reader, stream);

            try
            {
                var baseFont = BaseFont.CreateFont(
                    BaseFont.HELVETICA_BOLD,
                    BaseFont.CP1252,
                    BaseFont.NOT_EMBEDDED
                );

                for (int i = 1; i <= reader.NumberOfPages; i++)
                {
                    var contentByte = stamper.GetOverContent(i);

                    var pageSize = reader.GetPageSizeWithRotation(i);
                    var centerX = pageSize.Width / 2;
                    var centerY = pageSize.Height / 2;

                    var gstate = new PdfGState { FillOpacity = 0.3f };
                    contentByte.SetGState(gstate);

                    contentByte.BeginText();
                    contentByte.SetColorFill(BaseColor.RED);
                    contentByte.SetFontAndSize(baseFont, 48);
                    contentByte.ShowTextAligned(
                        Element.ALIGN_CENTER,
                        "CONFIDENTIAL",
                        centerX, centerY, 45
                    );
                    contentByte.EndText();
                }
            }
            finally
            {
                stamper.Close();
                reader.Close();
            }
        }
    }
}

Manual positioning required for every page. You must calculate center coordinates for each page size, create font objects, manage transparency through PdfGState, and iterate page-by-page. That's roughly 30 lines of watermark logic.

IronPDF — Apply Watermark

// Install-Package IronPdf

using IronPdf;
using IronPdf.Editing;

public class PdfWatermarker
{
    public void AddWatermark()
    {
        var pdf = PdfDocument.FromFile("original.pdf");

        var watermarkHtml = @"
            <div style='
                color: red;
                font-size: 48px;
                font-weight: bold;
                opacity: 0.3;
                text-align: center;'>
                CONFIDENTIAL
            </div>";

        pdf.ApplyWatermark(watermarkHtml, 45, VerticalAlignment.Middle, HorizontalAlignment.Center);
        pdf.SaveAs("watermarked.pdf");
    }
}

IronPDF's watermarking accepts HTML with CSS styling, eliminating manual coordinate calculations entirely. For image watermarks and advanced positioning, see the watermarking guide.

How Does Password Protection Compare?

iTextSharp — Encrypt PDF

// Install-Package iTextSharp

using System;
using System.IO;
using iTextSharp.text.pdf;

public class PdfSecurer
{
    public void ProtectPdf()
    {
        var inputFile = "original.pdf";
        var outputFile = "protected.pdf";
        var userPassword = "user123";
        var ownerPassword = "owner456";

        using (var reader = new PdfReader(inputFile))
        using (var stream = new FileStream(outputFile, FileMode.Create))
        {
            var stamper = new PdfStamper(reader, stream);

            try
            {
                stamper.SetEncryption(
                    PdfWriter.STRENGTH128BITS,
                    userPassword,
                    ownerPassword,
                    PdfWriter.ALLOW_PRINTING | PdfWriter.ALLOW_COPY
                );
            }
            finally
            {
                stamper.Close();
                reader.Close();
            }
        }
    }
}

Permissions use bitwise | operator with integer constants — no typed enums, unclear semantics between flags like ALLOW_DEGRADED_PRINTING vs ALLOW_PRINTING, and version-specific behavior depending on encryption strength.

IronPDF — Encrypt PDF

// Install-Package IronPdf

using IronPdf;

public class PdfSecurer
{
    public void ProtectPdf()
    {
        var pdf = PdfDocument.FromFile("original.pdf");

        pdf.Password = "user123";
        pdf.OwnerPassword = "owner456";

        pdf.SecuritySettings.AllowUserCopyPasteContent = true;
        pdf.SecuritySettings.AllowUserPrinting = PdfPrintSecurity.FullPrintRights;
        pdf.SecuritySettings.AllowUserFormData = false;
        pdf.SecuritySettings.AllowUserAnnotations = false;
        pdf.SecuritySettings.AllowUserAccessibilityExtractContent = true;

        pdf.SaveAs("protected.pdf");
    }
}

Explicit boolean properties and typed enums for permissions — no bitwise flag confusion. Each setting is self-documenting.

What Are the Known iTextSharp Bugs I Should Watch For?

These are commonly reported issues found in Stack Overflow discussions and GitHub issues. Verify against your specific version and use case.

CSS Table Rendering: XMLWorker mishandles border-collapse, padding, and nested tables — borders overlap or disappear.

Unicode Character Corruption: Non-ASCII characters (Arabic, Chinese, emoji) render as boxes or question marks without manual font embedding and encoding configuration.

Memory Growth in Loops: Creating PDFs in tight loops without explicit reader.Close() and document.Dispose() can cause memory to balloon until GC runs.

Thread Contention: Simultaneous PdfWriter instances on different threads can corrupt output files or throw IOException exceptions.

1-Based Indexing Errors: Developers used to 0-based C# collections frequently introduce off-by-one bugs when iterating pages.

When Should I Consider Migrating from iTextSharp?

Product discontinuation creates a hard ceiling for technical evolution. With iTextSharp frozen since 2016, there's no path to support modern .NET features introduced in C# 8-14 or .NET 6-10 improvements. Teams building new features face the choice: build workarounds in legacy code or migrate to an active library.

No support means unresolved bugs stay unresolved forever. If you hit a CSS rendering edge case in XMLWorker or a threading issue in PdfStamper, your options are Stack Overflow or source code surgery. For regulated industries where vendor support documentation is part of compliance, this becomes a hard blocker.

Missing features compound over time. When your UX team designs a dashboard with CSS Grid and you're stuck manually calculating coordinates, the gap between "design in Figma" and "output in PDF" widens. HTML email templates can't be reused for PDF reports. JavaScript-powered charts require server-side rendering workarounds.

Legacy architecture creates deployment friction. iTextSharp's .NET Framework roots limit container strategies (Windows-only images, larger attack surface) and complicate CI/CD pipelines. As teams standardize on .NET 8+ and Linux containers, running iTextSharp requires dual-targeting or separate build configurations.

Frequently Asked Questions

Is iTextSharp still safe to use in production?

Answer: iTextSharp 5.x still functions and receives critical security patches (the last was February 2024 for CVE-2025-8916). However, no bug fixes or feature development are happening. For stable, unchanging workloads it can continue running, but any new development against it accumulates technical debt with no vendor support path.

Can I migrate iTextSharp code to IronPDF incrementally?

Answer: Yes. The two libraries can coexist in the same project since they use different namespaces. You can migrate feature-by-feature — start with new PDF generation code using IronPDF while keeping existing iTextSharp code running. The API mapping table above shows the direct equivalents for each operation.

Does IronPDF support .NET Framework or only .NET Core?

Answer: IronPDF supports .NET Framework 4.6.2 and above, plus .NET 6, 7, 8, 9, and 10. It also runs cross-platform on Windows, Linux, and Docker. iTextSharp is limited to .NET Framework 2.0-4.8 and Windows only.

How does HTML rendering quality compare between the two?

Answer: iTextSharp's XMLWorker supports basic HTML 4 and CSS 2.1 — no Flexbox, Grid, viewport units, or JavaScript. IronPDF uses a full Chromium engine, so anything that renders in Chrome renders in IronPDF. This includes CSS3, JavaScript execution, responsive design with media queries, web fonts via @font-face, and full SVG support.

What about licensing differences?

Answer: iTextSharp 5.x uses the AGPL license, which requires you to release your source code if you distribute the software — or purchase a commercial license from iText. IronPDF uses a commercial license with a free trial available. There is no open-source obligation with IronPDF.

Is IronPDF thread-safe for concurrent PDF generation?

Answer: IronPDF has built-in async support with methods like RenderHtmlAsPdfAsync, and its architecture handles concurrent operations. iTextSharp's PdfWriter and PdfStamper are not thread-safe by default, requiring manual synchronization to avoid file corruption in concurrent environments.

How do I handle the 1-based vs 0-based page indexing difference?

Answer: iTextSharp uses 1-based page indexing (for (int i = 1; i <= NumberOfPages; i++)), which differs from standard C# conventions. IronPDF uses 0-based indexing consistent with C# arrays and lists. When migrating, subtract 1 from all page index references. This is a common source of off-by-one bugs during migration.

Decision Framework: Should You Migrate?

Stay with iTextSharp if: Your codebase is stable and unchanging, you have no new PDF features planned, you're committed to .NET Framework on Windows, you're comfortable with AGPL obligations or have a commercial license, and you accept no vendor support for unresolved bugs.

Migrate to IronPDF if: You need modern HTML/CSS rendering without manual PDF reconstruction, you're targeting .NET 6+ with Linux or Docker deployment, you're hitting documented bugs with no vendor to escalate to, compliance requirements demand active vendor support, or your team's PDF generation code is growing in complexity.

The core question is straightforward: iTextSharp is a deprecated library frozen in 2016 with no future roadmap. IronPDF is actively developed with a Chromium engine that handles modern web standards. Migration becomes mandatory when the cost of maintaining workarounds for iTextSharp's limitations exceeds the cost of switching.
