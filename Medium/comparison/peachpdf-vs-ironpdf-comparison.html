PeachPDF vs IronPDF: The .NET Developer's Decision Guide

One axis frames most HTML-to-PDF decisions in .NET: managed-only renderer versus embedded browser. PeachPDF sits firmly on the managed-only side — pure .NET, no Chromium to ship, a small NuGet footprint, and an async-first API. That deployment story holds up cleanly for text-based receipts and simple invoices, then bends when the same pipeline is asked to render a logo, styled tables with borders, and a multi-column layout matching the marketing site. The architectural question is not which library is "better" but which side of that axis matches the document complexity you actually need to render.

PeachPDF is an open-source HTML-to-PDF library built entirely in managed .NET code on top of PeachPDF.PdfSharpCore plus a custom HTML/CSS layout engine. IronPDF uses an embedded Chromium engine for browser-grade rendering. This comparison walks through the architectures, decision criteria with real code, and the practical differences that matter when you are choosing between deployment simplicity and rendering completeness.

What Is PeachPDF?

PeachPDF is an open-source HTML-to-PDF library (BSD-3-Clause) built entirely in managed .NET. It layers a custom HTML/CSS layout engine over PeachPDF.PdfSharpCore (a PDF construction library) and exposes an async PdfGenerator.GeneratePdf(html, config) API. The architecture avoids external processes — no Puppeteer, no wkhtmltopdf, no Chromium. This keeps deployment small (a single NuGet package, currently PeachPDF 0.7.x targeting net8.0) and resource usage minimal.

The tradeoff: CSS and HTML support is narrower than browser-based engines. Simple documents (text, basic tables, simple styling) render reliably; complex layouts (CSS Grid, Flexbox, advanced selectors) typically require workarounds or may render with visual differences from a browser — verify against your version.

What Is IronPDF?

IronPDF uses a different architecture: an embedded Chromium engine for HTML-to-PDF conversion. This means comprehensive HTML5, CSS3, and JavaScript support — if it renders in Chrome, it renders in the PDF. The tradeoff: larger deployment footprint (Chromium bundled) and higher memory usage (browser process overhead). For teams prioritizing rendering accuracy over deployment simplicity, Chromium's standards compliance reduces CSS troubleshooting.

IronPDF's workflow is: write HTML/CSS, call RenderHtmlAsPdf(), get the rendered output. The library handles web fonts, JavaScript execution, CSS media queries, and modern layout techniques without configuration. Installation is a single NuGet package with Chromium lifecycle managed automatically.

What Are the Key Limitations of PeachPDF?

Product status. Open-source project (BSD-3-Clause) on GitHub, actively maintained by a single primary maintainer. Currently pre-1.0 (latest 0.7.x as of late 2025) and targets net8.0 only — .NET Framework and .NET 6 projects need to upgrade or pick a different library. Community-driven support via GitHub Issues; no commercial support option is advertised. For teams requiring LTS or vendor support, that is a relevant input to risk planning.

HTML/CSS coverage. The renderer is a custom HTML/CSS layout engine over PeachPDF.PdfSharpCore, not a browser engine. Modern CSS features such as Flexbox, Grid, transforms, transitions, and animations are typically limited or unimplemented in this engine — verify against your version. JavaScript execution is not part of the rendering pipeline (the library targets static HTML). Web fonts via @font-face are supported. CSS media queries, complex selectors, SVG, and Canvas have limited or no coverage in common configurations.

Resource handling. Remote HTTP fetching for the document and its referenced resources is wired through a network adapter on PdfGenerateConfig.NetworkAdapter (for example, HttpClientNetworkAdapter with a base Uri); a self-contained MHTML adapter is also available. Without an adapter, images and stylesheets resolve from the local filesystem or data URIs. The CSS cascade and specificity rules are implemented at a level appropriate for simpler stylesheets — complex stylesheets may render with visual differences from a browser.

Support model. Community support via GitHub Issues. Documentation is primarily the README and source. Maintenance cadence depends on the maintainers' availability — a standard consideration for any small-team OSS dependency.

Architectural implications. A custom HTML/CSS parser means rendering will differ in places from Chromium or Gecko. Missing CSS features generally need to be addressed by HTML restructuring rather than configuration. There is no DOM inspector for the rendered output, so iteration on tricky layouts depends on producing the PDF and reviewing it.

How Do You Handle Images Not Displaying in Generated PDFs?

PeachPDF — Image Loading Issues

// Install-Package PeachPDF

using System;
using System.IO;
using System.Net.Http;
using System.Threading.Tasks;
using PeachPDF;
using PeachPDF.Network;
using PeachPDF.PdfSharpCore;

public class PdfImageExample
{
    public async Task GenerateWithDataUriImageAsync()
    {
        // Embed images as data URIs so they travel with the HTML payload
        byte[] imageBytes = File.ReadAllBytes("logo.png");
        string base64Image = Convert.ToBase64String(imageBytes);
        string dataUri = $"data:image/png;base64,{base64Image}";

        string html = $@"
            <html>
            <body>
                <h1>Report</h1>
                <img src='{dataUri}' />
            </body>
            </html>";

        var generator = new PdfGenerator();
        var document = await generator.GeneratePdf(html, new PdfGenerateConfig
        {
            PageSize = PageSize.Letter
        });

        using var stream = File.Create("output.pdf");
        document.Save(stream);
    }

    public async Task GenerateFromUrlAsync()
    {
        // URL fetching is wired through a network adapter; passing null HTML
        // tells PeachPDF to fetch the document via the adapter.
        var http = new HttpClient();
        var config = new PdfGenerateConfig
        {
            PageSize = PageSize.Letter,
            NetworkAdapter = new HttpClientNetworkAdapter(http, new Uri("https://example.com"))
        };

        var generator = new PdfGenerator();
        var document = await generator.GeneratePdf(null, config);

        using var stream = File.Create("output.pdf");
        document.Save(stream);
    }
}

Practical notes for image-heavy documents in this engine: data URIs embed cleanly but increase HTML payload size; remote image fetching is handled by the configured NetworkAdapter against the base Uri you supply. Without an adapter, set an explicit <base> tag or use absolute filesystem paths so the engine can resolve images consistently.

IronPDF — Image Handling (No Configuration Needed)

// Install-Package IronPdf

using IronPdf;

public class PdfImageHandling
{
    public void GenerateWithImages()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

            <html>
            <body>
                <h1>Report</h1>

Chromium handles HTTP requests, relative paths, and image formats the same way the browser does. See the HTML-to-PDF guide for advanced image handling.

How Does CSS Layout Rendering Compare?

PeachPDF — CSS Limitations

using System.IO;
using System.Threading.Tasks;
using PeachPDF;
using PeachPDF.PdfSharpCore;

public class PdfCssExample
{
    public async Task TableBasedLayoutAsync()
    {
        // Tables and floats are the safest cross-version layout primitives
        // for the custom CSS engine; verify Flexbox/Grid coverage against
        // the PeachPDF version you target.
        var htmlWithTable = @"
            <html>
            <head>
                <style>
                    table { width: 100%; border-collapse: collapse; }
                    td { width: 33.33%; padding: 10px; border: 1px solid black; }
                </style>
            </head>
            <body>
                <table>
                    <tr>
                        <td>Column 1</td>
                        <td>Column 2</td>
                        <td>Column 3</td>
                    </tr>
                </table>
            </body>
            </html>";

        var generator = new PdfGenerator();
        var document = await generator.GeneratePdf(htmlWithTable, new PdfGenerateConfig
        {
            PageSize = PageSize.Letter
        });

        using var stream = File.Create("output.pdf");
        document.Save(stream);
    }
}

Common CSS considerations with this engine: Flexbox and Grid coverage is typically limited (table or float-based layouts are the safe fallback), complex selectors such as :nth-child() may have partial support, pseudo-elements and media queries are limited in common configurations, and transforms/transitions are generally not part of the renderer's scope. Verify against the version you target.

How Does Font Rendering Compare?

PeachPDF — Custom Font Configuration

using System.IO;
using System.Threading.Tasks;
using PeachPDF;
using PeachPDF.PdfSharpCore;

public class PdfFontExample
{
    public async Task RenderWithSystemFontAsync()
    {
        // Available font families depend on what is installed on the host
        // and on the engine's font resolution; cross-platform behavior may
        // vary — verify on each target OS.
        var html = @"
            <html>
            <head>
                <style>
                    body { font-family: Arial, sans-serif; }
                </style>
            </head>
            <body>
                <h1>Invoice</h1>
                <p>Body text rendered with the configured font stack.</p>
            </body>
            </html>";

        var generator = new PdfGenerator();
        var document = await generator.GeneratePdf(html, new PdfGenerateConfig
        {
            PageSize = PageSize.Letter
        });

        using var stream = File.Create("output.pdf");
        document.Save(stream);
    }
}

Practical font notes for managed-only renderers: default font availability tracks the host OS, so the same HTML may resolve different glyphs on Windows vs. Linux vs. macOS — verify cross-platform output. Web fonts via @font-face are supported, and missing glyphs may fall back or render as blanks depending on engine version.

Chromium downloads and embeds web fonts automatically. System fonts work cross-platform with standard CSS font-family declarations.

What Does the Feature Comparison Look Like?

The two libraries optimize for different ends of the same axis. PeachPDF is an active OSS project (BSD-3-Clause) targeting net8.0 with a managed-only architecture, distributed via a small NuGet package and supported through GitHub Issues; its viability depends on a small group of maintainers. IronPDF is an active commercial product targeting .NET Framework and modern .NET on Windows, Linux, and macOS, distributed as a single NuGet that bundles Chromium and backed by commercial support channels.

HTML support tracks that split: PeachPDF renders basic HTML5 through a custom parser, with simple layouts coming through reliably; IronPDF renders the full HTML5 surface through Chromium, with the rendering fidelity you would expect from the browser engine.

What Does the Comprehensive Feature Breakdown Reveal?

Status & support.

  • License. PeachPDF is BSD-3-Clause open source; IronPDF is commercial.
  • Product status. Both are actively developed — PeachPDF as a pre-1.0 community project, IronPDF under a commercial entity.
  • Support. PeachPDF support runs through GitHub Issues; IronPDF offers multiple support channels.
  • .NET targets. PeachPDF currently targets net8.0 only; IronPDF supports .NET Framework and modern .NET — verify the current target list against the package's NuGet page.
  • Platforms. Both run on Windows, Linux, and macOS.

HTML/CSS support.

  • HTML5. PeachPDF parses a working subset; IronPDF uses Chromium's full HTML5 engine.
  • CSS3. PeachPDF support is limited to what its custom engine implements; IronPDF inherits Chromium's CSS3.
  • Flexbox and Grid. Typically limited or unimplemented in PeachPDF's engine — verify against your version. Fully supported in IronPDF via Chromium.
  • Transforms, transitions, media queries, complex selectors. Limited in PeachPDF's custom engine; full in IronPDF.
  • Web fonts. PeachPDF accepts fonts via configuration; IronPDF resolves @font-face automatically through Chromium.

Content handling.

  • Remote images. Resolved via PdfGenerateConfig.NetworkAdapter (e.g., HttpClientNetworkAdapter) or data-URI embedding in PeachPDF; handled natively by Chromium in IronPDF.
  • Data URIs. Supported by both.
  • JavaScript. PeachPDF targets static HTML — no JS execution. IronPDF executes JS through Chromium.
  • SVG and Canvas. Limited or no coverage in PeachPDF's engine; supported in IronPDF.
  • Forms. Static display only in PeachPDF; interactive form support in IronPDF.

PDF operations.

  • HTML-to-PDF. Supported by both.
  • Merge, split, text extraction, watermarks, encryption, digital signatures. Outside PeachPDF's current scope (it focuses on HTML-to-PDF); covered by IronPDF.
  • Edit existing PDFs. Not the focus of PeachPDF; limited but available in IronPDF.

Deployment.

  • Install size. PeachPDF is a small managed-only package; IronPDF bundles Chromium, so the on-disk footprint is materially larger.
  • Memory usage. Lower for PeachPDF; moderate for IronPDF due to the browser process.
  • Dependencies. Managed .NET only for PeachPDF; Chromium bundled by IronPDF (no separate install required).
  • Container friendliness. Both run in containers — PeachPDF with very small images, IronPDF with the standard Chromium-on-Linux setup.

What Does the API Mapping Between PeachPDF and IronPDF Look Like?

  • new PdfGenerator() — maps to new ChromePdfRenderer().
  • await generator.GeneratePdf(html, config) — maps to renderer.RenderHtmlAsPdf(html) (or RenderHtmlAsPdfAsync). PeachPDF's call is async; IronPDF exposes both sync and async overloads.
  • URL fetching via PdfGenerateConfig.NetworkAdapter = new HttpClientNetworkAdapter(http, baseUri) with a null HTML argument — maps to renderer.RenderUrlAsPdf(url).
  • Headers and footers — PeachPDF v0.7.x has no header/footer property; the pattern is to embed CSS-positioned markup inside the source HTML. In IronPDF, set renderer.RenderingOptions.HtmlHeader / HtmlFooter as HtmlHeaderFooter objects with HtmlFragment and layout properties.
  • Page size and orientation — PdfGenerateConfig.PageSize / PageOrientation; in IronPDF, set renderer.RenderingOptions.PaperSize and PaperOrientation.
  • Font configuration — PeachPDF resolves fonts against the host OS and supports @font-face; IronPDF resolves system and web fonts through Chromium without explicit registration in most cases.
  • Network resource loading — PeachPDF uses the configured NetworkAdapter (with MimeKitNetworkAdapter available for self-contained MHTML); in IronPDF, HTTP and relative-path resolution are handled by Chromium.
  • Output — PeachPDF returns a document whose Save(stream) writes to a Stream; IronPDF exposes pdf.SaveAs(path), pdf.BinaryData, and pdf.Stream.
  • PDF post-processing (merge, split, watermarks, metadata, security, signatures, text extraction) — outside PeachPDF's HTML-to-PDF scope; available on the PdfDocument surface in IronPDF (e.g., PdfDocument.Merge, pdf.MetaData, pdf.SecuritySettings).

Frequently Asked Questions

Is PeachPDF suitable for production use?

PeachPDF is an actively developed BSD-3-Clause project, currently pre-1.0. It targets net8.0 and is supported through GitHub Issues; commercial support and SLAs are not part of its model. For teams comfortable with the OSS support model, it is a reasonable fit for HTML-to-PDF in managed-only environments. Teams that need vendor SLAs or LTS guarantees usually weigh that against the project's small-team maintenance footprint.

Can PeachPDF render modern CSS like Flexbox and Grid?

PeachPDF uses a custom HTML/CSS engine rather than a browser engine, so coverage of Flexbox (display: flex) and CSS Grid (display: grid) is typically limited compared with Chromium — verify against your version. The common pattern is to express multi-column layouts using HTML tables or floats. Complex selectors, pseudo-elements, transforms, and transitions are similarly limited in this engine.

Why does IronPDF handle images and fonts without configuration?

IronPDF embeds a Chromium engine that handles HTTP requests, relative paths, image formats, and web font downloading natively — the same way Chrome does. PeachPDF wires resource fetching through a NetworkAdapter on PdfGenerateConfig and resolves fonts against the host OS, which means there is more explicit setup for non-trivial scenarios.

What drives migration from PeachPDF to IronPDF?

CSS coverage drives most evaluations. Teams adopt PeachPDF for deployment simplicity, then discover that Flexbox and Grid layouts need rework and complex selectors produce unexpected results in this engine. Resource handling (remote URLs, relative paths) and the absence of JavaScript execution add friction for teams whose source HTML was authored for a browser. When the visual gap between PDF output and the same HTML rendered in a browser becomes unacceptable, or when production support requirements exceed community-driven OSS risk tolerance, teams evaluate IronPDF's Chromium-based rendering.

Decision Framework: Which Library Fits Your Project?

PeachPDF solves a specific problem: HTML-to-PDF conversion in pure .NET without browser dependencies. The lightweight architecture (managed code only) keeps deployment simple — copy DLL, run anywhere. For teams prioritizing minimal resource usage and simple HTML documents (text, basic tables, minimal CSS), PeachPDF delivers without external process overhead. The BSD-3-Clause license provides transparency and flexibility.

Migration from PeachPDF becomes relevant when: (1) CSS gaps require enough HTML restructuring to outweigh the development velocity gains, (2) remote-resource handling creates more friction than a browser-based approach, (3) the visual gap between PDF output and the same HTML rendered in a browser becomes unacceptable, or (4) production support requirements exceed community-driven OSS risk tolerance.

IronPDF addresses comprehensive HTML rendering: the Chromium engine provides CSS3, HTML5, and JavaScript support matching browser standards. The tradeoff is a larger deployment footprint (Chromium bundled) and higher memory usage. For teams where rendering accuracy, CSS compatibility, and vendor support outweigh deployment size concerns, Chromium-based rendering reduces the troubleshooting surface for CSS and resource loading. The evaluation centers on deployment simplicity versus rendering completeness.
