PeachPDF vs IronPDF: The .NET Developer's Decision Guide for 2025

A common scenario in containerized .NET apps: teams want HTML-to-PDF conversion without the overhead of bundling Chromium or managing browser processes. PeachPDF appears as the solution — pure .NET, no external dependencies, lightweight. One team integrated it successfully for simple text-based receipts. Then the product manager requested adding a company logo, styled tables with borders, and a multi-column layout matching the website design. That's when troubleshooting started: images didn't load, CSS flexbox wasn't supported, multi-column layouts required manual workarounds. The "simple" library solved the deployment problem but created a rendering problem.

PeachPDF is an open-source HTML-to-PDF library built entirely in managed .NET code using PDFSharpCore plus a custom HTML parser. IronPDF uses an embedded Chromium engine for browser-grade rendering. This comparison walks through the limitations, decision criteria with real code, and the practical differences that matter when you're choosing between deployment simplicity and rendering completeness.

What Is PeachPDF?

Answer:

PeachPDF is an open-source HTML-to-PDF library built entirely in managed .NET code. It uses PDFSharpCore (a PDF construction library) plus a custom HTML parser to convert HTML into PDF documents. The architecture avoids external processes — no Puppeteer, no wkhtmltopdf, no Chromium. This makes deployment trivial (single NuGet package) and resource usage minimal (no browser overhead).

The tradeoff: CSS and HTML support is limited compared to browser-based engines. Simple documents (text, basic tables, simple styling) render reliably; complex layouts (CSS Grid, Flexbox, advanced selectors) require workarounds or fail silently.

What Is IronPDF?

Answer:

IronPDF uses a different architecture: embedded Chromium engine for HTML-to-PDF conversion. This means comprehensive HTML5, CSS3, and JavaScript support — if it renders in Chrome, it renders in the PDF. The tradeoff: larger deployment footprint (Chromium bundled) and higher memory usage (browser process overhead). For teams prioritizing rendering accuracy over deployment simplicity, Chromium's standards compliance eliminates CSS troubleshooting.

IronPDF's workflow is: write HTML/CSS, call RenderHtmlAsPdf(), get pixel-perfect output. The library handles web fonts, JavaScript execution, CSS media queries, and modern layout techniques without configuration. Installation is a single NuGet package with Chromium lifecycle managed automatically.

What Are the Key Limitations of PeachPDF?

Answer:

Product Status: Open-source project (MIT license) on GitHub, actively maintained by an individual developer. Requires .NET 8+ (no .NET Framework support). Relatively new library without extensive production track record. Community-driven support via GitHub Issues. No commercial support option. For teams requiring LTS or vendor support, this creates risk.

Missing Capabilities: No advanced CSS support — Flexbox, Grid, transforms, transitions, animations not implemented. JavaScript execution not supported (purely static HTML rendering). Web fonts via @font-face supported but require careful configuration. CSS media queries limited. No support for complex selectors (attribute selectors, pseudo-classes beyond basics). SVG support limited. Canvas elements not supported.

Technical Issues: Images must be local filesystem or data URIs — remote HTTP images require custom NetworkAdapter implementation. Base URLs for relative paths don't auto-detect — must manually configure. CSS cascade and specificity rules partially implemented — complex stylesheets produce unexpected results. Layout engine is custom, not standards-compliant — visual differences from browsers. Table rendering works for basic cases but struggles with complex nesting. Font fallback limited — missing glyphs render as blanks. Right-to-left text and complex scripts not fully supported.

Support Status: Community support only — GitHub Issues for bug reports and questions. No SLA, no guaranteed response time. Documentation limited to README file. No commercial entity backing the project. Maintenance depends on single maintainer's availability.

Architecture Problems: Custom HTML/CSS parser means non-standard rendering behavior. Missing CSS features require code workarounds or HTML restructuring. No inspector or debugging tools for layout issues — must iterate blind. Error messages sparse — silent failures for unsupported CSS. Memory usage unpredictable with complex HTML. PDF output quality varies based on HTML complexity.

How Do You Handle Images Not Displaying in Generated PDFs?

Answer:

PeachPDF — Image Loading Issues

// Install-Package PeachPDF

using System;
using System.IO;
using System.Threading.Tasks;
using PeachPDF;

public class PdfImageTroubleshooting
{
    public async Task GenerateWithImages()
    {
        // PROBLEM: Remote images don't load by default
        var htmlWithRemoteImage = @"
            <html>
            <body>
                <h1>Report</h1>
                <img src='https://example.com/logo.png' />
            </body>
            </html>";

        var config = new PdfGenerateConfig
        {
            PageSize = PageSize.Letter,
            PageOrientation = PageOrientation.Portrait
        };

        var generator = new PdfGenerator();

        // ISSUE: Image will be missing in PDF - no HTTP support by default
        using var stream = new MemoryStream();
        var document = await generator.GeneratePdf(htmlWithRemoteImage, config);
        document.Save(stream);
        // Result: PDF generated but image missing, no error thrown
    }

    public async Task SolutionDataUri()
    {
        // WORKAROUND 1: Convert images to data URIs
        byte[] imageBytes = File.ReadAllBytes("logo.png");
        string base64Image = Convert.ToBase64String(imageBytes);
        string dataUri = $"data:image/png;base64,{base64Image}";

        var html = $@"
            <html>
            <body>
                <h1>Report</h1>
                <img src='{dataUri}' />
            </body>
            </html>";

        var generator = new PdfGenerator();
        var config = new PdfGenerateConfig { PageSize = PageSize.Letter };

        using var stream = new MemoryStream();
        var document = await generator.GeneratePdf(html, config);
        document.Save(stream);
        // Result: Image displays correctly
    }

    public async Task SolutionHttpClient()
    {
        // WORKAROUND 2: Implement custom NetworkAdapter
        using var httpClient = new HttpClient();

        var config = new PdfGenerateConfig
        {
            PageSize = PageSize.Letter,
            // Custom network adapter for HTTP image loading
            NetworkAdapter = new HttpClientNetworkAdapter(
                httpClient,
                new Uri("https://example.com")
            )
        };

        var html = @"
            <html>
            <head>
                <base href='https://example.com/' />
            </head>
            <body>
                <h1>Report</h1>
                <img src='logo.png' />
            </body>
            </html>";

        var generator = new PdfGenerator();
        using var stream = new MemoryStream();
        var document = await generator.GeneratePdf(html, config);
        document.Save(stream);
        // Result: Image loads via HTTP
    }
}

Common troubleshooting issues: remote images missing without custom NetworkAdapter, relative paths fail without manual <base> tag, large images as base64 bloat HTML, failed image loads produce blank spaces without warnings.

IronPDF — Image Handling (No Configuration Needed)

// Install-Package IronPdf

using IronPdf;

public class PdfImageHandling
{
    public void GenerateWithImages()
    {
        // Remote images work automatically
        var html = @"
            <html>
            <body>
                <h1>Report</h1>
                <img src='https://example.com/logo.png' />
            </body>
            </html>";

        var renderer = new ChromePdfRenderer();

        // Result: Image loads and renders automatically
    }
}

Chromium handles HTTP requests, relative paths, and image formats natively. No configuration needed for standard scenarios. See the HTML-to-PDF guide for advanced image handling.

                        width: 33.33%;

                </style>
            </head>
            <body>
                <table>
                    <tr>
                        <td>Column 1</td>
                        <td>Column 2</td>
                        <td>Column 3</td>
                    </tr>
                </table>
            </body>
            </html>";

        var generator = new PdfGenerator();
        var config = new PdfGenerateConfig { PageSize = PageSize.Letter };

        using var stream = new MemoryStream();
        var document = await generator.GeneratePdf(htmlWithTable, config);
        document.Save(stream);
        // RESULT: Table layout renders correctly
    }
}

Common CSS troubleshooting issues: Flexbox and Grid not supported (use tables or floats instead), complex selectors like :nth-child() may not work, pseudo-elements limited, CSS specificity partially implemented, media queries limited, and transforms/transitions ignored entirely.

How Does Font Rendering Compare?

Answer:

PeachPDF — Custom Font Configuration

using System;
using System.IO;
using System.Threading.Tasks;
using PeachPDF;

public class PdfFontTroubleshooting
{
    public async Task DefaultFontIssue()
    {
        // ISSUE: Default font is Segoe UI (Windows-specific)
        var html = @"
            <html>
            <head>
                <style>
                    body { font-family: Arial; }
                </style>
            </head>
            <body>
                <h1>Invoice</h1>
                <p>This should use Arial font.</p>
            </body>
            </html>";

        var generator = new PdfGenerator();
        var config = new PdfGenerateConfig { PageSize = PageSize.Letter };

        using var stream = new MemoryStream();
        var document = await generator.GeneratePdf(html, config);
        document.Save(stream);
        // RESULT: Font may render as Segoe UI or fallback
        // On Linux/Mac: Segoe UI not available, font substitution occurs
    }

    public async Task CustomFontLoading()
    {
        // WORKAROUND: Load custom fonts at runtime
        var generator = new PdfGenerator();

        // Load TrueType font from stream
        using var fontStream = File.OpenRead("Arial.ttf");
        await generator.AddFontFromStream(fontStream);

        var html = @"
            <html>
            <head>
                <style>
                    body { font-family: Arial; }
                </style>
            </head>
            <body>
                <h1>Invoice</h1>
                <p>Font loaded explicitly.</p>
            </body>
            </html>";

        var config = new PdfGenerateConfig { PageSize = PageSize.Letter };

        using var stream = new MemoryStream();
        var document = await generator.GeneratePdf(html, config);
        document.Save(stream);
        // RESULT: Custom font renders correctly
    }
}

Common font troubleshooting issues: Segoe UI dependency not available on all platforms, font fallback limited (missing glyphs render as blanks), web fonts require local files, cross-platform inconsistency in font availability, and no system font enumeration.

Frequently Asked Questions

Is PeachPDF suitable for production use?

PeachPDF is an open-source project maintained by an individual developer under the MIT license. It requires .NET 8+ and has no commercial support option. There is no SLA, no guaranteed response time, and documentation is limited to the README file. For teams comfortable with the OSS support model and willing to contribute fixes, it works. For teams requiring vendor support or LTS guarantees, the single-maintainer dependency creates risk.

Can PeachPDF render modern CSS like Flexbox and Grid?

No. PeachPDF uses a custom HTML/CSS parser, not a browser engine. Flexbox (display: flex) is not supported — elements stack vertically instead. CSS Grid (display: grid) is also unsupported. The workaround is to rewrite layouts using HTML tables or floats. Complex selectors, pseudo-elements, transforms, and transitions are either limited or ignored entirely.

Why does IronPDF handle images and fonts without configuration?

IronPDF embeds a Chromium engine that handles HTTP requests, relative paths, image formats, and web font downloading natively — the same way Chrome browser does. PeachPDF requires custom NetworkAdapter implementation for remote images and manual font loading via AddFontFromStream(). The architectural difference means IronPDF works out of the box while PeachPDF requires workarounds for anything beyond local files.

What drives migration from PeachPDF to IronPDF?

CSS limitations drive most migration evaluations. Teams adopt PeachPDF for deployment simplicity, then discover Flexbox layouts fail, Grid doesn't work, and complex selectors produce unexpected results. Image loading complexity (remote URLs requiring custom NetworkAdapter, relative paths needing manual base URL configuration) and JavaScript absence (no dynamic content rendering) add friction. When rendering accuracy gaps between PDF output and browser preview become unacceptable, or when production support requirements exceed community-driven OSS risk tolerance, teams evaluate IronPDF's Chromium-based rendering.

Decision Framework: Which Library Fits Your Project?

Answer:

PeachPDF solves a specific problem: HTML-to-PDF conversion in pure .NET without browser dependencies. The lightweight architecture (managed code only, ~5MB) enables trivial deployment — copy DLL, run anywhere. For teams prioritizing minimal resource usage and simple HTML documents (text, basic tables, minimal CSS), PeachPDF delivers without external process overhead. The MIT license provides transparency and flexibility.

Migration from PeachPDF becomes relevant when: (1) CSS limitations require extensive HTML workarounds negating development velocity benefits, (2) image loading complexity creates more friction than browser-based automatic handling, (3) rendering accuracy gaps between PDF output and browser preview become unacceptable, or (4) production support requirements exceed community-driven OSS risk tolerance.

IronPDF addresses comprehensive HTML rendering: Chromium engine provides CSS3, HTML5, and JavaScript support matching browser standards. The tradeoff is larger deployment footprint (~100MB with Chromium) and higher memory usage. For teams where rendering accuracy, CSS compatibility, and vendor support outweigh deployment size concerns, Chromium-based rendering eliminates troubleshooting CSS limitations and image loading configurations. The evaluation centers on deployment simplicity versus rendering completeness.
