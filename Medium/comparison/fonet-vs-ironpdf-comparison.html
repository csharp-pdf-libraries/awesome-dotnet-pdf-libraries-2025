FO.NET vs IronPDF: The .NET Developer's Decision Guide

Two dates frame this comparison. The W3C closed the XSL-FO Working Group in 2013 — the specification has not advanced since. The most widely used Fonet NuGet package was last published in April 2011 against .NET Framework 2.0; the community Fonet.Standard fork last shipped in May 2020 against .NET Standard 2.0. If your reporting pipeline still consumes XSL-FO, the architectural question is whether to keep an obsolete-upstream dependency or move to a maintained renderer. FO.NET was designed for precise XML-based page layout, targeting applications where data arrives as XML and must render with millimeter-level positioning accuracy.

IronPDF approaches document generation from the opposite direction: it assumes your layout is expressed in HTML/CSS (the universal design language of the web) and renders PDFs using a Chromium engine. This architectural choice reflects the reality of modern .NET development, where most teams already use HTML for web UIs and prefer reusing those skills for PDF generation rather than learning XSL-FO's specialized syntax.

What is IronPDF?

IronPDF is a .NET library built around HTML-to-PDF conversion, leveraging Chromium's rendering engine to produce PDFs that match browser output. Teams adopt it when they need to convert web pages, Razor views, or dynamically generated HTML into distributable documents. The library handles modern CSS (Grid, Flexbox), JavaScript execution, and web fonts, ensuring that complex layouts render correctly without manual coordinate positioning.

Beyond rendering, IronPDF provides full PDF lifecycle management: merging documents, extracting text, filling forms, applying digital signatures, encrypting files, and manipulating pages. The API is designed for developers already familiar with HTML/CSS, eliminating the learning curve associated with XSL-FO's formatting object model.

What are the key limitations of FO.NET?

FO.NET is a one-way XSL-FO renderer, and that architectural choice defines its scope. The original CodePlex project is no longer active. The Fonet package on NuGet (v1.0.0, April 2011, targeting .NET Framework 2.0) and the community fork Fonet.Standard (v1.0.5, May 2020, targeting .NET Standard 2.0) are both still installable but neither receives active development. Several GitHub forks exist (nholik, hahmed, leekelleher) but show minimal activity in recent years. There is no central maintainer, no public roadmap, and no published security patch process. Teams running FO.NET in production are effectively maintaining the dependency themselves.

On the platform side, the legacy NuGet build targets .NET Framework 2.0 and the .NET Standard fork targets netstandard2.0; neither line tracks current .NET releases. The XSL-FO spec itself is a W3C Recommendation from 2006, and the W3C closed the XSL-FO Working Group in 2013, so the language has not absorbed any of the CSS3 features added since. FO.NET also does not accept HTML as input — it expects XSL-FO, which usually means an XSLT pre-processing step from your source XML. The library is scoped to rendering: it does support PDF user/owner password encryption via PdfRendererOptions (legacy 40/128-bit RC4 inherited from early Apache FOP), but interactive PDF forms, digital signatures, granular permission flags, and AES-256 encryption are outside its design.

The mental model is XSL-FO formatting objects (fo:block, fo:table, fo:page-sequence) rather than HTML/CSS, which is a separate skill set for most .NET teams. Font embedding is configured manually. Unicode coverage depends on the fonts you register. Large documents and complex tables tend to be the stress points; behavior here can vary by fork.

There is no commercial support channel. Community support flows through individual GitHub forks. Primary reference material is the W3C XSL-FO spec and the Apache FOP documentation, which is Java-oriented. Troubleshooting often means reading the source.

The two-stage pipeline (XML to XSL-FO to PDF) means failures can surface in either stage. XSLT can be hard to debug. Browser DevTools and CSS frameworks do not apply. The pipeline is tightly coupled to XML; JSON or POCO data needs conversion first. There is no browser-based preview — you generate the PDF to see the result.

How do page breaks compare between FO.NET and IronPDF?

Page break control is one of the areas where the architectural difference between XSL-FO and HTML/CSS becomes most apparent. In FO.NET, you work with formatting objects and attributes like break-before="page" and keep-together="always". In IronPDF, you use standard CSS properties that work the same way they do in a browser.

FO.NET: XSL-FO Page Break Control

<!-- XSL-FO document structure -->
<fo:root xmlns:fo="http://www.w3.org/1999/XSL/Format">
  <fo:layout-master-set>
    <fo:simple-page-master master-name="A4">
      <fo:region-body margin="1in"/>
    </fo:simple-page-master>
  </fo:layout-master-set>

  <fo:page-sequence master-reference="A4">
    <fo:flow flow-name="xsl-region-body">

      <fo:block font-size="18pt" space-after="12pt">
        Section 1: Introduction
      </fo:block>
      <fo:block>Content for section 1...</fo:block>

      <!-- Force page break -->
      <fo:block break-before="page"/>

      <fo:block font-size="18pt" space-after="12pt">
        Section 2: Details
      </fo:block>
      <fo:block>Content for section 2...</fo:block>

      <!-- Keep content together on same page -->
      <fo:block keep-together="always">
        <fo:block font-weight="bold">Critical Data:</fo:block>
        <fo:table>
          <fo:table-column column-width="3in"/>
          <fo:table-column column-width="2in"/>
          <fo:table-body>
            <fo:table-row>
              <fo:table-cell><fo:block>Metric A</fo:block></fo:table-cell>
              <fo:table-cell><fo:block>Value 1</fo:block></fo:table-cell>
            </fo:table-row>
          </fo:table-body>
        </fo:table>
      </fo:block>

    </fo:flow>
  </fo:page-sequence>
</fo:root>

A few practical notes when working with XSL-FO page breaks: break-before may not trigger as expected if preceding content does not fill the page; keep-together="always" can produce overflow conditions when content exceeds page height; table column widths typically need explicit values rather than relying on auto-sizing; and fonts must be registered manually before they will embed correctly. Behavior here can vary by fork — verify against the version you are running.

IronPDF: CSS-Based Page Control

using IronPdf;
using System;

namespace IronPdfPageBreakExample
{
    class Program
    {
        static void Main(string[] args)
        {
            IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

            var htmlWithPageBreaks = @"
                <html>
                <head>
                    <style>
                        body { font-family: Arial, sans-serif; }
                        h1 { font-size: 18pt; margin-bottom: 12pt; }
                        .page-break { page-break-before: always; }
                        .keep-together { page-break-inside: avoid; }
                        table { border-collapse: collapse; width: 100%; }
                        td { border: 1px solid #ccc; padding: 8px; }
                    </style>
                </head>
                <body>
                    <h1>Section 1: Introduction</h1>
                    <p>Content for section 1...</p>

                    <div class='page-break'></div>

                    <h1>Section 2: Details</h1>
                    <p>Content for section 2...</p>

                    <div class='keep-together'>
                        <strong>Critical Data:</strong>
                        <table>
                            <tr><td>Metric A</td><td>Value 1</td></tr>
                            <tr><td>Metric B</td><td>Value 2</td></tr>
                        </table>
                    </div>
                </body>
                </html>";

            var renderer = new ChromePdfRenderer();
            renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
            renderer.RenderingOptions.MarginTop = 72; // 1 inch = 72 points
            renderer.RenderingOptions.MarginBottom = 72;

            var pdf = renderer.RenderHtmlAsPdf(htmlWithPageBreaks);
            pdf.SaveAs("SectionedReport.pdf");

            Console.WriteLine("PDF created with controlled page breaks");
            pdf.Dispose();
        }
    }
}

CSS page-break-before and page-break-inside work as expected in browsers, making layout predictable. For complex page break scenarios — widow/orphan control, running headers — see IronPDF's HTML to PDF guide.

How does font rendering differ between FO.NET and IronPDF?

Font handling reflects FO.NET's XSL-FO origins. Fonts are registered manually in a configuration file with explicit paths to TTF files. Missing fonts typically fall back to a default sans-serif. Bold and italic variants are registered as separate font families. Unicode coverage tracks the fonts you register — there is no system font auto-discovery.

FO.NET: Manual Font Configuration

<!-- XSL-FO font configuration -->
<fo:root xmlns:fo="http://www.w3.org/1999/XSL/Format">
  <fo:layout-master-set>
    <fo:simple-page-master master-name="main">
      <fo:region-body margin="1in"/>
    </fo:simple-page-master>
  </fo:layout-master-set>

  <fo:page-sequence master-reference="main">
    <fo:flow flow-name="xsl-region-body">

      <fo:block font-family="Arial" font-size="12pt">
        Standard Arial text
      </fo:block>

      <fo:block font-family="CustomFont" font-size="14pt">
        Custom font text (requires font file registration)
      </fo:block>

      <fo:block font-family="Arial" font-weight="bold">
        Bold text
      </fo:block>

    </fo:flow>
  </fo:page-sequence>
</fo:root>

IronPDF: Automatic Font Handling

using IronPdf;

namespace IronPdfFontExample

                <head>

            renderer.RenderingOptions.PrintHtmlBackgrounds = true;

            pdf.SaveAs("FontTestReport.pdf");

            Console.WriteLine("PDF with various fonts created");
            pdf.Dispose();
        }
    }
}

IronPDF's Chromium engine handles fonts like a browser: system fonts work automatically, web fonts load from CDNs, and Unicode support is comprehensive. No manual configuration required.

How does dynamic table generation compare?

Tables show the architectural split clearly. In XSL-FO you typically declare column widths up front — proportional or fixed — because there is no auto-sizing equivalent to HTML's content-driven layout. Very large tables can be slow depending on the fork. In IronPDF, HTML tables handle column sizing, text wrapping, and header repetition (via thead) the same way they would in a browser.

FO.NET: XSL-FO Table Construction

<!-- XSL-FO table with data binding (typically generated via XSLT) -->
<fo:root xmlns:fo="http://www.w3.org/1999/XSL/Format">
  <fo:layout-master-set>
    <fo:simple-page-master master-name="table-page">
      <fo:region-body margin="0.5in"/>
    </fo:simple-page-master>
  </fo:layout-master-set>

  <fo:page-sequence master-reference="table-page">
    <fo:flow flow-name="xsl-region-body">

      <fo:table table-layout="fixed" width="100%">
        <fo:table-column column-width="proportional-column-width(2)"/>
        <fo:table-column column-width="proportional-column-width(1)"/>
        <fo:table-column column-width="proportional-column-width(1)"/>

        <fo:table-header>
          <fo:table-row background-color="#cccccc">
            <fo:table-cell border="1pt solid black" padding="4pt">
              <fo:block font-weight="bold">Product</fo:block>
            </fo:table-cell>
            <fo:table-cell border="1pt solid black" padding="4pt">
              <fo:block font-weight="bold">Quantity</fo:block>
            </fo:table-cell>
            <fo:table-cell border="1pt solid black" padding="4pt">
              <fo:block font-weight="bold">Price</fo:block>
            </fo:table-cell>
          </fo:table-row>
        </fo:table-header>

        <fo:table-body>
          <!-- Data rows - typically generated via XSLT loop -->
          <fo:table-row>
            <fo:table-cell border="1pt solid black" padding="4pt">
              <fo:block>Item A</fo:block>
            </fo:table-cell>
            <fo:table-cell border="1pt solid black" padding="4pt">
              <fo:block>100</fo:block>
            </fo:table-cell>
            <fo:table-cell border="1pt solid black" padding="4pt">
              <fo:block>$50.00</fo:block>
            </fo:table-cell>
          </fo:table-row>
        </fo:table-body>
      </fo:table>

    </fo:flow>
  </fo:page-sequence>
</fo:root>

IronPDF: HTML Table Rendering

using IronPdf;
using System;
using System.Text;

namespace IronPdfTableExample
{
    class Program
    {
        static void Main(string[] args)
        {
            IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

            var tableHtml = GenerateTableHtml(1000); // Generate large table

            var renderer = new ChromePdfRenderer();
            renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

            var pdf = renderer.RenderHtmlAsPdf(tableHtml);
            pdf.SaveAs("LargeDataTable.pdf");

            Console.WriteLine("Large table PDF created");
            pdf.Dispose();
        }

        static string GenerateTableHtml(int rowCount)
        {
            var sb = new StringBuilder();
            sb.AppendLine("<html><head><style>");
            sb.AppendLine("table { width: 100%; border-collapse: collapse; }");
            sb.AppendLine("th { background-color: #ccc; padding: 8px; border: 1px solid black; }");
            sb.AppendLine("td { padding: 8px; border: 1px solid black; }");
            sb.AppendLine("thead { display: table-header-group; }"); // Repeat header
            sb.AppendLine("</style></head><body>");
            sb.AppendLine("<table>");
            sb.AppendLine("<thead><tr><th>Product</th><th>Quantity</th><th>Price</th></tr></thead>");
            sb.AppendLine("<tbody>");

            for (int i = 0; i < rowCount; i++)
            {
                sb.AppendLine($"<tr><td>Item {i}</td><td>{i * 10}</td><td>${i * 5.50:F2}</td></tr>");
            }

            sb.AppendLine("</tbody></table></body></html>");
            return sb.ToString();
        }
    }
}

The thead element automatically repeats on page breaks in IronPDF. No manual configuration needed for header repetition, column sizing, or text wrapping.

How do the features compare?

The libraries sit at different points in their lifecycles and target different input formats, so most rows below are architectural facts rather than vendor-versus-vendor scores.

  • Project status. FO.NET is community-maintained via GitHub forks with no central maintainer or published roadmap; IronPDF is actively maintained by Iron Software with regular releases.
  • Support model. FO.NET support is community-driven through fork repos, with reference material being the W3C XSL-FO spec and Apache FOP documentation. IronPDF offers commercial support and first-party documentation and samples.
  • Layout language. FO.NET consumes XSL-FO (XML formatting objects). IronPDF consumes HTML/CSS via a Chromium engine, including CSS3 features and JavaScript execution. FO.NET does not accept HTML directly; IronPDF accepts XML only after conversion to HTML.
  • PDF lifecycle scope. FO.NET is a one-way renderer — by design it does not merge, split, extract text, edit existing PDFs, or apply digital signatures. IronPDF covers those operations as part of its API surface. PDF form filling in FO.NET is limited.
  • Security features. FO.NET supports user and owner passwords through PdfRendererOptions (legacy 40/128-bit RC4 only); it does not support digital signatures, granular permission flags, or AES-256. IronPDF supports AES 128/256-bit encryption, user/owner passwords, granular permission controls, and digital signatures.
  • Modern runtime support. The original Fonet NuGet targets .NET Framework 2.0 and the Fonet.Standard fork targets .NET Standard 2.0; neither line currently tracks .NET 5+. IronPDF supports .NET Framework 4.6.2+, .NET Core 3.1+, and .NET 5/6/7/8/9+. Linux and Docker support for FO.NET varies by fork; IronPDF supports both.
  • Fonts and Unicode. FO.NET requires manual font registration, with Unicode coverage tied to registered fonts. IronPDF inherits Chromium's font handling — system fonts, web fonts via CDN, and broad Unicode coverage out of the box.
  • Tables and large documents. XSL-FO tables typically need explicit column widths and large documents can be slow depending on the fork. IronPDF tables follow HTML semantics including thead repetition on page breaks.

FAQ and Troubleshooting Tips

Can I migrate from FO.NET to IronPDF without rewriting all my templates?

You will need to convert XSL-FO templates to HTML/CSS, but for most teams this is a net positive. XSL-FO formatting objects map to standard HTML elements: fo:block becomes <div> or <p>, fo:table becomes <table>, and break-before="page" becomes CSS page-break-before: always. The API mapping reference in the source covers the full set of equivalences. Most developers already know HTML/CSS, so the rewrite eliminates the XSL-FO learning curve permanently.

Does IronPDF handle the same XML data sources FO.NET uses?

FO.NET relies on XML input processed through XSLT to produce XSL-FO. With IronPDF, you can process the same XML data using C# (LINQ to XML, XmlDocument, or any serializer) and generate HTML strings dynamically. This replaces the XSLT transformation step with standard .NET code, which is easier to debug and maintain. For a complete walkthrough of dynamic HTML generation, see the C# PDF generation guide.

What are the known FO.NET issues I should watch for?

Based on historical fork-level reports and gaps in published documentation, common items to verify against your specific fork include: memory consumption in long-running processes without an upstream fix path; thread-safety, which typically requires external synchronization; image format coverage beyond JPEG/PNG (TIFF and BMP can be inconsistent); behavior with merged or nested tables; and SVG rendering. These vary by fork and may not be formally documented — test the patterns your application relies on.

How does installation compare between FO.NET and IronPDF?

For FO.NET, the typical install path is the Fonet NuGet package (v1.0.0, .NET Framework 2.0) or the Fonet.Standard fork (v1.0.5, .NET Standard 2.0). Namespaces are usually using Fonet with using Fonet.Render.Pdf for the PDF renderer. IronPDF installs via dotnet add package IronPdf and exposes using IronPdf;. The package is self-contained on official NuGet. See the IronPDF documentation for setup details.

When should you choose FO.NET vs IronPDF?

FO.NET served a specific need in the .NET ecosystem circa 2006-2012: teams with XML data pipelines who required precise, declarative page layout via XSL-FO standards. The library's dormancy reflects broader industry shifts — HTML/CSS emerged as the dominant layout language, XML processing moved to JSON/REST APIs, and .NET Framework-only libraries became incompatible with cloud-native architectures.

For existing FO.NET codebases, the migration decision hinges on pain points: if you are encountering .NET Core incompatibility, font rendering issues, or need features like digital signatures, the technical debt of maintaining FO.NET outweighs refactoring costs. IronPDF's HTML-based approach means most teams already possess the skills (HTML/CSS) to rebuild layouts, eliminating XSL-FO's learning curve. For more on what IronPDF offers, see the full feature overview.

Migration becomes mandatory when: deployment targets shift to .NET 5+; compliance requires modern encryption or signatures; or vendor-lock to unmaintained code becomes a security audit flag. In these cases, IronPDF's Chromium engine and active development provide a supported path forward.
