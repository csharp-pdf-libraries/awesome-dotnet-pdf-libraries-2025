ZetPDF vs IronPDF in C#: Which Should Your Team Choose?

A developer recently asked me about ZetPDF for a new project. They had seen it referenced in older blog posts and wanted to know if it could handle server-side PDF generation alongside their ASP.NET Core application. ZetPDF does ship a ZetPDF NuGet package with an HtmlToPdfConverter class, but the surrounding marketing material, blog posts, and documentation skew heavily toward Windows desktop viewing — which makes the architectural fit for ASP.NET Core worth examining in detail.

ZetPDF is marketed as a PDF viewer and rendering library for Windows desktop applications — WinForms, WPF, and Silverlight — with a small set of conversion helpers (HtmlToPdfConverter, PdfMerger). IronPDF is a server-side HTML-to-PDF generation engine built on headless Chromium. These libraries address overlapping but fundamentally different use cases, and the comparison matters because teams sometimes evaluate them side by side without realizing the architectural gap.

What is ZetPDF and what does it actually do?

ZetPDF is described as a PDF viewer and rendering library for Windows desktop applications. The marketing-facing functionality is displaying PDFs in UI controls — bookmark navigation, annotations, and attachment management — all oriented around interactive PDF viewing in WinForms/WPF/Silverlight environments. Alongside the viewer, the library exposes a small set of conversion helpers, including an HtmlToPdfConverter with ConvertHtmlToPdf and ConvertUrlToPdf methods, and a PdfMerger for combining files.

One thing worth checking before adoption is ZetPDF's current update cadence. Public-facing blog content dates from 2018 and there is no visible GitHub repository or recent release announcement, so teams should verify the latest package version and supported runtimes directly with the vendor. The NuGet package itself does exist (Install-Package ZetPDF), but the surrounding documentation surface is thin compared to libraries with active developer-relations programs.

How does ZetPDF differ from IronPDF architecturally?

The primary difference is emphasis. ZetPDF's product surface centers on displaying PDFs in desktop UI controls, with conversion helpers attached. IronPDF's product surface centers on generating PDFs from HTML in headless server environments. That emphasis shapes the API ergonomics and the runtime assumptions on each side.

ZetPDF's WinForms/WPF/Silverlight viewer controls are designed for desktop hosts and typically expect STAThread contexts and a Windows message loop. The conversion helpers (HtmlToPdfConverter, PdfMerger) are simple synchronous static-style APIs that write directly to a path; they are usable from server code, but verify behavior on non-Windows runtimes if you plan to deploy to Linux containers.

IronPDF's ChromePdfRenderer class converts HTML to PDF using embedded Chromium — designed for throughput, parallel processing, and stateless operation in ASP.NET Core, Azure Functions, AWS Lambda, or Docker containers. Async/await is supported throughout, with no UI thread or STAThread requirement.

Server-side PDF generation benefits from headless architectures: no UI thread dependencies, stateless operation for horizontal scaling, memory-efficient rendering without view buffers, and support for async patterns without STAThread constraints.

How does HTML-to-PDF conversion compare between ZetPDF and IronPDF?

The two libraries take different shapes here. ZetPDF exposes a thin synchronous API — HtmlToPdfConverter.ConvertHtmlToPdf(html, path) and ConvertUrlToPdf(url, path) — that writes directly to a file. IronPDF exposes a renderer-based API that returns a PdfDocument object you can compose, modify, and stream. The renderer also supports async operation. Detailed engine internals (rendering backend, CSS3/JS support, font loading) are not extensively documented for ZetPDF, so verify against your specific HTML inputs before committing.

ZetPDF: Synchronous HTML-to-PDF

// NuGet: Install-Package ZetPDF
using ZetPDF;
using System;

class Program
{
    static void Main()
    {
        var converter = new HtmlToPdfConverter();
        var htmlContent = "<html><body><h1>Hello World</h1></body></html>";
        converter.ConvertHtmlToPdf(htmlContent, "output.pdf");
        Console.WriteLine("PDF created successfully");
    }
}

The API is intentionally minimal. There is no native watermark, password, or async API surfaced on the converter — verify the current package version's feature set if any of those matter for your use case.

IronPDF: Headless HTML-to-PDF Generation

using IronPdf;
using System;
using System.Diagnostics;
using System.Threading.Tasks;

public class IronPdfGenerator
{
    public async Task<byte[]> GeneratePdfFromHtmlAsync(string html)
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var stopwatch = Stopwatch.StartNew();

        var renderer = new ChromePdfRenderer();

        // Headless Chromium rendering - no UI thread
        using var pdf = await renderer.RenderHtmlAsPdfAsync(html);

        stopwatch.Stop();
        Console.WriteLine($"Generation time: {stopwatch.ElapsedMilliseconds}ms");

        return pdf.BinaryData;
    }
}

// Usage
var generator = new IronPdfGenerator();
string html = @"
<!DOCTYPE html>
<html>
<head>
    <style>
        body { font-family: Arial; padding: 20px; }
        table { width: 100%; border-collapse: collapse; }
        th, td { border: 1px solid #ddd; padding: 8px; }
    </style>
</head>
<body>
    <h1>Performance Test Document</h1>
    <table>
        <tr><th>Metric</th><th>Value</th></tr>
        <tr><td>Throughput</td><td>High</td></tr>
    </table>
</body>
</html>";

byte[] pdf = await generator.GeneratePdfFromHtmlAsync(html);
// Typical: 200-400ms for simple document
// No UI thread, no control overhead

ZetPDF: Viewer Control Approach

Beyond the conversion helpers shown above, ZetPDF's primary surface is a viewer control for WinForms/WPF/Silverlight. That viewer is intended for desktop hosts — STAThread, Windows message loop, GDI-backed rendering buffers — and is a different abstraction from a server-side renderer. If your scenario is a thick-client app where end users open and annotate PDFs, the viewer is the right tool. If it is a web service that emits PDFs from HTML on demand, the conversion helpers (rather than the viewer control) are the relevant entry point, and you should benchmark them against your throughput target.

How do merging and batch processing compare?

ZetPDF exposes a PdfMerger.MergeFiles(paths, output) API that concatenates files synchronously by path. It works, and for low-volume merging it is straightforward. For batch generation under load, the synchronous file-by-file API style means orchestrating concurrency is the caller's problem — there is no built-in async surface and no batched renderer pool.

// NuGet: Install-Package ZetPDF
using ZetPDF;
using System.Collections.Generic;

var merger = new PdfMerger();
var files = new List<string> { "document1.pdf", "document2.pdf" };
merger.MergeFiles(files, "merged.pdf");

IronPDF processes batches in parallel with no UI thread bottleneck and supports async/await directly on the renderer.

IronPDF: Parallel Async Batch Processing

using IronPdf;
using System;
using System.Collections.Generic;
using System.Diagnostics;
using System.Linq;
using System.Threading.Tasks;

public class IronPdfBatchProcessor
{
    public async Task<List<byte[]>> ProcessBatchAsync(List<string> htmlDocuments)
    {
        var stopwatch = Stopwatch.StartNew();

        var renderer = new ChromePdfRenderer();

        // Process in parallel - no UI thread constraints
        var tasks = htmlDocuments.Select(async html =>
        {
            using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
            return pdf.BinaryData;
        });

        var results = (await Task.WhenAll(tasks)).ToList();

        stopwatch.Stop();
        Console.WriteLine($"Batch time: {stopwatch.Elapsed.TotalSeconds:F1}s");

        return results;
    }
}

// Usage
var processor = new IronPdfBatchProcessor();

var htmlDocs = Enumerable.Range(1, 100)
    .Select(i => $@"
        <!DOCTYPE html>
        <html><body>
            <h1>Document #{i}</h1>
            <p>Content for document {i}</p>
        </body></html>")
    .ToList();

var stopwatch = Stopwatch.StartNew();
var pdfs = await processor.ProcessBatchAsync(htmlDocs);
stopwatch.Stop();

Console.WriteLine($"Processed {pdfs.Count} PDFs in {stopwatch.Elapsed.TotalSeconds:F1}s");
Console.WriteLine($"Throughput: {pdfs.Count / stopwatch.Elapsed.TotalSeconds:F1} PDFs/sec");
// Typical: 5-10 seconds for 100 PDFs on multi-core system
// Throughput: 10-20 PDFs/sec depending on content complexity

IronPDF's parallel processing scales with CPU cores. See PDF generation settings for performance tuning options.

IronPDF: Server Deployment in ASP.NET Core

using Microsoft.AspNetCore.Mvc;
using IronPdf;
using System.Threading.Tasks;

[ApiController]
[Route("api/[controller]")]
public class PdfController : ControllerBase
{
    private readonly ChromePdfRenderer _renderer;

    public PdfController()
    {
        // Renderer is thread-safe and reusable
        _renderer = new ChromePdfRenderer();
    }

    [HttpPost]
    public async Task<IActionResult> GeneratePdf([FromBody] string html)
    {
        // No UI thread dependencies - works in ASP.NET Core
        using var pdf = await _renderer.RenderHtmlAsPdfAsync(html);

        return File(pdf.BinaryData, "application/pdf", "document.pdf");
    }
}

// Deployment benefits:
// - No STAThread requirements
// - Thread-safe concurrent requests
// - Horizontal scaling supported (multiple instances)
// - Azure/AWS/Docker compatible
// - Stateless operation
// - Memory-efficient (automatic cleanup)

IronPDF is designed for server environments from day one. No UI dependencies, supports containerization, scales horizontally.

How do the two libraries compare feature by feature?

The bullets below capture the dimensions worth evaluating. Several ZetPDF entries are framed as "verify" because the public documentation surface does not cover them in depth.

  • Primary use case. ZetPDF — PDF viewing in WinForms/WPF/Silverlight desktop applications, with conversion helpers attached. IronPDF — server-side PDF generation from HTML.
  • Update cadence and documentation. ZetPDF's public-facing blog content dates from 2018; the NuGet package exists but the surrounding docs and release notes are thin — verify current package version with the vendor. IronPDF ships regular updates with comprehensive documentation.
  • HTML-to-PDF API. ZetPDF — HtmlToPdfConverter.ConvertHtmlToPdf(html, path), synchronous, file-path output. IronPDF — ChromePdfRenderer.RenderHtmlAsPdf / RenderHtmlAsPdfAsync, returns a composable PdfDocument.
  • URL-to-PDF. Both supported — ConvertUrlToPdf(url, path) on ZetPDF, RenderUrlAsPdf on IronPDF.
  • Merge PDFs. Both supported — PdfMerger.MergeFiles(paths, output) on ZetPDF, PdfDocument.Merge(...) on IronPDF.
  • PDF viewing controls. ZetPDF offers WinForms/WPF/Silverlight viewer controls. IronPDF is generation-only and ships no viewer UI.
  • Async/await. Not surfaced on ZetPDF's converter API. Native on IronPDF.
  • Headless server operation. IronPDF is designed for headless server use (ASP.NET Core, Azure Functions, AWS Lambda, Docker). ZetPDF's viewer surface is desktop-oriented; verify the converter helpers' behavior on Linux/container hosts before depending on them.
  • Watermarks. No native watermark API documented on ZetPDF's converter. IronPDF — pdf.ApplyWatermark(html) with HTML/CSS.
  • Password / encryption. No native password API documented on ZetPDF's converter. IronPDF — pdf.SecuritySettings.UserPassword / OwnerPassword.
  • Text extraction, digital signatures, PDF/A. Verify in current ZetPDF docs. Supported on IronPDF.
  • Annotations and bookmarks. ZetPDF supports them as viewer features. IronPDF supports them as creation features.
  • Learning curve. ZetPDF's API is small and synchronous — easy to call, easy to outgrow. IronPDF — low curve if HTML/CSS is already familiar.

Frequently Asked Questions

Is ZetPDF still actively maintained?

Worth checking directly with the vendor. The public-facing blog content dates from 2018 and there is no visible GitHub repository or recent release announcement, but the ZetPDF NuGet package itself does exist. Confirm the latest published version, target frameworks, and support channels before adopting it for a new project.

Can ZetPDF generate PDFs from HTML on a server?

The library exposes an HtmlToPdfConverter with ConvertHtmlToPdf and ConvertUrlToPdf methods, so HTML-to-PDF is part of the API. The broader product surface, however, is oriented around WinForms/WPF/Silverlight viewer controls, and the converter's documented feature set is small (no native async, watermark, or password APIs). Verify rendering behavior — CSS3 fidelity, font handling, JavaScript execution, Linux/container support — against your specific HTML inputs before committing.

How does IronPDF handle installation compared to ZetPDF?

Both install via NuGet: Install-Package IronPdf and Install-Package ZetPDF. IronPDF has no UI dependencies and works out of the box in headless server environments. ZetPDF's package is published, but the surrounding documentation, samples, and release notes are thinner — confirm the latest version against the vendor before depending on it.

Can I use ZetPDF in ASP.NET Core or Docker containers?

ZetPDF's viewer controls (WinForms/WPF/Silverlight) carry desktop runtime assumptions — STAThread contexts and Windows message loops — that do not map cleanly onto ASP.NET Core's thread pool model or Linux containers. The conversion helpers are simpler and may work, but verify on your target runtime. IronPDF is designed for these environments — ASP.NET Core, Azure Functions, AWS Lambda, and Docker containers all work out of the box.

What is the performance difference for batch PDF generation?

IronPDF supports async/await directly on the renderer and parallel processing scales with available CPU cores. ZetPDF's converter API is synchronous and offers no built-in batch or async surface, so concurrency is the caller's responsibility. For high-throughput batch generation, benchmark both against representative inputs before choosing.

Does migrating from ZetPDF to IronPDF make sense?

Migration makes sense when the requirement shifts from viewing to generation, when server-side PDF creation becomes necessary, when a desktop application is replaced by a web application, when batch processing performance becomes critical, or when cloud deployment and containerization are required. The ChromePdfRenderer API is designed for stateless operation across all these scenarios.

Which should your team choose — ZetPDF or IronPDF?

ZetPDF and IronPDF address overlapping but distinct use cases. ZetPDF is marketed primarily as a PDF viewer library for Windows desktop applications, with a small set of conversion helpers attached. IronPDF is a server-side HTML-to-PDF generation engine, designed for headless operation in web applications and cloud services.

If your team needs to build a desktop application where users navigate, annotate, or interact with existing PDFs, a viewer library is the right category — verify ZetPDF's current package version, supported runtimes, and support channels with the vendor before adopting it, since the public-facing material is dated.

If your team needs server-side PDF generation, batch processing, HTML-to-PDF conversion, or cloud deployment, IronPDF is the clear choice. The headless Chromium architecture eliminates UI thread dependencies, supports async/await patterns, and scales horizontally. Install via Install-Package IronPdf and start converting HTML to PDF in minutes. The performance implications stem from architectural differences: desktop viewer controls introduce UI thread dependencies, STAThread requirements, and serial processing constraints that headless generation engines eliminate entirely.
