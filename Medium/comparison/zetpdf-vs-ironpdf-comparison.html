ZetPDF vs IronPDF in C#: Which Should Your Team Choose?

One architectural contrast frames this whole comparison: ZetPDF is a coordinate-based programmatic PDF toolkit in the PDFsharp lineage, while IronPDF is a headless Chromium HTML-to-PDF engine. The two libraries can both produce a PDF file, but the route they take is so different that picking the wrong one usually shows up as weeks of rework rather than a missing feature.

ZetPDF (zetpdf.com) is marketed as a proprietary, 100%-managed-code .NET PDF SDK whose API closely resembles PDFsharp — to the point that the PDFsharp community has questioned whether it is a closed fork. IronPDF is a server-side HTML-to-PDF generation engine built on embedded Chromium. These libraries address fundamentally different use cases, and the comparison matters because teams sometimes evaluate them side by side without realizing the architectural gap.

What is ZetPDF and what does it actually do?

ZetPDF is a commercial .NET PDF SDK focused on programmatic PDF construction and manipulation — creating documents page by page, drawing text and shapes via an XGraphics-style API, handling annotations and form fields, and exposing AES256 encryption settings on the document. The published feature surface (per zetpdf.com/net-pdf-sdk/) covers PDF view/print, annotations, AES256 encryption, form fields, and text extraction. There is no documented HTML-to-PDF or URL-to-PDF API in that surface.

One thing worth checking before adoption is ZetPDF's distribution and update cadence. ZetPDF is not distributed via NuGet — installation is a ZIP download from https://zetpdf.com/download/, and the ZetPDF.dll is referenced from the project manually. AlternativeTo's listing shows no public release activity since 2021, and there is no visible GitHub repository or recent release announcement. Verify the latest package version and supported runtimes directly with the vendor.

How does ZetPDF differ from IronPDF architecturally?

The primary difference is the rendering model. ZetPDF builds PDFs the way PDFsharp does: you create a PdfDocument, add pages, get an XGraphics off each page, and draw text, lines, rectangles, and images at explicit coordinates. The layout is whatever you compute. IronPDF builds PDFs from HTML/CSS, with Chromium handling text flow, page breaks, tables, images, fonts, and CSS resolution.

That difference cascades. Coordinate-based APIs need manual page-break detection, manual text measurement for wrapping, manual tabular layout, and per-element font and color management. HTML-based APIs delegate all of that to a layout engine. For a fixed-format document where you control every coordinate (a label, a receipt, a tax form), the programmatic approach is fine and arguably preferable. For a document whose shape varies with content — invoices with N line items, reports with embedded charts, anything that started as a designed web page — the HTML approach is dramatically less code.

IronPDF's ChromePdfRenderer class converts HTML to PDF using embedded Chromium and is designed for throughput, parallel processing, and stateless operation in ASP.NET Core, Azure Functions, AWS Lambda, or Docker containers. Async/await is supported throughout, with no UI thread or STAThread requirement.

How does HTML-to-PDF conversion compare between ZetPDF and IronPDF?

This is the cleanest line in the comparison: ZetPDF does not natively render HTML to PDF. The documented feature list does not include HTML-to-PDF or URL-to-PDF, and the API surface is built around XGraphics drawing rather than markup parsing. Practical migration paths usually pair ZetPDF with an external HTML renderer or move to a Chromium-based library.

IronPDF's HTML-to-PDF entry point is a renderer-based API that returns a composable PdfDocument and supports async operation:

IronPDF: Headless HTML-to-PDF Generation

// NuGet: Install-Package IronPdf
using IronPdf;
using System;
using System.Diagnostics;
using System.Threading.Tasks;

public class IronPdfGenerator
{
    public async Task<byte[]> GeneratePdfFromHtmlAsync(string html)
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var stopwatch = Stopwatch.StartNew();

        var renderer = new ChromePdfRenderer();

        // Headless Chromium rendering - no UI thread
        using var pdf = await renderer.RenderHtmlAsPdfAsync(html);

        stopwatch.Stop();
        Console.WriteLine($"Generation time: {stopwatch.ElapsedMilliseconds}ms");

        return pdf.BinaryData;
    }
}

// Usage
var generator = new IronPdfGenerator();
string html = @"
<!DOCTYPE html>
<html>
<head>
    <style>
        body { font-family: Arial; padding: 20px; }
        table { width: 100%; border-collapse: collapse; }
        th, td { border: 1px solid #ddd; padding: 8px; }
    </style>
</head>
<body>
    <h1>Performance Test Document</h1>
    <table>
        <tr><th>Metric</th><th>Value</th></tr>
        <tr><td>Throughput</td><td>High</td></tr>
    </table>
</body>
</html>";

byte[] pdf = await generator.GeneratePdfFromHtmlAsync(html);
// Typical: 200-400ms for simple document on modern hardware

ZetPDF: Coordinate-Based Programmatic PDF

By contrast, building a PDF in ZetPDF means positioning every element explicitly via XGraphics. The shape of the code is a long sequence of DrawString / DrawRectangle / DrawImage calls with calculated x/y coordinates:

// ZetPDF is NOT on NuGet — install via SDK ZIP from https://zetpdf.com/download/
using ZetPDF;
using ZetPDF.Drawing;

var document = new PdfDocument();
var page = document.AddPage();
var graphics = XGraphics.FromPdfPage(page);

var titleFont = new XFont("Arial", 24, XFontStyle.Bold);
var bodyFont = new XFont("Arial", 12);

graphics.DrawString("Company Report", titleFont, XBrushes.Navy, new XPoint(50, 50));
graphics.DrawString("This is the introduction paragraph.", bodyFont, XBrushes.Black, new XPoint(50, 80));
graphics.DrawString("Generated: " + DateTime.Now, bodyFont, XBrushes.Gray, new XPoint(50, 100));

document.Save("report.pdf");

For a fixed-format document this is direct and predictable. For anything where the content drives the layout — N-row tables, multi-page reports, designed templates — manual coordinate management is where most of the engineering effort goes.

How do merging and batch processing compare?

ZetPDF does not ship a one-line PdfMerger.MergeFiles() helper. Merging follows the PDFsharp pattern: open each input with PdfReader, loop pages into an output PdfDocument, and call Save(). It works, and for low-volume merging it is straightforward, but the synchronous file-by-file style means orchestrating concurrency is the caller's problem — there is no built-in async surface and no batched renderer pool.

IronPDF processes batches in parallel with no UI thread bottleneck and supports async/await directly on the renderer.

IronPDF: Parallel Async Batch Processing

// NuGet: Install-Package IronPdf
using IronPdf;
using System;
using System.Collections.Generic;
using System.Diagnostics;
using System.Linq;
using System.Threading.Tasks;

public class IronPdfBatchProcessor
{
    public async Task<List<byte[]>> ProcessBatchAsync(List<string> htmlDocuments)
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var stopwatch = Stopwatch.StartNew();

        var renderer = new ChromePdfRenderer();

        // Process in parallel - no UI thread constraints
        var tasks = htmlDocuments.Select(async html =>
        {
            using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
            return pdf.BinaryData;
        });

        var results = (await Task.WhenAll(tasks)).ToList();

        stopwatch.Stop();
        Console.WriteLine($"Batch time: {stopwatch.Elapsed.TotalSeconds:F1}s");

        return results;
    }
}

// Usage
var processor = new IronPdfBatchProcessor();

var htmlDocs = Enumerable.Range(1, 100)
    .Select(i => $@"
        <!DOCTYPE html>
        <html><body>
            <h1>Document #{i}</h1>
            <p>Content for document {i}</p>
        </body></html>")
    .ToList();

var sw = Stopwatch.StartNew();
var pdfs = await processor.ProcessBatchAsync(htmlDocs);
sw.Stop();

Console.WriteLine($"Processed {pdfs.Count} PDFs in {sw.Elapsed.TotalSeconds:F1}s");
Console.WriteLine($"Throughput: {pdfs.Count / sw.Elapsed.TotalSeconds:F1} PDFs/sec");
// Throughput scales with available CPU cores

IronPDF's parallel processing scales with CPU cores. See PDF generation settings for performance tuning options.

IronPDF: Server Deployment in ASP.NET Core

// NuGet: Install-Package IronPdf
using Microsoft.AspNetCore.Mvc;
using IronPdf;
using System.Threading.Tasks;

[ApiController]
[Route("api/[controller]")]
public class PdfController : ControllerBase
{
    private readonly ChromePdfRenderer _renderer;

    public PdfController()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        // Renderer is reusable across requests
        _renderer = new ChromePdfRenderer();
    }

    [HttpPost]
    public async Task<IActionResult> GeneratePdf([FromBody] string html)
    {
        // No UI thread dependencies - works in ASP.NET Core
        using var pdf = await _renderer.RenderHtmlAsPdfAsync(html);

        return File(pdf.BinaryData, "application/pdf", "document.pdf");
    }
}

// Deployment characteristics:
// - No STAThread requirements
// - Concurrent requests handled by the host's thread pool
// - Horizontal scaling supported (multiple instances)
// - Azure / AWS / Docker compatible
// - Stateless operation
// - Memory-efficient (automatic cleanup with using/Dispose)

IronPDF is designed for server environments from day one. No UI dependencies, supports containerization, scales horizontally.

How do the two libraries compare feature by feature?

The bullets below capture the dimensions worth evaluating. Several ZetPDF entries are framed as "verify with vendor" because the public documentation surface does not cover them in depth.

  • Primary use case. ZetPDF — programmatic PDF construction and manipulation via a coordinate-based XGraphics API in the PDFsharp lineage. IronPDF — server-side PDF generation from HTML.
  • Distribution. ZetPDF — ZIP download from zetpdf.com/download/; the ZetPDF.dll is referenced manually. IronPDF — NuGet (Install-Package IronPdf).
  • Update cadence and documentation. ZetPDF shows no public release activity since 2021 (per AlternativeTo's listing) and has no visible GitHub repository — verify current build with the vendor. IronPDF ships regular updates with comprehensive documentation.
  • HTML-to-PDF. Not part of ZetPDF's documented feature list; the API is coordinate-based drawing. IronPDF — ChromePdfRenderer.RenderHtmlAsPdf / RenderHtmlAsPdfAsync, returning a composable PdfDocument.
  • URL-to-PDF. Not part of ZetPDF's documented feature list. IronPDF — RenderUrlAsPdf / RenderUrlAsPdfAsync.
  • Merge PDFs. ZetPDF — manual loop using PdfReader.Open and copying pages, in the PDFsharp pattern. IronPDF — one-line PdfDocument.Merge(...).
  • Page layout model. ZetPDF — manual: caller calculates coordinates, page breaks, and text wrapping. IronPDF — automatic via Chromium's layout engine.
  • Async/await. Not surfaced on ZetPDF's API. Native on IronPDF.
  • Headless server operation. ZetPDF is 100%-managed-code per the vendor, so server use is plausible, but verify the converter/encryption helpers' behavior on Linux/container hosts before depending on them. IronPDF is designed for headless server use (ASP.NET Core, Azure Functions, AWS Lambda, Docker).
  • Watermarks. Manual via XGraphics drawing on each page. IronPDF — pdf.ApplyWatermark(html) with HTML/CSS.
  • Encryption / password. ZetPDF exposes AES256 encryption settings on the document. IronPDF — pdf.SecuritySettings.UserPassword / SecuritySettings.OwnerPassword.
  • Annotations, form fields, text extraction. Listed on ZetPDF's published feature surface; verify depth against your specific needs. Supported on IronPDF.
  • Digital signatures, PDF/A. Verify in current ZetPDF docs. Supported on IronPDF.
  • Learning curve. ZetPDF — comfortable if you already know PDFsharp; the cost is manual layout work. IronPDF — low curve if HTML/CSS is already familiar.

Frequently Asked Questions

Is ZetPDF still actively maintained?

Worth checking directly with the vendor. AlternativeTo's listing shows no public release activity since 2021, there is no visible GitHub repository, and the public-facing blog content is dated. Confirm the latest published version, target frameworks, and support channels before adopting it for a new project.

Can ZetPDF generate PDFs from HTML?

HTML-to-PDF and URL-to-PDF are not part of ZetPDF's documented feature list. The API is built around programmatic PDF construction via an XGraphics-style drawing surface, not markup parsing. Teams that need HTML rendering typically pair ZetPDF with an external renderer or move to a Chromium-based library.

How does IronPDF handle installation compared to ZetPDF?

IronPDF installs from NuGet with Install-Package IronPdf and works out of the box in headless server environments. ZetPDF is not on NuGet — installation is a ZIP download from https://zetpdf.com/download/, with the ZetPDF.dll added as a manual reference in the project file.

Can I use ZetPDF in ASP.NET Core or Docker containers?

ZetPDF markets itself as 100%-managed-code, so server use is plausible in principle, but the public documentation does not focus on Linux containers or ASP.NET Core scenarios — verify your specific runtime and feature set against the vendor before depending on it. IronPDF is designed for these environments — ASP.NET Core, Azure Functions, AWS Lambda, and Docker containers all work out of the box.

What is the performance difference for batch PDF generation?

IronPDF supports async/await directly on the renderer and parallel processing scales with available CPU cores. ZetPDF's API is synchronous and offers no built-in batch or async surface, so concurrency is the caller's responsibility. For high-throughput batch generation, benchmark both against representative inputs before choosing.

Does migrating from ZetPDF to IronPDF make sense?

Migration makes sense when the requirement shifts from coordinate-based drawing to HTML-driven layout, when server-side PDF creation becomes critical, when batch processing performance matters, or when cloud deployment and containerization are required. The ChromePdfRenderer API is designed for stateless operation across all these scenarios. A full step-by-step path is documented in the ZetPDF to IronPDF migration guide.

Which should your team choose — ZetPDF or IronPDF?

ZetPDF and IronPDF address different rendering models. ZetPDF is a programmatic PDF SDK in the PDFsharp lineage — coordinate-based drawing, AES256 encryption, annotations, form fields, text extraction — distributed as a ZIP rather than a NuGet package, with no documented HTML-to-PDF surface. IronPDF is a server-side HTML-to-PDF generation engine, designed for headless operation in web applications and cloud services.

If your team needs to build PDFs by drawing fixed-format documents at known coordinates — labels, receipts, forms with stable layout — and you are comfortable with a PDFsharp-shaped API, ZetPDF is the right category. Verify the current build, supported runtimes, and support channels with the vendor before adopting it, since the public-facing material is dated.

If your team needs server-side PDF generation, HTML-to-PDF conversion, batch processing, or cloud deployment, IronPDF is the clear choice. The headless Chromium architecture eliminates UI thread dependencies, supports async/await patterns, and scales horizontally. Install via Install-Package IronPdf and start converting HTML to PDF in minutes. The architectural difference — markup-driven layout versus manual coordinate management — is what shapes the engineering effort on each side, more than any single feature checkbox.
