Nutrient vs IronPDF: When a Comprehensive PDF SDK Is More Than You Need

A common scenario: a supply chain team needs to generate shipping manifests from HTML templates and let warehouse managers annotate them on tablets. The team picks Nutrient because the marketing surface shows both PDF generation and annotation in one product family. Two sprints in, a familiar shape emerges — a broad imaging-and-PDF toolkit doing the work of a focused HTML-to-PDF library, with the HTML pipeline routing through a file-based loader and a host-installed browser, and an annotation surface the backend batch process would never touch.

This is not a complaint about Nutrient's quality. It is a scope and product-line story. Nutrient (formerly PSPDFKit; rebranded on October 23, 2024) is a multi-product family. Its server-side .NET SDK is GdPicture.NET — the package Nutrient publishes as GdPicture on NuGet under the GdPicture14 namespace, after acquiring ORPALIS. Teams searching for ".NET PDF library" expecting an HTML-first generator can land on a broader imaging-plus-PDF toolkit. The troubleshooting scenarios below walk through where the mismatch shows up and where each tool actually fits.

What is Nutrient and where does it excel?

Nutrient.io (formerly PSPDFKit) excels at building interactive PDF viewers — think Dropbox's document preview pane or DocuSign's signing interface. Its annotation tools, form-filling UI, and collaboration features target enterprise document workflows.

The product is actively maintained and was rebranded from PSPDFKit on October 23, 2024. It provides cross-platform viewer SDKs (Web, iOS, Android, desktop) with consistent APIs. For teams building document-centric applications where users annotate, sign, and collaborate on PDFs, Nutrient's comprehensive SDK justifies its scope.

Where it gets complicated is when teams use Nutrient primarily for backend HTML-to-PDF generation. The server-side .NET offering is GdPicture.NET — Nutrient acquired ORPALIS and folded GdPicture in as their .NET line. It ships as the GdPicture NuGet package (latest 14.x) under the GdPicture14 namespace, with GdPicturePDF and GdPictureDocumentConverter as the primary classes. HTML-to-PDF is supported via the document converter, but the broader product surface covers imaging, scanning, OCR, and barcode work alongside PDF, and HTML rendering relies on a host-installed Chrome or Edge (or a portable path set via SetWebBrowserPath).

Problem 1 — What happens when HTML content doesn't render correctly?

This is a common pain point when an imaging-and-PDF toolkit gets used as an HTML-to-PDF generator. You pass an HTML invoice template to GdPictureDocumentConverter expecting a clean render, but CSS Grid layouts may collapse, web fonts may not load, or the result may not match your browser preview.

The root cause: GdPicture's HTML path delegates to a host-installed Chrome or Edge — the supported CSS surface depends on that browser version, and the loader takes a file or URL rather than an HTML string. If you do not have a browser on the host, you must point at one explicitly via converter.SetWebBrowserPath(...). If you are constructing PDFs programmatically with GdPicturePDF instead, you are working with a document object model and primitive draw calls rather than direct HTML rendering.

Diagnostic steps that tend to help: confirm whether the host has a recent Chrome or Edge installed and whether SetWebBrowserPath is pointing where you expect. Check that the converter is loading the HTML with DocumentFormat.DocumentFormatHTML. Test with minimal HTML first — <html><body>Test</body></html> — before adding CSS.

The GdPicture workaround for unsupported markup typically means writing the HTML to a temp file, normalizing layout to a more conservative CSS subset, or falling back to GdPicturePDF's drawing primitives — effectively a rewrite of your template logic against the document model.

IronPDF solves this differently. Because it embeds a Chromium Blink engine — the same renderer as Chrome browser — if your HTML displays correctly in Chrome print preview, IronPDF will match it exactly. No separate conversion layer or document model. For a deeper walkthrough, see the HTML file to PDF tutorial.

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// Direct HTML to PDF conversion
var renderer = new ChromePdfRenderer();

// CSS Grid and Flexbox work exactly as in Chrome

    <style>
        .invoice {
            display: grid;
            grid-template-columns: 2fr 1fr;
            gap: 20px;
        }
    </style>
    <div class='invoice'>
        <div>Left column content</div>
        <div>Right column content</div>
    </div>";

var pdf = renderer.RenderHtmlAsPdf(htmlContent);
pdf.SaveAs("invoice.pdf");

Problem 2 — Why does PDF generation require deploying a separate microservice?

This question typically arises when teams evaluate Nutrient's full product surface rather than just the .NET SDK. Nutrient ships a server-side conversion product (Document Engine, a Docker-deployed service) alongside the in-process GdPicture .NET SDK. Some HTML-to-PDF and Office-to-PDF flows are documented through the server product, which is where the "do I need a microservice?" question comes from.

The in-process SDK does expose HTML-to-PDF via GdPictureDocumentConverter, so a separate service is not strictly required for HTML-to-PDF — provided you accept the host-browser dependency for HTML rendering. The decision often comes down to whether your conversion needs (Office formats, large-scale rendering isolation, language-agnostic clients) push you toward Document Engine.

Before deciding, confirm what your Nutrient plan includes, check whether the in-process SDK covers your HTML-to-PDF needs, and — if you do go server-side — estimate the operational overhead of containers, monitoring, and the network round-trip latency.

IronPDF takes the in-process route by default. The entire conversion happens in-process — no external services, no microservices, no containers required. Installation is a single NuGet package, with no network calls or service dependencies, and the embedded Chromium ships with the package rather than relying on the host's Chrome or Edge.

IronPDF: Single package, no external services

using IronPdf;

// Installation: dotnet add package IronPdf
// No external services, no microservices, no containers required
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();

// Entire conversion happens in-process
var pdf = renderer.RenderHtmlAsPdf("<html><body>Content</body></html>");
pdf.SaveAs("output.pdf");

Problem 3 — How do you handle batch processing without memory blowup?

Batch generation is where the scope mismatch tends to hit hardest. A team batch-generating 500 PDFs from HTML templates may see memory climb steadily as GdPictureDocumentConverter and GdPicturePDF instances accumulate, particularly if they aren't disposed eagerly. HTML-path runs also pay a per-conversion cost for the host browser process.

The Nutrient .NET SDK (GdPicture) is broad — covering imaging, scanning, OCR, barcode, and PDF — and its API is largely synchronous and procedural. Both GdPicturePDF and GdPictureDocumentConverter implement IDisposable, so deterministic disposal via using is the practical pattern for keeping memory bounded.

Diagnostic path: profile memory to see which assemblies are loaded, confirm GdPicturePDF and GdPictureDocumentConverter instances are disposed after each generation, and verify which GdPicture features you actually pull in. Common workarounds are processing batches in smaller chunks, recycling the app domain between chunks, or scaling horizontally.

IronPDF is designed for batch scenarios. Chromium engine resources are pooled and reused across conversions. Proper disposal patterns prevent memory accumulation, and the thread-safe architecture supports parallel execution.

IronPDF: Parallel batch processing with disposal

using IronPdf;
using System.IO;
using System.Linq;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// Parallel processing with memory-efficient design
var renderer = new ChromePdfRenderer();
var htmlFiles = Directory.GetFiles("templates/", "*.html");

Parallel.ForEach(htmlFiles, htmlFile =>
{
    using var pdf = renderer.RenderHtmlFileAsPdf(htmlFile);
    pdf.SaveAs(Path.ChangeExtension(htmlFile, ".pdf"));
    // Explicit disposal releases Chromium resources immediately
});

What about licensing — does the pricing model fit backend generation?

Consider a workload generating 50,000 PDFs per month from a batch job running on three backend servers. Nutrient's pricing is sales-led — the nutrient.io/sdk/pricing page routes to a Contact Us flow rather than published rates, with quotes typically structured per-developer plus a deployment tier. Verify the specific terms with Nutrient for your scenario.

For pure backend generation, the licensing math depends on what comes back from that sales conversation. Some vendors offer document-transaction licensing for backend scenarios; whether Nutrient does is a question for their sales team.

IronPDF licensing is based on developers and deployment servers, not documents generated or end-user counts. A license covers unlimited PDF generation volume. You pay for developers writing code and servers running it, not for volume generated. See IronPDF Licensing Options for the full breakdown.

Can IronPDF handle modern CSS that Nutrient's HTML path struggles with?

This comes up when teams are using CSS Grid for layout and custom CSS properties (variables). Templates can look correct in the developer's browser but render with collapsed layouts or missing styles when converted via GdPicture's HTML-to-PDF path.

The root cause requires verification against your environment: GdPicture renders HTML by delegating to a host-installed Chrome or Edge, so the supported CSS surface depends on the version of that browser on each host — and on whether SetWebBrowserPath is pointing where you expect. Common workarounds are pinning the host browser version across environments, testing against the actual installed engine, or normalizing templates to a more conservative CSS subset.

IronPDF's Chromium Blink renderer supports CSS Grid, Flexbox, CSS Variables, Subgrid, Container Queries, and other modern features. If it works in current Chrome, it works in IronPDF. For the complete rendering options reference, see the JavaScript execution in PDFs guide.

How do the features compare?

  • Primary focus. Nutrient targets viewer and editor SDKs across Web, mobile, and desktop, with GdPicture.NET as its server-side .NET SDK. IronPDF targets HTML-to-PDF generation and PDF manipulation in .NET. Both are actively developed.
  • Documentation surface. Nutrient's docs are weighted toward viewer integration and interactive scenarios; GdPicture's docs cover imaging, OCR, barcode, scanning, and PDF; IronPDF's docs are weighted toward generation, rendering options, and PDF manipulation. SLA-backed support is available from both at enterprise tiers.
  • HTML-to-PDF. Nutrient's .NET SDK exposes HTML conversion via GdPictureDocumentConverter.LoadFromFile/LoadFromHttp followed by SaveAsPDF; rendering delegates to a host-installed Chrome or Edge (or a portable path set via SetWebBrowserPath), so CSS support tracks that browser. IronPDF renders via embedded Chromium Blink, so CSS Grid, Flexbox, CSS Variables, modern JavaScript, and WOFF2 fonts behave as they do in Chrome — with no system browser dependency.
  • Headers, footers, watermarks, page numbering. GdPicture composes these from primitive draw calls (SetFillAlpha, DrawTextBox, AddStandardFont, per-page loops). IronPDF exposes HtmlHeader/HtmlFooter, ApplyWatermark, and {page}/{total-pages} placeholders for HTML/CSS-based headers and footers.
  • PDF operations. Both cover merge (GdPictureDocumentConverter.CombineToPDF vs PdfDocument.Merge), split, text and image extraction, form filling, digital signatures, encryption, and password protection.
  • Interactive features. Nutrient includes annotation UI, form-filling UI, viewer components, and collaboration in its viewer SDKs. IronPDF does not include UI components — it is a server-side generation and manipulation library.
  • Specialized features. GdPicture bundles OCR, barcode reading, TWAIN scanning, and broader image processing. IronPDF keeps OCR and barcode in separate Iron Software libraries (IronOCR, IronBarcode); image processing is more basic and does not cover scanning.
  • Development model. GdPicture's .NET API is largely synchronous and procedural, with IDisposable on the main types. IronPDF offers both sync and async, is fully thread-safe, and is designed for parallel batch processing.
  • Page indexing. GdPicture is 1-based (SelectPage(1) targets page one). IronPDF's pdf.Pages and RemovePages(int) are 0-based — off-by-one is the most common porting bug.
  • Deployment. Both run cross-platform on Windows, Linux, and macOS, and in containers. Nutrient also offers Document Engine, a Docker-deployed server-side product for some conversion flows; IronPDF is in-process only.
  • .NET targets. Both support .NET Framework and modern .NET (.NET 6/7/8/10 on the GdPicture side). Verify the exact supported range against each vendor's release notes for your version.
  • Pricing. Nutrient is sales-led — nutrient.io/sdk/pricing routes to Contact Us. IronPDF publishes per-developer pricing on its site.

How do the concepts map between Nutrient and IronPDF?

  • GdPictureDocumentConverter.LoadFromFile(htmlPath, DocumentFormat.DocumentFormatHTML) + SaveAsPDF — maps to ChromePdfRenderer.RenderHtmlAsPdf(html) in IronPDF (string input directly, no temp file).
  • GdPictureDocumentConverter.LoadFromHttp(url) + SaveAsPDF — maps to ChromePdfRenderer.RenderUrlAsPdf(url).
  • GdPicturePDF document model with primitive draw calls — IronPDF uses direct HTML rendering with PdfDocument, no per-page drawing loop.
  • GdPictureDocumentConverter.CombineToPDF(paths, out, conformance) — maps to PdfDocument.Merge(pdfs).
  • SetFillAlpha + DrawTextBox + AddStandardFont per page — maps to a single pdf.ApplyWatermark(html, rotation, vAlign, hAlign) call.
  • Per-page DrawText for headers/footers — maps to RenderingOptions.HtmlHeader/HtmlFooter with {page}/{total-pages} placeholders.
  • Index-based form access (GetFormFieldsCount/GetFormFieldName/SetFormFieldValue) — maps to pdf.Form.FindFormField(name).Value.
  • Encryption via the SaveToFile overload with PdfEncryption.PdfEncryptionAES256 — maps to pdf.SecuritySettings.UserPassword/OwnerPassword plus AllowUserPrinting/AllowUserEdits.
  • pdf.GetPageText() in a per-page loop — maps to pdf.ExtractAllText().
  • Page manipulation (SelectPage + DeletePage, RotatePage) — maps to pdf.RemovePages(index), pdf.Pages[i].Rotation, and pdf.CopyPages(...). Note 1-based vs 0-based indexing.
  • Annotation and viewer UI components — not applicable in IronPDF (no UI surface).
  • OCR (GdPicture built-in) — not included in IronPDF; IronOCR is a separate Iron Software library.
  • Barcode reading (GdPicture built-in) — not included in IronPDF; IronBarcode is a separate library.
  • Redaction — PdfDocument.RedactTextByRegex() in IronPDF.

FAQ and Troubleshooting Tips

Can Nutrient generate PDFs from HTML without the Document Engine microservice?

Yes. The in-process .NET SDK (GdPicture) supports HTML-to-PDF via GdPictureDocumentConverter — typically LoadFromFile(htmlPath, DocumentFormat.DocumentFormatHTML) or LoadFromHttp(url) followed by SaveAsPDF. There is one practical caveat: HTML rendering delegates to a host-installed Chrome or Edge, so each deployment host needs a browser available (or an explicit path set via SetWebBrowserPath). With IronPDF, a single call to ChromePdfRenderer.RenderHtmlAsPdf() handles the entire conversion in-process using embedded Chromium, with no system browser dependency.

Why does my CSS Grid layout break in GdPicture but work in my browser?

GdPicture's HTML-to-PDF path renders through whatever Chrome or Edge is installed on the host — the supported CSS surface tracks that version, and a stale or missing browser will silently change the output. IronPDF uses an embedded Chromium Blink engine, so CSS Grid, Flexbox, and CSS Variables render identically to your browser preview without depending on host configuration.

How do I extract text and images from existing PDFs for archival workflows?

Both Nutrient (GdPicture) and IronPDF support text and image extraction. In IronPDF, use PdfDocument.ExtractAllText() for text and the corresponding methods for images. For a complete walkthrough, see the text and image extraction guide. If your archival workflow requires PDF/A compliance, IronPDF also supports that — see the PDF/A compliance guide.

Is IronPDF thread-safe for parallel batch processing?

Yes. IronPDF is fully thread-safe and designed for parallel execution. Chromium engine resources are pooled and reused across conversions. Use Parallel.ForEach with proper disposal patterns (the using statement) to process large batches without memory accumulation. GdPicture's main types (GdPicturePDF, GdPictureDocumentConverter) implement IDisposable, so deterministic disposal with using is also the recommended pattern there at scale.

What if I need both PDF generation and an interactive viewer?

This is where the decision gets nuanced. Nutrient provides both capabilities in one comprehensive SDK. However, evaluate whether combining IronPDF for generation with a simpler viewer library (PDF.js, browser native) costs less and integrates easier. If your viewer needs are basic (display and scroll), a lightweight option may suffice. If you need full annotation toolbars, form-filling interfaces, and real-time collaboration, Nutrient's viewer components are genuinely strong.

When should you use Nutrient vs IronPDF?

Use Nutrient when you're building interactive PDF experiences: document viewers with annotation toolbars, form-filling interfaces, real-time collaboration features, or apps where users need to markup and sign PDFs within your UI. Nutrient excels at these scenarios and provides cross-platform viewer SDKs (Web, iOS, Android, desktop) with consistent APIs.

Use IronPDF when your requirement is backend document generation: rendering invoices from HTML templates, exporting reports to PDF, batch-converting content for archival, or automated document assembly. IronPDF's direct HTML-to-PDF workflow and generation-focused licensing align with these use cases.

The red flag: you are using Nutrient purely for HTML-to-PDF generation without leveraging its viewer, editor, or imaging capabilities. The broader product surface means picking the correct package — GdPicture for in-process .NET, Document Engine for server-side conversion — and the correct HTML rendering path is not always obvious, and the host-browser dependency for HTML rendering adds operational moving parts your team may not have planned for.

The green flag: you need both generation and interactive viewing. In that case, evaluate honestly whether Nutrient's comprehensive approach makes sense — or whether combining IronPDF for generation with a simpler viewer library gives you better results for less cost and less integration friction.

For teams generating thousands of PDFs per day from HTML templates with no interactive viewing requirements, IronPDF's focused API and licensing structure typically prove more cost-effective. For teams building document-centric applications where users annotate, sign, and collaborate on PDFs, Nutrient's comprehensive SDK justifies its scope.
