Gotenberg vs IronPDF: Microservice or Library for .NET PDF Generation?

One generates PDFs over an HTTP boundary you operate. The other generates them with a method call inside your process. That single architectural choice — service vs. library — drives most of the downstream differences in latency, deployment, debugging, and operational footprint between Gotenberg and IronPDF.

Both produce high-quality PDFs from HTML using a Chromium-family engine. The decision really comes down to where you want the complexity to live: in your container orchestration platform, or in your application binary.

Here is a structured comparison to help you decide which approach fits your team and your deployment reality.

What is Gotenberg and how does it work?

Gotenberg represents a different architectural philosophy: PDF generation as a containerized microservice accessible via HTTP. .NET applications send HTML, URLs, or Office documents to a Gotenberg container and receive PDFs via HTTP response. This approach makes sense in polyglot environments where Python, Node.js, and .NET services all need PDF generation without language-specific SDKs.

Gotenberg version 8.x is actively developed as an open-source project. The Docker-first design means .NET developers manage container infrastructure, image updates, and service availability. For teams without existing container orchestration platforms, this introduces additional operational surface area.

IronPDF takes the opposite approach -- a native .NET library that integrates directly into your application process. It is a NuGet package that embeds a Chrome rendering engine directly into your .NET application. HTML-to-PDF conversion happens in-process, eliminating network calls and external service dependencies. The library supports .NET Framework 4.6.2+, .NET Core 3.1+, and .NET 6-10 with consistent behavior across Windows and Linux. Beyond HTML rendering, IronPDF provides PDF manipulation APIs within the same package: merging files, extracting text, filling forms, adding digital signatures. For detailed rendering options, see the HTML to PDF Conversion Tutorial.

What are the key trade-offs of Gotenberg for .NET teams?

Gotenberg's trade-offs fall into several categories that become apparent once you move past initial prototyping.

API surface. Gotenberg does not ship a .NET-native API. Interactions happen via HTTP multipart/form-data requests, so error handling typically involves parsing HTTP status codes and response bodies. There is no compile-time validation of form field names, and teams generally implement HTTP client logic, retry, and timeout handling themselves (or via a community wrapper such as Gotenberg.Sharp.API.Client).

Latency profile. The HTTP API adds round-trip overhead: serializing HTML/documents into a multipart request, network hop to the Gotenberg service, Chromium processing, network hop back, deserialization. For high-throughput scenarios this can accumulate, and synchronous calls hold a thread while waiting on the response.

Support model. Gotenberg is community-supported open-source software. There are no commercial SLAs out of the box; security patches and updates are tracked via GitHub releases and applied by rebuilding container images on your side.

Operational footprint. Running Gotenberg requires Docker runtime, container orchestration (Docker Compose, Kubernetes, etc.), and network configuration for service discovery. The Docker image bundles Chromium and LibreOffice and is typically in the 600-800MB range. Horizontal scaling involves load balancer configuration, and webhook-based async flows require shared session state coordination.

How do the code implementations compare?

The code comparison shows the difference in developer experience directly. Gotenberg has you build and maintain HTTP client plumbing. IronPDF exposes a typed .NET API.

Gotenberg -- HTML to PDF via HTTP API

using System;
using System.IO;
using System.Net.Http;
using System.Threading.Tasks;

public class GotenbergPdfGenerator
{
    private static readonly HttpClient httpClient = new HttpClient
    {
        BaseAddress = new Uri("http://gotenberg-service:3000"),
        Timeout = TimeSpan.FromMinutes(5)
    };

    public static async Task<byte[]> GeneratePdfFromHtml(string htmlContent)
    {
        try
        {
            // Create multipart form data
            using var multipartContent = new MultipartFormDataContent();

            // Add HTML file as form field
            var htmlBytes = System.Text.Encoding.UTF8.GetBytes(htmlContent);
            var htmlStream = new MemoryStream(htmlBytes);
            var htmlFileContent = new StreamContent(htmlStream);
            htmlFileContent.Headers.ContentType =
                new System.Net.Http.Headers.MediaTypeHeaderValue("text/html");

            multipartContent.Add(htmlFileContent, "files", "index.html");

            // Optional: Add custom headers for Gotenberg configuration
            multipartContent.Add(
                new StringContent("1080"),
                "emulatedMediaType",
                "screen"
            );

            // Send request to Gotenberg
            var response = await httpClient.PostAsync(
                "/forms/chromium/convert/html",
                multipartContent
            );

            // Check response status
            if (!response.IsSuccessStatusCode)
            {
                var errorBody = await response.Content.ReadAsStringAsync();
                throw new Exception(
                    $"Gotenberg error (HTTP {response.StatusCode}): {errorBody}"
                );
            }

            // Read PDF bytes from response
            var pdfBytes = await response.Content.ReadAsByteArrayAsync();

            Console.WriteLine($"Generated PDF: {pdfBytes.Length} bytes");
            return pdfBytes;
        }
        catch (HttpRequestException ex)
        {
            Console.WriteLine($"Network error connecting to Gotenberg: {ex.Message}");
            throw;
        }
        catch (TaskCanceledException ex)
        {
            Console.WriteLine($"Request timeout: {ex.Message}");
            throw;
        }
    }

    public static async Task GenerateInvoicePdf()
    {
        var htmlContent = @"
            <html>
            <head>
                <style>
                    body { font-family: Arial; padding: 40px; }
                    h1 { font-size: 24px; }
                    table { width: 100%; border-collapse: collapse; margin-top: 20px; }
                    th, td { padding: 10px; border: 1px solid #ddd; text-align: left; }
                    th { background: #f0f0f0; }
                </style>
            </head>
            <body>
                <h1>Invoice #12345</h1>
                <p>Date: " + DateTime.Now.ToString("yyyy-MM-dd") + @"</p>
                <p>Customer: Acme Corporation</p>

                <table>
                    <thead>
                        <tr><th>Item</th><th>Quantity</th><th>Price</th><th>Total</th></tr>
                    </thead>
                    <tbody>
                        <tr><td>Professional Services</td><td>40</td><td>$150</td><td>$6,000</td></tr>
                        <tr><td>Consulting</td><td>10</td><td>$200</td><td>$2,000</td></tr>
                    </tbody>
                </table>
                <p><strong>Total: $8,000</strong></p>
            </body>
            </html>";

        var pdfBytes = await GeneratePdfFromHtml(htmlContent);
        File.WriteAllBytes("invoice_gotenberg.pdf", pdfBytes);
        Console.WriteLine("Invoice created: invoice_gotenberg.pdf");
    }
}

The Gotenberg sample requires HttpClient configuration, multipart/form-data construction, Content-Type headers, HTTP status code handling, network exception handling, and timeout exception handling. Field names like emulatedMediaType are string literals validated at runtime, not compile time. Local development needs the Gotenberg container running.

IronPDF -- HTML to PDF in-process

using IronPdf;
using System;

public class IronPdfGenerator
{
    public static void GenerateInvoicePdf()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
        var renderer = new ChromePdfRenderer();

        var htmlContent = @"
            <html>
            <head>
                <style>
                    body { font-family: Arial; padding: 40px; }
                    h1 { font-size: 24px; }
                    table { width: 100%; border-collapse: collapse; margin-top: 20px; }
                    th, td { padding: 10px; border: 1px solid #ddd; text-align: left; }
                    th { background: #f0f0f0; }
                </style>
            </head>
            <body>
                <h1>Invoice #12345</h1>
                <p>Date: " + DateTime.Now.ToString("yyyy-MM-dd") + @"</p>
                <p>Customer: Acme Corporation</p>

                <table>
                    <thead>
                        <tr><th>Item</th><th>Quantity</th><th>Price</th><th>Total</th></tr>
                    </thead>
                    <tbody>
                        <tr><td>Professional Services</td><td>40</td><td>$150</td><td>$6,000</td></tr>
                        <tr><td>Consulting</td><td>10</td><td>$200</td><td>$2,000</td></tr>
                    </tbody>
                </table>

        var pdf = renderer.RenderHtmlAsPdf(htmlContent);

}

IronPDF gives you type-safe APIs with IntelliSense support, no network configuration or service dependencies, synchronous and async methods, rich exception context with stack traces, and debugging with breakpoints in the rendering pipeline. How do the features compare across all dimensions?

I have organized the full comparison by category so you can quickly find the capabilities that matter to your project.

Status and support

  • Development status. Gotenberg is an active open-source project on the v8.x line. IronPDF ships as an active commercial product.
  • Support model. Gotenberg relies on the GitHub community. IronPDF includes commercial support tied to the license.
  • SLA. Gotenberg does not offer SLAs out of the box. IronPDF offers SLA options under license.
  • Documentation. Gotenberg uses GitHub-hosted docs. IronPDF maintains a documentation site with worked examples.
  • Updates. Gotenberg upgrades happen by rebuilding container images. IronPDF upgrades happen via NuGet.
  • Security patches. Gotenberg patches are tracked through GitHub releases and rolled out by rebuilding the image. IronPDF patches arrive through NuGet package updates.

Deployment architecture

  • Deployment model. Gotenberg runs as a microservice container. IronPDF runs as an in-process .NET library.
  • Infrastructure. Gotenberg needs Docker plus orchestration. IronPDF needs a NuGet package reference.
  • Network. Gotenberg exposes an HTTP endpoint typically fronted by a load balancer. IronPDF has no network surface.
  • Service discovery. Required for Gotenberg in multi-replica setups; not applicable to IronPDF.
  • Horizontal scaling. Gotenberg scales by adding container replicas behind a balancer. IronPDF scales with the application's own instances.
  • Health monitoring. Gotenberg needs its own health checks; IronPDF rolls into application-level monitoring.

Development experience

  • API style. Gotenberg exposes an HTTP REST API with multipart/form-data payloads. IronPDF exposes a native .NET API.
  • Type safety. Gotenberg field names are strings on the wire. IronPDF options are typed properties with IntelliSense.
  • Local development. Gotenberg requires running the container locally. IronPDF integrates directly into the solution.
  • Debugging. Gotenberg debugging tends to use HTTP inspection tools (Fiddler, browser dev tools). IronPDF debugs through the Visual Studio debugger.
  • Error handling. Gotenberg surfaces errors as HTTP status codes and response bodies. IronPDF surfaces errors as managed .NET exceptions.
  • Async. Gotenberg is async over HTTP. IronPDF supports native async/await.

Performance characteristics

Gotenberg adds a network round-trip plus Chromium processing on every call, with throughput bounded by the service's configured capacity and queue size. Cold start includes the container's own startup time. The Docker image footprint is typically in the 600-800MB range because it bundles Chromium and LibreOffice.

IronPDF runs in-process, so there is no network hop. Throughput is bounded by the host application's CPU and memory and by the .NET thread pool. Cold start is the embedded Chrome engine initializing on first render. Memory footprint is generally lower than the Gotenberg container's because LibreOffice is not part of the package — verify against your workload.

Content creation

  • HTML to PDF. Gotenberg uses Chromium behind an HTTP endpoint. IronPDF embeds a Chrome engine in-process.
  • URL to PDF. Gotenberg via the URL endpoint; IronPDF via ChromePdfRenderer.RenderUrlAsPdf().
  • Office documents to PDF. Gotenberg supports DOCX/XLSX/PPTX conversion natively via LibreOffice. IronPDF does not convert Office documents directly — you would render via HTML or pair it with another component (e.g., IronWord for DOCX). This is a real gap if Office conversion is core to your workload.
  • Markdown to PDF. Gotenberg has a Markdown endpoint. IronPDF expects you to render Markdown to HTML (e.g., with Markdig) and then to PDF.
  • JavaScript and CSS. Both rely on a Chromium-family engine, so modern JavaScript execution and CSS3 are supported on both sides.

How do the APIs map between Gotenberg and IronPDF?

If you are migrating from Gotenberg to IronPDF, or comparing the two side by side, this mapping shows how each concept translates.

  • POST /forms/chromium/convert/html — ChromePdfRenderer.RenderHtmlAsPdf()
  • POST /forms/chromium/convert/url — ChromePdfRenderer.RenderUrlAsPdf()
  • POST /forms/libreoffice/convert — no direct IronPDF equivalent; Office conversion is not in scope for IronPDF
  • POST /forms/pdfengines/merge — PdfDocument.Merge()
  • Multipart form-data with file fields — direct method parameters and RenderingOptions properties
  • HTTP headers for configuration — typed ChromePdfRenderOptions properties
  • Webhook for async results — native async/await methods
  • HTTP status codes for errors — managed .NET exceptions
  • Manual HttpClient lifecycle — standard .NET object lifetime, no network plumbing required

For a full reference of IronPDF's rendering options, see the IronPDF render options documentation.

How do you install each tool?

The installation process highlights the architectural difference immediately. Gotenberg requires Docker infrastructure. IronPDF requires a NuGet package.

Gotenberg installation

Install Docker for your platform, then run the Gotenberg container:

# Basic deployment
docker run --rm -p 3000:3000 gotenberg/gotenberg:8

# Production deployment with Docker Compose
# docker-compose.yml
version: '3.8'
services:
  gotenberg:
    image: gotenberg/gotenberg:8
    ports:
      - "3000:3000"
    environment:
      - API_TIMEOUT=30s
      - CHROMIUM_MAX_QUEUE_SIZE=10
    restart: always

Configure your .NET HTTP client against the running service:

using System.Net.Http;

var httpClient = new HttpClient
{
    BaseAddress = new Uri("http://gotenberg-service:3000"),
    Timeout = TimeSpan.FromMinutes(5)
};

Your deployment checklist for Gotenberg includes: Docker installed on all environments, container orchestration configured, service discovery endpoints set, health checks configured, resource limits defined, and monitoring and logging integrated.

IronPDF installation

# NuGet Package Manager
Install-Package IronPdf

# .NET CLI
dotnet add package IronPdf

Configuration is minimal:

using IronPdf;

// Optional license configuration
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

Your deployment checklist for IronPDF: NuGet package referenced in project, license key configured for production. That is the complete list. Learn more in the IronPDF getting started guide.

FAQ

When should I choose Gotenberg over IronPDF?

Choose Gotenberg if you operate a polyglot microservices architecture where Python, Node.js, Java, and .NET all need PDF generation from a single shared service. Gotenberg also makes sense if you need to convert Office documents (Word, Excel, PowerPoint) to PDF via its built-in LibreOffice support, which IronPDF does not natively offer. You should also have existing Kubernetes or Docker Swarm infrastructure and a DevOps team comfortable managing container orchestration.

How much infrastructure does Gotenberg require compared to IronPDF?

Gotenberg requires Docker runtime on all environments (dev, staging, prod), a container orchestration platform, network configuration for service discovery, health check monitoring endpoints, resource limits for CPU and memory, image update processes for security patches, load balancer configuration for horizontal scaling, and firewall rules allowing HTTP traffic between services. IronPDF requires a NuGet package reference in your .csproj and a license key configuration for production use. Optionally, Linux deployments may need libgdiplus for certain image operations.

What are the known environment-specific issues for each tool?

Gotenberg considerations typically include Docker Desktop licensing for larger organizations, container startup time affecting cold-start behavior in serverless environments, network partitions that can propagate into the calling service, occasional Chromium process restarts in long-running containers, load balancer configuration for horizontal scaling, and an image size in the 600-800MB range that affects pull/deploy time. IronPDF considerations are more contained: an initial Chrome engine warm-up on the first render, Linux deployments may need libgdiplus for certain image operations depending on your .NET version, and Docker containers running IronPDF benefit from explicit memory configuration for the embedded Chrome process.

How does operational monitoring differ between the two approaches?

Gotenberg requires monitoring of service health endpoints, container resource usage metrics, HTTP request latency tracking, error rate monitoring for 4xx and 5xx responses, queue depth monitoring, Docker container log aggregation, Chromium crash detection and alerting, network connectivity monitoring between services, and load balancer health checks. This requires separate monitoring infrastructure with multiple failure points. IronPDF monitoring reduces to application-level exception tracking, performance counters for PDF generation timing, and memory usage monitoring for the Chrome engine -- a single process to monitor with your existing application monitoring tools.

How do the costs break down for each approach?

Gotenberg is free as open-source software, but operational costs include: container orchestration platform costs (EKS, AKS, GKE), load balancer costs, container registry storage, increased compute resources for running a separate service, DevOps time for container deployment and management, monitoring infrastructure costs, and network egress costs between services. IronPDF has a license cost (one-time or annual) but no additional infrastructure costs, no separate service management overhead, and deployment costs are included in your existing application deployment.

How should you decide?

Gotenberg and IronPDF represent fundamentally different architectural approaches to PDF generation. Gotenberg's microservice model makes sense in polyglot environments where multiple programming languages need PDF generation without implementing language-specific integrations. The Docker-based deployment provides isolation and enables horizontal scaling through container orchestration. However, this architecture introduces operational complexity: container management, service discovery, network configuration, health monitoring, and multi-stage deployment pipelines.

For .NET-centric applications, Gotenberg requires building and maintaining HTTP client infrastructure that handles multipart/form-data requests, parses response codes, implements retry logic, and manages timeouts. Each PDF generation adds network round-trip latency. Debugging involves HTTP inspection tools rather than stepping through managed code. Type safety disappears at the HTTP boundary -- configuration options become string literals susceptible to typos and runtime errors.

IronPDF consolidates PDF generation into a single NuGet package with a type-safe, IntelliSense-supported API. Deployment involves no Docker configuration, no service health monitoring, no network infrastructure. In-process execution eliminates network latency and simplifies debugging with Visual Studio breakpoints. For teams without existing container orchestration platforms, this reduces operational overhead significantly. For a deeper look at what IronPDF offers, explore the full HTML to PDF tutorial.

The choice depends on architectural context. Polyglot teams with mature Kubernetes infrastructure may find Gotenberg's service-based model aligns with existing patterns. .NET teams prioritizing development velocity and deployment simplicity typically benefit from IronPDF's library-based integration. Both approaches deliver high-quality PDF rendering; the decision hinges on infrastructure philosophy and operational complexity tolerance.
