Spire.PDF vs IronPDF: The Practical Breakdown for .NET

The interesting architectural question for any .NET PDF library in 2026 is not "can it draw a page?" — it is "what is the gap between the content you already have and the PDF you need to ship?" If your source content is HTML — invoice templates, generated reports, email-ready documents — a browser-based renderer reproduces what you already see in DevTools. If your source content is a Word or Excel file, a library with native Office conversion saves you an intermediate step. Picking the right tool starts with naming the input, not the output.

Spire.PDF is a commercial PDF library from E-iceblue that covers creating PDFs from scratch, editing existing documents, and converting various formats (HTML, images, text, and Office documents) to PDF. Unlike browser-based renderers, Spire.PDF uses a proprietary PDF generation engine, which means HTML-to-PDF conversion does not rely on Chromium or WebKit. For teams evaluating PDF libraries, this comparison walks through what Spire.PDF does well, where its architecture differs from a browser-based engine, and when IronPDF's HTML-first approach offers better alignment with modern .NET workflows.

What Is IronPDF?

IronPDF is purpose-built for HTML-to-PDF conversion using Google Chrome's rendering engine, with additional PDF manipulation capabilities (merge, split, sign, secure, extract). The Chrome-based rendering approach means your HTML renders pixel-perfectly as PDF — if it looks right in Chrome DevTools, it looks right in the PDF. IronPDF supports .NET 10, 9, 8, 7, 6, Core, Framework, and runs on Windows, Linux, macOS, Docker, and cloud platforms.

The library's workflow optimizes for a specific scenario: you already have HTML (web views, templates, generated content), and you need it as PDF. No learning a new document layout API; no rewriting content. Write HTML, call RenderHtmlAsPdf(), get PDF. For teams building web applications, this workflow reduces PDF integration to a rendering step, not a development project.

What Are the Key Limitations of Spire.PDF?

Product status. Commercial library maintained by E-iceblue. The "Free Spire.PDF" edition carries a documented 10-page-per-document limit on loading and creating PDFs, with further restrictions when converting to image, Word, HTML, or XPS — verify the current free-tier terms on E-iceblue's site before relying on the free build for production. The paid product has multiple license tiers (developer, site, OEM) with deployment terms that vary by tier — review the current license agreement before shipping.

Rendering engine. HTML-to-PDF conversion uses E-iceblue's own HTML parser rather than Chromium or WebKit. Modern CSS layout (CSS Grid, Flexbox), JavaScript execution, and web fonts via @font-face may behave differently from a browser — verify against your specific HTML and Spire.PDF version. There is no DevTools-equivalent preview, so HTML rendering behavior is engine-specific rather than WYSIWYG against Chrome.

HTML conversion API surface. The primary entry point is PdfDocument.LoadFromHTML(), which has multiple overloads — one accepts an HTML string directly, another expects a URL or file path with layout settings. Image loading from HTML typically requires accessible file paths or base64 encoding; remote HTTP image loads can fail silently depending on the overload and configuration.

Drawing API for PDFs from scratch. When you are not coming from HTML, Spire.PDF exposes a low-level drawing model: PdfPageBase, PdfFont, PdfBrush, PdfTable, and friends. This is powerful for vector-precise output but is a different mental model from HTML/CSS, so web-experienced teams will hit a learning curve.

Office-document conversion. Spire.PDF (and the wider E-iceblue suite) covers Word- and Excel-to-PDF conversion natively. IronPDF does not provide native Office-format conversion in the box — if your pipeline must consume .docx or .xlsx directly without an HTML-rendering intermediate, that is a legitimate Spire.PDF strength to keep in mind.

Support and distribution. Commercial support via E-iceblue email and forums; there is no public GitHub repository for issue tracking. Multiple assemblies (Spire.Pdf, Spire.License) ship with the package, with a typical footprint in the tens of MB.

How Does HTML String to PDF Conversion Compare?

Spire.PDF Implementation

// NuGet: Install-Package Spire.PDF
using Spire.Pdf;
using Spire.Pdf.Graphics;
using System;

class Program
{
    static void Main()
    {
        PdfDocument pdf = new PdfDocument();
        PdfPageSettings setting = new PdfPageSettings();
        setting.Size = PdfPageSize.A4;
        PdfHtmlLayoutFormat htmlLayoutFormat = new PdfHtmlLayoutFormat();
        htmlLayoutFormat.IsWaiting = false;

        string htmlString = "<html><body><h1>Hello World</h1><p>This is a PDF from HTML.</p></body></html>";

        // HTML-string overload: LoadFromHTML(string, bool autoDetectPageBreak, PdfPageSettings, PdfHtmlLayoutFormat)
        pdf.LoadFromHTML(htmlString, true, setting, htmlLayoutFormat);
        pdf.SaveToFile("output.pdf");
        pdf.Close();
    }
}

Spire.PDF's LoadFromHTML string overload is concise. A separate file/URL overload (LoadFromHTML(path, ..., settings, null)) exposes more layout settings but requires writing the HTML to a file or hosting it at a URL. Modern CSS layout (Grid, Flexbox) and JavaScript execution may behave differently from a browser-based engine — verify against your specific HTML and Spire.PDF version. Free Spire.PDF additionally caps output at 10 pages per document; verify current free-tier terms before using it.

IronPDF Implementation

// NuGet: Install-Package IronPdf
using IronPdf;

public void ConvertHtmlToPdf(string html, string outputPath)
{
    IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
    var renderer = new ChromePdfRenderer();
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs(outputPath);
}

IronPDF accepts an HTML string directly, with full CSS3 (including Grid and Flexbox) and JavaScript through the Chrome engine, web fonts via @font-face, and WYSIWYG output that matches Chrome's print preview. No table-count cap on the trial or paid licenses. See the IronPDF HTML rendering documentation for more details.

How Does Creating PDFs from Scratch with Text and Tables Compare?

Spire.PDF Implementation

using Spire.Pdf;
using Spire.Pdf.Graphics;
using Spire.Pdf.Tables;
using System.Data;
using System.Drawing;

public void CreatePdfFromScratch(string outputPath)
{
    // Create new PDF document
    PdfDocument doc = new PdfDocument();

    // Add page
    PdfPageBase page = doc.Pages.Add(PdfPageSize.A4);

    // Create font
    PdfFont font = new PdfFont(PdfFontFamily.Helvetica, 12f);
    PdfFont titleFont = new PdfFont(PdfFontFamily.Helvetica, 20f);

    // Create brush for text
    PdfBrush brush = PdfBrushes.Black;

    // Draw title
    page.Canvas.DrawString("Invoice Report", titleFont, brush,
        new PointF(50, 50));

    // Create table (Spire-specific table API)
    PdfTable table = new PdfTable();
    table.Style.CellPadding = 2;
    table.Style.BorderPen = new PdfPen(Color.Black, 0.5f);
    table.Style.DefaultStyle.Font = font;

    // Create data source
    DataTable dataTable = new DataTable();
    dataTable.Columns.Add("Item");
    dataTable.Columns.Add("Quantity");
    dataTable.Columns.Add("Price");
    dataTable.Rows.Add("Widget A", "10", "$50.00");
    dataTable.Rows.Add("Widget B", "5", "$75.00");
    dataTable.Rows.Add("Widget C", "20", "$100.00");

    // Set table data source
    table.DataSource = dataTable;

    // Draw table at specific position
    PdfLayoutResult result = table.Draw(page, new PointF(50, 100));

    // Add more content after table
    float yPos = result.Bounds.Bottom + 20;
    page.Canvas.DrawString("Total: $225.00", font, brush,
        new PointF(50, yPos));

    // Save document
    doc.SaveToFile(outputPath, FileFormat.PDF);
    doc.Close();
}

Spire.PDF's drawing API offers low-level control and precise positioning with table creation from a DataTable. The trade-off is the Spire-specific surface (PdfTable, PdfFont, PdfBrush) and manual layout calculations — Y-position tracking for stacking content, explicit unit handling.

How Does Adding Watermarks to Existing PDFs Compare?

Spire.PDF Implementation

using Spire.Pdf;
using Spire.Pdf.Graphics;
using System.Drawing;

public void AddWatermark(string inputPath, string outputPath, string watermarkText)
{
    // Load existing PDF
    PdfDocument doc = new PdfDocument(inputPath);

    // Configure watermark font and brush
    PdfFont font = new PdfFont(PdfFontFamily.Helvetica, 72f);
    PdfBrush brush = new PdfBrush(Color.FromArgb(100, 200, 200, 200)); // Semi-transparent

    // Iterate through pages
    foreach (PdfPageBase page in doc.Pages)
    {
        // Calculate position (center of page, rotated)
        SizeF pageSize = page.Size;
        float x = (pageSize.Width - font.MeasureString(watermarkText).Width) / 2;
        float y = (pageSize.Height - font.MeasureString(watermarkText).Height) / 2;

        // Save graphics state
        page.Canvas.Save();

        // Rotate and draw watermark
        page.Canvas.TranslateTransform(pageSize.Width / 2, pageSize.Height / 2);
        page.Canvas.RotateTransform(-45); // Diagonal
        page.Canvas.TranslateTransform(-pageSize.Width / 2, -pageSize.Height / 2);

        page.Canvas.DrawString(watermarkText, font, brush,
            new PointF(x, y), new PdfStringFormat());

        // Restore graphics state
        page.Canvas.Restore();
    }

    // Save modified PDF
    doc.SaveToFile(outputPath, FileFormat.PDF);
    doc.Close();
}

Spire.PDF offers precise control over watermark positioning with rotation and transparency support. The trade-off is manual calculation of positions and transformations, and a separate API path for image-based watermarks. If you are using the free edition, verify watermark behavior against E-iceblue's current free-tier terms before relying on it.

IronPDF Implementation

using IronPdf;

public void AddWatermark(string inputPath, string outputPath, string watermarkText)
{
    IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
    using var pdf = PdfDocument.FromFile(inputPath);

    // HTML-based watermark; opacity is an integer 0-100
    string watermarkHtml = $@"
        <div style='
            font-size: 72px;
            color: rgba(200, 200, 200, 0.4);
            font-weight: bold;
            font-family: Helvetica;'>
            {watermarkText}
        </div>";

    pdf.ApplyWatermark(watermarkHtml, rotation: 45, opacity: 40);
    pdf.SaveAs(outputPath);
}

IronPDF provides a simple, concise API with HTML-based watermarks supporting full CSS styling. It handles text and images, automatic centering and rotation, no manual position calculations, and works on all pages automatically. See the IronPDF watermark guide for more options.

What Does the API Mapping Between Spire.PDF and IronPDF Look Like?

  • PdfDocument.LoadFromHTML(html, ...) — ChromePdfRenderer.RenderHtmlAsPdf(html)
  • new PdfDocument() — renderer.RenderHtmlAsPdf(...) returns a PdfDocument
  • doc.SaveToFile(path, FileFormat.PDF) — pdf.SaveAs(path)
  • doc.Pages.Add() — HTML defines pages automatically; CSS controls page breaks
  • page.Canvas.DrawString(...) — HTML text elements + CSS
  • PdfTable API — HTML <table> elements
  • PdfFont, PdfBrush, PdfPen — CSS styling (font-family, color, border)
  • Manual X/Y positioning and transforms — CSS layout (Grid, Flexbox, transforms)
  • doc.Close() — automatic via using / IDisposable
  • doc.InsertPageRange(other, 0, count - 1) — PdfDocument.Merge(pdf1, pdf2)
  • doc.Security.UserPassword / OwnerPassword — pdf.SecuritySettings.UserPassword / OwnerPassword

How does Spire.PDF handle HTML-to-PDF conversion compared to IronPDF?

Conclusion

Spire.PDF and IronPDF address different PDF generation philosophies. Spire.PDF provides a broad PDF library with a low-level drawing API and editing capabilities — valuable when composing PDFs programmatically from scratch, and especially when your source content is Word or Excel that needs native conversion. Its HTML-to-PDF path uses a custom parser, so modern CSS layout and JavaScript behavior should be verified against your specific templates and the Spire.PDF version you ship.

IronPDF's Chrome-based HTML rendering targets teams with existing HTML content: web views, email templates, generated reports. The workflow is simple — write HTML (the content you already have), render to PDF, manipulate if needed. No proprietary drawing API to learn for layout, no manual Y-coordinate tracking. If your templates are HTML, IronPDF converts them with the fidelity you see in Chrome. If you are building PDFs from scratch with complex vector graphics, consider whether HTML can express your layout first.

For most .NET teams in 2026, the question is: do you have HTML to convert, or PDFs to draw? If you have HTML, IronPDF's architecture aligns naturally with your workflow. If you are drawing PDFs from scratch — or piping Word/Excel content into PDF without an HTML intermediate — Spire.PDF provides those tools. Most web applications fall into the first category, which is where IronPDF tends to be the pragmatic choice.
