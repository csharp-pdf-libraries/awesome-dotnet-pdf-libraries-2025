Spire.PDF vs IronPDF: The Practical Breakdown for .NET

Imagine onboarding a new developer who asks, "Can we generate PDFs from our existing HTML templates?" and discovering your current PDF library uses a proprietary drawing API that requires rewriting every template from scratch. One team I heard about spent two months migrating templates designed in HTML to a library-specific layout syntax before realizing they should have evaluated HTML rendering support upfront. In 2026, the question is not "does it create PDFs?" — it is "does it work with the content I already have?"

Spire.PDF is a comprehensive PDF library from E-iceblue that covers creating PDFs from scratch, editing existing documents, and converting various formats (HTML, images, text) to PDF. Unlike browser-based renderers, Spire.PDF uses a proprietary PDF generation engine, which means HTML-to-PDF conversion does not rely on Chromium or WebKit. For teams evaluating PDF libraries, this comparison provides a checklist-driven examination: what Spire.PDF does well, where it falls short, and when IronPDF's HTML-first approach offers better alignment with modern .NET workflows.

What Is IronPDF?

Answer:

IronPDF is purpose-built for HTML-to-PDF conversion using Google Chrome's rendering engine, with additional PDF manipulation capabilities (merge, split, sign, secure, extract). The Chrome-based rendering approach means your HTML renders pixel-perfectly as PDF — if it looks right in Chrome DevTools, it looks right in the PDF. IronPDF supports .NET 10, 9, 8, 7, 6, Core, Framework, and runs on Windows, Linux, macOS, Docker, and cloud platforms.

The library's workflow optimizes for a specific scenario: you already have HTML (web views, templates, generated content), and you need it as PDF. No learning a new document layout API; no rewriting content. Write HTML, call RenderHtmlAsPdf(), get PDF. For teams building web applications, this workflow reduces PDF integration to a rendering step, not a development project.

What Are the Key Limitations of Spire.PDF?

Answer:

Product Status: Commercial library maintained by E-iceblue with quarterly release cycle. Free edition has hard limits: 10 pages per document, 3 tables maximum, no watermarks. Exceeding limits adds evaluation watermark to output PDFs. Licensing model complex — multiple tiers (One Developer, One Site Small Business, etc.) with restrictions on deployment (e.g., no public-facing websites for certain licenses). Evaluation mode adds "Evaluation Warning" to PDFs unless licensed.

Missing Capabilities: HTML-to-PDF conversion exists but limited compared to browser-based renderers. CSS support incomplete — modern layout techniques (CSS Grid, Flexbox) not fully implemented or behave unpredictably. JavaScript execution not supported during HTML conversion — any dynamic content must be pre-rendered server-side. No Chromium/WebKit rendering; uses custom HTML parser with partial spec compliance. Font handling requires manual embedding; web fonts via @font-face support unclear. Responsive design support limited — no media queries or viewport-based rendering.

Technical Issues: HTML conversion uses PdfDocument.LoadFromHTML() method with unpredictable layout results for complex HTML. Teams report CSS positioning issues, table rendering miscalculations, and unexpected page breaks. Image loading from HTML requires accessible file paths or base64 encoding; remote HTTP images may fail silently. Threading model unclear — concurrent PDF generation may require creating separate PdfDocument instances per thread. Memory usage spikes with large documents (100+ pages); no built-in memory optimization for batch processing.

Support Status: Commercial support via E-iceblue email and forums; response time variable (hours to days). Documentation focuses on basic scenarios; advanced troubleshooting guidance sparse. No public GitHub for community contributions or issue tracking. Licensing activation required; trial key sent via email, can delay initial testing.

Architecture Problems: Proprietary PDF engine means HTML rendering behavior differs from browsers — no DevTools preview, no WYSIWYG. Drawing API (for creating PDFs from scratch) requires learning Spire-specific objects: PdfSection, PdfPageBase, PdfTextWidget, etc. Not intuitive for web developers familiar with HTML/CSS. Multiple assemblies (Spire.Pdf.dll, Spire.License.dll, potentially others) complicate dependency management. DLL sizes increase project footprint (~20-30MB), though smaller than browser-based renderers (~50MB).

How Does HTML String to PDF Conversion Compare?

Answer:

Spire.PDF Implementation

using Spire.Pdf;
using Spire.Pdf.Graphics;
using System.Drawing;
using System.IO;
using System.Threading;

public class SpirePdfHtmlConverter
{
    public void ConvertHtmlToPdf(string html, string outputPath)
    {
        // Spire.PDF requires creating PdfDocument
        PdfDocument doc = new PdfDocument();

        // Configure page settings
        PdfPageSettings settings = new PdfPageSettings();
        settings.Size = new SizeF(1000, 1000); // Custom page size
        settings.Margins = new Spire.Pdf.Graphics.PdfMargins(20);

        // LoadFromHTML expects URL or file path, not string
        // For HTML string, must write to temp file
        string tempFile = Path.GetTempFileName() + ".html";
        File.WriteAllText(tempFile, html);

        try
        {
            // Create thread for HTML loading (required by Spire.PDF)
            Thread thread = new Thread(() =>
            {
                // Load HTML with settings
                doc.LoadFromHTML(tempFile, false, false, false, settings, null);
            });

            // Set apartment state (required for some HTML rendering)
            thread.SetApartmentState(ApartmentState.STA);
            thread.Start();
            thread.Join(); // Wait for completion

            // Save to PDF
            doc.SaveToFile(outputPath, FileFormat.PDF);
        }
        finally
        {
            // Clean up temp file
            if (File.Exists(tempFile))
                File.Delete(tempFile);

            doc.Close();
        }
    }
}

Spire.PDF HTML conversion requires writing to a temp file (cannot load HTML string directly), mandatory thread creation with STA apartment state, incomplete CSS Grid/Flexbox support, no JavaScript execution, unclear web font support, manual page size specification, and unpredictable rendering of complex HTML layouts. The free tier limits tables to 3 per document.

IronPDF Implementation

using IronPdf;

public void ConvertHtmlToPdf(string html, string outputPath)
{
    var renderer = new ChromePdfRenderer();
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs(outputPath);
}

IronPDF supports direct HTML string input with no temp files, no threading complexity, full CSS Grid/Flexbox support via the Chrome engine, JavaScript execution, web fonts via @font-face, automatic page sizing from HTML content, and WYSIWYG output matching Chrome browser preview. No table count limitations. See the IronPDF HTML rendering documentation for more details.

How Does Creating PDFs from Scratch with Text and Tables Compare?

Answer:

Spire.PDF Implementation

using Spire.Pdf;
using Spire.Pdf.Graphics;
using Spire.Pdf.Tables;
using System.Data;
using System.Drawing;

public void CreatePdfFromScratch(string outputPath)
{
    // Create new PDF document
    PdfDocument doc = new PdfDocument();

    // Add page
    PdfPageBase page = doc.Pages.Add(PdfPageSize.A4);

    // Create font
    PdfFont font = new PdfFont(PdfFontFamily.Helvetica, 12f);
    PdfFont titleFont = new PdfFont(PdfFontFamily.Helvetica, 20f);

    // Create brush for text
    PdfBrush brush = PdfBrushes.Black;

    // Draw title
    page.Canvas.DrawString("Invoice Report", titleFont, brush,
        new PointF(50, 50));

    // Create table (Spire-specific table API)
    PdfTable table = new PdfTable();
    table.Style.CellPadding = 2;
    table.Style.BorderPen = new PdfPen(Color.Black, 0.5f);
    table.Style.DefaultStyle.Font = font;

    // Create data source
    DataTable dataTable = new DataTable();
    dataTable.Columns.Add("Item");
    dataTable.Columns.Add("Quantity");
    dataTable.Columns.Add("Price");
    dataTable.Rows.Add("Widget A", "10", "$50.00");
    dataTable.Rows.Add("Widget B", "5", "$75.00");
    dataTable.Rows.Add("Widget C", "20", "$100.00");

    // Set table data source
    table.DataSource = dataTable;

    // Draw table at specific position
    PdfLayoutResult result = table.Draw(page, new PointF(50, 100));

    // Add more content after table
    float yPos = result.Bounds.Bottom + 20;
    page.Canvas.DrawString("Total: $225.00", font, brush,
        new PointF(50, yPos));

    // Save document
    doc.SaveToFile(outputPath, FileFormat.PDF);
    doc.Close();
}

Spire.PDF's drawing API offers low-level control and precise positioning with table creation from DataTable. However, it requires learning Spire-specific APIs (PdfTable, PdfFont, PdfBrush, etc.), no HTML/CSS knowledge is transferable, and the code is verbose for complex layouts. Layout calculations are manual (calculating Y positions for stacking content), and the free tier is limited to 3 tables per document.

IronPDF Implementation (HTML-based)

using IronPdf;

public void CreatePdfFromScratch(string outputPath)
{
    string html = @"
<!DOCTYPE html>
<html>
<head>
    <style>
        body { font-family: Helvetica, Arial, sans-serif; margin: 50px; }
        h1 { font-size: 20pt; margin-bottom: 20px; }
        table { width: 100%; border-collapse: collapse; margin: 20px 0; }
        th, td { border: 1px solid black; padding: 8px; text-align: left; }
        .total { margin-top: 20px; font-size: 12pt; }
    </style>
</head>
<body>
    <h1>Invoice Report</h1>
    <table>
        <tr><th>Item</th><th>Quantity</th><th>Price</th></tr>
        <tr><td>Widget A</td><td>10</td><td>$50.00</td></tr>
        <tr><td>Widget B</td><td>5</td><td>$75.00</td></tr>
        <tr><td>Widget C</td><td>20</td><td>$100.00</td></tr>
    </table>
    <div class='total'>Total: $225.00</div>
</body>
</html>";

    var renderer = new ChromePdfRenderer();
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs(outputPath);
}

IronPDF uses familiar HTML/CSS with no new API to learn. Layout is handled automatically with no manual positioning, tables are unlimited, styling uses CSS for easier maintenance, and the code is shorter and more readable. For lower-level control when needed, see the IronPDF editing features.

How Does Adding Watermarks to Existing PDFs Compare?

Answer:

Spire.PDF Implementation

using Spire.Pdf;
using Spire.Pdf.Graphics;
using System.Drawing;

public void AddWatermark(string inputPath, string outputPath, string watermarkText)
{
    // Load existing PDF
    PdfDocument doc = new PdfDocument(inputPath);

    // Configure watermark font and brush
    PdfFont font = new PdfFont(PdfFontFamily.Helvetica, 72f);
    PdfBrush brush = new PdfBrush(Color.FromArgb(100, 200, 200, 200)); // Semi-transparent

    // Iterate through pages
    foreach (PdfPageBase page in doc.Pages)
    {
        // Calculate position (center of page, rotated)
        SizeF pageSize = page.Size;
        float x = (pageSize.Width - font.MeasureString(watermarkText).Width) / 2;
        float y = (pageSize.Height - font.MeasureString(watermarkText).Height) / 2;

        // Save graphics state
        page.Canvas.Save();

        // Rotate and draw watermark
        page.Canvas.TranslateTransform(pageSize.Width / 2, pageSize.Height / 2);
        page.Canvas.RotateTransform(-45); // Diagonal
        page.Canvas.TranslateTransform(-pageSize.Width / 2, -pageSize.Height / 2);

        page.Canvas.DrawString(watermarkText, font, brush,
            new PointF(x, y), new PdfStringFormat());

        // Restore graphics state
        page.Canvas.Restore();
    }

    // Save modified PDF
    doc.SaveToFile(outputPath, FileFormat.PDF);
    doc.Close();
}

Spire.PDF offers precise control over watermark positioning with rotation and transparency support. However, it requires manual calculation of positions and transformations, supports only text-based watermarks (image watermarks require a different API), and the code is verbose for a simple operation. The free tier may not support watermarks — verify before use.

IronPDF Implementation

using IronPdf;

public void AddWatermark(string inputPath, string outputPath, string watermarkText)
{
    using var pdf = PdfDocument.FromFile(inputPath);

    // Apply HTML-based watermark
    string watermarkHtml = $@"
        <div style='
            font-size: 72px;
            color: rgba(200, 200, 200, 0.4);
            font-weight: bold;
            font-family: Helvetica;'>
            {watermarkText}
        </div>";

    pdf.ApplyWatermark(watermarkHtml, rotation: 45, opacity: 40);
    pdf.SaveAs(outputPath);
}

IronPDF provides a simple, concise API with HTML-based watermarks supporting full CSS styling. It handles text and images, automatic centering and rotation, no manual position calculations, and works on all pages automatically. See the IronPDF watermark guide for more options.

Frequently Asked Questions

How does Spire.PDF handle HTML-to-PDF conversion compared to IronPDF?

Spire.PDF uses a proprietary custom HTML parser with incomplete CSS support and no JavaScript execution. It requires writing HTML to a temp file and creating a dedicated STA thread for conversion. IronPDF uses Google Chrome's rendering engine, accepting HTML strings directly with full CSS3, Flexbox, Grid, and JavaScript support. If your HTML looks right in Chrome DevTools, it looks right in the IronPDF output.

What are the free tier limitations of Spire.PDF?

Spire.PDF's free edition enforces hard limits: 10 pages per document, a maximum of 3 tables, and no watermark support. Exceeding these limits adds an "Evaluation Warning" watermark to output PDFs. IronPDF provides a 30-day full trial with no feature restrictions, allowing teams to evaluate every capability before purchasing.

Is Spire.PDF suitable for modern web template conversion?

Spire.PDF's HTML conversion uses a custom parser that does not fully implement modern CSS features like CSS Grid and Flexbox. JavaScript is not executed during conversion, and web font support via @font-face is unclear. For teams with existing HTML templates using modern CSS, IronPDF's Chrome-based rendering provides accurate, browser-identical conversion without rewriting templates.

What is the learning curve difference between Spire.PDF and IronPDF?

Spire.PDF requires learning a proprietary drawing API with Spire-specific objects like PdfSection, PdfPageBase, PdfTextWidget, PdfFont, and PdfBrush. Layout calculations are manual, requiring Y-position tracking for stacking content. IronPDF uses HTML and CSS — skills web developers already have. Write HTML, call RenderHtmlAsPdf(), and the Chrome engine handles layout automatically.

How does support differ between the two libraries?

Spire.PDF offers commercial support via E-iceblue email and forums with variable response times (hours to days). Documentation focuses on basic scenarios with sparse advanced troubleshooting. There is no public GitHub for community contributions. IronPDF provides 24/5 engineering support with guaranteed response times, comprehensive task-focused documentation, and active development with regular updates.

Conclusion

Spire.PDF and IronPDF address different PDF generation philosophies. Spire.PDF provides a comprehensive PDF library with low-level drawing API and editing capabilities — valuable when building PDFs programmatically from scratch. Its HTML-to-PDF conversion, however, uses a custom parser with incomplete CSS support and no JavaScript execution, making it unsuitable for modern web template conversion.

IronPDF's Chrome-based HTML rendering targets teams with existing HTML content: web views, email templates, generated reports. The workflow is simple — write HTML (the content you already have), render to PDF, manipulate if needed. No proprietary API to learn, no layout calculations, no manual positioning. If your templates are HTML, IronPDF converts them accurately. If you are building PDFs from scratch with complex vector graphics, evaluate whether HTML can represent your layout first.

For most .NET teams in 2026, the question is: do you have HTML to convert or PDFs to draw? If you have HTML, IronPDF's architecture aligns naturally with your workflow. If you are drawing PDFs from scratch and cannot express your layout in HTML, Spire.PDF provides those low-level tools. Most web applications fall into the first category — making IronPDF the pragmatic choice.
