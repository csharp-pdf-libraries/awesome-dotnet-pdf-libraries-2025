PDFSharp vs IronPDF in C#: Which Should Your Team Choose?

We benchmarked PDFSharp last quarter for a reporting system generating 10,000 monthly invoices. The test was simple: loop through order data, create PDF invoices, save to disk. PDFSharp clocked 47ms per invoice — impressive until we calculated the development cost. Each invoice required 200+ lines of positioning code: calculate header coordinates, draw company logo at (x, y), position each table cell, handle page breaks manually. When the client requested a layout change (add tax breakdown, move footer), we recalculated every coordinate downstream. The "fast" library cost us 40 hours of layout math per UI iteration.

PDFSharp is a low-level PDF construction library built for maximum control. You position every element using X/Y coordinates in points (1/72 inch). IronPDF trades explicit control for development velocity — you write HTML/CSS templates and let Chromium handle layout, text wrapping, pagination, and font rendering. This comparison walks through what each library does, how they differ, and which one fits your team's workflow.

What Is PDFSharp?

Answer:

PDFSharp is a low-level PDF construction library that gives you pixel-perfect precision through X/Y coordinate positioning. You place every element manually using points (1/72 inch). This gives maximum control but requires understanding PDF coordinate systems, font metrics, and layout algorithms. The library excels when you're building visualizations, rendering CAD drawings, or creating geometric designs where explicit positioning is the point.

It is open-source under the MIT license and actively maintained. Version 6.2.3 (December 2025) added .NET 9/10 support. Commercial support is available from empira Software GmbH. For document generation from templates, however, you are implementing a layout engine from scratch.

How Does IronPDF Differ from PDFSharp?

Answer:

IronPDF shifts the performance characteristic entirely. Instead of calculating coordinates, you write HTML/CSS templates and let the embedded Chromium engine handle layout, text wrapping, pagination, and font rendering. The architecture is straightforward: ChromePdfRenderer wraps a Chromium process, HTML/CSS determines layout, PdfDocument handles the result.

Installation is one NuGet package with Chromium bundled. The library runs cross-platform and supports modern CSS (Flexbox, Grid, media queries). For performance-critical batch operations, renderer reuse and async methods help amortize Chromium startup costs. For teams with web development skills, this is often a net win — you ship features instead of debugging coordinate math.

How Does HTML-to-PDF Generation Compare?

Answer:

PDFSharp has no HTML-to-PDF rendering — it is a graphics API, not a template engine. No CSS support, no JavaScript execution, no web fonts. You cannot reuse web designs or HTML templates. There are no built-in table layouts (you draw rectangles and text manually), no automatic text wrapping (calculate line breaks yourself), and no pagination engine (track height, insert page breaks manually).

IronPDF handles all of this through Chromium. Here is the contrast in code.

PDFSharp — Coordinate-Based Invoice

// Install-Package PDFsharp

using System;
using PdfSharp.Pdf;
using PdfSharp.Drawing;

public class InvoiceGenerator
{
    public void GenerateInvoice()
    {
        // Create document
        PdfDocument document = new PdfDocument();
        PdfPage page = document.AddPage();
        XGraphics gfx = XGraphics.FromPdfPage(page);

        // Define fonts (explicit sizing)
        XFont fontTitle = new XFont("Arial", 24, XFontStyle.Bold);
        XFont fontNormal = new XFont("Arial", 12, XFontStyle.Regular);
        XFont fontBold = new XFont("Arial", 12, XFontStyle.Bold);

        // Calculate positions (points from top-left)
        double leftMargin = 50;
        double topMargin = 50;
        double currentY = topMargin;

        // Draw title
        gfx.DrawString("INVOICE #2026-001", fontTitle, XBrushes.Black, leftMargin, currentY);
        currentY += 40;

        // Draw invoice info
        gfx.DrawString($"Date: {DateTime.Now:yyyy-MM-dd}", fontNormal, XBrushes.Black, leftMargin, currentY);
        currentY += 20;
        gfx.DrawString("Customer: Acme Corp", fontNormal, XBrushes.Black, leftMargin, currentY);
        currentY += 40;

        // Draw table header (manual rectangle + text positioning)
        double tableTop = currentY;
        double col1X = leftMargin;
        double col2X = leftMargin + 300;
        double rowHeight = 25;

        // Header background
        XRect headerRect = new XRect(col1X, tableTop, page.Width - 100, rowHeight);
        gfx.DrawRectangle(new XSolidBrush(XColor.FromArgb(240, 240, 240)), headerRect);

        // Header text
        gfx.DrawString("Description", fontBold, XBrushes.Black, col1X + 5, tableTop + 17);
        gfx.DrawString("Amount", fontBold, XBrushes.Black, col2X + 5, tableTop + 17);
        currentY += rowHeight;

        // Draw table row
        gfx.DrawString("Consulting Services", fontNormal, XBrushes.Black, col1X + 5, currentY + 17);
        gfx.DrawString("$5,000.00", fontNormal, XBrushes.Black, col2X + 5, currentY + 17);
        currentY += rowHeight;

        // Draw line
        gfx.DrawLine(XPens.Black, col1X, currentY, col2X + 100, currentY);
        currentY += 10;

        // Total
        gfx.DrawString("Total:", fontBold, XBrushes.Black, col1X + 5, currentY + 17);
        gfx.DrawString("$5,000.00", fontBold, XBrushes.Black, col2X + 5, currentY + 17);

        // Save
        document.Save("invoice_pdfsharp.pdf");
    }
}

Performance characteristics: fast execution with minimal overhead (~10-50ms per document), but high development cost at 40+ lines of positioning code per page. Memory efficient with streaming writes and low heap allocation. No startup delay. Every layout change requires coordinate recalculation.

IronPDF — HTML Template Invoice

// Install-Package IronPdf

using IronPdf;

public class InvoiceGenerator
{
    public void GenerateInvoice()
    {
        var htmlTemplate = @"
            <html>
            <head>
                <style>
                    body { font-family: Arial; margin: 50px; }
                    h1 { font-size: 24px; font-weight: bold; }
                    .info { margin: 20px 0; }
                    table { width: 100%; border-collapse: collapse; margin-top: 20px; }
                    th { background-color: #f0f0f0; padding: 10px; text-align: left; font-weight: bold; }
                    td { padding: 10px; border-bottom: 1px solid #ccc; }
                    .total { font-weight: bold; margin-top: 10px; }
                </style>
            </head>
            <body>
                <h1>INVOICE #2026-001</h1>
                <div class='info'>
                    <p>Date: 2026-02-11</p>
                    <p>Customer: Acme Corp</p>
                </div>
                <table>
                    <tr><th>Description</th><th>Amount</th></tr>
                    <tr><td>Consulting Services</td><td>$5,000.00</td></tr>
                </table>
                <p class='total'>Total: $5,000.00</p>
            </body>
            </html>";

        var renderer = new ChromePdfRenderer();
        var pdf = renderer.RenderHtmlAsPdf(htmlTemplate);
        pdf.SaveAs("invoice_ironpdf.pdf");
    }
}

Performance characteristics: higher per-document overhead with Chromium process startup adding ~200-500ms on first render (amortized with renderer reuse), but lower development cost at 25 lines of HTML/CSS. Layout changes are CSS edits. Designers update HTML/CSS without touching C# code.

How Do Merging and Signing Operations Compare?

Answer:

Both libraries support PDF merging, but the API surface differs significantly. PDFSharp requires iterating through each page and copying individually. IronPDF provides a single-call merge method.

PDFSharp — Manual Merge Implementation

// Install-Package PDFsharp

using System;
using System.IO;
using PdfSharp.Pdf;
using PdfSharp.Pdf.IO;

public class PdfMerger
{
    public void MergePdfs()
    {
        // Create output document
        using PdfDocument outputDocument = new PdfDocument();

        string[] inputFiles = { "report1.pdf", "report2.pdf", "report3.pdf" };

        foreach (string file in inputFiles)
        {
            // Open input document
            using PdfDocument inputDocument = PdfReader.Open(file, PdfDocumentOpenMode.Import);

            // Copy each page
            for (int pageIndex = 0; pageIndex < inputDocument.PageCount; pageIndex++)
            {
                PdfPage page = inputDocument.Pages[pageIndex];
                outputDocument.AddPage(page);
            }
        }

        // Save merged document
        outputDocument.Save("merged.pdf");
    }
}

Performance characteristics: fast merge with direct page copying and minimal overhead (~5-20ms per page). Memory efficient, streams pages without loading entire documents. No re-rendering — preserves original page content.

IronPDF — Single-Call Merge

// Install-Package IronPdf

using IronPdf;

public class PdfMerger
{
    public void MergePdfs()
    {
        var pdf1 = PdfDocument.FromFile("report1.pdf");
        var pdf2 = PdfDocument.FromFile("report2.pdf");
        var pdf3 = PdfDocument.FromFile("report3.pdf");

        var merged = PdfDocument.Merge(pdf1, pdf2, pdf3);
        merged.SaveAs("merged.pdf");
    }
}

PDFSharp uses lower overhead with more verbose code (~15 lines vs 5 lines). IronPDF provides a higher-level API with similar performance for merge operations. For advanced merge scenarios, see PDF merge documentation.

Frequently Asked Questions

Is PDFSharp still actively maintained?

Yes. PDFSharp is open-source under the MIT license with active maintenance. Version 6.2.3 (December 2025) added .NET 9/10 support. Commercial support is available from empira Software GmbH for enterprises needing SLAs. Pre-version-6 APIs differ significantly, but migration guides are available.

Can PDFSharp render HTML to PDF?

No. PDFSharp is a graphics API, not a template engine. There is no HTML-to-PDF rendering, no CSS support, no JavaScript execution, and no web fonts. You cannot reuse web designs or HTML templates. MigraDoc (a separate library) adds document abstractions but still lacks HTML rendering.

Why would I choose PDFSharp over IronPDF?

PDFSharp excels when you need pixel-perfect coordinate-level control — data visualizations, CAD output, or geometrically precise documents where explicit positioning is essential. The MIT license and zero per-document overhead (~10-50ms) also make it attractive for high-volume batch operations where development time is not a concern.

What triggers teams to migrate from PDFSharp to IronPDF?

Coordinate math burnout is the most common trigger. Teams start with simple single-page documents, then requirements expand to multi-column layouts, dynamic tables, and responsive page breaks. When the 10th "just move the footer down 5 points" ticket lands, developers evaluate alternatives. HTML skills reuse also drives migration for web-first teams — if your team knows CSS Grid and Flexbox, rewriting those layouts in coordinate-based code feels backwards.

How does batch performance compare between the two?

PDFSharp has lower per-document overhead (~10-50ms) with linear scaling and no startup delay. IronPDF has higher first-render overhead (~200-500ms for Chromium startup) but amortizes this with renderer reuse. For most teams, the development time savings — 4-8 hours for an HTML/CSS template versus 40+ hours of coordinate coding — dwarf per-document execution time differences.

Does IronPDF support modern CSS features that PDFSharp cannot?

Yes. IronPDF supports CSS3, Flexbox, Grid, media queries, modern JavaScript, and web fonts through its embedded Chromium engine. PDFSharp has no CSS support at all — it is a coordinate-based drawing API. For teams building modern document designs, PDFSharp's lack of CSS hits a ceiling quickly. For HTML template patterns and performance optimization, see IronPDF HTML rendering guide.

Which Should Your Team Choose?

PDFSharp delivers on its promise: complete control over PDF construction at the coordinate level. For teams building data visualizations, CAD output, or geometrically precise documents, this control is essential. The MIT license and active maintenance make it a sustainable open-source choice. If your documents are programmatically generated graphics — charts, diagrams, technical drawings — PDFSharp's API is designed for you.

Migration becomes necessary when coordinate-based positioning has become a maintenance burden exceeding any per-document performance gains, when your team's core competency is web development and rewriting HTML layouts in coordinates feels wasteful, when template-driven document generation (invoices, reports, letters) dominates your workload, or when design iteration speed matters more than raw execution speed.

IronPDF trades per-document performance for development velocity: HTML/CSS eliminates coordinate math, Chromium handles modern layout, templates enable designer collaboration. The performance profile shifts from "fastest per-document" to "fastest to market." For teams where developer hours cost more than server milliseconds, this is often the right trade.
