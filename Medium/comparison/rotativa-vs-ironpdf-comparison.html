Rotativa vs IronPDF in C#: Which Should Your Team Choose?

You are building an ASP.NET Core MVC application. Product owners want "Export to PDF" buttons on reports. You search ".NET MVC PDF" and Rotativa appears with clean examples showing return new ViewAsPdf(). It looks simple — and for the MVC-controller case, it is. The friction tends to surface later, in three places: deploying the wkhtmltopdf binary across platforms, configuring Azure App Service or container temp directories, and trying to generate PDFs from a background service only to find the API is built around ControllerContext. Rotativa is a focused wrapper for ASP.NET MVC controllers; what it does, it does cleanly, but its scope is narrower than a general PDF library.

Rotativa and IronPDF both generate PDFs from HTML in .NET, but they make fundamentally different architectural choices. Rotativa wraps wkhtmltopdf (an external Qt WebKit-based command-line tool) for MVC controller convenience. IronPDF embeds a Chromium rendering engine as a library component. This comparison covers deployment requirements, architectural constraints, and feature completeness to help your team decide which tool fits your stack.

What Is IronPDF?

IronPDF is a .NET library with an embedded Chromium rendering engine. It works in any .NET application context: MVC controllers, background services, console apps, Azure Functions, worker processes. You call ChromePdfRenderer.RenderHtmlAsPdf() from any code location, and IronPDF handles HTML parsing, CSS layout, JavaScript execution, and PDF generation in-process. No external binaries to manage, no command-line tool invocations, no temporary file handling. IronPDF runs on Windows, Linux, and macOS without platform-specific binary dependencies.

IronPDF supports .NET Core, .NET 5+, and .NET Framework 4.6.2+. Because it is a library, not a controller helper, it integrates with any architecture: microservices generating PDFs asynchronously, batch processors creating reports overnight, API endpoints streaming PDFs to clients, or background jobs rendering thousands of documents. See the HTML to PDF tutorial for implementation patterns.

How Does Rotativa Differ from IronPDF?

Rotativa (and Rotativa.AspNetCore) is an ASP.NET MVC library that wraps wkhtmltopdf, a Qt WebKit-based command-line tool for HTML-to-PDF conversion. Rotativa provides convenience methods like ViewAsPdf, UrlAsPdf, and ActionAsPdf that return ActionResult objects from MVC controllers. The library handles invoking wkhtmltopdf as an external process, passing HTML content via temporary files, and streaming the resulting PDF back to the HTTP response.

Rotativa is designed specifically for MVC controllers responding to web requests. It simplifies the common pattern of "render Razor view, convert to PDF, return to browser." However, this convenience comes with constraints: tight coupling to MVC request contexts, dependency on wkhtmltopdf binaries in specific file system locations, and limited support for non-web scenarios (background jobs, API services, console apps).

Rotativa.AspNetCore is a community-maintained third-party fork that continues to receive updates (v1.4.0 was released on 2024-11-06). The original Rotativa package for .NET Framework MVC 5 has not seen a release since 2017 (v1.7.3, 2017-09-29). The library inherits wkhtmltopdf's upstream status: the wkhtmltopdf GitHub repository was archived (read-only) on January 2, 2023, and the maintainers state no further releases are planned for its Qt WebKit rendering engine.

MVC-centric architecture: Rotativa's API is built around ActionResult types and MVC controller contexts. Generating PDFs outside controller actions (background jobs, console apps, Azure Functions) typically requires constructing a controller context manually or routing back through HTTP.

Generation-only scope: Rotativa generates PDFs from views; it does not provide APIs to merge, split, extract text, add watermarks programmatically, apply digital signatures, fill forms, or encrypt with passwords. For post-generation operations, you pair it with an additional library.

Rendering engine vintage: Rotativa uses wkhtmltopdf's Qt WebKit engine, which is based on a WebKit branch from the early 2010s. CSS Grid and Flexbox support is limited in this engine, and JavaScript ES6+ features may not execute reliably — verify against the version you ship.

Binary management: Rotativa requires wkhtmltopdf executables in a configured folder structure (typically wwwroot/Rotativa). These binaries are platform-specific (Windows .exe, Linux binaries with library dependencies, macOS variants). Shipping the correct binary for each target adds deployment steps.

File system writes: Rotativa writes temporary HTML files to disk, invokes wkhtmltopdf as an external process, and reads the resulting PDF from disk. In read-only file systems (some cloud environments, immutable containers), this can cause failures. Azure App Service, AWS Lambda, and Kubernetes pods with restricted write permissions typically require careful temp-directory configuration.

External process overhead: Every PDF generation spawns a wkhtmltopdf process. Process creation latency adds to per-document time. For high-throughput scenarios, this overhead compounds. Process cleanup must be reliable; leaked processes consume memory.

Synchronous API: Rotativa's BuildFile method is synchronous. For async controllers or high-concurrency scenarios, this blocks the calling thread during process invocation and file I/O.

Rotativa.AspNetCore has community support via GitHub issues. There is no commercial support offering or SLA for bug fixes — production support is via community channels.

How Does HTML-to-PDF Generation Compare?

Rotativa -- MVC Controller Action

Rotativa shines in its primary use case: returning PDFs from MVC controller actions.

using Microsoft.AspNetCore.Mvc;
using Rotativa.AspNetCore;
using Rotativa.AspNetCore.Options;
using System;
using System.Collections.Generic;

namespace MyApp.Controllers
{
    public class ReportController : Controller
    {
        public class ReportData
        {
            public string Title { get; set; }
            public DateTime GeneratedDate { get; set; }
            public List<ReportItem> Items { get; set; }
        }

        public class ReportItem
        {
            public string ProductName { get; set; }
            public int Quantity { get; set; }
            public decimal UnitPrice { get; set; }
            public decimal Total => Quantity * UnitPrice;
        }

        // Return PDF directly from controller action
        public IActionResult GeneratePdf()
        {
            var model = new ReportData
            {
                Title = "Q4 Sales Report",
                GeneratedDate = DateTime.Now,
                Items = new List<ReportItem>
                {
                    new ReportItem { ProductName = "Widget A", Quantity = 100, UnitPrice = 25.50m },
                    new ReportItem { ProductName = "Gadget B", Quantity = 50, UnitPrice = 47.99m }
                }
            };

            // Render Razor view as PDF and return to browser
            return new ViewAsPdf("ReportTemplate", model)
            {
                FileName = $"report-{DateTime.Now:yyyyMMdd}.pdf",
                PageSize = Size.A4,
                PageOrientation = Orientation.Portrait,
                PageMargins = new Margins(10, 10, 10, 10),
                // Pass additional wkhtmltopdf command-line switches
                CustomSwitches = "--enable-local-file-access"
            };
        }

        // For saving to disk instead of returning to browser
        public IActionResult SavePdfToDisk()
        {
            var model = new ReportData
            {
                Title = "Q4 Sales Report",
                GeneratedDate = DateTime.Now,
                Items = new List<ReportItem>
                {
                    new ReportItem { ProductName = "Widget A", Quantity = 100, UnitPrice = 25.50m }
                }
            };

            var pdf = new ViewAsPdf("ReportTemplate", model)
            {
                PageSize = Size.A4
            };

            // BuildFile materializes the PDF synchronously into a byte array
            byte[] pdfBytes = pdf.BuildFile(ControllerContext);
            System.IO.File.WriteAllBytes("output.pdf", pdfBytes);

            return Ok("PDF saved to disk");
        }
    }
}

Rotativa requires setup in Program.cs:

var builder = WebApplication.CreateBuilder(args);
builder.Services.AddControllersWithViews();

var app = builder.Build();
app.UseRouting();

// Configure Rotativa (points to wkhtmltopdf binary location)

app.MapControllerRoute(

Architectural notes: the API is centered on MVC controller contexts, so calling it from a background service or console app typically means constructing a controller context manually. BuildFile is synchronous, which blocks the calling thread during external-process invocation. The correct wkhtmltopdf binary must ship for each deployment platform. Read-only file systems (Azure App Service in some configurations, AWS Lambda, Kubernetes pods without writable temp mounts) require additional configuration. Post-generation operations like merging, watermarking, or signing are out of scope for Rotativa and need a separate library.

IronPDF -- Any Context (MVC, Background, Console)

IronPDF works in any .NET context. Here is the same report in an MVC controller:

using IronPdf;
using Microsoft.AspNetCore.Mvc;
using System;
using System.Collections.Generic;

namespace MyApp.Controllers
{
    public class ReportController : Controller
    {
        static ReportController()
        {
            IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
        }

        public class ReportData
        {
            public string Title { get; set; }
            public DateTime GeneratedDate { get; set; }
            public List<ReportItem> Items { get; set; }
        }

        public class ReportItem
        {
            public string ProductName { get; set; }
            public int Quantity { get; set; }
            public decimal UnitPrice { get; set; }
            public decimal Total => Quantity * UnitPrice;
        }

        // Generate PDF and return to browser
        public IActionResult GeneratePdf()
        {
            var model = new ReportData
            {
                Title = "Q4 Sales Report",
                GeneratedDate = DateTime.Now,
                Items = new List<ReportItem>
                {
                    new ReportItem { ProductName = "Widget A", Quantity = 100, UnitPrice = 25.50m },
                    new ReportItem { ProductName = "Gadget B", Quantity = 50, UnitPrice = 47.99m }
                }
            };

            // Build HTML (could also render Razor view to string)
            var html = $@"
                <!DOCTYPE html>
                <html>
                <head>
                    <style>
                        body {{ font-family: Arial, sans-serif; margin: 40px; }}
                        h1 {{ color: #333; }}
                        table {{ width: 100%; border-collapse: collapse; margin-top: 20px; }}
                        th, td {{ padding: 10px; border: 1px solid #ddd; text-align: left; }}
                        .total {{ text-align: right; font-weight: bold; }}
                    </style>
                </head>
                <body>
                    <h1>{model.Title}</h1>
                    <p>Generated: {model.GeneratedDate:yyyy-MM-dd}</p>
                    <table>
                        <thead>
                            <tr><th>Product</th><th>Qty</th><th>Price</th><th>Total</th></tr>
                        </thead>
                        <tbody>
                            {string.Join("", model.Items.Select(item => $@"
                            <tr>
                                <td>{item.ProductName}</td>
                                <td>{item.Quantity}</td>
                                <td>${item.UnitPrice:F2}</td>
                                <td class='total'>${item.Total:F2}</td>
                            </tr>"))}
                        </tbody>
                    </table>
                </body>
                </html>
            ";

            // Render HTML to PDF in-process
            var renderer = new ChromePdfRenderer();
            using var pdf = renderer.RenderHtmlAsPdf(html);

            // Return as file to browser
            return File(pdf.BinaryData, "application/pdf", $"report-{DateTime.Now:yyyyMMdd}.pdf");
        }

        // Async version for high-concurrency scenarios
        public async Task<IActionResult> GeneratePdfAsync()
        {
            var model = new ReportData
            {
                Title = "Q4 Sales Report",
                GeneratedDate = DateTime.Now,
                Items = new List<ReportItem>
                {
                    new ReportItem { ProductName = "Widget A", Quantity = 100, UnitPrice = 25.50m }
                }
            };

            var html = $@"<!DOCTYPE html><html><body><h1>{model.Title}</h1></body></html>";

            var renderer = new ChromePdfRenderer();
            using var pdf = await renderer.RenderHtmlAsPdfAsync(html);

            return File(pdf.BinaryData, "application/pdf", "report.pdf");
        }
    }
}

Deployment checklist for IronPDF: add the IronPdf NuGet package. That is it. No binaries, no dependencies, no configuration.

IronPDF also works outside MVC contexts. Here is a background service example:

public class ReportGenerationService : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            // Generate nightly reports
            var renderer = new ChromePdfRenderer();
            using var pdf = await renderer.RenderHtmlAsPdfAsync("<h1>Nightly Report</h1>");
            pdf.SaveAs($"reports/nightly-{DateTime.Now:yyyyMMdd}.pdf");

            await Task.Delay(TimeSpan.FromHours(24), stoppingToken);
        }
    }
}

This pattern is straightforward in IronPDF because the API does not require a controller context — the same code works in console apps, background services, and Azure Functions. Async support via RenderHtmlAsPdfAsync() enables non-blocking generation. The Chromium engine is bundled with the package, so there is no separate binary to ship. And merging, watermarking, signing, and encryption live in the same library — see the merge/split guide.

How Do Merging and Signing Compare?

Rotativa's scope is HTML-to-PDF generation. It does not expose APIs to merge, split, add watermarks programmatically, apply digital signatures, fill forms, or encrypt with passwords. For any post-generation operations, teams typically pair Rotativa with another library such as iText, PdfSharp, or IronPDF.

For watermarking specifically, the Rotativa pattern is to embed watermark styling directly in the HTML/CSS. The watermark becomes part of the rendered page content rather than a separate PDF annotation layer, so per-page variation or post-generation watermark application happens at the HTML layer rather than via a PDF API.

Rotativa -- No Built-in Watermark API

using Microsoft.AspNetCore.Mvc;
using Rotativa.AspNetCore;
using Rotativa.AspNetCore.Options;

public class ReportController : Controller
{
    public IActionResult GeneratePdfWithWatermark()
    {
        // Watermark applied via HTML/CSS as page content
        var html = @"
            <!DOCTYPE html>
            <html>
            <head>
                <style>
                    body { font-family: Arial; margin: 40px; position: relative; }
                    .watermark {
                        position: fixed;
                        top: 50%;
                        left: 50%;
                        transform: translate(-50%, -50%) rotate(-45deg);
                        font-size: 72px;
                        color: rgba(200, 200, 200, 0.3);
                        z-index: -1;
                        white-space: nowrap;
                    }
                </style>
            </head>
            <body>
                <div class='watermark'>CONFIDENTIAL</div>
                <h1>Quarterly Financial Report</h1>
                <p>This document contains sensitive information.</p>
            </body>
            </html>
        ";

        // Render the view that contains the styled watermark element
        return new ViewAsPdf
        {
            ViewName = "WatermarkTemplate",
            PageSize = Size.A4
        };
    }
}

IronPDF -- Built-in Watermark, Merge, and Sign APIs

IronPDF provides dedicated watermarking methods that apply true PDF annotation layers with configurable rotation, opacity, and positioning:

using IronPdf;

public class ReportController : Controller
{
    static ReportController()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
    }

    public IActionResult GeneratePdfWithWatermark()
    {
        // Render the report HTML
        var renderer = new ChromePdfRenderer();
        using var pdf = renderer.RenderHtmlAsPdf(@"
            <h1>Quarterly Financial Report</h1>
            <p>This document contains sensitive information.</p>
        ");

        // Apply watermark as a PDF annotation layer with rotation and opacity
        pdf.ApplyWatermark("<div style='font-size: 72px; color: #cccccc; font-weight: bold;'>CONFIDENTIAL</div>",
            rotation: -45,
            opacity: 30);

        return File(pdf.BinaryData, "application/pdf", "report-watermarked.pdf");
    }
}

Beyond watermarks, IronPDF supports merging via PdfDocument.Merge(), splitting and extracting pages, digital signatures via PdfDocument.Sign(), password protection via SecuritySettings.UserPassword/OwnerPassword, form filling, and text extraction via PdfDocument.ExtractAllText(). Rotativa supports none of these operations.

What Does the Full Feature Comparison Look Like?

  • Project status. Rotativa.AspNetCore is a community-maintained third-party fork (latest v1.4.0, 2024-11-06). IronPDF is commercially developed with regular releases. The underlying wkhtmltopdf GitHub repository was archived on January 2, 2023; IronPDF's Chromium engine is updated alongside upstream Chromium.
  • Support model. Rotativa support is via GitHub issues and community channels. IronPDF offers commercial engineer support via chat and email, with documentation covering tutorials, how-tos, and an API reference.
  • HTML engine. Rotativa renders via wkhtmltopdf's Qt WebKit engine, a branch with limited support for modern CSS Grid, Flexbox, and ES6+ JavaScript. IronPDF uses a current Chromium build with full HTML5, CSS3, and modern JavaScript.
  • External binary. Rotativa requires shipping the platform-specific wkhtmltopdf executable. IronPDF bundles its rendering engine via the NuGet package — no separate binary to ship.
  • Application context. Rotativa is built around MVC controllers and ControllerContext; calling it from background services, console apps, or Azure Functions typically requires workarounds. IronPDF works in any .NET context — controller, background service, console app, or function.
  • Async support. Rotativa's BuildFile is synchronous. IronPDF exposes RenderHtmlAsPdfAsync() for non-blocking generation.
  • PDF manipulation. Rotativa's scope is generation only; merging, splitting, watermarks, digital signatures, form filling, text extraction, and password protection are out of scope. IronPDF includes APIs for all of these operations in the same library.
  • Watermarks. Rotativa watermarks are HTML/CSS elements rendered as page content. IronPDF's ApplyWatermark applies a PDF annotation layer with configurable rotation and integer opacity 0–100.
  • Deployment. Rotativa deployments include the wkhtmltopdf binary, Linux library dependencies (X11, fontconfig, etc.), and a writable temp directory. IronPDF deploys as a standard NuGet dependency — no separate binary, no required temp writes for the basic path.
  • Cloud and container environments. Read-only file systems (some Azure App Service configurations, AWS Lambda, restricted Kubernetes pods) typically need extra configuration for Rotativa's temp-file workflow. IronPDF runs in standard .NET Docker images and on Azure App Service without additional configuration in most cases.
  • Platform coverage. Both libraries support .NET Core 3.1+, .NET 5+, and .NET Framework 4.6.2+. Linux support requires shipping Linux wkhtmltopdf binaries and their dependencies for Rotativa; IronPDF supports Linux and macOS natively.
  • Learning curve. Rotativa is straightforward for teams already on MVC — return a ViewAsPdf from an action. IronPDF is straightforward for HTML-driven scenarios — instantiate ChromePdfRenderer and call RenderHtmlAsPdf.

Frequently Asked Questions

Can I use Rotativa outside of an MVC controller?

Rotativa's API is built around ActionResult types and MVC controller contexts. Its convenience methods (ViewAsPdf, ActionAsPdf) depend on ControllerContext and MVC infrastructure. Background services, console apps, or microservices without MVC pipelines typically construct a controller context manually or route generation through an HTTP endpoint. IronPDF works in any .NET context without that step.

Does Rotativa support async PDF generation?

No. Rotativa's BuildFile method is synchronous. For async controllers or high-concurrency scenarios, this blocks threads during process invocation and file I/O, reducing scalability. IronPDF provides RenderHtmlAsPdfAsync() for non-blocking generation.

What are the deployment requirements for each library?

Rotativa requires wkhtmltopdf binaries in a specific folder structure, platform-specific binaries for each environment, Linux dependencies (libgdiplus, libfontconfig1, X11 libraries), file system write permissions for temp file creation, and process execution permissions. IronPDF requires only the NuGet package -- no additional binaries, dependencies, or configuration.

Does Rotativa support modern CSS like Grid and Flexbox?

Rotativa uses wkhtmltopdf's Qt WebKit engine, a branch from the early 2010s. CSS Grid and Flexbox support is limited in this engine, and JavaScript ES6+ may not execute reliably — verify against the version you ship. IronPDF uses an embedded Chromium engine with support for modern HTML5, CSS3, and JavaScript standards.

Can Rotativa merge PDFs or apply digital signatures?

No. Rotativa's scope is HTML-to-PDF generation; it does not provide APIs to merge, split, extract text, add watermarks programmatically, apply digital signatures, fill forms, or encrypt with passwords. Teams typically pair it with an additional library for those operations. IronPDF includes all of these capabilities in a single package: PdfDocument.Merge(), PdfDocument.Sign(), PdfDocument.ExtractAllText(), and SecuritySettings.UserPassword/OwnerPassword.

Will Rotativa work on Azure App Service or in Docker containers?

Rotativa can work in these environments with configuration. On Azure App Service, you ship the wkhtmltopdf binary and verify write permissions for the temp directory. In Docker, your base image includes wkhtmltopdf and its dependencies (libgdiplus, libfontconfig1, X11 libraries). In Kubernetes, you mount a writable temp directory. IronPDF runs on Azure App Service, in standard .NET Docker images, and in Kubernetes without those extra steps in most cases.

Which Should Your Team Choose?

Rotativa is a focused tool for ASP.NET MVC applications that need simple "Export to PDF" buttons on web pages. If your application is MVC-centric, you have file system write access for temp files, and your HTML stays within the CSS feature set the engine supports, Rotativa's convenience methods (ViewAsPdf, ActionAsPdf) provide quick integration. Rotativa.AspNetCore is community-maintained for its core use case.

The architectural fit matters most. When requirements expand to background services, console apps, Azure Functions, or serverless architectures, the controller-context dependency becomes friction. Modern CSS features such as Grid and Flexbox may render inconsistently in the wkhtmltopdf engine. Read-only file systems in some cloud environments require additional configuration. Post-generation operations (merging, watermarking, signing) live in separate libraries. And because the wkhtmltopdf repository was archived on January 2, 2023, the rendering engine itself will not gain new web-platform support over time.

IronPDF's embedded Chromium architecture is shaped around different trade-offs. It works in any .NET context (MVC, background services, console apps, functions), supports modern HTML5, CSS3, and JavaScript, ships its rendering engine via NuGet, and includes PDF manipulation features in a single package. The async API helps with high-concurrency scenarios, and commercial support is available for production environments. For teams building beyond MVC controller actions or deploying to modern cloud and container environments, IronPDF's library-based approach is worth evaluating.
