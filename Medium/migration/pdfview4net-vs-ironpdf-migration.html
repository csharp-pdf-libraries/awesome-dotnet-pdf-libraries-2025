Migrating from PDFView4NET to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

When the feature you need is not on the product page, you already know how the evaluation ends. PDFView4NET from O2 Solutions is, despite the name, exactly what the name says: a viewer, renderer, and print toolkit for WinForms and WPF. It ships as two NuGet packages — O2S.Components.PDFView4NET.Win and O2S.Components.PDFView4NET.WPF — and it loads, displays, prints, and extracts text from existing PDFs. Teams that adopted it for PDF viewing sometimes discover their requirements have expanded — they now need to generate PDFs from modern HTML/CSS, process documents in a server-side pipeline, or handle PDFs in a context where there is no UI. PDFView4NET was never built for any of that. O2 Solutions sells a separate product, PDF4NET, for programmatic creation, and even that ships no Chromium-class HTML renderer.

This guide covers migrating from PDFView4NET to IronPDF. It is useful for teams whose workload has shifted from viewing to generation, or who need to move PDF processing out of a UI layer into a service or API.

Why Should I Migrate Away from PDFView4NET?

The gap between a viewer-first toolkit and a Chromium-backed generation library is wider than it appears from the outside. Here are the issues teams commonly hit when their PDFView4NET usage needs to expand.

No HTML-to-PDF. PDFView4NET has no HtmlToPdfConverter, no HTML rendering API, no URL-to-PDF entry point. HTML rendering is out of scope for the toolkit. If your reports need to come from HTML/CSS templates, this is a hard wall — not a quality-of-rendering issue, but a missing feature.

Server-side and headless execution. The product is distributed as WinForms and WPF editions and is built around a viewer control. It does not run cleanly in web workers, Azure Functions, or Linux containers. If your architecture is moving toward hosted services, a desktop viewer toolkit becomes a liability.

ASP.NET Core integration. Wiring a WinForms/WPF component into a server-side PDF pipeline creates architectural friction. The paradigms simply do not align.

Cross-platform deployment. WinForms and WPF are Windows-only. Linux and Docker deployments are blocked entirely.

Bulk and automated processing. A viewer toolkit is not designed for batch PDF generation without UI context. If you need to produce hundreds of documents in a queue from templated HTML, this is the wrong tool.

Thin API surface for manipulation. Merge, split, watermark, encryption-on-create, and digital signatures are not part of PDFView4NET's documented API. Teams end up bolting on secondary libraries to fill the gaps — often O2's own PDF4NET, or PdfSharp/iTextSharp.

Thread model. UI controls require single-threaded access on the UI thread. Server-side PDF processing needs to be thread-safe or parallel-capable.

License model and pricing. PDFView4NET is sold per-developer per-edition (roughly US$699 per developer for either the WinForms or WPF edition, royalty-free runtime, with one year of support). Confirm current pricing at the vendor; the per-edition split matters if your organization needs both.

Maintenance cadence. O2 Solutions does still ship releases — the latest O2S.Components.PDFView4NET.Win at the time of writing is 11.3.30 — but the product scope is intentionally narrow.

How Do PDFView4NET and IronPDF Compare Feature by Feature?

  • Focus. PDFView4NET is a WinForms/WPF viewer/render/print toolkit centered on a PDFViewer control plus a PDFDocument object model. IronPDF is a Chromium-backed generation and manipulation library with no UI surface.
  • HTML-to-PDF. Not a PDFView4NET feature at all. IronPDF uses an embedded Chromium build, so modern CSS and JavaScript render the way designers expect.
  • Pricing. Both are commercial. PDFView4NET licensing is per developer per edition on o2sol.com; IronPDF licensing is on ironsoftware.com. Confirm current tiers at the source.
  • API style. PDFView4NET centers on a UI control event model (WinForms/WPF) with a non-visual PDFDocument for loading; IronPDF is a plain library API with no UI required.
  • Learning curve. Moderate for WinForms/WPF developers using PDFView4NET in its native habitat. IronPDF reads as ordinary .NET for most teams.
  • Package. PDFView4NET is O2S.Components.PDFView4NET.Win or O2S.Components.PDFView4NET.WPF (split by UI framework); IronPDF is the single IronPdf package.
  • Page indexing. Both are 0-based. The collection in PDFView4NET is document.Pages[i]; in IronPDF it is pdf.Pages[i].
  • Thread safety. The PDFView4NET viewer is UI-thread bound. ChromePdfRenderer can be called from any thread; consult current IronPDF guidance for high-volume concurrent use.

How Much Effort Does Each Feature Migration Require?

This is not a one-to-one API swap. PDFView4NET is built around a UI component for viewing and printing existing PDFs, and IronPDF is a server-side generation and manipulation library. The effort depends heavily on which features your codebase actually uses versus which are viewer-specific.

  • HTML string to PDF — New capability. Not present in PDFView4NET. IronPDF: ChromePdfRenderer.RenderHtmlAsPdfAsync().
  • URL to PDF — New capability. Not present in PDFView4NET. IronPDF: ChromePdfRenderer.RenderUrlAsPdfAsync().
  • Load existing PDF — Low. new PDFDocument(); document.Load(path) becomes PdfDocument.FromFile(path); the stream overload document.Load(stream) becomes PdfDocument.FromStream(stream).
  • Save PDF — Low. IronPDF: pdf.SaveAs(path). PDFView4NET does not write PDFs through a generic save API; it is a viewer.
  • Page rendering (display) — N/A. Native PDFView4NET viewer feature; IronPDF is not a viewer.
  • Annotations — High. Interactive viewer tools in PDFView4NET versus a programmatic IronPdf.Annotations API. Different use cases, not a 1:1 swap.
  • Form fields — High. PDFView4NET surfaces forms through the viewer for fill-in. IronPDF exposes them programmatically — read or write a field with pdf.Form.FindFormField(name). Verify the depth of AcroForm support against your form's complexity.
  • Merge PDFs — New capability. No first-class merge API in PDFView4NET; PdfDocument.Merge(pdf1, pdf2, ...) in IronPDF.
  • Watermark — New capability. No watermark API in PDFView4NET; pdf.ApplyWatermark(html) or the TextStamper / ImageStamper APIs in IronPDF.
  • Password protection — New capability. PDFView4NET can open password-protected PDFs through a constructor overload that accepts the password, but creating a new password-protected PDF is not part of its documented API. IronPDF uses pdf.SecuritySettings.
  • Text extraction — Low. document.Pages[i].ExtractText() per page becomes pdf.ExtractAllText() or pdf.ExtractTextFromPage(i).
  • Print — Low. document.Print() becomes pdf.Print(new PrintOptions { ... }).
  • Batch headless processing — New capability. Not what PDFView4NET is designed for; full support in IronPDF.

When Should I Migrate and When Should I Stay?

Not every team should migrate. Here is the decision matrix.

  • If you still need PDF viewing in a WinForms/WPF app and nothing else, PDFView4NET is appropriate. IronPDF is headless and does not ship a drop-in WinForms/WPF viewer control as a replacement.
  • If PDF generation from HTML for a server-side pipeline is your main need, switch — PDFView4NET cannot do HTML-to-PDF at all, and the desktop UI dependency rules out server hosting.
  • If you need both interactive viewing and programmatic generation, some teams run them side by side: PDFView4NET (or a WebView2-hosted viewer) for the UI, IronPDF for headless processing.
  • If you are moving from desktop to web/API architecture, switch. UI controls do not transfer to server-side hosting.

How Do I Set Up IronPDF and Replace PDFView4NET References?

Start by finding every PDFView4NET reference in your codebase, then swap the packages and namespaces.

Find All PDFView4NET References

# Search for PDFView4NET usages
rg -l "O2S\.Components\.PDFView4NET" --type cs
rg "PDFDocument|PDFViewer|PDFPage" --type cs -n

# Check project files (both editions)
grep -r "O2S.Components.PDFView4NET" *.csproj **/*.csproj 2>/dev/null

# Find UI-specific usage patterns that won't transfer server-side
rg "PDFViewer|\.Document\s*=" --type cs -n

Uninstall PDFView4NET and Install IronPDF

# Remove PDFView4NET (use whichever edition your project references)
dotnet remove package O2S.Components.PDFView4NET.Win
# or for WPF projects:
# dotnet remove package O2S.Components.PDFView4NET.WPF

# Install IronPDF
dotnet add package IronPdf

dotnet restore

Configure the License Key

using IronPdf;

// https://ironpdf.com/how-to/license-keys/
IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

Swap the Namespaces

Before:

using O2S.Components.PDFView4NET;

After:

using IronPdf;
using IronPdf.Rendering;

Basic Operation — Before (PDFView4NET PDFDocument)

using O2S.Components.PDFView4NET;
using System;
using System.IO;

class PdfBasicsBefore
{
    static void Main()
    {
        // Load and inspect an existing PDF (no UI required for the load itself)
        using (var fs = File.OpenRead("document.pdf"))
        {
            var document = new PDFDocument();
            document.Load(fs);
            Console.WriteLine($"Loaded {document.PageCount} pages");
            document.Close();
        }

        // PDFView4NET cannot generate a PDF from HTML — that feature does not
        // exist in the toolkit. The viewer/print/extract surface is the entire scope.
    }
}

Basic Operation — After (IronPDF Server-Side Generation)

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Headless — no UI required
var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Generated Report</h1></body></html>"
);
pdf.SaveAs("report.pdf");

Console.WriteLine($"Generated report.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

How Do the APIs Map Between PDFView4NET and IronPDF?

Because PDFView4NET centers on a viewer plus a small object model, and IronPDF is a generation/manipulation library, the mapping is not always one-to-one. Several PDFView4NET concepts have no IronPDF equivalent (interactive viewer display, page navigation events) and most of IronPDF's surface — HTML/URL-to-PDF, merge, watermarking, encryption-on-create, digital signatures — has no PDFView4NET equivalent in either direction.

Namespace Mapping

  • O2S.Components.PDFView4NET → IronPdf — core library
  • WinForms/WPF PDFViewer control → no IronPDF equivalent; host PDF bytes in a WebView2 control on desktop, or return them to the browser on the server
  • HTML rendering → IronPdf (ChromePdfRenderer) — new capability, not migrated from PDFView4NET
  • Annotation tools → IronPdf.Annotations — programmatic, not interactive

Core Class Mapping

  • new PDFDocument(); document.Load(path) → PdfDocument.FromFile(path) — load existing PDF from disk
  • new PDFDocument(); document.Load(stream) → PdfDocument.FromStream(stream) — load existing PDF from a stream
  • new PDFDocument(stream, password) → PdfDocument.FromFile(path, password) / PdfDocument.FromStream(stream, password) — load a password-protected PDF
  • document.Print() → pdf.Print(new PrintOptions { ... })
  • document.Close() → pdf.Dispose() (or wrap in using)
  • HTML/URL → PDF: no PDFView4NET equivalent; ChromePdfRenderer.RenderHtmlAsPdfAsync / RenderUrlAsPdfAsync in IronPDF

Document Loading Methods

  • Load from path: document.Load(path) → PdfDocument.FromFile(path)
  • Load from stream: document.Load(stream) → PdfDocument.FromStream(stream)
  • Load from bytes: PDFView4NET requires wrapping the byte array in a MemoryStream and calling Load; IronPDF exposes PdfDocument.FromBinaryData(bytes) directly
  • Generate from HTML: not available in PDFView4NET; renderer.RenderHtmlAsPdfAsync(html) in IronPDF
  • Generate from URL: not available in PDFView4NET; renderer.RenderUrlAsPdfAsync(url) in IronPDF

Page Operations

  • Page count: document.PageCount → pdf.PageCount
  • Access page: document.Pages[i] → pdf.Pages[i] (both 0-based)
  • Navigate to page: viewer-only in PDFView4NET; not applicable in headless IronPDF
  • Remove page: not part of the documented PDFView4NET API → pdf.RemovePages(index)
  • Extract text per page: document.Pages[i].ExtractText() → pdf.ExtractTextFromPage(i) or whole-doc pdf.ExtractAllText()

Merge / Split Operations

  • Merge: no merge API in PDFView4NET → PdfDocument.Merge(doc1, doc2, ...)
  • Split: no split API in PDFView4NET → pdf.CopyPage(i) / pdf.CopyPages(0, 1, 2, ...) — see the merge/split guide

What Do Complete Before-and-After Code Migrations Look Like?

Here are four migration examples covering the most common scenarios. The "before" snippets show real PDFView4NET APIs; where a feature is not available in PDFView4NET, the "before" block says so directly rather than fabricating an API.

HTML to PDF (New Capability)

Before — PDFView4NET has no HTML-to-PDF API. The toolkit can load and display existing PDFs, so teams that need HTML-to-PDF either pre-render with another tool and load the result for viewing, or pull in a separate library:

using O2S.Components.PDFView4NET;
using System;

class HtmlToPdfBefore
{
    static void Main()
    {
        // PDFView4NET cannot render an HTML string into a PDF.
        // The closest in-toolkit workflow is to load a PDF that was
        // produced upstream (by another product such as IronPDF or
        // O2's separate PDF4NET) and display or print it.
        var document = new PDFDocument();
        document.Load("pre-rendered-report.pdf");
        Console.WriteLine($"Loaded {document.PageCount} page(s) for viewing/printing.");
        document.Close();
    }
}

After:

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var html = @"
    <html>
    <head>
    <style>
        body { font-family: Arial, sans-serif; padding: 40px; }
        .header { font-size: 24px; font-weight: bold; border-bottom: 2px solid #333; }
        .section { margin-top: 20px; }
    </style>
    </head>
    <body>
        <div class='header'>Monthly Report — November 2026</div>
        <div class='section'>
            <h2>Summary</h2>
            <p>Report content generated from template data.</p>
        </div>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("monthly-report.pdf");

Console.WriteLine($"Generated monthly-report.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

Merge PDFs

Before — PDFView4NET has no merge API. The typical workaround is to load each input with PDFDocument and pull the pages into one output via a third-party library:

using O2S.Components.PDFView4NET;
using System;

class MergeBefore
{
    static void Main()
    {
        // PDFView4NET can load each input, but it does not expose a Merge()
        // entry point. Teams usually pair it with PdfSharp, iTextSharp, or
        // O2's separate PDF4NET to actually concatenate pages.
        var doc1 = new PDFDocument();
        doc1.Load("report-section1.pdf");
        var doc2 = new PDFDocument();
        doc2.Load("report-section2.pdf");

        Console.WriteLine($"Loaded {doc1.PageCount + doc2.PageCount} pages");
        // ...then hand off to a secondary library to write the merged PDF.

        doc1.Close();
        doc2.Close();
    }
}

After:

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Load PDFs from disk
var pdf1 = PdfDocument.FromFile("report-section1.pdf");
var pdf2 = PdfDocument.FromFile("report-section2.pdf");
var pdf3 = PdfDocument.FromFile("appendix.pdf");

// Merge: https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(pdf1, pdf2, pdf3);
merged.SaveAs("complete-report.pdf");

Console.WriteLine($"Merged: {merged.PageCount} total pages");

Watermark

Before — PDFView4NET has no watermark API. The viewer can display a PDF that already contains a watermark, but stamping is not part of the toolkit:

using O2S.Components.PDFView4NET;
using System;

class WatermarkBefore
{
    static void Main()
    {
        // PDFView4NET loads the document, but watermark/stamp generation
        // is not part of its API. Most teams reach for PdfSharp or iTextSharp,
        // which is one of the dependencies that consolidates after migration.
        var doc = new PDFDocument();
        doc.Load("document.pdf");

        Console.WriteLine($"Loaded {doc.PageCount} pages — watermark step needs a secondary library");
        doc.Close();
    }
}

After:

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var pdf = PdfDocument.FromFile("document.pdf");

// Watermark: https://ironpdf.com/how-to/custom-watermark/
var watermark = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronSoftware.Drawing.Color.DarkGray,
    Opacity = 20, // 0-100 scale
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("draft-document.pdf");
Console.WriteLine("Watermark applied");

Password Protection

Before — PDFView4NET can open a password-protected PDF through a PDFDocument constructor overload that takes the password, but creating a new password-protected PDF is not part of the documented API:

using O2S.Components.PDFView4NET;
using System;
using System.IO;

class PasswordBefore
{
    static void Main()
    {
        // Reading a password-protected PDF: PDFDocument accepts a password
        // through its constructor overloads.
        using var fs = File.OpenRead("protected.pdf");
        var doc = new PDFDocument(fs, "userpassword");
        Console.WriteLine($"Opened protected PDF ({doc.PageCount} pages)");
        doc.Close();

        // Creating a new password-protected PDF is not a documented PDFView4NET
        // capability — teams typically pair it with a secondary library.
    }
}

After:

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Confidential Document</h1></body></html>"
);

// Security: https://ironpdf.com/how-to/pdf-permissions-passwords/
pdf.SecuritySettings.UserPassword = "open123";
pdf.SecuritySettings.OwnerPassword = "admin456";
pdf.SecuritySettings.AllowUserCopyPasteContent = false;
pdf.SecuritySettings.AllowUserPrinting = PdfPrintSecurity.FullPrintRights;
pdf.SecuritySettings.AllowUserEdits = PdfEditSecurity.NoEdit;

pdf.SaveAs("protected.pdf");
Console.WriteLine("Created protected.pdf");

What Are the Most Common Migration Issues and How Do I Fix Them?

These are the issues I see most often when teams move from a viewer toolkit to a server-side generation library.

"The control requires UI thread access." The PDFView4NET viewer throws thread access exceptions when called from a background thread or async context. WinForms and WPF UI controls are thread-affine. This pattern does not transfer to server-side code. Replace the viewer control with IronPDF's library API — there is no adapter pattern here.

// The UI control pattern doesn't transfer:
// Task.Run(() => pdfViewer.Document = doc); // throws cross-thread exception

// IronPDF has no UI thread requirement:
var pdf = await renderer.RenderHtmlAsPdfAsync(html); // runs on any thread

"There's no built-in viewer in IronPDF." True — IronPDF is headless. To display a PDF on a desktop UI after migration, host the bytes in a WebView2 or WebBrowser control. On the server, return the bytes to the browser, which renders inline.

// Desktop (WPF/WinForms): host PDF in WebView2
// (Microsoft.Web.WebView2 NuGet)
webView2.Source = new Uri(System.IO.Path.GetFullPath("output.pdf"));

// ASP.NET Core: return PDF inline
return File(pdf.BinaryData, "application/pdf");

// System viewer fallback
System.Diagnostics.Process.Start(new ProcessStartInfo
{
    FileName = "output.pdf",
    UseShellExecute = true
});

"PDF generation worked in development but fails in IIS/Azure." IronPDF throws an exception in a hosted environment that does not occur locally. Common causes: license key not set in production, Azure-specific configuration needed, temp directory permissions, or missing system fonts in containers.

// 1. License key not set in production environment
// Verify environment variable is set: IRONPDF_LICENSE_KEY
// https://ironpdf.com/how-to/license-keys/

// 2. Azure/cloud environment — use IronPDF's Azure guidance
// https://ironpdf.com/how-to/azure/

// 3. Temp directory permissions
// IronPDF uses the system temp directory for Chromium working files
// Confirm the application pool identity has write access to %TEMP%

// 4. Missing system fonts in container
// Test with web fonts loaded via <link> or base64 @font-face
// rather than system font names

var renderer = new ChromePdfRenderer();

"Text extraction returns garbled output or empty strings." This is common with scanned PDFs (image-based, no text layer) or PDFs with embedded font subsets where character mapping is non-standard.

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var pdf = PdfDocument.FromFile("document.pdf");

// Check if PDF has extractable text layer
var text = pdf.ExtractAllText();
Console.WriteLine($"Extracted {text.Length} characters");

if (string.IsNullOrWhiteSpace(text))
{
    Console.WriteLine("PDF may be scanned/image-based — OCR required");
    // IronPDF does not include OCR natively — use IronOCR for scanned documents
    // https://ironsoftware.com/csharp/ocr/
}

// Extract per-page for more granular control
// var pageText = pdf.ExtractTextFromPage(0);
// https://ironpdf.com/how-to/extract-text-and-images/

"Rendering times out on complex HTML." RenderHtmlAsPdfAsync hangs or times out on pages with heavy JavaScript or async loading.

using IronPdf;

var renderer = new ChromePdfRenderer();

// Wait for JavaScript / network to settle before rendering
renderer.RenderingOptions.EnableJavaScript = true;
renderer.RenderingOptions.WaitFor.JavaScript(2000);   // wait up to 2s for JS

var pdf = await renderer.RenderHtmlAsPdfAsync(heavyHtml);

What Should I Know About Architecture, Threading, and Performance?

The most important distinction: PDFView4NET is a UI toolkit, IronPDF is a server-side library. If your codebase passes PDF content through a UI form context (events, control state, data binding), migrating to IronPDF requires restructuring that flow. You are moving PDF operations out of a UI layer into a service layer. That is a larger refactor than a typical library swap.

If the PDF-related logic is well-separated from the UI already, this is a low-risk migration. If it is tightly coupled to form event handlers, scope accordingly.

Annotation API differences. PDFView4NET's annotation tools are interactive — the user clicks to add annotations in the viewer. IronPDF's annotation API is programmatic. These are different use cases. If users need to interactively annotate documents, IronPDF alone does not replace the viewer. If you need to add annotations from code, IronPDF supports this. See the annotations guide.

Thread model. IronPDF does not require a UI thread. ChromePdfRenderer can be called from any thread. This enables patterns that PDFView4NET controls cannot support:

// Background processing pattern — not possible with a UI control
await Task.Run(async () =>
{
    var renderer = new ChromePdfRenderer();
    using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    pdf.SaveAs(outputPath);
});

Server-side batch processing. IronPDF renders without a display context. No monitor, no message pump, no repaint events. This is what makes it appropriate for server-side use:

using IronPdf;
using System;
using System.Diagnostics;
using System.Linq;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Server-side batch processing — not possible with a viewer control
var htmlDocuments = new[]
{
    "<html><body><h1>Document 1</h1></body></html>",
    "<html><body><h1>Document 2</h1></body></html>",
    "<html><body><h1>Document 3</h1></body></html>",
};

var sw = Stopwatch.StartNew();

// Parallel rendering
var tasks = htmlDocuments.Select(async html =>
{
    var renderer = new ChromePdfRenderer();
    return await renderer.RenderHtmlAsPdfAsync(html);
});

var pdfs = await Task.WhenAll(tasks);
sw.Stop();

Console.WriteLine($"Rendered {pdfs.Length} docs in {sw.Elapsed.TotalMilliseconds:F0}ms");

// Cleanup
foreach (var pdf in pdfs) pdf.Dispose();

Memory management. IronPDF's PdfDocument implements IDisposable. Always use using to release resources promptly in high-volume scenarios:

using IronPdf;
using System.IO;

var renderer = new ChromePdfRenderer();

using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
// Save to MemoryStream for downstream use without holding file handles
// https://ironpdf.com/how-to/pdf-memory-stream/
using var ms = new MemoryStream();
pdf.Stream.CopyTo(ms);
var pdfBytes = ms.ToArray();
// pdf disposed here — ms holds the bytes

Frequently Asked Questions

Can IronPDF replace the PDFView4NET viewer in my WinForms application?

Not as a drop-in control. IronPDF is headless — it does not ship a WinForms/WPF PdfViewer equivalent. The IronPDF docs recommend hosting PDF bytes in a WebView2 or WebBrowser control on desktop, returning bytes to the browser on the server, or shelling out to the system viewer. If your viewer needs are basic (display, zoom, page navigation), WebView2 covers those. If you depend on PDFView4NET's interactive annotation tools or specific event hooks, plan to keep PDFView4NET on the desktop side and use IronPDF for headless work.

Do I need to rewrite my entire codebase to migrate?

It depends on how tightly your PDF logic is coupled to the viewer control. If your PDF operations (load, print, text extraction) are already in a separate service layer, the migration is a straightforward API swap. If they are embedded in WinForms event handlers and control state, you will need to extract that logic into a service layer first.

Does IronPDF work on Linux and in Docker containers?

Yes. Unlike PDFView4NET, which requires Windows Forms or WPF (Windows-only), IronPDF runs on .NET 6/7/8/9 across Windows and Linux. This is often a primary driver for migration — teams moving from desktop applications to containerized deployments.

What happens to my PDFView4NET annotations after migration?

PDFView4NET's annotation tools are interactive (user-driven). IronPDF's annotation API is programmatic. Existing annotations embedded in PDF files will still be present when loaded by IronPDF. However, the interactive annotation workflow (click to add, drag to resize) does not transfer — that is a viewer feature.

Can IronPDF handle password-protected PDFs that were created with another tool?

Yes. IronPDF can open password-protected PDFs regardless of which library created them, as long as you provide the correct password. You can also create new password-protected PDFs using pdf.SecuritySettings. See the security guide.

How do I handle text extraction from scanned PDFs after migration?

IronPDF's pdf.ExtractAllText() works on PDFs with a text layer. For scanned or image-based PDFs, you need OCR. IronPDF does not include OCR natively, but IronOCR (a separate Iron Software product) handles this. If your PDFView4NET workflow included text extraction from scanned documents using a secondary library, evaluate whether IronOCR consolidates that dependency.

Conclusion

The scoping question that saves the most time: draw a line between PDFView4NET features used for display and printing (which IronPDF replaces only by hosting bytes in WebView2 or returning them to a browser) and features your team actually wants to add (HTML-to-PDF, merge, watermark, encrypt-on-create, headless batch). If most of the usage is display, the migration scope is small but the architectural boundary is real. If the requirements have outgrown viewing, the migration is straightforward.

PDFView4NET is a solid viewer/render/print toolkit for what it was designed to do. But if your requirements have outgrown viewing — if you need HTML-to-PDF generation, headless batch processing, cross-platform deployment, or server-side security and merge operations — IronPDF is built for that workload.
