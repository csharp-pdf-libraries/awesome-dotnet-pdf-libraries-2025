Migrating from GdPicture to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Would a .NET Team Migrate Away from GdPicture?

There is a particular kind of technical debt that accumulates quietly: the imaging SDK you licensed four years ago because you needed PDF thumbnail generation. Then someone added barcode scanning. Then OCR. Then PDF signing. Now you have a GdPicture.NET license that touches seven different workflows, the API has surface area in dozens of files, and when renewal comes, the conversation is less "should we renew" and more "can we even untangle this."

This guide is for the narrower scenario: you want to replace GdPicture.NET's PDF generation and manipulation features specifically, using IronPDF. If your GdPicture usage is purely imaging (TIFF processing, scanning, barcode, OCR), this article is less relevant — IronPDF does not replace those. If your primary pain is HTML-to-PDF, merge, watermark, and document security, read on.

Teams running GdPicture.NET for PDF work specifically, not imaging, commonly run into these friction points:

First, paying for breadth you do not use. GdPicture.NET ships as a full document-imaging suite — OCR, barcode, scanning, image processing — and the license cost reflects that whether you touch those modules or not. Second, the API is shaped around imaging concepts: GdPictureStatus return codes on every call, 1-indexed pages, manual Dispose() or Release(), and a version-stamped namespace (GdPicture14) that forces global rewrites on major upgrades. Third, ORPALIS (the original GdPicture vendor) was folded into Nutrient (formerly PSPDFKit), and documentation now spans two domains — gdpicture.com and nutrient.io — which slows down anyone trying to look up an exact method signature.

Beyond those, there are practical concerns around deployment assembly count, thread-safety model, upgrade paths across major versions, team familiarity for new engineers, and HTML/CSS rendering fidelity — GdPicture's HTML-to-PDF path exists (GdPictureDocumentConverter.LoadFromHTMLString) but is not Chromium-based, so modern CSS and JavaScript-heavy pages render less reliably than they would through IronPDF's Chromium renderer.

How Much Effort Does Each Feature Migration Require?

Not every part of a GdPicture integration maps to IronPDF. Some features translate directly. Others do not apply at all. Here is the effort breakdown I have seen across real migrations.

  • HTML string to PDF — Low. Direct method mapping. converter.LoadFromHTMLString + SaveAsPDF becomes renderer.RenderHtmlAsPdf + pdf.SaveAs. Expect rendering differences on modern CSS — IronPDF's Chromium engine generally renders closer to a browser.
  • URL to PDF — Low. Direct method mapping; LoadFromURL becomes renderer.RenderUrlAsPdf.
  • PDF merge — Low. pdf1.MergePages(pdf2) becomes the static PdfDocument.Merge(...).
  • PDF split — Low. ClonePage in a loop becomes pdf.CopyPage(index) or pdf.CopyPages(...).
  • Watermarking — Low. The coordinate-based SetTextColor / SetTextSize / DrawText loop collapses into a single pdf.ApplyWatermark call with HTML.
  • Password protection — Medium. SaveToFile with a long parameter list becomes pdf.SecuritySettings property assignment.
  • Text extraction — Low. Page-by-page GetPageText becomes pdf.ExtractAllText() or ExtractTextFromPage(i).
  • Form fields — Medium. Index-based GetFormFieldTitle/SetFormFieldValue becomes name-keyed pdf.Form.Fields["FieldName"].Value.
  • Digital signatures — Medium-High. Different certificate-handling model; verify your specific PAdES/timestamp requirements against IronPDF's signing API.
  • OCR — Not replaced. IronPDF does not include OCR. If you use GdPicture's OCR module, you either retain GdPicture for that path or adopt the separate IronOCR product.
  • Barcode — Not replaced. IronPDF does not read or write barcodes. Retain GdPicture or adopt the separate IronBarcode product.
  • Image processing / scanning / TIFF — Not replaced. IronPDF is a PDF library, not an imaging suite. Retain GdPicture for these workflows.

The key insight: OCR, barcode, and general imaging are out of scope entirely. IronPDF does not replace those GdPicture modules. If you use them, you will either retain GdPicture for imaging or pair IronPDF with IronOCR / IronBarcode as separate products.

What Does the Decision Matrix Look Like for Different Business Scenarios?

Not every team should migrate, and not every team should migrate everything at once. Here is how I think about it.

  • If your usage is PDF-only (HTML/URL to PDF, merge, watermark, security), the IronPDF migration is well-scoped and worth doing.
  • If your pipeline mixes PDF generation with OCR or barcode, split the responsibilities: IronPDF for the PDF layer, retain GdPicture for OCR/barcode — or adopt IronOCR/IronBarcode as companions.
  • If you have scanning or TIFF workflows interleaved with PDF, retain GdPicture. IronPDF will not replace imaging features; forcing a full-stack swap is the wrong move here.
  • If you are starting greenfield on .NET 8 and only need PDF, IronPDF is a reasonable default — evaluate against your specific requirements (signing depth, PDF/A compliance, deployment platform) before committing.

How Do the Two Libraries Compare Side by Side?

Before diving into code, it helps to see the structural differences at a glance. GdPicture.NET is a broad imaging and document SDK where PDF is one module among many. IronPDF is built specifically for PDF work.

  • Focus. GdPicture.NET is a broad imaging platform — scanning, OCR, barcode, image processing, and PDF. IronPDF is a focused PDF library centered on HTML-to-PDF rendering and document manipulation.
  • Pricing. GdPicture is sold as a modular commercial license; cost scales with which modules (PDF, OCR, barcode, etc.) you license. IronPDF is a single commercial license covering the PDF library.
  • API style. GdPicture is component- and handle-based — GdPicturePDF and GdPictureDocumentConverter objects, status-code returns, explicit Dispose. IronPDF is fluent and renderer-centric, with exception-based error handling and standard IDisposable.
  • Learning curve. GdPicture's full surface is large; the PDF-specific paths are moderate. IronPDF's surface is narrower and more familiar to anyone comfortable with HTML/CSS.
  • HTML rendering. GdPicture's GdPictureDocumentConverter renders HTML internally — adequate for simple markup but not Chromium-grade. IronPDF treats HTML-to-PDF as a primary design goal and ships with an embedded Chromium engine.
  • Page indexing. GdPicture is 1-based (SelectPage(1) is the first page). IronPDF is 0-based (Pages[0]). This is the single most common source of off-by-one bugs during migration.
  • Thread safety. GdPicture generally requires manual synchronization on shared instances. IronPDF's recommended pattern is one renderer per thread, with thread-safe document operations.
  • Namespace. GdPicture14 includes the major-version number, so upgrading GdPicture forces a global namespace rewrite. IronPDF uses the stable IronPdf namespace.

How Do I Handle License Configuration and Basic Setup?

The first thing to change is license initialization. GdPicture uses a static LicenseManager.RegisterKEY call. IronPDF uses a single static property.

GdPicture License Setup (Before)

using GdPicture14;

// Must be called before any GdPicture operations,
// typically once at application startup.
LicenseManager.RegisterKEY("YOUR-LICENSE-KEY");

IronPDF License Setup (After)

using IronPdf;

// Set once at application startup.
// See: https://ironpdf.com/how-to/license-keys/
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// Alternative: via environment variable IronPdf__LicenseKey
// or appsettings.json under "IronPdf" : { "LicenseKey": "..." }

Namespace imports simplify considerably. GdPicture's namespace includes the major version (GdPicture14), so upgrading the SDK is a global rewrite. IronPDF consolidates to a small, stable set.

GdPicture Namespaces (Before)

using GdPicture14;           // version-stamped; changes on major upgrades
using GdPicture14.PDF;       // PDF-specific types
using GdPicture14.Imaging;   // only if image processing is used

IronPDF Namespaces (After)

using IronPdf;
using IronPdf.Rendering;     // for ChromePdfRenderOptions
using IronPdf.Editing;       // for Stamper types

How Do I Migrate HTML-to-PDF Rendering?

This is typically the highest-value migration target. GdPicture exposes HTML-to-PDF through GdPictureDocumentConverter, with a status-code return on every call. IronPDF makes HTML-to-PDF a primary design goal with a Chromium-based renderer and exception-based error handling.

GdPicture HTML to PDF (Before)

using GdPicture14;
using System;

class Program
{
    static void Main()
    {
        LicenseManager.RegisterKEY("YOUR-LICENSE-KEY");

        using (GdPictureDocumentConverter converter = new GdPictureDocumentConverter())
        {
            string html = "<html><body><h1>Report</h1><p>Quarterly summary.</p></body></html>";
            GdPictureStatus status = converter.LoadFromHTMLString(html);

            if (status == GdPictureStatus.OK)
            {
                status = converter.SaveAsPDF("output.pdf");
                if (status != GdPictureStatus.OK)
                {
                    Console.WriteLine($"Save error: {status}");
                }
            }
            else
            {
                Console.WriteLine($"Load error: {status}");
            }
        }
    }
}

IronPDF HTML to PDF (After)

using IronPdf;

// See: https://ironpdf.com/how-to/html-string-to-pdf/
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

string html = "<html><body><h1>Report</h1><p>Quarterly summary.</p></body></html>";
using var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");
Console.WriteLine($"Generated {pdf.PageCount} page(s)");

How Do I Migrate PDF Merge Operations?

GdPicture's merge is an instance method on the base document — load the second document, then call MergePages to append it. IronPDF provides a static Merge method that handles a list of documents in a single call.

GdPicture Merge (Before)

using GdPicture14;

class MergeExample
{
    static void Main()
    {
        LicenseManager.RegisterKEY("YOUR-LICENSE-KEY");

        using (GdPicturePDF pdf1 = new GdPicturePDF())
        using (GdPicturePDF pdf2 = new GdPicturePDF())
        {
            pdf1.LoadFromFile("file1.pdf", false);
            pdf2.LoadFromFile("file2.pdf", false);

            pdf1.MergePages(pdf2);
            pdf1.SaveToFile("merged.pdf");
        }
    }
}

IronPDF Merge (After)

using IronPdf;
using System.Collections.Generic;

// See: https://ironpdf.com/how-to/merge-or-split-pdfs/
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf1 = PdfDocument.FromFile("file1.pdf");
var pdf2 = PdfDocument.FromFile("file2.pdf");

var merged = PdfDocument.Merge(new List<PdfDocument> { pdf1, pdf2 });
merged.SaveAs("merged.pdf");
Console.WriteLine($"Merged: {merged.PageCount} pages total");

How Do I Migrate Watermarking?

GdPicture watermarking iterates pages, selects each one with SelectPage, sets text color and size, then calls DrawText at fixed coordinates. IronPDF replaces that whole loop with a single ApplyWatermark call that takes an HTML fragment, an integer opacity (0–100), and alignment values.

GdPicture Watermark (Before)

using GdPicture14;
using System.Drawing;

class WatermarkExample
{
    static void Main()
    {
        LicenseManager.RegisterKEY("YOUR-LICENSE-KEY");

        using (GdPicturePDF pdf = new GdPicturePDF())
        {
            pdf.LoadFromFile("input.pdf", false);

            for (int i = 1; i <= pdf.GetPageCount(); i++)  // 1-indexed
            {
                pdf.SelectPage(i);
                pdf.SetTextColor(Color.Red);
                pdf.SetTextSize(48);
                pdf.DrawText("CONFIDENTIAL", 200, 400);
            }

            pdf.SaveToFile("watermarked.pdf");
        }
    }
}

IronPDF Watermark (After)

using IronPdf;
using IronPdf.Editing;

// See: https://ironpdf.com/how-to/custom-watermark/
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("input.pdf");

pdf.ApplyWatermark(
    "<h1 style='color:red;'>CONFIDENTIAL</h1>",
    50,                              // opacity 0–100 (integer)
    VerticalAlignment.Middle,
    HorizontalAlignment.Center);

pdf.SaveAs("watermarked.pdf");

How Do I Migrate Password Protection and Security?

GdPicture encrypts at save time: SaveToFile takes the encryption algorithm, both passwords, and a long list of permission booleans as positional arguments. IronPDF separates the two — set passwords and permissions on the SecuritySettings property, then save normally.

GdPicture Password Protection (Before)

using GdPicture14;

class SecurityExample
{
    static void Main()
    {
        LicenseManager.RegisterKEY("YOUR-LICENSE-KEY");

        using (GdPicturePDF pdf = new GdPicturePDF())
        {
            GdPictureStatus status = pdf.LoadFromFile("input.pdf", false);
            if (status != GdPictureStatus.OK) return;

            // Parameters: path, encryption, userPwd, ownerPwd,
            //             canPrint, canCopy, canModify, canAddNotes,
            //             canFillForms, canExtract, canAssemble, canPrintHQ
            pdf.SaveToFile(
                "protected.pdf",
                PdfEncryption.PdfEncryption256BitAES,
                "user123",
                "owner456",
                true, false, false, false, true, false, false, true);
        }
    }
}

IronPDF Password Protection (After)

using IronPdf;
using IronPdf.Security;

// See: https://ironpdf.com/how-to/pdf-permissions-passwords/
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("input.pdf");

pdf.SecuritySettings.UserPassword  = "user123";
pdf.SecuritySettings.OwnerPassword = "owner456";
pdf.SecuritySettings.AllowUserPrinting         = PdfPrintSecurity.FullPrintRights;
pdf.SecuritySettings.AllowUserCopyPasteContent = false;
pdf.SecuritySettings.AllowUserEdits            = PdfEditSecurity.NoEdit;
pdf.SecuritySettings.AllowUserFormData         = true;

pdf.SaveAs("protected.pdf");

What Are the Critical Technical Differences to Watch For?

Three things will bite you if you are not careful during migration.

First, page indexing. GdPicture.NET uses 1-based page indexing in most operations (e.g., SelectPage(1) for the first page). IronPDF uses 0-based indexing. This is the most common source of off-by-one bugs during migration.

// GdPicture:
pdf.SelectPage(1); // page 1 = first page

// IronPDF:
var firstPage = pdf.Pages[0]; // index 0 = first page
var lastPage  = pdf.Pages[pdf.PageCount - 1];

Second, error handling. GdPicture returns a GdPictureStatus enum from most operations. IronPDF throws exceptions. You need to replace status-code checks with try/catch blocks.

// IronPDF: always use try/catch for file I/O and rendering
try
{
    using var pdf = PdfDocument.FromFile("maybe-missing.pdf");
}
catch (System.IO.FileNotFoundException ex)
{
    Console.Error.WriteLine($"File not found: {ex.Message}");
}
catch (IronPdf.Exceptions.PdfException ex)
{
    Console.Error.WriteLine($"PDF error: {ex.Message}");
}

Third, disposal patterns. GdPicture requires explicit Dispose() on its objects. IronPDF's PdfDocument implements IDisposable, so use using blocks consistently. This is actually simpler, but you need to ensure every document object is wrapped.

What About Performance in Production?

ChromePdfRenderer carries initialization overhead. For batch processing, construct once and reuse across renders.

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

// Process batch
foreach (var task in renderQueue)
{
    using var pdf = renderer.RenderHtmlAsPdf(task.Html);
    pdf.SaveAs(task.OutputPath);
}

For web API or async service contexts, use the async rendering path to avoid blocking thread pool threads.

var renderer = new ChromePdfRenderer();
using var pdf = await renderer.RenderHtmlAsPdfAsync(htmlContent);
await File.WriteAllBytesAsync(outputPath, pdf.BinaryData);

For pipelines processing many large PDFs, dispose each PdfDocument promptly. Holding many open simultaneously increases GC pressure.

What Does the Full Feature Mapping Look Like?

Here are the complete API mappings for reference during your migration.

Namespaces.

  • GdPicture14 → IronPdf — core types
  • GdPicture14.PDF → IronPdf.Rendering — rendering options
  • GdPicture security types → IronPdf.Security — document security

Classes.

  • GdPicturePDF → PdfDocument — main document object
  • GdPictureDocumentConverter → ChromePdfRenderer — HTML-to-PDF rendering
  • Page operations on GdPicturePDF → PdfDocument.Pages[n] — page access
  • Security methods on GdPicturePDF → PdfDocument.SecuritySettings — encryption
  • LicenseManager → IronPdf.License — license management
  • GdPictureStatus → exceptions — error handling

Common operations.

  • Load from file. pdf.LoadFromFile(path, false) → PdfDocument.FromFile(path)
  • Load from stream. pdf.LoadFromStream(stream) → PdfDocument.FromStream(stream)
  • Load HTML string. converter.LoadFromHTMLString(html) → renderer.RenderHtmlAsPdf(html)
  • Load from URL. converter.LoadFromURL(url) → renderer.RenderUrlAsPdf(url)
  • Save. pdf.SaveToFile(path) / converter.SaveAsPDF(path) → pdf.SaveAs(path)
  • Page count. pdf.GetPageCount() → pdf.PageCount
  • Select / get page. pdf.SelectPage(i) (1-based) → pdf.Pages[i] (0-based)
  • Page width / height. pdf.GetPageWidth() / GetPageHeight() → page.Width / page.Height
  • Merge. pdf1.MergePages(pdf2) → PdfDocument.Merge(new List<PdfDocument> { ... })
  • Extract pages. pdf.ClonePage(...) in a loop → pdf.CopyPage(i) or pdf.CopyPages(indices)
  • Text extraction. pdf.GetPageText(i) → pdf.ExtractTextFromPage(i) or pdf.ExtractAllText()
  • Form field value. pdf.SetFormFieldValue(idx, value) → pdf.Form.Fields["Name"].Value = value
  • Watermark. SelectPage + DrawText loop → pdf.ApplyWatermark(html, opacity, vAlign, hAlign)

Frequently Asked Questions

Can I keep GdPicture for imaging and use IronPDF for PDF at the same time?

Yes, and this is often the right approach. GdPicture's strengths are in imaging, scanning, OCR, and barcode, which IronPDF does not replace. If your pipeline uses both imaging and PDF features, split the responsibilities: IronPDF for PDF generation, merge, watermark, and security; GdPicture for everything imaging-related. Both can coexist in the same .NET project without conflict.

What about page indexing differences between the two libraries?

GdPicture.NET uses 1-based page indexing in many operations, while IronPDF uses 0-based indexing. During migration, audit every page index reference. A SelectPage(1) call in GdPicture becomes Pages[0] in IronPDF. This is the single most common source of off-by-one bugs I see in migrations from GdPicture.

How do I handle GdPicture's status-code error pattern in IronPDF?

GdPicture returns integer status codes from many operations, where you check for success after each call. IronPDF throws exceptions instead. Replace your status-code checking patterns with try/catch blocks. This is actually a simplification in most cases, since you no longer need to check return values after every single API call.

What is the recommended approach for batch PDF rendering performance?

ChromePdfRenderer carries initialization overhead because it spins up a Chromium instance. For batch processing, construct one renderer and reuse it across all renders in your queue. For web API contexts, use the async rendering path (RenderHtmlAsPdfAsync) to avoid blocking thread pool threads. Dispose each PdfDocument promptly after use to manage memory pressure.

How Should I Make the Migration Decision?

The GdPicture.NET migration story has a useful structural property: GdPicture's breadth (imaging, OCR, barcode) is also the natural migration boundary.

Teams that used GdPicture primarily for PDF generation find the migration scope well-contained. The core operations, HTML-to-PDF, merge, watermark, and password protection, all map to IronPDF with low to medium effort. The API is more focused, the deployment footprint shrinks, and the learning curve for new team members drops.

Teams that used GdPicture for imaging and PDF interleaved need to be more deliberate about what stays and what moves. The split approach, IronPDF for PDF plus GdPicture retained for imaging, is a pragmatic middle ground that avoids forcing a full-stack replacement when only the PDF layer needs attention.

Start by running rg "using GdPicture" --type cs across your codebase. Understand which modules are actually in use. Then scope your migration to the PDF operations only, and evaluate from there.
