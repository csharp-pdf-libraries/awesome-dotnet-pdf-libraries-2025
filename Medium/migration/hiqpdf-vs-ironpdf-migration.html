Migrating from HiQPdf to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Here is a concrete scenario that comes up: you have been using HiQPdf for HTML-to-PDF conversion and it works, but you need to add PDF/A compliance output for document archiving. You open the HiQPdf docs looking for a PDF/A export option, and it is not there -- or it is marked as a feature in a tier you do not currently have. That is the trigger for this article. Not drama, just a gap in capability that forces a library evaluation.

This walkthrough covers the migration path from HiQPdf to IronPDF: what transfers cleanly, what needs attention, and what to measure before you commit.

Why Should I Migrate from HiQPdf to IronPDF?

Nine reasons teams evaluate alternatives to HiQPdf. Take what applies to your situation.

Missing features. PDF/A compliance, specific digital signature workflows, or advanced PDF manipulation operations may not be in the version you are on.

Platform support gap. HiQPdf's Linux support posture matters if you are moving workloads to Linux containers. Verify current state before assuming.

Commercial license model. Both HiQPdf and IronPDF are commercial. If you are re-evaluating, compare what is included at your tier.

CSS fidelity deltas. Even among Chromium-based renderers, version differences and configuration options produce different output. Teams with pixel-sensitive documents test this case by case.

API verbosity. Some operations require more boilerplate in HiQPdf than equivalent IronPDF calls.

Documentation coverage. If you are hitting underdocumented corners of HiQPdf, that is friction.

Thread safety and concurrency model. Verify how HiQPdf handles concurrent render requests before scaling.

Ecosystem integration. NuGet, Azure DevOps, GitHub Actions compatibility. Verify any CI/CD-specific package consumption nuances.

Support responsiveness. Commercial library support quality varies. If you are blocked on issues, that is a migration signal.

How Do HiQPdf and IronPDF Compare at a Glance?

  • Focus. HiQPdf centers on HTML-to-PDF with a converter-plus-document model; IronPDF covers HTML-to-PDF plus a full PDF manipulation surface (merge, stamp, security, PDF/A).
  • Pricing. Both are commercial. Compare the current license tiers at hiqpdf.com and ironsoftware.com against your feature list.
  • API style. HiQPdf uses an HtmlToPdf converter and a PdfDocument with AddDocument/WriteToFile; IronPDF uses a ChromePdfRenderer plus a PdfDocument with SaveAs/BinaryData.
  • Rendering engine. HiQPdf is WebKit-based (older); IronPDF uses a current Chromium build, which matters for modern CSS and JS frameworks.
  • Page indexing. IronPDF is 0-based across Pages, RemovePages, and CopyPages. Audit any HiQPdf page indexing assumptions before relying on the same numbers.
  • Thread safety. In IronPDF, create a separate ChromePdfRenderer per task rather than sharing across threads; verify HiQPdf's concurrent posture in its docs.
  • Namespace. HiQPdf → IronPdf.

How Much Effort Does Each Feature Take to Migrate?

I have mapped out the effort level for each feature area. Most conversions fall into the low-effort category, which means straightforward API substitutions.

  • HTML string to PDF — Low. HtmlToPdf.ConvertHtmlToMemory(html, baseUrl) becomes renderer.RenderHtmlAsPdf(html).
  • URL to PDF — Low. ConvertUrlToMemory(url) becomes renderer.RenderUrlAsPdf(url).
  • HTML file to PDF — Low. Use renderer.RenderHtmlFileAsPdf(path) on the IronPDF side.
  • PDF/A compliance — Low in IronPDF if starting from zero. HiQPdf's PDF/A coverage is tier-dependent; verify your contract before assuming. IronPDF exposes SaveAsPdfA directly.
  • Merge PDFs — Low. HiQPdf does this with PdfDocument.AddDocument; IronPDF uses static PdfDocument.Merge(doc1, doc2).
  • Watermark — Low. IronPDF uses TextStamper/ApplyStamp; in HiQPdf you build it from positioned elements on the document.
  • Password protection — Low. IronPDF uses pdf.SecuritySettings.UserPassword/OwnerPassword; HiQPdf exposes equivalent settings on the document.
  • Custom margins — Low. HiQPdf margins are points; IronPDF margins are millimeters. Convert with mm = points * 25.4 / 72.
  • Headers / footers — Medium. HiQPdf uses positioned HtmlToPdfVariableElement objects added to Document.Header/Document.Footer with placeholders like {CrtPage} and {PageCount}. IronPDF uses HtmlHeaderFooter (or TextHeaderFooter) with {page}/{total-pages}.
  • Digital signatures — Medium. Verify your signing requirements (PAdES level, timestamp authority) in both products' current docs before committing.
  • Concurrent rendering — Medium. In IronPDF, create one ChromePdfRenderer per task and use the async APIs; verify HiQPdf's thread model in its docs.

What Does the Decision Matrix Look Like?

  • If PDF/A compliance is required, verify your HiQPdf tier exposes it. IronPDF has it via SaveAsPdfA. Test both outputs against PAC or veraPDF before declaring conformance.
  • If you are Windows-only, both should work. Validate your CI runner image and any GDI+ dependencies before committing.
  • If you are deploying to Linux or Docker, check HiQPdf's current Linux posture carefully before assuming parity. IronPDF ships a Linux package, but native dependencies still need to be present in your container image.
  • Either way, both products are commercial. Compare the feature set per license tier against your real list of needs — PDF/A, signatures, OCR, concurrent rendering — rather than the marketing summary.

How Do I Convert HTML to PDF After Migration?

This is the most common operation. The pattern changes from a converter class returning bytes to a renderer class returning a document object.

HiQPdf (Before)

using HiQPdf;
using System;
using System.IO;

class HtmlToPdfExample
{
    static void Main()
    {
        var converter = new HtmlToPdf();
        converter.Document.PageSize = PdfPageSize.A4;
        converter.Document.PageOrientation = PdfPageOrientation.Portrait;

        byte[] pdfBytes = converter.ConvertHtmlToMemory(
            "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>",
            "" // optional base URL for resolving relative resources
        );

        File.WriteAllBytes("invoice.pdf", pdfBytes);
        Console.WriteLine("Saved: invoice.pdf");
    }
}

IronPDF (After)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>"
);
pdf.SaveAs("invoice.pdf");

You can find more details in the IronPDF HTML string to PDF guide.

How Do I Output PDF/A Compliant Documents?

This is often the feature gap that triggers migration in the first place. If HiQPdf does not expose PDF/A in your licensed tier, this is where IronPDF fills the gap directly.

HiQPdf (Before)

using HiQPdf;
using System.IO;

class PdfAExample
{
    static void Main()
    {
        // PDF/A coverage in HiQPdf is tier-dependent -- verify your contract
        // before assuming the property is exposed in your build
        var converter = new HtmlToPdf();
        // converter.Document.PdfStandard = PdfStandard.PdfA1b; // verify availability

        byte[] pdfBytes = converter.ConvertHtmlToMemory("<h1>Archived Document</h1>", "");
        File.WriteAllBytes("archived.pdf", pdfBytes);
    }
}

IronPDF (After)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<h1>Archived Document</h1>");

// Convert to PDF/A-3b for long-term archiving
pdf.SaveAsPdfA("archived_pdfa.pdf", IronPdf.PdfAVersions.PdfA3b);

For full details on PDF/A output, see the IronPDF PDF/A guide.

How Do I Add Watermarks After Migration?

HiQPdf does not expose a single-call watermark API the way IronPDF does — you typically build the effect by adding positioned elements to the document. In IronPDF, the stamper API makes this one call.

HiQPdf (Before)

using HiQPdf;
using System.IO;

class WatermarkExample
{
    static void Main()
    {
        // HiQPdf has no first-class "ApplyWatermark" call.
        // Teams typically render an HTML overlay with absolute-positioned
        // text and rely on Document elements / page templates to repeat it.
        var converter = new HtmlToPdf();
        string html = @"<div style='position:fixed; top:40%; left:20%;
            transform:rotate(-45deg); font-size:80px; color:lightgray;
            opacity:0.4;'>CONFIDENTIAL</div>
            <h1>Quarterly Report</h1>";

        byte[] pdfBytes = converter.ConvertHtmlToMemory(html, "");
        File.WriteAllBytes("watermarked.pdf", pdfBytes);
    }
}

IronPDF (After)

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("input.pdf");
var stamper = new TextStamper
{
    Text = "CONFIDENTIAL",
    FontColor = IronSoftware.Drawing.Color.LightGray,
    Opacity = 40,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");

See the IronPDF watermark guide for more stamper options.

How Do I Apply Password Protection After Migration?

Security settings move from converter-level properties to the PdfDocument's SecuritySettings object in IronPDF.

HiQPdf (Before)

using HiQPdf;
using System.IO;

class SecurityExample
{
    static void Main()
    {
        var converter = new HtmlToPdf();
        // HiQPdf exposes security settings on the document; confirm exact
        // property names (UserPassword / OwnerPassword) against your version
        converter.Document.Security.UserPassword = "userpass";
        converter.Document.Security.OwnerPassword = "ownerpass";

        byte[] pdf = converter.ConvertHtmlToMemory("<h1>Secure Document</h1>", "");
        File.WriteAllBytes("secured.pdf", pdf);
    }
}

IronPDF (After)

using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<h1>Secure Document</h1>");
pdf.SecuritySettings.UserPassword = "userpass";
pdf.SecuritySettings.OwnerPassword = "ownerpass";
pdf.SecuritySettings.AllowUserCopyPasteContent = false;
pdf.SaveAs("secured.pdf");

Full details are in the IronPDF permissions and passwords guide.

What Are the Core API Mapping Differences?

The namespace and class mappings are direct. Here is a complete reference for the core conversions you will make across your codebase.

Namespace Mapping

  • HiQPdf → IronPdf — core namespace
  • HiQPdf options classes → IronPdf.Rendering — render configuration
  • HiQPdf PDF manipulation types → IronPdf / IronPdf.Editing — manipulation ops

Core Class Mapping

  • HtmlToPdf → ChromePdfRenderer — main rendering entry point
  • Converter options on HtmlToPdf.Document → ChromePdfRenderOptions (via renderer.RenderingOptions) — render configuration
  • PdfDocument → PdfDocument — same name, different namespace
  • N/A in HiQPdf → static PdfDocument.Merge, PdfDocument.FromFile — document-level operations exposed at the type

Document Loading Methods

  • HTML string: converter.ConvertHtmlToMemory(html, baseUrl) → renderer.RenderHtmlAsPdf(html)
  • URL: converter.ConvertUrlToMemory(url) → renderer.RenderUrlAsPdf(url)
  • HTML file: converter.ConvertHtmlToFile(...) → renderer.RenderHtmlFileAsPdf(path)
  • Existing PDF: PdfDocument.FromFile(path) in both libraries

Page Operations

  • Page count: document.Pages.Count → pdf.PageCount
  • Paper size: Document.PageSize → RenderingOptions.PaperSize
  • Margins: Document.Margins.Top/Bottom/Left/Right (points) → RenderingOptions.MarginTop/Bottom/Left/Right (millimeters)
  • Orientation: Document.PageOrientation → RenderingOptions.PaperOrientation

Merge/Split Operations

  • Merge: document.AddDocument(other) + WriteToFile → static PdfDocument.Merge(pdf1, pdf2)
  • Split / extract: page-by-page loop in HiQPdf → pdf.CopyPages(startIndex, endIndex) in IronPDF (0-based)

What Critical Migration Notes Should I Know?

There are three key differences in how the two libraries handle output and errors. Getting these right will prevent subtle bugs.

Page Indexing

Verify HiQPdf's page indexing convention in its docs. IronPDF uses 0-based indexing throughout. If you are migrating page manipulation code, audit all index references.

Byte Array vs File Model

HiQPdf's converter returns byte[] from ConvertHtmlToMemory; IronPDF returns a PdfDocument object, which you then save or stream. Update call sites accordingly:

// HiQPdf pattern:
byte[] pdfBytes = converter.ConvertHtmlToMemory(html, "");
File.WriteAllBytes("output.pdf", pdfBytes);

// IronPDF pattern:
var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");

// Or to MemoryStream:
using var ms = new MemoryStream();
pdf.Stream.CopyTo(ms);

Error Handling Model

HiQPdf may return null or empty bytes on failure; IronPDF throws exceptions. Update error handling:

// IronPDF exception pattern
try
{
    var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs("output.pdf");
}
catch (IronPdf.Exceptions.IronPdfException ex)
{
    // Handle render errors
    Console.Error.WriteLine($"Render failed: {ex.Message}");
}

How Do I Optimize Performance After Migration?

Three patterns will get you the most out of IronPDF: renderer reuse, async rendering, and proper parallel execution.

Renderer Reuse

// Configure once, reuse for batch work
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
renderer.RenderingOptions.MarginTop = 10;

foreach (var html in templates)
{
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"output_{Guid.NewGuid()}.pdf");
}

Async for I/O-Heavy Workloads

// Async rendering for web applications
var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(html);
await pdf.SaveAsAsync("output.pdf");

Parallel Rendering

// Separate renderer per task -- don't share across threads
var results = await Task.WhenAll(
    htmlItems.Select(async html =>
    {
        var r = new ChromePdfRenderer();
        return await r.RenderHtmlAsPdfAsync(html);
    })
);

How Should I Benchmark HiQPdf vs IronPDF?

I want to be clear here: run these benchmarks in your environment before making a call. Do not rely on numbers from other contexts.

What to Measure

  • Single-page HTML-to-PDF time (ms). Baseline render performance — Stopwatch around the render call, 100 iterations, take the median.
  • Multi-page document render time. Scales differently across renderers — use a 20-page template, 50 iterations.
  • Memory allocation per render. Critical for high-volume services — sample GC.GetTotalMemory() before and after, or attach a memory profiler.
  • Concurrent renders throughput. Relevant if scaling horizontally — N threads, M renders each, measure wall clock.
  • Cold start time. Relevant for serverless or per-request scaling — compare the first render after process start against a warm run.
  • Output file size for same input. Affects storage and transfer costs — render the same HTML and compare byte counts.
  • CSS fidelity. Use your real templates and do a side-by-side screenshot diff; a single number won't capture template-specific rendering differences.

Benchmark Scaffold

using System;
using System.Diagnostics;
using System.Collections.Generic;
using IronPdf; // Swap to HiQPdf namespace when testing that library

class RenderBenchmark
{
    static void Main(string[] args)
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        const int warmupRounds = 3;
        const int timedRounds = 50;
        string html = System.IO.File.ReadAllText("test_template.html");

        var renderer = new ChromePdfRenderer();

        // Warm up -- exclude from timing
        for (int i = 0; i < warmupRounds; i++)
        {
            using var warmup = renderer.RenderHtmlAsPdf(html);
        }

        // Timed runs
        var times = new List<long>();
        var sw = new Stopwatch();

        for (int i = 0; i < timedRounds; i++)
        {
            sw.Restart();
            using var pdf = renderer.RenderHtmlAsPdf(html);
            sw.Stop();
            times.Add(sw.ElapsedMilliseconds);
        }

        times.Sort();
        Console.WriteLine($"Median: {times[timedRounds / 2]}ms");
        Console.WriteLine($"P95:    {times[(int)(timedRounds * 0.95)]}ms");
        Console.WriteLine($"P99:    {times[(int)(timedRounds * 0.99)]}ms");
        Console.WriteLine($"Min:    {times[0]}ms");
        Console.WriteLine($"Max:    {times[timedRounds - 1]}ms");
    }
}

Frequently Asked Questions

Is IronPDF a drop-in replacement for HiQPdf?

Not a literal drop-in -- the class names and method signatures differ. However, the migration effort for most operations is low. You are replacing HtmlToPdf with ChromePdfRenderer, swapping ConvertHtmlToMemory() for RenderHtmlAsPdf(), and changing from a byte[] return pattern to a PdfDocument object model. Most teams complete the core swap in a few hours.

Does IronPDF support PDF/A output?

Yes. This is often the feature that triggers the migration. Call pdf.SaveAsPdfA("output.pdf", IronPdf.PdfAVersions.PdfA3b) after rendering. Always validate the output with a tool like PAC or veraPDF to confirm conformance.

How does rendering fidelity compare between HiQPdf and IronPDF?

HiQPdf is built on an older WebKit engine; IronPDF uses a current Chromium build. That gap shows up most visibly on modern JavaScript frameworks (React, Angular, Vue), recent CSS features, and complex multi-page table layouts. Test with your actual HTML templates side by side before drawing conclusions.

Can I reuse my HiQPdf license key with IronPDF?

No. These are separate commercial products with separate licenses. You will need an IronPDF license key. Remove your HiQPdf license key from config and secrets management after migration is complete.

How does IronPDF handle concurrent rendering?

Create a separate ChromePdfRenderer instance per task -- do not share across threads. For async workloads, use RenderHtmlAsPdfAsync() and SaveAsAsync(). This pattern scales well for web applications and high-volume services.

What .NET versions does IronPDF support?

.NET 6+ is recommended. Verify IronPDF's current .NET version compatibility in their documentation before starting migration. Make sure your dev environment has HiQPdf currently working so you can do side-by-side render comparison testing.

What Should I Do After Completing the Migration?

The migration itself is mechanical -- the interesting question is whether the output matches. HiQPdf's WebKit engine and IronPDF's Chromium engine produce different output on modern CSS and JavaScript-heavy templates, so plan for visual review on anything pixel-sensitive.

After swapping the code, render each of your HTML templates through both libraries and compare the visual output side by side. Run the benchmark scaffold on your templates and record median and P95 numbers. If PDF/A was the trigger, validate the output with PAC or veraPDF.

Finally, remove your HiQPdf license key from config and secrets management, update your deployment docs, and monitor memory and CPU for the first production week.
