Migrating from HiQPdf to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Here is a concrete scenario that comes up: you have been using HiQPdf for HTML-to-PDF conversion and it works, but you need to add PDF/A compliance output for document archiving. You open the HiQPdf docs looking for a PDF/A export option, and it is not there -- or it is marked as a feature in a tier you do not currently have. That is the trigger for this article. Not drama, just a gap in capability that forces a library evaluation.

This walkthrough covers the migration path from HiQPdf to IronPDF: what transfers cleanly, what needs attention, and what to measure before you commit.

Why Should I Migrate from HiQPdf to IronPDF?

Nine reasons teams evaluate alternatives to HiQPdf. Take what applies to your situation.

Missing features. PDF/A compliance, specific digital signature workflows, or advanced PDF manipulation operations may not be in the version you are on.

Platform support gap. HiQPdf's Linux support posture matters if you are moving workloads to Linux containers. Verify current state before assuming.

Commercial license model. Both HiQPdf and IronPDF are commercial. If you are re-evaluating, compare what is included at your tier.

CSS fidelity deltas. Even among Chromium-based renderers, version differences and configuration options produce different output. Teams with pixel-sensitive documents test this case by case.

API verbosity. Some operations require more boilerplate in HiQPdf than equivalent IronPDF calls.

Documentation coverage. If you are hitting underdocumented corners of HiQPdf, that is friction.

Thread safety and concurrency model. Verify how HiQPdf handles concurrent render requests before scaling.

Ecosystem integration. NuGet, Azure DevOps, GitHub Actions compatibility. Verify any CI/CD-specific package consumption nuances.

Support responsiveness. Commercial library support quality varies. If you are blocked on issues, that is a migration signal.

How Do I Convert HTML to PDF After Migration?

This is the most common operation. The pattern changes from a converter class returning bytes to a renderer class returning a document object.

HiQPdf (Before)

using HiQPdf;
using System;
using System.IO;

class HtmlToPdfExample
{
    static void Main()
    {
        // Verify HiQPdf class names and method signatures in official docs
        // The pattern below is illustrative
        HtmlToPdf.SetLicense("YOUR_HIQPDF_KEY"); // verify method name

        var converter = new HtmlToPdf();
        // Verify property names for paper size, margins, etc.
        converter.Document.PageSize = HiQPdfPageSize.A4; // verify enum/class names

        byte[] pdfBytes = converter.ConvertHtmlString(
            "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>",
            "https://localhost/" // base URL -- verify if required
        );

        File.WriteAllBytes("invoice.pdf", pdfBytes);
        Console.WriteLine("Saved: invoice.pdf");
    }
}

IronPDF (After)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>"
);
pdf.SaveAs("invoice.pdf");

You can find more details in the IronPDF HTML string to PDF guide.

What Critical Migration Notes Should I Know?

Answer: There are three key differences in how the two libraries handle output and errors. Getting these right will prevent subtle bugs.

Page Indexing

Verify HiQPdf's page indexing convention in its docs. IronPDF uses 0-based indexing throughout. If you are migrating page manipulation code, audit all index references.

Byte Array vs File Model

HiQPdf commonly returns byte[] from conversion. IronPDF returns a PdfDocument object, which you then save or stream. Update call sites accordingly:

// HiQPdf pattern (illustrative -- verify):
byte[] pdfBytes = converter.ConvertHtmlString(html);
File.WriteAllBytes("output.pdf", pdfBytes);

// IronPDF pattern:
var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");

// Or to MemoryStream:
using var ms = new MemoryStream();
pdf.Stream.CopyTo(ms);

Error Handling Model

HiQPdf may return null or empty bytes on failure; IronPDF throws exceptions. Update error handling:

// IronPDF exception pattern
try
{
    var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs("output.pdf");
}
catch (IronPdf.Exceptions.IronPdfException ex)
{
    // Handle render errors
    Console.Error.WriteLine($"Render failed: {ex.Message}");
}

How Do I Optimize Performance After Migration?

Answer: Three patterns will get you the most out of IronPDF: renderer reuse, async rendering, and proper parallel execution.

Renderer Reuse

// Configure once, reuse for batch work
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
renderer.RenderingOptions.MarginTop = 10;

foreach (var html in templates)
{
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"output_{Guid.NewGuid()}.pdf");
}

Async for I/O-Heavy Workloads

// Async rendering for web applications
var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(html);
await pdf.SaveAsAsync("output.pdf");

Parallel Rendering

// Separate renderer per task -- don't share across threads
var results = await Task.WhenAll(
    htmlItems.Select(async html =>
    {
        var r = new ChromePdfRenderer();
        return await r.RenderHtmlAsPdfAsync(html);
    })
);

using System;
using System.Diagnostics;
using System.Collections.Generic;
using IronPdf; // Swap to HiQPdf namespace when testing that library

class RenderBenchmark
{
    static void Main(string[] args)
    {
        IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

        const int warmupRounds = 3;
        const int timedRounds = 50;
        string html = System.IO.File.ReadAllText("test_template.html");

        var renderer = new ChromePdfRenderer();

        // Warm up -- exclude from timing
        for (int i = 0; i < warmupRounds; i++)
        {
            using var warmup = renderer.RenderHtmlAsPdf(html);
        }

        // Timed runs
        var times = new List<long>();
        var sw = new Stopwatch();

        for (int i = 0; i < timedRounds; i++)
        {
            sw.Restart();
            using var pdf = renderer.RenderHtmlAsPdf(html);
            sw.Stop();
            times.Add(sw.ElapsedMilliseconds);
        }

        times.Sort();
        Console.WriteLine($"Median: {times[timedRounds / 2]}ms");
        Console.WriteLine($"P95:    {times[(int)(timedRounds * 0.95)]}ms");
        Console.WriteLine($"P99:    {times[(int)(timedRounds * 0.99)]}ms");
        Console.WriteLine($"Min:    {times[0]}ms");
        Console.WriteLine($"Max:    {times[timedRounds - 1]}ms");
    }
}

Frequently Asked Questions

Is IronPDF a drop-in replacement for HiQPdf?

Not a literal drop-in -- the class names and method signatures differ. However, the migration effort for most operations is low. You are replacing HtmlToPdf with ChromePdfRenderer, swapping ConvertHtmlString() for RenderHtmlAsPdf(), and changing from a byte[] return pattern to a PdfDocument object model. Most teams complete the core swap in a few hours.

Does IronPDF support PDF/A output?

Yes. This is often the feature that triggers the migration. Call pdf.SaveAsPdfA("output.pdf", IronPdf.PdfAVersions.PdfA3b) after rendering. Always validate the output with a tool like PAC or veraPDF to confirm conformance.

How does rendering fidelity compare between HiQPdf and IronPDF?

Both use Chromium-based rendering, so fidelity differences are likely to be subtle and template-specific rather than systematic. Test with your actual HTML templates side by side. Complex CSS and multi-page table layouts are where Chromium version differences tend to surface.

Can I reuse my HiQPdf license key with IronPDF?

No. These are separate commercial products with separate licenses. You will need an IronPDF license key. Remove your HiQPdf license key from config and secrets management after migration is complete.

How does IronPDF handle concurrent rendering?

Create a separate ChromePdfRenderer instance per task -- do not share across threads. For async workloads, use RenderHtmlAsPdfAsync() and SaveAsAsync(). This pattern scales well for web applications and high-volume services.

What .NET versions does IronPDF support?

.NET 6+ is recommended. Verify IronPDF's current .NET version compatibility in their documentation before starting migration. Make sure your dev environment has HiQPdf currently working so you can do side-by-side render comparison testing.

What Should I Do After Completing the Migration?

The migration itself is mechanical -- the interesting question is whether the output matches. Both HiQPdf and IronPDF use Chromium-based rendering, which means fidelity differences are likely to be subtle and template-specific rather than systematic.

After swapping the code, render each of your HTML templates through both libraries and compare the visual output side by side. Run the benchmark scaffold on your templates and record median and P95 numbers. If PDF/A was the trigger, validate the output with PAC or veraPDF.

Finally, remove your HiQPdf license key from config and secrets management, update your deployment docs, and monitor memory and CPU for the first production week.
