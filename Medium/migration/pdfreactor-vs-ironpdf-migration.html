Migrating from PDFreactor to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Consider Migrating from PDFreactor?

The concurrency problem shows up at scale. The job queue is filling faster than the renderer can drain it, you spin up more application instances, and then you hit the licensing ceiling or discover that the renderer is not safe to share across threads without external coordination. You are now building a rate-limiting wrapper around a PDF library, which is not what you set out to do.

Teams evaluating PDFreactor alternatives commonly encounter these conditions. Not all apply to every version, but I want to walk through them honestly.

Every deployment needs a JRE because PDFreactor has a Java runtime dependency. That increases container base image size and adds a runtime to manage. PDFreactor typically runs as a separate process or service, requiring health checks, startup coordination, and inter-process error handling.

Request queuing to the PDFreactor server introduces latency under load. Scaling requires multiple server instances. The licensing model at scale — whether concurrent connection or CPU-based — can become a bottleneck before infrastructure limits do.

Docker and container complexity is real. Java plus PDFreactor server plus your .NET app in one container or a separate sidecar adds orchestration complexity. The .NET-to-PDFreactor HTTP call adds measurable latency per document, especially at high volume.

PDFreactor's CSS Paged Media compliance is high-quality for print-exact typographic work, but that same specificity can require significant HTML and CSS preparation for web-derived content. Cold start time affects warm-up and auto-scaling behavior, and if the PDFreactor service goes down or becomes unresponsive, PDF generation for the entire application stops.

Comparison Table

  • Focus. PDFreactor targets CSS Paged Media and print-quality HTML-to-PDF; IronPDF covers general HTML-to-PDF plus PDF manipulation across the document lifecycle.
  • Pricing. Both are commercial licenses — PDFreactor at realobjects.com, IronPDF at ironsoftware.com. Compare at your target volume.
  • API style. PDFreactor's .NET integration is a Web Service client (REST over HTTP) talking to a Java/Jetty service; IronPDF exposes native .NET objects with no external process.
  • Learning curve. PDFreactor is medium — CSS Paged Media concepts help. IronPDF is low for .NET developers.
  • HTML rendering. PDFreactor implements W3C CSS Paged Media; IronPDF is Chromium-based.
  • Page indexing. IronPDF is 0-based. Audit any page-number arithmetic on the way over.
  • Thread safety. PDFreactor handles concurrency server-side via HTTP requests. Confirm IronPDF's concurrent-instance guidance against the official docs at your concurrency target.
  • Namespace. RealObjects.PDFreactor.Webservice.Client on the way in, IronPdf on the way out.

What Does the Migration Effort Actually Look Like?

I want to give you a realistic picture of the effort involved. Here is a feature-by-feature breakdown so you can estimate the work before committing.

Effort by Feature

  • HTML string to PDF — Low. PDFreactor.Convert() with config.Document = html maps to ChromePdfRenderer.RenderHtmlAsPdf().
  • URL to PDF — Low. config.Document = url maps to ChromePdfRenderer.RenderUrlAsPdf().
  • Save to bytes / file — Low. result.Document (byte array) maps to pdf.SaveAs() or pdf.BinaryData.
  • Custom page size — Low. CSS @page plus config.PageFormat moves to RenderingOptions.PaperSize.
  • Headers and footers — Medium. CSS Paged Media @page running elements move to RenderingOptions.HtmlHeader / HtmlFooter or TextHeader / TextFooter.
  • JavaScript rendering — Medium. config.EnableJavaScript maps to RenderingOptions.EnableJavaScript; PDFreactor's JavaScriptSettings.Timeout maps to IronPDF's WaitFor family (RenderDelay, NetworkIdle, JavaScript).
  • Merge PDFs — Medium. PDFreactor doesn't merge natively in the conversion SDK; IronPDF gives you PdfDocument.Merge() in-process.
  • Watermark — Medium. PDFreactor uses CSS background or running elements; IronPDF uses TextStamper / ImageStamper applied post-render.
  • Password protection — Medium. config.Encryption moves to pdf.SecuritySettings.
  • Async parallel — Medium-High. Parallel HTTP calls to the service become Task.WhenAll over async render methods, in-process.
  • CSS Paged Media features — High. PDFreactor supports the spec natively; IronPDF is limited to Chromium's CSS print subset.
  • Custom hyphenation and typographic spacing — High. PDFreactor handles these directly; IronPDF has no direct equivalent — verify whether your output truly needs these features before committing.

How Do You Decide Whether to Migrate?

Not every team should migrate. Here is a decision matrix to help you figure out where you fall.

  • If you produce print-exact typographic output — books, legal documents with strict CSS Paged Media requirements — PDFreactor may be better suited; evaluate carefully before switching.
  • If your workload is web-derived content, dashboard output, or reports from HTML templates, switch — Chromium fidelity is appropriate and deployment is simpler.
  • If you run on Kubernetes or Docker with throughput requirements, switch — no Java sidecar and no inter-process latency.
  • If you need concurrent high-volume rendering under tight licensing constraints, evaluate IronPDF's licensing model at your target volume before committing.

How Do You Set Up IronPDF and Replace PDFreactor?

I will walk you through the core migration. First, find your existing PDFreactor references, then swap the packages, and update your code.

Finding PDFreactor References in Your Codebase

# Search for PDFreactor API calls
rg -l "PDFreactor\|RealObjects\|pdfreactor" --type cs

# Check project files
grep -r "PDFreactor\|RealObjects" *.csproj **/*.csproj 2>/dev/null

# Find any PDFreactor server config files
find . -name "pdfreactor*.xml" -o -name "pdfreactor*.json" 2>/dev/null

# Find CSS files that use @page or CSS Paged Media rules
rg "@page\|prince-\|pdfreactor-" --type css

Remove PDFreactor and Install IronPDF

PDFreactor is not distributed via NuGet. The .NET integration ships as PDFreactor.dll inside the PDFreactor installer (under clients/netstandard2/bin/ or clients/netframework40/bin/), and it talks REST to a separate Java/Jetty PDFreactor Web Service (default port 9423). Removing PDFreactor means deleting the assembly reference from your project and decommissioning that service.

# Remove the PDFreactor.dll <Reference> from your .csproj
# Stop the PDFreactor Web Service (Java/Jetty, default port 9423)
#   Windows: net stop PDFreactor
#   Linux:   sudo systemctl stop pdfreactor
#   Docker:  docker stop <pdfreactor-container>

# Install IronPDF
dotnet add package IronPdf
dotnet restore

Basic HTML-to-PDF

Before:

using RealObjects.PDFreactor.Webservice.Client;
using System.IO;

class Program
{
    static void Main()
    {
        // Default service URL is http://localhost:9423/service/rest
        PDFreactor pdfReactor = new PDFreactor();

        Configuration config = new Configuration();
        config.Document = "<html><body><h1>Hello</h1></body></html>";

}

var pdf = renderer.RenderHtmlAsPdf("<html><body><h1>Hello</h1></body></html>");
pdf.SaveAs("output.pdf");

Console.WriteLine($"Saved output.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

What Does a Full HTML String to PDF Migration Look Like?

Let me show you the complete before and after for the most common operation teams need to migrate.

HTML String to PDF — Before

using RealObjects.PDFreactor.Webservice.Client;
using System.IO;

class HtmlToPdfBefore
{
    static void Main()
    {
        PDFreactor pdfReactor = new PDFreactor();

        Configuration config = new Configuration();
        config.Document = @"
            <html>
            <head><style>@page { size: A4; margin: 2cm; }</style></head>
            <body><h1>Report Q4 2024</h1><p>Financial summary here.</p></body>
            </html>";

        Result result = pdfReactor.Convert(config);
        File.WriteAllBytes("report.pdf", result.Document);
    }
}

HTML String to PDF — After

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var html = @"
    <html>
    <head>
    <style>
        @page { size: A4; margin: 2cm; }
        body { font-family: Arial, sans-serif; }
    </style>
    </head>

var renderer = new ChromePdfRenderer();

How Do You Merge PDFs After Migrating?

PDFreactor's conversion SDK doesn't merge directly — you generate each section, then bring in a secondary PDF library to combine the byte arrays. IronPDF merges in-process.

Merge PDFs — Before

using RealObjects.PDFreactor.Webservice.Client;
using System.Collections.Generic;
using System.IO;

class MergeBefore
{
    static void Main()
    {
        PDFreactor pdfReactor = new PDFreactor();

        // Generate each section separately
        var sections = new[] {
            "<html><body><h1>Section 1: Overview</h1></body></html>",
            "<html><body><h1>Section 2: Detail</h1></body></html>",
        };

        var pdfs = new List<byte[]>();
        foreach (var html in sections)
        {
            var config = new Configuration();
            config.Document = html;
            pdfs.Add(pdfReactor.Convert(config).Document);
        }

        // Combine via a secondary library (iText, PdfSharp, etc.) — adds a dependency
    }
}

Merge PDFs — After

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();

// Render sections concurrently: https://ironpdf.com/how-to/async/
var task1 = renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 1: Overview</h1></body></html>");
var task2 = renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 2: Detail</h1></body></html>");

var results = await Task.WhenAll(task1, task2);

// Merge locally: https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("combined-report.pdf");

Console.WriteLine($"Merged: {merged.PageCount} total pages");

How Do You Add Watermarks After Migrating?

PDFreactor handles watermarks through CSS Paged Media running elements. IronPDF provides a programmatic approach that is applied post-render.

Watermark — Before

using RealObjects.PDFreactor.Webservice.Client;
using System.Collections.Generic;
using System.IO;

class WatermarkBefore
{
    static void Main()
    {
        PDFreactor pdfReactor = new PDFreactor();

        Configuration config = new Configuration();
        config.Document = "<html><body><h1>Q4 Report</h1></body></html>";

        // PDFreactor uses CSS Paged Media or background-image for watermarks
        config.UserStyleSheets = new List<Resource>
        {
            new Resource
            {
                Content = @"
                    @page {
                        background-image: url('draft-watermark.png');
                        background-position: center;
                        background-repeat: no-repeat;
                        background-size: 50%;
                    }
                "
            }
        };

        Result result = pdfReactor.Convert(config);
        File.WriteAllBytes("draft-report.pdf", result.Document);
    }
}

Watermark — After

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Q4 Report</h1><p>Content...</p></body></html>"
);

// Programmatic watermark applied post-render
// https://ironpdf.com/how-to/custom-watermark/
var watermark = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronSoftware.Drawing.Color.Gray,
    Opacity = 15,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("draft-report.pdf");
Console.WriteLine("Watermark applied");

How Do You Add Password Protection After Migrating?

PDFreactor sets encryption through the Configuration.Encryption object before conversion. IronPDF applies it post-render through the SecuritySettings object on the resulting PdfDocument.

Password Protection — Before

using RealObjects.PDFreactor.Webservice.Client;
using System.IO;

class PasswordBefore
{
    static void Main()
    {
        PDFreactor pdfReactor = new PDFreactor();

        Configuration config = new Configuration();
        config.Document = "<html><body><h1>Confidential Report</h1></body></html>";

        config.Encryption = new Encryption
        {
            UserPassword = "open123",
            OwnerPassword = "admin456",
            AllowPrint = true,
            AllowCopy = false,
        };

        Result result = pdfReactor.Convert(config);
        File.WriteAllBytes("secured-report.pdf", result.Document);
    }
}

Password Protection — After

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Confidential Report</h1></body></html>"
);

// https://ironpdf.com/how-to/pdf-permissions-passwords/
pdf.SecuritySettings.UserPassword = "open123";
pdf.SecuritySettings.OwnerPassword = "admin456";

pdf.SaveAs("secured-report.pdf");
Console.WriteLine("Password-protected PDF saved");

What Are the Key Feature Differences Between PDFreactor and IronPDF?

Here are the full API mapping tables so you can plan your migration systematically.

Namespace Mapping

  • RealObjects.PDFreactor.Webservice.Client → IronPdf — core namespace.
  • PDFreactor (Web Service client class) → ChromePdfRenderer — no external service required.
  • Configuration class → ChromePdfRenderOptions — rendering options object.

Core Class Mapping

  • PDFreactor (service client) → ChromePdfRenderer — renders HTML or URL to PDF.
  • Configuration → ChromePdfRenderOptions — page size, margins, JavaScript settings.
  • Result → PdfDocument — the PDF object with manipulation methods.
  • No first-class merge in the conversion SDK → PdfDocument.Merge() — local in-process merge.

Document Loading Methods

  • HTML string. config.Document = html → renderer.RenderHtmlAsPdf(html) (or the async variant).
  • URL. config.Document = url → renderer.RenderUrlAsPdf(url).
  • HTML file. Read the file and pass through config.Document with BaseURL → renderer.RenderHtmlFileAsPdf(path).
  • Existing PDF. Not a PDFreactor concept — it's a converter, not a manipulation library. IronPDF: PdfDocument.FromFile(path).

Page Operations

PDFreactor is a converter, not a PDF-manipulation library, so post-render page operations have no native equivalent and typically involve a secondary library. In IronPDF, these live on the PdfDocument directly:

  • Page count. pdf.PageCount.
  • Remove page. pdf.RemovePages(index).
  • Extract page range. pdf.CopyPages(startIndex, endIndex).
  • Rotate. Per-page rotation via pdf.RotateAllPages(...); for partial rotation, see the IronPDF rotate page guide.

Merge / Split Operations

  • Merge. PDFreactor's conversion SDK doesn't expose merge directly — it's typically handled by combining byte arrays with a secondary PDF library. In IronPDF, PdfDocument.Merge(doc1, doc2).
  • Split. Same story — not in PDFreactor's converter remit. See the IronPDF merge or split guide.

What Are the Critical Migration Notes You Should Know?

PDFreactor has deep CSS Paged Media support — running headers and footers via @page running elements, complex footnotes, table header repetition. These features work because PDFreactor implements the W3C CSS Paged Media spec closely.

IronPDF's Chromium renderer supports a subset of CSS print properties — @page size and margins, page-break-*, orphans/widows. It does not implement the full CSS Paged Media spec. If your templates use CSS features like running(), string(), @footnote, or named page flows, audit those templates carefully before migration. You may need to migrate those header and footer patterns to IronPDF's RenderingOptions.HtmlHeader / HtmlFooter API instead.

CSS Paged Media Header Migration Example

// PDFreactor CSS running header: @page { @top-center { content: element(header); } }
// IronPDF equivalent: HtmlHeader/Footer via RenderingOptions

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.HtmlHeader = new HtmlHeaderFooter
{
    HtmlFragment = "<div style='text-align:center; font-size:10px;'>{page} of {total-pages}</div>",
    BaseUrl = new Uri(System.IO.Path.GetFullPath("."))
};
// https://ironpdf.com/how-to/rendering-options/

IronPDF uses 0-based page indexing. PDFreactor's CSS counter(page) is 1-based for display, so audit all page-number arithmetic in your migration before declaring it complete.

After migration is validated, decommission the PDFreactor server process from Docker images, CI pipelines, and startup scripts. This is where the deployment simplification is realized — removing the Java runtime dependency from container images can significantly reduce image size.

Async Pattern Alignment

PDFreactor's .NET SDK Convert() call is synchronous and blocks until the server returns. Replace with IronPDF's native async methods to free your request thread during rendering.

// Instead of blocking:
// Result result = pdfReactor.Convert(config); // synchronous

// Use:
var pdf = await renderer.RenderHtmlAsPdfAsync(html);
// See: https://ironpdf.com/how-to/async/

Frequently Asked Questions

Do I need to keep the Java runtime after migrating?

No. Java runtime removal is optional during migration, but you should decommission it after migration is validated. The deployment simplification from removing the Java sidecar or service dependency is the most concrete operational win. Removing the Java runtime dependency from container images can significantly reduce image size.

That said, keep it around until you have verified every template renders correctly in IronPDF.

What happens to my CSS Paged Media templates?

This is the most important pre-migration step. If your templates use basic @page sizing and page-break-*, migration is low-risk. If they use running elements, footnotes, or named page flows, those require rewriting before the migration produces comparable output.

IronPDF's Chromium renderer supports @page size and margins, page-break-*, orphans/widows, but does not implement the full CSS Paged Media spec. Audit your templates for features like running(), string(), and @footnote before committing to the migration.

How do I handle concurrent rendering differently?

With PDFreactor, concurrent rendering means parallel HTTP calls to the service. With IronPDF's async methods, you use Task.WhenAll with async render methods directly. There is no inter-process HTTP hop, which eliminates network latency from the equation. Confirm IronPDF's concurrent-instance guidance for your target concurrency level against the official documentation.

What is the recommended approach for high-volume benchmarking?

When benchmarking, account for the eliminated network hop. PDFreactor render time includes the HTTP call to the service plus Java render time. IronPDF render time is the local Chromium render time only. Run your benchmarks against your actual HTML templates, not synthetic examples. Hardware, template complexity, JavaScript usage, and image count all significantly affect render times.

Warm up the renderer before benchmarking to exclude Chromium initialization overhead from your measurements.

What Is the Decision Framework for This Migration?

The most important pre-migration step is the CSS Paged Media audit. If your templates use basic @page sizing and page-break-*, migration is low-risk. If they use running elements, footnotes, or named page flows, those require rewriting before the migration is comparable output.

The deployment simplification — removing the Java sidecar or service dependency — is the most concrete operational win and is verifiable before writing any migration code by checking your Docker image size and startup sequence.

For teams doing web-derived content, dashboard output, and reports from HTML templates, the switch is straightforward. For teams relying on print-exact typographic output with strict CSS Paged Media requirements, PDFreactor may remain the better fit. Evaluate honestly and test with your actual templates before committing.
