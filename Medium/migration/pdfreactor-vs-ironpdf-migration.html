Migrating from PDFreactor to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Consider Migrating from PDFreactor?

The concurrency problem shows up at scale. The job queue is filling faster than the renderer can drain it, you spin up more application instances, and then you hit the licensing ceiling or discover that the renderer is not safe to share across threads without external coordination. You are now building a rate-limiting wrapper around a PDF library, which is not what you set out to do.

Teams evaluating PDFreactor alternatives commonly encounter these conditions. Not all apply to every version, but I want to walk through them honestly.

Every deployment needs a JRE because PDFreactor has a Java runtime dependency. That increases container base image size and adds a runtime to manage. PDFreactor typically runs as a separate process or service, requiring health checks, startup coordination, and inter-process error handling.

Request queuing to the PDFreactor server introduces latency under load. Scaling requires multiple server instances. The licensing model at scale — whether concurrent connection or CPU-based — can become a bottleneck before infrastructure limits do.

Docker and container complexity is real. Java plus PDFreactor server plus your .NET app in one container or a separate sidecar adds orchestration complexity. The .NET-to-PDFreactor HTTP call adds measurable latency per document, especially at high volume.

PDFreactor's CSS Paged Media compliance is high-quality for print-exact typographic work, but that same specificity can require significant HTML and CSS preparation for web-derived content. Cold start time affects warm-up and auto-scaling behavior, and if the PDFreactor service goes down or becomes unresponsive, PDF generation for the entire application stops.

What Are the Critical Migration Notes You Should Know?

Answer:

PDFreactor has deep CSS Paged Media support — running headers and footers via @page running elements, complex footnotes, table header repetition. These features work because PDFreactor implements the W3C CSS Paged Media spec closely.

IronPDF's Chromium renderer supports a subset of CSS print properties — @page size and margins, page-break-*, orphans/widows. It does not implement the full CSS Paged Media spec. If your templates use CSS features like running(), string(), @footnote, or named page flows, audit those templates carefully before migration. You may need to migrate those header and footer patterns to IronPDF's RenderingOptions.HtmlHeader / HtmlFooter API instead.

CSS Paged Media Header Migration Example

// PDFreactor CSS running header: @page { @top-center { content: element(header); } }
// IronPDF equivalent: HtmlHeader/Footer via RenderingOptions

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.HtmlHeader = new HtmlHeaderFooter
{
    HtmlFragment = "<div style='text-align:center; font-size:10px;'>{page} of {total-pages}</div>",
    BaseUrl = new Uri(System.IO.Path.GetFullPath("."))
};
// Verify HtmlHeaderFooter class and properties in current IronPDF docs
// https://ironpdf.com/how-to/rendering-options/

IronPDF uses 0-based page indexing. If PDFreactor reports pages 1-based, audit all page number logic. Verify in current documentation.

After migration is validated, decommission the PDFreactor server process from Docker images, CI pipelines, and startup scripts. This is where the deployment simplification is realized — removing the Java runtime dependency from container images can significantly reduce image size.

Async Pattern Alignment

PDFreactor's .NET SDK may expose synchronous Convert() calls. Replace with IronPDF's native async methods.

// Instead of blocking:
// var result = pdfreactor.Convert(config); // synchronous

// Use:
var pdf = await renderer.RenderHtmlAsPdfAsync(html);

Frequently Asked Questions

Do I need to keep the Java runtime after migrating?

Answer:

No. Java runtime removal is optional during migration, but you should decommission it after migration is validated. The deployment simplification from removing the Java sidecar or service dependency is the most concrete operational win. Removing the Java runtime dependency from container images can significantly reduce image size.

That said, keep it around until you have verified every template renders correctly in IronPDF.

What happens to my CSS Paged Media templates?

Answer:

This is the most important pre-migration step. If your templates use basic @page sizing and page-break-*, migration is low-risk. If they use running elements, footnotes, or named page flows, those require rewriting before the migration produces comparable output.

IronPDF's Chromium renderer supports @page size and margins, page-break-*, orphans/widows, but does not implement the full CSS Paged Media spec. Audit your templates for features like running(), string(), and @footnote before committing to the migration.

How do I handle concurrent rendering differently?

Answer:

With PDFreactor, concurrent rendering means parallel HTTP calls to the service. With IronPDF's async methods, you use Task.WhenAll with async render methods directly. There is no inter-process HTTP hop, which eliminates network latency from the equation. Verify IronPDF's concurrent instance guidance for your target concurrency level.

What is the recommended approach for high-volume benchmarking?

Answer:

When benchmarking, account for the eliminated network hop. PDFreactor render time includes the HTTP call to the service plus Java render time. IronPDF render time is the local Chromium render time only. Run your benchmarks against your actual HTML templates, not synthetic examples. Hardware, template complexity, JavaScript usage, and image count all significantly affect render times.

Warm up the renderer before benchmarking to exclude Chromium initialization overhead from your measurements.

What Is the Decision Framework for This Migration?

Answer:

The most important pre-migration step is the CSS Paged Media audit. If your templates use basic @page sizing and page-break-*, migration is low-risk. If they use running elements, footnotes, or named page flows, those require rewriting before the migration is comparable output.

The deployment simplification — removing the Java sidecar or service dependency — is the most concrete operational win and is verifiable before writing any migration code by checking your Docker image size and startup sequence.

For teams doing web-derived content, dashboard output, and reports from HTML templates, the switch is straightforward. For teams relying on print-exact typographic output with strict CSS Paged Media requirements, PDFreactor may remain the better fit. Evaluate honestly and test with your actual templates before committing.
