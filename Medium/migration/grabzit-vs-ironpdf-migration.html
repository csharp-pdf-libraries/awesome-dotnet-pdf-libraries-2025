Migrating from GrabzIt to IronPDF: Copy-Paste and Ship

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Migrate from GrabzIt to IronPDF?

You have a service that generates PDFs and screenshots. It worked fine at one pod. Now you are running three, and the GrabzIt API calls are stepping on each other -- timeouts, corrupt outputs, occasional 429s from the cloud endpoint. The rate limiter does not care that your traffic is legitimate. You are scaling up, and the architecture does not support it.

I have seen teams run this migration for a range of neutral, practical reasons. Not all of these will apply to you, but here are the most common ones:

Concurrency limits hit. GrabzIt cloud API has rate limits. Under burst traffic, requests queue or fail.

Network dependency at render time. Every PDF generation makes an outbound HTTP call. Latency spikes in cloud environments, VPN-locked offices, or air-gapped systems.

Data residency requirements. HTML content is sent to GrabzIt servers. Some compliance frameworks prohibit sending document content to third-party endpoints.

Offline/edge deployments. Embedded systems, offline factory floors, or on-prem government environments cannot reach the GrabzIt API.

Docker/Kubernetes complexity. Managing API keys in secrets, retry logic for transient failures, and fallback strategies all add overhead that a local renderer eliminates.

Cost structure changes. SaaS pricing scales with usage. At volume, a one-time license can look different on a spreadsheet than per-render billing. Verify both against your own usage numbers.

CSS/JS fidelity. Cloud renderers have their own Chromium version and update cadence. Your HTML may render differently than expected.

Timeout handling. Async screenshot requests over HTTP introduce timeout complexity not present in in-process rendering.

Vendor lock-in. A thin wrapper pattern means your code is coupled to GrabzIt's API contract. If they change endpoints or deprecate methods, your builds break.

How Do GrabzIt and IronPDF Compare at a Glance?

Here is a side-by-side comparison of the two libraries across key dimensions.

  • Focus. GrabzIt is a cloud screenshot and PDF SaaS; IronPDF is a local PDF generation and manipulation library.
  • Pricing. GrabzIt uses subscription tiers tied to capture volume; IronPDF uses per-developer licensing.
  • API style. GrabzIt is a REST client wrapper with async callback or polling; IronPDF is an in-process .NET library with synchronous and async APIs.
  • Learning curve. GrabzIt is low setup but limited control; IronPDF has more API surface and more control.
  • HTML rendering. GrabzIt runs cloud Chromium with the version managed by the vendor; IronPDF ships a Chromium-based local renderer.
  • Page indexing. GrabzIt is 1-based; IronPDF is 0-based.
  • Thread safety. GrabzIt is rate-limited by API tier; IronPDF supports per-thread renderer instances.
  • Namespace. GrabzIt / GrabzItClient versus IronPdf.

How Much Effort Does Each Feature Take to Migrate?

Not every feature migrates with the same complexity. I have mapped the common GrabzIt features against the effort required to move them to IronPDF.

  • HTML string to PDF -- Low. GrabzIt uses HTMLToPDF(html, options); IronPDF replaces it with a single RenderHtmlAsPdf call.
  • URL to PDF -- Low. GrabzIt uses URLToPDF(url, options); IronPDF replaces it with RenderUrlAsPdf.
  • HTML/URL to image -- Medium. GrabzIt has HTMLToImage / URLToImage as first-class operations. IronPDF reaches images through pdf.ToBitmap() after rendering, which is a different shape.
  • Screenshot capture -- High. GrabzIt-specific feature; IronPDF does not do general-purpose web screenshots, only image rasterisation of rendered PDFs.
  • Merge PDFs -- Low. Not part of GrabzIt; teams typically bolt on PdfSharp or iTextSharp. IronPDF has PdfDocument.Merge built in.
  • Watermark -- Low. GrabzIt offers a dashboard-configured CustomWaterMark; IronPDF replaces it with ApplyWatermark using inline HTML.
  • Password protection -- Low. GrabzIt has a basic Password option; IronPDF replaces it with full SecuritySettings covering owner password and permissions.
  • Custom headers/footers -- Medium. GrabzIt limits these to pre-configured templates; IronPDF uses inline HtmlHeaderFooter.
  • Async batch rendering -- Medium. GrabzIt does this with API-level queueing and callbacks; IronPDF uses in-process parallel rendering.
  • Retry / fallback logic -- Medium. Already in your code today; review it because the failure modes change from network errors to renderer/memory pressure.
  • Data residency compliance -- High value. The architectural shift to in-process rendering removes the concern entirely.
  • Custom CSS / fonts -- Medium. Cloud renderer limits give way to local fonts; test your specific templates.

What Scenarios Favor Which Library?

  • If you need screenshot capture (not just PDF), GrabzIt remains relevant -- IronPDF does not replace general web-screenshot use cases.
  • For air-gapped or offline deployments, IronPDF or another local renderer is the only option, since GrabzIt requires the cloud API.
  • If you have data residency or compliance requirements, a local renderer is required; evaluate IronPDF, PuppeteerSharp, or wkhtmltopdf.
  • If you are scaling concurrent PDF generation, evaluate a local renderer and load test both before committing.

How Do I Set Up IronPDF After Removing GrabzIt?

First, find all GrabzIt references in your codebase so you know what needs to change.

Find GrabzIt References

# Find all files referencing GrabzIt
rg -l "GrabzIt" --type cs

# Find specific client instantiation
rg "GrabzItClient" --type cs -n

# Find all using statements
rg "using GrabzIt" --type cs -n

# Find HTMLToPDF / URLToPDF / HTMLToImage call sites
rg "HTMLToPDF|URLToPDF|HTMLToImage" --type cs -n

Then swap the NuGet packages.

Remove GrabzIt, Install IronPDF

# Remove GrabzIt NuGet package
dotnet remove package GrabzIt

# Install IronPDF
dotnet add package IronPdf

# Restore
dotnet restore

Next, replace the license configuration. GrabzIt requires an API key and secret at client instantiation, sent with every cloud request. IronPDF uses a single license key set once at app startup.

Before (GrabzIt API Key Setup)

using GrabzIt;

// GrabzIt requires your API key + secret at client instantiation
var grabzIt = new GrabzItClient("YOUR_APPLICATION_KEY", "YOUR_APPLICATION_SECRET");
// Key is sent with every API request to the cloud

After (IronPDF License Key)

using IronPdf;

// Set once at app startup -- appsettings or environment variable
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

Finally, update your namespace imports.

Before (GrabzIt Namespaces)

using GrabzIt;
using GrabzIt.Parameters;

After (IronPDF Namespaces)

using IronPdf;
using IronPdf.Rendering; // for ChromePdfRenderOptions

How Do the Core APIs Map Between GrabzIt and IronPDF?

Below are the mapping references covering namespaces, core classes, document loading, page operations, and merge/split.

Namespace Mapping

  • GrabzIt -> IronPdf -- top-level namespace.
  • GrabzIt.Parameters -> IronPdf.Rendering -- options and config classes.
  • GrabzIt.Screenshots -> N/A -- general web-screenshot feature is not in IronPDF.

Core Class Mapping

  • GrabzItClient -> ChromePdfRenderer -- main rendering entry point.
  • PDFOptions -> ChromePdfRenderOptions (used as renderer.RenderingOptions) -- render configuration.
  • ImageOptions -> ChromePdfRenderOptions plus pdf.ToBitmap() -- IronPDF rasterises a rendered PDF rather than rendering directly to image.
  • N/A -> PdfDocument -- represents a loaded or generated PDF.

Document Loading Methods

  • HTML string: HTMLToPDF(html, options) -> renderer.RenderHtmlAsPdf(html).
  • URL: URLToPDF(url, options) -> renderer.RenderUrlAsPdf(url).
  • HTML file: FileToPDF(path) -> renderer.RenderHtmlFileAsPdf(path).
  • HTML to image: HTMLToImage(html, options) -> renderer.RenderHtmlAsPdf(html) then pdf.ToBitmap().
  • Existing PDF: N/A -> PdfDocument.FromFile(path).

Page Operations

  • Page count: N/A in GrabzIt -> pdf.PageCount.
  • Extract page: N/A in GrabzIt -> pdf.CopyPages(startIndex, endIndex).
  • Page size: options.PageSize = PageSize.A4 -> RenderingOptions.PaperSize = PdfPaperSize.A4.
  • Margins: options.MarginTop etc. -> RenderingOptions.MarginTop etc.

Merge/Split Operations

  • Merge PDFs: not a GrabzIt feature -> PdfDocument.Merge(pdf1, pdf2).
  • Split PDF: not a GrabzIt feature -> pdf.CopyPages(0, n).

What Does a Basic HTML-to-PDF Migration Look Like?

This is the most common migration path. GrabzIt queues the conversion against the cloud API and you either wait via SaveTo polling or receive a callback. IronPDF renders in-process with a single synchronous call.

Before (GrabzIt HTML to PDF)

using GrabzIt;
using GrabzIt.Parameters;
using System;

class HtmlToPdfExample
{
    static void Main(string[] args)
    {
        var grabzIt = new GrabzItClient("YOUR_APPLICATION_KEY", "YOUR_APPLICATION_SECRET");

        var options = new PDFOptions();
        options.CustomId = "my-pdf";

        grabzIt.HTMLToPDF(
            "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>",
            options);

        // SaveTo blocks while polling the cloud API for the result
        bool saved = grabzIt.SaveTo("invoice.pdf");
        if (!saved)
            Console.WriteLine("Conversion failed or timed out");
        else
            Console.WriteLine("Saved: invoice.pdf");
    }
}

After (IronPDF)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>"
);
pdf.SaveAs("invoice.pdf");
// Docs: https://ironpdf.com/how-to/html-string-to-pdf/

How Do I Migrate PDF Merging from GrabzIt?

Merging is not a native GrabzIt feature. Teams typically bolt on a second library like PdfSharp or iTextSharp just for this. With IronPDF, merging is built in and you can drop that extra dependency.

Before (GrabzIt with PdfSharp Workaround)

using System.IO;
using PdfSharp.Pdf;      // Common workaround -- teams add a second library
using PdfSharp.Pdf.IO;

class MergePdfsExample
{
    static void Main()
    {
        // Generate each PDF via GrabzIt separately, then merge with PdfSharp
        // (or iTextSharp, or another library)
        using var output = new PdfDocument();

        foreach (var filePath in new[] { "part1.pdf", "part2.pdf" })
        {
            using var input = PdfReader.Open(filePath, PdfDocumentOpenMode.Import);
            foreach (PdfPage page in input.Pages)
                output.AddPage(page);
        }

        output.Save("merged.pdf");
        Console.WriteLine("Merged.");
    }
}

After (IronPDF Native Merge)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf1 = PdfDocument.FromFile("part1.pdf");
var pdf2 = PdfDocument.FromFile("part2.pdf");
var merged = PdfDocument.Merge(pdf1, pdf2);
merged.SaveAs("merged.pdf");
// Docs: https://ironpdf.com/how-to/merge-or-split-pdfs/

How Do I Add Watermarks After Migrating?

GrabzIt does support watermarks, but only ones you pre-create in the GrabzIt dashboard and reference by ID via SetCustomWaterMark. You cannot define watermark content inline in code, and you cannot adjust opacity, rotation, or per-document text without round-tripping through the dashboard. IronPDF takes inline HTML at the call site, so the watermark lives in your codebase next to everything else.

Before (GrabzIt CustomWaterMark)

using GrabzIt;
using GrabzIt.Parameters;

class WatermarkExample
{
    static void Main()
    {
        var grabzIt = new GrabzItClient("YOUR_APPLICATION_KEY", "YOUR_APPLICATION_SECRET");

        var options = new PDFOptions();
        // Watermark must be pre-created in the GrabzIt dashboard,
        // then referenced here by its ID
        options.SetCustomWaterMark(
            "watermark123",
            HorizontalPosition.Center,
            VerticalPosition.Middle);

        grabzIt.HTMLToPDF("<h1>Confidential Report</h1>", options);
        grabzIt.SaveTo("watermarked.pdf");
    }
}

After (IronPDF)

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<h1>Confidential Report</h1>");

// Inline HTML watermark -- no dashboard round-trip
pdf.ApplyWatermark(
    "<div style='color:red; font-size:48px; font-weight:bold; transform:rotate(-45deg);'>CONFIDENTIAL</div>",
    opacity: 30,
    VerticalAlignment.Middle,
    HorizontalAlignment.Center);

pdf.SaveAs("watermarked.pdf");
// Docs: https://ironpdf.com/how-to/custom-watermark/

How Do I Migrate Password Protection?

GrabzIt offers a single Password option on PDFOptions -- one user password, no separate owner password, no fine-grained permission flags. IronPDF replaces it with full SecuritySettings: separate user and owner passwords, plus print/edit/copy permissions.

Before (GrabzIt Password Option)

using GrabzIt;
using GrabzIt.Parameters;

class PasswordExample
{
    static void Main()
    {
        var grabzIt = new GrabzItClient("YOUR_APPLICATION_KEY", "YOUR_APPLICATION_SECRET");

        var options = new PDFOptions();
        options.Password = "secretpassword";

        grabzIt.HTMLToPDF("<h1>Sensitive Report</h1>", options);
        grabzIt.SaveTo("protected.pdf");
    }
}

After (IronPDF)

using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<h1>Sensitive Report</h1>");

pdf.SecuritySettings.OwnerPassword = "ownerpass";
pdf.SecuritySettings.UserPassword = "secretpassword";
pdf.SecuritySettings.AllowUserCopyPasteContent = false;
pdf.SecuritySettings.AllowUserPrinting = IronPdf.Security.PdfPrintSecurity.FullPrintRights;
pdf.SaveAs("protected.pdf");
// Docs: https://ironpdf.com/how-to/pdf-permissions-passwords/

What Are the Critical Gotchas During Migration?

There are four areas that catch teams during migration.

Page indexing. IronPDF uses 0-based page indexing. If your GrabzIt code references page 1 as the first page (1-based), adjust all page index references.

// If you had page 1 logic elsewhere:
// GrabzIt: page 1 = first page
// IronPDF: page 0 = first page
var firstPage = pdf.Pages[0]; // not [1]

Network vs in-process model. GrabzIt is fundamentally a network call. Your existing retry/timeout/circuit-breaker logic around those HTTP calls should be reviewed. It may no longer be necessary, or it may need to be rewritten around different failure modes like renderer startup and memory pressure.

Screenshot use case gap. IronPDF does not generate screenshots (PNG/JPEG captures of web pages). If your codebase uses GrabzIt's screenshot capability alongside PDF generation, you will need a separate tool for that use case. PuppeteerSharp is a common .NET option. Plan this before starting the migration.

Async patterns. GrabzIt is inherently async (cloud callback model). IronPDF has both sync and async APIs. If you are migrating to async IronPDF rendering, see the IronPDF async documentation.

How Should I Handle Performance and Concurrency?

There are three patterns to follow for optimal performance with IronPDF.

Renderer reuse. Instantiating ChromePdfRenderer is relatively lightweight, but doing it repeatedly in tight loops is inefficient. Reuse instances where possible.

Batch Rendering with Reuse

// Preferred for batch work
var renderer = new ChromePdfRenderer();
// Reuse renderer across renders in same scope
foreach (var html in htmlBatch)
{
    var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"output_{Guid.NewGuid()}.pdf");
}

Concurrency. For concurrent rendering, create separate renderer instances per thread rather than sharing one. See parallel rendering examples.

Per-Thread Renderer Instantiation

// Per-thread renderer instantiation is safe
Parallel.ForEach(htmlBatch, html =>
{
    var renderer = new ChromePdfRenderer(); // separate instance per thread
    var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"{Guid.NewGuid()}.pdf");
});

Disposal. PdfDocument implements IDisposable. Use using blocks or explicit disposal in long-running services.

Proper Disposal Pattern

using var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");
// Disposed at end of block

A few edge cases worth flagging: JavaScript-heavy pages may have timing issues with async JS content (use WaitFor options if needed), local fonts on your rendering server need to be present since GrabzIt managed this on their end, and local rendering has a higher memory footprint than a thin HTTP client so load test before going to production.

Frequently Asked Questions

Can IronPDF replace GrabzIt for screenshot capture?

No. IronPDF does not generate screenshots (PNG/JPEG captures of web pages). If your codebase uses GrabzIt's screenshot capability, you will need a separate tool for that use case. PuppeteerSharp is a common .NET alternative for screenshots.

Do I need to change my page indexing when migrating?

Yes. GrabzIt uses 1-based page indexing while IronPDF uses 0-based indexing. Audit all page index references in your code and subtract 1 from each. For example, pdf.Pages[0] is the first page in IronPDF, not pdf.Pages[1].

Can I keep my existing retry and circuit-breaker logic?

You should review it. GrabzIt is a network call, so retry/timeout logic made sense for HTTP failures. With IronPDF rendering in-process, the failure modes are different -- renderer startup issues, memory pressure, and font availability. Your existing retry logic may no longer be necessary, or it may need rewriting around these new failure modes.

Will my HTML templates render the same way in IronPDF?

They should be close, but test each one. Both use Chromium-based rendering, but the Chromium version and configuration differ. CSS/JS fidelity depends on the specific version. Render each template and visually compare the output before going to production. Pay special attention to custom fonts, which you now manage locally.

Can I remove secondary PDF libraries like PdfSharp or iTextSharp after migrating?

If those libraries were only present to supplement GrabzIt for features like merging, watermarking, or password protection, then yes. IronPDF handles merging, watermarking, and password protection natively. Audit your usage of those secondary libraries to confirm they are not used elsewhere before removing them.

Conclusion

The main structural difference in this migration is not the API surface -- it is the architectural shift from a cloud-dependent HTTP client to an in-process renderer. Your error handling, retry logic, and async patterns all need revisiting with that in mind.

Most of the code changes are straightforward: swap the NuGet package, replace GrabzItClient with ChromePdfRenderer, switch HTMLToPDF/URLToPDF calls to RenderHtmlAsPdf/RenderUrlAsPdf, update namespace imports, and adjust page indexing from 1-based to 0-based. The real win is dropping secondary libraries like PdfSharp or iTextSharp that you were using to patch GrabzIt's missing merge/split, plus moving watermark and password configuration out of the GrabzIt dashboard and into your codebase.

The one area where GrabzIt retains relevance is screenshot capture. If that is part of your workflow, plan a separate solution before you begin the migration. For everything else, the effort and mapping bullet lists above should give you a clear picture of what is involved. Load test under realistic concurrency before going to production, and watch memory usage in the first week of live traffic.
