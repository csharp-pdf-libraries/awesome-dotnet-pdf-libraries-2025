Migrating from ZetPDF to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate from ZetPDF to IronPDF?

ZetPDF (zetpdf.com) is a commercial .NET PDF SDK whose API closely resembles PDFsharp — to the point that the PDFsharp community has openly questioned whether it is a closed fork. It is not distributed on NuGet; installation is a ZIP download from zetpdf.com, after which you reference the ZetPDF.dll manually. The vendor's public release activity has been quiet since 2021, and the documented feature surface centers on viewer/print, annotations, AES-256 encryption, form fields, and text extraction.

The migration triggers I see most often follow from that shape. First, the API is coordinate-based. Building a PDF in ZetPDF means opening an XGraphics surface and calling DrawString, DrawRectangle, DrawImage at exact pixel offsets, calculating page breaks by hand, and measuring text manually. There is no HTML or CSS path in the documented surface, so any change in layout means recalculating coordinates by hand.

Second, the missing high-level helpers. ZetPDF does not ship a documented HTML-to-PDF or URL-to-PDF converter, nor a one-line merge helper, nor a built-in watermark API. Teams that need those features end up gluing ZetPDF to a second library or writing the page-iteration loops themselves.

Third, distribution and maintenance. The ZIP-only delivery and the long pause in public releases make dependency hygiene awkward: there is no automated package update, no transitive resolution, and limited community traffic on Stack Overflow when you hit an edge case. Smaller ecosystems carry higher update risk, especially when a .NET runtime upgrade or a CI base image change exposes something the vendor has not retested recently.

What Does the Migration Effort Look Like?

I want to give you a realistic picture of the work involved. Here is a feature-by-feature breakdown so you can estimate the scope before committing.

Effort by Feature

  • HTML string to PDF — High in ZetPDF, Low in IronPDF. ZetPDF has no documented HTML-to-PDF converter, so the existing code is almost certainly hand-laid XGraphics.DrawString calls. In IronPDF this collapses to ChromePdfRenderer.RenderHtmlAsPdfAsync() followed by pdf.SaveAs(path).
  • URL to PDF — High in ZetPDF, Low in IronPDF. No documented URL-to-PDF path in ZetPDF. IronPDF: renderer.RenderUrlAsPdfAsync(url).
  • Save to file — Low. ZetPDF: document.Save(path). IronPDF: pdf.SaveAs(path).
  • Save to stream — Low. IronPDF exposes pdf.Stream and pdf.BinaryData directly on the document.
  • Load existing PDF — Low. ZetPDF: PdfReader.Open(path). IronPDF: PdfDocument.FromFile(path).
  • Merge PDFs — Medium. ZetPDF has no one-line merge helper; merging is a manual page-iteration loop following the PDFsharp pattern. IronPDF: PdfDocument.Merge(IEnumerable<PdfDocument>).
  • Watermark — Medium. ZetPDF has no native watermark helper; teams hand-roll a rotated DrawString with a translucent brush. IronPDF: TextStamper / ImageStamper via ApplyStamp, or ApplyWatermark with HTML/CSS.
  • Password protection — Low. ZetPDF exposes AES-256 settings on the document. IronPDF exposes pdf.SecuritySettings for user/owner passwords and permission flags.
  • Text extraction — Low. Both libraries support text extraction; ZetPDF lists it among documented features and IronPDF exposes pdf.ExtractAllText().
  • Concurrent rendering — Medium. ZetPDF thread-safety is not clearly documented; IronPDF supports a per-task ChromePdfRenderer with Task.WhenAll.

Decision Matrix

  • If your application generates PDFs from HTML, reports, or web content, IronPDF's Chromium renderer removes the need to hand-position every element.
  • If your team is maintaining a large body of XGraphics drawing code and the layout rarely changes, the cost of rewriting may not pay back — budget for a phased migration instead.
  • If a .NET upgrade or container base image change has surfaced compatibility risk against a library that has not seen public updates since 2021, that risk is the migration trigger.
  • If you only need viewer, annotation, and form-fill features that ZetPDF does cover well, the migration urgency is lower; weigh the gap-filling features (HTML, watermark, merge, signing) against the cost of holding two libraries.

How Do You Set Up IronPDF and Replace ZetPDF?

I will walk you through the core migration. First, find your existing ZetPDF references, then swap the dependency, and finally update your code.

Finding ZetPDF References in Your Codebase

# Find ZetPDF usage (namespace casing is ZetPDF, uppercase)
rg -l "ZetPDF\b" --type cs
rg "XGraphics|XFont|PdfReader|ZetPDF\b" --type cs -n

# Find the ZetPDF.dll assembly reference inside your csproj files
grep -r "ZetPDF" *.csproj **/*.csproj 2>/dev/null

# Find in Dockerfiles and CI pipelines
grep -r "ZetPDF" Dockerfile* .github/**/*.yml .gitlab-ci.yml 2>/dev/null

Removing ZetPDF and Installing IronPDF

ZetPDF is not on NuGet. Remove the manual <Reference Include="ZetPDF"> entry (and the HintPath pointing at the downloaded DLL) from your .csproj, then add IronPDF from NuGet:

# Install IronPDF
dotnet add package IronPdf

dotnet restore

License Configuration

using IronPdf;

// https://ironpdf.com/how-to/license-keys/
IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY")
    ?? throw new InvalidOperationException("IRONPDF_LICENSE_KEY not set");

Namespace Swap

Before:

using ZetPDF;
using ZetPdf.Drawing;
using ZetPdf.Fonts;

After:

using IronPdf;
using IronPdf.Rendering;

Basic PDF Generation

Before (ZetPDF):

using ZetPDF;
using ZetPdf.Drawing;
using System;

var document = new PdfDocument();
var page = document.AddPage();
var graphics = XGraphics.FromPdfPage(page);
var font = new XFont("Arial", 24, XFontStyle.Bold);

graphics.DrawString("Hello", font, XBrushes.Black, new XPoint(50, 50));
document.Save("output.pdf");
Console.WriteLine("Saved output.pdf");

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Hello</h1></body></html>");
pdf.SaveAs("output.pdf");

Console.WriteLine($"Saved output.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

What Does Each Before/After Migration Look Like in Practice?

Let me walk through four complete migration examples so you can see exactly what changes in real code.

Tabular Document (Invoice)

Before (ZetPDF):

using ZetPDF;
using ZetPdf.Drawing;
using System;

var document = new PdfDocument();
var page = document.AddPage();
var graphics = XGraphics.FromPdfPage(page);

var titleFont = new XFont("Arial", 18, XFontStyle.Bold);
var headerFont = new XFont("Arial", 11, XFontStyle.Bold);
var bodyFont = new XFont("Arial", 11);

graphics.DrawString("Invoice #2024-0099", titleFont, XBrushes.Black,
    new XPoint(50, 60));

// Header row
double x = 50, y = 100, colWidth = 200, rowHeight = 22;
graphics.DrawRectangle(XBrushes.LightGray, x, y, colWidth * 2, rowHeight);
graphics.DrawString("Item", headerFont, XBrushes.Black, x + 6, y + 15);
graphics.DrawString("Price", headerFont, XBrushes.Black, x + colWidth + 6, y + 15);

// Data row
y += rowHeight;
graphics.DrawRectangle(XPens.Black, x, y, colWidth * 2, rowHeight);
graphics.DrawString("Widget", bodyFont, XBrushes.Black, x + 6, y + 15);
graphics.DrawString("$149.00", bodyFont, XBrushes.Black, x + colWidth + 6, y + 15);

document.Save("invoice.pdf");
Console.WriteLine("Saved invoice.pdf");

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var html = @"
    <html>
    <head><style>
    body { font-family: Arial; padding: 40px; }
    table { width: 100%; border-collapse: collapse; }
    th, td { border: 1px solid #ccc; padding: 6px; }
    </style></head>
    <body>
        <h1>Invoice #2024-0099</h1>
        <table>
            <tr><th>Item</th><th>Price</th></tr>
            <tr><td>Widget</td><td>$149.00</td></tr>
        </table>
    </body></html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");
Console.WriteLine($"Saved invoice.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

Merge PDFs

Before (ZetPDF):

using ZetPDF;
using ZetPdf.IO;
using System;

// ZetPDF has no one-line merge helper. Following the PDFsharp pattern,
// you open each input and copy pages into a fresh output document.
var output = new PdfDocument();

foreach (var path in new[] { "section1.pdf", "section2.pdf" })
{
    var input = PdfReader.Open(path, PdfDocumentOpenMode.Import);
    for (int i = 0; i < input.PageCount; i++)
    {
        output.AddPage(input.Pages[i]);
    }
}

output.Save("merged.pdf");
Console.WriteLine("PDFs merged successfully");

After (IronPDF):

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

var results = await Task.WhenAll(
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 1</h1></body></html>"),
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 2</h1></body></html>")
);

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("merged.pdf");
Console.WriteLine($"Merged: {merged.PageCount} pages");

Watermark

Before (ZetPDF):

using ZetPDF;
using ZetPdf.Drawing;
using ZetPdf.IO;
using System;

// ZetPDF has no native watermark helper. The common pattern is to open the
// document, draw a rotated translucent string on each page via XGraphics,
// and save.
var document = PdfReader.Open("document.pdf", PdfDocumentOpenMode.Modify);
var font = new XFont("Arial", 72, XFontStyle.Bold);
var brush = new XSolidBrush(XColor.FromArgb(40, 128, 128, 128));

foreach (var page in document.Pages)
{
    var graphics = XGraphics.FromPdfPage(page);
    graphics.RotateTransform(-45);
    graphics.DrawString("CONFIDENTIAL", font, brush, new XPoint(-100, 500));
}

document.Save("watermarked.pdf");

After (IronPDF):

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Document</h1></body></html>");

// https://ironpdf.com/how-to/custom-watermark/
var watermark = new TextStamper
{
    Text = "CONFIDENTIAL",
    FontColor = IronSoftware.Drawing.Color.Gray,
    Opacity = 15,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("watermarked.pdf");

Password Protection

Before (ZetPDF):

using ZetPDF;
using ZetPdf.Drawing;
using ZetPdf.Security;
using System;

var document = new PdfDocument();
var page = document.AddPage();
var graphics = XGraphics.FromPdfPage(page);
graphics.DrawString("Protected", new XFont("Arial", 24, XFontStyle.Bold),
    XBrushes.Black, new XPoint(50, 80));

// AES-256 encryption is exposed on PdfSecuritySettings.
var security = document.SecuritySettings;
security.UserPassword = "open123";
security.OwnerPassword = "admin456";

document.Save("secured.pdf");

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Protected</h1></body></html>");

// https://ironpdf.com/how-to/pdf-permissions-passwords/
pdf.SecuritySettings.UserPassword = "open123";
pdf.SecuritySettings.OwnerPassword = "admin456";

pdf.SaveAs("secured.pdf");

How Do the Two Libraries Compare Feature by Feature?

Here are the full mapping tables I use when planning a migration. These cover the overall comparison, namespaces, core classes, document loading, page operations, and merge/split.

Overall Comparison

  • Focus. ZetPDF centers on coordinate-based PDF construction, viewing/printing, annotations, AES-256 encryption, form fields, and text extraction. IronPDF centers on HTML-to-PDF rendering plus broader PDF manipulation, editing, security, and signing.
  • Pricing. ZetPDF sells perpetual licenses tiered by deployment scope — Single Project at $299, Multi at $449, OEM at $599. IronPDF licensing is published at ironsoftware.com.
  • Distribution. ZetPDF ships as a ZIP from zetpdf.com/download; there is no NuGet package. IronPDF is distributed via NuGet as IronPdf.
  • API style. ZetPDF inherits a PDFsharp-style coordinate API: open an XGraphics surface and call DrawString, DrawRectangle, DrawImage at exact offsets. IronPDF exposes a renderer (ChromePdfRenderer) that returns a PdfDocument you operate on.
  • Layout engine. ZetPDF has no documented HTML or CSS rendering. IronPDF uses an embedded Chromium build that renders HTML, CSS, and JavaScript before producing the PDF.
  • Maintenance. ZetPDF has shown no public release activity since 2021. IronPDF is on an active release cadence.
  • Namespace. ZetPDF lives under ZetPDF; IronPDF lives under IronPdf.

Namespace Mapping

  • ZetPDF → IronPdf — core namespace.
  • ZetPdf.Drawing → N/A — obsolete; HTML/CSS replaces XGraphics drawing.
  • ZetPdf.IO → IronPdf — document loading via PdfDocument.FromFile.
  • N/A → IronPdf.Rendering — rendering config.
  • N/A → IronPdf.Editing — watermark and stamp types.

Core Class Mapping

  • PdfDocument + XGraphics → ChromePdfRenderer + HTML/CSS — rendering entry point.
  • PdfReader.Open → PdfDocument.FromFile — load an existing PDF.
  • document.Save → pdf.SaveAs — write the document.
  • Manual foreach over input.Pages → PdfDocument.Merge — combine multiple PDFs.
  • Hand-rolled rotated DrawString → TextStamper / ApplyWatermark — watermarks.

Document Loading Methods

  • HTML string. Not part of ZetPDF's documented surface. IronPDF: renderer.RenderHtmlAsPdfAsync(html).
  • URL. Not part of ZetPDF's documented surface. IronPDF: renderer.RenderUrlAsPdfAsync(url).
  • Load existing PDF. ZetPDF: PdfReader.Open(path, PdfDocumentOpenMode.Modify). IronPDF: PdfDocument.FromFile(path).
  • Save. ZetPDF: document.Save(path). IronPDF: pdf.SaveAs(path).

Page Operations

  • Page count. ZetPDF: document.PageCount. IronPDF: pdf.PageCount.
  • Add page. ZetPDF: document.AddPage(). IronPDF: pages are produced by HTML rendering and CSS page-break rules.
  • Remove page. ZetPDF: document.Pages.RemoveAt(index). IronPDF: pdf.RemovePages(index).
  • Extract text. ZetPDF lists text extraction in its feature surface. IronPDF: pdf.ExtractAllText().

Merge/Split Operations

  • Merge. ZetPDF: manual loop opening each PdfReader and copying pages into an output document. IronPDF: PdfDocument.Merge(doc1, doc2).
  • Split. ZetPDF: copy a page range into a new document via the same loop pattern. IronPDF: see the merge and split guide.

What Are the Frequently Asked Questions?

I get these questions on nearly every ZetPDF migration I advise on. Here they are with direct answers.

How Do I Handle Container and CI Environment Differences?

ZetPDF markets itself as 100% managed code, which simplifies deployment relative to libraries with native dependencies. The deployment friction with ZetPDF is usually upstream of runtime: the ZIP-based distribution model means the DLL is checked into source or fetched manually, and version drift between dev and CI is easy to introduce. After migrating, IronPDF is restored from NuGet like any other package, which removes that whole class of friction.

IronPDF's Chromium component does have its own Linux system library requirements. Install them explicitly in your Dockerfile:

# IronPDF system dependencies for Linux:
FROM mcr.microsoft.com/dotnet/aspnet:8.0

RUN apt-get update && apt-get install -y \
    libnss3 \
    libatk-bridge2.0-0 \
    libdrm2 \
    libgbm1 \
    libxkbcommon0 \
    fonts-liberation \
    && rm -rf /var/lib/apt/lists/*

COPY --from=build /app/out .
ENTRYPOINT ["dotnet", "YourApp.dll"]

Why Does It Work After One Render but Fail on Subsequent Requests?

This is often a disposal issue. An object that should be reused per-request is being disposed after the first use, or a shared state object has single-use semantics. With IronPDF, the safe pattern is one renderer per operation:

using IronPdf;

// The safe pattern — one renderer per operation:
public async Task<byte[]> GeneratePdfAsync(string html)
{
    var renderer = new ChromePdfRenderer(); // new instance per call
    using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    return pdf.BinaryData; // copy bytes before pdf is disposed
}

Why Are PDFs Rendering with Black or Empty Pages in Production?

Font packages are the first suspect. IronPDF renders via Chromium and uses system fonts for font-family declarations. If a font requested in CSS is not available, Chromium falls back to a system default which may render differently or missing. Explicit web font loading is more reliable:

// Embed fonts via CSS to avoid system font dependency
var html = @"
    <html>
    <head>
    <link href='https://fonts.googleapis.com/css2?family=Inter:wght@400;700' rel='stylesheet'>
    <style>body { font-family: 'Inter', Arial, sans-serif; }</style>
    </head>
    <body><h1>Report</h1></body>
    </html>";

How Do I Fix the License Key Not Found Error in CI?

The IronPdf.Exceptions.IronPdfLicenseException appears in CI but not in local development. Add the license key to your CI secrets and reference it in your pipeline configuration:

# GitHub Actions
- name: Generate PDFs
  env:
    IRONPDF_LICENSE_KEY: ${{ secrets.IRONPDF_LICENSE_KEY }}
  run: dotnet test

# Azure DevOps
- task: DotNetCoreCLI@2
  env:
    IRONPDF_LICENSE_KEY: $(IRONPDF_LICENSE_KEY)
  inputs:
    command: test

// In startup code — fail early with a clear message
IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY")
    ?? throw new InvalidOperationException(
        "IRONPDF_LICENSE_KEY not set. Add it to CI secrets and local .env.");

How Do I Handle Parallel Rendering for Server-Side Workloads?

IronPDF supports concurrent rendering with separate renderer instances. Each task gets its own renderer for safe parallel work:

using IronPdf;
using System.Linq;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// https://ironpdf.com/examples/parallel/
var pdfs = await Task.WhenAll(htmlJobs.Select(async html =>
{
    var renderer = new ChromePdfRenderer();
    return await renderer.RenderHtmlAsPdfAsync(html);
}));

foreach (var pdf in pdfs) pdf.Dispose();
// See: https://ironpdf.com/how-to/async/

What Should I Watch Out for with Stream and Bytes Output?

The output pattern changes between ZetPDF and IronPDF. ZetPDF's document.Save writes to a file path or a Stream. IronPDF returns a PdfDocument that you can save with SaveAs or read bytes from via BinaryData:

// IronPDF equivalent:
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
var bytes = pdf.BinaryData; // equivalent byte[] copy
// OR: pdf.SaveAs(path);
// pdf disposed by 'using'

IronPDF uses 0-based page indexing. ZetPDF's document.Pages indexer follows the same PDFsharp-style 0-based convention, so the indexing math typically transfers cleanly — but audit page-manipulation code for off-by-one issues before testing.

What Is the Recommended Migration Path for Your Team?

The hardest part of this migration is conceptual rather than mechanical. ZetPDF's coordinate-based drawing model and IronPDF's HTML/CSS rendering model do not map line-for-line, so the work is rewriting layout logic in HTML and CSS rather than translating one API call to another. The payoff is that the rewritten layout is shorter, easier to maintain, and adapts automatically to content changes.

Capture the call sites from the codebase before starting:

# Capture all ZetPDF usage for the migration map
rg "XGraphics|XFont|PdfReader|ZetPDF" --type cs -n -A 2 > zetpdf-usage-map.txt
cat zetpdf-usage-map.txt
# Each line becomes a migration point

Do not remove ZetPDF code until each usage point is mapped to its IronPDF equivalent and tested. Work through the migration in order: pre-migration assessment, code migration, testing, and post-migration cleanup. The disposal pattern matters — wrap all PdfDocument usage in using blocks:

using IronPdf;

var renderer = new ChromePdfRenderer();
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("output.pdf");
// pdf disposed automatically
