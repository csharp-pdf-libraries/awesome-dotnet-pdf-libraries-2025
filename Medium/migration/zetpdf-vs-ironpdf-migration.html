Migrating from ZetPDF to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate from ZetPDF to IronPDF?

The build server and the developer machine diverge. ZetPDF renders correctly on every Windows development box the team has tested, but the GitHub Actions runner on Ubuntu fails with a native library error that nobody can reproduce locally. The CI fix that worked last month stopped working after a runner image update. The Docker build that deployed cleanly to staging won't deploy to production because the production container is a different base image.

Platform-specific library loading errors are not easy to debug in pipelines, and ZetPDF's narrow user community means Stack Overflow is quiet when you search for the exception message. I have seen this pattern repeatedly, and here are the migration triggers that come up most often.

First, CI/CD platform divergence. Native component behavior differs between Windows dev machines and Linux-based CI runners. Second, container image sensitivity. Native library loading depends on exact system library versions, and base image updates break deployments.

Maintenance uncertainty is another factor. Smaller library ecosystems carry higher update risk, and limited community support means fewer Stack Overflow questions and slower resolution when edge cases arise.

Then there is the API surface gap. ZetPDF exposes a small, high-level surface — HtmlToPdfConverter and PdfMerger — which covers basic generation and merging. Anything beyond that (watermarks, security, signing, forms, page-level editing) needs additional packages. Native dependency management means those native libraries require explicit management in Docker images and CI runners, and debugging friction from CI-only failures without local repro slows down incident response.

What Does the Migration Effort Look Like?

I want to give you a realistic picture of the work involved. Here is a feature-by-feature breakdown so you can estimate the scope before committing.

Effort by Feature

  • HTML string to PDF — Low. ZetPDF's HtmlToPdfConverter.ConvertHtmlToPdf(html, path) maps directly to ChromePdfRenderer.RenderHtmlAsPdfAsync() followed by pdf.SaveAs(path).
  • URL to PDF — Low. ZetPDF's HtmlToPdfConverter.ConvertUrlToPdf(url, path) maps to renderer.RenderUrlAsPdfAsync(url).
  • Save to file — Low. ZetPDF writes the file inside the converter call; IronPDF separates rendering from pdf.SaveAs(path).
  • Save to stream — Low. IronPDF exposes pdf.Stream and pdf.BinaryData; in ZetPDF you write to a path and read it back.
  • Load existing PDF — Low. IronPDF uses PdfDocument.FromFile(path); ZetPDF's surface is generation-focused, so loading-and-editing flows usually need a secondary package.
  • Merge PDFs — Medium. ZetPDF's PdfMerger.MergeFiles(paths, output) takes file paths; IronPDF's PdfDocument.Merge(IEnumerable<PdfDocument>) takes loaded documents, so the call shape changes.
  • Watermark — Medium. Not part of ZetPDF's documented surface; IronPDF uses TextStamper / ImageStamper.
  • Password protection — Low. Not part of ZetPDF's documented surface; IronPDF exposes pdf.SecuritySettings.
  • Text extraction — Medium. Not part of ZetPDF's documented surface; IronPDF uses pdf.ExtractAllText().
  • Concurrent rendering — Medium. Verify ZetPDF thread-safety in their docs; IronPDF supports a per-task ChromePdfRenderer with Task.WhenAll.

Decision Matrix

  • If CI/CD fails because of a platform-specific ZetPDF native issue, switching resolves the deployment constraint directly.
  • If maintenance and support are a hard requirement, switch once you have verified ZetPDF's release cadence is below your bar.
  • If ZetPDF is working with no deployment friction, weigh migration cost against the risk timeline before pulling the trigger.
  • If a .NET upgrade is blocked by ZetPDF compatibility, verify support at zetpdf.com and switch if the target framework is not on the matrix.

How Do You Set Up IronPDF and Replace ZetPDF?

I will walk you through the core migration in three steps. First, find your existing ZetPDF references, then swap the packages, and finally update your code.

Finding ZetPDF References in Your Codebase

# Find ZetPDF usage (namespace is ZetPDF, all uppercase)
rg -l "ZetPDF\b" --type cs
rg "HtmlToPdfConverter|PdfMerger|ZetPDF\b" --type cs -n

# Find native library loading patterns
rg "NativeLibrary\|DllImport\|Assembly\.Load" --type cs -n

# Find ZetPDF NuGet reference
grep -r "ZetPDF" *.csproj **/*.csproj 2>/dev/null

# Find in Dockerfiles and CI pipelines
grep -r "ZetPDF" Dockerfile* .github/**/*.yml .gitlab-ci.yml 2>/dev/null

Removing ZetPDF and Installing IronPDF

# Remove ZetPDF package
dotnet remove package ZetPDF

# Install IronPDF
dotnet add package IronPdf

dotnet restore

Step 1: License Configuration

using IronPdf;

// https://ironpdf.com/how-to/license-keys/
IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY")
    ?? throw new InvalidOperationException("IRONPDF_LICENSE_KEY not set");

Step 2: Namespace Swap

Before:

using ZetPDF;

After:

using IronPdf;
using IronPdf.Rendering;

Step 3: Basic PDF Generation

Before (ZetPDF):

using ZetPDF;
using System;

class Program
{
    static void Main()
    {
        var converter = new HtmlToPdfConverter();
        var html = "<html><body><h1>Hello</h1></body></html>";
        converter.ConvertHtmlToPdf(html, "output.pdf");
        Console.WriteLine("Saved output.pdf");
    }
}

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Hello</h1></body></html>");
pdf.SaveAs("output.pdf");

Console.WriteLine($"Saved output.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

HTML to PDF with Full Options

Before (ZetPDF):

using ZetPDF;
using System;

class HtmlToPdfBefore
{
    static void Main()
    {
        var html = @"
            <html>
            <head><style>
            body { font-family: Arial; padding: 40px; }
            table { width: 100%; border-collapse: collapse; }
            th, td { border: 1px solid #ccc; padding: 6px; }
            </style></head>
            <body>
                <h1>Invoice #2024-0099</h1>
                <table>
                    <tr><th>Item</th><th>Price</th></tr>
                    <tr><td>Widget</td><td>$149.00</td></tr>
                </table>
            </body></html>";

        var converter = new HtmlToPdfConverter();
        converter.ConvertHtmlToPdf(html, "invoice.pdf");

        Console.WriteLine("Saved invoice.pdf");
    }
}

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var html = @"
    <html>
    <head><style>
    body { font-family: Arial; padding: 40px; }
    table { width: 100%; border-collapse: collapse; }
    th, td { border: 1px solid #ccc; padding: 6px; }
    </style></head>
    <body>
        <h1>Invoice #2024-0099</h1>
        <table>
            <tr><th>Item</th><th>Price</th></tr>
            <tr><td>Widget</td><td>$149.00</td></tr>
        </table>
    </body></html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");
Console.WriteLine($"Saved invoice.pdf ({pdf.PageCount} page(s))");

Merge PDFs

Before (ZetPDF):

using ZetPDF;
using System;
using System.Collections.Generic;

class MergeBefore
{
    static void Main()
    {
        var converter = new HtmlToPdfConverter();
        converter.ConvertHtmlToPdf(
            "<html><body><h1>Section 1</h1></body></html>",
            "section1.pdf");
        converter.ConvertHtmlToPdf(
            "<html><body><h1>Section 2</h1></body></html>",
            "section2.pdf");

        var merger = new PdfMerger();
        var files = new List<string> { "section1.pdf", "section2.pdf" };
        merger.MergeFiles(files, "merged.pdf");
        Console.WriteLine("PDFs merged successfully");
    }
}

After (IronPDF):

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

var results = await Task.WhenAll(
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 1</h1></body></html>"),
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 2</h1></body></html>")
);

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("merged.pdf");
Console.WriteLine($"Merged: {merged.PageCount} pages");

Watermark

Before (ZetPDF):

using ZetPDF;
using System;

class WatermarkBefore
{
    static void Main()
    {
        // ZetPDF's documented surface (HtmlToPdfConverter, PdfMerger) does
        // not include a watermark/stamp API. Teams typically generate the
        // PDF with ZetPDF and then apply the stamp via a secondary library.
        var converter = new HtmlToPdfConverter();
        converter.ConvertHtmlToPdf(
            "<html><body><h1>Document</h1></body></html>",
            "document.pdf");

}

After (IronPDF):

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Document</h1></body></html>");

// https://ironpdf.com/how-to/custom-watermark/
var watermark = new TextStamper
{
    Text = "CONFIDENTIAL",
    FontColor = IronSoftware.Drawing.Color.Gray,
    Opacity = 15,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,

pdf.ApplyStamp(watermark);
pdf.SaveAs("watermarked.pdf");

How Do the Two Libraries Compare Feature by Feature?

Here are the full mapping tables I use when planning a migration. These cover the overall comparison, namespaces, core classes, document loading, page operations, and merge/split.

Overall Comparison

  • Focus. ZetPDF centers on HTML-to-PDF conversion and file-based merging. IronPDF covers HTML-to-PDF plus broader PDF manipulation, editing, security, and signing.
  • Pricing. Both are commercial; verify ZetPDF terms at zetpdf.com and IronPDF terms at ironsoftware.com.
  • API style. ZetPDF exposes high-level façades (HtmlToPdfConverter, PdfMerger) that take inputs and a destination path. IronPDF exposes a renderer (ChromePdfRenderer) that returns a PdfDocument you then operate on.
  • Learning curve. Both are approachable for .NET developers. IronPDF's surface is broader because the feature set is broader.
  • HTML rendering. Verify ZetPDF's underlying engine in their docs; IronPDF uses an embedded Chromium build.
  • Page indexing. Verify ZetPDF's indexing in their docs; IronPDF is 0-based.
  • Thread safety. Verify ZetPDF guidance in their docs; for IronPDF, verify concurrent-instance guidance in the IronPDF docs.
  • Namespace. ZetPDF lives under ZetPDF; IronPDF lives under IronPdf.

Namespace Mapping

  • ZetPDF → IronPdf — core namespace.
  • N/A → IronPdf.Rendering — rendering config.
  • N/A → IronPdf.Editing — watermark and stamp types.

Core Class Mapping

  • HtmlToPdfConverter → ChromePdfRenderer — HTML-to-PDF entry point.
  • PdfMerger → PdfDocument.Merge — combine multiple PDFs.
  • N/A → PdfDocument — loaded-document handle for manipulation.
  • N/A → ChromePdfRenderOptions — rendering options.
  • N/A → TextStamper — text watermark.

Document Loading Methods

  • HTML string. ZetPDF: converter.ConvertHtmlToPdf(html, path). IronPDF: renderer.RenderHtmlAsPdfAsync(html).
  • URL. ZetPDF: converter.ConvertUrlToPdf(url, path). IronPDF: renderer.RenderUrlAsPdfAsync(url).
  • Load existing PDF. Not part of ZetPDF's documented surface. IronPDF: PdfDocument.FromFile(path).
  • Save. ZetPDF writes the file inside the convert call. IronPDF: pdf.SaveAs(path).

Page Operations

  • Page count. Not part of ZetPDF's documented surface. IronPDF: pdf.PageCount.
  • Remove page. Not part of ZetPDF's documented surface. IronPDF: pdf.RemovePages(index).
  • Extract text. Not part of ZetPDF's documented surface. IronPDF: pdf.ExtractAllText().
  • Rotate. Not part of ZetPDF's documented surface. Verify the corresponding IronPDF API in their docs.

Merge/Split Operations

  • Merge. ZetPDF: new PdfMerger().MergeFiles(paths, output). IronPDF: PdfDocument.Merge(doc1, doc2).
  • Split. Not part of ZetPDF's documented surface. IronPDF: see the merge and split guide.

What Are the Frequently Asked Questions?

I get these questions on nearly every ZetPDF migration I advise on. Here they are with direct answers.

How Do I Handle the CI/CD Native Library Failure?

The DllNotFoundException: Unable to load shared library 'libzetpdf.so' pattern appears in CI on Ubuntu but not on Windows dev machines. Native library loading on Linux depends on LD_LIBRARY_PATH and system library presence. The .NET runtime's NativeLibrary.Load() behavior differs between Windows and Linux.

After migrating to IronPDF, the ZetPDF native library concern is eliminated. IronPDF's Chromium dependency has its own system library requirements. Install them explicitly in your Dockerfile:

# IronPDF system dependencies for Linux (verify complete list in IronPDF docs):
FROM mcr.microsoft.com/dotnet/aspnet:8.0

RUN apt-get update && apt-get install -y \
    libnss3 \
    libatk-bridge2.0-0 \
    libdrm2 \
    libgbm1 \
    libxkbcommon0 \
    fonts-liberation \
    && rm -rf /var/lib/apt/lists/*

COPY --from=build /app/out .
ENTRYPOINT ["dotnet", "YourApp.dll"]

Why Does It Work After One Render but Fail on Subsequent Requests?

This is often a disposal issue. An object that should be reused per-request is being disposed after the first use, or a shared state object has single-use semantics. With IronPDF, the safe pattern is one renderer per operation:

using IronPdf;

// The safe pattern — one renderer per operation:
public async Task<byte[]> GeneratePdfAsync(string html)
{
    var renderer = new ChromePdfRenderer(); // new instance per call
    using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    return pdf.BinaryData; // copy bytes before pdf is disposed
}

Why Are PDFs Rendering with Black or Empty Pages in Production?

Font packages are the first suspect. IronPDF renders via Chromium and uses system fonts for font-family declarations. If a font requested in CSS is not available, Chromium falls back to a system default which may render differently or missing. Explicit web font loading is more reliable:

// Embed fonts via CSS to avoid system font dependency
var html = @"
    <html>
    <head>
    <link href='https://fonts.googleapis.com/css2?family=Inter:wght@400;700' rel='stylesheet'>
    <style>body { font-family: 'Inter', Arial, sans-serif; }</style>
    </head>
    <body><h1>Report</h1></body>
    </html>";

How Do I Fix the License Key Not Found Error in CI?

The IronPdf.Exceptions.IronPdfLicenseException appears in CI but not in local development. Add the license key to your CI secrets and reference it in your pipeline configuration:

# GitHub Actions
- name: Generate PDFs
  env:
    IRONPDF_LICENSE_KEY: ${{ secrets.IRONPDF_LICENSE_KEY }}
  run: dotnet test

# Azure DevOps
- task: DotNetCoreCLI@2
  env:
    IRONPDF_LICENSE_KEY: $(IRONPDF_LICENSE_KEY)
  inputs:
    command: test

// In startup code — fail early with a clear message
IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY")
    ?? throw new InvalidOperationException(
        "IRONPDF_LICENSE_KEY not set. Add it to CI secrets and local .env.");

How Do I Handle Parallel Rendering for Server-Side Workloads?

IronPDF supports concurrent rendering with separate renderer instances. Each task gets its own renderer for safe parallel work:

using IronPdf;
using System.Linq;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// https://ironpdf.com/examples/parallel/
var pdfs = await Task.WhenAll(htmlJobs.Select(async html =>
{
    var renderer = new ChromePdfRenderer();
    return await renderer.RenderHtmlAsPdfAsync(html);
}));

foreach (var pdf in pdfs) pdf.Dispose();
// See: https://ironpdf.com/how-to/async/

What Should I Watch Out for with Stream and Bytes Output?

The output pattern changes between ZetPDF and IronPDF. ZetPDF's HtmlToPdfConverter.ConvertHtmlToPdf writes directly to a file path; IronPDF returns a PdfDocument that you can save with SaveAs or read bytes from via BinaryData:

// IronPDF equivalent:
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
var bytes = pdf.BinaryData; // equivalent byte[] copy
// OR: pdf.SaveAs(path);
// pdf disposed by 'using'

IronPDF uses 0-based page indexing. Verify ZetPDF's indexing in their documentation and audit any page manipulation code for off-by-one issues before testing.

What Is the Recommended Migration Path for Your Team?

The CI/CD divergence pattern — works locally, fails in pipeline — is the hardest kind of library bug to debug, and it is common with native-component dependencies on platforms the library was not primarily designed for. Once IronPDF replaces ZetPDF, that specific class of failure is resolved since IronPDF's Chromium dependencies are well-documented and consistently supported.

The main investment in this migration is mapping every ZetPDF call site to its IronPDF equivalent before removal. ZetPDF's documented surface is small — HtmlToPdfConverter for HTML and URL input, PdfMerger for combining files — so capture the call sites from the codebase itself before starting:

# Capture all ZetPDF usage for the migration map
rg "HtmlToPdfConverter|PdfMerger|ZetPDF" --type cs -n -A 2 > zetpdf-usage-map.txt
cat zetpdf-usage-map.txt
# Each line becomes a migration point

Do not remove ZetPDF code until each usage point is mapped to its IronPDF equivalent and tested. Work through the migration in order: pre-migration assessment, code migration, testing, and post-migration cleanup. The disposal pattern matters — wrap all PdfDocument usage in using blocks:

using IronPdf;

var renderer = new ChromePdfRenderer();
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("output.pdf");
// pdf disposed automatically
