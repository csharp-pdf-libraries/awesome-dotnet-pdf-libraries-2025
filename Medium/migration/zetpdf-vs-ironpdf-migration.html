Migrating from ZetPDF to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate from ZetPDF to IronPDF?

The build server and the developer machine diverge. ZetPDF renders correctly on every Windows development box the team has tested, but the GitHub Actions runner on Ubuntu fails with a native library error that nobody can reproduce locally. The CI fix that worked last month stopped working after a runner image update. The Docker build that deployed cleanly to staging won't deploy to production because the production container is a different base image.

Platform-specific library loading errors are not easy to debug in pipelines, and ZetPDF's narrow user community means Stack Overflow is quiet when you search for the exception message. I have seen this pattern repeatedly, and here are the migration triggers that come up most often.

First, CI/CD platform divergence. Native component behavior differs between Windows dev machines and Linux-based CI runners. Second, container image sensitivity. Native library loading depends on exact system library versions, and base image updates break deployments.

Maintenance uncertainty is another factor. Smaller library ecosystems carry higher update risk, and limited community support means fewer Stack Overflow questions and slower resolution when edge cases arise.

Then there is secondary library accumulation. Features not covered by ZetPDF, such as merge, watermark, and security, require additional packages. Native dependency management means native libraries require explicit management in Docker images and CI runners. And debugging friction from CI-only failures without local repro slows down incident response.

What Are the Frequently Asked Questions?

Answer: I get these questions on nearly every ZetPDF migration I advise on. Here they are with direct answers.

How Do I Handle the CI/CD Native Library Failure?

Answer: The DllNotFoundException: Unable to load shared library 'libzetpdf.so' pattern appears in CI on Ubuntu but not on Windows dev machines. Native library loading on Linux depends on LD_LIBRARY_PATH and system library presence. The .NET runtime's NativeLibrary.Load() behavior differs between Windows and Linux.

After migrating to IronPDF, the ZetPDF native library concern is eliminated. IronPDF's Chromium dependency has its own system library requirements. Install them explicitly in your Dockerfile:

# IronPDF system dependencies for Linux (verify complete list in IronPDF docs):
FROM mcr.microsoft.com/dotnet/aspnet:8.0

RUN apt-get update && apt-get install -y \
    libnss3 \
    libatk-bridge2.0-0 \
    libdrm2 \
    libgbm1 \
    libxkbcommon0 \
    fonts-liberation \
    && rm -rf /var/lib/apt/lists/*

COPY --from=build /app/out .
ENTRYPOINT ["dotnet", "YourApp.dll"]

Why Does It Work After One Render but Fail on Subsequent Requests?

Answer: This is often a disposal issue. An object that should be reused per-request is being disposed after the first use, or a shared state object has single-use semantics. With IronPDF, the safe pattern is one renderer per operation:

using IronPdf;

// The safe pattern — one renderer per operation:
public async Task<byte[]> GeneratePdfAsync(string html)
{
    var renderer = new ChromePdfRenderer(); // new instance per call
    using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    return pdf.BinaryData; // copy bytes before pdf is disposed
}

Why Are PDFs Rendering with Black or Empty Pages in Production?

Answer: Font packages are the first suspect. IronPDF renders via Chromium and uses system fonts for font-family declarations. If a font requested in CSS is not available, Chromium falls back to a system default which may render differently or missing. Explicit web font loading is more reliable:

// Embed fonts via CSS to avoid system font dependency
var html = @"
    <html>
    <head>
    <link href='https://fonts.googleapis.com/css2?family=Inter:wght@400;700' rel='stylesheet'>
    <style>body { font-family: 'Inter', Arial, sans-serif; }</style>
    </head>
    <body><h1>Report</h1></body>
    </html>";

How Do I Fix the License Key Not Found Error in CI?

Answer: The IronPdf.Exceptions.IronPdfLicenseException appears in CI but not in local development. Add the license key to your CI secrets and reference it in your pipeline configuration:

# GitHub Actions
- name: Generate PDFs
  env:
    IRONPDF_LICENSE_KEY: ${{ secrets.IRONPDF_LICENSE_KEY }}
  run: dotnet test

# Azure DevOps
- task: DotNetCoreCLI@2
  env:
    IRONPDF_LICENSE_KEY: $(IRONPDF_LICENSE_KEY)
  inputs:
    command: test

// In startup code — fail early with a clear message
IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY")
    ?? throw new InvalidOperationException(
        "IRONPDF_LICENSE_KEY not set. Add it to CI secrets and local .env.");

How Do I Handle Parallel Rendering for Server-Side Workloads?

Answer: IronPDF supports concurrent rendering with separate renderer instances. Each task gets its own renderer for safe parallel work:

using IronPdf;
using System.Linq;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// https://ironpdf.com/examples/parallel/
var pdfs = await Task.WhenAll(htmlJobs.Select(async html =>
{
    var renderer = new ChromePdfRenderer();
    return await renderer.RenderHtmlAsPdfAsync(html);
}));

foreach (var pdf in pdfs) pdf.Dispose();
// See: https://ironpdf.com/how-to/async/

What Should I Watch Out for with Stream and Bytes Output?

Answer: The output pattern likely changes between ZetPDF and IronPDF. If ZetPDF returned byte[] directly, the IronPDF equivalent wraps results in a PdfDocument object that you extract bytes from:

// IronPDF equivalent:
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
var bytes = pdf.BinaryData; // equivalent byte[] copy
// OR: pdf.SaveAs(path);
// pdf disposed by 'using'

IronPDF uses 0-based page indexing. Verify ZetPDF's indexing in their documentation and audit any page manipulation code for off-by-one issues before testing.

What Is the Recommended Migration Path for Your Team?

Answer: The CI/CD divergence pattern — works locally, fails in pipeline — is the hardest kind of library bug to debug, and it is common with native-component dependencies on platforms the library was not primarily designed for. Once IronPDF replaces ZetPDF, that specific class of failure is resolved since IronPDF's Chromium dependencies are well-documented and consistently supported.

The main investment in this migration is validating ZetPDF's API surface before removal. Since documentation is limited, capture the method signatures from the codebase itself before starting:

# Capture all ZetPDF usage for the migration map
rg "ZetPdf\|ZetPDF" --type cs -n -A 2 > zetpdf-usage-map.txt
cat zetpdf-usage-map.txt
# Each line becomes a migration point

Do not remove ZetPDF code until each usage point is mapped to its IronPDF equivalent and tested. Work through the migration in order: pre-migration assessment, code migration, testing, and post-migration cleanup. The disposal pattern matters — wrap all PdfDocument usage in using blocks:

using IronPdf;

var renderer = new ChromePdfRenderer();
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("output.pdf");
// pdf disposed automatically
