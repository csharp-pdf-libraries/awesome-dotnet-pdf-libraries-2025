Migrating from PDFPrinting.NET to IronPDF: Solving Common Migration Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

The application needs to evolve beyond printing. That is the recurring story I hear from teams using PDFPrinting.NET. The library is excellent at what it does — silently shipping an existing PDF to a Windows printer with fine-grained control over copies, duplex, and tray selection. But the moment a feature request lands that involves creating a PDF from an invoice template, capturing a web report, or stamping a draft watermark on the document before it prints, the team discovers there is no API surface for any of that. PDFPrinting.NET prints PDFs; it does not author them.

This article walks you through what to expect when you swap PDFPrinting.NET for IronPDF, which adds a Chromium-based HTML-to-PDF engine, full document manipulation, and cross-platform printing on top of the silent-print workflow you already rely on. You will have working migration code for the most common operations by the end.

Why Should You Migrate Away from PDFPrinting.NET?

Teams evaluating alternatives to PDFPrinting.NET commonly encounter these friction points. Not all apply to every version or use case, but this is a general list I have seen across migrations.

Print-only scope. PDFPrinting.NET creates no new PDF content. It prints, views, edits at the page level, and rasterizes existing PDFs. There is no HtmlToPdfConverter, no WebPageToPdfConverter, and no HTML or URL ingestion path of any kind. When the next ticket asks for an invoice generated from a template, you reach for a second library.

Windows-only print pipeline. The print path is tied to Windows print spooler infrastructure. Docker and Linux deployments are not part of the supported model.

Limited document manipulation. PdfPrintDocument covers basic merge, split, and extract on the input side, but watermarking, content authoring, form filling, digital signatures, and structured text extraction are not part of the API.

Licensing cost stacking. Pricing starts at $299 for a Site license and rises to $699 for a Redistributable license and $2,249 for a Full license. If your team has already added a second commercial PDF library to fill the generation gap, you are paying twice.

Two namespaces in the wild. Newer releases ship under PdfPrintingNet; older code may still reference the legacy TerminalWorks.PDFPrinting namespace with the PDFPrinter class. Both will be in your codebase if the project is more than a few years old.

What Does the Basic Silent-Printing Migration Look Like?

Install IronPDF and Remove PDFPrinting.NET

The NuGet package ID is PdfPrintingNet (the dot in the marketing name is not in the package ID):

# Remove PDFPrinting.NET
dotnet remove package PdfPrintingNet

# Install IronPDF
dotnet add package IronPdf

dotnet restore

Set Your License Key

using IronPdf;

// Set before any IronPDF usage — https://ironpdf.com/how-to/license-keys/
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

Swap the Namespaces

Replace the old namespace imports. Note that older code may use the legacy namespace:

// Before (newer PdfPrintingNet API)
using PdfPrintingNet;

// Before (legacy TerminalWorks API)
// using TerminalWorks.PDFPrinting;

// After
using IronPdf;
using IronPdf.Printing;

Print an Existing PDF

This is the bread-and-butter operation. PDFPrinting.NET exposes the PdfPrint class (or the legacy PDFPrinter) and takes a file path directly. IronPDF separates the load step from the print step using PdfDocument.FromFile(), which gives you a document handle you can also manipulate, save, or extract from.

// Before (PDFPrinting.NET)
using PdfPrintingNet;

var pdfPrint = new PdfPrint("license-owner", "license-key");
var status = pdfPrint.Print("document.pdf");

Console.WriteLine($"PDF printed: {status}");

// After (IronPDF)
using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("document.pdf");
pdf.Print();

Console.WriteLine("PDF printed successfully.");
// https://ironpdf.com/how-to/pdf-printing-print/

Print to a Specific Printer

PDFPrinting.NET sets PrinterName as a property on the PdfPrint instance. IronPDF accepts the printer name as an argument to Print():

// Before (PDFPrinting.NET)
var pdfPrint = new PdfPrint("license-owner", "license-key");
pdfPrint.PrinterName = "HP LaserJet Pro";
pdfPrint.Print("document.pdf");

// After (IronPDF)
var pdf = PdfDocument.FromFile("document.pdf");
pdf.Print("HP LaserJet Pro");

How Do You Migrate Custom Print Settings?

Copies, duplex, collation, grayscale, and tray selection live on the PdfPrint instance in the old library and on a PrintSettings object in IronPDF. The mapping is one-to-one for the common properties.

// Before (PDFPrinting.NET)
var pdfPrint = new PdfPrint("license-owner", "license-key");
pdfPrint.PrinterName = "Office Printer";
pdfPrint.Copies = 3;
pdfPrint.PageScaling = PageScaling.FitToPrintableArea;
pdfPrint.Duplex = true;
pdfPrint.Collate = true;
pdfPrint.PrintInColor = false;

pdfPrint.Print("report.pdf");

// After (IronPDF)
using IronPdf;
using IronPdf.Printing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("report.pdf");

var printSettings = new PrintSettings
{
    PrinterName = "Office Printer",
    NumberOfCopies = 3,
    DuplexMode = System.Drawing.Printing.Duplex.Vertical,
    Collate = true,
    GrayscaleOutput = true,
};

pdf.Print(printSettings);

How Do You Print a Page Range?

For finer-grained control — page ranges, paper sources, custom paper sizes — IronPDF exposes the underlying System.Drawing.Printing.PrintDocument through pdf.GetPrintDocument(). This gives you exactly the same surface PDFPrinting.NET’s PdfPrintDocument exposed (it inherits from the same .NET base class), so existing knowledge transfers directly.

// Before (PDFPrinting.NET)
var printDoc = new PdfPrintDocument("license-owner", "license-key", "document.pdf");
printDoc.PrinterSettings.PrinterName = "Network Printer";
printDoc.PrinterSettings.FromPage = 2;
printDoc.PrinterSettings.ToPage = 5;
printDoc.Print();

// After (IronPDF)
var pdf = PdfDocument.FromFile("document.pdf");
var printDoc = pdf.GetPrintDocument();

printDoc.PrinterSettings.PrinterName = "Network Printer";
printDoc.PrinterSettings.FromPage = 2;
printDoc.PrinterSettings.ToPage = 5;
printDoc.PrinterSettings.PrintRange = System.Drawing.Printing.PrintRange.SomePages;

printDoc.Print();

How Do You Generate a PDF from HTML and Then Print It?

This is the capability gap that most migrations are really about. PDFPrinting.NET cannot turn an HTML invoice template into a PDF — teams typically pair it with a separate generation library, then hand the resulting file to PdfPrint. IronPDF does both jobs in one package: render HTML through an embedded Chromium engine, then print the document directly without touching disk.

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var html = @"
    <html>
    <head>
    <style>
        body { font-family: Arial, sans-serif; }
        .invoice-table { width: 100%; border-collapse: collapse; }
        .invoice-table th, .invoice-table td { border: 1px solid #ddd; padding: 8px; }
    </style>
    </head>
    <body>
        <h1>Invoice #2024-001</h1>
        <table class='invoice-table'>
            <tr><th>Item</th><th>Qty</th><th>Price</th></tr>
            <tr><td>Widget A</td><td>10</td><td>$50</td></tr>
        </table>
    </body>
    </html>";

var pdf = await renderer.RenderHtmlAsPdfAsync(html);

// Print directly — or save and print later
pdf.Print("Invoice Printer");
pdf.SaveAs("invoice.pdf");
// https://ironpdf.com/how-to/html-string-to-pdf/

How Do You Merge PDFs Before Printing?

Combining a cover page, body, and appendix into a single print job is a common request. PDFPrinting.NET has no merge API; teams typically add PdfSharp or iTextSharp to fill the gap. IronPDF’s PdfDocument.Merge() covers it natively, so the secondary dependency can be removed.

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var cover    = PdfDocument.FromFile("cover.pdf");
var content  = PdfDocument.FromFile("content.pdf");
var appendix = PdfDocument.FromFile("appendix.pdf");

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var combined = PdfDocument.Merge(cover, content, appendix);

combined.Print("Office Printer");
combined.SaveAs("complete-report.pdf");

How Do You Add a Watermark Before Printing?

Stamping “DRAFT” or “CONFIDENTIAL” on a PDF before it hits the printer is another operation that PDFPrinting.NET does not offer. IronPDF provides a dedicated TextStamper API that applies a consistent watermark across all pages.

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("draft-report.pdf");

var watermark = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronSoftware.Drawing.Color.Red,
    Opacity = 30, // integer 0–100
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.Print();

How Do You Replace Secondary-Library Password Protection?

If you currently produce password-protected output by routing the PDF through a separate library, IronPDF’s built-in security settings let you consolidate.

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("sensitive.pdf");

// Security settings: https://ironpdf.com/how-to/pdf-permissions-passwords/
pdf.SecuritySettings.UserPassword = "open123";
pdf.SecuritySettings.OwnerPassword = "admin456";
pdf.SecuritySettings.AllowUserPrinting = IronPdf.Security.PdfPrintSecurity.FullPrintRights;

pdf.SaveAs("secured.pdf");

What Are the Most Common Troubleshooting Issues After Migration?

These are the patterns teams most frequently hit when switching from a Windows-only print library to a cross-platform Chromium-based stack.

Printer name exact match. Both libraries require an exact match on the printer name, including spaces and capitalization. On Linux under CUPS, spaces are typically replaced with underscores (HP_LaserJet rather than HP LaserJet). Enumerate available printers before assuming a name:

using System.Drawing.Printing;

foreach (string printer in PrinterSettings.InstalledPrinters)
{
    Console.WriteLine($"'{printer}'");
}

Resource disposal. PdfPrint in PDFPrinting.NET is largely stateless; PdfDocument in IronPDF holds resources and is disposable. Wrap it accordingly:

using (var pdf = PdfDocument.FromFile("document.pdf"))
{
    pdf.Print();
}

Async APIs. PDFPrinting.NET is synchronous-only. IronPDF exposes async methods like RenderHtmlAsPdfAsync, FromFileAsync, and SaveAsAsync. Avoid blocking with .Result or .Wait() in async contexts:

// Avoid:
// var pdf = renderer.RenderHtmlAsPdfAsync(html).Result; // blocks thread

// Prefer:
var pdf = await renderer.RenderHtmlAsPdfAsync(html);

Linux deployment dependencies. If you are taking the migration as an opportunity to move to Linux or Docker, IronPDF’s Chromium renderer needs the standard Chromium runtime libraries:

apt-get update && apt-get install -y \
    libnss3 libatk1.0-0 libatk-bridge2.0-0 libdrm2 \
    libxkbcommon0 libxcomposite1 libxdamage1 libxfixes3 \
    libxrandr2 libgbm1 libasound2 cups

How Does PDFPrinting.NET Compare to IronPDF Feature by Feature?

Here is how the two libraries line up across the dimensions that matter for migration.

  • Focus. PDFPrinting.NET is a silent-printing specialist for existing PDFs (with viewer, page-level edit, and rasterization as separately licensed components). IronPDF covers HTML-to-PDF, URL-to-PDF, full PDF manipulation (merge, split, watermark, security, forms, signatures), and printing.
  • PDF generation. PDFPrinting.NET does not generate PDFs from HTML or URLs — the API has no HtmlToPdfConverter and no WebPageToPdfConverter class. IronPDF generates from HTML strings, HTML files, URLs, and images.
  • Pricing. PDFPrinting.NET is commercial: Site $299+, Redistributable $699+, Full $2,249. IronPDF is also commercial; verify current pricing on the vendor site.
  • API style. PDFPrinting.NET centers on PdfPrint and PdfPrintDocument (with the legacy PDFPrinter class still present in older code). IronPDF is fluent and object-oriented around ChromePdfRenderer and PdfDocument.
  • Platform support. PDFPrinting.NET is Windows-only. IronPDF runs on Windows, Linux, and macOS.
  • Namespaces. PdfPrintingNet on the new side, with TerminalWorks.PDFPrinting still in legacy code. IronPDF uses IronPdf, IronPdf.Printing, IronPdf.Rendering, and IronPdf.Editing.
  • Async support. PDFPrinting.NET is synchronous-only. IronPDF ships built-in async methods.

And here is the migration effort breakdown by feature:

  • Silent print to default printer — Low. PdfPrint.Print(path) maps to PdfDocument.FromFile(path).Print().
  • Print to specific printer — Low. pdfPrint.PrinterName = "..." moves to pdf.Print(printerName).
  • Copies, duplex, collate, grayscale — Low. Properties on PdfPrint map to PrintSettings.NumberOfCopies, DuplexMode, Collate, and GrayscaleOutput.
  • Page range printing — Low. PdfPrintDocument.PrinterSettings.FromPage/ToPage maps to the same property surface on pdf.GetPrintDocument().PrinterSettings.
  • HTML string to PDF — Medium. Not available in PDFPrinting.NET; ChromePdfRenderer.RenderHtmlAsPdfAsync() in IronPDF.
  • URL to PDF — Medium. Not available in PDFPrinting.NET; ChromePdfRenderer.RenderUrlAsPdfAsync() in IronPDF.
  • Headers and footers — Medium. Not authorable in PDFPrinting.NET; IronPDF uses HTML fragments via RenderingOptions.HtmlHeader and HtmlFooter.
  • Merge PDFs — Medium. Basic merge available via PdfPrintDocument; IronPDF provides PdfDocument.Merge() with consistent document handles.
  • Watermark — Medium. Not available in PDFPrinting.NET; IronPDF provides TextStamper / ImageStamper and pdf.ApplyWatermark().
  • Password protection — Medium. Not available in PDFPrinting.NET; IronPDF provides pdf.SecuritySettings.
  • Linux/Docker deployment — Medium. Not supported on PDFPrinting.NET; IronPDF runs cross-platform with documented Linux dependencies and CUPS for printing.
  • Async rendering — Medium. Sync-only on PDFPrinting.NET; async throughout in IronPDF.

What Are the Frequently Asked Questions About This Migration?

Can I keep using my existing print settings logic?

Mostly yes. The properties on PdfPrint map closely to PrintSettings in IronPDF (Copies → NumberOfCopies, Duplex → DuplexMode, PrintInColor → inverted as GrayscaleOutput, and so on). For anything not exposed on PrintSettings, drop down to pdf.GetPrintDocument() and use the standard System.Drawing.Printing.PrinterSettings surface, which is the same base class PdfPrintDocument inherited from.

Can I remove my secondary PDF generation library after migrating?

In most cases, yes. Teams that paired PDFPrinting.NET with PdfSharp, iTextSharp, or another library specifically to generate PDFs from HTML or to merge documents can usually consolidate to IronPDF alone. Audit your NuGet dependencies after migration.

Does IronPDF work on Linux and Docker?

Yes. Moving from PDFPrinting.NET’s Windows-only print pipeline to a containerized deployment is one of the most common migration drivers. IronPDF supports Linux deployments. Avoid relying on system-installed fonts in containerized environments — embed fonts via @font-face with base64 or CDN URLs instead. For printing on Linux, install CUPS and use the CUPS printer name (which typically uses underscores rather than spaces).

What about my legacy TerminalWorks.PDFPrinting code?

Older PDFPrinting.NET releases shipped under the TerminalWorks.PDFPrinting namespace with a PDFPrinter class, and that code may still be in your project. Treat it the same way as the newer PdfPrintingNet code during migration: replace the using directive, replace the PDFPrinter instantiation with PdfDocument.FromFile(), and map the print settings to PrintSettings.

How do I handle synchronous code that calls the old library?

IronPDF provides async methods like RenderHtmlAsPdfAsync() and FromFileAsync(). If your existing code was synchronous, replace calls with their async equivalents. Do not block with .Result or .Wait() in async contexts.

When Is the Right Time to Migrate?

The clearest trigger is a feature request that PDFPrinting.NET cannot serve at all — generating an invoice from HTML, capturing a web report as PDF, stamping a watermark, applying password protection. Any one of those means a second library, a second commercial license, and a second integration to maintain. Migrating to IronPDF collapses both into one package.

The consolidation argument is also strong on its own. If your project currently depends on PDFPrinting.NET plus one or more secondary libraries for generation, merge, security, or watermarking, migrating lets you collapse those into a single dependency — fewer version conflicts, fewer license renewals, simpler build pipeline.

If your deployment target is moving to Linux or Docker, PDFPrinting.NET’s Windows-only print pipeline is a hard blocker. IronPDF supports Linux and macOS, handles web fonts via Chromium, and works with CUPS for printing — the migration becomes a deployment prerequisite rather than a nice-to-have.

The one area worth testing carefully: if your PDFPrinting.NET setup uses uncommon printer-driver-specific features (custom paper trays, vendor-specific PJL or PostScript hooks), exercise those on a real printer with IronPDF before cutting over — the print path is standard System.Drawing.Printing, but driver behavior varies.
