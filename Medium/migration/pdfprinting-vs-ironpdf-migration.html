Migrating from PDFPrinting.NET to IronPDF: Solving Common Migration Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

The output never quite matches the design. That is the recurring complaint I hear from teams using HTML-to-PDF tools built on aging rendering engines or non-Chromium stacks. A table that renders perfectly in Chrome looks slightly different in the PDF — column widths shift, fonts fall back, a flex layout collapses. You fix one page and something shifts on the next.

This article walks you through what to expect when you swap PDFPrinting.NET for IronPDF, which uses an embedded Chromium renderer. You will have working migration code for the four most common operations by the end.

Why Should You Migrate Away from PDFPrinting.NET?

Teams evaluating alternatives to PDFPrinting.NET commonly encounter these friction points. Not all apply to every version or use case, but this is a general list I have seen across migrations.

HTML/CSS fidelity gaps — output diverges from browser rendering, particularly with modern CSS (flex, grid, custom properties). The rendering engine age means older non-Chromium renderers do not track browser standards updates.

Windows-only or GDI-dependent pipeline — Docker and Linux deployments become problematic. Font handling tied to system fonts in Windows-only libraries breaks in containerized environments.

Limited .NET version support — newer .NET 6/7/8/9 targets may not be fully supported. Async API gaps mean synchronous-only rendering APIs do not integrate well with modern async .NET code.

Sparse documentation — limited examples for edge cases like headers/footers, custom page sizes, or JavaScript-rendered content. Maintenance uncertainty from infrequent NuGet updates suggests reduced active development.

Missing features — no built-in support for annotations, form fields, or digital signatures depending on version. Thread safety guidance is unclear or restrictive for concurrent usage.

// Before (verify actual PDFPrinting.NET namespace)
using PDFPrinting;

// After
using IronPdf;

Convert HTML String to PDF

This is the most common operation I see teams migrating. The IronPDF version uses ChromePdfRenderer, which renders CSS as Chrome does — so table layouts, flex, and grid all behave as you would expect in a browser.

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var html = @"
    <html>
    <head>
    <style>
        body { font-family: Arial, sans-serif; }
        .invoice-table { width: 100%; border-collapse: collapse; }
        .invoice-table th, .invoice-table td { border: 1px solid #ddd; padding: 8px; }
    </style>
    </head>
    <body>
        <h1>Invoice #2024-001</h1>
        <table class='invoice-table'>
            <tr><th>Item</th><th>Qty</th><th>Price</th></tr>
            <tr><td>Widget A</td><td>10</td><td>$50</td></tr>
        </table>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

// Chromium renderer — CSS table layouts render as in Chrome
var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");

Console.WriteLine($"Saved invoice.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

How Do You Handle PDF Merging Without a Secondary Library?

A common finding during PDFPrinting.NET migrations is that teams added PdfSharp, iTextSharp, or a similar library specifically to fill gaps like merging. With IronPDF, merging is built in — so you can remove that secondary dependency entirely.

Merge Two PDFs with IronPDF

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

var section1Html = "<html><body><h1>Section 1: Executive Summary</h1></body></html>";
var section2Html = "<html><body><h1>Section 2: Financial Detail</h1></body></html>";

var pdf1 = await renderer.RenderHtmlAsPdfAsync(section1Html);
var pdf2 = await renderer.RenderHtmlAsPdfAsync(section2Html);

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(pdf1, pdf2);
merged.SaveAs("report.pdf");

Console.WriteLine($"Merged: {merged.PageCount} total pages");

How Do You Add Watermarks Programmatically Instead of CSS Hacks?

With PDFPrinting.NET, teams typically injected watermarks via CSS pseudo-elements. The rendering fidelity of that approach varied depending on the engine. IronPDF provides a dedicated TextStamper API that applies consistent watermarks across all pages.

Apply a Text Watermark with IronPDF

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Quarterly Report</h1><p>Content here...</p></body></html>"
);

// Programmatic watermark — consistent across all pages
var watermark = new TextStamper
{
    Text = "CONFIDENTIAL",
    FontColor = IronPdf.Imaging.Color.Red,
    Opacity = 0.15,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("confidential-report.pdf");

How Do You Replace Secondary-Library Password Protection?

If PDFPrinting.NET did not support password protection natively, your team likely added iTextSharp or another library just for that. IronPDF has built-in security settings, so you can consolidate into a single package.

Set Password Protection with IronPDF

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Sensitive Document</h1></body></html>"
);

// Security settings: https://ironpdf.com/how-to/pdf-permissions-passwords/
pdf.SecuritySettings.UserPassword = "open123";
pdf.SecuritySettings.OwnerPassword = "admin456";

pdf.SaveAs("secured.pdf");

What Are the Most Common Troubleshooting Issues After Migration?

Answer:

These are the patterns teams most frequently hit when switching from an older HTML-to-PDF library to a Chromium-based renderer.

CSS layout differences. Tables or flex/grid layouts render differently after migration. PDFPrinting.NET may have used a non-Chromium rendering pipeline. IronPDF uses Chromium, which renders CSS as Chrome does. Most "broken" CSS is actually more correct in IronPDF — it was previously being rendered in a degraded mode. For print-specific styles, use @media print CSS:

var html = @"
    <html>
    <head>
    <style>
        @media print {
            .no-print { display: none; }
            table { page-break-inside: avoid; }
        }
    </style>
    </head>
    <body>...</body>
    </html>";

var renderer = new ChromePdfRenderer();
// See: https://ironpdf.com/how-to/html-to-pdf-responsive-css/

JavaScript-rendered content. Dynamic content from JavaScript frameworks (React, Vue) does not appear in the PDF. Use the WaitFor setting to let JavaScript execute before rendering:

var renderer = new ChromePdfRenderer();

// Wait for JavaScript execution before rendering
renderer.RenderingOptions.WaitFor.JavaScript(500); // ms — adjust as needed

var pdf = await renderer.RenderUrlAsPdfAsync("https://your-spa-url.com/report");

Font fallback issues. Custom fonts appear as system fallbacks. Embed fonts in your HTML using @font-face with base64 or CDN URLs. IronPDF's Chromium renderer handles web fonts as Chrome does. Avoid relying on system-installed fonts in containerized environments:

var html = @"
    <html>
    <head>
    <link href='https://fonts.googleapis.com/css2?family=Roboto' rel='stylesheet'>
    <style>body { font-family: 'Roboto', sans-serif; }</style>
    </head>
    <body>...</body>
    </html>";

What Are the Frequently Asked Questions About This Migration?

Answer:

Do I need to rewrite all my HTML templates?

No. IronPDF uses an embedded Chromium renderer, so your existing HTML and CSS will render as they do in Chrome. In fact, CSS that was intentionally simplified to work around the old renderer's limitations can often be restored to its full form. Test your HTML in Chrome DevTools first — if it looks correct there, it will look correct in IronPDF's HTML-to-PDF output.

Can I remove secondary PDF libraries after migrating?

In most cases, yes. A common finding during this migration is that projects added PdfSharp, iTextSharp, or a similar library specifically to fill gaps in PDFPrinting.NET — merge, security, watermark. After migrating to IronPDF, those secondary packages can often be removed. Audit your NuGet dependencies after migration.

Does IronPDF work on Linux and Docker?

Yes. One of the main migration drivers is moving from a Windows-only or GDI-dependent pipeline to something that runs in containers. IronPDF supports Linux deployments. Avoid relying on system-installed fonts in containerized environments — embed fonts via @font-face with base64 or CDN URLs instead.

How do I handle synchronous code that calls the old library?

IronPDF provides async methods like RenderHtmlAsPdfAsync(). If your existing code was synchronous, replace calls with their async equivalents. Do not block with .Result or .Wait() in async contexts:

// Avoid this pattern:
// var pdf = renderer.RenderHtmlAsPdfAsync(html).Result; // blocks thread

// Use this:
var pdf = await renderer.RenderHtmlAsPdfAsync(html);

What about page indexing differences?

IronPDF uses 0-based page indexing. If you have any page manipulation code, audit all page index references. This is a small but important detail that can cause off-by-one errors if overlooked. Verify the specifics in the IronPDF documentation.

When Is the Right Time to Migrate?

Answer:

The CSS fidelity improvement is usually the clearest win in this migration. Layouts that required workarounds for a weaker rendering engine tend to just work with Chromium. If your current output has visual errors on tables, flex, or grid layouts, that alone justifies the switch.

The consolidation argument is also strong. If your project currently depends on PDFPrinting.NET plus one or more secondary libraries for merge, security, or watermarking, migrating to IronPDF lets you collapse those into a single package. That means fewer dependencies, fewer version conflicts, and a simpler build pipeline.

If your deployment target is moving to Linux or Docker, and PDFPrinting.NET's Windows-only or GDI-dependent pipeline is blocking that move, the migration becomes a deployment prerequisite rather than a nice-to-have. IronPDF supports Linux and handles web fonts via its Chromium renderer the same way Chrome does — no system font dependencies to manage.

The one area worth testing carefully: if your PDFPrinting.NET setup had specific behavior for form fields or annotations, verify IronPDF's equivalents explicitly — the API surface differs.
