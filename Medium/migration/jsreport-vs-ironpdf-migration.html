Migrating from jsreport to IronPDF: No Fuss, No Fluff

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

The deployment worked fine on-prem. Then the team containerized the application for Kubernetes, and the PDF generation service started failing intermittently. The error logs pointed to jsreport's child process spawning — the embedded Node.js binary that jsreport.Local launches under the .NET host, with the reporting service trying to start headless Chromium inside that Node.js process. The layered process management, the user permission model in the container, the platform-specific binary packages — it all compounds.

Getting jsreport's local binaries running reliably in a container takes work that has nothing to do with generating PDFs. That's a common trigger for evaluating alternatives. This article is for .NET teams considering IronPDF as the replacement.

Why Should I Migrate from jsreport to IronPDF?

jsreport is a Node.js reporting platform that .NET teams typically consume through jsreport.Local, which downloads a Node.js binary (jsreport.Binary, jsreport.Binary.Linux, jsreport.Binary.OSX) and spawns it as a child process. IronPDF is a pure .NET library. That difference has practical implications before you write a single line of migration code. jsreport runs an external Node.js process — whether in utility mode (one-shot per render) or web server mode (persistent) — and communicates with your .NET code over local IPC or HTTP. IronPDF runs in-process, renders HTML strings or files via embedded Chromium, and ships no external runtime.

Here are eight specific reasons teams trigger this migration:

1. Container deployment complexity — jsreport's Chromium-in-Node.js-in-Docker requires managing layered process permissions, seccomp profiles, and memory limits that don't apply to an in-process library.

2. Node.js dependency in a .NET shop — maintaining a Node.js runtime, managing the platform-specific jsreport.Binary* packages, and keeping them updated adds operational overhead for teams without Node.js expertise.

3. Process spawn and IPC overhead — in utility mode, every RenderAsync call boots the Node.js binary; in web server mode, you pay a local HTTP round trip per render. In-process rendering eliminates both.

4. Service orchestration — jsreport in web server mode needs health checks, restart policies, and separate monitoring. An in-process library reduces the service count.

5. Azure App Service / serverless constraints — some Azure tiers restrict child process spawning, which breaks jsreport.Local. An in-process renderer avoids this constraint. See the IronPDF Azure documentation.

6. Handlebars / JsRender templating — jsreport's template engines run inside Node.js. After migration, templating moves to .NET (Razor, string interpolation, or Handlebars.NET) and lives in your repo under version control.

7. Output format scope — if PDF is the only output you actually use, maintaining a full multi-recipe reporting platform is overhead.

8. jsreport licensing tiers — if you're hitting limits on the community edition for your use case, evaluating alternatives is reasonable before upgrading tiers.

How Much Effort Does Each Feature Take to Migrate?

The migration scope depends heavily on which jsreport features you actually use. I've mapped every common feature to its migration effort below. If you're only generating PDFs from HTML, the migration is straightforward. If you depend on jsreport's multi-format output or template management UI, those pieces need separate handling.

  • HTML/Handlebars to PDF — Medium. The render call itself is a one-line swap; the work is moving template data binding from Handlebars (running in Node.js) to .NET string interpolation, Razor, or Handlebars.NET.
  • URL to PDF — Low. jsreport handles this through the ChromePdf recipe; IronPDF has a direct RenderUrlAsPdf method.
  • Headers and footers — Medium. Mechanically straightforward, but the placeholder syntax changes. jsreport's chrome-pdf recipe uses Chromium HTML span classes (<span class="pageNumber">, <span class="totalPages">, <span class="date">); the optional pdf-utils extension adds curly-brace forms ({#pageNum}, {#numPages}). IronPDF replaces both with simple merge fields: {page}, {total-pages}, {date}.
  • Margins — Low. Convert CSS units to millimeters ("2cm" → 20, "1in" → 25).
  • Platform-specific binaries — Low. Delete jsreport.Binary, jsreport.Binary.Linux, and jsreport.Binary.OSX; IronPDF auto-detects platform.
  • Server lifecycle (StartAsync / KillAsync) — Low. All such calls disappear; IronPDF is in-process.
  • Excel and CSV output — N/A. IronPDF does not cover non-PDF recipes; if you rely on jsreport for these, keep jsreport for them or add a dedicated library.
  • jsreport Studio (template management UI) — Not in IronPDF. If non-developers design templates in the browser, you need to build that workflow separately.
  • Scheduled reports — Not in IronPDF. Move to Hangfire, Quartz.NET, or a hosted scheduler.
  • Container deployment — Simpler. No Node.js layer, no platform-specific binaries.

And here is a quick decision matrix:

  • PDF only, container deployment issues. IronPDF is a direct fit and evaluates well.
  • Multi-format output (Excel, CSV) needed. IronPDF covers PDF only — retain jsreport for other formats, or add format-specific libraries.
  • Template management UI is business-critical. jsreport Studio has no equivalent in IronPDF; either keep jsreport or build a Razor-based editor in front of IronPDF.
  • Node.js runtime is already in your stack. jsreport overhead is lower for you, so migration ROI is smaller; the case is mostly about deployment simplification.

How Do jsreport and IronPDF Compare Feature by Feature?

Before diving into code, it helps to see the two libraries side by side. I've mapped the core aspects, API patterns, and class equivalents so you know exactly what changes during migration.

jsreport is a multi-format reporting platform; IronPDF is focused on PDF generation and manipulation. jsreport has a community (free) tier plus commercial tiers, IronPDF is commercial — pricing is on ironsoftware.com. Both render HTML through Chromium, so the visual output of equivalent templates is generally close.

The architectural difference is bigger than the API difference. jsreport.Local spawns a Node.js child process (utility mode) or runs an embedded Node.js web server (web server mode); the .NET side is essentially a client. IronPDF runs entirely in-process: a single ChromePdfRenderer instance renders directly. Page indexing in IronPDF's PdfDocument is 0-based — relevant once you start manipulating output. Renderer reuse is supported; for ASP.NET Core, register ChromePdfRenderer as a singleton.

Here is the namespace mapping:

  • jsreport.Local → IronPdf — entry point
  • jsreport.Types → IronPdf / IronPdf.Rendering — configuration types
  • jsreport.Binary, jsreport.Binary.Linux, jsreport.Binary.OSX → not needed — IronPDF auto-detects platform
  • jsreport.Client → not needed — no HTTP client (in-process)

And the core class mapping:

  • LocalReporting / ReportingService → ChromePdfRenderer — entry point for PDF generation
  • RenderRequest → method parameters on ChromePdfRenderer — no wrapper object
  • Template → method parameters — HTML string or file passed directly
  • Chrome (recipe options) → RenderingOptions — margins, paper size, headers, wait conditions
  • Report.Content (stream) → PdfDocument — output object with SaveAs, BinaryData, Stream
  • Recipe / Engine enums → not needed — IronPDF is always Chrome PDF; templating is C#

How Do I Set Up IronPDF and Run a Basic Migration?

The basic migration has three pieces: license configuration, namespace imports, and a one-line PDF generation swap. Before any of that, find the jsreport references in your codebase so you know the full scope.

Finding jsreport References

# Find jsreport using statements and client classes
rg -l "jsreport\.|JsReportBinary|LocalReporting|ReportingService" --type cs -i

# Find render call sites
rg "RenderAsync|RenderRequest|new Template" --type cs -n

# Find recipe / engine usage (Handlebars, JsRender, ChromePdf, Xlsx, etc.)
rg "Recipe\.|Engine\." --type cs -n

# Find NuGet package references
rg "jsreport\." -g "*.csproj"

Then remove jsreport and install IronPDF:

Swap the Packages

# Remove jsreport .NET packages
dotnet remove package jsreport.Local
dotnet remove package jsreport.Types
dotnet remove package jsreport.Binary
dotnet remove package jsreport.Binary.Linux  # if installed
dotnet remove package jsreport.Binary.OSX    # if installed
dotnet remove package jsreport.Client        # if you used the HTTP client

# Install IronPDF
dotnet add package IronPdf
dotnet restore

License Configuration

Before (jsreport — no .NET license key; the Node.js binary is configured at construction time):

using jsreport.Binary;

using jsreport.Types;

// No in-process license key. Configuration is on the LocalReporting builder.
var rs = new LocalReporting()

After (IronPDF):

using IronPdf;

// Guide: https://ironpdf.com/how-to/license-keys/

Basic PDF Generation

Before (jsreport with jsreport.Local):

using jsreport.Binary;
using jsreport.Local;
using jsreport.Types;
using System.IO;
using System.Threading.Tasks;

class QuickStartExample
{
    static async Task Main()
    {
        var rs = new LocalReporting()
            .UseBinary(JsReportBinary.GetBinary())
            .AsUtility()
            .Create();

        var report = await rs.RenderAsync(new RenderRequest
        {
            Template = new Template
            {
                Content = "<h1>Hello World</h1>",
                Engine = Engine.None,
                Recipe = Recipe.ChromePdf
            }
        });

        using var fs = File.Create("output.pdf");
        report.Content.CopyTo(fs);
    }
}

After (IronPDF):

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<h1>Hello World</h1>");
pdf.SaveAs("output.pdf");
// Guide: https://ironpdf.com/how-to/html-string-to-pdf/

How Do I Migrate HTML-to-PDF with Template Data?

With jsreport, template data binding happens in Handlebars running inside the Node.js binary. After migration, that logic moves to your .NET code. You can use string interpolation, Razor, or Handlebars.NET if you want to preserve your existing template syntax.

Before (jsreport RenderAsync with Handlebars Data)

using jsreport.Binary;
using jsreport.Local;
using jsreport.Types;
using System;
using System.IO;
using System.Threading.Tasks;

class HtmlToPdfExample
{
    static async Task Main()
    {
        var rs = new LocalReporting()
            .UseBinary(JsReportBinary.GetBinary())
            .AsUtility()
            .Create();

        // Template with Handlebars data binding
        string template = @"
            <html><body>
                <h1>Invoice #{{invoiceNumber}}</h1>
                <p>Customer: {{customerName}}</p>
                <p>Amount: ${{amount}}</p>
            </body></html>";

        var report = await rs.RenderAsync(new RenderRequest
        {
            Template = new Template
            {
                Content = template,
                Engine = Engine.Handlebars,
                Recipe = Recipe.ChromePdf,
                Chrome = new Chrome { MarginTop = "10mm", MarginBottom = "10mm" }
            },
            Data = new
            {
                invoiceNumber = "1234",
                customerName = "ACME Corp",
                amount = "500.00"
            }
        });

        using var fs = File.Create("invoice.pdf");
        report.Content.CopyTo(fs);
        Console.WriteLine("Saved: invoice.pdf");
    }
}

After (IronPDF — Template Logic in .NET)

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// Template data rendering moves to .NET (Razor, string interpolation, etc.)
var invoiceNumber = "1234";
var customerName = "ACME Corp";
var amount = "500.00";

string html = $@"
    <html><body>
        <h1>Invoice #{invoiceNumber}</h1>
        <p>Customer: {customerName}</p>
        <p>Amount: ${amount}</p>
    </body></html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.MarginTop = 10;
renderer.RenderingOptions.MarginBottom = 10;

var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("invoice.pdf");

For more complex templates, you have options:

// Option 1: String interpolation (simple cases)
string html = $"<h1>Invoice #{invoice.Number}</h1><p>Total: {invoice.Total:C}</p>";

// Option 2: Razor in a separate class library
// Use RazorLight or a similar Razor rendering library to compile .cshtml to string
// Then pass string to ChromePdfRenderer

// Option 3: Handlebars.NET
// NuGet: Handlebars.Net — allows reusing Handlebars template syntax in .NET
var template = Handlebars.Compile(handlebarsTemplate);
string html = template(data);
var pdf = renderer.RenderHtmlAsPdf(html);

How Do I Migrate URL-to-PDF and Header/Footer Templates?

These two scenarios cover the bulk of jsreport-to-IronPDF migration work in practice. URL rendering goes from a Chrome recipe option to a dedicated method. Header and footer templates carry the most subtle migration risk because the placeholder syntax differs. jsreport's chrome-pdf recipe expects Chromium-native HTML span classes inside header and footer templates — <span class="pageNumber">, <span class="totalPages">, <span class="date">, <span class="title">, <span class="url">. Teams that opted into jsreport's pdf-utils extension instead saw curly-brace forms like {#pageNum} and {#numPages}. IronPDF replaces both with simple merge fields: {page}, {total-pages}, {date}, {html-title}, {url}. Picking the wrong recipe (chrome-pdf span classes inside a template the engine treated as pdf-utils, or vice versa) can fail silently — the placeholder simply renders as literal text.

URL to PDF — Before (jsreport)

using jsreport.Binary;
using jsreport.Local;
using jsreport.Types;
using System.IO;
using System.Threading.Tasks;

class UrlToPdfExample
{
    static async Task Main()
    {
        var rs = new LocalReporting()
            .UseBinary(JsReportBinary.GetBinary())
            .AsUtility()
            .Create();

        // jsreport URL rendering: empty Content + Chrome.Url, or use a script redirect
        var report = await rs.RenderAsync(new RenderRequest
        {
            Template = new Template
            {
                Recipe = Recipe.ChromePdf,
                Engine = Engine.None,
                Content = "",
                Chrome = new Chrome
                {
                    Url = "https://example.com",
                    WaitForNetworkIddle = true,  // jsreport.Types property name (typo preserved by the SDK)
                    Format = "A4",
                    Landscape = true,
                    MediaType = MediaType.Print
                }
            }
        });

        using var fs = File.Create("webpage.pdf");
        report.Content.CopyTo(fs);
    }
}

URL to PDF — After (IronPDF)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
renderer.RenderingOptions.PaperOrientation = IronPdf.Rendering.PdfPaperOrientation.Landscape;
renderer.RenderingOptions.CssMediaType = IronPdf.Rendering.PdfCssMediaType.Print;
renderer.RenderingOptions.WaitFor.NetworkIdle();

var pdf = renderer.RenderUrlAsPdf("https://example.com");
pdf.SaveAs("webpage.pdf");
// Guide: https://ironpdf.com/how-to/url-to-pdf/

Headers and Footers — Before (jsreport)

using jsreport.Binary;
using jsreport.Local;
using jsreport.Types;
using System.IO;
using System.Threading.Tasks;

class HeaderFooterExample
{
    static async Task Main()
    {
        var rs = new LocalReporting()
            .UseBinary(JsReportBinary.GetBinary())
            .AsUtility()
            .Create();

        var report = await rs.RenderAsync(new RenderRequest
        {
            Template = new Template
            {
                Recipe = Recipe.ChromePdf,
                Engine = Engine.None,
                Content = "<h1>Document with Header and Footer</h1><p>Main content goes here.</p>",
                Chrome = new Chrome
                {
                    DisplayHeaderFooter = true,
                    HeaderTemplate = "<div style='font-size:10px; text-align:center; width:100%;'>Custom Header</div>",
                    FooterTemplate = "<div style='font-size:10px; text-align:center; width:100%;'>Page <span class='pageNumber'></span> of <span class='totalPages'></span></div>",
                    MarginTop = "2cm",
                    MarginBottom = "2cm"
                }
            }
        });

        using var fs = File.Create("document_with_headers.pdf");
        report.Content.CopyTo(fs);
    }
}

Headers and Footers — After (IronPDF)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();

renderer.RenderingOptions.TextHeader = new TextHeaderFooter
{
    CenterText = "Custom Header",
    FontSize = 10
};
renderer.RenderingOptions.TextFooter = new TextHeaderFooter
{
    CenterText = "Page {page} of {total-pages}",
    FontSize = 10
};
renderer.RenderingOptions.MarginTop = 20;    // mm (2cm)
renderer.RenderingOptions.MarginBottom = 20;

var pdf = renderer.RenderHtmlAsPdf("<h1>Document with Header and Footer</h1><p>Main content goes here.</p>");
pdf.SaveAs("document_with_headers.pdf");
// Guide: https://ironpdf.com/how-to/headers-and-footers/

If you need rich HTML headers (logos, multi-column layouts) rather than plain centered text, use HtmlHeader / HtmlFooter with HtmlHeaderFooter and the same {page}, {total-pages}, {date}, {html-title}, {url} placeholders.

What About Performance and Deployment Considerations?

The biggest performance win is eliminating the cross-process hop. With jsreport.Local utility mode, every render starts the Node.js binary and tears it down; in web server mode, you pay a localhost HTTP round trip per render. IronPDF renders in-process against an embedded Chromium, with no spawn or network overhead.

// jsreport (utility mode):  spawn Node.js binary -> Chromium -> stream back -> exit
// jsreport (web server mode): HTTP call to local Node.js -> Chromium -> response
// IronPDF:                   in-process -> embedded Chromium -> PdfDocument

For ASP.NET Core applications, register the renderer as a singleton for reuse:

// Register as singleton in ASP.NET Core for reuse
builder.Services.AddSingleton<ChromePdfRenderer>(sp =>
{
    var renderer = new ChromePdfRenderer();
    renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
    return renderer;
});

For async rendering in web handlers:

// Async pattern for web handlers
var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(html);

// Return as file download
return File(pdf.Stream, "application/pdf", "report.pdf");

IronPDF's in-process Chromium is more predictable in containerized environments than jsreport's Node.js + Chromium stack. Verify configuration for your specific Azure tier — see the IronPDF Azure documentation.

A few edge cases worth flagging: if your jsreport templates have complex Handlebars helpers or partials, inventory them before migrating — each helper needs a .NET equivalent. Images, fonts, and stylesheets referenced as jsreport assets need to be moved to accessible static paths or embedded in the HTML. And if your system uses jsreport's scheduling features, build those workflows separately in .NET.

Frequently Asked Questions

Can I keep jsreport for non-PDF outputs and use IronPDF for PDFs?

Yes. If you rely on jsreport for Excel, CSV, or other non-PDF output formats, you can run both side by side. IronPDF covers PDF generation and manipulation only. Retain jsreport for the formats it handles natively, or replace those with dedicated .NET libraries for each format.

What happens to my Handlebars templates?

You have three options. For simple templates, use C# string interpolation. For complex templates with partials and helpers, use Handlebars.NET (the NuGet package Handlebars.Net) which lets you reuse your existing Handlebars syntax directly in .NET. For teams already using Razor elsewhere, migrate templates to .cshtml files and use RazorLight to compile them to HTML strings before passing to ChromePdfRenderer.

Does IronPDF have an equivalent to jsreport Studio?

No. IronPDF has no built-in template management UI. If jsreport Studio is business-critical for your team — for example, non-developers design templates in the browser — you'll need to build that workflow separately. Some teams create a simple internal tool with a Razor-based editor and preview endpoint that calls IronPDF under the hood.

How do I handle jsreport running as a Docker service?

Remove the jsreport service entirely from your docker-compose.yml or Kubernetes config. If you were using jsreport.Local with bundled binaries, drop the jsreport.Binary* packages and the platform-detection code. If jsreport was the only consumer of Node.js in your Docker images, remove the Node.js runtime as well. Update monitoring to replace jsreport health checks with in-process metrics. IronPDF runs inside your existing .NET application container — no separate service needed.

Is IronPDF thread-safe for concurrent PDF generation?

Yes, you can reuse a ChromePdfRenderer instance across requests. For ASP.NET Core, register it as a singleton in your DI container. For high-concurrency scenarios, use the async rendering methods (RenderHtmlAsPdfAsync) to avoid blocking threads. See the IronPDF async documentation for details.

Conclusion

The architectural shift from a Node.js subprocess (or local jsreport server) to a pure in-process .NET library is the meaningful part of this migration. The API differences are straightforward once you've mapped the template engine and decided how non-PDF outputs will be handled.

Teams that hit friction tend to be the ones with large Handlebars template libraries — that migration is mechanical but takes time. And teams who depended on jsreport Studio for template management need to decide where that workflow goes.

If you're generating PDFs only and dealing with container deployment complexity, this migration pays for itself in reduced operational overhead. Start with the three-step quick start, migrate one template at a time, and validate output against your jsreport reference PDFs before cutting over.
