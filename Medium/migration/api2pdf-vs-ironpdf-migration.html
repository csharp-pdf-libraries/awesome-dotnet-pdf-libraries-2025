Migrating from Api2pdf to IronPDF: Solving Common Migration Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Would You Migrate Away from Api2pdf?

This is not an API-to-API migration. It is a paradigm shift from a SaaS REST service to an in-process .NET library. Api2pdf removes the pain of hosting Chrome or wkhtmltopdf yourself, and it does that job well. But teams outgrow it for reasons that have nothing to do with the quality of the service.

Latency per render. Every PDF is an HTTP round-trip: POST HTML, wait for the render, download the file from a temporary URL. For high-volume applications, this adds 1–5 seconds per PDF that an in-process library eliminates.

Temporary file URLs. Api2pdf returns a URL to the generated file, not the file itself. Your application must download the PDF before the 24-hour expiry. This introduces a race condition and a failure mode that does not exist with in-process generation.

External dependency for a core function. If Api2pdf's service goes down, your PDF generation stops. If your internet connection degrades, your PDF generation slows. For applications where PDFs are on the critical path — invoices, contracts, compliance docs — this is a risk.

Data leaves your network. Your HTML, which may contain customer names, addresses, and financial data, is sent to Api2pdf's servers for rendering. For regulated industries like healthcare, finance, and legal, this data transit may require additional compliance review.

Per-call cost at scale. Api2pdf's usage-based pricing is economical for low volume. At hundreds of thousands of calls per month, a one-time library license may be cheaper.

No offline capability. If your application needs to generate PDFs in disconnected environments — on-premise deployments, air-gapped networks — a REST API is not an option.

Limited post-processing. Api2pdf handles generation and merge well, but more advanced manipulation like watermarking, page-level security settings, and form filling requires additional API calls or is not supported.

wkhtmltopdf deprecation risk. Api2pdf offers wkhtmltopdf as a rendering option. wkhtmltopdf is based on a deprecated version of WebKit and has not received updates.

File size and timeout limits. Cloud APIs have practical limits on request payload size and processing time. Very large or complex HTML documents may timeout.

Debugging opacity. When a render fails or produces unexpected output, you cannot inspect the rendering engine's state. With an in-process library, you can step through with a debugger.

What Does the SaaS-to-Library Paradigm Shift Actually Mean?

You are changing the execution model entirely. With Api2pdf, rendering happens on their AWS/GCP servers. Your HTML travels over the internet, the PDF sits on a temporary URL for 24 hours, and your availability depends on their uptime.

With IronPDF, rendering happens in your own process. HTML stays in memory. Files live on your filesystem. Availability depends on your uptime, not someone else's.

ConcernApi2pdf (SaaS)IronPDF (Library)
Where rendering happensApi2pdf's AWS/GCP serversYour server / container
Data transitHTML sent over internetHTML stays in-process
AvailabilityDepends on Api2pdf uptimeDepends on your uptime
LatencyNetwork round-trip + render timeRender time only
File lifecycle24-hour temp URLYour filesystem / memory

The most significant performance gain is eliminating the network round-trip. Rough illustrative ranges: Api2pdf runs approximately 1,500–4,000ms per render (network + remote render + download), while IronPDF in-process runs approximately 200–1,000ms per render (render only, depends on HTML complexity). Your actual numbers will depend on network conditions, HTML complexity, and server resources.

How Do I Replace Html-to-PDF Generation?

This is the most common Api2pdf call, and it maps directly. The key difference: Api2pdf returns a URL you must download before it expires. IronPDF returns a PdfDocument object in memory.

Api2pdf HTML-to-PDF (before)

using Api2Pdf;
using System;
using System.IO;
using System.Net.Http;

class HtmlToPdfApi2pdf
{
    static void Main()
    {
        var a2p = new Api2Pdf.Api2Pdf("your-api-key");

        var request = new ChromeHtmlToPdfRequest
        {
            Html = @"<html><head><style>
                body { font-family: Arial; }
                .header { background: #0d47a1; color: white; padding: 20px; }
                table { width: 100%; border-collapse: collapse; margin: 20px 0; }
                td, th { padding: 8px; border: 1px solid #ddd; }
            </style></head><body>
                <div class='header'><h1>Statement</h1></div>
                <table>
                    <tr><th>Date</th><th>Description</th><th>Amount</th></tr>
                    <tr><td>2025-01-15</td><td>Service fee</td><td>$250.00</td></tr>
                    <tr><td>2025-01-20</td><td>Subscription</td><td>$99.00</td></tr>
                </table>
            </body></html>"
        };

        var result = a2p.Chrome.HtmlToPdf(request);

        if (!result.Success)
        {
            Console.WriteLine("API error: " + result.Error);
            return;
        }

        // Must download from temporary URL
        using var http = new HttpClient();
        var bytes = http.GetByteArrayAsync(result.FileUrl).Result;
        File.WriteAllBytes("statement.pdf", bytes);
        Console.WriteLine("Downloaded statement.pdf");
    }
}

IronPDF HTML-to-PDF (after)

using IronPdf;

class HtmlToPdfIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "IRONPDF-YOUR-KEY-HERE";

        var renderer = new ChromePdfRenderer();
        renderer.RenderingOptions.PrintHtmlBackgrounds = true;

        var pdf = renderer.RenderHtmlAsPdf(@"<html><head><style>
                body { font-family: Arial; }
                .header { background: #0d47a1; color: white; padding: 20px; }
                table { width: 100%; border-collapse: collapse; margin: 20px 0; }
                td, th { padding: 8px; border: 1px solid #ddd; }
            </style></head><body>
                <div class='header'><h1>Statement</h1></div>
                <table>
                    <tr><th>Date</th><th>Description</th><th>Amount</th></tr>
                    <tr><td>2025-01-15</td><td>Service fee</td><td>$250.00</td></tr>
                    <tr><td>2025-01-20</td><td>Subscription</td><td>$99.00</td></tr>
                </table>
            </body></html>");

        pdf.SaveAs("statement.pdf");
        Console.WriteLine("Saved locally — no download step.");
    }
}

No HTTP client. No temporary URL. No download race condition. See the IronPDF HTML-to-PDF tutorial for more rendering options.

How Do I Replace PDF Merging?

Api2pdf's merge endpoint requires URLs to existing PDFs hosted somewhere accessible. Your PDFs must be publicly reachable, and the merged result comes back as yet another temporary URL you must download. IronPDF merges local files, byte arrays, or in-memory PdfDocument objects directly.

Api2pdf merge (before)

using Api2Pdf;
using System;
using System.IO;
using System.Net.Http;
using System.Collections.Generic;

class MergeApi2pdf
{
    static void Main()
    {
        var a2p = new Api2Pdf.Api2Pdf("your-api-key");

        // Api2pdf merge requires URLs to existing PDFs
        var pdfUrls = new List<string>
        {
            "https://your-server.com/files/part1.pdf",
            "https://your-server.com/files/part2.pdf"
        };

        var result = a2p.PdfSharp.Merge(pdfUrls);

        if (!result.Success)
        {
            Console.WriteLine("Merge error: " + result.Error);
            return;
        }

        // Download the merged result
        using var http = new HttpClient();
        var bytes = http.GetByteArrayAsync(result.FileUrl).Result;
        File.WriteAllBytes("merged.pdf", bytes);
        Console.WriteLine("Merged and downloaded.");
    }
}

IronPDF merge (after)

using IronPdf;

class MergeIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "IRONPDF-YOUR-KEY-HERE";

        var pdf1 = PdfDocument.FromFile("part1.pdf");
        var pdf2 = PdfDocument.FromFile("part2.pdf");

        var merged = PdfDocument.Merge(pdf1, pdf2);
        merged.SaveAs("merged.pdf");
        Console.WriteLine("Merged locally.");
    }
}

No need to host PDFs at public URLs. No download step after the merge. No 24-hour expiry on the merged result. See the IronPDF merge documentation for additional options.

How Do I Add Watermarks and Password Protection?

These are two areas where the migration actually gives you capabilities you did not have before. Api2pdf does not have a native watermark endpoint. Password protection requires a separate API call to a URL-based endpoint. IronPDF handles both in-process with granular control.

Watermarking with IronPDF

using IronPdf;

class WatermarkIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "IRONPDF-YOUR-KEY-HERE";

        var pdf = PdfDocument.FromFile("input.pdf");

        pdf.ApplyWatermark(
            "<h1 style='color:rgba(0,0,0,0.15);font-size:60px;'>DRAFT</h1>",
            rotation: 45
        );

        pdf.SaveAs("watermarked.pdf");
        Console.WriteLine("Watermarked locally.");
    }
}

Password protection with Api2pdf (before)

using Api2Pdf;
using System;
using System.IO;
using System.Net.Http;

class PasswordApi2pdf
{
    static void Main()
    {
        var a2p = new Api2Pdf.Api2Pdf("your-api-key");

        // Api2pdf password endpoint requires a URL to an existing PDF
        var result = a2p.PdfSharp.AddPassword(
            "https://your-server.com/files/input.pdf",
            "user456"
        );

        if (!result.Success)
        {
            Console.WriteLine("Password error: " + result.Error);
            return;
        }

        using var http = new HttpClient();
        var bytes = http.GetByteArrayAsync(result.FileUrl).Result;
        File.WriteAllBytes("protected.pdf", bytes);
        Console.WriteLine("Password-protected and downloaded.");
    }
}

Password protection with IronPDF (after)

using IronPdf;

class PasswordIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "IRONPDF-YOUR-KEY-HERE";

        var pdf = PdfDocument.FromFile("input.pdf");

        pdf.SecuritySettings.OwnerPassword = "owner123";
        pdf.SecuritySettings.UserPassword = "user456";
        pdf.SecuritySettings.AllowUserPrinting = IronPdf.Security.PdfPrintSecurity.NoPrint;
        pdf.SecuritySettings.AllowUserCopyPasteContent = false;

        pdf.SaveAs("protected.pdf");
        Console.WriteLine("Protected locally.");
    }
}

IronPDF gives you granular permission control — not just a single password. Your PDF file never leaves your server. See the IronPDF security documentation for the full list of permission settings.

What Are the Most Common Questions About This Migration?

How do I handle async rendering after migrating?

Answer:

Api2pdf calls are inherently async because they are HTTP requests. IronPDF rendering is synchronous by default but offers async variants. Replace your awaited HTTP calls with:

var pdf = await renderer.RenderHtmlAsPdfAsync(html);

Your existing async/await patterns carry over cleanly. The difference is that the async operation is now local CPU-bound work, not a network call.

How does error handling change?

Answer:

Api2pdf returns result.Success and result.Error on every call. IronPDF throws exceptions. You need to replace your conditional checks with try/catch blocks:

// Api2pdf pattern
if (!result.Success) { log(result.Error); return; }

// IronPDF pattern
try { var pdf = renderer.RenderHtmlAsPdf(html); }
catch (Exception ex) { log(ex.Message); return; }

This is a structural change that affects every call site. Plan for it during your migration pass.

What about LibreOffice Office-to-PDF conversion?

Answer:

If you use Api2pdf's LibreOffice endpoint for Office-to-PDF conversion, IronPDF does not replace this. Run LibreOffice headless locally (libreoffice --headless --convert-to pdf input.docx) or use a dedicated service. Alternatively, the IronWord library from Iron Software handles DOCX-to-PDF within .NET.

Will my wkhtmltopdf HTML render the same in IronPDF's Chrome engine?

Answer:

If your existing Api2pdf integration uses wkhtmltopdf (not Chrome), switching to IronPDF's Chrome engine may produce different output for the same HTML. Chrome's rendering is more modern and accurate, but you should test your templates and adjust CSS if needed. This is a rendering engine change, not just an API change.

How do I reuse the renderer for performance?

Answer:

Create a single ChromePdfRenderer instance and reuse it. The renderer is stateless and safe to share. Dispose of individual PdfDocument objects when you are done with them:

private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

// In your method:
using var pdf = _renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");

This avoids the overhead of spinning up a new Chrome process for each render.

High-volume HTML-to-PDF with latency concernsMigrate — in-process rendering eliminates network overhead
Regulated industry with data residency requirementsMigrate — HTML never leaves your infrastructure
Low-volume, diverse format conversion (Office, email, images)Stay or partial migrate — keep Api2pdf for Office conversion, use IronPDF for HTML-to-PDF
Air-gapped or on-premise deploymentMigrate — Api2pdf requires internet

The core win in this migration is architectural: your PDF pipeline moves from a remote service dependency to an in-process operation. Your HTML never leaves your server, your files never sit on a temporary URL with a 24-hour countdown, and your rendering is not gated by network latency or third-party availability.

Start with the dependency swap, replace your highest-volume call site first, and verify the output matches. The rest of the migration follows the same pattern. Get a trial key from IronPDF's license page to test with your own templates before committing.
