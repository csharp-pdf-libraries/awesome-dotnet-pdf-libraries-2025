Migrating from PDFmyURL to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

The install story for PDFmyURL-based workflows usually goes the same way: a README with a dependency list that spills past the fold, a native binary that has to be coaxed into a Docker layer, and a CI pipeline that breaks whenever the base image updates. The rendering itself may be fine, but the setup overhead is a recurring tax on every new environment.

This article walks through replacing that setup with IronPDF — a .NET-native library with no external process dependencies. You will have copy-paste migration code for HTML-to-PDF, merge, watermark, and password protection by the end. Even if you do not adopt IronPDF, the comparison tables and checklist are useful for evaluating any replacement.

Why Should .NET Teams Migrate Away from PDFmyURL?

Answer:

PDFmyURL works well for straightforward URL-to-PDF use cases in environments where outbound HTTP is unrestricted. Teams commonly start looking at alternatives when one or more of these conditions appear:

1. Air-gapped or restricted network environments — every render requires an outbound HTTPS call to pdfmyurl.com.

2. Latency sensitivity — round-trip time to an external API adds to every document generation request.

3. Data privacy requirements — HTML content containing PII or confidential data leaves the network boundary on every call.

4. Offline or on-premise deployments — SaaS dependency conflicts with deployment requirements.

5. API rate limits — high-volume scenarios require plan upgrades or request throttling logic.

6. Bundle and container size concerns — while the client is thin, managing API credentials, retry logic, and timeout handling adds non-trivial code.

7. Error handling complexity — HTTP status codes from a remote service require different error-handling patterns than local exceptions.

8. CI/CD environment variability — network policy changes or firewall rules can silently break renders in build pipelines.

9. Dependency on third-party uptime — any SaaS outage directly impacts your PDF generation.

10. Local test environments — mocking an external API for unit tests adds friction.

What Are the Decision Criteria for Switching?

Answer:

Not every team needs to migrate. Here is the decision matrix I use when consulting with teams who are evaluating this move.

Business Scenario Recommendation
Air-gapped or on-premise deployment Switch — PDFmyURL cannot operate without outbound network
Low-volume, internet-connected, URL-only renders Staying on PDFmyURL is reasonable; evaluate based on latency/privacy needs
PII or confidential content in rendered HTML Switch — data leaves the network boundary on every PDFmyURL call
High-volume with complex PDF manipulation needs Switch — PDFmyURL lacks local merge/watermark/security features

The core rendering swap itself is straightforward. Here is the before and after for a basic URL-to-PDF conversion.

Before (PDFmyURL via HttpClient)

using System;
using System.Net.Http;
using System.Threading.Tasks;
using System.IO;
using System.Collections.Generic;

class Program
{
    static async Task Main()
    {
        using var client = new HttpClient();

        // PDFmyURL requires POST with form data — verify current API endpoint
        var formData = new FormUrlEncodedContent(new[]
        {
            new KeyValuePair<string, string>("url", "https://example.com"),
            new KeyValuePair<string, string>("apikey", Environment.GetEnvironmentVariable("PDFMYURL_KEY")),
        });

        var response = await client.PostAsync("https://api.pdfmyurl.com/api/save", formData);

        if (!response.IsSuccessStatusCode)
        {
            throw new Exception($"PDF generation failed: {response.StatusCode}");
        }

        var pdfBytes = await response.Content.ReadAsByteArrayAsync();
        await File.WriteAllBytesAsync("output.pdf", pdfBytes);
        Console.WriteLine("Saved output.pdf");
    }
}

After (IronPDF)

using IronPdf;
using System;

// License: https://ironpdf.com/how-to/license-keys/
IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

// Render URL directly — no network round-trip to an external service
var pdf = await renderer.RenderUrlAsPdfAsync("https://example.com");
pdf.SaveAs("output.pdf");

Console.WriteLine("Saved output.pdf");

What Do Complete Before/After Code Migrations Look Like?

Answer:

Here are four complete migrations covering the most common scenarios. Each one shows the full PDFmyURL code and its IronPDF replacement.

1. HTML String to PDF — Before (PDFmyURL via HttpClient)

using System;
using System.Net.Http;
using System.Collections.Generic;
using System.IO;
using System.Threading.Tasks;

class HtmlToPdfBefore
{
    static async Task Main()
    {
        var apiKey = Environment.GetEnvironmentVariable("PDFMYURL_KEY");
        var html = "<html><body><h1>Invoice #1042</h1><p>Amount due: $1,200</p></body></html>";

        using var client = new HttpClient();

        // PDFmyURL API — verify current endpoint and parameter names
        var formData = new FormUrlEncodedContent(new[]
        {
            new KeyValuePair<string, string>("value", html),
            new KeyValuePair<string, string>("apikey", apiKey),
            new KeyValuePair<string, string>("page_size", "A4"),
        });

        var response = await client.PostAsync("https://api.pdfmyurl.com/api/save", formData);

        if (!response.IsSuccessStatusCode)
            throw new Exception($"API error: {response.StatusCode} — check PDFmyURL docs for error codes");

        var bytes = await response.Content.ReadAsByteArrayAsync();
        await File.WriteAllBytesAsync("invoice.pdf", bytes);

        Console.WriteLine($"Saved {bytes.Length} bytes to invoice.pdf");
    }
}

1. HTML String to PDF — After (IronPDF)

using IronPdf;
using System;

// License: https://ironpdf.com/how-to/license-keys/
IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var html = "<html><body><h1>Invoice #1042</h1><p>Amount due: $1,200</p></body></html>";

var renderer = new ChromePdfRenderer();
// Rendering options: https://ironpdf.com/how-to/rendering-options/
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");

Console.WriteLine($"Saved invoice.pdf ({pdf.PageCount} page(s))");

2. Merge Multiple PDFs — Before (PDFmyURL — Manual Assembly)

using System;
using System.Net.Http;
using System.Collections.Generic;
using System.IO;
using System.Threading.Tasks;

class MergeBefore
{
    // PDFmyURL doesn't support merge natively.
    // Teams typically generate each PDF separately then use a separate library
    // (PdfSharp, iTextSharp, etc.) to combine the byte arrays.
    static async Task Main()
    {
        var apiKey = Environment.GetEnvironmentVariable("PDFMYURL_KEY");
        using var client = new HttpClient();

        var htmlPages = new[]
        {
            "<html><body><h1>Section 1</h1></body></html>",
            "<html><body><h1>Section 2</h1></body></html>",
        };

        var pdfBytesList = new List<byte[]>();

        foreach (var html in htmlPages)
        {
            var formData = new FormUrlEncodedContent(new[]
            {
                new KeyValuePair<string, string>("value", html),
                new KeyValuePair<string, string>("apikey", apiKey),
            });

            var response = await client.PostAsync("https://api.pdfmyurl.com/api/save", formData);
            if (!response.IsSuccessStatusCode)
                throw new Exception($"API error: {response.StatusCode}");

            pdfBytesList.Add(await response.Content.ReadAsByteArrayAsync());
        }

        // Now use a separate local library to merge — this is a common additional dependency
        // shown here as pseudo-code since it varies by library:
        // var merged = SomePdfLibrary.Merge(pdfBytesList);
        // File.WriteAllBytes("merged.pdf", merged);

        Console.WriteLine("Merge requires separate PDF library — verify in your implementation");
    }
}

2. Merge Multiple PDFs — After (IronPDF)

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

// Render both sections in parallel

// Merge: https://ironpdf.com/how-to/merge-or-split-pdfs/

merged.SaveAs("merged.pdf");

Console.WriteLine($"Merged PDF: {merged.PageCount} pages");

What Are the Critical Things to Watch During Migration?

Answer:

The main structural change in this migration is the shift from HTTP response handling to exception-based error handling — that is where most of the code rewrite lives. The render call itself is simpler.

PDFmyURL integration typically involves checking response.IsSuccessStatusCode and branching on HTTP error codes. IronPDF throws typed .NET exceptions instead. Audit all error-handling code that inspects status codes and convert to try/catch blocks.

Error Handling Pattern

// Pattern to refactor:
// if (!response.IsSuccessStatusCode) { /* handle */ }

// Replace with:
try
{
    var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    pdf.SaveAs("output.pdf");
}
catch (IronPdf.Exceptions.IronPdfException ex)
{
    // Handle IronPDF-specific errors
    Console.Error.WriteLine($"Render failed: {ex.Message}");
}

IronPDF uses 0-based page indexing. Keep this in mind when migrating any page manipulation logic.

Tests that mocked or stubbed the PDFmyURL HTTP endpoint can be simplified — IronPDF renders locally so no network mock is needed. Remove HttpMessageHandler mocks and test with actual rendered output.

PDFmyURL integration uses await for HTTP calls. IronPDF also exposes async methods (RenderHtmlAsPdfAsync, RenderUrlAsPdfAsync). The async call sites transfer cleanly. See the async guide.

How Should I Handle Performance and Renderer Reuse?

Answer:

Instantiating ChromePdfRenderer per request is functional but suboptimal for high-volume scenarios. A common pattern is to reuse a single renderer instance.

Renderer Reuse

// Reuse renderer — verify thread-safety guidance in IronPDF docs
var renderer = new ChromePdfRenderer();

// Render multiple documents
foreach (var job in renderQueue)
{
    var pdf = await renderer.RenderHtmlAsPdfAsync(job.Html);
    pdf.SaveAs(job.OutputPath);
}

Parallel Rendering

using IronPdf;
using System.Linq;
using System.Threading.Tasks;

// Parallel rendering: https://ironpdf.com/examples/parallel/
var htmlJobs = new[] { "<h1>Doc 1</h1>", "<h1>Doc 2</h1>", "<h1>Doc 3</h1>" };

var tasks = htmlJobs.Select(async html =>
{
    var r = new ChromePdfRenderer(); // verify per-task vs shared instance guidance
    return await r.RenderHtmlAsPdfAsync(html);
});

var results = await Task.WhenAll(tasks);
Console.WriteLine($"Rendered {results.Length} PDFs in parallel");

PdfDocument implements IDisposable. Use using blocks for proper cleanup:

using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("output.pdf");
// Disposed at end of block

One area worth benchmarking carefully: if your PDFmyURL integration was handling very large pages or complex CSS, measure local render times against your previous API response times. In network-constrained environments PDFmyURL calls may have been the bottleneck; in others, a cold Chromium startup may be. The tradeoff is different per workload.

Frequently Asked Questions

Do I need an internet connection for IronPDF to render PDFs?

Answer:

No. IronPDF embeds its own Chromium rendering engine and runs entirely locally. This is one of the primary reasons teams migrate away from PDFmyURL — every PDFmyURL render requires an outbound HTTPS call to pdfmyurl.com. With IronPDF, air-gapped and on-premise deployments work without any network dependency.

Can I merge and watermark PDFs without adding extra libraries?

Answer:

Yes. PDFmyURL does not support merge, watermark, or password protection natively — teams typically bolt on a separate local library like PdfSharp or iTextSharp for those features. IronPDF includes all of these capabilities in a single package: PdfDocument.Merge() for merging, TextStamper / ImageStamper for watermarks, and SecuritySettings for password protection. See the merge/split guide for details.

How do I handle errors differently after migrating?

Answer:

With PDFmyURL, you check response.IsSuccessStatusCode and branch on HTTP status codes. IronPDF throws typed .NET exceptions instead. Refactor all your status-code inspection logic into standard try/catch blocks using IronPdf.Exceptions.IronPdfException. This is where most of the code rewrite effort lives.

What .NET versions does IronPDF support?

Answer:

IronPDF supports .NET 6, 7, 8, and 9. Verify compatibility with your target framework before starting the migration. You can get a trial license key to test against your specific environment.

How Should I Decide Whether to Migrate?

Answer:

Use this framework. If your deployment is air-gapped or on-premise, the decision is already made — PDFmyURL cannot operate without outbound network access. If your rendered HTML contains PII or confidential data, that content leaves your network boundary on every PDFmyURL call, which may conflict with compliance requirements.

If you are in a low-volume, internet-connected, URL-only scenario, staying on PDFmyURL is reasonable. Evaluate based on latency and privacy needs rather than feature gaps.

For high-volume scenarios with complex PDF manipulation needs (merge, watermark, security), the switch pays for itself quickly because PDFmyURL lacks those features natively and you are likely already maintaining a second library to fill the gaps.

The main structural change in this migration is the shift from HTTP response handling to exception-based error handling. The render call itself is simpler on the IronPDF side. Start with the effort table above, scan your codebase for all PDFmyURL references, and estimate scope before committing.
