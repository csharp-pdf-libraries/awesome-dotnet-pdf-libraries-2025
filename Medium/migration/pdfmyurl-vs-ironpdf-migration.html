Migrating from PDFmyURL to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

The install story for PDFmyURL-based workflows usually goes the same way: a thin SDK wrapper, an API key wired into config, and a CI pipeline that depends on outbound network access to a third-party host. The rendering itself may be fine, but the network and credential plumbing is a recurring tax on every new environment.

This article walks through replacing that setup with IronPDF — a .NET-native library that renders locally with no remote API dependency. You will have copy-paste migration code for URL-to-PDF, HTML-to-PDF, merge, watermark, and password protection by the end. Even if you do not adopt IronPDF, the comparison and checklist are useful for evaluating any replacement.

Why Should .NET Teams Migrate Away from PDFmyURL?

PDFmyURL works well for straightforward URL-to-PDF use cases in environments where outbound HTTP is unrestricted. Teams commonly start looking at alternatives when one or more of these conditions appear:

1. Air-gapped or restricted network environments — every render requires an outbound HTTPS call to pdfmyurl.com.

2. Latency sensitivity — round-trip time to an external API adds to every document generation request.

3. Data privacy requirements — HTML content containing PII or confidential data leaves the network boundary on every call.

4. Offline or on-premise deployments — SaaS dependency conflicts with deployment requirements.

5. API rate limits — high-volume scenarios require plan upgrades or request throttling logic.

6. Bundle and container size concerns — while the client is thin, managing API credentials, retry logic, and timeout handling adds non-trivial code.

7. Error handling complexity — HTTP status codes from a remote service require different error-handling patterns than local exceptions.

8. CI/CD environment variability — network policy changes or firewall rules can silently break renders in build pipelines.

9. Dependency on third-party uptime — any SaaS outage directly impacts your PDF generation.

10. Local test environments — mocking an external API for unit tests adds friction.

  • Focus. PDFmyURL is a remote HTML-to-PDF SaaS API. IronPDF is a local .NET PDF library that handles HTML rendering and post-render manipulation in-process.
  • Pricing. PDFmyURL uses subscription tiers by page volume — verify current rates at pdfmyurl.com. IronPDF is a commercial license — verify at ironsoftware.com.
  • API style. PDFmyURL is a REST API consumed via the Pdfcrowd .NET SDK (HtmlToPdfClient) or raw HTTP. IronPDF is native C# objects and method calls.
  • Learning curve. Both are low-friction. PDFmyURL adds HTTP and credential handling; IronPDF abstracts the Chromium renderer internals.
  • HTML rendering. PDFmyURL runs a remote Chromium instance. IronPDF embeds Chromium in your process.
  • Page indexing. PDFmyURL is not a page-manipulation library. IronPDF is 0-based.
  • Thread safety. PDFmyURL calls are stateless HTTP and inherently parallelizable. IronPDF supports renderer reuse; verify concurrent instance guidance for your scenario.
  • Namespace. PDFmyURL ships as Pdfcrowd (the official SDK). IronPDF ships as IronPdf.

What Does the Migration Effort Look Like Feature by Feature?

I have put together a feature-by-feature breakdown so you can estimate the scope of work before touching any code. Most teams find the core rendering swap is trivial — the complexity lives in error handling and any post-render manipulation you bolted on separately.

  • URL to PDF — Low. PDFmyURL: client.convertUrlToFile(url, path). IronPDF: ChromePdfRenderer.RenderUrlAsPdf().
  • HTML string to PDF — Low. PDFmyURL: client.convertStringToFile(html, path). IronPDF: ChromePdfRenderer.RenderHtmlAsPdf().
  • Save to disk — Low. PDFmyURL writes via the SDK's convert*ToFile methods or response bytes. IronPDF: pdf.SaveAs().
  • Save to MemoryStream — Low. PDFmyURL: read the response stream from convert*ToStream. IronPDF: pdf.Stream.
  • Merge PDFs — Medium. Not a native PDFmyURL feature — requires multiple API calls plus a separate local library to combine the bytes. IronPDF: PdfDocument.Merge().
  • Watermark — Medium. Not a native PDFmyURL feature. IronPDF: pdf.ApplyWatermark() for HTML watermarks, or pdf.ApplyStamp() with TextStamper/ImageStamper for stamp-style placement.
  • Password protection — Medium. PDFmyURL exposes setEncrypt, setUserPassword, setOwnerPassword on the SDK client. IronPDF: pdf.SecuritySettings after rendering.
  • Page manipulation — Medium. Not a native PDFmyURL feature. IronPDF: pdf.RemovePages(), CopyPages(), RotateAllPages(), etc.
  • Custom headers/footers — Medium-High. PDFmyURL: setHeaderHtml/setFooterHtml with {page_number}/{total_pages} placeholders. IronPDF: RenderingOptions.HtmlHeader/HtmlFooter with {page}/{total-pages}.
  • Async parallel rendering — Medium. PDFmyURL: Task.WhenAll over HTTP-bound async calls. IronPDF: Task.WhenAll over local renders.
  • Error handling migration — Medium. PDFmyURL throws Pdfcrowd.Error from the SDK. IronPDF surfaces typed exceptions like IronPdfRenderingException — refactor any HTTP-status branching.

What Are the Decision Criteria for Switching?

Not every team needs to migrate. Here is how I frame the decision when consulting with teams who are evaluating this move.

  • If your deployment is air-gapped or on-premise, switch — PDFmyURL cannot operate without outbound network access.
  • If you are doing low-volume, internet-connected, URL-only renders, staying on PDFmyURL is reasonable. Evaluate based on latency and privacy needs rather than feature gaps.
  • If your rendered HTML contains PII or confidential content, switch — that data leaves the network boundary on every PDFmyURL call.
  • If you are doing high-volume work with complex PDF manipulation needs, switch — PDFmyURL lacks local merge, watermark, and security features, so you are likely already bolting on a second library.

The core rendering swap itself is straightforward. Here is the before and after for a basic URL-to-PDF conversion.

Before (PDFmyURL via Pdfcrowd SDK)

using System;
using Pdfcrowd;

class Program
{
    static void Main()
    {
        try
        {
            // API credentials required for every conversion
            var client = new HtmlToPdfClient("username", "apikey");

            // URL is sent to the PDFmyURL service for rendering
            client.convertUrlToFile("https://example.com", "output.pdf");

            Console.WriteLine("Saved output.pdf");
        }
        catch (Error why)
        {
            Console.WriteLine("Error: " + why);
        }
    }
}

After (IronPDF)

using IronPdf;
using System;

// License: https://ironpdf.com/how-to/license-keys/
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();

// Render URL directly — no network round-trip to an external service
var pdf = await renderer.RenderUrlAsPdfAsync("https://example.com");
pdf.SaveAs("output.pdf");

Console.WriteLine("Saved output.pdf");

How Do the APIs Map Across the Two Libraries?

Here is the full API mapping. These cover namespaces, core classes, document loading, page operations, and merge/split operations.

Namespace Mapping

  • Pdfcrowd → IronPdf — core SDK namespace replaces the PDFmyURL client
  • Pdfcrowd.HtmlToPdfClient → IronPdf.ChromePdfRenderer — renderer replaces the API client
  • API response object → IronPdf.PdfDocument — returned directly from the renderer
  • Manual byte/stream handling → pdf.SaveAs() / pdf.Stream — built-in save methods

Core Class Mapping

  • HtmlToPdfClient → ChromePdfRenderer — renders HTML or URL to PDF
  • Response bytes → PdfDocument — full PDF object with manipulation methods
  • setXxx client setters → ChromePdfRenderOptions properties — rendering config (page size, margins, headers)
  • No local manipulation → PdfDocument.Merge() — static merge method available locally

Document Loading Methods

  • client.convertUrlToFile(url, path) → renderer.RenderUrlAsPdfAsync(url) — renders a live URL
  • client.convertStringToFile(html, path) → renderer.RenderHtmlAsPdfAsync(html) — renders an HTML string
  • client.convertFileToFile(in, out) → renderer.RenderHtmlFileAsPdfAsync(path) — renders a local HTML file
  • No local equivalent → PdfDocument.FromFile(path) — loads an existing PDF for manipulation

Page Operations

  • Page count. Not available locally on PDFmyURL responses. IronPDF: pdf.PageCount.
  • Remove page. Not available in PDFmyURL. IronPDF: pdf.RemovePages(index).
  • Copy page. Not available in PDFmyURL. IronPDF: pdf.CopyPages(start, end).
  • Rotate page. Not available in PDFmyURL. IronPDF: pdf.RotateAllPages(PdfPageRotation.Rotate90).

Merge / Split Operations

  • Merge multiple PDFs. PDFmyURL requires multiple API calls plus manual byte assembly with a separate library. IronPDF: PdfDocument.Merge(doc1, doc2).
  • Split PDF. Not a native PDFmyURL feature. IronPDF: pdf.CopyPages(start, end) — see the merge/split guide.

What Do Complete Before/After Code Migrations Look Like?

Here are four complete migrations covering the most common scenarios. Each one shows the full PDFmyURL code and its IronPDF replacement.

1. HTML String to PDF — Before (PDFmyURL via Pdfcrowd SDK)

using System;
using Pdfcrowd;

class HtmlToPdfBefore
{
    static void Main()
    {
        try
        {
            var client = new HtmlToPdfClient("username", "apikey");
            client.setPageSize("A4");

            var html = "<html><body><h1>Invoice #1042</h1><p>Amount due: $1,200</p></body></html>";

            client.convertStringToFile(html, "invoice.pdf");

            Console.WriteLine("Saved invoice.pdf");
        }
        catch (Error why)
        {
            Console.WriteLine("Error: " + why);
        }
    }
}

1. HTML String to PDF — After (IronPDF)

using IronPdf;
using System;

// License: https://ironpdf.com/how-to/license-keys/
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var html = "<html><body><h1>Invoice #1042</h1><p>Amount due: $1,200</p></body></html>";

var renderer = new ChromePdfRenderer();
// Rendering options: https://ironpdf.com/how-to/rendering-options/
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");

Console.WriteLine($"Saved invoice.pdf ({pdf.PageCount} page(s))");
// See: https://ironpdf.com/how-to/html-string-to-pdf/

                pdfBytesList.Add(client.convertString(html));

            // shown here as pseudo-code since it varies by library:
            // var merged = SomePdfLibrary.Merge(pdfBytesList);

2. Merge Multiple PDFs — After (IronPDF)

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();

// Render both sections in parallel
var pdf1 = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 1</h1></body></html>");

// Merge: https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(pdf1, pdf2);
merged.SaveAs("merged.pdf");

Console.WriteLine($"Merged PDF: {merged.PageCount} pages");

4. Password Protection — Before (PDFmyURL via Pdfcrowd SDK)

using System;
using Pdfcrowd;

class PasswordBefore

    {
        try
        {
            var client = new HtmlToPdfClient("username", "apikey");

            // Enable encryption and set passwords
            client.setEncrypt(true);
            client.setUserPassword("open123");
            client.setOwnerPassword("admin456");

        catch (Error why)
        {
            Console.WriteLine("Error: " + why);
        }
    }
}

4. Password Protection — After (IronPDF)

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Confidential Report</h1></body></html>");

pdf.SecuritySettings.UserPassword = "open123";

pdf.SaveAs("protected.pdf");
Console.WriteLine("Saved protected.pdf");

What Are the Critical Things to Watch During Migration?

The main structural change in this migration is the shift from a remote-call error model to local exception-based handling — that is where most of the code rewrite lives. The render call itself is simpler.

PDFmyURL integration via the Pdfcrowd SDK throws Pdfcrowd.Error that wraps service-side errors (invalid credentials, rate limits, network failures, invalid URL). IronPDF throws typed .NET exceptions instead. Audit all error-handling code that catches Pdfcrowd.Error or inspects HTTP-style error categories and convert to try/catch blocks.

Error Handling Pattern

// Pattern to refactor:
// catch (Pdfcrowd.Error why) { /* handle service error */ }

// Replace with:
try
{
    var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    pdf.SaveAs("output.pdf");
}
catch (IronPdf.Exceptions.IronPdfException ex)
{
    // Handle IronPDF-specific errors
    Console.Error.WriteLine($"Render failed: {ex.Message}");
}

IronPDF uses 0-based page indexing. Keep this in mind when migrating any page manipulation logic.

Tests that stubbed the PDFmyURL service can be simplified — IronPDF renders locally so no network mock is needed. Remove SDK or HTTP mocks and test against actual rendered output.

The Pdfcrowd SDK exposes synchronous convert* methods. IronPDF exposes async methods (RenderHtmlAsPdfAsync, RenderUrlAsPdfAsync) that you can adopt directly. See the async guide.

How Should I Handle Performance and Renderer Reuse?

Instantiating ChromePdfRenderer per request is functional but suboptimal for high-volume scenarios. A common pattern is to reuse a single renderer instance.

Renderer Reuse

// Reuse renderer — verify thread-safety guidance in IronPDF docs
var renderer = new ChromePdfRenderer();

// Render multiple documents
foreach (var job in renderQueue)
{
    var pdf = await renderer.RenderHtmlAsPdfAsync(job.Html);
    pdf.SaveAs(job.OutputPath);
}

Parallel Rendering

using IronPdf;
using System.Linq;
using System.Threading.Tasks;

// Parallel rendering: https://ironpdf.com/examples/parallel/
var htmlJobs = new[] { "<h1>Doc 1</h1>", "<h1>Doc 2</h1>", "<h1>Doc 3</h1>" };

var tasks = htmlJobs.Select(async html =>
{
    var r = new ChromePdfRenderer(); // verify per-task vs shared instance guidance
    return await r.RenderHtmlAsPdfAsync(html);
});

var results = await Task.WhenAll(tasks);
Console.WriteLine($"Rendered {results.Length} PDFs in parallel");

PdfDocument implements IDisposable. Use using blocks for proper cleanup:

using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("output.pdf");
// Disposed at end of block

One area worth benchmarking carefully: if your PDFmyURL integration was handling very large pages or complex CSS, measure local render times against your previous API response times. In network-constrained environments PDFmyURL calls may have been the bottleneck; in others, a cold Chromium startup may be. The tradeoff is different per workload.

Frequently Asked Questions

Do I need an internet connection for IronPDF to render PDFs?

No. IronPDF embeds its own Chromium rendering engine and runs entirely locally. This is one of the primary reasons teams migrate away from PDFmyURL — every PDFmyURL render requires an outbound HTTPS call to the service. With IronPDF, air-gapped and on-premise deployments work without any network dependency.

Can I merge and watermark PDFs without adding extra libraries?

Yes. PDFmyURL does not support merge, watermark, or password protection natively — teams typically bolt on a separate local library like PdfSharp or iTextSharp for those features. IronPDF includes all of these capabilities in a single package: PdfDocument.Merge() for merging, TextStamper / ImageStamper for watermarks, and SecuritySettings for password protection. See the merge/split guide for details.

How do I handle errors differently after migrating?

With PDFmyURL via the Pdfcrowd SDK, you catch Pdfcrowd.Error for service-side failures. IronPDF throws typed .NET exceptions instead. Refactor that error handling into standard try/catch blocks using IronPdf.Exceptions.IronPdfException and its more specific subclasses (e.g. IronPdfRenderingException, IronPdfLicenseException). This is where most of the code rewrite effort lives.

What .NET versions does IronPDF support?

IronPDF supports .NET 6, 7, 8, and 9. Verify compatibility with your target framework before starting the migration. You can get a trial license key to test against your specific environment.

How Should I Decide Whether to Migrate?

If your deployment is air-gapped or on-premise, the decision is already made — PDFmyURL cannot operate without outbound network access. If your rendered HTML contains PII or confidential data, that content leaves your network boundary on every PDFmyURL call, which may conflict with compliance requirements.

If you are in a low-volume, internet-connected, URL-only scenario, staying on PDFmyURL is reasonable. Evaluate based on latency and privacy needs rather than feature gaps.

For high-volume scenarios with complex PDF manipulation needs (merge, watermark, security), the switch pays for itself quickly because PDFmyURL lacks those features natively and you are likely already maintaining a second library to fill the gaps.

The main structural change in this migration is the shift from remote-call error handling to local exception handling. The render call itself is simpler on the IronPDF side. Start with the feature breakdown above, scan your codebase for all PDFmyURL references, and estimate scope before committing.
