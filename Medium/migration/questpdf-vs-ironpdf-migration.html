Migrating from QuestPDF to IronPDF: Solving Common Migration Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Would You Migrate from QuestPDF to IronPDF?

It works on your machine. It generates the PDF exactly right. Then you push it through the CI pipeline — a Docker container running a minimal base image — and the font rendering breaks, or the runner exits with a native library error. QuestPDF has made real progress on cross-platform support, but teams on minimal Linux images occasionally still hit SkiaSharp or font provider issues that don't reproduce locally. That gap between local and pipeline behavior is the pain that typically triggers an evaluation.

I've walked through this migration with enough teams to know the patterns. Here are the conditions that most commonly push teams toward an alternative:

SkiaSharp native library issues in containers — libfontconfig, libfreetype, or libskia not present in minimal Docker base images like alpine or distroless. Font provider configuration — QuestPDF requires a font provider setup, and on CI runners without system fonts, this needs explicit configuration. No HTML rendering path — web developers maintaining templates in HTML can't directly use QuestPDF without rewriting everything to the fluent DSL. Limited PDF manipulation — QuestPDF is a generation library; merge, split, watermark, security, and text extraction all need additional libraries.

If any of those sound familiar, this article covers the complete migration path with before/after code for every standard operation.

Problem 1: SkiaSharp Native Dependencies Break Your CI Pipeline

The most common trigger I see is SkiaSharp failing in containers. QuestPDF depends on SkiaSharp for rendering, which requires native system libraries that minimal Docker images simply don't include. You end up adding libfontconfig1, libfreetype6, and libgdiplus to your Dockerfile just to get PDF generation working.

IronPDF uses an embedded Chromium renderer instead of SkiaSharp, so those native dependencies disappear entirely. Here is how the cleanup looks.

Removing QuestPDF and Installing IronPDF

# Remove QuestPDF (and SkiaSharp if only used by QuestPDF)
dotnet remove package QuestPDF

# Check if SkiaSharp can be removed too
dotnet remove package SkiaSharp  # only if no other dependency requires it

# Install IronPDF
dotnet add package IronPdf

dotnet restore

Cleaning Up Your Dockerfile

# Remove SkiaSharp system dependencies if they were only for QuestPDF:
# RUN apt-get install -y libfontconfig1 libfreetype6 libgdiplus

# IronPDF: no SkiaSharp; verify IronPDF's own system requirements in their Docker docs

After removing QuestPDF, verify that SkiaSharp packages can actually be removed:

dotnet list package | grep -i "skia"
# If SkiaSharp appears only as a QuestPDF transitive dependency, it can be removed

Problem 2: The Fluent DSL Locks You into Code-Only Document Changes

QuestPDF's fluent DSL is powerful, but every layout change requires a code change. Your designers can't touch the templates. Your frontend developers who already know HTML and CSS have to learn an entirely new document model with containers, columns, and custom components.

With IronPDF's HTML-to-PDF approach, your existing HTML templates become your PDF templates. Non-developers can modify them. Here is the fundamental shift.

QuestPDF Fluent DSL (Before)

using QuestPDF.Fluent;
using QuestPDF.Helpers;
using QuestPDF.Infrastructure;
using System;

class InvoiceBefore
{
    static void Main()
    {
        QuestPDF.Settings.License = LicenseType.Community;

        Document.Create(container =>
        {
            container.Page(page =>
            {
                page.Size(PageSizes.A4);
                page.Margin(2, Unit.Centimetre);

                page.Header()
                    .Text("Invoice #5001")
                    .SemiBold()
                    .FontSize(24);

                page.Content()
                    .Column(col =>
                    {
                        col.Item().Text("Client: Acme Corp").FontSize(12);
                        col.Item().PaddingTop(10).Text("Amount Due: $3,200").FontSize(12);
                    });

                page.Footer()
                    .AlignCenter()
                    .Text(x => x.CurrentPageNumber());
            });
        }).GeneratePdf("invoice.pdf");

        Console.WriteLine("Saved invoice.pdf");
    }
}

IronPDF HTML-Driven (After)

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var html = @"
    <html>
    <head>
    <style>
        body { font-family: Arial, sans-serif; padding: 2cm; }
        h1 { font-size: 24px; font-weight: 600; }
        p { font-size: 12px; margin: 4px 0; }
    </style>
    </head>
    <body>
        <h1>Invoice #5001</h1>
        <p>Client: Acme Corp</p>
        <p>Amount Due: $3,200</p>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

// Page numbers via footer
renderer.RenderingOptions.HtmlFooter = new HtmlHeaderFooter
{
    HtmlFragment = "<div style='text-align:center; font-size:10px'>{page}</div>",
};

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");

Console.WriteLine($"Saved invoice.pdf ({pdf.PageCount} page(s))");

The namespace swap is straightforward. Replace QuestPDF.Fluent, QuestPDF.Helpers, and QuestPDF.Infrastructure with IronPdf and IronPdf.Rendering. Remove the QuestPDF.Settings.License = LicenseType.Community call and add your IronPDF license key instead.

Problem 3: Merge, Watermark, and Security Require Secondary Libraries

QuestPDF is a generation library. If you need to merge documents, apply watermarks, or add password protection, you need a second (or third) library. With IronPDF, those operations are built in.

Merging PDFs — QuestPDF (Before)

using QuestPDF.Fluent;
using QuestPDF.Helpers;
using QuestPDF.Infrastructure;
using System;
using System.IO;

class MergeBefore
{
    static void Main()
    {
        QuestPDF.Settings.License = LicenseType.Community;

        // QuestPDF doesn't support merging separate documents.
        // Teams typically generate each section as a separate file
        // then use a secondary library to combine them.

        var sections = new[]
        {
            ("Section 1: Overview", "sec1.pdf"),
            ("Section 2: Detail", "sec2.pdf"),
        };

        foreach (var (title, filename) in sections)
        {
            Document.Create(container =>
            {
                container.Page(page =>
                {
                    page.Size(PageSizes.A4);
                    page.Content().Text(title).FontSize(20);
                });
            }).GeneratePdf(filename);
        }

        // Now merge sec1.pdf + sec2.pdf using secondary library (e.g., PDFsharp)
        Console.WriteLine("Merge requires secondary library — not built into QuestPDF");
    }
}

Merging PDFs — IronPDF (After)

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

// Render concurrently
var results = await Task.WhenAll(
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 1: Overview</h1></body></html>"),
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 2: Detail</h1></body></html>")
);

var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("combined.pdf");

Console.WriteLine($"Merged: {merged.PageCount} pages");

Watermark — QuestPDF (Before)

using QuestPDF.Fluent;
using QuestPDF.Helpers;
using QuestPDF.Infrastructure;
using QuestPDF.Drawing;
using SkiaSharp;
using System;

class WatermarkBefore
{
    static void Main()
    {
        QuestPDF.Settings.License = LicenseType.Community;

        // QuestPDF watermark via custom Canvas element — requires SkiaSharp directly
        Document.Create(container =>
        {
            container.Page(page =>
            {
                page.Size(PageSizes.A4);

                // Foreground layer for watermark
                page.Foreground().Canvas((canvas, size) =>
                {
                    using var paint = new SKPaint
                    {
                        Color = SKColors.Gray.WithAlpha(50),
                        TextSize = 72,
                        IsAntialias = true,
                    };
                    // Manual rotation and positioning required
                    canvas.Save();
                    canvas.Translate(size.Width / 2, size.Height / 2);
                    canvas.RotateDegrees(-45);
                    canvas.DrawText("DRAFT", 0, 0, paint);
                    canvas.Restore();
                });

                page.Content().Text("Report Content").FontSize(14);
            });
        }).GeneratePdf("watermarked.pdf");

        Console.WriteLine("SkiaSharp watermark applied");
    }
}

Watermark — IronPDF (After)

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Report Content</h1></body></html>"
);

// No SkiaSharp required — programmatic text stamp
var watermark = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronPdf.Imaging.Color.Gray,
    Opacity = 0.2,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("watermarked.pdf");

Password Protection — QuestPDF (Before)

using QuestPDF.Fluent;
using QuestPDF.Helpers;
using QuestPDF.Infrastructure;
using System;
using System.IO;

class PasswordBefore
{
    static void Main()
    {
        QuestPDF.Settings.License = LicenseType.Community;

        // Step 1: Generate PDF with QuestPDF
        var pdfBytes = Document.Create(container =>
        {
            container.Page(page =>
            {
                page.Size(PageSizes.A4);
                page.Content().Text("Confidential Document").FontSize(18);
            });
        }).GeneratePdf();

        // Step 2: Apply password with a secondary library (required)
        // QuestPDF has no security/password API
        // Pseudo-code:
        // var secured = SomePdfLib.SetPassword(pdfBytes, "open123", "admin456");
        // File.WriteAllBytes("secured.pdf", secured);

        Console.WriteLine("Password protection requires a secondary library with QuestPDF");
    }
}

Password Protection — IronPDF (After)

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Confidential Document</h1></body></html>"
);

pdf.SecuritySettings.UserPassword = "open123";
pdf.SecuritySettings.OwnerPassword = "admin456";

pdf.SaveAs("secured.pdf");

Frequently Asked Questions

How long does a typical QuestPDF to IronPDF migration take?

Answer:

The document rewrite from QuestPDF's fluent DSL to HTML is the main cost, and it scales linearly with the number of distinct document types in your codebase. Simple single-column documents — text, basic tables, logos — take roughly 30 minutes each to rewrite as HTML.

Before committing to a timeline, run rg "Document.Create" --type cs | wc -l to count how many document definitions exist. The namespace swap and package installation themselves take minutes. Complex multi-column layouts with precise spacing will take longer per document.

Do I need to rewrite every QuestPDF document from scratch?

Answer:

Yes. The fundamental migration challenge is that QuestPDF's fluent DSL doesn't map 1:1 to HTML. There is no automated converter. Every Document.Create() block needs to be rewritten as HTML/CSS and rendered through IronPDF's ChromePdfRenderer.

The upside is that once documents are in HTML, non-developers can modify templates without code changes, and your frontend team can reuse existing HTML/CSS skills directly.

How do I handle page numbers after migrating?

Answer:

QuestPDF's page numbering uses x.CurrentPageNumber() in the document model. IronPDF uses placeholder tokens in HtmlHeaderFooter. Here is the equivalent:

renderer.RenderingOptions.HtmlFooter = new HtmlHeaderFooter
{
    HtmlFragment = @"
        <div style='text-align:center; font-size:9px; color:#666'>
            Page {page} of {total-pages}
        </div>",
};

Verify the exact token names ({page}, {total-pages}) in the current IronPDF documentation.

Will IronPDF perform the same as QuestPDF for my documents?

Answer:

The performance profiles differ. QuestPDF renders via SkiaSharp — CPU-bound vector graphics rasterization with no HTML parse or layout pass. IronPDF renders via Chromium HTML parse, layout, and render, including JavaScript execution if present.

For simple text documents, QuestPDF may be faster. For complex HTML-derived content, IronPDF avoids the DSL rewrite overhead. You should measure for your specific templates using the benchmark patterns in this article, and run them in your actual environment rather than relying on synthetic benchmarks.

Can I remove SkiaSharp from my project after migrating?

Answer:

If SkiaSharp was only used as a transitive dependency of QuestPDF, yes. Run dotnet list package | grep -i "skia" to confirm. Also remove any SkiaSharp system library installs from your Dockerfiles — lines like RUN apt-get install -y libfontconfig1 libfreetype6 libgdiplus that were only there for QuestPDF.

If another part of your application uses SkiaSharp directly (image processing, for example), keep it. But in most cases I see, SkiaSharp is only present because of QuestPDF, and removing it simplifies your container images significantly.

When Should You Migrate from QuestPDF to IronPDF?

Business ScenarioRecommendation
Documents already designed as HTML/CSS templatesSwitch — direct HTML input eliminates DSL rewrite
Minimal Docker images (alpine, distroless) with SkiaSharp issuesSwitch — IronPDF avoids SkiaSharp native dependency
Complex programmatic layouts where DSL is working wellQuestPDF may be appropriate — evaluate migration cost
Documents need HTML-to-PDF from existing web contentSwitch — IronPDF accepts live HTML/URLs directly

The operational wins — removing SkiaSharp native library dependencies, eliminating font provider configuration, cleaning up Docker image system libraries — are immediate and verifiable before any code is written. If your CI pipeline has ever broken because of a missing native library in a container, that alone may justify the switch.

Start by counting your document definitions and categorizing them by complexity. The merge and security features that previously required secondary libraries come built in, so you may end up with fewer dependencies overall.
