Migrating from Pdfium to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should I Migrate from Pdfium to IronPDF?

The GitHub repository you depend on has 847 open issues and the last commit touching actual rendering logic is from 2021. The .NET wrapper around Pdfium has been in "works well enough" territory for years, but you are about to add HTML-to-PDF generation to a service that currently only renders existing PDFs to images, and the wrapper's issue backlog has six unresolved reports about HTML rendering behaviour. That is when it is time to assess alternatives rather than discover those issues in production.

I have seen nine reasons teams make this transition:

1. HTML-to-PDF gap — Pdfium renders PDFs, it does not convert HTML to PDF. Teams that need both add a second tool. Migration consolidates this.

2. Wrapper maintenance signal — community .NET wrappers for Pdfium vary in quality. Slow issue resolution and infrequent updates are common.

3. Native binary management — architecture-specific pdfium.dll / libpdfium.so must be sourced, bundled, and updated separately from your .NET packages.

4. P/Invoke fragility — native interop breaks across .NET runtime upgrades, OS updates, and architecture changes in non-obvious ways.

5. AGPL-adjacent licensing — some Pdfium builds have licensing considerations; verify the specific build you are using.

6. ARM64 deployment — ARM64 Pdfium binaries must be sourced separately.

7. Feature gaps — watermark, encrypt, merge operations may require additional code beyond your Pdfium wrapper.

8. Thread safety — Pdfium has documented thread-safety constraints that vary by operation; verify for your wrapper.

9. Docker complexity — native binary distribution in multi-arch Docker builds adds maintenance overhead.

How Do Pdfium and IronPDF Compare Feature by Feature?

  • Focus. Pdfium is a PDF rendering and parsing engine. IronPDF is built around HTML-to-PDF generation and PDF manipulation.
  • Pricing. Pdfium itself is open source (BSD-style) and free; commercial wrappers vary. IronPDF is commercial — verify current pricing at ironsoftware.com.
  • API style. Pdfium reaches .NET through P/Invoke wrappers such as PdfiumViewer or PdfiumSharp. IronPDF is a managed in-process .NET library.
  • Learning curve. High with Pdfium because it is wrapper-dependent. Medium with IronPDF.
  • HTML rendering. Pdfium does not support it natively. IronPDF uses an embedded Chromium engine.
  • Page indexing. Both libraries are 0-based at the C level; some wrappers shift this, so verify yours.
  • Thread safety. Pdfium has documented constraints. IronPDF works well with one renderer instance reused per thread.
  • Namespace. Wrapper-specific for Pdfium (e.g., PdfiumViewer); IronPDF uses IronPdf.

How Much Effort Does Each Feature Take to Migrate?

Not every feature migrates at the same cost. Here is my honest assessment of the effort involved for each capability you might be using today.

  • Render PDF page to image — High. Pdfium's native strength. IronPDF can rasterize via RasterizeToImageFiles, but if image rendering is your primary use case, expect more careful comparison work.
  • Text extraction — Medium. Native to Pdfium via GetPdfText. IronPDF offers ExtractAllText() and per-page page.Text — verify output against your corpus.
  • HTML to PDF — Low. Not native in Pdfium. Native and central in IronPDF.
  • Open/parse existing PDF — Low. Pdfium native; PdfDocument.FromFile in IronPDF.
  • Merge PDFs — Low. Wrapper-dependent in Pdfium (PdfiumViewer does not support it). Native in IronPDF.
  • Watermark — Low. Wrapper-dependent in Pdfium. Native in IronPDF via ApplyStamp.
  • Password protection — Low. Pdfium supports it at the C level; wrapper exposure varies. Native in IronPDF.
  • Page count / metadata — Low. Available via the wrapper (document.PageCount) and IronPDF (pdf.PageCount).
  • Native binary management — Eliminated. Required with Pdfium, gone after switching to IronPDF NuGet.
  • ARM64 deployment — Simplified. Manual binary sourcing under Pdfium becomes a NuGet install.
  • Thread-safe concurrent use — Constrained vs explicit pattern. Pdfium has documented constraints; IronPDF works well with one renderer per task.

And here is a decision matrix to help you decide whether this migration is right for your specific scenario.

  • If your primary use is rendering PDF pages to images, Pdfium remains well-suited; IronPDF can do it but is optimised differently.
  • If you need HTML-to-PDF alongside PDF rendering, IronPDF covers the HTML-to-PDF half cleanly, while image rendering is a different optimisation target.
  • If native binary management is your main pain, IronPDF via NuGet removes that whole class of problem.
  • If you have an open-source constraint, Pdfium (FOSS) wins; IronPDF is commercial.

How Do I Set Up My Project for the Migration?

Before touching any code, you need to find every Pdfium reference in your codebase. Here is how I recommend doing that.

Find Pdfium References

# Find the wrapper package -- check .csproj first
grep -r -i "pdfium\|fpdf" **/*.csproj *.csproj 2>/dev/null

# Find all usage in C# files
rg -l "Pdfium\|PdfiumViewer\|FpdfBitmap\|FPDF" --type cs -i

# Find P/Invoke declarations if using direct interop
rg "DllImport.*pdfium\|extern.*fpdf" --type cs -n -i

# Find native binary references
find . -name "*.dll" | grep -i pdfium
find . -name "*.so"  | grep -i pdfium

Remove Pdfium and Install IronPDF

# Remove wrapper -- verify exact package name
dotnet remove package PdfiumViewer  # or your specific wrapper (PdfiumSharp, PDFiumCore, PDFium.Net.SDK)

# Install IronPDF
dotnet add package IronPdf
dotnet restore

How Do Pdfium APIs Map to IronPDF APIs?

Here are the complete mapping tables I put together covering namespaces, core classes, document loading, page operations, and merge/split.

Namespace Mapping

  • Wrapper-specific (PdfiumViewer, PdfiumSharp, PDFiumCore) → IronPdf — core namespace.
  • Wrapper-specific render flags → IronPdf.Rendering — render options and paper size.
  • No Pdfium equivalent → IronPdf.Editing — stamp, annotate, watermark.

Core Class Mapping

  • Wrapper PdfDocument (e.g., PdfiumViewer.PdfDocument) → IronPdf.PdfDocument — the PDF document.
  • Wrapper page accessor (document.Pages[n]) → pdf.Pages[n] — per-page access, 0-based.
  • No Pdfium equivalent → ChromePdfRenderer — HTML-to-PDF renderer.
  • Wrapper bitmap/image classes (System.Drawing.Image via document.Render) → no direct bitmap class in IronPDF; rasterise via pdf.RasterizeToImageFiles().

Document Loading Methods

  • Open existing PDF. PdfDocument.Load(path) → PdfDocument.FromFile(path).
  • Open from a stream. PdfDocument.Load(stream) → PdfDocument.FromStream(stream).
  • Open from bytes. Wrapper-specific bytes overload → PdfDocument.FromBinaryData(bytes).
  • HTML to PDF. Not native in Pdfium → renderer.RenderHtmlAsPdf(html).
  • URL to PDF. Not native in Pdfium → renderer.RenderUrlAsPdf(url).

Page Operations

  • Page count. document.PageCount → pdf.PageCount.
  • Access a page. document.Pages[n] → pdf.Pages[n] (0-based).
  • Extract text. document.GetPdfText(i) per page → pdf.ExtractAllText() for the whole document or pdf.Pages[i].Text per page.
  • Render page to image. document.Render(i, w, h, dpiX, dpiY, forPrinting) → pdf.RasterizeToImageFiles(pattern, DPI: n); this is Pdfium's strength rather than IronPDF's primary focus.
  • Remove a page. Wrapper-specific → pdf.RemovePages(i).

Merge/Split Operations

  • Merge. Wrapper-dependent in Pdfium (PdfiumViewer does not expose merge) → PdfDocument.Merge(pdf1, pdf2) or PdfDocument.Merge(IEnumerable<string> paths).
  • Split. Wrapper-dependent in Pdfium → pdf.CopyPages(startIndex, endIndex).

How Do I Convert HTML to PDF After Migrating?

This is the most common reason teams migrate. Pdfium cannot convert HTML to PDF, so teams typically add a secondary tool like wkhtmltopdf. IronPDF eliminates that extra dependency entirely.

HTML to PDF — Before (Pdfium with Secondary Tool)

using System;
using System.Diagnostics;
using System.IO;

class HtmlToPdfWithSecondTool
{
    static void Main()
    {
        // Pdfium cannot convert HTML to PDF -- teams add a second tool
        // wkhtmltopdf subprocess is a common addition alongside Pdfium

        string html     = "<html><body><h1>Invoice #1234</h1></body></html>";
        string tempHtml = Path.GetTempFileName() + ".html";
        string tempPdf  = Path.GetTempFileName() + ".pdf";

        File.WriteAllText(tempHtml, html);

        var psi = new ProcessStartInfo("wkhtmltopdf", $"\"{tempHtml}\" \"{tempPdf}\"")
        {
            RedirectStandardError = true,
            UseShellExecute = false
        };
        using var proc = Process.Start(psi)!;
        proc.WaitForExit(30_000);

        if (proc.ExitCode != 0)
            throw new Exception($"wkhtmltopdf failed: {proc.StandardError.ReadToEnd()}");

        byte[] pdfBytes = File.ReadAllBytes(tempPdf);
        File.Delete(tempHtml);
        File.Delete(tempPdf);
        // Use pdfBytes...
        Console.WriteLine("Generated PDF via secondary tool");
    }
}

HTML to PDF — After (IronPDF)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>"
);
pdf.SaveAs("invoice.pdf");
// Guide: https://ironpdf.com/how-to/html-string-to-pdf/

How Do I Migrate PDF Merge Operations?

Merge PDFs — Before (Pdfium / PdfSharp Fallback)

// Pdfium can merge PDFs at the C library level
// Wrapper support varies -- verify your specific library

// If not supported, PdfSharp pattern:
using PdfSharp.Pdf;
using PdfSharp.Pdf.IO;
using System;

class MergeExample
{
    static void Main()
    {
        using var output = new PdfDocument();
        foreach (string path in new[] { "part1.pdf", "part2.pdf" })
        {
            using var input = PdfReader.Open(path, PdfDocumentOpenMode.Import);
            foreach (PdfPage page in input.Pages)
                output.AddPage(page);
        }
        output.Save("merged.pdf");
        Console.WriteLine("Merged.");
    }
}

merged.SaveAs("merged.pdf");

How Do I Add Watermarks After Migrating?

Watermark — Before (Pdfium with iTextSharp)

using iTextSharp.text;
using iTextSharp.text.pdf;
using System.IO;
using System;

class WatermarkExample
{
    static void Main()
    {
        // Pdfium doesn't expose high-level watermark API
        // Secondary library (iTextSharp) is common:
        using var reader  = new PdfReader("input.pdf");
        using var fs      = new FileStream("watermarked.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs);

        var font = BaseFont.CreateFont(BaseFont.HELVETICA_BOLD, BaseFont.CP1252, false);
        for (int page = 1; page <= reader.NumberOfPages; page++)
        {
            var cb = stamper.GetOverContent(page);
            cb.BeginText();
            cb.SetFontAndSize(font, 60);
            cb.SetColorFill(new BaseColor(180, 180, 180));
            cb.ShowTextAligned(Element.ALIGN_CENTER, "CONFIDENTIAL", 300, 420, 45);
            cb.EndText();
        }
        Console.WriteLine("Watermarked.");
    }
}

Watermark — After (IronPDF)

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

    FontSize = 60,
    Opacity = 30,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");

How Do I Migrate Password Protection?

Password Protection — Before (Pdfium with iTextSharp)

using iTextSharp.text.pdf;
using System.IO;
using System.Text;

class SecurityExample
{
    static void Main()
    {
        // Verify your Pdfium wrapper's encryption support
        // If not available -- secondary library (iTextSharp) pattern:
        byte[] userPass  = Encoding.ASCII.GetBytes("readonly");
        byte[] ownerPass = Encoding.ASCII.GetBytes("admin");

        using var reader  = new PdfReader("input.pdf");
        using var fs      = new FileStream("secured.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs, '\0', false);
        stamper.SetEncryption(
            userPass, ownerPass,
            PdfWriter.ALLOW_PRINTING,
            PdfWriter.ENCRYPTION_AES_128
        );
    }
}

pdf.SaveAs("secured.pdf");

What Are the Critical Migration Notes I Should Know?

Pdfium's primary strength — rendering PDF pages to bitmap images — is not IronPDF's primary feature. If your codebase relies heavily on this, consider a partial migration: keep Pdfium for image rendering while moving HTML-to-PDF and manipulation to IronPDF.

Both Pdfium (at the C level) and IronPDF use 0-based page indexing. Wrapper APIs may vary, so verify your wrapper's convention. In IronPDF, page access is straightforward:

// IronPDF: 0-based throughout
var firstPage = pdf.Pages[0];
var lastPage  = pdf.Pages[pdf.PageCount - 1];

If text extraction is a key feature, compare IronPDF's output against Pdfium's on your actual document corpus before committing:

var pdf = PdfDocument.FromFile("document.pdf");

// Full document text
string allText = pdf.ExtractAllText();

// Per-page
foreach (var page in pdf.Pages)
    Console.WriteLine($"Page {page.PageIndex}: {page.Text.Length} chars");

// Guide: https://ironpdf.com/how-to/extract-text-and-images/

How Should I Handle Performance and Benchmarking?

The numbers that matter are yours, not anyone else's. I always recommend running the benchmark scaffold below against your actual HTML templates and PDF workloads before making a capacity decision.

Benchmark Scaffold

using System;
using System.Collections.Generic;
using System.Diagnostics;
using System.Threading.Tasks;
using IronPdf;

class PdfBenchmark
{
    // Replace html with your actual production template content
    static readonly string TestHtml = System.IO.File.Exists("test_template.html")
        ? System.IO.File.ReadAllText("test_template.html")
        : "<html><body><h1>Test Document</h1><p>Paragraph content for benchmark.</p></body></html>";

    static void Main(string[] args)
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        const int warmupRounds = 5;
        const int timedRounds  = 50;

        var renderer = new ChromePdfRenderer();

        // Warm up -- don't time cold-start
        Console.WriteLine("Warming up...");
        for (int i = 0; i < warmupRounds; i++)
        {
            using var warmup = renderer.RenderHtmlAsPdf(TestHtml);
        }

        // Timed single-threaded renders
        var times = new List<long>();
        var sw    = new Stopwatch();

        Console.WriteLine($"Running {timedRounds} renders...");
        for (int i = 0; i < timedRounds; i++)
        {
            sw.Restart();
            using var pdf = renderer.RenderHtmlAsPdf(TestHtml);
            sw.Stop();
            times.Add(sw.ElapsedMilliseconds);
        }

        times.Sort();
        Console.WriteLine($"\n--- Single-threaded results ({timedRounds} renders) ---");
        Console.WriteLine($"Median:  {times[timedRounds / 2]}ms");
        Console.WriteLine($"P95:     {times[(int)(timedRounds * 0.95)]}ms");
        Console.WriteLine($"P99:     {times[(int)(timedRounds * 0.99)]}ms");
        Console.WriteLine($"Min:     {times[0]}ms");
        Console.WriteLine($"Max:     {times[timedRounds - 1]}ms");
        Console.WriteLine($"Average: {Average(times):F1}ms");
    }

    static double Average(List<long> list)
    {
        long sum = 0;
        foreach (var v in list) sum += v;
        return (double)sum / list.Count;
    }
}

What Metrics Should I Collect?

  • Median render time — typical-case latency. Stopwatch over 50+ iterations.
  • P95 render time — tail latency for SLAs. Sort the timings and index the 95th percentile.
  • Memory per render — container sizing. GC.GetTotalMemory before and after each render.
  • Concurrent throughput — scaling capacity. Task.WhenAll with a SemaphoreSlim to bound concurrency.
  • Output file size — storage cost. new FileInfo(path).Length.
  • Cold start time — serverless or first-request latency. Time the first render separately from steady state.
  • CSS fidelity — visual accuracy. Side-by-side review against Pdfium-rendered images of the same source.

Frequently Asked Questions

Can I keep Pdfium for PDF-to-image rendering alongside IronPDF?

Yes. Pdfium's primary strength is rendering PDF pages to bitmap images, and that is not IronPDF's primary feature. Many teams run a partial migration, keeping Pdfium for image rendering while using IronPDF for HTML-to-PDF generation and PDF manipulation.

If you want to evaluate alternatives to Pdfium for the image rendering portion, consider docnet (an open source Pdfium wrapper) or commercial PDF-to-image libraries.

Do I need to manage native binaries with IronPDF?

No. One of the biggest pain points with Pdfium is managing architecture-specific native binaries — pdfium.dll on Windows, libpdfium.so on Linux — that must be sourced, bundled, and updated separately. IronPDF ships as a NuGet package, so native binary management is eliminated.

After migration, clean up residual Pdfium binaries and remove any COPY/ADD references to them in your Dockerfiles.

What .NET versions does IronPDF support?

IronPDF targets .NET 6+. Before starting the migration, verify your project's target framework is compatible. Check the IronPDF documentation for the latest supported framework list.

Is IronPDF thread-safe for concurrent rendering?

IronPDF uses an explicit per-thread pattern. For concurrent rendering, create one ChromePdfRenderer per task and use a SemaphoreSlim to bound concurrency. The concurrent benchmark scaffold in the performance section above demonstrates this pattern.

Pdfium, by contrast, has documented thread-safety constraints that vary by operation and wrapper implementation.

How does page indexing compare between Pdfium and IronPDF?

Both Pdfium (at the C level) and IronPDF use 0-based page indexing. However, Pdfium wrapper APIs may vary in their convention, so verify your specific wrapper. In IronPDF, access pages with pdf.Pages[0] for the first page and pdf.Pages[pdf.PageCount - 1] for the last.

What if text extraction results differ between Pdfium and IronPDF?

If text extraction is a key feature in your application, I strongly recommend comparing IronPDF's output against Pdfium's on your actual document corpus before committing to the migration. Run both libraries against a representative set of PDFs and compare the extracted text for completeness and accuracy.

IronPDF provides both full-document extraction via pdf.ExtractAllText() and per-page extraction through the page.Text property. See the text extraction guide for details.

Conclusion

Pdfium's strength is PDF-to-image rendering. If that is your primary use case, this migration may be a partial one — keeping Pdfium for image rendering while moving HTML-to-PDF to IronPDF. The benchmark scaffold helps you understand whether the consolidated approach fits your performance requirements before you commit.

For teams that need HTML-to-PDF generation, PDF merging, watermarking, and password protection, IronPDF consolidates what previously required Pdfium plus one or two secondary libraries into a single NuGet package. The migration itself is straightforward for most operations, with the notable exception of PDF-to-image rendering where Pdfium remains strong.

I recommend starting with the three-step quick start — license configuration, namespace imports, and a basic PDF operation — then working through the API mapping tables to convert each feature systematically. Run the benchmark scaffold with your actual production templates before and after to validate performance in your environment.
