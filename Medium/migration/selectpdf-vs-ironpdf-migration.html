Migrating from SelectPdf to IronPDF: Solving Common Migration Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate Away from SelectPdf?

Page manipulation is the feature that breaks the evaluation. SelectPdf can generate PDFs from HTML, but when teams need to merge documents from multiple sources, extract text for indexing, add watermarks post-generation, or apply password encryption, they reach for another library. That second library adds a dependency, increases complexity, and sometimes conflicts.

I have watched teams accumulate PDFSharp, iTextSharp, or similar libraries alongside SelectPdf just to fill feature gaps. The missing-feature list is what typically puts SelectPdf on the replacement shortlist.

Here is what teams evaluating SelectPdf alternatives commonly encounter:

  • Manipulation features (digital signatures, full PDF/A compliance, form filling) are limited or absent, and watermarks and text extraction support is partial.
  • Community Edition limits: a hard cap of 5 pages per PDF, plus a watermark applied to every page until a license key is installed.
  • CSS rendering fidelity: the default engine is an internal WebKit that struggles with modern CSS (grid, flexbox gap, custom properties). SelectPdf does ship an optional Blink engine (Chromium 124.0.6367.201, April 2024), but it is opt-in.
  • Linux/Docker compatibility: SelectPdf's own documentation states it "currently require[s] Windows; [does] not run on Linux, macOS or Xamarin," which blocks Linux containers, Azure Functions, and AWS Lambda.
  • .NET version support: SelectPdf covers .NET Framework 4.6.1 through .NET 10 (the latter via .NET Standard 2.0), but only on Windows.
  • Synchronous API: SelectPdf's conversion methods are synchronous, making integration with async .NET code awkward.
  • Secondary library accumulation: teams often add PDFSharp, iTextSharp, or similar alongside SelectPdf for missing features, increasing complexity.

What Does the SelectPdf to IronPDF Migration Look Like at a High Level?

The good news is that SelectPdf and IronPDF share a conceptually similar API style. The converter-plus-options-to-document-to-save pattern is the same in both libraries. For simple HTML-to-PDF use cases, the migration is mostly find-and-replace.

The main divergences appear in three areas: the class and method names differ, IronPDF offers async methods alongside the synchronous SelectPdf-style calls, and IronPDF bundles manipulation features (merge, watermark, security) that SelectPdf may require secondary libraries to handle.

  • Focus. SelectPdf is HTML-to-PDF generation; IronPDF is HTML-to-PDF plus PDF manipulation in one library.
  • Pricing. SelectPdf offers a Community Edition (free, 5-page cap with per-page watermark) and perpetual paid tiers from $499 to $1,599 with one year of maintenance included; IronPDF is a commercial license with a generous trial.
  • API style. SelectPdf centers on an HtmlToPdf converter object; IronPDF uses ChromePdfRenderer plus PdfDocument.
  • Learning curve. Low in both; the patterns map cleanly for .NET developers.
  • HTML rendering. SelectPdf's default is an internal WebKit engine; an optional Blink engine ships with Chromium 124. IronPDF embeds an up-to-date Chromium build by default.
  • Page indexing. Both libraries are 0-based, so existing index math carries over.
  • Thread safety. Verify SelectPdf's guidance for your edition and IronPDF's concurrent-instance guidance before sharing instances across threads.
  • Namespace. SelectPdf becomes IronPdf.

Problem 1: How Do I Replace Basic HTML-to-PDF Conversion?

This is the most common migration task. SelectPdf uses HtmlToPdf and ConvertHtmlString(). IronPDF uses ChromePdfRenderer and RenderHtmlAsPdf(). The options mapping is nearly one-to-one.

Start by swapping your package references.

Uninstall SelectPdf and Install IronPDF

dotnet remove package SelectPdf
dotnet add package IronPdf
dotnet restore

Then set up your license key at application startup.

License Configuration

using IronPdf;

// https://ironpdf.com/how-to/license-keys/
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

Now replace the conversion code. Here is a before-and-after for an order confirmation PDF.

Before (SelectPdf)

using SelectPdf;
using System;

class HtmlToPdfBefore
{
    static void Main()
    {
        var html = @"
            <html>
            <head>
            <style>
                body { font-family: Arial; padding: 30px; }
                .header { font-size: 22px; font-weight: bold; }
                table { width: 100%; border-collapse: collapse; }
                td, th { border: 1px solid #ccc; padding: 6px; }
            </style>
            </head>
            <body>
                <div class='header'>Order Confirmation #ORD-8821</div>
                <table>
                    <tr><th>Product</th><th>Qty</th><th>Price</th></tr>
                    <tr><td>Widget Pro</td><td>2</td><td>$149.00</td></tr>
                </table>
            </body>
            </html>";

        var converter = new HtmlToPdf();
        converter.Options.PdfPageSize = PdfPageSize.A4;
        converter.Options.MarginTop = 30;
        converter.Options.MarginBottom = 30;
        converter.Options.MarginLeft = 25;
        converter.Options.MarginRight = 25;

        var doc = converter.ConvertHtmlString(html);
        doc.Save("order-confirmation.pdf");
        Console.WriteLine($"Saved order-confirmation.pdf ({doc.Pages.Count} pages)");
    }
}

After (IronPDF)

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var html = @"
    <html>
    <head>
    <style>
        body { font-family: Arial; padding: 30px; }
        .header { font-size: 22px; font-weight: bold; }
        table { width: 100%; border-collapse: collapse; }
        td, th { border: 1px solid #ccc; padding: 6px; }
    </style>
    </head>
    <body>
        <div class='header'>Order Confirmation #ORD-8821</div>
        <table>
            <tr><th>Product</th><th>Qty</th><th>Price</th></tr>
            <tr><td>Widget Pro</td><td>2</td><td>$149.00</td></tr>
        </table>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
renderer.RenderingOptions.MarginTop = 30;
renderer.RenderingOptions.MarginBottom = 30;
renderer.RenderingOptions.MarginLeft = 25;
renderer.RenderingOptions.MarginRight = 25;

var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("order-confirmation.pdf");

Console.WriteLine($"Saved order-confirmation.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

The key replacements are: new HtmlToPdf() becomes new ChromePdfRenderer(), converter.Options.* becomes renderer.RenderingOptions.*, ConvertHtmlString() becomes RenderHtmlAsPdf(), and doc.Save() becomes pdf.SaveAs().

Problem 2: How Do I Handle Merging PDFs Without a Secondary Library?

SelectPdf does include merge functionality, but in the Community Edition it inherits the 5-page cap and per-page watermark, which makes it unusable for production output without a license. With IronPDF's merge API, that constraint goes away and the merge call sits in the same library as your renderer.

Before (SelectPdf)

using SelectPdf;
using System;
using System.Collections.Generic;
using System.IO;

class MergeBefore
{
    static void Main()
    {
        var htmlSections = new[]
        {
            "<html><body><h1>Invoice Header</h1></body></html>",
            "<html><body><h1>Invoice Line Items</h1></body></html>",
        };

        var docs = new List<PdfDocument>();
        var converter = new HtmlToPdf();

        foreach (var html in htmlSections)
        {
            docs.Add(converter.ConvertHtmlString(html));
        }

        var merged = docs[0];
        for (int i = 1; i < docs.Count; i++)
        {
            merged.AppendDocument(docs[i]);
        }

        merged.Save("full-invoice.pdf");
        merged.Close();
    }
}

After (IronPDF)

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();

var header = renderer.RenderHtmlAsPdf("<html><body><h1>Invoice Header</h1></body></html>");
var lineItems = renderer.RenderHtmlAsPdf("<html><body><h1>Invoice Line Items</h1></body></html>");

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(header, lineItems);
merged.SaveAs("full-invoice.pdf");

Console.WriteLine($"Merged: {merged.PageCount} pages");

If you do need to overlap render times when generating many documents in a request handler, IronPDF also exposes RenderHtmlAsPdfAsync for use with Task.WhenAll.

Problem 3: How Do I Add Watermarks and Password Protection Without Extra Dependencies?

Watermarking in SelectPdf is manual — you compose it as HTML and stamp it through a template, or you draw it onto every page yourself. PDF security is supported but limited to basic user/owner password fields. IronPDF rolls both into first-class APIs.

Watermark Before (SelectPdf)

using SelectPdf;
using System;

class WatermarkBefore
{
    static void Main()
    {
        var converter = new HtmlToPdf();

        // SelectPdf watermarking is typically done by overlaying HTML
        // on every page via a template / header section.
        var watermarkHtml = "<div style='position:fixed; top:40%; left:30%; " +
                            "font-size:72px; color:#cccccc; transform:rotate(-30deg);'>DRAFT</div>";

        converter.Options.PdfPageCustomSize = new System.Drawing.SizeF(595, 842);
        converter.Header.Add(new PdfHtmlSection(watermarkHtml));

        var doc = converter.ConvertHtmlString("<html><body><h1>Report</h1></body></html>");
        doc.Save("watermarked-report.pdf");
        doc.Close();
    }
}

Watermark After (IronPDF)

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<html><body><h1>Report</h1></body></html>");

// https://ironpdf.com/how-to/custom-watermark/
var watermark = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronSoftware.Drawing.Color.Gray,
    Opacity = 15,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("watermarked-report.pdf");

Password protection follows the same pattern: a few lines in IronPDF, with no separate security namespace to navigate.

Password Protection Before (SelectPdf)

using SelectPdf;
using System;

class PasswordBefore
{
    static void Main()
    {
        var converter = new HtmlToPdf();
        var doc = converter.ConvertHtmlString("<html><body><h1>Secured</h1></body></html>");

        doc.Security.UserPassword = "open123";
        doc.Security.OwnerPassword = "admin456";

        doc.Save("secured.pdf");
        doc.Close();
    }
}

Password Protection After (IronPDF)

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<html><body><h1>Secured</h1></body></html>");

// https://ironpdf.com/how-to/pdf-permissions-passwords/
pdf.SecuritySettings.UserPassword = "open123";
pdf.SecuritySettings.OwnerPassword = "admin456";

pdf.SaveAs("secured.pdf");

What Does the Full Feature Migration Map Look Like?

Here is the complete effort-by-feature breakdown. I find this useful for scoping the migration before writing any code.

  • HTML string to PDF — Low. HtmlToPdf.ConvertHtmlString() becomes ChromePdfRenderer.RenderHtmlAsPdf().
  • URL to PDF — Low. HtmlToPdf.ConvertUrl() becomes ChromePdfRenderer.RenderUrlAsPdf().
  • Save to file — Low. doc.Save(path) becomes pdf.SaveAs(path).
  • Save to stream — Low. doc.Save(stream) becomes pdf.Stream or pdf.BinaryData.
  • Custom page size — Low. HtmlToPdfOptions.PdfPageSize becomes RenderingOptions.PaperSize.
  • Custom margins — Low. HtmlToPdfOptions.MarginTop and siblings become RenderingOptions.Margin*.
  • Headers/footers — Medium. HtmlToPdfOptions.PdfHtmlHeader/Footer becomes RenderingOptions.HtmlHeader/Footer; placeholder syntax also changes ({page_number} → {page}, {total_pages} → {total-pages}).
  • Merge PDFs — Medium. SelectPdf PdfDocument.AppendDocument() becomes IronPDF PdfDocument.Merge().
  • Watermark — Medium. SelectPdf's manual HTML-overlay approach becomes a single TextStamper or ImageStamper call in IronPDF.
  • Password protection — Medium. SelectPdf doc.Security.UserPassword / OwnerPassword becomes pdf.SecuritySettings.UserPassword / OwnerPassword in IronPDF.
  • Text extraction — Medium. SelectPdf has limited extraction; IronPDF uses pdf.ExtractAllText() across the whole document.
  • Digital signatures — Medium-High. SelectPdf does not ship native PAdES signing; IronPDF exposes pdf.Sign() with PFX/PKCS#12 certificates.

And here is the API mapping for the core classes and methods you will touch during migration.

  • Converter instantiation. new HtmlToPdf() → new ChromePdfRenderer().
  • Options property. converter.Options.* → renderer.RenderingOptions.*.
  • Convert HTML string. converter.ConvertHtmlString(html) → renderer.RenderHtmlAsPdf(html).
  • Convert URL. converter.ConvertUrl(url) → renderer.RenderUrlAsPdf(url).
  • Save. doc.Save(path) → pdf.SaveAs(path).

Frequently Asked Questions

Is the SelectPdf to IronPDF migration mostly find-and-replace?

For basic HTML-to-PDF generation, yes. The API styles are conceptually similar: you create a converter, set options, call a convert method, and save the result. The class names differ (HtmlToPdf vs ChromePdfRenderer, ConvertHtmlString vs RenderHtmlAsPdf), but the pattern is the same.

Where it stops being find-and-replace is if you decide to switch to IronPDF's async overloads. SelectPdf's conversion methods are synchronous; IronPDF offers both RenderHtmlAsPdf and RenderHtmlAsPdfAsync. The synchronous overload is the closest one-to-one match, but if you adopt the async variant you need to convert calling methods to async and use await.

Do I need to worry about page indexing differences?

No. Both SelectPdf and IronPDF use 0-based page indexing. This should be a non-issue for most migrations, but I would still audit any page manipulation code to confirm.

What happens to the Community edition watermark after migration?

SelectPdf's Community Edition applies a watermark to every page of every PDF until a license key is installed, and it caps output at 5 pages per document. Both restrictions are lifted only by purchasing a license. After migrating to IronPDF with a valid license key, verify that test PDFs no longer carry the watermark and that page counts beyond 5 render fully. It is a useful migration validation step.

Can I reuse a single ChromePdfRenderer instance for batch generation?

Yes. You can create one ChromePdfRenderer, configure its RenderingOptions, and reuse it across multiple render calls. For parallel rendering, create separate renderer instances per task. Verify thread-safety guidance in the IronPDF async documentation before sharing a single instance across threads.

How do I find all SelectPdf references in my codebase before starting?

Run these commands to locate every file that references SelectPdf APIs.

# Find SelectPdf usage
rg -l "SelectPdf|HtmlToPdf|ConvertHtmlString|ConvertUrl" --type cs
rg "SelectPdf|HtmlToPdf\b" --type cs -n

# Find project references
grep -r "SelectPdf" *.csproj **/*.csproj 2>/dev/null

# Count usage density to estimate migration scope
rg "HtmlToPdf\.|ConvertHtmlString|ConvertUrl\b" --type cs | wc -l

This gives you a clear picture of how many files need changes and whether you are dealing with simple conversion calls or more complex edition-specific features like merge and security.

When Should You Migrate from SelectPdf to IronPDF?

The decision depends on your current situation. Here is the decision matrix I use when advising teams.

  • If you are hitting the Community Edition's 5-page cap or its per-page watermark, switch — a paid IronPDF license has no page-count cap and no output watermark.
  • If you need digital signatures, full text extraction, or PDF/A compliance, switch — IronPDF covers all of them; SelectPdf does not.
  • If your HTML templates use modern CSS (grid, flexbox gap, custom properties), switch — IronPDF's embedded Chromium renders them where SelectPdf's default internal WebKit engine often does not. The optional Blink engine in SelectPdf narrows the gap but is opt-in.
  • If you must deploy to Linux, Docker, Azure Functions, or AWS Lambda, switch — SelectPdf is Windows-only by design.
  • If you are doing basic Windows-only HTML-to-PDF that already works and you have no manipulation needs, evaluate the cost of switching — SelectPdf may be sufficient.

If you are hitting Community edition limits, accumulating secondary libraries for missing features, or struggling with CSS rendering fidelity, the migration is straightforward and the payoff is consolidation. The API similarity between SelectPdf and IronPDF means the main work is resolving feature gaps, not learning a fundamentally different approach.

The most useful benchmark to run is not single-render latency but concurrent throughput. That is typically where the rendering engine difference shows up most clearly. Get a trial license key and test against your actual HTML templates in your environment.
