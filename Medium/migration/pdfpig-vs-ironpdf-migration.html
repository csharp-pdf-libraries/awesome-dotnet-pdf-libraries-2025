Migrating from PdfPig to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Consider Migrating from PdfPig to IronPDF?

Before you open a single pull request, it helps to be honest about the motivation. I have worked with teams whose CI pipelines broke because PdfPig holds a file handle via PdfDocument.Open(), and parallel test runners step on each other. It works on your machine because you run tests serially. The pipeline runs them in parallel.

That said, there are quick fixes within PdfPig itself. If you open from bytes instead of from a file path, you eliminate the file handle contention entirely. If those fixes resolve your CI failure, the migration may not be necessary.

Here are the nine reasons teams most commonly evaluate alternatives to PdfPig:

First, HTML-to-PDF is not supported. PdfPig is a reader, not a renderer. Teams needing HTML-to-PDF add a second library like wkhtmltopdf or NReco, which adds maintenance overhead. Second, document manipulation is limited. Merging, splitting, watermarking, and encrypting PDFs are not PdfPig's scope. Third, dependency sprawl becomes real when a codebase has PdfPig plus wkhtmltopdf plus PdfSharp plus iTextSharp all handling the same domain across four packages.

Fourth, parallel test isolation requires explicit workarounds for file handle management. Fifth, if your requirements have grown beyond parsing into generation and manipulation, a single library covering all three simplifies the stack. Sixth, PdfPig's word/letter model is powerful for layout analysis but verbose for simple text extraction tasks.

Seventh, cloud and serverless rendering with PdfPig requires a second deployment concern for the HTML-to-PDF binary. Eighth, tracking two or more PDF libraries through version updates is twice the maintenance work. Ninth, teams with HTML templates for reports need generation capabilities PdfPig does not provide.

How Do You Set Up IronPDF and Replace PdfPig?

I will walk you through the core migration in three steps. First, find your existing PdfPig references, swap the packages, and then update your code file by file.

Finding PdfPig References in Your Codebase

# Find all PdfPig usage
rg -l "PdfPig\|UglyToad" --type cs

# Find using statements
rg "using UglyToad\.PdfPig" --type cs -n

# Find document open patterns
rg "PdfDocument\.Open\|\.GetPages\|\.GetWords\|\.GetLetters" --type cs -n

# Find text extraction calls
rg "\.Text\b\|GetWords\(\)\|GetLetters\(\)" --type cs -n | grep -i pdf

# Find NuGet package references
grep -r -i "pdfpig\|uglytoad" **/*.csproj *.csproj 2>/dev/null

Removing PdfPig and Installing IronPDF

# Remove PdfPig
dotnet remove package PdfPig

# Also remove secondary libraries if added only to supplement PdfPig:
# dotnet remove package NReco.PdfGenerator  # if added for HTML-to-PDF
# dotnet remove package PdfSharp            # if added for merge/manipulation

# Install IronPDF
dotnet add package IronPdf
dotnet restore

On Windows via Package Manager Console: Install-Package IronPdf and Uninstall-Package PdfPig. You can verify the install pattern at the IronPDF license configuration guide.

What Does Each Before/After Migration Look Like in Practice?

Let me walk through four complete migration examples so you can see exactly what changes in real code.

Text Extraction (PdfPig's Primary Use Case)

Before (PdfPig — full text extraction per page):

using UglyToad.PdfPig;
using UglyToad.PdfPig.Content;
using System;
using System.Text;

class TextExtractionExample
{
    static void Main()
    {
        using var doc = PdfDocument.Open("invoice.pdf");

        Console.WriteLine($"PDF has {doc.NumberOfPages} pages");

        var fullText = new StringBuilder();

        // PdfPig: 1-based page access via GetPages()
        foreach (Page page in doc.GetPages())
        {
            Console.WriteLine($"\n--- Page {page.Number} ---");

            // page.Text: concatenated text content
            fullText.AppendLine(page.Text);
            Console.WriteLine(page.Text);

            // Word-level access with position data:
            int wordCount = 0;
            foreach (Word word in page.GetWords())
            {
                wordCount++;
                // word.Text, word.BoundingBox.Left, word.BoundingBox.Bottom
            }
            Console.WriteLine($"({wordCount} words)");
        }

        System.IO.File.WriteAllText("extracted.txt", fullText.ToString());
        Console.WriteLine("\nExtracted to: extracted.txt");
    }
}

After (IronPDF):

using IronPdf;
using System;
using System.IO;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

var pdf = PdfDocument.FromFile("invoice.pdf");
Console.WriteLine($"PDF has {pdf.PageCount} pages");

// Full document text — one call
string allText = pdf.ExtractAllText();
File.WriteAllText("extracted.txt", allText);
Console.WriteLine($"Extracted {allText.Length} chars to: extracted.txt");

// Per-page access (0-based):
for (int i = 0; i < pdf.PageCount; i++)
    Console.WriteLine($"Page {i + 1}: {pdf.Pages[i].Text.Length} chars");

// Guide: https://ironpdf.com/how-to/extract-text-and-images/

HTML to PDF (Secondary Tool Migration)

Before (wkhtmltopdf subprocess, commonly added alongside PdfPig for generation):

using System;
using System.Diagnostics;
using System.IO;

class HtmlToPdfWithSecondaryTool
{
    static void Main()
    {
        // PdfPig reads PDFs; it doesn't generate them
        // Teams add wkhtmltopdf or NReco for HTML-to-PDF generation
        string html    = "<html><body><h1>Report</h1><p>Data here</p></body></html>";
        string tmpHtml = Path.GetTempFileName() + ".html";
        string tmpPdf  = Path.GetTempFileName() + ".pdf";

        File.WriteAllText(tmpHtml, html);

        var psi = new ProcessStartInfo("wkhtmltopdf", $"\"{tmpHtml}\" \"{tmpPdf}\"")
        {
            RedirectStandardError = true,
            UseShellExecute = false,
            CreateNoWindow  = true
        };

        using var proc = Process.Start(psi)!;
        if (!proc.WaitForExit(30_000))
        {
            proc.Kill();
            throw new TimeoutException("wkhtmltopdf timed out");
        }
        if (proc.ExitCode != 0)
            throw new Exception($"wkhtmltopdf error: {proc.StandardError.ReadToEnd()}");

        byte[] pdfBytes = File.ReadAllBytes(tmpPdf);
        File.Delete(tmpHtml);
        File.Delete(tmpPdf);
        Console.WriteLine($"Generated {pdfBytes.Length} bytes");
    }
}

After (IronPDF — replaces secondary HTML-to-PDF tool):

using IronPdf;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Report</h1><p>Data here</p></body></html>"
);
pdf.SaveAs("report.pdf");
// Guide: https://ironpdf.com/how-to/html-string-to-pdf/

Merge PDFs (Secondary Library Migration)

Before (PdfSharp secondary library, common addition to PdfPig-based projects):

using PdfSharp.Pdf;
using PdfSharp.Pdf.IO;
using System;

class MergePdfsExample
{
    static void Main()
    {
        // PdfPig can't merge PDFs — read-only library
        // PdfSharp is a common addition
        using var output = new PdfDocument();

        foreach (string path in new[] { "chapter1.pdf", "chapter2.pdf", "chapter3.pdf" })
        {
            using var input = PdfReader.Open(path, PdfDocumentOpenMode.Import);
            foreach (PdfPage page in input.Pages)
                output.AddPage(page);
        }

        output.Save("full_document.pdf");
        Console.WriteLine("Merged to: full_document.pdf");
    }
}

After (IronPDF native — eliminates PdfSharp dependency):

using IronPdf;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

var merged = PdfDocument.Merge(
    PdfDocument.FromFile("chapter1.pdf"),
    PdfDocument.FromFile("chapter2.pdf"),
    PdfDocument.FromFile("chapter3.pdf")
);
merged.SaveAs("full_document.pdf");
// Guide: https://ironpdf.com/how-to/merge-or-split-pdfs/

Watermark (Secondary Library Migration)

Before (iTextSharp secondary library, commonly added alongside PdfPig):

using iTextSharp.text;
using iTextSharp.text.pdf;
using System.IO;
using System;

class WatermarkExample
{
    static void Main()
    {
        // PdfPig is read-only — watermarking requires a secondary library
        using var reader  = new PdfReader("document.pdf");
        using var fs      = new FileStream("watermarked.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs);

        var baseFont = BaseFont.CreateFont(BaseFont.HELVETICA_BOLD, BaseFont.CP1252, false);

        for (int page = 1; page <= reader.NumberOfPages; page++)
        {
            var cb = stamper.GetOverContent(page);
            cb.SaveState();
            cb.SetGState(new PdfGState { FillOpacity = 0.3f });
            cb.BeginText();
            cb.SetFontAndSize(baseFont, 60);
            cb.SetColorFill(BaseColor.GRAY);
            cb.ShowTextAligned(Element.ALIGN_CENTER, "DRAFT", 300, 420, 45);
            cb.EndText();
            cb.RestoreState();
        }

        Console.WriteLine("Watermarked: watermarked.pdf");
    }
}

After (IronPDF — eliminates iTextSharp dependency):

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

var pdf = PdfDocument.FromFile("document.pdf");
var stamper = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronSoftware.Drawing.Color.LightGray,
    FontSize = 60,
    Opacity = 30,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");
// Guide: https://ironpdf.com/how-to/custom-watermark/

What Are the Frequently Asked Questions?

Answer: I get these questions on nearly every PdfPig migration I advise on. Here they are with direct answers.

Will My Text Extraction Output Be Identical After Migration?

Answer: Not necessarily. PdfPig uses spatial reading order for text extraction, while IronPDF uses the PDF content stream order. For simple single-column documents, the results are usually comparable. For complex multi-column layouts, right-to-left text, or documents with ligatures, you should run a side-by-side comparison on your representative document corpus before committing to the migration.

I recommend a simple character count comparison as a first signal, then a deeper word-set diff on documents where extraction quality matters most.

What Happens to My Word-Level Layout Analysis Code?

Answer: This is the biggest migration consideration. PdfPig exposes individual Letter and Word objects with geometric bounding boxes, fonts, and colours. IronPDF's extraction model is text-string focused. If your application logic depends on word.BoundingBox.Left or letter.FontName, that data is not directly available in IronPDF's extraction API.

For teams who need layout analysis AND HTML-to-PDF generation, keeping PdfPig for parsing while using IronPDF for generation is a valid architecture. You do not have to choose one library for everything.

How Do I Handle the Page Indexing Difference?

Answer: This is the most common migration bug. PdfPig uses 1-based page numbering via doc.GetPage(1) for the first page. IronPDF uses 0-based indexing via pdf.Pages[0]. Audit every GetPage(n) call in your codebase and subtract 1 for the IronPDF equivalent. Also update doc.NumberOfPages to pdf.PageCount.

How Do Encrypted PDFs Work After Migration?

Answer: Both libraries support opening password-protected PDFs, but the API differs. PdfPig uses PdfDocument.Open("secured.pdf", new ParsingOptions { Password = "yourpassword" }). IronPDF uses PdfDocument.FromFile("secured.pdf", "yourpassword"). The IronPDF approach is more concise, but functionally equivalent for reading. IronPDF additionally supports writing encrypted PDFs natively, which PdfPig does not. You can learn more at the IronPDF security guide.

What About Performance Differences?

Answer: Both libraries extract text from the PDF content stream, but they take different approaches. PdfPig is fully managed .NET code. IronPDF uses its Chromium-based internals. Profile on your actual document set rather than relying on benchmarks. For the file-handle contention issue that surfaces in parallel CI tests, IronPDF resolves it because PdfDocument.FromFile() reads the file and closes the handle. For large PDFs, note that PdfPig streams page data lazily while IronPDF loads the document into memory. Test with your largest representative files.

Can I Keep Both Libraries During a Gradual Migration?

Answer: Yes. A partial migration is a valid pattern. Keep PdfPig for layout analysis where you depend on word-level bounding box data, and use IronPDF for HTML-to-PDF generation and document manipulation. This eliminates the secondary libraries (wkhtmltopdf, PdfSharp, iTextSharp) while preserving PdfPig's layout strengths. Over time, you can evaluate whether IronPDF's text extraction meets your needs for the remaining PdfPig use cases.

What Is the Decision Framework for Your Team?

Answer: Here is the migration checklist I recommend. Work through it in order: pre-migration assessment, code migration, testing, and post-migration cleanup.

Pre-Migration

Inventory all PdfPig features in use with rg "UglyToad\|PdfPig" --type cs -n. Distinguish word/letter layout analysis from simple text extraction. Identify secondary libraries added alongside PdfPig (wkhtmltopdf, PdfSharp, iTextSharp). Verify IronPDF .NET target framework compatibility. Confirm commercial license requirements internally. Set up an IronPDF trial license in your dev environment. Run side-by-side text extraction comparison on representative PDFs. Document all page indexing references with rg "GetPage\|\.Pages\[" --type cs -n.

Code Migration

Remove the PdfPig NuGet package. Remove secondary libraries if they only supplemented PdfPig. Add the IronPdf NuGet package. Replace using UglyToad.PdfPig with using IronPdf. Replace PdfDocument.Open() with PdfDocument.FromFile(). Replace doc.GetPages() / page.Text with pdf.Pages[n].Text using 0-based indexing. Replace doc.NumberOfPages with pdf.PageCount. Update all page index references: PdfPig page N becomes IronPDF index N-1. Replace HTML-to-PDF secondary tools with ChromePdfRenderer. Add your IronPDF license key to config.

Testing

Compare text extraction output on all PDF fixtures. Verify page index updates by regression-testing page-specific operations. Test encrypted PDF opening with password. Test merge, watermark, and security if migrating from secondary libraries. Run the parallel test suite to verify the CI/CD failure mode is resolved. Render HTML templates if HTML-to-PDF is a new requirement. Test in your Docker/CI environment.

Post-Migration

Remove secondary libraries that are no longer needed. Update CI/CD pipeline documentation. Commit the updated packages.lock.json if using locked restores. Monitor the first production week for any text extraction regression.

The most common reason a PdfPig migration hits complications is the word/letter layout model. Teams doing simple text extraction migrate in an afternoon. Teams with layout-aware parsing should budget more time to validate extraction output on their document corpus.
