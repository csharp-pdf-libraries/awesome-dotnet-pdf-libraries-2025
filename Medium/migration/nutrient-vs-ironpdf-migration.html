Migrating from Nutrient to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate from Nutrient to IronPDF?

You added a second pod. The PDF rendering service handled it fine. You added a third, and that's when the async pipeline started biting: every render is await PdfProcessor.CreateAsync(), every save is await document.SaveAsync(path), every annotation is await document.AddAnnotationAsync(i, watermark). The processor lifecycle wraps every workflow in using blocks and ceremony. What looked tidy at one pod looked like a wall of async/await at three.

If your team is in that exact position -- scaling up and finding that your Nutrient (formerly PSPDFKit) integration's async-first surface and platform ceremony don't match what you actually need from a PDF library -- here are the key reasons I see teams evaluate alternatives:

Async-first complexity for synchronous workflows -- Nutrient's .NET SDK forces await on initialization, generation, save, and even annotation. If your workflow is fundamentally synchronous, every call site is heavier than it needs to be.

Pricing tier -- Nutrient is a premium commercial SDK positioned for enterprise; pricing is opaque and requires a sales conversation. If your use case is primarily HTML-to-PDF generation, a published-pricing alternative may be more cost-effective at scale.

API surface breadth -- Nutrient has rebranded from a PDF SDK into a "document intelligence platform" with AI features, document workflow, and broader scope. If you only need HTML-to-PDF plus basic manipulation, you're paying for and managing surface you don't use.

Annotation-based watermarks and headers -- Nutrient models watermarks and headers as TextAnnotation objects positioned per page. Teams that prefer HTML/CSS for stamp content find this verbose.

Processor lifecycle -- using var processor = await PdfProcessor.CreateAsync() on every operation is more ceremony than the equivalent new ChromePdfRenderer().

Rebrand churn -- PSPDFKit becoming Nutrient means documentation, package names, and community answers reference both. Migrations between major versions of Nutrient itself can be non-trivial.

Feature set consolidation -- teams with secondary libraries for merge, watermark, and similar operations evaluate whether a single library can handle all PDF operations.

What Does a Side-by-Side Comparison Look Like?

  • Focus. Nutrient is a document intelligence platform; IronPDF is a focused HTML-to-PDF and PDF manipulation library.
  • Pricing. Nutrient is enterprise-tier with sales-led pricing; IronPDF publishes per-developer pricing.
  • API style. Nutrient is async-first (everything returns Task); IronPDF is sync by default with async variants where it matters (RenderHtmlAsPdfAsync).
  • Initialization. Nutrient requires await PdfProcessor.CreateAsync() with a disposable processor; IronPDF uses new ChromePdfRenderer().
  • HTML rendering. Nutrient has its own pipeline behind GeneratePdfFromHtmlStringAsync; IronPDF uses an embedded Chromium engine.
  • Page indexing. Both are 0-based, so loops like for (int i = 0; i < document.PageCount; i++) port directly.
  • Watermarks. Nutrient uses TextAnnotation objects with explicit position, opacity, and font size; IronPDF accepts an HTML string and places it via ApplyWatermark.
  • Namespaces. PSPDFKit.Pdf and PSPDFKit.Pdf.Annotation on the legacy/current package; IronPdf and IronPdf.Editing on the target.

How Much Effort Does Each Feature Require to Migrate?

  • HTML to PDF generation -- Low. processor.GeneratePdfFromHtmlStringAsync(html) maps directly to renderer.RenderHtmlAsPdf(html). Drop the await or use the async variant.
  • Merge PDFs -- Low. processor.MergeAsync(new List<PdfDocument> { d1, d2 }) becomes PdfDocument.Merge(pdf1, pdf2). No processor needed.
  • Watermark / stamp -- Low to Medium. Per-page TextAnnotation loops collapse into a single ApplyWatermark with HTML content. Restyle the existing FontSize/Opacity as CSS.
  • Headers and footers -- Low. Nutrient's manual annotation-per-page approach becomes RenderingOptions.HtmlHeader/HtmlFooter with {page} and {total-pages} placeholders.
  • Password protection -- Low. Nutrient's SecurityOptions object maps to IronPDF's SecuritySettings properties.
  • Form filling -- Low to Medium. document.GetFormFieldsAsync() becomes pdf.Form.Fields / pdf.Form.FindFormField(name); verify your specific field types (signature, choice, etc.) round-trip cleanly.
  • Text extraction -- Low. The per-page page.ExtractTextAsync() loop reduces to pdf.ExtractAllText().
  • PDF viewing / annotations in the browser -- N/A. Nutrient's web/mobile viewers have no IronPDF equivalent; that part of the stack is out of scope and would need to stay in place or be replaced separately.
  • Async-to-sync collapse -- Low. Most of the migration effort is removing await and the using var processor ceremony around it.

And here is the decision matrix for common scenarios:

  • If you need a PDF viewer in the browser, Nutrient's viewer has no IronPDF equivalent -- plan a partial migration that keeps the viewer and replaces only the server-side generation layer.
  • If you only need server-side generation, IronPDF fits well; verify feature coverage for any edge-case annotation or signing requirements.
  • If async-first ceremony is the trigger, IronPDF's sync-by-default surface with optional async variants is the most direct simplification.
  • If your workflow is annotation-heavy or form-heavy, verify IronPDF's coverage for your specific field types and annotation kinds before committing.

How Do I Diagnose Async/Lifecycle Problems Before Migrating?

Before looking at library options, make sure you've identified the exact failure mode. Pain in async-first PDF stacks comes from a few distinct sources: processor lifecycle thrash (creating and disposing PdfProcessor instances per render rather than reusing them), forced async fan-out in code paths that do not need it, and resource exhaustion under concurrent load when each call awaits a disposable processor.

Add logging around your render calls to surface where the time goes:

Diagnosis Pattern for Render Latency

// Add logging around render calls to surface timing issues
var sw = System.Diagnostics.Stopwatch.StartNew();
try
{
    // your render call here
    sw.Stop();
    _logger.LogInformation("Render completed in {ms}ms on thread {tid}",
        sw.ElapsedMilliseconds, System.Threading.Thread.CurrentThread.ManagedThreadId);
}
catch (Exception ex)
{
    _logger.LogError(ex, "Render failed after {ms}ms on thread {tid}",
        sw.ElapsedMilliseconds, System.Threading.Thread.CurrentThread.ManagedThreadId);
}

If the bulk of your time is in PdfProcessor.CreateAsync() on every request rather than in the actual render, your lifecycle is the bottleneck.

Step 2: Namespace Imports

// Before
using PSPDFKit.Pdf;

// After
using IronPdf;
using IronPdf.Rendering;

Step 3: Basic HTML-to-PDF, Async Collapsed

// Before (Nutrient / PSPDFKit -- async-first, processor lifecycle)
using PSPDFKit.Pdf;
using System.Threading.Tasks;

class PdfService
{
    public async Task<byte[]> GeneratePdfAsync(string html)
    {
        using var processor = await PdfProcessor.CreateAsync();
        var document = await processor.GeneratePdfFromHtmlStringAsync(html);
        await document.SaveAsync("output.pdf");
        return await document.ToBytesAsync();
    }
}

// After (IronPDF -- sync by default, async available)
using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

class PdfService
{
    public PdfDocument GeneratePdf(string html)
    {
        // No processor lifecycle, no required await
        var renderer = new ChromePdfRenderer();
        return renderer.RenderHtmlAsPdf(html);
    }

    // For high-throughput async paths
    public async Task<PdfDocument> GeneratePdfAsync(string html)
    {
        var renderer = new ChromePdfRenderer();
        return await renderer.RenderHtmlAsPdfAsync(html);
    }
}
// Async guide: https://ironpdf.com/how-to/async/

What Are the IronPDF Concurrency Patterns in Detail?

Once you drop the per-call using var processor ceremony, the threading model is mostly about not sharing a single renderer across threads while it is being mutated. Here's the explicit IronPDF concurrency approach with three patterns depending on your use case.

Pattern 1: Per-Request Renderer (Recommended for Web Apps)

// ASP.NET controller -- new renderer per request
// Each renderer is independent -- no shared state issues
[HttpGet("generate")]
public async Task<IActionResult> GeneratePdf([FromQuery] string templateId)
{
    string html = await _templateService.GetHtmlAsync(templateId);

    var renderer = new ChromePdfRenderer();
    renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

    using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    return File(pdf.Stream, "application/pdf", $"{templateId}.pdf");
}

Pattern 2: Parallel Batch Rendering

using System.Threading.Tasks;
using IronPdf;

// Render a batch in parallel -- each task gets its own renderer
async Task<PdfDocument[]> RenderBatchAsync(IEnumerable<string> htmlItems)
{
    return await Task.WhenAll(
        htmlItems.Select(async html =>
        {
            // One renderer per concurrent task -- explicitly safe
            var renderer = new ChromePdfRenderer();
            return await renderer.RenderHtmlAsPdfAsync(html);
        })
    );
}
// Parallel examples: https://ironpdf.com/examples/parallel/

Pattern 3: Bounded Concurrency with SemaphoreSlim

using System.Threading;
using System.Threading.Tasks;
using IronPdf;

class BoundedPdfRenderer
{
    private readonly SemaphoreSlim _semaphore;

    // Limit concurrent renders to avoid memory pressure
    public BoundedPdfRenderer(int maxConcurrent = 4)
        => _semaphore = new SemaphoreSlim(maxConcurrent);

    public async Task<PdfDocument> RenderAsync(string html)
    {
        await _semaphore.WaitAsync();
        try
        {
            var renderer = new ChromePdfRenderer();
            return await renderer.RenderHtmlAsPdfAsync(html);
        }
        finally
        {
            _semaphore.Release();
        }
    }
}

How Do I Map Nutrient Classes and Methods to IronPDF?

The namespace and class mapping is straightforward. Here are the core equivalences:

  • PSPDFKit.Pdf / Nutrient.Pdf → IronPdf -- core namespace
  • PSPDFKit.Pdf.Rendering → IronPdf.Rendering -- render configuration
  • PSPDFKit.Pdf.Annotation → IronPdf.Editing -- watermarks and stamps
  • PSPDFKit.Pdf.Forms → IronPdf.Forms -- form field handling
  • PdfProcessor (created via PdfProcessor.CreateAsync()) → ChromePdfRenderer -- render entry point, no async factory
  • PdfConfiguration → ChromePdfRenderOptions (accessed via renderer.RenderingOptions) -- render configuration as properties, not a separate object
  • PSPDFKit.Pdf.PdfDocument → IronPdf.PdfDocument -- document representation
  • TextAnnotation → HTML passed to ApplyWatermark or a TextStamper -- annotation objects collapse into HTML/CSS or stamp configuration

And here is how the document loading and page operations map:

  • HTML string. await processor.GeneratePdfFromHtmlStringAsync(html) → renderer.RenderHtmlAsPdf(html).
  • URL. await processor.CreatePdfFromUrlAsync(url, config) → renderer.RenderUrlAsPdf(url).
  • HTML file. await processor.GeneratePdfFromFileAsync(path) → renderer.RenderHtmlFileAsPdf(path).
  • Existing PDF. await processor.OpenAsync(path) → PdfDocument.FromFile(path).
  • Page count. document.PageCount → pdf.PageCount.
  • Paper size. config.PageSize = PageSize.A4 → renderer.RenderingOptions.PaperSize = PdfPaperSize.A4.
  • Margins. config.Margins = new Margins(t, r, b, l) → individual MarginTop/MarginRight/MarginBottom/MarginLeft properties.
  • Orientation. config.Orientation = Orientation.Landscape → renderer.RenderingOptions.PaperOrientation = PdfPaperOrientation.Landscape.
  • Merge. await processor.MergeAsync(new List<PdfDocument> { d1, d2 }) → PdfDocument.Merge(pdf1, pdf2).
  • Remove page. await document.RemovePageAsync(index) → pdf.RemovePages(index).
  • Save. await document.SaveAsync(path) → pdf.SaveAs(path).

How Do I Migrate HTML-to-PDF, Merge, Watermark, and Password Protection?

These are the four most common operations teams need to migrate. Let me walk through each one with complete before/after code.

HTML to PDF

// Before (Nutrient / PSPDFKit)
using PSPDFKit.Pdf;
using System.Threading.Tasks;

class HtmlToPdfExample
{
    static async Task Main()
    {
        var html = "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>";

        using var processor = await PdfProcessor.CreateAsync();
        var document = await processor.GeneratePdfFromHtmlStringAsync(html);
        await document.SaveAsync("invoice.pdf");
    }
}

// After (IronPDF -- sync, no processor lifecycle)
using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize    = IronPdf.Rendering.PdfPaperSize.A4;
renderer.RenderingOptions.MarginTop    = 10;
renderer.RenderingOptions.MarginBottom = 10;

var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>"
);
pdf.SaveAs("invoice.pdf");
// Guide: https://ironpdf.com/how-to/html-string-to-pdf/

Merge PDFs

// Before (Nutrient / PSPDFKit)
using PSPDFKit.Pdf;
using System.Collections.Generic;
using System.Threading.Tasks;

using var processor = await PdfProcessor.CreateAsync();
var document1 = await processor.OpenAsync("section1.pdf");
var document2 = await processor.OpenAsync("section2.pdf");

var mergedDocument = await processor.MergeAsync(
    new List<PdfDocument> { document1, document2 }
);
await mergedDocument.SaveAsync("merged.pdf");

// After (IronPDF native)
using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var merged = PdfDocument.Merge(
    PdfDocument.FromFile("section1.pdf"),
    PdfDocument.FromFile("section2.pdf")
);
merged.SaveAs("merged.pdf");
// Guide: https://ironpdf.com/how-to/merge-or-split-pdfs/

Watermark

// Before (Nutrient / PSPDFKit -- per-page TextAnnotation loop)
using PSPDFKit.Pdf;
using PSPDFKit.Pdf.Annotation;
using System.Threading.Tasks;

using var processor = await PdfProcessor.CreateAsync();
var document = await processor.OpenAsync("document.pdf");

for (int i = 0; i < document.PageCount; i++)
{
    var watermark = new TextAnnotation("CONFIDENTIAL")
    {
        Opacity  = 0.5,
        FontSize = 48
    };
    await document.AddAnnotationAsync(i, watermark);
}

await document.SaveAsync("watermarked.pdf");

// After (IronPDF -- single ApplyWatermark call, HTML content)
using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("document.pdf");

pdf.ApplyWatermark(
    "<h1 style='color:gray;opacity:0.5;font-size:48px;'>CONFIDENTIAL</h1>",
    50,
    VerticalAlignment.Middle,
    HorizontalAlignment.Center);

pdf.SaveAs("watermarked.pdf");

// Or use TextStamper for direct property control:
var stamper = new TextStamper
{
    Text              = "CONFIDENTIAL",
    FontColor         = IronSoftware.Drawing.Color.LightGray,
    FontSize          = 60,
    Opacity           = 30, // 0-100 integer, not 0.0-1.0
    Rotation          = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");
// Guide: https://ironpdf.com/how-to/custom-watermark/

Password Protection

// Before (Nutrient / PSPDFKit)
using PSPDFKit.Pdf;
using PSPDFKit.Pdf.Security;
using System.Threading.Tasks;

using var processor = await PdfProcessor.CreateAsync();
var document = await processor.OpenAsync("input.pdf");

var securityOptions = new SecurityOptions
{
    OwnerPassword = "ownerpass",
    UserPassword  = "userpass",
    Permissions = new Permissions
    {
        Printing      = PrintingPermission.HighResolution,
        ModifyContent = false,
        CopyContent   = false
    }
};

await document.ApplySecurityAsync(securityOptions);
await document.SaveAsync("secured.pdf");

// After (IronPDF)
using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("input.pdf");
pdf.SecuritySettings.UserPassword              = "userpass";
pdf.SecuritySettings.OwnerPassword             = "ownerpass";
pdf.SecuritySettings.AllowUserPrinting         = PdfPrintSecurity.FullPrintRights;
pdf.SecuritySettings.AllowUserCopyPasteContent = false;
pdf.SaveAs("secured.pdf");
// Guide: https://ironpdf.com/how-to/pdf-permissions-passwords/

What Are the Critical Migration Notes I Should Know?

There are three things that catch teams off guard during this migration.

Viewer functionality: If your application uses Nutrient's PDF viewer component (browser-side or mobile PDF rendering), that's out of scope for an IronPDF migration. IronPDF is a server-side generation and manipulation library. You can migrate the server-side component while keeping the viewer, or plan a separate viewer replacement.

Page indexing: Both Nutrient and IronPDF use 0-based page indexing, so existing loops over document.PageCount port directly. Audit any code that does i + 1 for display purposes -- that math stays the same.

// IronPDF: 0-based (same as Nutrient)
var firstPage = pdf.Pages[0];                  // first
var lastPage  = pdf.Pages[pdf.PageCount - 1];  // last

Output model: Nutrient's document.ToBytesAsync() and document.ToStream() map to IronPDF's pdf.BinaryData and pdf.Stream properties. Both are sync property accessors in IronPDF -- update call sites that currently await them.

DI Registration in ASP.NET

// If registering as singleton for performance:
builder.Services.AddSingleton<ChromePdfRenderer>(sp =>
{
    var renderer = new ChromePdfRenderer();
    renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
    return renderer;
});

// NOTE: If sharing a single instance, ensure thread safety by not
// modifying RenderingOptions after registration.
// For per-request options variation: use transient registration instead.
builder.Services.AddTransient<ChromePdfRenderer>();

How Do I Benchmark Performance After Migration?

The opening scenario was async ceremony multiplying across pods. Before finalizing the migration, benchmark IronPDF's behavior under your specific load:

Concurrency Benchmark

using System;
using System.Diagnostics;
using System.Threading.Tasks;
using System.Linq;
using IronPdf;

class ConcurrencyBenchmark
{
    static async Task Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
        const int totalRequests = 100;
        string html = "<html><body><h1>Test</h1><p>Content</p></body></html>";

        var sw = Stopwatch.StartNew();
        int completed = 0;

        // Simulate concurrent rendering
        await Task.WhenAll(
            Enumerable.Range(0, totalRequests).Select(async i =>
            {
                var renderer = new ChromePdfRenderer(); // one per task
                using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
                System.Threading.Interlocked.Increment(ref completed);
                if (completed % 10 == 0)
                    Console.WriteLine($"{completed}/{totalRequests} completed");
            })
        );

        sw.Stop();
        Console.WriteLine($"Total: {sw.ElapsedMilliseconds}ms for {totalRequests} renders");
        Console.WriteLine($"Throughput: {totalRequests / sw.Elapsed.TotalSeconds:F1} renders/sec");
    }
}

Always dispose PdfDocument after use in long-running services. Watch for cold start overhead on first render (Chromium initialization), memory footprint under concurrent load, and HTTP response streaming versus buffering to memory.

Frequently Asked Questions

Can I keep Nutrient's PDF viewer while migrating server-side generation to IronPDF?

Yes. The viewer and the server-side generator are separate concerns. Nutrient's PDF viewer component handles browser-side and mobile PDF rendering, which is out of scope for IronPDF. You can migrate the server-side generation component while keeping Nutrient's viewer in place.

Verify that Nutrient's viewer still functions correctly with IronPDF-generated output as part of your testing phase.

Does IronPDF let me drop the async ceremony Nutrient requires?

Yes. Nutrient's .NET SDK is async-first: PdfProcessor.CreateAsync, GeneratePdfFromHtmlStringAsync, OpenAsync, SaveAsync, AddAnnotationAsync, RemovePageAsync, and ToBytesAsync are all Task-returning. IronPDF is sync by default with optional async variants (RenderHtmlAsPdfAsync) for code paths that genuinely need them. Most call sites lose the await and the using var processor = await PdfProcessor.CreateAsync() wrapper.

For teams whose workflows are fundamentally synchronous, this collapse is the largest single readability win of the migration.

How does IronPDF handle thread safety differently from Nutrient?

IronPDF's per-request renderer pattern makes the threading model visible in the code. Each ChromePdfRenderer instance is independent -- no shared state issues. You don't need locking or synchronization. For web apps, create a new renderer per request. For batch jobs, create one per parallel task.

Check the IronPDF async documentation for the recommended patterns under concurrent load.

What .NET framework version do I need for IronPDF?

IronPDF requires .NET 6+ as the target framework. Before starting the migration, verify your project's target framework is compatible. Also confirm that your Nutrient SDK documentation is available for your specific version so you can map features accurately.

Run dotnet add package IronPdf after removing the PSPDFKit.Dotnet NuGet package, then dotnet restore to confirm everything resolves cleanly.

Will IronPDF handle my existing Nutrient merge and watermark workflows?

Yes. Merging is a single static call: PdfDocument.Merge(pdf1, pdf2). Watermarking uses a dedicated stamp API with precise control over text, opacity, rotation, and alignment via either an HTML string or a TextStamper object. Both are native to IronPDF -- no secondary libraries needed.

If you were previously using a secondary library like PdfSharp or iTextSharp alongside Nutrient for merge or watermark operations, you can consolidate down to IronPDF alone.

How should I benchmark Nutrient against IronPDF for my specific workload?

Run the concurrency benchmark from this article with your actual HTML templates and realistic traffic levels. Focus on concurrent rendering under load -- that's where the differences surface. First-render latency with IronPDF will be higher due to Chromium startup, but sustained throughput and memory behavior under concurrent load are what matter for server workloads.

Profile memory for at least 24 hours under production load. Watch Gen 2 GC frequency, working set trend (should be stable, not growing), and thread pool queue depth under burst load.

When Should You Migrate?

The async-first surface and processor-lifecycle ceremony in Nutrient's .NET SDK is fine when async fits your code, but it gets noisy when it doesn't. The sync-by-default surface in IronPDF, with async variants where they earn their keep, lets the code shape match the workflow shape.

If async ceremony in synchronous workflows is your trigger, IronPDF's surface collapses the noise. If you need server-side generation only, IronPDF fits well. If you need Nutrient's PDF viewer in the browser or on mobile, plan a partial migration -- keep the viewer, replace the generation layer.

The migration is mechanical for the core operations: HTML-to-PDF, merge, watermark, and password protection all have direct IronPDF equivalents. Benchmark before committing, verify feature coverage for your specific use case, and run the tests at production traffic levels before deploying.
