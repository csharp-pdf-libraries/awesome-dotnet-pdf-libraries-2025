Migrating from ComPDFKit to IronPDF: From Install to Ship

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Performance numbers are the first thing people reach for when justifying a library migration to management. They're also the easiest to misrepresent — a benchmark that doesn't match your workload is worse than no benchmark at all.

This article takes a benchmark-aware approach to migrating from ComPDFKit to IronPDF. That means: showing you how to measure what actually matters in your use case, providing before/after code for the operations most teams care about, and flagging where the libraries differ in architecture in ways that will affect throughput under real load. No synthetic numbers that don't apply to your environment.

You'll leave with working migration code, an API mapping table, and a repeatable benchmark harness you can run against your own workloads — because the only relevant performance number is the one measured with your data.

Why Should I Migrate from ComPDFKit to IronPDF?

Teams using ComPDFKit on .NET typically adopted it for PDF viewing, annotation, or interactive document scenarios. Server-side batch generation and HTML-to-PDF are where friction tends to appear:

ComPDFKit is viewer-focused and does not have a native HTML-to-PDF renderer — producing HTML-styled output requires manual text-widget placement through CPDFEditor, which is fine for stamps and annotations but does not handle modern CSS, flexbox, or responsive layouts. Libraries optimized for interactive viewers also have different performance characteristics at scale than batch generators, and you must remember to call document.Release() after every operation or memory leaks.

IronPDF is generation-focused. Merge, watermark, and encrypt operations require fewer setup steps; memory is handled by the GC; and the Chromium-based HTML-to-PDF renderer handles modern CSS out of the box.

How Do the Two Libraries Compare at a Glance?

  • Focus. ComPDFKit is a PDF viewer, annotation, forms, and editing SDK. IronPDF is a generation-first library covering HTML-to-PDF, edit, merge, and security.
  • API style. ComPDFKit uses C++-influenced SDK patterns (InitWith..., WriteToFilePath, manual Release()). IronPDF exposes a high-level renderer plus a fluent document model.
  • HTML rendering. ComPDFKit has no native HTML-to-PDF — you assemble pages manually through CPDFEditor. IronPDF ships a Chromium-based renderer that handles modern CSS.
  • Memory management. ComPDFKit requires explicit document.Release() and page.Release() calls. IronPDF leans on the GC, with optional using.
  • Page indexing. Both libraries are 0-based, so existing index math carries over.
  • Namespaces. ComPDFKit.PDFDocument / ComPDFKit.PDFPage / ComPDFKit.Import map down to a single IronPdf namespace.
  • Pricing. ComPDFKit is sold per-app or per-platform; IronPDF is per-developer with royalty-free production deployment.

How Much Effort Does Each Feature Take to Migrate?

The effort varies by feature. HTML-to-PDF is a net-new capability rather than a port, since ComPDFKit has no native HTML renderer. Annotations are the heaviest lift because ComPDFKit's annotation model is richer than IronPDF's.

  • HTML string to PDF — Low. Net-new capability vs ComPDFKit's manual CreateTextWidget placement; the IronPDF API is a one-liner.
  • HTML file / URL to PDF — Low. Same story as the string variant.
  • Merge PDFs — Low–Medium. ComPDFKit's ImportPagesAtIndex(doc, range, idx) becomes IronPDF's PdfDocument.Merge(...).
  • Split / extract pages — Medium. Map manual page-copy loops to pdf.CopyPages(start, end).
  • Watermark — Low. ComPDFKit uses the CPDFWatermark API (InitWatermark / SetText / CreateWatermark); IronPDF uses pdf.ApplyWatermark(html, ...) with HTML/CSS.
  • Password protection — Low. Both libraries support it; property names differ.
  • Annotation migration — High. ComPDFKit's annotation model is richer; round-trip parity needs explicit testing.
  • Form field editing — Medium. ComPDFKit's iterate-and-set pattern collapses to pdf.Form.SetFieldValue(name, value).
  • Server-side batch — Medium. Architecturally different (Chromium process vs in-process SDK); benchmark.
  • Async / parallel — Low–Medium. IronPDF exposes ...Async variants; ComPDFKit is primarily synchronous.

How Do I Set Up a Measurement Baseline Before Migrating?

Before removing ComPDFKit, capture timing data for your most frequent operations. This gives you apples-to-apples comparison after migration.

ComPDFKit Benchmark Harness

using System.Diagnostics;

// Benchmark harness — run against your actual workload before migration
class PdfBenchmark
{
    static void Main()
    {
        int iterations = 50;
        var sw = Stopwatch.StartNew();

        for (int i = 0; i < iterations; i++)
        {
            // Run your most common ComPDFKit operation here
            // Example: ConvertHtmlToPdf("sample.html", $"out_{i}.pdf")
        }

        sw.Stop();
        double avgMs = (double)sw.ElapsedMilliseconds / iterations;
        Console.WriteLine($"ComPDFKit average: {avgMs:F1} ms per operation ({iterations} iterations)");
        Console.WriteLine($"Total: {sw.ElapsedMilliseconds} ms");
    }
}

Run this benchmark. Record the numbers. Then run the equivalent IronPDF benchmark after migration. The delta in your environment is the only number that matters.

How Do the APIs Map Between ComPDFKit and IronPDF?

ComPDFKit .NET class names are stable across the ComPDFKit.NetCore and ComPDFKit.NetFramework packages. All IronPDF entries reflect the current public API.

Namespace Mapping

  • ComPDFKit.PDFDocument → IronPdf — main document operations
  • ComPDFKit.PDFPage → IronPdf — page-level operations
  • ComPDFKit.Import → IronPdf — import / merge helpers
  • ComPDFKit.PDFAnnotation → IronPdf — annotations
  • Native library init → not required — IronPDF is NuGet-only

Core Class Mapping

  • CPDFDocument → PdfDocument — PDF document object
  • CPDFPage → PdfPage — page reference
  • CPDFEditor + CPDFEditType.EditText → ChromePdfRenderer — replace manual editor placement with HTML rendering
  • License verifier → IronPdf.License — static key, no SDK init

Document Loading Methods

  • Load from file — CPDFDocument.InitWithFilePath(path) → PdfDocument.FromFile(path)
  • Create empty document — CPDFDocument.CreateDocument() → new PdfDocument()
  • Load from stream — CPDFDocument.InitWithStream(stream) → PdfDocument.FromStream(stream)
  • Load from bytes — via stream in ComPDFKit → PdfDocument.FromBinaryData(bytes)
  • Save — document.WriteToFilePath(path) → pdf.SaveAs(path)
  • Release — document.Release() → not required (GC)

Page Operations

  • Page count — doc.PageCount → pdf.PageCount
  • Get page — doc.PageAtIndex(n) → pdf.Pages[n] (both 0-based)
  • Remove page — doc.RemovePage(n) → pdf.RemovePages(n)
  • Insert page — doc.InsertPage(idx, w, h, "") → via merge / CopyPages
  • Rotate page — page.SetRotation(angle) → pdf.Pages[i].Rotation = ...

Merge/Split

  • Merge — doc1.ImportPagesAtIndex(doc2, range, idx) → PdfDocument.Merge(pdf1, pdf2) or the List<PdfDocument> overload
  • Extract pages — manual page-copy loop → pdf.CopyPages(start, end) or pdf.CopyPages(int[])

What Do Complete Before/After Code Migrations Look Like?

Here are four complete migrations covering the operations most teams care about: HTML-to-PDF, merge, watermark, and password protection.

1. HTML to PDF (Before — ComPDFKit)

using ComPDFKit.PDFDocument;
using System;
using System.Drawing;

class Program
{
    static void Main()
    {
        // ComPDFKit has no native HTML-to-PDF, so the "HTML" output
        // is built by hand: a page, an editor, and text widgets at coords.
        var document = CPDFDocument.CreateDocument();
        var page = document.InsertPage(0, 595, 842, "");

        var editor = page.GetEditor();
        editor.BeginEdit(CPDFEditType.EditText);
        editor.CreateTextWidget(new RectangleF(50, 50, 500, 700), "HTML content here");
        editor.EndEdit();

        document.WriteToFilePath("output.pdf");
        document.Release();
    }
}

1. HTML to PDF (After — IronPDF)

using System;
using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        // Chromium-based rendering — handles modern CSS
        // https://ironpdf.com/how-to/html-string-to-pdf/
        var renderer = new ChromePdfRenderer();

        string html = @"<html>
            <body style='font-family:Arial; padding:40px'>
                <h1 style='color:#1D4ED8'>Invoice #3810</h1>
                <table border='1' style='width:100%'>
                    <tr><th>Item</th><th>Amount</th></tr>
                    <tr><td>Service A</td><td>$800</td></tr>
                    <tr><td>Service B</td><td>$400</td></tr>
                </table>
            </body>
        </html>";

        using var pdf = renderer.RenderHtmlAsPdf(html);
        pdf.SaveAs("invoice.pdf");
        Console.WriteLine("Saved invoice.pdf");
    }
}

2. Merge PDFs (Before — ComPDFKit)

using ComPDFKit.PDFDocument;
using ComPDFKit.Import;
using System;

class Program
{
    static void Main()
    {
        var document1 = CPDFDocument.InitWithFilePath("file1.pdf");
        var document2 = CPDFDocument.InitWithFilePath("file2.pdf");

        // Page range is a string; insertion index is the tail of doc1.
        document1.ImportPagesAtIndex(
            document2,
            "0-" + (document2.PageCount - 1),
            document1.PageCount);

        document1.WriteToFilePath("merged.pdf");
        document1.Release();
        document2.Release();
    }
}

2. Merge PDFs (After — IronPDF)

using System;
using System.Collections.Generic;
using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        // https://ironpdf.com/how-to/merge-or-split-pdfs/
        var pdf1 = PdfDocument.FromFile("file1.pdf");
        var pdf2 = PdfDocument.FromFile("file2.pdf");

        using var merged = PdfDocument.Merge(new List<PdfDocument> { pdf1, pdf2 });
        merged.SaveAs("merged.pdf");
        Console.WriteLine($"Merged {merged.PageCount} pages to merged.pdf");
    }
}

3. Watermark (Before — ComPDFKit)

using ComPDFKit.PDFDocument;
using ComPDFKit.PDFWatermark;
using System;

class Program
{
    static void Main()
    {
        var document = CPDFDocument.InitWithFilePath("input.pdf");

        // ComPDFKit native watermark API: InitWatermark + CPDFWatermark chain
        CPDFWatermark watermark = document.InitWatermark(
            C_Watermark_Type.WATERMARK_TYPE_TEXT);
        watermark.SetText("CONFIDENTIAL");
        watermark.SetFontName("Helvetica");
        watermark.SetFontSize(48);
        watermark.SetTextRGBColor(255, 0, 0);
        watermark.SetRotation(45);
        watermark.SetOpacity(76);  // 0..255 (76 ≈ 30%)
        watermark.SetVertalign(C_Watermark_Vertalign.WATERMARK_VERTALIGN_CENTER);
        watermark.SetHorizalign(C_Watermark_Horizalign.WATERMARK_HORIZALIGN_CENTER);
        watermark.SetPages($"0-{document.PageCount - 1}");
        watermark.SetFront(true);
        watermark.CreateWatermark();

        document.WriteToFilePath("watermarked.pdf");
        document.Release();
    }
}

3. Watermark (After — IronPDF)

using System;
using IronPdf;
using IronPdf.Editing;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var pdf = PdfDocument.FromFile("input.pdf");

        // https://ironpdf.com/how-to/stamp-text-image/
        pdf.ApplyWatermark(
            "<h1 style='color:rgba(255,0,0,0.3);'>CONFIDENTIAL</h1>",
            rotation: 45,
            verticalAlignment: VerticalAlignment.Middle,
            horizontalAlignment: HorizontalAlignment.Center);

        pdf.SaveAs("watermarked.pdf");
        Console.WriteLine("Watermarked PDF saved");
    }
}

4. Password Protection (Before — ComPDFKit)

using ComPDFKit.PDFDocument;
using System;

class Program
{
    static void Main()
    {
        var document = CPDFDocument.InitWithFilePath("input.pdf");

        var permissions = new CPDFPermissionsInfo
        {

            AllowsPrinting = true,

        document.SetPassword("owner456", "user123");
        document.SetPermissions(permissions);
        document.WriteToFilePath("protected.pdf");
        document.Release();
    }
}

4. Password Protection (After — IronPDF)

using System;
using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        // https://ironpdf.com/how-to/pdf-permissions-passwords/
        using var pdf = PdfDocument.FromFile("input.pdf");

        pdf.SecuritySettings.UserPassword = "user123";
        pdf.SecuritySettings.OwnerPassword = "owner456";
        pdf.SecuritySettings.AllowUserPrinting =
            IronPdf.Security.PdfPrintSecurity.FullPrintRights;

        pdf.SaveAs("protected.pdf");
        Console.WriteLine("Password protected PDF saved");
    }
}

How Do I Benchmark IronPDF After Migration?

Run this harness after migration and compare against your ComPDFKit baseline numbers. It measures both sequential and parallel throughput so you can see how concurrency affects render time.

IronPDF Benchmark Harness

using System;
using System.Collections.Generic;
using System.Diagnostics;
using System.Threading.Tasks;
using IronPdf;

class IronPdfBenchmark
{
    // Reuse renderer — important for fair throughput measurement
    private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

    static async Task Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        int iterations = 50;

        // Sequential benchmark
        Console.WriteLine($"Sequential HTML-to-PDF ({iterations} iterations)...");
        var sw = Stopwatch.StartNew();
        for (int i = 0; i < iterations; i++)
        {
            using var pdf = await _renderer.RenderHtmlAsPdfAsync(GetHtml(i));
            // Don't save to disk — measures render time only
        }
        sw.Stop();
        double seqAvg = (double)sw.ElapsedMilliseconds / iterations;
        Console.WriteLine($"Sequential avg: {seqAvg:F1} ms/render");

        // Parallel benchmark
        Console.WriteLine($"\nParallel HTML-to-PDF ({iterations} iterations, 4-way)...");
        sw.Restart();
        var tasks = new List<Task>();
        for (int i = 0; i < iterations; i++)
        {
            int j = i;
            tasks.Add(Task.Run(async () =>
            {
                using var pdf = await _renderer.RenderHtmlAsPdfAsync(GetHtml(j));
            }));
            if (tasks.Count >= 4)
            {
                await Task.WhenAll(tasks);
                tasks.Clear();
            }
        }
        if (tasks.Count > 0) await Task.WhenAll(tasks);
        sw.Stop();
        double parAvg = (double)sw.ElapsedMilliseconds / iterations;
        Console.WriteLine($"Parallel avg: {parAvg:F1} ms/render effective");
    }

    static string GetHtml(int i) => $@"<html>
        <body style='font-family:Arial'>
            <h1>Document #{i}</h1>
            <p>Generated at {DateTime.Now:O}</p>
        </body>
    </html>";
}

Compare seqAvg and parAvg against your ComPDFKit baseline measurements.

What Are the Most Common Feature Fit Questions?

  • If you need an interactive PDF viewer or editor inside a desktop or mobile app, ComPDFKit is purpose-built for that — IronPDF is not a UI viewer.
  • If you need server-side HTML-to-PDF batch generation, IronPDF's Chromium renderer is a better fit.
  • If your workload is API-driven PDF operations (merge, stamp, encrypt), both work; IronPDF's API is more concise.
  • If you have a mixed viewer + server-generation workload, you may need both libraries — weigh consolidation cost against the complexity of two dependencies.

What Critical Details Should I Watch for During Migration?

ComPDFKit uses an SDK initialization step (loading native libraries, verifying license). IronPDF's license is a static string set once. The initialization model is simpler but the pattern in web applications differs:

License Key in Web Apps

// Startup.cs / Program.cs — set once, early
IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_KEY");

Don't set the license key inside hot paths or per-request code.

Audit every document.Release() and page.Release() in your codebase — IronPDF handles disposal through the GC, so these calls become no-ops at best and confusing dead code at worst. Delete them.

Both libraries use 0-based page indexing, so document.PageAtIndex(i) maps cleanly to pdf.Pages[i] without re-numbering math.

If your application creates annotations and expects them to survive PDF round-trips (save, reload, verify), test annotation persistence with IronPDF before committing to the migration. Annotation formats are standardized in PDF, but library implementations differ in which annotation types and properties they preserve — this is the genuinely unresolved area, since ComPDFKit's annotation model is richer than IronPDF's.

How Does IronPDF Perform Under Load?

IronPDF spawns a Chromium renderer process. This process startup is the largest latency contributor for the first render. Reusing ChromePdfRenderer across requests amortizes this cost. In a web application, register it as a singleton.

Run a memory profile under your target concurrent load. The Chromium renderer's memory behavior under concurrent requests is different from an in-process renderer. Test at your expected peak concurrency, not just sequentially.

Disk vs Memory Output

If your downstream process consumes the PDF as bytes (upload to S3, email attachment), avoid writing to disk:

using var pdf = renderer.RenderHtmlAsPdf(html);
byte[] bytes = pdf.BinaryData; // No disk write

// Or stream directly
using var stream = new MemoryStream();
pdf.Stream.CopyTo(stream);

See memory stream docs for patterns. See async rendering docs for patterns that scale under load.

Frequently Asked Questions

Can I run ComPDFKit and IronPDF side by side during migration?

Yes. Install IronPDF via NuGet alongside ComPDFKit and run both in parallel. This lets you capture benchmark numbers from both libraries against your actual workload before committing to the switch. Keep ComPDFKit until your IronPDF integration passes all tests.

Will my ComPDFKit annotations survive the migration?

Annotation formats are standardized in the PDF spec, but library implementations differ in which annotation types and properties they preserve. If your application creates annotations and expects them to survive round-trips (save, reload, verify), test annotation persistence with IronPDF before committing. ComPDFKit's annotation model is rich — this is the highest-effort area of the migration.

Does IronPDF support Linux and cloud deployment?

IronPDF runs on Windows, Linux, and macOS. For cloud targets like Azure App Service or AWS Lambda, IronPDF is distributed entirely via NuGet — no native binary distribution outside of NuGet is required.

How do I handle the Chromium renderer's first-render latency?

The Chromium renderer process startup is the largest latency contributor for the first render. Reuse ChromePdfRenderer across requests by registering it as a singleton in your DI container. In the benchmark harness, the renderer is created once and reused across all iterations — this is the pattern you want in production.

What if my use case is primarily a PDF viewer or editor?

IronPDF is not a UI viewer. If your primary use case is an interactive PDF viewer/editor application, evaluate carefully — ComPDFKit is designed for exactly this scenario. If you have a mixed workload (viewer plus server-side generation), you may need both libraries. Evaluate consolidation cost against the complexity of maintaining two dependencies.

That's the Migration

The migration from ComPDFKit to IronPDF is most straightforward when your use case is server-side generation — HTML-to-PDF, merge, stamp, encrypt. The rougher edges are annotation workflows and any code that depended on ComPDFKit's native rendering pipeline directly.

The benchmark harness matters here more than in most migrations because the performance characteristics of a Chromium-backed renderer versus a native PDF SDK are architecturally different. Don't assume — measure.
