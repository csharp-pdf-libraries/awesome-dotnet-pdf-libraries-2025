Migrating from PeachPDF to IronPDF: Solving Common Migration Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Would You Migrate Away from PeachPDF?

The .NET upgrade was supposed to be the boring part. Update the TFM, fix a few deprecated APIs, run the tests — done. But somewhere in that process, the PDF library started throwing exceptions it didn't throw before, or you discovered it doesn't have a NuGet package targeting the new framework version you need.

PeachPDF is a capable HTML-to-PDF library, but .NET version compatibility becomes a constraint when you're moving from .NET Framework or early .NET Core to the modern unified platform. I've seen teams hit this wall repeatedly, and the reasons for migrating tend to cluster around the same set of conditions:

.NET version incompatibility — the library doesn't publish a package for .NET 6/7/8/9, or the package exists but has runtime issues under the new TFM. Rendering engine age — non-Chromium HTML renderers often lag modern CSS capabilities. CSS fidelity gaps — flex layouts, CSS Grid, custom properties, or @media print rules don't render correctly.

Maintenance uncertainty — infrequent NuGet updates suggest the project is not actively tracking .NET releases. Cross-platform limitations — if the library has Windows-only native dependencies, Linux/Docker deployments are blocked. Missing manipulation features — no native merge, split, watermark, or security APIs in the library.

Async API support — older libraries often expose only synchronous APIs, which creates friction in async .NET code. Community support — smaller libraries have fewer Stack Overflow answers and GitHub issues for troubleshooting. Thread safety concerns — rendering under concurrent load may not be well-documented or tested.

What Does a .NET Version Incompatibility Look Like in Practice?

This is the most common trigger I see. Your team upgrades to .NET 8 or 9, and PeachPDF either doesn't have a compatible package or throws runtime exceptions under the new TFM. The build fails, and PeachPDF becomes the blocker for your entire upgrade.

The first step is finding every PeachPDF reference in your codebase.

Searching for PeachPDF API Usage

# Search for PeachPDF API usage — verify the actual namespace
rg -l "PeachPDF\|Peach\.Pdf\|PeachPdf" --type cs
rg "PeachPDF\|PeachPdf" --type cs -n

# Check project files
grep -r "PeachPDF\|PeachPdf" *.csproj **/*.csproj 2>/dev/null

# Search for HTML template strings that feed the PDF library
rg "RenderHtml\|ConvertHtml\|HtmlToPdf" --type cs -n

# Find any rendering options or configuration classes
rg "PageSize\|PaperSize\|Margin\|PdfOptions" --type cs -n

Once you know the scope, swap the packages.

Removing PeachPDF and Installing IronPDF

# Remove PeachPDF — verify exact package name at NuGet
dotnet remove package PeachPDF  # verify package name

# Install IronPDF
dotnet add package IronPdf

dotnet restore

If PeachPDF was blocking dotnet build on your new TFM, verify the build succeeds after this swap before writing any migration code. Then configure your IronPDF license key.

How Do You Handle CSS That Renders Differently After the Switch?

This is the second most common pain point. HTML that rendered acceptably in PeachPDF looks different in IronPDF because the rendering engines are fundamentally different. IronPDF uses Chromium; PeachPDF's engine is unknown. The new rendering may actually be more correct for modern CSS, but it can look different.

The debugging approach I recommend starts with Chrome DevTools.

Debugging CSS Rendering Differences

// Step 1: Test your HTML directly in Chrome DevTools print preview
// IronPDF's Chromium renders CSS as Chrome does with @media print rules

// Step 2: Add explicit @media print styles for predictable output
var html = @"
    <html>
    <head>
    <style>
        /* Screen styles */
        body { font-family: Arial; }

        /* Print-specific — IronPDF applies these via Chromium */
        @media print {
            .no-print { display: none; }
            .page-break { page-break-before: always; }
            table { page-break-inside: avoid; }
        }
    </style>
    </head>
    <body>...</body>
    </html>";

// Step 3: Adjust rendering options to match desired output
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
// See: https://ironpdf.com/how-to/rendering-options/
// See: https://ironpdf.com/how-to/pixel-perfect-html-to-pdf/

Another common issue: images, CSS files, or fonts referenced in the HTML don't appear in the PDF. Relative paths in HTML may not resolve correctly when rendering from a string.

Fixing External Resources That Don't Load

using IronPdf;
using System;
using System.IO;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

// Option 1: Set base URL so relative paths resolve correctly
// Verify BaseUrlPath / BaseUrl property name in current IronPDF docs
renderer.RenderingOptions.BaseUrlPath = new DirectoryInfo(
    Path.GetFullPath("./templates")
).FullName;

// Option 2: Use absolute URLs in the HTML
var html = @"
    <html>
    <head>
    <link rel='stylesheet' href='https://cdn.example.com/styles.css'>
    </head>
    <body>
        <img src='https://cdn.example.com/logo.png' alt='logo'>
    </body>
    </html>";

// Option 3: Embed resources as base64
// var logoBase64 = Convert.ToBase64String(File.ReadAllBytes("logo.png"));
// var html = $"<img src='data:image/png;base64,{logoBase64}'>";

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("output.pdf");

What Are the Most Common Questions During a PeachPDF to IronPDF Migration?

What if my build fails after swapping packages?

Answer:

The most common cause is a missing license key. IronPDF throws at runtime, not compile time, when the key is not set. Check that the license key is configured before any IronPDF API call.

using IronPdf;

// Check that the license key is set before any IronPDF API call
// A missing or invalid key will throw at runtime, not compile time
if (string.IsNullOrEmpty(IronPdf.License.LicenseKey))
{
    IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY")
        ?? throw new InvalidOperationException("Set IRONPDF_LICENSE_KEY environment variable");
}

// Validate: https://ironpdf.com/how-to/license-keys/
Console.WriteLine($"License valid: {IronPdf.License.IsValidLicense}");

If PeachPDF was the build blocker on your new TFM, verify the build succeeds after the package swap before writing any migration code.

How do I handle high memory usage under sustained rendering load?

Answer:

Always use using on PdfDocument — IronPDF wraps native Chromium resources that need timely disposal. For high-volume scenarios, process in batches.

using IronPdf;
using System;
using System.Collections.Generic;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Always use 'using' on PdfDocument — IronPDF wraps native Chromium resources
static async Task ProcessBatch(IEnumerable<string> htmlDocuments)
{
    var renderer = new ChromePdfRenderer();

    foreach (var html in htmlDocuments)
    {
        // 'using' ensures timely disposal of Chromium resources
        using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
        pdf.SaveAs($"output_{Guid.NewGuid():N}.pdf");
        // pdf disposed here — native resources released
    }
}

// For high-volume: consider processing in batches
// and explicitly calling GC.Collect() between batches if needed

Should I create one ChromePdfRenderer per request or reuse a single instance?

Answer:

Reusing a single renderer avoids repeated Chromium initialization overhead. For concurrent load, verify thread safety in the IronPDF documentation.

using IronPdf;
using System.Threading.Tasks;

// Option 1: Reuse a single renderer (verify thread safety in IronPDF docs)
public class PdfService
{
    private readonly ChromePdfRenderer _renderer;

    public PdfService()
    {
        IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");
        _renderer = new ChromePdfRenderer();
        // Configure once
        _renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
    }

    public async Task<byte[]> GeneratePdfAsync(string html)
    {
        using var pdf = await _renderer.RenderHtmlAsPdfAsync(html);
        return pdf.BinaryData;
    }
}

// Option 2: Per-request renderer (thread-isolated but slower to initialize)
// Choose based on IronPDF's thread-safety documentation and your load profile
// https://ironpdf.com/how-to/async/

How do I deploy IronPDF to Azure or cloud environments?

Answer:

IronPDF works locally but may throw exceptions in Azure App Service, Azure Functions, or similar cloud environments. Set Azure App Service to 64-bit, and verify that the hosting plan supports enough memory for Chromium rendering (Basic tier or above is typically needed).

// IronPDF has specific guidance for Azure deployments

How do I warm up IronPDF to avoid cold-start latency?

Answer:

Trigger Chromium initialization at application startup using a hosted service. This avoids cold-start latency on the first real request.

using IronPdf;
using Microsoft.Extensions.Hosting;
using System.Threading;
using System.Threading.Tasks;

// Warm up during application startup — avoids cold-start latency on first request
public class PdfWarmupService : IHostedService
{
    public async Task StartAsync(CancellationToken cancellationToken)
    {
        IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

        // Trigger Chromium initialization at startup
        var renderer = new ChromePdfRenderer();
        using var _ = await renderer.RenderHtmlAsPdfAsync("<html><body>warmup</body></html>");

        // Renderer is now warm for production traffic
    }

    public Task StopAsync(CancellationToken cancellationToken) => Task.CompletedTask;
}

When Is the Right Time to Migrate from PeachPDF to IronPDF?

The .NET version upgrade path is often the clearest migration case — there's a hard compatibility constraint, not a preference. If PeachPDF doesn't publish a package for your target framework, the decision is made for you.

Here is how I think about the decision matrix.

Business ScenarioRecommendation
.NET 6/7/8/9 compatibility required and PeachPDF doesn't support itSwitch — version compatibility is a hard constraint
CSS rendering fidelity is criticalSwitch — Chromium renders current CSS standards
All usage is basic HTML-to-PDF with no manipulation needsEvaluate carefully — simpler libraries may be sufficient
Cross-platform or Docker deployment neededVerify PeachPDF platform support; IronPDF supports Linux

The main work after the swap is verifying that the rendering output is equivalent. One practical note: if your .NET upgrade also includes moving from .NET Framework to a modern TFM, test IronPDF on Linux as early as possible — it's faster to catch platform issues in a local Docker container than in a CI pipeline.

IronPDF uses 0-based page indexing. If PeachPDF used 1-based indexing and you have page manipulation code, audit all page index references. And don't forget to check for secondary libraries you added to fill PeachPDF's gaps — after migrating to IronPDF, those secondary packages for merge, watermark, or security can typically be removed.
