Migrating from PeachPDF to IronPDF: Solving Common Migration Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Would You Migrate Away from PeachPDF?

The .NET upgrade was supposed to be the boring part. Update the TFM, fix a few deprecated APIs, run the tests — done. But somewhere in that process, the PDF library started throwing exceptions it didn't throw before, or you discovered it doesn't have a NuGet package targeting the new framework version you need.

PeachPDF is a newer, single-maintainer C#-native HTML-to-PDF library (BSD-3-Clause, currently at 0.7.26 as of October 2025). It targets net8.0 only, and its rendering layer is built on PeachPDF.PdfSharpCore — a managed layout engine, not a browser. The reasons for migrating tend to cluster around the same set of conditions:

.NET version incompatibility — PeachPDF publishes only a net8.0 target, so .NET Framework, .NET 6, and .NET 7 projects either upgrade or pick a different library. Rendering engine age — a managed HTML/CSS layout engine does not match Chromium's coverage of modern CSS. CSS fidelity gaps — flex layouts, CSS Grid, custom properties, and @media print rules render to a subset and may not match what the same markup looks like in Chrome.

Maturity — PeachPDF is pre-1.0 with a single GitHub maintainer (jhaygood86); IronPDF has been in production for years with 40M+ downloads. Missing manipulation features — PeachPDF is HTML-to-PDF only. There is no API to load an existing PDF, merge, split, watermark, sign, or password-protect; teams that need any of those today reach for a secondary library like PdfSharpCore alongside PeachPDF.

No JavaScript engine — scripts and SPA-style content do not execute, so dashboards, charts rendered client-side, and any HTML that depends on scripts will not render. Enterprise support — there is no commercial SLA or paid support channel for PeachPDF. Documentation depth — the project's documentation is essentially the README and the source.

What Does a .NET Version Incompatibility Look Like in Practice?

This is the most common trigger I see. Your team upgrades to .NET 9 (or you have a .NET 6/7 project that needs to stay on its TFM), and PeachPDF only ships net8.0. Either the package doesn't restore, or it restores but you cannot stay on the older TFM you intended.

The first step is finding every PeachPDF reference in your codebase.

Searching for PeachPDF API Usage

# Search for PeachPDF API usage — root namespace is `PeachPDF`,
# entry points are `PdfGenerator` and `PdfGenerateConfig`
rg -l "PeachPDF" --type cs
rg "PdfGenerator|PdfGenerateConfig|GeneratePdf\(" --type cs -n

# Check project files
rg "PeachPDF" -g "*.csproj"

# Find network-adapter usage (URL-to-PDF wiring)
rg "HttpClientNetworkAdapter|MimeKitNetworkAdapter|NetworkAdapter\s*=" --type cs -n

Once you know the scope, swap the packages.

Removing PeachPDF and Installing IronPDF

# Remove PeachPDF
dotnet remove package PeachPDF

# Install IronPDF
dotnet add package IronPdf

dotnet restore

If PeachPDF was blocking dotnet build on your new TFM, verify the build succeeds after this swap before writing any migration code. Then configure your IronPDF license key.

License Configuration

using IronPdf;

// Set at application startup — https://ironpdf.com/how-to/license-keys/
// Set via environment variable (recommended for server deployments):
// export IRONPDF_LICENSE_KEY=your-key-here

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY")
    ?? throw new InvalidOperationException("IRONPDF_LICENSE_KEY environment variable not set");

Namespace Swap — Before

using PeachPDF;
using PeachPDF.PdfSharpCore;

Namespace Swap — After

using IronPdf;
using IronPdf.Rendering;

Basic Conversion — Before

using PeachPDF;
using PeachPDF.PdfSharpCore;
using System.IO;
using System.Threading.Tasks;

class PeachPdfBefore
{
    static async Task Main()
    {
        var html = "<html><body><h1>Hello</h1></body></html>";

        var config = new PdfGenerateConfig
        {
            PageSize = PageSize.Letter,
            PageOrientation = PageOrientation.Portrait
        };

        var generator = new PdfGenerator();
        var document = await generator.GeneratePdf(html, config);

        using var stream = File.Create("output.pdf");
        document.Save(stream);
    }
}

Basic Conversion — After

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Hello</h1></body></html>");
pdf.SaveAs("output.pdf");

Console.WriteLine($"Saved output.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

How Do You Handle CSS That Renders Differently After the Switch?

This is the second most common pain point. HTML that rendered acceptably in PeachPDF looks different in IronPDF because the rendering engines are fundamentally different. IronPDF uses Chromium; PeachPDF is a newer managed C# HTML renderer that does not match Chromium's coverage of modern CSS. The new rendering may actually be more correct for modern CSS, but it can look different.

The debugging approach I recommend starts with Chrome DevTools.

Debugging CSS Rendering Differences

// Test your HTML in Chrome DevTools print preview first —
// IronPDF's Chromium renders CSS the same way Chrome does, including @media print

// Add explicit @media print styles for predictable output
var html = @"
    <html>
    <head>
    <style>
        /* Screen styles */
        body { font-family: Arial; }

        /* Print-specific — IronPDF applies these via Chromium */
        @media print {
            .no-print { display: none; }
            .page-break { page-break-before: always; }
            table { page-break-inside: avoid; }
        }
    </style>
    </head>
    <body>...</body>
    </html>";

// Adjust rendering options to match desired output
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
// See: https://ironpdf.com/how-to/rendering-options/
// See: https://ironpdf.com/how-to/pixel-perfect-html-to-pdf/

Another common issue: images, CSS files, or fonts referenced in the HTML don't appear in the PDF. Relative paths in HTML may not resolve correctly when rendering from a string.

Fixing External Resources That Don't Load

using IronPdf;
using System;
using System.IO;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

// Option 1: Set base URL so relative paths resolve correctly
renderer.RenderingOptions.BaseUrlPath = new DirectoryInfo(
    Path.GetFullPath("./templates")
).FullName;

// Option 2: Use absolute URLs in the HTML
var html = @"
    <html>
    <head>
    <link rel='stylesheet' href='https://cdn.example.com/styles.css'>
    </head>
    <body>
        <img src='https://cdn.example.com/logo.png' alt='logo'>
    </body>
    </html>";

// Option 3: Embed resources as base64
// var logoBase64 = Convert.ToBase64String(File.ReadAllBytes("logo.png"));
// var html = $"<img src='data:image/png;base64,{logoBase64}'>";

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("output.pdf");

What About Merging, Watermarks, and Security — Features PeachPDF May Not Have?

A common discovery during migration: a secondary library was added for watermarking, security, or merge because PeachPDF didn't support those features. After migrating to IronPDF, those secondary packages can typically be removed.

Merging PDFs — Before

using System;

// PeachPDF is HTML-to-PDF only. PdfGenerator only emits a freshly generated
// document; there is no API to load existing PDFs and combine them. Teams
// that need merge typically drop down to PdfSharpCore directly or add a
// secondary library (PdfPig, etc.) alongside PeachPDF.

class MergeBefore
{
    static void Main()
    {
        // Typical workaround pattern with a secondary library:
        // var pdf1 = PdfReader.Open("part1.pdf", PdfDocumentOpenMode.Import);
        // var pdf2 = PdfReader.Open("part2.pdf", PdfDocumentOpenMode.Import);
        // var output = new PdfDocument();
        // foreach (var p in pdf1.Pages) output.AddPage(p);
        // foreach (var p in pdf2.Pages) output.AddPage(p);
        // output.Save("combined.pdf");

Merging PDFs — After

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var pdf1 = PdfDocument.FromFile("part1.pdf");
var pdf2 = PdfDocument.FromFile("part2.pdf");
var pdf3 = PdfDocument.FromFile("part3.pdf");

// Merge: https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(pdf1, pdf2, pdf3);
merged.SaveAs("combined.pdf");

Console.WriteLine($"Merged: {merged.PageCount} pages");
pdf1.Dispose(); pdf2.Dispose(); pdf3.Dispose();

Watermark — Before (CSS-Based Workaround)

using PeachPDF;
using PeachPDF.PdfSharpCore;
using System.IO;
using System.Threading.Tasks;

// PeachPDF has no programmatic watermark/stamp API. The standard
// workaround is a CSS-positioned element baked into the source HTML
// before generating the PDF.

class WatermarkBefore
{
    static async Task Main()
    {
        var html = @"
            <html>
            <head>
            <style>
                .watermark {
                    position: fixed;
                    top: 50%;
                    left: 50%;
                    transform: translate(-50%, -50%) rotate(-45deg);
                    font-size: 80px;
                    color: rgba(150, 150, 150, 0.2);
                    z-index: 9999;
                    white-space: nowrap;
                    pointer-events: none;
                }
            </style>
            </head>
            <body>
                <div class='watermark'>INTERNAL USE ONLY</div>
                <h1>Document</h1>
            </body>
            </html>";

        var generator = new PdfGenerator();
        var document = await generator.GeneratePdf(html, new PdfGenerateConfig { PageSize = PageSize.Letter });

        using var stream = File.Create("watermarked.pdf");
        document.Save(stream);
    }
}

Watermark — After

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Document</h1><p>Content...</p></body></html>"
);

// Post-render programmatic watermark — consistent regardless of HTML content
// https://ironpdf.com/how-to/custom-watermark/
var watermark = new TextStamper
{
    Text = "INTERNAL USE ONLY",
    FontColor = IronSoftware.Drawing.Color.Gray,
    Opacity = 20, // 0–100 integer
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("internal-document.pdf");

How Does PeachPDF Compare to IronPDF Feature by Feature?

I find a side-by-side table is the fastest way to assess the scope of your migration. Here is how the two libraries compare across key dimensions.

  • Focus. PeachPDF is HTML-to-PDF only. IronPDF covers HTML-to-PDF plus full PDF manipulation (load, merge, split, stamp, sign, secure, edit).
  • Licensing. PeachPDF is BSD-3-Clause open source. IronPDF is commercial; pricing is at ironsoftware.com.
  • API style. PeachPDF splits an engine (PdfGenerator) from settings (PdfGenerateConfig) and emits a document whose Save takes a Stream. IronPDF uses a fluent ChromePdfRenderer + PdfDocument object model where SaveAs takes a path.
  • Learning curve. PeachPDF's surface is tiny — PdfGenerator.GeneratePdf plus a handful of PdfGenerateConfig properties. IronPDF is broader but maps cleanly to .NET conventions.
  • HTML rendering. PeachPDF runs a managed C#/PdfSharpCore HTML/CSS layout engine that does not match Chromium's coverage. IronPDF embeds Chromium.
  • JavaScript. PeachPDF does not execute scripts. IronPDF runs full Chromium and exposes wait hooks for SPA-style content.
  • Async. PeachPDF's GeneratePdf is async-only. IronPDF exposes both RenderHtmlAsPdf/RenderUrlAsPdf and async RenderHtmlAsPdfAsync/RenderUrlAsPdfAsync.
  • Target framework. PeachPDF ships net8.0 only. IronPDF supports .NET Framework 4.6.2+, .NET Core, .NET 5, and current .NET 6/7/8/9.
  • Namespace. PeachPDF → IronPdf.

And here is the feature-by-feature migration complexity breakdown.

  • HTML string to PDF — Low. await new PdfGenerator().GeneratePdf(html, config) maps directly to renderer.RenderHtmlAsPdfAsync(html).
  • URL to PDF — Low. The HttpClientNetworkAdapter + GeneratePdf(null, config) pattern collapses into renderer.RenderUrlAsPdfAsync(url).
  • Save to file — Low. document.Save(stream) against a FileStream becomes pdf.SaveAs(path).
  • Save to bytes / stream — Low. PeachPDF requires saving into a MemoryStream and calling ToArray(); IronPDF exposes pdf.BinaryData and pdf.Stream directly on PdfDocument.
  • Custom page size and margins — Low. PeachPDF's PdfGenerateConfig.PageSize / PageOrientation map to renderer.RenderingOptions.PaperSize / PaperOrientation; IronPDF adds explicit MarginTop / MarginBottom / MarginLeft / MarginRight.
  • Headers / footers — Medium. PeachPDF v0.7.x has no header/footer API at all; the markup is embedded in the source HTML with CSS positioning. IronPDF replaces that with first-class RenderingOptions.HtmlHeader / HtmlFooter using HtmlHeaderFooter objects and {page} / {total-pages} tokens.
  • JavaScript rendering — Medium. PeachPDF's layout engine does not execute scripts. IronPDF runs Chromium and exposes RenderingOptions.WaitFor hooks for SPAs.
  • Load existing PDF — High (no equivalent). PeachPDF cannot open existing PDFs; teams using it for reading or round-tripping are using PdfSharpCore directly. IronPDF exposes PdfDocument.FromFile / FromBinaryData.
  • Merge — Medium. PeachPDF has no native merge; this typically lives in a PdfSharpCore call today and consolidates onto PdfDocument.Merge(...).
  • Watermark — Medium. PeachPDF requires a CSS-positioned element in the source HTML; IronPDF uses programmatic TextStamper / ImageStamper.
  • Password protection — Medium. PeachPDF has no native security API; IronPDF exposes pdf.SecuritySettings.

What Are the Most Common Questions During a PeachPDF to IronPDF Migration?

What if my build fails after swapping packages?

The most common cause is a missing license key. IronPDF throws at runtime, not compile time, when the key is not set. Check that the license key is configured before any IronPDF API call.

using IronPdf;

// Check that the license key is set before any IronPDF API call
// A missing or invalid key will throw at runtime, not compile time
if (string.IsNullOrEmpty(IronPdf.License.LicenseKey))
{
    IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY")
        ?? throw new InvalidOperationException("Set IRONPDF_LICENSE_KEY environment variable");
}

// Validate: https://ironpdf.com/how-to/license-keys/
Console.WriteLine($"License valid: {IronPdf.License.IsValidLicense}");

If PeachPDF was the build blocker on your new TFM, verify the build succeeds after the package swap before writing any migration code.

How do I handle high memory usage under sustained rendering load?

Always use using on PdfDocument — IronPDF wraps native Chromium resources that need timely disposal. For high-volume scenarios, process in batches.

using IronPdf;
using System;
using System.Collections.Generic;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Always use 'using' on PdfDocument — IronPDF wraps native Chromium resources
static async Task ProcessBatch(IEnumerable<string> htmlDocuments)
{
    var renderer = new ChromePdfRenderer();

    foreach (var html in htmlDocuments)
    {
        // 'using' ensures timely disposal of Chromium resources
        using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
        pdf.SaveAs($"output_{Guid.NewGuid():N}.pdf");
        // pdf disposed here — native resources released
    }
}

// For high-volume: consider processing in batches
// and explicitly calling GC.Collect() between batches if needed

Should I create one ChromePdfRenderer per request or reuse a single instance?

Reusing a single renderer avoids repeated Chromium initialization overhead. For concurrent load, verify thread safety in the IronPDF documentation.

using IronPdf;
using System.Threading.Tasks;

// Option 1: Reuse a single renderer (verify thread safety in IronPDF docs)
public class PdfService
{
    private readonly ChromePdfRenderer _renderer;

    public PdfService()
    {
        IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");
        _renderer = new ChromePdfRenderer();
        // Configure once
        _renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
    }

    public async Task<byte[]> GeneratePdfAsync(string html)
    {
        using var pdf = await _renderer.RenderHtmlAsPdfAsync(html);
        return pdf.BinaryData;
    }
}

// Option 2: Per-request renderer (thread-isolated but slower to initialize)
// Choose based on IronPDF's thread-safety documentation and your load profile
// https://ironpdf.com/how-to/async/

How do I deploy IronPDF to Azure or cloud environments?

IronPDF works locally but may throw exceptions in Azure App Service, Azure Functions, or similar cloud environments. Set Azure App Service to 64-bit, and verify that the hosting plan supports enough memory for Chromium rendering (Basic tier or above is typically needed).

// IronPDF has specific guidance for Azure deployments
// https://ironpdf.com/how-to/azure/

// Common requirement: set Azure App Service to 64-bit
// Common requirement: verify that the hosting plan supports enough memory
// for Chromium rendering (Basic tier or above is typically needed)

// Cloud-safe initialization pattern — see Azure docs linked above for current recommendations
var renderer = new ChromePdfRenderer();

How do I warm up IronPDF to avoid cold-start latency?

Trigger Chromium initialization at application startup using a hosted service. This avoids cold-start latency on the first real request.

using IronPdf;
using Microsoft.Extensions.Hosting;
using System.Threading;
using System.Threading.Tasks;

// Warm up during application startup — avoids cold-start latency on first request
public class PdfWarmupService : IHostedService
{
    public async Task StartAsync(CancellationToken cancellationToken)
    {
        IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

        // Trigger Chromium initialization at startup
        var renderer = new ChromePdfRenderer();
        using var _ = await renderer.RenderHtmlAsPdfAsync("<html><body>warmup</body></html>");

        // Renderer is now warm for production traffic
    }

    public Task StopAsync(CancellationToken cancellationToken) => Task.CompletedTask;
}

When Is the Right Time to Migrate from PeachPDF to IronPDF?

The .NET version upgrade path is often the clearest migration case — there's a hard compatibility constraint, not a preference. If PeachPDF doesn't publish a package for your target framework, the decision is made for you.

Here is how I think about the decision matrix.

  • If .NET 6/7/8/9 compatibility is required and PeachPDF doesn't support your TFM, then switch — version compatibility is a hard constraint, not a preference.
  • If CSS rendering fidelity is critical, then switch — Chromium handles current CSS (Grid, Flexbox, custom properties, modern @media print) more completely than PeachPDF's managed engine.
  • If all usage is basic HTML-to-PDF with no manipulation needs, then evaluate carefully — a smaller library may be sufficient if rendering quality is acceptable.
  • If cross-platform or Docker deployment is required, then confirm PeachPDF works on your target Linux runtime; IronPDF officially supports Linux, macOS, and Windows.

The main work after the swap is verifying that the rendering output is equivalent. One practical note: if your .NET upgrade also includes moving from .NET Framework to a modern TFM, test IronPDF on Linux as early as possible — it's faster to catch platform issues in a local Docker container than in a CI pipeline.

IronPDF uses 0-based page indexing. PeachPDF does not expose per-page operations on existing PDFs at all, so any page manipulation in your codebase came from a secondary library — audit those index references when you consolidate. And don't forget to check for the secondary libraries you added to fill PeachPDF's gaps — after migrating to IronPDF, those packages for merge, watermark, or security can typically be removed.
