Migrating from PDFFilePrint to IronPDF: Practical .NET 8 Migration Notes

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Your team just committed to .NET 8 and the dependency audit came back with a flag on PDFFilePrint. The package targets .NET Framework 4.6.1 and the last NuGet release was 1.0.3 in February 2020. You have a working print library on a runtime you are leaving behind — that is the migration trigger, and this guide is for the period immediately after you have made the call to switch.

PDFFilePrint is a small open-source MIT wrapper around PdfiumViewer and Google Pdfium. Its job is narrow: take an existing PDF or XPS file and silently send it to a printer driver. It does not create PDFs, render HTML, fetch URLs, merge documents, or expose page-range and duplex configuration as a strongly-typed API. The troubleshooting patterns here apply broadly to any migration triggered by a runtime incompatibility, but the API translations below are specific to what PDFFilePrint actually does.

How Do I Diagnose the .NET Compatibility Failure?

Before migrating, confirm what is actually failing. .NET incompatibility manifests in several distinct ways, and I have seen teams start rewriting code when the real problem is a configuration issue.

Symptom: NuGet restore fails

# Check which .NET targets the package supports
dotnet nuget show package PDFFilePrint --source nuget.org

# Or check the NuGet page manually:
# https://www.nuget.org/packages/PDFFilePrint
# Look at "Supported frameworks" — verify .NET 8 is listed

Symptom: Package installs but throws PlatformNotSupportedException at runtime

# Check if the package depends on System.Drawing.Common
dotnet list package --include-transitive | grep -i drawing

# System.Drawing.Common on .NET 7+ Linux throws by default
# This affects packages that haven't updated their rendering path

Symptom: Build succeeds, runtime fails with missing method or type

# Check for binding redirects or multi-targeting issues
dotnet build --verbosity detailed 2>&1 | grep -i "warning\|error\|NU\d\{4\}"

# Also check for TFM mismatch warnings:
dotnet build 2>&1 | grep "NU1701\|NU1202\|NU1213"

Once you have confirmed the failure mode is genuine incompatibility (not configuration), proceed with migration.

Why Should I Migrate from PDFFilePrint?

Eight reasons a .NET version incompatibility becomes a migration trigger rather than a “wait for the maintainer” situation:

.NET 8 LTS timeline — if you are on the LTS track, staying on .NET Framework 4.6.1 to preserve a library is a security and support trade-off.

Stalled maintenance — PDFFilePrint 1.0.3 shipped in February 2020 and has not published a release since. There is no public source repository linked from the NuGet page.

Windows-only by construction — PDFFilePrint sits on PdfiumViewer's Win32 native binaries and the Windows printing subsystem. Linux and macOS targets are not on the table.

Native binary compatibility — print-driver and OS-dependent libraries may have broken native interop on new runtimes.

Feature gap at upgrade time — the incompatibility surfaces at upgrade time, creating a natural evaluation window.

No LTS support tier — open source libraries without a support SLA leave you waiting indefinitely on a fix that may never ship.

Secondary library consolidation — PDFFilePrint only prints. Teams that also need HTML-to-PDF, merge, or watermarking pull in a second library to cover those gaps. A forced migration is a chance to consolidate.

Configuration model — PDFFilePrint reads PrinterName, PaperName, Copies, PrintToFile, and DefaultPrintToDirectory from app.config via Properties.Settings.Default. Modern .NET deployments prefer strongly-typed options objects passed per call.

How Do PDFFilePrint and IronPDF Compare Feature by Feature?

  • Focus. PDFFilePrint is a print-only .NET library with a single FilePrint class that takes an existing PDF or XPS file and sends it to a printer. IronPDF is a full HTML-to-PDF and PDF manipulation toolkit that also handles printing.
  • Pricing. PDFFilePrint is MIT-licensed, low-traffic, and has no published commercial support tier — treat it as community-maintained. IronPDF is commercial; pricing at ironsoftware.com.
  • API style. PDFFilePrint exposes one class with a constructor that takes a PDF path and an optional XPS output path, plus a Print() method. Configuration lives in app.config. IronPDF separates rendering (ChromePdfRenderer) from the document model (PdfDocument) and accepts a System.Drawing.Printing.PrinterSettings instance per print call.
  • Learning curve. PDFFilePrint is trivial to learn because the surface is one class and a handful of config keys. IronPDF is medium — more types, but each one is purpose-built.
  • HTML rendering. PDFFilePrint does not render HTML at all; teams using it for HTML-to-print workflows pair it with a separate renderer. IronPDF uses Chromium — the same engine as Chrome — so HTML and CSS render the way modern browsers render them.
  • Page indexing. PDFFilePrint does not expose page-level operations; the printer driver handles the page concept. IronPDF is 0-based for PdfDocument.Pages.
  • Thread safety. Both libraries are best treated as per-job for printing. IronPDF is additionally designed for ChromePdfRenderer instance reuse across a rendering batch.
  • Namespace. PDFFilePrint → IronPdf, plus System.Drawing.Printing for printer settings.

How Hard Is Each Feature to Migrate?

  • Basic printing — Low. new FilePrint(pdfPath, null).Print() maps to PdfDocument.FromFile(pdfPath).Print(). Same two-step shape.
  • Named printer / copies — Low. Mutating Properties.Settings.Default before calling Print() becomes constructing a System.Drawing.Printing.PrinterSettings and passing it to pdf.GetPrintDocument(settings).Print().
  • Page range — Low (and a feature gain). PDFFilePrint has no page-range API; you would pre-split the PDF or rely on the driver default. IronPDF exposes PrinterSettings.FromPage / ToPage with PrintRange.SomePages.
  • Duplex / orientation — Low (and a feature gain). Same story — PDFFilePrint defers to the driver, IronPDF exposes PrinterSettings.Duplex and DefaultPageSettings.Landscape.
  • HTML-to-PDF — new capability. PDFFilePrint cannot do this at all. renderer.RenderHtmlAsPdf(html) in IronPDF replaces whatever secondary tool you were chaining together.
  • URL-to-PDF — new capability. Same story. renderer.RenderUrlAsPdf(url).
  • Merge PDFs — new capability. Native in IronPDF via PdfDocument.Merge.
  • Watermark — new capability. TextStamper + ApplyStamp in IronPDF.
  • Password protection — new capability. SecuritySettings in IronPDF.
  • Async rendering — Medium. RenderHtmlAsPdfAsync exists in IronPDF; PDFFilePrint is synchronous only.
  • .NET 8 compatibility — resolved in IronPDF. If your migration trigger is a runtime upgrade, this row is the whole point.

When Does It Make Sense to Migrate?

  • If your primary use is sending an existing PDF to a printer, IronPDF is a direct fit — same call shape, more configurability via PrinterSettings.
  • If you are already pairing PDFFilePrint with another library to render HTML or URLs to PDF before printing, IronPDF's Chromium-based ChromePdfRenderer collapses that two-step pipeline into one package.
  • If your workflow extends to merge, watermark, or encrypt operations that PDFFilePrint cannot do at all, and you are pulling in PdfSharp or iTextSharp alongside it, IronPDF consolidates the stack.
  • If a .NET 8 upgrade is not negotiable and PDFFilePrint has not shipped a compatible release, migration is required — use the window to evaluate alternatives once.

How Do I Migrate in 3 Quick Steps?

License configuration

Before (PDFFilePrint):

// PDFFilePrint has no license-key concept — it is MIT-licensed
// and the FilePrint class is used directly.
using PDFFilePrint;

After (IronPDF):

using IronPdf;

// Set once at application startup
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
// License guide: https://ironpdf.com/how-to/license-keys/

Namespace imports

Before:

using PDFFilePrint;

After:

using IronPdf;
using System.Drawing.Printing;

Basic PDF print

Before (PDFFilePrint):

using System;
using PDFFilePrint;

class BasicExample
{
    static void Main()
    {
        // PrinterName, PaperName, Copies are read from app.config
        // (Properties.Settings.Default). The second constructor argument
        // is the XPS / output filename when PrintToFile is enabled, or null.
        var fileprint = new FilePrint("document.pdf", null);
        fileprint.Print();
    }
}

After (IronPDF):

using IronPdf;
using System.Drawing.Printing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("document.pdf");

// Default printer:
pdf.Print();

// Or with strongly-typed PrinterSettings:
var settings = new PrinterSettings { PrinterName = "HP LaserJet Pro", Copies = 1 };
pdf.GetPrintDocument(settings).Print();
// Guide: https://ironpdf.com/docs/questions/print-pdf/

How Do PDFFilePrint APIs Map to IronPDF?

Namespace mapping

  • PDFFilePrint → IronPdf — core namespace.
  • System.Configuration (Properties.Settings.Default) → System.Drawing.Printing — PrinterSettings replaces app.config keys.

Core class mapping

  • FilePrint(pdfPath, xpsOrNull) → PdfDocument.FromFile(pdfPath) followed by pdf.Print() or pdf.GetPrintDocument(settings).Print().
  • fileprint.Print() → pdf.Print() (default printer) or pdf.Print(printerName).
  • Properties.Settings.Default.PrinterName → PrinterSettings.PrinterName.
  • Properties.Settings.Default.Copies → PrinterSettings.Copies.
  • Properties.Settings.Default.PaperName → PrinterSettings.DefaultPageSettings.PaperSize.
  • Properties.Settings.Default.PrintToFile + DefaultPrintToDirectory → PrinterSettings.PrintToFile + PrintFileName.

Capabilities not present in PDFFilePrint

  • HTML string to PDF: renderer.RenderHtmlAsPdf(html).
  • URL to PDF: renderer.RenderUrlAsPdf(url).
  • HTML file to PDF: renderer.RenderHtmlFileAsPdf(path).
  • Save: pdf.SaveAs(path).
  • Page-range printing: PrinterSettings.FromPage / ToPage with PrintRange.SomePages.
  • Duplex: PrinterSettings.Duplex.
  • Paper size and orientation per call: PrinterSettings.DefaultPageSettings.PaperSize and Landscape; for HTML rendering, renderer.RenderingOptions.PaperSize and renderer.RenderingOptions.PaperOrientation.
  • Margins: renderer.RenderingOptions.MarginTop/Bottom/Left/Right.
  • Page count on a loaded document: pdf.PageCount.
  • Merge: PdfDocument.Merge(pdf1, pdf2).
  • Split / extract pages: pdf.CopyPages(startIndex, endIndex).

What Do Complete Before/After Code Migrations Look Like?

Silent print with named printer and copy count

Before (PDFFilePrint):

using System;
using PDFFilePrint;

class PrintExample
{
    static void Main()
    {
        // PDFFilePrint reads PrinterName, Copies, PaperName from
        // Properties.Settings.Default (app.config). To override at runtime
        // you mutate the settings object before instantiating FilePrint.
        Properties.Settings.Default.PrinterName = "HP LaserJet Pro";
        Properties.Settings.Default.Copies      = 3;

        var fileprint = new FilePrint("invoice.pdf", null);
        fileprint.Print();
    }
}

After (IronPDF):

using IronPdf;
using System.Drawing.Printing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("invoice.pdf");

var settings = new PrinterSettings
{
    PrinterName = "HP LaserJet Pro",
    Copies      = 3
};

pdf.GetPrintDocument(settings).Print();
// Guide: https://ironpdf.com/docs/questions/print-pdf/

HTML to PDF (new capability — PDFFilePrint cannot render HTML)

Before (PDFFilePrint, paired with a separate renderer):

// PDFFilePrint cannot convert HTML to PDF. Teams that need HTML-to-print
// pipelines render the HTML with another tool first, then hand the file
// to FilePrint. A typical pairing was wkhtmltopdf called via Process.Start:
using PDFFilePrint;
using System.Diagnostics;

var psi = new ProcessStartInfo("wkhtmltopdf.exe", "input.html invoice.pdf");
Process.Start(psi).WaitForExit();

new FilePrint("invoice.pdf", null).Print();

After (IronPDF — render and save in one library):

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize    = IronPdf.Rendering.PdfPaperSize.A4;
renderer.RenderingOptions.MarginTop    = 10;
renderer.RenderingOptions.MarginBottom = 10;
renderer.RenderingOptions.MarginLeft   = 15;
renderer.RenderingOptions.MarginRight  = 15;

var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>"
);
pdf.SaveAs("invoice.pdf");
// Or render and print in one pass:
// pdf.Print("HP LaserJet Pro");
// Rendering options: https://ironpdf.com/how-to/rendering-options/

Merge PDFs

Before (PDFFilePrint — via PdfSharp, since PDFFilePrint has no merge API):

using System;
// PDFFilePrint's FilePrint class has no merge method.
// Teams typically pull in PdfSharp as a secondary library:
using PdfSharp.Pdf;
using PdfSharp.Pdf.IO;

class MergePdfsExample
{
    static void Main()
    {
        using var output = new PdfDocument();

        foreach (string path in new[] { "part1.pdf", "part2.pdf" })
        {
            using var input = PdfReader.Open(path, PdfDocumentOpenMode.Import);
            foreach (PdfPage page in input.Pages)
                output.AddPage(page);
        }

        output.Save("merged.pdf");
        Console.WriteLine("Merged to: merged.pdf");
    }
}

After (IronPDF):

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var merged = PdfDocument.Merge(
    PdfDocument.FromFile("part1.pdf"),
    PdfDocument.FromFile("part2.pdf")
);
merged.SaveAs("merged.pdf");
// Guide: https://ironpdf.com/how-to/merge-or-split-pdfs/

Watermark

Before (PDFFilePrint — via iTextSharp, since PDFFilePrint has no watermark API):

using iTextSharp.text;
using iTextSharp.text.pdf;
using System.IO;
using System;

class WatermarkExample
{
    static void Main()
    {
        // PDFFilePrint does not expose stamping — iTextSharp pattern:
        using var reader  = new PdfReader("input.pdf");
        using var fs      = new FileStream("watermarked.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs);

        var baseFont = BaseFont.CreateFont(BaseFont.HELVETICA_BOLD, BaseFont.CP1252, false);
        for (int page = 1; page <= reader.NumberOfPages; page++)
        {
            var cb = stamper.GetOverContent(page);
            cb.SaveState();
            cb.SetGState(new PdfGState { FillOpacity = 0.3f });
            cb.BeginText();
            cb.SetFontAndSize(baseFont, 60);
            cb.SetColorFill(BaseColor.GRAY);
            cb.ShowTextAligned(Element.ALIGN_CENTER, "CONFIDENTIAL", 300, 420, 45);
            cb.EndText();
            cb.RestoreState();
        }
        Console.WriteLine("Watermarked.");
    }
}

After (IronPDF):

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("input.pdf");
var stamper = new TextStamper
{
    Text = "CONFIDENTIAL",
    FontColor = IronSoftware.Drawing.Color.Gray,
    FontSize = 60,
    Opacity = 30,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");
// Guide: https://ironpdf.com/how-to/custom-watermark/

Password protection

Before (PDFFilePrint — via iTextSharp, since PDFFilePrint has no security API):

using iTextSharp.text.pdf;
using System.IO;
using System.Text;

class SecurityExample
{
    static void Main()
    {
        // PDFFilePrint does not expose encryption — iTextSharp pattern:
        byte[] userPass  = Encoding.ASCII.GetBytes("readonly");
        byte[] ownerPass = Encoding.ASCII.GetBytes("admin");

        using var reader  = new PdfReader("input.pdf");
        using var fs      = new FileStream("secured.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs, '\0', false);
        stamper.SetEncryption(
            userPass, ownerPass,
            PdfWriter.ALLOW_PRINTING,
            PdfWriter.ENCRYPTION_AES_128
        );
    }
}

After (IronPDF):

using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("input.pdf");
pdf.SecuritySettings.UserPassword  = "readonly";
pdf.SecuritySettings.OwnerPassword = "admin";
pdf.SecuritySettings.AllowUserPrinting = PdfPrintSecurity.FullPrintRights;
pdf.SecuritySettings.AllowUserCopyPasteContent = false;
pdf.SaveAs("secured.pdf");
// Guide: https://ironpdf.com/how-to/pdf-permissions-passwords/

What Critical Details Should I Watch During Migration?

Page indexing on loaded documents

PDFFilePrint does not expose page-level operations — the printer driver handles paging. If you start using IronPDF's PdfDocument APIs (page count, copy pages, stamp specific pages) note that pdf.Pages is 0-based:

// IronPDF: page 0 = first page
var firstPage = pdf.Pages[0];
for (int i = 0; i < pdf.PageCount; i++)
{
    // operate on pdf.Pages[i]
}

Printing surface differences

PDFFilePrint's FilePrint.Print() sends the PDF (passed in via the constructor) to whatever printer is configured under Properties.Settings.Default.PrinterName, with copy count and paper-name pulled from the same settings object. There is no first-class API for page range, duplex, or orientation; those flow through the printer driver default.

IronPDF's equivalent is pdf.Print() — or pdf.Print(printerName) for a named printer — and a System.Drawing.Printing.PrinterSettings overload via pdf.GetPrintDocument(settings).Print() that surfaces copies, page range, duplex, orientation, paper size, and the print-to-file flags directly on the call. If your code was mutating Settings.Default before each print, the migration is constructing a PrinterSettings instance per call instead.

Error model migration

Replace any status-code or return-value error handling with try/catch:

try
{
    var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs("output.pdf");
}
catch (IronPdf.Exceptions.IronPdfException ex)
{
    _logger.LogError(ex, "PDF generation failed: {msg}", ex.Message);
    throw;
}

How Do I Optimize Performance After Migration?

Renderer reuse

Instantiate once per batch — do not create a new renderer per render:

// Instantiate once per batch
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

foreach (var html in documentQueue)
{
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"output_{Guid.NewGuid()}.pdf");
}

Async in ASP.NET

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(html);
return File(pdf.Stream, "application/pdf", "document.pdf");

Frequently Asked Questions

Can I migrate from PDFFilePrint to IronPDF if my primary use case is sending PDFs to a printer?

Yes — that is the most direct migration path. new FilePrint(path, null).Print(); becomes PdfDocument.FromFile(path).Print();. Same two-step shape. IronPDF additionally accepts a System.Drawing.Printing.PrinterSettings instance via pdf.GetPrintDocument(settings).Print(), exposing copies, duplex, page range, paper size, and orientation as strongly-typed properties — configuration that PDFFilePrint reads from app.config (or defers to the printer driver entirely).

If your source content originates as HTML, IronPDF can render and print in one pass without writing to disk: renderer.RenderHtmlAsPdf(html).Print(printerName).

Does IronPDF work on .NET 8 in Linux Docker containers?

Yes. IronPDF supports .NET 8 on Linux. You will need to install Chromium dependencies in your Docker image — libglib2.0-0, libnss3, libatk1.0-0, libatk-bridge2.0-0, and libx11-6 at minimum.

Use multi-stage builds and --no-install-recommends to keep image size manageable. Verify current requirements at the IronPDF Docker deployment guide.

How do I handle page ranges when migrating from PDFFilePrint?

PDFFilePrint has no page-range API — the printer driver default applies, or you pre-split the PDF with another tool. IronPDF exposes the page range directly on PrinterSettings:

var settings = new PrinterSettings
{
    PrinterName = "HP LaserJet Pro",
    PrintRange  = PrintRange.SomePages,
    FromPage    = 2,
    ToPage      = 5
};
pdf.GetPrintDocument(settings).Print();

If you start using PdfDocument APIs that take page indices (such as pdf.CopyPages), note that those are 0-based: pdf.Pages[0] is the first page.

Will my printed output look different after migrating to IronPDF?

For straight pass-through printing of an existing PDF, output should be functionally equivalent — both libraries hand the document to the Windows printing subsystem.

If you are also moving an HTML-to-PDF pipeline into IronPDF (which PDFFilePrint cannot do natively), expect rendering differences from whatever previous renderer was producing the PDF. Specify fonts explicitly with fallbacks in your HTML templates and visually compare a representative document set before completing the migration.

Can IronPDF handle merge, watermark, and password protection without secondary libraries?

Yes. IronPDF handles all three natively. Merging uses PdfDocument.Merge(), watermarking uses TextStamper via ApplyStamp(), and password protection uses SecuritySettings on the PdfDocument object.

This is a consolidation advantage — if you were using secondary libraries like PdfSharp or iTextSharp alongside PDFFilePrint, you can remove those dependencies.

What Is the Bottom Line on This Migration?

.NET version incompatibility is one of the cleaner migration triggers — the decision is often made for you. The ambiguity lives in the feature audit: PDFFilePrint's surface is small (one FilePrint class that takes a PDF path and an optional XPS output path, plus a handful of app.config keys), so most teams that picked it have either outgrown it or supplemented it with a second library before the runtime upgrade forces the issue. The audit is mainly about cataloguing where you are mutating Properties.Settings.Default and where you are chaining a separate renderer.

If your primary use case is sending existing PDFs to a printer, IronPDF's PdfDocument.Print() and GetPrintDocument(PrinterSettings) overloads are a direct fit with a richer configuration surface. If your workflow extends to HTML-to-PDF generation, PDF merging, watermarking, or encryption, IronPDF covers those with the same NuGet package on .NET 8 across Windows and Linux. The code samples above show the API translations.

Run the codebase scan, swap the packages, translate the calls using the mapping above, and verify printed output on your target printers. That is the migration in practice.
