Migrating from iTextSharp to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should .NET Teams Migrate Away from iTextSharp?

The signal was quiet at first. A GitHub issue sitting open for fourteen months. A Stack Overflow answer from a core contributor saying "this package is not actively maintained." Then you actually look at the commit history: the last meaningful code change is years back.

iTextSharp, the old .NET wrapper around iText, has been in maintenance-only mode for a long time. The actively maintained successor, iText 7 for .NET, has a different API and a different licensing model. Teams using iTextSharp face a choice: migrate to iText 7 (a significant API rewrite), or use a different library entirely.

There are eight reasons I see teams making this switch repeatedly:

First, there is the maintenance signal. Commit history and issue response times indicate iTextSharp is in wind-down. The successor (iText 7) has a different API and licensing model entirely.

Second, AGPL licensing friction is real. iText's AGPL license requires that any software using it also be AGPL, unless you purchase a commercial license. Teams with proprietary applications either buy a license or move to a differently-licensed library.

Third, there is a .NET Framework dependency. iTextSharp targets .NET Framework; teams moving to .NET 6/8 encounter compatibility friction.

Fourth, HTML-to-PDF fidelity is limited. iTextSharp's HTMLWorker is not a full HTML renderer. Modern CSS is largely unsupported. Teams needing accurate HTML rendering end up adding a second library anyway.

Fifth, API verbosity adds up. Building a simple table requires substantial boilerplate. HTML/CSS templates with a renderer library often require less code.

Sixth, debugging document layout is harder when the PDF is built from programmatic API calls rather than an HTML template you can preview in a browser.

Seventh, the maintenance burden of iTextSharp-specific code is significant. The API has quirks, and developers joining a team often have no prior iTextSharp exposure.

Eighth, there are feature gaps. iTextSharp does not include a Chromium-based renderer, so teams needing that capability add a second library.

How Does the Migration Effort Break Down by Feature?

Before committing to this migration, I always recommend teams assess their codebase feature by feature. Not every iTextSharp usage requires the same level of effort to replace.

Here is the effort breakdown I have seen across these migrations. The key factor is whether your code uses iTextSharp for HTML rendering or as a programmatic document builder.

  • HTML string to PDF — Low. HTMLWorker.ParseToList() in iTextSharp 5.x (or HtmlConverter.ConvertToPdf() in iText 7's pdfHTML add-on) maps cleanly to renderer.RenderHtmlAsPdf().
  • Programmatic document building — High. The Document + Paragraph + Table API has no direct equivalent; rewrite as an HTML template.
  • Merge PDFs — Low. PdfCopy in iTextSharp 5.x or PdfMerger in iText 7 collapses to a one-line PdfDocument.Merge() call.
  • Split PDF — Low. PdfCopy with page iteration becomes pdf.CopyPages().
  • Watermark / stamp — Low. PdfStamper + content stream becomes pdf.ApplyStamp() with a TextStamper.
  • Password protection — Low. PdfStamper.SetEncryption() or WriterProperties.SetStandardEncryption() becomes property assignments on SecuritySettings.
  • Form filling — Medium. AcroFields via PdfStamper — verify IronPDF's form API coverage against your specific form types.
  • Digital signatures — Medium. PdfSignatureAppearance — verify IronPDF signing docs against your signature requirements (PAdES levels, timestamp authority).
  • Reading existing PDF text — Medium. PdfTextExtractor — verify IronPDF's text extraction against your test corpus, especially for multi-column or scanned PDFs.
  • PDF metadata — Low. PdfReader.Info becomes pdf.MetaData property assignments.
  • Page rotation — Low. PdfReader.GetPageRotation() reads cleanly; verify the equivalent IronPDF accessor for your version.

The fundamental architecture difference matters here. iTextSharp works by constructing PDF documents in code — you add paragraphs, tables, images, and layout elements through a document object model. IronPDF works primarily as an HTML renderer — you give it HTML/CSS, and Chromium renders it to PDF. It also supports PDF manipulation (merge, split, watermark, and more).

This means the migration path splits into two tracks. If you are using iTextSharp to generate PDFs from HTML, it is a straightforward substitution. If you are using iTextSharp as a document builder, you rewrite the generation logic as HTML templates and then render them.

What Are the Decision Criteria for Choosing IronPDF Over iText 7?

When teams ask me whether they should migrate to iText 7 or to IronPDF, I walk them through the following comparison and scenario matrix.

  • Focus. iTextSharp is a low-level PDF builder plus manipulation toolkit; IronPDF is HTML-to-PDF plus PDF manipulation.
  • Pricing. iTextSharp ships under AGPL (free for open source) or commercial; IronPDF is commercial — verify current terms at ironsoftware.com.
  • API style. iTextSharp uses a document object model with programmatic construction; IronPDF combines an HTML renderer with a document model.
  • Learning curve. iTextSharp is high (low-level API, many PDF-spec concepts); IronPDF is medium for developers already comfortable with HTML and CSS.
  • HTML rendering. iTextSharp's HTMLWorker handles a limited CSS subset; IronPDF embeds Chromium for full CSS3.
  • Page indexing. iTextSharp is 1-based (PdfReader.NumberOfPages); IronPDF is 0-based.
  • Thread safety. iTextSharp is not thread-safe by default; with IronPDF, reuse a renderer instance — see the async docs.
  • Namespace. iTextSharp.text → IronPdf.

Decision Scenarios

  • If you are already generating PDFs from HTML templates, IronPDF is a natural fit and the rewrite is minimal.
  • If you rely on programmatic PDF construction (complex tables, precise absolute positioning), evaluate whether HTML and CSS templates can replace it — in most cases they can, but plan for the rewrite.
  • If the AGPL constraint is the trigger, both iText 7 commercial and IronPDF address it — compare feature sets against your actual workload.
  • If you do heavy PDF form filling (AcroForms), verify IronPDF's form API coverage against your specific form templates before committing.

HTML to PDF: Before and After

// iText 7 pdfHTML add-on (separate license from core iText 7)
using iText.Html2pdf;
using System.IO;

class HtmlToPdfMigration
{
    static void Main()
    {
        string html = @"
            <html><body>
                <h1>Invoice #1234</h1>
                <table border='1' cellpadding='4'>
                    <tr><th>Item</th><th>Price</th></tr>
                    <tr><td>Widget A</td><td>$50.00</td></tr>
                    <tr><td>Widget B</td><td>$75.00</td></tr>
                </table>
                <p><b>Total: $125.00</b></p>
            </body></html>";

        using var fs = new FileStream("invoice.pdf", FileMode.Create);
        HtmlConverter.ConvertToPdf(html, fs);
    }
}
// IronPDF approach (full CSS3 support via Chromium)
using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

string html = @"
    <html><head><style>
        h1 { color: #333; }
        table { border-collapse: collapse; width: 100%; }
        th, td { border: 1px solid #ccc; padding: 8px; }
        th { background: #f5f5f5; }
    </style></head><body>
        <h1>Invoice #1234</h1>
        <table>
            <tr><th>Item</th><th>Price</th></tr>
            <tr><td>Widget A</td><td>$50.00</td></tr>
            <tr><td>Widget B</td><td>$75.00</td></tr>
        </table>
        <p><strong>Total: $125.00</strong></p>
    </body></html>";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("invoice.pdf");

Merge PDFs: Before and After

// iText 7 PdfMerger pattern
using iText.Kernel.Pdf;
using iText.Kernel.Utils;

class MergePdfsExample
{
    static void MergePdfs(string[] inputPaths, string outputPath)
    {
        using var writer = new PdfWriter(outputPath);
        using var pdfDoc = new PdfDocument(writer);
        var merger = new PdfMerger(pdfDoc);

        foreach (string path in inputPaths)
        {
            using var sourcePdf = new PdfDocument(new PdfReader(path));
            merger.Merge(sourcePdf, 1, sourcePdf.GetNumberOfPages());
        }
    }

    static void Main() => MergePdfs(
        new[] { "part1.pdf", "part2.pdf", "part3.pdf" },
        "merged.pdf"
    );
}
// IronPDF merge (3 lines)
using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var merged = PdfDocument.Merge(
    PdfDocument.FromFile("part1.pdf"),
    PdfDocument.FromFile("part2.pdf"),
    PdfDocument.FromFile("part3.pdf")
);
merged.SaveAs("merged.pdf");

Watermark: Before and After

// iText 7 PdfCanvas watermark approach
using iText.Kernel.Pdf;
using iText.Kernel.Pdf.Canvas;
using iText.Kernel.Pdf.Extgstate;
using iText.Kernel.Font;
using iText.IO.Font.Constants;
using System;

class WatermarkExample
{
    static void Main()
    {
        using var reader = new PdfReader("input.pdf");
        using var writer = new PdfWriter("watermarked.pdf");
        using var pdfDoc = new PdfDocument(reader, writer);

        var font = PdfFontFactory.CreateFont(StandardFonts.HELVETICA_BOLD);
        var gs   = new PdfExtGState().SetFillOpacity(0.3f);

        for (int i = 1; i <= pdfDoc.GetNumberOfPages(); i++)
        {
            var page     = pdfDoc.GetPage(i);
            var pageSize = page.GetPageSize();
            var canvas   = new PdfCanvas(page);

            canvas.SaveState()
                  .SetExtGState(gs)
                  .BeginText()
                  .SetFontAndSize(font, 60)
                  .SetTextMatrix(
                      (float)Math.Cos(Math.PI / 4), (float)Math.Sin(Math.PI / 4),
                      -(float)Math.Sin(Math.PI / 4), (float)Math.Cos(Math.PI / 4),
                      pageSize.GetWidth() / 4, pageSize.GetHeight() / 2)
                  .ShowText("CONFIDENTIAL")
                  .EndText()
                  .RestoreState();
        }
    }
}
// IronPDF watermark approach
using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("input.pdf");
var stamper = new TextStamper
{
    Text = "CONFIDENTIAL",
    FontColor = IronSoftware.Drawing.Color.Gray,
    FontSize = 60,
    Opacity = 30,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");

Password Protection: Before and After

// iText 7 WriterProperties encryption
using iText.Kernel.Pdf;
using System.Text;

class SecurityExample
{
    static void Main()
    {
        byte[] userPassword  = Encoding.UTF8.GetBytes("readonlyaccess");
        byte[] ownerPassword = Encoding.UTF8.GetBytes("admincontrol");

        var writerProperties = new WriterProperties()
            .SetStandardEncryption(
                userPassword,
                ownerPassword,
                EncryptionConstants.ALLOW_PRINTING | EncryptionConstants.ALLOW_COPY,
                EncryptionConstants.ENCRYPTION_AES_128);

        using var reader = new PdfReader("document.pdf");
        using var writer = new PdfWriter("secured.pdf", writerProperties);
        using var pdfDoc = new PdfDocument(reader, writer);
    }
}
// IronPDF password protection
using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("document.pdf");
pdf.SecuritySettings.UserPassword  = "readonlyaccess";
pdf.SecuritySettings.OwnerPassword = "admincontrol";
pdf.SecuritySettings.AllowUserCopyPasteContent = true;
pdf.SecuritySettings.AllowUserPrinting = PdfPrintSecurity.FullPrintRights;
pdf.SaveAs("secured.pdf");

What Are the Complete API Mappings Between iTextSharp and IronPDF?

I maintain these mapping tables for my team when onboarding projects that are mid-migration. They cover namespaces, core classes, document loading, page operations, and merge/split workflows.

Namespace Mapping

  • iTextSharp.text → IronPdf — core types
  • iTextSharp.text.pdf → IronPdf — PDF operations
  • iTextSharp.text.html.simpleparser → N/A — replaced by IronPDF's Chromium renderer

Core Class Mapping

  • PdfWriter + Document → ChromePdfRenderer — PDF generation entry point
  • PdfReader → PdfDocument.FromFile() — open an existing PDF
  • PdfStamper → PdfDocument + instance methods — modify an existing PDF
  • PdfCopy → PdfDocument.Merge() — combine PDFs

Document Loading Methods

  • HTML string: HTMLWorker.Parse() → renderer.RenderHtmlAsPdf(html)
  • URL: not native in iTextSharp → renderer.RenderUrlAsPdf(url)
  • Existing PDF: new PdfReader(path) → PdfDocument.FromFile(path)
  • From bytes: new PdfReader(bytes) → PdfDocument.FromStream(stream)

Page Operations

  • Page count: reader.NumberOfPages → pdf.PageCount
  • Paper size: PageSize.A4 constant → ChromePdfRenderOptions.PaperSize
  • Margins: Document constructor → ChromePdfRenderOptions.Margin*
  • Page rotation: reader.GetPageRotation(n) → verify the equivalent IronPDF accessor for your version

Merge and Split Operations

  • Merge: PdfCopy + PdfReader per file → PdfDocument.Merge(pdf1, pdf2)
  • Split: PdfCopy + page iteration → pdf.CopyPages(startIndex, endIndex)

Page Indexing: A Critical Difference

// iTextSharp: page 1 = first page
for (int i = 1; i <= reader.NumberOfPages; i++) { ... }

// IronPDF: page 0 = first page
for (int i = 0; i < pdf.PageCount; i++) { ... }
var firstPage = pdf.Pages[0]; // not [1]

What About Performance and Async Patterns?

I get this question frequently from teams running PDF generation inside ASP.NET controllers or batch processing jobs. The rendering model is different, so the performance profile is different too.

Renderer Reuse for Batch Generation

// Reuse renderer for batch generation
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

foreach (var template in GetDocumentTemplates())
{
    var html = RenderTemplate(template); // your template engine
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"doc_{template.Id}.pdf");
}

Async Pattern for Web Applications

// Async pattern for ASP.NET controllers
[HttpGet("generate")]
public async Task<IActionResult> GeneratePdf()
{
    var renderer = new ChromePdfRenderer();
    var html = await _templateEngine.RenderAsync(model);
    var pdf = await renderer.RenderHtmlAsPdfAsync(html);

    return File(pdf.Stream, "application/pdf", "document.pdf");
}

Memory Disposal

// PdfDocument implements IDisposable
using var pdf = PdfDocument.FromFile("input.pdf");
// ... operations ...
pdf.SaveAs("output.pdf");
// Disposed at end of block

There are a few edge cases worth flagging. Complex table layouts in iTextSharp's programmatic table API gave precise control over cell sizing — HTML/CSS tables in a renderer work differently, so test your most complex layouts explicitly. Exact positioning via absolute coordinates also differs: in HTML, absolute positioning is CSS-based. And font embedding works differently between the two libraries, so verify font rendering in output PDFs, especially with custom fonts.

Frequently Asked Questions

Is this migration a simple find-and-replace operation?

It depends on your usage pattern. If you are primarily using iTextSharp's HTMLWorker for HTML-to-PDF conversion, the migration is genuinely straightforward — you replace HTMLWorker.Parse() calls with IronPDF's ChromePdfRenderer and get better CSS support in the process.

If your codebase builds PDFs programmatically using Document, Paragraph, PdfPTable, and related classes, migrating means rewriting those as HTML templates. This is not a mechanical substitution — it is a design change. For each PDF type your system generates, create an HTML template using any templating engine (Razor, Liquid, Handlebars), render it to a string, and pass the string to IronPDF's renderer.

What is the biggest gotcha during migration?

Page indexing. iTextSharp uses 1-based page indexing throughout — PdfReader.NumberOfPages returns 1-based values, and loops start at 1. IronPDF uses 0-based indexing. Review every page reference in your codebase. A single off-by-one error here can cause pages to be skipped or duplicated during merge and split operations.

If your code uses AcroFields to fill PDF forms, verify IronPDF's form API coverage before committing to the migration. This feature set should be tested against your specific form types.

Can IronPDF handle everything iTextSharp does?

For the common operations — HTML to PDF, merge, split, watermark, and password protection — IronPDF covers those with less code. For specialized operations like digital signatures and AcroForm filling, I recommend verifying IronPDF's specific API coverage against your requirements before committing. The honest assessment is that the 1:1 conceptual mapping does not always exist between a low-level document builder and an HTML renderer.

How do I handle the AGPL licensing concern?

If AGPL was the migration trigger, both iText 7 commercial and IronPDF address it — compare feature sets for your use case. iText's AGPL license requires that any software using it also be AGPL, unless you purchase a commercial license. IronPDF uses a commercial license model. Verify that IronPDF's licensing terms cover your use case before switching. This is a legal and business decision, not purely a technical one.

How Should Teams Make the Final Migration Decision?

The hardest part of an iTextSharp migration is not usually the API mapping. It is the cases where someone spent significant effort building precise document layouts programmatically that now need to be reimplemented as HTML templates.

I recommend a three-step decision framework.

First, classify your PDF generation code. Run rg "iTextSharp\|PdfWriter\|PdfStamper\|PdfCopy" --type cs across your codebase. Separate everything into two buckets: HTML-based generation (using HTMLWorker) and programmatic document construction (using Document, Paragraph, PdfPTable). The first bucket migrates quickly. The second bucket requires planning.

Second, verify feature coverage for your specific needs. If you use form filling (AcroFields), digital signatures, or text extraction, test those features against your actual documents with IronPDF's trial license before committing. Do not assume coverage — test it.

Third, budget time based on your dominant pattern. For teams that are primarily HTML-to-PDF, the migration is genuinely straightforward. For teams with complex programmatic document construction, budget more time for the HTML template rewrite. The HTML template approach is typically more maintainable and easier to iterate on, but it does require the initial rewrite investment.
