Migrating from DynamicPDF to IronPDF: A Working Migration in an Afternoon

If your DynamicPDF project is generating PDFs from HTML and the output looks like it was rendered in 2009, you're not alone. DynamicPDF's HTML pipeline ships as a separate add-on (HtmlConverter) layered on top of a programmatic Generator/Merger core. Teams that started with invoice templates and then got feature-creep'd into "render this React-generated HTML report" often hit a wall.

I've seen this pattern dozens of times. This article gives you a structured migration path to IronPDF, with real API mappings, four complete before/after code samples, and a copy-pasteable migration scaffolding you can use today.

Why Should You Migrate from DynamicPDF to IronPDF?

Teams typically reach for a different library when one or more of these triggers accumulates:

Product fragmentation. DynamicPDF is sold as separate products with separate licenses: Generator (create), Merger (merge/split/manipulate), HtmlConverter (HTML add-on), ReportWriter, and Print Manager. Most production stacks need three or four of these. IronPDF is a single package.

HTML fidelity. DynamicPDF's HtmlConverter is layered on top of a Generator core, not a full browser engine. Complex CSS (flexbox, grid, custom fonts, SVG) often degrades.

JavaScript execution. If your templates use JS (charting libraries, dynamic tables), HtmlConverter's support is limited. IronPDF uses Chromium internally and waits for JS execution.

Linux/container deployment. Some DynamicPDF versions require GDI+ shims. Verify their current Linux native-deps story for your environment.

API verbosity. DynamicPDF's coordinate-based Generator API is expressive but verbose for simple "render this URL to PDF" tasks.

Async support. IronPDF has documented async/parallel patterns; HtmlConverter is synchronous.

Modern .NET. IronPDF targets .NET 6/7/8/9 natively. Verify your CoreSuite version's support for your .NET target.

PDF/A and compliance. DynamicPDF has PDF/A support; IronPDF also has PDF/A generation -- compare feature parity for your compliance requirements.

How Do DynamicPDF and IronPDF Compare Side by Side?

The fundamental difference is architectural. DynamicPDF is a fragmented suite of programmatic products (Generator, Merger, HtmlConverter, ReportWriter), each licensed separately. IronPDF is a single package built around an embedded Chromium engine.

  • Product model. DynamicPDF ships as five-plus separate products with separate licenses; IronPDF is one all-in-one library.
  • API style. DynamicPDF uses coordinate-based positioning (X, Y, width, height in points); IronPDF uses HTML/CSS plus a manipulation API.
  • HTML rendering. DynamicPDF requires the HtmlConverter add-on with limited CSS3 support; IronPDF uses Chromium with full CSS3 (Flexbox, Grid) and ES6+ JavaScript.
  • Learning curve. DynamicPDF requires learning a coordinate layout model across multiple product APIs; IronPDF leverages web technologies you already know.
  • Page indexing. Both libraries are 0-based.
  • Modern .NET. IronPDF targets .NET 6/7/8/9 natively; DynamicPDF is .NET Standard 2.0.
  • Namespace. ceTe.DynamicPDF versus IronPdf.

What Does the Migration Effort Look Like Feature by Feature?

It depends heavily on whether your codebase is HTML-heavy or layout-heavy. Here's the effort breakdown by feature, followed by a decision matrix to help you determine the right approach.

  • HTML string to PDF -- Low. Direct 1:1 if you were using HtmlConverter.
  • URL to PDF -- Low. Both support it (HtmlConverter accepts a Uri; IronPDF uses RenderUrlAsPdf).
  • Programmatic page layout -- High. No coordinate-based equivalent in IronPDF; the work is converting Generator code to HTML/CSS templates.
  • Text extraction -- Medium. pdfDoc.GetText() maps to pdf.ExtractAllText(); per-page and per-area extraction map cleanly.
  • Merge documents -- Low. MergeDocument.Append() becomes PdfDocument.Merge() or PdfDocument.InsertPdf().
  • Split documents -- Low. Similar complexity; use pdf.CopyPages().
  • Watermark / stamp -- Medium. DynamicPDF uses Template + Label or ImageWatermark; IronPDF uses ApplyWatermark or ApplyStamp.
  • Password protection -- Low. Aes256Security maps to SecuritySettings.
  • Digital signatures -- Medium-High. Both support certificate signing; PAdES-level conformance should be verified per project.
  • Form filling -- Low-Medium. pdfDoc.Form.Fields["x"].Value maps to pdf.Form.GetFieldByName("x").Value.
  • JavaScript-rendered HTML -- New capability. HtmlConverter has limited JS support; IronPDF's Chromium runs full ES6+.

Decision Matrix

  • If all PDF generation is coordinate-based Generator code, expect a real rewrite -- the work is converting layout to HTML/CSS, not swapping using statements.
  • If you have a mix of Generator layout and HtmlConverter rendering, run both libraries side by side and migrate one rendering path at a time.
  • If you're primarily on HtmlConverter or doing URL-to-PDF, IronPDF is a natural fit and the migration is largely mechanical.
  • If you're targeting Linux containers at high volume, verify both libraries' Linux native-deps story for your specific runtime before committing.

How Do I Prepare My Codebase Before Migrating?

Start by inventorying every DynamicPDF reference, then swap NuGet packages, and set your license key. Here's the full pre-migration workflow.

Find DynamicPDF References

# Find all files referencing DynamicPDF namespaces
rg "ceTe\.DynamicPDF" --type cs -l

# Find specific Document/Page usages (Generator)
rg "new Document\(\)|new Page\(" --type cs

# Find HtmlConverter usages -- these are your HTML migration targets
rg "HtmlConverter" --type cs

# Find MergeDocument usages -- these become PdfDocument.Merge
rg "MergeDocument" --type cs

Uninstall and Install

# Remove DynamicPDF (each product is a separate package)
dotnet remove package ceTe.DynamicPDF.CoreSuite.NET
dotnet remove package ceTe.DynamicPDF.Generator.NET
dotnet remove package ceTe.DynamicPDF.Merger.NET
dotnet remove package ceTe.DynamicPDF.HtmlConverter.NET

# Install IronPDF
dotnet add package IronPdf

# Verify
dotnet list package

Set your license key before running:

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
// Or via environment variable: IRONPDF_LICENSEKEY
// Or in appsettings.json

What Does a Quick Start Migration Look Like in Three Steps?

The core migration is license, namespaces, and render call. Each step is a mechanical replacement.

Step 1: License

// Before (DynamicPDF)
// ceTe.DynamicPDF.License.SetLicenseKey("YOUR-LICENSE-KEY");

// After (IronPDF)
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

Step 2: Namespace Imports

// Before
using ceTe.DynamicPDF;
using ceTe.DynamicPDF.PageElements;
using ceTe.DynamicPDF.Conversion;
using ceTe.DynamicPDF.Merger;

// After
using IronPdf;

Step 3: Basic HTML to PDF Conversion

// Before (DynamicPDF HtmlConverter add-on)
// using ceTe.DynamicPDF.Conversion;
// string html = "<h1>Hello</h1>";
// HtmlConverter converter = new HtmlConverter(html);
// converter.Convert("output.pdf");

// After (IronPDF)
using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<h1>Hello</h1>");
pdf.SaveAs("output.pdf");

How Do DynamicPDF APIs Map to IronPDF APIs?

I've built out the full mapping below covering namespaces, core classes, document loading, page operations, and merge/split. Keep these handy during your migration.

Namespace Mapping

  • ceTe.DynamicPDF → IronPdf -- core namespace
  • ceTe.DynamicPDF.PageElements → IronPdf -- no separate elements namespace; HTML drives layout
  • ceTe.DynamicPDF.Conversion → IronPdf -- HtmlConverter becomes ChromePdfRenderer
  • ceTe.DynamicPDF.Merger → IronPdf -- merge is a static method on PdfDocument
  • ceTe.DynamicPDF.Cryptography → IronPdf -- security via SecuritySettings
  • ceTe.DynamicPDF.Forms → IronPdf -- form access via the Form property

Core Class Mapping

  • Document (Generator) → ChromePdfRenderer -- for generating new PDFs from HTML
  • Document / PdfDocument (Merger) → PdfDocument -- for loading and manipulating existing PDFs
  • Page + coordinate elements → HTML/CSS -- pages are an output of rendering, not constructed by hand
  • HtmlConverter → ChromePdfRenderer -- HTML rendering engine
  • MergeDocument → PdfDocument.Merge() -- static merge method
  • Template → HtmlHeaderFooter -- for repeating headers, footers, and watermarks

Document Loading

  • Load from file: new PdfDocument(path) → PdfDocument.FromFile(path)
  • Load from bytes: DynamicPDF accepts a byte[] in PdfDocument (Merger) → PdfDocument.FromBinaryData(bytes)
  • Load from stream: DynamicPDF PdfDocument(Stream) → PdfDocument.FromStream(stream)
  • Render HTML string: new HtmlConverter(html).Convert(path) → renderer.RenderHtmlAsPdf(html).SaveAs(path)
  • Render URL: new HtmlConverter(new Uri("...")) → renderer.RenderUrlAsPdf("...")

Page Operations

  • Page count: doc.Pages.Count → pdf.PageCount
  • Get page: doc.Pages[i] → pdf.Pages[i]
  • Add page: doc.Pages.Add(page) → pdf.InsertPdf(other, index) or render fresh HTML
  • Remove page: pdfDoc.Pages.Remove(page) → pdf.RemovePages(index)
  • Extract page text: pdfDoc.Pages[i].GetText() → pdf.ExtractTextFromPage(i)

Merge and Split

  • Append PDFs: new MergeDocument(a).Append(b) → PdfDocument.Merge(pdfA, pdfB)
  • Insert at index: mergeDoc.InsertDocument(other, i) → pdf.InsertPdf(other, i)
  • Split / extract pages: copy a range out of an existing document → pdf.CopyPages(0, n)

What Do Complete Before/After Code Migrations Look Like?

Below are four real migration examples covering the most common DynamicPDF workloads: HTML to PDF, merging, watermarking, and password protection.

1. HTML to PDF

Before (DynamicPDF HtmlConverter add-on):

using ceTe.DynamicPDF.Conversion;

class Program
{
    static void Main()
    {
        string html = "<h1>Quarterly Report</h1><p>Data as of Q4.</p>";

        HtmlConverter converter = new HtmlConverter(html);
        converter.Convert("report.pdf");
    }
}

After (IronPDF):

using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        string html = "<h1>Quarterly Report</h1><p>Data as of Q4.</p>";

        var renderer = new ChromePdfRenderer();

        using var pdf = renderer.RenderHtmlAsPdf(html);
        pdf.SaveAs("report.pdf");
    }
}

2. Merge PDFs

Before (DynamicPDF Merger):

using ceTe.DynamicPDF.Merger;

class MergeSample
{
    static void Main()
    {
        MergeDocument document = new MergeDocument("doc_a.pdf");
        document.Append("doc_b.pdf");
        document.Draw("merged.pdf");
    }
}

After (IronPDF):

using IronPdf;

class MergeSample
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var docA = PdfDocument.FromFile("doc_a.pdf");
        using var docB = PdfDocument.FromFile("doc_b.pdf");

        using var merged = PdfDocument.Merge(docA, docB);
        merged.SaveAs("merged.pdf");
    }
}

3. Watermark

Before (DynamicPDF Template approach):

using ceTe.DynamicPDF;
using ceTe.DynamicPDF.PageElements;
using ceTe.DynamicPDF.Merger;

class WatermarkSample
{
    static void Main()
    {
        PdfDocument pdfDoc = new PdfDocument("input.pdf");
        Document document = new Document(pdfDoc);

        Template watermark = new Template();
        Label label = new Label("CONFIDENTIAL", 0, 400, 612, 50,
            Font.HelveticaBold, 48);
        label.Align = TextAlign.Center;
        label.TextColor = new RgbColor(200, 200, 200);
        watermark.Elements.Add(label);

        document.Template = watermark;
        document.Draw("watermarked.pdf");
    }
}

After (IronPDF):

using IronPdf;
using IronPdf.Editing;

class WatermarkSample
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var pdf = PdfDocument.FromFile("input.pdf");

        var stamper = new TextStamper
        {
            Text = "CONFIDENTIAL",
            FontSize = 40,
            Opacity = 50,
            Rotation = 45
        };

        pdf.ApplyStamp(stamper);
        pdf.SaveAs("watermarked.pdf");
    }
}

4. Password Protection

Before (DynamicPDF):

using ceTe.DynamicPDF;
using ceTe.DynamicPDF.Cryptography;
using ceTe.DynamicPDF.Merger;

class SecuritySample
{
    static void Main()
    {
        PdfDocument pdfDoc = new PdfDocument("input.pdf");
        Document document = new Document(pdfDoc);

        Aes256Security security = new Aes256Security("user123", "owner456");
        security.AllowPrint = true;
        security.AllowCopy = false;
        security.AllowEdit = false;
        document.Security = security;

        document.Draw("secured.pdf");
    }
}

After (IronPDF):

using IronPdf;
using IronPdf.Security;

class SecuritySample
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var pdf = PdfDocument.FromFile("input.pdf");

        pdf.SecuritySettings.UserPassword = "user123";
        pdf.SecuritySettings.OwnerPassword = "owner456";
        pdf.SecuritySettings.AllowUserPrinting = IronPdf.Security.PdfPrintSecurity.FullPrintRights;

        pdf.SaveAs("secured.pdf");
    }
}

What Are the Critical Migration Gotchas I Should Know About?

There are four sharp edges that will bite you if you're not watching for them. I've seen each of these cause bugs in real migrations.

Coordinate Layout. DynamicPDF's Generator uses points-from-top-left coordinates for every element (new Label("Text", 100, 200, 300, 50)). IronPDF has no coordinate-based layout API. If your codebase is full of Label, TextArea, Image, and Table2 placement calls, you're not migrating a library -- you're migrating to HTML/CSS templates. Plan accordingly.

Draw vs SaveAs. DynamicPDF uses doc.Draw(path) or doc.Draw(stream) as its output method. IronPDF uses pdf.SaveAs(path) or pdf.Stream / pdf.BinaryData for stream output. This is a mechanical find-replace, but watch for spots where you're passing the draw output to another method -- the IronPDF equivalent returns a byte array or MemoryStream directly from the render call.

Page Numbering Placeholders. DynamicPDF uses %%CP%% and %%TP%% in PageNumberingLabel. IronPDF's HtmlHeaderFooter uses {page} and {total-pages}. Search-and-replace the placeholder syntax when porting headers and footers.

Multiple Products, Multiple Licenses. If your code mixes ceTe.DynamicPDF.Generator, ceTe.DynamicPDF.Merger, ceTe.DynamicPDF.HtmlConverter, and ceTe.DynamicPDF.Cryptography, each is a separately-licensed product. Removing all four and dropping in a single IronPdf reference simplifies build, deploy, and renewal. Audit your using statements before you uninstall.

How Should I Handle Performance When Migrating to IronPDF?

The biggest performance win is renderer reuse, and the biggest pitfall is cold start. Here are the patterns I recommend.

Renderer Reuse. ChromePdfRenderer is designed to be reused. Instantiate once, use many times. Avoid newing it per request in a server context.

public class PdfService
{
    private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

    public byte[] GeneratePdf(string html)
    {
        using var pdf = _renderer.RenderHtmlAsPdf(html);
        return pdf.BinaryData;
    }
}

Disposal. PdfDocument implements IDisposable. Use using consistently, especially in loops or server workloads. For parallel rendering patterns, see the IronPDF parallel docs.

Async. IronPDF exposes async render methods. For high-throughput servers, prefer await renderer.RenderHtmlAsPdfAsync(html) over the synchronous form. See the async docs.

Cold Start. IronPDF's first render is slower than subsequent ones because of Chromium initialization. Warm up the renderer at application startup if latency matters. DynamicPDF's layout engine has a lighter startup profile; factor this into your p50/p99 analysis.

Benchmark Aspects Worth Measuring

  • Simple HTML to PDF throughput. Renders/second and p95 latency, measured with BenchmarkDotNet.
  • Memory usage per render. Peak MB and GC pressure, tracked with dotMemory or dotnet-counters.
  • Complex CSS fidelity. Visual diff of output, e.g., Playwright screenshot comparison against a reference render.
  • Parallel render scaling. Renders/sec at N threads, with a custom load test that mirrors your real workload.
  • Cold start time. First-render latency (Chromium init), measured with a Stopwatch around the first call.

Frequently Asked Questions

Can I run DynamicPDF and IronPDF side by side during migration?

Yes. Both libraries use distinct namespaces (ceTe.DynamicPDF vs IronPdf), so they can coexist in the same project. This is the recommended approach for a gradual migration -- migrate one rendering path at a time, validate the output, and remove DynamicPDF references only after the new code is verified in production.

What happens to my programmatic layout code that uses Label, TextArea, and Table2?

This is the sharpest edge in the migration. IronPDF does not have a coordinate-based layout API equivalent to DynamicPDF Generator. If your codebase is full of Label, TextArea, Image, and Table2 placement calls, you're migrating a rendering architecture, not a library. That work belongs in HTML/CSS templates. For HtmlConverter-heavy codebases, the migration is mechanical and the payoff is immediate.

Does IronPDF support the same output method as DynamicPDF's Draw?

DynamicPDF uses doc.Draw(path) for file output and doc.Draw(stream) for stream output. IronPDF uses pdf.SaveAs(path) for files, and exposes pdf.Stream and pdf.BinaryData for stream and byte array output respectively. This is a straightforward find-replace, but check any code that chains the draw output into another method call.

How does IronPDF handle JavaScript in HTML templates compared to DynamicPDF?

DynamicPDF's HtmlConverter has limited JavaScript support. IronPDF uses an embedded Chromium engine that fully executes JavaScript before rendering, which means charting libraries, dynamic tables, and any other JS-dependent content render correctly. This is often one of the primary reasons teams migrate.

What should my post-migration testing checklist look like?

At minimum: visual regression test all PDF outputs using screenshot diffs, test with your most complex HTML/CSS template, verify password-protected output with an actual PDF reader (not just code), confirm merge output page counts equal the sum of inputs, test watermark opacity and rotation, run at least 10 parallel renders under concurrent load, and validate PDF/A compliance with a conformance validator if required.

What Is the Bottom Line on Migrating from DynamicPDF to IronPDF?

The sharpest edge in a DynamicPDF to IronPDF migration is the coordinate-based Generator code. If your codebase is full of Label, TextArea, Image, and Table2 placement calls, you're not migrating a library -- you're migrating a rendering architecture. That work belongs in HTML/CSS templates, not in a swap of using statements. Plan accordingly.

For HtmlConverter-heavy codebases, the migration is mechanical and the payoff is immediate -- full CSS/JS rendering with far less boilerplate, plus collapsing three or four DynamicPDF product licenses into a single IronPDF package. You get Chromium-quality output, proper JavaScript execution, and a simpler API surface. The four code examples above cover the most common patterns, and the API mappings should handle the rest.

Start with a single rendering path, run both libraries side by side, and expand from there. The effort breakdown and decision matrix above will help you scope the work realistically before you commit.
