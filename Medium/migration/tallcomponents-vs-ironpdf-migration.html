Migrating from TallComponents to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should .NET Teams Migrate Away from TallComponents?

The local build works and the unit tests pass. Then a NuGet update warning fires, the vendor's website 301-redirects somewhere unfamiliar, and the support email bounces. TallComponents was acquired by Apryse on 27 May 2025, and Apryse has stopped selling new TallComponents licenses; existing customers are being routed to iText SDK. The NuGet packages (TallComponents.PDFKit5, TallComponents.TallPDF5, and so on) are still published, but no road-map work is happening on the brand. If you're still on TallComponents in 2026, migration is not optional anymore.

I built IronPDF as the modern .NET path for teams in this exact situation. Here are the reasons teams come to us from TallComponents:

Closed to new licenses -- post-acquisition, the only forward path Apryse offers new buyers is the iText SDK; existing TallComponents customers receive support but no road-map development.

XHTML-only HTML pipeline -- TallPDF's XhtmlParagraph parses XHTML 1.0 Strict / XHTML 1.1 + CSS 2.1, not modern HTML5/CSS3/JavaScript. That is unusable for most current marketing or dashboard HTML.

Document-model API -- TallPDF builds PDFs from a Section/TextParagraph/XhtmlParagraph object graph; modern teams want HTML/CSS as the input.

Secondary library accumulation -- teams pairing TallComponents with wkhtmltopdf or Prince to handle real HTML5 end up with two PDF stacks to maintain.

Legacy package shape -- PDFKit 4.x targets .NET Framework 2.0; PDFKit5 targets .NET Standard 2.0. There is no .NET 8+ TFM published, and none is coming.

Strategic dead end -- development energy at Apryse is concentrated on Apryse SDK (PDFTron) and iText, not TallComponents.

How Does Migration Effort Break Down by Feature?

Before committing to a migration, you need to know exactly how much work each feature area requires. I've mapped every common operation to its complexity level so your team can plan sprints accurately.

  • HTML string to PDF -- Low. TallPDF's XhtmlParagraph only parses XHTML 1.0/1.1 + CSS 2.1; IronPDF's ChromePdfRenderer.RenderHtmlAsPdfAsync() uses full Chromium and renders modern HTML5/CSS3.
  • URL to PDF -- Low. XhtmlParagraph.Path accepts an HTTP URL but only for XHTML-compliant pages; renderer.RenderUrlAsPdfAsync() in IronPDF handles JavaScript-driven dashboards.
  • Load existing PDF -- Low. new Document(stream) maps to PdfDocument.FromFile(path).
  • Save to file -- Low. document.Write(stream) maps to pdf.SaveAs(path).
  • Merge PDFs -- Medium. TallComponents iterates pages with outputDoc.Pages.Add(page.Clone()); IronPDF collapses this to PdfDocument.Merge(pdf1, pdf2).
  • Watermark -- Medium. TallComponents uses page.Overlay.Add(TextShape) per page; IronPDF uses TextStamper / ImageStamper with one ApplyStamp call.
  • Password protection -- Medium. Document.Security = new PasswordSecurity { ... } maps to pdf.SecuritySettings; permission flags rename (e.g., AllowPrint becomes AllowUserPrinting, which is an enum rather than a bool).
  • Form field read/write -- Medium. doc.Fields["FieldName"] in TallComponents has analogues in IronPDF -- verify form API parity for the specific field types you use.
  • Text extraction -- Medium. Use pdf.ExtractAllText() in IronPDF; layout fidelity differs from TallComponents.
  • Digital signatures -- Medium-High. SignatureField.Sign(cert) in TallComponents.PDF.Signing maps to PdfSignature + pdf.Sign(); verify PAdES level requirements before committing.

What Are the Key Decision Criteria for Switching?

Not every team should migrate. Here is the decision matrix I walk teams through when they ask me whether switching makes sense for their specific situation.

  • If you are still on TallComponents at all in 2026, plan the switch -- new licenses are no longer sold post-Apryse acquisition and the engine is on maintenance-only support.
  • If your primary need is HTML-to-PDF, IronPDF replaces TallPDF's XHTML pipeline and any secondary HTML5 tool you bolted on with one library.
  • If complex PDF form manipulation is the core use case, verify IronPDF form-fill API parity for the specific field types you rely on before committing.
  • If you depend on PAdES-level signing, verify IronPDF's signing options against your TallComponents SignatureField setup.
  • If a .NET version upgrade is blocked by TallComponents compatibility (PDFKit5 stops at .NET Standard 2.0), that is no longer fixable on the TallComponents side -- switch.

HTML to PDF -- Chromium Replaces an XHTML 1.x Parser

TallPDF accepts HTML through XhtmlParagraph, but the parser only handles XHTML 1.0 Strict / XHTML 1.1 with CSS 2.1 -- no HTML5, no flexbox or grid, no JavaScript, no modern web fonts. Teams either accept that constraint or pair TallComponents with a secondary CLI tool like wkhtmltopdf or Prince to render real-world HTML. With IronPDF, one library uses an embedded Chromium engine and handles both generation and manipulation.

Before (TallComponents -- TallPDF5):

using TallComponents.PDF.Layout;
using TallComponents.PDF.Layout.Paragraphs;
using System.IO;

class HtmlToPdfBefore
{
    static void Main()
    {
        // XhtmlParagraph supports XHTML 1.0/1.1 + CSS 2.1 only.
        // Modern HTML5 / CSS3 / JavaScript will not render reliably.
        Document document = new Document();
        Section section = document.Sections.Add();

        XhtmlParagraph xhtml = new XhtmlParagraph();
        xhtml.Text = "<html><body><h1>Document</h1></body></html>";
        section.Paragraphs.Add(xhtml);

        using (FileStream fs = new FileStream("doc.pdf", FileMode.Create))
        {
            document.Write(fs);
        }
    }
}

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Single library handles generation and manipulation
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Document</h1></body></html>"
);
pdf.SaveAs("doc.pdf");

Console.WriteLine($"Generated doc.pdf ({pdf.PageCount} page(s)) -- one library");

Merging PDFs

Before (TallComponents):

using TallComponents.PDF;
using System.IO;

class MergeBefore
{
    static void Main()
    {
        // PDFKit.NET has no dedicated Merger class -- you append pages
        // by cloning them into a target Document.
        Document outputDoc = new Document();

        using (FileStream fs1 = new FileStream("section1.pdf", FileMode.Open, FileAccess.Read))
        {
            Document doc1 = new Document(fs1);
            foreach (Page page in doc1.Pages)
                outputDoc.Pages.Add(page.Clone());
        }

        using (FileStream fs2 = new FileStream("section2.pdf", FileMode.Open, FileAccess.Read))
        {
            Document doc2 = new Document(fs2);
            outputDoc.Pages.AddRange(doc2.Pages.CloneToArray());
        }

        using (FileStream output = new FileStream("merged.pdf", FileMode.Create))
        {
            outputDoc.Write(output);
        }
    }
}

After (IronPDF):

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

var results = await Task.WhenAll(
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 1</h1></body></html>"),
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 2</h1></body></html>")
);

var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("merged.pdf");
Console.WriteLine($"Merged: {merged.PageCount} pages");

Watermarking

Before (TallComponents):

using TallComponents.PDF;
using TallComponents.PDF.Shapes;
using System.IO;
using System.Drawing;

class WatermarkBefore
{
    static void Main()
    {
        using (FileStream fs = new FileStream("input.pdf", FileMode.Open, FileAccess.Read))
        {
            Document document = new Document(fs);

            foreach (Page page in document.Pages)
            {
                TextShape watermark = new TextShape();
                watermark.Text = "CONFIDENTIAL";
                watermark.Font = new Font("Arial", 60);
                watermark.Pen = new Pen(Color.FromArgb(128, 255, 0, 0));
                watermark.X = 200;
                watermark.Y = 400;
                // TextShape has no Rotate property; rotation goes through Transform.
                watermark.Transform = new RotateTransform(45);

                // Page.Overlay is itself the canvas -- there is no Shapes collection.
                page.Overlay.Add(watermark);
            }

            using (FileStream output = new FileStream("watermarked.pdf", FileMode.Create))
            {
                document.Write(output);
            }
        }
    }
}

After (IronPDF):

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Document</h1></body></html>"
);

var watermark = new TextStamper
{
    Text = "CONFIDENTIAL",
    FontSize = 60,
    Opacity = 50,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("watermarked.pdf");

Password Protection

Before (TallComponents):

using TallComponents.PDF;
using TallComponents.PDF.Security;
using System.IO;

class PasswordBefore
{
    static void Main()
    {
        using (FileStream fs = new FileStream("input.pdf", FileMode.Open, FileAccess.Read))
        {
            Document document = new Document(fs);

            // Security takes a Security object (PasswordSecurity or
            // CertificateSecurity) -- not flat fields on Document.
            PasswordSecurity security = new PasswordSecurity();
            security.UserPassword = "open123";
            security.OwnerPassword = "admin456";
            security.AllowPrint = false;
            security.AllowCopy = false;
            security.KeyLength = KeyLength.Aes256;
            document.Security = security;

            using (FileStream output = new FileStream("protected.pdf", FileMode.Create))
            {
                document.Write(output);
            }
        }
    }
}

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Protected Document</h1></body></html>"
);

pdf.SecuritySettings.UserPassword = "open123";
pdf.SecuritySettings.OwnerPassword = "admin456";

pdf.SaveAs("protected.pdf");

How Do TallComponents and IronPDF Compare Feature by Feature?

Here is the full feature comparison. I've kept this honest -- where I say "verify," it is genuinely worth checking against your specific TallComponents build before you commit.

  • Status. TallComponents is closed to new licenses post-Apryse acquisition (May 2025); IronPDF is actively developed and supported.
  • Focus. TallComponents PDFKit is a document-object API for manipulating existing PDFs, paired with TallPDF for layout-driven generation; IronPDF is HTML-to-PDF plus PDF manipulation, built on Chromium.
  • API style. TallPDF uses a Section/TextParagraph/XhtmlParagraph layout model; PDFKit uses a Document/Page/Overlay manipulation model. IronPDF uses an HTML renderer plus a PdfDocument for manipulation.
  • HTML rendering. TallPDF parses XHTML 1.0/1.1 + CSS 2.1 only; IronPDF embeds Chromium for full HTML5 / CSS3 / ES2024.
  • Learning curve. TallComponents requires learning its own document model; IronPDF is low-friction for any .NET developer who knows HTML/CSS.
  • Pricing. TallComponents licenses can no longer be purchased; IronPDF is commercial -- check ironsoftware.com for current pricing and trial keys.
  • Page indexing. Both PDFKit and IronPDF use 0-based Pages collections, so direct index porting is usually safe; spot-check any arithmetic that assumed 1-based numbering.
  • Thread safety. Verify concurrent-instance guidance for both libraries before fanning out renders in a service.
  • Namespaces. TallComponents.PDF, TallComponents.PDF.Shapes, TallComponents.PDF.Security, TallComponents.PDF.Layout, and TallComponents.PDF.Layout.Paragraphs collapse into IronPdf.

And the core API mapping for the migration itself:

  • renderer.RenderHtmlAsPdfAsync(html) -- generate from HTML; replaces TallPDF's XhtmlParagraph-into-Section pattern.
  • PdfDocument.FromFile(path) -- load existing PDF; replaces new Document(stream).
  • pdf.SaveAs(path) -- save to file; replaces document.Write(stream).
  • pdf.Stream / pdf.BinaryData -- save to stream / bytes; replaces document.Write(stream).
  • pdf.PageCount -- page count; replaces document.Pages.Count.
  • pdf.RemovePages(index) -- remove a page; TallComponents uses document.Pages.RemoveAt(index).
  • pdf.ExtractAllText() -- text extraction; equivalent to walking PDFKit's content stream API.
  • PdfDocument.Merge(doc1, doc2) -- merge; collapses the TallComponents pattern of cloning pages into a target Document.

What Are the Most Common Migration Questions?

Q: Does IronPDF work on Linux CI runners where TallComponents failed?

Yes. IronPDF runs on Linux without COM dependencies or native Windows DLLs. You do need to install a few system packages in your CI container. Here is the GitHub Actions configuration:

# GitHub Actions -- add IronPDF system dependencies for Linux
- name: Install system dependencies
  run: |
    sudo apt-get update
    sudo apt-get install -y libnss3 libatk-bridge2.0-0 libdrm2 libgbm1 libxkbcommon0
    # Verify complete list in IronPDF docs

For Docker deployments, install base font packages to avoid rendering issues:

FROM mcr.microsoft.com/dotnet/aspnet:8.0

RUN apt-get update && apt-get install -y \
    fonts-liberation \
    fonts-noto-core \
    && rm -rf /var/lib/apt/lists/*

COPY --from=build /app/out .
ENTRYPOINT ["dotnet", "YourApp.dll"]

Q: Can I replace both TallComponents and my secondary HTML-to-PDF tool with IronPDF?

Yes -- and that is one of the biggest wins of this migration. TallPDF's XhtmlParagraph only handles XHTML 1.x with CSS 2.1, so teams typically pair it with wkhtmltopdf, Chrome CLI, or Prince when they need modern CSS or JavaScript. The migration replaces both tools with IronPDF.

Audit your pipeline to find secondary tools that can be removed:

# Find all PDF generation tools in the codebase
rg "wkhtmltopdf\|chrome.*headless\|princexml\|PhantomJS" --type cs -n

# These can be removed after IronPDF handles generation

Q: How do I handle stream differences between the two libraries?

TallComponents writes to streams via its own API. IronPDF exposes pdf.Stream (a MemoryStream) and pdf.BinaryData. Map the pattern like this:

// TallComponents pattern:
// document.Write(outputStream);

// IronPDF patterns:
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);

// Pattern 1: Copy stream
pdf.Stream.CopyTo(existingOutputStream);

// Pattern 2: Get bytes
byte[] bytes = pdf.BinaryData;

// Pattern 3: Save to file
pdf.SaveAs("output.pdf");

A common root cause of corruption is a MemoryStream being disposed before the PDF bytes are read. With IronPDF, use pdf.BinaryData or pdf.SaveAs() before the stream is disposed.

Q: How do I set up the IronPDF license key in CI so it doesn't fail in the pipeline?

Store the key as a CI secret and set it as an environment variable. Here is the GitHub Actions configuration:

# GitHub Actions -- set license as secret
- name: Run tests
  env:
    IRONPDF_LICENSE_KEY: ${{ secrets.IRONPDF_LICENSE_KEY }}
  run: dotnet test

In your startup code, fail early with a clear message:

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY")
    ?? throw new InvalidOperationException(
        "IRONPDF_LICENSE_KEY environment variable not set. Add it to CI secrets.");

Get a trial key at ironpdf.com/how-to/license-keys.

What Does the Migration Decision Framework Look Like?

Here is my honest framework for deciding whether this migration makes sense for your team.

Migrate now in almost every case. TallComponents is closed to new licenses on maintenance-only support, and there is no path forward to .NET 8+ or to a Chromium-class HTML engine on the TallComponents side. The effort for most teams is low: load/save/generate operations map directly, and you can eliminate any secondary HTML-to-PDF tool in the same pass.

Verify before committing if complex PDF form manipulation is your core use case. Form APIs differ meaningfully between PDF libraries, so check IronPDF's documentation for the specific form operations you rely on, and the same applies if you depend on PAdES-level digital signing.

One operational note for the cutover: spot-check any custom page-numbering logic that built display strings from document.Pages -- the index is 0-based on both sides, but humans usually see "Page 1 of N", not "Page 0 of N". And always use the using pattern on PdfDocument to avoid memory issues in high-throughput scenarios.
