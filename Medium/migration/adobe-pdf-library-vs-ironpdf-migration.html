Migrating from Adobe PDF Library to IronPDF: A Working Migration in an Afternoon

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

We were four months into a compliance overhaul when the invoice came. Not the Datalogics renewal itself — we expected that — but the line item for the Forms Extension add-on that nobody remembered purchasing, attached to a server that had been decommissioned in Q2. The Adobe PDF Library SDK had been part of our stack for six years, and somewhere in those six years, the gap between what we needed (turn HTML into PDFs, stamp them, encrypt them, ship them) and what we were paying for (a low-level PDF construction toolkit built on Acrobat source code) had become a canyon.

This is not a story about the Adobe PDF Library being bad software. It is, by any measure, the most comprehensive low-level PDF SDK on the market — it literally shares source code with Acrobat. This is a story about what happens when a team that writes HTML all day is forced to think in PDF content streams, page description operators, and coordinate geometry just to generate a report.

This guide walks through the migration to IronPDF. Eighty percent of the content is transferable: dependency auditing, API surface mapping, testing strategies. The remaining twenty percent is IronPDF-specific. Whether you end up using IronPDF, Puppeteer Sharp, or something else entirely, the framework applies.

Why Migrate from Adobe PDF Library to IronPDF?

The Adobe PDF Library is the nuclear reactor of PDF SDKs. It will do anything. The question is whether you need fission when all you want is a light bulb. Here are the triggers that push teams to evaluate.

Paradigm mismatch. Your team writes HTML and CSS. APDFL thinks in content streams, coordinate pairs, and font resource dictionaries. Every "simple" task requires understanding PDF internals.

Licensing complexity. APDFL is licensed through Adobe via Datalogics on a per-deployment, OEM, or ISV basis. The process involves sales conversations, not a "buy now" button. For smaller teams or startups, the procurement overhead is a real cost.

No built-in HTML renderer. APDFL's documented conversion list covers PDF/A, PDF/X, ZUGFeRD, EPS, PS, XPS, and Office formats — but not HTML. Producing a PDF from an HTML string with APDFL alone means hand-building pages, fonts, and content runs, or pairing APDFL with an external HTML-to-PDF engine such as headless Chrome or wkhtmltopdf. Either way, another dependency and often another license.

NuGet package sprawl. A working APDFL installation pulls in Adobe.PDF.Library.LM.NET (or Adobe.PDF.Library.LM.NETFramework) plus platform-specific native runtime payloads, and optionally Adobe.PDF.Library.FormsExtension.LM.NET. Getting the right combination of packages for your target platform takes trial and error.

Learning curve. APDFL's API mirrors the PDF specification. That is an advantage for PDF specialists and a barrier for everyone else. The difference between a PDEElement, a PDEContainer, and a PDEContent is not obvious to a C# developer who just wants to add a watermark.

Native dependency management. APDFL includes significant native C++ code. Platform-specific binaries must be present at runtime. Docker builds need the right base image and runtime packages.

Forms Extension is a separate purchase. If you work with PDF forms — especially XFA forms — the Forms Extension is an add-on with its own license and NuGet package.

COS-level debugging. When something goes wrong with APDFL, the error messages reference COS objects and PDF internal structures. Debugging requires a PDF specification alongside your C# debugger.

.NET version cadence. The current Adobe.PDF.Library.LM.NET package targets .NET 6+, with .NET Framework support split into a separate Adobe.PDF.Library.LM.NETFramework package. Verify your target framework is still on Datalogics' supported matrix before planning a multi-year deployment.

Overkill for generation workflows. If 90% of your PDF work is "take this HTML, produce a PDF, stamp it, encrypt it, email it," you are paying for — and maintaining dependencies on — a full PDF manipulation SDK when a focused HTML-to-PDF library would do.

How Does Adobe PDF Library Compare to IronPDF?

  • Focus. APDFL targets full PDF specification manipulation, rendering, and compliance work. IronPDF is HTML-first PDF generation with manipulation on top.
  • Pricing. APDFL is licensed via Adobe/Datalogics on a per-deployment basis. IronPDF offers per-developer, perpetual licenses.
  • API style. APDFL is low-level — COS objects, PDE elements, content streams. IronPDF is high-level — HTML to PDF and property-based manipulation.
  • Learning curve. APDFL requires PDF specification knowledge. IronPDF is productive quickly for HTML/CSS developers.
  • HTML rendering. APDFL has no built-in HTML-to-PDF engine — teams either build pages programmatically or pair APDFL with an external renderer (headless Chrome, wkhtmltopdf). IronPDF includes a built-in Chrome rendering engine in the core package.
  • Page indexing. Both libraries are 0-based.
  • Thread safety. APDFL requires library-level initialization. IronPDF's ChromePdfRenderer is stateless and safe to share.
  • Namespace. Datalogics.PDFL becomes IronPdf.

How Much Effort Does Each Feature Require to Migrate?

Not every feature migrates at the same difficulty. I have mapped out the effort by feature so you can plan your sprint accordingly.

  • HTML to PDF — Low. APDFL has no native HTML renderer; you build pages by hand or shell out to an external tool. IronPDF is purpose-built for it and ships the renderer in the core package.
  • Programmatic PDF construction — High. APDFL's PDE-level construction has no direct IronPDF equivalent; rethink as HTML templates.
  • Merge — Low. Both offer merge; API shape differs significantly.
  • Split and extract pages — Low. Different API, same concept.
  • Watermark — Medium. APDFL uses PDE layer drawing; IronPDF uses HTML overlay.
  • Password and encryption — Medium. APDFL uses PDDocSetPermissions; IronPDF uses SecuritySettings.
  • Form filling (AcroForms) — Medium. Both support AcroForms; IronPDF uses the FormField collection.
  • XFA forms — High or not applicable. APDFL with the Forms Extension supports dynamic XFA; IronPDF does not handle XFA.
  • PDF/A compliance — Medium-High. APDFL has deep PDF/A support; IronPDF supports PDF/A export — verify version coverage.
  • PDF/X and PDF/UA — High. APDFL's strength; IronPDF support varies — verify in docs.
  • Text extraction — Low. Both offer text extraction with different APIs.
  • Image extraction and rendering — Medium. APDFL renders pages to images natively; verify IronPDF rasterization support.

Decision Matrix

  • If your work is primarily HTML-to-PDF generation with basic manipulation, then migrate — IronPDF eliminates the paradigm mismatch.
  • If you depend on deep PDF/A, PDF/X, or PDF/UA compliance workflows, then evaluate carefully — APDFL's compliance coverage is the industry standard.
  • If you process XFA forms, then stay on APDFL or find a dedicated XFA tool — IronPDF does not handle XFA.
  • If you do programmatic PDF construction with drawing and coordinate layout, then rethink as HTML templates in IronPDF, or keep APDFL for those specific workflows.

How Do I Find and Replace APDFL References in My Codebase?

Before writing any new code, you need an inventory. Here is how I find every touchpoint.

Finding APDFL References

# Find all APDFL using statements
rg -l "Datalogics\.PDFL|using Datalogics" --glob "*.cs"

# Find PDE/COS API usage (indicates deep PDF manipulation)
rg -c "PDEElement|PDEContent|PDEText|CosObj|CosDict" --glob "*.cs"

# Find project references
rg -l "Datalogics" --glob "*.csproj"

If you don't have rg:

grep -rl "Datalogics\|PDFL\|PDEElement" --include="*.cs" .

Key insight: If the rg -c "PDEElement|CosObj" count is high, you have significant low-level PDF construction code. This code cannot be mechanically translated — it needs to be rethought as HTML templates. If the count is low or zero, your codebase is mostly using APDFL for tasks that IronPDF handles directly, and migration will be straightforward.

Swapping NuGet Packages

# Remove APDFL packages
dotnet remove package Adobe.PDF.Library.LM.NET            # .NET 6+
dotnet remove package Adobe.PDF.Library.LM.NETFramework   # .NET Framework
dotnet remove package Adobe.PDF.Library.FormsExtension.LM.NET  # if used

# Install IronPDF
dotnet add package IronPdf

How Do I Migrate License Configuration and Namespaces?

The very first thing that changes is how you initialize the library. APDFL requires a Library object to be alive for the duration of all PDF operations. IronPDF has no such requirement.

Before: APDFL License and Initialization

using Datalogics.PDFL;
using System;

class Program
{
    static void Main()
    {
        // APDFL requires library initialization before any operations.
        // The Library object manages the lifecycle and must stay alive
        // for the duration of all PDF operations.
        Library.LicenseKey = "ADOBE-KEY";
        using (Library lib = new Library())
        {
            Console.WriteLine("APDFL Library initialized.");

            // All APDFL operations must happen within the Library scope
            // ...
        }
    }
}

After: IronPDF License Configuration

using IronPdf;

class Program
{
    static void Main()
    {
        // No library initialization ceremony
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        Console.WriteLine($"IronPDF licensed: {License.IsLicensed}");
        // Start using IronPDF immediately — no scope restrictions
    }
}

The Library initialization object and its lifecycle scope go away entirely.

Namespace Mapping

Here is the before and after for your using statements.

Before:

using Datalogics.PDFL; // single root namespace covers Library,
                       // Document, Page, PDE editing, and rendering

After:

using IronPdf;
using IronPdf.Editing;    // watermarks, headers, footers
using IronPdf.Rendering;  // render options
  • Datalogics.PDFL.Library → no equivalent — IronPDF needs no lifecycle init.
  • Datalogics.PDFL.Document → IronPdf.PdfDocument — document creation and manipulation.
  • Datalogics.PDFL.Watermark / WatermarkTextParams → IronPdf.Editing watermark APIs — content modification.

APDFL exposes a single root namespace, Datalogics.PDFL; Document, Page, Content, Font, and Color all live there. There are no Datalogics.PDFL.Document / Datalogics.PDFL.Page sub-namespaces to swap out.

How Do I Convert Basic PDF Creation from APDFL to IronPDF?

This is where the paradigm shift hits hardest. APDFL constructs PDFs from the inside out — fonts, text runs, content streams, page objects. IronPDF starts from the outside — HTML that your team already writes — and converts it to PDF.

Before: APDFL Programmatic Construction (~25 Lines)

using Datalogics.PDFL;
using System;

class Program
{
    static void Main()
    {
        using (Library lib = new Library())
        {
            // Create a new document
            using (Document doc = new Document())
            {
                // Add a page with specific dimensions (points: 612x792 = Letter)
                Rect pageRect = new Rect(0, 0, 612, 792);
                using (Page page = doc.CreatePage(Document.BeforeFirstPage, pageRect))
                {
                    Content content = page.Content;

                    // PDE-level construction: create a font, build a text
                    // run with size and position, then add it to the page
                    // content stream and update the page.
                    Font font = new Font("Helvetica", FontCreateFlags.Embedded);
                    Text title = new Text();
                    title.AddRun(new TextRun("Monthly Report", font, 24,
                        new Point(72, 720)));
                    content.AddElement(title);
                    page.UpdateContent();
                }

                doc.Save(SaveFlags.Full, "report_apdfl.pdf");
            }
            Console.WriteLine("Saved with APDFL.");
        }
    }
}

After: IronPDF HTML-First Approach (12 Lines)

using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var renderer = new ChromePdfRenderer();
        renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.Letter;

        // Your "PDF construction" is now HTML
        var pdf = renderer.RenderHtmlAsPdf(@"
            <h1 style='font-family:Helvetica;font-size:24pt;'>Monthly Report</h1>
            <p>Generated on " + DateTime.Now.ToString("yyyy-MM-dd") + @"</p>");

        pdf.SaveAs("report_ironpdf.pdf");
        Console.WriteLine("Saved with IronPDF.");
    }
}

The IronPDF HTML-to-PDF tutorial covers rendering options in depth.

How Do I Migrate HTML-to-PDF Generation?

APDFL has no built-in HTML renderer. Teams typically take one of two approaches: build pages, fonts, and text runs programmatically (shown below), or shell out to wkhtmltopdf or headless Chrome as an external process and then optionally post-process with APDFL. IronPDF replaces both patterns with a single in-process renderer in the core package.

Before: APDFL Programmatic Page Construction (Stand-in for HTML Rendering)

using Datalogics.PDFL;
using System;

class HtmlToPdfApdfl
{
    static void Main()
    {
        using (Library lib = new Library())
        {
            // Programmatic construction substitutes for HTML rendering.
            using (Document doc = new Document())
            {
                // US Letter in points (8.5 x 11 inches @ 72 DPI)
                Rect pageRect = new Rect(0, 0, 612, 792);
                using (Page page = doc.CreatePage(Document.BeforeFirstPage, pageRect))
                {
                    Content content = page.Content;
                    Font font = new Font("Helvetica", FontCreateFlags.Embedded);

                    Text text = new Text();
                    text.AddRun(new TextRun("Quarterly Report", font, 24,
                        new Point(72, 720)));
                    content.AddElement(text);

                    page.UpdateContent();
                }
                doc.Save(SaveFlags.Full, "report_apdfl.pdf");
            }
        }
        Console.WriteLine("Generated with APDFL.");
    }
}

// The common alternative is to shell out to wkhtmltopdf or headless
// Chrome and post-process with APDFL — an extra binary on every
// server and an extra process per render.

After: IronPDF Built-In HTML Rendering

using IronPdf;

class HtmlToPdfIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var renderer = new ChromePdfRenderer();
        renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.Letter;
        renderer.RenderingOptions.MarginTop = 20;
        renderer.RenderingOptions.MarginBottom = 20;
        renderer.RenderingOptions.PrintHtmlBackgrounds = true;
        renderer.RenderingOptions.CssMediaType = IronPdf.Rendering.PdfCssMediaType.Screen;

        string html = @"
            <html>
            <head><style>
                body { font-family: 'Segoe UI', sans-serif; margin: 0; }
                .header { background: linear-gradient(135deg, #667eea, #764ba2);
                           color: white; padding: 40px; }
                .content { padding: 30px; }
                table { width: 100%; border-collapse: collapse; margin-top: 20px; }
                th { background: #f8f9fa; text-align: left; }
                td, th { padding: 12px; border-bottom: 1px solid #e9ecef; }
            </style></head>
            <body>
                <div class='header'>
                    <h1>Quarterly Compliance Report</h1>
                    <p>Generated: " + DateTime.Now.ToString("MMMM dd, yyyy") + @"</p>
                </div>
                <div class='content'>
                    <table>
                        <tr><th>Control</th><th>Status</th><th>Last Audit</th></tr>
                        <tr><td>SOC 2 Type II</td><td>Compliant</td><td>2025-12-15</td></tr>
                        <tr><td>PCI DSS</td><td>Compliant</td><td>2025-11-20</td></tr>
                        <tr><td>HIPAA</td><td>Review Needed</td><td>2025-10-01</td></tr>
                    </table>
                </div>
            </body>
            </html>";

        var pdf = renderer.RenderHtmlAsPdf(html);
        pdf.SaveAs("compliance_report.pdf");
        Console.WriteLine("Saved with IronPDF.");
    }
}

The hand-rolled page, font, and text-run construction collapses into a single RenderHtmlAsPdf call. If you were on the wkhtmltopdf fallback, the external binary, Process.Start, and exit-code checking all disappear. Either way, CSS gradients, web fonts, and modern layouts render natively in-process.

How Do I Migrate PDF Merging?

Merge is one of the simplest operations to port. APDFL uses InsertPages with index arithmetic and flags. IronPDF provides a static Merge method.

Before: APDFL InsertPages Merge

using Datalogics.PDFL;
using System;

class MergeApdfl
{
    static void Main()
    {
        using (Library lib = new Library())
        {
            using (Document doc1 = new Document("part1.pdf"))
            using (Document doc2 = new Document("part2.pdf"))
            {
                // Insert all pages from doc2 at the end of doc1.
                // InsertPages takes a destination index, source doc,
                // source start index, count, and an insert flag.
                doc1.InsertPages(
                    Document.LastPage,
                    doc2,
                    0,
                    Document.AllPages,
                    PageInsertFlags.None
                );

                doc1.Save(SaveFlags.Full, "merged_apdfl.pdf");
            }
            Console.WriteLine("Merged with APDFL.");
        }
    }
}

After: IronPDF Static Merge

using IronPdf;

class MergeIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var pdf1 = PdfDocument.FromFile("part1.pdf");
        var pdf2 = PdfDocument.FromFile("part2.pdf");

        var merged = PdfDocument.Merge(pdf1, pdf2);
        merged.SaveAs("merged.pdf");
        Console.WriteLine("Merged with IronPDF.");
    }
}

No page-index arithmetic. No insert flags. See the IronPDF merge documentation.

How Do I Migrate Watermarking?

In APDFL, watermarking is a configurable helper: build a WatermarkParams object with opacity, rotation, and scale, build a WatermarkTextParams object with the text and color, then call doc.Watermark(textParams, watermarkParams). In IronPDF, it is a single method call with an HTML string.

Before: APDFL Document.Watermark

using Datalogics.PDFL;
using System;

class WatermarkApdfl
{
    static void Main()
    {
        using (Library lib = new Library())
        using (Document doc = new Document("input.pdf"))
        {
            // Configure watermark placement / appearance
            WatermarkParams watermarkParams = new WatermarkParams();
            watermarkParams.Opacity = 0.5;
            watermarkParams.Rotation = 45.0;
            watermarkParams.Scale = -1; // auto-fit

            // Configure the text content of the watermark
            WatermarkTextParams textParams = new WatermarkTextParams();
            textParams.Text = "CONFIDENTIAL";
            textParams.Color = new Color(0.8, 0.8, 0.8);
            textParams.TextAlign = HorizontalAlignment.Center;

            // Apply the watermark via the Document.Watermark(...) method.
            doc.Watermark(textParams, watermarkParams);

            doc.Save(SaveFlags.Full, "watermarked_apdfl.pdf");
            Console.WriteLine("Watermarked with APDFL.");
        }
    }
}

After: IronPDF HTML Watermark

using IronPdf;

class WatermarkIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var pdf = PdfDocument.FromFile("input.pdf");

        pdf.ApplyWatermark(
            "<h1 style='color:rgba(200,200,200,0.5);font-size:48px;font-family:Helvetica;'>CONFIDENTIAL</h1>",
            rotation: 45
        );

        pdf.SaveAs("watermarked.pdf");
        Console.WriteLine("Watermarked with IronPDF.");
    }
}

Two parameter objects and a method call — replaced by one CSS-styled HTML string. IronPDF's opacity argument takes an integer 0–100; APDFL's WatermarkParams.Opacity takes a 0.0–1.0 fraction. Watch the scale when porting values.

How Do I Migrate Password Protection?

APDFL constructs an EncryptionHandler with user and owner passwords plus permission flags, attaches it to the document, then saves with SaveFlags.Encrypted. IronPDF uses named properties with typed enums on the SecuritySettings object.

Before: APDFL EncryptionHandler

using Datalogics.PDFL;
using System;

class PasswordApdfl
{
    static void Main()
    {
        using (Library lib = new Library())
        {
            using (Document doc = new Document("input.pdf"))
            {
                // Build permissions — restrict everything by passing
                // an empty flag set, or OR together specific allowances.
                PermissionFlags permissions = PermissionFlags.None;

                EncryptionHandler encHandler = new EncryptionHandler(
                    "user456",          // user password
                    "owner123",         // owner password
                    permissions,
                    EncryptionMethod.AES256
                );

                doc.SetEncryptionHandler(encHandler);
                doc.Save(SaveFlags.Full | SaveFlags.Encrypted, "protected_apdfl.pdf");
            }
            Console.WriteLine("Protected with APDFL.");
        }
    }
}

After: IronPDF SecuritySettings

using IronPdf;

class PasswordIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var pdf = PdfDocument.FromFile("input.pdf");

        pdf.SecuritySettings.OwnerPassword = "owner123";
        pdf.SecuritySettings.UserPassword = "user456";
        pdf.SecuritySettings.AllowUserPrinting = IronPdf.Security.PdfPrintSecurity.NoPrint;
        pdf.SecuritySettings.AllowUserCopyPasteContent = false;
        pdf.SecuritySettings.AllowUserEdits = IronPdf.Security.PdfEditSecurity.NoEdit;

        pdf.SaveAs("protected.pdf");
        Console.WriteLine("Protected with IronPDF.");
    }
}

See the IronPDF security documentation for the full list of permission options.

What Are the Core API Mappings Between APDFL and IronPDF?

Here are the class-level and method-level mappings I used during our migration. Keep these tables open in a second monitor.

Core Class Mapping

  • Library → no equivalent — APDFL requires global init; IronPDF does not.
  • Document → PdfDocument — the PDF document object.
  • Page → pdf.Pages[n] — page access (both 0-based).
  • Font / PDEFont → CSS font-family in HTML — IronPDF uses web fonts via CSS.

Document Loading Methods

  • Open from file. new Document("file.pdf") → PdfDocument.FromFile("file.pdf").
  • Open with password. new Document("file.pdf", "password") → PdfDocument.FromFile("file.pdf", "pass").
  • Create new empty. new Document() → renderer.RenderHtmlAsPdf("<html>...</html>").
  • Save. doc.Save(SaveFlags.Full, "output.pdf") → pdf.SaveAs("output.pdf").

Page Operations

  • Get page count. doc.NumPages → pdf.PageCount.
  • Get specific page. doc.GetPage(index) → pdf.Pages[index] (both 0-based).
  • Create page. doc.CreatePage(position, rect) → render HTML content (pages auto-created).
  • Delete page. doc.DeletePages(...) → pdf.RemovePages(index).

Merge / Split Operations

  • Merge documents. doc.InsertPages(...) → PdfDocument.Merge(pdf1, pdf2).
  • Extract pages. doc.DeletePages loop or page copy → pdf.CopyPages(start, end).

What Are the Critical Edge Cases and Performance Considerations?

The most important thing to understand about this migration is the paradigm shift. APDFL constructs PDFs by assembling low-level elements — it is a PDF assembler. IronPDF converts web content to PDFs — it is a PDF compiler, where HTML is the source language. If your codebase has significant PDEElement, PDEText, CosObj, or content stream code, you cannot simply translate it to IronPDF API calls. You need to express the same visual output as HTML/CSS and let IronPDF's Chrome engine render it.

Library initialization scope. APDFL requires a Library object to be alive for the duration of all PDF operations. Forgetting to keep it in scope or disposing it early causes crashes. IronPDF has no such requirement — call any API at any time after setting the license key.

Save flags. APDFL's Save method takes SaveFlags that control incremental vs. full saves. IronPDF's SaveAs always performs a full save. If you relied on incremental saves for performance with large documents, test the IronPDF SaveAs performance for your file sizes.

Font handling. In APDFL, you explicitly create Font objects and embed them. In IronPDF, fonts come from CSS. If you use custom fonts, ensure they are installed on the server or loaded via @font-face in your HTML. IronPDF's Chrome engine handles font embedding automatically during render.

Eliminating the external process. If your APDFL setup used wkhtmltopdf or headless Chrome as a subprocess for HTML-to-PDF, migrating to IronPDF eliminates that process boundary. The rendering engine runs in-process, which removes process startup/teardown overhead per render, temp file I/O between the subprocess and your .NET code, and port or file-handle management for the external process.

Renderer Reuse

// Create once, reuse across all requests
private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

public PdfDocument RenderReport(string html)
{
    return _renderer.RenderHtmlAsPdf(html);
}

Disposal

using var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");

PDF/A output: APDFL has deep PDF/A-1a, 1b, 2a, 2b, 3a, 3b support. IronPDF supports PDF/A export — verify which conformance levels are covered for your requirements.

XFA forms: If you use APDFL's Forms Extension for dynamic XFA, there is no IronPDF equivalent. AcroForms (static forms) are supported. Consider flattening XFA to AcroForms before migration, or maintaining APDFL for that specific workflow.

Print-quality rendering: APDFL can render PDFs to TIFF at 300+ DPI for print workflows. If you use this, verify IronPDF's rasterization capabilities meet your DPI requirements.

COS-level manipulation: If your code directly manipulates the COS object tree (the raw PDF internal structure), this cannot be ported. Evaluate whether the same outcome can be achieved through IronPDF's higher-level API or HTML-based generation.

Frequently Asked Questions

Can I migrate incrementally, keeping APDFL and IronPDF side by side?

Yes, and I recommend it. Both libraries can coexist in the same solution since they have completely separate namespaces (Datalogics.PDFL vs. IronPdf). Migrate one workflow at a time — start with HTML-to-PDF generation since that is the lowest-complexity swap. Keep APDFL for any XFA or COS-level workflows until you have verified replacements.

The key constraint is that APDFL's Library initialization must remain alive for any code path still using APDFL. IronPDF has no such constraint, so adding it alongside existing code introduces no lifecycle complications.

What happens to my XFA form workflows?

IronPDF does not handle XFA forms. If you rely on APDFL's Forms Extension for dynamic XFA processing, you have three options: keep APDFL for that specific workflow, flatten XFA forms to AcroForms before migration (IronPDF supports AcroForms via the FormField collection), or find a dedicated XFA tool.

This is the one area where APDFL has no direct IronPDF replacement. Do not attempt to force-fit XFA processing into IronPDF.

Will IronPDF match APDFL's PDF/A compliance output?

APDFL has the deepest PDF/A support in the industry, covering PDF/A-1a, 1b, 2a, 2b, 3a, and 3b conformance levels. IronPDF supports PDF/A export, but you should verify which specific conformance levels are covered for your requirements. If you need PDF/X or PDF/UA, evaluate carefully — these are APDFL's strengths and IronPDF support varies by version.

I recommend validating output with a tool like veraPDF before committing to the migration for compliance-critical workflows.

How does the performance compare when replacing the wkhtmltopdf subprocess?

If your current pipeline shells out to wkhtmltopdf alongside APDFL, migrating to IronPDF eliminates that entire process boundary. You remove process startup and teardown overhead per render, temp file I/O between the subprocess and your .NET code, and port or file-handle management. IronPDF's Chrome engine runs in-process. Reuse the ChromePdfRenderer instance across requests — it is stateless and safe to share.

Beyond the architectural improvement, you also stop depending on wkhtmltopdf, which is a deprecated WebKit-based renderer with known security vulnerabilities that stopped receiving updates.

What is the minimum .NET version required for IronPDF?

IronPDF supports .NET 6+ and .NET Framework 4.6.2+. Datalogics ships Adobe.PDF.Library.LM.NET for .NET 6+ and a separate Adobe.PDF.Library.LM.NETFramework package for .NET Framework targets — verify their current support matrix before committing to a long-lived deployment. Wrapping Up

Six years of APDFL taught our team a lot about the PDF specification. That knowledge does not go away — it makes you better at debugging any PDF library's output. But the daily reality of constructing compliance reports in content streams when we could express the same thing in HTML and CSS was a cost we no longer needed to pay.

If your situation is similar — if most of your PDF work is generation from web content with some manipulation bolted on — the migration to an HTML-first library saves real engineering hours. If your situation involves deep PDF specification work (XFA, PDF/UA tagging, COS-level surgery), APDFL earns its complexity.

If you are currently shelling out to wkhtmltopdf alongside APDFL, you are running a deprecated, WebKit-based renderer that has known security vulnerabilities and stopped receiving updates. Regardless of where you migrate, replacing that subprocess should be a priority.
