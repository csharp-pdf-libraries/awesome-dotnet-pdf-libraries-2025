Migrating from FastReport to IronPDF: The Decision Guide for .NET Teams

Here's a scenario that comes up more often than it should: a .NET application has been generating invoices, purchase orders, and reports via FastReport for years. The reports look exactly right — pixel-perfect, precisely positioned, every field bound to data. Then someone asks: "Can we just take this HTML email template and turn it into a PDF?" And that's where things get complicated.

FastReport is a reporting tool. It excels at data-bound, template-designed documents. IronPDF is an HTML-to-PDF renderer. They solve different problems, but they overlap in the "produce a PDF" outcome enough that teams regularly need to make a choice. This article is for teams maintaining both tools because FastReport does the reports and something else does the HTML rendering — and they want to consolidate.

By the end, you'll have the migration patterns, API mappings, and code samples for the HTML-to-PDF half of that consolidation.

Why Should You Migrate Away from FastReport for HTML-to-PDF?

These triggers apply specifically to teams trying to reduce the FastReport footprint for HTML-rendering use cases.

First, HTML input isn't FastReport's model. FastReport uses .frx template files and a designer. Teams needing to render HTML emails, Razor-generated pages, or web templates find FastReport is the wrong tool.

Second, the template designer dependency. FastReport's report model requires visual designer tooling for layout changes. HTML templates can be edited in any editor.

Third, data binding model mismatch. FastReport binds data to report bands via RegisterData() and DataBand objects. If your data is already in an HTML template, FastReport adds unnecessary ceremony.

Fourth, deployment weight. FastReport is a full reporting suite that ships as multiple NuGet packages (FastReport.OpenSource, FastReport.OpenSource.Export.PdfSimple, FastReport.OpenSource.Web, etc.). If your use case is "HTML to PDF," a targeted HTML renderer may be lighter.

Fifth, license scope. The commercial FastReport.Net edition is published on FastReport's private NuGet server (not on nuget.org — only the page-limited FastReport.Net.Demo is). The MIT-licensed FastReport.OpenSource is on nuget.org but ships FastReport.OpenSource.Export.PdfSimple, which rasterizes pages — output PDFs contain images of text rather than selectable text. Selectable-text PDF, encryption, digital signing, and PDF/A all require the commercial edition. If you're paying for FastReport solely to get real PDF export, evaluate fit.

Sixth, CSS fidelity in HTML exports. FastReport's HtmlObject renders a constrained subset of HTML inside a banded report. Web-standard styling — Flexbox, Grid, modern CSS — going through a report engine will vary.

Seventh, maintenance of .frx files. Report template files are XML, not version-control-friendly in practice. HTML templates are plain text.

Eighth, async rendering. Verify whether FastReport's rendering pipeline supports async patterns for high-throughput server use; IronPDF exposes RenderHtmlAsPdfAsync directly.

Ninth, dynamic content. For PDFs whose structure changes based on data (not just field values but layout structure), HTML templates are often more flexible than banded report templates.

FastReport is a data-bound report designer with multi-format export, while IronPDF renders HTML or URLs to PDF via Chromium. FastReport ships in two editions: a commercial FastReport.Net on a private NuGet feed and an MIT-licensed FastReport.OpenSource on nuget.org with image-only PDF export. IronPDF is commercial; check the vendor pricing page for current numbers. The API style differs sharply: FastReport loads .frx templates, registers data, and exports via PDFSimpleExport or PDFExport; IronPDF passes HTML or a URL into ChromePdfRenderer and gets a PdfDocument back. The learning curve reflects that — the report designer model takes a few projects to internalize, while HTML-first workflows are immediate. HTML rendering quality is a clear differentiator: FastReport routes HTML through its HtmlObject with limited CSS fidelity, while IronPDF runs full Chromium with CSS and JavaScript. Thread safety: ChromePdfRenderer is reusable as a singleton; treat Report instances as single-use and dispose after export. The core namespace is FastReport on one side and IronPdf on the other.

How Complex Is the Migration Effort?

The complexity depends entirely on what you're migrating. If you're moving HTML-to-PDF paths, it's straightforward. If you're trying to replace banded report generation, I'd recommend against it — that's FastReport's strength.

  • HTML string to PDF — Low. FastReport doesn't do this natively, so this is net-new functionality on the IronPDF side rather than a rewrite.
  • URL to PDF — Low. Same story — net-new for FastReport teams; RenderUrlAsPdf in IronPDF is a one-liner.
  • Data-bound report to PDF — High. Core FastReport use case. Don't migrate this; you'd be rebuilding the band model in HTML for no real gain.
  • Banded layout to HTML template — High. Architectural change, not a library swap. Estimate template-by-template.
  • Headers, footers, and page numbers — Low. FastReport's PageHeaderBand/PageFooterBand with [Page]/[TotalPages] placeholders maps cleanly to IronPDF's HtmlHeaderFooter with {page}/{total-pages}.
  • Master-detail and nested data — Medium. FastReport's nested DataBand with RegisterData(ds, name, maxNesting) becomes a foreach-in-foreach in your HTML template engine.
  • Multi-page reports with page breaks — Low. DataBand.StartNewPage maps to CSS page-break-before: always.
  • Dynamic charts — Medium. FastReport has built-in chart objects; IronPDF renders JS charts (Chart.js, D3, Highcharts) inside the HTML.
  • Long-running report (100+ pages) — Verify. FastReport is optimized for this; verify IronPDF for large docs against your specific workload before committing.

What's the Right Decision for Your Scenario?

The most honest migration advice here: if your FastReport .frx templates are working and your stakeholders don't want to move to HTML-based templates, don't migrate. FastReport does its job. The migration makes sense when you're already generating HTML and want to convert it, not when you're trying to recreate report templates in a different tool.

If all your PDFs are data-bound banded reports, keep FastReport — IronPDF won't improve this workload. If you have a mix of banded reports and HTML-template PDFs, keep both and use each for what it's built for. If you're producing HTML-template PDFs only, with no banded reports, migrate fully to IronPDF and retire FastReport. And if you specifically need to render existing .frx reports to PDF programmatically, stay on FastReport for that workload.

Teams often find themselves maintaining two PDF pathways: FastReport for structured reports and a separate library (or hand-rolled solution) for HTML-to-PDF. A common pattern:

Invoice report       → FastReport (.frx template) → PDF (works)
Email confirmation   → HtmlString + ad-hoc lib    → PDF (fragile)
Dynamic dashboard    → Razor view                 → PDF (broken CSS)
Compliance doc       → HTML template              → PDF (no JS)

The consolidation case is: replace the ad-hoc HTML-to-PDF paths with IronPDF, and keep FastReport for the banded reports that use .frx templates. This reduces the number of "how do we convert X to PDF" conversations without disrupting working report templates.

The scenario you want to avoid: trying to rewrite working FastReport templates as HTML because you want one library. Unless there's a specific reason to do this (maintenance burden on .frx files, designer tooling cost), it adds risk for marginal benefit.

How Do I Set Up and Start the Migration?

Before you touch any code, identify the split in your codebase between report-to-PDF paths (keep FastReport) and HTML-to-PDF paths (migrate to IronPDF).

Identify the HTML-to-PDF vs. Report-to-PDF Split

# Find FastReport report loading — these are your report-PDF paths (keep FastReport)
rg "new Report\(\)|\.Load\(" --type cs

# Find HTML string being passed to FastReport or converted elsewhere — migration targets
rg "HtmlContent\|htmlString\|renderHtml" --type cs

# Find Razor view rendering to PDF — migration candidates
rg "RenderToString\|ViewResult.*pdf\|html.*pdf" --type cs -l

Install IronPDF for HTML-to-PDF Paths

# If fully replacing FastReport (only if you've confirmed no banded report usage):
dotnet remove package FastReport.OpenSource
dotnet remove package FastReport.OpenSource.Export.PdfSimple

# Install IronPDF for HTML-to-PDF paths

dotnet list package

Set Your License

How Do I Convert HTML Strings to PDF with IronPDF?

This is the core migration path. FastReport doesn't have a native HTML-to-PDF API — its HtmlObject embeds a small block of HTML inside a banded report — so teams typically have ad-hoc workarounds or a separate library for raw HTML rendering. IronPDF makes it a first-class operation.

License Setup

// IronPDF license
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

Namespace Imports

// Before (FastReport HTML-via-HtmlObject path)
using FastReport;
using FastReport.Export.PdfSimple;
using System.IO;

// After (IronPDF)
using IronPdf;

Basic HTML to PDF

// Before (FastReport — HtmlObject inside a banded report — the closest FastReport gets to HTML-to-PDF)
using FastReport;
using FastReport.Export.PdfSimple;
using System.Drawing;
using System.IO;

using (Report report = new Report())
{
    ReportPage page = new ReportPage();
    report.Pages.Add(page);
    page.ReportTitle = new ReportTitleBand { Height = Units.Millimeters * 100 };

    HtmlObject htmlObject = new HtmlObject();
    htmlObject.Bounds = new RectangleF(0, 0, Units.Millimeters * 190, Units.Millimeters * 100);
    htmlObject.Text = "<html><body><h1>Hello World</h1></body></html>";
    page.ReportTitle.Objects.Add(htmlObject);

    report.Prepare();

    // PDFSimpleExport rasterizes pages — text is not selectable
    var pdfExport = new PDFSimpleExport();
    using (var fs = new FileStream("output.pdf", FileMode.Create))
    {
        report.Export(pdfExport, fs);
    }
}

// After (IronPDF)
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
var renderer = new ChromePdfRenderer();
using var pdf = renderer.RenderHtmlAsPdf("<html><body><h1>Hello World</h1></body></html>");
pdf.SaveAs("output.pdf");
// Docs: https://ironpdf.com/how-to/html-string-to-pdf/

How Do the FastReport and IronPDF APIs Map to Each Other?

The two libraries have fundamentally different models, but I can give you concrete mappings for the overlapping operations. Use these as a quick reference during migration.

Namespace Mapping

  • FastReport → IronPdf — core namespace
  • FastReport.Export.Pdf → IronPdf — PDF export, built into the main namespace
  • FastReport.Export.PdfSimple → IronPdf — simplified export, also built in
  • FastReport.Utils → IronPdf.Rendering — rendering utilities
  • FastReport.Data → standard .NET data access — no FastReport-specific layer needed

Core Class Mapping

  • Report → ChromePdfRenderer — main rendering class (different model)
  • PDFExport / PDFSimpleExport → ChromePdfRenderer + PdfDocument.SaveAs() — rendering plus output
  • Report.Prepare() → renderer.RenderHtmlAsPdf() — render trigger
  • ReportPage → HTML <body> or <div>
  • TextObject → HTML <p>, <span>, <div>
  • TableObject → HTML <table>
  • DataBand → loop in template — foreach in your HTML generator
  • PageHeaderBand / PageFooterBand → HtmlHeaderFooter
  • HtmlObject → direct HTML rendering via RenderHtmlAsPdf
  • PictureObject → HTML <img>
  • BarcodeObject → IronBarCode (separate package) or SVG
  • MemoryStream + Export → pdf.BinaryData or pdf.Stream — in-memory output

Document Loading (HTML Path Only)

  • HTML string to PDF — FastReport has no native API; IronPDF: renderer.RenderHtmlAsPdf(html)
  • URL to PDF — FastReport has no native API; IronPDF: renderer.RenderUrlAsPdf(url)
  • Load existing PDF — not a FastReport workflow; IronPDF: PdfDocument.FromFile(path)
  • Load from stream — not a FastReport workflow; IronPDF: PdfDocument.FromStream(stream)

Page Operations

  • Page count — FastReport: report.Pages.Count; IronPDF: pdf.PageCount
  • Page size — FastReport: report template setting; IronPDF: RenderingOptions.PaperSize
  • Remove page — FastReport: edit the template; IronPDF: pdf.RemovePages(index)
  • Margins — FastReport: report template setting; IronPDF: RenderingOptions.MarginTop/Bottom/Left/Right

Page Number Placeholders

  • [Page] → {page} — current page number
  • [TotalPages] → {total-pages} — total page count
  • [PageNofM] → {page} of {total-pages} — combined
  • [Date] → {date} — current date
  • [Time] → {time} — current time

Merge / Split

  • Merge reports — FastReport: prepare multiple reports and combine via a third-party PDF merge; IronPDF: PdfDocument.Merge(a, b)
  • Split — FastReport: not a first-class operation; IronPDF: pdf.CopyPages(start, end)

What Does a Full Before/After Migration Look Like for HTML to PDF?

Let me walk through the most common migration pattern. FastReport's model is load a .frx template, bind data, export to PDF. For raw HTML input, teams typically use workarounds. IronPDF handles HTML natively.

FastReport Before Code (Report-Based Workaround)

using FastReport;
using FastReport.Export.PdfSimple;
using System.IO;

class Program
{
    static void Main()
    {
        // FastReport's model: load a .frx template, bind data, export to PDF.
        // For HTML input, teams typically:
        //   Option A: Embed an HtmlObject inside a banded report
        //   Option B: Use a separate HTML-to-PDF library alongside FastReport
        //   Option C: Render the report to HTML then run another tool over it

        using (var report = new Report())
        {
            report.Load("template.frx"); // designer-created template

            // Bind data
            report.RegisterData(myDataSource, "DataSource");
            report.GetDataSource("DataSource").Enabled = true;
            report.Prepare();

            // Export to PDF
            var pdfExport = new PDFSimpleExport();
            using var ms = new MemoryStream();
            report.Export(pdfExport, ms);

            File.WriteAllBytes("report.pdf", ms.ToArray());
        }
    }
}

IronPDF After Code (Direct HTML Input)

using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
        // Docs: https://ironpdf.com/how-to/license-keys/

        // HTML template populated by your Razor/Scriban/Handlebars engine
        string html = BuildHtmlTemplate(myData);

        var renderer = new ChromePdfRenderer();
        renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
        renderer.RenderingOptions.MarginTop = 15;
        renderer.RenderingOptions.MarginBottom = 15;
        // Docs: https://ironpdf.com/how-to/rendering-options/

        using var pdf = renderer.RenderHtmlAsPdf(html);
        pdf.SaveAs("report.pdf");
    }

    static string BuildHtmlTemplate(object data)
    {
        // Your existing template engine (Razor, Scriban, Handlebars, etc.)
        return $"<html><body><h1>Report for {data}</h1></body></html>";
    }
}

How Do I Replace RegisterData and DataBand Binding?

Data binding is the heart of FastReport. The RegisterData + DataBand pair iterates rows from a DataTable or DataSet and emits one band per row. With IronPDF you do the iteration yourself in your template engine — usually a foreach in Razor or string interpolation — and the result is just HTML that the Chromium renderer turns into PDF.

FastReport Before Code (Tabular Data via RegisterData)

using FastReport;
using FastReport.Export.PdfSimple;
using System.Data;

class Program
{
    static void Main()
    {
        // Build the data
        DataTable products = new DataTable("Products");
        products.Columns.Add("Name", typeof(string));
        products.Columns.Add("Price", typeof(decimal));
        products.Columns.Add("Quantity", typeof(int));
        products.Rows.Add("Widget A", 29.99m, 100);
        products.Rows.Add("Widget B", 49.99m, 50);
        products.Rows.Add("Widget C", 19.99m, 200);

        using (var report = new Report())
        {
            report.Load("products.frx");                  // template with DataBand
            report.RegisterData(products, "Products");
            report.GetDataSource("Products").Enabled = true;
            report.Prepare();

            var pdfExport = new PDFSimpleExport();
            using var fs = new FileStream("products.pdf", FileMode.Create);
            report.Export(pdfExport, fs);
        }
    }
}

IronPDF After Code (Loop in HTML Template)

using IronPdf;
using System.Text;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var products = new[]
        {
            new { Name = "Widget A", Price = 29.99m, Quantity = 100 },
            new { Name = "Widget B", Price = 49.99m, Quantity = 50 },
            new { Name = "Widget C", Price = 19.99m, Quantity = 200 }
        };

        var html = new StringBuilder();
        html.Append(@"
            <html><head><style>
              body { font-family: Arial, sans-serif; padding: 20px; }
              table { width: 100%; border-collapse: collapse; }
              th { background: #4CAF50; color: white; padding: 12px; text-align: left; }
              td { border: 1px solid #ddd; padding: 10px; }
              tr:nth-child(even) { background: #f2f2f2; }
            </style></head><body>
              <h1>Product Catalog</h1>
              <table>
                <thead><tr><th>Product</th><th>Price</th><th>Quantity</th></tr></thead>
                <tbody>");

        foreach (var p in products)
        {
            html.Append($"<tr><td>{p.Name}</td><td>${p.Price:F2}</td><td>{p.Quantity}</td></tr>");
        }

        html.Append("</tbody></table></body></html>");

        var renderer = new ChromePdfRenderer();
        using var pdf = renderer.RenderHtmlAsPdf(html.ToString());
        pdf.SaveAs("products.pdf");
    }
}

For master-detail data, FastReport's RegisterData(ds, name, maxNestingLevel) with a parent DataBand containing a child DataBand becomes a nested foreach in your HTML builder. The hierarchy lives in your domain model (Order with a List<OrderDetail> Details) instead of in a DataRelation on a DataSet.

How Do I Migrate Headers, Footers, and Page Numbers?

FastReport puts headers and footers into PageHeaderBand and PageFooterBand objects on each ReportPage, and uses [Page] and [TotalPages] placeholders in TextObjects for pagination. IronPDF uses a pair of HtmlHeaderFooter objects on the renderer's RenderingOptions, with {page} and {total-pages} tokens. The mapping is direct.

FastReport Before Code (PageHeaderBand / PageFooterBand)

using FastReport;
using FastReport.Export.PdfSimple;
using System.Drawing;
using System.IO;

class Program
{
    static void Main()
    {
        using (var report = new Report())
        {
            report.Load("template.frx");
            var page = (ReportPage)report.Pages[0];

            page.PageHeader = new PageHeaderBand { Height = Units.Millimeters * 15 };
            var headerText = new TextObject
            {
                Bounds = new RectangleF(0, 0, Units.Millimeters * 190, Units.Millimeters * 10),
                Text = "Document Header",
                HorzAlign = HorzAlign.Center
            };
            page.PageHeader.Objects.Add(headerText);

            page.PageFooter = new PageFooterBand { Height = Units.Millimeters * 15 };
            var footerText = new TextObject
            {
                Bounds = new RectangleF(0, 0, Units.Millimeters * 190, Units.Millimeters * 10),
                Text = "Page [Page] of [TotalPages]",
                HorzAlign = HorzAlign.Right
            };
            page.PageFooter.Objects.Add(footerText);

            report.Prepare();

            var pdfExport = new PDFSimpleExport();
            using var fs = new FileStream("report.pdf", FileMode.Create);
            report.Export(pdfExport, fs);
        }
    }
}

IronPDF After Code (HtmlHeaderFooter with {page} / {total-pages})

using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var renderer = new ChromePdfRenderer();

        renderer.RenderingOptions.HtmlHeader = new HtmlHeaderFooter
        {
            HtmlFragment = "<div style='text-align:center; font-family:Arial; font-weight:bold;'>Document Header</div>",
            DrawDividerLine = true
        };

        renderer.RenderingOptions.HtmlFooter = new HtmlHeaderFooter
        {
            HtmlFragment = "<div style='text-align:right;'>Page {page} of {total-pages}</div>",
            DrawDividerLine = true
        };

        using var pdf = renderer.RenderHtmlAsPdf("<html><body><h1>Report Content</h1></body></html>");
        pdf.SaveAs("report.pdf");
        // Docs: https://ironpdf.com/how-to/html-headers-and-footers/
    }
}

How Do I Merge PDFs After Migration?

FastReport doesn't have a built-in merge. Teams typically prepare multiple reports, export to memory streams, and then use a separate PDF merge library. IronPDF has native merge support.

FastReport Before Code (Multi-Report Export)

using FastReport;
using FastReport.Export.PdfSimple;
using System.IO;

class MergeSample
{
    static void Main()
    {
        // FastReport: prepare multiple reports, export to memory, then merge with a separate library

        using var reportA = new Report();
        reportA.Load("report_a.frx");
        reportA.Prepare();

        using var reportB = new Report();
        reportB.Load("report_b.frx");
        reportB.Prepare();

        var pdfExport = new PDFSimpleExport();
        using var msA = new MemoryStream();
        using var msB = new MemoryStream();

        reportA.Export(pdfExport, msA);
        reportB.Export(pdfExport, msB);

        // Then merge byte[] from msA and msB using a third-party PDF library
    }
}

IronPDF After Code (Native Merge)

using IronPdf;

class MergeSample
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var docA = PdfDocument.FromFile("doc_a.pdf");
        using var docB = PdfDocument.FromFile("doc_b.pdf");

        using var merged = PdfDocument.Merge(docA, docB);
        merged.SaveAs("merged.pdf");
        // Docs: https://ironpdf.com/how-to/merge-or-split-pdfs/
    }
}

What Are the Critical Migration Notes I Should Know?

FastReport uses a designer-template model: create a .frx file, register data sources, call Prepare(), export. IronPDF uses an HTML-in-PDF-out model. These are not interchangeable. The migration only makes sense for the HTML-to-PDF portions of your application.

Memory management matters. FastReport's Report object should be disposed after export. IronPDF's PdfDocument is IDisposable — use using. In server scenarios, ChromePdfRenderer should be a singleton.

Do not delete .frx templates until you have confirmed that all reports generated from them are either migrated or retired. These templates can be hard to reconstruct.

IronPDF Singleton Pattern for Server Use

// IronPDF singleton pattern for server use
public class PdfRenderService
{
    private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

    public byte[] RenderDocument(string html)
    {
        // Async alternative: await _renderer.RenderHtmlAsPdfAsync(html)
        using var pdf = _renderer.RenderHtmlAsPdf(html);
        return pdf.BinaryData;
    }
}

Parallel Batch Generation

// Parallel render using Task.WhenAll
var tasks = htmlDocuments.Select(html => Task.Run(() =>
{
    using var pdf = _renderer.RenderHtmlAsPdf(html);
    return pdf.BinaryData;
}));

var results = await Task.WhenAll(tasks);

FastReport's performance characteristics are optimized for banded report generation — large datasets, cross-tab aggregations, grouping. IronPDF's performance characteristics are optimized for HTML rendering throughput. They're not directly comparable; profile the specific workload.

Frequently Asked Questions

Should I migrate everything off FastReport to IronPDF?

No. If your FastReport .frx templates are working and your stakeholders don't want to move to HTML-based templates, keep FastReport for those paths. The migration makes sense for the HTML-to-PDF paths — the ad-hoc workarounds, the separate libraries, the Razor-to-PDF chains that break on CSS. Consolidate those onto IronPDF's HTML rendering, and leave the banded reports on FastReport.

What if I need both FastReport and IronPDF in the same project?

That's the recommended approach for many teams. Use FastReport for data-bound banded reports where the .frx designer model is a strength, and use IronPDF for HTML-template PDFs where Chromium rendering gives you full CSS and JavaScript support. There are no namespace conflicts between FastReport and IronPdf.

How do I handle dynamic charts that FastReport generates natively?

FastReport has built-in chart objects. With IronPDF, you render JavaScript-based charts (Chart.js, D3, Highcharts) inside your HTML templates, and the Chromium engine executes the JavaScript before generating the PDF. This gives you more flexibility in chart styling but requires you to build the chart in HTML/JS rather than through a report designer. See the rendering options documentation for controlling JavaScript execution timing.

Is IronPDF suitable for long-running reports with 100+ pages?

FastReport is specifically optimized for large banded reports with complex data grouping. For IronPDF, verify performance with your specific document size and complexity. Use the singleton ChromePdfRenderer pattern, and for batch operations, leverage parallel rendering with Task.WhenAll. Profile both approaches against your actual workload before committing to a migration of large-document paths.

How Do I Decide What to Migrate?

The FastReport migration story is really a consolidation story: stop solving HTML-to-PDF with workarounds and use a library designed for it. If FastReport's banded reports are working, leave them alone. The migration target is the code that was trying to shoehorn HTML into a report engine, or the parallel PDF library you've been maintaining alongside FastReport.

The pattern worth asking yourself: how many different code paths does your application currently use to produce a PDF? If the answer is more than two, that's a sign that consolidation has value independent of any specific library choice.

For teams running FastReport alongside an HTML renderer — consider the case where a "report" needs to include a section that's dynamically generated HTML (like a chart from a JS library). With IronPDF, you render that HTML section as its own PDF and merge it with the FastReport output, giving you the best of both tools without trying to force one to do the other's job.
