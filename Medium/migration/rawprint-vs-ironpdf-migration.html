Migrating from RawPrint to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Would a Team Migrate from RawPrint .NET to IronPDF?

I see this pattern regularly. A team started with RawPrint .NET because they needed to send bytes to a physical printer. Win32 APIs, DOCINFO structs, StartDocPrinter, WritePrinter — the whole P/Invoke stack. It solved a real problem. But somewhere along the way, the requirement shifted from "print this document" to "generate a PDF file and archive/email/display it."

That shift is what makes this migration worth considering. RawPrint .NET wraps Win32 printer APIs. IronPDF generates and manipulates PDF documents. They solve different problems, and if your problem has changed, so should your tooling.

Here are the reasons teams typically make this move:

Windows-only dependency. RawPrint .NET uses Win32 printer APIs. Cross-platform or Linux deployment is impossible.

Install footprint. Win32 printer drivers, spooler configuration, and printer-name management add operational complexity.

Shifted requirements. What started as "print this document" has become "generate a PDF file and archive/email/display it."

No HTML input. Raw printing sends pre-formed bytes (ZPL, PCL, PostScript). HTML-to-PDF requires a completely different pipeline.

No PDF manipulation. Merge, watermark, security, and page extraction aren't part of a raw printing workflow.

Printer availability dependency. Raw print requires a named printer to be configured. Development and test environments must have matching printer setup.

Silent failures. Win32 printer API errors are difficult to diagnose. Errors often surface as empty output or spooler failures.

Docker incompatibility. Win32 printer APIs aren't available in Linux containers.

No PDF preview. Raw print jobs go directly to the printer. There's no intermediate PDF artifact for review or archival.

Driver compatibility. PCL/PostScript output may behave differently across printer models. A PDF is model-independent.

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Document</h1></body></html>"
);

// Post-render watermark — applied consistently on all pages
// https://ironpdf.com/how-to/custom-watermark/
var watermark = new TextStamper
{
    Text = "COPY",
    FontColor = IronPdf.Imaging.Color.Red,
    Opacity = 0.15,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("copy-watermarked.pdf");
Console.WriteLine("Watermark applied");

What Questions Come Up Most During This Migration?

Q: Can IronPDF replace RawPrint .NET for sending bytes directly to a physical printer?

Answer:

No. These libraries solve different problems. RawPrint .NET sends pre-formed bytes (ZPL, PCL, PostScript, or PDF) directly to a named Windows printer via Win32 APIs. IronPDF generates and manipulates PDF documents.

If you still need raw printing to physical devices after migration, handle that via System.Drawing.Printing.PrintDocument or a similar Windows print API — separately from PDF generation.

Q: What happens to my Win32 P/Invoke declarations after migrating?

Answer:

They can be removed entirely. Raw printing code often includes direct Win32 P/Invoke declarations for winspool.drv, StartDocPrinter, WritePrinter, and related functions. After migration, these are dead code. Search for remaining references before deleting:

# Find P/Invoke declarations that can be removed
rg "DllImport.*winspool|AddPrinter|StartDocPrinter|WritePrinter|EndDocPrinter|ClosePrinter" --type cs -n

# Verify each has no remaining call sites before deleting

Q: Will my PDFs look the same as the original print output?

Answer:

Not automatically. The original print output may have been generated by a separate system (ERP, report server) and sent as PCL or PostScript. IronPDF generates PDFs from HTML — the visual output depends on your HTML template, not on reverse-engineering the original printer data format. This is a template design task, not a bug. Start by screenshotting the original print output and recreating the layout in HTML/CSS.

Q: Does IronPDF work on Linux and Docker, unlike RawPrint?

Answer:

Yes. IronPDF runs on Linux and macOS. Moving PDF generation to IronPDF enables Linux/Docker deployment that was impossible with RawPrint's Windows-only Win32 API dependency. For Docker, use a supported 64-bit base image (not Alpine) and install the required system libraries for Chromium-based rendering:

# Use a supported base image — verify IronPDF Linux requirements in their docs
FROM mcr.microsoft.com/dotnet/aspnet:8.0

# Common Linux deps for Chromium-based rendering:
RUN apt-get update && apt-get install -y \
    libnss3 \
    libatk-bridge2.0-0 \
    libdrm2 \
    libxkbcommon0 \
    libgbm1 \
    # Verify complete list in IronPDF docs
    && rm -rf /var/lib/apt/lists/*

COPY --from=build /app/out .
ENTRYPOINT ["dotnet", "YourApp.dll"]

How Should I Decide Whether to Migrate?

This decision comes down to one question: has your requirement shifted from "send bytes to a printer" to "generate a PDF file"?

If it has, the migration makes sense. You're replacing a multi-step pipeline — generate document elsewhere, configure printer, send bytes — with a single-step pipeline: render HTML to PDF. The footprint reduction is real. You remove Win32 P/Invoke declarations, printer-name configuration, and Windows-only API dependencies. What replaces it is HTML templates and a renderer call.

If you still need raw printing to physical devices, that requirement should be handled separately from PDF generation. IronPDF and a printer API can coexist — IronPDF generates the PDF, and a separate print API sends it to the device.

The interesting benchmark in this migration is not render time. It's the comparison between the complexity of the old end-to-end pipeline versus the new single-step pipeline. For teams that have already made this shift in requirements, the IronPDF setup takes minutes and the additional capabilities — merge, watermark, password protection — come included in one library instead of requiring multiple dependencies.
