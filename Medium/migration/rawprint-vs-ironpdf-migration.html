Migrating from RawPrint to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Would a Team Migrate from RawPrint .NET to IronPDF?

I see this pattern regularly. A team started with RawPrint .NET because they needed to send bytes to a physical printer. Win32 APIs, DOCINFO structs, StartDocPrinter, WritePrinter — the whole P/Invoke stack. It solved a real problem. But somewhere along the way, the requirement shifted from "print this document" to "generate a PDF file and archive/email/display it."

That shift is what makes this migration worth considering. RawPrint .NET wraps Win32 printer APIs. IronPDF generates and manipulates PDF documents. They solve different problems, and if your problem has changed, so should your tooling.

Here are the reasons teams typically make this move:

Windows-only dependency. RawPrint .NET uses Win32 printer APIs. Cross-platform or Linux deployment is impossible.

Install footprint. Win32 printer drivers, spooler configuration, and printer-name management add operational complexity.

Shifted requirements. What started as "print this document" has become "generate a PDF file and archive/email/display it."

No HTML input. Raw printing sends pre-formed bytes (ZPL, PCL, PostScript). HTML-to-PDF requires a completely different pipeline.

No PDF manipulation. Merge, watermark, security, and page extraction aren't part of a raw printing workflow.

Printer availability dependency. Raw print requires a named printer to be configured. Development and test environments must have matching printer setup.

Silent failures. Win32 printer API errors are difficult to diagnose. Errors often surface as empty output or spooler failures.

Docker incompatibility. Win32 printer APIs aren't available in Linux containers.

No PDF preview. Raw print jobs go directly to the printer. There's no intermediate PDF artifact for review or archival.

Driver compatibility. PCL/PostScript output may behave differently across printer models. A PDF is model-independent.

How Does the Migration Effort Break Down?

This isn't a line-by-line library swap. It's a workflow change. The effort depends on which features you need in the new pipeline.

  • Send bytes to printer — N/A. RawPrint uses RawPrinterHelper.SendBytesToPrinter(); IronPDF is not a raw printer API and does not replace this directly.
  • Generate PDF from HTML — Low. ChromePdfRenderer.RenderHtmlAsPdf() replaces the entire "shell out to a PDF tool then send bytes" pipeline.
  • Generate PDF from URL — Low. ChromePdfRenderer.RenderUrlAsPdf() renders a live page with CSS and JS, not the raw HTML source.
  • Print a generated PDF — Medium. IronPDF exposes pdf.Print() for high-level printing, including printer name and copy count.
  • Merge PDFs — Medium. PdfDocument.Merge() combines multiple documents in one call.
  • Watermark — Medium. TextStamper and ImageStamper apply post-render stamps.
  • Password protection — Medium. pdf.SecuritySettings sets user and owner passwords.
  • Cross-platform support — Low (inherit). RawPrint is Windows-only; IronPDF runs on Linux and Docker.
  • Save PDF to file/stream — Low. pdf.SaveAs() and pdf.Stream cover both cases.

Most of the "low complexity" items are things RawPrint .NET simply didn't do. You're not replacing functionality — you're adding it.

What Does the Decision Criteria Look Like in Practice?

The key question is whether your requirement is still "send bytes to a printer" or whether it has become "produce a PDF file." Here's how that breaks down by scenario.

  • If you still need raw print jobs sent to physical printers, RawPrint .NET serves that case directly; IronPDF does not replace a raw printer API.
  • If you need PDF files for email, archive, or web display, switch — IronPDF generates PDF files; RawPrint .NET does not.
  • If you need to print PDF files to a printer, IronPDF can do both — it generates the PDF and exposes pdf.Print() for the print step.
  • If you are moving from Windows to Linux/Docker deployment, switch — RawPrint .NET can't run on Linux.

Basic PDF Generation — Before (RawPrint .NET)

using System;
using System.IO;

class Program
{
    static void Main()
    {
        // RawPrint sends pre-formed document bytes to a named Windows printer
        // The bytes must already be in a format the printer understands (ZPL, PCL, PDF, etc.)

        var pdfBytes = File.ReadAllBytes("document.pdf"); // pre-generated PDF
        var printerName = "PrinterOnNetwork";

        RawPrinterHelper.SendBytesToPrinter(printerName, pdfBytes);

        Console.WriteLine("Bytes sent to printer — note RawPrint sends to a device, not a file");
    }
}

Basic PDF Generation — After (IronPDF)

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Generate the PDF document
var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Document</h1><p>Content here.</p></body></html>"
);

// Save to file — or use pdf.BinaryData for in-memory/email/API
pdf.SaveAs("document.pdf");

Console.WriteLine($"Generated document.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

// If you still need to print the PDF, IronPDF exposes pdf.Print():

Watermark — After (IronPDF)

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Document</h1></body></html>"
);

// Post-render watermark — applied consistently on all pages
// https://ironpdf.com/how-to/custom-watermark/
var watermark = new TextStamper
{
    Text = "COPY",
    FontColor = IronSoftware.Drawing.Color.Red,
    Opacity = 15,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("copy-watermarked.pdf");
Console.WriteLine("Watermark applied");

How Do the Two Libraries Compare Feature by Feature?

This list gives you the full picture. Remember — these libraries serve fundamentally different purposes, so many points show "Not applicable" on the RawPrint side.

  • Focus. RawPrint .NET sends raw bytes to Windows printers; IronPDF does PDF generation and manipulation.
  • Pricing. RawPrint is a small open-source package on NuGet; IronPDF is a commercial license — see ironsoftware.com.
  • API style. RawPrint is a Win32 wrapper that takes a printer name and byte array; IronPDF works with .NET objects and accepts HTML or existing PDF input.
  • Learning curve. RawPrint is low for raw printing but offers nothing for PDF generation; IronPDF is low for .NET developers.
  • HTML rendering. RawPrint has none; IronPDF uses an embedded Chromium engine.
  • Page indexing. Not applicable to RawPrint; IronPDF is 0-based.
  • Thread safety. Not applicable to the printer API; verify IronPDF concurrent instance guidance for parallel rendering.
  • Namespace. RawPrint versus IronPdf.

Namespace Mapping

  • RawPrint → IronPdf — core namespace
  • System.Runtime.InteropServices (P/Invoke) → N/A — removed; no Win32 P/Invoke needed
  • N/A → IronPdf.Editing — watermark and stamp APIs

Core Class Mapping

  • Printer helper class → N/A — different purpose. IronPDF generates PDFs; it does not directly replace a raw print transport.
  • Byte array (pre-formed document) → ChromePdfRenderer — generates PDF from HTML.
  • Printer name string → N/A for generation. IronPDF outputs to file or stream; pdf.Print() takes a printer name when you need to print.
  • N/A → PdfDocument — the PDF object you save, merge, and manipulate.

Document Loading Methods

  • Generate PDF from HTML — RawPrint: not applicable; IronPDF: renderer.RenderHtmlAsPdf(html).
  • Generate PDF from URL — RawPrint: not applicable; IronPDF: renderer.RenderUrlAsPdf(url).
  • Load existing PDF — RawPrint: N/A; IronPDF: PdfDocument.FromFile(path).
  • Save to stream — RawPrint: N/A; IronPDF: pdf.Stream.

Page Operations

  • Page count — RawPrint: N/A; IronPDF: pdf.PageCount.
  • Remove page — RawPrint: N/A; IronPDF: pdf.RemovePages(index).
  • Extract text — RawPrint: N/A; IronPDF: pdf.ExtractAllText().
  • Rotate — RawPrint: N/A; IronPDF: pdf.RotateAllPages() and per-page rotation are available — verify the exact overload for your version.

Merge / Split Operations

  • Merge — RawPrint: not applicable; IronPDF: PdfDocument.Merge(doc1, doc2).
  • Split — RawPrint: not applicable; IronPDF: see the merge and split guide.

What Questions Come Up Most During This Migration?

Q: Can IronPDF replace RawPrint .NET for sending bytes directly to a physical printer?

Partly. RawPrint .NET sends pre-formed bytes (ZPL, PCL, PostScript, or PDF) directly to a named Windows printer via Win32 APIs. IronPDF doesn't speak ZPL or PCL, but it does expose pdf.Print() for sending a generated PDF to a Windows printer through the standard print subsystem.

If you specifically need to push pre-formed PCL or ZPL byte streams to a printer, keep RawPrint or use System.Drawing.Printing.PrintDocument for that step — separately from PDF generation.

Q: What happens to my Win32 P/Invoke declarations after migrating?

They can be removed entirely. Raw printing code often includes direct Win32 P/Invoke declarations for winspool.drv, StartDocPrinter, WritePrinter, and related functions. After migration, these are dead code. Search for remaining references before deleting:

# Find P/Invoke declarations that can be removed
rg "DllImport.*winspool|AddPrinter|StartDocPrinter|WritePrinter|EndDocPrinter|ClosePrinter" --type cs -n

# Verify each has no remaining call sites before deleting

Q: Will my PDFs look the same as the original print output?

Not automatically. The original print output may have been generated by a separate system (ERP, report server) and sent as PCL or PostScript. IronPDF generates PDFs from HTML — the visual output depends on your HTML template, not on reverse-engineering the original printer data format. This is a template design task, not a bug. Start by screenshotting the original print output and recreating the layout in HTML/CSS.

Q: Does IronPDF work on Linux and Docker, unlike RawPrint?

Yes. IronPDF runs on Linux and macOS. Moving PDF generation to IronPDF enables Linux/Docker deployment that was impossible with RawPrint's Windows-only Win32 API dependency. For Docker, use a supported 64-bit base image (not Alpine) and install the required system libraries for Chromium-based rendering:

# Use a supported base image — verify IronPDF Linux requirements in their docs
FROM mcr.microsoft.com/dotnet/aspnet:8.0

# Common Linux deps for Chromium-based rendering:
RUN apt-get update && apt-get install -y \
    libnss3 \
    libatk-bridge2.0-0 \
    libdrm2 \
    libxkbcommon0 \
    libgbm1 \
    # Verify complete list in IronPDF docs
    && rm -rf /var/lib/apt/lists/*

COPY --from=build /app/out .
ENTRYPOINT ["dotnet", "YourApp.dll"]

How Should I Decide Whether to Migrate?

This decision comes down to one question: has your requirement shifted from "send bytes to a printer" to "generate a PDF file"?

If it has, the migration makes sense. You're replacing a multi-step pipeline — generate document elsewhere, configure printer, send bytes — with a single-step pipeline: render HTML to PDF. The footprint reduction is real. You remove Win32 P/Invoke declarations, printer-name configuration, and Windows-only API dependencies. What replaces it is HTML templates and a renderer call.

If you still need raw printing to physical devices, that requirement should be handled separately from PDF generation. IronPDF and a printer API can coexist — IronPDF generates the PDF, and a separate print API sends it to the device.

The interesting benchmark in this migration is not render time. It's the comparison between the complexity of the old end-to-end pipeline versus the new single-step pipeline. For teams that have already made this shift in requirements, the IronPDF setup takes minutes and the additional capabilities — merge, watermark, password protection — come included in one library instead of requiring multiple dependencies.
