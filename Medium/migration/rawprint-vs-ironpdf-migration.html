Migrating from RawPrint to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Would a Team Move from RawPrint to IronPDF?

I see this pattern regularly. A team started with RawPrint because they needed to push pre-formed bytes to a physical printer — ESC/POS to a thermal receipt printer, ZPL to a Zebra label printer, or PCL/PostScript bytes to an enterprise MFP. RawPrint is a thin P/Invoke wrapper over winspool.Drv that ships a byte stream straight to the Windows print spooler with the RAW datatype. It solved a real problem and still does.

The honest framing of this migration is "complement," not "replace." RawPrint and IronPDF solve different problems. RawPrint sends raw bytes; IronPDF generates and manipulates PDF documents. If your application's requirement has shifted from "push these bytes to a printer" to "generate a PDF file and archive/email/display it," the migration makes sense. If your raw-byte channel is genuinely raw — ESC/POS to a thermal printer, ZPL to a Zebra — keep RawPrint for that channel.

One thing to note before reading further: the RawPrint NuGet package (frogmorecs/RawPrint) has been unlisted as legacy on nuget.org. The last published version is 0.5.0, released September 2019. It still works on .NET Framework 4.0+, and a community fork called RawPrint.NetStd exists separately. That status alone is enough reason to think about your forward path.

Here are the reasons teams typically rebuild the PDF-producing half of the pipeline on IronPDF:

Windows-only dependency. RawPrint targets the Windows print spooler. Cross-platform or Linux deployment is impossible for that pipeline.

Install footprint. Spooler configuration and printer-name management add operational complexity that PDF-only workflows do not need.

Shifted requirements. What started as "print this document" has become "generate a PDF file and archive/email/display it."

No HTML input. Raw printing sends pre-formed bytes. HTML-to-PDF requires a completely different pipeline.

No PDF manipulation. Merge, watermark, security, and page extraction aren't part of a raw printing workflow.

Printer availability dependency. Raw print requires a named printer to be configured. Development and test environments must have matching printer setup.

Silent failures. Win32 spooler errors are difficult to diagnose. Errors often surface as empty output or spooler failures.

Docker incompatibility. The Windows spooler isn't available in Linux containers.

No PDF preview. Raw print jobs go directly to the printer. There's no intermediate PDF artifact for review or archival.

Driver compatibility. PCL/PostScript output may behave differently across printer models. A PDF is model-independent.

How Does the Migration Effort Break Down?

This isn't a line-by-line library swap. It's a workflow change. The effort depends on which features you need in the new pipeline.

  • Push bytes to printer — N/A. RawPrint uses Printer.PrintRawFile() and Printer.PrintRawStream() on the IPrinter interface; IronPDF is not a RAW byte channel and does not replace this. ESC/POS, ZPL, and hand-built PCL stay on RawPrint.
  • Generate PDF from HTML — Low. ChromePdfRenderer.RenderHtmlAsPdf() replaces the "produce a PDF with another library, then push the bytes" pipeline.
  • Generate PDF from URL — Low. ChromePdfRenderer.RenderUrlAsPdf() renders a live page with CSS and JS, not the raw HTML source.
  • Print a generated PDF — Medium. IronPDF exposes pdf.Print(), which hands the document to the operating system print system rather than streaming RAW bytes to the spooler.
  • Merge PDFs — Medium. PdfDocument.Merge() combines multiple documents in one call.
  • Watermark — Medium. TextStamper and ImageStamper apply post-render stamps.
  • Password protection — Medium. pdf.SecuritySettings sets user and owner passwords.
  • Cross-platform support — Low (inherit). RawPrint is Windows-only; IronPDF runs on Linux and Docker.
  • Save PDF to file/stream — Low. pdf.SaveAs() and pdf.Stream cover both cases.

Most of the "low complexity" items are things RawPrint simply didn't do. On the PDF-generation side, you're not replacing functionality — you're adding it.

What Does the Decision Criteria Look Like in Practice?

The key question is whether your requirement is still "push bytes to a printer" or whether it has become "produce a PDF file." Here's how that breaks down by scenario.

  • If you need ESC/POS, ZPL/EPL, or hand-built PCL bytes sent to physical printers, RawPrint serves that case directly; IronPDF is not a RAW byte channel and cannot stand in for it.
  • If you need PDF files for email, archive, or web display, switch — IronPDF generates PDF files; RawPrint does not.
  • If you need to render and then print PDF files on Windows, IronPDF can do both — it generates the PDF and exposes pdf.Print() for the print step.
  • If you are moving from Windows to Linux/Docker deployment, switch — RawPrint targets the Windows spooler and can't run on Linux.

Where RawPrint Stays — ESC/POS and ZPL

This is the part of the workflow IronPDF cannot and should not replace. If you have an ESC/POS thermal receipt printer or a Zebra label printer, RawPrint is the right tool and the code stays.

using RawPrint;
using System.IO;
using System.Text;

// ESC/POS to a thermal receipt printer:
// ESC @ resets the printer, then text, then a partial cut command.
byte[] receipt = Encoding.ASCII.GetBytes("\x1B@Hello\nWorld\n\n\n\x1DV\x01");

IPrinter printer = new Printer();
using (var stream = new MemoryStream(receipt))
{
    printer.PrintRawStream("EPSON TM-T20II", stream, "Receipt", false);
}

// ZPL to a Zebra label printer:
string zpl = "^XA^FO50,50^ADN,36,20^FDHello Zebra^FS^XZ";
byte[] zplBytes = Encoding.ASCII.GetBytes(zpl);

using (var stream = new MemoryStream(zplBytes))
{
    printer.PrintRawStream("ZDesigner GK420d", stream, "Label", false);
}

IronPDF doesn't speak ESC/POS or ZPL — those are printer-side languages, not document formats. If your application has both pipelines, the right architecture is RawPrint for the printer-language side and IronPDF for PDF generation. They run side by side.

How Do the Two Libraries Compare Feature by Feature?

This list gives you the full picture. These libraries serve fundamentally different purposes, so many points show "Not applicable" on the RawPrint side.

  • Focus. RawPrint pushes raw bytes to Windows printers via the spooler; IronPDF does PDF generation and manipulation.
  • Pricing. RawPrint is a small NuGet package, now unlisted as legacy; IronPDF is a commercial license — see ironsoftware.com.
  • API style. RawPrint exposes a single Printer class implementing IPrinter, taking a printer name and a file path or stream; IronPDF works with .NET objects and accepts HTML or existing PDF input.
  • Learning curve. RawPrint is low for raw printing but offers nothing for PDF generation; IronPDF is low for .NET developers.
  • HTML rendering. RawPrint has none; IronPDF uses an embedded Chromium engine.
  • Printer languages. RawPrint handles ESC/POS, ZPL/EPL, PCL, and PostScript byte streams; IronPDF does not.
  • Page indexing. Not applicable to RawPrint; IronPDF is 0-based.
  • Thread safety. Not applicable to the printer API; verify IronPDF concurrent instance guidance for parallel rendering.
  • Namespace. RawPrint versus IronPdf.

Namespace Mapping

  • RawPrint → IronPdf — core namespace on the PDF side; the RawPrint namespace stays for ESC/POS or ZPL channels
  • System.Runtime.InteropServices (hand-rolled P/Invoke) → N/A — not needed by IronPDF; RawPrint hides its own P/Invoke internally
  • N/A → IronPdf.Editing — watermark and stamp APIs

Core Class Mapping

  • Printer / IPrinter → N/A — different purpose. IronPDF generates PDFs; it does not replace a RAW byte transport.
  • PrintRawFile(printer, path, paused) → PdfDocument.FromFile(path).Print() — IronPDF re-renders to the OS print system rather than streaming RAW bytes.
  • PrintRawStream(printer, stream, doc, paused) → new PdfDocument(stream).Print() — same caveat.
  • OnJobCreated event → not applicable; use IronPDF print options instead.
  • N/A → ChromePdfRenderer — generates PDF from HTML or URL.
  • N/A → PdfDocument — the PDF object you save, merge, and manipulate.

Document Loading Methods

  • Generate PDF from HTML — RawPrint: not applicable; IronPDF: renderer.RenderHtmlAsPdf(html).
  • Generate PDF from URL — RawPrint: not applicable; IronPDF: renderer.RenderUrlAsPdf(url).
  • Load existing PDF — RawPrint: N/A; IronPDF: PdfDocument.FromFile(path).
  • Save to stream — RawPrint: N/A; IronPDF: pdf.Stream.

Page Operations

  • Page count — RawPrint: N/A; IronPDF: pdf.PageCount.
  • Remove page — RawPrint: N/A; IronPDF: pdf.RemovePages(index).
  • Extract text — RawPrint: N/A; IronPDF: pdf.ExtractAllText().
  • Rotate — RawPrint: N/A; IronPDF: pdf.RotateAllPages() and per-page rotation are available — verify the exact overload for your version.

Merge / Split Operations

  • Merge — RawPrint: not applicable; IronPDF: PdfDocument.Merge(doc1, doc2).
  • Split — RawPrint: not applicable; IronPDF: see the merge and split guide.

What Questions Come Up Most During This Migration?

Q: Can IronPDF replace RawPrint for pushing bytes directly to a physical printer?

Partly. RawPrint sends pre-formed bytes (ESC/POS, ZPL, PCL, PostScript, or PDF) directly to a named Windows printer via the spooler with the RAW datatype. IronPDF doesn't speak ESC/POS, ZPL, or PCL, but it does expose pdf.Print() for handing a generated PDF to the operating system print system.

If you specifically need to push pre-formed PCL, ZPL, or ESC/POS byte streams to a printer, keep RawPrint for that channel — alongside IronPDF for the PDF-generation side, not as a replacement for it.

Q: What happens to hand-rolled Win32 P/Invoke wrappers after migrating?

If your codebase has its own P/Invoke helpers (sometimes named things like RawPrinterHelper.SendBytesToPrinter) calling into winspool.Drv, those are not part of the RawPrint package — they're hand-rolled wrappers built on the same Win32 surface RawPrint hides internally (OpenPrinter, StartDocPrinter, WritePrinter, EndDocPrinter, ClosePrinter). For the PDF half of the pipeline, those helpers can be removed; for any remaining ESC/POS or ZPL channel, replace them with the real RawPrint API (Printer.PrintRawFile / Printer.PrintRawStream) rather than maintaining your own P/Invoke. Search for remaining references before deleting:

# Find P/Invoke declarations that can be removed or consolidated
rg "DllImport.*winspool|StartDocPrinter|WritePrinter|EndDocPrinter|ClosePrinter|OpenPrinter" --type cs -n

# Confirm no call sites remain before deleting

Q: Will my PDFs look the same as the original print output?

Not automatically. The original print output may have been generated by a separate system (ERP, report server) and sent as PCL or PostScript. IronPDF generates PDFs from HTML — the visual output depends on your HTML template, not on reverse-engineering the original printer data format. This is a template design task, not a bug. Start by screenshotting the original print output and recreating the layout in HTML/CSS.

Q: Does IronPDF work on Linux and Docker, unlike RawPrint?

Yes. IronPDF runs on Linux and macOS. Moving PDF generation to IronPDF enables Linux/Docker deployment that was impossible with RawPrint's Windows-spooler dependency. For Docker, use a supported 64-bit base image (not Alpine) and install the required system libraries for Chromium-based rendering:

# Use a supported base image — verify IronPDF Linux requirements in their docs
FROM mcr.microsoft.com/dotnet/aspnet:8.0

# Common Linux deps for Chromium-based rendering:
RUN apt-get update && apt-get install -y \
    libnss3 \
    libatk-bridge2.0-0 \
    libdrm2 \
    libxkbcommon0 \
    libgbm1 \
    # Confirm complete list against IronPDF Linux docs
    && rm -rf /var/lib/apt/lists/*

COPY --from=build /app/out .
ENTRYPOINT ["dotnet", "YourApp.dll"]

How Should I Decide Whether to Migrate?

This decision comes down to one question: has the PDF-producing half of your pipeline shifted from "push pre-formed bytes" to "generate a PDF file"?

If it has, the migration makes sense. You're replacing a multi-step pipeline — generate document elsewhere, configure printer, push bytes — with a single-step pipeline: render HTML to PDF. The footprint reduction is real. You remove ad-hoc P/Invoke wrappers, printer-name configuration on the document side, and Windows-only spooler dependencies. What replaces it is HTML templates and a renderer call.

If you still need raw printing to physical devices — ESC/POS, ZPL, hand-built PCL — that requirement stays on RawPrint. IronPDF and RawPrint coexist cleanly: IronPDF generates the PDFs your application archives, emails, or displays, and RawPrint pushes printer-language bytes to the devices that need them.

The interesting comparison in this migration is not render time. It's the gap between the complexity of the old end-to-end pipeline (external PDF library, temp file, spooler call) versus the new single-step pipeline. For teams that have already made this shift in requirements, the IronPDF setup takes minutes and the additional capabilities — merge, watermark, password protection — come included in one library instead of requiring multiple dependencies.
