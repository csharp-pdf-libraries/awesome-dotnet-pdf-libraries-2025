Migrating from Telerik Document Processing to IronPDF: Less Config, Same Output

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Consider Migrating from Telerik Document Processing?

The conversation usually starts with a dependency audit. Someone runs dotnet list package --include-transitive and the Telerik Document Processing transitive dependency tree scrolls for several screens. There are multiple Telerik.Documents.* packages — some for PDF (Telerik.Documents.Fixed), some for Word (Telerik.Documents.Flow), some for spreadsheets (Telerik.Documents.Spreadsheet) — and the application only uses PDF generation. The install footprint is wider than the actual use case.

Teams evaluating Telerik Document Processing alternatives commonly cite these triggers. First, the package footprint: TDP involves multiple NuGet packages, and teams using only PDF generation often pull in Telerik.Documents.Flow as well because that is where HtmlFormatProvider lives. Second, the Telerik license dependency: TDP is part of the Telerik ecosystem, and access typically requires a Telerik subscription that bundles PDF with UI components you may not need.

Third, the HTML path goes through an intermediate model. TDP's HTML-to-PDF route is HtmlFormatProvider.Import(html) → RadFlowDocument → PdfFormatProvider.Export(). The flow model is built for Word-style content, so modern CSS layout (Flexbox, Grid, Bootstrap columns) is flattened in the conversion rather than rendered like a browser would. Fourth, the fixed-document path (RadFixedDocument + FixedContentEditor) is verbose for layout-heavy documents compared to HTML/CSS templates.

Fifth, upgrade coupling. TDP version is often coupled to Telerik UI versions, so upgrading one may require upgrading others. Sixth, if your report content starts as HTML, the flow-document round trip adds intermediate steps that the CSS layout gap creates.

Seventh, Docker and cross-platform deployment: verify which TDP operations are supported cross-platform, as some operations may have platform-specific behavior. Eighth, different Telerik.Documents packages for PDF vs Word vs Excel mean version management across several package IDs. Ninth, some features like text extraction, watermark, and security may require specific TDP packages or configuration. And tenth, TDP documentation is thorough, but community Q&A is concentrated in the Telerik forums rather than Stack Overflow.

What Does the Migration Effort Actually Look Like?

I want to give you a realistic picture of the effort involved. Here is a feature-by-feature breakdown so you can estimate the work before writing any code.

Effort by Feature

  • HTML string to PDF — Low. TDP routes through HtmlFormatProvider.Import() + PdfFormatProvider.Export() via RadFlowDocument. IronPDF uses ChromePdfRenderer.RenderHtmlAsPdfAsync() directly — the intermediate flow document is gone.
  • URL to PDF — Low. TDP requires you to fetch the HTML yourself with HttpClient and feed it to HtmlFormatProvider. IronPDF calls renderer.RenderUrlAsPdfAsync(url).
  • Save to file — Low. PdfFormatProvider.Export(doc, stream) becomes pdf.SaveAs(path).
  • Save to stream — Low. Same TDP export call becomes pdf.Stream or pdf.BinaryData.
  • Document creation — High (rewrite). Code that hand-builds a RadFixedDocument with FixedContentEditor coordinates needs to be reimagined as an HTML/CSS template. This is the bulk of the migration cost.
  • Merge PDFs — Low to Medium. TDP merges by importing both PDFs to RadFixedDocument and copying pages into a third document. IronPDF: PdfDocument.Merge(doc1, doc2).
  • Watermark — Medium. TDP draws watermarks via FixedContentEditor and RgbColor. IronPDF uses TextStamper / ImageStamper or pdf.ApplyWatermark(html).
  • Password protection — Low. PdfExportSettings.UserPassword / OwnerPassword become pdf.SecuritySettings.UserPassword / OwnerPassword.
  • Text extraction — Medium. TDP iterates page.Content looking for TextFragment elements. IronPDF: pdf.ExtractAllText().
  • Digital signatures — Medium-High. Verify signature scope on both sides; PAdES profile support and timestamping are the gaps to confirm before committing.

How Do You Decide Whether to Migrate?

Not every team should migrate. Here is a decision matrix to help you figure out where you fall.

  • If PDF generation from HTML templates is your primary need, switch. IronPDF is HTML-first via Chromium, while TDP's flow-document conversion flattens modern CSS layout.
  • If TDP is doing PDF, Word, and Excel in the same project, evaluate carefully. TDP covers more document formats; IronPDF is PDF-only and does not natively convert Word or Excel files. Migrate the PDF surface and keep the other Telerik.Documents packages.
  • If a Telerik subscription is already bundled with your UI license, weigh migration cost against the ongoing dependency footprint. The savings here are operational, not financial.
  • If you deploy to Docker or cloud where package footprint matters, switch. IronPDF has a smaller, focused dependency set than the multi-package Telerik.Documents suite.

How Do You Set Up IronPDF and Replace Telerik Document Processing?

I will walk you through the core migration in three steps. First, find your existing Telerik references, then swap the packages, and finally update your code.

Finding Telerik Document Processing References in Your Codebase

# Find TDP PDF API usage (covers Flow + Fixed paths)
rg -l "Telerik\.Windows\.Documents\.(Flow|Fixed)|HtmlFormatProvider|PdfFormatProvider|RadFlowDocument|RadFixedDocument|FixedContentEditor" --type cs
rg "RadFlowDocument|RadFixedDocument|HtmlFormatProvider|PdfFormatProvider|FixedContentEditor" --type cs -n

# Count document model usage (higher migration cost)
rg "RadFlow|RadFixed|FixedContent|ContentEditor" --type cs | wc -l

# Find TDP packages in project files
grep -r "Telerik\.Documents" *.csproj **/*.csproj 2>/dev/null

# List all Telerik packages in use
dotnet list package | grep -i telerik

Removing Telerik and Installing IronPDF

# Remove Telerik Document Processing packages
# Note: removing these may affect other Telerik packages -- check dependencies first
dotnet remove package Telerik.Documents.Core
dotnet remove package Telerik.Documents.Fixed
dotnet remove package Telerik.Documents.Flow
dotnet remove package Telerik.Documents.Flow.FormatProviders.Pdf
# Keep Telerik.Documents.Spreadsheet if still used for Excel

# Install IronPDF
dotnet add package IronPdf

dotnet restore

Step 3: Basic PDF Generation

Before (Telerik Flow document — HTML import path):

using Telerik.Windows.Documents.Flow.FormatProviders.Html;
using Telerik.Windows.Documents.Flow.FormatProviders.Pdf;
using Telerik.Windows.Documents.Flow.Model;
using System.IO;

string html = "<html><body><h1>Hello World</h1><p>This is a PDF document.</p></body></html>";

HtmlFormatProvider htmlProvider = new HtmlFormatProvider();

PdfFormatProvider pdfProvider = new PdfFormatProvider();
using (FileStream output = File.OpenWrite("output.pdf"))
{
    pdfProvider.Export(document, output);
}

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Direct Chromium rendering -- no flow document round-trip
string html = "<html><body><h1>Hello World</h1><p>This is a PDF document.</p></body></html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("output.pdf");

Console.WriteLine($"Saved output.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

What Does Each Before/After Migration Look Like in Practice?

Let me walk through four complete migration examples so you can see exactly what changes in real code.

1. HTML to PDF

Before (Telerik Document Processing — Flow document HTML import):

using Telerik.Windows.Documents.Flow.FormatProviders.Html;
using Telerik.Windows.Documents.Flow.FormatProviders.Pdf;
using Telerik.Windows.Documents.Flow.Model;
using System.IO;

string html = @"
    <html><body>
    <h1>Invoice #2024-0099</h1>
    <p>Customer: Acme Corp</p>
    <p><strong>Total: $4,200.00</strong></p>
    </body></html>";

HtmlFormatProvider htmlProvider = new HtmlFormatProvider();
RadFlowDocument document = htmlProvider.Import(html);

// Modern CSS (Flexbox, Grid, Bootstrap columns) is flattened during
// the HTML -> RadFlowDocument conversion. Layout-heavy templates lose
// their CSS structure here.

PdfFormatProvider pdfProvider = new PdfFormatProvider();
using (FileStream output = File.OpenWrite("invoice.pdf"))
{
    pdfProvider.Export(document, output);
}

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// HTML template replaces RadFixedDocument model -- no manual position calculation
var html = @"
    <html>
    <head>
    <style>
        body { font-family: Arial, sans-serif; padding: 40px; }
        .title { font-size: 20px; font-weight: bold; }
        .field { margin-top: 8px; font-size: 13px; }
        .total { font-size: 16px; font-weight: bold; margin-top: 20px; }
    </style>
    </head>
    <body>
        <div class='title'>Invoice #2024-0099</div>
        <div class='field'>Customer: Acme Corp</div>
        <div class='total'>Total: $4,200.00</div>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");
Console.WriteLine($"Saved invoice.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

2. Merge PDFs

Before (Telerik Document Processing):

using Telerik.Windows.Documents.Fixed.FormatProviders.Pdf;
using Telerik.Windows.Documents.Fixed.Model;
using System.IO;

PdfFormatProvider provider = new PdfFormatProvider();

RadFixedDocument document1;
using (FileStream input = File.OpenRead("document1.pdf"))
{
    document1 = provider.Import(input);
}

RadFixedDocument document2;
using (FileStream input = File.OpenRead("document2.pdf"))
{
    document2 = provider.Import(input);
}

// Build the merged document by copying pages
RadFixedDocument mergedDocument = new RadFixedDocument();
foreach (var page in document1.Pages)
{
    mergedDocument.Pages.Add(page);
}
foreach (var page in document2.Pages)
{
    mergedDocument.Pages.Add(page);
}

using (FileStream output = File.OpenWrite("merged.pdf"))
{
    provider.Export(mergedDocument, output);
}

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var pdf1 = PdfDocument.FromFile("document1.pdf");
var pdf2 = PdfDocument.FromFile("document2.pdf");

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(pdf1, pdf2);
merged.SaveAs("merged.pdf");
Console.WriteLine($"Merged: {merged.PageCount} pages");

3. Watermark

Before (Telerik Document Processing):

using Telerik.Windows.Documents.Fixed.FormatProviders.Pdf;
using Telerik.Windows.Documents.Fixed.Model;
using Telerik.Windows.Documents.Fixed.Model.ColorSpaces;
using Telerik.Windows.Documents.Fixed.Model.Editing;
using System.IO;

PdfFormatProvider provider = new PdfFormatProvider();
RadFixedDocument document;
using (FileStream s = File.OpenRead("source.pdf"))
{
    document = provider.Import(s);
}

foreach (RadFixedPage page in document.Pages)
{
    FixedContentEditor editor = new FixedContentEditor(page);
    editor.GraphicProperties.IsFilled = true;
    editor.GraphicProperties.FillColor = new RgbColor(200, 255, 0, 0); // alpha + RGB

    // Manual centring + rotation
    editor.Position.Translate(page.Size.Width / 2, page.Size.Height / 2);
    editor.Position.Rotate(45);

    Block block = new Block();
    block.TextProperties.FontSize = 72;
    block.InsertText("DRAFT");
    editor.DrawBlock(block);
}

using (FileStream output = File.OpenWrite("watermarked.pdf"))
{
    provider.Export(document, output);
}

After (IronPDF):

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var pdf = PdfDocument.FromFile("source.pdf");

// https://ironpdf.com/how-to/custom-watermark/
var watermark = new TextStamper
{
    Text = "DRAFT",
    FontFamily = "Arial",
    FontSize = 72,
    FontColor = IronSoftware.Drawing.Color.Gray.ToString(),
    Opacity = 15, // 0-100, integer
    Rotation = 45,
    VerticalAlignment = IronSoftware.Drawing.VerticalAlignment.Middle,
    HorizontalAlignment = IronSoftware.Drawing.HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("watermarked.pdf");
Console.WriteLine("Watermark applied");

4. Password Protection

Before (Telerik Document Processing):

using Telerik.Windows.Documents.Fixed.FormatProviders.Pdf;
using Telerik.Windows.Documents.Fixed.FormatProviders.Pdf.Export;
using Telerik.Windows.Documents.Fixed.Model;
using System.IO;

PdfFormatProvider provider = new PdfFormatProvider();

RadFixedDocument document;
using (FileStream input = File.OpenRead("document.pdf"))
{
    document = provider.Import(input);
}

PdfExportSettings settings = new PdfExportSettings();
settings.UserPassword = "open123";
settings.OwnerPassword = "admin456";
// Permission flags (printing, copy, edit) are configured on settings
// as well -- consult the TDP encryption reference.

provider.ExportSettings = settings;

using (FileStream output = File.OpenWrite("secured.pdf"))
{
    provider.Export(document, output);
}

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Confidential</h1></body></html>"
);

// https://ironpdf.com/how-to/pdf-permissions-passwords/
pdf.SecuritySettings.UserPassword = "open123";
pdf.SecuritySettings.OwnerPassword = "admin456";

pdf.SaveAs("secured.pdf");
Console.WriteLine("Saved secured.pdf");

How Do the Two Libraries Compare Feature by Feature?

Here are the full mapping tables I use when planning a migration. These cover the overall comparison, namespaces, core classes, document loading, page operations, and merge/split.

Overall Comparison

  • Focus. TDP is multi-format document generation (PDF, Word, Excel) via document models. IronPDF is HTML-to-PDF plus PDF manipulation, no native Office conversion.
  • Pricing. TDP ships under a Telerik subscription that bundles UI components; IronPDF is a standalone commercial license. Verify current pricing at telerik.com and ironsoftware.com.
  • API style. TDP uses RadFlowDocument for HTML import and RadFixedDocument + FixedContentEditor for fixed-layout PDF. IronPDF uses ChromePdfRenderer with HTML input and PdfDocument for manipulation.
  • Learning curve. TDP is medium-to-high — two document models with many types. IronPDF is low for web developers because HTML/CSS is the input language.
  • HTML rendering. TDP routes HTML through a flow-document conversion, which flattens modern CSS layout. IronPDF embeds Chromium and renders the HTML directly.
  • Page indexing. Both libraries are 0-based, so page-index code translates directly.
  • Thread safety. Verify in the Telerik docs for the version you target. For IronPDF, follow the concurrent instance guidance — one renderer per task in Task.WhenAll patterns.
  • Namespace. TDP lives under Telerik.Windows.Documents.Flow.* and Telerik.Windows.Documents.Fixed.*. IronPDF lives under IronPdf.

Namespace Mapping

  • Telerik.Windows.Documents.Flow.Model → IronPdf — core namespace; flow document is replaced by direct HTML
  • Telerik.Windows.Documents.Flow.FormatProviders.Html → IronPdf — HtmlFormatProvider becomes ChromePdfRenderer
  • Telerik.Windows.Documents.Flow.FormatProviders.Pdf → IronPdf — export is handled by pdf.SaveAs() / pdf.Stream
  • Telerik.Windows.Documents.Fixed.Model → IronPdf — RadFixedDocument manipulation maps to PdfDocument
  • Telerik.Windows.Documents.Fixed.Model.Editing → N/A for HTML path — FixedContentEditor is replaced by HTML/CSS

Core Class Mapping

  • HtmlFormatProvider → ChromePdfRenderer — direct HTML rendering, no flow conversion
  • RadFlowDocument → not needed — the intermediate document is gone; HTML goes straight to PDF
  • RadFixedDocument → PdfDocument — the in-memory PDF object for manipulation (load, merge, stamp)
  • PdfFormatProvider → pdf.SaveAs() / pdf.Stream / pdf.BinaryData — export is on the pdf object
  • FixedContentEditor → N/A — HTML/CSS handles layout, no editor required
  • Section / Paragraph → HTML elements (<section>, <p>) — structure lives in the markup

Document Loading Methods

  • Create from HTML. TDP: HtmlFormatProvider.Import(html) → RadFlowDocument. IronPDF: renderer.RenderHtmlAsPdfAsync(html) → PdfDocument.
  • Render URL. TDP requires HttpClient.GetStringAsync() followed by HtmlFormatProvider.Import(); external resources may not load. IronPDF: renderer.RenderUrlAsPdfAsync(url) with full resource resolution.
  • Load existing PDF. TDP: PdfFormatProvider.Import(stream) → RadFixedDocument. IronPDF: PdfDocument.FromFile(path).
  • Save to stream. TDP: PdfFormatProvider.Export(doc, stream). IronPDF: pdf.Stream.CopyTo(stream).

Page Operations

  • Page count. TDP: doc.Pages.Count. IronPDF: pdf.PageCount.
  • Remove page. TDP: doc.Pages.Remove(page). IronPDF: pdf.RemovePages(index).
  • Extract text. TDP iterates page.Content for TextFragment elements. IronPDF: pdf.ExtractAllText(), or page.Text per page.
  • Rotate. Verify the rotation API on both sides for the version you target.

Merge/Split Operations

  • Merge. TDP: import each PDF to a RadFixedDocument, build a third document, copy pages from each source into the merged document, then export. IronPDF: PdfDocument.Merge(doc1, doc2).
  • Split. Verify TDP split support for your version. IronPDF: see the merge or split guide.

What Are the Critical Migration Notes?

The biggest shift in this migration is conceptual. Telerik's RadFixedDocument + FixedContentEditor approach uses an absolute-position layout model — translate X, Y, draw content. IronPDF uses HTML/CSS, which is a flow layout model. The translation is conceptual, not mechanical.

The Document Model to HTML Rewrite

  • editor.Position.Translate(40, 40); DrawText("Title") → <h1 style='margin-left:40px'>Title</h1>
  • editor.DrawRectangle(rect) → <div style='border:1px solid #333; width:...; height:...'>
  • Explicit font size / color via Block.TextProperties → inline CSS, e.g. style='font-size:16px; color:#333'
  • Absolute page coordinates → CSS position:absolute when needed, otherwise flow layout

For complex positioned layouts, this is a design exercise. The HTML template is what ships, not translated coordinate code.

PdfFormatProvider.Export() to pdf.SaveAs()

The most common Telerik export pattern maps directly.

// Telerik:
// var provider = new PdfFormatProvider();
// using var stream = File.Create("out.pdf");
// provider.Export(doc, stream);

// IronPDF:
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("out.pdf");
// Or: pdf.Stream.CopyTo(yourStream);

Package Removal Is Gradual

If your application uses Telerik for PDF and also for Word/Excel documents, remove Telerik.Documents.Fixed and the PDF-format-provider packages while keeping the packages your Word/Excel code still depends on. Telerik.Documents.Flow is shared — it hosts both Word document manipulation and the HtmlFormatProvider used in the HTML-to-PDF path — so check Word usage before removing it. Do not remove the entire Telerik.Documents suite at once if it serves multiple use cases.

# Check what each Telerik.Documents package is used for before removing
rg "Telerik\.Windows\.Documents\.Fixed" --type cs  # Fixed PDF model (load/manipulate)
rg "Telerik\.Windows\.Documents\.Flow" --type cs   # Flow document -- Word AND HTML-to-PDF path
rg "Telerik\.Windows\.Documents\.Spreadsheet" --type cs  # Excel-specific
# Note: HtmlFormatProvider lives in Flow, so Flow may be in use even if Word is not.

Parallel Rendering

using IronPdf;
using System.Linq;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// https://ironpdf.com/examples/parallel/
var pdfs = await Task.WhenAll(reportData.Select(async data =>
{
    var renderer = new ChromePdfRenderer();
    return await renderer.RenderHtmlAsPdfAsync(BuildHtml(data));
}));

foreach (var pdf in pdfs) pdf.Dispose();
// See: https://ironpdf.com/how-to/async/

Disposal vs PdfFormatProvider Pattern

using IronPdf;
using System.IO;

// Telerik: provider.Export(doc, stream) -- stream stays open
// IronPDF: 'using' on PdfDocument handles disposal

var renderer = new ChromePdfRenderer();
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);

// Safe export patterns:
pdf.SaveAs("output.pdf");               // file
pdf.Stream.CopyTo(existingStream);      // stream
var bytes = pdf.BinaryData;             // bytes
// pdf disposed automatically at end of 'using' block

What Are the Frequently Asked Questions?

I get these five questions on almost every Telerik Document Processing migration I advise on. Here they are with direct answers.

Will My PDFs Look Exactly the Same After Migration?

No, and that is expected. Telerik Document Processing has two paths to PDF: a fixed-position document model (RadFixedDocument) where you place content at explicit coordinates, and a flow-document path (RadFlowDocument) that imports HTML through a Word-style intermediate model. IronPDF uses Chromium to render HTML/CSS straight into PDF. The visual output will be equivalent in content but the layout pipeline is fundamentally different — modern CSS that the flow document flattened will now render correctly, and absolute coordinates need to be re-expressed as HTML/CSS. For simple documents like invoices and reports, the HTML/CSS approach typically produces cleaner output with less code. For complex positioned layouts with overlapping elements, you will need to invest design time in the HTML template.

Both libraries use 0-based page indexing, so manipulation code that references page numbers translates directly.

How Do I Handle the RadFixedDocument to HTML Rewrite?

This is the main creative work in the migration. The TDP pattern of editor.Position.Translate(x, y) followed by DrawText() maps to HTML elements with CSS styling. Start with the simplest document templates first. Single-column report layouts translate quickly. Complex multi-column layouts with precise positioning take more HTML design work. Treat each document type as its own mini-project.

The payoff is that HTML templates are easier to maintain, can be previewed in a browser, and do not require recompilation to update styling.

What About Performance and Concurrent Rendering?

IronPDF supports concurrent rendering with separate renderer instances. The parallel rendering pattern using Task.WhenAll works well for batch jobs. Each task gets its own ChromePdfRenderer instance. Monitor memory consumption during the first production week since the Chromium rendering engine has a different memory profile than TDP's document model approach.

Can I Migrate Gradually If I Use TDP for Multiple Document Formats?

Yes. If your application uses Telerik for PDF, Word, and Excel, you can remove the PDF-only packages (Telerik.Documents.Fixed and the PDF format-provider packages) while keeping the Word and Excel packages. The wrinkle is that HtmlFormatProvider — the TDP HTML-to-PDF entry point — lives in Telerik.Documents.Flow, which is also where Word manipulation lives. Run rg "Telerik\.Windows\.Documents\.Fixed" --type cs and rg "RadFlowDocument|HtmlFormatProvider" --type cs to map PDF-specific usage, and migrate those files first. Spreadsheet packages stay untouched.

What Is the Biggest Risk in This Migration?

The document model rewrite. The effort list above marks "Document creation" as High complexity for a reason. Every RadFixedDocument layout needs to be reimagined as an HTML/CSS template. The API swap for merge, watermark, and password protection is straightforward and low risk. The template rewrite is where the time goes. Scope it per document type and test each one individually against reference output from the TDP version.

What Is the Decision Framework for Your Team?

Here is the migration checklist I recommend. Work through it in order: pre-migration assessment, code migration, testing, and post-migration cleanup.

Pre-Migration

Count RadFixedDocument / FixedContentEditor usage with rg "RadFixed\|FixedContent" --type cs | wc -l. Identify which Telerik.Documents packages are PDF-specific vs Word/Excel. Verify Telerik license model — are you paying for the full bundle or PDF specifically? Measure current install footprint with dotnet list package --include-transitive | wc -l. Document features in use: merge, security, signatures, form fields. Obtain an IronPDF license key. Verify IronPDF .NET version compatibility. Plan the HTML template rewrite for each document type.

Code Migration

Install IronPDF with dotnet add package IronPdf. Remove Telerik.Documents.Core, Telerik.Documents.Fixed, and the PDF-related Telerik.Documents.Flow packages once Word usage is confirmed gone. Add the license key at application startup. Replace the HtmlFormatProvider → RadFlowDocument → PdfFormatProvider chain with ChromePdfRenderer.RenderHtmlAsPdfAsync(). Convert any RadFixedDocument + FixedContentEditor layouts to HTML templates. Replace PdfFormatProvider.Export() with pdf.SaveAs() or pdf.Stream. Replace the import-and-copy-pages merge pattern with PdfDocument.Merge(). Replace document-model watermarks (FixedContentEditor.DrawBlock) with TextStamper / ImageStamper. Replace PdfExportSettings password fields with pdf.SecuritySettings. Handle URL rendering via renderer.RenderUrlAsPdfAsync(). Replace page.Content iteration for text with pdf.ExtractAllText().

Testing

Compare PDF output visually against TDP reference exports. Verify package footprint reduction with dotnet list package | grep Telerik. Test merge output page count and order. Test password protection. Benchmark render time vs TDP baseline. Test Docker/Linux deployment. Verify concurrent rendering behavior.

Post-Migration

Remove Telerik.Documents.Fixed and related packages. Confirm remaining Telerik packages are still needed for Word/Excel use cases. Update bundle size metrics if applicable. Archive TDP document model code for reference.

The package footprint reduction is often the most immediately measurable outcome. Running dotnet list package --include-transitive before and after the PDF-specific Telerik packages are removed shows the size delta. The document model-to-HTML rewrite is the main creative work — the most complex TDP layouts take the most HTML design work to replicate, while simpler single-column report layouts are fast.
