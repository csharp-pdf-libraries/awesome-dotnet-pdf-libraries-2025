Migrating from Telerik Document Processing to IronPDF: Less Config, Same Output

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Consider Migrating from Telerik Document Processing?

The conversation usually starts with a dependency audit. Someone runs dotnet list package --include-transitive and the Telerik Document Processing transitive dependency tree scrolls for several screens. There are multiple Telerik.Documents.* packages — some for PDF, some for Word, some for spreadsheets, some for imaging — and the application only uses PDF generation. The install footprint is wider than the actual use case.

Teams evaluating Telerik Document Processing alternatives commonly cite these triggers. First, the package footprint: TDP involves multiple NuGet packages, and teams using only PDF generation pull in the entire Documents suite transitively. Second, the Telerik license dependency: TDP is part of the Telerik ecosystem, and access typically requires a Telerik subscription that bundles PDF with UI components you may not need.

Third, document model verbosity. The RadFixedDocument / FixedContentEditor API is verbose for layout-heavy documents compared to HTML/CSS templates. Fourth, there is no HTML input path — generating PDFs from HTML requires additional packages and integration steps.

Fifth, upgrade coupling. TDP version is often coupled to Telerik UI versions, so upgrading one may require upgrading others. Sixth, if your report content starts as HTML, converting it to a RadFixedDocument model adds intermediate steps that the CSS layout gap creates.

Seventh, Docker and cross-platform deployment: verify which TDP operations are supported cross-platform, as some operations may have platform-specific behavior. Eighth, different Telerik documents packages for PDF vs Word vs Excel means version management across several package IDs. Ninth, some features like text extraction, watermark, and security may require specific TDP packages or configuration. And tenth, TDP documentation is thorough, but community Q&A is concentrated in the Telerik forums rather than Stack Overflow.

How Do You Set Up IronPDF and Replace Telerik Document Processing?

I will walk you through the core migration in three steps. First, find your existing Telerik references, then swap the packages, and finally update your code.

Finding Telerik Document Processing References in Your Codebase

# Find TDP PDF API usage
rg -l "Telerik\.Windows\.Documents\|RadFixedDocument\|PdfFormatProvider\|FixedContentEditor" --type cs
rg "RadFixedDocument\|PdfFormatProvider\|FixedContentEditor" --type cs -n

# Count document model usage (higher migration cost)
rg "RadFixed\|FixedContent\|ContentEditor" --type cs | wc -l

# Find TDP packages in project files
grep -r "Telerik\.Documents" *.csproj **/*.csproj 2>/dev/null

# List all Telerik packages in use
dotnet list package | grep -i telerik

Removing Telerik and Installing IronPDF

# Remove Telerik Document Processing packages
# Note: removing these may affect other Telerik packages -- check dependencies first
dotnet remove package Telerik.Documents.Core
dotnet remove package Telerik.Documents.Fixed
# Remove other Telerik.Documents.* packages specific to your use

# Install IronPDF
dotnet add package IronPdf

dotnet restore

Step 3: Basic PDF Generation

Before (Telerik RadFixedDocument model):

using Telerik.Windows.Documents.Fixed.Model;
using Telerik.Windows.Documents.Fixed.Model.Editing;
using Telerik.Windows.Documents.Fixed.FormatProviders.Pdf;
using System;
using System.IO;

class Program
{
    static void Main()
    {
        var doc = new RadFixedDocument();
        var page = doc.Pages.AddPage();
        page.Size = new System.Windows.Size(595, 842); // A4 in points

        var editor = new FixedContentEditor(page);
        editor.Position.Translate(40, 40);
        editor.DrawText("Hello, PDF!"); // verify method signature

        var provider = new PdfFormatProvider();
        using var outStream = File.Create("output.pdf");
        provider.Export(doc, outStream);
        Console.WriteLine("Saved output.pdf -- verify Telerik API names");
    }
}

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// HTML replaces RadFixedDocument model
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body style='font-family:Arial; padding:40px'><h1>Hello, PDF!</h1></body></html>"
);
pdf.SaveAs("output.pdf");

Console.WriteLine($"Saved output.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

What Does Each Before/After Migration Look Like in Practice?

Let me walk through four complete migration examples so you can see exactly what changes in real code.

1. HTML to PDF

Before (Telerik Document Processing — document model approach):

using Telerik.Windows.Documents.Fixed.Model;
using Telerik.Windows.Documents.Fixed.Model.Editing;
using Telerik.Windows.Documents.Fixed.Model.Resources;

using System.IO;
using System.Windows;

class ReportBefore
{
    static void Main()
    {
        // Telerik: build document via model -- no HTML input
        var doc = new RadFixedDocument();
        var page = doc.Pages.AddPage();
        page.Size = new Size(595, 842);

        var editor = new FixedContentEditor(page);

        // Header -- verify font API in your Telerik version
        editor.Position.Translate(40, 40);
        editor.DrawText("Invoice #2024-0099");  // verify method

        editor.Position.Translate(40, 80);
        editor.DrawText("Customer: Acme Corp"); // verify method

        editor.Position.Translate(40, 120);
        editor.DrawText("Total: $4,200.00");    // verify method

        .field { margin-top: 8px; font-size: 13px; }
        .total { font-size: 16px; font-weight: bold; margin-top: 20px; }
    </style>
    </head>
    <body>
        <div class='title'>Invoice #2024-0099</div>
        <div class='field'>Customer: Acme Corp</div>
        <div class='total'>Total: $4,200.00</div>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");

What Are the Critical Migration Notes?

The biggest shift in this migration is conceptual. Telerik's RadFixedDocument + FixedContentEditor approach uses an absolute-position layout model — translate X, Y, draw content. IronPDF uses HTML/CSS, which is a flow layout model. The translation is conceptual, not mechanical.

The Document Model to HTML Rewrite

TDP PatternHTML/CSS Equivalent
editor.Position.Translate(40, 40); DrawText("Title")<h1 style='margin-left:40px'>Title</h1>
editor.DrawRectangle(rect)<div style='border:1px solid #333; width:...; height:...'>
Explicit font size / colorstyle='font-size:16px; color:#333'
Absolute page coordinatesCSS position:absolute or flow layout

For complex positioned layouts, this is a design exercise. The HTML template is what ships, not translated coordinate code.

PdfFormatProvider.Export() to pdf.SaveAs()

The most common Telerik export pattern maps directly.

// Telerik:
// var provider = new PdfFormatProvider();
// using var stream = File.Create("out.pdf");
// provider.Export(doc, stream);

// IronPDF:
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("out.pdf");
// Or: pdf.Stream.CopyTo(yourStream);

Package Removal Is Gradual

If your application uses Telerik for PDF and also for Word/Excel documents, remove Telerik.Documents.Fixed and PDF-specific packages while keeping Word/Excel packages. Do not remove the entire Telerik.Documents suite at once if it serves multiple use cases.

# Check what each Telerik.Documents package is used for before removing
rg "Telerik\.Windows\.Documents\.Fixed" --type cs  # PDF-specific
rg "Telerik\.Windows\.Documents\.Flow" --type cs   # Word-specific
rg "Telerik\.Windows\.Documents\.Spreadsheet" --type cs  # Excel-specific

What Are the Frequently Asked Questions?

Answer: I get these five questions on almost every Telerik Document Processing migration I advise on. Here they are with direct answers.

Will My PDFs Look Exactly the Same After Migration?

Answer: No, and that is expected. Telerik Document Processing uses a fixed-position document model where you place content at explicit coordinates. IronPDF uses Chromium to render HTML/CSS into PDF. The visual output will be equivalent in content but the layout approach is fundamentally different. For simple documents like invoices and reports, the HTML/CSS approach typically produces cleaner output with less code. For complex positioned layouts with overlapping elements, you will need to invest design time in the HTML template.

Both libraries use 0-based page indexing, so manipulation code that references page numbers translates directly.

How Do I Handle the RadFixedDocument to HTML Rewrite?

Answer: This is the main creative work in the migration. The TDP pattern of editor.Position.Translate(x, y) followed by DrawText() maps to HTML elements with CSS styling. Start with the simplest document templates first. Single-column report layouts translate quickly. Complex multi-column layouts with precise positioning take more HTML design work. Treat each document type as its own mini-project.

The payoff is that HTML templates are easier to maintain, can be previewed in a browser, and do not require recompilation to update styling.

What About Performance and Concurrent Rendering?

Answer: IronPDF supports concurrent rendering with separate renderer instances. The parallel rendering pattern using Task.WhenAll works well for batch jobs. Each task gets its own ChromePdfRenderer instance. Monitor memory consumption during the first production week since the Chromium rendering engine has a different memory profile than TDP's document model approach.

Can I Migrate Gradually If I Use TDP for Multiple Document Formats?

Answer: Yes. If your application uses Telerik for PDF, Word, and Excel, you can remove the PDF-specific packages (Telerik.Documents.Fixed) while keeping the Word and Excel packages. Run rg "Telerik\.Windows\.Documents\.Fixed" --type cs to find PDF-specific usage, and migrate those files first. The other Telerik.Documents packages remain untouched.

What Is the Biggest Risk in This Migration?

Answer: The document model rewrite. The effort table above marks "Document model creation" as High complexity for a reason. Every RadFixedDocument layout needs to be reimagined as an HTML/CSS template. The API swap for merge, watermark, and password protection is straightforward and low risk. The template rewrite is where the time goes. Scope it per document type and test each one individually against reference output from the TDP version.

What Is the Decision Framework for Your Team?

Answer: Here is the migration checklist I recommend. Work through it in order: pre-migration assessment, code migration, testing, and post-migration cleanup.

Pre-Migration

Count RadFixedDocument / FixedContentEditor usage with rg "RadFixed\|FixedContent" --type cs | wc -l. Identify which Telerik.Documents packages are PDF-specific vs Word/Excel. Verify Telerik license model — are you paying for the full bundle or PDF specifically? Measure current install footprint with dotnet list package --include-transitive | wc -l. Document features in use: merge, security, signatures, form fields. Obtain an IronPDF license key. Verify IronPDF .NET version compatibility. Plan the HTML template rewrite for each document type.

Code Migration

Install IronPDF with dotnet add package IronPdf. Remove Telerik.Documents.Core, Telerik.Documents.Fixed, and related packages. Add license key at application startup. Convert RadFixedDocument layouts to HTML templates. Replace PdfFormatProvider.Export() with pdf.SaveAs() or pdf.Stream. Replace RadFixedDocumentMerger with PdfDocument.Merge(). Replace document model watermark with TextStamper / ImageStamper. Replace PdfExportSettings.EncryptionSettings with pdf.SecuritySettings. Handle URL rendering via renderer.RenderUrlAsPdfAsync(). Verify text extraction via pdf.ExtractAllText() if used.

Testing

Compare PDF output visually against TDP reference exports. Verify package footprint reduction with dotnet list package | grep Telerik. Test merge output page count and order. Test password protection. Benchmark render time vs TDP baseline. Test Docker/Linux deployment. Verify concurrent rendering behavior.

Post-Migration

Remove Telerik.Documents.Fixed and related packages. Confirm remaining Telerik packages are still needed for Word/Excel use cases. Update bundle size metrics if applicable. Archive TDP document model code for reference.

The package footprint reduction is often the most immediately measurable outcome. Running dotnet list package --include-transitive before and after the PDF-specific Telerik packages are removed shows the size delta. The document model-to-HTML rewrite is the main creative work — the most complex TDP layouts take the most HTML design work to replicate, while simpler single-column report layouts are fast.
