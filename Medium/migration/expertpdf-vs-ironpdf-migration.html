Migrating from ExpertPdf to IronPDF: Solving Common Migration Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate from ExpertPdf to IronPDF?

Performance regressions are usually what surfaces ExpertPdf as a migration candidate. Teams running batch document generation at scale start noticing memory pressure, render-time variance, or output fidelity issues on complex templates. The question isn't whether ExpertPdf can produce a PDF -- it can -- but whether it can hold up under your specific workload profile.

I built IronPDF on Chromium's rendering engine specifically to address the pain points that custom HTML engines create. Here are the key reasons teams migrate:

CSS rendering fidelity -- ExpertPdf uses a legacy embedded Chrome build whose documentation has not been refreshed since 2018. Modern CSS (Grid, Flexbox, CSS variables, custom fonts via @font-face) may render inconsistently against current browsers.

JavaScript execution -- The legacy rendering pipeline lags current Chromium, so newer ES features and JS-generated DOM may not render the way they do in a current browser.

Linux deployment -- Container environments may need non-obvious packages for ExpertPdf's native dependencies.

Performance at scale -- Under concurrent load, rendering throughput and memory behavior are worth profiling.

Fragmented product suite -- ExpertPdf splits HTML-to-PDF, merging, security, splitting, and PDF-to-image across separate packages (ExpertPdf.HtmlToPdf, ExpertPdf.PDFMerge, ExpertPdf.PDFSecurity, ExpertPdf.PDFSplit, ExpertPdf.PdfToImage), each licensed individually. IronPDF ships everything in one NuGet.

Async rendering -- ExpertPdf is primarily synchronous; verify its current async surface against your ASP.NET pipeline.

Thread safety under concurrency -- Verify whether PdfConverter is thread-safe for concurrent use. In IronPDF, ChromePdfRenderer is reusable across threads.

Debug experience -- Legacy-engine rendering failures can be opaque. A current Chromium engine tends to produce more actionable output.

What Does a Side-by-Side Comparison Look Like?

  • Focus. Both libraries center on HTML and URL to PDF. ExpertPdf uses a legacy embedded engine; IronPDF uses a current Chromium build.
  • Pricing. Both are commercial. ExpertPdf historically lists $550-$1,200 per-component pricing across its split product suite; IronPDF licenses the full feature set as one package. Confirm current rates with each vendor.
  • API style. ExpertPdf exposes a PdfConverter with a PdfDocumentOptions settings object; IronPDF exposes a ChromePdfRenderer with a RenderingOptions object.
  • Learning curve. Low on both sides for basic HTML-to-PDF flows.
  • HTML rendering. ExpertPdf relies on an older embedded engine; IronPDF runs full CSS3 and JavaScript through current Chromium.
  • Page indexing. Both libraries are 0-based, so loop logic transfers without an off-by-one shift.
  • Thread safety. Verify PdfConverter behavior under concurrency in your ExpertPdf version. ChromePdfRenderer is reusable across threads.
  • Namespace. ExpertPdf.HtmlToPdf (plus ExpertPdf.PDFMerge, ExpertPdf.PDFSecurity, etc.) collapses to a single IronPdf namespace.

Problem 1: How Do I Map ExpertPdf Classes and Methods to IronPDF?

The class mapping is straightforward. ExpertPdf's converter pattern maps directly to IronPDF's renderer pattern. Core class equivalences:

  • PdfConverter → ChromePdfRenderer -- main rendering class
  • PdfDocumentOptions → ChromePdfRenderOptions (accessed via renderer.RenderingOptions) -- per-render settings
  • ExpertPdf's converter output (byte[]) → IronPdf.PdfDocument -- the post-render document object
  • PdfPageSize → IronPdf.Rendering.PdfPaperSize -- page size enum
  • PdfPageOrientation → PdfPaperOrientation -- orientation enum
  • PdfHeaderOptions / PdfFooterOptions → HtmlHeaderFooter or TextHeaderFooter -- header/footer configuration
  • PDFMerge (separate ExpertPdf package) → PdfDocument.Merge(...) -- merging built into the core IronPDF NuGet
  • PdfSecurityOptions (separate ExpertPdf package) → pdf.SecuritySettings on the rendered PdfDocument

And here is how the core operations map:

  • Render HTML string. pdfConverter.GetPdfBytesFromHtmlString(html) → renderer.RenderHtmlAsPdf(html).
  • Render URL. pdfConverter.GetPdfBytesFromUrl(url) → renderer.RenderUrlAsPdf(url).
  • Render HTML file. pdfConverter.GetPdfBytesFromHtmlFile(path) → renderer.RenderHtmlFileAsPdf(path).
  • Save directly to file. pdfConverter.SavePdfFromUrlToFile(url, path) → renderer.RenderUrlAsPdf(url).SaveAs(path) (two-step in IronPDF).
  • Merge PDFs. pdfMerge.AppendPDFFile(...) + SaveMergedPDFToFile(...) → PdfDocument.Merge(a, b).
  • Split / extract pages. Use pdf.CopyPages(start, end).
  • Page count. pdf.PageCount.
  • Remove pages. pdf.RemovePages(index).

One critical difference: ExpertPdf's converter typically returns byte[] directly. IronPDF returns a PdfDocument object from which you can get BinaryData (bytes), save to file, or return as a stream. Update callers that expect raw bytes:

Updating Callers That Expect byte[]

// Old pattern -- byte[] from ExpertPdf
// byte[] pdf = converter.GetPdfBytesFromHtmlString(html);
// return File(pdf, "application/pdf");

// New pattern -- extract BinaryData from PdfDocument
using var pdf = renderer.RenderHtmlAsPdf(html);
return File(pdf.BinaryData, "application/pdf", "document.pdf");
// PdfDocument is disposed after BinaryData is copied

Problem 2: How Do I Migrate HTML-to-PDF Rendering with Margins and Page Settings?

This is the most common migration path. The namespace swap is simple, and the rendering options map closely between the two libraries. Here is the complete before/after.

Step 1: Swap License and Namespace

// Before
using ExpertPdf.HtmlToPdf;

// After
using IronPdf;

// Before (ExpertPdf -- license set per-converter instance)
// pdfConverter.LicenseKey = "YOUR-LICENSE-KEY";

// After (IronPDF -- set globally once at startup)
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

Step 2: Replace the Converter with a Renderer

// Before (ExpertPdf)
// PdfConverter pdfConverter = new PdfConverter();
// pdfConverter.PdfDocumentOptions.PdfPageSize = PdfPageSize.A4;
// byte[] result = pdfConverter.GetPdfBytesFromHtmlString("<h1>Hello</h1>");
// File.WriteAllBytes("output.pdf", result);

// After (IronPDF)
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
var renderer = new ChromePdfRenderer();
using var pdf = renderer.RenderHtmlAsPdf("<h1>Hello</h1>");
pdf.SaveAs("output.pdf");

Full HTML-to-PDF Migration with Styling and Margins

using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        string html = @"
            <html>
            <head>
                <style>
                    body { font-family: Arial, sans-serif; }
                    .report-header { background: #0077cc; color: white; padding: 20px; }
                    table { width: 100%; border-collapse: collapse; }
                    td, th { border: 1px solid #ccc; padding: 8px; }
                </style>
            </head>
            <body>
                <div class='report-header'><h1>Sales Report</h1></div>
                <table>
                    <tr><th>Product</th><th>Units</th><th>Revenue</th></tr>
                    <tr><td>Widget A</td><td>1,200</td><td>$24,000</td></tr>
                    <tr><td>Widget B</td><td>800</td><td>$32,000</td></tr>
                </table>
            </body>
            </html>";

        var renderer = new ChromePdfRenderer();
        renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
        renderer.RenderingOptions.MarginTop = 15;
        renderer.RenderingOptions.MarginBottom = 15;

        using var pdf = renderer.RenderHtmlAsPdf(html);
        pdf.SaveAs("sales_report.pdf");
    }
}

Problem 3: How Do I Migrate Watermarks, Merge, and Password Protection?

These are the operations where ExpertPdf and IronPDF diverge the most in API design. Each one in turn.

Merging PDFs

ExpertPdf ships merging as a separate ExpertPdf.PDFMerge NuGet, which exposes a PDFMerge class with AppendPDFFile and SaveMergedPDFToFile. In IronPDF, merging is built into the core package and reduces to a single static call:

using IronPdf;

class MergeSample
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var docA = PdfDocument.FromFile("doc_a.pdf");
        using var docB = PdfDocument.FromFile("doc_b.pdf");

        using var merged = PdfDocument.Merge(docA, docB);
        merged.SaveAs("merged.pdf");
    }
}

Watermarking with the Stamp API

ExpertPdf typically handles watermarks through header HTML overlays or background images. IronPDF has a dedicated stamp API that gives you precise control:

using IronPdf;
using IronPdf.Editing;

class WatermarkSample
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var pdf = PdfDocument.FromFile("input.pdf");

        var stamper = new TextStamper
        {
            Text = "CONFIDENTIAL",
            FontSize = 45,
            Opacity = 30,
            Rotation = 45,
            VerticalAlignment = VerticalAlignment.Middle,
            HorizontalAlignment = HorizontalAlignment.Center
        };

        pdf.ApplyStamp(stamper); // all pages
        pdf.SaveAs("watermarked.pdf");
    }
}

Password Protection and Security

Both libraries support password protection. ExpertPdf exposes it through PdfSecurityOptions in the separate ExpertPdf.PDFSecurity NuGet (UserPassword, OwnerPassword, CanPrint, CanCopyContent, CanEditContent, CanFillFormFields). IronPDF surfaces it as SecuritySettings on the rendered PdfDocument with granular permission enums:

using IronPdf;
using IronPdf.Security;

class SecuritySample
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var pdf = PdfDocument.FromFile("input.pdf");

        pdf.SecuritySettings.UserPassword = "user123";
        pdf.SecuritySettings.OwnerPassword = "owner456";
        pdf.SecuritySettings.AllowUserPrinting = IronPdf.Security.PdfPrintSecurity.FullPrintRights;
        pdf.SecuritySettings.AllowUserCopyPasteContent = false;

        pdf.SaveAs("secured.pdf");
    }
}

How Does Migration Complexity Break Down by Feature?

  • HTML string to PDF -- Low. Direct class swap from PdfConverter to ChromePdfRenderer.
  • URL to PDF -- Low. Both libraries support it; the API names differ.
  • HTML file to PDF -- Low. Path input, similar pattern.
  • Rendering options (margins, size) -- Low. The option objects differ in shape, but property names are mostly similar.
  • Text extraction -- Medium. Verify the extraction API your ExpertPdf version exposes before mapping it to IronPDF.
  • Merge PDFs -- Low. ExpertPdf needs the separate PDFMerge package; IronPDF has it built in.
  • Split PDFs -- Low. ExpertPdf ships PDFSplit as a separate package; IronPDF uses pdf.CopyPages.
  • Watermark -- Medium. ExpertPdf typically uses HTML overlays; IronPDF has a dedicated stamp API.
  • Password / security -- Low. Both expose security settings; ExpertPdf needs its PDFSecurity package, IronPDF does not.
  • Headers / footers -- Medium. ExpertPdf uses &p;/&P; tokens; IronPDF uses {page}/{total-pages} placeholders inside HtmlHeaderFooter or TextHeaderFooter.
  • JavaScript rendering -- N/A to Low. Effectively a new capability in IronPDF for templates that depend on JS-generated DOM.

How Do I Optimize Performance After Migration?

The biggest performance win is reusing your ChromePdfRenderer instance. Don't create a new one per request. Here's the pattern I recommend for ASP.NET and worker services:

Singleton Renderer Pattern

// One renderer per application -- not per request
public class DocumentService
{
    private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

    public async Task<byte[]> GenerateAsync(string html)
    {
        using var pdf = await _renderer.RenderHtmlAsPdfAsync(html);
        return pdf.BinaryData;
    }
}

Cold Start Warm-Up

// Warm up in IHostedService or Startup
public async Task StartAsync(CancellationToken _)
{
    IronPdf.License.LicenseKey = config["IronPdf:LicenseKey"];
    using var warmup = await _renderer.RenderHtmlAsPdfAsync("<p>init</p>");
    _logger.LogInformation("IronPDF renderer warmed up");
}

After migrating, profile memory for at least 24 hours under production load. Watch Gen 2 GC frequency (large objects from PDF bytes), working set trend (should be stable, not growing), and thread pool queue depth under burst load.

Frequently Asked Questions

Can I run IronPDF in Linux containers?

Yes. IronPDF runs on Linux with Chromium dependencies. One of the reasons teams migrate from ExpertPdf is that container environments may need non-obvious packages for ExpertPdf's native dependencies. With IronPDF, the Chromium runtime is bundled, so Linux deployment is straightforward. Test your container base image to confirm all dependencies resolve.

After deployment, update your Dockerfile if Linux dependencies changed from your ExpertPdf setup.

Do I need to change every caller that expects byte[] from ExpertPdf?

Yes, but the change is minimal. ExpertPdf's converter returns byte[] directly. IronPDF returns a PdfDocument object. Access the bytes through pdf.BinaryData. List all methods that return byte[] from ExpertPdf before migrating -- those callers need updating.

The PdfDocument object is disposable, so wrap it in a using statement to ensure proper cleanup.

Will my CSS-heavy templates render differently after migration?

They will likely render better. ExpertPdf relies on a legacy embedded Chrome build that lags current browsers, so modern CSS (Grid, Flexbox, CSS variables, custom fonts) may render inconsistently. IronPDF uses a current Chromium with full CSS and JavaScript support. Capture visual screenshots of key templates before migration, then run a visual diff after migration to verify fidelity.

Specifically test CSS Grid, Flexbox, @font-face custom fonts, and @media print rules -- these are the areas where current Chromium diverges most from older builds.

Is ChromePdfRenderer thread-safe for concurrent use?

Yes. ChromePdfRenderer is reusable across threads, which is why I recommend the singleton pattern above. With ExpertPdf, you should verify whether PdfConverter is thread-safe for concurrent use. In IronPDF, a single renderer instance handles parallel rendering safely.

For batch generation, you can run parallel tasks against a single renderer instance without needing a renderer pool.

How do I benchmark ExpertPdf against IronPDF for my workload?

Choose templates that represent your actual workload distribution: simple HTML (baseline), CSS-heavy (rendering fidelity and timing), image-heavy (memory and throughput), long documents of 100+ pages (scaling behavior), and complex tables with 1000+ rows (streaming versus in-memory). Run these tests on your own hardware.

Use BenchmarkDotNet for structured measurement. First-render latency with IronPDF will be higher due to Chromium startup, but sustained throughput and memory behavior under concurrent load are what matter for server workloads. Measure both, then decide. Check the rendering options documentation for tuning parameters.

When Should You Migrate?

The ExpertPdf migration is relatively mechanical -- both libraries are HTML-first converters, so the conceptual model transfers directly. Scenarios where I recommend migrating now versus waiting:

  • If CSS Grid or Flexbox templates render incorrectly today, migrate -- a current Chromium engine will resolve those layout issues.
  • If you have simple HTML templates with no CSS complexity, the urgency is lower; migrate when it fits the schedule.
  • If you run batch PDF generation and the workload is performance-sensitive, benchmark both libraries on your hardware before deciding.
  • If you deploy to Linux or containers and hit rendering issues, verify ExpertPdf's published Linux support first, then plan the migration.

The trickiest parts are the options object property mapping and any callers expecting byte[] directly from the conversion call rather than a PdfDocument object. Read the IronPDF HTML-to-PDF documentation rather than guessing property names.

The performance story is worth measuring rather than assuming. First-render latency with IronPDF will be higher due to Chromium startup, but sustained throughput and memory behavior under concurrent load are what matter for server workloads. Run the benchmarks before making claims.
