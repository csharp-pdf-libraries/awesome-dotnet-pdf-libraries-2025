Migrating from Sumatra PDF to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate from Sumatra PDF to IronPDF?

The on-premises deployment works fine. Sumatra PDF is on the file path, the process invocation is stable, and the PDFs print correctly. Then the team moves toward containers — and Sumatra PDF is a Windows GUI application. There is no Linux binary, no Docker image, no server mode. A process that shells out to a .exe on a Windows path simply does not exist in a Linux container, and the deployment model collapses entirely.

I have seen this pattern repeat across teams moving from Sumatra PDF to IronPDF. Here are the migration triggers that come up most often.

Windows-only constraint. Sumatra PDF has no Linux binary. Docker on Linux is impossible with this dependency.

Process invocation overhead. Each print or render operation spawns a new process, adding latency and requiring error-handling around exit codes.

Binary management. Bundling SumatraPDF.exe with the application adds to the deployment artifact and requires explicit version management.

Printing vs generation gap. Sumatra PDF prints existing PDFs; it does not generate PDFs from HTML. HTML-to-PDF requires a separate pipeline stage.

Cloud deployment. Azure App Service, AWS Lambda, and similar platforms do not have Windows printer contexts. Sumatra PDF's printing use case does not apply.

Silent failure modes. Sumatra PDF CLI errors surface via process exit codes. Error messages go to stderr and require explicit capture.

No PDF manipulation API. Sumatra PDF does not expose merge, watermark, security, or text extraction via its CLI.

Version pinning. Application behavior depends on the bundled binary version. Updates require explicit deployment of a new binary.

Licensing. Verify Sumatra PDF's commercial use terms. AGPLv3 has implications for closed-source applications.

Audit trail. Shelling out to a binary makes rendering auditing harder than in-process library calls.

How Do You Set Up IronPDF and Replace Sumatra PDF?

I will walk you through the core migration steps. First, find your existing Sumatra PDF references, then swap the packages, and finally update your code.

Finding Sumatra PDF References in Your Codebase

# Find SumatraPDF process invocations
rg -l "SumatraPDF\|sumatrapdf\|SumatraPDF\.exe" --type cs
rg "SumatraPDF\|sumatrapdf" --type cs -n

# Find Process.Start() calls that may invoke Sumatra
rg "Process\.Start.*[Ss]umatra\|StartInfo.*[Ss]umatra" --type cs -n

# Find binary references in deployment scripts
grep -r "SumatraPDF\|sumatra" Dockerfile* .github/**/*.yml deploy/**/*.ps1 2>/dev/null

# Find bundled binary in project
find . -name "SumatraPDF.exe" -o -name "sumatrapdf.exe" 2>/dev/null

Removing Sumatra PDF and Installing IronPDF

# No NuGet package to remove — Sumatra PDF is a bundled binary
# Remove the binary from the project:
# rm -f Tools/SumatraPDF.exe
# rm -f wwwroot/SumatraPDF.exe  # if web-accessible (not recommended)

# Install IronPDF
dotnet add package IronPdf

dotnet restore

Remove from COPY steps in Dockerfile:

# Remove:
# COPY Tools/SumatraPDF.exe /app/Tools/
# ENV SUMATRA_PATH=/app/Tools/SumatraPDF.exe

# IronPDF — no binary copy step needed

Step 3: Basic PDF Generation

Before (Sumatra PDF printing existing PDF via CLI):

using System;
using System.Diagnostics;
using System.IO;

class Program
{
    static void Main()
    {
        // Sumatra PDF: print an existing PDF to a named printer
        // This requires a pre-generated PDF — Sumatra doesn't generate from HTML

        var pdfPath = "existing-document.pdf";
        var printerName = "HP LaserJet Office";

        var psi = new ProcessStartInfo
        {
            FileName = @"C:\Tools\SumatraPDF.exe",   // must be on this path
            // VERIFY: -print-to flag name and syntax in current Sumatra version
            Arguments = $"-print-to \"{printerName}\" \"{pdfPath}\"",
            UseShellExecute = false,
            RedirectStandardError = true,
        };

        using var process = Process.Start(psi)!;
        process.WaitForExit();
        if (process.ExitCode != 0)
            Console.Error.WriteLine("SumatraPDF error: " + process.StandardError.ReadToEnd());
    }
}

After (IronPDF generates PDF; print separately if needed):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Generate the PDF — IronPDF replaces the generation step (if any) upstream of Sumatra
var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Document</h1><p>Content.</p></body></html>"
);
pdf.SaveAs("generated.pdf");

Console.WriteLine($"Generated generated.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

// If raw printing to a physical printer is still needed,
// use System.Drawing.Printing.PrintDocument — IronPDF is a generation library

HTML to PDF (Sumatra Gap to New IronPDF Capability)

Before (Sumatra PDF — requires pre-generated PDF; HTML generation was done separately):

using System;
using System.Diagnostics;
using System.IO;

class HtmlToPdfBefore
{
    static void Main()
    {
        // Sumatra PDF cannot generate from HTML.
        // A typical pattern before this migration:
        // 1. Generate HTML string from data
        // 2. Use another tool to convert HTML to PDF (wkhtmltopdf, Chrome CLI, etc.)
        // 3. Pass the resulting PDF to Sumatra for printing

        var html = "<html><body><h1>Invoice #1042</h1><p>Total: $2,800</p></body></html>";

        // Step 2: Write HTML to temp file and invoke another tool (illustrative)
        var tmpHtml = Path.GetTempFileName() + ".html";
        File.WriteAllText(tmpHtml, html);

        // This needed a SECOND binary alongside Sumatra PDF
        var psi = new ProcessStartInfo
        {
            FileName = @"C:\Tools\wkhtmltopdf.exe",   // or Chrome, Prince, etc.
            Arguments = $"\"{tmpHtml}\" invoice.pdf",
            UseShellExecute = false,
        };
        using var proc = Process.Start(psi)!;
        proc.WaitForExit();
        File.Delete(tmpHtml);

        Console.WriteLine("Needed TWO external binaries for HTML to PDF to print workflow");
    }
}

After:

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Single in-process step replaces the two-binary pipeline
var html = @"
    <html>
    <head>
    <style>
        body { font-family: Arial, sans-serif; padding: 40px; }
        h1 { font-size: 22px; }
        .amount { font-weight: bold; font-size: 18px; margin-top: 20px; }
    </style>
    </head>
    <body>
        <h1>Invoice #1042</h1>
        <p>Date: 2024-09-30 | Customer: Acme Corp</p>
        <div class='amount'>Total Due: $2,800.00</div>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");

Console.WriteLine($"Generated invoice.pdf ({pdf.PageCount} page(s)) — in-process, no external binary");
// https://ironpdf.com/how-to/html-string-to-pdf/

        // 1. Generate PDF A from some source
        // 2. Generate PDF B from another source
        // 3. Merge via secondary library (iTextSharp, PDFSharp, etc.)
        // 4. Pass merged PDF to SumatraPDF.exe for printing

        Console.WriteLine("Merge was done before Sumatra — separate library required");

After:

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

var results = await Task.WhenAll(
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Part A: Summary</h1></body></html>"),
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Part B: Detail</h1></body></html>")
);

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("merged-document.pdf");

Console.WriteLine($"Merged: {merged.PageCount} pages — no external binary required");

What Are the Frequently Asked Questions?

Answer: I get these questions on nearly every Sumatra PDF migration I advise on. Here they are with direct answers.

Is This a Direct Library Swap?

Answer: No. Sumatra PDF and IronPDF solve fundamentally different problems. Sumatra PDF views and prints existing PDF files as a Windows-only CLI application. IronPDF generates and manipulates PDF files as a cross-platform .NET library. The migration is replacing a Windows-only binary invocation workflow with an in-process library. The scope depends on whether your use case was printing existing PDFs (partial migration — IronPDF handles generation, printing still needs OS API) or generating PDFs from HTML (full replacement).

What Replaces Sumatra PDF's Printing Capability?

Answer: IronPDF is a PDF generation library, not a printer API. For physical printer output on Windows, System.Drawing.Printing.PrintDocument is the standard .NET approach. Verify IronPDF's print methods in their documentation as they may cover this use case. You can also keep Sumatra for printing and add IronPDF for generation if both use cases exist in the same codebase.

Will Font Rendering Look Different?

Answer: Yes. Sumatra PDF and IronPDF use different rendering engines. Sumatra displays existing PDFs using its own renderer. IronPDF generates PDFs from HTML via Chromium. These are different pipelines. If the source of truth is an HTML template, render that in Chrome browser with @media print styles enabled (DevTools, Rendering, Emulate CSS media, print) to preview IronPDF output before rendering. The Chromium rendering in IronPDF matches Chrome browser rendering closely. See the pixel-perfect guide for details.

What About Docker Deployments?

Answer: After removing Sumatra (Windows binary) and adding IronPDF, Docker builds succeed but rendering may throw at runtime if system libraries are missing. IronPDF uses embedded Chromium, and on minimal Linux base images Chromium may require libraries that are not present. Use a non-minimal base image or install dependencies like libnss3, libatk-bridge2.0-0, libdrm2, libxkbcommon0, libgbm1, and fonts-liberation. The key benefit is that there is no SumatraPDF.exe to copy into the container.

How Does Error Handling Change?

Answer: Existing error handling that checks process.ExitCode != 0 for Sumatra failures needs to become try/catch. IronPDF uses exception-based error handling via IronPdf.Exceptions.IronPdfException. This is a straightforward pattern replacement across the codebase.

// Before (Sumatra exit code pattern):
// using var proc = Process.Start(psi)!;
// proc.WaitForExit();
// if (proc.ExitCode != 0)
//     throw new Exception($"SumatraPDF failed: exit {proc.ExitCode}");

// After (IronPDF exception pattern):
try
{
    using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    pdf.SaveAs(outputPath);
}
catch (IronPdf.Exceptions.IronPdfException ex)
{
    Console.Error.WriteLine($"PDF generation failed: {ex.Message}");
    throw;
}

What About Performance When Moving from Process Spawn to In-Process?

Answer: Each Sumatra PDF operation spawned a new process. IronPDF renders in-process. The process spawn overhead is non-trivial, especially if Sumatra was invoked per-document in a batch. The first render initializes Chromium (warm-up), but subsequent renders reuse the instance. For parallel workloads, each task can get its own renderer instance. See the async rendering guide for concurrent rendering patterns.

using IronPdf;
using System;
using System.Diagnostics;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Warm-up (first render initializes Chromium)
var renderer = new ChromePdfRenderer();
using var _ = await renderer.RenderHtmlAsPdfAsync("<html><body>warmup</body></html>");

// Batch generation — compare against Sumatra process-per-document baseline
var sw = Stopwatch.StartNew();
for (int i = 0; i < 10; i++)
{
    using var pdf = await renderer.RenderHtmlAsPdfAsync($"<html><body><h1>Document {i}</h1></body></html>");
    pdf.SaveAs($"doc-{i}.pdf");
}
sw.Stop();
Console.WriteLine($"10 documents in {sw.Elapsed.TotalMilliseconds:F0}ms (in-process, no process spawn)");

What Is the Migration Checklist for Your Team?

Answer: Here is the migration checklist I recommend. Work through it in order: pre-migration assessment, code migration, testing, and post-migration cleanup.

Pre-Migration

Identify all Sumatra PDF invocations with rg "SumatraPDF\|sumatrapdf" --type cs. Determine primary use case: printing existing PDFs, or rendering/generation. Identify how PDFs were generated before being passed to Sumatra — this is what IronPDF replaces. Find bundled SumatraPDF.exe in project files. Find Sumatra references in Dockerfiles and CI configs. Verify Sumatra licensing terms if that was a concern. Obtain an IronPDF license key. Verify IronPDF .NET version compatibility.

Code Migration

Install IronPDF with dotnet add package IronPdf. Add license key at application startup. Replace the HTML generation to file to Sumatra pipeline with ChromePdfRenderer. Remove all Process.Start("SumatraPDF.exe") patterns. Replace exit code error handling with try/catch blocks. Remove binary path configuration from appsettings and environment variables. Add merge capability via PdfDocument.Merge() where previously a secondary library was needed. Add watermark via TextStamper. Add password protection via pdf.SecuritySettings. Handle physical printing separately if still required.

Testing

Verify PDF generation in Linux/Docker — the original deployment blocker. Compare PDF output against Sumatra baseline where applicable. Test Docker image builds without the SumatraPDF.exe copy step. Benchmark generation time versus the Sumatra-based pipeline including process spawn. Verify error handling with exceptions instead of exit codes. Test merge, watermark, and security if implemented. Test font rendering in the deployment environment.

Post-Migration

Remove SumatraPDF.exe from project and deployment artifacts. Remove SUMATRA_PATH and related environment variables. Remove COPY SumatraPDF.exe from Dockerfiles. Remove secondary PDF libraries that were used for merge, watermark, or security alongside Sumatra. The deployment blocker — Windows binary in a Linux container — is resolved structurally. The underlying problem goes away, not just the symptom.
