Migrating from Sumatra PDF to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate from Sumatra PDF to IronPDF?

The on-premises deployment works fine. Sumatra PDF is on the file path, the process invocation is stable, and the PDFs print correctly. Then the team moves toward containers — and Sumatra PDF is a Windows GUI application. There is no Linux binary, no Docker image, no server mode. A process that shells out to a .exe on a Windows path simply does not exist in a Linux container, and the deployment model collapses entirely.

I have seen this pattern repeat across teams moving from Sumatra PDF to IronPDF. Here are the migration triggers that come up most often.

Windows-only constraint. Sumatra PDF has no Linux binary. Docker on Linux is impossible with this dependency.

Process invocation overhead. Each print or render operation spawns a new process, adding latency and requiring error-handling around exit codes.

Binary management. Bundling SumatraPDF.exe with the application adds to the deployment artifact and requires explicit version management.

Printing vs generation gap. Sumatra PDF prints existing PDFs; it does not generate PDFs from HTML. HTML-to-PDF requires a separate pipeline stage.

Cloud deployment. Azure App Service, AWS Lambda, and similar platforms do not have Windows printer contexts. Sumatra PDF's printing use case does not apply.

Silent failure modes. Sumatra PDF CLI errors surface via process exit codes. Error messages go to stderr and require explicit capture.

No PDF manipulation API. Sumatra PDF does not expose merge, watermark, security, or text extraction via its CLI.

Version pinning. Application behavior depends on the bundled binary version. Updates require explicit deployment of a new binary.

Licensing. Sumatra PDF is distributed under GPLv3 (with portions under BSD); confirm the current terms in the project's COPYING file. GPLv3's static-linking copyleft has implications if you redistribute Sumatra binaries with a closed-source application or rely on it as more than an out-of-process tool.

Audit trail. Shelling out to a binary makes rendering auditing harder than in-process library calls.

Comparison Table

  • Focus. Sumatra PDF is a viewer, printer, and image renderer; IronPDF is a generation and manipulation library.
  • Pricing. Sumatra PDF is open source under GPLv3 (with some BSD-licensed code) — verify GPLv3 redistribution terms if you ship the binary with a closed-source application. IronPDF is commercial — verify pricing at ironsoftware.com.
  • API style. Sumatra PDF is invoked via Process.Start() on a Windows binary. IronPDF is a native .NET library with no external process.
  • Learning curve. Sumatra is low for basic printing through a handful of CLI flags. IronPDF is low for .NET devs with idiomatic class APIs.
  • HTML rendering. Not applicable to Sumatra — it only renders existing PDFs. IronPDF uses an embedded Chromium engine.
  • Page indexing. Sumatra's CLI flags are 1-based; IronPDF is 0-based.
  • Thread safety. Sumatra gets process-level isolation by virtue of being a separate binary. Verify IronPDF concurrent instance guidance for in-process parallelism.
  • Namespace. Sumatra has no .NET namespace — it is a CLI binary. IronPDF lives under IronPdf.

What Does the Migration Effort Look Like?

I want to give you a realistic picture of the effort involved. This is not a typical library swap — it is a workflow replacement. Sumatra PDF views and prints existing PDF files as a Windows-only CLI application. IronPDF generates and manipulates PDF files as a cross-platform .NET library. The scope depends on whether your use case was printing existing PDFs or generating PDFs from HTML.

Effort by Feature

  • Print PDF to printer — Medium. SumatraPDF.exe -print-to PrinterName becomes pdf.Print("PrinterName") (or pdf.Print(xDpi, yDpi, "PrinterName") for DPI control). Note: IronPDF's printing API is Windows-only.
  • Print to default printer — Medium. Sumatra's -print-to-default becomes pdf.Print(); Windows-only.
  • Render PDF page to image — Low (new capability). Not exposed by Sumatra's documented CLI — teams typically shelled out to a second tool. IronPDF provides a native image extraction API; verify the current method names in the docs.
  • Open / display PDF — Low. Sumatra was launched as the viewer via Process.Start("SumatraPDF.exe", path). With IronPDF, save the file and open with the system-associated viewer using UseShellExecute = true.
  • Extract text — Low (new capability). Not applicable to Sumatra (the .cs sample shells out to pdftotext.exe); IronPDF exposes pdf.ExtractAllText().
  • Generate PDF from HTML — Low (new capability). Not applicable to Sumatra; IronPDF uses ChromePdfRenderer.RenderHtmlAsPdfAsync().
  • Generate PDF from URL — Low (new capability). Not applicable to Sumatra; IronPDF uses renderer.RenderUrlAsPdfAsync().
  • Cross-platform — Low (inherited). Sumatra is Windows only; IronPDF runs on Linux, macOS, and Windows.
  • Process spawn overhead — Low (eliminated). Sumatra spawned a process per operation; IronPDF runs in-process.

Decision Matrix

  • If you print PDFs to physical printers on Windows, IronPDF exposes pdf.Print() for the default printer and overloads accepting a printer name and optional DPI: pdf.Print("PrinterName"), pdf.Print(dpi), pdf.Print(xDpi, yDpi, "PrinterName"). For low-level control (copies, duplex, page range), use pdf.GetPrintDocument(printerSettings) with a System.Drawing.Printing.PrinterSettings. IronPDF's printing API is Windows-only.
  • If you generate PDFs from HTML or URLs (new requirement), switch — IronPDF is designed for this and replaces any wkhtmltopdf-style intermediary used upstream of Sumatra.
  • If you are moving to Linux or Docker containers, switch — this eliminates the Windows-only binary dependency that blocked the move.
  • If you render PDF pages to images, evaluate IronPDF's image rendering API — verify method names in the current docs.
  • If you only opened PDFs in a GUI viewer, note that IronPDF is not a viewer. Save with pdf.SaveAs() and hand off to the system-associated viewer via Process.Start with UseShellExecute = true.

How Do You Set Up IronPDF and Replace Sumatra PDF?

I will walk you through the core migration steps. First, find your existing Sumatra PDF references, then swap the packages, and finally update your code.

Finding Sumatra PDF References in Your Codebase

# Find SumatraPDF process invocations
rg -l "SumatraPDF\|sumatrapdf\|SumatraPDF\.exe" --type cs
rg "SumatraPDF\|sumatrapdf" --type cs -n

# Find Process.Start() calls that may invoke Sumatra
rg "Process\.Start.*[Ss]umatra\|StartInfo.*[Ss]umatra" --type cs -n

# Find binary references in deployment scripts
grep -r "SumatraPDF\|sumatra" Dockerfile* .github/**/*.yml deploy/**/*.ps1 2>/dev/null

# Find bundled binary in project
find . -name "SumatraPDF.exe" -o -name "sumatrapdf.exe" 2>/dev/null

Removing Sumatra PDF and Installing IronPDF

# No NuGet package to remove — Sumatra PDF is a bundled binary
# Remove the binary from the project:
# rm -f Tools/SumatraPDF.exe
# rm -f wwwroot/SumatraPDF.exe  # if web-accessible (not recommended)

# Install IronPDF
dotnet add package IronPdf

dotnet restore

Remove from COPY steps in Dockerfile:

# Remove:
# COPY Tools/SumatraPDF.exe /app/Tools/
# ENV SUMATRA_PATH=/app/Tools/SumatraPDF.exe

# IronPDF — no binary copy step needed

Step 3: Basic PDF Generation

Before (Sumatra PDF printing existing PDF via CLI):

using System;
using System.Diagnostics;
using System.IO;

class Program
{
    static void Main()
    {
        // Sumatra PDF: print an existing PDF to a named printer
        // This requires a pre-generated PDF — Sumatra doesn't generate from HTML

        var pdfPath = "existing-document.pdf";
        var printerName = "HP LaserJet Office";

        var psi = new ProcessStartInfo
        {
            FileName = @"C:\Tools\SumatraPDF.exe",   // must be on this path
            // VERIFY: -print-to flag name and syntax in current Sumatra version
            Arguments = $"-print-to \"{printerName}\" \"{pdfPath}\"",
            UseShellExecute = false,
            RedirectStandardError = true,
        };

        using var process = Process.Start(psi)!;
        process.WaitForExit();
        if (process.ExitCode != 0)
            Console.Error.WriteLine("SumatraPDF error: " + process.StandardError.ReadToEnd());
    }
}

After (IronPDF generates PDF; print separately if needed):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Generate the PDF — IronPDF replaces the generation step (if any) upstream of Sumatra
var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Document</h1><p>Content.</p></body></html>"
);
pdf.SaveAs("generated.pdf");

Console.WriteLine($"Generated generated.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

// If raw printing to a physical printer is still needed,
// use System.Drawing.Printing.PrintDocument — IronPDF is a generation library

What Does Each Before/After Migration Look Like in Practice?

Here are four complete before/after migrations covering the most common scenarios teams encounter when moving from Sumatra PDF to IronPDF.

HTML to PDF (Sumatra Gap to New IronPDF Capability)

Before (Sumatra PDF — requires pre-generated PDF; HTML generation was done separately):

using System;
using System.Diagnostics;
using System.IO;

class HtmlToPdfBefore
{
    static void Main()
    {
        // Sumatra PDF cannot generate from HTML.
        // A typical pattern before this migration:
        // 1. Generate HTML string from data
        // 2. Use another tool to convert HTML to PDF (wkhtmltopdf, Chrome CLI, etc.)
        // 3. Pass the resulting PDF to Sumatra for printing

        var html = "<html><body><h1>Invoice #1042</h1><p>Total: $2,800</p></body></html>";

        // Step 2: Write HTML to temp file and invoke another tool (illustrative)
        var tmpHtml = Path.GetTempFileName() + ".html";
        File.WriteAllText(tmpHtml, html);

        // This needed a SECOND binary alongside Sumatra PDF
        var psi = new ProcessStartInfo
        {
            FileName = @"C:\Tools\wkhtmltopdf.exe",   // or Chrome, Prince, etc.
            Arguments = $"\"{tmpHtml}\" invoice.pdf",
            UseShellExecute = false,
        };
        using var proc = Process.Start(psi)!;
        proc.WaitForExit();
        File.Delete(tmpHtml);

        Console.WriteLine("Needed TWO external binaries for HTML to PDF to print workflow");
    }
}

After:

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Single in-process step replaces the two-binary pipeline
var html = @"
    <html>
    <head>
    <style>
        body { font-family: Arial, sans-serif; padding: 40px; }
        h1 { font-size: 22px; }
        .amount { font-weight: bold; font-size: 18px; margin-top: 20px; }
    </style>
    </head>
    <body>
        <h1>Invoice #1042</h1>
        <p>Date: 2024-09-30 | Customer: Acme Corp</p>
        <div class='amount'>Total Due: $2,800.00</div>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");

Console.WriteLine($"Generated invoice.pdf ({pdf.PageCount} page(s)) — in-process, no external binary");
// https://ironpdf.com/how-to/html-string-to-pdf/

Open / Display PDF

Before (Sumatra PDF — launched as external viewer):

using System.Diagnostics;

class OpenDisplayBefore
{
    static void Main()
    {
        var pdfPath = "document.pdf";

        // Sumatra excels at viewing PDFs — but only on Windows,
        // and only when SumatraPDF.exe is on the path or installed.
        var startInfo = new ProcessStartInfo
        {
            FileName = "SumatraPDF.exe",
            Arguments = $"\"{pdfPath}\"",
            UseShellExecute = true,
        };

        Process.Start(startInfo);

        // Optional: open at a specific page
        // Arguments = $"-page 5 \"{pdfPath}\""
    }
}

After (IronPDF generates / loads, then hands off to the system viewer):

using IronPdf;
using System;
using System.Diagnostics;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Load (or generate) the PDF in-process — no SumatraPDF.exe required
var pdf = PdfDocument.FromFile("document.pdf");
Console.WriteLine($"Page count: {pdf.PageCount}");

// Save any edits, then hand off to the OS-associated viewer
pdf.SaveAs("modified.pdf");
Process.Start(new ProcessStartInfo("modified.pdf") { UseShellExecute = true });
// IronPDF is not itself a GUI viewer — the OS picks the viewer
// (Edge, Acrobat, the user's default) via UseShellExecute.

Printing a PDF

Before (Sumatra PDF CLI print):

using System;
using System.Diagnostics;

class PrintBefore
{
    static void Main()
    {
        var pdfPath = "document.pdf";

        // Print to the default printer via Sumatra CLI
        var psi = new ProcessStartInfo
        {
            FileName = @"C:\Tools\SumatraPDF.exe",
            // Verify the -print-to-default flag against the current Sumatra version
            Arguments = $"-print-to-default \"{pdfPath}\"",
            UseShellExecute = false,
            CreateNoWindow = true,
            RedirectStandardError = true,
        };

        using var process = Process.Start(psi)!;
        process.WaitForExit();
        if (process.ExitCode != 0)
            Console.Error.WriteLine("SumatraPDF error: " + process.StandardError.ReadToEnd());
    }
}

After:

using IronPdf;
using System;
using System.Drawing.Printing;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var pdf = PdfDocument.FromFile("document.pdf");

// Print to the default printer — in-process, no Sumatra binary
pdf.Print();

// Or print to a named printer (with optional DPI control)
pdf.Print("HP LaserJet Office");
pdf.Print(300, 300, "HP LaserJet Office");

// For copies / duplex / page-range, configure PrinterSettings explicitly
var settings = new PrinterSettings
{
    PrinterName = "HP LaserJet Office",
    Copies = 2,
};
pdf.GetPrintDocument(settings).Print();

// IronPDF printing is Windows-only; on Linux/Docker, generate the PDF and
// hand off to a Windows print server or system tool.
// https://ironpdf.com/how-to/print-pdf/

Extract Text from a PDF

Before (Sumatra has no text-extraction API — the .cs sample shells out to pdftotext.exe):

using System;
using System.Diagnostics;
using System.IO;

class ExtractTextBefore
{
    static void Main()
    {
        var pdfFile = "document.pdf";

        // Sumatra is a viewer; it does not expose a programmatic
        // text-extraction API. Teams typically shell out to a second
        // binary like pdftotext.exe (Poppler) and read the result.
        var psi = new ProcessStartInfo
        {
            FileName = "pdftotext.exe",
            Arguments = $"\"{pdfFile}\" output.txt",
            UseShellExecute = false,
        };

        Process.Start(psi)?.WaitForExit();

        var extractedText = File.ReadAllText("output.txt");
        Console.WriteLine(extractedText);
    }
}

After:

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var pdf = PdfDocument.FromFile("document.pdf");

// Whole-document text — in-process, no second binary
var allText = pdf.ExtractAllText();
Console.WriteLine(allText);

// Per-page text (0-based; Sumatra CLI was 1-based)
for (int i = 0; i < pdf.PageCount; i++)
{
    var pageText = pdf.ExtractTextFromPage(i);
    Console.WriteLine($"Page {i + 1}: {pageText}");
}
// https://ironpdf.com/how-to/extract-text-and-images/

How Do the Two Libraries Compare Feature by Feature?

Here is a complete mapping of the API surface between Sumatra PDF and IronPDF. I find it helpful to see these side by side when planning the migration.

Namespace Mapping

  • None (System.Diagnostics for Process.Start) → IronPdf — core namespace
  • None → IronPdf.Rendering — rendering configuration
  • None → IronPdf.Editing — watermark and stamp APIs

Core Class Mapping

  • SumatraPDF.exe -print-to → pdf.Print("PrinterName") (or pdf.Print(xDpi, yDpi, "PrinterName")) — in-process, Windows-only
  • Process.Start() wrapper around the Sumatra binary → ChromePdfRenderer — in-process rendering, no external binary
  • Exit-code error handling → try / catch (IronPdf.Exceptions.IronPdfProductException) — exceptions instead of integer exit codes
  • No equivalent → PdfDocument — the in-memory PDF object for manipulation, extraction, and security

Document Loading Methods

  • Generate from HTML. Not applicable to Sumatra; IronPDF uses renderer.RenderHtmlAsPdfAsync(html).
  • Render from URL. Not applicable to Sumatra; IronPDF uses renderer.RenderUrlAsPdfAsync(url).
  • Load existing PDF. Sumatra received a file path on its CLI; IronPDF uses PdfDocument.FromFile(path).
  • Save to file. Sumatra wrote output via its CLI flags; IronPDF uses pdf.SaveAs(path).

Page Operations

  • Page count. Not exposed by Sumatra's CLI; IronPDF uses pdf.PageCount.
  • Remove page. Not applicable to Sumatra; IronPDF uses pdf.RemovePages(index).
  • Extract text. Not applicable to Sumatra; IronPDF uses pdf.ExtractAllText() and pdf.ExtractTextFromPage(i).
  • Rotate. Sumatra exposes rotation only via print-settings flags; verify the equivalent IronPDF rotate API in docs.

Merge / Split Operations

  • Merge. Not applicable to Sumatra; IronPDF uses PdfDocument.Merge(doc1, doc2).
  • Split. Not applicable to Sumatra; see the IronPDF merge / split guide.

What Are the Frequently Asked Questions?

I get these questions on nearly every Sumatra PDF migration I advise on. Here they are with direct answers.

Is This a Direct Library Swap?

No. Sumatra PDF and IronPDF solve fundamentally different problems. Sumatra PDF views and prints existing PDF files as a Windows-only CLI application. IronPDF generates and manipulates PDF files as a cross-platform .NET library. The migration is replacing a Windows-only binary invocation workflow with an in-process library. The scope depends on whether your use case was printing existing PDFs (IronPDF exposes pdf.Print(), with System.Drawing.Printing as a fallback for low-level control) or generating PDFs from HTML (a full replacement, plus the wkhtmltopdf-style intermediary upstream of Sumatra goes away).

What Replaces Sumatra PDF's Printing Capability?

IronPDF exposes pdf.Print() for the default printer and overloads accepting a printer name plus optional DPI: pdf.Print("PrinterName"), pdf.Print(dpi), and pdf.Print(xDpi, yDpi, "PrinterName"). For low-level control over copies, duplex, and page range, configure a System.Drawing.Printing.PrinterSettings and pass it to pdf.GetPrintDocument(settings).Print(). IronPDF's printing API is Windows-only — on Linux or Docker deployments, generate the PDF in-process and hand off to a Windows print server or system tool for physical-printer dispatch. You can also keep Sumatra for printing and add IronPDF for generation if both use cases coexist in the same codebase.

Will Font Rendering Look Different?

Yes. Sumatra PDF and IronPDF use different rendering engines. Sumatra displays existing PDFs using its own renderer. IronPDF generates PDFs from HTML via Chromium. These are different pipelines. If the source of truth is an HTML template, render that in Chrome browser with @media print styles enabled (DevTools, Rendering, Emulate CSS media, print) to preview IronPDF output before rendering. The Chromium rendering in IronPDF matches Chrome browser rendering closely. See the pixel-perfect guide for details.

What About Docker Deployments?

After removing Sumatra (Windows binary) and adding IronPDF, Docker builds succeed but rendering may throw at runtime if system libraries are missing. IronPDF uses embedded Chromium, and on minimal Linux base images Chromium may require libraries that are not present. Use a non-minimal base image or install the Chromium dependencies Iron documents (typically libnss3, libatk-bridge2.0-0, libgbm1, plus distro-specific packages); see the IronPDF Linux troubleshooting guide for the authoritative, distro-by-distro install list. The key benefit is that there is no SumatraPDF.exe to copy into the container.

How Does Error Handling Change?

Existing error handling that checks process.ExitCode != 0 for Sumatra failures needs to become try/catch. IronPDF uses exception-based error handling. The IronPdf.Exceptions namespace exposes concrete types including IronPdfProductException, IronPdfDeploymentException, IronPdfNativeException, and IronPdfUnsupportedException — catch the most relevant type, or fall back to System.Exception when wrapping multiple. This is a straightforward pattern replacement across the codebase.

// Before (Sumatra exit code pattern):
// using var proc = Process.Start(psi)!;
// proc.WaitForExit();
// if (proc.ExitCode != 0)
//     throw new Exception($"SumatraPDF failed: exit {proc.ExitCode}");

// After (IronPDF exception pattern):
try
{
    using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    pdf.SaveAs(outputPath);
}
catch (IronPdf.Exceptions.IronPdfProductException ex)
{
    Console.Error.WriteLine($"PDF generation failed: {ex.Message}");
    throw;
}

What About Performance When Moving from Process Spawn to In-Process?

Each Sumatra PDF operation spawned a new process. IronPDF renders in-process. The process spawn overhead is non-trivial, especially if Sumatra was invoked per-document in a batch. The first render initializes Chromium (warm-up), but subsequent renders reuse the instance. For parallel workloads, each task can get its own renderer instance. See the async rendering guide for concurrent rendering patterns.

using IronPdf;
using System;
using System.Diagnostics;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Warm-up (first render initializes Chromium)
var renderer = new ChromePdfRenderer();
using var _ = await renderer.RenderHtmlAsPdfAsync("<html><body>warmup</body></html>");

// Batch generation — compare against Sumatra process-per-document baseline
var sw = Stopwatch.StartNew();
for (int i = 0; i < 10; i++)
{
    using var pdf = await renderer.RenderHtmlAsPdfAsync($"<html><body><h1>Document {i}</h1></body></html>");
    pdf.SaveAs($"doc-{i}.pdf");
}
sw.Stop();
Console.WriteLine($"10 documents in {sw.Elapsed.TotalMilliseconds:F0}ms (in-process, no process spawn)");

What Is the Migration Checklist for Your Team?

Here is the migration checklist I recommend. Work through it in order: pre-migration assessment, code migration, testing, and post-migration cleanup.

Pre-Migration

Identify all Sumatra PDF invocations with rg "SumatraPDF\|sumatrapdf" --type cs. Determine primary use case: printing existing PDFs, or rendering/generation. Identify how PDFs were generated before being passed to Sumatra — this is what IronPDF replaces. Find bundled SumatraPDF.exe in project files. Find Sumatra references in Dockerfiles and CI configs. Verify Sumatra GPLv3 terms if redistributing the binary with a closed-source application. Obtain an IronPDF license key. Verify IronPDF .NET version compatibility.

Code Migration

Install IronPDF with dotnet add package IronPdf. Add the license key at application startup. Replace the HTML-generation-to-file-to-Sumatra pipeline with ChromePdfRenderer. Remove all Process.Start("SumatraPDF.exe") patterns — viewing becomes UseShellExecute = true against the OS-associated viewer, printing becomes pdf.Print(), and text extraction becomes pdf.ExtractAllText(). Replace exit-code error handling with try/catch around IronPdfException. Remove binary-path configuration from appsettings and environment variables. Drop any secondary pdftotext.exe or wkhtmltopdf.exe shell-outs that fed Sumatra. Add merge via PdfDocument.Merge(), watermark via TextStamper, and password protection via pdf.SecuritySettings if those gaps existed.

Testing

Verify PDF rendering and printing in Linux/Docker — the original deployment blocker. Compare PDF output against the Sumatra baseline where applicable. Test Docker image builds without the SumatraPDF.exe copy step. Benchmark in-process rendering and printing versus the Sumatra-based pipeline including process spawn. Verify error handling with exceptions instead of exit codes. Test text extraction parity against the prior pdftotext.exe output. Test font rendering in the deployment environment.

Post-Migration

Remove SumatraPDF.exe from project and deployment artifacts. Remove SUMATRA_PATH and related environment variables. Remove COPY SumatraPDF.exe from Dockerfiles. Remove secondary PDF libraries that were used for merge, watermark, or security alongside Sumatra. The deployment blocker — Windows binary in a Linux container — is resolved structurally. The underlying problem goes away, not just the symptom.
