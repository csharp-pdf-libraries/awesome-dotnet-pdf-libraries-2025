Migrating from ActivePDF to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Three separate ActivePDF NuGet packages. Three separate DLL references. Three separate licensing conversations with procurement. Your WebGrabber handles HTML conversion, Toolkit handles manipulation, and DocConverter handles everything else -- but none of them talk to each other cleanly, and all of them require Windows Server.

Then the Foxit acquisition happened, and now your support tickets route through a portal that looks nothing like the one your team bookmarked two years ago. The product still works. But the question every architect eventually asks is: do we really need three components for what is, fundamentally, one pipeline?

This guide walks you through consolidating ActivePDF into IronPDF -- a single NuGet package that covers HTML-to-PDF, manipulation, merge, watermark, and security.

Why Should I Migrate Away from ActivePDF?

ActivePDF has served enterprise .NET shops for over a decade. Here is why teams re-evaluate:

Modular overhead. You buy and license Toolkit, WebGrabber, and DocConverter separately. Each has its own DLL, its own NuGet package, its own API surface. A single PDF generation workflow might touch all three.

Windows Server lock-in. ActivePDF components require Windows Server. If you are moving to Linux containers, Docker on ARM, or Azure App Service on Linux, ActivePDF does not follow you.

MSI installer dependencies. WebGrabber and DocConverter require MSI installation on the host -- not just a NuGet restore. This breaks immutable infrastructure patterns and complicates CI/CD.

Acquisition uncertainty. Foxit acquired ActivePDF and merged support portals. The product roadmap, pricing, and long-term component strategy may shift toward Foxit's own PDF SDK.

IE-era rendering engine. WebGrabber historically offered Native and Internet Explorer rendering engines. Modern HTML5/CSS3 applications need a Chrome-class renderer.

API verbosity. Toolkit uses COM-style patterns inherited from its Classic ASP roots. Method names like SetMasterPassword and CopyForm work, but they are verbose compared to fluent property-based APIs.

Limited cross-platform path. Even though Toolkit's NuGet targets .NET Standard 1.0, the underlying native code and WebGrabber/DocConverter dependencies are Windows-only.

Separate HTML conversion path. If you need both HTML-to-PDF and PDF manipulation, you are calling WebGrabber to generate the PDF, then loading it into Toolkit to manipulate it. That is two separate APIs for one workflow.

DocConverter as a Windows service. DocConverter runs as a watched-folder service or COM automation host. In a microservices architecture, this model feels like a detour.

License complexity. Each ActivePDF component is licensed separately, often per-server. For teams scaling horizontally, the licensing math gets expensive.

How Do ActivePDF and IronPDF Compare Feature by Feature?

  • Focus. ActivePDF is modular -- Toolkit handles manipulation, WebGrabber handles HTML-to-PDF. IronPDF unifies generation and manipulation in a single package.
  • Pricing. ActivePDF licenses per component, typically per server. IronPDF uses per-developer licensing, with perpetual options available.
  • API style. ActivePDF carries COM-influenced, procedural methods. IronPDF exposes fluent properties in idiomatic C#.
  • Learning curve. ActivePDF requires learning multiple component APIs; IronPDF lives in a single namespace with discoverable members.
  • HTML rendering. WebGrabber historically used Native and Internet Explorer engines. IronPDF uses a Chrome rendering engine that handles modern web standards.
  • Page indexing. Toolkit is 1-based. IronPDF is 0-based -- a known off-by-one trap during migration.
  • Thread safety. Toolkit's threading model varies by version and component -- verify before sharing instances. IronPDF's ChromePdfRenderer is stateless and safe to share.
  • Namespace. APToolkit / APWebGrabber versus IronPdf.

How Much Effort Does Each Feature Require to Migrate?

  • HTML to PDF (from WebGrabber) -- Low. Direct replacement; IronPDF eliminates the separate WebGrabber dependency.
  • PDF manipulation (from Toolkit) -- Medium. API shape differs; Toolkit's COM-style methods map to IronPDF properties.
  • Merge -- Low. PdfDocument.Merge() replaces Toolkit's merge methods.
  • Split / extract pages -- Low. IronPDF's page indexer replaces Toolkit page operations.
  • Watermark / stamping -- Medium. Toolkit uses SetHeaderWatermark; IronPDF uses HTML-based ApplyWatermark.
  • Password / encryption -- Low. Both support owner/user passwords; property names differ.
  • Form filling -- Medium. Toolkit has extensive form APIs; IronPDF uses a FormField collection -- verify your specific AcroForm depth before migrating.
  • Document conversion (300+ formats) -- High / N/A. DocConverter has no direct IronPDF equivalent. Use IronPDF for HTML/URL; for Office to PDF, consider a dedicated converter.
  • Print-to-PDF (Server) -- N/A. No direct equivalent in IronPDF. Consider OS-level print drivers or alternative tools.
  • Watched-folder automation (Meridian) -- N/A. Architectural pattern -- implement with FileSystemWatcher or message queues.
  • Digital signatures -- Medium. Different API surface; verify IronPDF signing documentation against your PAdES requirements.

Which Migration Scenario Fits My Project?

  • If your stack is HTML-to-PDF plus basic manipulation with no DocConverter, migrate -- IronPDF unifies WebGrabber and Toolkit into one NuGet.
  • If you depend heavily on DocConverter for Office or CAD conversions, take a partial migration: use IronPDF for HTML/PDF operations and keep DocConverter, or evaluate a dedicated converter on a separate timeline.
  • If you are scaling to Linux, Docker, or cross-platform deployment, migrate -- ActivePDF components are Windows-only.
  • If you have deep Toolkit form automation, evaluate IronPDF's form API against your specific AcroForm workflows before committing.

How Do I Set Up the Migration Step by Step?

You will need .NET 6+ (or .NET Framework 4.6.2+), a trial or licensed IronPDF key, and an inventory of which ActivePDF components your codebase uses.

Find ActivePDF References

# Find all ActivePDF using statements and DLL references
rg -l "APToolkit|APWebGrabber|ActivePDF|APDocConverter" --glob "*.cs"

# Count Toolkit instantiations
rg -c "new APToolkit\(\)|new Toolkit\(\)" --glob "*.cs"

# Find project/config references
rg -l "ActivePDF" --glob "*.csproj" --glob "*.config"

If you don't have rg:

grep -rl "APToolkit\|APWebGrabber\|ActivePDF" --include="*.cs" .

In PowerShell, Get-ChildItem -Recurse -Filter *.cs | Select-String "APToolkit|APWebGrabber|ActivePDF" covers the same ground.

Swap NuGet Packages

# Remove ActivePDF packages
dotnet remove package ActivePDF.Toolkit
dotnet remove package ActivePDF.WebGrabber  # if present as NuGet

# Install IronPDF (replaces both)
dotnet add package IronPdf

If WebGrabber or DocConverter were installed via MSI rather than NuGet, remove the DLL references from your .csproj manually and uninstall the MSI from your build/deploy hosts.

Step 3: Basic HTML-to-PDF

Before (ActivePDF WebGrabber -- ~18 lines):

// ActivePDF WebGrabber -- verify exact property names against your installed version
using System;

class Program
{
    static void Main()
    {
        var grabber = new WebGrabber();
        grabber.SetLicenseKey("YOUR-LICENSE-KEY");
        grabber.CreateFromHTMLText = true;

        grabber.HTMLText = @"
            <html><body>
                <h1>Invoice #1042</h1>
                <p>Amount: $5,200.00</p>
            </body></html>";

        grabber.OutputDirectory = @"C:\output\";
        grabber.NewDocumentName = "invoice.pdf";

        grabber.PageWidth = 8.5f;   // inches
        grabber.PageHeight = 11.0f; // inches
        grabber.MarginTop = 0.5f;
        grabber.MarginBottom = 0.5f;

        int result = grabber.ConvertToPDF();
        if (result != 0)
        {
            Console.WriteLine("WebGrabber error: " + result);
        }
        else
        {
            Console.WriteLine("Saved with WebGrabber.");
        }
    }
}

After (IronPDF -- 10 lines):

using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var renderer = new ChromePdfRenderer();
        renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.Letter;
        renderer.RenderingOptions.MarginTop = 12; // mm
        renderer.RenderingOptions.MarginBottom = 12;

        var pdf = renderer.RenderHtmlAsPdf(@"
            <html><body>
                <h1>Invoice #1042</h1>
                <p>Amount: $5,200.00</p>
            </body></html>");

        pdf.SaveAs("invoice.pdf");
        Console.WriteLine("Saved with IronPDF.");
    }
}

No output directory configuration. No status-code checking. No separate install. One NuGet, one renderer, one SaveAs. See the IronPDF HTML-to-PDF tutorial for URL-based and file-based rendering.

How Do I Map ActivePDF API Calls to IronPDF?

Namespace Mapping

  • APToolkit (or ActivePDF.Toolkit) → IronPdf -- document manipulation
  • APWebGrabber → IronPdf -- HTML-to-PDF conversion
  • N/A (COM properties on Toolkit) → IronPdf.Editing -- watermarks, headers, footers

Core Class Mapping

  • APToolkit / Toolkit → PdfDocument -- PDF loading, manipulation, saving
  • WebGrabber → ChromePdfRenderer -- HTML and URL to PDF conversion
  • DocConverter → no direct equivalent -- 300+ format conversion requires dedicated tools
  • Integer status code returns → exceptions (IronPdfProductException) -- error handling pattern shift

Document Loading Methods

  • Open existing PDF: toolkit.OpenInputFile("file.pdf") → PdfDocument.FromFile("file.pdf")
  • Open with password: toolkit.OpenInputFile("file.pdf"); toolkit.InputPassword = "pass" → PdfDocument.FromFile("file.pdf", "pass")
  • Create from HTML: WebGrabber as a separate component → renderer.RenderHtmlAsPdf(html)
  • Create from URL: WebGrabber with the URL property → renderer.RenderUrlAsPdf("https://...")

Page Operations

  • Get page count: toolkit.NumPages() → pdf.PageCount
  • Copy specific pages: toolkit.CopyForm(input, startPage, endPage) → pdf.CopyPages(start, end)
  • Add pages from another PDF: toolkit.AddPDF(file) or toolkit.MergeFile(file, startPage, endPage) → PdfDocument.Merge(pdf1, pdf2)
  • Delete pages: toolkit.RemovePages(start, end) → pdf.RemovePages(index)

Merge / Split Operations

  • Merge multiple PDFs: loop with toolkit.AddPDF() or toolkit.MergeFile() → PdfDocument.Merge(pdf1, pdf2, pdf3)
  • Split by page range: toolkit.CopyForm() with a page range → pdf.CopyPages(start, end)

Merge PDFs

Before (ActivePDF Toolkit):

using ActivePDF.Toolkit;
using System;

class MergeActivePdf
{
    static void Main()
    {
        var toolkit = new Toolkit();

        int result = toolkit.OpenOutputFile(@"C:\output\merged.pdf");
        if (result != 0) { Console.WriteLine("Output error"); return; }

        toolkit.AddPDF(@"C:\input\part1.pdf");
        toolkit.AddPDF(@"C:\input\part2.pdf");

        toolkit.CloseOutputFile();
        Console.WriteLine("Merged with ActivePDF.");
    }
}

After (IronPDF):

using IronPdf;

class MergeIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var pdf1 = PdfDocument.FromFile("part1.pdf");
        var pdf2 = PdfDocument.FromFile("part2.pdf");

        var merged = PdfDocument.Merge(pdf1, pdf2);
        merged.SaveAs("merged.pdf");
        Console.WriteLine("Merged with IronPDF.");
    }
}

The open-output, merge-file, close-output ceremony reduces to a single static call. See the IronPDF merge documentation for additional options.

Password Protection

Before (ActivePDF Toolkit):

using ActivePDF.Toolkit;
using System;

class PasswordActivePdf
{
    static void Main()
    {
        var toolkit = new Toolkit();
        toolkit.SetLicenseKey("YOUR-LICENSE-KEY");

        int result = toolkit.OpenOutputFile("protected.pdf");
        if (result != 0) return;

        toolkit.OpenInputFile("input.pdf");

        // Property names for owner/user passwords and permission flags vary
        // by Toolkit version -- confirm against your installed reference.
        toolkit.SetMasterPassword("owner123");
        toolkit.SetUserPassword("user456");

        toolkit.AllowPrint = false;
        toolkit.AllowCopy = false;
        toolkit.AllowModify = false;

        toolkit.CopyForm(0, 0); // copy all pages with security applied
        toolkit.CloseOutputFile();
        Console.WriteLine("Protected with ActivePDF.");
    }
}

After (IronPDF):

using IronPdf;

class PasswordIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var pdf = PdfDocument.FromFile("input.pdf");

        pdf.SecuritySettings.OwnerPassword = "owner123";
        pdf.SecuritySettings.UserPassword = "user456";
        pdf.SecuritySettings.AllowUserPrinting = IronPdf.Security.PdfPrintSecurity.NoPrint;
        pdf.SecuritySettings.AllowUserCopyPasteContent = false;
        pdf.SecuritySettings.AllowUserEdits = IronPdf.Security.PdfEditSecurity.NoEdit;

        pdf.SaveAs("protected.pdf");
        Console.WriteLine("Protected with IronPDF.");
    }
}

No open-output/close-output dance. No separate input/output file streams. See IronPDF password protection documentation for the full security API.

What Are the Critical Migration Gotchas?

Status Codes to Exceptions

ActivePDF returns integer status codes from nearly every method (0 = success, non-zero = error). IronPDF throws exceptions on failure. Your error handling needs to shift from if (result != 0) checks to try/catch blocks:

// ActivePDF pattern
int result = toolkit.MergeFile("file.pdf", 0, 0);
if (result != 0) { /* handle error */ }

// IronPDF pattern
try
{
    var pdf = PdfDocument.FromFile("file.pdf");
}
catch (Exception ex)
{
    // handle error
}

Page Indexing Differences

ActivePDF Toolkit uses 1-based page numbering in most operations. IronPDF uses 0-based indexing. Audit every page reference in your code.

Input/Output File Pattern

ActivePDF Toolkit separates "open input" and "open output" into distinct calls. IronPDF loads a document and saves it -- there is no separate output file concept. Any code that relies on the input/output split needs restructuring:

// ActivePDF pattern
toolkit.OpenOutputFile("output.pdf");
toolkit.OpenInputFile("input.pdf");
toolkit.CopyForm(0, 0);
toolkit.CloseOutputFile();

// IronPDF pattern
var pdf = PdfDocument.FromFile("input.pdf");
// ... manipulate ...
pdf.SaveAs("output.pdf");

Unit Conversion

ActivePDF WebGrabber uses inches for page dimensions. IronPDF uses millimeters for margins. The conversion: 1 inch = 25.4 mm. So MarginTop = 0.5f (inches) becomes MarginTop = 12.7 (mm) in IronPDF.

How Do I Optimize Performance After Migration?

Eliminating the Component Hop

With ActivePDF, a typical HTML-to-manipulated-PDF workflow looks like: WebGrabber converts HTML, saves to disk, Toolkit opens the file, manipulates, saves again. That is two disk round-trips and two component initializations.

IronPDF keeps everything in memory:

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(html); // in-memory PdfDocument
pdf.ApplyWatermark("<h1>DRAFT</h1>");     // still in memory
pdf.SecuritySettings.UserPassword = "x";   // still in memory
pdf.SaveAs("final.pdf");                   // one write to disk

Renderer Reuse

ChromePdfRenderer is stateless. Create one and share it:

private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

Disposal

PdfDocument is IDisposable. In high-throughput services, wrap in using:

using var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");

Edge Cases Worth Flagging

DocConverter replacement: If you depend on DocConverter for Office-to-PDF, IronPDF does not natively convert .docx or .xlsx to PDF. Consider pairing IronPDF with a library like IronWord or a dedicated conversion service.

Watched-folder patterns: If Meridian's watched-folder feature is part of your pipeline, replace it with a FileSystemWatcher + IronPDF processing loop. This gives you more control over error handling and retry logic.

Large batch processing: IronPDF supports Parallel.ForEach with a shared renderer instance. ActivePDF Toolkit may have required sequential processing depending on your threading configuration.

Frequently Asked Questions

Can I migrate incrementally or do I have to replace everything at once?

You can migrate incrementally. Start with HTML-to-PDF workflows by replacing WebGrabber with ChromePdfRenderer, then move Toolkit manipulation code to PdfDocument. DocConverter workflows can remain in place while you evaluate dedicated alternatives. The two libraries can coexist in the same solution during transition.

What happens to my ActivePDF status-code error handling?

ActivePDF returns integer status codes from nearly every method (0 = success). IronPDF throws exceptions on failure. You need to convert every if (result != 0) check to a try/catch block. This is a pattern-level change that affects every call site.

Does IronPDF work on Linux and in Docker containers?

Yes. IronPDF runs on Windows, Linux, and macOS. It works in Docker containers, Azure App Service on Linux, and AWS Lambda. ActivePDF components are Windows-only, so this is one of the primary reasons teams migrate.

How do I handle the page indexing difference?

ActivePDF Toolkit uses 1-based page numbering. IronPDF uses 0-based indexing. Audit every page reference in your codebase and subtract 1. This is a subtle bug source -- automated tests for page operations are essential.

What about ActivePDF DocConverter's 300+ format support?

DocConverter has no direct IronPDF equivalent. IronPDF handles HTML, URL, and image-to-PDF conversion. For Office-to-PDF, consider pairing IronPDF with IronWord or a dedicated conversion service. For CAD-to-PDF or other specialized formats, you will need a separate tool.

How does IronPDF licensing compare to ActivePDF's per-server model?

ActivePDF licenses each component separately, often per-server. IronPDF uses per-developer perpetual licensing. For teams scaling horizontally across multiple servers or containers, IronPDF's model avoids the per-server multiplication that makes ActivePDF expensive at scale. Get a trial key at ironpdf.com.

Which Approach Should I Take?

If your codebase is primarily WebGrabber + Toolkit with no DocConverter dependency, migrate everything. IronPDF unifies both components into a single NuGet package, eliminates the MSI installation requirement, and gives you cross-platform deployment. The API surface is smaller and more discoverable.

If you have heavy DocConverter usage for Office, CAD, or other specialized format conversions, take a partial approach. Migrate HTML-to-PDF and PDF manipulation to IronPDF, keep DocConverter for format conversion, and evaluate dedicated alternatives on a separate timeline.

If cross-platform deployment is your primary driver -- Linux containers, Docker, Azure App Service on Linux -- the migration is straightforward. ActivePDF is Windows-only. IronPDF is not. That single fact may be the entire business case.

The biggest win in this migration is consolidation. Going from three separate components to a single NuGet package simplifies your dependency tree, your licensing, and your deployment pipeline. The biggest risk is DocConverter: if you rely on Office-to-PDF conversion for 300+ formats, plan that replacement separately.

Worth knowing even without IronPDF: the open-output, merge-file, close-output pattern that ActivePDF Toolkit uses is fragile -- forget to close and you get corrupted files. Any modern PDF library that uses an in-memory document model eliminates this class of bug.
