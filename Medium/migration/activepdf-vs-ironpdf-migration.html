Migrating from ActivePDF to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Three separate ActivePDF NuGet packages. Three separate DLL references. Three separate licensing conversations with procurement. Your WebGrabber handles HTML conversion, Toolkit handles manipulation, and DocConverter handles everything else -- but none of them talk to each other cleanly, and all of them require Windows Server.

Then the Apryse acquisition happened, and now your support tickets route through a portal that looks nothing like the one your team bookmarked two years ago. The product still works. But the question every architect eventually asks is: do we really need three components for what is, fundamentally, one pipeline?

This guide walks you through consolidating ActivePDF into IronPDF -- a single NuGet package that covers HTML-to-PDF, manipulation, merge, watermark, and security.

Why Should I Migrate Away from ActivePDF?

Answer:

ActivePDF has served enterprise .NET shops for over a decade. Here is why teams re-evaluate:

Modular overhead. You buy and license Toolkit, WebGrabber, and DocConverter separately. Each has its own DLL, its own NuGet package, its own API surface. A single PDF generation workflow might touch all three.

Windows Server lock-in. ActivePDF components require Windows Server. If you are moving to Linux containers, Docker on ARM, or Azure App Service on Linux, ActivePDF does not follow you.

MSI installer dependencies. WebGrabber and DocConverter require MSI installation on the host -- not just a NuGet restore. This breaks immutable infrastructure patterns and complicates CI/CD.

Acquisition uncertainty. Apryse acquired ActivePDF and merged support portals. The product roadmap, pricing, and long-term component strategy may shift toward Apryse's own PDFTron SDK.

IE-era rendering engine. WebGrabber historically offered Native and Internet Explorer rendering engines. Modern HTML5/CSS3 applications need a Chrome-class renderer.

API verbosity. Toolkit uses COM-style patterns inherited from its Classic ASP roots. Method names like SetMasterPassword and CopyForm work, but they are verbose compared to fluent property-based APIs.

Limited cross-platform path. Even though Toolkit's NuGet targets .NET Standard 1.0, the underlying native code and WebGrabber/DocConverter dependencies are Windows-only.

Separate HTML conversion path. If you need both HTML-to-PDF and PDF manipulation, you are calling WebGrabber to generate the PDF, then loading it into Toolkit to manipulate it. That is two separate APIs for one workflow.

DocConverter as a Windows service. DocConverter runs as a watched-folder service or COM automation host. In a microservices architecture, this model feels like a detour.

License complexity. Each ActivePDF component is licensed separately, often per-server. For teams scaling horizontally, the licensing math gets expensive.

How Do I Set Up the Migration Step by Step?

Answer:

You will need .NET 6+ (or .NET Framework 4.6.2+), a trial or licensed IronPDF key, and an inventory of which ActivePDF components your codebase uses.

Find ActivePDF References

# Find all ActivePDF using statements and DLL references
rg -l "APToolkit|APWebGrabber|ActivePDF|APDocConverter" --glob "*.cs"

# Count Toolkit instantiations
rg -c "new APToolkit\(\)|new Toolkit\(\)" --glob "*.cs"

# Find project/config references
rg -l "ActivePDF" --glob "*.csproj" --glob "*.config"

If you don't have rg:

grep -rl "APToolkit\|APWebGrabber\|ActivePDF" --include="*.cs" .

In PowerShell, Get-ChildItem -Recurse -Filter *.cs | Select-String "APToolkit|APWebGrabber|ActivePDF" covers the same ground.

Swap NuGet Packages

# Remove ActivePDF packages
dotnet remove package ActivePDF.Toolkit
dotnet remove package ActivePDF.WebGrabber  # if present as NuGet

# Install IronPDF (replaces both)
dotnet add package IronPdf

If WebGrabber or DocConverter were installed via MSI rather than NuGet, remove the DLL references from your .csproj manually and uninstall the MSI from your build/deploy hosts.

What Are the Critical Migration Gotchas?

Answer:

Status Codes to Exceptions

ActivePDF returns integer status codes from nearly every method (0 = success, non-zero = error). IronPDF throws exceptions on failure. Your error handling needs to shift from if (result != 0) checks to try/catch blocks:

// ActivePDF pattern
int result = toolkit.MergeFile("file.pdf", 0, 0);
if (result != 0) { /* handle error */ }

// IronPDF pattern
try
{
    var pdf = PdfDocument.FromFile("file.pdf");
}
catch (Exception ex)
{
    // handle error
}

Page Indexing Differences

ActivePDF Toolkit uses 1-based page numbering in most operations. IronPDF uses 0-based indexing. Audit every page reference in your code.

Input/Output File Pattern

ActivePDF Toolkit separates "open input" and "open output" into distinct calls. IronPDF loads a document and saves it -- there is no separate output file concept. Any code that relies on the input/output split needs restructuring:

// ActivePDF pattern
toolkit.OpenOutputFile("output.pdf");
toolkit.OpenInputFile("input.pdf");
toolkit.CopyForm(0, 0);
toolkit.CloseOutputFile();

// IronPDF pattern
var pdf = PdfDocument.FromFile("input.pdf");
// ... manipulate ...
pdf.SaveAs("output.pdf");

Unit Conversion

ActivePDF WebGrabber uses inches for page dimensions. IronPDF uses millimeters for margins. The conversion: 1 inch = 25.4 mm. So MarginTop = 0.5f (inches) becomes MarginTop = 12.7 (mm) in IronPDF.

How Do I Optimize Performance After Migration?

Answer:

Eliminating the Component Hop

With ActivePDF, a typical HTML-to-manipulated-PDF workflow looks like: WebGrabber converts HTML, saves to disk, Toolkit opens the file, manipulates, saves again. That is two disk round-trips and two component initializations.

IronPDF keeps everything in memory:

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(html); // in-memory PdfDocument
pdf.ApplyWatermark("<h1>DRAFT</h1>");     // still in memory
pdf.SecuritySettings.UserPassword = "x";   // still in memory
pdf.SaveAs("final.pdf");                   // one write to disk

Renderer Reuse

ChromePdfRenderer is stateless. Create one and share it:

private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

Disposal

PdfDocument is IDisposable. In high-throughput services, wrap in using:

using var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");

Edge Cases Worth Flagging

DocConverter replacement: If you depend on DocConverter for Office-to-PDF, IronPDF does not natively convert .docx or .xlsx to PDF. Consider pairing IronPDF with a library like IronWord or a dedicated conversion service.

Watched-folder patterns: If Meridian's watched-folder feature is part of your pipeline, replace it with a FileSystemWatcher + IronPDF processing loop. This gives you more control over error handling and retry logic.

Large batch processing: IronPDF supports Parallel.ForEach with a shared renderer instance. ActivePDF Toolkit may have required sequential processing depending on your threading configuration.

Frequently Asked Questions

Can I migrate incrementally or do I have to replace everything at once?

Answer: You can migrate incrementally. Start with HTML-to-PDF workflows by replacing WebGrabber with ChromePdfRenderer, then move Toolkit manipulation code to PdfDocument. DocConverter workflows can remain in place while you evaluate dedicated alternatives. The two libraries can coexist in the same solution during transition.

What happens to my ActivePDF status-code error handling?

Answer: ActivePDF returns integer status codes from nearly every method (0 = success). IronPDF throws exceptions on failure. You need to convert every if (result != 0) check to a try/catch block. This is a pattern-level change that affects every call site.

Does IronPDF work on Linux and in Docker containers?

Answer: Yes. IronPDF runs on Windows, Linux, and macOS. It works in Docker containers, Azure App Service on Linux, and AWS Lambda. ActivePDF components are Windows-only, so this is one of the primary reasons teams migrate.

How do I handle the page indexing difference?

Answer: ActivePDF Toolkit uses 1-based page numbering. IronPDF uses 0-based indexing. Audit every page reference in your codebase and subtract 1. This is a subtle bug source -- automated tests for page operations are essential.

What about ActivePDF DocConverter's 300+ format support?

Answer: DocConverter has no direct IronPDF equivalent. IronPDF handles HTML, URL, and image-to-PDF conversion. For Office-to-PDF, consider pairing IronPDF with IronWord or a dedicated conversion service. For CAD-to-PDF or other specialized formats, you will need a separate tool.

How does IronPDF licensing compare to ActivePDF's per-server model?

Answer: ActivePDF licenses each component separately, often per-server. IronPDF uses per-developer perpetual licensing. For teams scaling horizontally across multiple servers or containers, IronPDF's model avoids the per-server multiplication that makes ActivePDF expensive at scale. Get a trial key at ironpdf.com.

Which Approach Should I Take?

Answer:

If your codebase is primarily WebGrabber + Toolkit with no DocConverter dependency, migrate everything. IronPDF unifies both components into a single NuGet package, eliminates the MSI installation requirement, and gives you cross-platform deployment. The API surface is smaller and more discoverable.

If you have heavy DocConverter usage for Office, CAD, or other specialized format conversions, take a partial approach. Migrate HTML-to-PDF and PDF manipulation to IronPDF, keep DocConverter for format conversion, and evaluate dedicated alternatives on a separate timeline.

If cross-platform deployment is your primary driver -- Linux containers, Docker, Azure App Service on Linux -- the migration is straightforward. ActivePDF is Windows-only. IronPDF is not. That single fact may be the entire business case.

The biggest win in this migration is consolidation. Going from three separate components to a single NuGet package simplifies your dependency tree, your licensing, and your deployment pipeline. The biggest risk is DocConverter: if you rely on Office-to-PDF conversion for 300+ formats, plan that replacement separately.

Worth knowing even without IronPDF: the open-output, merge-file, close-output pattern that ActivePDF Toolkit uses is fragile -- forget to close and you get corrupted files. Any modern PDF library that uses an in-memory document model eliminates this class of bug.
