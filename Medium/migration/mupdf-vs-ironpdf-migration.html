Migrating from MuPDF to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate Away from MuPDF?

Setting up MuPDF in a .NET project means dealing with native binaries before you write a line of PDF code. You are pulling in libmupdf, figuring out the right platform-specific binary for your target environment, adding interop wrappers or a third-party NuGet shim, and verifying it loads correctly in both your dev machine and your Docker image. That is setup overhead that has nothing to do with your actual requirement.

Here are nine reasons I see teams looking at alternatives to MuPDF-based .NET integrations:

Native binary management. Platform-specific binaries must be bundled correctly for each target OS and architecture. Docker multi-arch builds become significantly more complex.

No official .NET NuGet package. Community wrapper quality varies. You may be depending on a package with one maintainer and no SLA.

AGPL licensing constraints. MuPDF's open-source license is AGPL. Applications that use it must also be AGPL, unless a commercial license is purchased from Artifex.

HTML-to-PDF not native. MuPDF renders PDFs, it does not convert HTML to PDF. Teams needing HTML-to-PDF add a second tool.

P/Invoke complexity. Native interop via P/Invoke is fragile across .NET versions, OS versions, and architecture changes.

ARM64 deployment. ARM64 native binaries for MuPDF must be sourced and managed separately.

Feature scope. MuPDF is excellent at rendering and parsing, but merge, watermark, and encryption require additional code or libraries.

CI/CD binary distribution. Native binaries in CI/CD pipelines create supply chain and caching complexity.

Maintenance overhead. Tracking security updates for a native C library and its .NET wrapper is a separate maintenance track from your application code.

HTML to PDF — Before (MuPDF with Secondary Tool)

using System;
using System.Diagnostics;
using System.IO;

class HtmlToPdfWithSecondTool
{
    static void Main()
    {
        // MuPDF doesn't do HTML-to-PDF natively
        // Common patterns teams use alongside MuPDF:
        // 1. wkhtmltopdf subprocess
        // 2. Puppeteer/PuppeteerSharp
        // 3. Another library entirely

        // Example: wkhtmltopdf subprocess (common workaround)
        string html = "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>";
        string tempHtml = Path.GetTempFileName() + ".html";
        string tempPdf  = Path.GetTempFileName() + ".pdf";

        File.WriteAllText(tempHtml, html);
        var psi = new ProcessStartInfo("wkhtmltopdf", $"\"{tempHtml}\" \"{tempPdf}\"")
        {
            RedirectStandardError = true,
            UseShellExecute = false
        };
        using var proc = Process.Start(psi)!;
        proc.WaitForExit(30_000);

        if (proc.ExitCode != 0)
            throw new Exception($"wkhtmltopdf failed: {proc.StandardError.ReadToEnd()}");

        byte[] pdfBytes = File.ReadAllBytes(tempPdf);
        File.Delete(tempHtml);
        File.Delete(tempPdf);
        // Use pdfBytes...
        Console.WriteLine("Generated invoice PDF");
    }
}

Watermark — Before (MuPDF with iTextSharp)

using iTextSharp.text;
using iTextSharp.text.pdf;
using System.IO;
using System;

class WatermarkExample
{
    static void Main()
    {
        // MuPDF doesn't watermark natively
        // Teams add iTextSharp or PDFsharp alongside MuPDF for this

        using var reader  = new PdfReader("input.pdf");
        using var fs      = new FileStream("watermarked.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs);

        var font = BaseFont.CreateFont(BaseFont.HELVETICA_BOLD, BaseFont.CP1252, false);
        for (int i = 1; i <= reader.NumberOfPages; i++)
        {
            var cb = stamper.GetOverContent(i);
            cb.BeginText();
            cb.SetFontAndSize(font, 50);
            cb.SetColorFill(new BaseColor(200, 200, 200));
            cb.ShowTextAligned(Element.ALIGN_CENTER, "DRAFT", 300, 400, 45);
            cb.EndText();
        }

        Console.WriteLine("Watermarked: watermarked.pdf");
    }
}

Watermark — After (IronPDF)

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

var pdf = PdfDocument.FromFile("input.pdf");
var stamper = new TextStamper

    Text = "DRAFT",
    FontColor = IronSoftware.Drawing.Color.LightGray,
    FontSize = 50,
    Opacity = 40,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");

Password Protection — Before (MuPDF with iTextSharp)

using iTextSharp.text.pdf;
using System.IO;
using System.Text;

class SecurityExample
{
    static void Main()
    {
        // MuPDF can read encrypted PDFs (verify your wrapper)
        // Writing encrypted PDFs typically requires a secondary library

        byte[] userPass  = Encoding.ASCII.GetBytes("viewonly");
        byte[] ownerPass = Encoding.ASCII.GetBytes("admincontrol");

        using var reader  = new PdfReader("input.pdf");
        using var fs      = new FileStream("secured.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs, '\0', false);
        stamper.SetEncryption(
            userPass, ownerPass,
            PdfWriter.ALLOW_PRINTING,
            PdfWriter.ENCRYPTION_AES_128
        );
    }
}

Password Protection — After (IronPDF)

using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

var pdf = PdfDocument.FromFile("input.pdf");
pdf.SecuritySettings.UserPassword  = "viewonly";
pdf.SecuritySettings.OwnerPassword = "admincontrol";
pdf.SecuritySettings.AllowUserPrinting = PdfPrintSecurity.FullPrintRights;
pdf.SecuritySettings.AllowUserCopyPasteContent = false;
pdf.SaveAs("secured.pdf");
// Guide: https://ironpdf.com/how-to/pdf-permissions-passwords/

What Should I Watch Out for During Migration?

Answer: There are several critical areas to verify. Native binary cleanup, page indexing conventions, AGPL licensing implications, and text extraction feature parity are the ones that catch teams off guard.

Native binary cleanup. When migrating away from MuPDF, audit your deployment artifacts. Remove libmupdf binaries from your Docker images, CI/CD pipelines, and project directories. IronPDF ships everything via NuGet, so manual binary management is eliminated.

Page indexing. MuPDF uses 0-based page indexing internally, but wrapper APIs may vary. IronPDF uses 0-based throughout. Verify your wrapper's convention before migrating page-specific code.

// IronPDF: 0-based throughout
var firstPage = pdf.Pages[0];
var lastPage  = pdf.Pages[pdf.PageCount - 1];

AGPL licensing. If you are migrating in part because of MuPDF's AGPL, verify IronPDF's commercial license covers your use case. This is a business and legal decision.

Text extraction feature parity. MuPDF's text extraction is a primary feature. IronPDF's is secondary. If text extraction is a core function, test IronPDF's output against your representative PDF corpus before committing.

// IronPDF text extraction
var pdf = PdfDocument.FromFile("document.pdf");
string fullText = pdf.ExtractAllText();

// Per-page extraction:
foreach (var page in pdf.Pages)
{
    string pageText = page.Text;
    Console.WriteLine($"Page {page.PageIndex}: {pageText.Length} chars");
}
// Guide: https://ironpdf.com/how-to/extract-text-and-images/

Performance. MuPDF's P/Invoke pattern has native interop overhead per call. IronPDF operates in managed code with its own Chromium process. The characteristics are different. Profile your specific workload. Reuse the renderer instance for batch work:

// Instantiate once for batch HTML-to-PDF work
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

foreach (var html in htmlTemplates)
{
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"output_{Guid.NewGuid()}.pdf");
}

Edge cases. ARM64: IronPDF ships ARM64 binaries via NuGet, so no manual binary sourcing is required. Alpine Linux: verify IronPDF's Alpine compatibility in current docs, since Chromium on Alpine has historically required specific dependencies. Memory: Chromium-based rendering has a different memory profile than MuPDF's C renderer. Profile under load before sizing.

What Questions Come Up Most During This Migration?

Does IronPDF replace MuPDF's PDF-to-image rendering?

Not directly. MuPDF's primary strength is rendering PDF pages to images. IronPDF's primary strength is HTML-to-PDF generation and PDF manipulation. If PDF-to-image rendering is your core use case, MuPDF may still be the better fit. If you need both, evaluate whether IronPDF's rasterization capabilities meet your quality requirements.

Will IronPDF eliminate my native binary management burden?

Yes. IronPDF distributes everything through NuGet, including its Chromium dependency. You no longer need to bundle platform-specific libmupdf binaries, manage per-architecture builds, or maintain Dockerfile COPY commands for native libraries. The tradeoff is that IronPDF's Chromium distribution is larger than MuPDF's native binaries, so container image size may increase.

What about the AGPL licensing concern?

MuPDF's open-source license is AGPL, which means applications that use it must also be AGPL unless a commercial license is purchased from Artifex. IronPDF uses a commercial license model. Evaluate both license terms against your project requirements. This is a legal decision, not a technical one.

How do I handle parallel rendering with IronPDF?

Use one renderer per thread. Do not share a single ChromePdfRenderer instance across concurrent calls.

// One renderer per thread — don't share across concurrent calls
await Task.WhenAll(htmlBatch.Select(async html =>
{
    var renderer = new ChromePdfRenderer();
    using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    await pdf.SaveAsAsync($"{Guid.NewGuid()}.pdf");
}));

What Is the Decision Framework for This Migration?

Answer: The install footprint complaint that opens most MuPDF-to-managed-library migrations is valid. Native binary distribution across multiple architectures is real operational overhead. The tradeoff is that IronPDF's Chromium distribution is larger than MuPDF's native binaries, so container image size may increase. Measure both.

If your primary use case is HTML-to-PDF, the migration is straightforward because MuPDF never did this natively. You are replacing a workaround with a purpose-built tool. If your primary use case is PDF-to-image rendering, MuPDF is well-suited and switching may not be the right call.

For teams where the pain point is native binary management, AGPL licensing, or needing merge, watermark, and encryption without bolting on additional libraries, IronPDF consolidates those capabilities into a single managed NuGet package. Run the effort table against your actual feature inventory, test text extraction against your PDF corpus, and profile rendering performance under your expected load before committing.
