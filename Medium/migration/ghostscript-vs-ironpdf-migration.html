Migrating from Ghostscript to IronPDF in C#: The Complete Guide

Ghostscript is a powerful PostScript/PDF interpreter, but it's not a .NET library. It's a native C binary that you typically drive from .NET via the third-party Ghostscript.NET wrapper or by shelling out to gswin64c.exe / gs. If you've been running it in production .NET applications, you know the drill: locating gsdll64.dll, passing string switches to GhostscriptProcessor, parsing stderr for error detection, and installing the native binary in every container and CI runner.

I built IronPDF as a managed .NET library that eliminates that entire layer of native-binary complexity. The migration isn't just swapping one PDF tool for another — it's replacing a native binary plus wrapper with a single NuGet package. In this guide, I'll walk you through why teams make this move, what the effort looks like for each feature, and show you complete before/after code for the operations the .cs samples in this repo cover.

Why Should I Migrate from Ghostscript to IronPDF?

Teams using Ghostscript in .NET applications encounter a familiar set of issues depending on their use case and deployment environment.

AGPL license obligations — Ghostscript's AGPL terms require source disclosure for software that distributes or networks it, unless you buy a commercial license from Artifex. This alone drives migration for commercial software.

No first-party .NET library — Ghostscript is a native C tool. The common .NET path is the Ghostscript.NET wrapper, which still requires the native gsdll64.dll / gsdll32.dll on disk. Path resolution and DLL bitness mismatches are error-prone across environments.

Native binary dependency — Ghostscript must be installed on every machine, container, and CI runner. Version mismatches cause subtle failures, and 32-bit vs 64-bit DLL selection adds conditional logic at startup.

No HTML-to-PDF capability — Ghostscript doesn't render HTML. If you've added a web-based report layer, you're already using a second tool such as wkhtmltopdf.

String-switch API — Operations are driven by cryptic command-line switches like -dNOPAUSE -dBATCH -sDEVICE=pdfwrite. Errors surface through stderr text rather than typed exceptions, and concurrency requires careful per-process isolation.

Container overhead — apt-get install ghostscript in Dockerfiles pulls native dependencies. Image sizes grow and dependency audits become more complex.

How Do the Two Libraries Compare Feature by Feature?

  • Focus. Ghostscript is a PostScript/PDF interpreter built around rasterization and PS-to-PDF conversion. IronPDF is an HTML-to-PDF renderer plus document-manipulation library.
  • Licensing. Ghostscript ships under AGPL, with a commercial license from Artifex for closed-source distribution. IronPDF is commercial with clear per-product terms.
  • API style. Ghostscript uses string switches passed to GhostscriptProcessor.Process(string[]) or to a gs subprocess. IronPDF is a typed .NET library with IntelliSense.
  • Learning curve. Ghostscript expects familiarity with the PostScript model and dozens of -d* / -s* switches. IronPDF is a standard .NET object API.
  • HTML rendering. Ghostscript does not render HTML at all. IronPDF treats HTML-to-PDF as the primary capability via embedded Chromium.
  • Page indexing. Ghostscript CLI uses 1-based pages (-dFirstPage=1); IronPDF is 0-based. Every page-number calculation needs adjustment.
  • Thread safety. Ghostscript isolates work via separate processes or careful wrapper reuse. IronPDF is thread-safe with one ChromePdfRenderer per task.
  • Namespace. Ghostscript.NET (third-party wrapper) or System.Diagnostics.Process with the gs binary, versus the single first-party IronPdf namespace.

The key takeaway: Ghostscript is a PostScript interpreter you call from .NET. IronPDF is a .NET library you call from .NET. The integration model is fundamentally different.

What Is the Migration Effort for Each Feature?

  • HTML to PDF — Low. Ghostscript cannot do this at all; IronPDF gains it natively via Chromium.
  • PDF to image rasterization — Medium-High. This is Ghostscript's core strength. Benchmark IronPDF's ToBitmap / RasterizeToImageFiles output against Ghostscript on your real documents before committing.
  • PDF merge — Low. Ghostscript merges via -sDEVICE=pdfwrite with multiple inputs. IronPDF replaces this with PdfDocument.Merge(IEnumerable<PdfDocument>).
  • PDF split / page extraction — Medium. Ghostscript uses -dFirstPage / -dLastPage. IronPDF uses CopyPages with 0-based indices.
  • PDF compression — Medium. Ghostscript's -dPDFSETTINGS=/screen|/ebook|/printer presets map to pdf.CompressImages(quality) on a 0–100 scale.
  • PostScript to PDF — N/A. Ghostscript-only. There is no IronPDF equivalent for .ps input. Keep Ghostscript for that step or restructure upstream.
  • Font embedding — Medium. Ghostscript controls fonts through pdfwrite options; IronPDF embeds via Chromium's font handling, so output may differ.
  • Password protection — Low. Ghostscript uses -sOwnerPassword / -sUserPassword plus a permissions bitmask. IronPDF exposes typed SecuritySettings properties.
  • PDF/A conversion — Low. Ghostscript needs -dPDFA=1 plus a PDFA_def.ps file. IronPDF sets renderer.RenderingOptions.PdfA = true.
  • Batch processing — Medium. Subprocess-per-file batching becomes Parallel.ForEachAsync with managed disposal.

Be honest with yourself about PostScript processing. If your pipeline processes .ps input files, IronPDF cannot replace that. You have two options: retain Ghostscript for that single step, or restructure the input pipeline to start from HTML/PDF.

How Do I Set Up My Project for IronPDF?

Remove Ghostscript and Install IronPDF

// Remove the Ghostscript.NET wrapper
dotnet remove package Ghostscript.NET

// Add IronPDF
dotnet add package IronPdf

dotnet restore

The Ghostscript native binary installation is separate from any NuGet wrapper. If you remove all Ghostscript usages, you can also remove the native binary from container images, Dockerfiles, and CI provisioning scripts.

Configure Your License Key

Before (Ghostscript — no .NET license initialization):

// Ghostscript has no .NET license initialization step.
// Licensing is governed by AGPL or by an Artifex commercial license,
// not by code. The .NET-side setup is the wrapper's DLL location.
using Ghostscript.NET;
using Ghostscript.NET.Processor;

// Typical Ghostscript.NET wrapper setup:
GhostscriptVersionInfo gvi = new GhostscriptVersionInfo("gsdll64.dll");
// On 32-bit processes you would point at gsdll32.dll instead.

After (IronPDF):

using IronPdf;

// See: https://ironpdf.com/how-to/license-keys/
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
// One line. No DLL path resolution. No 32/64-bit branching.

One line replaces a GhostscriptVersionInfo lookup plus the deployment work of shipping gsdll64.dll. That's the difference between a native binary and a managed library. You can get your IronPDF license key here.

How Do I Migrate HTML-to-PDF Generation?

This is the most common reason teams migrate. Ghostscript does not render HTML to PDF. If you have an HTML-to-PDF requirement with Ghostscript in your stack, you're likely using a second tool — wkhtmltopdf, headless Chrome, or some intermediate PostScript conversion.

Ghostscript Approach (Not Applicable)

using Ghostscript.NET;
using Ghostscript.NET.Processor;

// Ghostscript cannot render HTML directly. The typical workaround is
// to first convert HTML to PostScript with another tool (wkhtmltopdf,
// headless Chrome, or a print driver), then feed the .ps file through
// pdfwrite. That setup looks like this:

string psFile    = "temp.ps";   // produced by some other HTML-to-PS step
string outputPdf = "output.pdf";

GhostscriptProcessor processor = new GhostscriptProcessor();

var switches = new List<string>
{
    "-dNOPAUSE",
    "-dBATCH",
    "-dSAFER",
    "-sDEVICE=pdfwrite",
    $"-sOutputFile={outputPdf}",
    psFile
};

processor.Process(switches.ToArray());

IronPDF HTML-to-PDF Rendering

using IronPdf;

// See: https://ironpdf.com/how-to/html-string-to-pdf/
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

string html = "<html><body><h1>Report</h1><p>Generated via IronPDF.</p></body></html>";
using var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");
Console.WriteLine($"Created {pdf.PageCount} page(s)");

With IronPDF, you get full Chromium rendering. Your HTML templates look exactly like they do in a browser. Learn more about HTML string to PDF conversion.

How Do I Migrate PDF Merging?

PDF merging is one of the most common Ghostscript operations in .NET. The Ghostscript approach uses -sDEVICE=pdfwrite with multiple input files passed as switches to GhostscriptProcessor. It works, but it's a lot of switch-list ceremony for what should be a one-line call.

Ghostscript Merge via Ghostscript.NET

using Ghostscript.NET;
using Ghostscript.NET.Processor;
using System.Collections.Generic;

class MergeExample
{
    static void Main()
    {
        string outputPdf = "merged.pdf";
        string[] inputFiles = { "file1.pdf", "file2.pdf", "file3.pdf" };

        GhostscriptProcessor processor = new GhostscriptProcessor();

        var switches = new List<string>
        {
            "-dNOPAUSE",
            "-dBATCH",
            "-dSAFER",
            "-sDEVICE=pdfwrite",
            $"-sOutputFile={outputPdf}"
        };

        switches.AddRange(inputFiles);

        processor.Process(switches.ToArray());
    }
}

IronPDF Merge

using IronPdf;

// See: https://ironpdf.com/how-to/merge-or-split-pdfs/
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var docs = new[] { "file1.pdf", "file2.pdf", "file3.pdf" }
              .Select(PdfDocument.FromFile)
              .ToList();

using var merged = PdfDocument.Merge(docs);
merged.SaveAs("merged.pdf");
foreach (var d in docs) d.Dispose();
Console.WriteLine($"Merged: {merged.PageCount} pages");

Three lines of actual logic replace the switch-list construction. No DLL path resolution, no argument string assembly, no stderr parsing. See the full merge and split documentation.

How Do I Migrate Watermarking?

Ghostscript watermarking requires injecting PostScript content via a stamp/overlay file. It involves writing a temporary .ps file with font and positioning commands, then merging it with the target PDF through pdfwrite. IronPDF replaces all of that with a TextStamper object.

Ghostscript Watermark via PostScript Injection

using Ghostscript.NET;
using Ghostscript.NET.Processor;
using System.Collections.Generic;
using System.IO;

class WatermarkExample
{
    static void Main()
    {
        // Step 1: write a PostScript watermark overlay
        string watermarkPs = Path.GetTempFileName() + ".ps";
        File.WriteAllText(watermarkPs, @"
            /watermark {
                gsave
                0.3 setgray
                /Helvetica-Bold 40 selectfont
                200 400 moveto
                45 rotate
                (CONFIDENTIAL) show
                grestore
            } def
        ");

        // Step 2: feed overlay + source PDF through pdfwrite
        // (simplified illustration — real overlays use a control script)
        GhostscriptProcessor processor = new GhostscriptProcessor();

        var switches = new List<string>
        {
            "-dNOPAUSE",
            "-dBATCH",
            "-dSAFER",
            "-sDEVICE=pdfwrite",
            "-sOutputFile=watermarked.pdf",
            watermarkPs,
            "input.pdf"
        };

        processor.Process(switches.ToArray());
        File.Delete(watermarkPs);
    }
}

IronPDF Watermark

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

using var pdf = PdfDocument.FromFile("input.pdf");

var stamp = new TextStamper
{
    Text                = "CONFIDENTIAL",
    FontSize            = 40,
    Opacity             = 30,            // integer 0–100
    Rotation            = -45,
    VerticalAlignment   = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};

pdf.ApplyStamp(stamp);
pdf.SaveAs("watermarked.pdf");

No temporary PostScript files. No PostScript language knowledge. A TextStamper object with readable properties replaces the overlay file and the switch-list invocation.

How Do I Migrate Password Protection?

Ghostscript handles encryption via command-line arguments: -sOwnerPassword, -sUserPassword, -dEncryptionR, and a permission bitmask that requires looking up the PDF spec. IronPDF replaces all of that with typed properties.

Ghostscript Password Protection

using Ghostscript.NET;
using Ghostscript.NET.Processor;
using System.Collections.Generic;

class SecurityExample
{
    static void Main()
    {
        GhostscriptProcessor processor = new GhostscriptProcessor();

        // Ghostscript password protection via -sOwnerPassword / -sUserPassword.
        // -dPermissions is a signed bitmask — bits map to PDF spec permission flags;
        // the exact value depends on which permissions you want to deny.
        var switches = new List<string>
        {
            "-dNOPAUSE",
            "-dBATCH",
            "-dSAFER",
            "-sDEVICE=pdfwrite",
            "-dEncryptionR=3",
            "-dKeyLength=128",
            "-sOwnerPassword=owner456",
            "-sUserPassword=user123",
            "-dPermissions=-3904",
            "-sOutputFile=protected.pdf",
            "input.pdf"
        };

        processor.Process(switches.ToArray());
    }
}

IronPDF Password Protection

using IronPdf;

// See: https://ironpdf.com/how-to/pdf-permissions-passwords/
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

using var pdf = PdfDocument.FromFile("input.pdf");

pdf.SecuritySettings.UserPassword  = "user123";
pdf.SecuritySettings.OwnerPassword = "owner456";
pdf.SecuritySettings.AllowUserPrinting         = PdfPrintSecurity.FullPrintRights;
pdf.SecuritySettings.AllowUserCopyPasteContent = false;

pdf.SaveAs("protected.pdf");

No permission bitmask. No encryption revision numbers. Typed boolean properties tell you exactly what each setting does. See the full permissions and passwords documentation.

What Are the Critical Migration Gotchas?

There are three things that will catch you if you don't plan for them.

Page Indexing: 1-Based to 0-Based

Ghostscript CLI uses 1-based page numbering (-dFirstPage=1). IronPDF uses 0-based indexing. Every page number calculation in your codebase needs adjustment.

// Ghostscript: -dFirstPage=1 -dLastPage=3 extracts pages 1, 2, 3
// IronPDF equivalent:
int[] pageIndices = { 0, 1, 2 }; // 0-based: first three pages
using var extracted = pdf.CopyPages(pageIndices);

Error Handling: Stderr Parsing to Exceptions

Ghostscript surfaces failures through stderr text and (for the gs binary) exit codes. Ghostscript.NET wraps these into GhostscriptException. IronPDF raises typed IronPdf.Exceptions.PdfException. Remove stderr-parsing logic and exit-code checks in favor of a normal try/catch.

// Replace stderr or exit-code checks:
// if (proc.ExitCode != 0) throw new Exception(...)
// or:  catch (GhostscriptException gex) { ParseStderr(gex.Message); }

// With structured exception handling:
try
{
    using var pdf = PdfDocument.FromFile("input.pdf");
    // operations
}
catch (IronPdf.Exceptions.PdfException ex)
{
    Console.Error.WriteLine(ex.Message);
}

No PostScript Support

IronPDF cannot interpret PostScript (.ps) files. If your pipeline processes .ps input, you have two options: convert PostScript to PDF first (retain Ghostscript for that step only), or restructure the input pipeline to start from HTML/PDF.

How Does Parallel Processing Compare?

Ghostscript parallel processing requires managing multiple OS processes. IronPDF parallel processing uses standard .NET Task/Parallel patterns with one ChromePdfRenderer per task.

var results = await Task.WhenAll(
    htmlDocuments.Select(async html =>
    {
        var r = new ChromePdfRenderer();
        using var pdf = await r.RenderHtmlAsPdfAsync(html);
        return pdf.BinaryData;
    })
);

Frequently Asked Questions

Can IronPDF replace Ghostscript for PostScript processing?

No. IronPDF cannot interpret PostScript (.ps) files. If your pipeline requires PostScript interpretation, retain Ghostscript for that specific step. You can migrate all other PDF operations — merging, splitting, watermarking, password protection, HTML-to-PDF — to IronPDF while keeping a minimal Ghostscript deployment for PostScript only.

Alternatively, restructure your input pipeline so PostScript files are converted to PDF upstream, then IronPDF handles everything from that point forward.

How do I handle the page indexing difference between Ghostscript and IronPDF?

Ghostscript CLI uses 1-based page numbering (-dFirstPage=1). IronPDF uses 0-based indexing. Audit every page number calculation in your codebase and subtract 1. For example, Ghostscript's -dFirstPage=1 -dLastPage=3 becomes pdf.CopyPages(new[] { 0, 1, 2 }) in IronPDF.

This is the most common source of off-by-one bugs during migration. Add explicit unit tests for page extraction to catch these early.

What happens to my Ghostscript error handling code?

Ghostscript writes errors to stderr; the Ghostscript.NET wrapper raises GhostscriptException wrapping that text. IronPDF raises structured IronPdf.Exceptions.PdfException. Remove any stderr-parsing or exit-code checks and replace them with standard try/catch blocks.

The exception messages from IronPDF are descriptive — no need to parse text output to determine what went wrong.

Is IronPDF suitable for PDF-to-image rasterization?

IronPDF supports rasterization, but this is Ghostscript's core strength. If rasterization quality and performance are critical to your workflow, benchmark IronPDF's output against Ghostscript's before committing to the migration for that specific feature.

The effort rating for this migration path is Medium-High. Test with your actual documents and verify the output meets your requirements.

Do I need to change my Docker configuration?

Yes. Remove apt-get install ghostscript from your Dockerfiles. IronPDF handles its Chromium dependencies automatically. The native Ghostscript binary, any wrapper configurations, and CI runner setup scripts that install Ghostscript can all be removed once you've verified the migration is complete.

Measure your container image size before and after — Ghostscript and IronPDF have different dependency footprints.

What licensing considerations should I be aware of?

Ghostscript ships under AGPL, which requires source disclosure for software that distributes or networks it unless you buy a commercial license from Artifex Software. This is a significant concern for closed-source distribution. IronPDF uses a straightforward commercial license — one line of code sets your key and you're done.

Confirm your current Ghostscript licensing situation with your legal team before and during the migration. If AGPL compliance is driving the move, document the removal of all Ghostscript references for your compliance audit.

What Is the Bottom Line on This Migration?

The Ghostscript migration is often the most impactful from an architecture perspective, because Ghostscript isn't a .NET library — it's a native binary driven through a wrapper or subprocess. Replacing it with a managed NuGet package removes an entire category of deployment complexity: gsdll64.dll path resolution, 32/64-bit branching, stderr parsing, and native package management in containers.

The capability gaps are real. PostScript interpretation and the highest-fidelity rasterization workflows don't have direct IronPDF equivalents. For those, retaining a minimal Ghostscript deployment or restructuring the pipeline upstream is the honest answer.

For everything else — HTML-to-PDF rendering, PDF merging, watermarking, password protection, and page manipulation — the migration path is well-defined and the code reduction is dramatic. Audit your Ghostscript switch lists, identify which argument patterns map to IronPDF operations, and migrate feature by feature.
