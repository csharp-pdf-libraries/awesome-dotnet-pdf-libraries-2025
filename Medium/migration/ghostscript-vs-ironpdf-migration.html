Migrating from Ghostscript to IronPDF in C#: The Complete Guide

Ghostscript is a powerful PostScript/PDF interpreter, but it's not a .NET library. It's a native C binary that you invoke via subprocess, P/Invoke, or a third-party wrapper like Ghostscript.NET. If you've been running it in production .NET applications, you know the drill: platform-specific binary paths, argument string construction, stderr parsing for error detection, and native package management in every container and CI runner.

I built IronPDF as a managed .NET library that eliminates that entire layer of subprocess complexity. The migration isn't just swapping one PDF tool for another — it's replacing a native binary dependency with a NuGet package. In this guide, I'll walk you through why teams make this move, what the effort looks like for each feature, and show you complete before/after code for every common operation.

Why Should I Migrate from Ghostscript to IronPDF?

Teams using Ghostscript in .NET applications encounter a familiar set of issues depending on their use case and deployment environment.

GPL/AGPL license obligations — Ghostscript under GPL requires derivative works to also be GPL unless a commercial license is purchased. This alone drives migration for commercial software.

No first-party .NET library — Ghostscript requires subprocess invocation or third-party wrappers. Both add fragility. Path resolution, process management, and argument escaping are error-prone across environments.

Native binary dependency — Ghostscript must be installed on every machine, container, and CI runner. Version mismatches cause subtle failures. gswin64c.exe vs. gs adds conditional logic to process invocations.

No HTML-to-PDF capability — Ghostscript doesn't render HTML. If you've added a web-based report layer, you're already using a second tool.

Error handling and thread safety — Ghostscript returns exit codes and stderr output. Structured error handling requires parsing text. Running multiple Ghostscript processes concurrently requires process management that a managed library handles natively.

Container overhead — apt-get install ghostscript in Dockerfiles pulls native dependencies. Image sizes grow and dependency audits become more complex.

How Do I Set Up My Project for IronPDF?

Remove Ghostscript and Install IronPDF

// Remove Ghostscript.NET wrapper (if used — verify package name)
dotnet remove package Ghostscript.NET

// Add IronPDF
dotnet add package IronPdf

dotnet restore

The Ghostscript native binary installation is separate from any NuGet wrapper. If you remove all Ghostscript usages, you can also remove the native binary from container images, Dockerfiles, and CI provisioning scripts.

Configure Your License Key

Before (Ghostscript — no .NET license initialization):

// Ghostscript doesn't have a .NET license initialization step.
// Licensing is handled at the OS/binary level.
// GPL usage is governed by Ghostscript's AGPL license terms — verify legal compliance.
using System.Diagnostics;

// Typical Ghostscript process setup:
var psi = new ProcessStartInfo
{
    FileName  = RuntimeInformation.IsOSPlatform(OSPlatform.Windows)
                    ? "gswin64c.exe"
                    : "gs",
    Arguments = "-dNOPAUSE -dBATCH ...", // varies per operation
    RedirectStandardOutput = true,
    RedirectStandardError  = true,
    UseShellExecute = false
};

After (IronPDF):

using IronPdf;

// See: https://ironpdf.com/how-to/license-keys/

// One line. No binary path resolution. No platform branching.

One line replaces an entire ProcessStartInfo block with platform branching. That's the difference between a native binary and a managed library. You can get your IronPDF license key here.

How Do I Migrate PDF Merging?

PDF merging is one of the most common Ghostscript operations in .NET. The Ghostscript approach uses -sDEVICE=pdfwrite with multiple input files. It works, but it's a lot of subprocess ceremony for what should be a simple operation.

Ghostscript Merge via Subprocess

using System.Diagnostics;
using System.Runtime.InteropServices;

class MergeExample
{
    static void Main(string[] args)
    {
        string[] inputFiles = { "file1.pdf", "file2.pdf", "file3.pdf" };
        string outputFile   = "merged.pdf";

        string gs = RuntimeInformation.IsOSPlatform(OSPlatform.Windows)
                        ? "gswin64c.exe" : "gs";

        // Escape and join input paths
        string inputs = string.Join(" ", inputFiles.Select(f => $"\"{f}\""));

        var psi = new ProcessStartInfo
        {
            FileName  = gs,
            Arguments = $"-dNOPAUSE -dBATCH -sDEVICE=pdfwrite " +
                        $"-dCompatibilityLevel=1.7 " +
                        $"-sOutputFile=\"{outputFile}\" {inputs}",
            RedirectStandardOutput = true,
            RedirectStandardError  = true,
            UseShellExecute        = false
        };

        using var proc = Process.Start(psi)!;
        string stderr = proc.StandardError.ReadToEnd();
        proc.WaitForExit();

        if (proc.ExitCode != 0)
            throw new Exception($"Ghostscript merge failed (exit {proc.ExitCode}): {stderr}");

        Console.WriteLine($"Merged to {outputFile}");
    }
}

IronPDF Merge

using IronPdf;

// See: https://ironpdf.com/how-to/merge-or-split-pdfs/
IronPdf.License.LicenseKey = "YOUR_KEY";

var docs = new[] { "file1.pdf", "file2.pdf", "file3.pdf" }
              .Select(PdfDocument.FromFile)
              .ToList();

using var merged = PdfDocument.Merge(docs);
merged.SaveAs("merged.pdf");
foreach (var d in docs) d.Dispose();
Console.WriteLine($"Merged: {merged.PageCount} pages");

Three lines of actual logic replace 25+ lines of subprocess management. No binary path resolution, no argument string construction, no exit code parsing. See the full merge and split documentation.

How Do I Migrate Watermarking?

Ghostscript watermarking requires injecting PostScript content via a stamp/overlay file. It's a non-trivial approach that involves creating a temporary PostScript file, defining font and positioning commands, then merging it with the target PDF. IronPDF replaces all of that with a TextStamper object.

Ghostscript Watermark via PostScript Injection

using System.Diagnostics;
using System.IO;

class WatermarkExample
{
    static void Main(string[] args)
    {
        // Ghostscript watermarking requires injecting PostScript content
        // via a stamp/overlay file — a non-trivial approach

        // Step 1: Create a PostScript watermark file
        string watermarkPs = Path.GetTempFileName() + ".ps";
        File.WriteAllText(watermarkPs, @"
            /watermark {
                gsave
                0.3 setgray
                /Helvetica-Bold 40 selectfont
                200 400 moveto
                45 rotate
                (CONFIDENTIAL) show
                grestore
            } def
        ");

        // Step 2: Apply via Ghostscript — exact args vary
        // This is a simplified illustration; real implementation is more involved
        string gs = System.Runtime.InteropServices.RuntimeInformation
                         .IsOSPlatform(System.Runtime.InteropServices.OSPlatform.Windows)
                         ? "gswin64c.exe" : "gs";

        var psi = new ProcessStartInfo
        {
            FileName  = gs,
            Arguments = $"-dNOPAUSE -dBATCH -sDEVICE=pdfwrite " +
                        $"-sOutputFile=\"watermarked.pdf\" " +
                        $"\"{watermarkPs}\" \"input.pdf\"",
            UseShellExecute = false
        };
        using var proc = Process.Start(psi)!;
        proc.WaitForExit();
        File.Delete(watermarkPs);
    }
}

IronPDF Watermark

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR_KEY";

using var pdf = PdfDocument.FromFile("input.pdf");

var stamp = new TextStamper
{
    Text                = "CONFIDENTIAL",
    FontSize            = 40,
    Opacity             = 0.3,
    Rotation            = -45,
    VerticalAlignment   = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};

pdf.ApplyStamp(stamp);
pdf.SaveAs("watermarked.pdf");

No temporary PostScript files. No PostScript language knowledge. A TextStamper object with readable properties replaces an entire PostScript code block and subprocess invocation.

How Do I Migrate Password Protection?

Ghostscript handles encryption via command-line arguments: -sOwnerPassword, -sUserPassword, -dEncryptionR, and a permission bitmask that requires looking up the PDF spec. IronPDF replaces all of that with typed properties.

Ghostscript Password Protection

using System.Diagnostics;
using System.Runtime.InteropServices;

class SecurityExample
{
    static void Main(string[] args)
    {
        string gs = RuntimeInformation.IsOSPlatform(OSPlatform.Windows)
                        ? "gswin64c.exe" : "gs";

        // Ghostscript password protection via -sOwnerPassword / -sUserPassword
        var psi = new ProcessStartInfo
        {
            FileName  = gs,
            Arguments = "-dNOPAUSE -dBATCH -sDEVICE=pdfwrite " +
                        "-dEncryptionR=3 -dKeyLength=128 " +
                        "-sOwnerPassword=owner456 " +
                        "-sUserPassword=user123 " +
                        "-dPermissions=-3904 " +      // permission bitmask — verify values
                        "-sOutputFile=\"protected.pdf\" \"input.pdf\"",
            RedirectStandardError = true,
            UseShellExecute       = false
        };

        using var proc = Process.Start(psi)!;
        proc.WaitForExit();

        if (proc.ExitCode != 0)
            throw new Exception($"Ghostscript encryption failed: {proc.StandardError.ReadToEnd()}");

        Console.WriteLine("Protected PDF created");
    }
}

IronPDF Password Protection

using IronPdf;

// See: https://ironpdf.com/how-to/pdf-permissions-passwords/
IronPdf.License.LicenseKey = "YOUR_KEY";

using var pdf = PdfDocument.FromFile("input.pdf");

pdf.SecuritySettings.UserPassword  = "user123";
pdf.SecuritySettings.OwnerPassword = "owner456";
pdf.SecuritySettings.AllowUserPrinting         = PdfPrintSecurity.FullPrintRights;
pdf.SecuritySettings.AllowUserCopyPasteContent = false;

pdf.SaveAs("protected.pdf");

No permission bitmask. No encryption revision numbers. Typed boolean properties tell you exactly what each setting does. See the full permissions and passwords documentation.

What Are the Critical Migration Gotchas?

There are three things that will catch you if you don't plan for them.

Page Indexing: 1-Based to 0-Based

Ghostscript CLI uses 1-based page numbering (-dFirstPage=1). IronPDF uses 0-based indexing. Every page number calculation in your codebase needs adjustment.

// Ghostscript: -dFirstPage=1 -dLastPage=3 extracts pages 1, 2, 3
// IronPDF equivalent:
int[] pageIndices = { 0, 1, 2 }; // 0-based: first three pages
using var extracted = pdf.CopyPages(pageIndices);

Error Handling: Exit Codes to Exceptions

Ghostscript returns exit codes. IronPDF raises exceptions. Remove all ExitCode != 0 checks and replace with try/catch.

// Replace exit code checks:
// if (proc.ExitCode != 0) throw new Exception(...)

// With exception handling:
try
{
    using var pdf = PdfDocument.FromFile("input.pdf");
    // operations
}
catch (IronPdf.Exceptions.PdfException ex)
{
    // Structured exception with message, no stderr parsing
    Console.Error.WriteLine(ex.Message);
}

No PostScript Support

IronPDF cannot interpret PostScript (.ps) files. If your pipeline processes .ps input, you have two options: convert PostScript to PDF first (retain Ghostscript for that step only), or restructure the input pipeline to start from HTML/PDF.

How Does Parallel Processing Compare?

Ghostscript parallel processing requires managing multiple OS processes. IronPDF parallel processing uses standard .NET Task/Parallel patterns with one ChromePdfRenderer per task.

var results = await Task.WhenAll(
    htmlDocuments.Select(async html =>
    {
        var r = new ChromePdfRenderer();
        using var pdf = await r.RenderHtmlAsPdfAsync(html);
        return pdf.BinaryData;
    })
);

Frequently Asked Questions

Can IronPDF replace Ghostscript for PostScript processing?

Answer: No. IronPDF cannot interpret PostScript (.ps) files. If your pipeline requires PostScript interpretation, retain Ghostscript for that specific step. You can migrate all other PDF operations — merging, splitting, watermarking, password protection, HTML-to-PDF — to IronPDF while keeping a minimal Ghostscript deployment for PostScript only.

Alternatively, restructure your input pipeline so PostScript files are converted to PDF upstream, then IronPDF handles everything from that point forward.

How do I handle the page indexing difference between Ghostscript and IronPDF?

Answer: Ghostscript CLI uses 1-based page numbering (-dFirstPage=1). IronPDF uses 0-based indexing. Audit every page number calculation in your codebase and subtract 1. For example, Ghostscript's -dFirstPage=1 -dLastPage=3 becomes pdf.CopyPages(new[] { 0, 1, 2 }) in IronPDF.

This is the most common source of off-by-one bugs during migration. Add explicit unit tests for page extraction to catch these early.

What happens to my Ghostscript error handling code?

Answer: Ghostscript returns exit codes and writes errors to stderr. IronPDF raises structured IronPdf.Exceptions.PdfException exceptions. Remove all proc.ExitCode != 0 checks and stderr parsing. Replace them with standard try/catch blocks.

The exception messages from IronPDF are descriptive — no need to parse text output to determine what went wrong.

Is IronPDF suitable for PDF-to-image rasterization?

Answer: IronPDF supports rasterization, but this is Ghostscript's core strength. If rasterization quality and performance are critical to your workflow, benchmark IronPDF's output against Ghostscript's before committing to the migration for that specific feature.

The effort rating for this migration path is Medium-High. Test with your actual documents and verify the output meets your requirements.

Do I need to change my Docker configuration?

Answer: Yes. Remove apt-get install ghostscript from your Dockerfiles. IronPDF handles its Chromium dependencies automatically. The native Ghostscript binary, any wrapper configurations, and CI runner setup scripts that install Ghostscript can all be removed once you've verified the migration is complete.

Measure your container image size before and after — Ghostscript and IronPDF have different dependency footprints.

What licensing considerations should I be aware of?

Answer: Ghostscript under GPL/AGPL requires derivative works to also be GPL unless a commercial license is purchased from Artifex Software. This is a significant concern for commercial software distribution. IronPDF uses a straightforward commercial license — one line of code sets your key and you're done.

Verify your current Ghostscript licensing situation with your legal team before and during the migration. If GPL compliance is driving the migration, document the removal of all Ghostscript references for your compliance audit.

What Is the Bottom Line on This Migration?

The Ghostscript migration is often the most impactful from an architecture perspective, because Ghostscript isn't a .NET library — it's a native binary invoked via subprocess. Replacing it with a managed NuGet package removes an entire category of deployment complexity: binary path resolution, platform-specific process names, stderr parsing for error detection, and native package management in containers.

The capability gaps are real. PostScript interpretation and certain rasterization workflows don't have direct IronPDF equivalents. For those, retaining a minimal Ghostscript deployment or restructuring the pipeline is the honest answer.

For everything else — HTML-to-PDF rendering, PDF merging, watermarking, password protection, and page manipulation — the migration path is well-defined and the code reduction is dramatic. Start by auditing your Ghostscript subprocess calls, identify which argument patterns map to IronPDF operations, and migrate feature by feature.
