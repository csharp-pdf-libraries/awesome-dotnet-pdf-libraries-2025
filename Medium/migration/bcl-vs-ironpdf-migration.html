Migrating from BCL easyPDF to IronPDF: Solving Common Migration Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

The support ticket that forces the migration conversation usually has a subject line like: "PDF generation failing on new server." The stack trace points to a missing COM component, or a 32-bit DLL that doesn't load in a 64-bit process, or a Windows-only native dependency that was never going to survive the move to Linux containers.

BCL EasyPDF SDK works well in the environment it was designed for. The problems start when that environment changes — containerized CI/CD, cross-platform deployments, or .NET Core workloads that never anticipated a COM dependency in the PDF path.

I wrote this guide to walk through the real failure modes you will hit and how to solve them with IronPDF. The focus is on the troubleshooting patterns and migration steps, not a theoretical comparison.

Why Should I Migrate from BCL EasyPDF SDK?

Teams running BCL EasyPDF SDK in stable Windows environments often have no reason to move. The trigger is usually an infrastructure change, not a library failure. Here are the common migration drivers I see:

Docker/Linux containerization — BCL EasyPDF SDK's native Windows dependencies don't translate to Linux containers without significant workaround effort.

.NET Core / .NET 5+ incompatibility — COM-based components require [STAThread] or specific interop configuration that conflicts with modern ASP.NET Core threading models. Verify current .NET support before assuming.

32-bit/64-bit mismatch — Historical BCL SDK versions ship 32-bit native components. 64-bit IIS and modern .NET runtime defaults cause load failures.

NuGet-first dependency management — BCL EasyPDF distributed via installer, not NuGet, creates friction in modern CI/CD pipelines.

Missing MSI in CI — Build agents provisioned from images don't have the BCL installer baked in; each provisioning cycle requires manual intervention.

Azure App Service / serverless — Managed hosting environments often prohibit COM registration or native library installation.

HTML rendering limitations — BCL's HTML-to-PDF engine may not handle modern CSS. Verify your specific version's CSS support.

Support escalation path — If BCL Technologies support is slow to respond to issues, teams start evaluating alternatives.

License key management — Machine-bound or per-server licensing creates friction at scale.

Consolidation — Teams running multiple PDF tools prefer one well-maintained library.

How Do BCL EasyPDF and IronPDF Compare Feature by Feature?

  • Focus. BCL EasyPDF is Windows-centric HTML/Word to PDF; IronPDF covers HTML-to-PDF, edit, merge, and security in one package.
  • Pricing. BCL is per-developer or per-server; IronPDF is per-developer or royalty-free.
  • API style. BCL is COM interop or a .NET wrapper over native components; IronPDF is fully managed .NET, NuGet-native.
  • Learning curve. BCL is simple for basic use but exposes COM quirks for advanced scenarios; IronPDF's Chromium model helps with modern CSS.
  • HTML rendering. BCL uses a proprietary engine; IronPDF uses Chromium.
  • Page indexing. BCL page indexing varies by version — verify in current docs; IronPDF is 0-based.
  • Thread safety. BCL follows the COM STA model — verify concurrent patterns against your version; IronPDF renderers are reusable, verify thread-safety docs for your scenario.
  • Namespace. BCL namespace casing varies by version (the working .cs samples in this repo use BCL.EasyPDF); IronPDF is always IronPdf.

How Hard Is Each Feature to Migrate?

  • HTML string to PDF — Low to Medium. API shape differs; test CSS output carefully.
  • HTML file to PDF — Low to Medium. Path and base URL handling may differ.
  • Merge PDFs — Medium. BCL's merge API varies by version; verify against your SDK.
  • Split PDFs — Medium to High. Verify BCL page-extraction support before assuming parity.
  • Watermark — Medium. Stamping model differs.
  • Password protection — Medium. Both libraries support it; property names differ.
  • COM interop removal — High. Core architectural change — audit your threading model.
  • 32-bit to 64-bit — High. Any P/Invoke or unsafe code must be audited.
  • Azure / container deploy — High. New dependency surface; test thoroughly.
  • Async patterns — Medium. Verify IronPDF's async API against your existing usage.

When Does It Make Sense to Migrate?

  • If you are on Windows-only, stable infrastructure, evaluate whether migration provides enough benefit before moving.
  • If you are moving to Linux containers or Azure App Service, migration is strongly worth evaluating.
  • If you target .NET Core 3.1+ or .NET 5/6/7/8/9 workloads, verify BCL compatibility first; IronPDF is modern .NET native.
  • If your CI/CD pipeline cannot bake in an MSI installer, NuGet-based IronPDF simplifies dependency management.

What Should I Troubleshoot Before Writing Any Migration Code?

Before writing migration code, diagnose what is actually failing. I have seen teams waste days rewriting code when the real issue is a missing runtime dependency.

Diagnostic step 1 — What is BCL actually doing?

# Find all BCL EasyPDF references
rg "EasyPDF\|BCL\|PdfPrint\|PdfDocument" --type cs -l

# Find COM interop usage
rg "CreateObject\|Activator\.CreateInstance\|Marshal\." --type cs

# Find P/Invoke signatures
rg "\[DllImport" --type cs

Diagnostic step 2 — Check process architecture

// Run this in your current app to confirm 32/64-bit context
Console.WriteLine($"Process is 64-bit: {Environment.Is64BitProcess}");
Console.WriteLine($"OS is 64-bit: {Environment.Is64BitOperatingSystem}");
Console.WriteLine($"Runtime: {System.Runtime.InteropServices.RuntimeInformation.ProcessArchitecture}");

Diagnostic step 3 — Confirm COM availability

// If BCL uses COM, this will fail in environments without the component registered
try
{
    // Replace with actual BCL COM ProgID — verify in your docs
    var type = Type.GetTypeFromProgID("BCL.EasyPDF"); // hypothetical — verify
    if (type == null)
        Console.WriteLine("COM type not found — likely unregistered or missing");
    var obj = Activator.CreateInstance(type);
    Console.WriteLine("COM instance created successfully");
}
catch (Exception ex)
{
    Console.WriteLine($"COM instantiation failed: {ex.Message}");
    // This is the error to address before migration
}

Install IronPDF alongside BCL (don't remove BCL yet)

# Add IronPDF while BCL is still present
dotnet add package IronPdf

# Restore
dotnet restore

Run a parallel test — both libraries generating the same PDF — before removing BCL. This lets you compare outputs visually and catch rendering differences early.

How Do I Handle the COM Threading Problem?

If BCL EasyPDF SDK uses COM under the hood, your code may rely on [STAThread] or manual STA thread marshaling. IronPDF is fully managed and does not require STA threading. After migration, you can remove [STAThread] from entry points and simplify async code that worked around COM STA restrictions.

Removing the STAThread requirement

// Before — STA required for COM-based PDF generation
[STAThread]
static void Main()
{
    // BCL PDF call
}

// After — no threading constraint
static void Main()
{
    // IronPDF call
}

IronPDF is fully managed .NET, so it works in any threading context. You can safely remove [STAThread] attributes and simplify any STA marshaling workarounds. COM-based failures that threw COMException with HRESULT codes are gone entirely — IronPDF throws standard .NET exception types instead.

Update your catch blocks accordingly:

// Before
catch (System.Runtime.InteropServices.COMException comEx)
{
    Log($"COM error 0x{comEx.HResult:X}: {comEx.Message}");
}

// After
catch (Exception ex)
{
    Log($"IronPDF error: {ex.GetType().Name}: {ex.Message}");
}

How Do I Fix the 32-bit to 64-bit Migration Issue?

If your project was forced to target x86 due to BCL's 32-bit native components, you may be able to switch to AnyCPU or x64 after migration.

Update your project file

<!-- Before -->
<PlatformTarget>x86</PlatformTarget>

<!-- After -->
<PlatformTarget>AnyCPU</PlatformTarget>

Once BCL's native 32-bit components are out of the picture, you can switch to AnyCPU or x64. Rebuild and test thoroughly — other dependencies may have had hidden x86 assumptions. Also audit any [DllImport] or P/Invoke signatures that may have assumed a 32-bit process.

IronPDF pages are 0-based. If BCL used 1-based indexing, update all page index references. This is the most common silent bug in PDF library migrations — it won't throw, it will just operate on the wrong page.

How Do I Migrate HTML-to-PDF Conversion?

Before (BCL EasyPDF SDK)

using System;
using BCL.EasyPDF;

class Program
{
    [STAThread] // COM-based BCL may require STA thread model
    static void Main()
    {
        var pdf = new PDFDocument();
        var htmlConverter = new HTMLConverter();

        string html = @"<html>
            <body style='font-family:Arial'>
                <h1>Invoice #2071</h1>
                <p>Amount: $1,200.00</p>
                <p>Due: 2025-12-01</p>
            </body>
        </html>";

        htmlConverter.ConvertHTML(html, pdf);
        pdf.Save("invoice.pdf");
        pdf.Close();
        Console.WriteLine("Saved invoice.pdf");
    }
}

After (IronPDF)

using System;
using IronPdf;

class Program
{
    static void Main()
    {
        // No STA thread requirement — works in any threading context
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var renderer = new ChromePdfRenderer();

        string html = @"<html>
            <body style='font-family:Arial'>
                <h1>Invoice #2071</h1>
                <p>Amount: $1,200.00</p>
                <p>Due: 2025-12-01</p>
            </body>
        </html>";

        using var pdf = renderer.RenderHtmlAsPdf(html);
        pdf.SaveAs("invoice.pdf");
        Console.WriteLine("Saved invoice.pdf");
    }
}

The key change is replacing BCL's HTMLConverter + PDFDocument pair with IronPDF's single ChromePdfRenderer. The Chromium-based rendering engine gives you modern CSS support out of the box. No COM, no STA thread, no installer — just a NuGet package.

How Do I Migrate PDF Merging?

Before (BCL EasyPDF SDK)

using System;
using BCL.EasyPDF;

class Program
{
    static void Main()
    {
        var pdf1 = new PDFDocument("part1.pdf");
        var pdf2 = new PDFDocument("part2.pdf");

        // BCL merges by appending one document onto another in place
        pdf1.Append(pdf2);

        pdf1.Save("merged.pdf");
        pdf1.Close();
        pdf2.Close();
        Console.WriteLine("Merged to merged.pdf");
    }
}

After (IronPDF)

using System;
using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var pdf1 = PdfDocument.FromFile("part1.pdf");
        var pdf2 = PdfDocument.FromFile("part2.pdf");
        var pdf3 = PdfDocument.FromFile("part3.pdf");

        using var merged = PdfDocument.Merge(pdf1, pdf2, pdf3);
        merged.SaveAs("merged.pdf");
        Console.WriteLine("Merged to merged.pdf");
    }
}

IronPDF's merge API takes any number of PdfDocument objects and combines them in one call, instead of mutating one document by appending another the way BCL's Append does. Load each PDF with PdfDocument.FromFile(), merge with PdfDocument.Merge(), and save.

How Do I Migrate Watermarking and Stamping?

Before (BCL — verify against your SDK version)

using System;
// BCL watermark/stamp API surface varies significantly by SDK version —
// some builds expose stamping on PDFDocument, others require building it
// into source HTML before conversion. Verify what your installed version
// supports before porting.

class Program
{
    static void Main()
    {
        // Representative BCL watermark pattern — verify exact API in your version:
        // var doc = new PDFDocument("input.pdf");
        // doc.Pages[0].Watermark("CONFIDENTIAL", opacity: 0.4f);
        // doc.Save("watermarked.pdf");
        // doc.Close();
    }
}

After (IronPDF)

using System;
using IronPdf;
using IronPdf.Editing;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var pdf = PdfDocument.FromFile("input.pdf");

        var stamper = new TextStamper
        {
            Text = "CONFIDENTIAL",
            FontSize = 48,
            Opacity = 40,
            Rotation = 45,
            VerticalAlignment = VerticalAlignment.Middle,
            HorizontalAlignment = HorizontalAlignment.Center
        };

        pdf.ApplyStamp(stamper);
        pdf.SaveAs("watermarked.pdf");
        Console.WriteLine("Watermarked PDF saved");
    }
}

IronPDF uses a TextStamper model instead of a watermark-specific API. You configure the stamp text, font size, opacity (integer 0–100), rotation, and alignment as properties, then apply it with ApplyStamp(). This gives you more control over positioning than most watermark-only APIs.

How Do I Migrate Password Protection?

Before (BCL EasyPDF SDK)

using System;
using BCL.EasyPDF;

class Program
{
    static void Main()
    {
        var doc = new PDFDocument("input.pdf");

        // BCL exposes password and permissions as method calls on PDFDocument
        doc.SetPassword("user123");
        doc.SetOwnerPassword("owner456");
        doc.SetPrintPermission(false);
        doc.SetCopyPermission(false);

        doc.Save("protected.pdf");
        doc.Close();
    }
}

After (IronPDF)

using System;
using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var pdf = PdfDocument.FromFile("input.pdf");

        pdf.SecuritySettings.UserPassword = "user123";
        pdf.SecuritySettings.OwnerPassword = "owner456";
        pdf.SecuritySettings.AllowUserPrinting =
            IronPdf.Security.PdfPrintSecurity.FullPrintRights;
        pdf.SecuritySettings.AllowUserCopyPasteContent = false;

        pdf.SaveAs("protected.pdf");
        Console.WriteLine("Password protected PDF saved");
    }
}

IronPDF handles security through the SecuritySettings property on PdfDocument. Where BCL exposes SetPassword, SetOwnerPassword, SetPrintPermission, and SetCopyPermission as method calls, IronPDF surfaces the same controls as properties — UserPassword, OwnerPassword, AllowUserPrinting, AllowUserCopyPasteContent. No separate security class to instantiate.

How Do I Handle Async PDF Generation After Migration?

BCL's COM-based architecture makes async/await painful — COM STA objects can't be awaited across threads without explicit marshaling. IronPDF's managed API works cleanly with async.

IronPDF async rendering

public async Task<byte[]> RenderAsync(string html)
{
    var renderer = new ChromePdfRenderer();
    using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    return pdf.BinaryData;
}

In ASP.NET Core, register ChromePdfRenderer as a singleton or reuse it at the service level. Creating one per request adds startup overhead from Chromium process initialization. For Linux containers, IronPDF requires native libraries — check the Azure deployment docs for the current authoritative dependency list.

Docker configuration for Linux

FROM mcr.microsoft.com/dotnet/aspnet:8.0

# Install IronPDF native dependencies (verify current list in IronPDF docs)
RUN apt-get update && apt-get install -y \
    libgdiplus \
    libx11-6 \
    libc6 \
    libgtk-3-0 \
    # verify current required packages at https://ironpdf.com/how-to/azure/
    && rm -rf /var/lib/apt/lists/*

What Are the Key API Mappings I Need to Know?

Namespace Mapping

  • BCL.EasyPDF → IronPdf — core namespace.
  • COM ProgID usage → not applicable — IronPDF is fully managed.
  • BCL.EasyPDF.Interop / BCL.EasyPDF.Printer / BCL.EasyPDF.PDFConverter → IronPdf — everything collapses into one namespace.

Core Class Mapping

  • Printer → ChromePdfRenderer — main HTML/URL conversion class.
  • HTMLConverter → ChromePdfRenderer — HTML-to-PDF; ConvertHTML(html, doc) becomes renderer.RenderHtmlAsPdf(html).
  • PDFDocument → PdfDocument — loaded PDF document (note BCL's all-caps casing).
  • PrinterConfiguration → ChromePdfRenderOptions — rendering options.
  • SecurityHandler → PdfDocument.SecuritySettings — security configuration.
  • License class → IronPdf.License — license configuration.
  • COM object via Activator → not applicable — IronPDF requires no COM.

Document Loading Methods

  • Load from file: BCL new PDFDocument("file.pdf") → IronPDF PdfDocument.FromFile("file.pdf").
  • Load from stream: verify exact BCL API in your version → IronPDF PdfDocument.FromStream(stream).
  • HTML string: BCL htmlConverter.ConvertHTML(html, doc) → IronPDF renderer.RenderHtmlAsPdf(html).
  • HTML file: BCL printer.RenderFileToPDF(path, output) → IronPDF renderer.RenderHtmlFileAsPdf(path).
  • URL: BCL htmlConverter.ConvertURL(url, doc) → IronPDF renderer.RenderUrlAsPdf(url).

Page Operations

  • Page count: BCL doc.GetPageCount() → IronPDF pdf.PageCount.
  • Get page: BCL accessors vary by version — verify → IronPDF pdf.Pages[0] (0-based).
  • Delete page: BCL doc.DeletePage(index) (1-based) → IronPDF pdf.RemovePages(0) (0-based).
  • Extract pages: BCL doc.ExtractPages(start, end) (1-based) → IronPDF pdf.CopyPages(start, end) (0-based).
  • Rotate: BCL doc.RotatePage(index, angle) → IronPDF pdf.RotatePage(index, angle).

Merge/Split

  • Merge PDFs: BCL doc1.Append(doc2) in sequence → IronPDF PdfDocument.Merge(pdf1, pdf2, ...) in one call.
  • Split / extract: BCL doc.ExtractPages(start, end) (1-based) → IronPDF pdf.CopyPages(start, end) (0-based).

Frequently Asked Questions

Can I run IronPDF and BCL EasyPDF side by side during migration?

Yes, and I strongly recommend it. Install IronPDF via NuGet alongside your existing BCL references. Run both libraries generating the same PDF, then compare outputs visually before removing BCL. This lets you catch rendering differences early and validate your migration incrementally.

Do I need to change my project's platform target after removing BCL?

If your project was forced to target x86 due to BCL's 32-bit native components, you can switch to AnyCPU or x64 after migration. Update <PlatformTarget> in your project file, rebuild, and test thoroughly. Other dependencies may have had hidden x86 assumptions that surface only at runtime.

Will my existing catch blocks for COMException still work?

No. COM-based failures throw COMException with HRESULT codes. IronPDF throws standard .NET exception types. You need to update your catch blocks from catch (COMException) to catch (Exception) or more specific .NET exception types. This is a quick find-and-replace but easy to miss.

Does IronPDF support async PDF generation?

Yes. Unlike BCL's COM-based architecture where async/await requires painful STA thread marshaling, IronPDF's managed API works cleanly with async. Use renderer.RenderHtmlAsPdfAsync(html) for non-blocking PDF generation. In ASP.NET Core, register ChromePdfRenderer as a singleton to avoid per-request Chromium startup overhead.

What dependencies does IronPDF need on Linux containers?

IronPDF on Linux requires native libraries including libgdiplus, libx11-6, libc6, and libgtk-3-0. Add these to your Dockerfile's apt-get install step. Check the IronPDF Azure deployment docs for the current authoritative dependency list, as requirements may change between versions.
