Migrating from TX Text Control to IronPDF: Solving Common Migration Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate from TX Text Control to IronPDF?

I have worked with teams that hit the same ceiling with TextControl. The library is a powerful rich-text editor, but when you are using it only for PDF export, you are carrying the full library footprint for one feature. And the scaling constraints become apparent quickly under load.

TextControl's core components require STA thread affinity in many configurations. At fifty requests per hour, that works fine. At five hundred, the behavior changes. PDF output becomes inconsistent, some documents render with corrupted layout, and the logs show thread collisions in the rendering layer. The fix involves queues, semaphores, and fixed thread pools — infrastructure to work around a library constraint.

Here are the migration triggers I see most often. Thread safety and STA constraints force multi-threaded server workloads into workarounds. The STA constraint means one document at a time per thread, so horizontal scaling is limited by threading infrastructure, not compute. Installation complexity from native components and COM registration creates deployment friction in Docker and cloud environments. Generating PDFs from HTML content requires conversion to TextControl's document model first. And multiple TXTextControl.* packages for a PDF-only use case adds unnecessary NuGet surface area.

If your primary use case is HTML-to-PDF generation at scale, IronPDF is designed specifically for that. It uses an embedded Chromium engine, runs on any thread, and requires no COM registration or STA threading.

What Are the STA Threading Problems and How Does IronPDF Solve Them?

The concurrency issue is usually the trigger for evaluating alternatives. TextControl's ServerTextControl typically requires STA thread affinity in many configurations — verify against your specific TextControl version and license edition. That means every PDF generation call needs to be wrapped in a dedicated STA thread with SetApartmentState, Start, and Join boilerplate. Under concurrent load, you end up building request queues and semaphores just to serialize access.

Here is what the workaround pattern looks like in a typical TextControl codebase.

TextControl STA Threading Pattern (Before)

using TXTextControl;
using System;
using System.IO;
using System.Threading;

class HtmlToPdfBefore
{
    static void Main()
    {
        // STA threading wrapper required — the concurrency problem
        Exception threadException = null;
        var thread = new Thread(() =>
        {
            try
            {
                using var tx = new ServerTextControl();
                tx.Create();

                var html = @"
                    <html><body>
                    <h1>Invoice #2024-0099</h1>
                    <p>Customer: Acme Corp | Total: $4,200.00</p>
                    </body></html>";

                tx.Load(html, StreamType.HTMLFormat);

                tx.Save("invoice.pdf", StreamType.AdobePDF);
                Console.WriteLine("Saved invoice.pdf");
            }
            catch (Exception ex)
            {
                threadException = ex;
            }
        });

        thread.SetApartmentState(ApartmentState.STA);
        thread.Start();
        thread.Join();

        if (threadException != null)
            throw new Exception("TextControl error in STA thread", threadException);
    }
}

With IronPDF, all of that infrastructure disappears. No STA thread, no exception capture wrapper, no Join blocking.

IronPDF Replacement (After)

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// No STA thread wrapper, no thread.Start/Join, no exception capture
var html = @"
    <html>
    <head>
    <style>body { font-family: Arial, sans-serif; padding: 40px; }
    h1 { font-size: 22px; } .total { font-weight: bold; font-size: 16px; }
    </style></head>
    <body>
        <h1>Invoice #2024-0099</h1>
        <p>Customer: Acme Corp</p>
        <div class='total'>Total: $4,200.00</div>
    </body></html>";

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");
Console.WriteLine($"Saved invoice.pdf ({pdf.PageCount} page(s))");

For concurrent rendering, the difference is even more significant. Instead of an STA thread pool with semaphores, you use Task.WhenAll directly.

Concurrent Rendering Without STA Overhead

using IronPdf;
using System;
using System.Diagnostics;
using System.Linq;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var sw = Stopwatch.StartNew();

var requests = Enumerable.Range(1, 20)
    .Select(i => $"<html><body><h1>Document {i}</h1></body></html>")
    .ToArray();

var pdfs = await Task.WhenAll(requests.Select(async html =>
{
    var renderer = new ChromePdfRenderer();
    return await renderer.RenderHtmlAsPdfAsync(html);
}));

sw.Stop();
Console.WriteLine($"20 concurrent PDFs in {sw.Elapsed.TotalMilliseconds:F0}ms — no STA overhead");
foreach (var pdf in pdfs) pdf.Dispose();

How Do You Handle PDF Merging After Migration?

TextControl can append PDFs through its document model by loading each PDF with LoadAppendMode.Append and saving the combined result as PDF. It works, but every load and save runs through the same ServerTextControl document pipeline — you are routing PDFs through a word processor to concatenate them, with the same threading constraints as HTML rendering.

IronPDF has a dedicated PdfDocument.Merge for stitching PDFs without going through a document model. You can render sections concurrently and merge them in a single call.

TextControl Merge Approach (Before)

using TXTextControl;
using System;
using System.IO;
using System.Threading;

class MergeBefore
{
    static void Main()
    {
        var thread = new Thread(() =>
        {
            using var tx = new ServerTextControl();
            tx.Create();

            byte[] pdf1 = File.ReadAllBytes("document1.pdf");
            tx.Load(pdf1, StreamType.AdobePDF);

            byte[] pdf2 = File.ReadAllBytes("document2.pdf");
            tx.Load(pdf2, StreamType.AdobePDF, LoadAppendMode.Append);

            tx.Save("merged.pdf", StreamType.AdobePDF);
        });
        thread.SetApartmentState(ApartmentState.STA);
        thread.Start();
        thread.Join();

        Console.WriteLine("Merged via ServerTextControl document pipeline");
    }
}

IronPDF Merge (After)

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();

var results = await Task.WhenAll(
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 1</h1></body></html>"),
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 2</h1></body></html>")
);

var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("merged.pdf");
Console.WriteLine($"Merged: {merged.PageCount} pages — no STA thread needed");

How Do You Add Watermarks and Password Protection Without a Secondary Library?

With TextControl, watermarking on PDF output is typically expressed through the document model (background text or background images on a section), and the exact API depends on your TextControl version — verify against the edition you have licensed. PDF security export options (user/owner password) are exposed through SaveSettings in recent versions.

IronPDF handles both natively. No additional dependencies.

Watermark with IronPDF

using IronPdf;
using IronPdf.Editing;
using IronSoftware.Drawing;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Report</h1></body></html>"
);

var watermark = new TextStamper
{
    Text = "DRAFT",
    FontColor = Color.Gray,
    Opacity = 15,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("watermarked.pdf");

Password Protection with IronPDF

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Confidential Document</h1></body></html>"
);

pdf.SecuritySettings.UserPassword = "open123";
pdf.SecuritySettings.OwnerPassword = "admin456";

pdf.SaveAs("secured.pdf");

How Do the Two Libraries Compare Feature by Feature?

Here is a side-by-side comparison of the core aspects that matter for a migration decision.

  • Focus. TextControl is a rich text editor with document export (PDF, DOCX, RTF). IronPDF is HTML-to-PDF plus PDF manipulation.
  • Pricing. Both are commercial — verify current numbers at textcontrol.com and ironsoftware.com.
  • API style. TextControl uses a document load/edit/save model with STA thread affinity in many configurations. IronPDF exposes ChromePdfRenderer with no threading constraints in its renderer surface.
  • Learning curve. TextControl has its own document model to learn. IronPDF takes HTML and CSS as input, which most .NET teams already know.
  • HTML rendering. TextControl loads HTML into its document model via conversion. IronPDF uses embedded Chromium directly.
  • Page indexing. IronPDF is 0-based. TextControl page indexing depends on the API surface you are using — verify in TextControl docs.
  • Thread safety. TextControl has STA constraints in some configurations — verify. Verify IronPDF concurrent-instance guidance for your deployment topology.
  • Namespace. TXTextControl.* → IronPdf.

And here is the migration effort breakdown by feature.

  • HTML string to PDF — Low. TextControl loads HTML through its document model; IronPDF uses ChromePdfRenderer.RenderHtmlAsPdfAsync().
  • DOCX to PDF — High (keep TextControl for this). Native export in TextControl; not in IronPDF's scope.
  • Mail merge to PDF — Medium-High. TextControl has a built-in merge engine; IronPDF combines an HTML template engine with RenderHtmlAsPdfAsync().
  • Save to file — Low. doc.Save(path, format) → pdf.SaveAs(path).
  • Save to stream — Low. doc.Save(stream, format) → pdf.Stream.
  • Concurrent rendering — Low (eliminate workaround). TextControl typically needs an STA thread pool workaround; IronPDF needs none in the renderer surface.
  • Merge PDFs — Medium. TextControl appends through its document pipeline; IronPDF uses PdfDocument.Merge().
  • Watermark — Medium. Verify the TextControl background-text or stamp API for your version; IronPDF uses TextStamper or ImageStamper.
  • Password protection — Low. Verify TextControl export settings for your edition; IronPDF uses pdf.SecuritySettings.
  • Headers/footers — Medium. TextControl uses section headers and footers in the document model; IronPDF uses RenderingOptions.HtmlHeader/Footer or pdf.AddTextHeader/AddTextFooter.

What Are the Most Common Questions About This Migration?

Can I keep TextControl for DOCX and use IronPDF for PDF?

Yes, and this is the recommended approach if your codebase uses TextControl for both DOCX editing and PDF export. Keep TextControl for the DOCX creation and editing path. Replace only the PDF export path with IronPDF for HTML-to-PDF generation. Remove the DOCX-to-TextControl-to-PDF path if you can serve the same output through HTML templates instead.

What replaces TextControl's mail merge?

IronPDF is an HTML renderer, not a document editor, so there is no direct mail merge equivalent. The recommended replacement is a .NET template engine like Scriban or Fluid (both MIT licensed) that generates HTML, which you then render to PDF with IronPDF. If you are already in the ASP.NET Core stack, Razor views work as well. You render the view to an HTML string and pass it to RenderHtmlAsPdfAsync.

How do I find all TextControl references in my codebase?

Run these searches to locate every touchpoint before starting migration.

# Find TextControl API usage
rg -l "TXTextControl\|ServerTextControl\|TXTextControl" --type cs
rg "TXTextControl\|ServerTextControl\|StreamType\." --type cs -n

# Find STA thread workarounds (the scaling problem)
rg "ApartmentState\.STA\|STAThread\|SemaphoreSlim\|ConcurrentQueue" --type cs -n

# Find TextControl NuGet references
grep -r "TXTextControl" *.csproj **/*.csproj 2>/dev/null

What .NET versions does IronPDF support?

IronPDF supports .NET 6, 7, 8, and 9. You will need an IronPDF license key to get started. The installation is a single NuGet package (dotnet add package IronPdf) with no native COM components or registration requirements.

What about the disposal pattern?

The pattern is similar to TextControl. Use a using block on the PdfDocument result. For API responses, access pdf.BinaryData. For file output, call pdf.SaveAs(outputPath). The document is disposed at the end of the using block, just as you would with ServerTextControl.

var renderer = new ChromePdfRenderer();
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);

// For API response:
return pdf.BinaryData;

// For file:
pdf.SaveAs(outputPath);

When Should You Migrate?

The decision matrix from my experience working with teams making this switch breaks down cleanly.

  • If concurrent PDF generation is the primary scaling bottleneck, switch — this eliminates the STA threading constraint.
  • If HTML-to-PDF is the primary use case, switch — IronPDF is HTML-first, while TextControl requires a conversion step through its document model.
  • If DOCX editing plus PDF export is the primary use case, keep TextControl for the DOCX path and evaluate IronPDF for HTML-to-PDF separately.
  • If mail merge documents are the primary workload, evaluate a template engine (Scriban, Fluid, or Razor) as the mail-merge replacement before migrating.

If the STA threading constraint is your bottleneck, the migration pays for itself immediately. The concurrent rendering capacity is no longer limited by thread pool size and STA serialization — compute and I/O become the actual constraints. If DOCX editing is your primary use case, keep TextControl for that path and evaluate IronPDF separately for the HTML-to-PDF workload.

The remaining scope to plan for is mail merge replacement if that was a TextControl use case, and the DOCX path decision. Start with IronPDF's HTML-to-PDF guide to validate the approach against your actual templates.
