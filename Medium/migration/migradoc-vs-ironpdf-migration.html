Migrating from MigraDoc to IronPDF: Solving Common Migration Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

The pipeline worked fine on your dev machine. Then it went into the CI container and started failing with font-related errors on every build. MigraDoc's renderer depends on GDI+ internals that behave differently on Linux — in Alpine-based containers particularly, the libgdiplus dependency either isn't present, isn't the right version, or produces rendering artefacts that don't show up locally.

I wrote this guide to walk through the real failure modes you will hit migrating from MigraDoc to IronPDF, with a troubleshooting-first structure: the problems that commonly surface, how to diagnose them, and the code changes that resolve them.

Why Should I Migrate from MigraDoc to IronPDF?

Beyond CI/CD failures, here are the common reasons teams make this switch:

System.Drawing.Common deprecation on Linux — breaking change in .NET 7+ that affects MigraDoc versions with this dependency.

Programmatic vs HTML-first — MigraDoc requires building documents in code. Teams with HTML templates don't have a natural migration path within MigraDoc.

CSS/modern layout not supported — MigraDoc has its own style system, not CSS. Web-first teams find the mental model mismatch expensive.

No HTML renderer — converting existing HTML templates requires either converting them to MigraDoc's DOM or adding a second library.

Limited manipulation features — MigraDoc's strength is document generation; merging, watermarking, and encrypting existing PDFs requires PDFsharp directly or additional libraries.

Licensing trajectory — verify current licensing status; ensure commercial use terms haven't changed.

Debug difficulty — visual debugging of layout issues is harder via a DOM API than via HTML preview in a browser.

Limited async support — verify current async API surface in your MigraDoc version.

How Do I Diagnose CI/CD Failures Before Migrating?

Before touching migration code, understand exactly what's breaking in your pipeline. The failure mode shapes the fix.

Symptom: TypeInitializationException or DllNotFoundException referencing libgdi+

# Check if libgdiplus is present in your container
docker run --rm your-image ldconfig -p | grep gdi

# If missing, the quick fix for Debian/Ubuntu base images:
RUN apt-get update && apt-get install -y libgdiplus

This is a temporary fix. It adds an OS-level dependency to your image and must be validated each time the base image updates.

Symptom: Fonts render differently in CI than locally

MigraDoc relies on font resolution that can differ between Windows (dev) and Linux (CI).

# List available fonts in your container
docker run --rm your-image fc-list

# If font-config not installed:
RUN apt-get install -y fontconfig

# Install specific fonts if needed (e.g., liberation fonts)
RUN apt-get install -y fonts-liberation

Symptom: Works in .NET 6 but breaks in .NET 7+

System.Drawing.Common was restricted to Windows in .NET 6 (with a warning) and throws PlatformNotSupportedException on Linux in .NET 7+ by default. MigraDoc versions that depend on System.Drawing.Common will fail.

# Check if your MigraDoc version has this dependency
dotnet list package --include-transitive | grep -i drawing

If it does, you need either a newer MigraDoc version or a different library.

How Do I Convert MigraDoc Tables to HTML for IronPDF?

MigraDoc tables use an explicit column/row/cell model with precise width control. HTML tables are more flexible but require CSS for the same level of control. This is the conversion pattern I see teams struggle with most.

MigraDoc table (before)

using MigraDoc.DocumentObjectModel;
using MigraDoc.DocumentObjectModel.Tables;
using MigraDoc.Rendering;
using PdfSharp.Pdf;

class TableExample
{
    static void Main()
    {
        var doc = new Document();
        var section = doc.AddSection();

        var table = section.AddTable();
        table.Borders.Width = 0.75;

        // Define columns with explicit widths
        table.AddColumn("3cm");
        table.AddColumn("5cm");
        table.AddColumn("3cm");

        // Header row
        var row = table.AddRow();
        row.Shading.Color = MigraDoc.DocumentObjectModel.Colors.LightGray;
        row.Cells[0].AddParagraph("Product");
        row.Cells[1].AddParagraph("Description");
        row.Cells[2].AddParagraph("Price");

        // Data row
        row = table.AddRow();
        row.Cells[0].AddParagraph("Widget A");
        row.Cells[1].AddParagraph("Standard widget");
        row.Cells[2].AddParagraph("$50.00");

        var renderer = new PdfDocumentRenderer(unicode: true);
        renderer.Document = doc;
        renderer.RenderDocument();
        renderer.PdfDocument.Save("table.pdf");
    }
}

IronPDF HTML/CSS table (after)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

string html = @"
<html><head><style>
  table { border-collapse: collapse; width: 100%; }
  th, td { border: 1px solid #999; padding: 6px 10px; }
  th { background: #d3d3d3; font-weight: bold; }
  .col-product { width: 27%; }
  .col-desc    { width: 46%; }
  .col-price   { width: 27%; }
</style></head><body>
  <table>
    <tr>
      <th class='col-product'>Product</th>
      <th class='col-desc'>Description</th>
      <th class='col-price'>Price</th>
    </tr>
    <tr>
      <td>Widget A</td>
      <td>Standard widget</td>
      <td>$50.00</td>
    </tr>
  </table>
</body></html>";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("table.pdf");

Answer: The key change is replacing MigraDoc's programmatic Table / Row / Cell model with standard HTML <table> elements styled with CSS. Map MigraDoc's explicit column widths (e.g., "3cm", "5cm") to CSS percentage widths or fixed units. The advantage is you can preview the HTML table in a browser before rendering to PDF.

Here is the style mapping reference for the conversion:

MigraDoc ConceptHTML/CSS Equivalent
Style.Font.Size = 12font-size: 12pt
Style.Font.Bold = truefont-weight: bold
Style.Font.Color = Colors.Redcolor: red
ParagraphFormat.SpaceBeforemargin-top
ParagraphFormat.Alignment = Centertext-align: center
Section.PageSetup.LeftMarginChromePdfRenderOptions.MarginLeft

How Do I Migrate Headers, Footers, and Document Generation?

MigraDoc uses HeaderFooter objects attached to sections. IronPDF uses HtmlHeaderFooter, which gives you the same HTML/CSS control you have in the document body.

Headers and footers migration

// MigraDoc header pattern:
// section.Headers.Primary.AddParagraph("Page Header");

// IronPDF equivalent:
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.HtmlHeader = new HtmlHeaderFooter
{
    HtmlFragment = "<div style='text-align:right;font-size:9pt;'>Page Header</div>",
    DrawDividerLine = false
};
// Headers/footers guide: https://ironpdf.com/how-to/headers-and-footers/

Full document generation (before)

using MigraDoc.DocumentObjectModel;
using MigraDoc.Rendering;
using System;

class DocumentGenerationExample
{
    static void Main()
    {
        var doc = new Document();
        doc.Info.Title = "Invoice #1234";

        var style = doc.Styles["Normal"];
        style.Font.Name = "Arial";
        style.Font.Size = 10;

        var section = doc.AddSection();
        section.PageSetup.TopMargin = "2cm";
        section.PageSetup.BottomMargin = "2cm";

        var title = section.AddParagraph("Invoice #1234");
        title.Format.Font.Size = 18;
        title.Format.Font.Bold = true;
        title.Format.SpaceAfter = "0.5cm";

        var content = section.AddParagraph("Amount Due: $500.00");
        content.Format.Font.Size = 12;

        var pdfRenderer = new PdfDocumentRenderer(unicode: true)
        {
            Document = doc
        };
        pdfRenderer.RenderDocument();
        pdfRenderer.PdfDocument.Save("invoice.pdf");
        Console.WriteLine("Saved: invoice.pdf");
    }
}

Full document generation (after)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

string html = @"
<html><head><style>
  body { font-family: Arial, sans-serif; font-size: 10pt;
         margin: 2cm 2cm 2cm 2cm; }
  h1   { font-size: 18pt; margin-bottom: 0.5cm; }
  .amount { font-size: 12pt; }
</style></head><body>
  <h1>Invoice #1234</h1>
  <p class='amount'>Amount Due: $500.00</p>
</body></html>";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("invoice.pdf");
// Rendering options: https://ironpdf.com/how-to/rendering-options/

Answer: The core architectural change is replacing MigraDoc's Document DOM (sections, paragraphs, styles) with HTML templates and CSS. You define structure in HTML, styling in CSS, and render with ChromePdfRenderer. The HTML approach is typically more maintainable long-term — you can preview in a browser before rendering, and your web team already knows the styling language.

For the document model migration strategy: list every document type your system generates, create an HTML/CSS template for each (browser-preview it first), wire up your data model to populate the template (Razor, Handlebars.NET, string interpolation), then render via IronPDF's HTML-to-PDF renderer.

How Do I Migrate Merge, Watermark, and Password Protection?

These features required PDFsharp alongside MigraDoc. In IronPDF, they are native to the library.

PDF merge (before — PDFsharp)

using PdfSharp.Pdf;
using PdfSharp.Pdf.IO;
using System;

class MergePdfsExample
{
    static void Main()
    {
        using var outputDoc = new PdfDocument();

        foreach (string path in new[] { "section1.pdf", "section2.pdf" })
        {
            using var input = PdfReader.Open(path, PdfDocumentOpenMode.Import);
            foreach (PdfPage page in input.Pages)
                outputDoc.AddPage(page);
        }

        outputDoc.Save("merged.pdf");
        Console.WriteLine("Merged to: merged.pdf");
    }
}

PDF merge (after — IronPDF)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

var merged = PdfDocument.Merge(
    PdfDocument.FromFile("section1.pdf"),
    PdfDocument.FromFile("section2.pdf")
);
merged.SaveAs("merged.pdf");
// Guide: https://ironpdf.com/how-to/merge-or-split-pdfs/

Watermark (before — PDFsharp)

using PdfSharp.Pdf;
using PdfSharp.Pdf.IO;
using PdfSharp.Drawing;
using System.IO;
using System;

class WatermarkExample
{
    static void Main()
    {
        using var document = PdfReader.Open("input.pdf", PdfDocumentOpenMode.Modify);

        foreach (PdfPage page in document.Pages)
        {
            using var gfx = XGraphics.FromPdfPage(page, XGraphicsPdfPageOptions.Append);
            var font = new XFont("Arial", 60, XFontStyle.Bold);
            gfx.TranslateTransform(page.Width / 2, page.Height / 2);
            gfx.RotateTransform(-45);
            gfx.DrawString(
                "CONFIDENTIAL",
                font,
                new XBrush[] { XBrushes.LightGray }[0],
                new XPoint(0, 0)
            );
        }

        document.Save("watermarked.pdf");
        Console.WriteLine("Watermarked.");
    }
}

Watermark (after — IronPDF)

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

var pdf = PdfDocument.FromFile("input.pdf");
var stamper = new TextStamper
{
    Text = "CONFIDENTIAL",
    FontColor = IronSoftware.Drawing.Color.LightGray,
    FontSize = 60,
    Opacity = 35,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");
// Guide: https://ironpdf.com/how-to/custom-watermark/

Password protection (before — PDFsharp)

using PdfSharp.Pdf;
using PdfSharp.Pdf.IO;
using PdfSharp.Pdf.Security;
using System;

class SecurityExample
{
    static void Main()
    {
        using var document = PdfReader.Open("input.pdf", PdfDocumentOpenMode.Modify);

        var security = document.SecuritySettings;
        security.UserPassword  = "readpass";
        security.OwnerPassword = "adminpass";
        security.PermitPrint   = true;
        security.PermitModifyDocument = false;

        document.Save("secured.pdf");
        Console.WriteLine("Encrypted: secured.pdf");
    }
}

Password protection (after — IronPDF)

using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

var pdf = PdfDocument.FromFile("input.pdf");
pdf.SecuritySettings.UserPassword  = "readpass";
pdf.SecuritySettings.OwnerPassword = "adminpass";
pdf.SecuritySettings.AllowUserPrinting = PdfPrintSecurity.FullPrintRights;
pdf.SecuritySettings.AllowUserAnnotations = false;
pdf.SaveAs("secured.pdf");
// Guide: https://ironpdf.com/how-to/pdf-permissions-passwords/

Answer: All three operations — merge, watermark, and password protection — required PDFsharp as a separate library alongside MigraDoc. In IronPDF, they are first-class operations on PdfDocument. Merge uses PdfDocument.Merge(), watermarking uses TextStamper with ApplyStamp(), and security uses pdf.SecuritySettings directly. No second library needed.

Frequently Asked Questions

Can I run IronPDF and MigraDoc side by side during migration?

Answer: Yes, and I recommend it. Install IronPDF via NuGet alongside your existing MigraDoc packages. Run both libraries generating the same PDF, then compare outputs visually before removing MigraDoc. This lets you catch rendering differences — especially table column widths, font rendering, and spacing — early in the migration process.

How do I handle batch rendering performance after migration?

Answer: Reuse a single ChromePdfRenderer instance across multiple renders. Creating one per document adds startup overhead from Chromium process initialization. IronPDF bundles Chromium binaries, so the package is larger than MigraDoc + PDFsharp, but you eliminate the OS-level libgdiplus dependency. For web workloads, use the async API: await renderer.RenderHtmlAsPdfAsync(html).

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

foreach (var data in documentDataSet)
{
    string html = BuildHtml(data); // your template engine
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"doc_{data.Id}.pdf");
}

What about font handling differences between MigraDoc and IronPDF?

Answer: MigraDoc lets you specify fonts by name from the system font list. IronPDF's Chromium renderer does the same via CSS font-family. Verify that fonts your templates reference are installed in your deployment environment. IronPDF has more predictable cross-platform rendering behavior, but system fonts still need to be present for custom font usage. In Docker, add fonts explicitly:

# In your Dockerfile — add fonts before build
RUN apt-get install -y fonts-liberation fonts-dejavu

# Verify fonts are accessible inside the container
docker run --rm your-image fc-list | grep -i liberation

What is the page indexing difference I need to watch for?

Answer: IronPDF uses 0-based page indexing. MigraDoc doesn't expose a post-render page model, but PDFsharp (used for manipulation) uses 0-based as well — verify your specific version. This is the most common silent bug in PDF library migrations: it won't throw an error, it will just operate on the wrong page. Audit all page index references during migration.

How do I choose an HTML templating approach for the migration?

Answer: You have three main options: Razor templates for teams already in ASP.NET, Handlebars.NET for logic-light templates, or simple string interpolation for straightforward documents. The choice depends on your document complexity. Teams with complex multi-section reports with precise column widths and custom styles should budget a day or two per document type. Teams with simple invoices or letters can typically move faster.
