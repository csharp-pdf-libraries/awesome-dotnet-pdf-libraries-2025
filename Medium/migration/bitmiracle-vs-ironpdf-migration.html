Migrating from Bit Miracle Docotic to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Docotic.Pdf is a genuinely well-engineered library. The API is clean, the NuGet package is lean, and it handles PDF parsing and programmatic construction reliably across platforms. The migration decision rarely comes from Docotic.Pdf breaking -- it comes from a requirement that pushes past what Docotic was primarily designed for.

The most common trigger: someone adds a feature that requires rendering a modern HTML template -- with custom fonts, flex layout, web-style CSS -- to PDF. Docotic.Pdf does support HTML-to-PDF, but only through a separate add-on package (BitMiracle.Docotic.Pdf.HtmlToPdf) with an async-only HtmlConverter API. That extra moving part -- another NuGet, a separate license tier, an async-only entry point that downloads Chromium on first use -- is friction that teams often want to remove once HTML rendering becomes a first-class requirement.

I wrote this guide to walk you through the decision and the technical steps for migrating from BitMiracle Docotic.Pdf to IronPDF. The migration is relatively low-drama because both are managed .NET libraries with no COM dependencies. The complexity concentrates in two places: the different mental model (Docotic's document construction vs IronPDF's renderer-first approach), and any text extraction or annotation code that relies on Docotic's parse-side API.

Why Should You Migrate?

Here are the neutral triggers I see teams run into most often:

Browser-quality HTML rendering in one package -- Docotic's HTML path requires the BitMiracle.Docotic.Pdf.HtmlToPdf add-on and an async-only HtmlConverter. IronPDF includes the Chromium renderer in the core package, with both sync and async APIs.

CSS rendering fidelity -- Programmatic document construction in Docotic doesn't match what a designer produces in HTML/CSS. IronPDF's Chromium renderer closes this gap for HTML-based workflows.

Template maintenance burden -- Programmatic PDF construction (positioning text objects, drawing borders) requires developer changes for every layout update. HTML templates can be maintained by non-developers.

Beyond those core triggers, teams also cite font handling complexity (Docotic requires explicit font loading and glyph management while IronPDF's Chromium renderer handles font resolution as a browser would), deployment simplicity (IronPDF's all-in-one approach reduces the number of libraries in the dependency tree), API surface for new developers (ChromePdfRenderer.RenderHtmlAsPdf(html) is a shorter onboarding path than document construction for HTML-heavy workloads), PDF/A output workflows, license changes, and consolidation of parsing and rendering into one library.

How Do the Two Libraries Compare Side by Side?

  • Focus. Docotic.Pdf is parse, edit, construct, and extract. IronPDF is HTML-to-PDF first, plus edit, merge, and security.
  • Pricing. Docotic.Pdf is free with a watermark and offers a commercial license. IronPDF is per-developer or royalty-free.
  • API style. Docotic.Pdf is a document object model with explicit construction. IronPDF is a high-level renderer plus document model.
  • Learning curve. Docotic.Pdf is gradual for parsing and steeper for programmatic layout. IronPDF is gradual for HTML and moderate for parse ops.
  • HTML rendering. Docotic.Pdf requires the separate BitMiracle.Docotic.Pdf.HtmlToPdf add-on (async-only Chromium-based API). IronPDF ships the Chromium renderer in the core package.
  • Page indexing. Both libraries are 0-based.
  • Thread safety. Verify concurrent document access patterns in Docotic.Pdf. The IronPDF renderer is reusable; verify thread-safety docs for your version.
  • Namespace. Docotic.Pdf uses BitMiracle.LibTiff and BitMiracle.Docotic.Pdf. IronPDF uses IronPdf.

What Does the Migration Effort Actually Look Like?

I've broken down the effort by feature so you can estimate the scope for your codebase:

  • HTML string to PDF -- Low. A meaningful upgrade in fidelity for most Docotic users.
  • HTML file to PDF -- Low. Same as above.
  • Merge PDFs -- Low. Both support it; the API differs.
  • Split/extract pages -- Medium. Page extraction model differs.
  • Text watermark -- Medium. Docotic's canvas drawing model vs IronPDF's stamper.
  • Image watermark -- Medium. Coordinate model differs.
  • Password protection -- Low. Both support owner and user passwords.
  • Text extraction -- Medium-High. Docotic's text model is detailed; verify IronPDF parity for your needs.
  • Annotation handling -- High. Docotic's annotations API is rich; verify IronPDF coverage.
  • Form field editing -- High. Verify IronPDF's form API against your Docotic usage.
  • PDF/A output -- Medium. Test against your validator.

Which Scenario Matches Your Team?

  • Primarily programmatic PDF construction, no HTML. Evaluate whether Docotic's model already serves you; migration cost may exceed benefit.
  • HTML templates requiring browser-quality CSS. IronPDF's Chromium renderer addresses this directly.
  • Heavy text extraction and parse workflows. Audit IronPDF's extraction API against your Docotic usage before committing.
  • Mixed HTML and parse requirements. IronPDF can handle both; test extraction output quality on representative documents.

What Does the Quick Start Migration Look Like?

I break this into three steps: license configuration, namespace imports, and a basic PDF operation.

Step 1 -- License Configuration (Before)

using BitMiracle.Docotic;

// Docotic applies the license string before any API call
LicenseManager.AddLicenseData("YOUR-LICENSE-KEY");

Step 1 -- License Configuration (After)

using IronPdf;

// https://ironpdf.com/how-to/license-keys/
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

Step 2 -- Namespace Imports (Before)

using BitMiracle.Docotic.Pdf;
// Add when using the HtmlToPdf add-on:
// using BitMiracle.Docotic.Pdf; // HtmlConverter lives in this same namespace

Step 2 -- Namespace Imports (After)

using IronPdf;

Step 3 -- Basic PDF Operation: Open and Extract Text (Before)

using BitMiracle.Docotic.Pdf;
using System;

using (var pdf = new PdfDocument("document.pdf"))
{
    string allText = "";
    foreach (var page in pdf.Pages)
    {
        allText += page.GetText();
    }
    Console.WriteLine(allText);
}

Step 3 -- Basic PDF Operation: Open and Extract Text (After)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
var pdf = PdfDocument.FromFile("document.pdf");
// https://ironpdf.com/how-to/extract-text-and-images/
string text = pdf.ExtractAllText();
Console.WriteLine(text);

How Do the APIs Map Between Libraries?

Namespace Mapping

  • BitMiracle.Docotic.Pdf -> IronPdf -- core namespace (also hosts HtmlConverter from the HtmlToPdf add-on)
  • BitMiracle.Docotic.Pdf.Layout -> IronPdf -- layout in IronPDF is HTML/CSS-based
  • BitMiracle.Docotic.Pdf.HtmlToPdf package -> built into IronPdf; no separate add-on required

Core Class Mapping

  • PdfDocument -> PdfDocument -- same name, different API
  • PdfPage -> PdfPage -- page object
  • PdfCanvas -> no direct equivalent -- IronPDF uses a stamper model for drawing
  • LicenseManager.AddLicenseData() -> IronPdf.License.LicenseKey -- license setup

Document Loading Methods

  • Load from file. Docotic: new PdfDocument("file.pdf"). IronPDF: PdfDocument.FromFile("file.pdf").
  • Load from stream. Docotic: new PdfDocument(stream). IronPDF: PdfDocument.FromStream(stream).
  • Create from HTML. Docotic: requires the HtmlToPdf add-on, then await HtmlConverter.CreateAsync() followed by await converter.CreatePdfFromStringAsync(html). IronPDF: renderer.RenderHtmlAsPdf(html) (or the async variant) in the core package.
  • Load from bytes. Docotic: new PdfDocument(byteArray). IronPDF: PdfDocument.FromBinaryData(bytes).

Page Operations

  • Page count. Docotic: doc.PageCount. IronPDF: pdf.PageCount.
  • Get page. Docotic: doc.Pages[n] (0-based). IronPDF: pdf.Pages[n] (0-based).
  • Remove page. Docotic: doc.RemovePage(page). IronPDF: pdf.RemovePages(n).
  • Page size. Docotic: page.Width, page.Height. IronPDF: PdfPage.Width, PdfPage.Height.

Merge/Split

  • Merge PDFs. Docotic: doc1.Append("file.pdf") -- Append takes a file path, Stream, or byte[], not another PdfDocument. IronPDF: PdfDocument.Merge(pdf1, pdf2) or PdfDocument.Merge(new List<PdfDocument> { ... }).
  • Extract pages. Docotic: copy pages into a new document. IronPDF: pdf.CopyPages(start, end).

How Does HTML-to-PDF Migration Work in Practice?

This is the most common migration scenario. Docotic.Pdf's HTML-to-PDF lives in a separate add-on package (BitMiracle.Docotic.Pdf.HtmlToPdf), exposes an async-only HtmlConverter, and downloads Chromium on first use. IronPDF includes Chromium rendering in the core package and offers both sync and async entry points.

HTML to PDF -- Docotic HtmlConverter Add-on (Before)

// Requires both NuGet packages:
//   BitMiracle.Docotic.Pdf
//   BitMiracle.Docotic.Pdf.HtmlToPdf
using BitMiracle.Docotic.Pdf;
using System;
using System.Threading.Tasks;

class Program
{
    static async Task Main()
    {
        // HtmlConverter is async-only and downloads Chromium on first use
        using var converter = await HtmlConverter.CreateAsync();

        string html = "<html><body><h1>Hello World</h1><p>This is HTML to PDF conversion.</p></body></html>";

        using var pdf = await converter.CreatePdfFromStringAsync(html);
        pdf.Save("output.pdf");

        Console.WriteLine("PDF created successfully");
    }
}

HTML to PDF -- IronPDF Chromium Renderer (After)

using IronPdf;
using System;

class Program
{
    static void Main()
    {
        // https://ironpdf.com/how-to/html-string-to-pdf/
        var renderer = new ChromePdfRenderer();
        string html = "<html><body><h1>Hello World</h1><p>This is HTML to PDF conversion.</p></body></html>";

        var pdf = renderer.RenderHtmlAsPdf(html);
        pdf.SaveAs("output.pdf");

        Console.WriteLine("PDF created successfully");
    }
}

How Do You Migrate PDF Merging?

Merging is one of the simpler migrations. Docotic appends source files (or streams or byte arrays) into an open document; IronPDF has a static method that takes a list of PdfDocument instances.

Merge PDFs -- Docotic Append (Before)

using BitMiracle.Docotic.Pdf;
using System;

class Program
{
    static void Main()
    {
        using (var pdf1 = new PdfDocument("document1.pdf"))
        {
            // PdfDocument.Append accepts a file path, Stream, or byte[] --
            // not another PdfDocument instance.
            pdf1.Append("document2.pdf");
            pdf1.Save("merged.pdf");
        }

        Console.WriteLine("PDFs merged successfully");
    }
}

Merge PDFs -- IronPDF Static Merge (After)

using IronPdf;
using System;
using System.Collections.Generic;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        // https://ironpdf.com/how-to/merge-or-split-pdfs/
        var pdf1 = PdfDocument.FromFile("document1.pdf");
        var pdf2 = PdfDocument.FromFile("document2.pdf");

        var merged = PdfDocument.Merge(new List<PdfDocument> { pdf1, pdf2 });
        merged.SaveAs("merged.pdf");

        Console.WriteLine("PDFs merged successfully");
    }
}

How Do You Migrate Watermarks?

Docotic uses a canvas drawing model where you position everything with coordinates. IronPDF uses an alignment-based stamper model, which is simpler but requires a mental shift if you're used to absolute positioning.

Watermark -- Docotic Canvas Drawing (Before)

using System;
using BitMiracle.Docotic.Pdf;

class Program
{
    static void Main()
    {
        using var doc = new PdfDocument("input.pdf");

        // Docotic: draw text watermark on each page's canvas
        foreach (var page in doc.Pages)
        {
            var canvas = page.Canvas;

            canvas.SaveState();
            canvas.SetTransparency(0.3); // 30% opacity
            canvas.FontSize = 72;
            canvas.FillColor = new PdfRgbColor(200, 0, 0);
            canvas.Rotate(45, page.Width / 2, page.Height / 2);
            canvas.DrawString(page.Width / 4, page.Height / 2, "CONFIDENTIAL");
            canvas.RestoreState();
        }

        doc.Save("watermarked.pdf");
        Console.WriteLine("Watermarked PDF saved");
    }
}

Watermark -- IronPDF Stamper Model (After)

using System;
using IronPdf;
using IronPdf.Editing;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var pdf = PdfDocument.FromFile("input.pdf");

        // https://ironpdf.com/how-to/stamp-text-image/
        var stamper = new TextStamper
        {
            Text = "CONFIDENTIAL",
            FontSize = 48,
            Opacity = 40,
            Rotation = 45,
            VerticalAlignment = VerticalAlignment.Middle,
            HorizontalAlignment = HorizontalAlignment.Center
        };

        // Applies to all pages by default
        pdf.ApplyStamp(stamper);
        pdf.SaveAs("watermarked.pdf");
        Console.WriteLine("Watermarked PDF saved");
    }
}

How Do You Migrate Password Protection?

Password protection is a straightforward swap. Both libraries support owner and user passwords with permission controls.

Password Protection -- Docotic Encryption Settings (Before)

using System;
using BitMiracle.Docotic.Pdf;

class Program
{
    static void Main()
    {
        using var doc = new PdfDocument("input.pdf");

        // Docotic applies encryption through a single Encrypt() call
        var permissions = PdfPermissions.Printing | PdfPermissions.ContentExtraction;

        doc.Encrypt(
            ownerPassword: "owner456",
            userPassword: "user123",
            permissions: permissions,
            encryptionAlgorithm: PdfEncryptionAlgorithm.Aes256
        );

        doc.Save("protected.pdf");
        Console.WriteLine("Password protected PDF saved");
    }
}

Password Protection -- IronPDF Security Settings (After)

using System;
using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        // https://ironpdf.com/how-to/pdf-permissions-passwords/
        using var pdf = PdfDocument.FromFile("input.pdf");

        pdf.SecuritySettings.UserPassword = "user123";
        pdf.SecuritySettings.OwnerPassword = "owner456";
        pdf.SecuritySettings.AllowUserPrinting =
            IronPdf.Security.PdfPrintSecurity.FullPrintRights;

        pdf.SaveAs("protected.pdf");
        Console.WriteLine("Password protected PDF saved");
    }
}

What Are the Critical Differences to Watch For?

There are four areas where the migration can trip you up if you're not careful.

Text Extraction Differences. Docotic.Pdf has a detailed text model -- it exposes character positions, font information, and reading order metadata. IronPDF's text extraction returns the text content but with less granularity on positional data. If your code depends on character-level positioning (e.g., table detection, column parsing), test IronPDF's output carefully before committing.

Text Extraction Comparison

// Docotic -- granular character data
using var doc = new PdfDocument("input.pdf");
foreach (var page in doc.Pages)
{
    foreach (var chunk in page.GetWords())
    {
        Console.WriteLine($"{chunk.Text} at ({chunk.Bounds.X}, {chunk.Bounds.Y})");
    }
}

// IronPDF -- text content
// https://ironpdf.com/how-to/extract-text-and-images/
using var pdf = PdfDocument.FromFile("input.pdf");
string allText = pdf.ExtractAllText();
// For page-level extraction:
// string pageText = pdf.ExtractTextFromPage(0);

Annotation API Differences. Docotic.Pdf's annotation model is explicit -- you create PdfAnnotation objects with coordinates and properties. IronPDF has annotation support, but the API shape differs. Teams with heavy annotation workflows should test annotation round-trips (create, save, reload, verify) before migrating. See the IronPDF annotations docs for the current API.

Page Indexing. Both Docotic.Pdf and IronPDF use 0-based page indexing. This makes the page index migration simpler than moving from 1-based libraries, but still audit all page index references -- off-by-one errors are silent.

Canvas vs Stamper. Docotic's PdfCanvas is a drawing context -- you position everything with coordinates. IronPDF's stamper model is alignment-based. If you have precise coordinate requirements for stamps or overlays, check the stamper's IsStampedOnAllPages, HorizontalOffset, and VerticalOffset properties. For highly precise positioning, verify whether IronPDF's HTML-stamp approach (stamping a rendered HTML element) gives better control than the stamper API.

What About Performance and Deployment?

Neither Docotic.Pdf nor IronPDF require COM or native interop for core operations on Windows. This means the migration doesn't introduce new deployment complexity on Windows. On Linux, IronPDF's Chromium renderer does require system libraries -- Docotic.Pdf has fewer Linux-side requirements if your workload is parse-heavy.

If you're adding HTML-to-PDF as a new capability during this migration, reuse ChromePdfRenderer:

Renderer Reuse Pattern

// Register in DI container as singleton in ASP.NET Core
services.AddSingleton<ChromePdfRenderer>();

// Or reuse at class level
private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

Docotic.Pdf documents are IDisposable. IronPDF PdfDocument is also IDisposable. The migration shouldn't change disposal discipline if your existing code uses using consistently. Audit for any PdfDocument instances created without using or explicit .Dispose() calls.

If you generate PDFs in parallel, verify IronPDF's concurrent rendering behavior at the parallel examples. The Chromium renderer process is separate from the managed layer -- concurrent requests share the renderer process differently than Docotic's in-process rendering.

What Is the Full Migration Checklist?

Pre-Migration:

  • Run rg "using BitMiracle" --type cs -l to find all affected files
  • Capture baseline PDF outputs for visual regression comparison
  • Identify all PdfCanvas drawing operations (will need refactoring)
  • Inventory annotation-heavy code -- verify IronPDF annotation API parity
  • Check form field editing usage -- verify IronPDF form API
  • Note any character-level text extraction dependencies
  • Confirm target .NET version supports IronPDF
  • Test IronPDF output in parallel with Docotic before removing Docotic

Code Migration:

  • Replace BitMiracle.Docotic.Pdf (and any add-ons such as BitMiracle.Docotic.Pdf.HtmlToPdf) NuGet with IronPdf
  • Replace LicenseManager.AddLicenseData() with IronPdf.License.LicenseKey = "..."
  • Replace using BitMiracle.Docotic.Pdf with using IronPdf
  • Replace new PdfDocument("file.pdf") with PdfDocument.FromFile("file.pdf")
  • Replace programmatic layout with HTML templates where feasible
  • Replace PdfCanvas drawing with TextStamper or ImageStamper
  • Replace Docotic merge (Append(path) / Append(stream)) with PdfDocument.Merge()
  • Replace doc.Encryption settings with pdf.SecuritySettings properties
  • Replace doc.GetText() with pdf.ExtractAllText() -- verify parity
  • Update annotation handling to IronPDF API -- verify coverage

Testing:

  • Visual comparison: programmatic layouts vs HTML-rendered equivalents
  • Text extraction output -- compare character data if positional data needed
  • Annotation round-trip: create, save, reload, verify
  • Password protection: open with user password, confirm owner restrictions
  • Merge page count and order verification
  • Test on Linux if target environment -- verify system library requirements
  • Load test concurrent rendering if parallel workloads exist

Post-Migration:

  • Remove BitMiracle.Docotic.Pdf from all project files
  • Update Docker images for Linux deployment with IronPDF dependencies
  • Update internal documentation on PDF generation approach
  • Archive Docotic license for audit purposes

What Questions Come Up Most During This Migration?

Q: Can I keep Docotic.Pdf alongside IronPDF during the transition?

Yes. Both are managed .NET libraries with no COM dependencies, so they coexist in the same project without conflict. I recommend running them side by side during testing so you can compare outputs before fully removing Docotic. Both use a class named PdfDocument, so you will need namespace aliases to avoid ambiguity.

Q: Does IronPDF match Docotic's text extraction granularity?

Not exactly. Docotic.Pdf exposes character positions, font information, and reading order metadata. IronPDF's ExtractAllText() returns the text content but with less granularity on positional data. If your code depends on character-level positioning for table detection or column parsing, test carefully. Check the current API at ironpdf.com/how-to/extract-text-and-images.

Q: What about PdfCanvas drawing code -- is there a direct equivalent?

There is no direct equivalent. Docotic's PdfCanvas is a coordinate-based drawing context. IronPDF uses an alignment-based stamper model (TextStamper, ImageStamper) for overlays. For highly precise positioning, IronPDF's HTML-stamp approach -- stamping a rendered HTML element -- may give you better control than the stamper API alone.

Q: Does the migration introduce new Linux deployment requirements?

On Windows, no new complexity. On Linux, IronPDF's Chromium renderer requires system libraries that Docotic.Pdf does not need. If your workload is parse-heavy and runs on Linux, factor in the additional system library dependencies when updating your Docker images or deployment scripts.

How Should You Make the Final Decision?

The Docotic.Pdf to IronPDF migration is generally straightforward for the common operations -- loading, merging, password protection. The complexity concentrates in two areas that deserve careful evaluation before starting: annotation workflows and text extraction. If either is central to your use case, run a spike against the IronPDF API before committing to the migration path.

The real win for most teams isn't the API swap -- it's getting HTML-to-PDF that renders like a browser, which eliminates the programmatic layout code that no one wants to maintain.
