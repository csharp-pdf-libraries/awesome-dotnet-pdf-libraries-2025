Migrating from Apryse to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

The renewal email arrived on a Tuesday. Attached was a PDF — ironic, given the context — outlining the new licensing structure for the Apryse SDK, formerly PDFTron. The per-platform, per-feature, add-on pricing model had been reorganized. Again. Our legal team had questions. Our procurement team had questions. Our engineering team had one question: how much of this SDK do we actually use?

The answer, after two days of grepping, was humbling. We used HTML2PDF to convert Razor templates into reports. We used PDFDoc.MergePages to combine those reports into bundles. We used SecurityHandler to set passwords. That was it. Three features out of an SDK with a hundred.

Apryse builds enterprise document infrastructure. It is a serious platform for serious document workflows. But if your use case is "HTML in, encrypted PDF out," the licensing, API surface, and deployment weight of a full document platform create friction that compounds over time. This guide walks through migrating to IronPDF — a library focused specifically on the workflow we actually needed.

Why Migrate from Apryse to IronPDF?

Migration decisions rarely come from a single trigger. Here are the reasons I see most often from teams making this move.

Feature-to-usage ratio is the primary trigger. You are licensed for an enterprise document platform. You use three features. The delta between what you pay for and what you use is what starts the conversation.

Licensing complexity is the second driver. Apryse uses per-platform, per-module licensing with add-ons for OCR, redaction, digital signatures, CAD conversion, and more. Understanding what you are licensed for — and what you are not — requires reading contract attachments.

Pricing at scale compounds the issue. Apryse SDK pricing is enterprise-tier. For teams that need HTML-to-PDF and basic manipulation, the cost-per-feature is high relative to focused libraries.

API surface area creates cognitive overhead. PDFNet's API covers PDF internals, annotations, appearance objects, widget types, and content streams. For a team that just wants RenderHtmlAsPdf(), navigating the full SDK is real friction.

The HTML2PDF module has its own limitations. Apryse's HTML2PDF module uses its own rendering engine. Teams report issues with CSS fidelity compared to Chrome-based renderers, particularly with modern CSS grid, flexbox, and web fonts.

Native binary footprint adds deployment weight. Apryse ships platform-specific native libraries. Docker images grow, and ensuring the right binaries are present for each target platform requires careful configuration.

The PDFTron rebrand created a migration-within-a-migration. The rebrand from PDFTron to Apryse introduced package name and namespace changes. Teams on older PDFTron versions face a migration just to stay on the current Apryse SDK — which creates an opportunity to re-evaluate the dependency entirely.

Add-on fragmentation increases cost and complexity. Need OCR? That is a module. Need redaction? Another module. Need the WebViewer? Another license. Each add-on increases the licensing surface and the deployment footprint.

Documentation can be overwhelming. Apryse documentation is extensive but covers multiple platforms and languages. Multiple users report that finding the right entry point for a specific task takes trial and error.

Vendor consolidation introduces uncertainty. Apryse has expanded through acquisition — most recently TallComponents in May 2025 — bringing multiple .NET PDF product lines under one corporate umbrella. For teams concerned about long-term product strategy — which product line gets continued investment, which gets sunset — this consolidation raises questions worth asking your account manager.

  • Focus. Apryse is an enterprise document platform spanning PDF and roughly 30 other formats; IronPDF is HTML-first PDF generation and manipulation.
  • Pricing. Apryse uses enterprise, per-platform, add-on module licensing; IronPDF uses per-developer licensing with perpetual options.
  • API style. Apryse layers low-level PDFNet APIs with higher-level convenience classes; IronPDF is fluent, HTML-centric, and property-based.
  • Learning curve. Apryse is high — large API surface, multiple modules. IronPDF is low — HTML/CSS developers are productive quickly.
  • HTML rendering. Apryse uses its HTML2PDF module with a proprietary engine; IronPDF uses Chrome.
  • Page indexing. Apryse is 1-based via PageIterator; IronPDF is 0-based.
  • Thread safety. Apryse requires PDFNet.Initialize() and a managed threading model; IronPDF's ChromePdfRenderer is effectively stateless and safe to share.
  • Namespace. Apryse lives under pdftron.PDF and pdftron.SDF; IronPDF lives under IronPdf.

How Complex Is the Migration from Apryse to IronPDF?

The honest answer depends entirely on which Apryse features your codebase actually calls. Here is the effort breakdown by feature.

  • HTML to PDF — Low. Replace the HTML2PDF module with ChromePdfRenderer; expect better CSS fidelity from Chrome.
  • Merge — Low. Both libraries support merge; the API patterns differ.
  • Split / extract pages — Low. Different API, same concept.
  • Watermark — Medium. Apryse uses stamper and appearance objects; IronPDF uses an HTML overlay.
  • Password / encryption — Medium. Apryse uses SecurityHandler; IronPDF uses SecuritySettings.
  • Digital signatures — Medium-High. Signing APIs differ; verify IronPDF coverage for your specific PAdES level.
  • Annotation — High / N/A. Apryse has deep annotation support; IronPDF does not focus on annotations.
  • Redaction — N/A. No IronPDF equivalent — this is an Apryse-specific feature.
  • OCR — N/A. Not an IronPDF feature; consider IronOCR separately.
  • WebViewer — N/A. No IronPDF equivalent — the browser-based viewer is a separate Apryse product.
  • CAD / Office conversion — N/A. Not an IronPDF feature; use dedicated converters.

The key insight: if your Annot|Redactor|OCR|WebViewer usage count is zero but your HTML2PDF|PDFDoc|SecurityHandler count is non-zero, you are using 10% of the SDK. That is the ideal migration profile.

Find Your Apryse/PDFTron References

# Find all Apryse/PDFTron using statements
rg -l "pdftron|PDFTron|PDFNet|PDFDoc|HTML2PDF|SecurityHandler" --glob "*.cs"

# Count usage of advanced features (annotation, redaction, OCR)
rg -c "Annot|Redactor|OCR|WebViewer|ContentReplacer" --glob "*.cs"

# Count HTML2PDF usage (the feature you're likely migrating)
rg -c "HTML2PDF" --glob "*.cs"

# Find project references
rg -l "PDFTron|pdftron|Apryse" --glob "*.csproj"

Swap NuGet Packages

# Remove whichever Apryse/PDFTron package(s) you have installed.
# The package IDs were not renamed during the 2023 Apryse rebrand;
# common ones depending on your target framework and architecture:
dotnet remove package PDFTron.NET.x64                # primary .NET / .NET Core x64
dotnet remove package PDFTron.NetFramework.x64       # .NET Framework 4.5.1+ x64
dotnet remove package PDFTron.NETCore.Windows.x64    # .NET Core on Windows x64
dotnet remove package PDFNet                         # legacy combined Windows package

# Install IronPDF
dotnet add package IronPdf

What Does the Step-by-Step Migration Look Like in Code?

I will walk through the three foundational steps — license configuration, namespace imports, and basic HTML-to-PDF — then show four complete before/after migrations covering the most common operations.

License Configuration

Apryse requires a global initialization and termination lifecycle. IronPDF has none.

Before: Apryse License Setup

using pdftron;     
using pdftron.PDF; 
using System;

class Program
{
    static void Main()
    {
        // PDFNet requires global initialization before any operations
        PDFNet.Initialize("YOUR-LICENSE-KEY");

        // Some configurations require setting resource paths
        // PDFNet.SetResourcesPath("path/to/resources");

        Console.WriteLine("PDFNet initialized.");

        // All PDFNet operations must happen after Initialize
        // ...

        PDFNet.Terminate();
    }
}

After: IronPDF License Setup

using IronPdf;

class Program
{
    static void Main()
    {
        // No global initialization. No terminate. No resource paths.
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        Console.WriteLine($"IronPDF licensed: {License.IsLicensed}");
    }
}

The PDFNet.Initialize() / PDFNet.Terminate() lifecycle goes away entirely.

Namespace Imports

Apryse spreads across multiple namespaces. IronPDF consolidates into one primary namespace with optional sub-namespaces.

Before: Apryse Namespaces

using pdftron;          
using pdftron.PDF;      
using pdftron.SDF;      
using pdftron.Common;

After: IronPDF Namespaces

using IronPdf;

HTML to PDF — Before (Apryse HTML2PDF)

using pdftron;
using pdftron.PDF;
using System;

class Program
{
    static void Main()
    {
        PDFNet.Initialize("YOUR-LICENSE-KEY");

        using var doc = new PDFDoc();

        // HTML2PDF converter
        var converter = new HTML2PDF();
        converter.SetPaperSize(
            PrinterMode.PaperSize.e_letter
        );

        // Insert HTML content
        converter.InsertFromHtmlString(
            "<h1>Quarterly Report</h1><p>Generated by Apryse HTML2PDF</p>"
        );

        bool success = converter.Convert(doc);
        if (!success)
        {
            Console.WriteLine("Conversion failed: " + converter.GetHttpErrorCode());
            PDFNet.Terminate();
            return;
        }

        doc.Save("report_apryse.pdf", SDFDoc.SaveOptions.e_linearized);
        Console.WriteLine("Saved with Apryse HTML2PDF.");

        PDFNet.Terminate();
    }
}

HTML to PDF — After (IronPDF)

using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var renderer = new ChromePdfRenderer();
        renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.Letter;

        var pdf = renderer.RenderHtmlAsPdf(
            "<h1>Quarterly Report</h1><p>Generated by IronPDF</p>");

        pdf.SaveAs("report_ironpdf.pdf");
        Console.WriteLine("Saved with IronPDF.");
    }
}

No initialization. No termination. No converter object. No success-check boilerplate. The Chrome engine renders modern CSS faithfully. See the IronPDF HTML-to-PDF tutorial for more details.

How Do I Migrate HTML-to-PDF, Merge, Watermark, and Password Protection?

These are the four operations that cover 90% of what teams actually use from Apryse. Here are the complete before/after migrations for each one.

Before: HTML to PDF with Apryse (Full Example)

using pdftron;
using pdftron.PDF;
using System;

class HtmlToPdfApryse
{
    static void Main()
    {
        PDFNet.Initialize("YOUR-LICENSE-KEY");

        using var doc = new PDFDoc();
        var converter = new HTML2PDF();

        converter.SetPaperSize(PrinterMode.PaperSize.e_letter);
        converter.SetMargins("20", "20", "20", "20");

        string html = @"
            <html><head><style>
                body { font-family: 'Segoe UI', sans-serif; }
                .banner { background: #1b5e20; color: white; padding: 30px; }
                .metrics { display: grid; grid-template-columns: 1fr 1fr; gap: 20px; padding: 20px; }
                .card { border: 1px solid #e0e0e0; border-radius: 8px; padding: 20px; }
            </style></head><body>
                <div class='banner'><h1>Environment Impact Report</h1></div>
                <div class='metrics'>
                    <div class='card'><h3>Carbon Offset</h3><p>1,242 tons</p></div>
                    <div class='card'><h3>Energy Saved</h3><p>85,000 kWh</p></div>
                </div>
            </body></html>";

        converter.InsertFromHtmlString(html);

        if (!converter.Convert(doc))
        {
            Console.WriteLine("HTML2PDF failed");
            PDFNet.Terminate();
            return;
        }

        doc.Save("report_apryse.pdf", SDFDoc.SaveOptions.e_linearized);
        PDFNet.Terminate();
        Console.WriteLine("Saved with Apryse.");
    }
}

After: HTML to PDF with IronPDF (Full Example)

using IronPdf;
using System;

class HtmlToPdfIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var renderer = new ChromePdfRenderer();
        renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.Letter;
        renderer.RenderingOptions.MarginTop = 20;
        renderer.RenderingOptions.MarginBottom = 20;
        renderer.RenderingOptions.PrintHtmlBackgrounds = true;
        renderer.RenderingOptions.CssMediaType = IronPdf.Rendering.PdfCssMediaType.Screen;

        string html = @"
            <html><head><style>
                body { font-family: 'Segoe UI', sans-serif; }
                .banner { background: #1b5e20; color: white; padding: 30px; }
                .metrics { display: grid; grid-template-columns: 1fr 1fr; gap: 20px; padding: 20px; }
                .card { border: 1px solid #e0e0e0; border-radius: 8px; padding: 20px; }
            </style></head><body>
                <div class='banner'><h1>Environment Impact Report</h1></div>
                <div class='metrics'>
                    <div class='card'><h3>Carbon Offset</h3><p>1,242 tons</p></div>
                    <div class='card'><h3>Energy Saved</h3><p>85,000 kWh</p></div>
                </div>
            </body></html>";

        var pdf = renderer.RenderHtmlAsPdf(html);
        pdf.SaveAs("report_ironpdf.pdf");
        Console.WriteLine("Saved with IronPDF.");
    }
}

CSS grid renders correctly because IronPDF uses a real Chrome engine. Apryse's HTML2PDF module may not fully support CSS grid — verify with your templates.

Before: Merge PDFs with Apryse

using pdftron;
using pdftron.PDF;
using System;

class MergeApryse
{
    static void Main()
    {
        PDFNet.Initialize("YOUR-LICENSE-KEY");

        using var doc1 = new PDFDoc("part1.pdf");
        using var doc2 = new PDFDoc("part2.pdf");

        // Insert all pages from doc2 into doc1
        doc1.InsertPages(
            doc1.GetPageCount() + 1, // insert position (1-based)
            doc2,
            1,                        // start page
            doc2.GetPageCount(),      // end page
            PDFDoc.InsertFlag.e_none
        );

        doc1.Save("merged_apryse.pdf", SDFDoc.SaveOptions.e_linearized);
        PDFNet.Terminate();
        Console.WriteLine("Merged with Apryse.");
    }
}

After: Merge PDFs with IronPDF

using IronPdf;

class MergeIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var pdf1 = PdfDocument.FromFile("part1.pdf");
        var pdf2 = PdfDocument.FromFile("part2.pdf");

        var merged = PdfDocument.Merge(pdf1, pdf2);
        merged.SaveAs("merged.pdf");
        Console.WriteLine("Merged with IronPDF.");
    }
}

No page-count arithmetic. No insert position. No flags. See IronPDF merge documentation for additional patterns.

Before: Watermark with Apryse (Stamper-Based)

using pdftron;
using pdftron.PDF;
using System;

class WatermarkApryse
{
    static void Main()
    {
        PDFNet.Initialize("YOUR-LICENSE-KEY");

        using var doc = new PDFDoc("input.pdf");

        // Apryse Stamper API
        using var stamper = new Stamper(
            Stamper.SizeType.e_font_size,
            48,                             // font size
            0                               // unused size param
        );

        stamper.SetAlignment(
            Stamper.HorizontalAlignment.e_horizontal_center,
            Stamper.VerticalAlignment.e_vertical_center
        );
        stamper.SetFontColor(new ColorPt(0.8, 0.8, 0.8));
        stamper.SetRotation(45);
        stamper.SetAsBackground(false);                     

        var pgSet = new PageSet(1, doc.GetPageCount());
        stamper.StampText(doc, "CONFIDENTIAL", pgSet);

        doc.Save("watermarked_apryse.pdf", SDFDoc.SaveOptions.e_linearized);
        PDFNet.Terminate();
        Console.WriteLine("Watermarked with Apryse.");
    }
}

After: Watermark with IronPDF

using IronPdf;

class WatermarkIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var pdf = PdfDocument.FromFile("input.pdf");

        pdf.ApplyWatermark(
            "<h1 style='color:rgba(200,200,200,0.5);font-size:48px;font-family:Helvetica;'>CONFIDENTIAL</h1>",
            rotation: 45
        );

        pdf.SaveAs("watermarked.pdf");
        Console.WriteLine("Watermarked with IronPDF.");
    }
}

Stamper objects, ColorPt values, PageSet ranges, and alignment enums reduce to one HTML string.

Before: Password Protection with Apryse (SecurityHandler)

using pdftron;
using pdftron.PDF;
using pdftron.SDF;
using System;

class PasswordApryse
{
    static void Main()
    {
        PDFNet.Initialize("YOUR-LICENSE-KEY");

        using var doc = new PDFDoc("input.pdf");

        // Create security handler
        var handler = new SecurityHandler(
            SecurityHandler.AlgorithmType.e_AES,
            128                                    // key length
        );

        handler.ChangeUserPassword("user456");
        handler.ChangeMasterPassword("owner123");

        // Set permissions
        handler.SetPermission(
            SecurityHandler.Permission.e_print,
            false
        );
        handler.SetPermission(
            SecurityHandler.Permission.e_doc_modify,
            false
        );

        doc.SetSecurityHandler(handler);
        doc.Save("protected_apryse.pdf", SDFDoc.SaveOptions.e_linearized);

        PDFNet.Terminate();
        Console.WriteLine("Protected with Apryse.");
    }
}

After: Password Protection with IronPDF

using IronPdf;

class PasswordIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var pdf = PdfDocument.FromFile("input.pdf");

        pdf.SecuritySettings.OwnerPassword = "owner123";
        pdf.SecuritySettings.UserPassword = "user456";
        pdf.SecuritySettings.AllowUserPrinting = IronPdf.Security.PdfPrintSecurity.NoPrint;
        pdf.SecuritySettings.AllowUserEdits = IronPdf.Security.PdfEditSecurity.NoEdit;
        pdf.SecuritySettings.AllowUserCopyPasteContent = false;

        pdf.SaveAs("protected.pdf");
        Console.WriteLine("Protected with IronPDF.");
    }
}

No SecurityHandler construction, no algorithm type enum, no permission flag method calls. See IronPDF security documentation for all available options.

What Does the Full API Mapping Look Like Between Apryse and IronPDF?

I have compiled the complete namespace, class, and operation mappings below. These tables give you a reference for every common operation you will need to translate during migration.

Namespace Mapping

  • pdftron → no equivalent — IronPDF has no global SDK init step
  • pdftron.PDF → IronPdf — PDF document operations
  • pdftron.SDF → no equivalent — low-level document structure is not exposed in IronPDF

Core Class Mapping

  • PDFNet → IronPdf.License — global initialization and licensing
  • PDFDoc → PdfDocument — the PDF document object
  • HTML2PDF → ChromePdfRenderer — HTML-to-PDF conversion
  • SecurityHandler → pdf.SecuritySettings — encryption and password protection

Document Loading Methods

  • Open from file. new PDFDoc("file.pdf") → PdfDocument.FromFile("file.pdf")
  • Open with password. new PDFDoc("file.pdf"); doc.InitStdSecurityHandler("pass") → PdfDocument.FromFile("file.pdf", "pass")
  • Create new. new PDFDoc() → render via renderer.RenderHtmlAsPdf(html)
  • Save. doc.Save("out.pdf", saveOptions) → pdf.SaveAs("out.pdf")

Page Operations

  • Get page count. doc.GetPageCount() → pdf.PageCount
  • Get page. doc.GetPage(pageNum) (1-based) → pdf.Pages[index] (0-based)
  • Delete page. doc.PageRemove(pageIterator) → pdf.RemovePages(index)
  • Insert pages. doc.InsertPages(...) → PdfDocument.Merge() or page copy

Merge / Split Operations

  • Merge. doc.InsertPages() or doc.MergePages() → PdfDocument.Merge(pdf1, pdf2)
  • Extract pages. Apryse uses a page iterator into a new doc; IronPDF uses pdf.CopyPages(start, end).

What Critical Differences Will Break My Code If I Miss Them?

Five things will catch you if you are not paying attention: the initialization lifecycle, page indexing, save options, the HTML rendering engine, and features that do not migrate at all.

The PDFNet.Initialize() / PDFNet.Terminate() lifecycle goes away entirely. Apryse requires PDFNet.Initialize() before any SDK call and PDFNet.Terminate() at shutdown. Forgetting either causes crashes or resource leaks. IronPDF has no global lifecycle — set the license key and start working.

Page indexing changes from 1-based to 0-based. Apryse uses 1-based page numbering in most operations. IronPDF uses 0-based indexing. Audit every page reference in your codebase.

Apryse's Save method takes SDFDoc.SaveOptions flags — linearized, incremental, remove unused, and more. IronPDF's SaveAs performs a standard full save. If you relied on linearized output for web streaming, verify IronPDF's output meets your needs.

The HTML rendering engines are fundamentally different. Apryse's HTML2PDF module uses a proprietary rendering engine. IronPDF uses Chrome. The same HTML may render differently — particularly for CSS grid, flexbox, @font-face, and JavaScript-dependent content. Test your templates and expect IronPDF's Chrome output to be closer to what you see in a desktop browser.

Be explicit about features that do not migrate: annotation, redaction, OCR, WebViewer, CAD conversion, real-time collaboration. If any of these exist in your codebase — even unused — document them before removing the Apryse dependency.

Renderer Reuse

private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

Disposal Pattern

Additional edge cases worth flagging: linearized PDF output may require verification if your use case depends on web-optimized loading. Annotation preservation works in the sense that IronPDF can open Apryse-annotated files, but annotations cannot be programmatically modified. And if you are still on older PDFTron package names, you face a namespace migration regardless — this is a natural point to evaluate whether IronPDF better fits your actual usage.

Frequently Asked Questions

Is IronPDF a full replacement for Apryse?

For HTML-to-PDF rendering, merging, watermarking, and password protection — yes, it is a direct replacement with a simpler API. For deep annotation, redaction, OCR, and WebViewer features, Apryse's document platform features have no IronPDF equivalent. If your application performs both PDF generation and annotation collaboration, your architecture may need to separate those concerns.

What .NET versions does IronPDF support for this migration?

IronPDF supports .NET 6+ and .NET Framework 4.6.2+. Apryse SDK targets multiple platforms with platform-specific binaries. If you are on a modern .NET version, IronPDF's support is straightforward. You can get started by obtaining a license key.

How do I handle the PDFTron-to-Apryse rebrand during migration?

Teams on older PDFTron versions face a namespace migration just to stay on the current Apryse SDK. The rebrand introduced package name and namespace changes. If you are already facing that migration, it is the ideal time to evaluate whether Apryse is the right target or whether IronPDF better fits your actual usage. You avoid migrating twice.

What about performance differences between Apryse HTML2PDF and IronPDF?

Apryse's HTML2PDF module and IronPDF's Chrome engine are fundamentally different renderers. IronPDF's Chrome engine handles modern CSS faithfully because it is, essentially, Chrome. Apryse's engine may render simpler HTML faster. Test with your specific templates. For throughput scenarios, reuse the ChromePdfRenderer instance across requests rather than constructing per-call.

When Should I Migrate — and When Should I Stay?

Here is the honest decision framework.

  • If you need HTML-to-PDF, merge, and encrypt — and no viewer, annotation, or OCR — migrate. IronPDF covers this at a fraction of the API surface and cost.
  • If you depend on deep annotation, redaction, or real-time collaboration features, stay. Apryse's document platform features have no IronPDF equivalent.
  • If WebViewer is embedded in your application, stay for the viewer; you can still use IronPDF for server-side generation.
  • If you are already evaluating during the PDFTron-to-Apryse rebrand migration, this is a good time to right-size — migrate the parts you actually use to IronPDF instead of migrating twice.

Two days of grepping revealed that our six-figure document platform was a three-feature PDF library with extra weight. The migration to IronPDF took a week — most of that was testing HTML templates in the Chrome engine against the Apryse HTML2PDF output to verify CSS fidelity.

The lesson is not "Apryse is bad." It is "right-size your dependencies." If you need annotation, redaction, OCR, and a WebViewer, Apryse earns its place. If you need HTML to PDF to merge to encrypt, a focused library eliminates licensing conversations, reduces your API surface, and simplifies your deployment.

Worth knowing even without IronPDF: if you decide to stay with Apryse but want better HTML rendering, you can replace just the HTML2PDF module by rendering with Puppeteer Sharp or Playwright first, then loading the resulting PDF into Apryse for post-processing. This gives you Chrome-quality HTML rendering while keeping Apryse's annotation and manipulation features.
