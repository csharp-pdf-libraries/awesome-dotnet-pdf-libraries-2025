Migrating from Apache PDFBox to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

The pull request sat open for three weeks. Someone had added a seven-line method to merge two PDFs -- except it was seven lines of Java wrapping Apache PDFBox, running inside a microservice that the rest of the .NET monolith called over HTTP. The service existed solely because, years ago, nobody could find a .NET PDF library that handled merge without a commercial license. PDFBox was free. PDFBox was Java. So somebody spun up a JVM sidecar, and now every invoice merge takes a network hop, a serialization round-trip, and a deployment that requires both dotnet publish and mvn package.

If that sounds familiar -- or if you are running PDFBox via IKVM, a Kotlin interop layer, or a Lambda function that your .NET team cannot debug -- this guide is for you.

Apache PDFBox is a solid open-source Java library. The migration question is not about quality. It is about whether maintaining a Java dependency in a .NET ecosystem costs more than it saves. This guide covers the shift to IronPDF, but eighty percent of the framework -- dependency auditing, API mapping, testing patterns -- applies to any .NET-native PDF library.

Why Should I Migrate from Apache PDFBox to IronPDF?

PDFBox is one of the most widely deployed PDF libraries in the world. Here is why .NET teams specifically move away from it:

Language boundary. PDFBox is Java. Your application is .NET. Every integration requires a bridge: a microservice, IKVM transpilation, a CLI wrapper, or a serverless function. Each bridge has latency, deployment overhead, and debugging friction.

No HTML-to-PDF. PDFBox does not convert HTML. If you need HTML rendering, you are pairing PDFBox with another tool -- adding a second dependency in a second language.

JVM deployment overhead. Running PDFBox means running a JVM. In Docker, that is an additional base image layer. In CI, it is an additional SDK. In production, it is an additional runtime to patch.

IKVM compatibility gaps. Teams using IKVM to run PDFBox inside .NET hit compatibility issues with newer PDFBox versions, Bouncy Castle updates, and Java 11+ features that IKVM does not fully support.

API verbosity for common operations. PDFBox's API mirrors the PDF specification, which makes it powerful but verbose. Merging two PDFs requires loading both documents, creating a PDFMergerUtility, adding sources, and calling mergeDocuments(MemoryUsageSetting).

Memory management in Java. PDFBox documents hold native resources and must be explicitly closed. In a Java-to-.NET bridge, the garbage collection semantics differ, increasing the risk of resource leaks.

Text extraction ordering. PDFBox extracts text in the order it appears in the PDF content stream, which is not necessarily visual order. Sorting requires explicit configuration -- a common source of bugs for teams unfamiliar with PDF internals.

No async support. PDFBox's API is synchronous. In an async .NET pipeline, blocking on JVM calls ties up thread pool threads.

Font handling across platforms. PDFBox uses Java's font infrastructure. Font availability and rendering may differ between the JVM on your dev machine and the JVM in your production Linux container.

Two ecosystems to maintain. Maven for PDFBox, NuGet for everything else. Two package managers, two dependency trees, two vulnerability scanners.

PDFBox is Apache-licensed, so there is no license key. But you need a JVM, Maven, and a Java project structure. IronPDF requires a single license key line -- and no JVM at all.

// PDFBox is Apache-licensed — no license key required

        document.close();
    }
}

Step 3: Basic PDF Creation

PDFBox PDF Creation (Java, ~20 lines)

import org.apache.pdfbox.pdmodel.PDDocument;
import org.apache.pdfbox.pdmodel.PDPage;
import org.apache.pdfbox.pdmodel.PDPageContentStream;
import org.apache.pdfbox.pdmodel.font.PDType1Font;
import org.apache.pdfbox.pdmodel.common.PDRectangle;

public class CreatePdf {
    public static void main(String[] args) throws Exception {
        PDDocument document = new PDDocument();
        PDPage page = new PDPage(PDRectangle.LETTER);
        document.addPage(page);

        PDPageContentStream content = new PDPageContentStream(document, page);
        content.beginText();
        content.setFont(PDType1Font.HELVETICA, 24);
        content.newLineAtOffset(50, 700); // x, y from bottom-left
        content.showText("Monthly Report");
        content.endText();
        content.close();

        document.save("report_pdfbox.pdf");
        document.close();
        System.out.println("Saved with PDFBox.");
    }
}

How Do I Merge PDFs After Migrating?

PDFBox merging requires creating a PDFMergerUtility, adding sources, and configuring memory settings. IronPDF reduces this to a single static method call.

PDFBox Merge (Java)

import org.apache.pdfbox.multipdf.PDFMergerUtility;
import org.apache.pdfbox.io.MemoryUsageSetting;
import java.io.File;

public class MergePdfbox {
    public static void main(String[] args) throws Exception {
        PDFMergerUtility merger = new PDFMergerUtility();

        merger.addSource(new File("part1.pdf"));
        merger.addSource(new File("part2.pdf"));
        merger.setDestinationFileName("merged_pdfbox.pdf");

        // MemoryUsageSetting controls temp file vs. memory usage
        merger.mergeDocuments(MemoryUsageSetting.setupMainMemoryOnly());

        System.out.println("Merged with PDFBox.");
    }
}

IronPDF Merge (C#)

using IronPdf;

class MergeIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "IRONPDF-YOUR-KEY-HERE";

        var pdf2 = PdfDocument.FromFile("part2.pdf");

        var merged = PdfDocument.Merge(pdf1, pdf2);

        Console.WriteLine("Merged with IronPDF.");

}

How Do I Apply Watermarks After Migrating?

PDFBox watermarking requires content stream operations, matrix transforms, and per-page loops. IronPDF replaces all of that with a single HTML string.

PDFBox Watermark (Java, content stream drawing)

import org.apache.pdfbox.pdmodel.PDDocument;
import org.apache.pdfbox.pdmodel.PDPage;
import org.apache.pdfbox.pdmodel.PDPageContentStream;
import org.apache.pdfbox.pdmodel.font.PDType1Font;
import org.apache.pdfbox.util.Matrix;
import java.io.File;

public class WatermarkPdfbox {
    public static void main(String[] args) throws Exception {
        PDDocument doc = PDDocument.load(new File("input.pdf"));

        for (PDPage page : doc.getPages()) {
            PDPageContentStream cs = new PDPageContentStream(
                doc, page, PDPageContentStream.AppendMode.APPEND, true, true);

            cs.setFont(PDType1Font.HELVETICA, 48);
            cs.setNonStrokingColor(200, 200, 200); // light gray

            // Rotate 45 degrees around page center
            float cx = page.getMediaBox().getWidth() / 2;
            float cy = page.getMediaBox().getHeight() / 2;
            cs.transform(Matrix.getRotateInstance(
                Math.toRadians(45), cx, cy));

            cs.beginText();
            cs.newLineAtOffset(cx - 100, cy); // approximate centering
            cs.showText("CONFIDENTIAL");
            cs.endText();
            cs.close();
        }

        doc.save("watermarked_pdfbox.pdf");
        doc.close();
        System.out.println("Watermarked with PDFBox.");
    }
}

How Do I Add Password Protection After Migrating?

PDFBox encryption requires creating AccessPermission and StandardProtectionPolicy objects. IronPDF exposes the same controls through a single SecuritySettings property.

PDFBox Password Protection (Java)

import org.apache.pdfbox.pdmodel.PDDocument;
import org.apache.pdfbox.pdmodel.encryption.AccessPermission;
import org.apache.pdfbox.pdmodel.encryption.StandardProtectionPolicy;
import java.io.File;

public class PasswordPdfbox {
    public static void main(String[] args) throws Exception {
        PDDocument doc = PDDocument.load(new File("input.pdf"));

        AccessPermission perms = new AccessPermission();
        perms.setCanPrint(false);
        perms.setCanModify(false);
        perms.setCanExtractContent(false);

        StandardProtectionPolicy policy = new StandardProtectionPolicy(
            "owner123",  // owner password
            "user456",   // user password
            perms
        );
        policy.setEncryptionKeyLength(128);

        doc.protect(policy);
        doc.save("protected_pdfbox.pdf");
        doc.close();
        System.out.println("Protected with PDFBox.");
    }
}

IronPDF Password Protection (C#)

using IronPdf;

class PasswordIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "IRONPDF-YOUR-KEY-HERE";

        var pdf = PdfDocument.FromFile("input.pdf");

        pdf.SecuritySettings.OwnerPassword = "owner123";
        pdf.SecuritySettings.UserPassword = "user456";
        pdf.SecuritySettings.AllowUserPrinting = IronPdf.Security.PdfPrintSecurity.NoPrint;
        pdf.SecuritySettings.AllowUserCopyPasteContent = false;
        pdf.SecuritySettings.AllowUserEdits = IronPdf.Security.PdfEditSecurity.NoEdit;

        pdf.SaveAs("protected.pdf");
        Console.WriteLine("Protected with IronPDF.");
    }
}

No AccessPermission + StandardProtectionPolicy ceremony. See IronPDF security documentation.

What Should I Watch Out for During Migration?

The language shift is the migration. Unlike migrating between two .NET libraries, this migration crosses a language boundary. You are not just swapping API calls -- you are eliminating a JVM dependency, a Java build tool, and potentially a microservice. Plan for decommissioning the Java infrastructure, not just rewriting the code.

Page indexing. Both PDFBox and IronPDF use 0-based page indexing. This is the one thing that does not change.

Text extraction ordering. PDFBox's PDFTextStripper extracts text in content stream order by default, which can produce scrambled output. You must set setSortByPosition(true) for visual reading order. IronPDF's text extraction may handle ordering differently -- test with your specific documents.

Resource management. PDFBox's PDDocument must be explicitly closed (document.close() in Java). IronPDF's PdfDocument implements IDisposable -- use using blocks in C#. The pattern maps naturally.

Bouncy Castle dependency. PDFBox uses Bouncy Castle for encryption in Java. IronPDF handles encryption internally in .NET. You can remove the Bouncy Castle Java dependency entirely.

Eliminating the network hop. If PDFBox runs in a sidecar microservice, every PDF operation incurs HTTP serialization overhead. Moving to IronPDF makes PDF operations in-process:

// Before: HTTP call to Java PDFBox service
var response = await httpClient.PostAsync("http://pdf-service/merge",
    new StringContent(JsonSerializer.Serialize(request)));

// After: in-process, no network
var merged = PdfDocument.Merge(pdf1, pdf2);

Renderer reuse. Create one ChromePdfRenderer and reuse it:

private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

Disposal. Always wrap PdfDocument instances in using blocks:

using var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");

What Edge Cases Should I Test?

Text extraction quality. PDFBox's PDFTextStripper is well-regarded for extraction quality. Test IronPDF's text extraction against your specific document corpus. For extraction-heavy workloads, also consider PdfPig, a .NET-native PDF text extraction library that works without IronPDF.

PDF/A validation. PDFBox's Preflight module validates PDF/A compliance. IronPDF generates PDF/A but may not validate existing documents -- if you need validation, consider keeping Preflight as a separate tool or using veraPDF.

Large file memory. PDFBox offers MemoryUsageSetting to control memory vs. temp-file trade-offs for large documents. IronPDF handles this internally -- test with your largest production documents.

Frequently Asked Questions

Is Apache PDFBox free and IronPDF paid?

Answer: Yes. Apache PDFBox is free under the Apache License v2.0. IronPDF uses per-developer perpetual licenses. The cost trade-off is not just the license fee -- it is the JVM infrastructure, the cross-language bridge maintenance, and the deployment complexity that PDFBox requires in a .NET ecosystem.

Many teams find the operational savings from eliminating the Java dependency outweigh the license cost. A trial key lets you evaluate before committing.

Can I migrate incrementally or do I have to replace everything at once?

Answer: You can migrate incrementally. Start with the highest-value operation -- typically HTML-to-PDF, since PDFBox cannot do this at all. Leave lower-priority operations (like text extraction) running through the existing PDFBox bridge while you validate IronPDF's output.

The key is that each operation you migrate removes one HTTP call or interop boundary. You do not need to decommission the Java service until every operation has been ported.

What happens to my PDFBox Preflight (PDF/A validation) workflows?

Answer: IronPDF supports PDF/A export but may not validate existing documents the way PDFBox's Preflight module does. If PDF/A validation is critical, consider keeping Preflight as a standalone validation tool or switching to veraPDF.

This is one area where the migration is not a clean one-to-one swap. Audit your PDF/A workflows before migrating.

How do I handle text extraction differences between PDFBox and IronPDF?

Answer: PDFBox's PDFTextStripper extracts text in content stream order by default, which is not necessarily visual order. You must explicitly call setSortByPosition(true) for reading-order output. IronPDF may handle ordering differently.

Test with at least 10 diverse documents from your production corpus. Compare output line by line. For extraction-heavy workloads, PdfPig is a .NET-native alternative worth evaluating alongside IronPDF.

Do I need to change my page indexing code?

Answer: No. Both PDFBox and IronPDF use 0-based page indexing. This is one of the few areas where the migration requires zero code changes beyond the API surface swap.

The method names change -- document.getPage(index) becomes pdf.Pages[index] -- but the index values stay the same.

What is the recommended migration scenario for my team?

Answer: It depends on your current integration pattern. If your .NET app calls a Java PDFBox sidecar for basic operations, migrate -- eliminate the cross-language bridge entirely. If your full .NET stack needs HTML-to-PDF plus manipulation, migrate -- IronPDF covers both in one NuGet package.

If PDFBox is used only for text extraction, evaluate carefully. IronPDF extracts text, but you might also consider lighter .NET-native alternatives like PdfPig. If PDFBox is used for PDF/A validation via Preflight, keep Preflight or find a dedicated validator.

Conclusion

The real migration here is not API-to-API. It is ecosystem-to-ecosystem. You are removing a JVM, a Maven build, and a network boundary from your architecture. The code translation is the easy part. The deployment simplification is where the time savings compound.

If you are currently bridging to PDFBox via IKVM, know that IKVM's compatibility with newer Java versions is limited. The project is actively maintained but does not cover all Java 11+ features. If you stay in .NET but want a free option, PdfPig handles text extraction and basic manipulation in pure C# under Apache 2.0, though it does not do HTML-to-PDF.

For teams that need HTML rendering, merging, watermarking, and encryption in one package without maintaining a second language runtime, IronPDF consolidates everything into a single NuGet install. Start with a trial key and migrate one operation at a time.
