Migrating from Apache PDFBox to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

The pull request sat open for three weeks. Someone had added a seven-line method to merge two PDFs using Pdfbox-IKVM, one of the community .NET ports of Apache PDFBox. It compiled. It even ran. But on the third invoice the team tried, PDFMergerUtility.mergeDocuments() failed on a code path the upstream Java library handles fine. The port was years behind — its last NuGet release shipped in 2017 — and nobody on the team owned it. The original reason it was chosen — "PDFBox is free" — was about the Java library, not the abandoned bindings the .NET app was actually calling.

If that sounds familiar — or if you are running an Apache PDFBox community port that has not seen a release in years, ships without async support, and offers no commercial backing when something breaks — this guide is for you.

Apache PDFBox itself is a solid open-source Java library (current line: PDFBox 3.0.7, Apache License 2.0). The migration question is not about the upstream project. It is about whether an unofficial .NET port — IKVM-bundled, JCOBridge-wrapped, or hand-translated — costs more than it saves once your application grows past the trivial cases. This guide covers the shift to IronPDF, but eighty percent of the framework — dependency auditing, API mapping, testing patterns — applies to any .NET-native PDF library.

Why Should I Migrate from Apache PDFBox to IronPDF?

Upstream Apache PDFBox is one of the most widely deployed PDF libraries in the world. Here is why .NET teams specifically move away from its community .NET ports:

No official .NET build; the community options are uneven. Apache only publishes Java artifacts. The available .NET ports are Pdfbox 1.1.1 (2013, abandoned), Pdfbox-IKVM 1.8.9 (2017, abandoned, IKVM-bundled JVM reimplementation), PdfBox_DotNet_Version 2.0.15 (2019), and MASES.NetPDF 3.0.10 (March 2026, active — a JCOBridge wrapper that requires a real JVM at runtime). Three of the four have not shipped in years.

No HTML-to-PDF. PDFBox does not convert HTML, and every .NET port inherits that gap. If you need HTML rendering, you are pairing the port with a second library or shelling out to an external renderer.

JVM baggage either way. The IKVM-based ports bundle a .NET reimplementation of the JVM. MASES.NetPDF calls into a real JVM via JCOBridge. Either path drags Java runtime concerns into a .NET deployment that idiomatic .NET libraries avoid.

No commercial support. When a port misbehaves, there is no vendor to file a ticket with. Issues sit on community GitHubs for months — and for the abandoned ports, indefinitely. For a library that sits underneath an invoicing or compliance pipeline, that is a hard risk to carry.

API verbosity for common operations. PDFBox's API mirrors the PDF specification, which makes it powerful but verbose. Merging two PDFs requires loading both documents, creating a PDFMergerUtility, adding sources, and calling mergeDocuments().

Manual resource management. PDDocument in the port follows the Java close() contract directly — IKVM and JCOBridge preserve method casing, so you call document.close(), not Dispose(). Forgetting it leaks file handles.

Text extraction ordering. PDFBox extracts text in the order it appears in the PDF content stream, which is not necessarily visual order. Sorting requires explicit configuration — a common source of bugs for teams unfamiliar with PDF internals.

No async support. The ported API is synchronous. In an async .NET pipeline, blocking on it ties up thread pool threads.

Font handling across platforms. The port relies on whatever font discovery the underlying JDK-style code does. Font availability and rendering can differ between your dev machine and your production Linux container.

Java-flavored API in a C# project. Because IKVM and JCOBridge preserve Java's lowercase method names, the surface still reads as Java: PDDocument.load(), stripper.getText(), merger.addSource(). New .NET hires read it as foreign code.

How Do Apache PDFBox and IronPDF Compare?

The community .NET ports of PDFBox focus on PDF manipulation, text extraction, and form filling. Their API mirrors the PDF specification with classes like PDDocument and PDPage exposed under org.apache.pdfbox.* (IKVM ports preserve Java's lowercase package form) or Org.Apache.Pdfbox.* (MASES.NetPDF title-cases for C#) — powerful, but it assumes you know PDF internals. The library is free under Apache License v2.0 and has no HTML rendering capability in either the upstream Java project or any of the .NET ports.

IronPDF takes the opposite approach: HTML-first PDF generation plus manipulation, exposed through a property-based C# API in the IronPdf namespace. The learning curve is low for any developer who already knows HTML and CSS, since rendering goes through a built-in Chrome engine. Licensing is per-developer with perpetual options available.

A few specifics worth noting up front: both libraries use 0-based page indexing, so index values stay the same when you migrate. PDDocument in the PDFBox ports is not safe to share across threads, while IronPDF's ChromePdfRenderer is stateless and safe to share.

How Much Effort Does Each Feature Require to Migrate?

  • HTML to PDF — Low. PDFBox cannot do this; IronPDF is purpose-built for it.
  • Text extraction — Medium. Both extract text; API shape and sort behavior differ.
  • Merge — Low. Both offer merge; IronPDF is a one-liner via PdfDocument.Merge().
  • Split / extract pages — Low. Different API, same concept (pdf.CopyPages() in IronPDF).
  • Watermark — Medium. PDFBox uses PDPageContentStream drawing; IronPDF uses HTML overlay via ApplyWatermark.
  • Password / encryption — Medium. PDFBox uses AccessPermission + StandardProtectionPolicy; IronPDF uses SecuritySettings.
  • Form filling (AcroForms) — Medium. Both support AcroForms with different API surfaces.
  • Digital signatures — Medium-High. Different libraries and API patterns.
  • PDF/A validation — High / N/A. PDFBox has Preflight for PDF/A validation; IronPDF supports PDF/A export but validation parity is not guaranteed -- verify before relying on it.
  • Image extraction — Medium. Both support image extraction with different APIs.
  • PDF rendering to image — Medium. PDFBox renders pages to BufferedImage; IronPDF rasterizes via ToBitmap(dpi).

    {

Step 3: Basic PDF Creation

Apache PDFBox Port PDF Creation (C#, ~20 lines)

using org.apache.pdfbox.pdmodel;
using org.apache.pdfbox.pdmodel.common;
using org.apache.pdfbox.pdmodel.font;
using System;

class CreatePdf
{
    static void Main()
    {
        PDDocument document = new PDDocument();
        try
        {
            PDPage page = new PDPage(PDRectangle.LETTER);
            document.addPage(page);

            PDPageContentStream content = new PDPageContentStream(document, page);
            content.beginText();
            content.setFont(PDType1Font.HELVETICA, 24);
            content.newLineAtOffset(50, 700); // x, y from bottom-left
            content.showText("Monthly Report");
            content.endText();
            content.close();

            document.save("report_pdfbox.pdf");
        }
        finally
        {
            document.close();
        }
        Console.WriteLine("Saved with the PDFBox .NET port.");
    }
}

How Do I Convert HTML to PDF After Migrating?

The PDFBox .NET ports inherit PDFBox's lack of HTML-to-PDF support. Neither the Java original nor any of the .NET ports (Pdfbox-IKVM, PdfBox_DotNet_Version, MASES.NetPDF) ships an HTML/CSS renderer. The workaround is shelling out to an external tool like wkhtmltopdf, or pairing the port with a separate HTML renderer. With IronPDF, HTML rendering is built in.

Apache PDFBox HTML-to-PDF Workaround (C#)

// PDFBox is a low-level PDF manipulation library — there is no API that
// takes "
Hello
" and produces a styled PDF. The usual workaround
// is to shell out to an external renderer.
using System;
using System.Diagnostics;
using org.apache.pdfbox.pdmodel;
using java.io;

class HtmlToPdfPdfbox
{
    static void Main()
    {
        // Option A: Shell out to wkhtmltopdf
        var psi = new ProcessStartInfo("wkhtmltopdf",
            "--page-size Letter --margin-top 20 --margin-bottom 20 input.html report.pdf")
        {
            RedirectStandardError = true,
            UseShellExecute = false
        };
        using (var process = Process.Start(psi))
        {
            process.WaitForExit();
            if (process.ExitCode != 0)
            {
                Console.Error.WriteLine("wkhtmltopdf failed: " + process.ExitCode);
                return;
            }
        }

        // Option B: Then post-process with the PDFBox port if needed
        // PDDocument doc = PDDocument.load(new File("report.pdf"));
        // try { /* ... manipulate ... */ doc.save("report_final.pdf"); }
        // finally { doc.close(); }

        Console.WriteLine("Generated with wkhtmltopdf + PDFBox port.");
    }
}

How Do I Merge PDFs After Migrating?

PDFBox merging requires creating a PDFMergerUtility, adding sources, and calling mergeDocuments() with a MemoryUsageSetting — and on the abandoned ports specifically, the call site has been a known sore spot when bugs go unfixed. IronPDF reduces this to a single static method.

Apache PDFBox Port Merge (C#)

// IKVM-style .NET port (e.g. Pdfbox-IKVM on nuget.org). Method names
// stay Java-style (camelCase) because IKVM exposes the Java API directly.
using org.apache.pdfbox.multipdf;
using org.apache.pdfbox.io;
using System;

class MergePdfbox
{
    static void Main()
    {
        PDFMergerUtility merger = new PDFMergerUtility();

        merger.addSource("part1.pdf");
        merger.addSource("part2.pdf");
        merger.setDestinationFileName("merged_pdfbox.pdf");

        // MemoryUsageSetting governs heap vs temp-file buffering
        merger.mergeDocuments(MemoryUsageSetting.setupMainMemoryOnly());

        Console.WriteLine("Merged with the PDFBox .NET port.");
    }
}

IronPDF Merge (C#)

using IronPdf;

class MergeIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var pdf1 = PdfDocument.FromFile("part1.pdf");
        var pdf2 = PdfDocument.FromFile("part2.pdf");

        var merged = PdfDocument.Merge(pdf1, pdf2);
        merged.SaveAs("merged.pdf");
        Console.WriteLine("Merged with IronPDF.");
    }
}

How Do I Apply Watermarks After Migrating?

Watermarking through the PDFBox port requires content stream operations, matrix transforms, and per-page loops — and on the abandoned ports, you are at the mercy of whatever helpers the maintainers shipped before walking away. IronPDF replaces all of that with a single HTML string.

Apache PDFBox Port Watermark (C#, content stream drawing)

using org.apache.pdfbox.pdmodel;
using org.apache.pdfbox.pdmodel.font;
using org.apache.pdfbox.util;
using java.io;
using System;

class WatermarkPdfbox
{
    static void Main()
    {
        PDDocument doc = PDDocument.load(new File("input.pdf"));
        try
        {
            foreach (PDPage page in doc.getPages())
            {
                PDPageContentStream cs = new PDPageContentStream(
                    doc, page, PDPageContentStream.AppendMode.APPEND, true, true);

                cs.setFont(PDType1Font.HELVETICA, 48);
                cs.setNonStrokingColor(200, 200, 200); // light gray

                // Rotate 45 degrees around page center
                float cx = page.getMediaBox().getWidth() / 2;
                float cy = page.getMediaBox().getHeight() / 2;
                cs.transform(Matrix.getRotateInstance(
                    Math.PI / 4, cx, cy));

                cs.beginText();
                cs.newLineAtOffset(cx - 100, cy); // approximate centering
                cs.showText("CONFIDENTIAL");
                cs.endText();
                cs.close();
            }

            doc.save("watermarked_pdfbox.pdf");
        }
        finally
        {
            doc.close();
        }
        Console.WriteLine("Watermarked with the PDFBox .NET port.");
    }
}

IronPDF Watermark (C#)

How Do I Add Password Protection After Migrating?

Encryption through the PDFBox port requires creating AccessPermission and StandardProtectionPolicy objects — and on abandoned ports, verifying that the encryption code path is actually wired up in the version you are on. IronPDF exposes the same controls through a single SecuritySettings property.

Apache PDFBox Port Password Protection (C#)

using org.apache.pdfbox.pdmodel;
using org.apache.pdfbox.pdmodel.encryption;
using java.io;
using System;

class PasswordPdfbox
{
    static void Main()
    {
        PDDocument doc = PDDocument.load(new File("input.pdf"));
        try
        {
            AccessPermission perms = new AccessPermission();
            perms.setCanPrint(false);
            perms.setCanModify(false);
            perms.setCanExtractContent(false);

            StandardProtectionPolicy policy = new StandardProtectionPolicy(
                "owner123",  // owner password
                "user456",   // user password
                perms
            );
            policy.setEncryptionKeyLength(128);

            doc.protect(policy);
            doc.save("protected_pdfbox.pdf");
        }
        finally
        {
            doc.close();
        }
        Console.WriteLine("Protected with the PDFBox .NET port.");
    }
}

IronPDF Password Protection (C#)

using IronPdf;

class PasswordIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var pdf = PdfDocument.FromFile("input.pdf");

        pdf.SecuritySettings.OwnerPassword = "owner123";
        pdf.SecuritySettings.UserPassword = "user456";
        pdf.SecuritySettings.AllowUserPrinting = IronPdf.Security.PdfPrintSecurity.NoPrint;
        pdf.SecuritySettings.AllowUserCopyPasteContent = false;
        pdf.SecuritySettings.AllowUserEdits = IronPdf.Security.PdfEditSecurity.NoEdit;

        pdf.SaveAs("protected.pdf");
        Console.WriteLine("Protected with IronPDF.");
    }
}

No AccessPermission + StandardProtectionPolicy ceremony. See IronPDF security documentation.

What Are the Complete API Mappings Between Apache PDFBox and IronPDF?

These map the PDFBox .NET port API to IronPDF. Both sides are C#, so the change is API-shape and idiom rather than language. The IKVM ports preserve Java's lowercase package names (org.apache.pdfbox.*); MASES.NetPDF title-cases them (Org.Apache.Pdfbox.*). Method names stay lowercase on both because the underlying Java API is exposed directly.

Namespace Mapping

  • org.apache.pdfbox.pdmodel → IronPdf — core document model
  • org.apache.pdfbox.pdmodel.font → CSS font-family in HTML — font handling
  • org.apache.pdfbox.multipdf → IronPdf — merge/split utilities
  • org.apache.pdfbox.text → IronPdf — text extraction
  • org.apache.pdfbox.pdmodel.encryption → IronPdf.Security — security/encryption

Core Class Mapping

  • PDDocument → PdfDocument — the PDF document object
  • PDPage → pdf.Pages[n] — individual page access
  • PDPageContentStream → HTML/CSS via ChromePdfRenderer — IronPDF uses HTML instead of drawing commands
  • PDFMergerUtility → PdfDocument.Merge() — document merging
  • PDFTextStripper → pdf.ExtractAllText() — text extraction

Document Loading Methods

  • Open from file: PDDocument.load(new File("f.pdf")) → PdfDocument.FromFile("f.pdf")
  • Open with password: PDDocument.load(new File("f.pdf"), "pass") → PdfDocument.FromFile("f.pdf", "pass")
  • Create new: new PDDocument() → renderer.RenderHtmlAsPdf(html)
  • Save: document.save("out.pdf") → pdf.SaveAs("out.pdf")
  • Close: document.close() → using statement / Dispose()

Page Operations

  • Get page count: document.getNumberOfPages() → pdf.PageCount
  • Get page: document.getPage(index) → pdf.Pages[index] — both 0-based
  • Add page: document.addPage(new PDPage()) → render HTML, pages are auto-created
  • Remove page: document.removePage(index) → pdf.RemovePages(index)

Merge / Split Operations

  • Merge: PDFMergerUtility → addSource() → mergeDocuments() becomes PdfDocument.Merge(pdf1, pdf2)
  • Extract pages: document.getPage() + new doc + import page becomes pdf.CopyPages(start, end)

What Should I Watch Out for During Migration?

Abandoned ports. Three of the four .NET ports — Pdfbox (2013), Pdfbox-IKVM (2017), PdfBox_DotNet_Version (2019) — have not shipped a release in years. The active option, MASES.NetPDF, requires a real JVM at runtime via JCOBridge. Plan for some archeology when you encounter Java exceptions surfacing through IKVM stack traces.

Page indexing. Both PDFBox and IronPDF use 0-based page indexing. This is the one thing that does not change.

Text extraction ordering. PDFBox's PDFTextStripper extracts text in content stream order by default, which can produce scrambled output. You must call stripper.setSortByPosition(true) for visual reading order. IronPDF's text extraction may handle ordering differently — verify against your specific documents.

Resource management. The PDFBox port's PDDocument follows the Java close() contract — you call document.close() explicitly inside a try/finally. IronPDF's PdfDocument implements IDisposable, so the natural pattern is a using block. The verbosity drops noticeably.

Encryption parity. PDFBox uses Bouncy Castle for encryption upstream. The IKVM ports embed it; MASES.NetPDF calls into the real Java implementation. IronPDF handles encryption internally and consistently across releases.

Eliminating the JVM dependency. Replacing the PDFBox port with IronPDF removes either an IKVM-bundled JVM reimplementation or a JCOBridge dependency on a real JVM from your deployment:

// Before: PDFBox via IKVM port, with java.io.File and Java exceptions
PDDocument doc = PDDocument.load(new File("input.pdf"));
try { /* ... */ }
finally { doc.close(); }

// After: IronPDF, native .NET
using var pdf = PdfDocument.FromFile("input.pdf");
var merged = PdfDocument.Merge(pdf, otherPdf);

Renderer reuse. Create one ChromePdfRenderer and reuse it:

private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

Disposal. Always wrap PdfDocument instances in using blocks:

using var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");

What Edge Cases Should I Test?

Text extraction quality. PDFBox's PDFTextStripper is well-regarded for extraction quality. Test IronPDF's text extraction against your specific document corpus. For extraction-heavy workloads, also consider PdfPig, a .NET-native PDF text extraction library that works without IronPDF.

PDF/A validation. PDFBox's Preflight module validates PDF/A compliance. IronPDF generates PDF/A but may not validate existing documents -- if you need validation, consider keeping Preflight as a separate tool or using veraPDF.

Large file memory. PDFBox offers MemoryUsageSetting to control memory vs. temp-file trade-offs for large documents. IronPDF handles this internally -- test with your largest production documents.

Frequently Asked Questions

Is Apache PDFBox free and IronPDF paid?

Yes. Apache PDFBox is free under the Apache License v2.0, and the community .NET ports inherit that. IronPDF uses per-developer perpetual licenses. The cost trade-off is not just the license fee — it is the time spent working around abandoned ports, debugging Java exceptions surfacing through IKVM, and managing a JVM dependency in a .NET deployment.

Many teams find the engineering hours saved on a fully supported library outweigh the license cost. A trial key lets you evaluate before committing.

Can I migrate incrementally or do I have to replace everything at once?

You can migrate incrementally. Start with the highest-value operation — typically HTML-to-PDF, since none of the PDFBox ports support that at all. Leave lower-priority operations (like text extraction) running through the port while you validate IronPDF's output.

Each operation you migrate removes one dependency on the community port. You do not need to remove the PDFBox NuGet package until every operation has been ported.

What happens to my PDFBox Preflight (PDF/A validation) workflows?

Upstream PDFBox has a Preflight module for PDF/A validation. The .NET ports do not consistently expose it, so if you were already using Preflight you were probably calling it through some out-of-process bridge. IronPDF supports PDF/A export but may not validate existing documents — verify against your validation requirements; if validation is critical, consider veraPDF or keep an external validator.

This is one area where the migration is not a clean one-to-one swap. Audit your PDF/A workflows before migrating.

How do I handle text extraction differences between PDFBox and IronPDF?

PDFBox's PDFTextStripper extracts text in content stream order by default, which is not necessarily visual order. You must explicitly call setSortByPosition(true) for reading-order output. IronPDF may handle ordering differently.

Test with at least 10 diverse documents from your production corpus. Compare output line by line. For extraction-heavy workloads, PdfPig is a .NET-native alternative worth evaluating alongside IronPDF.

Do I need to change my page indexing code?

No. Both PDFBox and IronPDF use 0-based page indexing. This is one of the few areas where the migration requires zero changes beyond the API surface swap.

The method names change — document.getPage(index) becomes pdf.Pages[index] — but the index values stay the same.

What is the recommended migration scenario for my team?

It depends on your current integration pattern. If you are running one of the abandoned ports (Pdfbox, Pdfbox-IKVM, PdfBox_DotNet_Version) for any non-trivial PDF work, migrate — the lack of maintenance is a long-term liability. If you are on MASES.NetPDF and want to drop the JVM dependency, migrate. If your .NET stack needs HTML-to-PDF plus manipulation, migrate — IronPDF covers both in one NuGet package.

If the port is used only for text extraction, evaluate carefully. IronPDF extracts text, but you might also consider lighter .NET-native alternatives like PdfPig. If you depend on PDFBox Preflight for PDF/A validation, keep an external validator like veraPDF.

Conclusion

The real migration here is not language-to-language. Both sides are C#. It is moving from an unofficial .NET port — IKVM-bundled or JCOBridge-wrapped — to a fully supported native .NET library. The code translation is mostly idiom: PDDocument.load(new File(...)) becomes PdfDocument.FromFile, content-stream drawing becomes HTML, and the Java-shaped surface gives way to property-based C#.

If you stay on a PDFBox port, plan for the same friction across the board: no HTML-to-PDF, abandoned NuGet packages, and a JVM dependency one way or another. If you stay in .NET but want a free option for the narrow case of text extraction, PdfPig handles that in pure C# under Apache 2.0, though it does not do HTML-to-PDF either.

For teams that need HTML rendering, merging, watermarking, and encryption in one package with vendor support, IronPDF consolidates everything into a single NuGet install. Start with a trial key and migrate one operation at a time.
