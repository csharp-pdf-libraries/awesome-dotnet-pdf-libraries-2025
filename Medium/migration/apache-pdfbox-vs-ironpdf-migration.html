Migrating from Apache PDFBox to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

The pull request sat open for three weeks. Someone had added a seven-line method to merge two PDFs using PdfBoxDotNet, the community .NET port of Apache PDFBox. It compiled. It even ran. But on the third invoice the team tried, PDFMergerUtility.MergeDocuments() threw a NotImplementedException on a code path the upstream Java library handles fine. The port was years behind. Nobody on the team owned it. And the original reason it was chosen -- "PDFBox is free" -- was about the Java library, not the half-finished bindings the .NET app was actually calling.

If that sounds familiar -- or if you are running an Apache PDFBox community port that drops methods between releases, ships without async support, and offers no commercial backing when something breaks -- this guide is for you.

Apache PDFBox itself is a solid open-source Java library. The migration question is not about the upstream project. It is about whether an unofficial, incomplete .NET port costs more than it saves once your application grows past the trivial cases. This guide covers the shift to IronPDF, but eighty percent of the framework -- dependency auditing, API mapping, testing patterns -- applies to any .NET-native PDF library.

Why Should I Migrate from Apache PDFBox to IronPDF?

Upstream Apache PDFBox is one of the most widely deployed PDF libraries in the world. Here is why .NET teams specifically move away from its community .NET ports:

Unofficial, incomplete community port. Apache does not publish a .NET build. Packages like PdfBoxDotNet are community ports of the Java source. They lag the upstream releases, skip features the maintainers have not gotten to yet, and ship with methods that compile but throw at runtime.

No HTML-to-PDF. PDFBox does not convert HTML, and the .NET ports inherit that gap. If you need HTML rendering, you are pairing the port with a second library -- the .cs samples in this repo flag this explicitly.

API surface gaps between releases. Because the port is reverse-engineered from Java, classes and method signatures shift between versions. Code that worked against last year's NuGet package may not compile against this year's, and the diff is not always documented.

No commercial support. When the port misbehaves, there is no vendor to file a ticket with. Issues sit on a community GitHub for months. For a library that sits underneath an invoicing or compliance pipeline, that is a hard risk to carry.

API verbosity for common operations. PDFBox's API mirrors the PDF specification, which makes it powerful but verbose. Merging two PDFs requires loading both documents, creating a PDFMergerUtility, adding sources, and calling MergeDocuments().

Manual resource management. PDDocument in the port follows the Java close() contract. It implements IDisposable, but the API still exposes Java-style lifecycle calls, and forgetting them leaks file handles.

Text extraction ordering. PDFBox extracts text in the order it appears in the PDF content stream, which is not necessarily visual order. Sorting requires explicit configuration -- a common source of bugs for teams unfamiliar with PDF internals.

No async support. The ported API is synchronous. In an async .NET pipeline, blocking on it ties up thread pool threads.

Font handling across platforms. The port relies on whatever font discovery the underlying JDK-style code did. Font availability and rendering can differ between your dev machine and your production Linux container.

Java-flavored API in a C# project. Even after the port translates packages to namespaces, the surface still feels Java: PDDocument, PDFTextStripper, setSortByPosition-style helpers. New .NET hires read it as foreign code.

How Do Apache PDFBox and IronPDF Compare?

The community .NET port of PDFBox focuses on PDF manipulation, text extraction, and form filling. Its API mirrors the PDF specification with classes like PDDocument and PDPage exposed through the PdfBoxDotNet.Pdmodel, PdfBoxDotNet.Text, and PdfBoxDotNet.Multipdf namespaces -- powerful, but it assumes you know PDF internals, and the port itself is incomplete relative to upstream Apache PDFBox. It is free under Apache License v2.0 and has no HTML rendering capability.

IronPDF takes the opposite approach: HTML-first PDF generation plus manipulation, exposed through a property-based C# API in the IronPdf namespace. The learning curve is low for any developer who already knows HTML and CSS, since rendering goes through a built-in Chrome engine. Licensing is per-developer with perpetual options available.

A few specifics worth noting up front: both libraries use 0-based page indexing, so index values stay the same when you migrate. PDDocument in the PDFBox port is not safe to share across threads, while IronPDF's ChromePdfRenderer is stateless and safe to share.

How Much Effort Does Each Feature Require to Migrate?

  • HTML to PDF — Low. PDFBox cannot do this; IronPDF is purpose-built for it.
  • Text extraction — Medium. Both extract text; API shape and sort behavior differ.
  • Merge — Low. Both offer merge; IronPDF is a one-liner via PdfDocument.Merge().
  • Split / extract pages — Low. Different API, same concept (pdf.CopyPages() in IronPDF).
  • Watermark — Medium. PDFBox uses PDPageContentStream drawing; IronPDF uses HTML overlay via ApplyWatermark.
  • Password / encryption — Medium. PDFBox uses AccessPermission + StandardProtectionPolicy; IronPDF uses SecuritySettings.
  • Form filling (AcroForms) — Medium. Both support AcroForms with different API surfaces.
  • Digital signatures — Medium-High. Different libraries and API patterns.
  • PDF/A validation — High / N/A. PDFBox has Preflight for PDF/A validation; IronPDF supports PDF/A export but validation parity is not guaranteed -- verify before relying on it.
  • Image extraction — Medium. Both support image extraction with different APIs.
  • PDF rendering to image — Medium. PDFBox renders pages to BufferedImage; IronPDF rasterizes via ToBitmap(dpi).

Step 1: License Configuration

PDFBox is Apache-licensed, so the community .NET port has no license key. But you do inherit whatever the port maintainers have implemented (and not implemented) in the current NuGet package. IronPDF requires a single license key line and ships as a fully supported .NET library.

// Apache PDFBox has no official .NET port; this is a community port.
// NuGet: Install-Package PdfBoxDotNet (community-maintained, incomplete)

Step 3: Basic PDF Creation

PDFBox-dotnet PDF Creation (C#, ~20 lines)

using PdfBoxDotNet.Pdmodel;
using PdfBoxDotNet.Pdmodel.Common;
using PdfBoxDotNet.Pdmodel.Font;
using System;

class CreatePdf
{
    static void Main()
    {
        using (var document = new PDDocument())
        {
            var page = new PDPage(PDRectangle.LETTER);
            document.AddPage(page);

            var content = new PDPageContentStream(document, page);
            content.BeginText();
            content.SetFont(PDType1Font.HELVETICA, 24);
            content.NewLineAtOffset(50, 700); // x, y from bottom-left
            content.ShowText("Monthly Report");
            content.EndText();
            content.Close();

            document.Save("report_pdfbox.pdf");
        }
        Console.WriteLine("Saved with PDFBox-dotnet.");
    }
}

IronPDF PDF Creation (C#, 10 lines)

using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var renderer = new ChromePdfRenderer();
        renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.Letter;

        var pdf = renderer.RenderHtmlAsPdf(
            "<h1 style='font-family:Helvetica;font-size:24pt;'>Monthly Report</h1>");

        pdf.SaveAs("report_ironpdf.pdf");

}

How Do I Convert HTML to PDF After Migrating?

The PDFBox-dotnet port inherits PDFBox's lack of HTML-to-PDF support -- the .cs samples in this repo flag it explicitly: "Community ports like PDFBox-dotnet are incomplete and do not support HTML to PDF conversion natively." The workaround is shelling out to an external tool like wkhtmltopdf, or pairing the port with a separate HTML renderer. With IronPDF, HTML rendering is built in.

PDFBox-dotnet HTML-to-PDF Workaround (C#)

// Apache PDFBox does not have an official .NET port.
// Community ports like PDFBox-dotnet are incomplete and
// do not support HTML to PDF conversion natively.
// The usual workaround: shell out to an external renderer.
using System;
using System.Diagnostics;

class HtmlToPdfPdfbox
{
    static void Main()
    {
        // Option A: Shell out to wkhtmltopdf
        var psi = new ProcessStartInfo("wkhtmltopdf",
            "--page-size Letter --margin-top 20 --margin-bottom 20 input.html report.pdf")
        {
            RedirectStandardError = true,
            UseShellExecute = false
        };
        using (var process = Process.Start(psi))
        {
            process.WaitForExit();
            if (process.ExitCode != 0)
            {
                Console.Error.WriteLine("wkhtmltopdf failed: " + process.ExitCode);
                return;
            }
        }

        // Option B: Then post-process with PDFBox-dotnet if needed
        // using (var doc = PDDocument.Load("report.pdf")) {
        //     // ... manipulate ...
        //     doc.Save("report_final.pdf");
        // }

        Console.WriteLine("Generated with wkhtmltopdf + PDFBox-dotnet.");
    }
}

How Do I Merge PDFs After Migrating?

PDFBox-dotnet merging requires creating a PDFMergerUtility, adding sources, and calling MergeDocuments() -- and on the port specifically, that call has been a known sore spot when API coverage drifts. IronPDF reduces this to a single static method.

PDFBox-dotnet Merge (C#)

// Apache PDFBox .NET port attempt (incomplete support)
using PdfBoxDotNet.Pdmodel;
using PdfBoxDotNet.Multipdf;
using System;

class MergePdfbox
{
    static void Main()
    {
        // PDFBox-dotnet ports have incomplete API coverage
        var merger = new PDFMergerUtility();

        merger.AddSource("part1.pdf");
        merger.AddSource("part2.pdf");
        merger.SetDestinationFileName("merged_pdfbox.pdf");

        merger.MergeDocuments();

        Console.WriteLine("Merged with PDFBox-dotnet.");
    }
}

IronPDF Merge (C#)

using IronPdf;

class MergeIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var pdf1 = PdfDocument.FromFile("part1.pdf");
        var pdf2 = PdfDocument.FromFile("part2.pdf");

        var merged = PdfDocument.Merge(pdf1, pdf2);
        merged.SaveAs("merged.pdf");
        Console.WriteLine("Merged with IronPDF.");
    }
}

No merger utility class. No incomplete-port roulette. See IronPDF merge documentation.

How Do I Apply Watermarks After Migrating?

Watermarking through PDFBox-dotnet requires content stream operations, matrix transforms, and per-page loops -- assuming the port has implemented every helper you reach for. IronPDF replaces all of that with a single HTML string.

PDFBox-dotnet Watermark (C#, content stream drawing)

using PdfBoxDotNet.Pdmodel;
using PdfBoxDotNet.Pdmodel.Font;
using PdfBoxDotNet.Util;
using System;

class WatermarkPdfbox
{
    static void Main()
    {
        using (var doc = PDDocument.Load("input.pdf"))
        {
            foreach (var page in doc.GetPages())
            {
                var cs = new PDPageContentStream(
                    doc, page, PDPageContentStream.AppendMode.APPEND, true, true);

                cs.SetFont(PDType1Font.HELVETICA, 48);
                cs.SetNonStrokingColor(200, 200, 200); // light gray

                // Rotate 45 degrees around page center
                float cx = page.GetMediaBox().GetWidth() / 2;
                float cy = page.GetMediaBox().GetHeight() / 2;
                cs.Transform(Matrix.GetRotateInstance(
                    Math.PI / 4, cx, cy));

                cs.BeginText();
                cs.NewLineAtOffset(cx - 100, cy); // approximate centering
                cs.ShowText("CONFIDENTIAL");
                cs.EndText();
                cs.Close();
            }

            doc.Save("watermarked_pdfbox.pdf");
        }
        Console.WriteLine("Watermarked with PDFBox-dotnet.");
    }
}

IronPDF Watermark (C#)

using IronPdf;

How Do I Add Password Protection After Migrating?

Encryption through PDFBox-dotnet requires creating AccessPermission and StandardProtectionPolicy objects -- and verifying that the encryption code path is actually wired up in the version of the port you're on. IronPDF exposes the same controls through a single SecuritySettings property.

PDFBox-dotnet Password Protection (C#)

using PdfBoxDotNet.Pdmodel;
using PdfBoxDotNet.Pdmodel.Encryption;
using System;

class PasswordPdfbox
{
    static void Main()
    {
        using (var doc = PDDocument.Load("input.pdf"))
        {
            var perms = new AccessPermission();
            perms.SetCanPrint(false);
            perms.SetCanModify(false);
            perms.SetCanExtractContent(false);

            var policy = new StandardProtectionPolicy(
                "owner123",  // owner password
                "user456",   // user password
                perms
            );
            policy.SetEncryptionKeyLength(128);

            doc.Protect(policy);
            doc.Save("protected_pdfbox.pdf");
        }
        Console.WriteLine("Protected with PDFBox-dotnet.");
    }
}

IronPDF Password Protection (C#)

using IronPdf;

class PasswordIronPdf
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        var pdf = PdfDocument.FromFile("input.pdf");

        pdf.SecuritySettings.OwnerPassword = "owner123";
        pdf.SecuritySettings.UserPassword = "user456";
        pdf.SecuritySettings.AllowUserPrinting = IronPdf.Security.PdfPrintSecurity.NoPrint;
        pdf.SecuritySettings.AllowUserCopyPasteContent = false;
        pdf.SecuritySettings.AllowUserEdits = IronPdf.Security.PdfEditSecurity.NoEdit;

        pdf.SaveAs("protected.pdf");
        Console.WriteLine("Protected with IronPDF.");
    }
}

No AccessPermission + StandardProtectionPolicy ceremony. See IronPDF security documentation.

What Are the Complete API Mappings Between PDFBox-dotnet and IronPDF?

These map the PdfBoxDotNet community-port API to IronPDF. Both sides are C#, so the change is API-shape and idiom rather than language.

Namespace Mapping

  • PdfBoxDotNet.Pdmodel → IronPdf — core document model
  • PdfBoxDotNet.Pdmodel.Font → CSS font-family in HTML — font handling
  • PdfBoxDotNet.Multipdf → IronPdf — merge/split utilities
  • PdfBoxDotNet.Text → IronPdf — text extraction

Core Class Mapping

  • PDDocument → PdfDocument — the PDF document object
  • PDPage → pdf.Pages[n] — individual page access
  • PDPageContentStream → HTML/CSS via ChromePdfRenderer — IronPDF uses HTML instead of drawing commands
  • PDFMergerUtility → PdfDocument.Merge() — document merging
  • PDFTextStripper → pdf.ExtractAllText() — text extraction

Document Loading Methods

  • Open from file: PDDocument.Load("f.pdf") → PdfDocument.FromFile("f.pdf")
  • Open with password: PDDocument.Load("f.pdf", "pass") → PdfDocument.FromFile("f.pdf", "pass")
  • Create new: new PDDocument() → renderer.RenderHtmlAsPdf(html)
  • Save: document.Save("out.pdf") → pdf.SaveAs("out.pdf")

Page Operations

  • Get page count: document.GetNumberOfPages() → pdf.PageCount
  • Get page: document.GetPage(index) → pdf.Pages[index] — both 0-based
  • Add page: document.AddPage(new PDPage()) → render HTML, pages are auto-created
  • Remove page: document.RemovePage(index) → pdf.RemovePages(index)

Merge / Split Operations

  • Merge: PDFMergerUtility → AddSource() → MergeDocuments() becomes PdfDocument.Merge(pdf1, pdf2)
  • Extract pages: document.GetPage() + new doc + import page becomes pdf.CopyPages(start, end)

What Should I Watch Out for During Migration?

API surface gaps in the port. Unlike migrating between two well-supported .NET libraries, you are leaving a community port whose published API does not always match what is actually implemented. Plan for some NotImplementedException traces during code archeology, especially around encryption, signatures, and AcroForm operations.

Page indexing. Both PDFBox and IronPDF use 0-based page indexing. This is the one thing that does not change.

Text extraction ordering. PDFBox's PDFTextStripper extracts text in content stream order by default, which can produce scrambled output. You must set SortByPosition = true (or call the equivalent setter the port exposes) for visual reading order. IronPDF's text extraction may handle ordering differently -- test with your specific documents.

Resource management. PDFBox-dotnet's PDDocument still follows the Java close() contract -- the .cs samples in this repo wrap it in a using block. IronPDF's PdfDocument also implements IDisposable. The pattern maps naturally; the verbosity drops.

Encryption parity. PDFBox uses Bouncy Castle for encryption upstream. The .NET port wires through whatever its current build supports, which has historically been a moving target. IronPDF handles encryption internally and consistently across releases.

Eliminating the unsupported dependency. Replacing PDFBox-dotnet with IronPDF removes a community-maintained NuGet package from your dependency tree:

// Before: PdfBoxDotNet community port (incomplete API coverage)
using (var doc = PDDocument.Load("input.pdf"))
{
    // ... operations may or may not be implemented in your version
}

// After: IronPDF, fully supported
using var pdf = PdfDocument.FromFile("input.pdf");
var merged = PdfDocument.Merge(pdf, otherPdf);

Renderer reuse. Create one ChromePdfRenderer and reuse it:

private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

Disposal. Always wrap PdfDocument instances in using blocks:

using var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");

What Edge Cases Should I Test?

Text extraction quality. PDFBox's PDFTextStripper is well-regarded for extraction quality. Test IronPDF's text extraction against your specific document corpus. For extraction-heavy workloads, also consider PdfPig, a .NET-native PDF text extraction library that works without IronPDF.

PDF/A validation. PDFBox's Preflight module validates PDF/A compliance. IronPDF generates PDF/A but may not validate existing documents -- if you need validation, consider keeping Preflight as a separate tool or using veraPDF.

Large file memory. PDFBox offers MemoryUsageSetting to control memory vs. temp-file trade-offs for large documents. IronPDF handles this internally -- test with your largest production documents.

Frequently Asked Questions

Is Apache PDFBox free and IronPDF paid?

Yes. Apache PDFBox is free under the Apache License v2.0, and the community .NET port inherits that. IronPDF uses per-developer perpetual licenses. The cost trade-off is not just the license fee -- it is the time spent working around an incomplete port, debugging methods that throw NotImplementedException, and tracking community-driven release cadence.

Many teams find the engineering hours saved on a fully supported library outweigh the license cost. A trial key lets you evaluate before committing.

Can I migrate incrementally or do I have to replace everything at once?

You can migrate incrementally. Start with the highest-value operation -- typically HTML-to-PDF, since the PDFBox port cannot do this at all. Leave lower-priority operations (like text extraction) running through the port while you validate IronPDF's output.

Each operation you migrate removes one dependency on the community port. You do not need to remove the PDFBox-dotnet NuGet package until every operation has been ported.

What happens to my PDFBox Preflight (PDF/A validation) workflows?

Upstream PDFBox has a Preflight module for PDF/A validation. The .NET ports typically do not include it, so if you were already using Preflight you were probably calling it through some out-of-process bridge. IronPDF supports PDF/A export but may not validate existing documents -- if validation is critical, consider veraPDF or keep an external validator.

This is one area where the migration is not a clean one-to-one swap. Audit your PDF/A workflows before migrating.

How do I handle text extraction differences between PDFBox and IronPDF?

PDFBox's PDFTextStripper extracts text in content stream order by default, which is not necessarily visual order. You must explicitly enable position-based sorting for reading-order output. IronPDF may handle ordering differently.

Test with at least 10 diverse documents from your production corpus. Compare output line by line. For extraction-heavy workloads, PdfPig is a .NET-native alternative worth evaluating alongside IronPDF.

Do I need to change my page indexing code?

No. Both PDFBox and IronPDF use 0-based page indexing. This is one of the few areas where the migration requires zero changes beyond the API surface swap.

The method names change -- document.GetPage(index) becomes pdf.Pages[index] -- but the index values stay the same.

What is the recommended migration scenario for my team?

It depends on your current integration pattern. If you are running PDFBox-dotnet for any non-trivial PDF work, migrate -- the port's incomplete coverage is a long-term liability. If your full .NET stack needs HTML-to-PDF plus manipulation, migrate -- IronPDF covers both in one NuGet package.

If the port is used only for text extraction, evaluate carefully. IronPDF extracts text, but you might also consider lighter .NET-native alternatives like PdfPig. If you depend on PDFBox Preflight for PDF/A validation, keep an external validator like veraPDF.

Conclusion

The real migration here is not language-to-language. Both sides are C#. It is moving from an unofficial, incomplete community port to a fully supported .NET library. The code translation is mostly idiom: PDDocument.Load becomes PdfDocument.FromFile, content-stream drawing becomes HTML, and the Java-shaped API gives way to a property-based C# surface.

If you stay on the PDFBox-dotnet port, plan for the same friction the .cs samples in this repo flag explicitly: incomplete API coverage, no HTML-to-PDF, no commercial backing. If you stay in .NET but want a free option for the narrow case of text extraction, PdfPig handles that in pure C# under Apache 2.0, though it does not do HTML-to-PDF either.

For teams that need HTML rendering, merging, watermarking, and encryption in one package with vendor support, IronPDF consolidates everything into a single NuGet install. Start with a trial key and migrate one operation at a time.
