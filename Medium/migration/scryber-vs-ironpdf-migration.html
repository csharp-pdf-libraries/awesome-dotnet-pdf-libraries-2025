Migrating from Scryber to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate from Scryber.Core to IronPDF?

License renewal usually starts this conversation. The bill shows up, someone checks the NuGet page for the latest version, and they find the last release was a long time ago. Or the issue tracker has open bugs that nobody has touched in months. Scryber.Core is a capable open-source PDF generation library that renders an HTML/CSS subset (and an XML template flavor) to PDF, but before committing further, teams often do a quick evaluation of alternatives. That is typically how this migration begins.

Here are the reasons I see most often when teams evaluate Scryber.Core alternatives.

First, maintenance cadence. Slower update cycles mean lag on .NET version support. You should verify the current release frequency before making assumptions.

Second, CSS subset limitations. Scryber.Core implements its own HTML/CSS parser, not a browser engine. Modern features like Flexbox, Grid, custom properties, and most JavaScript-driven layouts behave differently than they would in Chrome — or not at all.

Third, no JavaScript execution. Scryber renders static markup. Anything that depends on client-side scripting to populate the DOM will not appear in the output.

Fourth, community size. A smaller community means fewer Stack Overflow answers and fewer community examples to draw from when you hit edge cases.

Fifth, missing manipulation features. Merge, split, watermark, security, and text extraction commonly require supplementary libraries on top of Scryber.Core.

Sixth, no live URL rendering. Scryber generates from an HTML string or template. To render a live page you fetch the HTML yourself with HttpClient first, which means no JavaScript, no follow-up XHR calls, and no images that load after the initial response.

Seventh, documentation gaps. Complex layout scenarios may have limited examples in the official docs.

Eighth, font handling. Custom font registration in Scryber.Core has its own configuration requirements that differ from standard web font workflows.

Ninth, error messaging. Template and binding errors can surface as opaque runtime exceptions that are difficult to debug.

Comparison Table

Scryber.Core focuses on rendering an HTML/CSS subset (plus an XML template variant) into a PDF. IronPDF focuses on full HTML-to-PDF via an embedded Chromium plus a broad set of PDF manipulation features. Scryber is open source under LGPL, which can require source disclosure in some commercial scenarios; IronPDF is commercially licensed — verify current pricing at ironsoftware.com.

API style differs in tone, not in surface area. Scryber takes an HTML string (or XML template) and returns a document object you save with SaveAsPDF. IronPDF instantiates a ChromePdfRenderer, hands it HTML or a URL, and returns a PdfDocument you can save, edit, merge, secure, or stamp. The core namespaces are Scryber.Core / Scryber.Core.Html / Scryber.Drawing on one side and IronPdf on the other.

The learning curve is the most honest difference. Scryber's CSS is a subset, so styling tends to be a process of finding what works; IronPDF rendering matches what your team already sees in Chrome DevTools. Page indexing and concurrent-instance guidance for Scryber are worth confirming in its current docs against your version.

What Does the Migration Effort Actually Look Like?

I want to give you a realistic picture of the work involved. Most Scryber.Core migrations are mainly an API swap plus a CSS pass — not a wholesale template rewrite — because Scryber already accepts HTML strings via Document.ParseDocument(html, ParseSourceType.DynamicContent). If your project also uses XML .pdfx templates, those need to be redone in HTML. Here is the feature-by-feature breakdown.

Effort by Feature

  • HTML to PDF — Low. Document.ParseDocument(html, ParseSourceType.DynamicContent) becomes ChromePdfRenderer.RenderHtmlAsPdf(html). Mostly a one-liner change.
  • Save to file — Low. doc.SaveAsPDF(path) becomes pdf.SaveAs(path).
  • Save to stream — Low. doc.SaveAsPDF(stream) becomes pdf.Stream or pdf.BinaryData.
  • XML .pdfx templates — Medium. Scryber's XML document model has no IronPDF equivalent. Each .pdfx file is rewritten as an HTML template. There is no automated converter.
  • Data binding — Medium. Scryber's doc.Params["key"] = value and {{model.X}} binding becomes C# string interpolation, Razor, or a template engine like Scriban.
  • Custom page size — Low. doc.RenderOptions.PaperSize = PaperSize.A4 becomes renderer.RenderingOptions.PaperSize = PdfPaperSize.A4.
  • CSS styling — Medium. Scryber's CSS subset becomes full Chromium CSS. Most of this work is removing workarounds, not adding them.
  • Merge PDFs — Medium. Native in IronPDF via PdfDocument.Merge(); in Scryber it typically means a second library.
  • Watermark — Medium. TextStamper / ImageStamper programmatically; in Scryber it lives in the template or needs a secondary library.
  • Password protection — Medium. pdf.SecuritySettings in IronPDF; verify current Scryber security API surface in your version.
  • Headers/footers — Medium. RenderingOptions.HtmlHeader / HtmlFooter in IronPDF; XML header/footer sections in Scryber templates.
  • Text extraction — Medium. pdf.ExtractAllText() in IronPDF; verify Scryber support if you rely on it.

Decision Matrix

  • If your project is HTML-string-driven and your CSS is well within Scryber's subset, the migration is a straight API swap and you mostly gain manipulation features.
  • If you have XML .pdfx templates, evaluate the rewrite cost first — Scryber templates do not auto-convert to HTML.
  • If you need full modern CSS (flex, grid, custom properties) or any client-side JavaScript to run before rendering, switch — Chromium covers both; Scryber covers neither.
  • If you need to render live web pages with images, scripts, and async XHR, switch — IronPDF's RenderUrlAsPdf uses a real browser.
  • If you need programmatic PDF manipulation (merge, security, annotations, stamping), switch — IronPDF covers all of these natively.

Step 3: Basic Conversion

Before (Scryber.Core):

using Scryber.Core;
using Scryber.Core.Html;
using System;

class Program
{
    static void Main()
    {
        string html = "<html><body><h1>Q3 2024 Report</h1>" +
                      $"<p>Generated: {DateTime.UtcNow:yyyy-MM-dd}</p></body></html>";

        using (var doc = Document.ParseDocument(html, ParseSourceType.DynamicContent))
        {
            doc.SaveAsPDF("output.pdf");
        }

    }
}

After:

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Same HTML input — Chromium replaces Scryber's HTML/CSS subset parser
var html = $@"
    <html>
    <head>
    <style>
        body {{ font-family: Arial, sans-serif; padding: 40px; }}
        h1 {{ font-size: 22px; }}
        .meta {{ color: #666; font-size: 12px; }}
    </style>
    </head>
    <body>
        <h1>Q3 2024 Report</h1>
        <div class='meta'>Generated: {DateTime.UtcNow:yyyy-MM-dd}</div>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");

Console.WriteLine($"Saved output.pdf ({pdf.PageCount} page(s))");

HTML to PDF (Invoice Example)

Before (Scryber.Core):

using Scryber.Core;
using Scryber.Core.Html;
using System;

class InvoiceBefore
{
    static void Main()
    {
        var customerName = "Acme Corp";
        var invoiceNumber = "INV-2024-0042";
        var totalAmount = "$4,200.00";
        var dueDate = DateTime.UtcNow.AddDays(30).ToString("yyyy-MM-dd");

        // String interpolation feeds Scryber's HTML/CSS subset
        string html = $@"
            <html><body style='font-family: Arial, sans-serif; padding: 40px;'>
                <div style='font-size: 22px; font-weight: bold;'>Invoice {invoiceNumber}</div>
                <div>Customer: {customerName}</div>
                <div>Due Date: {dueDate}</div>
                <div style='font-size: 16px; font-weight: bold;'>Total: {totalAmount}</div>
            </body></html>";

        using (var doc = Document.ParseDocument(html, ParseSourceType.DynamicContent))
        {
            doc.SaveAsPDF("invoice.pdf");
        }

        Console.WriteLine("Saved invoice.pdf");
    }
}

After:

using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

Merge PDFs

Before (Scryber.Core):

using Scryber.Core;
using Scryber.Core.Html;
using System;
using System.Collections.Generic;
using System.IO;

class MergeBefore
{
    static void Main()
    {
        // Scryber.Core does not expose a native cross-document merge.
        // Teams typically render each section, then add a secondary library
        // (PdfSharp, iTextSharp, etc.) to combine the resulting bytes.

        var sections = new[] { BuildSection1Html(), BuildSection2Html() };
        var pdfBytes = new List<byte[]>();

        foreach (var html in sections)
        {
            using var doc = Document.ParseDocument(html, ParseSourceType.DynamicContent);
            using var ms = new MemoryStream();
            doc.SaveAsPDF(ms);
            pdfBytes.Add(ms.ToArray());
        }

        // Merge via secondary library:
        // var merged = SomePdfLib.Merge(pdfBytes);
        // File.WriteAllBytes("merged.pdf", merged);

        Console.WriteLine("Render done; merge requires a secondary library");
    }

    static string BuildSection1Html() => "<html><body><h1>Section 1</h1></body></html>";
    static string BuildSection2Html() => "<html><body><h1>Section 2</h1></body></html>";
}

After:

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

var results = await Task.WhenAll(
    renderer.RenderHtmlAsPdfAsync(BuildSection1Html()),
    renderer.RenderHtmlAsPdfAsync(BuildSection2Html())
);

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("combined.pdf");

Console.WriteLine($"Merged: {merged.PageCount} pages");

Watermark

Before (Scryber.Core):

using Scryber.Core;
using Scryber.Core.Html;
using System;

class WatermarkBefore
{
    static void Main()
    {
        // In Scryber, a watermark is typically a positioned, semi-transparent
        // element inside the source HTML/template — there is no first-class
        // post-render Stamp API. For an existing PDF you'd use a secondary library.

        string html = @"
            <html><body style='position: relative;'>
                <div style='position: fixed; top: 40%; left: 25%;
                            font-size: 80px; color: #888; opacity: 0.15;
                            transform: rotate(-30deg);'>DRAFT</div>
                <h1>Draft Report</h1>
            </body></html>";

        using var doc = Document.ParseDocument(html, ParseSourceType.DynamicContent);
        doc.SaveAsPDF("watermarked.pdf");

        Console.WriteLine("Watermark embedded in source HTML");
    }
}

After:

using IronPdf;
using IronPdf.Editing;
using IronSoftware.Drawing;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Draft Report</h1></body></html>"
);

// https://ironpdf.com/how-to/custom-watermark/
var watermark = new TextStamper
{
    Text = "DRAFT",
    FontColor = Color.Gray,
    Opacity = 15,                  // integer 0–100
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("watermarked.pdf");
Console.WriteLine("Watermark applied");

What Are the Most Common Migration Pitfalls?

After helping teams through this migration, these are the issues that come up most often.

CSS Property Not Rendering as Expected

Scryber.Core implements a CSS subset. Your existing HTML may have been written to work within those constraints. IronPDF uses Chromium, which implements current CSS standards. After migration, flex, grid, and custom properties all work. Markup that was specifically limited to avoid Scryber's CSS gaps will often look better under Chromium. Review and simplify CSS where workarounds were added.

// After migration, test your templates in Chrome browser with @media print styles:
// DevTools → More tools → Rendering → Emulate CSS media type → print
// IronPDF renders these print styles as Chrome does
// See: https://ironpdf.com/how-to/pixel-perfect-html-to-pdf/

Template Data Binding Gap

Scryber's doc.Params["key"] = value and {{model.X}} patterns have no direct equivalent in IronPDF. IronPDF takes HTML strings as input. You have several options depending on your preference.

// Option 1: C# string interpolation (simplest)
var html = $"<h1>Invoice {invoiceNumber}</h1><p>Customer: {customerName}</p>";

// Option 2: Template engine (better for complex templates)
// Scriban (MIT): https://github.com/scriban/scriban
// var template = Template.Parse(File.ReadAllText("invoice.sbn"));
// var html = template.Render(new { invoiceNumber, customerName });

// Option 3: Razor (if already using ASP.NET Core)
// var html = await razorRenderer.RenderToStringAsync("Invoice", model);

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(html);

.pdfx Template File Handling

If your project uses Scryber's XML .pdfx templates, those files have no IronPDF equivalent import path. Each .pdfx file needs to be converted to an HTML template manually. There is no automated converter. The XML structure gives you a visual reference for the layout to recreate in HTML. Projects that already feed Scryber HTML strings via Document.ParseDocument(html, ParseSourceType.DynamicContent) skip this step entirely.

Font Not Appearing in Output

Scryber.Core has its own font registration system. IronPDF renders via Chromium, which uses system fonts or web fonts. If a font specified in the template renders as a fallback, use one of these approaches.

// Option 1: Use system fonts available in your deployment environment
var html = "<html><head><style>body { font-family: Arial, sans-serif; }</style></head>...";

// Option 2: Embed via Google Fonts CDN
var html = @"
    <html>
    <head>
    <link href='https://fonts.googleapis.com/css2?family=Open+Sans' rel='stylesheet'>
    <style>body { font-family: 'Open Sans', sans-serif; }</style>
    </head>...";

// Option 3: Base64 embed for offline/air-gapped environments
// var fontBase64 = Convert.ToBase64String(File.ReadAllBytes("MyFont.woff2"));
// CSS: @font-face { font-family: MyFont; src: url(data:font/woff2;base64,...); }

Page Size / Orientation Mismatch

Scryber configures page size via doc.RenderOptions.PaperSize (or page styles in a template). IronPDF uses RenderingOptions on the renderer.

using IronPdf;
using IronPdf.Rendering;

var renderer = new ChromePdfRenderer();

// Map Scryber page size to IronPDF enum
renderer.RenderingOptions.PaperSize = PdfPaperSize.A4;  // or Letter, Legal, etc.
renderer.RenderingOptions.PaperOrientation = PdfPaperOrientation.Portrait;

// For custom sizes: see https://ironpdf.com/how-to/rendering-options/

What Does the Full API Mapping Look Like?

Here are the complete mapping references so you can look up any Scryber.Core API call and find its IronPDF equivalent.

Namespace Mapping

  • Scryber.Core → IronPdf — core namespace
  • Scryber.Core.Html → IronPdf — HTML parsing entry point on the Scryber side; in IronPDF the HTML path is the default
  • Scryber.Drawing → IronPdf.Rendering + IronSoftware.Drawing — page sizes, paper, and color types
  • Scryber styles / CSS namespace → N/A — CSS lives inside the HTML <style> block

Core Class Mapping

  • Document (with Document.ParseDocument) → ChromePdfRenderer — primary rendering object
  • doc.RenderOptions → ChromePdfRenderOptions — rendering configuration
  • doc.Params / template data model → HTML template string built by your code or a template engine
  • N/A in Scryber → PdfDocument — first-class object for post-render manipulation

Document Loading Methods

  • Render HTML string. Document.ParseDocument(html, ParseSourceType.DynamicContent) → renderer.RenderHtmlAsPdf(html) (or RenderHtmlAsPdfAsync).
  • Load XML .pdfx template. Document.ParseDocument(path) → no direct equivalent; rewrite as HTML and render via RenderHtmlAsPdf.
  • Render URL. Scryber requires a manual HttpClient fetch followed by ParseDocument on the result, with no JS execution → renderer.RenderUrlAsPdf(url) (real Chromium, JS executes).
  • Load existing PDF. Verify Scryber support for opening existing PDFs in your version → PdfDocument.FromFile(path).

Page Operations

  • Page count. pdf.PageCount in IronPDF; verify the equivalent on Scryber's document object for your version.
  • Remove page. pdf.RemovePages(index) in IronPDF; not a native Scryber operation post-render.
  • Extract text. pdf.ExtractAllText() in IronPDF; verify Scryber support if you rely on it.
  • Rotate. See IronPDF's editing docs; verify Scryber equivalent for your version.

Merge / Split Operations

  • Merge. PdfDocument.Merge(doc1, doc2) in IronPDF; no first-class merge in Scryber — typically a secondary library.
  • Split. See IronPDF's merge and split guide; not a native Scryber capability.

Frequently Asked Questions

How long does a typical Scryber.Core to IronPDF migration take?

If you are already feeding Scryber HTML strings via Document.ParseDocument(html, ParseSourceType.DynamicContent), the migration is usually a couple of hours of API rewiring plus a CSS pass. If your project uses XML .pdfx templates, the .pdfx count and complexity becomes the main variable. A simple single-column .pdfx file typically converts to HTML in 20 to 30 minutes.

Run find . -name "*.pdfx" | wc -l as your first step to scope the work. Then categorize templates by complexity and data binding depth. The namespace and package swap takes minutes; the template rewrites are the real time investment.

How do I handle Scryber's data binding model in IronPDF?

Scryber's doc.Params["key"] = value and {{model.X}} patterns do not have a direct equivalent in IronPDF. You have three options. The simplest is C# string interpolation. For complex templates, use a lightweight template engine like Scriban or Fluid. If you are already on ASP.NET Core, Razor views work well.

Using a template engine like Scriban gives you a separation of template and code that is closer to Scryber's model than raw string interpolation. This is the approach I recommend for teams with many templates.

Do I need to manually convert every .pdfx file?

Yes, for the XML template flavor. There is no automated converter from Scryber's .pdfx format to HTML. Each .pdfx file needs to be manually recreated as an HTML template. The XML structure gives you a visual reference for the layout. Projects that only feed Scryber raw HTML strings have nothing to convert here.

I recommend archiving your .pdfx files for reference rather than deleting them immediately. They serve as your layout specification during the conversion process.

Will my CSS look different after migrating?

It may look different, but usually better. Scryber.Core implements a CSS subset. IronPDF uses Chromium, which implements current CSS standards including Flexbox, Grid, and custom properties. Markup that was specifically limited to avoid Scryber's CSS gaps may render differently under Chromium.

Test your templates in Chrome browser using DevTools, navigating to More tools, then Rendering, then Emulate CSS media type set to print. IronPDF renders print styles exactly as Chrome does. You can learn more at the pixel-perfect rendering guide.

How does IronPDF handle page sizing compared to Scryber?

Scryber configures page size via doc.RenderOptions.PaperSize (or per-page CSS in a template). IronPDF uses RenderingOptions.PaperSize with enum values like PdfPaperSize.A4 or PdfPaperSize.Letter.

You can also set orientation via RenderingOptions.PaperOrientation. For custom sizes, check the IronPDF rendering options documentation.

Can IronPDF handle the same merge and security features that required secondary libraries with Scryber?

Yes. Scryber.Core's merge and security capabilities are limited or absent, which is why teams commonly add a second library like PdfSharp or iTextSharp. IronPDF handles all of these natively. Merging uses PdfDocument.Merge(), password protection uses pdf.SecuritySettings, and watermarking uses TextStamper or ImageStamper.

This consolidation from two or three dependencies down to one is often a significant motivator for the migration. You can explore the full set of manipulation features at the merge and split guide.

What Is the Bottom Line?

For HTML-string projects the migration is mostly an API swap and a CSS cleanup pass. For XML .pdfx projects, the template count and complexity is the key variable, and the rewrite from XML to HTML is where the real time investment lies. Running find . -name "*.pdfx" | wc -l and categorizing templates by complexity is the highest-value pre-migration step you can take.

For the data binding gap, I recommend using a lightweight template engine like Scriban or Fluid. It gives you a separation of template and code that is closer to Scryber's model than raw string interpolation, and it scales better as your template count grows.

IronPDF consolidates rendering, merging, watermarking, and security into a single dependency. If your Scryber.Core setup currently requires secondary libraries for those features, migration simplifies your dependency graph as well as your code.
