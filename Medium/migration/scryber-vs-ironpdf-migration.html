Migrating from Scryber to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate from Scryber.Core to IronPDF?

License renewal usually starts this conversation. Scryber.Core (maintained by Richard Hewitson, current version 9.3.1, LGPL-3.0) is a capable open-source PDF generation library that renders XHTML and a CSS subset to PDF using its own layout engine — not a browser. Before committing further, teams often do a quick evaluation of alternatives. That is typically how this migration begins.

Here are the reasons I see most often when teams evaluate Scryber.Core alternatives.

First, the LGPL-3.0 license. Source-disclosure obligations apply if you modify Scryber itself; static linking in closed-source apps is permitted, but the terms still constrain how you ship modifications.

Second, CSS subset limitations. Scryber.Core implements its own XHTML and CSS layout engine, not a browser. Modern features like CSS Grid, container queries, and most of Flexbox behave differently than they would in Chrome — or not at all.

Third, no JavaScript execution. Scryber renders static markup. Anything that depends on client-side scripting to populate the DOM will not appear in the output.

Fourth, XHTML strictness. Scryber requires well-formed XHTML with the xhtml namespace declared on the root element. Most real-world HTML has to be cleaned up (HtmlAgilityPack or similar) before Scryber will parse it.

Fifth, missing manipulation features. Merge, split, watermark, security, and digital signatures are out of scope for Scryber and commonly require supplementary libraries on top.

Sixth, no live URL rendering. Scryber generates from an XHTML string or template. To render a live page you fetch the HTML yourself with HttpClient first, which means no JavaScript, no follow-up XHR calls, and no images that load after the initial response.

Seventh, smaller community. Around 226 GitHub stars and 36 forks at this writing, which means fewer Stack Overflow answers and fewer worked examples to draw from when you hit edge cases.

Eighth, font handling. Custom font registration in Scryber has its own configuration path that differs from standard web font workflows.

Ninth, error messaging. Template and binding errors can surface as opaque runtime exceptions that are difficult to debug.

Comparison Table

Scryber.Core focuses on rendering XHTML and a CSS subset into a PDF using its own layout engine. IronPDF focuses on full HTML-to-PDF via an embedded Chromium plus a broad set of PDF manipulation features. Scryber is open source under LGPL-3.0, which can constrain how modifications are shipped in closed-source apps; IronPDF is commercially licensed — current pricing is on ironsoftware.com.

API style differs in tone, not in surface area. Scryber parses an XHTML string through a TextReader via Document.ParseDocument, then writes to a stream with SaveAsPDF. IronPDF instantiates a ChromePdfRenderer, hands it HTML or a URL, and returns a PdfDocument you can save, edit, merge, secure, or stamp. The core namespaces are Scryber.Components, Scryber.Drawing, Scryber.PDF, and Scryber.Styles on one side and IronPdf on the other.

The learning curve is the most honest difference. Scryber's CSS is a subset and its input must be valid XHTML, so styling tends to be a process of finding what works; IronPDF rendering matches what your team already sees in Chrome DevTools.

What Does the Migration Effort Actually Look Like?

I want to give you a realistic picture of the work involved. Most Scryber.Core migrations are mainly an API swap plus a template-binding rewrite — not a layout overhaul — because Scryber 6 and later already takes XHTML+CSS as input. The two sides of the work are switching the parse-and-save calls over to IronPDF's ChromePdfRenderer, and re-expressing Scryber's Handlebars-style {{ }} bindings (and any <template data-bind> sections) in Razor or string interpolation. Here is the feature-by-feature breakdown.

Effort by Feature

  • HTML to PDF — Low. Document.ParseDocument(reader, "", ParseSourceType.DynamicContent) followed by doc.SaveAsPDF(stream) becomes renderer.RenderHtmlAsPdf(html) followed by pdf.SaveAs(path).
  • Save to stream / bytes — Low. doc.SaveAsPDF(stream) becomes pdf.Stream or pdf.BinaryData.
  • XHTML to plain HTML — Low. Drop the xmlns="http://www.w3.org/1999/xhtml" attribute and stop hand-cleaning input markup. Chromium accepts well-formed HTML5 directly.
  • Data binding — Medium. Scryber's doc.Params["model"] = model and {{model.X}} patterns (including <template data-bind> loops) become C# string interpolation, Razor, or a template engine like Scriban.
  • Page size and margins — Low. @page { size: A4 portrait; margin: 20pt; } in template CSS becomes renderer.RenderingOptions.PaperSize = PdfPaperSize.A4 with explicit MarginTop/MarginBottom/MarginLeft/MarginRight properties.
  • CSS styling — Medium. Scryber's CSS subset becomes full Chromium CSS. Most of this work is removing workarounds, not adding them.
  • Headers/footers — Medium. @page margin boxes with counter(page) become RenderingOptions.HtmlHeader / HtmlFooter with {page} and {total-pages} placeholders.
  • Merge PDFs — Medium. Native in IronPDF via PdfDocument.Merge(); Scryber has no PDF-to-PDF merge API, so the task typically requires a second library.
  • Watermark — Medium. TextStamper / ImageStamper programmatically; in Scryber a watermark lives inside the source XHTML as a positioned overlay.
  • Password protection — Medium. pdf.SecuritySettings in IronPDF; encryption and password protection are not part of Scryber's public API.
  • Digital signatures — Medium. Native in IronPDF; not supported in Scryber.

Decision Matrix

  • If your project is XHTML-string-driven and your CSS is well within Scryber's subset, the migration is a straight API swap plus a binding rewrite, and you mostly gain manipulation features.
  • If your templates rely heavily on {{ }} binding and <template data-bind> loops, plan for that rewrite — Razor or Scriban is the natural target.
  • If you need full modern CSS (Grid, container queries, full Flexbox) or any client-side JavaScript to run before rendering, switch — Chromium covers both; Scryber covers neither.
  • If you need to render live web pages with images, scripts, and async XHR, switch — IronPDF's RenderUrlAsPdf uses a real browser.
  • If you need programmatic PDF manipulation (merge, security, signatures, stamping), switch — IronPDF covers all of these natively.

How Do You Set Up IronPDF and Replace Scryber.Core?

The core migration has three phases: find your existing Scryber references, swap the packages, then update your code.

Finding Scryber References in Your Codebase

Basic Conversion

Before (Scryber.Core):

using Scryber.Components;
using System;
using System.IO;

class Program
{
    static void Main()
    {
        // Scryber requires well-formed XHTML with the xhtml namespace.
        string html = $@"<html xmlns='http://www.w3.org/1999/xhtml'>
            <body>
                <h1>Q3 2024 Report</h1>
                <p>Generated: {DateTime.UtcNow:yyyy-MM-dd}</p>
            </body>
        </html>";

        using (var reader = new StringReader(html))
        using (var doc = Document.ParseDocument(reader, string.Empty, ParseSourceType.DynamicContent))
        using (var stream = new FileStream("output.pdf", FileMode.Create))
        {
            doc.SaveAsPDF(stream);
        }

        Console.WriteLine("Saved output.pdf");
    }
}

After:

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// Chromium replaces Scryber's XHTML/CSS layout engine — well-formed HTML5 is fine
var html = $@"
    <html>
    <head>
    <style>
        body {{ font-family: Arial, sans-serif; padding: 40px; }}
        h1 {{ font-size: 22px; }}
        .meta {{ color: #666; font-size: 12px; }}
    </style>
    </head>
    <body>
        <h1>Q3 2024 Report</h1>
        <div class='meta'>Generated: {DateTime.UtcNow:yyyy-MM-dd}</div>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");

Console.WriteLine($"Saved output.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

HTML to PDF (Invoice Example)

Before (Scryber.Core):

using Scryber.Components;
using System;
using System.IO;

class InvoiceBefore
{
    static void Main()
    {
        var customerName = "Acme Corp";
        var invoiceNumber = "INV-2024-0042";
        var totalAmount = "$4,200.00";
        var dueDate = DateTime.UtcNow.AddDays(30).ToString("yyyy-MM-dd");

        // String interpolation feeds Scryber's XHTML/CSS layout engine
        string html = $@"
            <html xmlns='http://www.w3.org/1999/xhtml'>
            <body style='font-family: Arial, sans-serif; padding: 40px;'>
                <div style='font-size: 22px; font-weight: bold;'>Invoice {invoiceNumber}</div>
                <div>Customer: {customerName}</div>
                <div>Due Date: {dueDate}</div>
                <div style='font-size: 16px; font-weight: bold;'>Total: {totalAmount}</div>
            </body></html>";

        using (var reader = new StringReader(html))
        using (var doc = Document.ParseDocument(reader, string.Empty, ParseSourceType.DynamicContent))
        using (var stream = new FileStream("invoice.pdf", FileMode.Create))
        {
            doc.SaveAsPDF(stream);
        }

        Console.WriteLine("Saved invoice.pdf");
    }
}

Merge PDFs

Before (Scryber.Core):

using Scryber.Components;
using System;
using System.Collections.Generic;
using System.IO;

class MergeBefore
{
    static void Main()
    {
        // Scryber has no PDF-to-PDF merge API — its mandate is generating
        // new PDFs from XHTML. Teams typically render each section, then add
        // a secondary library (PdfSharp, iText, etc.) to combine the bytes.

        var sections = new[] { BuildSection1Html(), BuildSection2Html() };
        var pdfBytes = new List<byte[]>();

        foreach (var html in sections)
        {
            using var reader = new StringReader(html);
            using var doc = Document.ParseDocument(reader, string.Empty, ParseSourceType.DynamicContent);
            using var ms = new MemoryStream();
            doc.SaveAsPDF(ms);
            pdfBytes.Add(ms.ToArray());
        }

        // Merge via secondary library:
        // var merged = SomePdfLib.Merge(pdfBytes);
        // File.WriteAllBytes("merged.pdf", merged);

        Console.WriteLine("Render done; merge requires a secondary library");
    }

    static string BuildSection1Html() => "<html xmlns='http://www.w3.org/1999/xhtml'><body><h1>Section 1</h1></body></html>";
    static string BuildSection2Html() => "<html xmlns='http://www.w3.org/1999/xhtml'><body><h1>Section 2</h1></body></html>";
}

After:

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();

var results = await Task.WhenAll(
    renderer.RenderHtmlAsPdfAsync(BuildSection1Html()),
    renderer.RenderHtmlAsPdfAsync(BuildSection2Html())
);

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("combined.pdf");

Console.WriteLine($"Merged: {merged.PageCount} pages");

Watermark

Before (Scryber.Core):

using Scryber.Components;
using System;
using System.IO;

class WatermarkBefore
{
    static void Main()
    {
        // In Scryber, a watermark is a positioned, semi-transparent element
        // inside the source XHTML — there is no first-class post-render
        // Stamp API. For an existing PDF you would need a secondary library.

        string html = @"
            <html xmlns='http://www.w3.org/1999/xhtml'>
            <body style='position: relative;'>
                <div style='position: fixed; top: 40%; left: 25%;
                            font-size: 80px; color: #888; opacity: 0.15;
                            transform: rotate(-30deg);'>DRAFT</div>
                <h1>Draft Report</h1>
            </body></html>";

        using var reader = new StringReader(html);
        using var doc = Document.ParseDocument(reader, string.Empty, ParseSourceType.DynamicContent);
        using var stream = new FileStream("watermarked.pdf", FileMode.Create);
        doc.SaveAsPDF(stream);

        Console.WriteLine("Watermark embedded in source XHTML");
    }
}

After:

using IronPdf;
using IronPdf.Editing;
using IronSoftware.Drawing;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Draft Report</h1></body></html>"
);

// https://ironpdf.com/how-to/custom-watermark/
var watermark = new TextStamper
{
    Text = "DRAFT",
    FontColor = Color.Gray,
    Opacity = 15,                  // integer 0–100
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("watermarked.pdf");
Console.WriteLine("Watermark applied");

What Are the Most Common Migration Pitfalls?

After helping teams through this migration, these are the issues that come up most often.

CSS Property Not Rendering as Expected

Scryber.Core implements a CSS subset on its own layout engine. Your existing XHTML may have been written to work within those constraints. IronPDF uses Chromium, which implements current CSS standards. After migration, Grid, full Flexbox, and custom properties all work. Markup that was specifically limited to avoid Scryber's CSS gaps will often look better under Chromium. Review and simplify CSS where workarounds were added.

// After migration, test your templates in Chrome browser with @media print styles:
// DevTools → More tools → Rendering → Emulate CSS media type → print
// IronPDF renders these print styles as Chrome does
// See: https://ironpdf.com/how-to/pixel-perfect-html-to-pdf/

Template Data Binding Gap

Scryber's doc.Params["model"] = model and {{model.X}} patterns — including <template data-bind> loops — have no direct equivalent in IronPDF. IronPDF takes HTML strings as input. You have several options depending on your preference.

// Option 1: C# string interpolation (simplest)
var html = $"<h1>Invoice {invoiceNumber}</h1><p>Customer: {customerName}</p>";

// Option 2: Template engine (better for complex templates)
// Scriban (MIT): https://github.com/scriban/scriban
// var template = Template.Parse(File.ReadAllText("invoice.sbn"));
// var html = template.Render(new { invoiceNumber, customerName });

// Option 3: Razor / RazorLight (if already using ASP.NET Core)
// var html = await engine.CompileRenderStringAsync("Invoice", template, model);

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(html);

XHTML Strictness Goes Away

Scryber requires every template to be valid XHTML with the xmlns="http://www.w3.org/1999/xhtml" attribute on the root element, and any cleanup pass that fed Scryber unsanctioned HTML (HtmlAgilityPack output, third-party widgets, copy-pasted snippets) had to produce well-formed XML. Drop those guards. Chromium accepts well-formed HTML5 directly, so the namespace attribute and the cleanup pipeline can both come out.

Font Not Appearing in Output

Scryber.Core has its own font registration system. IronPDF renders via Chromium, which uses system fonts or web fonts. If a font specified in the template renders as a fallback, use one of these approaches.

// Option 1: Use system fonts available in your deployment environment
var html = "<html><head><style>body { font-family: Arial, sans-serif; }</style></head>...";

// Option 2: Embed via Google Fonts CDN
var html = @"
    <html>
    <head>
    <link href='https://fonts.googleapis.com/css2?family=Open+Sans' rel='stylesheet'>
    <style>body { font-family: 'Open Sans', sans-serif; }</style>
    </head>...";

// Option 3: Base64 embed for offline/air-gapped environments
// var fontBase64 = Convert.ToBase64String(File.ReadAllBytes("MyFont.woff2"));
// CSS: @font-face { font-family: MyFont; src: url(data:font/woff2;base64,...); }

Page Size / Orientation Mismatch

Scryber configures page size in template CSS via @page { size: A4 portrait; margin: 20pt; }; output-side options like compression sit on doc.RenderOptions. IronPDF puts both on RenderingOptions on the renderer.

using IronPdf;
using IronPdf.Rendering;

var renderer = new ChromePdfRenderer();

// Map Scryber page size to IronPDF enum
renderer.RenderingOptions.PaperSize = PdfPaperSize.A4;  // or Letter, Legal, etc.
renderer.RenderingOptions.PaperOrientation = PdfPaperOrientation.Portrait;

// For custom sizes: see https://ironpdf.com/how-to/rendering-options/

What Does the Full API Mapping Look Like?

Here are the complete mapping references so you can look up any Scryber.Core API call and find its IronPDF equivalent.

Namespace Mapping

  • Scryber.Components → IronPdf — Document, ParseSourceType, and the core component tree
  • Scryber.PDF → IronPdf — RenderOptions and OutputCompressionType
  • Scryber.Drawing → IronPdf.Rendering + IronSoftware.Drawing — units, page sizes, paper, and color types
  • Scryber.Styles → N/A — CSS lives inside the HTML <style> block

Core Class Mapping

  • Document (with Document.ParseDocument) → ChromePdfRenderer — primary rendering object
  • doc.RenderOptions → renderer.RenderingOptions — rendering configuration
  • doc.Params / template data model → HTML template string built by your code or a template engine
  • doc.Info.Title / doc.Info.Author → pdf.MetaData.Title / pdf.MetaData.Author
  • N/A in Scryber → PdfDocument — first-class object for post-render manipulation

Document Loading Methods

  • Render XHTML string. Document.ParseDocument(reader, "", ParseSourceType.DynamicContent) → renderer.RenderHtmlAsPdf(html) (or RenderHtmlAsPdfAsync).
  • Render HTML file. Document.ParseDocument(path) → renderer.RenderHtmlFileAsPdf(path).
  • Render URL. Scryber requires a manual HttpClient fetch followed by ParseDocument on the result, with no JS execution → renderer.RenderUrlAsPdf(url) (real Chromium, JS executes).
  • Load existing PDF. Not part of Scryber's public API — its mandate is generating new PDFs from XHTML → PdfDocument.FromFile(path).

Page Operations

  • Page count. pdf.PageCount in IronPDF; not a first-class concern for Scryber post-render.
  • Remove page. pdf.RemovePages(index) in IronPDF; not a native Scryber operation.
  • Extract text. pdf.ExtractAllText() in IronPDF; not part of Scryber's public API.
  • Rotate. See IronPDF's editing docs; not a native Scryber capability.

Merge / Split Operations

  • Merge. PdfDocument.Merge(doc1, doc2) in IronPDF; no first-class merge in Scryber — typically a secondary library.
  • Split. See IronPDF's merge and split guide; not a native Scryber capability.

Frequently Asked Questions

How long does a typical Scryber.Core to IronPDF migration take?

If your templates are static XHTML with no data binding, the migration is usually a couple of hours: namespace and package swap, the ParseDocument/SaveAsPDF calls swing over to RenderHtmlAsPdf/SaveAs, and a CSS pass to delete subset workarounds. The longer phase is the data-binding rewrite. Each template that uses {{model.X}} placeholders or <template data-bind> loops needs to be re-expressed in Razor or string interpolation.

Run a quick grep for doc.Params and {{ in your XHTML to scope the binding work. The namespace and package swap takes minutes; the binding rewrite is the real time investment.

How do I handle Scryber's data binding model in IronPDF?

Scryber's doc.Params["model"] = model and {{model.X}} patterns do not have a direct equivalent in IronPDF. You have three options. The simplest is C# string interpolation. For complex templates, use a lightweight template engine like Scriban or Fluid. If you are already on ASP.NET Core, Razor views (or RazorLight outside ASP.NET) work well.

Using a template engine like Scriban or RazorLight gives you a separation of template and code that is closer to Scryber's model than raw string interpolation. This is the approach I recommend for teams with many templates.

Do I need to clean up my XHTML to feed it to IronPDF?

No — and that is one of the upsides. Scryber requires every template to be valid XHTML with the xmlns="http://www.w3.org/1999/xhtml" attribute. Any cleanup pipeline you ran before ParseDocument (HtmlAgilityPack, manual sanitizing, namespace injection) can come out. Chromium accepts well-formed HTML5 directly.

You can keep the existing XHTML as-is for the first migration pass and modernize it later. The xhtml namespace attribute is harmless under Chromium; you just no longer need to maintain the cleanup machinery that put it there.

Will my CSS look different after migrating?

It may look different, but usually better. Scryber.Core implements a CSS subset on its own layout engine. IronPDF uses Chromium, which implements current CSS standards including full Flexbox, Grid, and custom properties. Markup that was specifically limited to avoid Scryber's CSS gaps may render differently under Chromium.

Test your templates in Chrome browser using DevTools, navigating to More tools, then Rendering, then Emulate CSS media type set to print. IronPDF renders print styles exactly as Chrome does. You can learn more at the pixel-perfect rendering guide.

How does IronPDF handle page sizing compared to Scryber?

Scryber configures page size in template CSS via @page { size: A4 portrait; margin: 20pt; }. IronPDF uses RenderingOptions.PaperSize with enum values like PdfPaperSize.A4 or PdfPaperSize.Letter, plus explicit MarginTop/MarginBottom/MarginLeft/MarginRight properties.

You can also set orientation via RenderingOptions.PaperOrientation. For custom sizes, check the IronPDF rendering options documentation.

Can IronPDF handle the same merge and security features that required secondary libraries with Scryber?

Yes. Scryber.Core has no PDF-to-PDF merge API and no encryption in its public API, which is why teams commonly add a second library like PdfSharp or iText. IronPDF handles all of these natively. Merging uses PdfDocument.Merge(), password protection uses pdf.SecuritySettings, and watermarking uses TextStamper or ImageStamper.

This consolidation from two or three dependencies down to one is often a significant motivator for the migration. You can explore the full set of manipulation features at the merge and split guide.

What Is the Bottom Line?

The Scryber-to-IronPDF migration is mostly an API swap, a CSS cleanup pass, and a binding rewrite. The API swap is mechanical: Document.ParseDocument over a TextReader becomes renderer.RenderHtmlAsPdf, and doc.SaveAsPDF(stream) becomes pdf.SaveAs(path). The binding rewrite is where the real time goes if your templates lean on {{model.X}} placeholders and <template data-bind> loops.

For the data binding gap, I recommend using a lightweight template engine like Scriban, Fluid, or RazorLight. It gives you a separation of template and code that is closer to Scryber's model than raw string interpolation, and it scales better as your template count grows.

IronPDF consolidates rendering, merging, watermarking, and security into a single dependency. If your Scryber.Core setup currently requires secondary libraries for those features, migration simplifies your dependency graph as well as your code.
