Migrating from Scryber to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate from Scryber.Core to IronPDF?

License renewal usually starts this conversation. The bill shows up, someone checks the NuGet page for the latest version, and they find the last release was a long time ago. Or the issue tracker has open bugs that nobody has touched in months. Scryber.Core is a capable PDF generation library built on an XML-based template model, but before renewing or committing further, teams often do a quick evaluation of alternatives. That is typically how this migration begins.

Here are the reasons I see most often when teams evaluate Scryber.Core alternatives.

First, maintenance cadence. Slower update cycles mean lag on .NET version support. You should verify the current release frequency before making assumptions.

Second, template model specificity. Scryber.Core uses an XML/PDFX template format that requires learning its own document model rather than reusing HTML/CSS knowledge your team already has.

Third, CSS subset limitations. Scryber.Core implements a subset of CSS. Not all CSS properties behave as they would in a browser. If you need modern CSS features like Flexbox or Grid, this becomes a blocker.

Fourth, community size. A smaller community means fewer Stack Overflow answers and fewer community examples to draw from when you hit edge cases.

Fifth, missing manipulation features. Merge, split, watermark, security, and text extraction may require supplementary libraries on top of Scryber.Core.

Sixth, no live URL rendering. Scryber.Core generates from templates and code. Rendering a live web page is not a primary use case.

Seventh, documentation gaps. Complex layout scenarios may have limited examples in the official docs.

Eighth, font handling. Custom font registration in Scryber.Core has its own configuration requirements that differ from standard web font workflows.

Ninth, error messaging. XML template errors can surface as opaque runtime exceptions that are difficult to debug.

What Are the Most Common Migration Pitfalls?

After helping teams through this migration, these are the issues that come up most often.

CSS Property Not Rendering as Expected

Scryber.Core implements a CSS subset. Your existing templates may have been designed to work within those constraints. IronPDF uses Chromium, which implements current CSS standards. After migration, flex, grid, and custom properties all work. However, templates that were specifically limited to avoid Scryber's CSS gaps may look different (usually better) under Chromium. Review and simplify CSS where workarounds were added.

// After migration, test your templates in Chrome browser with @media print styles:
// DevTools → More tools → Rendering → Emulate CSS media type → print
// IronPDF renders these print styles as Chrome does
// See: https://ironpdf.com/how-to/pixel-perfect-html-to-pdf/

Template Data Binding Gap

Scryber's doc.Params["key"] = value pattern has no direct equivalent in IronPDF. IronPDF takes HTML strings as input and does not have a params dictionary. You have several options depending on your preference.

// Option 1: C# string interpolation (simplest)
var html = $"<h1>Invoice {invoiceNumber}</h1><p>Customer: {customerName}</p>";

// Option 2: Template engine (better for complex templates)
// Scriban (MIT): https://github.com/scriban/scriban
// var template = Template.Parse(File.ReadAllText("invoice.sbn"));
// var html = template.Render(new { invoiceNumber, customerName });

// Option 3: Razor (if already using ASP.NET Core)
// var html = await razorRenderer.RenderToStringAsync("Invoice", model);

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(html);

.pdfx Template File Handling

Existing .pdfx files have no IronPDF equivalent import path. Scryber.Core uses a proprietary XML-based template format, and IronPDF uses HTML. Each .pdfx file needs to be converted to an HTML template manually. There is no automated converter. The XML structure gives you a visual reference for the layout to recreate in HTML.

Font Not Appearing in Output

Scryber.Core has its own font registration system. IronPDF renders via Chromium, which uses system fonts or web fonts. If a font specified in the template renders as a fallback, use one of these approaches.

// Option 1: Use system fonts available in your deployment environment
var html = "<html><head><style>body { font-family: Arial, sans-serif; }</style></head>...";

// Option 2: Embed via Google Fonts CDN
var html = @"
    <html>
    <head>
    <link href='https://fonts.googleapis.com/css2?family=Open+Sans' rel='stylesheet'>
    <style>body { font-family: 'Open Sans', sans-serif; }</style>
    </head>...";

// Option 3: Base64 embed for offline/air-gapped environments
// var fontBase64 = Convert.ToBase64String(File.ReadAllBytes("MyFont.woff2"));
// CSS: @font-face { font-family: MyFont; src: url(data:font/woff2;base64,...); }

Page Size / Orientation Mismatch

Scryber configures page size in the .pdfx template. IronPDF uses RenderingOptions instead.

using IronPdf;
using IronPdf.Rendering;

var renderer = new ChromePdfRenderer();

// Map Scryber template page size to IronPDF enum
renderer.RenderingOptions.PaperSize = PdfPaperSize.A4;  // or Letter, Legal, etc.
renderer.RenderingOptions.PaperOrientation = PdfPaperOrientation.Portrait; // verify enum

// For custom sizes: verify in IronPDF docs
// See: https://ironpdf.com/how-to/rendering-options/

Frequently Asked Questions

How long does a typical Scryber.Core to IronPDF migration take?

Answer:

The .pdfx template count and complexity is the key variable. The API swap itself is straightforward. A simple single-column .pdfx file typically converts to HTML in 20 to 30 minutes. Complex multi-column layouts take longer.

Run find . -name "*.pdfx" | wc -l as your first step to scope the work. Then categorize templates by complexity and data binding depth. The namespace and package swap takes minutes; the template rewrites are the real time investment.

How do I handle Scryber's data binding model in IronPDF?

Answer:

Scryber's doc.Params["key"] = value pattern does not have a direct equivalent in IronPDF. You have three options. The simplest is C# string interpolation. For complex templates, use a lightweight template engine like Scriban or Fluid. If you are already on ASP.NET Core, Razor views work well.

Using a template engine like Scriban gives you a separation of template and code that is closer to Scryber's model than raw string interpolation. This is the approach I recommend for teams with many templates.

Do I need to manually convert every .pdfx file?

Answer:

Yes. There is no automated converter from Scryber's proprietary XML/PDFX format to HTML. Each .pdfx file needs to be manually recreated as an HTML template. The XML structure gives you a visual reference for the layout to recreate.

I recommend archiving your .pdfx files for reference rather than deleting them immediately. They serve as your layout specification during the conversion process.

Will my CSS look different after migrating?

Answer:

It may look different, but usually better. Scryber.Core implements a CSS subset. IronPDF uses Chromium, which implements current CSS standards including Flexbox, Grid, and custom properties. Templates that were specifically limited to avoid Scryber's CSS gaps may render differently under Chromium.

Test your templates in Chrome browser using DevTools, navigating to More tools, then Rendering, then Emulate CSS media type set to print. IronPDF renders print styles exactly as Chrome does. You can learn more at the pixel-perfect rendering guide.

How does IronPDF handle page sizing compared to Scryber's template-based approach?

Answer:

Scryber configures page size in the .pdfx template using attributes like @page-width and @page-height. IronPDF uses RenderingOptions.PaperSize with enum values like PdfPaperSize.A4 or PdfPaperSize.Letter.

You can also set orientation via RenderingOptions.PaperOrientation. For custom sizes, check the IronPDF rendering options documentation.

Can IronPDF handle the same merge and security features that required secondary libraries with Scryber?

Answer:

Yes. Scryber.Core's merge and security capabilities are limited or absent, which is why teams commonly add a second library like PdfSharp or iTextSharp. IronPDF handles all of these natively. Merging uses PdfDocument.Merge(), password protection uses pdf.SecuritySettings, and watermarking uses TextStamper or ImageStamper.

This consolidation from two or three dependencies down to one is often a significant motivator for the migration. You can explore the full set of manipulation features at the merge and split guide.

What Is the Bottom Line?

The .pdfx template count and complexity is the key variable before scoping this migration. The API swap is straightforward. The template rewrite from XML to HTML is where the real time investment lies. Running find . -name "*.pdfx" | wc -l and categorizing templates by complexity is the highest-value pre-migration step you can take.

For the data binding gap, I recommend using a lightweight template engine like Scriban or Fluid. It gives you a separation of template and code that is closer to Scryber's model than raw string interpolation, and it scales better as your template count grows.

IronPDF consolidates rendering, merging, watermarking, and security into a single dependency. If your Scryber.Core setup currently requires secondary libraries for those features, migration simplifies your dependency graph as well as your code.
