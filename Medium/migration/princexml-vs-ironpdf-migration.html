Migrating from PrinceXML to IronPDF: No Fuss, No Fluff

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Consider Migrating from PrinceXML?

The deployment works perfectly in the on-premises environment and breaks as soon as you try to containerize it. PrinceXML is a native binary — it is not a .NET library, it is an executable that you shell out to or call via an HTTP API. In a Docker context, that means either bundling the binary in your image (licensing implications, Linux ABI compatibility, container size) or running PrinceXML as a sidecar service (orchestration complexity, network hop on every render, single point of failure). Neither option is frictionless.

Here are the conditions I see most often when teams evaluate alternatives:

Docker/container deployment — PrinceXML is a native binary; Docker images require bundling the binary or running a sidecar process.

Cloud platform restrictions — serverless environments (Azure Functions Consumption, AWS Lambda) often cannot execute arbitrary native binaries.

Licensing per server — PrinceXML's per-server license model requires tracking and paying per deployment node; horizontal scaling has direct cost implications.

Process invocation overhead — shelling out to PrinceXML via Process.Start() per document adds process spawn latency and error-handling complexity.

Sidecar service management — running PrinceXML as an HTTP service introduces health checks, failover, and inter-service error handling.

CSS Paged Media specificity — PrinceXML's strict CSS Paged Media compliance is ideal for print-typographic work but requires specific HTML preparation for web-derived content.

Missing manipulation features — PrinceXML generates PDFs but does not provide merge, split, watermark, or security APIs out of the box.

Binary version management — updating PrinceXML requires updating binaries in every environment, which is different from a NuGet version bump.

Linux ABI compatibility — verify that the PrinceXML binary for your Linux distribution is available for your container base image.

Concurrent request throughput — a single PrinceXML process handles one render at a time; concurrency requires multiple processes or a queuing wrapper.

Step 3 — Basic Conversion

Before (PrinceXML via Process.Start — common pattern):

using System;
using System.Diagnostics;
using System.IO;

class Program
{
    static void Main()
    {
        var htmlPath = Path.GetTempFileName() + ".html";
        var pdfPath = "output.pdf";

        File.WriteAllText(htmlPath,
            "<html><body><h1>Hello</h1></body></html>");

        var psi = new ProcessStartInfo
        {
            FileName = "prince",  // must be on PATH; varies by install
            Arguments = $"\"{htmlPath}\" -o \"{pdfPath}\"",
            RedirectStandardError = true,
            UseShellExecute = false,
        };

        using var process = Process.Start(psi)!;
        var errors = process.StandardError.ReadToEnd();
        process.WaitForExit();

        if (process.ExitCode != 0)
            throw new Exception($"Prince failed: {errors}");

        Console.WriteLine("Saved output.pdf");
        File.Delete(htmlPath);
    }
}

After:

using IronPdf;
using System;

// No native binary required — renders locally via .NET
IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Hello</h1></body></html>");
pdf.SaveAs("output.pdf");

Console.WriteLine($"Saved output.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

Merge PDFs

Before (PrinceXML — no native merge; common pattern):

using System;
using System.Collections.Generic;
using System.Diagnostics;
using System.IO;
using System.Threading.Tasks;

class MergeBefore
{
    static async Task Main()
    {
        // PrinceXML can concatenate inputs on CLI (verify current options):
        // prince section1.html section2.html -o merged.pdf
        // But for dynamically generated PDFs, teams typically generate each
        // and then use a secondary library to merge the PDF files.

        var htmlSections = new[]
        {
            ("<html><body><h1>Section 1: Overview</h1></body></html>", "sec1.pdf"),
            ("<html><body><h1>Section 2: Detail</h1></body></html>", "sec2.pdf"),
        };

        foreach (var (html, outFile) in htmlSections)
        {
            var tmpHtml = Path.GetTempFileName() + ".html";
            await File.WriteAllTextAsync(tmpHtml, html);

            var psi = new ProcessStartInfo
            {
                FileName = "prince",
                Arguments = $"\"{tmpHtml}\" -o \"{outFile}\"",
                RedirectStandardError = true,
                UseShellExecute = false,
            };
            using var p = Process.Start(psi)!;
            await p.WaitForExitAsync();
            File.Delete(tmpHtml);
        }

        // Now merge sec1.pdf + sec2.pdf using secondary library
        Console.WriteLine("PrinceXML merge requires secondary library for programmatic use");
    }
}

After:

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

// Render concurrently
var t1 = renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 1</h1></body></html>");
var t2 = renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 2</h1></body></html>");
var results = await Task.WhenAll(t1, t2);

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("merged-report.pdf");

Console.WriteLine($"Merged: {merged.PageCount} pages");

What Critical Issues Should You Watch for During Migration?

Answer:

There are three areas that catch teams off guard. I will cover each one.

CSS Paged Media Gap

PrinceXML implements W3C CSS Paged Media closely. If your templates use running(), string(), @footnote, named page flows, or Prince-specific extensions (-prince-* properties), those will not transfer to IronPDF's Chromium renderer.

Before migrating, audit your CSS:

# Find Prince-specific CSS properties
rg "\-prince\-\|@footnote\|running(\|@page.*string(" --type css
rg "\-prince\-\|@footnote\|running(" --type html

For headers/footers, migrate from CSS Paged Media @page running elements to IronPDF's HtmlHeader/Footer API:

using IronPdf;
using IronPdf.Rendering.Abstractions;

var renderer = new ChromePdfRenderer();

// https://ironpdf.com/how-to/headers-and-footers/
renderer.RenderingOptions.HtmlHeader = new HtmlHeaderFooter
{
    HtmlFragment = "<div style='font-size:10px; text-align:right'>{page} of {total-pages}</div>",
};

// Replace: @page { @top-right { content: counter(page); } }

Process Exit Code Handling

PrinceXML via Process.Start() returns exit codes; your existing code likely checks process.ExitCode. Replace with try/catch for IronPDF exceptions:

// Remove this pattern:
// if (process.ExitCode != 0) throw new Exception($"Prince error: {stderr}");

// Replace with:
try
{
    var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    pdf.SaveAs(outputPath);
}
catch (IronPdf.Exceptions.IronPdfException ex)
{
    Console.Error.WriteLine($"Render failed: {ex.Message}");
    throw;
}

Temp File Cleanup

PrinceXML Process.Start() workflows often write HTML to temp files. With IronPDF, you pass HTML strings directly — remove the temp file write/delete pattern.

# Find temp file patterns to remove
rg "GetTempFileName\|GetTempPath\|\.html.*temp\|temp.*\.html" --type cs -n

How Does Concurrent Rendering Compare?

Answer:

With PrinceXML, a single process handles one render at a time. Concurrency requires multiple processes or a queuing wrapper. With IronPDF, you use Task.WhenAll and async render methods in-process.

Concurrent Throughput Benchmark Structure

using IronPdf;
using System;
using System.Diagnostics;
using System.Linq;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Prince: concurrent renders require multiple processes (no shared state)
// IronPDF: multiple async renders via Task.WhenAll
// https://ironpdf.com/examples/parallel/

async Task<double> BenchmarkConcurrency(int concurrency)
{
    var htmlJobs = Enumerable.Range(1, concurrency)
        .Select(i => $"<html><body><h1>Document {i}</h1></body></html>")
        .ToArray();

    var sw = Stopwatch.StartNew();
    var tasks = htmlJobs.Select(async html =>
    {
        var renderer = new ChromePdfRenderer();
        using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
        return pdf.PageCount;
    });

    await Task.WhenAll(tasks);
    sw.Stop();
    Console.WriteLine($"Concurrency {concurrency}: {sw.Elapsed.TotalMilliseconds:F0}ms total");
    return sw.Elapsed.TotalMilliseconds;
}

await BenchmarkConcurrency(5);
await BenchmarkConcurrency(10);
await BenchmarkConcurrency(20);
// See: https://ironpdf.com/how-to/async/

// First render includes Chromium initialization overhead

var renderer = new ChromePdfRenderer();

Frequently Asked Questions

Can I use IronPDF as a drop-in replacement for PrinceXML?

Answer:

For basic HTML-to-PDF rendering, yes — the migration is straightforward. Replace your Process.Start("prince") calls with ChromePdfRenderer.RenderHtmlAsPdfAsync() and you are running. The temp file pattern goes away entirely since IronPDF accepts HTML strings directly.

The exception is CSS Paged Media. If your templates rely on running(), @footnote, named page flows, or -prince-* extensions, those features need rewriting. Run the CSS audit commands above before estimating effort.

How does IronPDF handle features that PrinceXML does not provide natively?

Answer:

PrinceXML focuses on rendering. Merge, split, watermark, and password protection require secondary libraries or CLI flags. IronPDF provides all of these through PdfDocument.Merge(), TextStamper/ImageStamper, and pdf.SecuritySettings — no extra dependencies.

This means you can remove whatever secondary PDF library you bolted on for post-processing. One NuGet package handles the full pipeline from HTML-to-PDF rendering through to merge/split and encryption.

What happens to my Docker deployment pipeline?

Answer:

It gets simpler. Remove the PrinceXML binary download, the tar extraction, the install.sh step, and the license file copy from your Dockerfile. IronPDF ships as a NuGet package, so dotnet restore handles everything. No native binary install, no sidecar service, no license file management in the image.

If you were running PrinceXML as a sidecar service, you also remove that container from your orchestration config — eliminating health checks, failover logic, and the network hop on every render.

How does concurrent rendering differ between PrinceXML and IronPDF?

Answer:

PrinceXML handles one render per process. To render concurrently, you spawn multiple prince processes or queue requests to an HTTP sidecar. IronPDF renders in-process using async methods — you call Task.WhenAll with multiple RenderHtmlAsPdfAsync() tasks. No process management, no queuing wrapper.

Warm up the renderer during application startup to absorb the first-render Chromium initialization cost. After that, concurrent requests run without process spawn overhead. See the async rendering guide for patterns.

What is the biggest risk in this migration?

Answer:

CSS Paged Media feature parity. PrinceXML implements the W3C CSS Paged Media specification closely. If your templates use advanced features like running() elements, string() functions, @footnote, or -prince-* properties, those will not work in IronPDF's Chromium renderer and need rewriting.

If your CSS audit comes back clean — basic @page sizing, standard print properties only — this migration is low-risk. If it surfaces Prince-specific properties, those sections need rewriting before the migration produces comparable output.

Conclusion

The clearest operational win in this migration is the Dockerfile simplification — removing a native binary install, license file management, and sidecar service orchestration. The biggest technical risk is CSS Paged Media feature parity, and the CSS audit is the first thing to run before writing any migration code.

If the CSS audit comes back clean (basic @page sizing, standard print properties only), this migration is low-risk. If it surfaces running() elements, footnotes, or -prince-* properties, those sections need rewriting before the migration is comparable output.
