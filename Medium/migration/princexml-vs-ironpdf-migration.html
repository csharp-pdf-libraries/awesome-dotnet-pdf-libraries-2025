Migrating from PrinceXML to IronPDF: No Fuss, No Fluff

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Consider Migrating from PrinceXML?

The deployment works perfectly in the on-premises environment and breaks as soon as you try to containerize it. PrinceXML is a native binary — it is not a .NET library. The common .NET integration is the PrinceXMLWrapper NuGet package (which exposes the PrinceXML.Wrapper namespace), and it wraps the Prince executable, shelling out to it under the hood. In a Docker context, that still means either bundling the binary in your image (licensing implications, Linux ABI compatibility, container size) or running PrinceXML as a sidecar service (orchestration complexity, network hop on every render, single point of failure). Neither option is frictionless.

Here are the conditions I see most often when teams evaluate alternatives:

Docker/container deployment — even with PrinceXMLWrapper, Docker images still need the Prince binary bundled or a sidecar process.

Cloud platform restrictions — serverless environments (Azure Functions Consumption, AWS Lambda) often cannot execute arbitrary native binaries.

Licensing per server — PrinceXML's per-server license model requires tracking and paying per deployment node; horizontal scaling has direct cost implications.

Wrapper-to-process overhead — prince.Convert() in PrinceXMLWrapper still spawns the Prince executable per document, with the same process spawn latency and error-handling complexity as a raw Process.Start().

Sidecar service management — running PrinceXML as an HTTP service introduces health checks, failover, and inter-service error handling.

CSS Paged Media specificity — PrinceXML's strict CSS Paged Media compliance is ideal for print-typographic work but requires specific HTML preparation for web-derived content.

Missing manipulation features — PrinceXML generates PDFs but does not provide merge, split, watermark, or security APIs out of the box.

Binary version management — updating PrinceXML requires updating binaries in every environment, which is different from a NuGet version bump.

Linux ABI compatibility — the PrinceXML binary must match your Linux distribution's ABI, which constrains your choice of container base image.

Concurrent request throughput — a single PrinceXML process handles one render at a time; concurrency requires multiple processes or a queuing wrapper.

Comparison Table

The two libraries solve overlapping problems but with very different shapes. PrinceXML focuses on W3C CSS Paged Media HTML/XML-to-PDF rendering and ships as a per-server commercial native binary; IronPDF focuses on HTML-to-PDF plus full PDF manipulation and ships as a .NET-native commercial library — pricing details on princexml.com and ironsoftware.com respectively.

The API shapes diverge accordingly. Prince's .NET surface is the PrinceXMLWrapper package (exposing types under PrinceXML.Wrapper) wrapping the prince executable; IronPDF is a direct .NET library with no external process. Prince has a medium learning curve once you bring in CSS Paged Media concepts; IronPDF is low-friction for .NET developers because the API maps directly onto C# objects.

Rendering engines differ too: Prince is W3C CSS Paged Media compliant, IronPDF is Chromium-based. Prince doesn't expose page indexing because it is generation-only; IronPDF is 0-based for page operations — that is the standard off-by-one footgun versus 1-based libraries. Prince concurrency is process-level (one render per process); IronPDF runs concurrent renders in-process via Task.WhenAll. The namespace swap is from PrinceXML.Wrapper to IronPdf.

What Does the Migration Effort Actually Look Like?

I want to give you a realistic picture of the effort involved. Here is a feature-by-feature breakdown so you can estimate the work before writing a single line of code.

Effort by Feature

  • HTML string to PDF — Low. Prince exposes prince.ConvertString(html, output), which writes the resulting PDF to disk; IronPDF takes the string directly via ChromePdfRenderer.RenderHtmlAsPdfAsync() and returns an in-memory document.
  • URL to PDF — Low. Prince accepts a URL as input to prince.Convert(url, output); IronPDF uses ChromePdfRenderer.RenderUrlAsPdfAsync().
  • Save to file — Low. Prince writes via the output argument to Convert; IronPDF uses pdf.SaveAs(path).
  • Save to bytes — Low. Prince requires reading the output file from disk; IronPDF exposes pdf.BinaryData directly.
  • Custom page size — Low. Prince expects CSS @page { size: A4; }; IronPDF uses RenderingOptions.PaperSize.
  • Headers/footers — Medium. Prince uses CSS Paged Media @page running elements; IronPDF uses RenderingOptions.HtmlHeader/Footer.
  • CSS Paged Media features — High. Prince implements the full W3C spec including running(), string(), and @footnote; IronPDF supports the Chromium subset of the print spec. This is the migration's main risk surface.
  • Merge PDFs — Medium. Prince has no native programmatic merge for dynamically generated PDFs — teams add a secondary library; IronPDF has PdfDocument.Merge() built in.
  • Watermark — Medium. Prince relies on CSS @page backgrounds or a post-process library; IronPDF uses TextStamper or ImageStamper.
  • Password protection — Medium. Prince uses the Encrypt, UserPassword, and OwnerPassword properties on the wrapper; IronPDF uses pdf.SecuritySettings.UserPassword/OwnerPassword.
  • Concurrent rendering — Medium. Prince needs multiple processes (one render per process); IronPDF uses Task.WhenAll with the async render methods.

How Do You Decide Whether to Migrate?

Not every team should migrate. Here is a decision matrix to help you figure out where you fall.

  • If you need print-exact typographic output (books, legal documents, complex footnotes), PrinceXML may still be better suited — evaluate CSS Paged Media needs carefully before committing to switch.
  • If Docker or cloud deployment is required, switch — this eliminates the native binary dependency entirely.
  • If you generate web-derived HTML reports and dashboards, switch — Chromium rendering matches what users already see in their browser.
  • If you are horizontally scaling and per-server licensing is a concern, switch — IronPDF's NuGet distribution is container-friendly without per-node license fees.

How Do You Set Up IronPDF and Replace PrinceXML?

The core migration has three parts: find your existing PrinceXML references, swap the packages, and update your basic conversion code.

Finding PrinceXML References in Your Codebase

# Find PrinceXMLWrapper invocation patterns
rg -l "PrinceXML\.Wrapper|PrinceXMLWrapper|Prince\(|prince\.Convert" --type cs
rg "PrinceXML\.Wrapper|new Prince|prince\.Convert|prince\.ConvertString|\.JavaScript\s*=|\.Encrypt\s*=|\.PdfTitle\s*=" --type cs -n

# Find raw Process.Start calls that might invoke Prince directly
rg "Process\.Start.*prince|StartInfo.*prince" --type cs -n

# Find prince in Docker files and CI
grep -r "prince\|princexml" Dockerfile* docker-compose.yml .github/**/*.yml 2>/dev/null

# Find project references

Basic Conversion

Before (PrinceXMLWrapper):

using PrinceXML.Wrapper;
using System;

class Program
{
    static void Main()
    {
        var html = "<html><body><h1>Hello</h1></body></html>";

        // Wrapper still spawns the prince.exe binary under the hood
        Prince prince = new Prince("C:\\Program Files\\Prince\\engine\\bin\\prince.exe");
        prince.ConvertString(html, "output.pdf");

        Console.WriteLine("Saved output.pdf");
    }
}

After:

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Hello</h1></body></html>");
pdf.SaveAs("output.pdf");

Console.WriteLine($"Saved output.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

Merge PDFs

Before (PrinceXMLWrapper — no native programmatic merge):

using PrinceXML.Wrapper;
using System;

class MergeBefore
{
    static void Main()
    {
        // PrinceXML can concatenate input files on the CLI, but for
        // dynamically generated PDFs teams typically generate each and
        // then bring in a secondary library to merge the resulting files.

        var htmlSections = new[]
        {
            ("<html><body><h1>Section 1: Overview</h1></body></html>", "sec1.pdf"),
            ("<html><body><h1>Section 2: Detail</h1></body></html>", "sec2.pdf"),
        };

        Prince prince = new Prince("C:\\Program Files\\Prince\\engine\\bin\\prince.exe");

        foreach (var (html, outFile) in htmlSections)
        {
            prince.ConvertString(html, outFile);
        }

        // Now merge sec1.pdf + sec2.pdf using a secondary library —
        // PrinceXMLWrapper does not expose a programmatic merge API.
        Console.WriteLine("PrinceXML merge requires a secondary library for programmatic use");
    }
}

After:

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

// Render concurrently
var t1 = renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 1</h1></body></html>");
var t2 = renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 2</h1></body></html>");
var results = await Task.WhenAll(t1, t2);

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("merged-report.pdf");

Console.WriteLine($"Merged: {merged.PageCount} pages");

What Are the Complete API Mappings Between PrinceXML and IronPDF?

Here are the full mapping tables so you can look up any PrinceXML concept and find its IronPDF equivalent.

Namespace Mapping

  • PrinceXML.Wrapper → IronPdf — core namespace.
  • Prince wrapper properties / CLI flags → IronPdf.Rendering.ChromePdfRenderOptions — rendering configuration.
  • N/A in Prince → IronPdf.Editing — watermark and stamp APIs.

Core Class Mapping

  • Prince wrapper class invoking prince.exe → ChromePdfRenderer — no external process required.
  • prince.Convert(input, output) and prince.ConvertString(html, output) write a file → PdfDocument — an in-memory PDF object that exposes manipulation methods.
  • Wrapper properties (JavaScript, Encrypt, PdfTitle) → ChromePdfRenderOptions — page size, margins, print options.
  • N/A in Prince → PdfDocument.Merge() — local merge of multiple PDFs.

Document Loading Methods

  • HTML file. prince.Convert("input.html", "output.pdf") → renderer.RenderHtmlFileAsPdfAsync(path).
  • HTML string. prince.ConvertString(html, "output.pdf") → renderer.RenderHtmlAsPdfAsync(html) — both accept a string directly.
  • URL. prince.Convert("https://...", "output.pdf") → renderer.RenderUrlAsPdfAsync(url).
  • Load existing PDF. Not supported in Prince → PdfDocument.FromFile(path).

Page Operations

  • Page count. Not exposed in Prince (output only) → pdf.PageCount.
  • Remove page. Not supported in Prince → pdf.RemovePages(index) (note: 0-based).
  • Extract text. Not supported in Prince → pdf.ExtractAllText().
  • Rotate. Not supported in Prince → available via IronPDF's page rotation API; see the IronPDF docs for the current method signature.

Merge / Split Operations

  • Merge. Not native in PrinceXMLWrapper — requires a secondary library → PdfDocument.Merge(doc1, doc2).
  • Split. Not native in PrinceXMLWrapper → see the IronPDF merge/split guide.

What Critical Issues Should You Watch for During Migration?

There are three areas that catch teams off guard. I will cover each one.

CSS Paged Media Gap

PrinceXML implements W3C CSS Paged Media closely. If your templates use running(), string(), @footnote, named page flows, or Prince-specific extensions (-prince-* properties), those will not transfer to IronPDF's Chromium renderer.

Before migrating, audit your CSS:

# Find Prince-specific CSS properties
rg "\-prince\-\|@footnote\|running(\|@page.*string(" --type css
rg "\-prince\-\|@footnote\|running(" --type html

For headers/footers, migrate from CSS Paged Media @page running elements to IronPDF's HtmlHeader/Footer API:

using IronPdf;
using IronPdf.Rendering.Abstractions;

var renderer = new ChromePdfRenderer();

// https://ironpdf.com/how-to/headers-and-footers/
renderer.RenderingOptions.HtmlHeader = new HtmlHeaderFooter
{
    HtmlFragment = "<div style='font-size:10px; text-align:right'>{page} of {total-pages}</div>",
};

// Replace: @page { @top-right { content: counter(page); } }

Process Exit Code Handling

PrinceXMLWrapper translates the underlying prince.exe exit code into thrown exceptions or false return values, and raw Process.Start() code typically checks process.ExitCode directly. Either way, replace with try/catch for IronPDF exceptions:

// Remove this pattern:
// if (process.ExitCode != 0) throw new Exception($"Prince error: {stderr}");

// Replace with:
try
{
    var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    pdf.SaveAs(outputPath);
}
catch (IronPdf.Exceptions.IronPdfException ex)
{
    Console.Error.WriteLine($"Render failed: {ex.Message}");
    throw;
}

Temp File Cleanup

Codebases that pre-date ConvertString often write HTML to a temp file with File.WriteAllText("temp.html", html) and then call prince.Convert("temp.html", ...). With IronPDF, you pass HTML strings directly to RenderHtmlAsPdfAsync(), so the temp file write/delete pattern goes away entirely — and the resulting PdfDocument stays in memory rather than round-tripping through disk.

# Find temp file patterns to remove
rg "GetTempFileName\|GetTempPath\|\.html.*temp\|temp.*\.html" --type cs -n

How Does Concurrent Rendering Compare?

With PrinceXML, a single process handles one render at a time. Concurrency requires multiple processes or a queuing wrapper. With IronPDF, you use Task.WhenAll and async render methods in-process.

Concurrent Throughput Benchmark Structure

using IronPdf;
using System;
using System.Diagnostics;
using System.Linq;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// Prince: concurrent renders require multiple processes (no shared state)
// IronPDF: multiple async renders via Task.WhenAll
// https://ironpdf.com/examples/parallel/

async Task<double> BenchmarkConcurrency(int concurrency)
{
    var htmlJobs = Enumerable.Range(1, concurrency)
        .Select(i => $"<html><body><h1>Document {i}</h1></body></html>")
        .ToArray();

    var sw = Stopwatch.StartNew();
    var tasks = htmlJobs.Select(async html =>
    {
        var renderer = new ChromePdfRenderer();
        using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
        return pdf.PageCount;
    });

    await Task.WhenAll(tasks);
    sw.Stop();
    Console.WriteLine($"Concurrency {concurrency}: {sw.Elapsed.TotalMilliseconds:F0}ms total");
    return sw.Elapsed.TotalMilliseconds;
}

await BenchmarkConcurrency(5);
await BenchmarkConcurrency(10);
await BenchmarkConcurrency(20);
// See: https://ironpdf.com/how-to/async/

Renderer Warm-Up

using IronPdf;

// First render includes Chromium initialization overhead
// Warm up during application startup for consistent request latency
var renderer = new ChromePdfRenderer();
using var _ = await renderer.RenderHtmlAsPdfAsync("<html><body>warmup</body></html>");
// Renderer is now ready for production traffic

Memory Management

using IronPdf;
using System.IO;

var renderer = new ChromePdfRenderer();

// Always use 'using' on PdfDocument

// MemoryStream pattern for in-memory processing
// https://ironpdf.com/how-to/async/

// pdf disposed at end of 'using' block

Frequently Asked Questions

Can I use IronPDF as a drop-in replacement for PrinceXML?

For basic HTML-to-PDF rendering, yes — the migration is straightforward. Replace your prince.Convert(input, output) or prince.ConvertString(html, output) calls (or any raw Process.Start("prince")) with ChromePdfRenderer.RenderHtmlAsPdfAsync() and you are running. Any legacy temp-file patterns built around earlier wrapper versions go away too, since IronPDF's PdfDocument stays in memory until you explicitly save.

The exception is CSS Paged Media. If your templates rely on running(), @footnote, named page flows, or -prince-* extensions, those features need rewriting. Run the CSS audit commands above before estimating effort.

How does IronPDF handle features that PrinceXML does not provide natively?

PrinceXML focuses on rendering. Merge, split, watermark, and password protection require secondary libraries or CLI flags. IronPDF provides all of these through PdfDocument.Merge(), TextStamper/ImageStamper, and pdf.SecuritySettings — no extra dependencies.

This means you can remove whatever secondary PDF library you bolted on for post-processing. One NuGet package handles the full pipeline from HTML-to-PDF rendering through to merge/split and encryption.

What happens to my Docker deployment pipeline?

It gets simpler. Remove the PrinceXML binary download, the tar extraction, the install.sh step, and the license file copy from your Dockerfile. IronPDF ships as a NuGet package, so dotnet restore handles everything. No native binary install, no sidecar service, no license file management in the image.

If you were running PrinceXML as a sidecar service, you also remove that container from your orchestration config — eliminating health checks, failover logic, and the network hop on every render.

How does concurrent rendering differ between PrinceXML and IronPDF?

PrinceXML handles one render per process. To render concurrently, you spawn multiple prince processes or queue requests to an HTTP sidecar. IronPDF renders in-process using async methods — you call Task.WhenAll with multiple RenderHtmlAsPdfAsync() tasks. No process management, no queuing wrapper.

Warm up the renderer during application startup to absorb the first-render Chromium initialization cost. After that, concurrent requests run without process spawn overhead. See the async rendering guide for patterns.

What is the biggest risk in this migration?

CSS Paged Media feature parity. PrinceXML implements the W3C CSS Paged Media specification closely. If your templates use advanced features like running() elements, string() functions, @footnote, or -prince-* properties, those will not work in IronPDF's Chromium renderer and need rewriting.

If your CSS audit comes back clean — basic @page sizing, standard print properties only — this migration is low-risk. If it surfaces Prince-specific properties, those sections need rewriting before the migration produces comparable output.

Conclusion

The clearest operational win in this migration is the Dockerfile simplification — removing a native binary install, license file management, and sidecar service orchestration. The biggest technical risk is CSS Paged Media feature parity, and the CSS audit is the first thing to run before writing any migration code.

If the CSS audit comes back clean (basic @page sizing, standard print properties only), this migration is low-risk. If it surfaces running() elements, footnotes, or -prince-* properties, those sections need rewriting before the migration produces comparable output.
