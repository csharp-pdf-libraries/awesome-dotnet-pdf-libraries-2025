Migrating from Spire.PDF to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should Your Team Consider Migrating from Spire.PDF?

The .NET upgrade is usually what forces the conversation. Spire.PDF has been working fine on .NET Framework, but the team is moving to .NET 8, and the NuGet page shows a version that claims compatibility — except there is a regression with a specific API path you are using, or the behavior changed between major versions, or the thread-safety characteristics shifted in a way that does not surface until load testing. Whatever the specific trigger, the upgrade timeline becomes the opportunity to evaluate whether this dependency is the right one for the next five years.

I have seen teams encounter one or more of these pain points when evaluating whether to stay on Spire.PDF:

.NET upgrade friction. Spire.PDF version requirements for newer .NET versions may force API changes even when the underlying need has not changed.

HTML rendering fidelity. Spire.PDF's default HTML-to-PDF path is the legacy IE/QtWebKit engine; modern CSS (flex, grid, custom properties) does not render reliably there. Spire 10.7.21 added an opt-in ChromeHtmlConverter in the Spire.Additions.Chrome package, but it requires a system-installed Google Chrome binary on the host.

Community edition limits. FreeSpire.PDF caps load and create operations at 10 pages, and only the first 3 pages convert when you target image, Word, HTML, or XPS output. The cap is silent — output beyond that is truncated rather than reported as an error.

Drawing API maintenance overhead. Programmatic PDF creation via a drawing API (PdfPageBase.Canvas.DrawString, etc.) is verbose and requires manual layout calculation.

Licensing model. Spire.PDF is sold as a perpetual, scope-based license: Developer Small Business is $999, Developer OEM (required for any public-facing, SaaS, or Docker deployment) is $2,549, Site Small Business is $4,599, and Site OEM is $13,088. Spire.Doc and Spire.XLS are sold separately if you need Office input.

Cross-dependency accumulation. Complex workflows may require Spire.PDF + Spire.Doc + Spire.XLS, adding multiple packages.

Thread safety. Concurrent rendering behavior has shifted between Spire.PDF major releases; benchmark your specific version under load before assuming previous behavior holds.

PDF/A and compliance. PDF/A support varies by Spire edition — check that yours covers the conformance level your output requires.

Missing features. Digital signatures, advanced annotations, and specific PDF standards may be edition-gated.

API verbosity. The drawing API for layout is significantly more verbose than HTML template rendering for report-style documents.

Spire.PDF vs IronPDF Comparison

  • Focus. Spire.PDF covers PDF creation, editing, conversion, and text extraction; IronPDF is HTML-to-PDF plus PDF manipulation.
  • Pricing. Spire ships a page-limited free edition alongside perpetual paid tiers from $999 to $13,088 depending on developer count and deployment scope; IronPDF is commercial only, with perpetual tiers from $799 (Lite) to $4,799 (Unlimited).
  • API style. Spire combines a drawing API with a document model and gets verbose for layout; IronPDF pairs an HTML renderer with manipulation objects.
  • Learning curve. Spire is medium-to-high for drawing-API layout code and low for simple operations; IronPDF is low for web devs because HTML/CSS is the input.
  • HTML rendering. Spire's default path uses a legacy IE/QtWebKit engine that supports only a CSS subset; the opt-in ChromeHtmlConverter requires a host-installed Chrome. IronPDF embeds Chromium, so no system browser is required.
  • Page indexing. Both libraries are 0-based, so loops port directly.
  • Namespace. Spire.Pdf (note the lowercase d) and Spire.Pdf.Graphics become IronPdf.

How Much Effort Will the Migration Actually Take?

This is the first question every team lead asks, and it is the right one. The answer depends almost entirely on how much of your codebase uses Spire's drawing API versus its HTML conversion path. Here is how each feature area maps to migration complexity.

  • HTML string to PDF — Low–Medium. pdf.LoadFromHTML() + SaveToFile becomes ChromePdfRenderer.RenderHtmlAsPdfAsync().
  • Drawing text/shapes — High (rewrite). page.Canvas.DrawString() and friends have no direct equivalent; recreate the layout as an HTML/CSS template.
  • Save to file — Low. doc.SaveToFile(path) becomes pdf.SaveAs(path).
  • Save to stream — Low. doc.SaveToStream(stream) becomes pdf.Stream.
  • Custom page size — Low. PdfDocument.PageSettings.Size becomes RenderingOptions.PaperSize.
  • Headers/footers — Medium. Per-page drawing-API calls become RenderingOptions.HtmlHeader / HtmlFooter.
  • Merge PDFs — Low–Medium. PdfDocument.AppendPage() / InsertPageRange() become PdfDocument.Merge().
  • Watermark — Low. PdfWatermark or a drawing layer becomes TextStamper / ImageStamper.
  • Password protection — Low. PdfSecurity.KeySize plus password fields become pdf.SecuritySettings.
  • Text extraction — Low. PdfPageBase.ExtractText() becomes pdf.ExtractAllText().
  • Word/Excel to PDF — High. Spire ships native Office conversion via Spire.Doc/XLS; IronPDF has no native Office input path, so this work either stays on Spire or moves to a separate Office-to-HTML pipeline before rendering.

What Are the Key Decision Criteria for Choosing Between Spire.PDF and IronPDF?

The decision comes down to your primary use case and migration cost tolerance. Here is the decision matrix I use when advising teams:

  • If your primary use case is HTML-to-PDF from web templates, switch — Chromium rendering handles modern CSS that Spire's HTML engine does not.
  • If you have extensive use of Spire's drawing API for complex layouts, evaluate migration cost first — the drawing-API-to-HTML rewrite is significant.
  • If Word/Excel-to-PDF conversion is the primary need, Spire is likely better suited; IronPDF is HTML-first and does not have a native Office input path.
  • If a .NET 8 (or later) migration is blocked by a Spire compatibility issue, switching resolves the blocker, but assess feature parity for the rest of your usage first.

If your codebase uses Spire.PDF's drawing API (Canvas.DrawString, DrawRectangle, DrawImage, PdfFont, PdfBrush, etc.) to construct document layouts, that code has no direct IronPDF equivalent. IronPDF is HTML-first — the drawing operations need to be rewritten as HTML/CSS. Audit how much of your codebase falls into each category before committing to a timeline.

Namespace Swap

The first concrete step is replacing your using statements.

Before (Spire.PDF):

using Spire.Pdf;
using Spire.Pdf.Graphics;
using Spire.Pdf.Security;

After (IronPDF):

using IronPdf;
using IronPdf.Rendering;
using IronPdf.Editing;

HTML to PDF Conversion

This is the most common migration path. Here is how a basic HTML-to-PDF conversion changes.

Before (Spire.PDF):

using Spire.Pdf;
using Spire.Pdf.Graphics;
using System;

class HtmlToPdfBefore
{
    static void Main()
    {
        var pdf = new PdfDocument();

        string html = "<html><body><h1>Q3 Invoice #4421</h1><p>Total: $8,400</p></body></html>";

        var setting = new PdfPageSettings();
        setting.Size = PdfPageSize.A4;
        var layout = new PdfHtmlLayoutFormat();
        layout.IsWaiting = false;

        // HTML-string overload: LoadFromHTML(string, bool autoDetectPageBreak,
        //                                   PdfPageSettings, PdfHtmlLayoutFormat)
        // The 4-bool overload — LoadFromHTML(url, enableJS, enableHyperlinks, autoDetectPageBreak)
        // — is the URL path, not for inline HTML strings.
        pdf.LoadFromHTML(html, true, setting, layout);

        pdf.SaveToFile("invoice.pdf");
        pdf.Close();
        Console.WriteLine("Saved invoice.pdf");
    }
}

After (IronPDF HTML to PDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var html = @"
    <html>
    <head>
    <style>
        body { font-family: Arial, sans-serif; padding: 40px; }
        h1 { font-size: 22px; }
        .total { font-weight: bold; font-size: 16px; margin-top: 20px; }
    </style>
    </head>
    <body>
        <h1>Q3 Invoice #4421</h1>
        <p>Total: $8,400</p>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");

Console.WriteLine($"Saved invoice.pdf ({pdf.PageCount} page(s))");

Merging PDFs

Spire's merge pattern loads each source document then copies a page range into the target. IronPDF takes complete document objects, which simplifies the code considerably.

Before (Spire.PDF):

using Spire.Pdf;
using System;

class MergeBefore
{
    static void Main()
    {
        var pdf1 = new PdfDocument();
        pdf1.LoadFromFile("document1.pdf");

        var pdf2 = new PdfDocument();
        pdf2.LoadFromFile("document2.pdf");

        pdf1.InsertPageRange(pdf2, 0, pdf2.Pages.Count - 1);

        pdf1.SaveToFile("merged.pdf");
        pdf1.Close();
        pdf2.Close();
    }
}

After (IronPDF Merge):

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

var results = await Task.WhenAll(
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 1: Overview</h1></body></html>"),
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 2: Detail</h1></body></html>")
);

var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("merged.pdf");

Console.WriteLine($"Merged: {merged.PageCount} pages");

Watermarking

The watermark migration is one of the clearest examples of how the drawing API translates to a simpler declarative approach.

Before (Spire.PDF drawing API):

using Spire.Pdf;
using Spire.Pdf.Graphics;
using System;
using System.Drawing;

class WatermarkBefore
{
    static void Main()
    {
        var doc = new PdfDocument();
        doc.LoadFromFile("source.pdf");

        foreach (PdfPageBase page in doc.Pages)
        {
            var font = new PdfFont(PdfFontFamily.Helvetica, 60f);
            var brush = new PdfSolidBrush(Color.FromArgb(30, Color.Gray));

            var state = page.Canvas.Save();
            page.Canvas.TranslateTransform(
                (float)(page.ActualSize.Width / 2),
                (float)(page.ActualSize.Height / 2));
            page.Canvas.RotateTransform(-45f);
            page.Canvas.DrawString("DRAFT", font, brush, 0, 0,
                new PdfStringFormat(PdfTextAlignment.Center));
            page.Canvas.Restore(state);
        }

        doc.SaveToFile("watermarked.pdf");
        doc.Close();
    }
}

After (IronPDF Watermark):

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Document</h1></body></html>"
);

var watermark = new TextStamper
{
    Text = "DRAFT",
    FontSize = 60,
    FontColor = "#808080",
    Opacity = 15,
    Rotation = -45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("watermarked.pdf");

Password Protection

Security settings are straightforward to migrate. The property names differ but the concepts map directly.

Before (Spire.PDF):

using Spire.Pdf;
using Spire.Pdf.Security;
using System;

class PasswordBefore
{
    static void Main()
    {
        var doc = new PdfDocument();
        doc.LoadFromFile("source.pdf");

        doc.Security.KeySize = PdfEncryptionKeySize.Key128Bit;
        doc.Security.OwnerPassword = "admin456";
        doc.Security.UserPassword = "open123";
        doc.Security.Permissions = PdfPermissionsFlags.Default & ~PdfPermissionsFlags.EditContent;

        doc.SaveToFile("secured.pdf");
        doc.Close();
    }
}

After (IronPDF Password Protection):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Protected Document</h1></body></html>"
);

pdf.SecuritySettings.UserPassword = "open123";
pdf.SecuritySettings.OwnerPassword = "admin456";

pdf.SaveAs("secured.pdf");

What Does the Full Feature Mapping Look Like?

Below are the API mappings I reference during migrations. These cover namespace mapping, core classes, document loading, page operations, and merge/split operations.

Namespace Mapping

  • Spire.Pdf → IronPdf — core namespace.
  • Spire.Pdf.Graphics → not needed on the HTML path — the drawing API is replaced by HTML/CSS.
  • Spire.Pdf.Security → IronPdf — security lives on the PdfDocument instance via SecuritySettings.

Core Class Mapping

  • PdfDocument (Spire, used to render HTML) → ChromePdfRenderer — the HTML-to-PDF renderer.
  • PdfDocument (Spire, used to edit a file) → PdfDocument (IronPDF) — the PDF manipulation object.
  • PdfPageBase / PdfPage → no equivalent — drawing operations are replaced by HTML/CSS templates.
  • PdfSecurity → pdf.SecuritySettings — password and permission settings.

Document Loading Methods

  • HTML string. doc.LoadFromHTML(html, autoDetectPageBreak, PdfPageSettings, PdfHtmlLayoutFormat) becomes renderer.RenderHtmlAsPdfAsync(html).
  • URL. doc.LoadFromHTML(url, enableJS, enableHyperlinks, autoDetectPageBreak) — the four-bool overload — becomes renderer.RenderUrlAsPdfAsync(url).
  • HTML file. Spire's path varies by version; IronPDF uses renderer.RenderHtmlFileAsPdfAsync(path).
  • Load existing PDF. new PdfDocument(); doc.LoadFromFile(path) becomes PdfDocument.FromFile(path).

Page Operations

  • Page count. doc.Pages.Count becomes pdf.PageCount.
  • Remove page. doc.Pages.RemoveAt(index) becomes pdf.RemovePages(index).
  • Extract text. doc.Pages[i].ExtractText() becomes pdf.ExtractAllText(); per-page text is available via pdf.Pages[i].Text.
  • Rotate. Spire uses page.Rotation = PdfPageRotateAngle.*; IronPDF exposes page rotation on the page object — verify the exact property name in your IronPDF version.

Merge / Split Operations

  • Merge. doc.AppendPage(otherDoc) / InsertPageRange() become PdfDocument.Merge(doc1, doc2).
  • Split. Spire offers doc.Split(outputPaths, pageRanges); IronPDF uses CopyPages() on a loaded document — see the split guide.

What Are the Most Common Questions Teams Have During Migration?

Q1: What happens to my FreeSpire.PDF code that was silently truncating at 10 pages?

FreeSpire.PDF enforces two distinct caps that catch teams off guard. Loading or creating a document is hard-limited to 10 pages, and conversions to image, Word, HTML, or XPS only emit the first 3 pages. Both caps are silent — output beyond the limit is truncated rather than reported as an error, and an evaluation watermark is stamped onto what remains. Once you migrate to IronPDF, previously-truncated PDFs render in full and the watermark is gone, which is often where teams discover that reports they thought were complete had been losing data for months.

Q2: How do I handle the Spire.PDF drawing API code that has no direct IronPDF equivalent?

If the drawing API count dominates your codebase, estimate the HTML template rewrite effort before committing to a timeline. A complex multi-column report layout done in drawing API code may take several hours to recreate in HTML/CSS. A useful benchmark: compare the line count of the drawing API code with the HTML/CSS equivalent. In most report scenarios, the HTML version is significantly shorter.

Q3: How does resource cleanup differ between Spire.PDF and IronPDF?

Spire.PDF requires explicit doc.Close() for resource cleanup. With IronPDF, the standard C# using pattern handles disposal automatically:

// Spire pattern:
// var doc = new PdfDocument();
// try { ... } finally { doc.Close(); }

// IronPDF pattern:
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("output.pdf");
// Disposed automatically

Q4: Can I render multiple PDFs in parallel with IronPDF?

Yes. IronPDF supports concurrent rendering. Here is the pattern I recommend:

using IronPdf;
using System.Linq;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var htmlJobs = reportDataList.Select(data => BuildHtml(data)).ToArray();

var pdfs = await Task.WhenAll(htmlJobs.Select(async html =>
{
    var renderer = new ChromePdfRenderer();
    return await renderer.RenderHtmlAsPdfAsync(html);
}));

Console.WriteLine($"Generated {pdfs.Length} PDFs");
foreach (var pdf in pdfs) pdf.Dispose();

What Decision Framework Should I Use to Decide Whether to Migrate?

Start with a quantitative audit. Run these commands against your codebase to understand your migration scope:

# Drawing API usage (higher migration cost)
rg "Canvas\.Draw\|DrawString\|DrawLine\|DrawRectangle\|DrawImage" --type cs -n | wc -l

# HTML/URL conversion usage (lower migration cost)
rg "LoadFromHtml\|LoadFromUrl\|ConvertHtmlString" --type cs -n | wc -l

If HTML conversion dominates, the migration is straightforward. If drawing API code dominates, you need to budget for HTML/CSS template rewrites.

Then walk through this checklist before writing any migration code:

Pre-Migration: Count HTML-to-PDF vs drawing API usage. Verify Spire.PDF edition (Free vs paid). Identify all Spire cross-product dependencies (Spire.Doc, Spire.XLS). Document the current .NET version issue that triggered the evaluation. Obtain an IronPDF license key.

Code Migration: Install IronPDF. Remove Spire packages. Add the license key at application startup. Replace LoadFromHtml() with ChromePdfRenderer.RenderHtmlAsPdfAsync(). Replace doc.SaveToFile() with pdf.SaveAs(). Replace the Spire merge pattern with PdfDocument.Merge(). Replace drawing API watermarks with TextStamper / ImageStamper. Replace doc.Security.* with pdf.SecuritySettings.*.

Testing: Render each HTML template and compare visual output. Verify previously-truncated Free edition PDFs now render all pages. Test merge output page count and order. Test password protection. Benchmark render time vs Spire baseline. Test in your target .NET version — the original upgrade trigger.

The .NET version trigger that started this evaluation tends to resolve cleanly once the HTML-to-PDF path is migrated. The drawing API code is the variable — measure how much of your PDF generation uses it before committing to a timeline.
