Migrating from Spire.PDF to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should Your Team Consider Migrating from Spire.PDF?

The .NET upgrade is usually what forces the conversation. Spire.PDF has been working fine on .NET Framework, but the team is moving to .NET 8, and the NuGet page shows a version that claims compatibility — except there is a regression with a specific API path you are using, or the behavior changed between major versions, or the thread-safety characteristics shifted in a way that does not surface until load testing. Whatever the specific trigger, the upgrade timeline becomes the opportunity to evaluate whether this dependency is the right one for the next five years.

I have seen teams encounter one or more of these pain points when evaluating whether to stay on Spire.PDF:

.NET upgrade friction. Spire.PDF version requirements for newer .NET versions may force API changes even when the underlying need has not changed.

HTML rendering fidelity. Spire.PDF's HTML-to-PDF engine has its own rendering path. Modern CSS (flex, grid, custom properties) may not render as expected.

Community edition limits. FreeSpire.PDF has page limits (10 pages for most operations). Verify current constraints at e-iceblue.com.

Drawing API maintenance overhead. Programmatic PDF creation via a drawing API (PdfPageBase.Canvas.DrawString, etc.) is verbose and requires manual layout calculation.

Licensing model. Spire.PDF licensing is per-developer/server. Verify current terms for your team size and deployment count.

Cross-dependency accumulation. Complex workflows may require Spire.PDF + Spire.Doc + Spire.XLS, adding multiple packages.

Thread safety. Verify concurrent rendering behavior in your Spire.PDF version. Behavior can vary between major releases.

PDF/A and compliance. Verify PDF/A support in your specific edition.

Missing features. Digital signatures, advanced annotations, or specific PDF standards may vary by edition.

API verbosity. The drawing API for layout can be significantly more verbose than HTML template rendering for report-style documents.

What Are the Key Decision Criteria for Choosing Between Spire.PDF and IronPDF?

Answer:

The decision comes down to your primary use case and migration cost tolerance. Here is a decision matrix I use when advising teams.

Business ScenarioRecommendation
Primary use case is HTML-to-PDF from web templatesSwitch — Chromium rendering handles modern CSS
Extensive use of Spire's drawing API for complex layoutsEvaluate migration cost — drawing API to HTML rewrite is significant
Word/Excel-to-PDF conversion is the primary needSpire may be better suited; IronPDF is HTML-first
.NET 8 migration blocked by Spire compatibility issueSwitch resolves the blocker; assess feature parity first

If your codebase uses Spire.PDF's drawing API (Canvas.DrawString, DrawRectangle, DrawImage, PdfFont, PdfBrush, etc.) to construct document layouts, that code has no direct IronPDF equivalent. IronPDF is HTML-first — the drawing operations need to be rewritten as HTML/CSS. Audit how much of your codebase falls into each category before committing to a timeline.

Namespace Swap

The first concrete step is replacing your using statements.

Before (Spire.PDF):

using Spire.Pdf;
using Spire.Pdf.Graphics;
using Spire.Pdf.Security;

After (IronPDF):

using IronPdf;
using IronPdf.Rendering;
using IronPdf.Editing;

HTML to PDF Conversion

This is the most common migration path. Here is how a basic HTML-to-PDF conversion changes.

Before (Spire.PDF):

using Spire.Pdf;
using Spire.Pdf.HtmlConverter;  // verify namespace for your version
using System;

class HtmlToPdfBefore
{
    static void Main()
    {
        var doc = new PdfDocument();

        var settings = new HtmlToPdfSettings();   // verify class name
        settings.WebPageWidth = 1280;             // verify property name
        settings.WebPageHeight = 0;               // 0 = auto

        PdfConverter.Convert(                     // verify class + method
            "<html><body><h1>Q3 Invoice #4421</h1><p>Total: $8,400</p></body></html>",
            doc,
            settings);

        doc.SaveToFile("invoice.pdf", FileFormat.PDF);
        doc.Close();
        Console.WriteLine("Saved invoice.pdf");
    }
}

After (IronPDF HTML to PDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var html = @"
    <html>
    <head>
    <style>
        body { font-family: Arial, sans-serif; padding: 40px; }
        h1 { font-size: 22px; }
        .total { font-weight: bold; font-size: 16px; margin-top: 20px; }
    </style>
    </head>
    <body>
        <h1>Q3 Invoice #4421</h1>
        <p>Total: $8,400</p>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");

Console.WriteLine($"Saved invoice.pdf ({pdf.PageCount} page(s))");

Merging PDFs

Spire's merge pattern is append-page-by-page. IronPDF takes complete document objects, which simplifies the code considerably.

Before (Spire.PDF):

using Spire.Pdf;
using System;

class MergeBefore
{
    static void Main()
    {
        var doc1 = new PdfDocument();
        doc1.SaveToFile("section1.pdf");
        doc1.Close();

        var doc2 = new PdfDocument();
        doc2.SaveToFile("section2.pdf");
        doc2.Close();

        var merged = new PdfDocument();
        merged.LoadFromFile("section1.pdf");

        var appendDoc = new PdfDocument();
        appendDoc.LoadFromFile("section2.pdf");

        foreach (PdfPageBase page in appendDoc.Pages)
        {
            merged.AppendPage(appendDoc, page.PageIndex);
        }

        merged.SaveToFile("merged.pdf");
        merged.Close();
        appendDoc.Close();
    }
}

After (IronPDF Merge):

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

var results = await Task.WhenAll(
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 1: Overview</h1></body></html>"),
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 2: Detail</h1></body></html>")
);

var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("merged.pdf");

Console.WriteLine($"Merged: {merged.PageCount} pages");

Watermarking

The watermark migration is one of the clearest examples of how the drawing API translates to a simpler declarative approach.

Before (Spire.PDF drawing API):

using Spire.Pdf;
using Spire.Pdf.Graphics;
using System;
using System.Drawing;

class WatermarkBefore
{
    static void Main()
    {
        var doc = new PdfDocument();
        doc.LoadFromFile("source.pdf");

        foreach (PdfPageBase page in doc.Pages)
        {
            var font = new PdfFont(PdfFontFamily.Helvetica, 60f);
            var brush = new PdfSolidBrush(Color.FromArgb(30, Color.Gray));

            var state = page.Canvas.Save();
            page.Canvas.TranslateTransform(
                (float)(page.ActualSize.Width / 2),
                (float)(page.ActualSize.Height / 2));
            page.Canvas.RotateTransform(-45f);
            page.Canvas.DrawString("DRAFT", font, brush, 0, 0,
                new PdfStringFormat(PdfTextAlignment.Center));
            page.Canvas.Restore(state);
        }

        doc.SaveToFile("watermarked.pdf");
        doc.Close();
    }
}

After (IronPDF Watermark):

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Document</h1></body></html>"
);

var watermark = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronPdf.Imaging.Color.Gray,
    Opacity = 0.15,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("watermarked.pdf");

Password Protection

Security settings are straightforward to migrate. The property names differ but the concepts map directly.

Before (Spire.PDF):

using Spire.Pdf;
using Spire.Pdf.Security;
using System;

class PasswordBefore
{
    static void Main()
    {
        var doc = new PdfDocument();
        doc.LoadFromFile("source.pdf");

        doc.Security.KeySize = PdfEncryptionKeySize.Key128Bit;
        doc.Security.OwnerPassword = "admin456";
        doc.Security.UserPassword = "open123";
        doc.Security.Permissions = PdfPermissionsFlags.Default & ~PdfPermissionsFlags.EditContent;

        doc.SaveToFile("secured.pdf");
        doc.Close();
    }
}

After (IronPDF Password Protection):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Protected Document</h1></body></html>"
);

pdf.SecuritySettings.UserPassword = "open123";
pdf.SecuritySettings.OwnerPassword = "admin456";

pdf.SaveAs("secured.pdf");

What Are the Most Common Questions Teams Have During Migration?

Answer:

Q1: What happens to my FreeSpire.PDF code that was silently truncating at 10 pages?

If your application uses FreeSpire.PDF and relies on its behavior for documents over 10 pages, the free edition silently truncates output beyond that limit. After migrating to IronPDF, verify that previously-truncated PDFs now render fully. This is a common source of pleasant surprises — reports that were missing data suddenly show everything.

Q2: How do I handle the Spire.PDF drawing API code that has no direct IronPDF equivalent?

If the drawing API count dominates your codebase, estimate the HTML template rewrite effort before committing to a timeline. A complex multi-column report layout done in drawing API code may take several hours to recreate in HTML/CSS. A useful benchmark: compare the line count of the drawing API code with the HTML/CSS equivalent. In most report scenarios, the HTML version is significantly shorter.

Q3: How does resource cleanup differ between Spire.PDF and IronPDF?

Spire.PDF requires explicit doc.Close() for resource cleanup. With IronPDF, the standard C# using pattern handles disposal automatically:

// Spire pattern:
// var doc = new PdfDocument();
// try { ... } finally { doc.Close(); }

// IronPDF pattern:
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("output.pdf");
// Disposed automatically

Q4: Can I render multiple PDFs in parallel with IronPDF?

Yes. IronPDF supports concurrent rendering. Here is the pattern I recommend:

using IronPdf;
using System.Linq;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var htmlJobs = reportDataList.Select(data => BuildHtml(data)).ToArray();

var pdfs = await Task.WhenAll(htmlJobs.Select(async html =>
{
    var renderer = new ChromePdfRenderer();
    return await renderer.RenderHtmlAsPdfAsync(html);
}));

Console.WriteLine($"Generated {pdfs.Length} PDFs");
foreach (var pdf in pdfs) pdf.Dispose();

What Decision Framework Should I Use to Decide Whether to Migrate?

Answer:

Start with a quantitative audit. Run these commands against your codebase to understand your migration scope:

# Drawing API usage (higher migration cost)
rg "Canvas\.Draw\|DrawString\|DrawLine\|DrawRectangle\|DrawImage" --type cs -n | wc -l

# HTML/URL conversion usage (lower migration cost)
rg "LoadFromHtml\|LoadFromUrl\|ConvertHtmlString" --type cs -n | wc -l

If HTML conversion dominates, the migration is straightforward. If drawing API code dominates, you need to budget for HTML/CSS template rewrites.

Then walk through this checklist before writing any migration code:

Pre-Migration: Count HTML-to-PDF vs drawing API usage. Verify Spire.PDF edition (Free vs paid). Identify all Spire cross-product dependencies (Spire.Doc, Spire.XLS). Document the current .NET version issue that triggered the evaluation. Obtain an IronPDF license key.

Code Migration: Install IronPDF. Remove Spire packages. Add the license key at application startup. Replace LoadFromHtml() with ChromePdfRenderer.RenderHtmlAsPdfAsync(). Replace doc.SaveToFile() with pdf.SaveAs(). Replace the Spire merge pattern with PdfDocument.Merge(). Replace drawing API watermarks with TextStamper / ImageStamper. Replace doc.Security.* with pdf.SecuritySettings.*.

Testing: Render each HTML template and compare visual output. Verify previously-truncated Free edition PDFs now render all pages. Test merge output page count and order. Test password protection. Benchmark render time vs Spire baseline. Test in your target .NET version — the original upgrade trigger.

The .NET version trigger that started this evaluation tends to resolve cleanly once the HTML-to-PDF path is migrated. The drawing API code is the variable — measure how much of your PDF generation uses it before committing to a timeline.
