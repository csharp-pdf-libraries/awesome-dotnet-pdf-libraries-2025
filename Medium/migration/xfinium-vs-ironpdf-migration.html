Migrating from XFINIUM.PDF to IronPDF: What Actually Changes in Your Code

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate from XFINIUM.PDF to IronPDF?

Two major versions separate where you are from where the documentation is. XFINIUM.PDF shipped breaking API changes between versions, the upgrade guide is incomplete, and the version you're on has a known rendering issue with complex table layouts that the newer version supposedly fixes. Staying on the old version is a tax. Upgrading is an unknown risk. Switching is on the table.

I see teams evaluating XFINIUM.PDF alternatives for the same reasons over and over:

Version pinning friction hits first. XFINIUM.PDF has had breaking changes between major versions, and upgrade paths can be non-trivial. Then there is the HTML input gap — XFINIUM.PDF creates PDFs from a document model, not HTML. If your content starts as HTML templates, you must convert to the XFINIUM document model, adding intermediate steps. The drawing API itself requires explicit coordinate positioning for layout-heavy PDF generation, which means verbose code that is hard to maintain.

Teams with web backgrounds find HTML/CSS templates easier to maintain than drawing API code. Features available in newer XFINIUM versions may require an upgrade that introduces breaking changes. And .NET 6/7/8/9 support should be verified for your specific XFINIUM version.

How Do You Replace XFINIUM.PDF HTML-to-PDF Generation with IronPDF?

This is where the biggest architectural shift happens. XFINIUM builds PDFs from a document model — pages, graphics, fonts, coordinates. IronPDF renders PDFs from HTML. These are different mental models, and the code reflects it.

XFINIUM Document Model (Before)

using Xfinium.Pdf;          // VERIFY
using Xfinium.Pdf.Graphics; // VERIFY
using System;
using System.IO;
using System.Text;

class ReportBefore
{
    static void Main()
    {
        // XFINIUM: build document model — no HTML input
        // All names below are illustrative — VERIFY at xfinium.com

        var doc = new PdfFixedDocument();
        var page = doc.Pages.Add();
        var graphics = page.Graphics;

        var titleFont = new PdfStandardFont(PdfStandardFontName.HelveticaBold, 18);
        var bodyFont  = new PdfStandardFont(PdfStandardFontName.Helvetica, 12);
        var brush     = new PdfBrush(PdfRgbColor.Black);

        // Manual positioning for each element
        graphics.DrawString("Q3 Invoice #4421", titleFont, brush, 40, 40);
        graphics.DrawString("Customer: Acme Corp",    bodyFont, brush, 40, 80);
        graphics.DrawString("Total Due: $8,400.00",   bodyFont, brush, 40, 110);

        using var stream = File.Create("invoice.pdf");
        doc.Save(stream);
        Console.WriteLine("Saved invoice.pdf — verify all XFINIUM API names");
    }
}

IronPDF HTML Rendering (After)

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

// HTML/CSS handles layout — no manual coordinate math
var html = @"
    <html>
    <head>
    <style>
        body { font-family: Arial, sans-serif; padding: 40px; }
        h1 { font-size: 20px; }
        .field { margin-top: 8px; font-size: 13px; }
        .total { font-weight: bold; font-size: 16px; margin-top: 20px; }
    </style>
    </head>
    <body>
        <h1>Q3 Invoice #4421</h1>
        <div class='field'>Customer: Acme Corp</div>
        <div class='total'>Total Due: $8,400.00</div>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");
Console.WriteLine($"Saved invoice.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

The coordinate math disappears entirely. Your layout lives in CSS, which your team already knows how to maintain.

How Do You Merge PDFs After Migrating from XFINIUM?

XFINIUM's merge approach involves loading documents and appending pages manually. With IronPDF, merging becomes a single static method call.

XFINIUM Page Append (Before)

using Xfinium.Pdf;  // VERIFY
using System;
using System.IO;

class MergeBefore
{
    static void Main()
    {
        // VERIFY: XFINIUM merge API — may be AppendPage, AppendDocument, or Clone
        // All names below are illustrative

        using var s1 = File.OpenRead("section1.pdf");
        var doc1 = new PdfFixedDocument(s1); // VERIFY

        using var s2 = File.OpenRead("section2.pdf");
        var doc2 = new PdfFixedDocument(s2); // VERIFY

        // VERIFY: XFINIUM page append/merge API
        // for (int i = 0; i < doc2.Pages.Count; i++)
        //     doc1.Pages.Add(doc2.Pages[i]); // verify method

        using var outStream = File.Create("merged.pdf");
        doc1.Save(outStream); // VERIFY
        Console.WriteLine("Merged — verify XFINIUM page append API");
    }
}

IronPDF Merge (After)

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

var results = await Task.WhenAll(
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 1</h1></body></html>"),
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 2</h1></body></html>")
);

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("merged.pdf");
Console.WriteLine($"Merged: {merged.PageCount} pages");

The loop-and-append pattern collapses into PdfDocument.Merge(). You can also render multiple documents in parallel with Task.WhenAll before merging, as shown above. See the full merge and split guide for additional patterns.

How Do You Apply Watermarks After Migrating from XFINIUM?

In XFINIUM, watermarking means drawing semi-transparent text on the graphics layer of each page, managing transforms and opacity manually. IronPDF replaces that with a declarative stamper object.

XFINIUM Drawing Layer Watermark (Before)

using Xfinium.Pdf;           // VERIFY
using Xfinium.Pdf.Graphics;  // VERIFY
using System;
using System.IO;

class WatermarkBefore
{
    static void Main()
    {
        // XFINIUM: apply watermark by drawing semi-transparent text on each page
        // VERIFY: all API names, especially transparency/opacity support

        using var s = File.OpenRead("source.pdf");
        var doc = new PdfFixedDocument(s); // VERIFY

        foreach (var page in doc.Pages)
        {
            var graphics = page.Graphics;
            // VERIFY: XFINIUM transparency API for watermark text
            var font  = new PdfStandardFont(PdfStandardFontName.Helvetica, 60); // VERIFY
            var brush = new PdfBrush(new PdfRgbColor(0.5, 0.5, 0.5)); // VERIFY alpha

            graphics.SaveGraphicsState();
            // VERIFY: translate + rotate transform API
            graphics.DrawString("DRAFT", font, brush, 200, 350); // VERIFY
            graphics.RestoreGraphicsState();
        }

        using var outStream = File.Create("watermarked.pdf");
        doc.Save(outStream);
        Console.WriteLine("Verify XFINIUM watermark API — transparency support may vary by version");
    }
}

IronPDF TextStamper Watermark (After)

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Document</h1></body></html>");

// https://ironpdf.com/how-to/custom-watermark/
var watermark = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronPdf.Imaging.Color.Gray,
    Opacity = 0.15,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("watermarked.pdf");
Console.WriteLine("Watermark applied");

No more manual graphics state management or coordinate math. The watermark API handles positioning, opacity, and alignment declaratively.

How Do You Set Password Protection After Migrating from XFINIUM?

XFINIUM uses a PdfStandardSecurity class that you assign to the document before saving. IronPDF exposes security settings directly on the PdfDocument object.

XFINIUM Security API (Before)

using Xfinium.Pdf;           // VERIFY
using Xfinium.Pdf.Security;  // VERIFY namespace
using System;
using System.IO;

class PasswordBefore
{
    static void Main()
    {
        var doc = new PdfFixedDocument(); // VERIFY
        doc.Pages.Add();
        // ... build document ...

        // VERIFY: XFINIUM security API — class and property names vary by version
        var security = new PdfStandardSecurity(); // VERIFY class name
        security.UserPassword = "open123";         // VERIFY property
        security.OwnerPassword = "admin456";       // VERIFY property
        // security.Permissions = ...;             // VERIFY permissions API
        doc.Security = security;                   // VERIFY assignment

        using var stream = File.Create("secured.pdf");
        doc.Save(stream);
        Console.WriteLine("Saved secured.pdf — verify XFINIUM security API names");
    }
}

IronPDF Security Settings (After)

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Protected</h1></body></html>");

// https://ironpdf.com/how-to/pdf-permissions-passwords/
pdf.SecuritySettings.UserPassword = "open123";
pdf.SecuritySettings.OwnerPassword = "admin456";

pdf.SaveAs("secured.pdf");
Console.WriteLine("Saved secured.pdf");

The separate security object goes away. You set passwords directly on the document before saving. See the full permissions and passwords guide for additional options.

What Are the Common Troubleshooting Issues During This Migration?

I have seen the same patterns come up repeatedly when teams move from XFINIUM to IronPDF. Here are the ones worth preparing for.

Answer:

Version Breaking Changes — Which API Names Are Correct?

Symptom: CS0117 or CS1061 errors after switching between XFINIUM versions; method names in old code do not exist in the new version.

XFINIUM has had API surface changes between major versions. Before migration, capture the exact version in use and map the exact method signatures. Before removing XFINIUM, document every method call. Each one becomes a migration point. The IronPDF equivalent is HTML for drawing API calls; the API calls map closely for save/load/security operations.

Drawing API Code Is Too Large to Rewrite Immediately

Symptom: The codebase has dozens of graphics.DrawString, DrawRectangle, FillPath calls that all need to become HTML/CSS. The migration scope is larger than the sprint allows.

Migrate in phases. Start with the HTML generation path and leave the drawing API code in place temporarily with both libraries installed. Identify which code paths generate from HTML vs the drawing API, migrate HTML-first paths as new capability, then migrate drawing API paths to HTML templates over subsequent sprints.

Text Extraction Behavior Differs

Symptom: After migrating text extraction from PdfContentExtractionSink to pdf.ExtractAllText(), the extracted text differs in whitespace, order, or Unicode handling.

Text extraction output varies by library implementation. If exact text positioning matters, verify what the new output produces before replacing any downstream logic that parses extracted text. See the text extraction guide for per-page extraction options.

Frequently Asked Questions

Can I run XFINIUM.PDF and IronPDF side by side during migration?

Answer:

Yes. Both libraries can coexist in the same project. This is the recommended approach when your codebase has extensive drawing API code that cannot be rewritten in a single sprint. Install IronPDF alongside XFINIUM, migrate HTML generation paths first, then convert drawing API code to HTML templates over subsequent sprints.

The namespaces do not conflict — XFINIUM uses Xfinium.Pdf.* and IronPDF uses IronPdf.

How do I handle the save pattern difference between XFINIUM and IronPDF?

Answer:

XFINIUM uses doc.Save(stream) which writes to a stream and keeps it open. IronPDF gives you three options: pdf.SaveAs("output.pdf") for file output, pdf.Stream.CopyTo(yourOutputStream) for stream output, or pdf.BinaryData for raw bytes. Copy bytes before disposal if using a using block.

The key difference is that IronPDF is object-centric rather than stream-centric.

Does IronPDF support the same page indexing as XFINIUM?

Answer:

Both XFINIUM and IronPDF use 0-based page indexing. This typically does not require code changes during migration. Page count moves from doc.Pages.Count to pdf.PageCount, and page removal moves from doc.Pages.RemoveAt(i) to pdf.RemovePage(index).

What is the biggest variable in migration timeline?

Answer:

The drawing API scope. Document model code that draws shapes and places text with explicit coordinates needs HTML equivalents. The more DrawString, DrawRectangle, and FillPath calls you have, the longer the migration takes. HTML input paths migrate in hours. Save, load, merge, watermark, and security operations are straightforward low-complexity swaps.

Use rg "DrawString|DrawRect|FillPath" --type cs | wc -l to measure the scope before committing.

How do I handle parallel rendering in IronPDF?

Answer:

IronPDF supports async rendering natively. You can render multiple documents in parallel using Task.WhenAll with separate ChromePdfRenderer instances:

var pdfs = await Task.WhenAll(reportDataList.Select(async data =>
{
    var renderer = new ChromePdfRenderer();
    return await renderer.RenderHtmlAsPdfAsync(BuildHtml(data));
}));

Dispose each PdfDocument when done, just as you would dispose PdfFixedDocument in XFINIUM. See the async rendering guide for details.

Conclusion

The version pinning constraint — staying on an old version to avoid breaking changes while watching features accumulate in the newer version — is a common forcing function. Migrating to a different library removes that constraint and resets the version timeline.

The drawing API scope is the variable that determines timeline. Document model code that draws shapes and places text needs HTML equivalents; the more of it there is, the longer the migration takes. HTML input paths migrate in hours.

Start by auditing your XFINIUM usage with the search commands in this article, scope the drawing API rewrite, and migrate in phases. The save/load/merge/security operations are straightforward swaps. The HTML rendering path is a net-new capability that eliminates the intermediate conversion step entirely.
