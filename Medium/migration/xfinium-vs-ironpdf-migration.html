Migrating from XFINIUM.PDF to IronPDF: What Actually Changes in Your Code

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate from XFINIUM.PDF to IronPDF?

Two major versions separate where you are from where the documentation is. XFINIUM.PDF shipped breaking API changes between versions, the upgrade guide is incomplete, and the version you're on has a known rendering issue with complex table layouts that the newer version supposedly fixes. Staying on the old version is a tax. Upgrading is an unknown risk. Switching is on the table.

I see teams evaluating XFINIUM.PDF alternatives for the same reasons over and over:

Version pinning friction hits first. XFINIUM.PDF has had breaking changes between major versions, and upgrade paths can be non-trivial. Then there is the HTML input gap — XFINIUM has a PdfFlowDocument + PdfFlowContent.AppendHtml path, but it is a flow-document HTML interpreter, not a real browser. CSS coverage is limited, and if your templates rely on flexbox, grid, web fonts, or JavaScript, you end up either rewriting them or layering in the drawing API. The drawing API itself requires explicit coordinate positioning for layout-heavy PDF generation, which means verbose code that is hard to maintain.

Teams with web backgrounds find Chromium-rendered HTML/CSS templates easier to maintain than the flow-document or drawing API. Features available in newer XFINIUM versions may require an upgrade that introduces breaking changes. And .NET 6/7/8/9 support should be verified for your specific XFINIUM version.

XFINIUM.PDF vs IronPDF Comparison

  • Focus. XFINIUM is PDF creation and editing via a fixed document model and drawing API, with a flow-document HTML path on the side. IronPDF is HTML-to-PDF first, with full PDF manipulation on the same object.
  • Pricing. Both are commercial — confirm current numbers at xfinium.com and ironsoftware.com.
  • API style. XFINIUM exposes PdfFixedDocument, PdfPage.Graphics, and PdfFlowDocument. IronPDF exposes ChromePdfRenderer and PdfDocument, with HTML or URL as the input.
  • Learning curve. XFINIUM is medium — you learn its document model and drawing primitives. IronPDF is low for web developers; HTML/CSS is the input.
  • HTML rendering. XFINIUM uses a flow-document HTML interpreter with limited CSS coverage. IronPDF uses embedded Chromium.
  • Page indexing. XFINIUM uses 0-based indexing on document.Pages. IronPDF is also 0-based.
  • Thread safety. Verify in the XFINIUM docs for your version. For IronPDF, prefer separate ChromePdfRenderer instances per thread.
  • Namespace. Xfinium.Pdf.* versus IronPdf.

How Much Effort Does Each Feature Require to Migrate?

Not every part of the migration is equally complex. I have mapped out the effort by feature so you can scope the work before committing to a timeline.

  • HTML string to PDF — Low. XFINIUM has PdfFlowDocument + PdfFlowContent.AppendHtml, but CSS coverage is limited; IronPDF uses ChromePdfRenderer.RenderHtmlAsPdfAsync() through Chromium. Templates that already work in a browser usually port over directly.
  • Document model to PDF — High (rewrite). XFINIUM's drawing API (PdfPage, page.Graphics) becomes an HTML/CSS template. The amount of DrawString/DrawImage/DrawLine in your codebase is the main timeline driver.
  • Save to file — Low. document.Save(path) becomes pdf.SaveAs(path).
  • Save to stream — Low. document.Save(stream) becomes pdf.Stream or pdf.BinaryData.
  • Load existing PDF — Low. new PdfFixedDocument(stream) becomes PdfDocument.FromFile(path) or PdfDocument.FromBinaryData(bytes).
  • Merge PDFs — Low. The XFINIUM loop that does output.Pages.Add(pdf1.Pages[i]) collapses to PdfDocument.Merge(pdf1, pdf2).
  • Watermark — Low. Graphics-layer drawing is replaced by a declarative TextStamper or ImageStamper.
  • Password protection — Low. document.Security becomes pdf.SecuritySettings with the same conceptual properties.
  • Text extraction — Low. PdfContentExtractionSink becomes pdf.ExtractAllText(); verify whitespace and ordering against your downstream parsers.
  • Form fields — Medium. XFINIUM has a full AcroForm API; IronPDF form-fill depth varies by field type — verify against the IronPDF forms documentation for your specific field set.
  • Annotations — Medium. XFINIUM exposes a broad annotations API; IronPDF coverage is narrower — verify against the IronPDF annotations docs for the annotation types you actually use.

Decision Matrix

  • If you are stuck on an old XFINIUM version because the upgrade introduces breaking changes, switching resolves the version-pinning constraint outright.
  • If HTML-to-PDF from web templates is the primary need and you are hitting CSS gaps in PdfFlowDocument, switch — IronPDF is HTML-first and uses Chromium.
  • If your codebase is heavy on programmatic drawing or AcroForm work, evaluate migration cost first — the drawing API to HTML rewrite is the main effort.
  • If a .NET version upgrade is blocked by XFINIUM compatibility, verify support at xfinium.com and switch if it isn't covered.

How Do You Replace XFINIUM.PDF HTML-to-PDF Generation with IronPDF?

This is where the biggest architectural shift happens. XFINIUM builds PDFs from a fixed document model — pages, graphics, fonts, coordinates — and offers a side path through PdfFlowDocument that interprets a subset of HTML. IronPDF renders PDFs from HTML through Chromium. The mental models are different, and the code reflects it.

XFINIUM Document Model (Before)

using Xfinium.Pdf;
using Xfinium.Pdf.Graphics;
using Xfinium.Pdf.Core;
using System;
using System.IO;

class ReportBefore
{
    static void Main()
    {
        // XFINIUM: build the fixed document model with the drawing API.
        var doc = new PdfFixedDocument();
        var page = doc.Pages.Add();
        var graphics = page.Graphics;

        var titleFont = new PdfStandardFont(PdfStandardFontFace.HelveticaBold, 18);
        var bodyFont  = new PdfStandardFont(PdfStandardFontFace.Helvetica, 12);
        var brush     = new PdfBrush(PdfRgbColor.Black);

        // Manual positioning for each element
        graphics.DrawString("Q3 Invoice #4421", titleFont, brush, 40, 40);
        graphics.DrawString("Customer: Acme Corp",    bodyFont, brush, 40, 80);
        graphics.DrawString("Total Due: $8,400.00",   bodyFont, brush, 40, 110);

        doc.Save("invoice.pdf");
        Console.WriteLine("Saved invoice.pdf");
    }
}

IronPDF HTML Rendering (After)

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// HTML/CSS handles layout — no manual coordinate math
var html = @"
    <html>
    <head>
    <style>
        body { font-family: Arial, sans-serif; padding: 40px; }
        h1 { font-size: 20px; }
        .field { margin-top: 8px; font-size: 13px; }
        .total { font-weight: bold; font-size: 16px; margin-top: 20px; }
    </style>
    </head>
    <body>
        <h1>Q3 Invoice #4421</h1>
        <div class='field'>Customer: Acme Corp</div>
        <div class='total'>Total Due: $8,400.00</div>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");
Console.WriteLine($"Saved invoice.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

The coordinate math disappears entirely. Your layout lives in CSS, which your team already knows how to maintain.

How Do You Merge PDFs After Migrating from XFINIUM?

XFINIUM's merge approach involves loading documents and appending pages manually. With IronPDF, merging becomes a single static method call.

XFINIUM Page Append (Before)

using Xfinium.Pdf;
using System;
using System.IO;

class MergeBefore
{
    static void Main()
    {
        var output = new PdfFixedDocument();

        using var file1 = File.OpenRead("section1.pdf");
        var pdf1 = new PdfFixedDocument(file1);

        using var file2 = File.OpenRead("section2.pdf");
        var pdf2 = new PdfFixedDocument(file2);

        for (int i = 0; i < pdf1.Pages.Count; i++)
            output.Pages.Add(pdf1.Pages[i]);

        for (int i = 0; i < pdf2.Pages.Count; i++)
            output.Pages.Add(pdf2.Pages[i]);

        output.Save("merged.pdf");
        Console.WriteLine("Merged section1.pdf and section2.pdf");
    }
}

IronPDF Merge (After)

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();

var results = await Task.WhenAll(
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 1</h1></body></html>"),
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section 2</h1></body></html>")
);

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("merged.pdf");
Console.WriteLine($"Merged: {merged.PageCount} pages");

The loop-and-append pattern collapses into PdfDocument.Merge(). You can also render multiple documents in parallel with Task.WhenAll before merging, as shown above. See the full merge and split guide for additional patterns.

How Do You Apply Watermarks After Migrating from XFINIUM?

In XFINIUM, watermarking means drawing semi-transparent text on the graphics layer of each page, managing transforms and opacity manually. IronPDF replaces that with a declarative stamper object.

XFINIUM Drawing Layer Watermark (Before)

using Xfinium.Pdf;
using Xfinium.Pdf.Graphics;
using Xfinium.Pdf.Core;
using System;
using System.IO;

class WatermarkBefore
{
    static void Main()
    {
        // XFINIUM: apply watermark by drawing large gray text on each page.
        // Verify the exact transparency/transform API for your version —
        // alpha and rotation helpers vary across XFINIUM major releases.

        using var s = File.OpenRead("source.pdf");
        var doc = new PdfFixedDocument(s);

        var font  = new PdfStandardFont(PdfStandardFontFace.Helvetica, 60);
        var brush = new PdfBrush(new PdfRgbColor(180, 180, 180));

        foreach (var page in doc.Pages)
        {
            var graphics = page.Graphics;
            graphics.SaveGraphicsState();
            graphics.DrawString("DRAFT", font, brush, 200, 350);
            graphics.RestoreGraphicsState();
        }

        doc.Save("watermarked.pdf");
        Console.WriteLine("Watermarked all pages");
    }
}

IronPDF TextStamper Watermark (After)

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Document</h1></body></html>");

// https://ironpdf.com/how-to/custom-watermark/
var watermark = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronSoftware.Drawing.Color.Gray,
    Opacity = 15,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("watermarked.pdf");
Console.WriteLine("Watermark applied");

No more manual graphics state management or coordinate math. The watermark API handles positioning, opacity, and alignment declaratively.

How Do You Set Password Protection After Migrating from XFINIUM?

XFINIUM exposes security settings on document.Security, which you populate before saving. IronPDF exposes the same conceptual settings on pdf.SecuritySettings.

XFINIUM Security API (Before)

using Xfinium.Pdf;
using Xfinium.Pdf.Security;
using System;

class PasswordBefore
{
    static void Main()
    {
        var doc = new PdfFixedDocument();
        doc.Pages.Add();
        // ... build document ...

        // Permissions and encryption-algorithm property names vary by
        // XFINIUM major version — verify against your version's docs.
        doc.Security.UserPassword = "open123";
        doc.Security.OwnerPassword = "admin456";
        doc.Security.AllowPrinting = false;
        doc.Security.AllowContentCopying = false;
        doc.Security.EncryptionAlgorithm = PdfEncryptionAlgorithm.Aes256;

        doc.Save("secured.pdf");
        Console.WriteLine("Saved secured.pdf");
    }
}

IronPDF Security Settings (After)

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Protected</h1></body></html>");

// https://ironpdf.com/how-to/pdf-permissions-passwords/
pdf.SecuritySettings.UserPassword = "open123";
pdf.SecuritySettings.OwnerPassword = "admin456";
pdf.SecuritySettings.AllowUserPrinting = PdfPrintSecurity.NoPrint;
pdf.SecuritySettings.AllowUserCopyPasteContent = false;

pdf.SaveAs("secured.pdf");
Console.WriteLine("Saved secured.pdf");

The separate security object goes away. You set passwords directly on the document before saving. See the full permissions and passwords guide for additional options.

What Are the Key API Mappings Between XFINIUM.PDF and IronPDF?

I have compiled the full mapping tables so you can look up any XFINIUM call and find its IronPDF equivalent in one place.

Namespace Mapping

  • Xfinium.Pdf → IronPdf — core namespace.
  • Xfinium.Pdf.Graphics → not needed when you migrate to the HTML path; drawing is replaced by HTML/CSS.
  • Xfinium.Pdf.Security → IronPdf — security lives on pdf.SecuritySettings on the output document.
  • Xfinium.Pdf.FlowDocument → not needed; ChromePdfRenderer.RenderHtmlAsPdfAsync replaces the flow-document HTML path.

Core Class Mapping

  • PdfFixedDocument → ChromePdfRenderer — different mental model; HTML is the input.
  • PdfFlowDocument + PdfFlowContent.AppendHtml → ChromePdfRenderer.RenderHtmlAsPdfAsync — Chromium replaces the flow-document HTML interpreter.
  • PdfPage / page.Graphics → not used on the HTML path; drawing is replaced by HTML and CSS.
  • document.Security → pdf.SecuritySettings — passwords and permissions live on the output document.
  • (no XFINIUM equivalent) → PdfDocument — IronPDF's manipulation surface for load, merge, stamp, extract.

Document Loading Methods

  • Create from model. new PdfFixedDocument() + pages → renderer.RenderHtmlAsPdfAsync(html).
  • Load existing PDF. new PdfFixedDocument(stream) → PdfDocument.FromFile(path) or PdfDocument.FromBinaryData(bytes).
  • Save to file. document.Save(path) → pdf.SaveAs(path).
  • Save to stream. document.Save(stream) → pdf.Stream.CopyTo(stream).

Page Operations

  • Page count. document.Pages.Count → pdf.PageCount.
  • Remove page. document.Pages.RemoveAt(i) → pdf.RemovePages(index).
  • Extract text. PdfContentExtractionSink → pdf.ExtractAllText().
  • Rotate. Verify the rotation API for your XFINIUM version; IronPDF exposes per-page rotation through pdf.RotateAllPages / pdf.RotatePage — confirm the signatures against the current IronPDF docs.

Merge / Split Operations

  • Merge. The XFINIUM loop that calls output.Pages.Add(pdf1.Pages[i]) across documents → PdfDocument.Merge(doc1, doc2).
  • Split. Verify the split helper for your XFINIUM version; IronPDF supports pdf.CopyPage and pdf.CopyPages — see the merge and split guide.

What Are the Common Troubleshooting Issues During This Migration?

I have seen the same patterns come up repeatedly when teams move from XFINIUM to IronPDF. Here are the ones worth preparing for.

Version Breaking Changes — Which API Names Are Correct?

Symptom: CS0117 or CS1061 errors after switching between XFINIUM versions; method names in old code do not exist in the new version.

XFINIUM has had API surface changes between major versions. Before migration, capture the exact version in use and map the exact method signatures. Before removing XFINIUM, document every method call. Each one becomes a migration point. The IronPDF equivalent is HTML for drawing API calls; the API calls map closely for save/load/security operations.

Drawing API Code Is Too Large to Rewrite Immediately

Symptom: The codebase has dozens of graphics.DrawString, DrawRectangle, FillPath calls that all need to become HTML/CSS. The migration scope is larger than the sprint allows.

Migrate in phases. Start with the HTML generation path and leave the drawing API code in place temporarily with both libraries installed. Identify which code paths generate from HTML vs the drawing API, migrate HTML-first paths as new capability, then migrate drawing API paths to HTML templates over subsequent sprints.

Text Extraction Behavior Differs

Symptom: After migrating text extraction from PdfContentExtractionSink to pdf.ExtractAllText(), the extracted text differs in whitespace, order, or Unicode handling.

Text extraction output varies by library implementation. If exact text positioning matters, verify what the new output produces before replacing any downstream logic that parses extracted text. See the text extraction guide for per-page extraction options.

Frequently Asked Questions

Can I run XFINIUM.PDF and IronPDF side by side during migration?

Yes. Both libraries can coexist in the same project. This is the recommended approach when your codebase has extensive drawing API code that cannot be rewritten in a single sprint. Install IronPDF alongside XFINIUM, migrate HTML generation paths first, then convert drawing API code to HTML templates over subsequent sprints.

The namespaces do not conflict — XFINIUM uses Xfinium.Pdf.* and IronPDF uses IronPdf.

How do I handle the save pattern difference between XFINIUM and IronPDF?

XFINIUM uses doc.Save(stream) which writes to a stream and keeps it open. IronPDF gives you three options: pdf.SaveAs("output.pdf") for file output, pdf.Stream.CopyTo(yourOutputStream) for stream output, or pdf.BinaryData for raw bytes. Copy bytes before disposal if using a using block.

The key difference is that IronPDF is object-centric rather than stream-centric.

Does IronPDF support the same page indexing as XFINIUM?

Both XFINIUM and IronPDF use 0-based page indexing. This typically does not require code changes during migration. Page count moves from document.Pages.Count to pdf.PageCount, and page removal moves from document.Pages.RemoveAt(i) to pdf.RemovePages(index).

What is the biggest variable in migration timeline?

The drawing API scope. Document model code that draws shapes and places text with explicit coordinates needs HTML equivalents. The more DrawString, DrawRectangle, and FillPath calls you have, the longer the migration takes. HTML input paths migrate in hours. Save, load, merge, watermark, and security operations are straightforward low-complexity swaps.

Use rg "DrawString|DrawRect|FillPath" --type cs | wc -l to measure the scope before committing.

How do I handle parallel rendering in IronPDF?

IronPDF supports async rendering natively. You can render multiple documents in parallel using Task.WhenAll with separate ChromePdfRenderer instances:

var pdfs = await Task.WhenAll(reportDataList.Select(async data =>
{
    var renderer = new ChromePdfRenderer();
    return await renderer.RenderHtmlAsPdfAsync(BuildHtml(data));
}));

Dispose each PdfDocument when done, just as you would dispose PdfFixedDocument in XFINIUM. See the async rendering guide for details.

Conclusion

The version pinning constraint — staying on an old version to avoid breaking changes while watching features accumulate in the newer version — is a common forcing function. Migrating to a different library removes that constraint and resets the version timeline.

The drawing API scope is the variable that determines timeline. Document model code that draws shapes and places text needs HTML equivalents; the more of it there is, the longer the migration takes. HTML input paths migrate in hours.

Start by auditing your XFINIUM usage with the search commands in this article, scope the drawing API rewrite, and migrate in phases. The save/load/merge/security operations are straightforward swaps. The HTML rendering path is a net-new capability that eliminates the intermediate conversion step entirely.
