Migrating from NReco to IronPDF: What Breaks, What Doesn't

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

The PDF looks close. The heading font is right, the table renders, the footer is in the right position — but the column widths are slightly off, the image didn't align correctly, and the multi-line cell that looked fine in browser preview wrapped differently on page 3. You've spent hours tweaking CSS and the gap keeps closing but never quite closes.

That is the CSS fidelity wall that teams hit with wkhtmltopdf-based generators, and NReco PDF Generator is a .NET wrapper around wkhtmltopdf. This article covers migrating from NReco PDF Generator to IronPDF and the troubleshooting patterns for the most common breakage points.

Why Should I Migrate from NReco PDF Generator?

Eight neutral triggers for leaving NReco PDF Generator:

WebKit vs Chromium rendering gap — modern CSS (Grid, Flexbox, CSS variables, some pseudo-elements) works in Chromium-based renderers but not in wkhtmltopdf's WebKit.

wkhtmltopdf is no longer actively developed — the upstream repository was archived on January 2, 2023.

Binary dependency management — NReco embeds wkhtmltopdf 0.12.6 binaries for Windows, but Linux and macOS deployments still require installing the wkhtmltopdf binary separately. Docker images require explicit installation steps.

No PDF manipulation — NReco generates PDFs; it doesn't merge, split, watermark, or encrypt them. Secondary libraries are needed.

Subprocess or P/Invoke architecture — both patterns add complexity: process management, error handling, timeout management, interop fragility.

JavaScript rendering — wkhtmltopdf's JS support is limited. Dynamic content generated via modern JS frameworks may not render.

Font handling differences — system fonts on the render server must match what the template expects; font subsetting behavior differs from Chromium.

Stalled maintenance — NReco.PdfGenerator 1.2.1 was published on January 8, 2023 with no further releases on NuGet, and the underlying wkhtmltopdf 0.12.6 carries unpatched CVEs including CVE-2020-21365 (directory traversal / local file read) and CVE-2022-35583 (SSRF, CVSS 9.8).

How Do NReco PDF Generator and IronPDF Compare Feature by Feature?

  • Focus. NReco is HTML-to-PDF via wkhtmltopdf; IronPDF does HTML-to-PDF plus PDF manipulation in one library.
  • Pricing. NReco.PdfGenerator is free for non-SaaS single-server production; SaaS, multi-server, or redistribution scenarios require the $199 enterprise pack. IronPDF is commercial.
  • API style. NReco wraps the wkhtmltopdf subprocess/native binary; IronPDF runs in-process as a .NET library.
  • Learning curve. Low for basic NReco use; medium for IronPDF given the broader surface area.
  • HTML rendering. NReco uses WebKit (wkhtmltopdf, pre-modern CSS); IronPDF uses Chromium with full CSS3.
  • Page indexing. NReco doesn't expose a page model post-render; IronPDF is 0-based.
  • Thread safety. NReco gets process-level isolation from the subprocess; IronPDF supports renderer instance reuse — see async docs.
  • Namespace. NReco.PdfGenerator vs IronPdf.

How Hard Is Each Feature to Migrate?

  • HTML string to PDF — Low. HtmlToPdfConverter.GeneratePdf() maps directly to RenderHtmlAsPdf().
  • URL to PDF — Low. NReco uses GeneratePdfFromFile(url, null); IronPDF has a dedicated RenderUrlAsPdf().
  • Custom margins — Low. NReco uses a PageMargins object or CustomWkHtmlArgs; IronPDF uses individual MarginTop/MarginBottom/MarginLeft/MarginRight properties.
  • Headers / footers — Medium. NReco's PageHeaderHtml/PageFooterHtml map to IronPDF's HtmlHeader/HtmlFooter with the HtmlHeaderFooter type, and the placeholder syntax changes.
  • wkhtmltopdf CLI flags — Medium. Each CustomWkHtmlArgs flag maps to a typed ChromePdfRenderOptions property.
  • Merge PDFs — Low. Required an external library with NReco; native in IronPDF via PdfDocument.Merge.
  • Watermark — Low. Required an external library with NReco; native in IronPDF via TextStamper.
  • Password protection — Low. Required an external library with NReco; native in IronPDF via SecuritySettings.
  • CSS Grid / Flexbox layouts — Low to fix. Broken or partial in WebKit; automatic in Chromium.
  • JavaScript rendering — Low to improve. Limited in WebKit; Chromium supports modern JS frameworks.
  • Binary management — Eliminated. wkhtmltopdf binary deployment goes away entirely.

When Does It Make Sense to Migrate?

  • If CSS fidelity is the primary pain, a Chromium-based renderer is the fix — evaluate IronPDF or PuppeteerSharp.
  • If wkhtmltopdf works fine for your templates, there is no urgent reason to migrate — though the wkhtmltopdf maintenance trajectory is a factor worth weighing.
  • If you need merge, watermark, or security in the same library, IronPDF consolidates these and removes the secondary library dependency.
  • If budget rules out commercial licensing, PuppeteerSharp (free, Chromium-based) is a viable alternative.

How Do I Diagnose Fidelity Problems Before Migrating?

Before migrating, confirm that the fidelity gap is renderer-related (WebKit vs Chromium) rather than a configuration issue. Some CSS fidelity problems with NReco/wkhtmltopdf are fixable.

Flexbox or CSS Grid layout broken

wkhtmltopdf uses a WebKit build from approximately 2012. CSS Flexbox and Grid have very limited support:

# Check your wkhtmltopdf version
wkhtmltopdf --version
# Below 0.12.6, Flexbox may not work at all
# Even 0.12.6 has incomplete Flexbox — Grid is largely unsupported

Fix within NReco: Convert layouts to display: table or float-based layouts (pre-Flexbox techniques). This works but means maintaining two CSS codebases if your HTML also renders in a browser.

Fix via migration: Chromium-based renderer supports current CSS — no layout workarounds needed.

@media print rules not applying

// NReco: enable media type setting
var htmlToPdf = new NReco.PdfGenerator.HtmlToPdfConverter();
htmlToPdf.CustomWkHtmlArgs = "--print-media-type";
// Set this flag if your CSS uses @media print rules

If it was already set and still broken, the issue is WebKit-level — a renderer change is needed.

Background colors/images not rendering

// NReco: enable background rendering
var htmlToPdf = new NReco.PdfGenerator.HtmlToPdfConverter();
htmlToPdf.CustomWkHtmlArgs = "--background";
// Also check: --disable-smart-shrinking

If after applying these fixes the output still doesn't match, the gap is likely renderer-level and migration is the more efficient path.

How Do I Set Up the Migration?

Find NReco references in your codebase

# Find all NReco.PdfGenerator usage
rg -l "NReco\|HtmlToPdfConverter" --type cs

# Find converter instantiation
rg "HtmlToPdfConverter\|new.*PdfConverter" --type cs -n

# Find CustomWkHtmlArgs usage — flags to migrate to options
rg "CustomWkHtmlArgs\|CustomWkHtmlPageArgs\|PdfToolPath\|WkHtmlToPdfExeName" --type cs -n

# Find using directives
rg "using NReco" --type cs -n

# Find wkhtmltopdf binary references
rg "wkhtmltopdf" . -n 2>/dev/null
find . -name "wkhtmltopdf*" 2>/dev/null

Remove NReco, install IronPDF

# Remove NReco PDF Generator
dotnet remove package NReco.PdfGenerator

# Install IronPDF
dotnet add package IronPdf
dotnet restore

Also remove wkhtmltopdf from Docker images and CI/CD:

# Find Docker references to wkhtmltopdf
rg "wkhtmltopdf" Dockerfile* -n

# Find CI/CD install steps
rg "wkhtmltopdf" .github/ azure-pipelines.yml -l 2>/dev/null

How Do I Migrate Basic HTML-to-PDF Conversion?

Before (NReco PDF Generator)

using NReco.PdfGenerator;
using System;
using System.IO;

class HtmlToPdfExample
{
    static void Main()
    {
        var htmlToPdf = new HtmlToPdfConverter();

        // Common NReco configuration pattern
        htmlToPdf.CustomWkHtmlArgs =
            "--print-media-type --background --dpi 180 " +
            "--margin-top 10mm --margin-bottom 10mm " +
            "--margin-left 15mm --margin-right 15mm";

        // Optional: set explicit binary path (Linux/macOS)
        // htmlToPdf.PdfToolPath = "/usr/local/bin/";
        // htmlToPdf.WkHtmlToPdfExeName = "wkhtmltopdf";

        string html = @"
            <html><head><style>
                body { font-family: Arial; font-size: 10pt; }
                h1 { font-size: 18pt; }
                .total { font-weight: bold; }
            </style></head><body>
                <h1>Invoice #1234</h1>
                <p>Customer: ACME Corp</p>
                <p class='total'>Total: $500.00</p>
            </body></html>";

        byte[] pdfBytes = htmlToPdf.GeneratePdf(html);
        File.WriteAllBytes("invoice.pdf", pdfBytes);
        Console.WriteLine("Saved: invoice.pdf");
    }
}

After (IronPDF)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize  = IronPdf.Rendering.PdfPaperSize.A4;
renderer.RenderingOptions.MarginTop    = 10;
renderer.RenderingOptions.MarginBottom = 10;
renderer.RenderingOptions.MarginLeft   = 15;
renderer.RenderingOptions.MarginRight  = 15;

var pdf = renderer.RenderHtmlAsPdf(@"
    <html><head><style>
        body { font-family: Arial; font-size: 10pt; }
        h1 { font-size: 18pt; }
        .total { font-weight: bold; }
    </style></head><body>
        <h1>Invoice #1234</h1>
        <p>Customer: ACME Corp</p>
        <p class='total'>Total: $500.00</p>
    </body></html>");

pdf.SaveAs("invoice.pdf");
// Rendering options: https://ironpdf.com/how-to/rendering-options/

The key change is replacing NReco's HtmlToPdfConverter with IronPDF's ChromePdfRenderer. Each CustomWkHtmlArgs flag maps to a typed property on RenderingOptions — no more string-based configuration. The Chromium-based rendering engine gives you modern CSS support out of the box.

NReco returns byte[] while IronPDF returns a PdfDocument. If you need bytes for HTTP responses or storage, use pdf.Stream or save to a MemoryStream.

How Do I Map wkhtmltopdf Flags to IronPDF Options?

Map each CustomWkHtmlArgs flag to a ChromePdfRenderOptions property:

  • --page-size A4 → RenderingOptions.PaperSize = PdfPaperSize.A4
  • --margin-top 10mm → RenderingOptions.MarginTop = 10
  • --orientation Landscape → RenderingOptions.PaperOrientation = PdfPaperOrientation.Landscape
  • --print-media-type → not needed — Chromium handles this; if you need to force print styles, set RenderingOptions.CssMediaType = PdfCssMediaType.Print
  • --background → on by default in Chromium; toggle with RenderingOptions.PrintHtmlBackgrounds
  • --disable-smart-shrinking → RenderingOptions.FitToPaperMode
  • --javascript-delay 1000 → RenderingOptions.WaitFor.RenderDelay(1000) (or WaitFor.JavaScript(condition, timeout) for an event-driven wait)

Mapping NReco options to IronPDF

// Mapping NReco options to IronPDF
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize    = IronPdf.Rendering.PdfPaperSize.A4;
renderer.RenderingOptions.MarginTop    = 10; // mm
renderer.RenderingOptions.MarginBottom = 10;
renderer.RenderingOptions.MarginLeft   = 15;
renderer.RenderingOptions.MarginRight  = 15;
renderer.RenderingOptions.PaperOrientation = IronPdf.Rendering.PdfPaperOrientation.Portrait;
// Full options: https://ironpdf.com/how-to/rendering-options/

Flags like --print-media-type and --background are not needed with Chromium — it handles these by default. For JavaScript-heavy content, prefer WaitFor.JavaScript() over a fixed delay for more reliable rendering.

How Do I Migrate Headers and Footers?

NReco passes HTML headers/footers to wkhtmltopdf via specific properties. IronPDF uses HtmlHeaderFooter:

Before (NReco)

// NReco header pattern:
// htmlToPdf.PageHeaderHtml = "<div style='text-align:right'>Page [page] of [topage]</div>";
// htmlToPdf.HeaderSpacing = 5;

After (IronPDF)

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.HtmlHeader = new HtmlHeaderFooter
{
    HtmlFragment = "<div style='text-align:right;font-size:9pt;'>" +
                   "Page {page} of {total-pages}</div>",
    DrawDividerLine = true
};
// Headers guide: https://ironpdf.com/how-to/headers-and-footers/

IronPDF's HtmlHeaderFooter gives you full HTML control over headers and footers with built-in placeholder tokens like {page} and {total-pages}. The DrawDividerLine property adds a visual separator without extra CSS. Test multi-page documents carefully — header/footer rendering is where renderer differences show up most.

How Do I Migrate PDF Merging?

Before (NReco — not supported natively; secondary library)

using PdfSharp.Pdf;
using PdfSharp.Pdf.IO;
using System;

class MergePdfsExample
{
    static void Main()
    {
        // NReco doesn't merge PDFs
        // PdfSharp is a common secondary choice
        using var output = new PdfDocument();

        foreach (string path in new[] { "section1.pdf", "section2.pdf", "section3.pdf" })
        {
            using var input = PdfReader.Open(path, PdfDocumentOpenMode.Import);
            foreach (PdfPage page in input.Pages)
                output.AddPage(page);
        }

        output.Save("merged.pdf");
        Console.WriteLine("Merged: merged.pdf");
    }
}

After (IronPDF native)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var merged = PdfDocument.Merge(
    PdfDocument.FromFile("section1.pdf"),
    PdfDocument.FromFile("section2.pdf"),
    PdfDocument.FromFile("section3.pdf")
);
merged.SaveAs("merged.pdf");
// Guide: https://ironpdf.com/how-to/merge-or-split-pdfs/

IronPDF's merge API takes any number of PdfDocument objects and combines them in one call — no secondary library like PdfSharp needed.

How Do I Migrate Watermarking?

Before (NReco — not supported; secondary library)

using iTextSharp.text;
using iTextSharp.text.pdf;
using System.IO;
using System;

class WatermarkExample
{
    static void Main()
    {
        // NReco generates; iTextSharp stamps
        using var reader  = new PdfReader("generated.pdf");
        using var fs      = new FileStream("watermarked.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs);

        var baseFont = BaseFont.CreateFont(BaseFont.HELVETICA_BOLD, BaseFont.CP1252, false);
        for (int page = 1; page <= reader.NumberOfPages; page++)
        {
            var cb = stamper.GetOverContent(page);
            cb.SaveState();
            var gs1 = new PdfGState { FillOpacity = 0.3f };
            cb.SetGState(gs1);
            cb.BeginText();
            cb.SetFontAndSize(baseFont, 60);
            cb.SetColorFill(BaseColor.GRAY);
            cb.ShowTextAligned(Element.ALIGN_CENTER, "CONFIDENTIAL", 300, 420, 45);
            cb.EndText();
            cb.RestoreState();
        }
        Console.WriteLine("Watermarked: watermarked.pdf");
    }
}

After (IronPDF)

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("generated.pdf");
var stamper = new TextStamper
{
    Text = "CONFIDENTIAL",
    FontColor = IronSoftware.Drawing.Color.Gray,
    FontSize = 60,
    Opacity = 30,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");
// Guide: https://ironpdf.com/how-to/custom-watermark/

IronPDF uses a TextStamper model instead of the low-level iTextSharp approach of manually managing graphics state, fonts, and coordinates. You configure text, font size, opacity, rotation, and alignment as properties, then apply with ApplyStamp(). The 20+ lines of iTextSharp boilerplate compress down to a handful of property assignments.

How Do I Migrate Password Protection?

Before (NReco — not supported; secondary library)

using iTextSharp.text.pdf;
using System.IO;
using System.Text;

class SecurityExample
{
    static void Main()
    {
        byte[] userPass  = Encoding.ASCII.GetBytes("readpass");
        byte[] ownerPass = Encoding.ASCII.GetBytes("adminpass");

        using var reader  = new PdfReader("generated.pdf");
        using var fs      = new FileStream("secured.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs, '\0', false);
        stamper.SetEncryption(
            userPass, ownerPass,
            PdfWriter.ALLOW_PRINTING,
            PdfWriter.ENCRYPTION_AES_128
        );
    }
}

After (IronPDF)

using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("generated.pdf");
pdf.SecuritySettings.UserPassword  = "readpass";
pdf.SecuritySettings.OwnerPassword = "adminpass";
pdf.SecuritySettings.AllowUserPrinting = PdfPrintSecurity.FullPrintRights;
pdf.SecuritySettings.AllowUserCopyPasteContent = false;
pdf.SaveAs("secured.pdf");
// Guide: https://ironpdf.com/how-to/pdf-permissions-passwords/

IronPDF handles security through the SecuritySettings property on PdfDocument. Set UserPassword, OwnerPassword, and granular permissions directly as properties. No byte array encoding, no separate stamper class, no encryption constant flags.

What Are the Key API Mappings I Need to Know?

Namespace Mapping

  • NReco.PdfGenerator → IronPdf — core namespace
  • (no NReco equivalent) → IronPdf.Rendering — options and configuration enums
  • (no NReco equivalent) → IronPdf.Editing — manipulation operations (stampers, etc.)

Core Class Mapping

  • HtmlToPdfConverter → ChromePdfRenderer — HTML-to-PDF entry point
  • CustomWkHtmlArgs (string of CLI flags) → ChromePdfRenderOptions — typed render configuration
  • (no NReco equivalent) → PdfDocument — PDF document model you can save, stream, or manipulate
  • byte[] return → PdfDocument return — call SaveAs, Stream, or BinaryData at the final step

Document Loading Methods

  • HTML string. htmlToPdf.GeneratePdf(html) → renderer.RenderHtmlAsPdf(html)
  • URL. htmlToPdf.GeneratePdfFromFile(url, null) → renderer.RenderUrlAsPdf(url)
  • HTML file. NReco requires reading the file to a string first; IronPDF has renderer.RenderHtmlFileAsPdf(path)
  • Existing PDF. Out of scope for NReco; IronPDF uses PdfDocument.FromFile(path)

Page Operations

  • Paper size. NReco's PageWidth/PageHeight or --page-size flag → RenderingOptions.PaperSize (or SetCustomPaperSizeInMillimeters)
  • Margins. NReco's Margins = new PageMargins { ... } or --margin-* flags → RenderingOptions.MarginTop/MarginBottom/MarginLeft/MarginRight
  • Orientation. Orientation = PageOrientation.Landscape or --orientation flag → RenderingOptions.PaperOrientation
  • Page count. Not accessible from NReco's byte[] output → pdf.PageCount

Merge/Split Operations

  • Merge. External library required with NReco → PdfDocument.Merge(pdf1, pdf2, ...)
  • Split. External library required with NReco → pdf.CopyPages(startIndex, endIndex)

How Do I Handle the byte[] to PdfDocument Output Change?

NReco returns byte[]. IronPDF returns PdfDocument. This is the change that ripples through the most call sites:

Updating output handling

// Before: byte[] pdfBytes = htmlToPdf.GeneratePdf(html);
// After:
var pdf = renderer.RenderHtmlAsPdf(html);

// If you need bytes (for HTTP response, storage, etc.):
pdf.SaveAs("output.pdf");
// Or stream to memory:
using var ms = new MemoryStream();
pdf.Stream.CopyTo(ms);
byte[] pdfBytes = ms.ToArray();
// Memory stream guide: https://ironpdf.com/how-to/pdf-memory-stream/

The PdfDocument object is more powerful than raw bytes because you can chain operations — merge, watermark, encrypt — before saving. If downstream code expects byte[], use pdf.Stream or pdf.BinaryData at the final step. IronPDF pages are 0-based; NReco doesn't expose a page model post-render, but if you are feeding output to page-aware tools, verify their indexing convention.

How Do I Handle Async Rendering and Performance?

NReco spawns wkhtmltopdf as a process (or calls native library). IronPDF runs in-process. Subprocess spawn cost (~50-200ms per call, system-dependent) is eliminated.

Reusing the renderer for batch work

// Reuse renderer for batch work — amortizes Chromium initialization
var renderer = new ChromePdfRenderer();

foreach (var html in batchTemplates)
{
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"doc_{Guid.NewGuid()}.pdf");
}

Async rendering for web controllers

// Async for web controllers — NReco doesn't have native async
var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(html);
return File(pdf.Stream, "application/pdf", "document.pdf");
// Async guide: https://ironpdf.com/how-to/async/

Reuse the ChromePdfRenderer instance across calls to amortize Chromium initialization overhead. For ASP.NET Core, register it as a singleton. Chromium has higher baseline memory than wkhtmltopdf, so profile under load. Test custom fonts explicitly — system font resolution differs between WebKit and Chromium. For JavaScript-heavy templates, verify timing with WaitFor options.

Frequently Asked Questions

Can I run NReco and IronPDF side by side during migration?

Yes. Install IronPDF via NuGet while NReco is still present. Run both libraries generating the same PDF from the same HTML, then compare outputs visually. This lets you catch rendering differences early — especially for Flexbox/Grid layouts, background colors, and custom fonts — before removing NReco and wkhtmltopdf from your project.

Do I need to remove wkhtmltopdf from my deployment pipeline?

Yes, once migration is complete. Search your Dockerfiles, CI/CD scripts, and appsettings for wkhtmltopdf, PdfToolPath, and WkHtmlToPdfExeName references. Removing the binary dependency is one of the main benefits of migrating — IronPDF ships as a NuGet package with no external binary installation required.

Will my CSS render differently after migration?

It will likely render better. wkhtmltopdf uses a WebKit build from approximately 2012 with incomplete Flexbox and no Grid support. IronPDF's Chromium-based renderer supports current CSS3. However, test all templates carefully — some edge cases like specific table rendering or page break behavior may differ between WebKit and Chromium.

How do I handle NReco's CustomWkHtmlArgs flags?

Each wkhtmltopdf flag maps to a typed property on ChromePdfRenderOptions. Flags like --print-media-type and --background are not needed at all with Chromium. Flags like --page-size, --margin-*, and --orientation map to PaperSize, Margin*, and PaperOrientation properties. See the rendering options guide for the full mapping.

Does IronPDF support async PDF generation?

Yes. NReco does not have native async support since it wraps a subprocess. IronPDF provides RenderHtmlAsPdfAsync() for non-blocking PDF generation in ASP.NET Core controllers and other async contexts. Register ChromePdfRenderer as a singleton to avoid per-request Chromium startup overhead.
