Migrating from PdfiumViewer to IronPDF: Solving Common Migration Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should I Migrate from PdfiumViewer to IronPDF?

PdfiumViewer worked on-prem for years. The developer machine, the test server, the production Windows VM — all fine. Then the team moved the PDF rendering service to a Linux Docker container on Kubernetes and the first deploy failed with a DllNotFoundException on pdfium.dll. PdfiumViewer is a thin .NET wrapper around the native Pdfium DLLs and a Windows Forms PdfRenderer control. You're on Linux now. The migration decision practically makes itself.

I've seen this pattern over and over. Here's why the migration makes sense:

Windows-only by design. PdfiumViewer targets Windows Forms and ships native Windows Pdfium DLLs. Linux, macOS, and container environments can't load those binaries.

Repository archived. The upstream pvginkel/PdfiumViewer repository was archived on August 2, 2019, and the last NuGet release (2.13.0) shipped on November 6, 2017. No further fixes, .NET 6+ updates, or CVE patches are coming from upstream — any patches you pick up will be from community forks.

.NET Framework heritage. The library was written for the .NET Framework / WinForms era. .NET 6+ and modern System.Drawing portability constraints are not first-class.

Viewer + rasterizer, not a PDF toolkit. PdfiumViewer is built to display PDFs in a WinForms control and rasterize pages to System.Drawing.Image. It does not generate, merge, watermark, sign, or secure PDFs.

No HTML-to-PDF or URL-to-PDF. If your service needs to produce PDFs from HTML or web content, PdfiumViewer was never the right tool — teams typically bolt on wkhtmltopdf or another library for that side.

Cloud deployment incompatible. Azure App Service on Linux, AWS Lambda, GCP Cloud Run: all fail with Windows-only native libraries.

CI/CD on Linux agents. Build agents running Linux (GitHub Actions ubuntu runners, GitLab CI) can't run PdfiumViewer-dependent tests.

The honest framing of this migration is viewer → renderer + viewer. IronPDF is a server-side rendering and manipulation library; it is not a WinForms control. You replace the page-to-image rasterization side with IronPDF on .NET 6+ Linux, and replace the WinForms PdfRenderer control with a separate viewer (browser-based, Syncfusion, DevExpress, Telerik). This article focuses on the renderer half — page rasterization and reading existing PDFs — because that's what the .NET code actually has to swap.

How Do I Diagnose the PdfiumViewer Deployment Failure?

Before writing migration code, confirm the exact failure mode. Linux and Docker failures with PdfiumViewer are predictable, and I want you to know exactly what you're dealing with.

The most common symptom is a DllNotFoundException for pdfium.dll. This is a Windows PE binary that won't load on Linux regardless of configuration.

Confirming the Windows-Only Dependency

# Confirm the library is Windows-targeting
docker run --rm your-image dotnet --info | grep OS

# The error will reference pdfium.dll — a Windows PE binary
# This won't load on Linux regardless of configuration

You may also see a PlatformNotSupportedException at startup:

# PdfiumViewer or its dependencies may explicitly check OS
docker run --rm your-image dotnet run 2>&1 | grep -i "platform\|not.*supported\|windows"

Or a native process crash:

# Native exception from trying to load a Windows DLL on Linux
docker run --rm your-image dotnet run 2>&1 | grep -i "segfault\|abort\|SIGSEGV\|native"

Once confirmed as a Windows-binary dependency issue, there's no configuration path — the library must be replaced for Linux/Docker deployment.

How Do I Replace PdfiumViewer with IronPDF in My Project?

The swap itself is straightforward. I'll walk you through the three essential steps: removing PdfiumViewer, installing IronPDF, and updating your code.

First, find all PdfiumViewer references in your codebase:

Finding PdfiumViewer References

# Find all PdfiumViewer usage
rg -l "PdfiumViewer\|PdfDocument.*Pdfium\|PdfRenderer" --type cs

# Find using statements
rg "using PdfiumViewer" --type cs -n

# Find NuGet references
grep -r -i "pdfiumviewer" **/*.csproj *.csproj 2>/dev/null

Then remove PdfiumViewer and install IronPDF. Don't forget the native binary packages — PdfiumViewer ships its Pdfium DLLs in side packages and they need to come out too:

Swapping NuGet Packages

# Remove PdfiumViewer and its native binary packages
dotnet remove package PdfiumViewer
dotnet remove package PdfiumViewer.Native.x86.v8-xfa
dotnet remove package PdfiumViewer.Native.x86_64.v8-xfa

# Install IronPDF
dotnet add package IronPdf
dotnet restore

Remove the native binary from your project directories and Docker files:

# Remove pdfium.dll from project directories
find . -name "pdfium.dll" -exec echo "Remove: {}" \;

# Remove from Docker COPY commands
rg "pdfium\.dll\|PdfiumViewer" Dockerfile* -n

Now update your namespace imports. Replace the old PdfiumViewer namespaces:

Updating Namespace Imports

// Before:
using PdfiumViewer;
using System.Drawing; // for bitmap rendering — Windows Forms dependency
using System.Drawing.Imaging;

// After:
using IronPdf;
using IronPdf.Rendering;
using IronPdf.Editing;

And set up your IronPDF license key:

Configuring the License Key

using IronPdf;

// Set once at application startup
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
// License guide: https://ironpdf.com/how-to/license-keys/

What Does the Core PDF Operation Code Look Like After Migration?

The biggest difference is that PdfiumViewer is a page renderer focused on rasterizing PDF pages to System.Drawing.Image bitmaps, while IronPDF is a server-side renderer with a much broader API surface — it can also rasterize pages, but pages-to-image is one feature among many. For a straight viewer/rasterizer migration, the swap is small.

Page-to-Image Rendering — Before (PdfiumViewer)

using PdfiumViewer;
using System;
using System.Drawing.Imaging;

class BasicPdfiumViewerExample
{
    static void Main()
    {
        // PdfiumViewer: open existing PDF and render pages to bitmap.
        // This is the primary use case — Windows only.
        using var document = PdfDocument.Load("document.pdf");

        int pageCount = document.PageCount;
        Console.WriteLine($"Pages: {pageCount}");

        // Render page 0 at 300 DPI (Windows-only System.Drawing)
        using (var image = document.Render(0, 300, 300, true))
        {
            image.Save("page1.png", ImageFormat.Png);
        }

        // Render every page at 150 DPI
        for (int i = 0; i < document.PageCount; i++)
        {
            using var pageImage = document.Render(i, 150, 150, true);
            pageImage.Save($"page_{i + 1}.png", ImageFormat.Png);
        }
    }
}

Page-to-Image Rendering — After (IronPDF)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// Open the existing PDF
PdfDocument pdf = PdfDocument.FromFile("document.pdf");
Console.WriteLine($"Pages: {pdf.PageCount}");

// Rasterize page 0 to a bitmap
var firstPageImage = pdf.ToBitmap(0);
firstPageImage[0].Save("page1.png");

// Rasterize every page
var allPageImages = pdf.ToBitmap();
for (int i = 0; i < allPageImages.Length; i++)
{
    allPageImages[i].Save($"page_{i + 1}.png");
}
// Guide: https://ironpdf.com/how-to/rasterize-pdf-to-image/

Both libraries use 0-based page indexing, so that's one less thing to worry about:

// PdfiumViewer: doc.Render(page: 0, ...) — 0-based
// IronPDF: pdf.Pages[0], pdf.ToBitmap(0) — 0-based
// Consistent between them

How Do I Migrate Page Info and Page Size Lookups?

If your PdfiumViewer code reads metadata — page counts, page sizes — it ports almost line-for-line. The naming differs but the shape is the same.

Page Info — Before (PdfiumViewer)

using PdfiumViewer;
using System;

void GetPageInfo(string pdfPath)
{
    using var document = PdfDocument.Load(pdfPath);
    Console.WriteLine($"Total pages: {document.PageCount}");

    for (int i = 0; i < document.PageCount; i++)
    {
        var size = document.PageSizes[i];
        Console.WriteLine($"Page {i + 1}: {size.Width} x {size.Height} points");
    }
}

Page Info — After (IronPDF)

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

void GetPageInfo(string pdfPath)
{
    var pdf = PdfDocument.FromFile(pdfPath);
    Console.WriteLine($"Total pages: {pdf.PageCount}");

    for (int i = 0; i < pdf.PageCount; i++)
    {
        var page = pdf.Pages[i];
        Console.WriteLine($"Page {i + 1}: {page.Width} x {page.Height} points");
    }
}

Note the shape change: PdfiumViewer exposes page sizes through a separate PageSizes indexer, while IronPDF gives you a Pages collection of objects with Width and Height properties on each.

How Do I Migrate Text Extraction?

PdfiumViewer does expose page-level text extraction through PdfDocument.GetPdfText(int page), which returns the raw text for a single page. What it does not give you is layout reconstruction, per-word coordinates, or OCR for image-only pages — and you have to loop the pages and concatenate yourself. IronPDF collapses that to a single call.

Text Extraction — Before (PdfiumViewer)

using PdfiumViewer;
using System;
using System.Text;

using var document = PdfDocument.Load("document.pdf");

var sb = new StringBuilder();
for (int i = 0; i < document.PageCount; i++)
{
    // GetPdfText returns raw page text — no layout, no per-word
    // coordinates, no OCR for image-only PDFs.
    sb.AppendLine($"--- Page {i + 1} ---");
    sb.AppendLine(document.GetPdfText(i));
}
Console.WriteLine(sb.ToString());

Text Extraction — After (IronPDF)

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

PdfDocument pdf = PdfDocument.FromFile("document.pdf");

// Pull text from the whole document
string allText = pdf.ExtractAllText();
Console.WriteLine(allText);

// Or one page at a time
string firstPageText = pdf.ExtractTextFromPage(0);
Console.WriteLine(firstPageText);

The IronPDF version is a one-liner for the common case and drops the StringBuilder loop. The caveat applies to both libraries: these APIs read the embedded text layer; neither is OCR. For scanned PDFs that have no text layer, you still want IronOCR or a dedicated OCR engine.

How Do I Fix IronPDF Failures in Docker on Linux?

You've removed PdfiumViewer and installed IronPDF. Now the Docker build succeeds but IronPDF's Chromium renderer fails at runtime. This is the most common post-migration issue I see.

IronPDF's Chromium renderer requires specific Linux libraries in your Docker base image. Here's the Dockerfile configuration that works:

Docker Base Image with IronPDF Dependencies

FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS runtime

# IronPDF Chromium dependencies on Debian/Ubuntu base images
RUN apt-get update && apt-get install -y \
    libglib2.0-0 \
    libnss3 \
    libatk1.0-0 \
    libatk-bridge2.0-0 \
    libx11-6 \
    libxcomposite1 \
    libxdamage1 \
    libxext6 \
    libxfixes3 \
    libxrandr2 \
    libgbm1 \
    libxkbcommon0 \
    libpango-1.0-0 \
    libcairo2 \
    --no-install-recommends \
    && rm -rf /var/lib/apt/lists/*

# Verify current IronPDF Docker requirements:
# https://ironpdf.com/how-to/azure/

If fonts render incorrectly, install font packages:

Fixing Missing Fonts in Docker

# Check available fonts in container
docker run --rm your-image fc-list | grep -i arial

# Install fonts if missing:
RUN apt-get install -y fonts-liberation fonts-dejavu fontconfig \
    && fc-cache -fv

A few edge cases to watch for:

Alpine Linux: Alpine uses musl libc; Chromium requires glibc. Verify IronPDF's Alpine support or switch to a glibc-based base image.

ARM64: Verify IronPDF provides ARM64 Chromium binaries for your target architecture.

Memory: Chromium in-process has a higher memory footprint than PdfiumViewer's native Windows rendering. Profile under load before sizing Kubernetes pods.

What Are the Key API Differences Between PdfiumViewer and IronPDF?

Understanding the API mapping makes the find-and-replace portion of migration much faster. Here are the mappings from my experience working with teams making this switch.

  • Focus. PdfiumViewer is a Windows Forms viewer control plus a page-to-image rasterizer. IronPDF is a server-side renderer and PDF manipulation library — page rasterization is one feature, not the whole product.
  • Pricing. PdfiumViewer is open source (Apache 2.0 / community-maintained). IronPDF is commercial — verify the current edition and trial terms at ironsoftware.com.
  • API style. PdfiumViewer wraps a Windows Forms control around the native Pdfium DLL. IronPDF is an in-process .NET library with no UI surface.
  • Learning curve. PdfiumViewer is small and easy if you only need to view or rasterize; it was never designed for server-side use. IronPDF has a broader API and a moderate learning curve.
  • HTML rendering. PdfiumViewer cannot render HTML. IronPDF uses an embedded Chromium renderer.
  • Page indexing. Both libraries are 0-based.
  • Threading. PdfiumViewer follows the WinForms threading model (UI-thread coupled). IronPDF works server-side; reuse a single ChromePdfRenderer across calls.
  • Namespace. PdfiumViewer → IronPdf.

Document Loading Methods

  • Open existing PDF. PdfDocument.Load(path) → PdfDocument.FromFile(path).
  • Open from stream. PdfDocument.Load(stream, ...) → PdfDocument.FromStream(stream).
  • Open from bytes. PdfDocument.Load(bytes) → PdfDocument.FromBinaryData(bytes).
  • HTML to PDF. Not supported in PdfiumViewer; renderer.RenderHtmlAsPdf(html) in IronPDF.
  • URL to PDF. Not supported in PdfiumViewer; renderer.RenderUrlAsPdf(url) in IronPDF.

Page Operations

  • Page count. doc.PageCount → pdf.PageCount.
  • Render page to bitmap. doc.Render(page, dpiX, dpiY, forPrinting) → pdf.ToBitmap(pageIndex), or pdf.ToBitmap() for all pages.
  • Page size. doc.PageSizes[n] → pdf.Pages[n].Width / pdf.Pages[n].Height.
  • Access a page object. No equivalent in PdfiumViewer (it is render-focused) → pdf.Pages[n] in IronPDF.

Text Extraction

  • Whole-document text. Loop doc.GetPdfText(i) in PdfiumViewer → pdf.ExtractAllText() in IronPDF.
  • Single-page text. doc.GetPdfText(pageIndex) → pdf.ExtractTextFromPage(pageIndex).
  • Layout / per-word coordinates. Not exposed by PdfiumViewer's GetPdfText; available through IronPDF's text APIs.

Features PdfiumViewer Doesn't Have

  • Merge. Not in PdfiumViewer; PdfDocument.Merge(pdf1, pdf2) in IronPDF.
  • Split. Not in PdfiumViewer; pdf.CopyPages(startIndex, endIndex) in IronPDF.
  • Watermark / stamp. Not in PdfiumViewer; pdf.ApplyStamp(...) / pdf.ApplyWatermark(...) in IronPDF.
  • Headers and footers. Not in PdfiumViewer; HtmlHeaderFooter on RenderingOptions in IronPDF.
  • Password / encryption. Not in PdfiumViewer; pdf.SecuritySettings in IronPDF.
  • Form filling. Not in PdfiumViewer; pdf.Form in IronPDF — verify field-type coverage against your specific forms.

Migration Effort by Feature

  • Page-to-bitmap rasterization — Low. PdfiumViewer's primary feature; maps cleanly to pdf.ToBitmap().
  • Open / read existing PDF — Low. PdfDocument.Load → PdfDocument.FromFile.
  • Page count / metadata — Low. doc.PageCount stays as pdf.PageCount.
  • Print PDF — Low. doc.CreatePrintDocument() → pdf.Print(printerName).
  • Text extraction — Low. Page-loop with doc.GetPdfText(i) collapses to pdf.ExtractAllText().
  • HTML / URL to PDF — Low (new capability). If you were running wkhtmltopdf or another tool alongside PdfiumViewer, you can delete that subprocess.
  • Merge / watermark / password — Low (new capability). Native in IronPDF; you can drop PDFSharp or iTextSharp.
  • Linux / Docker deployment — Resolved. The whole point of the migration.
  • Cross-platform .NET 6/8 — Resolved. IronPDF supports modern .NET on Windows, Linux, and macOS.
  • Display PDF in a WinForms control — High (out of scope). IronPDF has no UI control. Replace PdfRenderer with a separate viewer (browser-based PDF.js, Syncfusion, DevExpress, Telerik).

Frequently Asked Questions

Can IronPDF Replace PdfiumViewer's WinForms PDF Viewer?

No. IronPDF is a server-side generation and manipulation library. PdfiumViewer as a Windows Forms control (PdfRenderer) has no IronPDF equivalent. If you have a WinForms application that uses PdfiumViewer for display, those view components need a separate replacement — pdf.js for browser-based viewing, or DevExpress, Syncfusion, or Telerik for WinForms controls.

What About PdfiumViewer's PDF-to-Image Rendering?

IronPDF can rasterize pages to bitmaps cross-platform via pdf.ToBitmap() (single page or whole document) and to disk via pdf.RasterizeToImageFiles(path, DPI). That's a like-for-like replacement for PdfiumViewer's document.Render(...) call, and it runs on Linux. If you specifically want a thin Pdfium wrapper without IronPDF's broader feature set, docnet is an open-source cross-platform option.

Do I Need to Change Page Indexing in My Code?

No. Both PdfiumViewer and IronPDF use 0-based page indexing, so the loop bounds and indices in your existing code carry over unchanged.

What Should I Do About System.Drawing Dependencies?

PdfiumViewer's image rendering returns System.Drawing.Image objects — those are Windows-only. Remove System.Drawing dependencies from your server-side code entirely after migration. Run rg "System\.Drawing\|Bitmap\|ImageFormat" --type cs -n to find remaining references.

How Do I Optimize IronPDF's Chromium Renderer for Performance?

If you do generate HTML-to-PDFs alongside your rasterization workload, reuse the ChromePdfRenderer instance across calls. Creating a new renderer per PDF is unnecessary overhead:

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

foreach (var html in templates)
{
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"output_{Guid.NewGuid()}.pdf");
}

For pure rasterization workloads, PdfDocument.FromFile is cheap and you don't need a renderer at all. Cache the apt-get layer in your Dockerfile separately from your application layer so rebuilds stay fast.

When Should I Migrate from PdfiumViewer to IronPDF?

The PdfiumViewer migration is one of the cleaner cases because the constraint is definitive — it's a Windows library and you need Linux.

Migrate now if: You're moving to Linux, Docker, or any cloud environment. You're modernizing to .NET 6+. You need HTML-to-PDF generation, merge, watermark, or password protection without bolting on secondary libraries.

Evaluate separately if: Your primary need is a WinForms PDF viewer replacement — IronPDF is not a viewer. If you need PDF-to-image rendering on Linux, consider docnet or Ghostscript alongside IronPDF. If you have both viewing and server-side generation, split the responsibilities: IronPDF for the server side, a separate viewer component for the client.

The ambiguity in this migration is whether the WinForms viewer component needs a parallel replacement or if it's only server-side code that's moving. Inventory your PdfiumViewer usage, separate the viewer concerns from the server concerns, and migrate the server side to IronPDF. The rest follows naturally.
