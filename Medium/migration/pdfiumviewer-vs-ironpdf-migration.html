Migrating from PdfiumViewer to IronPDF: Solving Common Migration Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should I Migrate from PdfiumViewer to IronPDF?

PdfiumViewer worked on-prem for years. The developer machine, the test server, the production Windows VM — all fine. Then the team moved the PDF generation service to a Linux Docker container on Kubernetes and the first deploy failed with a DllNotFoundException on pdfiumviewer.dll. It's a Windows-targeting library. You're on Linux now. The migration decision practically makes itself.

I've seen this pattern over and over. Here's why the migration makes sense:

Windows-only limitation. PdfiumViewer targets Windows Forms and native Windows DLLs. Linux, macOS, and container environments can't use it.

Repository archived. PdfiumViewer appears unmaintained. Maintenance has likely stopped.

.NET Framework target. Limited or no .NET 6+ support means incompatibility as the stack modernizes.

PDF viewer vs server-side generator. PdfiumViewer is designed as a desktop viewer control. Server-side PDF generation is not its primary design target.

HTML-to-PDF not supported. PdfiumViewer renders existing PDFs to images; it doesn't generate PDFs from HTML.

No manipulation features. Merge, watermark, password protection require additional libraries.

Cloud deployment incompatible. Azure App Service on Linux, AWS Lambda, GCP Cloud Run: all fail with Windows-only native libraries.

CI/CD on Linux agents. Build agents running Linux (GitHub Actions ubuntu runners, GitLab CI) can't run PdfiumViewer-dependent tests.

With IronPDF, all of these constraints disappear. It runs cross-platform on .NET 6+, works in Docker containers, and handles HTML-to-PDF generation natively.

How Do I Diagnose the PdfiumViewer Deployment Failure?

Before writing migration code, confirm the exact failure mode. Linux and Docker failures with PdfiumViewer are predictable, and I want you to know exactly what you're dealing with.

Answer:

The most common symptom is a DllNotFoundException for pdfium.dll. This is a Windows PE binary that won't load on Linux regardless of configuration.

Confirming the Windows-Only Dependency

# Confirm the library is Windows-targeting
docker run --rm your-image dotnet --info | grep OS

# The error will reference pdfium.dll — a Windows PE binary
# This won't load on Linux regardless of configuration

You may also see a PlatformNotSupportedException at startup:

# PdfiumViewer or its dependencies may explicitly check OS
docker run --rm your-image dotnet run 2>&1 | grep -i "platform\|not.*supported\|windows"

Or a native process crash:

# Native exception from trying to load a Windows DLL on Linux
docker run --rm your-image dotnet run 2>&1 | grep -i "segfault\|abort\|SIGSEGV\|native"

Once confirmed as a Windows-binary dependency issue, there's no configuration path — the library must be replaced for Linux/Docker deployment.

How Do I Replace PdfiumViewer with IronPDF in My Project?

The swap itself is straightforward. I'll walk you through the three essential steps: removing PdfiumViewer, installing IronPDF, and updating your code.

Answer:

First, find all PdfiumViewer references in your codebase:

Finding PdfiumViewer References

# Find all PdfiumViewer usage
rg -l "PdfiumViewer\|PdfDocument.*Pdfium\|PdfRenderer" --type cs

# Find using statements
rg "using PdfiumViewer" --type cs -n

# Find NuGet references
grep -r -i "pdfiumviewer" **/*.csproj *.csproj 2>/dev/null

Then remove PdfiumViewer and install IronPDF:

Swapping NuGet Packages

# Remove PdfiumViewer
dotnet remove package PdfiumViewer

# Install IronPDF
dotnet add package IronPdf
dotnet restore

Remove the native binary from your project directories and Docker files:

# Remove pdfium.dll from project directories
find . -name "pdfium.dll" -exec echo "Remove: {}" \;

# Remove from Docker COPY commands
rg "pdfium\.dll\|PdfiumViewer" Dockerfile* -n

Now update your namespace imports. Replace the old PdfiumViewer namespaces:

Updating Namespace Imports

// Before:
using PdfiumViewer;
using System.Drawing; // for bitmap rendering — Windows Forms dependency
using System.Drawing.Imaging;

// After:
using IronPdf;
using IronPdf.Rendering;
using IronPdf.Editing;
using IronPdf.Security;

And set up your IronPDF license key:

Configuring the License Key

using IronPdf;

// Set once at application startup
IronPdf.License.LicenseKey = "YOUR_IRONPDF_LICENSE_KEY";
// License guide: https://ironpdf.com/how-to/license-keys/

What Does the Core PDF Operation Code Look Like After Migration?

The biggest difference is that PdfiumViewer is a page renderer focused on converting PDF pages to bitmap images, while IronPDF is an HTML-to-PDF generator and manipulation library. The API surface is fundamentally different.

Answer:

Basic PDF Operation — Before (PdfiumViewer)

using PdfiumViewer;
using System;
using System.Drawing;
using System.Drawing.Imaging;

class BasicPdfiumViewerExample
{
    static void Main()
    {
        // PdfiumViewer: open existing PDF and render page to image
        // This is the primary use case — Windows only
        using var document = PdfDocument.Load("input.pdf");

        int pageCount = document.PageCount;
        Console.WriteLine($"Pages: {pageCount}");

        // Render page 0 to bitmap (Windows-only System.Drawing)
        using var image = document.Render(
            page: 0,
            width: 1240,
            height: 1754,
            dpiX: 150,
            dpiY: 150,
            forPrinting: false
        );

        image.Save("page1.png", ImageFormat.Png);
        Console.WriteLine("Rendered page 1 to page1.png");
    }
}

Basic PDF Operation — After (IronPDF)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

// If replacing server-side PDF generation:
var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<h1>Hello World</h1>");
pdf.SaveAs("output.pdf");
// Guide: https://ironpdf.com/how-to/html-string-to-pdf/

// If you need to open and read an existing PDF:
// var existing = PdfDocument.FromFile("input.pdf");
// Console.WriteLine($"Pages: {existing.PageCount}");

Both libraries use 0-based page indexing, so that's one less thing to worry about:

// PdfiumViewer: doc.Render(page: 0, ...) — 0-based
// IronPDF: pdf.Pages[0] — 0-based
// Consistent between them

How Do I Migrate HTML-to-PDF Generation That Used a Secondary Tool?

Since PdfiumViewer can't generate PDFs from HTML, teams commonly add wkhtmltopdf or another subprocess tool alongside it. This is one of the cleanest wins in the migration — you eliminate that external process entirely.

Answer:

HTML to PDF — Before (wkhtmltopdf subprocess)

using System;
using System.Diagnostics;
using System.IO;

class HtmlToPdfWithSecondary
{
    static void Main()
    {
        // PdfiumViewer can't generate PDFs from HTML
        // Teams add wkhtmltopdf or another tool for this
        string html    = "<html><body><h1>Invoice</h1></body></html>";
        string tmpHtml = Path.GetTempFileName() + ".html";
        string tmpPdf  = Path.GetTempFileName() + ".pdf";

        File.WriteAllText(tmpHtml, html);

        var psi = new ProcessStartInfo("wkhtmltopdf", $"\"{tmpHtml}\" \"{tmpPdf}\"")
        {
            RedirectStandardError = true,
            UseShellExecute = false,
            CreateNoWindow = true
        };

        using var proc = Process.Start(psi)!;
        if (!proc.WaitForExit(30_000)) { proc.Kill(); throw new TimeoutException(); }
        if (proc.ExitCode != 0)
            throw new Exception(proc.StandardError.ReadToEnd());

        byte[] pdfBytes = File.ReadAllBytes(tmpPdf);
        File.Delete(tmpHtml);
        File.Delete(tmpPdf);
        Console.WriteLine($"Generated {pdfBytes.Length} bytes");
    }
}

HTML to PDF — After (IronPDF)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Invoice</h1><p>Amount: $500</p></body></html>"
);
pdf.SaveAs("invoice.pdf");
// Guide: https://ironpdf.com/how-to/html-string-to-pdf/

No temp files. No subprocess management. No timeout handling. The ChromePdfRenderer handles it all in-process.

How Do I Migrate Merge, Watermark, and Security Operations?

PdfiumViewer doesn't support merge, watermark, or password protection — so teams typically bolted on PDFSharp or iTextSharp for these features. With IronPDF, all of these are built in.

Answer:

Merging PDFs — Before (PDFSharp secondary library)

using PdfSharp.Pdf;
using PdfSharp.Pdf.IO;
using System;

class MergePdfsExample
{
    static void Main()
    {
        // PdfiumViewer has no merge — secondary library needed
        using var output = new PdfDocument();

        foreach (string path in new[] { "section1.pdf", "section2.pdf" })
        {
            using var input = PdfReader.Open(path, PdfDocumentOpenMode.Import);
            foreach (PdfPage page in input.Pages)
                output.AddPage(page);
        }

        output.Save("merged.pdf");
        Console.WriteLine("Merged to: merged.pdf");
    }
}

Merging PDFs — After (IronPDF)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

var merged = PdfDocument.Merge(
    PdfDocument.FromFile("section1.pdf"),
    PdfDocument.FromFile("section2.pdf")
);
merged.SaveAs("merged.pdf");
// Guide: https://ironpdf.com/how-to/merge-or-split-pdfs/

Watermarking — Before (iTextSharp secondary library)

using iTextSharp.text;
using iTextSharp.text.pdf;
using System.IO;
using System;

class WatermarkExample
{
    static void Main()
    {
        using var reader  = new PdfReader("input.pdf");
        using var fs      = new FileStream("watermarked.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs);

        var baseFont = BaseFont.CreateFont(BaseFont.HELVETICA_BOLD, BaseFont.CP1252, false);
        for (int i = 1; i <= reader.NumberOfPages; i++)
        {
            var cb = stamper.GetOverContent(i);
            cb.BeginText();
            cb.SetFontAndSize(baseFont, 60);
            cb.SetColorFill(new BaseColor(180, 180, 180));
            cb.ShowTextAligned(Element.ALIGN_CENTER, "DRAFT", 300, 420, 45);
            cb.EndText();
        }
        Console.WriteLine("Watermarked.");
    }
}

Watermarking — After (IronPDF)

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

var pdf = PdfDocument.FromFile("input.pdf");
var stamper = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronSoftware.Drawing.Color.LightGray,
    FontSize = 60,
    Opacity = 30,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");
// Guide: https://ironpdf.com/how-to/custom-watermark/

Password Protection — Before (iTextSharp secondary library)

using iTextSharp.text.pdf;
using System.IO;
using System.Text;

class SecurityExample
{
    static void Main()
    {
        byte[] userPass  = Encoding.ASCII.GetBytes("viewonly");
        byte[] ownerPass = Encoding.ASCII.GetBytes("admin");

        using var reader  = new PdfReader("input.pdf");
        using var fs      = new FileStream("secured.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs, '\0', false);
        stamper.SetEncryption(
            userPass, ownerPass,
            PdfWriter.ALLOW_PRINTING,
            PdfWriter.ENCRYPTION_AES_128
        );
    }
}

Password Protection — After (IronPDF)

using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR_LICENSE_KEY";

var pdf = PdfDocument.FromFile("input.pdf");
pdf.SecuritySettings.UserPassword  = "viewonly";
pdf.SecuritySettings.OwnerPassword = "admin";
pdf.SecuritySettings.AllowUserPrinting = PdfPrintSecurity.FullPrintRights;
pdf.SaveAs("secured.pdf");
// Guide: https://ironpdf.com/how-to/pdf-permissions-passwords/

How Do I Fix IronPDF Failures in Docker on Linux?

You've removed PdfiumViewer and installed IronPDF. Now the Docker build succeeds but IronPDF's Chromium renderer fails at runtime. This is the most common post-migration issue I see.

Answer:

IronPDF's Chromium renderer requires specific Linux libraries in your Docker base image. Here's the Dockerfile configuration that works:

Docker Base Image with IronPDF Dependencies

FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS runtime

# IronPDF Chromium dependencies on Debian/Ubuntu base images
RUN apt-get update && apt-get install -y \
    libglib2.0-0 \
    libnss3 \
    libatk1.0-0 \
    libatk-bridge2.0-0 \
    libx11-6 \
    libxcomposite1 \
    libxdamage1 \
    libxext6 \
    libxfixes3 \
    libxrandr2 \
    libgbm1 \
    libxkbcommon0 \
    libpango-1.0-0 \
    libcairo2 \
    --no-install-recommends \
    && rm -rf /var/lib/apt/lists/*

# Verify current IronPDF Docker requirements:
# https://ironpdf.com/how-to/azure/

If fonts render incorrectly, install font packages:

Fixing Missing Fonts in Docker

# Check available fonts in container
docker run --rm your-image fc-list | grep -i arial

# Install fonts if missing:
RUN apt-get install -y fonts-liberation fonts-dejavu fontconfig \
    && fc-cache -fv

A few edge cases to watch for:

Alpine Linux: Alpine uses musl libc; Chromium requires glibc. Verify IronPDF's Alpine support or switch to a glibc-based base image.

ARM64: Verify IronPDF provides ARM64 Chromium binaries for your target architecture.

Memory: Chromium in-process has a higher memory footprint than PdfiumViewer's native Windows rendering. Profile under load before sizing Kubernetes pods.

Frequently Asked Questions

Can IronPDF Replace PdfiumViewer's WinForms PDF Viewer?

No. IronPDF is a server-side generation and manipulation library. PdfiumViewer as a Windows Forms control (PdfRenderer) has no IronPDF equivalent. If you have a WinForms application that uses PdfiumViewer for display, those view components need a separate replacement — pdf.js for browser-based viewing, or DevExpress, Syncfusion, or Telerik for WinForms controls.

What About PdfiumViewer's PDF-to-Image Rendering?

This is a genuine feature gap, not a configuration issue. PdfiumViewer's primary purpose is rendering PDF pages to bitmap images. For cross-platform PDF-to-image on Linux, alternatives include docnet (an open-source, cross-platform Pdfium wrapper), Ghostscript, or ImageMagick. If you also need HTML-to-PDF, IronPDF handles that side while you use a separate tool for image rendering.

Do I Need to Change Page Indexing in My Code?

No. Both PdfiumViewer and IronPDF use 0-based page indexing. This is one less thing to worry about during migration. However, verify your specific PdfiumViewer wrapper's convention if you're using a third-party wrapper.

What Should I Do About System.Drawing Dependencies?

PdfiumViewer's image rendering returns System.Drawing.Image objects — those are Windows-only. Remove System.Drawing dependencies from your server-side code entirely after migration. Run rg "System\.Drawing\|Bitmap\|ImageFormat" --type cs -n to find remaining references.

How Do I Optimize IronPDF's Chromium Renderer for Performance?

Reuse the ChromePdfRenderer instance across multiple render calls. Creating a new renderer for each PDF is unnecessary overhead:

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

foreach (var html in templates)
{
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"output_{Guid.NewGuid()}.pdf");
}

Cache the apt-get layer in your Dockerfile separately from your application layer so rebuilds are faster.

When Should I Migrate from PdfiumViewer to IronPDF?

The PdfiumViewer migration is one of the cleaner cases because the constraint is definitive — it's a Windows library and you need Linux.

Migrate now if: You're moving to Linux, Docker, or any cloud environment. You're modernizing to .NET 6+. You need HTML-to-PDF generation, merge, watermark, or password protection without bolting on secondary libraries.

Evaluate separately if: Your primary need is a WinForms PDF viewer replacement — IronPDF is not a viewer. If you need PDF-to-image rendering on Linux, consider docnet or Ghostscript alongside IronPDF. If you have both viewing and server-side generation, split the responsibilities: IronPDF for the server side, a separate viewer component for the client.

The ambiguity in this migration is whether the WinForms viewer component needs a parallel replacement or if it's only server-side code that's moving. Inventory your PdfiumViewer usage, separate the viewer concerns from the server concerns, and migrate the server side to IronPDF. The rest follows naturally.
