Migrating from Kaizen to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

The version pin in your .csproj has been sitting at 2.x for eight months. A security advisory came through, the fix landed in 3.x, and the upgrade path involves breaking API changes you haven't had time to assess. Meanwhile the rest of the stack has moved to .NET 8 and the old package doesn't play cleanly with the new runtime.

You're not looking for a migration out of enthusiasm — you're looking for a path forward. I wrote this guide to map the code changes needed to move from Kaizen.io HTML-to-PDF to IronPDF. Kaizen.io HTML-to-PDF is a remote service with a thin .NET wrapper; IronPDF is an in-process Chromium-based renderer. The mapping tables and checklist are useful reference material regardless of which library you end up choosing.

Why Should I Migrate from Kaizen.io to IronPDF?

Version pinning and upgrade friction are common triggers, but the bigger story with Kaizen.io is the cloud-API model itself. Here are the reasons teams move off it:

Cloud dependency — every PDF requires a network round-trip to Kaizen.io's servers. Outages, DNS issues, or firewall changes break PDF generation.

Data privacy — HTML and any embedded data leaves your network on every request. For regulated workloads (healthcare, finance, internal-only documents) that is a hard blocker.

Latency — in-process rendering avoids the 100–500 ms network round-trip that hits every Kaizen.io call.

Per-request pricing — SaaS costs scale linearly with volume; an in-process license has a fixed ceiling.

Rate limiting — Kaizen.io throttles during traffic spikes. IronPDF is bounded only by hardware.

Offline / air-gapped environments — on-prem or restricted networks cannot reach a remote API at all.

Vendor lock-in — if Kaizen.io changes pricing, deprecates the API, or shuts down, every PDF endpoint stops working at once.

Customization ceiling — you only get the rendering options the API exposes. IronPDF gives you full Chromium control (JavaScript wait conditions, custom paper sizes, headers/footers, BaseUrl, etc.).

Commercial support requirements — teams needing SLA-backed support may need a library with an enterprise support tier.

How Do Kaizen.io and IronPDF Compare Feature by Feature?

  • Focus. Kaizen.io is a remote HTML-to-PDF service consumed via a thin .NET wrapper. IronPDF is HTML-to-PDF plus full in-process PDF manipulation (merge, split, stamp, encrypt, sign).
  • Deployment model. Kaizen.io runs on Kaizen's servers; every conversion is a network call. IronPDF runs entirely in your process.
  • Pricing. Kaizen.io is per-request / subscription — check kaizen.io for current tiers. IronPDF is a commercial per-developer or per-deployment license — see ironsoftware.com.
  • HTML rendering. Kaizen.io uses a server-side renderer behind the API (engine not exposed to the caller). IronPDF embeds Chromium directly, so what renders in Chrome renders in your PDF.
  • JavaScript / SPA support. Kaizen.io supports a WaitForComplete/delay flag at the API level. IronPDF gives you full Chromium control including RenderDelay, WaitFor.JavaScript(...), and network-idle waits.
  • Page indexing. Kaizen.io's API is page-output rather than page-manipulation, so indexing is mostly N/A. IronPDF is 0-based for any post-render page operations you add.
  • Namespace. The .NET wrapper for Kaizen.io exposes Kaizen.IO; IronPDF lives under IronPdf.
  • Offline use. Kaizen.io cannot run without network access. IronPDF works in air-gapped environments.

How Hard Is Each Feature to Migrate?

  • HTML string to PDF — Low. converter.Convert(html) maps directly to renderer.RenderHtmlAsPdf(html).
  • URL to PDF — Low. converter.ConvertUrl(url, options) maps to renderer.RenderUrlAsPdf(url).
  • HTML file to PDF — Low. Read the file and call RenderHtmlAsPdf, or use RenderHtmlFileAsPdf(path).
  • Custom margins and page size — Low. Move from ConversionOptions.MarginTop etc. to renderer.RenderingOptions.MarginTop. PageSize becomes PdfPaperSize.
  • Headers / footers — Medium. Kaizen.io uses HeaderOptions.HtmlContent; IronPDF uses HtmlHeaderFooter.HtmlFragment. The {total} placeholder becomes {total-pages}.
  • Async rendering — Medium. ConvertAsync maps to RenderHtmlAsPdfAsync; the call shape is identical.
  • JavaScript / SPA pages — Medium. Replace Kaizen's JavaScriptDelay with RenderingOptions.RenderDelay or WaitFor.JavaScript(...) for explicit wait conditions.
  • Merge / split — New capability. Not part of Kaizen.io's HTML-to-PDF API. If you were combining Kaizen.io with PdfSharp or similar for merge, IronPDF's PdfDocument.Merge consolidates that.
  • Watermark, encryption, digital signatures — New capability. Not in Kaizen.io's surface area. IronPDF covers all three in-process; teams that bolted on iTextSharp for these can drop the secondary dependency.
  • Removing rate-limit / retry / network-error code — Low. All of it deletes once you're in-process. Verify your error handling still covers IronPdfException instead.

When Does It Make Sense to Migrate?

  • If you need offline / air-gapped PDF generation, Kaizen.io is a non-starter and IronPDF is the obvious move.
  • If data privacy or compliance prohibits sending HTML payloads to a third party, in-process rendering is the only correct answer — IronPDF or another local renderer.
  • If your volume has outgrown per-request pricing, a fixed-cost license usually wins; do a back-of-envelope on monthly Kaizen.io spend versus IronPDF's tiers before committing.
  • If your only requirement is open source, IronPDF is commercial — evaluate PuppeteerSharp or a wkhtmltopdf wrapper instead.
  • If your use case is plain HTML-to-PDF and Kaizen.io's fidelity is fine, the migration buys you privacy, latency, and cost predictability rather than feature breadth.
  • If you need merge, watermark, or encryption alongside HTML-to-PDF, IronPDF consolidates these into one library and removes a secondary dependency.

How Do I Set Up My Codebase for Migration?

Before writing any migration code, take a full inventory of Kaizen.io call sites. You will need .NET 6+ target framework and NuGet access or an offline package cache.

Find Kaizen.io references in your codebase

# Find all files referencing Kaizen
rg -l "Kaizen" --type cs

# Find using statements (Kaizen.IO is the .NET wrapper namespace)
rg "using Kaizen" --type cs -n

# Find the converter and options classes
rg "HtmlToPdfConverter|ConversionOptions|ConvertUrl|ConvertAsync" --type cs -n

# Find NuGet package references in project files
rg -i "kaizen" -g "*.csproj"

Remove Kaizen.io, install IronPDF

# Remove Kaizen package — the exact package id varies; check your .csproj
dotnet remove package Kaizen.HtmlToPdf

# Install IronPDF
dotnet add package IronPdf

# Restore
dotnet restore

On Windows PowerShell via Package Manager Console: Install-Package IronPdf and Uninstall-Package Kaizen.HtmlToPdf.

How Do I Migrate License Configuration and Namespaces?

Before (Kaizen.io)

using Kaizen.IO;

// Kaizen.io is keyed per request: the API key is passed to the converter constructor.
// Some integrations also wrap the SaaS endpoint themselves — if your code does that,
// the API key lives wherever you store HttpClient credentials.
var converter = new HtmlToPdfConverter("YOUR-KAIZEN-API-KEY");

After (IronPDF)

using IronPdf;

// Set once at application startup
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
// License guide: https://ironpdf.com/how-to/license-keys/

IronPDF uses a single static property for license configuration. Set IronPdf.License.LicenseKey once at application startup, and you are done. No per-request key, no separate license class.

For namespace imports, replace Kaizen.io namespaces with the IronPDF equivalents:

Before (Kaizen.io namespaces)

using Kaizen.IO; // HtmlToPdfConverter, ConversionOptions, HeaderOptions, FooterOptions, PageSize, Orientation

After (IronPDF namespaces)

using IronPdf;
using IronPdf.Rendering;  // for ChromePdfRenderOptions, PdfPaperSize, PdfPaperOrientation
using IronPdf.Editing;    // for stamping/watermark (new capability)
using IronPdf.Security;   // for passwords (new capability)

How Do I Migrate Basic HTML-to-PDF Conversion?

Before (Kaizen.io)

using Kaizen.IO;
using System.IO;

class HtmlToPdfExample
{
    static void Main()
    {
        var converter = new HtmlToPdfConverter();
        var options = new ConversionOptions
        {
            PageSize = PageSize.A4,
            Orientation = Orientation.Portrait,
            MarginTop = 10,
            MarginBottom = 10
        };

        byte[] pdfBytes = converter.Convert(
            "<html><body><h1>Invoice</h1></body></html>",
            options
        );
        File.WriteAllBytes("invoice.pdf", pdfBytes);
    }
}

After (IronPDF)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
renderer.RenderingOptions.MarginTop = 10;
renderer.RenderingOptions.MarginBottom = 10;

var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>"
);
pdf.SaveAs("invoice.pdf");
// Full guide: https://ironpdf.com/how-to/html-string-to-pdf/

The shape of the call survives the migration: instantiate a renderer, configure options, hand it HTML. The differences are that HtmlToPdfConverter becomes ChromePdfRenderer, ConversionOptions moves onto the renderer as RenderingOptions, and the call returns a PdfDocument instead of byte[]. If your downstream code passes the bytes around, use pdf.BinaryData.

The bigger change is what's no longer happening: the Convert call is no longer hitting Kaizen.io's servers. There's no network request, no API key in the call, no rate limit to worry about, and the HTML never leaves your process.

How Do I Migrate URL-to-PDF with Headers and Footers?

Before (Kaizen.io)

using Kaizen.IO;
using System.IO;

class UrlToPdfExample
{
    static void Main()
    {
        var converter = new HtmlToPdfConverter();
        var options = new ConversionOptions
        {
            Header = new HeaderOptions
            {
                HtmlContent = "<div style='text-align:center'>Company Header</div>"
            },
            Footer = new FooterOptions
            {
                HtmlContent = "<div style='text-align:center'>Page {page} of {total}</div>"
            },
            MarginTop = 20,
            MarginBottom = 20
        };
        var pdfBytes = converter.ConvertUrl("https://example.com", options);
        File.WriteAllBytes("webpage.pdf", pdfBytes);
    }
}

After (IronPDF)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.HtmlHeader = new HtmlHeaderFooter
{
    HtmlFragment = "<div style='text-align:center'>Company Header</div>",
    MaxHeight = 25
};
renderer.RenderingOptions.HtmlFooter = new HtmlHeaderFooter
{
    HtmlFragment = "<div style='text-align:center'>Page {page} of {total-pages}</div>",
    MaxHeight = 25
};
renderer.RenderingOptions.MarginTop = 20;
renderer.RenderingOptions.MarginBottom = 20;

var pdf = renderer.RenderUrlAsPdf("https://example.com");
pdf.SaveAs("webpage.pdf");
// Guide: https://ironpdf.com/how-to/headers-and-footers/

Three things shift here. HeaderOptions.HtmlContent becomes HtmlHeaderFooter.HtmlFragment. The page-count placeholder {total} becomes {total-pages}. And MaxHeight on the IronPDF header/footer reserves space at the top and bottom of the page that your margins won't cover — size it to fit your tallest header content. ConvertUrl(url) maps cleanly to RenderUrlAsPdf(url), and the call still flows out to a saved file.

How Do I Migrate HTML File Conversion?

Before (Kaizen.io)

using Kaizen.IO;
using System.IO;

class HtmlFileExample
{
    static void Main()
    {
        var converter = new HtmlToPdfConverter();
        var htmlContent = File.ReadAllText("input.html");
        var options = new ConversionOptions
        {
            PageSize = PageSize.A4,
            Orientation = Orientation.Portrait
        };
        var pdfBytes = converter.Convert(htmlContent, options);
        File.WriteAllBytes("document.pdf", pdfBytes);
    }
}

After (IronPDF)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
renderer.RenderingOptions.PaperOrientation = IronPdf.Rendering.PdfPaperOrientation.Portrait;

// Option 1: read the file yourself, render the string
var html = File.ReadAllText("input.html");
var pdf = renderer.RenderHtmlAsPdf(html);

// Option 2: let IronPDF read the file and resolve relative resources
// var pdf = renderer.RenderHtmlFileAsPdf("input.html");

pdf.SaveAs("document.pdf");
// Guide: https://ironpdf.com/how-to/html-file-to-pdf/

If your HTML file references local images, CSS, or fonts via relative paths, prefer RenderHtmlFileAsPdf — it resolves relative resources against the file's directory. RenderHtmlAsPdf(string) on raw HTML has no implicit base path, so you'd need to set RenderingOptions.BaseUrl for the same effect.

What New Capabilities Become Available After Migration?

Kaizen.io's API surface is HTML–in, PDF–out. Once you're in-process with IronPDF, three operations that previously required a second library (PdfSharp, iTextSharp, etc.) are part of the same package:

Merge — load PDFs with PdfDocument.FromFile and combine them with PdfDocument.Merge(pdf1, pdf2, ...). Guide.

Watermark / stamp — configure a TextStamper (text, font color, font size, opacity 0–100, rotation, alignment) and call pdf.ApplyStamp(stamper). Guide.

Password protection — set pdf.SecuritySettings.UserPassword, OwnerPassword, and per-permission flags like AllowUserPrinting, then save. Guide.

If your application currently glues Kaizen.io to a second PDF library to get any of these, the migration also collapses that dependency.

What Are the Key API Mappings I Need to Know?

Namespace Mapping

  • Kaizen.IO → IronPdf — core namespace
  • Options live on the same namespace in Kaizen → IronPdf.Rendering — render configuration (paper size, orientation, margins)
  • N/A → IronPdf.Editing — new capability: stamping and watermarking
  • N/A → IronPdf.Security — new capability: passwords and permissions

Core Class Mapping

  • HtmlToPdfConverter → ChromePdfRenderer — PDF generation entry point
  • ConversionOptions → ChromePdfRenderOptions (accessed via renderer.RenderingOptions)
  • HeaderOptions / FooterOptions → HtmlHeaderFooter (assigned to HtmlHeader / HtmlFooter)
  • PageSize enum → PdfPaperSize enum
  • Orientation enum → PdfPaperOrientation enum
  • byte[] return → PdfDocument — use pdf.BinaryData when raw bytes are needed

Document Loading Methods

  • converter.Convert(html) → renderer.RenderHtmlAsPdf(html)
  • converter.Convert(html, options) → configure RenderingOptions, then RenderHtmlAsPdf(html)
  • converter.ConvertUrl(url) → renderer.RenderUrlAsPdf(url)
  • converter.ConvertFile(path) → renderer.RenderHtmlFileAsPdf(path)
  • converter.ConvertAsync(...) → renderer.RenderHtmlAsPdfAsync(...)
  • N/A → PdfDocument.FromFile(path) — load an existing PDF for manipulation

Page Operations

  • options.PageSize → RenderingOptions.PaperSize
  • options.Orientation → RenderingOptions.PaperOrientation
  • options.MarginTop/Bottom/Left/Right → RenderingOptions.MarginTop/Bottom/Left/Right (millimetres in both)
  • options.PageWidth/PageHeight → RenderingOptions.SetCustomPaperSizeInMillimeters(width, height)
  • N/A on the converter → pdf.PageCount on the resulting PdfDocument

Header / Footer Placeholder Mapping

  • {page} → {page}
  • {total} → {total-pages}
  • {date} → {date}
  • {time} → {time}
  • {title} → {html-title}
  • {url} → {url}

Merge/Split Operations (new in IronPDF)

  • Not in Kaizen.io → PdfDocument.Merge(pdf1, pdf2, ...)
  • Not in Kaizen.io → pdf.CopyPages(startIndex, endIndex) — split into a sub-document

What Critical Migration Notes Should I Watch For?

Output model difference

Kaizen.io's Convert returns byte[]. IronPDF's RenderHtmlAsPdf returns a PdfDocument object. Update call sites that expect raw bytes:

// Kaizen.io pattern:
// byte[] pdfBytes = converter.Convert(html);

// IronPDF returns PdfDocument — save or stream as needed:
var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");

// Or get raw bytes when downstream code expects them:
byte[] pdfBytes = pdf.BinaryData;

// Or to MemoryStream:
// https://ironpdf.com/how-to/pdf-memory-stream/
using var ms = new MemoryStream();
pdf.Stream.CopyTo(ms);

Placeholder syntax

Kaizen.io's footer template "Page {page} of {total}" needs {total} rewritten to {total-pages} for IronPDF. {title} becomes {html-title}. The other common placeholders ({page}, {date}, {time}, {url}) carry over unchanged. Grep your codebase for the old placeholders before going live — this is the most common visible regression after migration.

Page indexing (only if you start using IronPDF's manipulation APIs)

Kaizen.io is HTML-in/PDF-out, so it has no notion of page indexing exposed to you. The moment you start using IronPDF's post-render features (split, page deletion, stamping a specific page) you're in 0-based territory:

// IronPDF: 0-based
var firstPage = pdf.Pages[0]; // first page
var lastPage  = pdf.Pages[pdf.PageCount - 1]; // last page

Error handling

Kaizen.io's failure modes are HTTP-shaped: 401 on bad keys, 429 on rate limits, HttpRequestException on network errors. IronPDF's failures are local: render exceptions and licensing exceptions. Most of the catch blocks you wrote for the network model can simply be deleted:

try
{
    var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs("output.pdf");
}
catch (IronPdf.Exceptions.IronPdfException ex)
{
    // Log render failure (malformed HTML, missing fonts, etc.)
    Console.Error.WriteLine($"Render failed: {ex.Message}");
}
// No 429 retry, no HttpRequestException, no rate-limit backoff.

How Should I Handle Performance After Migration?

Renderer reuse

// Instantiate once, reuse for batch work
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

foreach (var html in templates)
{
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"output_{Guid.NewGuid()}.pdf");
}

Concurrent rendering

// Separate renderer per thread — don't share across concurrent operations
Parallel.ForEach(htmlBatch, html =>
{
    var renderer = new ChromePdfRenderer();
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"{Guid.NewGuid()}.pdf");
});
// Parallel examples: https://ironpdf.com/examples/parallel/

Disposal

// PdfDocument implements IDisposable
using var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");
// Auto-disposed at end of block

Chromium-based renderers have a startup cost. In serverless or per-request scaling scenarios, the first render will be slower — pre-warm if latency is critical. System fonts must be present on the render server, so verify font availability in Docker images. Profile under production-representative load before sizing containers.

Frequently Asked Questions

Can I run IronPDF and Kaizen.io side by side during migration?

Yes, and I recommend it. Install IronPDF via NuGet alongside your existing Kaizen.io references, run both generating the same PDF, and compare outputs visually before removing Kaizen.io. Cutting over one endpoint at a time catches rendering differences early.

What is the minimum .NET version required for IronPDF?

IronPDF requires .NET 6+ target framework. If your project is still on .NET Framework or early .NET Core, you will need to upgrade your target framework before migrating. Check the IronPDF docs for current target framework compatibility (it widens over time).

Does IronPDF handle merge, watermark, and encryption without secondary libraries?

Yes. IronPDF consolidates merge, watermark, and encryption into a single library. If you were using secondary libraries like PdfSharp for merging or iTextSharp for watermarking alongside Kaizen.io, you can remove those dependencies after migration.

How do I handle the byte[] to PdfDocument output model change?

If your code passes raw bytes to downstream systems, use pdf.BinaryData or copy pdf.Stream to a MemoryStream. Update all call sites that previously consumed byte[] from converter.Convert. This is the most common migration oversight.

What about my retry-and-backoff code for 429s?

Delete it. IronPDF runs in-process and has no rate limit, so HTTP-shaped error handling (rate-limit retries, 401 detection, network-error fallbacks) becomes dead code. Keep the catch around IronPdfException and IronPdfLicensingException, drop the rest.

How should I structure testing after migration?

Render each HTML template with both libraries and visually diff the output. Verify page counts match for multi-page documents. Verify {total-pages} renders correctly in headers and footers. Test async and concurrent rendering at expected peak load. Regression-test every PDF-generating endpoint.

What Is the Bottom Line on This Migration?

Migrating from Kaizen.io to IronPDF is mostly a shape-preserving substitution: HtmlToPdfConverter → ChromePdfRenderer, ConversionOptions → RenderingOptions, byte[] → PdfDocument. The harder work is around the edges — deleting retry/backoff/network-error code, updating header/footer placeholders, and confirming font availability on whatever box runs the render.

What you gain is qualitative: PDFs no longer leave your network, no rate limits, no per-request bill, and a wider surface area (merge, stamp, encrypt) inside a single dependency. Single-concern codebases that only do HTML-to-PDF migrate in a few hours.

Start with the codebase search commands above to inventory your Kaizen.io usage, run IronPDF in parallel for visual comparison, and cut over one endpoint at a time. The IronPDF getting started guide is the fastest way to validate the basic conversion path before committing to a full migration.
