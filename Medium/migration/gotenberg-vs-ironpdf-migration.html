Migrating from Gotenberg to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Would a .NET Team Migrate Away from Gotenberg?

This migration is architecturally different from a typical library swap. Gotenberg is a network service — a Docker-based microservice exposing HTTP endpoints for document conversion. IronPDF is an in-process .NET library. That distinction changes everything about how you evaluate this move.

Teams running Gotenberg specifically for a .NET service's PDF generation often hit these friction points:

Network latency. Every PDF generation is an HTTP round-trip. That adds measurable latency compared to in-process rendering.

Service dependency. Gotenberg must be running, healthy, and reachable. That's an operational dependency your .NET service doesn't need if it can render PDFs itself.

Container orchestration overhead. Managing the Gotenberg container — health checks, restarts, resource limits — adds DevOps complexity. The Gotenberg Docker image is ~1–1.5GB depending on enabled modules, which affects pull time and registry cost.

Multipart form construction. Gotenberg uses multipart HTTP requests. Building those request bodies in .NET requires more boilerplate than a direct library call.

Error handling across the network. HTTP errors, timeouts, and service unavailability are failure modes you simply don't have with library exceptions.

Local development friction. Developers need Docker running to test PDF generation locally. Remove the container and that friction disappears.

Scaling coupling. Scaling the Gotenberg service separately from your .NET service adds infrastructure complexity. In-process rendering scales with your application.

Debugging difficulty. PDF rendering issues require logs from a separate container, not just your application logs.

How Do I Convert HTML to PDF After Migrating?

This is the most common Gotenberg use case I see teams migrating. The shift from HTTP multipart to a direct library call is dramatic in terms of code reduction.

Before: Gotenberg HTML to PDF via HttpClient

using System;
using System.Net.Http;
using System.Net.Http.Headers;
using System.Text;
using System.Threading.Tasks;
using System.IO;

class Program
{
    static async Task Main(string[] args)
    {
        var client = new HttpClient { BaseAddress = new Uri("http://localhost:3000") };

        string html = @"
            <html>
            <head><style>body { font-family: Arial; margin: 40px; }</style></head>
            <body>
                <h1>Quarterly Report</h1>
                <p>Generated via Gotenberg.</p>
            </body>
            </html>";

        using var content = new MultipartFormDataContent();

        var htmlBytes = Encoding.UTF8.GetBytes(html);
        var htmlPart  = new ByteArrayContent(htmlBytes);
        htmlPart.Headers.ContentType = new MediaTypeHeaderValue("text/html");
        content.Add(htmlPart, "files", "index.html");

        content.Add(new StringContent("A4"), "paperWidth");
        content.Add(new StringContent("A4"), "paperHeight");

        var response = await client.PostAsync("/forms/chromium/convert/html", content);
        response.EnsureSuccessStatusCode();

        byte[] pdfBytes = await response.Content.ReadAsByteArrayAsync();
        await File.WriteAllBytesAsync("output.pdf", pdfBytes);
        Console.WriteLine($"PDF written: {pdfBytes.Length} bytes");
    }
}

After: IronPDF HTML String to PDF

using IronPdf;
using System.IO;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = "YOUR_KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize   = IronPdf.Rendering.PdfPaperSize.A4;
renderer.RenderingOptions.MarginTop   = 10;
renderer.RenderingOptions.MarginBottom = 10;

string html = @"
    <html>
    <head><style>body { font-family: Arial; margin: 40px; }</style></head>
    <body>
        <h1>Quarterly Report</h1>
        <p>Generated via IronPDF.</p>
    </body>
    </html>";

using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
await File.WriteAllBytesAsync("output.pdf", pdf.BinaryData);
Console.WriteLine($"PDF written: {pdf.BinaryData.Length} bytes");

Answer: The Gotenberg version requires constructing a MultipartFormDataContent object, attaching an index.html file, setting content type headers, and making an HTTP POST to /forms/chromium/convert/html. The IronPDF version creates a ChromePdfRenderer, sets paper size and margins via typed properties, and calls RenderHtmlAsPdfAsync. No Docker container, no HTTP client, no multipart form construction.

How Do I Merge PDFs Without the Gotenberg Merge Endpoint?

Gotenberg's merge endpoint requires building multipart form data with each PDF as a named file attachment. The filename ordering determines merge order. Here's how that simplifies.

Before: Gotenberg PDF Merge via HTTP

using System;
using System.Net.Http;
using System.Net.Http.Headers;
using System.IO;
using System.Threading.Tasks;

class MergeExample
{
    static async Task Main(string[] args)
    {
        var client = new HttpClient { BaseAddress = new Uri("http://localhost:3000") };

        string[] filePaths = { "file1.pdf", "file2.pdf" };

        using var content = new MultipartFormDataContent();

        foreach (var path in filePaths)
        {
            var bytes   = await File.ReadAllBytesAsync(path);
            var part    = new ByteArrayContent(bytes);
            part.Headers.ContentType = new MediaTypeHeaderValue("application/pdf");
            content.Add(part, "files", Path.GetFileName(path));
        }

        var response = await client.PostAsync("/forms/pdfengines/merge", content);
        response.EnsureSuccessStatusCode();

        byte[] merged = await response.Content.ReadAsByteArrayAsync();
        await File.WriteAllBytesAsync("merged.pdf", merged);
        Console.WriteLine($"Merged: {merged.Length} bytes");
    }
}

After: IronPDF Merge

using IronPdf;
using System.IO;

IronPdf.License.LicenseKey = "YOUR_KEY";

using var doc1 = PdfDocument.FromFile("file1.pdf");
using var doc2 = PdfDocument.FromFile("file2.pdf");

using var merged = PdfDocument.Merge(doc1, doc2);
merged.SaveAs("merged.pdf");
Console.WriteLine($"Merged: {merged.PageCount} pages");

Answer: Gotenberg merge requires reading each PDF into byte arrays, constructing multipart form data with correct content types, and POSTing to /forms/pdfengines/merge. IronPDF loads files directly with PdfDocument.FromFile and merges with a single PdfDocument.Merge call. You also get PageCount on the result, which Gotenberg doesn't expose.

What New Capabilities Do I Gain That Gotenberg Doesn't Offer?

This is the part of the migration that surprised most teams I've worked with. Gotenberg is excellent at what it does — document conversion via HTTP — but it intentionally doesn't expose page-level manipulation, watermarking, or security features.

Watermarking: Before (Gotenberg Workaround) and After (IronPDF)

// Gotenberg: no direct watermark API.
// Teams inject watermark text via HTML/CSS before sending to Gotenberg.
string htmlWithWatermark = @"
    <html>
    <head>
    <style>
        .watermark {
            position: fixed;
            top: 50%;
            left: 50%;
            transform: translate(-50%, -50%) rotate(-45deg);
            font-size: 60px;
            color: rgba(128, 128, 128, 0.3);
            z-index: 9999;
            pointer-events: none;
        }
    </style>
    </head>
    <body>
        <div class='watermark'>CONFIDENTIAL</div>
        <h1>Document Content</h1>
        <p>Body text here.</p>
    </body>
    </html>";

// Then POST this HTML to Gotenberg
// This approach has limitations: doesn't apply to existing PDFs

After: IronPDF Watermark on Existing PDFs

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR_KEY";

using var pdf = PdfDocument.FromFile("input.pdf");

var stamp = new TextStamper
{
    Text                = "CONFIDENTIAL",
    FontSize            = 40,
    Opacity             = 0.3,
    Rotation            = -45,
    VerticalAlignment   = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};

pdf.ApplyStamp(stamp);
pdf.SaveAs("watermarked.pdf");

Password Protection: Not Available in Gotenberg

// Gotenberg does not expose PDF password/encryption via its API routes.
// Teams typically post-process PDF bytes with a second tool to add encryption.

Console.WriteLine("Password protection is new capability — not available in Gotenberg routes");

After: IronPDF Password Protection

using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR_KEY";

using var pdf = PdfDocument.FromFile("output.pdf");

pdf.SecuritySettings.UserPassword  = "user123";
pdf.SecuritySettings.OwnerPassword = "owner456";
pdf.SecuritySettings.AllowUserPrinting         = PdfPrintSecurity.FullPrintRights;
pdf.SecuritySettings.AllowUserCopyPasteContent = false;

pdf.SaveAs("protected.pdf");
Console.WriteLine("Password protection applied — was not available via Gotenberg");

Answer: Gotenberg's Chromium routes don't expose watermark, stamp, or password protection APIs. Teams either inject watermarks via CSS in the source HTML (which can't apply to existing PDFs) or post-process with a second tool for encryption. IronPDF provides TextStamper for watermarks on any PDF and SecuritySettings for password protection and permission control — all in-process, no extra tooling.

What About Error Handling and Async Patterns?

Gotenberg surfaces errors as HTTP status codes. IronPDF throws exceptions. This is a fundamental shift in how your code handles failures.

Error Handling Migration

// Before: HTTP error handling
var response = await client.PostAsync("/forms/chromium/convert/html", content);
if (!response.IsSuccessStatusCode)
{
    string body = await response.Content.ReadAsStringAsync();
    throw new Exception($"Gotenberg error {response.StatusCode}: {body}");
}

// After: exception handling
try
{
    using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    // ...
}
catch (IronPdf.Exceptions.PdfException ex)
{
    Console.Error.WriteLine($"Render error: {ex.Message}");
}

Async Processing in ASP.NET Core

public class PdfService
{
    private readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

    public async Task<byte[]> GenerateAsync(string html)
    {
        using var pdf = await _renderer.RenderHtmlAsPdfAsync(html);
        return pdf.BinaryData;
    }
}

Answer: Replace HTTP status code checks with try/catch on IronPdf.Exceptions.PdfException. For async patterns, IronPDF's RenderHtmlAsPdfAsync integrates directly with ASP.NET Core's async pipeline — no HttpClient management, no multipart form construction, just a renderer instance and an await call.

Frequently Asked Questions

Can I keep Gotenberg running alongside IronPDF during migration?

Answer: Yes. Since Gotenberg is a separate service, you can run both in parallel. Migrate one endpoint at a time, verify PDF output visually against the Gotenberg baseline, and decommission the Gotenberg container only after all .NET callers have been migrated. If non-.NET services also consume your Gotenberg instance, those callers remain unaffected.

What happens to my Office document conversion if I drop Gotenberg?

Answer: IronPDF does not convert Office documents (Word, Excel, PowerPoint) to PDF. If you use Gotenberg's LibreOffice routes (/forms/libreoffice/convert), you'll need to either retain Gotenberg solely for those routes, use a separate Office conversion library, or restructure your source documents to HTML/CSS templates. This is the single most common blocker I see in this migration.

How does memory usage change when moving from a Gotenberg container to in-process IronPDF?

Answer: Gotenberg runs Chromium in a separate container, so memory usage is isolated from your .NET process. IronPDF runs Chromium in-process, meaning memory pressure affects your application's heap. For memory-constrained deployments, evaluate whether in-process Chromium is acceptable. For high-frequency conversions (100+ per minute), measure both approaches in your environment before concluding which is better for your workload.

Do I need Docker at all after migrating to IronPDF?

Answer: Not for PDF generation. One of the tangible developer experience wins is removing the Docker dependency for local PDF testing. Your .NET application handles rendering in-process. If you were running Docker solely for Gotenberg, you can drop it from your local development setup entirely. Update your local dev setup docs accordingly — no more Gotenberg Docker required.

How Do I Decide Whether This Migration Is Right for My Team?

Before a single line of code changes, answer these questions honestly.

Is the external service model a requirement? If other services (non-.NET) also call Gotenberg, IronPDF won't replace those callers. You'd be migrating only the .NET path.

Are you using Gotenberg's LibreOffice routes? If your pipeline converts Word or Excel documents to PDF via /forms/libreoffice/convert, that capability has no IronPDF equivalent. This is the question that most often stalls the migration.

What's your deployment constraint? Moving from microservice to in-process changes where the CPU/memory overhead lands. Gotenberg isolates Chromium in a container; IronPDF puts it in your application process.

If you're a .NET-only Gotenberg consumer doing HTML and URL conversion, this migration is well-scoped and the code reduction is significant. You gain merge and split capabilities, password protection, watermarking, and page-level manipulation — none of which Gotenberg exposes through its API.

The trade-off is real: you're moving from service isolation to in-process rendering. For teams where that trade-off works, the result is less infrastructure, simpler local development, and a substantially smaller codebase for PDF generation.
