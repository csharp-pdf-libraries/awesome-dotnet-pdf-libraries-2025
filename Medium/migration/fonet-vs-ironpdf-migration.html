Migrating from FO.NET to IronPDF in C#: The Complete Guide

The FO.NET migration question usually starts with a specific failure: an XSL-FO stylesheet that worked for a decade stops rendering correctly on a new server, or the team discovers FO.NET has had no meaningful updates since the mid-2000s, or a new requirement comes in that FO.NET simply cannot fulfill. XSL-FO is a powerful layout specification, but FO.NET is one of the more obscure implementation choices for .NET projects, and its maintenance trajectory is what it is.

This is not a drop-in library swap. FO.NET takes XSL Formatting Objects as input. IronPDF takes HTML. The conceptual gap between XSL-FO and HTML-to-PDF is real and worth understanding before writing any code. In this guide, I'll walk you through identifying what's broken, deciding what to migrate, and making the switch to IronPDF systematically.

Why Should I Migrate Away from FO.NET?

There are several compelling reasons to move, and most of them are not optional — they're forced by the state of the library itself.

Abandonment. FO.NET has no active maintenance. .NET runtime changes (especially .NET Core+) may break it silently or loudly.

.NET Core / .NET 5+ incompatibility. FO.NET was built for .NET Framework. You need to verify if it even runs on your target runtime.

XSL-FO expertise drain. The team that understood XSL-FO may have left. HTML + CSS is far more accessible to modern .NET developers.

Runtime exceptions on upgrade. Upgrading the .NET runtime or OS frequently reveals FO.NET incompatibilities.

Docker/Linux failures. FO.NET's rendering pipeline may depend on Windows GDI+. Linux containers will fail.

Feature gap. Modern requirements (CSS animations as print snapshots, JS-rendered charts) are incompatible with XSL-FO processing.

Tooling scarcity. XSL-FO editors and validators are rare. HTML templates can be built and debugged in any browser.

Pipeline complexity. XSLT to XSL-FO to FO.NET to PDF is a multi-step transformation chain. HTML to IronPDF to PDF is one step.

Error diagnosis. FO.NET errors are often cryptic XML processing exceptions with no actionable message.

How Do FO.NET and IronPDF Compare Side by Side?

The fundamental difference is the input model. FO.NET processes XSL Formatting Objects. IronPDF renders HTML through a real Chromium engine. Here's how they stack up across every major dimension.

  • Focus. FO.NET is an XSL-FO processor that emits PDF; IronPDF renders HTML and URLs through Chromium.
  • Pricing. FO.NET is free and open source but abandoned; IronPDF is commercial.
  • API style. FO.NET is XML/stream processing; IronPDF exposes simple render methods.
  • Learning curve. XSL-FO is complex and unfamiliar to most .NET developers; HTML-first workflows are immediate.
  • HTML rendering. Not applicable in FO.NET — XSL-FO input only. IronPDF gives you Chromium with full CSS/JS.
  • Page indexing. N/A in FO.NET (no page-level access). IronPDF is 0-based.
  • Thread safety. FO.NET's threading guarantees are undocumented — verify before sharing a FonetDriver across threads. IronPDF's ChromePdfRenderer is reusable.
  • Namespace. Fonet versus IronPdf.

How Complex Is This Migration, Feature by Feature?

I want to be upfront about this: some parts of the migration are straightforward, and some require real engineering effort. The template rewrite from XSL-FO to HTML/CSS is likely the bulk of the work. Here's the effort breakdown by feature.

  • XML data to PDF — Medium-High. Pipeline changes from XSL-FO output to HTML template conversion.
  • XSL-FO layout to HTML/CSS — High. Significant design work; this is a template rewrite, not a code swap.
  • Tables with complex spans — High. XSL-FO table model versus HTML table: similar concept, different syntax.
  • Running headers/footers — Medium. <fo:static-content> maps to IronPDF's HtmlHeader and HtmlFooter.
  • Multi-column layout — High. XSL-FO has explicit multi-column controls; the CSS Multi-column spec differs.
  • Page numbering — Medium. <fo:page-number> becomes IronPDF's HTML header placeholders.
  • Merge output — Low (in IronPDF). FO.NET has no native merge, so this is a net new capability.
  • Password/security — Low (in IronPDF). Net new capability; FO.NET had no real encryption support.
  • Watermark — Medium. Net new capability via IronPDF's stamp API.
  • Conditional page masters — High. XSL-FO page master sequences map to CSS @page selectors with caveats.

When Should I Actually Pull the Trigger on Migration?

Not every situation demands immediate migration. Here's a decision matrix to help you figure out where you stand.

  • If FO.NET no longer runs on your .NET version, you must migrate — there's no patch coming.
  • If your XSL-FO templates work today and the team understands them, urgency is lower; validate a long-term maintenance plan instead.
  • If new requirements can't be met by XSL-FO, migrate those requirements to IronPDF first and let coverage grow from there.
  • If you need Linux container support, FO.NET will likely fail; migrate.

How Do I Diagnose Common FO.NET Problems Before Migrating?

Before you write any migration code, it helps to understand what's actually failing. Here are the most common FO.NET problems I've seen and the path forward for each.

FO.NET Won't Load on .NET 6+

The symptom is a System.BadImageFormatException or System.IO.FileNotFoundException when loading the FO.NET DLL. The root cause: FO.NET DLLs target .NET Framework. On .NET Core / .NET 6+, they may not load.

# Check FO.NET DLL target framework
# On Windows (PowerShell):
[System.Reflection.Assembly]::ReflectionOnlyLoadFrom("Fonet.dll").ImageRuntimeVersion

# Check your project target framework:
grep -r "TargetFramework" *.csproj

If the DLL won't load, you can't keep FO.NET. The migration is forced.

XSL-FO Rendering Produces Malformed PDF

The PDF opens but has missing sections, garbled text, or overlapping regions — no exception thrown. Here's the typical FO.NET pipeline for context.

// Typical FO.NET pipeline:

// Step 1: Load XML data
var xmlDoc = new XmlDocument();
xmlDoc.Load("data.xml");

// Step 2: Apply XSLT to get XSL-FO
var transform = new XslCompiledTransform();
transform.Load("stylesheet.xsl");
var foString = new StringBuilder();
transform.Transform(xmlDoc, null, new StringWriter(foString));

// Step 3: Pass XSL-FO to FO.NET
FonetDriver driver = FonetDriver.Make();
using var outputStream = new FileStream("output.pdf", FileMode.Create);
driver.Render(new StringReader(foString.ToString()), outputStream);

If the XSL-FO renders correctly but FO.NET produces bad PDF, the data and transformation logic is salvageable. The pipeline changes from:

[OLD]: XML → XSLT → XSL-FO → FO.NET → PDF
[NEW]: XML → XSLT → HTML → IronPDF → PDF

The key insight: the XML data and the transformation are often good. The output target changes from XSL-FO to HTML.

Missing Fonts in PDF Output

FO.NET outputs PDF with fonts substituted or missing. Font configuration in FO.NET is non-trivial and may involve configuration files pointing to system font directories. With IronPDF, Chromium handles font rendering natively.

using IronPdf;

// Embed custom fonts via HTML @font-face
string htmlWithFont = @"
<html>
<head>
<style>
@font-face {
    font-family: 'CustomFont';
    src: url('https://your-domain.com/fonts/custom.woff2') format('woff2');
}
body { font-family: 'CustomFont', sans-serif; }
</style>
</head>
<body><p>Custom font text</p></body>
</html>";

var renderer = new ChromePdfRenderer();
using var pdf = renderer.RenderHtmlAsPdf(htmlWithFont);
pdf.SaveAs("output_with_fonts.pdf");

// For local font files, use BaseUrl to serve from local file system:
// renderer.RenderingOptions.BaseUrl = new Uri("file:///path/to/fonts/dir/");

How Do I Handle Page Numbering After Migration?

XSL-FO's <fo:page-number> was a first-class feature. Here's what the XSL-FO pattern looked like, for reference:

<!-- XSL-FO page numbering — this becomes HTML headers/footers in IronPDF -->
<fo:static-content flow-name="xsl-region-after">
  <fo:block text-align="center">
    Page <fo:page-number/> of <fo:page-number-citation ref-id="last-page"/>
  </fo:block>
</fo:static-content>

And here's the IronPDF equivalent using HTML headers and footers:

using IronPdf;

var renderer = new ChromePdfRenderer();

// HTML footer with page number placeholder
renderer.RenderingOptions.HtmlFooter = new HtmlHeaderFooter
{
    HtmlFragment = "<div style='text-align:center; font-size:10px;'>Page {page} of {total-pages}</div>",
    DrawDividerLine = false
};

string html = "<html><body><h1>Multi-page Document</h1></body></html>";
using var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("numbered.pdf");
// Docs: https://ironpdf.com/how-to/rendering-options/

How Do I Handle Complex Table Layouts That Repeat Headers Across Pages?

XSL-FO tables with table-header and table-footer that repeated across pages work in FO.NET, but you need the HTML equivalent. The good news: Chromium respects CSS print table rules.

<!-- HTML table headers repeat on page breaks via print CSS -->
<html>
<head>
<style>
  thead { display: table-header-group; }  /* repeat on each page */
  tfoot { display: table-footer-group; }  /* repeat at bottom */
  @media print {
    table { page-break-inside: auto; }
    tr    { page-break-inside: avoid; page-break-after: auto; }
  }
</style>
</head>
<body>
  <table>
    <thead>
      <tr><th>Column A</th><th>Column B</th></tr>
    </thead>
    <tbody>
      <!-- data rows -->
    </tbody>
  </table>
</body>
</html>

IronPDF uses Chromium's print rendering, so display: table-header-group is respected.

How Do I Set Up the Migration Step by Step?

Find FO.NET References in Your Codebase

# Find Fonet/FO.NET references
rg "Fonet\|FonetDriver\|fo\.net" --type cs -i -l

# Find XSL-FO pipeline stages
rg "XslCompiledTransform" --type cs        # XSLT stage
rg "XSL-FO\|xsl-fo\|fo:root" --type xsl   # XSL-FO stylesheets

# Find FO.NET DLL references in project files
rg "Fonet\|fo\.net" --include="*.csproj" --include="*.fsproj" -l

Inventory Your XSL-FO Stylesheets

# List all XSLT stylesheets that produce XSL-FO output
grep -rl "xmlns:fo=\"http://www.w3.org/1999/XSL/Format\"" . --include="*.xsl"
grep -rl "xmlns:fo=" . --include="*.xslt"

Important: These XSL-FO stylesheets represent significant layout work. If you're migrating to HTML output, you'll be writing equivalent HTML/CSS templates. Factor this effort into your migration plan — it's not a code swap, it's a template rewrite.

Uninstall FO.NET and Install IronPDF

# Remove FO.NET reference (may be DLL reference or NuGet)
# If NuGet:
dotnet remove package Fonet

# If DLL reference: remove from .csproj manually

# Install IronPDF
dotnet add package IronPdf

dotnet list package

What Does the Quick Start Migration Look Like in Code?

Step 1: License Setup

// FO.NET: no license setup (public domain / MIT-style)
// IronPDF: license before first render
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
// Docs: https://ironpdf.com/how-to/license-keys/

Step 2: Namespace Imports

// Before (FO.NET)
using Fonet;
using Fonet.Render.Pdf;

// After
using IronPdf;
// Keep: using System.Xml; using System.Xml.Xsl; — for your data transformation

Step 3: Pipeline Restructure

The core change is the XSLT output target: instead of XSL-FO, your XSLT now produces HTML.

// Before: XSLT → XSL-FO → FO.NET → PDF
// var xslfo = ApplyXslt(xmlData, "to_fo.xsl");   // XSLT to XSL-FO
// FonetDriver.Make().Render(new StringReader(xslfo), outputStream);

// After: XSLT → HTML → IronPDF → PDF
using System.Xml;
using System.Xml.Xsl;
using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// Step 1: Transform XML data to HTML (change your XSLT output to HTML)
var transform = new XslCompiledTransform();
transform.Load("to_html.xsl"); // new XSLT targeting HTML output

var xmlDoc = new XmlDocument();
xmlDoc.Load("data.xml");

var htmlWriter = new System.Text.StringBuilder();
transform.Transform(xmlDoc, null, new System.IO.StringWriter(htmlWriter));
string html = htmlWriter.ToString();

// Step 2: IronPDF renders HTML
var renderer = new ChromePdfRenderer();
using var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf");
// Docs: https://ironpdf.com/how-to/html-string-to-pdf/

How Do the APIs Map Between FO.NET and IronPDF?

Here are the complete API mapping tables so you can quickly find the IronPDF equivalent for any FO.NET operation you're currently using.

Namespace Mapping

  • Fonet → IronPdf — core namespace
  • XSL-FO markup → HTML + CSS — input format changes
  • XSLT output method="xml" → XSLT output method="html" — stylesheet retarget

Core Class Mapping

  • FonetDriver → ChromePdfRenderer — main rendering class
  • FonetDriver.Render() → renderer.RenderHtmlAsPdf() — render trigger
  • XSL-FO document (stream) → HTML string — input type changes
  • Output stream → PdfDocument.SaveAs() / BinaryData — output

Document Loading

  • Render XSL-FO stream. FO.NET: driver.Render(stream, outputStream). IronPDF: N/A — the input changes to HTML.
  • Render HTML string. FO.NET: not supported. IronPDF: renderer.RenderHtmlAsPdf(html).
  • Load existing PDF. FO.NET: not supported (it's a renderer, not a PDF object model). IronPDF: PdfDocument.FromFile(path).
  • Load from bytes. FO.NET: not supported. IronPDF: PdfDocument.FromBinaryData(bytes).

Page Operations

  • Running headers. fo:static-content in XSL-FO → RenderingOptions.HtmlHeader.
  • Running footers. fo:static-content in XSL-FO → RenderingOptions.HtmlFooter.
  • Page size. fo:simple-page-master → RenderingOptions.PaperSize.
  • Margins. fo:simple-page-master attributes → RenderingOptions.Margin*.

Merge / Split

  • Merge multiple docs. Not supported natively in FO.NET. IronPDF: PdfDocument.Merge(a, b).
  • Split. Not supported natively in FO.NET. IronPDF: pdf.CopyPages(start, end).

What Do the Complete Before/After Code Migrations Look Like?

Let me walk through four complete migration examples covering the most common use cases.

XML Data to PDF (Core Pipeline)

This is the most common FO.NET use case. The XML data and XSLT stay; the output target changes.

// BEFORE: XSL-FO pipeline
using System.Xml;
using System.Xml.Xsl;
using System.IO;
using Fonet;

class Program
{
    static void Main()
    {
        // Step 1: Load XML data
        var xmlDoc = new XmlDocument();
        xmlDoc.Load("invoice_data.xml");

        // Step 2: XSLT transforms XML → XSL-FO
        var transform = new XslCompiledTransform();
        transform.Load("invoice_to_fo.xsl"); // XSL-FO stylesheet

        var foOutput = new StringBuilder();
        transform.Transform(xmlDoc, null, new StringWriter(foOutput));
        string xslfo = foOutput.ToString();

        // Step 3: FO.NET renders XSL-FO to PDF
        FonetDriver driver = FonetDriver.Make();
        using var outputStream = File.OpenWrite("invoice.pdf");
        driver.Render(new StringReader(xslfo), outputStream);
    }
}
// AFTER: HTML pipeline with IronPDF
using System.Xml;
using System.Xml.Xsl;
using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
        // Docs: https://ironpdf.com/how-to/license-keys/

        // Step 1: Load XML data (unchanged)
        var xmlDoc = new XmlDocument();
        xmlDoc.Load("invoice_data.xml");

        // Step 2: XSLT now produces HTML instead of XSL-FO
        // (requires rewriting invoice_to_fo.xsl → invoice_to_html.xsl)
        var transform = new XslCompiledTransform();
        transform.Load("invoice_to_html.xsl");

        var htmlOutput = new System.Text.StringBuilder();
        transform.Transform(xmlDoc, null, new System.IO.StringWriter(htmlOutput));
        string html = htmlOutput.ToString();

        // Step 3: IronPDF renders HTML to PDF
        var renderer = new ChromePdfRenderer();
        renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;
        // Docs: https://ironpdf.com/how-to/rendering-options/

        using var pdf = renderer.RenderHtmlAsPdf(html);
        pdf.SaveAs("invoice.pdf");
    }
}

Watermark

<!-- BEFORE: XSL-FO watermark via fo:block-container -->

<fo:block-container position="fixed" top="0" left="0"

What Are the Critical Migration Notes I Need to Know?

There are several architectural realities that will affect your migration. I want to lay them out clearly so you're not caught off guard.

XSL-FO is a different input model. FO.NET takes XSL Formatting Objects as input. IronPDF takes HTML. If your pipeline generates XSL-FO via XSLT, you have two options:

Option A: Retarget XSLT to HTML output. Change your XSLT xsl:output method="xml" to method="html". Rewrite layout rules from XSL-FO vocabulary (fo:block, fo:table, fo:simple-page-master) to HTML + CSS equivalents. Higher up-front effort but results in maintainable HTML templates.

Option B: Add an XSL-FO to HTML/PDF transformation step. Use a third-party XSL-FO processor that supports HTML output, or transform XSL-FO to HTML using a separate XSLT stylesheet. Less upfront work but adds complexity.

Neither option is a configuration change. Budget engineering time for template work.

FO.NET feature parity. FO.NET supported a subset of the XSL-FO specification. Some features may have been silently unsupported. Test your migrated HTML output against the original PDF output visually — don't assume 1:1 fidelity in either direction.

Running headers and footers. XSL-FO's fo:static-content with flow-name="xsl-region-before/after" is a first-class feature. IronPDF uses RenderingOptions.HtmlHeader and HtmlFooter with {page} and {total-pages} placeholders. For complex headers (images, company logos, variable data), embed them in the HTML header fragment.

Page master sequences. XSL-FO's fo:page-sequence-master with different masters for first page, odd/even pages has no direct equivalent in IronPDF's HTML model. Use CSS @page :first, @page :left, @page :right selectors and verify Chromium's support for them.

How Do I Handle Performance in the Migration?

FO.NET was a synchronous, single-threaded processor. IronPDF is Chromium-based with async support. For high-volume document generation, this matters.

Async Rendering

// Async render — preferred for server contexts
public async Task<byte[]> GenerateAsync(string html)
{
    using var pdf = await _renderer.RenderHtmlAsPdfAsync(html);
    return pdf.BinaryData;
}

Parallel Batch Processing

using IronPdf;

// Singleton renderer — initialize once
private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

// Parallel render
var tasks = documents.Select(async doc =>
{
    string html = TransformToHtml(doc.XmlData, doc.XsltPath);
    using var pdf = await _renderer.RenderHtmlAsPdfAsync(html);
    return (doc.Id, pdf.BinaryData);
});

var results = await Task.WhenAll(tasks);

Cold start note: IronPDF initializes Chromium on first render. FO.NET's cold start was lighter. Plan a warm-up render in your application startup.

Frequently Asked Questions

Is this a drop-in library replacement?

No, and this is the most important thing to understand. FO.NET takes XSL Formatting Objects as input. IronPDF takes HTML. The input model is fundamentally different.

If your pipeline generates XSL-FO via XSLT, you need to retarget your XSLT to output HTML instead. The XML data and transformation logic are salvageable, but the stylesheet itself needs a rewrite from XSL-FO vocabulary to HTML + CSS.

Can I keep my existing XML data pipeline?

Yes. Your XML data sources stay the same. Your XSLT transformations stay as the mechanism. What changes is the XSLT output target — from XSL-FO markup to HTML markup.

The pipeline goes from XML to XSLT to XSL-FO to FO.NET to PDF, and becomes XML to XSLT to HTML to IronPDF to PDF. The first two stages are preserved.

How do I handle fo:static-content headers and footers?

XSL-FO's fo:static-content maps to IronPDF's RenderingOptions.HtmlHeader and RenderingOptions.HtmlFooter. You write the header/footer content as an HTML fragment with placeholders like {page} and {total-pages}.

For complex headers with images, logos, or variable data, embed them directly in the HTML header fragment. IronPDF renders these through Chromium, so full CSS styling is supported.

Does IronPDF support the same page layout controls as XSL-FO?

The concepts map, but the syntax is different. fo:simple-page-master for page size maps to RenderingOptions.PaperSize. Margin attributes map to RenderingOptions.Margin* properties. Multi-column layouts use CSS Multi-column instead of XSL-FO's explicit column controls.

For conditional page masters (different layouts for first page, odd/even), use CSS @page :first, @page :left, and @page :right selectors.

Will IronPDF work in my Linux Docker container where FO.NET failed?

Yes. FO.NET's rendering pipeline may depend on Windows GDI+, which is why it fails in Linux containers. IronPDF ships with its own Chromium binary and works on Linux without GDI+ dependencies. Update your Dockerfile to account for IronPDF's Linux requirements.

Test on your specific container base image early in the migration process.

How much engineering time should I budget for the template rewrite?

The template rewrite from XSL-FO to HTML/CSS is likely the bulk of the migration effort. Simple documents with basic tables and text may take a day per template. Complex documents with conditional page masters, multi-column layouts, and intricate table spanning could take significantly longer.

Keep your old XSL-FO stylesheets as reference documentation — they describe the layout intent precisely. Use them to guide your HTML/CSS equivalents, page by page. The XSL-FO to HTML layout concept mapping is well-understood territory; most constructs have direct CSS equivalents.

Conclusion

FO.NET migrations are the most conceptually complex because the input model changes — not just the library. The combination of an abandoned tool, .NET Core incompatibility, and an input format that most modern .NET developers don't know makes this feel more daunting than it is. The XSL-FO to HTML/CSS translation is well-understood territory; most layout concepts map directly, and the CSS Print specification covers the print-specific features you need.

The best approach: keep your old XSL-FO stylesheets as reference documentation. They describe the layout intent precisely. Use them to guide your HTML/CSS equivalents, page by page. Start with the HTML-to-PDF quick start, retarget one XSLT stylesheet at a time, and validate visually against the original output. The XML data pipeline stays; only the rendering target changes.
