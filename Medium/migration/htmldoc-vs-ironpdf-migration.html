Migrating from HTMLDOC to IronPDF: Solving Common Migration Pain Points

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Migrate from HTMLDOC to IronPDF?

You're upgrading your stack to .NET 8 and the PDF generation pipeline just became a problem. HTMLDOC is a command-line tool from the early 2000s. The shell-out wrapper your team wrote years ago is getting less tenable as the surrounding infrastructure modernises. The ASP.NET app targeting .NET 8, the Docker images, the deployment pipeline — they've all moved forward. The PDF generator hasn't.

This is a practical guide to replacing an HTMLDOC-based pipeline with IronPDF. Even if you end up with a different library, the audit steps and migration pattern here transfer.

Here are the neutral triggers I see most often for this migration:

.NET version upgrade. .NET 8+ targets and the HTMLDOC subprocess wrapper starts feeling like technical debt. The architecture mismatch becomes visible.

Modern CSS support. HTMLDOC's HTML engine predates CSS Grid, Flexbox, and most CSS3. Templates designed with modern HTML/CSS won't render correctly.

Subprocess management overhead. Error handling, process cleanup, stderr capture, timeout management, retry logic for a CLI tool is non-trivial code. In-process rendering eliminates this class of problem.

Docker image complexity. Installing HTMLDOC in container images means managing OS-level package installs, which differ by base image and create maintenance overhead.

Async PDF generation. Spawning synchronous processes fits awkwardly with async/await patterns. An in-process library integrates cleanly.

No PDF manipulation. HTMLDOC generates PDFs but doesn't merge, split, watermark, or encrypt them. Teams end up adding secondary libraries.

Output quality on complex documents. HTMLDOC's renderer has known limitations with tables, page breaks, and multi-column layouts.

CI/CD portability. HTMLDOC binary availability varies by OS and image. A NuGet package is more portable across build environments.

Problem 1: How Do I Replace the HTMLDOC Shell-Out Pattern with In-Process Rendering?

The subprocess-to-in-process architectural shift is the most significant change in this migration. Your current code writes HTML to a temp file, shells out to the htmldoc binary, captures stderr, handles exit codes, reads the output file, and cleans up. All of that collapses into a few lines with IronPDF's ChromePdfRenderer.

First, install IronPDF and remove the HTMLDOC dependency.

Remove HTMLDOC, Install IronPDF

# Install IronPDF via NuGet
dotnet add package IronPdf
dotnet restore

# Remove from Docker image (update your Dockerfile):
# Remove lines like:
#   RUN apt-get install -y htmldoc
#   COPY htmldoc /usr/local/bin/

# Remove from CI/CD pipeline:
# Find and remove htmldoc install steps in Azure DevOps / GitHub Actions
rg "htmldoc" .github/ azure-pipelines.yml Dockerfile -l 2>/dev/null

Next, set up your license key once at application startup.

License Configuration

using System;
using System.Diagnostics;

// Typical HTMLDOC wrapper pattern
static string GetHtmlDocPath() => "/usr/bin/htmldoc"; // or from config
// No license key — open source CLI tool

After (IronPDF):

using IronPdf;

Now here is the core migration. This is a realistic production HTMLDOC wrapper that handles temp files, timeouts, error codes, and cleanup.

Before: HTMLDOC Subprocess Wrapper

using System;
using System.Diagnostics;
using System.IO;
using System.Threading;

class HtmlToPdfService
{
    private readonly string _htmlDocPath;

    public HtmlToPdfService(string htmlDocPath = "htmldoc")
        => _htmlDocPath = htmlDocPath;

    public byte[] Render(string html, string margins = "1cm")
    {
        var tempHtml = Path.Combine(Path.GetTempPath(), $"{Guid.NewGuid()}.html");
        var tempPdf  = Path.Combine(Path.GetTempPath(), $"{Guid.NewGuid()}.pdf");

        try
        {
            File.WriteAllText(tempHtml, html, System.Text.Encoding.UTF8);

            var args = $"--webpage -t pdf --outfile \"{tempPdf}\" " +
                       $"--top {margins} --bottom {margins} " +
                       $"--left {margins} --right {margins} \"{tempHtml}\"";

            var psi = new ProcessStartInfo(_htmlDocPath, args)
            {
                RedirectStandardError = true,
                RedirectStandardOutput = true,
                UseShellExecute = false,
                CreateNoWindow = true
            };

            using var proc = new Process { StartInfo = psi };
            proc.Start();
            if (!proc.WaitForExit(60_000))
            {
                proc.Kill();
                throw new TimeoutException("htmldoc timed out after 60 seconds");
            }

            if (proc.ExitCode != 0)
                throw new InvalidOperationException(
                    $"htmldoc exit {proc.ExitCode}: {proc.StandardError.ReadToEnd()}"
                );

            return File.ReadAllBytes(tempPdf);
        }
        finally
        {
            if (File.Exists(tempHtml)) File.Delete(tempHtml);
            if (File.Exists(tempPdf))  File.Delete(tempPdf);
        }
    }
}

After: IronPDF In-Process Rendering

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.MarginTop    = 10; // mm
renderer.RenderingOptions.MarginBottom = 10;
renderer.RenderingOptions.MarginLeft   = 10;
renderer.RenderingOptions.MarginRight  = 10;

var pdf = renderer.RenderHtmlAsPdf("<html><body><h1>Invoice</h1></body></html>");
pdf.SaveAs("invoice.pdf");
// Rendering options: https://ironpdf.com/how-to/rendering-options/

No temp files. No process management. No stderr capture. The entire subprocess wrapper class is replaced by a few lines. Error handling also changes — you replace exit code checks with standard try/catch:

Error Handling Migration

// Replace: if (proc.ExitCode != 0) throw ...
// With:
try
{
    var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs("output.pdf");
}
catch (IronPdf.Exceptions.IronPdfException ex)
{
    // Log and handle
}

Problem 2: How Do I Consolidate Merge, Watermark, and Security Operations into One Library?

HTMLDOC generates PDFs but cannot merge, watermark, or encrypt them. Most teams I talk to have added PdfSharp or iTextSharp alongside HTMLDOC to cover these gaps. With IronPDF, all of these operations are native, so you can drop the secondary libraries entirely.

Here is the merge pattern. HTMLDOC has no merge feature, so teams typically use PdfSharp.

Before: Merge with PdfSharp (HTMLDOC Cannot Merge)

using PdfSharp.Pdf;
using PdfSharp.Pdf.IO;
using System;
using System.IO;

class MergePdfsExample
{
    static void Main()
    {
        // HTMLDOC has no merge feature
        // Teams add PdfSharp or iTextSharp alongside it
        using var outputDoc = new PdfDocument();

        foreach (string path in new[] { "section1.pdf", "section2.pdf", "section3.pdf" })
        {
            if (!File.Exists(path))
                throw new FileNotFoundException($"PDF not found: {path}");

            using var input = PdfReader.Open(path, PdfDocumentOpenMode.Import);
            foreach (PdfPage page in input.Pages)
                outputDoc.AddPage(page);
        }

        outputDoc.Save("merged.pdf");
        Console.WriteLine("Merged to: merged.pdf");
    }
}

After: Merge with IronPDF

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var merged = PdfDocument.Merge(
    PdfDocument.FromFile("section1.pdf"),
    PdfDocument.FromFile("section2.pdf"),
    PdfDocument.FromFile("section3.pdf")
);
merged.SaveAs("merged.pdf");
// Guide: https://ironpdf.com/how-to/merge-or-split-pdfs/

Here is the watermark pattern. HTMLDOC cannot watermark PDFs, so teams typically use iTextSharp.

Before: Watermark with iTextSharp (HTMLDOC Cannot Watermark)

using iTextSharp.text;
using iTextSharp.text.pdf;
using System.IO;
using System;

class WatermarkExample
{
    static void Main()
    {
        // HTMLDOC cannot watermark PDFs
        // Common pattern: generate via HTMLDOC, then stamp via iTextSharp
        using var reader  = new PdfReader("generated.pdf");
        using var fs      = new FileStream("watermarked.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs);

        var font = BaseFont.CreateFont(BaseFont.HELVETICA_BOLD, BaseFont.CP1252, false);

        for (int page = 1; page <= reader.NumberOfPages; page++)
        {
            var cb = stamper.GetOverContent(page);
            cb.SaveState();
            cb.BeginText();
            cb.SetFontAndSize(font, 60);
            cb.SetColorFill(new BaseColor(200, 200, 200));
            cb.ShowTextAligned(Element.ALIGN_CENTER, "DRAFT", 297, 420, 45);
            cb.EndText();
            cb.RestoreState();
        }

        Console.WriteLine("Watermarked: watermarked.pdf");
    }
}

After: Watermark with IronPDF

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("generated.pdf");
var stamper = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronSoftware.Drawing.Color.LightGray,
    FontSize = 60,
    Opacity = 30,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");
// Guide: https://ironpdf.com/how-to/custom-watermark/

And here is the password protection pattern.

Before: Password Protection with iTextSharp (HTMLDOC Has No Encryption)

using iTextSharp.text.pdf;
using System.IO;
using System.Text;

class SecurityExample
{
    static void Main()
    {
        // HTMLDOC has no PDF encryption
        // Secondary library required
        byte[] userPass  = Encoding.ASCII.GetBytes("viewonly");
        byte[] ownerPass = Encoding.ASCII.GetBytes("adminaccess");

        using var reader  = new PdfReader("generated.pdf");
        using var fs      = new FileStream("secured.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs, '\0', false);

        stamper.SetEncryption(
            userPass, ownerPass,
            PdfWriter.ALLOW_PRINTING,
            PdfWriter.ENCRYPTION_AES_128
        );
    }
}

After: Password Protection with IronPDF

using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<h1>Confidential Report</h1>");

pdf.SecuritySettings.UserPassword  = "viewonly";
pdf.SecuritySettings.OwnerPassword = "adminaccess";
pdf.SecuritySettings.AllowUserCopyPasteContent = false;
pdf.SecuritySettings.AllowUserPrinting = PdfPrintSecurity.FullPrintRights;
pdf.SaveAs("secured.pdf");
// Guide: https://ironpdf.com/how-to/pdf-permissions-passwords/

Problem 3: How Do I Handle Performance, Concurrency, and Docker Differences After Migrating?

The architectural shift from a CLI tool to an in-process Chromium-based renderer changes how performance, concurrency, and containerisation work. Here is what I recommend you plan for.

Process startup overhead is gone. HTMLDOC incurs process spawn overhead per call — roughly 50-200ms depending on the system. IronPDF's in-process renderer amortises startup cost across calls when you reuse the instance:

Reusing the Renderer Instance

// Process spawn cost: ~50-200ms per call (system dependent)
// In-process renderer: startup amortized across calls when reusing instance
var renderer = new ChromePdfRenderer(); // instantiate once
for (int i = 0; i < 1000; i++)
{
    using var pdf = renderer.RenderHtmlAsPdf(GetHtml(i));
    pdf.SaveAs($"output_{i}.pdf");
}

Concurrent rendering works differently. HTMLDOC achieves parallelism naturally through separate processes. IronPDF requires separate renderer instances per thread:

Parallel Rendering with IronPDF

Parallel.ForEach(htmlItems, new ParallelOptions { MaxDegreeOfParallelism = 4 }, html =>
{
    var renderer = new ChromePdfRenderer(); // one per thread
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"{Guid.NewGuid()}.pdf");
});

Memory footprint is higher. A Chromium-based in-process renderer has a higher baseline memory footprint than spawning a lightweight CLI tool. Profile this in your environment before sizing containers.

Docker images change. You remove the HTMLDOC binary install and IronPDF handles its Chromium binaries via NuGet:

Dockerfile Changes

# Remove from Dockerfile:
# RUN apt-get install -y htmldoc

# IronPDF installs its Chromium binaries via NuGet
# Verify IronPDF Docker setup at: https://ironpdf.com/how-to/azure/

Render output will differ. HTMLDOC uses an HTML 4.01-era rendering engine. IronPDF uses Chromium. The output will not be byte-for-byte identical, and for simple HTML it likely won't be visually identical either. Plan for a visual review of all templates. For most teams this is an improvement, but don't assume — verify.

How Do HTMLDOC and IronPDF Compare Feature by Feature?

  • Focus. HTMLDOC is HTML-to-PDF/PS via a CLI binary. IronPDF is HTML-to-PDF plus full PDF manipulation in-process.
  • Licensing. HTMLDOC is GPL with a paid commercial license available from its maintainer. IronPDF ships under a commercial license; pricing is on the Iron Software site.
  • API style. HTMLDOC uses CLI flags driven by a subprocess. IronPDF is a native .NET library called in-process.
  • Learning curve. HTMLDOC is low for basic use, with shell-out plumbing as the hidden cost. IronPDF is medium — the API surface is larger because the library does more.
  • HTML rendering. HTMLDOC uses a custom HTML 4.01-era engine. IronPDF renders with embedded Chromium.
  • Page indexing. HTMLDOC has none — it only generates. IronPDF is 0-based when you manipulate pages.
  • Thread safety. HTMLDOC gets parallelism naturally through process isolation. IronPDF expects a separate ChromePdfRenderer per thread.
  • Namespace. HTMLDOC has no .NET namespace. IronPDF lives under IronPdf.

Migration Effort by Feature

  • HTML string to PDF — Low. HTMLDOC writes to a temp file and shells out; IronPDF takes the string directly via RenderHtmlAsPdf.
  • HTML file to PDF — Low. HTMLDOC shells out with a file arg; IronPDF uses RenderHtmlFileAsPdf.
  • URL to PDF — Low. HTMLDOC shells out with a URL arg; IronPDF uses RenderUrlAsPdf.
  • Custom margins — Low. CLI flags --top, --bottom, and friends become properties on RenderingOptions.
  • Headers and footers — Medium. CLI flags --header / --footer with cryptic positional codes become HtmlHeaderFooter or TextHeaderFooter objects with full HTML control.
  • Table of contents — Medium. HTMLDOC has a --toc flag; IronPDF generates a TOC via RenderingOptions.CreateTableOfContents().
  • Merge PDFs — Low. Not an HTMLDOC feature; teams add an external library. Native in IronPDF via PdfDocument.Merge.
  • Watermark — Low. Not an HTMLDOC feature; native in IronPDF via TextStamper / ImageStamper.
  • Password protection — Low. Not an HTMLDOC feature; native in IronPDF via SecuritySettings.
  • Modern CSS support — Low to benefit. HTMLDOC does not support CSS3, Grid, or Flexbox. With Chromium, you simply get it.
  • Async integration — Low. Subprocesses fit awkwardly with async/await. IronPDF exposes a native async API.

API Mapping: Document Loading

  • HTML string. HTMLDOC writes the string to a file and passes the file path as an arg — IronPDF takes it directly with renderer.RenderHtmlAsPdf(html).
  • HTML file. htmldoc --outfile output.pdf input.html — renderer.RenderHtmlFileAsPdf(path).
  • URL. htmldoc --webpage url — renderer.RenderUrlAsPdf(url).
  • Existing PDF. Not an HTMLDOC feature — PdfDocument.FromFile(path).

API Mapping: Page Operations

  • Paper size. --size A4 — RenderingOptions.PaperSize.
  • Margins. --top 1cm --bottom 1cm — RenderingOptions.MarginTop, MarginBottom, MarginLeft, MarginRight (millimetres).
  • Orientation. --landscape — RenderingOptions.PaperOrientation.
  • DPI. --jpeg / --resolution — RenderingOptions.DPI.

API Mapping: Merge and Split

  • Merge. Not an HTMLDOC feature; teams reach for an external library — PdfDocument.Merge(pdf1, pdf2).
  • Split. Not an HTMLDOC feature — pdf.CopyPages(startIndex, endIndex).

Frequently Asked Questions

Is HTMLDOC still maintained?

HTMLDOC is open source under the GPL license, with a commercial license available from its maintainer for teams that cannot accept GPL terms. It is a command-line tool from the early 2000s with an HTML 4.01-era rendering engine. It does what it does for simple HTML, but it does not support modern CSS features like Grid, Flexbox, or CSS3.

For teams on .NET 8+ with modern HTML templates, the architecture mismatch becomes visible. If your templates work with HTMLDOC's rendering and you don't need PDF manipulation features, it may still serve your needs.

How difficult is the migration from HTMLDOC to IronPDF?

Most core operations — HTML string to PDF, HTML file to PDF, URL to PDF, and custom margins — are low effort to migrate. Headers/footers and table of contents are medium effort.

The biggest architectural change is moving from subprocess management (temp files, exit codes, stderr capture, timeout handling) to in-process rendering with standard try/catch error handling. If your team also uses secondary libraries like PdfSharp or iTextSharp for merge, watermark, or encryption, those can be consolidated into IronPDF as well, which simplifies the overall dependency graph.

Will my PDF output look the same after migrating?

No. HTMLDOC uses a custom HTML 4.01-era rendering engine. IronPDF uses Chromium. The output will not be byte-for-byte identical, and for simple HTML it likely won't be visually identical either.

Plan for a visual review of all templates. Pay close attention to tables, column layouts, page breaks, and fonts. For most teams this is an improvement — modern CSS features that didn't work at all in HTMLDOC will render correctly — but don't assume. Verify.

What about Docker and CI/CD pipeline changes?

HTMLDOC requires OS-level binary installs in your Docker images and CI/CD pipelines. These differ by base image and create maintenance overhead. IronPDF distributes via NuGet and manages its Chromium binaries through the package, which eliminates the binary management problem.

Remove the apt-get install htmldoc lines from your Dockerfile and the corresponding install steps from your CI/CD pipelines. Be aware that the Chromium-based renderer has a higher baseline memory footprint than the lightweight HTMLDOC CLI, so profile your environment before sizing containers.

Can I use IronPDF with async/await patterns?

Yes. HTMLDOC spawns synchronous processes, which fits awkwardly with async/await patterns. IronPDF provides a native async API that integrates cleanly with modern .NET async pipelines.

For concurrent rendering, create separate ChromePdfRenderer instances per thread rather than sharing a single instance. HTMLDOC achieved parallelism naturally through separate processes; IronPDF requires renderer instance management but avoids the process spawn overhead of roughly 50-200ms per call.

When Should You Migrate from HTMLDOC to IronPDF?

Here is my decision matrix based on the scenarios I see most often:

  • If you only need open source plus simple HTML and no modern CSS, HTMLDOC may still serve — evaluate output quality on your real templates first.
  • If your templates use modern CSS (Grid, Flexbox, CSS3), switch to a Chromium-based renderer — IronPDF, PuppeteerSharp, or similar.
  • If you are air-gapped and cannot get a commercial license approved, HTMLDOC, wkhtmltopdf, or PuppeteerSharp are the open-source options.
  • If you are on .NET 8+ with Docker and care about CI/CD portability, a NuGet-distributed library eliminates the OS-level binary management problem.

The subprocess-to-in-process architectural shift is the most significant change here, not the API surface. Error handling, logging, retry logic, and Docker image management all change character when you go from a CLI tool to a library. If your team is on .NET 8+, uses modern CSS in templates, or needs native merge/watermark/encryption without bolting on secondary libraries, that is the right time to make the switch.
