Migrating from PDF Duo to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Consider Migrating from PDF Duo?

Renewal season has a way of making you look at the dependency list with fresh eyes. The PDF generation library works, but "works" is doing a lot of heavy lifting — you have got a workaround for the encoding issue, a secondary library handling merge, and a comment in the code about a rendering quirk that nobody has touched in months.

Here are eight neutral reasons teams change PDF libraries at renewal time.

Cost structure reassessment. At renewal, the per-developer or per-deployment cost is compared against alternatives. Do this evaluation yourself with current pricing from both vendors.

Maintenance signal. Verify current PDF Duo maintenance status before citing this reason. If commits have slowed and issues go unanswered, it is a legitimate concern.

Feature gaps. PDF/A compliance, digital signatures, advanced manipulation — verify what is missing in PDF Duo versus what IronPDF provides.

Dependency sprawl. If PDF Duo handles generation and you have added two more libraries for manipulation, consolidation is a valid motivation.

Rendering engine. If your templates have CSS fidelity issues, the underlying rendering engine matters. IronPDF uses Chromium. Verify what PDF Duo uses.

Native binary management. If PDF Duo requires native binaries, those add deployment overhead.

.NET version compatibility. Verify PDF Duo's trajectory with your target .NET version.

Vendor lock-in. Thin wrappers around external services create API-contract coupling. An in-process library removes this risk.

Watermark

Before (PDF Duo — verify native support; iTextSharp if secondary):

using iTextSharp.text;
using iTextSharp.text.pdf;
using System.IO;
using System;

class WatermarkExample
{
    static void Main()
    {
        // Verify PDF Duo watermark capability — if not native, iTextSharp typical
        using var reader  = new PdfReader("input.pdf");
        using var fs      = new FileStream("watermarked.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs);

        var font = BaseFont.CreateFont(BaseFont.HELVETICA_BOLD, BaseFont.CP1252, false);
        for (int i = 1; i <= reader.NumberOfPages; i++)
        {
            var cb = stamper.GetOverContent(i);
            cb.SaveState();
            cb.SetGState(new PdfGState { FillOpacity = 0.3f });
            cb.BeginText();
            cb.SetFontAndSize(font, 60);
            cb.SetColorFill(BaseColor.GRAY);
            cb.ShowTextAligned(Element.ALIGN_CENTER, "DRAFT", 297, 420, 45);
            cb.EndText();
            cb.RestoreState();
        }
        Console.WriteLine("Watermarked.");
    }
}

What Are the Frequently Asked Questions?

Answer: I get these four questions on almost every PDF Duo migration I advise on. Here they are with direct answers.

Will My PDFs Look Exactly the Same After Migration?

Answer: Renderer changes will affect CSS rendering. If PDF Duo uses a different engine than Chromium, you should expect differences. Systematic approach: render all templates with PDF Duo as baseline, render again with IronPDF, and diff visually.

Priority areas to check after a renderer change:

CSS FeatureRisk Level
Flexbox / GridHigh if switching from WebKit to Chromium
CSS variablesMedium
@media printMedium
Background colorsMedium (requires explicit browser flag in WebKit)
Custom fontsMedium — font resolution differs
Table layoutLow-Medium — test complex tables
page-break-* CSSLow — standard behavior

IronPDF uses 0-based page indexing for all manipulation operations. Verify PDF Duo's convention and adjust any downstream consumers accordingly.

How Do I Handle Encoding Issues That Follow Me to the New Library?

Answer: Encoding problems in PDF generation are often in the HTML source or the rendering pipeline, not the library. Ensure your HTML is explicitly UTF-8 and verify encoding at file read time.

// Ensure HTML is explicitly UTF-8
string html = @"<html>
<head><meta charset='utf-8' /></head>
<body><p>Content with special chars: ñ, ü, 中文</p></body>
</html>";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(html);
// Chromium handles UTF-8 natively — no special encoding flag needed

// If rendering from a file, verify encoding at read time:
// string html = File.ReadAllText("template.html", System.Text.Encoding.UTF8);

What About the Output Model Change from byte[] to PdfDocument?

Answer: PDF Duo likely returned byte[]. IronPDF returns a PdfDocument object. You will need to update downstream consumers that expect raw bytes.

// PDF Duo likely returned byte[] — update downstream consumers

// Old pattern: byte[] pdfBytes = duoConverter.Convert(html);

// IronPDF returns PdfDocument:
var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf"); // to disk

// To byte[]:
using var ms = new MemoryStream();
pdf.Stream.CopyTo(ms);
byte[] pdfBytes = ms.ToArray();
// Memory stream guide: https://ironpdf.com/how-to/pdf-memory-stream/

// To HTTP response (ASP.NET):
return File(pdf.Stream, "application/pdf", "document.pdf");

What About Performance and Concurrent Rendering?

Answer: Reuse a single renderer instance for batch work. For web applications, use the async API. If you were using secondary libraries alongside PDF Duo, removing them simplifies the dependency graph and can improve startup time.

// One renderer, many renders — efficient for batch processing
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

foreach (var (id, html) in documentQueue)
{
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"output_{id}.pdf");
}

// Async for web applications
[HttpPost("generate")]
public async Task<IActionResult> GeneratePdf([FromBody] string html)
{
    var renderer = new ChromePdfRenderer();
    using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    return File(pdf.Stream, "application/pdf", "document.pdf");
    // Async guide: https://ironpdf.com/how-to/async/
}

What Is the Decision Framework for Your Team?

Answer: Here is the migration checklist I recommend. Work through it in order: pre-migration assessment, code migration, testing, and post-migration cleanup.

Pre-Migration

Verify PDF Duo NuGet package name and current maintenance status. Build usage inventory table covering operations, files, and secondary libraries. Document all current workarounds and what they address. Pull all HTML templates for render comparison testing. Find all secondary libraries added alongside PDF Duo. Verify IronPDF .NET target framework compatibility. Confirm commercial license requirements. Set up IronPDF trial license in dev environment.

Code Migration

Remove PDF Duo NuGet package. Remove secondary libraries if they only supplemented PDF Duo. Add the IronPdf NuGet package. Replace PDF Duo namespace imports. Replace license initialization. Replace HTML-to-PDF calls with ChromePdfRenderer. Replace merge operations with PdfDocument.Merge(). Replace watermark operations with pdf.ApplyStamp(). Replace security operations with pdf.SecuritySettings. Update output handling from byte[] to PdfDocument model. Add IronPDF license key to config.

Testing

Render each HTML template and visually compare against PDF Duo output. Focus on CSS features your templates rely on. Verify workarounds that were in the old code are no longer needed. Test merge with representative document sets. Test watermark on multi-page documents. Test password protection with correct and incorrect credentials. Load test concurrent rendering at expected peak.

Post-Migration

Remove PDF Duo license key from config and secrets. Remove secondary libraries from project if no longer needed. Update deployment documentation. Monitor first production week for any regression.

The renewal-trigger migration often reveals how many secondary workarounds have accumulated around a library over time. The migration audit — finding the workarounds, understanding what they address, verifying whether they follow you to the new library — is frequently the most valuable part of the process, independent of which library you end up with.
