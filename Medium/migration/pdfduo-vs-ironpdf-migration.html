Migrating from PDF Duo to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Consider Migrating from PDF Duo?

Renewal season has a way of making you look at the dependency list with fresh eyes. The PDF generation library works, but "works" is doing a lot of heavy lifting — you have got a workaround for the encoding issue, a secondary library handling merge, and a comment in the code about a rendering quirk that nobody has touched in months. With PDF Duo .NET, that conversation is sharper than usual: the last public release (v2.4) is dated December 10, 2010, the supported runtimes are .NET Framework 1.1 through 3.5, and the component is distributed as a DLL download from duodimension.com — there is no NuGet package.

Here are eight neutral reasons teams move off PDF Duo .NET.

Effectively abandoned. v2.4 in December 2010 was the last release. Roughly fifteen years without an update is a hard signal, especially for code that has to ship on .NET Framework 4.x or .NET 6/7/8/9/10.

No NuGet, no SemVer. Distribution is "download a zip from duodimension.com and reference PDFDuo.dll." There is no dotnet add package, no transitive dependency resolution, and no automated CVE surface for security tooling.

Pre-modern .NET targeting. The vendor lists supported runtimes as .NET Framework 1.1, 2.0, 3.0, and 3.5 on Windows. There is no documented support for .NET Framework 4.x, .NET Core, .NET 5+, Linux, macOS, or Docker.

Feature gaps. The documented API is DuoDimension.HtmlToPdf with OpenHTML(...) and SavePDF(...). There is no documented native API for watermarks, password protection, digital signatures, form filling, text extraction, PDF/A, or PDF merging. IronPDF covers all of these natively.

Dependency sprawl. Because the documented surface is HTML-to-PDF only, teams that needed merge or stamping in the 2010s typically paired PDF Duo with iTextSharp 4.x. Consolidating onto a single library is a valid motivation on its own.

Undocumented rendering engine. The vendor describes PDF Duo as a self-contained component with no Office or Acrobat dependency, but does not publish the HTML/CSS engine it uses. CSS3, modern JavaScript, web fonts, and flex/grid layout are not claimed. IronPDF uses Chromium.

Sparse documentation. A vendor product page, a couple of sample snippets, and a 2010 press release. There is no API reference site, no GitHub repository, and no Stack Overflow tag of meaningful size.

Vendor risk. A single-product 2010-era component with no visible roadmap concentrates risk in one place. An actively maintained library removes it.

What Does the Migration Effort Actually Look Like?

I want to give you a realistic picture of the effort involved. Here is a feature-by-feature breakdown so you can scope the work before writing a single line of migration code.

Effort by Feature

  • HTML file to PDF — Low. PDF Duo uses conv.OpenHTML(htmlPath); conv.SavePDF(pdfPath);. IronPDF uses renderer.RenderHtmlFileAsPdf(htmlPath).SaveAs(pdfPath).
  • URL to PDF — Low. OpenHTML also accepts a URL string. IronPDF uses renderer.RenderUrlAsPdf(url).
  • HTML string to PDF — Low. PDF Duo has no documented "render this HTML string" call — teams write the markup to a temp file first, then call OpenHTML. IronPDF renders strings directly via RenderHtmlAsPdf(html).
  • Custom margins — Medium. The documented HtmlToPdf surface does not include a settings object for paper size, orientation, or margins. IronPDF exposes individual RenderingOptions.Margin* properties.
  • Headers / footers — Medium. No native support in PDF Duo. Add during migration via HtmlHeaderFooter.
  • Merge PDFs — Medium. No native merge API in PDF Duo — teams typically combined it with iTextSharp 4.x. IronPDF consolidates this with PdfDocument.Merge(pdf1, pdf2).
  • Watermark — Medium. No native watermark API in PDF Duo. IronPDF uses pdf.ApplyWatermark(html) or a TextStamper.
  • Password protection — Medium. No native security API in PDF Duo. IronPDF uses pdf.SecuritySettings.
  • Async rendering — Medium. PDF Duo's async story is undocumented; the supported runtimes pre-date async/await. IronPDF exposes RenderHtmlAsPdfAsync and friends.
  • PDF/A compliance — Low in IronPDF. Not present in PDF Duo. IronPDF supports it via save options.
  • Digital signatures — Medium. Not present in PDF Duo. IronPDF uses pdf.SignWithFile(); PAdES depth varies by use case — verify against your compliance requirement.

How Do You Decide Whether to Migrate?

Not every team should migrate. Here is the decision framework I use when advising teams on whether the switch makes sense for their situation.

  • Cost is the only trigger and PDF Duo works well. Evaluate total cost of migration versus renewal — migration has hidden costs in regression testing. That said, "works well" is harder to defend on a 2010 codebase pinned to .NET Framework 3.5.
  • You are stuck on .NET Framework 1.1 / 2.0 / 3.0 / 3.5. If you genuinely cannot move forward of those runtimes, IronPDF will not help — it targets .NET Framework 4.6.2+ and .NET 6/7/8/9/10. Most teams in this position are also planning a runtime upgrade.
  • Dependency sprawl with secondary libraries. IronPDF likely consolidates — everything PDF Duo covers, plus the manipulation features your secondary libraries provide.
  • Maintenance or end-of-life concern. A 2010-dated last release is end-of-life by any reasonable definition; plan migration before a forced runtime upgrade forces it for you.
  • Open source budget requirement. IronPDF is commercial — evaluate PuppeteerSharp or a wkhtmltopdf wrapper instead.

Before you begin, audit your actual usage. Build an inventory of every PDF operation in your codebase.

Inventory Your PDF Duo Usage

# What PDF operations does your codebase perform?
rg "DuoDimension\|HtmlToPdf\b\|OpenHTML\|SavePDF" --type cs -i -n

# Find all PDF-generating code paths
rg "class.*Pdf\|\.pdf\|GeneratePdf\|CreatePdf\|RenderPdf" --type cs -n -i

# Find secondary libraries added alongside PDF Duo
grep -r "iTextSharp\|PdfSharp\|wkhtmltopdf\|MuPdf" **/*.csproj *.csproj 2>/dev/null

Build an inventory like this before writing migration code:

  • HTML-to-PDF — used in OrderService.cs, ReportService.cs — no secondary library.
  • Merge — used in InvoiceBundle.cs — secondary library: iTextSharp 4.x.
  • Watermark — used in DraftPdf.cs — secondary library: iTextSharp 4.x.
  • Password — used in SecureReport.cs — secondary library: iTextSharp 4.x.

This inventory tells you the actual migration scope and whether you can simplify the dependency tree.

Basic HTML-to-PDF

Before (PDF Duo):

using DuoDimension;
using System;
using System.IO;

class BasicConversionExample
{
    static void Main()
    {
        // OpenHTML accepts a file path or URL. There is no documented
        // overload that takes an HTML string directly, so write the
        // markup to a temp file first.
        var tempHtml = Path.GetTempFileName() + ".html";
        File.WriteAllText(tempHtml, "<h1>Hello World</h1>");

        var conv = new HtmlToPdf();
        conv.OpenHTML(tempHtml);
        conv.SavePDF("output.pdf");
        Console.WriteLine("PDF created successfully!");
    }
}

After (IronPDF):

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<h1>Hello World</h1>");
pdf.SaveAs("output.pdf");
// Guide: https://ironpdf.com/how-to/html-string-to-pdf/

What Does Each Before/After Migration Look Like in Practice?

Here are four complete migration examples so you can see exactly what changes in real code.

HTML to PDF with Full Options

Before (PDF Duo):

using DuoDimension;
using System;
using System.IO;

class HtmlToPdfExample
{
    static void Main()
    {
        // The documented HtmlToPdf surface does not include a
        // settings object for paper size, margins, or orientation.
        // OpenHTML / SavePDF is the entire public surface.
        var tempHtml = Path.GetTempFileName() + ".html";
        File.WriteAllText(tempHtml,
            "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>");

        var conv = new HtmlToPdf();
        conv.OpenHTML(tempHtml);
        conv.SavePDF("invoice.pdf");
    }
}

After (IronPDF):

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize    = IronPdf.Rendering.PdfPaperSize.A4;
renderer.RenderingOptions.MarginTop    = 10;
renderer.RenderingOptions.MarginBottom = 10;
renderer.RenderingOptions.MarginLeft   = 15;
renderer.RenderingOptions.MarginRight  = 15;

var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>"
);
pdf.SaveAs("invoice.pdf");
// Full guide: https://ironpdf.com/how-to/html-string-to-pdf/

Merge PDFs

Before (PDF Duo — not native; requires a second library):

// PDF Duo .NET has no native merge API. The 2010-era pattern was to
// render each HTML source to its own PDF with DuoDimension.HtmlToPdf,
// then concatenate the outputs with a separate library such as
// iTextSharp 4.x.
using DuoDimension;
// using iTextSharp.text.pdf;  // separate dependency

var conv = new HtmlToPdf();
conv.OpenHTML("page1.html"); conv.SavePDF("document1.pdf");

conv = new HtmlToPdf();
conv.OpenHTML("page2.html"); conv.SavePDF("document2.pdf");

// Concatenate document1.pdf and document2.pdf with a different PDF
// library; PDF Duo itself does not expose a merge call.

After (IronPDF native):

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var merged = PdfDocument.Merge(
    PdfDocument.FromFile("part1.pdf"),
    PdfDocument.FromFile("part2.pdf")
);
merged.SaveAs("merged.pdf");
// Guide: https://ironpdf.com/how-to/merge-or-split-pdfs/

Watermark

Before (PDF Duo has no native watermark; iTextSharp is the typical fallback):

using iTextSharp.text;
using iTextSharp.text.pdf;
using System.IO;
using System;

class WatermarkExample
{
    static void Main()
    {
        // PDF Duo does not expose a watermark API; secondary library required.
        using var reader  = new PdfReader("input.pdf");
        using var fs      = new FileStream("watermarked.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs);

        var font = BaseFont.CreateFont(BaseFont.HELVETICA_BOLD, BaseFont.CP1252, false);
        for (int i = 1; i <= reader.NumberOfPages; i++)
        {
            var cb = stamper.GetOverContent(i);
            cb.SaveState();
            cb.SetGState(new PdfGState { FillOpacity = 0.3f });
            cb.BeginText();
            cb.SetFontAndSize(font, 60);
            cb.SetColorFill(BaseColor.GRAY);
            cb.ShowTextAligned(Element.ALIGN_CENTER, "DRAFT", 297, 420, 45);
            cb.EndText();
            cb.RestoreState();
        }
        Console.WriteLine("Watermarked.");
    }
}

After (IronPDF):

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("input.pdf");
var stamper = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronSoftware.Drawing.Color.Gray,
    FontSize = 60,
    Opacity = 30,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");
// Guide: https://ironpdf.com/how-to/custom-watermark/

Password Protection

Before (PDF Duo has no native security API):

// PDF Duo .NET does not expose a password / permissions API.
// Teams typically post-process the output with PdfSharp or iTextSharp,
// or migrate to a library that supports security natively.

Console.WriteLine("No PDF Duo security API — this is a feature gap.");

After (IronPDF):

using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("input.pdf");
pdf.SecuritySettings.UserPassword  = "userpass";
pdf.SecuritySettings.OwnerPassword = "ownerpass";
pdf.SecuritySettings.AllowUserPrinting = PdfPrintSecurity.FullPrintRights;
pdf.SecuritySettings.AllowUserAnnotations = false;
pdf.SaveAs("secured.pdf");
// Guide: https://ironpdf.com/how-to/pdf-permissions-passwords/

How Do the Two Libraries Compare Feature by Feature?

Here are the full mappings I use when planning a migration. These cover the overall comparison, namespaces, core classes, document loading, page operations, and merge/split.

Overall Comparison

  • Focus. PDF Duo is a basic HTML-to-PDF converter (file path or URL in, PDF file out). IronPDF is HTML-to-PDF plus full PDF manipulation.
  • Distribution. PDF Duo ships as a DLL download from duodimension.com — no NuGet package exists. IronPDF is on NuGet as IronPdf.
  • Last release. PDF Duo .NET v2.4 in December 2010. IronPDF ships regular releases.
  • Runtime support. PDF Duo targets .NET Framework 1.1 / 2.0 / 3.0 / 3.5 on Windows only. IronPDF supports .NET Framework 4.6.2+, .NET 6/7/8/9/10, Linux, macOS, and Docker.
  • Pricing. Confirm PDF Duo pricing at the vendor's site. IronPDF is commercial with published pricing at ironsoftware.com.
  • API style. PDF Duo exposes one class, DuoDimension.HtmlToPdf, with OpenHTML(...) and SavePDF(path). IronPDF is a fluent in-process .NET library with property-based configuration.
  • HTML rendering. PDF Duo's engine is not disclosed by the vendor. IronPDF is Chromium-based.
  • Page indexing. N/A in PDF Duo — there are no documented page-manipulation methods. IronPDF is 0-based for all manipulation operations.
  • Async support. Not documented in PDF Duo — the supported runtimes pre-date async/await. IronPDF exposes RenderHtmlAsPdfAsync and friends.
  • Namespace. DuoDimension → IronPdf.

Namespace Mapping

  • DuoDimension → IronPdf — core namespace.
  • N/A → IronPdf.Rendering — render-options types (no documented PDF Duo equivalent).
  • N/A → IronPdf.Editing — manipulation (no PDF Duo equivalent).
  • N/A → IronPdf.Security — passwords and permissions (no PDF Duo equivalent).

Core Class Mapping

  • DuoDimension.HtmlToPdf → ChromePdfRenderer — HTML-to-PDF entry point.
  • N/A → ChromePdfRenderOptions / RenderingOptions — render configuration (PDF Duo does not document one).
  • N/A → PdfDocument.Merge static — merge entry point (PDF Duo has no native merge).
  • N/A → PdfDocument — PDF document model for manipulation, no equivalent in PDF Duo.

Document Loading Methods

  • HTML file. conv.OpenHTML(htmlPath); conv.SavePDF(pdfPath); → renderer.RenderHtmlFileAsPdf(htmlPath).SaveAs(pdfPath).
  • URL. conv.OpenHTML(url); conv.SavePDF(pdfPath); → renderer.RenderUrlAsPdf(url).SaveAs(pdfPath).
  • HTML string. Write the markup to a temp file, then OpenHTML it → renderer.RenderHtmlAsPdf(html) directly.
  • Existing PDF. Not documented in PDF Duo → PdfDocument.FromFile(path).

Page Operations

  • Paper size. Not documented in PDF Duo → RenderingOptions.PaperSize.
  • Margins. Not documented in PDF Duo → individual RenderingOptions.Margin* properties (top, right, bottom, left, in millimetres).
  • Orientation. Not documented in PDF Duo → RenderingOptions.PaperOrientation.
  • Page count. Not documented in PDF Duo → pdf.PageCount.

Merge/Split Operations

  • Merge. Not native in PDF Duo — teams paired it with iTextSharp 4.x or similar → PdfDocument.Merge(pdf1, pdf2).
  • Split. Not native in PDF Duo → pdf.CopyPages(startIndex, endIndex).

What Are the Frequently Asked Questions?

I get these four questions on almost every PDF Duo migration I advise on. Here they are with direct answers.

Will My PDFs Look Exactly the Same After Migration?

Renderer changes will affect CSS rendering. PDF Duo's engine is undocumented, so expect differences when moving to Chromium. The systematic approach: render all templates with PDF Duo as the baseline, render again with IronPDF, and diff visually.

Priority areas to check after a renderer change:

  • Flexbox / Grid. High risk if PDF Duo's engine is WebKit-derived or older.
  • CSS variables. Medium risk — modern Chromium handles them; older engines may not.
  • @media print. Medium risk — behavior differs across engines.
  • Background colors. Medium risk — some engines need an explicit print-background flag.
  • Custom fonts. Medium risk — font resolution paths differ.
  • Table layout. Low-to-medium risk — test complex tables with merged cells.
  • page-break-* CSS. Low risk — standard behavior across modern engines.

IronPDF uses 0-based page indexing for all manipulation operations. Verify PDF Duo's convention and adjust any downstream consumers accordingly.

How Do I Handle Encoding Issues That Follow Me to the New Library?

Encoding problems in PDF generation are often in the HTML source or the rendering pipeline, not the library. Ensure your HTML is explicitly UTF-8 and verify encoding at file read time.

// Ensure HTML is explicitly UTF-8
string html = @"<html>
<head><meta charset='utf-8' /></head>
<body><p>Content with special chars: ñ, ü, 中文</p></body>
</html>";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(html);
// Chromium handles UTF-8 natively — no special encoding flag needed

// If rendering from a file, verify encoding at read time:
// string html = File.ReadAllText("template.html", System.Text.Encoding.UTF8);

What About the Output Model Change from a File Path to PdfDocument?

PDF Duo's HtmlToPdf.SavePDF(path) writes directly to a file. There is no documented in-memory byte API. IronPDF returns a PdfDocument object that you then save to disk, expose as bytes, stream, or further manipulate. Update downstream consumers that expect the file-path-only signature.

// Old PDF Duo pattern: conv.OpenHTML(htmlPath); conv.SavePDF("output.pdf");

// IronPDF returns PdfDocument:
var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf"); // to disk

// To byte[]:
byte[] pdfBytes = pdf.BinaryData;
// Memory stream guide: https://ironpdf.com/how-to/pdf-memory-stream/

// To HTTP response (ASP.NET):
return File(pdf.Stream, "application/pdf", "document.pdf");

What About Performance and Concurrent Rendering?

Reuse a single renderer instance for batch work. For web applications, use the async API. If you were using secondary libraries alongside PDF Duo, removing them simplifies the dependency graph and can improve startup time.

// One renderer, many renders — efficient for batch processing
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

foreach (var (id, html) in documentQueue)
{
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"output_{id}.pdf");
}

// Async for web applications
[HttpPost("generate")]
public async Task<IActionResult> GeneratePdf([FromBody] string html)
{
    var renderer = new ChromePdfRenderer();
    using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    return File(pdf.Stream, "application/pdf", "document.pdf");
    // Async guide: https://ironpdf.com/how-to/async/
}

What Is the Decision Framework for Your Team?

Here is the migration checklist I recommend. Work through it in order: pre-migration assessment, code migration, testing, and post-migration cleanup.

Pre-Migration

Locate the vendored PDFDuo.dll (and PDFDuoNET.dll if present) in your /lib folder or alongside the project — PDF Duo .NET is not on NuGet. Build a usage inventory covering operations, files, and secondary libraries. Document all current workarounds and what they address. Pull all HTML templates for render comparison testing. Find all secondary libraries added alongside PDF Duo, especially anything iTextSharp 4.x-shaped that handled merge or stamping. Verify IronPDF .NET target framework compatibility. Confirm commercial license requirements. Set up an IronPDF trial license in the dev environment.

Code Migration

Remove the project reference to PDFDuo.dll and delete the vendored binary. Remove secondary libraries if they only supplemented PDF Duo. Add the IronPdf NuGet package. Replace using DuoDimension; with the IronPDF namespaces. Replace license initialization. Replace conv.OpenHTML(...) / conv.SavePDF(...) calls with ChromePdfRenderer.RenderHtmlAsPdf / RenderUrlAsPdf / RenderHtmlFileAsPdf followed by pdf.SaveAs. Replace any temp-file detour you used to render HTML strings — IronPDF renders strings directly. Replace the iTextSharp-based merge step with PdfDocument.Merge(). Add pdf.ApplyStamp() or ApplyWatermark() where you had no watermark before. Add pdf.SecuritySettings where you had no security before. Update the output handling: PDF Duo wrote to a path directly; IronPDF returns a PdfDocument you save explicitly. Add the IronPDF license key to config.

Testing

Render each HTML template and visually compare against PDF Duo output. Focus on CSS features your templates rely on. Verify workarounds in the old code are no longer needed. Test merge with representative document sets. Test watermark on multi-page documents. Test password protection with correct and incorrect credentials. Load test concurrent rendering at expected peak.

Post-Migration

Remove any PDF Duo license key or DLL path from config and secrets. Remove secondary libraries from the project if no longer needed. Update deployment documentation. Monitor the first production week for any regression.

The renewal-trigger migration often reveals how many secondary workarounds have accumulated around a library over time. The migration audit — finding the workarounds, understanding what they address, verifying whether they follow you to the new library — is frequently the most valuable part of the process, independent of which library you end up with.
