Migrating from PDF Duo to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Consider Migrating from PDF Duo?

Renewal season has a way of making you look at the dependency list with fresh eyes. The PDF generation library works, but "works" is doing a lot of heavy lifting — you have got a workaround for the encoding issue, a secondary library handling merge, and a comment in the code about a rendering quirk that nobody has touched in months.

Here are eight neutral reasons teams change PDF libraries at renewal time.

Cost structure reassessment. At renewal, the per-developer or per-deployment cost is compared against alternatives. Do this evaluation yourself with current pricing from both vendors.

Maintenance signal. PDF Duo .NET shows weak signals here — no visible public repository, sparse NuGet activity, and support forums with posts from 2019. Verify current status before citing this reason, but the gap is real.

Feature gaps. PDF Duo covers basic HTML-to-PDF and merge. It does not provide headers/footers, watermarks, password protection, form filling, digital signatures, text extraction, or PDF/A. IronPDF covers all of these natively.

Dependency sprawl. If PDF Duo handles generation and you have added two more libraries for manipulation, consolidation is a valid motivation.

Rendering engine. If your templates have CSS fidelity issues, the underlying rendering engine matters. IronPDF uses Chromium. PDF Duo's rendering engine is undocumented — that is itself a signal.

Native binary management. If PDF Duo requires native binaries, those add deployment overhead.

.NET version compatibility. Verify PDF Duo's trajectory with your target .NET version — the lack of public release notes makes modern .NET support hard to confirm.

Vendor lock-in. Thin wrappers around external services create API-contract coupling. An in-process library removes this risk.

What Does the Migration Effort Actually Look Like?

I want to give you a realistic picture of the effort involved. Here is a feature-by-feature breakdown so you can scope the work before writing a single line of migration code.

Effort by Feature

  • HTML string to PDF — Low. PDF Duo uses HtmlToPdfConverter.ConvertHtmlString(html, path). IronPDF uses ChromePdfRenderer.RenderHtmlAsPdf(html) followed by SaveAs.
  • URL to PDF — Low. PDF Duo uses converter.ConvertUrl(url, path). IronPDF uses renderer.RenderUrlAsPdf(url).
  • HTML file to PDF — Low. PDF Duo uses converter.ConvertFile(htmlPath, pdfPath). IronPDF uses renderer.RenderHtmlFileAsPdf(path).
  • Custom margins — Low. PDF Duo passes a Margins object on a settings instance; IronPDF uses individual RenderingOptions.Margin* properties.
  • Headers / footers — Medium. PDF Duo has no native support, so this is a feature you add during migration via HtmlHeaderFooter.
  • Merge PDFs — Low. PDF Duo has a native PdfMerger with AddFile / Merge. IronPDF uses PdfDocument.Merge(pdf1, pdf2).
  • Watermark — Low. PDF Duo has no native watermark support; you likely have a secondary library doing this. IronPDF uses pdf.ApplyWatermark(html) or a TextStamper.
  • Password protection — Low. Not native in PDF Duo. IronPDF uses pdf.SecuritySettings.
  • Async rendering — Medium. PDF Duo's async story is undocumented. IronPDF exposes RenderHtmlAsPdfAsync and friends.
  • PDF/A compliance — Low in IronPDF. Not present in PDF Duo. IronPDF supports it via save options.
  • Digital signatures — Medium. Not present in PDF Duo. IronPDF uses pdf.SignWithFile(); PAdES depth varies by use case — verify against your compliance requirement.

How Do You Decide Whether to Migrate?

Not every team should migrate. Here is the decision framework I use when advising teams on whether the switch makes sense for their situation.

  • Cost is the only trigger and PDF Duo works well. Evaluate total cost of migration versus renewal — migration has hidden costs in regression testing.
  • Dependency sprawl with secondary libraries. IronPDF likely consolidates — everything PDF Duo covers, plus the manipulation features your secondary libraries provide.
  • Maintenance or end-of-life concern. PDF Duo's signals point this way; if confirmed for your version, plan migration before a forced upgrade.
  • Open source budget requirement. IronPDF is commercial — evaluate wkhtmltopdf wrappers or PuppeteerSharp instead.

Before you begin, audit your actual usage. Build an inventory of every PDF operation in your codebase.

Inventory Your PDF Duo Usage

# What PDF operations does your codebase perform?
rg "PDFDuo\|HtmlToPdfConverter\|PdfMerger" --type cs -i -n

# Find all PDF-generating code paths

  • Merge — used in InvoiceBundle.cs — secondary library: PdfSharp.
  • Watermark — used in DraftPdf.cs — secondary library: iTextSharp.

What Does Each Before/After Migration Look Like in Practice?

Here are four complete migration examples so you can see exactly what changes in real code.

HTML to PDF with Full Options

Before (PDF Duo):

using PDFDuo;
using System;

class HtmlToPdfExample
{
    static void Main()
    {
        var settings = new PDFSettings
        {
            PageSize = PageSize.A4,
            Margins  = new Margins(top: 10, right: 15, bottom: 10, left: 15),
            Orientation = Orientation.Portrait
        };

        var converter = new HtmlToPdfConverter(settings);
        converter.ConvertHtmlString(
            "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>",
            "invoice.pdf"
        );
    }
}

After (IronPDF):

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize    = IronPdf.Rendering.PdfPaperSize.A4;
renderer.RenderingOptions.MarginTop    = 10;
renderer.RenderingOptions.MarginBottom = 10;
renderer.RenderingOptions.MarginLeft   = 15;
renderer.RenderingOptions.MarginRight  = 15;

var pdf = renderer.RenderHtmlAsPdf(
    "<html><body><h1>Invoice #1234</h1><p>Amount: $500</p></body></html>"
);
pdf.SaveAs("invoice.pdf");
// Full guide: https://ironpdf.com/how-to/html-string-to-pdf/

Merge PDFs

Before (PDF Duo native):

using PDFDuo;
using System;

class MergePdfsExample
{
    static void Main()
    {
        var merger = new PdfMerger();
        merger.AddFile("part1.pdf");
        merger.AddFile("part2.pdf");
        merger.Merge("merged.pdf");
        Console.WriteLine("Merged: merged.pdf");
    }
}

After (IronPDF native):

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var merged = PdfDocument.Merge(
    PdfDocument.FromFile("part1.pdf"),
    PdfDocument.FromFile("part2.pdf")
);
merged.SaveAs("merged.pdf");
// Guide: https://ironpdf.com/how-to/merge-or-split-pdfs/

Watermark

Before (PDF Duo has no native watermark; iTextSharp is the typical fallback):

using iTextSharp.text;
using iTextSharp.text.pdf;
using System.IO;
using System;

class WatermarkExample
{
    static void Main()
    {
        // PDF Duo does not expose a watermark API; secondary library required.
        using var reader  = new PdfReader("input.pdf");
        using var fs      = new FileStream("watermarked.pdf", FileMode.Create);
        using var stamper = new PdfStamper(reader, fs);

        var font = BaseFont.CreateFont(BaseFont.HELVETICA_BOLD, BaseFont.CP1252, false);
        for (int i = 1; i <= reader.NumberOfPages; i++)
        {
            var cb = stamper.GetOverContent(i);
            cb.SaveState();
            cb.SetGState(new PdfGState { FillOpacity = 0.3f });
            cb.BeginText();
            cb.SetFontAndSize(font, 60);
            cb.SetColorFill(BaseColor.GRAY);
            cb.ShowTextAligned(Element.ALIGN_CENTER, "DRAFT", 297, 420, 45);
            cb.EndText();
            cb.RestoreState();
        }
        Console.WriteLine("Watermarked.");
    }
}

After (IronPDF):

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("input.pdf");
var stamper = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronSoftware.Drawing.Color.Gray,
    FontSize = 60,
    Opacity = 30,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");
// Guide: https://ironpdf.com/how-to/custom-watermark/

Password Protection

Before (PDF Duo has no native security API):

// PDF Duo .NET does not expose a password / permissions API.
// Teams typically post-process the output with PdfSharp or iTextSharp,
// or migrate to a library that supports security natively.

Console.WriteLine("No PDF Duo security API — this is a feature gap.");

After (IronPDF):

using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("input.pdf");
pdf.SecuritySettings.UserPassword  = "userpass";
pdf.SecuritySettings.OwnerPassword = "ownerpass";
pdf.SecuritySettings.AllowUserPrinting = PdfPrintSecurity.FullPrintRights;
pdf.SecuritySettings.AllowUserAnnotations = false;
pdf.SaveAs("secured.pdf");
// Guide: https://ironpdf.com/how-to/pdf-permissions-passwords/

How Do the Two Libraries Compare Feature by Feature?

Here are the full mappings I use when planning a migration. These cover the overall comparison, namespaces, core classes, document loading, page operations, and merge/split.

Overall Comparison

  • Focus. PDF Duo is a basic HTML-to-PDF library with merge support. IronPDF is HTML-to-PDF plus full PDF manipulation.
  • Pricing. PDF Duo's pricing is hard to confirm publicly — verify at the vendor's site. IronPDF is commercial with published pricing at ironsoftware.com.
  • API style. PDF Duo uses a settings-object pattern passed to HtmlToPdfConverter. IronPDF is a fluent in-process .NET library with property-based configuration.
  • Learning curve. PDF Duo is small and limited; IronPDF surface area is larger but well documented.
  • HTML rendering. PDF Duo's engine is undocumented. IronPDF is Chromium-based.
  • Page indexing. Verify PDF Duo's convention against your code. IronPDF is 0-based for all manipulation operations.
  • Thread safety. Verify in PDF Duo. IronPDF supports renderer instance reuse — see async docs.
  • Namespace. PDFDuo → IronPdf.

Namespace Mapping

  • PDFDuo → IronPdf — core namespace.
  • PDFDuo settings types → IronPdf.Rendering — configuration types.
  • N/A → IronPdf.Editing — manipulation (no PDF Duo equivalent).
  • N/A → IronPdf.Security — passwords and permissions (no PDF Duo equivalent).

Core Class Mapping

  • HtmlToPdfConverter → ChromePdfRenderer — HTML-to-PDF entry point.
  • PDFSettings → ChromePdfRenderOptions — render configuration.
  • PdfMerger → PdfDocument.Merge static — merge entry point.
  • N/A → PdfDocument — PDF document model for manipulation, no equivalent in PDF Duo.

Document Loading Methods

  • HTML string. converter.ConvertHtmlString(html, path) → renderer.RenderHtmlAsPdf(html).
  • URL. converter.ConvertUrl(url, path) → renderer.RenderUrlAsPdf(url).
  • HTML file. converter.ConvertFile(htmlPath, pdfPath) → renderer.RenderHtmlFileAsPdf(path).
  • Existing PDF. PDFDocument.Load(path) → PdfDocument.FromFile(path).

Page Operations

  • Paper size. settings.PageSize → RenderingOptions.PaperSize.
  • Margins. new Margins(t, r, b, l) → individual RenderingOptions.Margin* properties.
  • Orientation. settings.Orientation → RenderingOptions.PaperOrientation.
  • Page count. Verify on PDF Duo. IronPDF: pdf.PageCount.

Merge/Split Operations

  • Merge. new PdfMerger() / AddFile / Merge(path) → PdfDocument.Merge(pdf1, pdf2).
  • Split. Not native in PDF Duo. IronPDF: pdf.CopyPages(startIndex, endIndex).

What Are the Frequently Asked Questions?

I get these four questions on almost every PDF Duo migration I advise on. Here they are with direct answers.

Will My PDFs Look Exactly the Same After Migration?

Renderer changes will affect CSS rendering. PDF Duo's engine is undocumented, so expect differences when moving to Chromium. The systematic approach: render all templates with PDF Duo as the baseline, render again with IronPDF, and diff visually.

Priority areas to check after a renderer change:

  • Flexbox / Grid. High risk if PDF Duo's engine is WebKit-derived or older.
  • CSS variables. Medium risk — modern Chromium handles them; older engines may not.
  • @media print. Medium risk — behavior differs across engines.
  • Background colors. Medium risk — some engines need an explicit print-background flag.
  • Custom fonts. Medium risk — font resolution paths differ.
  • Table layout. Low-to-medium risk — test complex tables with merged cells.
  • page-break-* CSS. Low risk — standard behavior across modern engines.

IronPDF uses 0-based page indexing for all manipulation operations. Verify PDF Duo's convention and adjust any downstream consumers accordingly.

How Do I Handle Encoding Issues That Follow Me to the New Library?

Encoding problems in PDF generation are often in the HTML source or the rendering pipeline, not the library. Ensure your HTML is explicitly UTF-8 and verify encoding at file read time.

// Ensure HTML is explicitly UTF-8
string html = @"<html>
<head><meta charset='utf-8' /></head>
<body><p>Content with special chars: ñ, ü, 中文</p></body>
</html>";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf(html);
// Chromium handles UTF-8 natively — no special encoding flag needed

// If rendering from a file, verify encoding at read time:
// string html = File.ReadAllText("template.html", System.Text.Encoding.UTF8);

What About the Output Model Change from a File Path to PdfDocument?

PDF Duo's HtmlToPdfConverter.ConvertHtmlString writes directly to a path. IronPDF returns a PdfDocument object that you then save, stream, or further manipulate. Update downstream consumers that expect the file-path-only signature.

// Old PDF Duo pattern: converter.ConvertHtmlString(html, "output.pdf");

// IronPDF returns PdfDocument:
var pdf = renderer.RenderHtmlAsPdf(html);
pdf.SaveAs("output.pdf"); // to disk

// To byte[]:
byte[] pdfBytes = pdf.BinaryData;
// Memory stream guide: https://ironpdf.com/how-to/pdf-memory-stream/

// To HTTP response (ASP.NET):
return File(pdf.Stream, "application/pdf", "document.pdf");

What About Performance and Concurrent Rendering?

Reuse a single renderer instance for batch work. For web applications, use the async API. If you were using secondary libraries alongside PDF Duo, removing them simplifies the dependency graph and can improve startup time.

// One renderer, many renders — efficient for batch processing
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

foreach (var (id, html) in documentQueue)
{
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"output_{id}.pdf");
}

// Async for web applications
[HttpPost("generate")]
public async Task<IActionResult> GeneratePdf([FromBody] string html)
{
    var renderer = new ChromePdfRenderer();
    using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
    return File(pdf.Stream, "application/pdf", "document.pdf");
    // Async guide: https://ironpdf.com/how-to/async/
}

What Is the Decision Framework for Your Team?

Here is the migration checklist I recommend. Work through it in order: pre-migration assessment, code migration, testing, and post-migration cleanup.

Pre-Migration

Verify the PDF Duo NuGet package name (PDFDuo.NET in most projects) and current maintenance status. Build a usage inventory covering operations, files, and secondary libraries. Document all current workarounds and what they address. Pull all HTML templates for render comparison testing. Find all secondary libraries added alongside PDF Duo. Verify IronPDF .NET target framework compatibility. Confirm commercial license requirements. Set up an IronPDF trial license in the dev environment.

Code Migration

Remove the PDFDuo.NET NuGet package. Remove secondary libraries if they only supplemented PDF Duo. Add the IronPdf NuGet package. Replace using PDFDuo; with the IronPDF namespaces. Replace license initialization. Replace HtmlToPdfConverter.ConvertHtmlString / ConvertUrl calls with ChromePdfRenderer.RenderHtmlAsPdf / RenderUrlAsPdf. Replace PdfMerger with PdfDocument.Merge(). Add pdf.ApplyStamp() or ApplyWatermark() where you had no watermark before. Add pdf.SecuritySettings where you had no security before. Update the output handling: PDF Duo wrote to a path directly; IronPDF returns a PdfDocument you save explicitly. Add the IronPDF license key to config.

Testing

Render each HTML template and visually compare against PDF Duo output. Focus on CSS features your templates rely on. Verify workarounds in the old code are no longer needed. Test merge with representative document sets. Test watermark on multi-page documents. Test password protection with correct and incorrect credentials. Load test concurrent rendering at expected peak.

Post-Migration

Remove any PDF Duo license key from config and secrets. Remove secondary libraries from the project if no longer needed. Update deployment documentation. Monitor the first production week for any regression.

The renewal-trigger migration often reveals how many secondary workarounds have accumulated around a library over time. The migration audit — finding the workarounds, understanding what they address, verifying whether they follow you to the new library — is frequently the most valuable part of the process, independent of which library you end up with.
