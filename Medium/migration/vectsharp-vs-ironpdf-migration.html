Migrating from VectSharp to IronPDF: The Drawing API to HTML Transition Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Consider Migrating from VectSharp to IronPDF?

VectSharp is a vector graphics library. It draws shapes, lines, text, and figures with precision. What it does not do natively is take an HTML string and produce a paginated PDF from it. Teams that chose VectSharp for diagram and chart rendering eventually hit the point where they also need report-style HTML output, and the two requirements are architecturally separate.

Adding a second library to handle HTML-to-PDF creates a dependency split that compounds over time. Here are the migration triggers I see most often.

First, there is the HTML input gap. VectSharp generates PDFs from drawing operations. Generating from HTML requires converting HTML to VectSharp drawing calls, which is a significant intermediate step.

Second, the layout calculation burden. Positioning text and layout elements requires explicit coordinate calculation. HTML and CSS handle this automatically.

Third, CSS and web content sources. If your input data comes from a web template or Razor view, VectSharp requires converting that to drawing API calls.

Fourth, table and list rendering. Complex tabular layouts that are trivial in HTML with <table> require manual coordinate math in VectSharp.

Fifth, font management. VectSharp has its own font loading mechanism via FontLibrary. Web fonts from Google Fonts or embedded CSS @font-face are not directly usable.

Sixth, there is no PDF manipulation. VectSharp creates PDFs but it does not merge, watermark, add passwords, or extract text from existing PDFs.

Seventh, the two-library pattern. Teams using VectSharp for diagrams plus another library for HTML-to-PDF carry both dependencies.

Eighth, page numbering. Adding page numbers across a multi-page document in VectSharp requires manual tracking. HTML footer patterns handle this automatically.

Ninth, image-heavy report layouts. Embedding images in VectSharp PDFs requires explicit rasterization or SVG embedding steps.

Tenth, team familiarity. Teams with web development backgrounds find HTML/CSS templates easier to maintain than coordinate-based drawing code.

What Does the Migration Effort Actually Look Like?

I want to give you a realistic picture of the effort involved. This is not a simple library swap — it is a model change. VectSharp's model is: describe what to draw, where to draw it. IronPDF's model is: provide HTML, get back a PDF. Here is a feature-by-feature breakdown.

Effort by Feature

  • HTML string to PDF — Low (new capability). Not applicable in VectSharp; in IronPDF it is ChromePdfRenderer.RenderHtmlAsPdfAsync().
  • Drawing-based PDF — High (rewrite). VectSharp's Graphics.* drawing API becomes an HTML/CSS template in IronPDF.
  • Save to file — Low. Document.SaveAsPDF(path) maps directly to pdf.SaveAs(path).
  • Save to stream — Low. Document.SaveAsPDF(stream) maps to pdf.Stream or pdf.BinaryData.
  • Custom page size — Low. Page(width, height) becomes RenderingOptions.PaperSize.
  • Merge PDFs — Medium. Not native to VectSharp; IronPDF provides PdfDocument.Merge().
  • Watermark — Medium. Manual drawing layer in VectSharp; TextStamper or ImageStamper in IronPDF.
  • Password protection — Low. Not native to VectSharp; pdf.SecuritySettings in IronPDF.
  • Text extraction — Medium. Not applicable in VectSharp; pdf.ExtractAllText() in IronPDF.
  • Diagrams/charts — High. VectSharp's native strength is vector drawing; IronPDF approaches this via HTML plus JavaScript chart libraries.
  • Custom fonts — Medium. VectSharp's FontLibrary system becomes CSS @font-face or Google Fonts.

How Do You Decide Whether to Migrate or Keep Both?

Not every team should do a full replacement. Here is a decision matrix to help you figure out where you fall.

  • If your primary need is HTML/CSS report generation, switch — VectSharp is drawing-first and IronPDF is HTML-first.
  • If your primary need is precise diagram or chart rendering from data, keep VectSharp for the diagrams and add IronPDF for HTML reports.
  • If you need both diagrams and HTML reports, run both libraries together and scope the boundary carefully.
  • If you need PDF merge, watermark, or security on existing PDFs, switch for those operations — VectSharp does not cover them.

How Do the Two Libraries Compare Side by Side?

Before diving into code, here is a direct feature comparison so you can see the architectural differences at a glance.

  • Focus. VectSharp is built for vector graphics creation with PDF, SVG, and raster output. IronPDF is built for HTML-to-PDF plus PDF manipulation.
  • Pricing. VectSharp is open source under LGPL-3.0-only. IronPDF is commercial; current pricing is on ironsoftware.com.
  • API style. VectSharp exposes drawing calls like Graphics.FillText, FillPath, and FillRectangle. IronPDF exposes ChromePdfRenderer plus HTML input.
  • Learning curve. VectSharp is low for developers with a graphics background but harder for report layouts. IronPDF is low for web developers because HTML and CSS are the input.
  • HTML rendering. Not applicable in VectSharp; embedded Chromium in IronPDF.
  • Page indexing. Both libraries are 0-based.
  • Thread safety. VectSharp's per-Document/Page objects are not designed for shared mutation across threads; IronPDF supports concurrent rendering using separate ChromePdfRenderer instances.
  • Namespace. VectSharp and VectSharp.PDF become IronPdf.

I will walk you through the core migration. First, find your existing VectSharp references, then swap the packages, and finally update your code.

Basic PDF Generation

Before (VectSharp drawing API):

using VectSharp;
using VectSharp.PDF;
using System;

class Program
{
    static void Main()
    {
        // VectSharp: draw content via graphics API
        var doc = new Document();
        var page = new Page(595, 842); // A4 in points
        doc.Pages.Add(page);

        var g = page.Graphics;

        var font = new Font(FontFamily.ResolveFontFamily(FontFamily.StandardFontFamilies.Helvetica), 24);
        g.FillText(40, 40, "Hello, PDF!", font, Colours.Black);

        doc.SaveAsPDF("output.pdf");
        Console.WriteLine("Saved output.pdf");
    }
}

After:

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// HTML replaces drawing API — no coordinate calculation
var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body style='font-family:Arial; padding:40px'><h1>Hello, PDF!</h1></body></html>"
);
pdf.SaveAs("output.pdf");

Console.WriteLine($"Saved output.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

How Do You Migrate Report Generation from Drawing Calls to HTML?

This is the most common migration scenario. Teams have invoice or report code that manually positions every element with coordinates. Here is a complete before-and-after.

VectSharp Drawing API for a Report

using VectSharp;
using VectSharp.PDF;
using System;

class ReportBefore
{
    static void Main()
    {
        // VectSharp: explicit coordinate positioning for every element

        var doc = new Document();
        var page = new Page(595, 842); // A4 in points (72 DPI)
        doc.Pages.Add(page);
        var g = page.Graphics;

        // Title — manual coordinate (40, 40)
        var titleFont = new Font(
            FontFamily.ResolveFontFamily(FontFamily.StandardFontFamilies.HelveticaBold), 20);
        g.FillText(40, 40, "Q3 Invoice #4421", titleFont, Colours.Black);

        // Customer field — manual coordinate (40, 80)
        var bodyFont = new Font(
            FontFamily.ResolveFontFamily(FontFamily.StandardFontFamilies.Helvetica), 12);
        g.FillText(40, 80, "Customer: Acme Corp", bodyFont, Colours.Black);

        // Amount — manual coordinate (40, 110)
        g.FillText(40, 110, "Total: $8,400.00", bodyFont, Colours.Black);

        doc.SaveAsPDF("invoice.pdf");
        Console.WriteLine("Saved invoice.pdf — manual coordinates for every element");
    }
}

IronPDF HTML Template Equivalent

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

// HTML/CSS handles layout — no manual coordinate math
var html = @"
    <html>
    <head>
    <style>
        body { font-family: Arial, sans-serif; padding: 40px; color: #222; }
        h1 { font-size: 20px; font-weight: bold; }
        .field { margin-top: 8px; font-size: 13px; }
        .total { font-size: 16px; font-weight: bold; margin-top: 20px; }
    </style>
    </head>
    <body>
        <h1>Q3 Invoice #4421</h1>
        <div class='field'>Customer: Acme Corp</div>
        <div class='total'>Total: $8,400.00</div>
    </body>
    </html>";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("invoice.pdf");
Console.WriteLine($"Saved invoice.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

How Do You Handle PDF Merge, Watermark, and Password Protection?

These are the features VectSharp does not cover natively. Teams commonly bolt on a second library for them. With IronPDF, they are built in.

Merge PDFs — Before (VectSharp)

using VectSharp;
using VectSharp.PDF;
using System;

class MergeBefore
{
    static void Main()
    {
        // VectSharp builds PDFs from drawing operations — it does not parse or import
        // existing PDF files. To compose a multi-section document, build all sections
        // as Pages on a single Document, then call SaveAsPDF once.

        var merged = new Document();

        // Section A
        var pageA = new Page(595, 842);
        var gA = pageA.Graphics;
        // ... draw section A content ...
        merged.Pages.Add(pageA);

        // Section B
        var pageB = new Page(595, 842);
        var gB = pageB.Graphics;
        // ... draw section B content ...
        merged.Pages.Add(pageB);

        merged.SaveAsPDF("merged.pdf");
        Console.WriteLine("Note: merging existing PDF files requires a separate library.");
    }
}

Merge PDFs — After (IronPDF)

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();

var results = await Task.WhenAll(
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section A</h1></body></html>"),
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section B</h1></body></html>")
);

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("merged.pdf");
Console.WriteLine($"Merged: {merged.PageCount} pages");

Watermark — Before (VectSharp)

using VectSharp;
using VectSharp.PDF;
using System;

class WatermarkBefore
{
    static void Main()
    {
        // VectSharp creates PDFs from scratch — it cannot load and modify an existing
        // PDF. Watermarks are drawn directly onto each Page during construction.

        var doc = new Document();
        var page = new Page(595, 842);
        doc.Pages.Add(page);
        var g = page.Graphics;

        // Main content
        var bodyFont = new Font(
            FontFamily.ResolveFontFamily(FontFamily.StandardFontFamilies.Helvetica), 14);
        g.FillText(40, 40, "Document Content", bodyFont, Colours.Black);

        // Draw rotated semi-transparent text — manual coordinate transform required.
        // Colour.FromRgba accepts four doubles (0–1) for r, g, b, alpha.
        var watermarkFont = new Font(
            FontFamily.ResolveFontFamily(FontFamily.StandardFontFamilies.Helvetica), 60);
        g.Save();
        g.Translate(297, 421); // center of A4 page
        g.Rotate(-Math.PI / 4); // -45 degrees
        g.FillText(-100, 0, "DRAFT", watermarkFont, Colour.FromRgba(0.5, 0.5, 0.5, 0.15));
        g.Restore();

        doc.SaveAsPDF("watermarked.pdf");
        Console.WriteLine("Watermark drawn during page construction.");
    }
}

Watermark — After (IronPDF)

using IronPdf;
using IronPdf.Editing;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(
    "<html><body><h1>Document Content</h1></body></html>"
);

// https://ironpdf.com/how-to/custom-watermark/
var watermark = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronSoftware.Drawing.Color.Gray,
    Opacity = 15,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center,
};

pdf.ApplyStamp(watermark);
pdf.SaveAs("watermarked.pdf");
Console.WriteLine("Watermark applied");

What Are the Most Common Troubleshooting Issues During Migration?

Here are the issues I see teams run into most frequently when moving from VectSharp to IronPDF.

Drawing API Has No HTML Equivalent for Complex Diagrams

If you have a VectSharp drawing that generates a precise chart or diagram with exact coordinate positioning, bezier curves, and custom paths, there is no direct HTML/CSS equivalent. The bridge is VectSharp's SVG output.

Keep VectSharp for diagrams, export to SVG, and embed that SVG in HTML for IronPDF to render. Here is how that looks:

// Render diagram as SVG with VectSharp
using VectSharp;
using VectSharp.SVG;
using System;

var doc = new Document();
var page = new Page(400, 300);
doc.Pages.Add(page);
var g = page.Graphics;
// ... draw your diagram ...
page.SaveAsSVG("diagram.svg"); // extension method from VectSharp.SVG

// Embed SVG in HTML for IronPDF
using IronPdf;
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var svgContent = System.IO.File.ReadAllText("diagram.svg");
var html = $@"
    <html><body style='padding:40px'>
        <h1>Report with Embedded Diagram</h1>
        {svgContent}
        <p>Diagram analysis...</p>
    </body></html>";

var renderer = new ChromePdfRenderer();
var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("report-with-diagram.pdf");

Font Mismatch After Migration

If a report was using VectSharp's FontFamily.StandardFontFamilies.Helvetica and after migration the font looks different, this is expected. Helvetica is a licensed font. VectSharp uses Type 1 standard fonts. IronPDF renders via Chromium using system fonts or web fonts, and Helvetica may not be available.

Specify the font explicitly in your HTML to get consistent results:

var html = @"
    <html>
    <head>
    <style>
        /* Use Arial as the near-equivalent to Helvetica on most systems */
        body { font-family: Arial, 'Helvetica Neue', Helvetica, sans-serif; }
        /* Or use Google Fonts for exact web font */
        /* @import url('https://fonts.googleapis.com/css2?family=Inter'); */
    </style>
    </head>
    <body><h1>Report Title</h1></body>
    </html>";

Page Size Units Differ Between Libraries

VectSharp pages are specified in points (new Page(595, 842) for A4). IronPDF rendering options use an enum or millimeters. Instead of specifying raw points, use the enum:

using IronPdf.Rendering;

// Instead of specifying raw points, use the enum:
renderer.RenderingOptions.PaperSize = PdfPaperSize.A4;
// Or Letter, A3, A5, etc.

// For custom sizes, use SetCustomPaperSizeinPixelsOrPoints(width, height)
// See: https://ironpdf.com/how-to/rendering-options/

Stream Output After SaveAsPDF Is Gone

If VectSharp's document.SaveAsPDF(stream) was used to write to a MemoryStream for API responses, here is the IronPDF equivalent:

using IronPdf;
using System.IO;

var renderer = new ChromePdfRenderer();
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);

// Option 1: get bytes directly
var bytes = pdf.BinaryData;
return File(bytes, "application/pdf", "output.pdf");

// Option 2: copy stream
using var ms = new MemoryStream();
pdf.Stream.CopyTo(ms);
return ms.ToArray();
// pdf disposed at end of 'using' block

Frequently Asked Questions

Is this migration a simple library swap?

No. The migration from VectSharp to IronPDF is a model change, not a drop-in replacement. VectSharp's model is to describe what to draw and where to draw it with explicit coordinates. IronPDF's model is to provide HTML and get back a PDF.

If your existing codebase builds content via VectSharp drawing calls, those drawing calls become HTML and CSS templates. If it already builds content via HTML templates, IronPDF is the natural fit and the migration is more direct.

Should I keep VectSharp for diagrams and add IronPDF for reports?

Yes, this is a legitimate architecture. For precise data-driven diagrams, VectSharp's vector drawing API can be more appropriate than trying to achieve equivalent precision with SVG and JavaScript chart libraries. The bridge is SVG export from VectSharp embedded in HTML rendered by IronPDF.

Consider keeping VectSharp for the diagram component and adding IronPDF for HTML report generation. Both libraries can coexist in the same project.

Does page indexing change between the two libraries?

No. Both VectSharp and IronPDF use 0-based page indexing. This typically does not require changes for page manipulation code.

Can IronPDF handle parallel rendering for batch reports?

Yes. You can render multiple PDFs concurrently using Task.WhenAll. Each renderer instance handles its own Chromium process. Here is the pattern:

using IronPdf;
using System.Linq;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdfs = await Task.WhenAll(reportDataList.Select(async data =>
{
    var renderer = new ChromePdfRenderer();
    return await renderer.RenderHtmlAsPdfAsync(BuildHtml(data));
}));

foreach (var pdf in pdfs) pdf.Dispose();

What about disposal patterns — does IronPDF require explicit cleanup?

VectSharp's Document uses managed objects with no explicit disposal. IronPDF's PdfDocument should be wrapped in a using block. This is a small but important change:

using IronPdf;

var renderer = new ChromePdfRenderer();
using var pdf = await renderer.RenderHtmlAsPdfAsync(html);
pdf.SaveAs("output.pdf");
// pdf disposed automatically at end of scope

Conclusion

The drawing API to HTML transition is the core challenge in this migration. For report-style documents — tabular data, text, simple formatting — HTML is generally simpler to write and maintain than coordinate-based drawing code. For precision diagrams, VectSharp's approach is often more appropriate, and the SVG bridge keeps both tools working in the same pipeline.

The missing features like merge, watermark, and security migrate with minimal effort since they are additive — new API calls on existing rendered output.

If you are starting from report-style code that manually positions every text element with coordinates, the move to HTML templates will feel like a significant improvement. If you are working with precise vector diagrams, keep VectSharp for that and let IronPDF handle everything else.
