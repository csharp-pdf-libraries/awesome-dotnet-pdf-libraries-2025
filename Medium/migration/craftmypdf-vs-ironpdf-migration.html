Migrating from CraftMyPDF to IronPDF in C#: The Complete Guide

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should I Migrate from CraftMyPDF to IronPDF?

Migrating off a cloud PDF service is architecturally different from migrating between two .NET libraries. You're not swapping a NuGet package — you're moving from an HTTP dependency to a local process, which changes your latency profile, cost model, data residency posture, and offline availability.

CraftMyPDF solves a real problem — PDF generation without running PDF infrastructure. The friction appears when requirements change. Here are the scenarios where I've seen teams make the move:

Data residency and compliance. PDF content (customer PII, financial data) leaves your network with every API call. Some compliance frameworks prohibit or complicate this.

Network latency. Each PDF generation round-trips to CraftMyPDF's servers. For low-latency or high-volume scenarios, this adds measurable delay.

API cost at scale. Cloud services price per operation. Above a threshold, local rendering is cheaper. Calculate your current API call volume times the rate.

Offline and air-gapped environments. CraftMyPDF requires internet access. Air-gapped servers, offline modes, or edge deployments can't use cloud APIs.

Vendor dependency risk. API changes, service outages, or pricing changes from a third-party affect your application.

Template flexibility. CraftMyPDF templates are managed in their UI. Complex dynamic logic (conditional sections, loops, computed fields) requires their template language rather than full C# control flow.

CI/CD test environment. Integration tests that hit a live cloud API are slower and fragile. Local rendering makes tests hermetic.

Response time SLA. If PDF generation is in a synchronous request path, cloud round-trip adds to your p99 latency.

Custom fonts and assets. Managing font assets in a cloud template system can be less flexible than local filesystem access.

On-premises deployment requirements. Some enterprise environments prohibit cloud service dependencies in production workloads.

How Do CraftMyPDF and IronPDF Compare?

The architectural difference is the entire migration. CraftMyPDF is a cloud API — you send data over HTTP (typically with RestSharp or HttpClient) and receive a PDF back. IronPDF is a local .NET library that renders PDFs on your own machine using a Chromium engine.

  • Focus. CraftMyPDF is cloud template PDF generation; IronPDF is local HTML-to-PDF, edit, merge, and security.
  • Pricing. CraftMyPDF is per-API-call or subscription; IronPDF is per-developer or royalty-free.
  • API style. CraftMyPDF is REST/HTTP with JSON template data; IronPDF is .NET method calls.
  • Learning curve. CraftMyPDF is simple for template-driven work but limited for complex logic; IronPDF is gradual and your existing HTML/CSS knowledge transfers directly.
  • HTML rendering. CraftMyPDF runs a hosted cloud renderer behind their REST API; IronPDF is Chromium-based and runs in your process.
  • Page indexing. Not applicable for CraftMyPDF cloud output; IronPDF is 0-based.
  • Thread safety. CraftMyPDF is stateless HTTP; IronPDF's renderer is reusable — consult IronPDF's current thread-safety guidance before sharing a single instance across threads.
  • Namespace. CraftMyPDF has no local SDK (you use RestSharp or HttpClient); IronPDF lives under IronPdf.

How Much Effort Does Each Feature Require to Migrate?

Not every migration task is equal. Template-driven PDFs with complex conditional logic will take the most effort since you're rebuilding that logic in C# or Razor. Simple HTML-to-PDF conversions are straightforward — you're just replacing an HTTP call with a local renderer call.

  • Simple HTML-to-PDF — Low. Replace the HTTP call with a local renderer.
  • Template-driven PDF (data injection) — Medium. Replace the JSON template payload with HTML plus C# string interpolation or Razor.
  • URL-to-PDF — Low. One method call replaces the cloud round-trip and any second download request.
  • Headers and footers — Low. CraftMyPDF passes header/footer HTML inside the template data; IronPDF exposes RenderingOptions.TextHeader/TextFooter or the HTML equivalents.
  • Merge PDFs — Medium. CraftMyPDF exposes a merge endpoint that requires uploading files first; locally it becomes a single PdfDocument.Merge call.
  • Watermark — Medium. Watermarks move from the template editor (or a dedicated endpoint) to programmatic stamping.
  • Password protection — Low. A new local capability with a simple property-based API.
  • Complex conditional templates — High. Rebuild the logic in C# or Razor.
  • Batch generation at scale — Medium. Test local throughput against your API throughput.
  • Test environment hermeticity — Low. Tests no longer need network access.
  • Error handling model — Medium. HTTP status codes become .NET exceptions.

I also recommend thinking about this from a business-scenario perspective.

  • If data residency is a compliance requirement, local rendering is strongly worth evaluating.
  • If API cost is significant and growing, calculate the break-even point — local is often cheaper at scale.
  • If you need offline or air-gapped deployment, the cloud API is not viable and local rendering is required.
  • If templates are managed by non-developers, CraftMyPDF's UI is a genuine advantage; evaluate the migration cost honestly.

How Do I Set Up IronPDF and Replace the CraftMyPDF API Key?

The first step is swapping the authentication model. CraftMyPDF uses an API key sent in HTTP headers. IronPDF uses a local license key set once at application startup.

CraftMyPDF API Key Pattern (Before)

// No local license — auth via API key in HTTP headers
using RestSharp;

var client = new RestClient("https://api.craftmypdf.com/v1/create");
var request = new RestRequest(Method.POST);
request.AddHeader("X-API-KEY", "YOUR-LICENSE-KEY");

IronPDF License Configuration (After)

using IronPdf;

// Set once at startup — https://ironpdf.com/how-to/license-keys/
IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

Install IronPDF with dotnet add package IronPdf, then run dotnet restore. Store the license key in your environment variables or appsettings.json — never hardcode it in source.

How Do I Convert an API Call to a Local Render?

This is the core migration step. You're replacing an HTTP round-trip to CraftMyPDF's servers with a local Chromium-based render. The data that used to go into a JSON payload now goes into an HTML template method.

CraftMyPDF API Call (Before)

// NuGet: Install-Package RestSharp
using System;
using RestSharp;
using System.IO;

class Program
{
    static void Main()
    {
        var client = new RestClient("https://api.craftmypdf.com/v1/create");
        var request = new RestRequest(Method.POST);
        request.AddHeader("X-API-KEY", "YOUR-LICENSE-KEY");

        // Template ID and data sent to CraftMyPDF cloud
        request.AddJsonBody(new
        {
            template_id = "your-template-id",
            data = new
            {
                customer_name = "Alice Smith",
                invoice_number = "INV-2071",
                amount = 1200.00,
                due_date = "2025-12-01"
            }
        });

        // Round-trip to CraftMyPDF servers
        var response = client.Execute(request);
        File.WriteAllBytes("invoice.pdf", response.RawBytes);
        Console.WriteLine("Invoice downloaded from CraftMyPDF");
    }
}

IronPDF Local Render (After)

using System;
using IronPdf;

class Program
{
    // Reuse renderer — don't create per-request
    private static readonly ChromePdfRenderer _renderer = new ChromePdfRenderer();

    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        // No network call — renders locally with Chromium
        // https://ironpdf.com/how-to/html-string-to-pdf/
        string html = BuildInvoiceHtml("Alice Smith", "INV-2071", 1200.00m, "2025-12-01");
        using var pdf = _renderer.RenderHtmlAsPdf(html);
        pdf.SaveAs("invoice.pdf");
        Console.WriteLine("Invoice generated locally");
    }

    static string BuildInvoiceHtml(string name, string invoiceNum, decimal amount, string due) =>
        $@"<!DOCTYPE html><html><body style='font-family:Arial;padding:40px'>
            <h1>Invoice {System.Net.WebUtility.HtmlEncode(invoiceNum)}</h1>
            <p>Customer: {System.Net.WebUtility.HtmlEncode(name)}</p>
            <p>Amount: ${amount:F2}</p>
            <p>Due: {due}</p>
        </body></html>";
}

How Do I Convert CraftMyPDF Templates to HTML?

The template conversion is the most time-consuming part of this migration. Don't try to migrate all templates at once. Work template-by-template using this pattern:

1. Screenshot the CraftMyPDF template output.
2. Write an HTML equivalent.
3. Render with IronPDF.
4. Visual diff against the screenshot.
5. Adjust CSS until output matches.
6. Move to next template.

Template Conversion Pattern

// Template method replacing CraftMyPDF template ID
static string GetInvoiceHtml(string name, decimal amount, string due) => $@"
<!DOCTYPE html>
<html>
<head>
    <style>
        body {{ font-family: Arial, sans-serif; padding: 40px; }}
        h1 {{ color: #2563EB; }}
        .amount {{ font-size: 24px; font-weight: bold; }}
    </style>
</head>
<body>
    <h1>Invoice for {System.Net.WebUtility.HtmlEncode(name)}</h1>
    <p class='amount'>Amount: ${amount:F2}</p>
    <p>Due: {due}</p>
</body>
</html>";

For Razor-based applications, use partial views instead of string interpolation. This gives you the same C# control flow you'd use anywhere else in your app.

How Do I Merge PDFs Without the Cloud API?

CraftMyPDF exposes a merge endpoint, but it expects the source PDFs to be reachable as URLs — meaning you upload first, then merge. With IronPDF's merge API, this becomes a single local operation against files already on disk or in memory.

CraftMyPDF Merge Pattern (Before)

// NuGet: Install-Package RestSharp
using System;
using RestSharp;
using System.IO;

class Program
{
    static void Main()
    {
        var client = new RestClient("https://api.craftmypdf.com/v1/merge");
        var request = new RestRequest(Method.POST);
        request.AddHeader("X-API-KEY", "YOUR-LICENSE-KEY");

        // Files must already be reachable as URLs — upload first
        request.AddJsonBody(new
        {
            urls = new[]
            {
                "https://craftmypdf.com/uploads/file1.pdf",
                "https://craftmypdf.com/uploads/file2.pdf"
            }
        });

        var response = client.Execute(request);
        // Download merged PDF from response URL...
        Console.WriteLine("Merged via CraftMyPDF cloud");
    }
}

IronPDF Merge (After)

using System;
using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        // https://ironpdf.com/how-to/merge-or-split-pdfs/
        var pdf1 = PdfDocument.FromFile("section1.pdf");
        var pdf2 = PdfDocument.FromFile("section2.pdf");

        using var merged = PdfDocument.Merge(pdf1, pdf2);
        merged.SaveAs("merged.pdf");
        Console.WriteLine($"Merged: {merged.PageCount} pages");
    }
}

How Do I Add Watermarks Without the Template Editor?

CraftMyPDF exposes a dedicated /v1/add-watermark endpoint, but it operates on a PDF that's already hosted in their storage — so you typically upload, then call the watermark endpoint, then download. Watermarks can also be configured directly in the template editor UI for static cases. With IronPDF, you stamp watermarks programmatically with full control over every property.

CraftMyPDF Watermark (Before)

// NuGet: Install-Package RestSharp
using RestSharp;

var client = new RestClient("https://api.craftmypdf.com/v1/add-watermark");
var request = new RestRequest(Method.POST);
request.AddHeader("X-API-KEY", "YOUR-LICENSE-KEY");
request.AddJsonBody(new
{
    url = "https://storage.craftmypdf.com/your-pdf.pdf",
    watermark_text = "CONFIDENTIAL",
    opacity = 0.3,
    rotation = 45
});

var response = client.Execute(request);
// Download the watermarked PDF from the response URL...

IronPDF Watermark (After)

using System;
using IronPdf;
using IronPdf.Editing;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var pdf = PdfDocument.FromFile("input.pdf");

        // Full control over watermark properties — https://ironpdf.com/how-to/stamp-text-image/
        var stamper = new TextStamper
        {
            Text = "DRAFT",
            FontSize = 72,
            Opacity = 30,
            Rotation = 45,
            VerticalAlignment = VerticalAlignment.Middle,
            HorizontalAlignment = HorizontalAlignment.Center
        };

        pdf.ApplyStamp(stamper);
        pdf.SaveAs("watermarked.pdf");
        Console.WriteLine("Watermarked PDF saved");
    }
}

How Do I Add Password Protection?

CraftMyPDF's documented endpoints (/v1/create, /v1/merge, /v1/add-watermark) don't expose client-specified password encryption. Most teams I've seen using CraftMyPDF were either skipping encryption or applying it post-download with a separate library. IronPDF lets you do it in the same step as generation.

CraftMyPDF Password Protection (Before)

// CraftMyPDF's public endpoints (/v1/create, /v1/merge, /v1/add-watermark)
// don't expose client-specified password encryption.
// If you currently encrypt post-download with another library, that
// separate library code is your true "before" state. The CraftMyPDF leg
// of the work is just downloading the PDF:

// var pdfBytes = await DownloadFromCraftMyPdf(...);
// [existing post-download encryption code here]

IronPDF Password Protection (After)

using System;
using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        // Generate or load the PDF
        var renderer = new ChromePdfRenderer();
        using var pdf = renderer.RenderHtmlAsPdf("<html><body><p>Confidential</p></body></html>");

        // Encrypt in the same step — https://ironpdf.com/how-to/pdf-permissions-passwords/
        pdf.SecuritySettings.UserPassword = "user123";
        pdf.SecuritySettings.OwnerPassword = "owner456";
        pdf.SecuritySettings.AllowUserPrinting =
            IronPdf.Security.PdfPrintSecurity.FullPrintRights;

        pdf.SaveAs("protected.pdf");
        Console.WriteLine("Encrypted PDF saved");
    }
}

How Do I Handle the Error Model Change?

This is a subtle but important shift. Cloud APIs fail with HTTP status codes and JSON error bodies. Local library calls throw .NET exceptions. You need to update your error handling pattern throughout the codebase.

Error Handling Migration

// Before — HTTP error handling with RestSharp
var response = client.Execute(request);
if (!response.IsSuccessful)
{
    throw new Exception(
        $"CraftMyPDF API error {(int)response.StatusCode}: {response.Content}");
}

// After — exception handling
try
{
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs("output.pdf");
}
catch (Exception ex)
{
    _logger.LogError(ex, "PDF generation failed");
    throw;
}

One more thing to watch: CraftMyPDF calls are HTTP — they fit naturally into async code paths. IronPDF rendering is CPU-bound. The first render after creating a new ChromePdfRenderer has startup overhead. Subsequent renders are faster. Use async rendering to avoid blocking threads:

using var pdf = await renderer.RenderHtmlAsPdfAsync(html);

What Does the Full API Mapping Look Like?

Here are the complete operation mappings to reference during your migration.

Transport and auth layer:

  • RestClient/HttpClient + X-API-KEY header → IronPdf.License.LicenseKey — no HTTP client needed.
  • application/json POST body → method parameters — data becomes HTML template inputs.
  • Download URL from response → pdf.BinaryData or pdf.SaveAs() — local output.

Endpoint-to-method mapping:

  • POST /v1/create with template_id → renderer.RenderHtmlAsPdf(html) — generate PDF.
  • Template UI design → HTML + CSS string or Razor view — template format.
  • JSON data field injection → C# string interpolation or Razor model — data binding.
  • Download from CDN URL → pdf.BinaryData or pdf.Stream — PDF retrieval.

Operation-by-operation:

  • Generate from template. CraftMyPDF: API call with template_id + data. IronPDF: renderer.RenderHtmlAsPdf(html).
  • Merge documents. CraftMyPDF: POST /v1/merge with hosted URLs. IronPDF: PdfDocument.Merge(pdf1, pdf2).
  • Add watermark. CraftMyPDF: POST /v1/add-watermark or template editor. IronPDF: pdf.ApplyStamp(stamper).
  • Password protect. CraftMyPDF: not exposed by the public endpoints. IronPDF: pdf.SecuritySettings.*.

How Do I Handle Font Migration?

Fonts uploaded to CraftMyPDF need to be available locally when you switch to IronPDF. You have two main options.

First, reference local font files in your HTML templates:

<!-- Reference local font in HTML template -->
<style>
    @font-face {
        font-family: 'CustomFont';
        src: url('/fonts/custom-font.woff2') format('woff2');
    }
    body { font-family: 'CustomFont', Arial, sans-serif; }
</style>

Second, use system fonts or Google Fonts. Since IronPDF uses Chromium, any font that works in Chrome will work in your PDFs.

Frequently Asked Questions

Is this migration replacing a NuGet package swap?

No. This is a fundamentally different kind of migration. You're moving from an HTTP dependency to a local process. CraftMyPDF doesn't have a local .NET SDK — it's a REST API. That means you're replacing RestSharp or HttpClient calls with IronPDF method calls, and converting JSON template data into HTML templates.

The execution model changes completely: no more network round-trips, no more CDN downloads, no more API key authentication. Budget time accordingly — this is more than a find-and-replace.

How do I handle templates managed by non-developers?

This is CraftMyPDF's genuine advantage — their visual template editor lets non-technical users design PDF layouts. When you move to IronPDF, templates become HTML/CSS files or Razor views. If non-developers currently manage templates, you'll need to either train them on basic HTML or create an internal template management workflow.

For Razor-based applications, you can use partial views that separate layout from data, which makes it easier for designers to work on templates without touching C# logic.

What about latency differences?

CraftMyPDF adds network round-trip time to every PDF generation. IronPDF renders locally, so latency is CPU-bound instead of network-bound. The first render after creating a new ChromePdfRenderer has startup overhead as Chromium initializes. Subsequent renders are faster.

For synchronous request paths, removing the cloud round-trip typically shows up clearly in p99 latency improvements. Use renderer.RenderHtmlAsPdfAsync(html) to avoid blocking threads during CPU-bound rendering.

Can I run IronPDF in Docker or Linux containers?

Yes. Install the native dependencies in your Dockerfile:

RUN apt-get update && apt-get install -y \
    libgdiplus libx11-6 libc6 libgtk-3-0 libxcomposite1 libxdamage1 \
    libxrandr2 libgbm1 libxkbcommon0 libasound2

Test PDF generation in your target container before writing the full migration. This ensures the Chromium renderer works correctly in your specific Linux environment.

How long should I budget for the template conversion?

Budget roughly one working day per complex template for visual regression testing. Simple templates with straightforward data injection can be converted in an hour or two. Templates with complex conditional logic, loops, and computed values take longer because you're rebuilding that logic in C# or Razor.

Work template-by-template: screenshot the CraftMyPDF output, write the HTML equivalent, render with IronPDF, visual diff, and adjust CSS until it matches.

What happens to my CraftMyPDF API costs after migration?

They go to zero. IronPDF uses a per-developer or royalty-free license model instead of per-API-call pricing. For high-volume scenarios, calculate the break-even point: IronPDF license cost plus infrastructure (CPU and memory for the Chromium renderer) versus your current monthly CraftMyPDF API spend.

Above a certain call volume threshold, local rendering is almost always cheaper. Export your current monthly API call volume from the CraftMyPDF dashboard to run this calculation.

What Should I Remember After Migrating?

Moving from a cloud PDF API to a local library is one of the less common migration patterns in .NET, but it comes up whenever data residency, offline requirements, or cost-at-scale push teams to bring generation in-house. The template conversion is the most time-consuming step — budget roughly one working day per complex template for visual regression testing.

The payoff is removing a network dependency from a synchronous code path, which tends to show up clearly in p99 latency numbers once it's gone.

After the migration is complete, clean up thoroughly: remove the CraftMyPDF API key from all configuration files and secrets stores, remove it from CI/CD environment variables, and delete or archive CraftMyPDF templates (download PDFs of each as reference first). Update your runbooks to reflect that PDF generation is now a local dependency rather than a cloud API dependency.
