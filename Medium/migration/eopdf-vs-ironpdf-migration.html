Migrating from EO.Pdf to IronPDF in C#: The Complete Guide

The error message that sends most teams down the EO.Pdf migration path looks something like this: NullReferenceException somewhere inside HtmlToPdf.ConvertHtml, triggered by a perfectly valid HTML file that rendered fine last deploy. EO.Pdf has been in maintenance territory for years, and as .NET versions advance, the surface area for subtle compatibility issues grows.

I've watched this pattern play out across dozens of teams. In this guide, I'll walk you through the full migration from EO.Pdf to IronPDF — structured around the problems teams actually encounter, with complete before/after code examples and every API mapping you need.

Why Should I Migrate Away from EO.Pdf?

Migrations off EO.Pdf tend to happen when one of these pressure points tips over. Here are the ten most common reasons I see teams make the switch:

Static, global rendering options. EO.Pdf configures rendering through HtmlToPdf.Options, a static singleton. Two requests on different threads stomp on each other's PageSize, BaseUrl, and event handlers. This is the single most common reason teams migrate.

Runtime exceptions on newer .NET. EO.Pdf was built for .NET Framework. Running it on net6+ via compatibility shims can produce hard-to-diagnose NREs and SEHExceptions.

Platform portability. Despite cross-platform marketing, EO.Pdf's Linux story is thin. Containers frequently fail without GDI+ shims and a hand-curated set of native dependencies.

Heavy package footprint. EO.Pdf bundles its own browser engine, producing roughly a 126MB deployment footprint that bloats Docker layers and CI artifacts.

Legacy IE-era rendering baggage. EO.Pdf migrated from an IE-based engine to Chromium over its lifetime, and the API still carries artifacts from that transition — inconsistent option naming and version-specific behavior gaps.

ACM (Advanced Content Model) lock-in. Teams that built reports with AcmRender, AcmText, and AcmBlock are running an imperative coordinate-positioned layout engine that nobody outside EO.Pdf understands. HTML/CSS is a more portable target.

Limited async support. Synchronous-only rendering becomes a bottleneck in async ASP.NET controller pipelines.

Debug experience. When EO.Pdf fails, the exception stack is often inside the library internals with no actionable message. Troubleshooting involves trial and error.

Memory growth in long-running processes. Teams running EO.Pdf in Windows services or worker roles sometimes report unbounded memory growth.

Per-license pricing. EO.Pdf lists at $799 per developer license — competitive in 2010, less so now that several alternatives bundle similar feature sets.

How Do the Two Libraries Compare Side by Side?

Before diving into code, let me lay out the architectural differences. The headline contrast is configuration model: EO.Pdf is a static, global, converter-pattern API; IronPDF is an instance-based renderer with per-instance options. That single difference changes how you structure services, how you handle concurrency, and how you reason about thread safety.

EO.Pdf's HTML engine started life pre-Chromium and migrated forward over the years; IronPDF runs Chromium directly, so modern CSS (Grid, Flexbox, custom properties) and JavaScript-driven content render the way Chrome renders them. Both libraries are commercial, both use 0-based page indexing, and both ship a PdfDocument class — which causes a name collision during phased migrations. The namespaces are EO.Pdf and IronPdf.

How Complex Is the Migration Effort?

Not all features take the same amount of work. Here is the effort breakdown by feature area, so you can plan your migration sprint accurately.

  • HTML string to PDF — Low. Static HtmlToPdf.ConvertHtml becomes an instance ChromePdfRenderer.RenderHtmlAsPdf. Mostly mechanical.
  • URL to PDF — Low. EO.Pdf has HtmlToPdf.ConvertUrl; IronPDF has RenderUrlAsPdf. Direct swap, with the bonus that IronPDF executes JavaScript on the page first.
  • HTML file to PDF — Low. Both accept a path; the IronPDF method is RenderHtmlFileAsPdf.
  • Static options to instance options — Medium. Every HtmlToPdf.Options.X = … line needs to move onto a renderer instance. The mechanical part is fast; the design part is auditing where you assumed global config was safe.
  • Merge documents — Low. EO.Pdf's Append in a loop becomes IronPDF's static PdfDocument.Merge.
  • Watermark — Medium. EO.Pdf typically uses ACM-positioned text overlays; IronPDF has a dedicated ApplyWatermark / stamper API.
  • Headers and footers — Medium. EO.Pdf wires per-page rendering through the AfterRenderPage event; IronPDF uses HtmlHeaderFooter templates with {page} / {total-pages} placeholders. The translation is conceptually clean but every callback has to be rewritten as HTML.
  • Password protection — Low. Both libraries expose user/owner passwords and permissions; the property names differ but the concepts map directly.
  • Page indexing — Low. Both are 0-based; no off-by-one.
  • ACM (Advanced Content Model) — High. If your codebase leans on AcmRender, AcmText, AcmBlock, AcmTable, plan to rebuild those sections in HTML/CSS. The good news is HTML/CSS is far more expressive than ACM — flexbox alone replaces hundreds of lines of coordinate math.
  • JavaScript rendering — N/A to Low. New capability in IronPDF; no migration cost, only a template-QA pass.
  • CSS Grid / Flexbox — N/A to Low. New capability — may require template QA where EO.Pdf was silently dropping unsupported CSS.

And here is my decision matrix for whether migrating makes sense for your scenario:

  • If you're stuck on .NET Framework with no migration plans, EO.Pdf may continue working — verify support timelines before committing further investment.
  • If you're targeting net6+ or Linux containers, the migration is a strong recommendation. EO.Pdf's cross-platform story is fragile.
  • If complex CSS templates are not rendering correctly, migrate. Chromium fidelity resolves most CSS issues without template changes.
  • If your codebase leans heavily on ACM, schedule the migration deliberately — the ACM-to-HTML rewrite is the long pole.
  • If you have minimal HTML, simple output, and no rendering issues, urgency is lower — migrate when convenient.

How Do I Handle the Quick Start Migration?

The basic migration is three steps. Let me walk through each one.

Step 1: License Setup

// Before (EO.Pdf)
// EO.Pdf typically requires the runtime DLL on disk; no inline license call
// for the trial. Production licensing is per the EO.Pdf vendor docs.

// After (IronPDF)
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
// Docs: https://ironpdf.com/how-to/license-keys/

Step 2: Namespace Imports

// Before
using EO.Pdf;

// After
using IronPdf;

Step 3: Basic HTML to PDF

// Before (EO.Pdf)
HtmlToPdf.ConvertHtml("<h1>Hello</h1>", "output.pdf");

// After (IronPDF)
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
var renderer = new ChromePdfRenderer();
using var pdf = renderer.RenderHtmlAsPdf("<h1>Hello</h1>");
pdf.SaveAs("output.pdf");
// Docs: https://ironpdf.com/how-to/html-string-to-pdf/

What Are the Core API Mappings?

These mappings are the reference you'll keep coming back to during the migration. Below are every major EO.Pdf class and operation paired with its IronPDF equivalent.

Namespace Mapping

  • EO.Pdf → IronPdf — core namespace
  • EO.Pdf.Acm → HTML/CSS — IronPDF has no ACM equivalent; rebuild as HTML
  • EO.Pdf.Contents → IronPdf.Editing — low-level content operations
  • EO.Pdf.Drawing → HTML/CSS or IronPdf.Editing — graphics primitives

Core Class Mapping

  • HtmlToPdf (static) → ChromePdfRenderer (instance) — main HTML rendering class
  • EO.Pdf.PdfDocument → IronPdf.PdfDocument — document object; name collision, use fully qualified names during phased migration
  • HtmlToPdfOptions → ChromePdfRenderOptions (via renderer.RenderingOptions) — rendering configuration
  • AcmRender, AcmText, AcmBlock, AcmImage, AcmTable → HTML/CSS — no direct IronPDF class, rewrite as markup
  • PdfDocument.Security → PdfDocument.SecuritySettings — passwords and permissions

Document Loading

  • Load from file: new PdfDocument(path) → PdfDocument.FromFile(path)
  • Load from stream: new PdfDocument(stream) → PdfDocument.FromStream(stream)
  • Load from bytes: not directly supported in EO.Pdf → PdfDocument.FromBinaryData(bytes)
  • Render HTML string: HtmlToPdf.ConvertHtml(html, path) → renderer.RenderHtmlAsPdf(html) + SaveAs(path)
  • Render URL: HtmlToPdf.ConvertUrl(url, path) → renderer.RenderUrlAsPdf(url) + SaveAs(path)

Page Operations

  • Page count: doc.Pages.Count → pdf.PageCount
  • Get specific page: doc.Pages[i] → pdf.Pages[i]
  • Remove page: doc.Pages.RemoveAt(i) → pdf.RemovePages(i)
  • Copy pages: doc.Pages iteration → pdf.CopyPages(start, end)

Merge / Split

  • Merge PDFs: doc1.Append(doc2) in a loop → PdfDocument.Merge(new List<PdfDocument>{ a, b })
  • Split by page range: manual page iteration → pdf.CopyPages(start, end)

How Do I Migrate HTML to PDF Rendering?

This is the most common migration path. Here is the full before/after for a typical report generation scenario.

EO.Pdf (Before)

using EO.Pdf;

class Program
{
    static void Main()
    {
        string html = @"
            <html>
            <body>
                <h1 style='color: #333;'>Monthly Report</h1>
                <p>Generated: <strong>2025-01</strong></p>
                <table border='1'>
                    <tr><th>Item</th><th>Value</th></tr>
                    <tr><td>Revenue</td><td>$120,000</td></tr>
                </table>
            </body>
            </html>";

        // Static, global options — affects every thread in this process
        HtmlToPdf.Options.PageSize = PdfPageSizes.A4;
        HtmlToPdf.Options.OutputArea = new RectangleF(0.5f, 0.5f, 7.5f, 10.5f);

        HtmlToPdf.ConvertHtml(html, "report.pdf");
    }
}

IronPDF (After)

using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        string html = @"
            <html>
            <body>
                <h1 style='color: #333;'>Monthly Report</h1>
                <p>Generated: <strong>2025-01</strong></p>
                <table border='1'>
                    <tr><th>Item</th><th>Value</th></tr>
                    <tr><td>Revenue</td><td>$120,000</td></tr>
                </table>
            </body>
            </html>";

        // Instance-scoped options — safe across threads and requests
        var renderer = new ChromePdfRenderer();
        renderer.RenderingOptions.PaperSize = PdfPaperSize.A4;
        renderer.RenderingOptions.MarginTop = 12.7;    // 0.5 inch in mm
        renderer.RenderingOptions.MarginBottom = 12.7;
        renderer.RenderingOptions.MarginLeft = 12.7;
        renderer.RenderingOptions.MarginRight = 12.7;

        using var pdf = renderer.RenderHtmlAsPdf(html);
        pdf.SaveAs("report.pdf");
    }
}

Note the unit change: EO.Pdf's OutputArea uses inches; IronPDF's margin properties use millimeters. Multiply inches by 25.4 to convert.

How Do I Migrate PDF Merging?

Merging is a common operation, and the API shapes differ in two ways: EO.Pdf uses an instance Append method on a destination document, while IronPDF uses a static Merge that takes a list.

EO.Pdf (Before)

using EO.Pdf;

class MergeSample
{
    static void Main()
    {
        var doc1 = new PdfDocument("file1.pdf");
        var doc2 = new PdfDocument("file2.pdf");

        var merged = new PdfDocument();
        merged.Append(doc1);
        merged.Append(doc2);

        merged.Save("merged.pdf");
    }
}

IronPDF (After)

using IronPdf;
using System.Collections.Generic;

class MergeSample
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var pdf1 = PdfDocument.FromFile("file1.pdf");
        using var pdf2 = PdfDocument.FromFile("file2.pdf");

        using var merged = PdfDocument.Merge(new List<PdfDocument> { pdf1, pdf2 });
        merged.SaveAs("merged.pdf");
        // Docs: https://ironpdf.com/how-to/merge-or-split-pdfs/
    }
}

How Do I Migrate Watermarks?

EO.Pdf typically handles watermarks through ACM — an AcmText rendered onto each page via AcmRender with explicit coordinates and styling. IronPDF has a dedicated stamp / watermark API that takes HTML and applies it across pages.

EO.Pdf (Before)

using EO.Pdf;
using EO.Pdf.Acm;
using System.Drawing;

class WatermarkSample
{
    static void Main()
    {
        var doc = new PdfDocument("input.pdf");

        foreach (PdfPage page in doc.Pages)
        {
            var watermark = new AcmText("DRAFT");
            watermark.Style.FontSize = 72;
            watermark.Style.ForegroundColor = Color.FromArgb(80, 200, 200, 200);
            watermark.Style.HorizontalAlign = AcmHorizontalAlign.Center;

            var render = new AcmRender(page, 0, page.Size.Height / 2 - 50,
                page.Size.Width, 100);
            render.Render(watermark);
        }

        doc.Save("watermarked.pdf");
    }
}

IronPDF (After)

using IronPdf;

class WatermarkSample
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var pdf = PdfDocument.FromFile("input.pdf");

        pdf.ApplyWatermark(
            "<h1 style='color:rgba(200,200,200,0.5); font-size:72pt;'>DRAFT</h1>",
            rotation: 45,
            opacity: 30);

        pdf.SaveAs("watermarked.pdf");
        // Docs: https://ironpdf.com/how-to/stamp-text-image/
    }
}

Note that IronPDF's opacity argument is an integer 0–100, not a 0.0–1.0 decimal.

How Do I Migrate Password Protection?

Security settings are straightforward to migrate. The property names change, but the concepts map directly.

EO.Pdf (Before)

using EO.Pdf;

class SecuritySample
{
    static void Main()
    {
        var doc = new PdfDocument("input.pdf");

        doc.Security.UserPassword = "user123";
        doc.Security.OwnerPassword = "owner456";
        doc.Security.AllowPrint = true;
        doc.Security.AllowCopy = false;
        doc.Security.AllowModify = false;

        doc.Save("secured.pdf");
    }
}

IronPDF (After)

using IronPdf;
using IronPdf.Security;

class SecuritySample
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var pdf = PdfDocument.FromFile("input.pdf");

        pdf.SecuritySettings.UserPassword = "user123";
        pdf.SecuritySettings.OwnerPassword = "owner456";
        pdf.SecuritySettings.AllowUserPrinting = PdfPrintSecurity.FullPrintRights;
        pdf.SecuritySettings.AllowUserCopyPasteContent = false;
        pdf.SecuritySettings.AllowUserEdits = PdfEditSecurity.NoEdit;

        pdf.SaveAs("secured.pdf");
        // Docs: https://ironpdf.com/how-to/pdf-permissions-passwords/
    }
}

What Are the Critical Migration Gotchas?

There are a few sharp corners in this migration that I want to flag explicitly.

Static options to instance options. This is the design-level shift, not just a syntax change. Every HtmlToPdf.Options.X = … in your codebase was setting global state. When you move to IronPDF's per-renderer RenderingOptions, audit every place that assumed previous configuration was still in effect — those assumptions silently break.

The PdfDocument name collision. Both EO.Pdf and IronPDF have a class named PdfDocument. If you're doing a phased migration with both packages installed, you'll get ambiguous reference errors. Use fully qualified names or aliases:

// Alias approach during transition
using EoPdfDoc = EO.Pdf.PdfDocument;
using IronPdfDoc = IronPdf.PdfDocument;

Once EO.Pdf is fully removed, drop the aliases.

Page indexing. Both libraries use 0-based page indexing, so there's no off-by-one to translate — but if you mix in any 1-based external system, the silent failure mode is the wrong page being modified without an exception.

Event-based page rendering. EO.Pdf's HtmlToPdf.Options.AfterRenderPage event lets you inject per-page content (headers, footers, watermarks) imperatively. IronPDF doesn't use events; it uses HtmlHeaderFooter templates with {page} / {total-pages} placeholders for headers and footers, and ApplyStamp / ApplyWatermark for overlays. If you're relying heavily on per-page events, plan extra time — it requires restructuring how you inject per-page content.

Margin units. EO.Pdf's OutputArea takes a RectangleF in inches; IronPDF's margin properties are in millimeters. Multiply by 25.4.

How Do I Handle Performance After Migration?

IronPDF's Chromium engine has different performance characteristics than EO.Pdf's renderer. Here is what to plan for.

Cold start. IronPDF initializes a Chromium instance on first render. Plan for a 1-3 second startup cost. I recommend warming up at application startup:

// In Program.cs / startup
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
var renderer = new ChromePdfRenderer();
using var warmup = renderer.RenderHtmlAsPdf("<p>init</p>");
// Now your first real render won't pay the startup cost

Concurrency. ChromePdfRenderer can be reused across requests. For high-concurrency workloads, register it as a singleton:

// Singleton pattern for ASP.NET
builder.Services.AddSingleton<ChromePdfRenderer>();

Disposal discipline. Every PdfDocument should be in a using block. In ASP.NET, returning a File() result after disposal is safe — BinaryData is a byte array that survives disposal:

using var pdf = renderer.RenderHtmlAsPdf(html);
return File(pdf.BinaryData, "application/pdf", "report.pdf");
// pdf disposed after this line — BinaryData already copied

How Do I Troubleshoot Common EO.Pdf Problems During Migration?

The actual runtime problems teams hit, and how IronPDF handles each one differently:

TypeInitializationException or SEHException on Linux. In EO.Pdf, this is caused by missing native GDI+ or rendering libraries in Docker containers based on Alpine or minimal Debian. IronPDF uses Chromium and requires its own set of Linux packages:

# IronPDF Linux dependencies
apt-get install -y \
  libnss3 libatk1.0-0 libatk-bridge2.0-0 libcups2 \
  libdrm2 libxkbcommon0 libxcomposite1 libxdamage1 \
  libxrandr2 libgbm1 libpango-1.0-0 libcairo2 libasound2
# Full current list: https://ironpdf.com/how-to/azure/

NullReferenceException inside HtmlToPdf.ConvertHtml. In EO.Pdf, this is typically triggered by malformed HTML, missing resources, or certain CSS properties the engine can't parse. IronPDF exceptions are standard .NET exceptions with useful messages:

using IronPdf;

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.WaitFor.RenderDelay(300);

try
{
    using var pdf = renderer.RenderHtmlAsPdf("<h1>Test</h1>");
    pdf.SaveAs("output.pdf");
}
catch (IronPdfException ex)
{
    Console.WriteLine(ex.Message);
}

CSS not rendering correctly. EO.Pdf's older rendering paths may silently ignore unsupported CSS. IronPDF uses Chromium — what Chrome renders, IronPDF renders. Use browser DevTools to debug CSS first, then hand the same HTML to IronPDF:

using IronPdf;

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.CssMediaType = IronPdf.Rendering.PdfCssMediaType.Print;

using var pdf = renderer.RenderHtmlAsPdf(htmlWithComplexCss);
pdf.SaveAs("output.pdf");
// Docs: https://ironpdf.com/how-to/rendering-options/

Memory growth in long-running processes. In IronPDF, PdfDocument is IDisposable. Always use using:

using IronPdf;

var renderer = new ChromePdfRenderer(); // reuse this
using var pdf = renderer.RenderHtmlAsPdf(html);
byte[] data = pdf.BinaryData;
// pdf is disposed here — no lingering memory

Frequently Asked Questions

Can I run both EO.Pdf and IronPDF in the same project during migration?

Yes, you can install both NuGet packages side by side for a phased migration. The main issue you'll hit is the PdfDocument name collision — both libraries use that class name. Use fully qualified names or using aliases to resolve it:

using EoPdfDoc = EO.Pdf.PdfDocument;
using IronPdfDoc = IronPdf.PdfDocument;

Once all EO.Pdf code is migrated, remove the package and drop the aliases.

Does IronPDF support Linux containers like Docker?

Yes. IronPDF uses Chromium and requires specific Linux packages to be installed in your container. The dependency list differs from EO.Pdf's GDI+ requirements. IronPDF has documented Linux setup at ironpdf.com/how-to/azure. EO.Pdf's Linux story is less documented and often requires trial and error with libgdiplus and X11 dependencies.

Will my existing HTML templates work without changes in IronPDF?

In most cases, yes — and they'll likely render better. IronPDF uses the actual Chromium rendering engine, so any CSS that works in Chrome will work in IronPDF. Templates that relied on EO.Pdf's quirks (CSS features it silently ignored, or older engine behaviors) may render differently because IronPDF actually processes them correctly. I recommend visual regression testing every template after migration.

How do I handle the cold start cost of IronPDF's Chromium engine?

IronPDF initializes a Chromium instance on first render, which takes 1-3 seconds depending on hardware. Warm up at application startup by rendering a dummy document. For ASP.NET, register ChromePdfRenderer as a singleton so the engine stays warm across requests.

What about EO.Pdf's event-based page callbacks — how do I replace those?

This is the most complex part of the migration. EO.Pdf uses HtmlToPdf.Options.AfterRenderPage for per-page operations like headers, footers, and overlays. IronPDF doesn't use events — it uses HtmlHeaderFooter templates for headers/footers and the stamp/watermark API for overlays. You'll need to restructure any per-page callback logic into HTML templates or stamp operations. Plan extra time for this area.

What about EO.Pdf's ACM (Advanced Content Model) — how do I migrate that?

ACM has no direct IronPDF equivalent, and that's intentional. ACM is an imperative coordinate-positioned layout engine; IronPDF replaces it with HTML/CSS, which is more expressive and far more portable. AcmBlock becomes <div>, AcmText becomes <span> or <p>, AcmTable becomes <table>, and AcmStyle properties become CSS. The migration is mechanical at the leaf level but often produces cleaner code overall, because flexbox and grid replace explicit X/Y math.

Is IronPDF thread-safe for high-concurrency workloads?

Yes. ChromePdfRenderer is reusable across threads and requests. For ASP.NET, register it as a singleton. For high-concurrency scenarios, IronPDF supports parallel render patterns. Every PdfDocument should be in a using block to ensure proper disposal — BinaryData is a byte array that survives disposal, so you can safely return file results after the document is disposed. This is a meaningful upgrade over EO.Pdf's static HtmlToPdf.Options, where two threads racing on global config was a real bug source.

Conclusion

The EO.Pdf migration is mostly mechanical for HTML-to-PDF workflows, with three sharp corners: the static-to-instance options shift, the PdfDocument name collision, and the event-based page handling. The name collision is a compile error you'll catch immediately; the options shift requires a thread-safety audit; the event model takes design time. ACM-heavy codebases add a fourth corner — a real rewrite into HTML/CSS, but one that usually produces simpler code afterwards.

For teams using EO.Pdf primarily because "it was already there" and experiencing CSS rendering oddities, IronPDF's Chromium backing tends to resolve those without template changes — what the browser renders is what you get.

The quick start is three steps: swap the NuGet package, change your namespace imports, and replace static HtmlToPdf.ConvertHtml with an instance ChromePdfRenderer.RenderHtmlAsPdf. From there, work through the API mapping in this guide and test each template. You can run both libraries side by side during the transition, so there's no need for a big-bang rewrite.
