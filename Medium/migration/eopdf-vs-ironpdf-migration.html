Migrating from EO.Pdf to IronPDF in C#: The Complete Guide

The error message that sends most teams down the EO.Pdf migration path looks something like this: NullReferenceException somewhere in HtmlToPdfConverter, triggered by a perfectly valid HTML file that rendered fine last deploy. EO.Pdf has been in maintenance territory for years, and as .NET versions advance, the surface area for subtle compatibility issues grows.

I've watched this pattern play out across dozens of teams. In this guide, I'll walk you through the full migration from EO.Pdf to IronPDF — structured around the problems teams actually encounter, with complete before/after code examples and every API mapping you need.

Why Should I Migrate Away from EO.Pdf?

Migrations off EO.Pdf tend to happen when one of these pressure points tips over. Here are the ten most common reasons I see teams make the switch:

Runtime exceptions on newer .NET. EO.Pdf was built for .NET Framework. Running it on net6+ via compatibility shims can produce hard-to-diagnose NREs and SEHExceptions.

Platform portability. Linux containers frequently fail with EO.Pdf if native GDI+ or rendering dependencies aren't present. Docker base images don't include these by default.

CSS support gaps. The EO.Pdf rendering engine predates modern CSS. Flexbox, CSS Grid, and custom properties (var(--color)) may be partially or fully unsupported.

No JavaScript execution. EO.Pdf renders HTML at parse time. Charts, lazy-loaded content, or any JS-driven DOM changes are invisible to it.

@page and print media queries. Behavior may differ from browser-based rendering. If templates were designed for screen, print fidelity may vary.

No async API. Synchronous-only rendering becomes a bottleneck in async ASP.NET controller pipelines.

Debug experience. When EO.Pdf fails, the exception stack is often inside the library internals with no actionable message. Troubleshooting involves trial and error.

Thread isolation. Some rendering libraries have static state that causes issues under load. Verify EO.Pdf's behavior when called concurrently.

Memory leaks in long-running processes. Teams running EO.Pdf in Windows services or worker roles sometimes report memory growth.

No URL rendering. EO.Pdf takes HTML input; rendering live URLs directly requires you to fetch HTML first.

How Do I Migrate HTML to PDF Rendering?

This is the most common migration path. Let me show you the full before/after for a typical report generation scenario.

EO.Pdf (Before)

using EO.Pdf;
using System.IO;

class Program
{
    static void Main()
    {
        // EO.Pdf license activation
        // PdfRuntime.Activate("YOUR-KEY");

        string html = @"
            <html>
            <body>
                <h1 style='color: #333;'>Monthly Report</h1>
                <p>Generated: <strong>2025-01</strong></p>
                <table border='1'>
                    <tr><th>Item</th><th>Value</th></tr>
                    <tr><td>Revenue</td><td>$120,000</td></tr>
                </table>
            </body>
            </html>";

        var converter = new HtmlToPdfConverter();

        var pdfDoc = new EO.Pdf.PdfDocument();
        converter.ConvertHtml(html, pdfDoc);

        pdfDoc.Save("report.pdf");
    }
}

IronPDF (After)

using IronPdf;

class Program
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        string html = @"
            <html>
            <body>
                <h1 style='color: #333;'>Monthly Report</h1>
                <p>Generated: <strong>2025-01</strong></p>
                <table border='1'>
                    <tr><th>Item</th><th>Value</th></tr>
                    <tr><td>Revenue</td><td>$120,000</td></tr>
                </table>
            </body>
            </html>";

        var renderer = new ChromePdfRenderer();
        renderer.RenderingOptions.MarginTop = 20;

        using var pdf = renderer.RenderHtmlAsPdf(html);
        pdf.SaveAs("report.pdf");
    }
}

How Do I Migrate Watermarks?

EO.Pdf may handle watermarking via HTML header/footer injection or page annotation. IronPDF has a dedicated stamp API that's much more intuitive.

{
    static void Main()
    {
        var doc = new EO.Pdf.PdfDocument("input.pdf");

        // EO.Pdf watermarking — may handle via HTML header/footer injection
        // or via page annotation
        // var converter = new HtmlToPdfConverter();
        // converter.Options.PageHeaderHtml = "<div style='opacity:0.3'>DRAFT</div>";

        doc.Save("watermarked.pdf");
    }
}

IronPDF (After)

using IronPdf;
using IronPdf.Editing;

class WatermarkSample
{
    static void Main()
    {
        IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

        using var pdf = PdfDocument.FromFile("input.pdf");

        {
            Text = "DRAFT",
            FontSize = 50,
            Opacity = 30,
            Rotation = 45,
            VerticalAlignment = VerticalAlignment.Middle,
            HorizontalAlignment = HorizontalAlignment.Center
        };

        pdf.ApplyStamp(stamper);
        pdf.SaveAs("watermarked.pdf");
        // Docs: https://ironpdf.com/how-to/stamp-text-image/
    }
}

What Are the Critical Migration Gotchas?

There are a few sharp corners in this migration that I want to flag explicitly.

The PdfDocument Name Collision. Both EO.Pdf and IronPDF have a class named PdfDocument. If you're doing a phased migration with both packages installed, you'll get ambiguous reference errors. Use fully qualified names or aliases:

// Alias approach during transition
using EoPdfDoc = EO.Pdf.PdfDocument;
using IronPdfDoc = IronPdf.PdfDocument;

Once EO.Pdf is fully removed, drop the aliases.

Page Indexing. Both libraries use 0-based page indexing. Off-by-one errors are silent — they'll affect the wrong page without throwing an exception.

Event-Based Page Rendering. EO.Pdf has a PdfPageEventCallback or similar event model for per-page operations (headers, footers, overlays). IronPDF doesn't use events; it uses HTML templates for headers/footers and stamps for overlays. If you're relying heavily on per-page events, plan extra time for this — it requires restructuring how you inject per-page content.

Exception Model. EO.Pdf may use return codes or event-based error reporting in some versions. IronPDF throws standard .NET exceptions. Review your error-handling code as part of migration.

How Do I Handle Performance After Migration?

IronPDF's Chromium engine has different performance characteristics than EO.Pdf's lightweight renderer. Here's what to plan for.

Cold Start. IronPDF initializes a Chromium instance on first render. Plan for a 1-3 second startup cost. I recommend warming up at application startup:

// In Program.cs / startup
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
var renderer = new ChromePdfRenderer();
using var warmup = renderer.RenderHtmlAsPdf("<p>init</p>");
// Now your first real render won't pay the startup cost

Concurrency. ChromePdfRenderer can be reused across requests. For high-concurrency workloads, consider a renderer pool or singleton pattern:

// Singleton pattern for ASP.NET
builder.Services.AddSingleton<ChromePdfRenderer>();

Disposal Discipline. Every PdfDocument should be in a using block. In ASP.NET, returning a File() result after disposing is safe — BinaryData is a byte array that survives disposal:

using var pdf = renderer.RenderHtmlAsPdf(html);
return File(pdf.BinaryData, "application/pdf", "report.pdf");
// pdf disposed after this line — BinaryData already copied

How Do I Troubleshoot Common EO.Pdf Problems During Migration?

Let me address the actual runtime problems teams hit, and how IronPDF handles each one differently.

TypeInitializationException or SEHException on Linux. In EO.Pdf, this is caused by missing native GDI+ or rendering libraries in Docker containers based on Alpine or minimal Debian. IronPDF uses Chromium and requires its own set of Linux packages:

# IronPDF Linux dependencies
apt-get install -y \
  libnss3 libatk1.0-0 libatk-bridge2.0-0 libcups2 \
  libdrm2 libxkbcommon0 libxcomposite1 libxdamage1 \
  libxrandr2 libgbm1 libpango-1.0-0 libcairo2 libasound2
# Full current list: https://ironpdf.com/how-to/azure/

NullReferenceException Inside HtmlToPdfConverter. In EO.Pdf, this is typically triggered by malformed HTML, missing resources, or certain CSS properties the engine can't parse. IronPDF exceptions are standard .NET exceptions with useful messages:

using IronPdf;

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.WaitFor.RenderDelay(300);

try
{
    using var pdf = renderer.RenderHtmlAsPdf("<h1>Test</h1>");
    pdf.SaveAs("output.pdf");
}
catch (IronPdfException ex)
{
    Console.WriteLine(ex.Message);
}

CSS Not Rendering Correctly. EO.Pdf's custom rendering engine may silently ignore unsupported CSS. IronPDF uses Chromium — what Chrome renders, IronPDF renders. Use browser DevTools to debug CSS first, then hand the same HTML to IronPDF:

using IronPdf;

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.CssMediaType = IronPdf.Rendering.PdfCssMediaType.Print;

using var pdf = renderer.RenderHtmlAsPdf(htmlWithComplexCss);
pdf.SaveAs("output.pdf");
// Docs: https://ironpdf.com/how-to/rendering-options/

Memory Growth in Long-Running Processes. In IronPDF, PdfDocument is IDisposable. Always use using:

using IronPdf;

var renderer = new ChromePdfRenderer(); // reuse this
using var pdf = renderer.RenderHtmlAsPdf(html);
byte[] data = pdf.BinaryData;
// pdf is disposed here — no lingering memory

Frequently Asked Questions

Can I run both EO.Pdf and IronPDF in the same project during migration?

Answer: Yes, you can install both NuGet packages side by side for a phased migration. The main issue you'll hit is the PdfDocument name collision — both libraries use that class name. Use fully qualified names or using aliases to resolve it:

using EoPdfDoc = EO.Pdf.PdfDocument;
using IronPdfDoc = IronPdf.PdfDocument;

Once all EO.Pdf code is migrated, remove the package and drop the aliases.

Does IronPDF support Linux containers like Docker?

Answer: Yes. IronPDF uses Chromium and requires specific Linux packages to be installed in your container. The dependency list differs from EO.Pdf's GDI+ requirements. IronPDF has documented Linux setup at ironpdf.com/how-to/azure. EO.Pdf's Linux story is less documented and often requires trial and error with libgdiplus and X11 dependencies.

Will my existing HTML templates work without changes in IronPDF?

Answer: In most cases, yes — and they'll likely render better. IronPDF uses the actual Chromium rendering engine, so any CSS that works in Chrome will work in IronPDF. Templates that relied on EO.Pdf's quirks (like CSS features it silently ignored) may render differently because IronPDF actually processes them correctly. I recommend visual regression testing every template after migration.

How do I handle the cold start cost of IronPDF's Chromium engine?

Answer: IronPDF initializes a Chromium instance on first render, which takes 1-3 seconds depending on hardware. Warm up at application startup by rendering a dummy document. For ASP.NET, register ChromePdfRenderer as a singleton so the engine stays warm across requests.

What about EO.Pdf's event-based page callbacks — how do I replace those?

Answer: This is the most complex part of the migration. EO.Pdf uses a PdfPageEventCallback model for per-page operations like headers, footers, and overlays. IronPDF doesn't use events — it uses HTML templates for headers/footers and the stamp API for overlays. You'll need to restructure any per-page callback logic into HTML templates or stamp operations. Plan extra time for this area.

Is IronPDF thread-safe for high-concurrency workloads?

Answer: Yes. ChromePdfRenderer is reusable across threads and requests. For ASP.NET, register it as a singleton. For high-concurrency scenarios, IronPDF supports parallel render patterns. Every PdfDocument should be in a using block to ensure proper disposal — BinaryData is a byte array that survives disposal, so you can safely return file results after the document is disposed.

Conclusion

The EO.Pdf migration is mostly mechanical for HTML-to-PDF workflows, with two sharp corners: the PdfDocument name collision and the event-based page handling. The name collision is a compile error you'll catch immediately; the event model takes design time.

For teams using EO.Pdf primarily because "it was already there" and experiencing CSS rendering oddities, IronPDF's Chromium backing tends to resolve those without template changes — what the browser renders is what you get.

The quick start is three steps: swap the NuGet package, change your namespace imports, and replace HtmlToPdfConverter with ChromePdfRenderer. From there, work through the API mapping tables in this guide and test each template. You can run both libraries side by side during the transition, so there's no need for a big-bang rewrite.
