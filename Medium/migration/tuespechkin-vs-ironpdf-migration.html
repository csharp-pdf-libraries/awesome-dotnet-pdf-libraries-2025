Migrating from TuesPechkin to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Consider Migrating from TuesPechkin?

I have watched this conversation play out many times. Someone checks the NuGet page and the realization hits: TuesPechkin's last release was 2.1.1 in January 2015, and the wkhtmltopdf binary it wraps had its upstream repository archived on January 2, 2023, with the packaging repository archived on August 28, 2023. The conversation shifts from "someday" to "this sprint."

TuesPechkin ships under a Creative Commons Attribution 3.0 license and has had no activity since 2015. .NET 6+ compatibility is not officially maintained, and the wkhtmltopdf maintainers have stated the project will not receive further security fixes. CVE-2022-35583 (CVSS 9.8 SSRF) is unpatched and will remain so. If you are building on a modern stack, that is a real concern.

The bundled wkhtmltopdf binary is platform-specific, with separate Win32, Win64, Linux, and macOS deployments wired through TuesPechkin's IDeployment chain. Docker on Linux works only with the matching native build, and cross-platform deployment requires careful binary management. If your team is moving toward containerized deployments, this is a real friction point.

The rendering engine is Qt WebKit 4.8, which predates modern Chrome. Flexbox, CSS Grid, and custom properties do not render correctly. Your browser preview and your PDF output will not match.

Concurrency is a recurring pain point. The standard TuesPechkin pattern wires StandardConverter through RemotingToolset<PdfToolset> so the native wkhtmltopdf binary runs in an isolated AppDomain. That isolation is what keeps the process alive, but it makes the converter awkward to share across threads, and most production setups end up with a singleton converter funneling requests through one render pipeline.

TuesPechkin has no native PDF manipulation. Merge, watermark, security, and text extraction all require additional libraries alongside TuesPechkin. That means more dependencies, more maintenance, and more surface area for breaks.

The wkhtmltopdf binary adds roughly 14MB to your Docker image or deployment artifact. Cookies, session tokens, and authenticated page rendering are complex with wkhtmltopdf. And while wkhtmltopdf executes JavaScript, the older engine means complex JS-rendered content may not render correctly.

What Does the Migration Effort Actually Look Like?

I want to give you a realistic picture of the effort involved. Here is a feature-by-feature breakdown so you can scope the work before writing any code.

Effort by Feature

  • HTML string to PDF — Low. IConverter.Convert(doc) in TuesPechkin becomes ChromePdfRenderer.RenderHtmlAsPdf().
  • URL to PDF — Low. ObjectSettings.PageUrl = url becomes renderer.RenderUrlAsPdf(url).
  • HTML file to PDF — Low. ObjectSettings.PageUrl = "file:///..." becomes renderer.RenderHtmlFileAsPdf(path).
  • Save to bytes — Low. Convert() returns byte[]; IronPDF exposes the same via pdf.BinaryData.
  • Save to file — Low. File.WriteAllBytes(path, bytes) becomes pdf.SaveAs(path).
  • Page size — Low. GlobalSettings.PaperSize becomes RenderingOptions.PaperSize.
  • Margins — Low. GlobalSettings.Margins.* becomes RenderingOptions.MarginTop/Bottom/Left/Right.
  • Headers/footers — Medium. HeaderSettings/FooterSettings becomes RenderingOptions.HtmlHeader/HtmlFooter, with token syntax changing from [page] to {page}.
  • Merge PDFs — Medium. Not native in TuesPechkin; IronPDF provides PdfDocument.Merge().
  • Watermark — Medium. Not native in TuesPechkin; IronPDF provides TextStamper and ImageStamper.
  • Password protection — Medium. Not native in TuesPechkin; IronPDF provides pdf.SecuritySettings.
  • Text extraction — Medium. Not native in TuesPechkin; IronPDF provides pdf.ExtractAllText().

How Do You Decide Whether to Migrate?

Not every team should migrate on the same timeline. Here is a decision matrix I use when advising teams.

  • If .NET 6+ compatibility is the primary concern, switch — TuesPechkin has no official .NET 6+ support.
  • If Docker or Linux deployment is needed, switch — wkhtmltopdf requires a platform-specific native binary and the Windows-embedded build will not run there.
  • If modern CSS (flex, grid, custom properties) is required, switch — Chromium handles all of it; Qt WebKit 4.8 does not.
  • If you need basic HTML-to-PDF on Windows only and wkhtmltopdf is working well, evaluate migration cost. The upstream-abandonment caveat still applies, but your timeline is more flexible.

Finding TuesPechkin References in Your Codebase

# Show line-level usage for review

Removing TuesPechkin and Installing IronPDF

# Remove TuesPechkin packages
dotnet remove package TuesPechkin
dotnet remove package TuesPechkin.Wkhtmltox.Win64
dotnet remove package TuesPechkin.Wkhtmltox.Win32

# Install IronPDF
dotnet add package IronPdf

dotnet restore

# Verify packages
dotnet list package | grep -i "tuespechkin\|wkhtmltox\|ironpdf"

If you manually copied wkhtmltopdf.exe into your project or Docker image, remove those references too. IronPDF ships its own Chromium binaries. The IronPDF license configuration guide covers the install pattern.

Basic HTML-to-PDF Conversion

Before (TuesPechkin):

using TuesPechkin;
using System;
using System.Drawing.Printing;
using System.IO;

class Program
{
    static void Main()
    {
        var converter = new StandardConverter(
            new RemotingToolset<PdfToolset>(
                new Win64EmbeddedDeployment(
                    new TempFolderDeployment())));

        var doc = new HtmlToPdfDocument
        {
            GlobalSettings = { PaperSize = PaperKind.A4 },
            Objects =
            {
                new ObjectSettings
                {
                    HtmlText = "<html><body><h1>Hello</h1></body></html>",
                    WebSettings = { DefaultEncoding = "utf-8" },
                }
            }
        };

        byte[] pdfBytes = converter.Convert(doc);
        File.WriteAllBytes("output.pdf", pdfBytes);
        Console.WriteLine("Saved output.pdf");
    }
}

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = PdfPaperSize.A4;

var pdf = renderer.RenderHtmlAsPdf("<html><body><h1>Hello</h1></body></html>");
pdf.SaveAs("output.pdf");

Console.WriteLine($"Saved output.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

What Does Each Before/After Migration Look Like in Practice?

Let me walk through four complete migration examples so you can see exactly what changes in real code.

HTML String to PDF with Full Options

Before (TuesPechkin):

using TuesPechkin;
using System;
using System.Drawing.Printing;
using System.IO;

class HtmlToPdfBefore
{
    static void Main()
    {
        var converter = new StandardConverter(
            new RemotingToolset<PdfToolset>(
                new Win64EmbeddedDeployment(new TempFolderDeployment())));

        var doc = new HtmlToPdfDocument
        {
            GlobalSettings =
            {
                PaperSize = PaperKind.A4,
                Margins = new MarginSettings
                {
                    Unit = Unit.Millimeters,
                    Top = 20, Bottom = 20, Left = 25, Right = 25,
                },
            },
            Objects =
            {
                new ObjectSettings
                {
                    HtmlText = "<html><body><h1>Invoice #9901</h1></body></html>",
                    WebSettings = { DefaultEncoding = "utf-8" },
                }
            }
        };

        byte[] pdfBytes = converter.Convert(doc);
        File.WriteAllBytes("invoice.pdf", pdfBytes);
        Console.WriteLine($"Saved invoice.pdf ({pdfBytes.Length:N0} bytes)");
    }
}

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = PdfPaperSize.A4;
renderer.RenderingOptions.MarginTop = 20;
renderer.RenderingOptions.MarginBottom = 20;
renderer.RenderingOptions.MarginLeft = 25;
renderer.RenderingOptions.MarginRight = 25;

var pdf = renderer.RenderHtmlAsPdf("<html><body><h1>Invoice #9901</h1></body></html>");
pdf.SaveAs("invoice.pdf");
Console.WriteLine($"Saved invoice.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

Merge PDFs

Before (TuesPechkin — not native; secondary library required):

using TuesPechkin;
using System;
using System.Collections.Generic;
using System.IO;

class MergeBefore
{
    static void Main()
    {
        var converter = new StandardConverter(
            new RemotingToolset<PdfToolset>(
                new Win64EmbeddedDeployment(new TempFolderDeployment())));

        var htmlSections = new[] {
            "<html><body><h1>Section A</h1></body></html>",
            "<html><body><h1>Section B</h1></body></html>",
        };
        var pdfFiles = new List<string>();

        for (int i = 0; i < htmlSections.Length; i++)
        {
            var doc = new HtmlToPdfDocument
            {
                Objects = { new ObjectSettings { HtmlText = htmlSections[i] } }
            };
            byte[] bytes = converter.Convert(doc);

            var outPath = $"section{i + 1}.pdf";
            File.WriteAllBytes(outPath, bytes);
            pdfFiles.Add(outPath);
        }

        // Merge requires a secondary library — TuesPechkin has no merge.
        Console.WriteLine("Merge requires secondary library");
    }
}

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();

var sectionA = renderer.RenderHtmlAsPdf("<html><body><h1>Section A</h1></body></html>");
var sectionB = renderer.RenderHtmlAsPdf("<html><body><h1>Section B</h1></body></html>");

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(sectionA, sectionB);
merged.SaveAs("merged.pdf");
Console.WriteLine($"Merged: {merged.PageCount} pages");

How Do the Two Libraries Compare Feature by Feature?

Here are the full mapping tables I use when planning a TuesPechkin migration. These cover the overall comparison, namespaces, core classes, document loading, page operations, and merge/split.

Overall Comparison

  • Focus. TuesPechkin is an HTML-to-PDF wrapper around wkhtmltopdf; IronPDF covers HTML-to-PDF plus PDF manipulation in one library.
  • Pricing. TuesPechkin is licensed under Creative Commons Attribution 3.0 and the bundled wkhtmltopdf binary ships under LGPL; IronPDF is commercially licensed — check current pricing at ironsoftware.com.
  • API style. TuesPechkin uses IConverter.Convert(document) with HtmlToPdfDocument + settings objects; IronPDF uses ChromePdfRenderer with a single RenderingOptions bag.
  • Learning curve. TuesPechkin is low for basic use, but the deployment chain and remoting toolset add ceremony; IronPDF is low for any .NET developer with no native binary management to learn.
  • HTML rendering. TuesPechkin renders with Qt WebKit 4.8 (the pre-Chrome era); IronPDF renders with embedded Chromium.
  • Page indexing. wkhtmltopdf token syntax is 1-based in headers/footers; IronPDF page APIs are 0-based.
  • Concurrency. The RemotingToolset<PdfToolset> + StandardConverter pattern is typically run as a singleton funneling renders through one AppDomain. IronPDF supports concurrent ChromePdfRenderer usage; verify guidance for your target version.
  • Namespace. TuesPechkin versus IronPdf.

Namespace Mapping

  • TuesPechkin → IronPdf — core namespace
  • No equivalent → IronPdf.Rendering — rendering configuration types
  • No equivalent → IronPdf.Editing — watermark and stamp types

Core Class Mapping

  • IConverter / StandardConverter / ThreadSafeConverter → ChromePdfRenderer — primary rendering class
  • HtmlToPdfDocument → no equivalent — the HTML string itself is the input
  • GlobalSettings → ChromePdfRenderOptions — page size, margins, global settings
  • ObjectSettings → ChromePdfRenderOptions plus the HTML input parameter — per-document settings
  • RemotingToolset<PdfToolset> / Win64EmbeddedDeployment / TempFolderDeployment → no equivalent — IronPDF ships its own Chromium and needs no native deployment chain

Document Loading Methods

  • HTML string. ObjectSettings.HtmlText = html in TuesPechkin; renderer.RenderHtmlAsPdf(html) in IronPDF.
  • URL. ObjectSettings.PageUrl = url in TuesPechkin; renderer.RenderUrlAsPdf(url) in IronPDF.
  • HTML file. ObjectSettings.PageUrl = "file:///path" in TuesPechkin; renderer.RenderHtmlFileAsPdf(path) in IronPDF.
  • Get bytes. converter.Convert(doc) returns byte[] in TuesPechkin; pdf.BinaryData in IronPDF.

Page Operations

  • Page count. Not returned from TuesPechkin's Convert(); IronPDF exposes pdf.PageCount.
  • Remove page. Not applicable in TuesPechkin (no PDF manipulation); IronPDF uses pdf.RemovePages(index).
  • Extract text. Not applicable in TuesPechkin; IronPDF uses pdf.ExtractAllText().
  • Rotate. wkhtmltopdf relies on CSS @page or its own orientation setting; IronPDF exposes page rotation via the RenderingOptions.PaperOrientation setting and per-page rotation APIs — verify the exact entry point against current IronPDF docs.

Merge/Split Operations

  • Merge. Not native in TuesPechkin (a secondary library is required); IronPDF uses PdfDocument.Merge(doc1, doc2).
  • Split. Not native in TuesPechkin; IronPDF supports it via CopyPages — see the merge/split guide.

What Are the Frequently Asked Questions?

I get these four questions on almost every TuesPechkin migration I advise on. Here they are with direct answers.

Will My PDFs Look Exactly the Same After Migration?

TuesPechkin uses Qt WebKit 4.8 via wkhtmltopdf. IronPDF uses Chromium. For most templates this produces visually similar output, but not identical. Modern CSS features like flex and grid will actually render better with IronPDF since Qt WebKit did not support them correctly.

IronPDF uses 0-based page indexing while wkhtmltopdf uses 1-based indexing in some contexts. If you are feeding output into other tools, verify the consumer side too. Run a visual comparison across all your HTML templates before cutting over.

How Do I Handle the Converter and Deployment Boilerplate Removal?

Count your converter and deployment wrappers before starting. Each one is a migration site. Every StandardConverter, every ThreadSafeConverter, every RemotingToolset, every Win64EmbeddedDeployment / Win32EmbeddedDeployment, and any singleton or ThreadLocal wrapper around the converter gets removed entirely. IronPDF does not require any of this infrastructure.

# TuesPechkin: count these and remove all of them
rg "StandardConverter|ThreadSafeConverter|MultiplexingConverter" --type cs -n
rg "RemotingToolset|Win64EmbeddedDeployment|Win32EmbeddedDeployment|TempFolderDeployment" --type cs -n
rg "ThreadLocal<IConverter>|SetApartmentState" --type cs -n

What About Header and Footer Token Migration?

wkhtmltopdf and TuesPechkin use [page] and [topage] as header/footer tokens. IronPDF uses {page} and {total-pages}. Any header or footer HTML that referenced the old tokens needs updating. This is a find-and-replace operation, but easy to miss if you do not audit systematically.

// wkhtmltopdf/TuesPechkin footer tokens:
// "Page [page] of [topage]"

// IronPDF footer tokens:
// https://ironpdf.com/how-to/headers-and-footers/
renderer.RenderingOptions.HtmlFooter = new HtmlHeaderFooter
{
    HtmlFragment = "<div style='font-size:9px; text-align:right; padding: 0 20px'>Page {page} of {total-pages}</div>",
};

Can I Remove Secondary PDF Libraries After Migration?

TuesPechkin commonly accumulates secondary libraries for missing features like merge, watermark, and security. After IronPDF covers those natively, you can remove them. Run dotnet list package | grep -i "pdfsharp\|itextsharp\|pdfdocument\|dinkto" to find candidates. Verify each library was only used to supplement TuesPechkin before removing it.

What Is the Decision Framework for Your Team?

Here is the migration framework I recommend. Work through it in order: pre-migration assessment, code migration, testing, and post-migration cleanup.

Pre-Migration

Find all TuesPechkin usages with rg -l "TuesPechkin" --type cs. Count the converter and deployment-chain wrappers (StandardConverter, ThreadSafeConverter, RemotingToolset, Win64EmbeddedDeployment) to understand scope. Check whether your secondary PDF libraries (PdfSharp, iTextSharp) are only there to supplement TuesPechkin and can be removed. Inventory all HTML templates for render comparison testing. Verify IronPDF target framework compatibility. Note all GlobalSettings and ObjectSettings properties currently in use.

Code Migration

Remove the TuesPechkin, TuesPechkin.Wkhtmltox.Win64, and TuesPechkin.Wkhtmltox.Win32 NuGet packages. Remove wkhtmltopdf binaries from Docker images and deployment scripts. Install the IronPdf NuGet package. Replace using TuesPechkin with using IronPdf. Remove the entire converter, toolset, and deployment chain (StandardConverter / ThreadSafeConverter + RemotingToolset<PdfToolset> + Win64EmbeddedDeployment + TempFolderDeployment). Replace HtmlToPdfDocument + GlobalSettings + ObjectSettings with ChromePdfRenderOptions. Replace converter.Convert(doc) with renderer.RenderHtmlAsPdf(html). Replace File.WriteAllBytes(path, pdfBytes) with pdf.SaveAs(path). Update header/footer tokens from [page]/[topage] to {page}/{total-pages}. Remove secondary merge/watermark/encrypt libraries. Add your IronPDF license key to CI/CD secrets.

Testing

Render each HTML template and visually compare TuesPechkin vs IronPDF output. Check that modern CSS (flex, grid) now renders correctly. Verify page size and margins match reference. Test headers and footers, paying attention to the page number token format difference. Run concurrent rendering tests at 10/50/100 simultaneous requests so you can confirm IronPDF handles the load that previously had to funnel through one TuesPechkin singleton. Test merge output for correct page order. Test watermark for opacity, position, and text. Test password protection with correct and incorrect credentials. Test in Docker on Linux. Verify Docker image builds without the wkhtmltopdf binary.

Post-Migration

Remove TuesPechkin NuGet package and all related packages. Remove the converter, toolset, and deployment-chain singleton infrastructure. Remove wkhtmltopdf binary from Docker image copy steps. Remove secondary PDF libraries now replaced by IronPDF. Update Dockerfile with IronPDF Linux system dependencies. Verify deployment artifact size reduced. Archive wkhtmltopdf-specific header/footer HTML files for reference. Monitor memory usage in the first production week.

The immediate wins after migration — Linux/Docker support without managing native binaries, modern CSS rendering, and concurrent rendering without funneling through one converter — are measurable from day one.
