Migrating from TuesPechkin to IronPDF: The Decision Guide for .NET Teams

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Should You Consider Migrating from TuesPechkin?

I have watched this conversation play out many times. The license renewal arrives, someone checks the NuGet page, and the realization hits: TuesPechkin's last release was years ago, and the wkhtmltopdf binary it wraps has not seen an update either. The conversation shifts from "someday" to "this sprint."

TuesPechkin and wkhtmltopdf are both in low or no active development. .NET 6+ compatibility is not officially maintained. If you are building on a modern stack, that is a real concern.

The bundled wkhtmltopdf binary is Windows-only. Docker on Linux or any cross-platform deployment fails outright. If your team is moving toward containerized deployments, this is a hard blocker.

The rendering engine is Qt WebKit, which is pre-2016. Modern CSS features like flex, grid, and custom properties do not render correctly. Your browser preview and your PDF output will not match.

Thread safety is a recurring pain point. wkhtmltopdf has COM-like STA affinity requirements, so concurrent rendering requires a dedicated thread. Most production setups end up using an STA-threaded AppDomain or ThreadLocal<IConverter> to manage this constraint.

TuesPechkin has no native PDF manipulation. Merge, watermark, security, and text extraction all require additional libraries alongside TuesPechkin. That means more dependencies, more maintenance, and more surface area for breaks.

The wkhtmltopdf binary adds roughly 14MB to your Docker image or deployment artifact. Cookies, session tokens, and authenticated page rendering are complex with wkhtmltopdf. And while wkhtmltopdf executes JavaScript, the older engine means complex JS-rendered content may not render correctly.

Finding TuesPechkin References in Your Codebase

Before touching any code, run these commands and document the results. The output determines your scope and risk.

# Find all TuesPechkin API usage files
rg -l "TuesPechkin\|IConverter\|HtmlToPdfDocument\|ObjectSettings\|GlobalSettings" --type cs

# Show line-level usage for review
rg "TuesPechkin\|IConverter\|HtmlToPdfDocument" --type cs -n

# Find STA threading workaround code
rg "ApartmentState\.STA\|SetApartmentState\|ThreadLocal.*Converter" --type cs -n

# Find the converter factory / singleton pattern
rg "ThreadSafeConverter\|MultiplexingConverter\|ThreadLocal<IConverter>" --type cs -n

# Find URL rendering vs HTML string rendering
rg "ObjectSettings.*Page\s*=\|Page\s*=.*http" --type cs -n

# Count usage density
rg "IConverter\.\|HtmlToPdfDocument\|ObjectSettings" --type cs | wc -l

Removing TuesPechkin and Installing IronPDF

# Remove TuesPechkin packages
dotnet remove package TuesPechkin
dotnet remove package TuesPechkin.Wkhtmltopdf.Win64
dotnet remove package TuesPechkin.Wkhtmltopdf.Win32

# Install IronPDF
dotnet add package IronPdf

dotnet restore

# Verify packages
dotnet list package | grep -i "tuespechkin\|wkhtmltopdf\|ironpdf"

If you manually copied wkhtmltopdf.exe into your project or Docker image, remove those references too. IronPDF ships its own Chromium binaries. You can verify the install pattern at the IronPDF license configuration guide.

Step 3: Basic HTML-to-PDF Conversion

Before (TuesPechkin with STA thread):

using TuesPechkin;
using System;
using System.IO;
using System.Threading;

class Program
{
    private static readonly ThreadLocal<IConverter> _converter = new ThreadLocal<IConverter>(
        () => new ThreadSafeConverter(
            new PdfToolset(new Win64EmbeddedDeployment(
                new TempFolderDeployment()))));

    static void Main()
    {
        byte[] pdfBytes = null;
        var thread = new Thread(() =>
        {
            var doc = new HtmlToPdfDocument
            {
                GlobalSettings = { PaperSize = PaperKind.A4 },
                Objects =
                {
                    new ObjectSettings
                    {
                        HtmlText = "<html><body><h1>Hello</h1></body></html>",
                        WebSettings = { DefaultEncoding = "utf-8" },
                    }
                }
            };
            pdfBytes = _converter.Value.Convert(doc);
        });

        thread.SetApartmentState(ApartmentState.STA);
        thread.Start();
        thread.Join();

        File.WriteAllBytes("output.pdf", pdfBytes);
        Console.WriteLine("Saved output.pdf");
    }
}

After (IronPDF):

using IronPdf;
using System;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

var pdf = await renderer.RenderHtmlAsPdfAsync("<html><body><h1>Hello</h1></body></html>");
pdf.SaveAs("output.pdf");

Console.WriteLine($"Saved output.pdf ({pdf.PageCount} page(s))");
// https://ironpdf.com/how-to/html-string-to-pdf/

HTML String to PDF with Full Options

Before (TuesPechkin):

using TuesPechkin;
using System;
using System.IO;
using System.Threading;

class HtmlToPdfBefore
{

            new PdfToolset(

    static void Main()
    {
        byte[] pdfBytes = null;

        var thread = new Thread(() =>
        {
            var doc = new HtmlToPdfDocument
            {
                GlobalSettings =
                {
                    PaperSize = PaperKind.A4,
                    Margins = { Top = 20, Bottom = 20, Left = 25, Right = 25, Unit = Unit.Millimeters },
                },
                Objects =
                {
                    new ObjectSettings
                    {
                        HtmlText = "<html><body><h1>Invoice #9901</h1></body></html>",
                        WebSettings = { DefaultEncoding = "utf-8" },
                    }
                }
            };
            pdfBytes = _converter.Value.Convert(doc);
        });

        thread.SetApartmentState(ApartmentState.STA);
        thread.Start();
        thread.Join();

        File.WriteAllBytes("invoice.pdf", pdfBytes);
        Console.WriteLine($"Saved invoice.pdf ({pdfBytes.Length:N0} bytes)");
    }
}

Merge PDFs

Before (TuesPechkin — not native; secondary library required):

using TuesPechkin;
using System;
using System.Collections.Generic;
using System.IO;
using System.Threading;

class MergeBefore
{
    private static readonly ThreadLocal<IConverter> _converter = new(() =>
        new ThreadSafeConverter(
            new PdfToolset(new Win64EmbeddedDeployment(new TempFolderDeployment()))));

    static void Main()
    {
        var htmlSections = new[] {
            "<html><body><h1>Section A</h1></body></html>",
            "<html><body><h1>Section B</h1></body></html>",
        };
        var pdfFiles = new List<string>();

        for (int i = 0; i < htmlSections.Length; i++)
        {
            var html = htmlSections[i];
            var outPath = $"section{i + 1}.pdf";
            byte[] bytes = null;

            var thread = new Thread(() =>
            {
                var doc = new HtmlToPdfDocument
                {
                    Objects = { new ObjectSettings { HtmlText = html } }
                };
                bytes = _converter.Value.Convert(doc);
            });
            thread.SetApartmentState(ApartmentState.STA);
            thread.Start();
            thread.Join();

            File.WriteAllBytes(outPath, bytes);
            pdfFiles.Add(outPath);
        }

        // Merge requires secondary library — TuesPechkin has no merge
        Console.WriteLine("Merge requires secondary library — STA thread per render");
    }
}

After (IronPDF):

using IronPdf;
using System;
using System.Threading.Tasks;

IronPdf.License.LicenseKey = Environment.GetEnvironmentVariable("IRONPDF_LICENSE_KEY");

var renderer = new ChromePdfRenderer();

var results = await Task.WhenAll(
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section A</h1></body></html>"),
    renderer.RenderHtmlAsPdfAsync("<html><body><h1>Section B</h1></body></html>")
);

// https://ironpdf.com/how-to/merge-or-split-pdfs/
var merged = PdfDocument.Merge(results[0], results[1]);
merged.SaveAs("merged.pdf");
Console.WriteLine($"Merged: {merged.PageCount} pages — no STA thread needed");

What Are the Frequently Asked Questions?

Answer: I get these four questions on almost every TuesPechkin migration I advise on. Here they are with direct answers.

Will My PDFs Look Exactly the Same After Migration?

Answer: TuesPechkin uses Qt WebKit (pre-2016) via wkhtmltopdf. IronPDF uses Chromium. For most templates this produces visually similar output, but not identical. Modern CSS features like flex and grid will actually render better with IronPDF since Qt WebKit did not support them correctly.

IronPDF uses 0-based page indexing while wkhtmltopdf uses 1-based indexing in some contexts. If you are feeding output into other tools, verify the consumer side too. Run a visual comparison across all your HTML templates before cutting over.

How Do I Handle the STA Thread Boilerplate Removal?

Answer: Count your STA thread wrappers before starting. Each one is a migration site. Run rg "SetApartmentState\(ApartmentState\.STA\)" --type cs -n | wc -l to get the count. Every ThreadLocal<IConverter>, every ThreadSafeConverter, every MultiplexingConverter singleton, and every STA thread creation block gets removed entirely. IronPDF does not require any of this infrastructure.

// TuesPechkin: count these and remove all of them
rg "SetApartmentState\(ApartmentState\.STA\)" --type cs -n | wc -l
rg "ThreadSafeConverter\|MultiplexingConverter\|ThreadLocal<IConverter>" --type cs -n

What About Header and Footer Token Migration?

Answer: wkhtmltopdf and TuesPechkin use [page] and [topage] as header/footer tokens. IronPDF uses {page} and {total-pages}. Any header or footer HTML that referenced the old tokens needs updating. This is a find-and-replace operation, but easy to miss if you do not audit systematically.

// wkhtmltopdf/TuesPechkin footer tokens:
// "Page [page] of [topage]"

// IronPDF footer tokens:
// https://ironpdf.com/how-to/headers-and-footers/
renderer.RenderingOptions.HtmlFooter = new HtmlHeaderFooter
{
    HtmlFragment = "<div style='font-size:9px; text-align:right; padding: 0 20px'>Page {page} of {total-pages}</div>",
};

Can I Remove Secondary PDF Libraries After Migration?

Answer: TuesPechkin commonly accumulates secondary libraries for missing features like merge, watermark, and security. After IronPDF covers those natively, you can remove them. Run dotnet list package | grep -i "pdfsharp\|itextsharp\|pdfdocument\|dinkto" to find candidates. Verify each library was only used to supplement TuesPechkin before removing it.

What Is the Decision Framework for Your Team?

Answer: Here is the migration framework I recommend. Work through it in order: pre-migration assessment, code migration, testing, and post-migration cleanup.

Pre-Migration

Find all TuesPechkin usages with rg -l "TuesPechkin" --type cs. Count the STA thread wrappers to understand scope. Check if your secondary PDF libraries (PdfSharp, iTextSharp) are only there to supplement TuesPechkin and can be removed. Inventory all HTML templates for render comparison testing. Verify IronPDF target framework compatibility. Note all GlobalSettings and ObjectSettings properties currently in use.

Code Migration

Remove the TuesPechkin and TuesPechkin.Wkhtmltopdf.Win64 NuGet packages. Remove wkhtmltopdf binaries from Docker images and deployment scripts. Install the IronPdf NuGet package. Replace using TuesPechkin with using IronPdf. Remove the entire ThreadSafeConverter / MultiplexingConverter singleton infrastructure. Remove all STA thread boilerplate. Replace HtmlToPdfDocument + GlobalSettings + ObjectSettings with ChromePdfRenderOptions. Replace converter.Convert(doc) with renderer.RenderHtmlAsPdfAsync(html). Replace File.WriteAllBytes(path, pdfBytes) with pdf.SaveAs(path). Update header/footer tokens from [page]/[topage] to {page}/{total-pages}. Remove secondary merge/watermark/encrypt libraries. Add your IronPDF license key to CI/CD secrets.

Testing

Render each HTML template and visually compare TuesPechkin vs IronPDF output. Check that modern CSS (flex, grid) now renders correctly. Verify page size and margins match reference. Test headers and footers, paying attention to the page number token format difference. Remove the STA thread wrapper from at least one call site and verify rendering still works. Run concurrent rendering tests at 10/50/100 simultaneous requests. Verify no ThreadAbortException or lock contention errors. Test merge output for correct page order. Test watermark for opacity, position, and text. Test password protection with correct and incorrect credentials. Test in Docker on Linux. Verify Docker image builds without the wkhtmltopdf binary.

Post-Migration

Remove TuesPechkin NuGet package and all related packages. Remove ThreadSafeConverter / MultiplexingConverter singleton infrastructure. Remove wkhtmltopdf binary from Docker image copy steps. Remove secondary PDF libraries now replaced by IronPDF. Update Dockerfile with IronPDF Linux system dependencies. Verify deployment artifact size reduced. Archive wkhtmltopdf-specific header/footer HTML files for reference. Monitor memory usage in the first production week.

The immediate wins after migration — Linux/Docker support, modern CSS rendering, concurrent rendering without thread infrastructure — are measurable from day one. The STA thread boilerplate removal alone simplifies your codebase significantly.
