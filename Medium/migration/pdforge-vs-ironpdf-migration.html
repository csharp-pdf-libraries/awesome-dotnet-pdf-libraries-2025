Migrating from pdforge to IronPDF: What Actually Changes in Your Code

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Migrate from pdforge to IronPDF?

The compliance review surfaced it first. Every PDF the billing service generated was being POSTed to a third-party endpoint with the customer's HTML payload — invoice line items, addresses, the lot — and the auditor flagged it. The HtmlToPdfConverter calls in the codebase are six lines each and look harmless, but each call leaves your infrastructure, hits pdforge's servers, and comes back with a byte[]. For a small marketing site that is fine. For regulated workloads it is a problem you have to answer for.

You have two options: keep paying for pdforge and write a data-processing addendum, or move PDF generation back inside your process. I am covering the second path here — migrating from pdforge to IronPDF — so you have concrete code to evaluate before making the call.

Here are eight neutral reasons the analysis tips toward migration:

1. Data leaves your infrastructure — every ConvertHtmlString call sends HTML and any embedded data to pdforge's cloud. For PII, PHI, or financial records that is a compliance conversation you would rather not have.

2. Network latency on the hot path — HTTP round-trip + render + response per document. Local rendering removes the round-trip.

3. Subscription cost vs perpetual license — monthly fees scale with volume; a perpetual license does not.

4. Rate limits — plan-tier throttling can throttle high-volume batch jobs. Local rendering is bounded by your hardware, not a quota.

5. Outbound dependency — if pdforge has an outage or you lose egress, your PDF pipeline stops.

6. Vendor lock-in on a service contract — you depend on the provider's uptime, API stability, and pricing decisions.

7. Limited PDF manipulation — pdforge focuses on generation. Merge, split, watermark, sign, and form-fill require either supplementary libraries or are not available.

8. .NET version future-proofing — verify pdforge's .NET 8/9 client compatibility as you plan runtime upgrades.

How Do pdforge and IronPDF Compare at a Glance?

The fundamental difference is where rendering happens. pdforge is a cloud service: you instantiate a converter (or, in older client patterns, a PdfClient with an API key), call ConvertHtmlString or ConvertUrl, and the bytes come back over HTTP. Pricing is subscription-based, scope is HTML/URL-to-PDF, and data privacy depends on the vendor's terms.

IronPDF is an in-process .NET library. ChromePdfRenderer ships with a Chromium-based engine that runs locally, returns a PdfDocument object, uses 0-based page indexing, and is licensed perpetually. The namespace is IronPdf. There are no rate limits, no per-document network calls, and the API surface covers manipulation (merge, split, stamp, sign, forms, security) on top of HTML-to-PDF.

What Is the Migration Effort by Feature?

  • HTML string to PDF — Low. converter.ConvertHtmlString(html) maps directly to renderer.RenderHtmlAsPdf(html).
  • URL to PDF — Low. converter.ConvertUrl(url) maps directly to renderer.RenderUrlAsPdf(url).
  • HTML file to PDF — Low. Read the file and call RenderHtmlAsPdf, or use RenderHtmlFileAsPdf(path).
  • Page size / orientation — Low. pdforge's PageSize / Orientation properties become RenderingOptions.PaperSize / PaperOrientation.
  • Custom margins — Low. Property names match (MarginTop, MarginBottom, etc.); they live on RenderingOptions in IronPDF.
  • Headers / footers — Medium. Placeholder syntax shifts from {totalPages} to {total-pages}; HTML headers move from string properties to HtmlHeaderFooter objects.
  • Password protection — Low. pdforge accepts Password on the request; IronPDF exposes SecuritySettings on the PdfDocument.
  • Merge / split — Low (gained). pdforge does not natively merge; IronPDF ships PdfDocument.Merge and CopyPages.
  • Watermark / stamp — Low (gained). IronPDF's TextStamper / ImageStamper covers stamping; pdforge generally does not.
  • Digital signatures — Medium (gained). IronPDF's PdfSignature takes a .pfx + password; pdforge does not provide signing.
  • Async pattern — Low. pdforge clients are typically async by necessity; IronPDF is sync by default with async variants where useful.

The decision matrix is short:

  • Compliance is the primary trigger. Migration is the right call — data needs to stay local.
  • Cost is primary, low volume. Stay on pdforge if you are inside the free or cheap tier; migrate if subscription scales past license cost.
  • Manipulation features needed (merge, sign, forms). Migration consolidates onto IronPDF and removes any supplementary libraries.
  • Outbound network is unreliable or restricted. Migration removes the runtime dependency.

How Do I Set Up IronPDF After Removing pdforge?

Start by removing pdforge and installing IronPDF via NuGet. Then configure your license key and update your namespace imports.

Remove pdforge and install IronPDF

dotnet remove package PdfForge

dotnet add package IronPdf
dotnet restore

License configuration — pdforge (before)

// pdforge: API key authentication, often per-client
// using PdfForge;
// using PdfForge.Client;

// Older client pattern:
// var client = new PdfClient("your-pdforge-api-key");
//
// Newer / simpler pattern in this repo's samples:
// var converter = new HtmlToPdfConverter();
// (key picked up from configuration or constructor; verify in your version)

License configuration — IronPDF (after)

using IronPdf;

// Set once at application startup
IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";
// License guide: https://ironpdf.com/how-to/license-keys/

Namespace imports — pdforge (before)

using PdfForge;
using PdfForge.Client;   // older client-based pattern
using PdfForge.Models;   // request/options types

Namespace imports — IronPDF (after)

using IronPdf;
using IronPdf.Rendering;
using IronPdf.Editing;
using IronPdf.Security;

How Do I Convert HTML to PDF with IronPDF?

The basic HTML-to-PDF conversion is the most common migration target. I will show the pdforge pattern alongside the IronPDF replacement, followed by a version with custom rendering options.

Basic HTML-to-PDF — pdforge (before)

using PdfForge;
using System.IO;

class Program
{
    static void Main()
    {
        var converter = new HtmlToPdfConverter();
        var html = "<html><body><h1>Hello World</h1></body></html>";
        byte[] pdf = converter.ConvertHtmlString(html);
        File.WriteAllBytes("output.pdf", pdf);
    }
}

Basic HTML-to-PDF — IronPDF (after)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<html><body><h1>Hello World</h1></body></html>");
pdf.SaveAs("output.pdf");
// Guide: https://ironpdf.com/how-to/html-string-to-pdf/

HTML-to-PDF with custom options — pdforge (before)

using PdfForge;
using System.IO;

class Program
{
    static void Main()
    {
        var converter = new HtmlToPdfConverter();
        converter.PageSize    = PageSize.A4;
        converter.Orientation = PageOrientation.Landscape;

        var htmlContent = File.ReadAllText("input.html");
        byte[] pdf = converter.ConvertHtmlString(htmlContent);
        File.WriteAllBytes("invoice.pdf", pdf);
    }
}

HTML-to-PDF with custom options — IronPDF (after)

using IronPdf;
using IronPdf.Rendering;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize        = PdfPaperSize.A4;
renderer.RenderingOptions.PaperOrientation = PdfPaperOrientation.Landscape;
renderer.RenderingOptions.MarginTop        = 10;
renderer.RenderingOptions.MarginBottom     = 10;
renderer.RenderingOptions.MarginLeft       = 15;
renderer.RenderingOptions.MarginRight      = 15;

var htmlContent = System.IO.File.ReadAllText("input.html");
var pdf = renderer.RenderHtmlAsPdf(htmlContent);
pdf.SaveAs("invoice.pdf");
// Rendering options: https://ironpdf.com/how-to/rendering-options/

How Do I Convert a URL to PDF After Migrating?

URL-to-PDF is pdforge's other primary entry point. The mapping is one-for-one; the only thing that changes is that the request never leaves your process.

URL to PDF — pdforge (before)

using PdfForge;
using System.IO;

class Program
{
    static void Main()
    {
        var converter = new HtmlToPdfConverter();
        byte[] pdf = converter.ConvertUrl("https://example.com");
        File.WriteAllBytes("webpage.pdf", pdf);
    }
}

URL to PDF — IronPDF (after)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderUrlAsPdf("https://example.com");
pdf.SaveAs("webpage.pdf");
// Guide: https://ironpdf.com/how-to/url-to-pdf/

How Do I Merge PDFs After Migrating?

Merging is one of the features pdforge does not cover natively, so teams often pull in PdfSharp or iTextSharp alongside it. After migrating, the secondary library can usually go.

Merge PDFs — PdfSharp pattern alongside pdforge (before)

using PdfSharp.Pdf;
using PdfSharp.Pdf.IO;

class Program
{
    static void Main()
    {
        using var output = new PdfDocument();

        foreach (string path in new[] { "part1.pdf", "part2.pdf" })
        {
            using var input = PdfReader.Open(path, PdfDocumentOpenMode.Import);
            foreach (PdfPage page in input.Pages)
                output.AddPage(page);
        }

        output.Save("merged.pdf");
    }
}

Merge PDFs — IronPDF (after)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var merged = PdfDocument.Merge(
    PdfDocument.FromFile("part1.pdf"),
    PdfDocument.FromFile("part2.pdf")
);
merged.SaveAs("merged.pdf");
// Guide: https://ironpdf.com/how-to/merge-or-split-pdfs/

How Do I Add a Watermark with IronPDF?

pdforge does not ship a watermark API, so this is typically a feature added during migration. IronPDF's TextStamper covers text watermarks; ImageStamper covers image watermarks. Both apply through PdfDocument.ApplyStamp.

Watermark — IronPDF

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("input.pdf");
var stamper = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronSoftware.Drawing.Color.LightGray,
    FontSize = 60,
    Opacity = 30,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");
// Guide: https://ironpdf.com/how-to/custom-watermark/

Apply watermark to specific pages only

// Watermark only the first page:
pdf.ApplyStamp(textStamper, 0);

// Watermark pages 2-4 (0-based):
for (int i = 1; i <= 3; i++)
    pdf.ApplyStamp(textStamper, i);

How Do I Add Password Protection with IronPDF?

IronPDF exposes security settings directly on the PdfDocument object. There is no separate security class to instantiate — you set properties and save.

Password protection — pdforge (before)

using PdfForge;
using PdfForge.Client;
using System.IO;
using System.Threading.Tasks;

class Program
{
    static async Task Main()
    {
        var client = new PdfClient("your-api-key");

        var request = new HtmlToPdfRequest
        {
            Html = "<h1>Confidential Document</h1>",
            Password      = "secret123",   // user password
            OwnerPassword = "admin456"
        };

        byte[] pdf = await client.GenerateAsync(request);
        File.WriteAllBytes("secure.pdf", pdf);
    }
}

Password protection — IronPDF (after)

using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<h1>Confidential Document</h1>");

pdf.SecuritySettings.UserPassword              = "secret123";
pdf.SecuritySettings.OwnerPassword             = "admin456";
pdf.SecuritySettings.AllowUserPrinting         = PdfPrintSecurity.FullPrintRights;
pdf.SecuritySettings.AllowUserCopyPasteContent = false;
pdf.SaveAs("secure.pdf");
// Guide: https://ironpdf.com/how-to/pdf-permissions-passwords/

What Are the API Mapping Tables Between pdforge and IronPDF?

These mappings give you a systematic reference for every pdforge call site. I recommend exporting your call sites first and working through them row by row.

Namespace mapping

  • PdfForge → IronPdf — core namespace
  • PdfForge.Client → IronPdf / IronPdf.Rendering — renderer + render config (no separate client; license is global)
  • PdfForge.Models → IronPdf.Rendering — request/option types collapse into RenderingOptions

Core class mapping

  • HtmlToPdfConverter / PdfClient → ChromePdfRenderer — HTML and URL to PDF
  • HtmlToPdfRequest / UrlToPdfRequest → ChromePdfRenderOptions — render configuration on renderer.RenderingOptions
  • API response (byte[]) → PdfDocument — result object, with BinaryData and Stream for raw bytes

Document loading methods

  • HTML string. converter.ConvertHtmlString(html) → renderer.RenderHtmlAsPdf(html)
  • URL. converter.ConvertUrl(url) → renderer.RenderUrlAsPdf(url)
  • HTML file. converter.ConvertHtmlString(File.ReadAllText(path)) → renderer.RenderHtmlFileAsPdf(path)
  • Existing PDF. n/a in pdforge → PdfDocument.FromFile(path)

Page operations

  • Paper size. converter.PageSize = PageSize.A4 → RenderingOptions.PaperSize = PdfPaperSize.A4
  • Orientation. converter.Orientation = PageOrientation.Landscape → RenderingOptions.PaperOrientation = PdfPaperOrientation.Landscape
  • Margins. request.MarginTop etc. → RenderingOptions.MarginTop etc.
  • Page count. n/a on the request side → pdf.PageCount

Headers, footers, and placeholders

  • Plain text header / footer. request.Header = "..." / Footer = "..." → RenderingOptions.TextHeader / TextFooter as TextHeaderFooter with CenterText.
  • HTML header / footer. request.HeaderHtml / FooterHtml → RenderingOptions.HtmlHeader / HtmlFooter as HtmlHeaderFooter with HtmlFragment.
  • Total-pages placeholder. {totalPages} → {total-pages} (note the hyphen). {page}, {date}, {time}, {url} are unchanged; {title} becomes {html-title}.

Merge/split operations

  • Merge. not native in pdforge → PdfDocument.Merge(pdf1, pdf2)
  • Split. not native in pdforge → pdf.CopyPages(startIndex, endIndex)

What Critical Details Should I Watch During Migration?

Three areas consistently trip up teams migrating from pdforge: the sync vs async shift, the output model difference, and error handling now that the network is out of the picture.

Sync vs async

pdforge's PdfClient.GenerateAsync is async because every call is an HTTP request. IronPDF runs locally and is sync by default; wrap in Task.Run or use the Async variants only when you genuinely need to free a request thread:

// pdforge: always async
byte[] pdfBytes = await client.GenerateAsync(request);

// IronPDF: sync by default
var pdf = renderer.RenderHtmlAsPdf(html);

// IronPDF: async variant when you need it
var pdf = await renderer.RenderHtmlAsPdfAsync(html);

Output model

pdforge returns byte[]. IronPDF returns PdfDocument:

// IronPDF PdfDocument to byte[]:
var pdf = renderer.RenderHtmlAsPdf(html);

byte[] pdfBytes = pdf.BinaryData;       // direct
// or via stream:
using var ms = new MemoryStream();
pdf.Stream.CopyTo(ms);
byte[] alsoBytes = ms.ToArray();
// Memory stream: https://ironpdf.com/how-to/pdf-memory-stream/

Error handling

pdforge surfaces network and quota errors (RateLimitException, ApiTimeoutException, ApiAuthenticationException). After migration, those are gone — replace them with IronPDF's local exception types:

try
{
    var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs("output.pdf");
}
catch (IronPdf.Exceptions.IronPdfException ex)
{
    _logger.LogError(ex, "PDF render failed: {msg}", ex.Message);
    throw;
}

Performance — renderer reuse

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

foreach (var html in documentQueue)
{
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"doc_{Guid.NewGuid()}.pdf");
}

Linux deployment

IronPDF's Chromium engine has native dependencies on Linux. The first run downloads a Chromium build (~150MB, one-time); the runtime needs the usual headless-Chromium libraries (libnss3, libatk-bridge, libgbm, etc.). pdforge has none of this because rendering happens server-side — this is a real deployment-checklist difference, not a gotcha.

Frequently Asked Questions

What .NET versions does IronPDF support?

IronPDF targets .NET 6 and above. If your project is on .NET 6+, you are ready to go. Verify pdforge's .NET 8/9 client compatibility as you plan runtime upgrades.

How do I handle transitive dependencies after removing pdforge?

If pdforge pulled in HTTP-client or serialization packages, check whether any are still needed after migration. Run dotnet restore and commit the updated packages.lock.json if you are using locked restores. Also update any PackageReference version constraints in .csproj files rather than just removing the package.

Can IronPDF replace secondary libraries I added to supplement pdforge?

Yes. If you have added PdfSharp, iTextSharp, or other libraries to fill gaps in pdforge — merging, watermarking, signing, form-fill — IronPDF consolidates those capabilities. The merge example above shows how a PdfSharp-based merge pattern collapses into a single IronPDF call. Identify all secondary libraries during pre-migration and remove them if they were only supplementing pdforge.

Is there a like-for-like async migration path?

Mostly yes. RenderHtmlAsPdfAsync and RenderUrlAsPdfAsync mirror pdforge's GenerateAsync, so controllers that await the call do not need restructuring. The behavioural difference is that the async-ness is no longer hiding a network request — it is freeing the calling thread during a local render.

How does data privacy actually change after migration?

Every ConvertHtmlString or ConvertUrl on pdforge sends payload to the vendor's servers. After migrating to IronPDF, the same call runs in your process and the bytes never leave. For regulated workloads (PII, PHI, financial records) that is the headline change — the rest of this guide is mechanics.

Conclusion

The compliance-driven migration has an underappreciated upside: while you are pulling pdforge out, you usually find PdfSharp or iTextSharp sitting next to it covering merges, watermarks, or signing. Those can come out too. The dependency tree gets shorter, the network round-trip on every call disappears, and the audit conversation gets a lot simpler.

I have walked through the migration path from pdforge to IronPDF — removing the cloud client, configuring a local license, and replacing every ConvertHtmlString / ConvertUrl call site with the in-process equivalent, plus the manipulation features (merge, watermark, password, signing) that pdforge does not cover natively. The effort table shows that most features are low-effort migrations, with headers/footers and digital signatures sitting at medium effort.

If the analysis points to migration, the code samples and mapping reference above give you everything you need to start. If pdforge is still the right fit for your volume and compliance posture, stay there — do not migrate when staying is the right call.
