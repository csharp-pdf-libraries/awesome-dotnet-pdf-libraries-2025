Migrating from pdforge to IronPDF: What Actually Changes in Your Code

Written by Jacob Mellor, CTO at Iron Software. Jacob is the creator of IronPDF and leads teams building .NET document processing libraries that help developers at scale.

Why Migrate from pdforge to IronPDF?

The compliance review surfaced it first. Every PDF the billing service generated was being POSTed to a third-party endpoint with the customer's HTML payload — invoice line items, addresses, the lot — and the auditor flagged it. The HttpClient.PostAsync call to api.pdfnoodle.com is fifteen lines including the Bearer header and the signed-URL fetch, and looks harmless on its own. But each call leaves your infrastructure, hits the vendor's servers, and comes back with a JSON envelope you have to download a second time to get the PDF bytes. For a small marketing site that is fine. For regulated workloads it is a problem you have to answer for.

(A naming note before we go further: pdforge rebranded to "pdf noodle" in 2026, and the public docs now live at docs.pdfnoodle.com. The api.pdforge.com hostname continues to work via 301 redirects through end of 2026, so existing integrations keep running — but new code in this guide uses the api.pdfnoodle.com hostname.)

You have two options: keep paying the subscription and write a data-processing addendum, or move PDF generation back inside your process. I am covering the second path here — migrating from pdforge to IronPDF — so you have concrete code to evaluate before making the call.

Here are eight neutral reasons the analysis tips toward migration:

1. Data leaves your infrastructure — every POST /v1/html-to-pdf/sync sends HTML and any embedded data to the vendor's cloud. For PII, PHI, or financial records that is a compliance conversation you would rather not have.

2. Network latency on the hot path — HTTP round-trip + render + signed-URL download per document. Local rendering removes both round-trips.

3. Subscription cost vs perpetual license — monthly fees scale with volume; a perpetual license does not.

4. Rate limits — plan-tier throttling can throttle high-volume batch jobs. Local rendering is bounded by your hardware, not a quota.

5. Outbound dependency — if the service has an outage or you lose egress, your PDF pipeline stops.

6. Vendor lock-in on a service contract — you depend on the provider's uptime, API stability, and pricing decisions.

7. Limited PDF manipulation — pdforge focuses on generation. Merge, split, watermark, sign, and form-fill require either supplementary libraries or are not available.

8. No official .NET SDK — integration is hand-rolled HttpClient + JSON, which means request shapes, error handling, and signed-URL plumbing are all on your team to maintain.

How Do pdforge and IronPDF Compare at a Glance?

The fundamental difference is where rendering happens. pdforge is a cloud service: you POST a JSON body containing your html and a pdfParams object to https://api.pdfnoodle.com/v1/html-to-pdf/sync with a Bearer token, parse the JSON envelope that comes back, and download the PDF from the signedUrl field. There is no official .NET SDK on NuGet; integration is System.Net.Http against the documented REST endpoints. Pricing is subscription-based, scope is HTML-to-PDF, and data privacy depends on the vendor's terms.

IronPDF is an in-process .NET library. ChromePdfRenderer ships with a Chromium-based engine that runs locally, returns a PdfDocument object, uses 0-based page indexing, and is licensed perpetually. The namespace is IronPdf. There are no rate limits, no per-document network calls, and the API surface covers manipulation (merge, split, stamp, sign, forms, security) on top of HTML-to-PDF.

What Is the Migration Effort by Feature?

  • HTML string to PDF — Low. A POST to /v1/html-to-pdf/sync with { html } maps directly to renderer.RenderHtmlAsPdf(html).
  • URL to PDF — Low. pdforge has no dedicated URL endpoint — you fetch the page yourself and POST its HTML. IronPDF exposes renderer.RenderUrlAsPdf(url) directly.
  • HTML file to PDF — Low. Read the file and POST its contents, or call renderer.RenderHtmlFileAsPdf(path).
  • Page size / orientation — Low. The Chromium-style format: "A4" and landscape: true fields on pdfParams become RenderingOptions.PaperSize / PaperOrientation.
  • Custom margins — Low. The nested margin: { top: "20px", ... } object becomes plain MarginTop/MarginBottom/MarginLeft/MarginRight properties on RenderingOptions (numeric millimetres, no unit string).
  • Headers / footers — Medium. pdforge uses Chromium's HTML-template format with <span class="pageNumber"> / <span class="totalPages"> elements; IronPDF uses bracketed placeholders ({page}, {total-pages}) inside TextHeaderFooter or HtmlHeaderFooter objects.
  • Password protection — Low (gained). pdforge has no documented password parameter; teams typically encrypt downstream. IronPDF exposes SecuritySettings on the PdfDocument.
  • Merge / split — Low (gained). Not in pdforge; IronPDF ships PdfDocument.Merge and CopyPages.
  • Watermark / stamp — Low (gained). Not in pdforge; IronPDF's TextStamper / ImageStamper covers stamping.
  • Digital signatures — Medium (gained). Not in pdforge; IronPDF's PdfSignature takes a .pfx + password.
  • Async pattern — Low. pdforge calls are async by necessity (every call is HTTP); IronPDF is sync by default with async variants where useful.

The decision matrix is short:

  • Compliance is the primary trigger. Migration is the right call — data needs to stay local.
  • Cost is primary, low volume. Stay on pdforge if you are inside the free or cheap tier; migrate if subscription scales past license cost.
  • Manipulation features needed (merge, sign, forms). Migration consolidates onto IronPDF and removes any supplementary libraries.
  • Outbound network is unreliable or restricted. Migration removes the runtime dependency.

How Do I Set Up IronPDF After Removing pdforge?

Because pdforge has no .NET package on NuGet, there is nothing to dotnet remove. The migration is an additive install of IronPDF, followed by deleting your HttpClient call sites once the new code is in place.

Install IronPDF

dotnet add package IronPdf
dotnet restore

// pdforge: Bearer token sent on every HTTP request — no SDK to remove.

How Do I Convert HTML to PDF with IronPDF?

The basic HTML-to-PDF conversion is the most common migration target. Below is the pdforge REST pattern alongside the IronPDF replacement, followed by a version with custom rendering options.

Basic HTML-to-PDF — pdforge (before)

using System.IO;
using System.Net.Http;
using System.Net.Http.Headers;
using System.Text;
using System.Text.Json;
using System.Threading.Tasks;

class Program
{
    static async Task Main()
    {
        using var http = new HttpClient();
        http.DefaultRequestHeaders.Authorization =
            new AuthenticationHeaderValue("Bearer", "pdfnoodle_api_YOUR_KEY");

        var body = new { html = "<html><body><h1>Hello World</h1></body></html>" };
        var json = new StringContent(JsonSerializer.Serialize(body), Encoding.UTF8, "application/json");

        // The sync endpoint returns a JSON envelope with a signedUrl pointing to the PDF.
        var resp = await http.PostAsync("https://api.pdfnoodle.com/v1/html-to-pdf/sync", json);
        resp.EnsureSuccessStatusCode();

        using var doc = JsonDocument.Parse(await resp.Content.ReadAsStringAsync());
        var signedUrl = doc.RootElement.GetProperty("signedUrl").GetString();
        var pdfBytes = await http.GetByteArrayAsync(signedUrl);
        File.WriteAllBytes("output.pdf", pdfBytes);
    }
}

Basic HTML-to-PDF — IronPDF (after)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<html><body><h1>Hello World</h1></body></html>");
pdf.SaveAs("output.pdf");
// Guide: https://ironpdf.com/how-to/html-string-to-pdf/

HTML-to-PDF with custom options — pdforge (before)

using System.IO;
using System.Net.Http;
using System.Net.Http.Headers;
using System.Text;
using System.Text.Json;
using System.Threading.Tasks;

class Program
{
    static async Task Main()
    {
        using var http = new HttpClient();
        http.DefaultRequestHeaders.Authorization =
            new AuthenticationHeaderValue("Bearer", "pdfnoodle_api_YOUR_KEY");

        var htmlContent = File.ReadAllText("input.html");

        var body = new
        {
            html = htmlContent,
            pdfParams = new
            {
                format    = "A4",
                landscape = true,
                margin    = new { top = "10mm", bottom = "10mm", left = "15mm", right = "15mm" }
            }
        };
        var json = new StringContent(JsonSerializer.Serialize(body), Encoding.UTF8, "application/json");

        var resp = await http.PostAsync("https://api.pdfnoodle.com/v1/html-to-pdf/sync", json);
        resp.EnsureSuccessStatusCode();

        using var doc = JsonDocument.Parse(await resp.Content.ReadAsStringAsync());
        var pdfBytes = await http.GetByteArrayAsync(doc.RootElement.GetProperty("signedUrl").GetString());
        File.WriteAllBytes("invoice.pdf", pdfBytes);
    }
}

HTML-to-PDF with custom options — IronPDF (after)

using IronPdf;
using IronPdf.Rendering;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize        = PdfPaperSize.A4;
renderer.RenderingOptions.PaperOrientation = PdfPaperOrientation.Landscape;
renderer.RenderingOptions.MarginTop        = 10;
renderer.RenderingOptions.MarginBottom     = 10;
renderer.RenderingOptions.MarginLeft       = 15;
renderer.RenderingOptions.MarginRight      = 15;

var htmlContent = System.IO.File.ReadAllText("input.html");
var pdf = renderer.RenderHtmlAsPdf(htmlContent);
pdf.SaveAs("invoice.pdf");
// Rendering options: https://ironpdf.com/how-to/rendering-options/

How Do I Convert a URL to PDF After Migrating?

URL-to-PDF is where the API-shape mismatch is most visible. pdforge has no dedicated URL endpoint — the canonical pattern is to fetch the page yourself with HttpClient.GetStringAsync, then submit the resulting HTML to /v1/html-to-pdf/sync. IronPDF exposes the URL path directly as a one-liner, and the request never leaves your process.

URL to PDF — pdforge (before)

using System.IO;
using System.Net.Http;
using System.Net.Http.Headers;
using System.Text;
using System.Text.Json;
using System.Threading.Tasks;

class Program
{
    static async Task Main()
    {
        using var http = new HttpClient();
        http.DefaultRequestHeaders.Authorization =
            new AuthenticationHeaderValue("Bearer", "pdfnoodle_api_YOUR_KEY");

        // Two-step: fetch the source page, then POST its HTML.
        var sourceHtml = await http.GetStringAsync("https://example.com");

        var body = new { html = sourceHtml };
        var json = new StringContent(JsonSerializer.Serialize(body), Encoding.UTF8, "application/json");

        var resp = await http.PostAsync("https://api.pdfnoodle.com/v1/html-to-pdf/sync", json);
        resp.EnsureSuccessStatusCode();

        using var doc = JsonDocument.Parse(await resp.Content.ReadAsStringAsync());
        var pdfBytes = await http.GetByteArrayAsync(doc.RootElement.GetProperty("signedUrl").GetString());
        File.WriteAllBytes("webpage.pdf", pdfBytes);
    }
}

URL to PDF — IronPDF (after)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderUrlAsPdf("https://example.com");
pdf.SaveAs("webpage.pdf");
// Guide: https://ironpdf.com/how-to/url-to-pdf/

How Do I Merge PDFs After Migrating?

Merging is one of the features pdforge does not cover, so teams often pull in PdfSharp or iTextSharp alongside it. After migrating, the secondary library can usually go.

Merge PDFs — PdfSharp pattern alongside pdforge (before)

using PdfSharp.Pdf;
using PdfSharp.Pdf.IO;

class Program
{
    static void Main()
    {
        using var output = new PdfDocument();

        foreach (string path in new[] { "part1.pdf", "part2.pdf" })
        {
            using var input = PdfReader.Open(path, PdfDocumentOpenMode.Import);
            foreach (PdfPage page in input.Pages)
                output.AddPage(page);
        }

        output.Save("merged.pdf");
    }
}

Merge PDFs — IronPDF (after)

using IronPdf;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var merged = PdfDocument.Merge(
    PdfDocument.FromFile("part1.pdf"),
    PdfDocument.FromFile("part2.pdf")
);
merged.SaveAs("merged.pdf");
// Guide: https://ironpdf.com/how-to/merge-or-split-pdfs/

How Do I Add a Watermark with IronPDF?

pdforge does not ship a watermark API, so this is typically a feature added during migration. IronPDF's TextStamper covers text watermarks; ImageStamper covers image watermarks. Both apply through PdfDocument.ApplyStamp.

Watermark — IronPDF

using IronPdf;
using IronPdf.Editing;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var pdf = PdfDocument.FromFile("input.pdf");
var stamper = new TextStamper
{
    Text = "DRAFT",
    FontColor = IronSoftware.Drawing.Color.LightGray,
    FontSize = 60,
    Opacity = 30,
    Rotation = 45,
    VerticalAlignment = VerticalAlignment.Middle,
    HorizontalAlignment = HorizontalAlignment.Center
};
pdf.ApplyStamp(stamper);
pdf.SaveAs("watermarked.pdf");
// Guide: https://ironpdf.com/how-to/custom-watermark/

Apply watermark to specific pages only

// Watermark only the first page:
pdf.ApplyStamp(textStamper, 0);

// Watermark pages 2-4 (0-based):
for (int i = 1; i <= 3; i++)
    pdf.ApplyStamp(textStamper, i);

How Do I Add Password Protection with IronPDF?

pdforge does not expose password protection or encryption in its documented HTML-to-PDF parameters, so teams generally generate the PDF, download it, and encrypt it locally with a separate library. After migration that two-step pattern collapses: IronPDF exposes security settings directly on the PdfDocument object — you set properties and save.

Password protection — pdforge (before)

using System.IO;
using System.Net.Http;
using System.Net.Http.Headers;
using System.Text;
using System.Text.Json;
using System.Threading.Tasks;

class Program
{
    static async Task Main()
    {
        using var http = new HttpClient();
        http.DefaultRequestHeaders.Authorization =
            new AuthenticationHeaderValue("Bearer", "pdfnoodle_api_YOUR_KEY");

        var body = new { html = "<h1>Confidential Document</h1>" };
        var json = new StringContent(JsonSerializer.Serialize(body), Encoding.UTF8, "application/json");

        var resp = await http.PostAsync("https://api.pdfnoodle.com/v1/html-to-pdf/sync", json);
        resp.EnsureSuccessStatusCode();

        using var doc = JsonDocument.Parse(await resp.Content.ReadAsStringAsync());
        var pdfBytes = await http.GetByteArrayAsync(doc.RootElement.GetProperty("signedUrl").GetString());
        File.WriteAllBytes("secure.pdf", pdfBytes);

        // Encrypt secure.pdf with a separate library — pdforge has no built-in
        // user/owner password parameter on its HTML-to-PDF endpoint.
    }
}

Password protection — IronPDF (after)

using IronPdf;
using IronPdf.Security;

IronPdf.License.LicenseKey = "YOUR-LICENSE-KEY";

var renderer = new ChromePdfRenderer();
var pdf = renderer.RenderHtmlAsPdf("<h1>Confidential Document</h1>");

pdf.SecuritySettings.UserPassword              = "secret123";
pdf.SecuritySettings.OwnerPassword             = "admin456";
pdf.SecuritySettings.AllowUserPrinting         = PdfPrintSecurity.FullPrintRights;
pdf.SecuritySettings.AllowUserCopyPasteContent = false;
pdf.SaveAs("secure.pdf");
// Guide: https://ironpdf.com/how-to/pdf-permissions-passwords/

What Are the API Mapping Tables Between pdforge and IronPDF?

These mappings give you a systematic reference for every pdforge call site. I recommend grepping for api.pdfnoodle.com / api.pdforge.com first and working through the call sites row by row.

Endpoint and class mapping

  • HttpClient + Authorization: Bearer → ChromePdfRenderer — main PDF generator (no client object to construct, license is global)
  • JSON request body { html, pdfParams } → renderer.RenderingOptions — render configuration moves from a serialized object to a typed property bag
  • POST /v1/html-to-pdf/sync with { html } → renderer.RenderHtmlAsPdf(html) — HTML-string conversion
  • Fetch URL via HttpClient.GetStringAsync, then POST → renderer.RenderUrlAsPdf(url) — URL conversion (one step instead of two)
  • File.ReadAllText(path) then POST → renderer.RenderHtmlFileAsPdf(path) — HTML-file conversion
  • JSON envelope with signedUrl → PdfDocument — result object exposing BinaryData and Stream for raw bytes; no second download required

Page operations

  • Paper size. pdfParams.format = "A4" → RenderingOptions.PaperSize = PdfPaperSize.A4
  • Orientation. pdfParams.landscape = true → RenderingOptions.PaperOrientation = PdfPaperOrientation.Landscape
  • Margins. pdfParams.margin = { top: "20px", ... } → RenderingOptions.MarginTop etc. (numeric millimetres, no unit string)
  • Custom page size. pdfParams.width / height (with unit) → RenderingOptions.SetCustomPaperSizeInInches(...) or SetCustomPaperSizeInMillimeters(...)
  • Print backgrounds. pdfParams.printBackground = true → RenderingOptions.PrintHtmlBackgrounds = true
  • Page count. n/a on the request side → pdf.PageCount

Headers, footers, and placeholders

This is the area with the largest behavioural shift. pdforge inherits Chromium's HTML-template format: header and footer are HTML fragments containing <span class="pageNumber"></span> / <span class="totalPages"></span> elements that the engine substitutes at render time, and they only render when pdfParams.displayHeaderFooter is true. IronPDF uses bracketed placeholders inside either a TextHeaderFooter.CenterText or an HtmlHeaderFooter.HtmlFragment, and visibility is implicit when the property is set.

The substitution mapping:

  • <span class="pageNumber"></span> → {page}
  • <span class="totalPages"></span> → {total-pages} (note the hyphen)
  • <span class="date"></span> → {date}
  • (no built-in time class) → {time}
  • <span class="title"></span> → {html-title}
  • <span class="url"></span> → {url}

Merge/split operations

  • Merge. not native in pdforge → PdfDocument.Merge(pdf1, pdf2)
  • Split. not native in pdforge → pdf.CopyPages(startIndex, endIndex)

What Critical Details Should I Watch During Migration?

Three areas consistently trip up teams migrating from pdforge: the sync vs async shift, the output model difference, and error handling now that the network is out of the picture.

Sync vs async

Every pdforge call is async because every call is HTTP. IronPDF runs locally and is sync by default; wrap in Task.Run or use the Async variants only when you genuinely need to free a request thread:

// pdforge: always async (HTTP POST + signed-URL fetch)
var resp = await http.PostAsync("https://api.pdfnoodle.com/v1/html-to-pdf/sync", json);

// IronPDF: sync by default
var pdf = renderer.RenderHtmlAsPdf(html);

// IronPDF: async variant when you need it
var pdf = await renderer.RenderHtmlAsPdfAsync(html);

Output model

pdforge returns a JSON envelope; you parse out signedUrl and download the PDF bytes from it. IronPDF returns PdfDocument directly:

// IronPDF PdfDocument to byte[]:
var pdf = renderer.RenderHtmlAsPdf(html);

byte[] pdfBytes = pdf.BinaryData;       // direct
// or via stream:
using var ms = new MemoryStream();
pdf.Stream.CopyTo(ms);
byte[] alsoBytes = ms.ToArray();
// Memory stream: https://ironpdf.com/how-to/pdf-memory-stream/

Error handling

pdforge errors arrive as HTTP status codes — 401 Unauthorized for a bad Bearer token, 429 Too Many Requests for rate-limit hits, and HttpRequestException / TaskCanceledException for transport problems. After migration, those are gone — replace them with IronPDF's local exception types:

try
{
    var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs("output.pdf");
}
catch (IronPdf.Exceptions.IronPdfException ex)
{
    _logger.LogError(ex, "PDF render failed: {msg}", ex.Message);
    throw;
}

Performance — renderer reuse

var renderer = new ChromePdfRenderer();
renderer.RenderingOptions.PaperSize = IronPdf.Rendering.PdfPaperSize.A4;

foreach (var html in documentQueue)
{
    using var pdf = renderer.RenderHtmlAsPdf(html);
    pdf.SaveAs($"doc_{Guid.NewGuid()}.pdf");
}

Linux deployment

IronPDF's Chromium engine has native dependencies on Linux. The first run downloads a Chromium build (~150MB, one-time); the runtime needs the usual headless-Chromium libraries (libnss3, libatk-bridge, libgbm, etc.). pdforge has none of this because rendering happens server-side — this is a real deployment-checklist difference, not a gotcha.

Frequently Asked Questions

What .NET versions does IronPDF support?

IronPDF targets .NET 6 and above. If your project is on .NET 6+, you are ready to go. pdforge integration is just System.Net.Http and System.Text.Json, so the runtime is whatever your project already targets — there is no SDK version to coordinate.

How do I handle transitive dependencies after removing pdforge?

Because there was never a NuGet package, the cleanup is mostly about your own code: delete the HttpClient call sites, the JSON request models, and any System.Text.Json wrappers you wrote around them. Run dotnet restore and commit the updated packages.lock.json if you are using locked restores.

Can IronPDF replace secondary libraries I added to supplement pdforge?

Yes. If you have added PdfSharp, iTextSharp, qpdf, or others to fill gaps in pdforge — merging, watermarking, signing, password-protecting downloaded files — IronPDF consolidates those capabilities. The merge example above shows how a PdfSharp-based merge pattern collapses into a single IronPDF call. Identify all secondary libraries during pre-migration and remove them if they were only supplementing pdforge.

Is there a like-for-like async migration path?

Mostly yes. RenderHtmlAsPdfAsync and RenderUrlAsPdfAsync mirror the shape of an awaited HttpClient.PostAsync, so controllers that await the call do not need restructuring. The behavioural difference is that the async-ness is no longer hiding a network request — it is freeing the calling thread during a local render.

How does data privacy actually change after migration?

Every POST to api.pdfnoodle.com sends your payload to the vendor's servers, and the resulting PDF is held on their storage long enough for you to fetch it from a signed URL. After migrating to IronPDF, the same call runs in your process and the bytes never leave. For regulated workloads (PII, PHI, financial records) that is the headline change — the rest of this guide is mechanics.

Conclusion

The compliance-driven migration has an underappreciated upside: while you are pulling the pdforge HttpClient code out, you usually find PdfSharp or iTextSharp sitting next to it covering merges, watermarks, or post-download encryption. Those can come out too. The dependency tree gets shorter, the network round-trip on every call disappears, and the audit conversation gets a lot simpler.

I have walked through the migration path from pdforge to IronPDF — deleting the Bearer-token plumbing, replacing every POST /v1/html-to-pdf/sync + signed-URL fetch with the in-process equivalent, and picking up the manipulation features (merge, watermark, password, signing) that the REST API does not cover. The effort table shows that most features are low-effort migrations, with headers/footers and digital signatures sitting at medium effort.

If the analysis points to migration, the code samples and mapping reference above give you everything you need to start. If pdforge is still the right fit for your volume and compliance posture, stay there — do not migrate when staying is the right call.
